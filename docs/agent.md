# Agent 概念与基于 LLM 的 Agent 开发

## 1) 是什么（What）

### 1.1 Agent 的定义
Agent（智能体）可以理解为：**能够围绕目标自主执行任务的系统**。  
它通常具备以下能力：
- 感知输入（用户问题、系统状态、外部数据）
- 任务分解与规划
- 调用工具（搜索、数据库、代码执行、API 等）
- 根据结果反思并迭代
- 最终交付可执行结果

在 LLM 语境下，Agent 常被实现为“**LLM + 工具 + 控制流程**”的组合，而不是单次问答模型。

### 1.2 常见术语
- **LLM**：Large Language Model，大语言模型
- **Prompt**：给模型的指令与上下文
- **Tool Calling / Function Calling**：模型按约定结构请求调用外部工具
- **Planner**：负责拆解任务、生成步骤的模块
- **Executor**：执行具体步骤（可能调用工具）
- **Memory**：记忆机制（短期会话记忆、长期知识记忆）
- **RAG**：检索增强生成，从外部知识库检索后再生成
- **Observation**：工具返回结果或环境反馈
- **Reflection / Self-Critique**：对中间结果进行反思修正

### 1.3 Agent 与 ChatBot 的关系
- ChatBot 偏重“对话回答”
- Agent 偏重“完成任务”
- 一个系统可以同时具备两者能力，但 Agent 需要更强的流程控制与外部执行能力

---

## 2) 为什么（Why）

### 2.1 它解决了什么问题
- 单轮问答无法处理复杂、跨步骤任务
- 模型参数内知识可能过时，需要外部检索和实时数据
- 仅靠自然语言生成结果不可靠，需要可验证的工具执行
- 企业场景需要“从问题到结果”的自动化链路，而不仅是“回答建议”

### 2.2 核心价值
- **自动化复杂任务**：把多步骤流程变成可复用能力
- **提升准确性**：通过工具与检索降低幻觉
- **可扩展性强**：新增工具即可拓展能力边界
- **可追踪与可审计**：每步决策、调用、结果都有日志
- **人机协同**：在关键步骤做人类确认，兼顾效率与安全

---

## 3) 怎么做（How）

### 3.1 典型开发流程
1. 明确目标与边界（输入、输出、成功标准、失败策略）
2. 定义工具集合（可调用 API、数据库、检索、执行器）
3. 设计状态机/流程（计划 -> 执行 -> 观察 -> 反思 -> 收敛）
4. 设计 Prompt（系统规则、角色、工具使用约束、输出格式）
5. 实现执行循环（控制最大步骤、超时、重试、降级）
6. 引入记忆机制（会话上下文 + 持久化知识）
7. 接入可观测性（日志、Tracing、Token 成本、错误率）
8. 做评测与对抗测试（正确率、稳定性、安全性、成本）

### 3.2 最小可行架构（MVP）
- **输入层**：用户请求 + 上下文
- **规划层**：LLM 生成执行步骤或下一动作
- **工具层**：标准化工具接口（入参校验、权限控制）
- **执行层**：按步骤调用工具并记录结果
- **总结层**：LLM 汇总最终答案与证据
- **治理层**：安全策略、预算控制、审计日志

### 3.3 任务规划常见策略
- **Plan-and-Execute**：先出全局计划，再逐步执行
- **ReAct**：思考与行动交替进行，边做边修正
- **Tree/Graph of Thoughts**：并行探索多条推理路径后择优
- **Workflow + Agent 混合**：确定性流程中嵌入 Agent 节点

## 4) 何时用（When）

### 4.1 适用场景
- 任务步骤多、依赖外部系统、且结果需落地执行
- 需要跨系统编排（如“查数据 -> 分析 -> 生成报告 -> 发邮件”）
- 需要持续迭代直到满足目标（而不是一次生成）
- 需要工具可验证结果（如 SQL 查询、代码执行、API 调用）

### 4.2 触发条件
- 单次 Prompt 无法稳定完成任务
- 业务流程存在明显可自动化步骤
- 结果正确性可通过外部工具验证

---

## 5) 何时不用（When not）

### 5.1 不建议使用 Agent 的情况
- 需求是单步静态问答，普通 RAG 或模板化调用即可
- 业务对确定性要求极高且容错极低（优先纯规则引擎）
- 工具权限敏感且风险不可控
- 成本和时延预算非常严格，不允许多轮推理与调用

### 5.2 常见边界
- Agent 不是“万能自动化”，需要清晰约束
- 不能把“高风险最终决策”完全交给模型
- 外部系统质量差时，Agent 表现会被上游限制

---

## 6) 与谁像（Compare）

### 6.1 与普通 LLM 问答
- 相同点：都依赖语言模型推理与生成
- 不同点：Agent 增加了目标驱动、工具执行与闭环反馈

### 6.2 与 RAG
- 相同点：都可引入外部知识提升正确率
- 不同点：RAG核心是“检索+生成”；Agent 核心是“规划+执行+反馈”

### 6.3 与传统工作流引擎
- 相同点：都可编排多步骤任务
- 不同点：工作流是确定性节点，Agent 对不确定问题自适应更强

### 6.4 与规则引擎
- 相同点：都可做决策控制
- 不同点：规则引擎可解释性强但泛化弱；Agent 泛化强但可控性需额外治理

---

## 7) 常见坑（Pitfalls）

### 7.1 误区
- 误把 Agent 当“更聪明的聊天机器人”
- 认为接入工具后就一定正确
- 忽视成本与时延，只关注功能可用

### 7.2 常见错误用法
- 没有定义“停止条件”，导致无限循环
- 工具接口不做参数校验，产生错误调用
- 缺少权限隔离，出现越权风险
- 缺乏失败重试与降级策略，系统不稳定

### 7.3 失败案例模式
- Prompt 过度复杂且无结构，导致行为漂移
- 工具返回数据未标准化，LLM 无法稳定理解
- 没有评测集与回归测试，更新后质量退化

## 8) 技术实现的底层原理

### 8.1 核心执行循环（Agent Loop）
Agent 本质上是一个“感知-决策-行动-反馈”的循环：
1. 读取当前状态（用户请求、历史、工具结果）
2. 用 LLM 决定下一步动作（回答、调用工具、继续规划）
3. 执行动作并得到观察结果
4. 更新状态并判断是否结束
5. 未结束则继续循环

### 8.2 为什么 LLM 能“调用工具”
模型并不直接执行函数，它做的是：
- 按协议输出结构化调用意图（函数名 + 参数）
- 运行时框架解析该结构并真实执行工具
- 将工具结果再回填给模型进行下一步推理

因此，**工具执行能力来自框架，决策能力来自模型**。

### 8.3 记忆机制原理
- **短期记忆**：当前会话上下文（最近对话与中间结果）
- **长期记忆**：向量库或数据库中的历史信息
- **检索机制**：通过 embedding 相似度召回相关片段，再注入上下文

### 8.4 可控性与安全原理
- **策略约束**：系统 Prompt、工具白名单、权限隔离
- **运行约束**：步数上限、时间上限、成本预算
- **输出约束**：结构化输出、敏感信息过滤
- **审计追踪**：记录每次决策和工具调用链路

### 8.5 评测与优化原理
- 用任务集评估“成功率、正确率、延迟、成本、稳定性”
- 对失败样本做归因（规划错、调用错、检索错、总结错）
- 通过 Prompt、工具设计、流程控制、模型选择迭代优化

---

## 9) 一个实用落地建议（从 0 到 1）

如果你要在业务里落地 Agent，建议先做一个“单场景 MVP”：
- 选择高价值、低风险、流程清晰的任务
- 先接 1-2 个关键工具，不要一次接太多
- 保留人工确认节点，逐步提升自动化比例
- 先把可观测性和评测体系搭好，再扩展场景

这样能在可控风险下，快速验证 Agent 的真实业务价值。
