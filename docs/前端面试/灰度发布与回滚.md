# 灰度发布与回滚专题

## 使用方式

这个专题真正考察的是，团队有没有把发布当成稳定性工程来建设，而不是只会点发布按钮。成熟回答通常会覆盖发布策略、风险点、技术手段、故障处置和复盘治理这几个阶段。

---

## 一、基础认知

1. 什么是灰度发布？
答：灰度发布就是不把一个新版本一次性推给所有用户，而是按比例、按人群、按地区、按渠道或按环境逐步放量，让团队在可控范围内观察真实表现。它的本质不是“慢一点发”，而是把一次高风险的全量变更拆成多个可观察、可中止、可回退的小步骤，用更小的影响面去验证新版本是否真的稳定。

2. 为什么前端也必须理解灰度发布？
答：因为前端发布直接面向最终用户，而且前端的问题往往会马上变成“页面打不开”“按钮点不了”“资源加载失败”这种高感知故障。很多人误以为灰度主要是后端的事情，但前端同样会遇到缓存不一致、静态资源失效、接口不兼容、浏览器兼容问题，所以如果前端只会构建和上传，而不理解灰度和回滚，线上风险其实非常高。

3. 灰度发布主要解决什么问题？
答：它主要解决的是两个问题，一是降低新版本一次性扩散到全量用户的风险，二是让团队能够在真实流量和真实设备环境中尽早发现测试环境不容易暴露的问题。很多性能波动、浏览器兼容问题、配置异常和缓存问题，在本地与测试环境都可能表现正常，只有放到小规模真实用户上，风险才会被提前看见。

4. 灰度发布和正式发布是什么关系？
答：灰度不是正式发布的对立面，而是正式发布过程里最重要的一段风险控制机制。理想流程通常不是“开发完直接全量”，而是先在小范围流量上验证，再结合监控和反馈逐步放量，最后才到全量，所以正式发布可以理解成一个阶段性过程，而灰度是这个过程中负责探路和止损的关键环节。

5. 前端灰度发布相比后端有什么特殊性？
答：前端最大的特殊性在于代码运行在用户浏览器里，静态资源还会经过 CDN、浏览器缓存和各种中间代理，这意味着用户究竟拿到哪个版本并不总是像后端切流那样可控。也就是说，前端发布不只是“切一个服务版本”，而是要同时考虑 HTML、JS、CSS、资源清单、接口契约和缓存传播是否保持一致。

6. 面试里如何解释“发布风险”？
答：我会把发布风险拆成代码逻辑风险、静态资源风险、缓存风险、浏览器兼容风险、接口兼容风险、配置风险和操作风险几个层面。这样回答的好处是，能体现你理解发布不是“代码写得对不对”这么单一，而是一次上线过程中多个环节可能叠加放大问题，真正成熟的 owner 视角会主动识别这些风险是如何联动的。

7. 什么样的需求更应该做灰度？
答：凡是涉及核心链路、用户覆盖面大、依赖复杂、改动范围广、引入新基础设施、存在较强不确定性或者可能影响关键转化的需求，都更适合做灰度。并不是所有小改动都必须走同样重的流程，但风险越高、回滚越复杂、影响越广的变更，就越应该用更谨慎的发布策略去保护线上。

8. 灰度放量时一般看什么？
答：我通常会重点看错误率、白屏率、资源加载失败率、接口成功率、关键转化、核心路径性能指标和用户反馈，并且一定会结合版本维度和历史基线去对比。灰度不是“先放一点，没炸就继续”，而是一个持续判断的过程，要看新版本相对旧版本是否出现明显恶化，而不是只看绝对值有没有超过某个阈值。

## 二、灰度、AB 实验与特性开关

9. 灰度发布和 `AB` 实验有什么区别？
答：灰度发布的主要目标是控制上线风险、验证稳定性，核心问题是“新版本能不能安全上线”；`AB` 实验更偏产品策略和效果验证，核心问题是“哪个方案的业务表现更好”。两者都可能使用分流能力，但一个强调风险控制，一个强调方案对比，如果概念不分清，发布和实验很容易相互干扰。

10. 为什么不能把灰度和 `AB` 实验混为一谈？
答：因为一旦把版本灰度和实验分桶混在一起，出了问题时你很难判断到底是代码故障、灰度放量策略异常，还是实验本身的设计导致效果变差。成熟团队通常会尽量把“发布控制”和“业务实验”分层处理，哪怕它们都依赖同一套分流平台，也会在概念、配置和观测上区分清楚。

11. 什么是特性开关 `Feature Flag`？
答：特性开关本质上是一种把“代码是否已发布”和“功能是否对用户生效”拆开的机制，也就是代码可以先进入线上，但功能是否真正打开由远端配置控制。这样做的意义非常大，因为它把发布和启用解耦了，团队可以先把代码安全送到生产环境，再按小范围逐步开启，出了问题也能更快关停。

12. 特性开关对前端有什么价值？
答：前端很多风险并不是整个版本都坏了，而是某个新入口、某个新弹窗、某个实验模块或某段新交互逻辑出了问题。这个时候如果只能整包回滚，代价会很大；而有了特性开关，就能把故障收敛到功能粒度，快速关闭有问题的部分，保留其他已经稳定的改动，这对发布效率和止血速度都很有帮助。

13. 特性开关应该怎样设计才不会失控？
答：开关系统最怕的是只加不清，最后变成一堆谁也不敢删的历史债务，所以必须配合命名规范、归属人、创建原因、失效时间和清理机制。成熟的开关体系不只是“支持开和关”，而是能回答这个开关是为什么存在、影响谁、什么时候该删除，以及出了问题谁负责处理。

14. 灰度放量通常按什么维度切流？
答：常见的切流维度包括用户 id、设备 id、地区、渠道、白名单、账号等级和租户维度，选择哪种方式取决于业务形态和风险控制目标。这里一个很重要的原则是规则必须稳定可复现，也就是同一个用户在同一阶段最好落到同一个桶里，否则今天看到新版本、明天又回到旧版本，不仅体验混乱，也不利于问题定位。

15. 前端如何拿到灰度或实验配置？
答：常见做法包括通过配置中心下发、由 BFF 注入首屏配置、服务端渲染时注水，或者在首屏关键接口里返回配置结果。关键点不在“从哪里拿”，而在于拿得够早、失败时有默认值、缓存策略可控，否则页面渲染和开关判断可能相互打架，导致用户先看到一种状态，随后又被切成另一种状态。

16. 为什么说特性开关是一种“线上止血手段”？
答：因为它允许团队在不重新发版的前提下，快速关闭某个高风险功能，把影响面尽快收住。对于前端来说，重新构建、重新发 CDN、等待缓存传播往往没有关闭远端开关那么快，所以当主站整体正常、只是某个新功能导致大量报错时，开关往往就是最有效的第一道止血手段。

17. 做 `AB` 实验时为什么也要考虑发布风险？
答：因为实验不是只改一个文案或颜色，它常常会带来新代码、新资源、新埋点、新接口依赖，如果只盯转化提升而忽略稳定性，实验本身也完全可能变成一次线上事故。真正成熟的实验体系会把效果指标和稳定性指标同时纳入守护条件，也就是说，不仅要看谁转化高，还要看谁更稳。

18. 如果面试官问“灰度和开关有什么关系”，怎么回答？
答：我会说灰度是一种发布策略，解决的是“新版本先给谁、放多少、怎么逐步扩散”；特性开关是一种实现和控制手段，解决的是“某段功能逻辑是否真正生效”。它们经常配合使用，但不完全等价，把这层关系讲清楚，会显得你对发布系统的理解更分层。

## 三、版本不一致与静态资源缓存

19. 为什么前端比后端更容易遇到版本不一致问题？
答：因为后端版本通常运行在服务端，切流路径更集中，而前端代码散落在用户浏览器里，HTML、JS、CSS、接口和配置还可能分别来自不同时间点的缓存和 CDN 节点。结果就是同一次发布期间，线上往往会同时存在多个前端版本，这种“混合态”是前端发布天然要面对的现实。

20. 什么是前端版本不一致？
答：前端版本不一致指的是同一个用户最终拿到的 HTML、主包、异步 chunk、配置或接口语义没有落在同一个版本快照上，比如新 HTML 引用了旧 JS、旧页面逻辑访问了新接口字段、主包更新了但按需加载的 chunk 还是旧的。很多前端线上事故的本质，都是这类看不见的版本组合出了问题。

21. 静态资源缓存为什么既是性能手段又是发布风险来源？
答：缓存可以大幅降低资源加载时间和源站压力，所以它对前端性能非常重要，但缓存也意味着资源更新传播不是瞬时完成的，而是渐进发生的。只要资源命名、缓存策略或清理策略设计得不够稳妥，就会出现一部分用户拿到新资源、一部分用户还停留在旧资源的情况，从而把性能优化手段变成发布风险来源。

22. 为什么前端静态资源通常要做文件名 hash？
答：因为 `hash` 能让“资源内容变化”和“资源 URL 变化”绑定在一起，浏览器和 CDN 看到新 URL 就会把它当成新资源拉取，而不会继续命中旧缓存。它的底层思想其实很简单，就是用内容指纹避免脏缓存，让带 `hash` 的 JS、CSS 可以安全地长缓存，同时保证新版本仍然能够准确生效。

23. 为什么入口 HTML 的缓存策略往往和 JS/CSS 不一样？
答：因为入口 HTML 往往承担“告诉浏览器当前应该加载哪一组资源”的角色，它本身像一份资源索引，所以更适合短缓存或协商缓存，以便尽快指向新的版本清单；而带 `hash` 的 JS/CSS 文件内容稳定、路径唯一，更适合长缓存。两者配合起来，才能同时兼顾资源更新及时生效和缓存性能收益。

24. 什么是“旧 HTML 引用新资源”和“新 HTML 引用旧资源”问题？
答：这两种情况本质上都是版本快照不一致，只是错位方向不同。旧 HTML 引用新资源，可能导致页面逻辑和资源清单不匹配；新 HTML 引用旧资源，则可能出现新模板依赖的逻辑根本不存在，或者加载到错误的 chunk，最终都可能表现为运行时报错、白屏或者某些页面无法进入，所以发布时要尽量让一整套构建产物以完整快照协同生效。

25. 异步 chunk 更新为什么容易出线上故障？
答：因为单页应用里用户可能在页面停留很久，首次进站时加载的是主包，但某个二级页面对应的异步 chunk 可能在十分钟后才请求。此时如果平台已经发布了新版本、旧 chunk 又被清掉或清单已切换，就容易出现动态导入失败，表现成“刚才还好好的，一点某个页面就白屏”，这类问题非常典型。

26. 如何降低 chunk 加载失败的风险？
答：通常可以通过保留旧版本静态资源一段时间、让发布平台支持资源保留窗口、为 chunk 加载失败增加刷新或回退兜底、避免过早清理旧清单来降低风险。这里的核心思想是承认前端线上存在“用户停留很久才去加载后续资源”的现实，而不是假设用户一定会瞬间切到最新快照。

27. 什么是前端发布中的接口兼容风险？
答：接口兼容风险指的是前端新版本依赖的新字段、新状态或新协议还没有在后端完全准备好，或者旧版本前端还在请求的字段和接口已经被后端删掉。因为前端线上天然会同时存在多个版本，所以后端接口往往不能只面向“当前最新版前端”设计，而要考虑一个过渡窗口内的多版本兼容。

28. 如果线上存在多个前端版本，后端接口应该如何配合？
答：更成熟的做法是为前端预留版本过渡期，保证旧版本在一段时间内仍能正常工作，新字段按增量兼容方式引入，破坏性变更则延后处理。前后端协作真正需要的是共同设计“升级窗口”和“淘汰节奏”，而不是简单地互相等版本或赌用户都会立刻刷新到最新页面。

## 四、回滚机制与发布风险控制

29. 什么是回滚？前端回滚和后端回滚有什么不同？
答：前端回滚可以理解成把用户可见版本尽快切回一个已知稳定的状态，它可能涉及 HTML、资源清单、静态资源、远端配置和特性开关，而不只是“把代码回退一个 commit”。和后端相比，前端回滚最大的难点在于缓存传播不可瞬间统一，所以即使平台已经切回旧版本，也不代表所有用户立刻都恢复到了同一状态。

30. 前端回滚一般有哪些手段？
答：常见手段包括回切入口 HTML 或资源清单、关闭特性开关、回退远端配置、切回上一个稳定构建版本，以及在必要时刷新 CDN 缓存或恢复旧资源访问。不同故障适合的手段不同，回滚的核心不是动作形式统一，而是尽快把用户恢复到稳定体验，所以不能把“回滚”理解成只有一种固定操作。

31. 为什么说前端回滚不一定比重新发版更快？
答：因为有些问题只是某个功能开关开错了，直接关开关会比整站回滚快得多；有些问题是缓存导致的，即使你回滚了代码，用户也不一定马上命中旧资源。前端应急真正追求的是“最快恢复可用性”，而不是形式上一定要执行一个名叫回滚的动作，所以有时局部修复、快速补发甚至比传统回滚更有效。

32. 发布前应该做哪些风险控制？
答：我会在发布前做发布清单检查、核心链路回归、接口兼容确认、静态资源和资源清单校验、监控告警预设、灰度计划确认以及回滚预案确认。成熟发布不是“测试通过就上线”，而是上线前就明确如果出问题看哪些指标、由谁负责决策、能用什么方式最快止血。

33. 为什么发布预案和回滚预案同样重要？
答：因为真正线上出问题时，团队最缺的不是技术点子，而是时间和冷静，如果没有预案，大家很容易在高压下临时试错，反而扩大影响。提前准备好故障分级、负责人、开关位置、回滚目标版本、关键看板和观察指标，能把应急从慌乱操作变成标准化响应。

34. 如何判断应该继续放量还是停止灰度？
答：判断标准通常不是“有没有完全挂掉”，而是新版本相对旧版本和历史基线是否出现明显恶化，尤其要看错误率、白屏率、资源失败率、核心转化和关键性能指标是否有显著异常。如果核心路径已经明显变差，就不该抱着侥幸心理继续放量，灰度的意义本来就是尽早止损，而不是硬着头皮往前推。

35. 前端发布平台最值得建设哪些能力？
答：我会优先建设版本管理、灰度分流、发布记录、特性开关联动、资源清单原子切换、监控看板联动、快速回切和旧资源保留策略。发布平台的真正价值不只是“帮大家点按钮更方便”，而是把经验性的发布和回滚能力沉淀成平台化、可复用、可审计的稳定性基础设施。

36. 什么是“原子发布”思想？
答：所谓原子发布，并不是前端一定能做到像数据库事务那样绝对原子，而是尽量让一个版本的 HTML、资源清单、静态文件和配置以完整快照的方式一起生效，减少跨版本混搭。这个思想的核心，是尽量让用户拿到的始终是一组彼此匹配的产物，而不是东一块新版本、西一块旧版本。

37. 发布过程中为什么要特别关注监控和告警窗口？
答：因为很多前端故障会在发版后的几分钟内快速暴露出来，如果这个窗口没人盯、告警没接通、指标看板也没有版本对比，问题就可能在团队还没意识到时继续扩散到更大范围。发布并不是点完按钮就结束，真正危险的阶段往往是发布后那段短但关键的观察窗口。

38. 如果面试官问“你如何设计一个安全的前端发布流程”，怎么答？
答：我会从发布前检查、核心链路回归、按批灰度、指标守护、特性开关配合、资源快照管理、快速回滚和复盘治理几个阶段去讲。这样回答的重点，不是展示自己会不会发包，而是体现你理解前端发布本质上是在做一套线上风险控制流程。

## 五、线上故障止血与应急处理

39. 线上故障出现后，前端第一步应该做什么？
答：第一步通常不是立刻改代码，而是先判断影响范围和故障类型，例如是全站白屏、某个路由异常、静态资源 404、配置错误、接口不兼容，还是只影响某个地区、浏览器或渠道。只有先把问题归到大致正确的层面，后续止血动作才不会乱，否则很容易边查边试，把故障拖得更久。

40. 什么叫“止血”？
答：止血就是先把故障扩散停下来，尽快恢复用户基本可用性，再去做更彻底的根因修复。它强调的是顺序感，线上应急时最忌讳的就是一边让影响继续扩大，一边直接在生产环境里试错，所以止血本质上是一种优先级判断能力。

41. 前端线上故障常见的止血手段有哪些？
答：常见手段包括暂停继续放量、关闭特性开关、回切稳定版本、回退远端配置、恢复旧资源、临时降级高风险入口、增加刷新兜底以及必要时引导用户重新加载。止血手段没有固定模板，关键是先判断问题落在代码、配置、资源、缓存还是外部依赖，再选择影响最小、恢复最快的方案。

42. 如果是静态资源 404 或 chunk 加载失败，通常怎么止血？
答：这类问题一般会优先确认旧资源是否被过早清理、HTML 或清单是否引用错版本、CDN 缓存是否异常，然后尽快恢复旧资源访问、回切到稳定的资源清单，并在必要时给前端增加刷新或回退兜底。因为这类故障很多时候不是业务逻辑有 bug，而是资源快照被打散了，所以处理重点也应放在恢复正确资源组合上。

43. 如果是某个新功能导致大量报错，但主站其他功能正常，怎么处理更合适？
答：这种情况下通常更合适的做法是优先关掉对应特性开关、隐藏入口或局部降级，而不是整站回滚。原因很简单，故障既然是局部的，止血最好也保持局部，这样既能快速恢复大多数用户体验，也能避免把其他已经稳定的新改动一起撤掉。

44. 故障处理后为什么必须复盘？
答：因为真正能提升稳定性的不是“这次修好了”，而是通过复盘把这次事故沉淀成机制改进，比如补监控、补回滚预案、补接口兼容检查、调整缓存策略、完善发布平台和流程。没有复盘，团队下次遇到同类问题时大概率还会用同样混乱的方式再处理一遍。

45. 如果面试官让你总结“灰度发布与回滚的核心价值”，你会怎么答？
答：我会说它们的核心价值，是把前端上线从“全量暴露、难以及时发现、出了问题影响巨大”变成“分批放量、持续观察、能够快速止损和恢复”。这背后体现的不只是某个工具会不会用，而是团队有没有把发布当成稳定性工程，而不是把上线当成一次碰运气的操作。

## 六、灰度发布与 `AB` 测试怎么真正落地做

46. 如果不是面试背答案，而是团队要真的把灰度发布做起来，最核心的设计思想是什么？
答：我会先强调一句，灰度发布不是单一按钮能力，而是一整套“分批放量 + 可观测 + 可止损 + 可回滚”的闭环系统。真正落地时至少要同时具备五个基础：第一，有稳定分流能力，保证同一用户不会来回抖动；第二，有版本与功能解耦能力，通常就是版本发布叠加特性开关；第三，有按版本、按桶、按地区、按终端维度观察指标的能力；第四，有快速止血能力，比如关开关、暂停放量、回切版本；第五，有规范化流程，明确谁审批、谁观察、谁决策继续放量、谁执行回滚。很多团队不是不会做灰度，而是只做了“按比例发”，却没有把观察和止损也产品化，所以一旦出问题还是靠人肉排查。

47. 一个比较成熟的前端灰度发布流程，通常应该怎么设计？
答：一个比较完整的流程，通常会分成发布前、灰度中、全量后和复盘四段。发布前要确认需求风险等级、核心链路回归、接口兼容、资源快照和回滚预案；灰度中要先从白名单或内部账号开始，再到极小流量，比如 `1% -> 5% -> 10% -> 25% -> 50% -> 100%`，每一档都要有明确观察窗口和守护指标；全量后不能立刻离场，而要继续观察一段时间，因为很多问题会在长会话、异步 chunk、缓存扩散和峰值流量下才暴露；最后要做复盘，把这次放量过程中的风险点、拦截效果、监控缺口和手工操作沉淀成平台能力。真正成熟的流程一定不是“今天谁有空谁来盯”，而是每一步都标准化。

48. 灰度发布到底灰的是什么？前端应该按什么层次去灰？
答：前端灰度至少可以分成三个层次。第一层是版本级灰度，也就是让不同用户拿到不同前端构建版本；第二层是功能级灰度，也就是代码已经上线，但通过特性开关决定某个模块是否真正生效；第三层是配置级灰度，也就是同一份代码下，不同用户拿到不同配置、文案、接口策略或推荐逻辑。实际工程里最推荐的不是只做其中一种，而是“版本灰度负责验证整体稳定性，功能灰度负责控制单点风险，配置灰度负责低成本试验”。这样分层以后，团队在处理问题时才知道该关版本、关功能，还是回退配置。

49. 前端灰度分流规则怎么设计才稳？
答：最核心的是稳定、一致、可复现。稳定指的是同一用户在观察窗口内不要反复横跳；一致指的是 Web、H5、小程序、App 内嵌页这些多端如果业务要求统一体验，就要尽量落到同一实验或灰度桶；可复现指的是线上出了问题后，团队能根据用户 id、设备 id、租户 id 或请求日志重新推导出这个用户为什么命中了某个版本。实现上常见做法是对用户唯一标识做 hash，再映射到固定桶区间，比如 `0-9999`，这样发 `10%` 就是命中 `0-999`，扩容到 `20%` 就变成 `0-1999`，之前已命中的用户不会抖动。这里一个经常被忽略的点是匿名用户，如果产品存在大量未登录访问，就需要提前设计设备 id、cookie id 或 session id 的稳定方案，否则灰度流量会非常乱。

50. 灰度发布时，前端和后端/BFF/配置中心应该怎么协同？
答：比较理想的方式是让“分流判定”尽量收敛到一个统一层，不要前端自己判一套、BFF 再判一套、后端实验平台又判一套，否则桶位很容易不一致。常见做法有两种，一种是服务端判桶，由 BFF 或网关把版本、开关、实验结果直接注入首屏数据，前端只负责消费；另一种是实验平台提供统一 SDK，前后端都基于同一份分桶规则计算。无论哪种方式，都要特别注意首屏时机，因为前端如果先按默认值渲染，等异步拿到配置后再切换，就可能出现闪烁、埋点污染和实验曝光不准确的问题。所以真正落地时，关键不是“配置从哪来”，而是“首屏之前能否拿到、不同层是否一致、失败时默认值是否安全”。

51. 灰度放量的节奏应该怎么定，为什么不能只凭经验拍脑袋？
答：放量节奏本质上是风险和样本量之间的平衡。放得太慢，会拖长验证周期；放得太快，一旦有问题就会把影响面迅速扩大。更成熟的做法是把节奏和风险等级绑定，例如普通低风险需求可以走 `5% -> 20% -> 50% -> 100%`，核心链路或高风险重构则走 `白名单 -> 1% -> 5% -> 10% -> 20% -> 50% -> 100%`，并且每一档都设最短观察时间、最低样本量和明确的守护阈值。也就是说，是否继续放量不该靠“看着还行”，而应该满足“样本已足够 + 守护指标无显著恶化 + 关键路径人工抽查通过”这三个条件。灰度真正专业的地方，就在于把继续放量的决策条件说清楚，而不是靠感觉。

52. 灰度发布时到底看哪些指标，怎么判断该不该继续？
答：我会把指标拆成三类。第一类是稳定性指标，比如 `JS error rate`、白屏率、资源加载失败率、接口失败率、chunk 加载失败率、页面崩溃率；第二类是体验性能指标，比如首屏时间、`LCP`、`INP`、首屏接口耗时、关键交互耗时；第三类是业务守护指标，比如支付成功率、下单转化率、注册成功率、表单提交成功率。判断时不能只看新版本自己的绝对值，而要同时和旧版本、历史同期、相同流量结构做对比。比如新版本 `JS error rate` 从 `0.2%` 升到 `0.6%`，哪怕绝对值看起来不高，也可能已经是 3 倍恶化，这时就不该继续放量。成熟团队还会给不同指标设置不同级别的阈值，例如“达到阈值立刻暂停”“连续恶化但未爆阈值需要人工复核”，让灰度决策更一致。

53. 前端做灰度发布时，为什么经常强调版本发布和特性开关要配合？
答：因为只做版本灰度，止损粒度太粗；只做开关灰度，又很难验证整包升级的真实稳定性。两者配合时，比较推荐的方式是先把代码安全发布到线上，但默认关闭高风险功能，然后先灰版本验证基础稳定性，再逐步开功能。如果某个功能出问题，可以只关功能开关，不必整站回滚；如果发现是整包构建或依赖层面的问题，再执行版本回切。这个设计的本质，是把“上线代码”和“对外生效”拆成两个可独立控制的阶段，从而让灰度既能验证真实环境，又保留快速止血的灵活性。

54. 什么样的前端技术条件，决定了你能不能把灰度做得足够安全？
答：我会重点看六个条件。第一，静态资源是否内容 hash 化，HTML 是否短缓存，旧资源是否有保留窗口；第二，是否有统一分流与配置平台；第三，是否能按版本、按桶、按页面、按浏览器维度观测监控；第四，是否有一键暂停放量、关开关、回切版本的能力；第五，埋点体系是否能区分用户属于哪个灰度桶、哪个实验桶、哪个版本；第六，前后端接口是否遵守向前兼容和过渡期策略。少了这些基础设施，团队也能“做一点灰度”，但大概率只能做到流程层面的谨慎，做不到工程层面的可控。

55. 如果要做 `AB` 测试，第一步不是开发页面，而应该先想什么？
答：第一步应该先把实验设计说清楚，而不是先写代码。至少要先回答六个问题：实验目标是什么，主指标是什么，守护指标是什么，实验对象是谁，分流单位是什么，实验持续多久。比如想验证“新的结算页交互能不能提升支付转化”，那主指标可能是支付转化率，守护指标可能是接口报错率、页面停留时长异常、投诉率，实验对象可能是移动端已登录用户，分流单位可能是用户 id，持续时间则要覆盖完整业务周期。很多失败实验并不是方案差，而是一开始就没有把指标、样本、分流单位和观察周期设计清楚，最后得到的结论不可信。

56. `AB` 测试里的分流单位为什么非常关键？
答：因为它决定了实验是否会被污染。比如购物流程这种跨多次访问、跨多个页面的链路，如果你按页面访问随机分流，同一个用户今天看到 A、明天看到 B，数据就会混乱，用户体验也会割裂；这类场景通常更适合按用户 id 固定分流。再比如社交或内容分发场景，如果一个实验结果会受到社交关系或流量网络影响，就要考虑群组污染问题，不能只把每个用户当成完全独立样本。分流单位选错，后面不管统计做得多精细，结论都不稳。

57. `AB` 测试怎么避免“看起来提升了，其实结论不可靠”？
答：这里面最常见的坑有四类。第一，样本量不够，今天涨了、明天跌了，最后只是波动；第二，实验提前偷看，看到短期数据变好就提前结束，导致假阳性；第三，实验对象不纯，比如一部分用户没真正曝光到实验页面却被算进实验桶；第四，同时叠加多个实验，彼此互相干扰。更稳妥的做法是，实验前先做样本量估算和实验期预估，实验中定义清晰的曝光口径和成功口径，实验期间尽量避免频繁改规则，实验结束后同时看主指标、守护指标和分层结果。也就是说，`AB` 测试不是“谁数字高谁赢”，而是一个需要设计、执行和解释都足够严谨的过程。

58. 前端在 `AB` 测试里，除了改页面本身，还要承担哪些工程职责？
答：前端至少要承担四类职责。第一，正确渲染实验版本，保证实验组和对照组拿到的 UI、逻辑和资源是可控的；第二，正确上报曝光、点击、转化和异常埋点，保证数据可解释；第三，避免实验闪烁和错分桶，比如首屏拿不到实验配置导致用户先看见 A 后又切成 B；第四，处理实验关闭后的代码治理，包括下线冗余分支、删除临时开关、清理过期埋点。很多人把 `AB` 测试看成产品或数据团队的事，但前端如果没有把曝光时机、埋点质量和代码清理做好，实验结论和代码质量都会一起出问题。

59. 灰度发布和 `AB` 测试在一个真实项目里，最推荐怎么配合？
答：比较推荐的顺序通常是“先灰度、后实验、再全量”。也就是先用灰度发布验证新代码、新资源、新依赖、新埋点在真实环境下是否稳定，确认没有明显稳定性风险后，再在已经稳定的版本内做 `AB` 测试验证业务收益。这样做的好处是，如果实验效果不好，你知道这是方案问题；如果稳定性异常，你知道优先回到发布链路查，而不是把实验效果和上线风险混在一起。很多团队的问题恰恰在于把“稳定性验证”和“效果验证”一次性叠加，结果实验结论不干净，故障定位也很慢。

60. 如果面试官追问“请你设计一套前端灰度发布 + `AB` 实验体系”，应该怎么高分回答？
答：我会按“目标分层 + 平台能力 + 运行流程 + 风险治理”四层来答。目标分层上，先区分版本灰度、功能开关和 `AB` 实验三者职责；平台能力上，强调统一分流、配置中心、版本管理、监控告警、埋点分析和快速回滚；运行流程上，强调发布前评审、白名单验证、分批放量、守护指标、全量观察和实验复盘；风险治理上，强调静态资源快照、接口兼容、实验污染防控、开关清理和事故复盘。这样回答的好处是，既能体现你理解业务实验，也能体现你理解线上稳定性工程，不会让人觉得你只是会配几个实验桶。

## 七、真正实施方案：现在就在项目里引入灰度发布和 `AB` 测试

下面这一部分不再讲抽象概念，而是按“你现在要开始做”来写，分为通用实施步骤、`React Web` 实施方案、`Flutter` 实施方案三部分。对于大多数团队，我最推荐的是 **统一分流平台 + 服务端判桶 + 客户端消费配置 + 客户端负责曝光与埋点** 这一套模式，因为它最容易保证多端一致、首屏稳定和问题可追踪。

这里先明确一个非常重要的点：**灰度发布和 `AB` 测试不是同一个业务需求，它们只是共享了一部分底层基础设施。**  
灰度发布解决的是“新版本或新功能能不能安全上线”，关注的是稳定性、风险控制、放量和回滚；`AB` 测试解决的是“多个方案谁效果更好”，关注的是实验设计、曝光口径、转化指标和统计结论。  
也就是说：

1. 灰度发布的目标是安全上线
2. `AB` 测试的目标是比较收益
3. 两者都可能用到分桶、配置、埋点、监控，所以底层能力可以共建
4. 但在实施层面，应该拆成两套独立流程，不能混成一个发布动作

所以下面我会先讲 **共用底座怎么建**，再单独讲 **灰度发布怎么做** 和 **`AB` 测试怎么做**。

### 7.1 先定实施方案，不要一上来就写代码

在真正开始接入前，先把方案定清楚。常见有三种：

1. 纯客户端判桶
   优点是接入快、对后端依赖少。
   缺点是多端一致性差、首屏闪烁风险高、实验曝光口径容易乱，不太适合正式长期体系。

2. 纯服务端判桶
   优点是规则统一、可控性强、多端结果一致。
   缺点是客户端灵活性差，纯前端局部实验会变得比较重。

3. 混合方案
   服务端负责计算用户属于哪个灰度桶、实验桶，客户端只消费结果并执行 UI 分支、功能开关和埋点上报。
   这是大多数中大型项目更推荐的方案，也是下面实施步骤默认采用的方案。

如果你现在就要落地，我建议直接选第三种，不要为了“先快点做起来”而长期停留在前端本地随机分桶上，因为后面多端对齐、统计口径、实验复盘都会很痛苦。

### 7.2 引入前必须先定下来的 8 个核心决策

在项目正式接入前，先把下面 8 件事定下来，否则做着做着一定返工：

1. 分流单位是什么
   通常优先用 `userId`，未登录用户用首次生成并持久化的 `anonymousId` 或 `deviceId`。

2. 判桶层放在哪里
   推荐放在 BFF、网关或统一实验平台，不推荐由各端自己各算一遍。

3. 首屏配置从哪里拿
   推荐首屏接口直接返回，或服务端渲染/HTML 注水注入，不推荐页面渲染后再异步补拿。

4. 功能开关和实验配置的数据结构是什么
   要统一字段命名、默认值、失效时间、归属人和版本范围。

5. 曝光口径是什么
   是“用户分到实验组就算曝光”，还是“实验模块真正渲染到屏幕才算曝光”，这必须先统一。更推荐后者。

6. 守护指标是什么
   至少要定稳定性指标、性能指标、业务指标三类，不然只会看转化，很危险。

7. 回滚路径是什么
   先关功能、再停灰度、再回版本，还是直接回版本，要提前定清楚。

8. 谁来决策继续放量
   必须有明确 owner，不要线上出问题时一群人一起猜。

### 7.2.1 什么叫“分桶”

很多人第一次接触灰度和实验时，最容易被“分桶”这个词卡住。  
所谓 **分桶**，其实就是把用户按照某套稳定规则，放进不同的“桶”里，再根据桶来决定这个用户看到哪个版本、命中哪个开关、进入哪个实验组。

你可以把它理解成：

1. 先给每个用户一个稳定编号
2. 再把编号映射到很多小格子里
3. 每个格子就是一个桶
4. 不同桶命中不同规则

比如：

1. 用户 `A` 经过 hash 后落到 `bucket = 123`
2. 用户 `B` 经过 hash 后落到 `bucket = 6789`
3. 配置规定 `0-999` 走灰度新版本
4. 配置规定 `1000-9999` 继续走旧版本

那结果就是：

1. 用户 `A` 命中新版本
2. 用户 `B` 继续看旧版本

这就叫“按桶分流”。

#### 为什么叫分桶

因为它的思想很像把一大堆用户放进很多固定箱子里。  
后续不管你做灰度发布还是 `AB` 测试，本质上都是先问一句：**这个用户现在属于哪个桶？**

#### 分桶和随机有什么区别

分桶不是“每次来都随机一下”。  
真正可用的分桶，强调的是 **稳定** 和 **可复现**。

比如一个用户今天命中实验组 `B`，明天再打开页面，最好还是 `B`，否则会出现：

1. 用户体验来回跳
2. 埋点数据被污染
3. 实验结论不可信
4. 问题难以复盘

所以工程里常见做法不是 `Math.random()`，而是：

1. 取 `userId` 或 `anonymousId`
2. 拼上 `experimentKey` 或 `featureKey`
3. 做稳定 hash
4. 再映射到固定区间，比如 `0-9999`

示意：

```ts
bucket = stableHash(`${identity}:${key}`) % 10000;
```

这样只要 `identity` 和 `key` 不变，这个用户长期都会落到同一个桶里。

#### 灰度发布里的分桶是什么意思

在灰度发布里，分桶的作用是 **决定谁先看到新版本或新功能**。

例如：

1. `0-99` 先看到新版本，表示 `1%`
2. 后续扩到 `0-499`，表示 `5%`
3. 再扩到 `0-999`，表示 `10%`

这样做的好处是，放量时只是扩大区间，已经进来的用户不会来回抖动。

#### `AB` 测试里的分桶是什么意思

在 `AB` 测试里，分桶的作用是 **决定用户属于对照组还是实验组**。

例如：

1. `0-4999` 进入 `control`
2. `5000-7499` 进入 `A`
3. `7500-9999` 进入 `B`

这里的重点不是“谁先上新版本”，而是“不同方案分给谁看，且长期保持稳定”。

#### 为什么灰度和 `AB` 测试都要用分桶

因为两者虽然目标不同，但都需要一个稳定的分流基础：

1. 灰度发布需要知道谁先看到新能力
2. `AB` 测试需要知道谁属于哪个实验组
3. 两者都要求用户不要来回跳组
4. 两者都要求问题可以回放、可追溯

所以你可以把分桶理解为一套 **底层分流机制**，上层既可以服务灰度发布，也可以服务 `AB` 测试。

#### 分桶时最容易犯的错

最常见的错误有这些：

1. 直接随机，不做稳定分桶
2. 同一个项目里的前端、BFF、后端各算各的，结果不一致
3. 未登录用户没有稳定 `anonymousId`
4. 放量时重算规则，导致老用户跳桶
5. 一个实验和另一个实验互相污染，没有隔离

所以真正要落地时，你应该默认追求的是 **稳定分桶**、**同一业务链路内结果一致**、**可追溯**，而不是“能随机分一下就行”。

### 7.2.2 有没有第三方工具可以直接用

有，而且不少。  
但这里也要先分清：有些平台更偏 **灰度发布 / Feature Flag**，有些平台更偏 **实验平台 / `AB` 测试**，还有一些是两者都覆盖，但强项不同。

#### 第一类：更偏灰度发布和 Feature Flag

这类工具更适合做：

1. 功能开关
2. 渐进放量
3. 白名单
4. 快速回滚
5. 多环境配置

典型代表有：

1. `LaunchDarkly`
   在 `React Web` 里它很常见，适合做成熟的 Feature Flag、渐进式放量、用户定向和实时开关。  
   如果你的重点是“把 Web 灰度发布做扎实”，它是很强的商业方案。  
   但如果你想做完整实验分析，通常还要结合它自己的实验能力或你们自己的埋点分析体系一起用。

2. `Flagsmith`
   在 Web 项目里也比较常见，既能做 Feature Flag，也支持 multivariate flag，适合做灰度和简单 `AB/n`。  
   它的优势是比较灵活，也有自托管路线，适合希望保留一定可控性的团队。  
   不过如果你们很强调实验统计和分析看板，往往还是要自己接数据分析链路。

3. `ConfigCat`
   在 `React Web` 侧接入体验很好，适合做开关和灰度。  
   这类产品思路和 `LaunchDarkly` 接近，偏“远程配置 + 灰度放量 + 用户定向”。  
   如果你只做 Web，它是比较直接的选择；不需要为了未来可能有移动端，就先强行按双端统一来选。

#### 第二类：更偏 `AB` 测试和实验平台

这类工具更适合做：

1. 对照组和实验组分配
2. 曝光口径管理
3. sticky bucketing
4. 实验分析
5. 胜者推广

典型代表有：

1. `GrowthBook`
   在 `React Web` 场景里是比较常见的实验平台方案，既支持 Feature Flag，也更强调实验和 sticky bucketing。  
   如果你们更关注 Web 端的实验体系，而不是只做开关，`GrowthBook` 是比较值得关注的方案。

2. `Statsig`
   `Statsig` 也是这一类里很值得关注的方案，它的定位通常比“单纯 Feature Flag 平台”更偏 **实验平台 + Feature Flag + 数据分析联动**。  
   在 `React Web` 和 `Flutter/Dart` 都有 SDK，既能做 gate/check，也能做 experiment、layer 和事件上报。  
   如果你的重点不是“只想远程开关一下功能”，而是想把灰度、实验、指标分析尽量放在一套体系里，`Statsig` 很值得重点评估。

3. `Firebase Remote Config + Firebase A/B Testing`
   这套更适合 `Flutter` 或移动端团队，尤其是本来就在 Firebase 生态里的项目。  
   它适合做远程配置和实验，移动端接入心智成本相对更低。  
   但如果你的后端、埋点、数据分析平台完全不在 Firebase 体系里，工程整合成本要提前评估。

#### 如果 `Flutter` 项目不在 Firebase 体系里，主流方案有哪些

如果你们的 `Flutter` 项目没有接 Firebase，也完全没打算把埋点、配置、实验分析往 Firebase 里迁，那就不要硬上这套。  
更常见、也更实际的选择通常是下面几类：

1. `LaunchDarkly`
   这是偏企业级、偏成熟治理的一条路。  
   适合你们已经比较重视灰度发布、审批流程、用户定向、渐进式放量和快速回滚，愿意接受商业平台成本。  
   如果你的重点是“移动端发版后还能强控制功能开关”，它是很常见的主流方案。

2. `Flagsmith`
   这是很多团队在“功能开关 + 灰度放量 + 一定自控能力”之间比较平衡的选择。  
   它适合做远程配置、身份定向、百分比放量、简单多变体实验，也比较适合不想完全绑死在大商业平台上的团队。  
   如果你们既想做灰度，又希望保留一定自托管或平台控制空间，`Flagsmith` 很值得评估。

3. `GrowthBook`
   如果你们不只是想做开关，还很关注 `AB` 测试、sticky bucketing、实验管理和结果分析，那么 `GrowthBook` 是 `Flutter` 里很值得关注的主流方案。  
   它更偏“实验平台 + Feature Flag”一体化，不只是简单远程开关。  
   所以当你的重点是“验证不同方案谁更好”，而不只是“能不能安全上线”，它通常比纯开关平台更合适。

4. `Statsig`
   如果你们很重视实验、指标、事件上报和产品数据闭环，`Statsig` 也是 `Flutter` 非 Firebase 场景里很值得关注的主流方案。  
   它不是只做“开关开或关”，而是更强调 gate、experiment、layer 和事件分析联动。  
   所以它更适合对实验成熟度要求更高、希望灰度和实验都围绕数据做决策的团队。

5. `ConfigCat`
   如果你们更想要一套相对轻量、接入成本低、以 Feature Flag 和远程配置为主的方案，也可以评估 `ConfigCat`。  
   它更适合“先把灰度发布、功能开关和远程配置做起来”，实验分析能力一般还是要结合你们自己的埋点体系来看。  
   如果你们团队规模不大、需求偏实用主义，这类方案会比较顺手。

6. 自建最小可用方案
   如果你们已经有自己的后端、BFF、配置中心、埋点平台和监控平台，也完全可以不依赖 Firebase，直接自建最小可用版。  
   最少只要有稳定身份、分桶逻辑、配置接口、开关读取层、曝光埋点和紧急关停能力，就能先把灰度和实验跑起来。  
   这种方式最适合平台能力比较强、又不想引入外部商业服务的团队。

#### 没有 Firebase 时，`Flutter` 项目怎么选更实际

可以按目标来粗分：

1. 如果你最关注灰度发布、开关治理、快速关停
   优先看 `LaunchDarkly`、`Flagsmith`、`ConfigCat`

2. 如果你最关注实验体系、分组稳定性、方案对比
   优先看 `GrowthBook`、`Statsig`

3. 如果你们后端和数据平台已经比较完整
   可以评估自建最小版本，不一定非要接第三方

4. 如果你们团队刚起步，先想快速落地
   通常先上 `Flagsmith` 或 `ConfigCat` 这类更容易推进的方案，会比一开始就造完整平台更现实

### 7.2.3 真正做决策时，应该怎么选 
更实际的问题应该是：**我的团队当前最缺的到底是什么能力，我应该优先为哪种能力付费或付出接入成本。**

这里先给一个核心判断：

1. 如果你最在意的是安全上线、灰度放量、远程关停、审批治理
   那你优先看的应该是 **Feature Flag / 灰度发布平台**

2. 如果你最在意的是实验分组、方案比较、指标分析、数据驱动决策
   那你优先看的应该是 **实验平台**

3. 如果你两者都想要
   就不要只看“能不能开关”，而要看这个平台有没有把 **分组、事件、指标、实验结果** 一起做好

换句话说，很多团队的误区不是不会选工具，而是没有先搞清楚自己到底是“发布问题优先”，还是“实验问题优先”。

#### 一、先看五个方案的定位差异

##### 1. `LaunchDarkly`

定位：更偏 **企业级 Feature Flag / 灰度发布平台**

它最强的地方通常不是实验分析，而是：

1. 开关治理
2. 渐进式放量
3. 用户定向
4. 紧急关停
5. 发布安全控制
6. 权限、审批、治理能力

如果你们团队首先担心的是“新功能上线别把线上搞挂”，而不是“哪个方案转化更高”，那它通常会比偏实验平台的工具更对路。

##### 2. `Flagsmith`

定位：更偏 **功能开关 + 灰度放量 + 一定实验能力** 的均衡型方案

它的特点是：

1. 灰度和开关能力够实用
2. 没有那么重
3. 对很多中小团队更容易推进
4. 既可以先做开关和灰度，也能支持一些简单实验
5. 如果你比较在意控制感和灵活度，它通常比纯大平台更容易接受

它不一定是“实验平台里的最强选手”，但经常是“最容易真正落地”的那一类。

##### 3. `GrowthBook`

定位：更偏 **实验平台 + Feature Flag**，强调实验和数据自主

它比较适合：

1. 认真做 `AB` 测试
2. 强调 sticky bucketing
3. 希望实验逻辑可控
4. 团队本身就有一定数据和平台能力

如果你们只想远程开关，它可能有点重；但如果你们想真正把实验做成能力，它会比“纯开关平台”更有吸引力。

##### 4. `Statsig`

定位：更偏 **实验平台 + Feature Flag + 数据分析联动**

它和单纯 Feature Flag 平台最大的不同，是它更强调：

1. gate
2. experiment
3. layer
4. 事件上报
5. 指标分析
6. 实验驱动决策

如果你的目标不只是“把功能开关放到远端”，而是想把灰度、实验、事件、数据闭环尽量串起来，`Statsig` 通常会更接近你要的方向。

##### 5. `ConfigCat`

定位：更偏 **轻量远程配置 / Feature Flag**

它比较适合：

1. 先快速接入
2. 团队规模不大
3. 当前最重要的是开关、远程配置、简单灰度
4. 不急着把实验体系做得很重

如果你现在最急的是先把远程开关和灰度发布做起来，它会比重平台更轻；但如果你期待它直接变成一整套实验决策平台，就容易预期过高。

#### 三、按 `React Web` 项目看，应该怎么选

`React Web` 项目和移动端最大的不同，是它往往更频繁地做：

1. 转化率优化
2. 页面布局实验
3. 文案实验
4. 推荐策略实验
5. 快速试错

所以 Web 端工具选型时，实验能力的重要性通常会更高。

##### 如果你是 `React Web` 项目，我更推荐这样理解

1. 如果你最关注实验、转化、指标分析
   优先评估 `Statsig`

2. 如果你更关注上线安全、灰度放量、开关治理
   优先评估 `LaunchDarkly`

3. 如果你想要实验能力，但更强调自控和平台可掌握
   优先评估 `GrowthBook`

4. 如果你想先轻量把灰度和开关做起来
   优先评估 `Flagsmith`、`ConfigCat`

##### 为什么 `React Web` 我会更看重 `Statsig`

因为很多 Web 团队真正的长期诉求不是“远程开关一下功能”，而是：

1. 不同页面方案谁更好
2. 哪个结算流程转化更高
3. 哪个文案点击率更高
4. 实验结果能不能直接指导产品决策

在这种场景里，纯开关平台虽然也能凑出实验，但体验通常不如本身就偏实验平台的方案顺。

#### 四、按 `Flutter` 项目看，应该怎么选

`Flutter` 移动端和 Web 不一样，它通常先面对的是：

1. 发版成本高
2. 审核慢
3. 回包慢
4. 出问题止血要求高

所以移动端很多团队虽然也说想做 `AB` 测试，但第一优先级往往还是：

1. 发版后可控
2. 能远程关功能
3. 能灰度放量
4. 能避免因为重新发包而止血太慢

##### 如果你是 `Flutter` 项目，我更推荐这样理解

1. 如果你当前第一诉求是灰度发布、功能开关、快速止血
   优先评估 `Flagsmith`

2. 如果你是大团队、治理要求高、预算也更充足
   优先评估 `LaunchDarkly`

3. 如果你不只是想做开关，而是真的要做实验平台和数据闭环
   优先评估 `Statsig`

4. 如果你更希望实验能力可控，且团队本身有平台/数据能力
   优先评估 `GrowthBook`

5. 如果你们只想先轻量做远程配置和开关
   优先评估 `ConfigCat`

##### 为什么 `Flutter` 我不会默认直接推最重的实验平台

因为很多移动端团队当前最痛的并不是“实验平台不够强”，而是：

1. 新功能上线后不能快速关
2. 线上出问题只能等重新发包
3. 没有远程配置能力
4. 没有灰度放量能力

在这种阶段，如果你一开始就上很重的实验平台，反而可能投入很大，但先解决不了最痛的发布控制问题。

#### 五、为什么 `Flutter` 和 `Web` 的发版机制差异，会直接影响工具选择

很多人第一次看这类选型，会觉得奇怪：  
为什么同样都是前端，`React Web` 往往更容易优先选实验平台，而 `Flutter` 移动端经常先选灰度发布 / 开关平台？

根本原因不是“移动端不重视实验”，而是 **两类客户端的发版机制和止血机制完全不一样**。  
而发版机制不同，会直接改变团队最优先要解决的问题，所以最后选型自然也不同。

##### 1. `Web` 为什么更容易优先考虑实验能力

`Web` 的最大特点是：**发版快、回滚也快、代码生效快**。

通常一个 Web 项目上线新代码，大致是这样：

1. 前端打包
2. 上传静态资源
3. 发到 CDN
4. 用户刷新或重新访问后逐步拿到新版本

这意味着 Web 有几个天然优势：

1. 发布速度快
   很多团队一天能发多次，甚至几十次。

2. 回滚速度也快
   如果只是资源或配置问题，通常比移动端重新提审快很多。

3. 试错成本低
   页面方案、按钮文案、流程顺序都可以很快试。

4. 真实流量实验更容易持续进行
   因为版本切换成本低，所以更适合不断做页面和转化实验。

所以 Web 团队在很多阶段最先想优化的，不是“怎么发版才不痛”，而是：

1. 哪个方案转化更高
2. 哪个页面布局更好
3. 哪个推荐策略更有效
4. 怎么用实验结果指导产品决策

这就是为什么 `React Web` 项目里，像 `Statsig`、`GrowthBook` 这种偏实验平台的价值会被放大。  
不是因为 Web 不需要灰度，而是因为 Web 在“发版可操作性”这件事上，本来就比移动端轻松很多，所以团队更容易把注意力放到“效果优化”上。

##### 2. `Flutter` 为什么更容易优先考虑灰度发布和开关能力

`Flutter` 移动端的最大特点是：**发版慢、回滚慢、止血难、版本长期分裂**。

一个移动端新版本上线，现实中往往会遇到这些限制：

1. 提审需要时间
   无论是 iOS 还是 Android，不是说发就能立刻让所有用户拿到新版本。

2. 用户更新不及时
   就算新版本已经发布，很多用户也不会马上更新。

3. 多版本长期共存
   线上会长期同时存在旧版本和新版本，不可能像 Web 那样很快统一。

4. 真正回包慢
   如果线上功能出了问题，重新发一个修复包，链路远比 Web 慢。

5. 发布错误的代价更高
   如果是首页白屏、支付异常、崩溃上升，止血窗口比 Web 更紧张。

这就导致移动端团队更容易先关注这些问题：

1. 新功能上线后能不能远程关掉
2. 能不能只放给一小部分用户
3. 能不能按版本控制配置
4. 老版本用户还能不能安全工作
5. 不重新发包的前提下，能不能快速止血

这就是为什么 `Flutter` 端很多时候先优先 `Flagsmith`、`LaunchDarkly`、`ConfigCat` 这类更偏开关和灰度的平台。  
不是因为实验不重要，而是因为 **移动端先活下来、先可控、先能止血** 往往比“实验分析做到最强”更优先。

##### 3. 发版机制差异为什么会直接改变选型顺序

工具选型本质上是在回答：**当前最值得优先解决的瓶颈是什么。**

对于 `Web` 来说，常见瓶颈更像是：

1. 页面方案太多，不知道哪个更优
2. 需要快速反复试验
3. 希望实验和转化分析形成闭环

所以它更容易优先考虑：

1. `Statsig`
2. `GrowthBook`
3. 其他偏实验平台的方案

对于 `Flutter` 来说，常见瓶颈更像是：

1. 发版后不可控
2. 出问题无法快速关功能
3. 多版本并存导致兼容复杂
4. 审核和更新链路让修复速度变慢

所以它更容易优先考虑：

1. `Flagsmith`
2. `LaunchDarkly`
3. `ConfigCat`
4. 然后再补更强实验能力

也就是说，不是 Web 和 Flutter 的团队认知不同，而是 **平台约束不同，导致最优先解决的问题不同**。

##### 4. `Flutter` 移动端是不是一般不做 `AB` 测试

不是。  
更准确的说法应该是：**移动端不是不做 `AB` 测试，而是很多团队会把“可控发布”排在“复杂实验”前面。**

也就是说：

1. 移动端当然做 `AB` 测试
2. 而且大厂、成熟产品、增长型团队都会做
3. 只是移动端做实验时，通常要比 Web 更谨慎
4. 很多团队不会在没有远程开关和灰度能力之前，就直接上很重的实验体系

##### 5. 为什么移动端的 `AB` 测试会更谨慎

因为移动端实验比 Web 多了几层现实约束：

1. 用户版本不统一
   同一个实验，可能只有一部分版本支持。

2. 配置兼容要求更高
   老版本看不懂的新实验字段，必须安全降级。

3. 崩溃和性能风险更敏感
   App 一旦卡顿、崩溃、闪退，用户容忍度通常更低。

4. 修复周期更长
   Web 出问题可以很快改，移动端很多时候不能立刻靠新包修。

5. 首屏和启动链路更敏感
   实验配置如果影响启动流程，失败代价比 Web 更大。

所以移动端做 `AB` 测试，不是不能做，而是更常见的路径是：

1. 先有远程配置
2. 先有功能开关
3. 先有灰度放量
4. 再在稳定基础上做实验

这也是为什么很多移动端团队前期看起来“灰度比实验更重”，但那不等于他们以后不做实验，而是阶段优先级不同。

##### 6. 什么样的 `Flutter` 团队会非常重视 `AB` 测试

下面这些移动端团队，通常会非常重视实验：

1. 增长型产品团队
   比如拉新、留存、转化高度依赖产品优化。

2. 内容平台
   比如推荐流、排序、曝光策略、互动入口需要持续优化。

3. 电商或交易型产品
   比如支付流程、商品详情、活动页、优惠展示等对转化很敏感。

4. 已经有成熟埋点和数据分析体系的团队
   因为他们具备把实验结果真正用起来的基础。

这类团队的移动端照样会认真做 `AB` 测试，只是他们通常不会跳过开关和灰度治理，直接裸做实验。

##### 7. 一句话总结这件事

可以把这件事记成一句特别实用的话：

1. `Web` 因为发版快、试错快，所以更容易优先为“实验效率”买单
2. `Flutter` 因为发版慢、止血难，所以更容易优先为“发布可控性”买单
3. `Flutter` 不是不做 `AB` 测试，而是更常见的顺序是：先把灰度和开关做稳，再把实验做深

所以当你看到 Web 更容易优先选 `Statsig`、`GrowthBook`，而移动端更常见先选 `Flagsmith`、`LaunchDarkly` 这类方案时，不要把它理解成“谁更先进”，而应该理解成“谁先解决自己最痛的问题”。

#### 这些工具是不是拿来就能直接解决问题

可以，但要分清“能直接用 SDK”和“能直接解决你的所有问题”不是一回事。

这些第三方平台通常能帮你直接解决：

1. 远程配置拉取
2. Feature Flag 判断
3. 用户定向
4. 百分比放量
5. 实验组分配

但通常仍然需要你自己补这些工程能力：

1. 统一身份体系
2. 首屏初始化时机
3. 曝光埋点口径
4. 核心业务转化埋点
5. 看板和告警
6. 实验结束后的代码清理

### 7.3 通用实施步骤：不区分 `React` 或 `Flutter`

如果你现在要从 0 到 1 建立灰度和实验体系，推荐按下面 10 步走。

#### 第 1 步：先梳理你要控制的对象

先把未来要灰度和实验控制的对象分成三类：

1. 版本发布
   比如新版本首页、新依赖升级、新构建链路改造。

2. 功能开关
   比如新入口、新弹窗、新支付流程、新推荐位。

3. 实验配置
   比如按钮文案、排序策略、推荐算法、结算页布局。

如果这三类对象不分开，后面线上一旦出问题，你根本说不清到底该关哪个。

#### 第 2 步：定义统一配置协议

无论 `Web` 还是 `Flutter`，都应该消费同一种配置结构。最少要有这些字段：

```ts
type FeatureFlag = {
  key: string;
  enabled: boolean;
  owner: string;
  expireAt?: string;
};

type ExperimentAssignment = {
  key: string;
  variant: 'control' | 'A' | 'B' | 'C';
  bucket: number;
  version: string;
};

type GrayConfigPayload = {
  appVersion: string;
  flags: Record<string, FeatureFlag>;
  experiments: Record<string, ExperimentAssignment>;
  fetchedAt: string;
};
```

这里最重要的不是字段多，而是 **统一**。只要配置结构统一，`React Web`、`Flutter`、埋点系统、监控平台、BFF 都能说同一种语言。

#### 第 3 步：确定稳定身份体系

分桶要稳定，先解决身份问题：

1. 已登录用户
   直接用 `userId`。

2. 未登录用户
   首次进入时生成 `anonymousId`，持久化到浏览器本地或移动端存储。

3. 多端一致需求
   如果要求用户在 Web 和 App 上命中同一实验，就必须以登录态用户 id 为主，不要各端自己随机。

#### 第 4 步：实现统一判桶逻辑

判桶逻辑必须稳定，可复现。典型做法：

1. 取 `identity + experimentKey`
2. 做稳定 hash
3. 映射到 `0-9999`
4. 按比例区间分配实验组或灰度组

例如：

```ts
bucket = stableHash(`${identity}:${experimentKey}`) % 10000;
```

然后配置：

- `0-4999` 进入 `control`
- `5000-7499` 进入 `A`
- `7500-9999` 进入 `B`

这样以后从 `10%` 扩到 `20%`，只是扩大区间，不会让老用户跳桶。

#### 第 5 步：实现首屏引导接口

你需要一个统一的“启动配置接口”或“首屏注水数据”，返回：

1. 当前用户身份
2. 当前 app/web 版本
3. 命中的 flags
4. 命中的 experiments
5. 关键守护开关

这个接口要尽量在首屏渲染前拿到，因为实验和功能分支一旦晚到，就会出现：

1. 页面先按默认逻辑渲染
2. 配置回来后再次切换
3. 用户看到闪烁
4. 埋点曝光失真

#### 第 6 步：建立统一 SDK 或适配层

不要让业务组件到处直接读原始配置。应该封一层统一能力，比如：

1. `isFeatureEnabled(flagKey)`
2. `getExperimentVariant(experimentKey)`
3. `trackExperimentExposure(experimentKey, variant)`
4. `trackExperimentGoal(goalKey, metadata)`

这样后面不管你切配置平台、自研还是第三方，业务层都不用大改。

#### 第 7 步：定义曝光、点击、转化埋点口径

`AB` 实验成败很多时候不取决于页面代码，而取决于埋点口径是否可信。至少要定义三类事件：

1. 曝光事件
   实验组件真正可见时才上报，不要只要分到桶就算曝光。

2. 点击事件
   用户真正触发操作时上报。

3. 转化事件
   比如支付成功、表单提交成功、注册完成。

每条事件都应该带上这些字段：

1. `experimentKey`
2. `variant`
3. `bucket`
4. `appVersion`
5. `userId/anonymousId`
6. `pageRoute`
7. `timestamp`

#### 第 8 步：建立守护指标与回滚阈值

至少建立下面三类守护：

1. 稳定性
   `JS error rate`、崩溃率、白屏率、接口错误率、资源加载失败率。

2. 性能
   `LCP`、`INP`、首屏耗时、关键交互耗时。

3. 业务
   注册成功率、支付成功率、下单率、留存等。

同时要明确：

1. 达到什么阈值立刻暂停灰度
2. 达到什么阈值只关功能开关
3. 达到什么阈值必须回版本

#### 第 9 步：把发布流程标准化

真正的上线流程建议固定成：

1. 合并代码，但默认关闭高风险功能
2. 小流量灰版本
3. 观察稳定性
4. 逐步开功能
5. 再做 `AB` 实验
6. 实验结束后选胜者
7. 清理旧分支和旧开关

#### 第 10 步：建立清理机制

灰度和实验体系最容易烂掉的地方就是“只加不删”。每个 flag 和 experiment 都必须有：

1. owner
2. 创建时间
3. 预期下线时间
4. 关联需求
5. 是否还在采数

实验结束后，必须删掉无效分支，不要让项目代码永久活在 `if/else` 分叉里。

### 7.3.1 灰度发布的独立实施方案

上面 10 步更偏“共用底座建设”。如果你现在单独要落地的是 **灰度发布**，那真正执行时应该按下面这套流程走，而不是直接套 `AB` 实验思路。

#### 灰度发布要解决的核心问题

灰度发布的目标只有一个：**把新版本安全送上线，并在出问题时能快速止损。**

所以它的核心判断不是：

1. 哪个方案转化更高
2. 哪个文案点击更好

而是：

1. 新版本会不会把线上搞挂
2. 新功能会不会让错误率、性能、关键链路变差
3. 出问题时能不能快速停掉、关掉、回掉

#### 灰度发布的标准执行步骤

如果你现在要在项目里真正开始灰度发布，建议按下面 8 步执行。

##### 第 1 步：先圈定灰度对象

先确定这次到底灰什么：

1. 灰整个版本
   比如新首页、依赖升级、构建升级、路由重构。

2. 灰某个功能
   比如新支付页、新推荐位、新搜索面板。

3. 灰某类用户
   比如员工、白名单、某地区、某渠道、某端版本。

这里最怕的是“这次到底在灰什么”都说不清，最后出问题时没人知道该回版本、关功能还是停放量。

##### 第 2 步：给新能力加默认关闭的开关

灰度发布不是一上来就让所有新逻辑直接生效。更稳妥的做法是：

1. 代码先发布
2. 高风险功能默认关闭
3. 通过远端配置逐步开启

这样做的意义是，版本上线和功能生效可以拆开控制，线上出问题时第一反应不是重发版，而是先关开关止血。

##### 第 3 步：先跑白名单，不要直接上比例

推荐顺序：

1. 本地和测试环境验证
2. 员工账号白名单验证
3. 内部测试账号验证
4. 极小流量验证，比如 `0.5%` 或 `1%`
5. 再逐步放量

白名单阶段最重要的不是数据量，而是验证“真实生产链路是否正常”，比如：

1. 配置能否正确下发
2. 旧页面和新页面能否正确切换
3. 埋点是否带上版本和桶信息
4. 开关关闭后能否立刻恢复旧逻辑

##### 第 4 步：定义灰度放量节奏

灰度发布要明确放量计划，不能临场拍脑袋。推荐预先写清楚：

1. `1% -> 5% -> 10% -> 25% -> 50% -> 100%`
2. 每一档最少观察多久
3. 每一档看哪些指标
4. 谁批准进入下一档

高风险需求更保守，低风险需求可以快一点，但无论如何都要有明确节奏。

##### 第 5 步：灰度期间重点看守护指标

灰度阶段关注的是守护指标，不是实验收益指标。至少看三类：

1. 稳定性
   `JS error rate`、白屏率、崩溃率、接口失败率、资源加载失败率。

2. 性能
   `LCP`、`INP`、首屏耗时、关键交互耗时。

3. 核心链路
   登录成功率、下单率、支付成功率、提交成功率。

如果这些指标在灰度版本中明显恶化，就应该暂停放量，而不是继续赌后面会变好。

##### 第 6 步：明确三条止血路径

灰度发布一定要提前准备三条止血路径：

1. 暂停继续放量
   适合“还没确认问题，但不敢继续扩散”。

2. 关闭功能开关
   适合“问题集中在某个新功能，但整站不用回”。

3. 回切稳定版本
   适合“整包有问题、依赖有问题、资源快照有问题”。

真正专业的灰度流程，不是只会放量，而是知道不同级别故障分别该怎么收。

##### 第 7 步：灰度完成后再决定是否全量

灰度到 `50%` 或更高时，不代表一定自动进入全量。必须确认：

1. 指标稳定
2. 重点路径人工抽查通过
3. 没有明显地区、浏览器、端版本异常
4. 回滚预案仍然可用

确认没问题后再进入 `100%`，而不是“都已经放一半了，干脆全开”。

##### 第 8 步：灰度结束后清理临时状态

灰度结束后要收尾：

1. 白名单规则清理
2. 临时观测配置清理
3. 失效开关清理
4. 灰度看板与结论归档
5. 问题复盘

否则下一次灰度时，这些历史状态会反过来污染新流程。

#### 灰度发布一句话总结

灰度发布是一套 **“先小范围验证稳定性，再逐步放量，出问题能快速止血”** 的上线流程，它不是为了选最优方案，而是为了控制生产风险。

### 7.3.2 `AB` 测试的独立实施方案

如果你现在单独要落地的是 **`AB` 测试**，那执行方式和灰度发布完全不一样。  
`AB` 测试不是为了验证“能不能上”，而是为了验证“哪个更好”。

#### `AB` 测试要解决的核心问题

`AB` 测试的目标是：

1. 比较多个方案的业务效果
2. 用真实用户数据而不是主观感觉做决策
3. 找出胜出版本，再推广到全量

所以它的核心不是“逐步放量观察错误率”，而是：

1. 主指标怎么定义
2. 曝光口径是否准确
3. 样本量是否足够
4. 实验是否被污染
5. 结论是否可信

#### `AB` 测试的标准执行步骤

如果你现在要在项目里真正开始做 `AB` 测试，建议按下面 8 步执行。

##### 第 1 步：先写清实验假设

实验一开始就要明确：

1. 你想验证什么变化
2. 你认为为什么会提升
3. 提升哪个指标

例如：

1. 假设：新版结算页减少干扰项，会提升支付转化率
2. 主指标：支付转化率
3. 守护指标：错误率、页面停留时长异常、投诉率

如果连实验假设都写不清，那最后很容易变成“大家看看数据，好像这个版本更顺眼”。

##### 第 2 步：定义实验对象和分流单位

这里要明确两件事：

1. 谁进入实验
   比如移动端登录用户、Web 首页访客、新注册用户等。

2. 按什么分流
   通常优先按 `userId`，不要按单次访问随机。

实验对象定义不清，或者分流单位不稳定，后面的结果就会被污染。

##### 第 3 步：设计实验组和对照组

最少要有：

1. `control`
   当前线上方案。

2. `A/B`
   新方案或多个备选方案。

推荐一开始先做简单对比，不要一次做太多组。组越多，分析越复杂，样本分散越严重。

##### 第 4 步：定义曝光、点击、转化口径

`AB` 测试最关键的是数据口径。至少要明确定义：

1. 曝光事件
   用户真正看到实验模块才算曝光。

2. 点击事件
   用户真实点击实验元素时才记录。

3. 转化事件
   用户完成目标行为时记录，比如支付成功、注册成功、提交成功。

如果曝光口径错了，比如用户其实没看到实验模块却被记成曝光，那整个实验结论都会被污染。

##### 第 5 步：估算样本量和实验周期

不要今天开实验，明天看见上涨就结束。至少要提前定：

1. 实验跑多久
2. 预期最少样本量
3. 是否覆盖完整业务周期

比如周末流量和工作日流量差异很大，那实验至少要覆盖完整周期，不然很容易把自然波动当成实验效果。

##### 第 6 步：实验期间保持规则稳定

实验一旦开始，尽量不要频繁改：

1. 分桶规则
2. 页面逻辑
3. 埋点口径
4. 实验对象

否则同一个实验前后根本不是同一套条件，最后数据无法解释。

##### 第 7 步：实验结束后一起看主指标和守护指标

判断胜者时不能只看主指标。要同时看：

1. 主指标是否提升
2. 守护指标是否变差
3. 分层结果是否稳定
4. 是否只在某一类用户里有效

如果一个方案提升了点击率，却明显拉高了错误率或投诉率，那它不应该被简单视为胜者。

##### 第 8 步：实验结束后做“胜者推广”和“代码收口”

实验结束后要做两件事：

1. 胜者推广
   把最终选中的方案逐步替换为默认方案。

2. 代码收口
   删除失败分支、无效埋点和临时实验配置。

很多团队实验结束后最容易忘的是第二件事，结果项目里永久堆积大量历史实验代码。

#### `AB` 测试一句话总结

`AB` 测试是一套 **“先提出假设，再做稳定分组，基于可信曝光和转化数据比较方案优劣”** 的实验流程，它不是为了安全上线，而是为了科学选方案。

### 7.4 `React Web` 项目实施方案

这里假设你的项目是典型 `React` 单页应用或 `Next.js` / `SSR` Web 项目。

#### 第 1 步：先搭一层 `bootstrap config`

最推荐的做法是，在应用初始化阶段拿一次配置，然后放进全局 store 或 context 中。

推荐数据流：

1. 打开页面
2. 请求 `bootstrap` 接口
3. 返回用户身份、flags、experiments、版本号
4. 初始化全局 `ExperimentProvider`
5. 业务组件通过 hook 读取

可以抽象成：

```ts
type BootstrapData = {
  identity: {
    userId?: string;
    anonymousId: string;
  };
  grayConfig: GrayConfigPayload;
};
```

#### 第 2 步：封装统一 Hook

业务层不要自己解析 JSON。至少提供两个 Hook：

```ts
function useFeatureFlag(flagKey: string): boolean {}

function useExperimentVariant(
  experimentKey: string
): 'control' | 'A' | 'B' | 'C' {}
```

业务组件这样用：

```tsx
const showNewCheckout = useFeatureFlag('checkout_v2_enabled');
const checkoutLayoutVariant = useExperimentVariant('checkout_layout_test');
```

这样未来你换平台、换字段、换缓存策略，业务组件基本不用动。

#### 第 3 步：在路由和组件层都加守卫

`React Web` 常见问题是只在组件里判断，但路由层没拦，导致用户能通过直链进入未开放页面。更稳妥的做法是：

1. 路由守卫控制页面是否可进入
2. 组件守卫控制页面内部新模块是否展示
3. 接口层守卫保证危险操作不能只靠前端页面隐藏

也就是说，前端开关只能作为体验层控制，真正有风险的业务动作，后端也要校验。

#### 第 4 步：处理首屏闪烁问题

这是 Web 接实验最容易踩的坑。建议：

1. 有服务端渲染时，直接把实验配置注水到 HTML
2. 纯前端应用时，在 `bootstrap` 完成前先渲染骨架屏或稳定占位
3. 不要让页面先渲染 `control`，配置回来后再瞬间改成 `A/B`

如果首屏必需依赖实验结果，就宁可多等一个启动请求，也不要让用户看见闪烁后的“假曝光”。

#### 第 5 步：曝光埋点要绑定真实渲染时机

Web 里推荐在组件真正进入可视区域时再报曝光，不要组件一 mount 就报。更稳妥的方式是：

1. 组件渲染完成
2. 使用 `IntersectionObserver`
3. 首次进入可视区时上报曝光
4. 去重，避免重复曝光

这样得到的数据更接近“用户真的看到了”。

#### 第 6 步：接口层带上实验上下文

很多团队只把实验埋到前端日志里，后端完全不知道用户属于哪个实验组，后面排查问题很难。更推荐：

1. 页面请求头带 `x-app-version`
2. 请求头带 `x-gray-bucket`
3. 请求头带关键实验，比如 `x-exp-checkout-layout: B`

这样 BFF、网关、后端日志、监控平台都能按实验组聚合分析。

#### 第 7 步：React 项目的发布顺序建议

对于一个新功能，比如“新版结算页”，建议按下面顺序上线：

1. 代码上线，但 `checkout_v2_enabled = false`
2. 内部白名单打开
3. 观察错误率、支付成功率、核心链路性能
4. 小比例灰度 `1% -> 5% -> 10%`
5. 基础稳定后，在灰度人群里做 `AB` 布局实验
6. 选出胜者后，对全量用户逐步打开
7. 删除失败版本代码和临时开关

这套顺序的重点是：**先验证能不能稳，再验证哪种更好**。

#### 第 8 步：React 项目最少应有的代码组织方式

推荐单独抽一个目录，不要把实验逻辑散到每个业务组件里：

```text
src/
  experiments/
    types.ts
    provider.tsx
    hooks.ts
    tracker.ts
    guards.ts
  features/
    checkout/
      CheckoutPage.tsx
      CheckoutV1.tsx
      CheckoutV2.tsx
```

这样后面清理实验、排查逻辑、统一替换都更容易。

### 7.5 `Flutter` 项目实施方案

这里按你常用的 `Flutter 3 + Riverpod + go_router + Dio` 来讲。移动端和 Web 最大的不同，是版本发布更慢、审核更重，所以 **功能开关和实验平台的重要性甚至比 Web 更高**。

#### 第 1 步：把“发布版本”和“功能生效”彻底拆开

`Flutter` 项目里，不要把“功能是否展示”直接绑定到 app 发版。推荐策略：

1. 代码随安装包上线
2. 默认关闭新功能
3. App 启动时拉配置
4. 根据远端开关与实验结果决定是否展示

这样线上出了问题，第一优先是远端关开关，而不是重新发包等审核。

#### 第 2 步：先做启动阶段配置拉取

推荐在应用启动时拉取一次 `bootstrap config`，写入 `Riverpod` 状态层。流程建议：

1. App 启动
2. 读取本地 `userId` 或 `anonymousId`
3. 请求配置接口
4. 返回 flags、experiments、版本信息
5. 写入 `StateNotifier` / `NotifierProvider`
6. 再进入主路由

如果首屏强依赖实验结果，可以让 `Splash` 页承担这一段初始化，不要先进首页后再切换。

#### 第 3 步：匿名身份要持久化

未登录状态下，建议首次启动就生成稳定 `anonymousId`，保存到本地，比如：

1. `shared_preferences`
2. `flutter_secure_storage`（如果你们希望更稳妥）

核心不是用哪个库，而是 **安装周期内稳定不变**，否则实验会抖动。

#### 第 4 步：用 `Riverpod` 封装统一读取入口

不要在任意 widget 里到处直接读原始 map。建议抽象 provider：

```dart
final featureFlagProvider = Provider.family<bool, String>((ref, key) {
  final config = ref.watch(grayConfigProvider);
  return config.flags[key]?.enabled ?? false;
});

final experimentVariantProvider =
    Provider.family<String, String>((ref, key) {
  final config = ref.watch(grayConfigProvider);
  return config.experiments[key]?.variant ?? 'control';
});
```

业务层只消费：

```dart
final enabled = ref.watch(featureFlagProvider('checkout_v2_enabled'));
final variant = ref.watch(experimentVariantProvider('checkout_layout_test'));
```

#### 第 5 步：在 `go_router` 层做页面级开关控制

如果某个新页面还没开放，不要只在页面内部隐藏内容，最好直接在路由层拦住。比如：

1. 未命中新功能开关时
2. `go_router` redirect 到旧页面或降级页
3. 页面内部只负责局部实验分支

这样比“用户先进新页面，再看到一个空白态”更稳。

#### 第 6 步：在 Widget 层做实验分支，但控制重建范围

`Flutter` 做实验时，最怕的是把整页都包在一个大实验判断里，导致 rebuild 范围过大。更推荐：

1. 路由级决定页面能不能进
2. 模块级决定新旧组件怎么切
3. 尽量把实验分支收敛到局部 widget

这样能减少不必要重建，也更方便实验结束后删除旧分支。

#### 第 7 步：通过 `Dio` 统一透传实验上下文

推荐在 `Dio Interceptor` 里统一加请求头，至少带：

1. `x-app-version`
2. `x-platform: flutter`
3. `x-gray-bucket`
4. `x-exp-*`

这样后端日志、网关观测、接口异常排查时，能快速知道问题是不是只发生在某个实验组或某个灰度版本。

#### 第 8 步：曝光埋点要绑定页面真正可见

移动端不要只在 provider 初始化时就报曝光。更合理的时机通常是：

1. 页面进入前台
2. 目标模块真正渲染
3. 首次可见时上报曝光

如果是列表流里的实验卡片，还要考虑滚动可见才算曝光。

#### 第 9 步：Flutter 项目的上线顺序建议

移动端推荐的顺序通常比 Web 更保守：

1. 代码进包，但默认关闭新功能
2. 员工账号或测试账号白名单开启
3. 线上稳定后，先小人群开开关
4. 确认稳定后再做 `AB` 实验
5. 选胜者后全量开启
6. 下一版本移除旧逻辑和过期开关

原因是 App 发版和回包的成本高，所以功能开关和实验配置要承担更强的线上控制职责。

#### 第 10 步：Flutter 项目要特别注意版本兼容

移动端线上一定会长期存在多个 App 版本，所以配置平台必须支持：

1. 按客户端版本下发不同配置
2. 老版本看不懂的新字段必须有默认值
3. 新版本依赖的新接口不能让旧版本直接崩掉

也就是说，移动端的灰度和实验，从来不是“只考虑当前最新版”。

### 7.6 我最推荐你现在就执行的落地顺序

如果你现在要真的开始干，而不是继续讨论，我建议直接按下面顺序推进：

1. 先定统一配置协议
   明确 flags、experiments、版本号、bucket、identity 结构。

2. 先做身份体系
   已登录用 `userId`，未登录生成稳定 `anonymousId`。

3. 先做 `bootstrap` 接口
   首屏把用户命中的灰度与实验一次性拿回来。

4. 再做客户端适配层
   `React` 封 `Provider + hooks`，`Flutter` 封 `Riverpod providers`。

5. 再补埋点和曝光
   没有可信曝光数据，实验基本等于白做。

6. 再接守护指标和告警
   至少能看错误率、性能、核心转化。

7. 先选一个低风险功能试点
   不要第一枪就上登录、支付、下单主链路。

8. 验证“白名单 -> 小流量 -> 扩量 -> 关开关 -> 回滚”整条链路都可用
   这个演练非常重要，别等线上出事时才第一次用。

### 7.7 如果你团队资源有限，最低可用版本应该怎么做

如果你们现在还没有成熟实验平台，也可以先做一个最小可用版，但最少要有这 6 个能力：

1. 稳定 `identity`
2. 稳定 hash 分桶
3. 启动配置接口
4. 功能开关读取层
5. 实验曝光埋点
6. 一键关闭某个功能或实验

也就是说，你们可以先不追求很华丽的平台后台，但千万不要跳过“稳定分桶、统一配置、可止血”这三件事。

### 7.8 最后给一个真正可执行的项目清单

你可以把下面这张清单直接当实施 checklist：

1. 明确使用 `userId + anonymousId` 作为统一身份体系
2. 明确由 BFF / 网关 / 实验平台统一判桶
3. 定义统一 `grayConfig` 协议
4. 建立 `bootstrap` 启动接口
5. `React` 实现 `Provider + useFeatureFlag + useExperimentVariant`
6. `Flutter` 实现 `Riverpod provider + go_router redirect + Dio interceptor`
7. 建立曝光、点击、转化三类埋点
8. 给所有事件附带 `experimentKey / variant / bucket / version`
9. 建立灰度看板和实验看板
10. 预置暂停灰度、关开关、回版本三条止血路径
11. 选低风险功能试点一次完整流程
12. 试点成功后再推广到核心业务链路

如果这 12 步你都做完了，你的项目就不再只是“有几个开关判断”，而是真正开始具备灰度发布和 `AB` 测试体系了。
