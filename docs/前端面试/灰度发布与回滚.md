# 灰度发布与回滚专题

## 使用方式

这个专题真正考察的是，团队有没有把发布当成稳定性工程来建设，而不是只会点发布按钮。成熟回答通常会覆盖发布策略、风险点、技术手段、故障处置和复盘治理这几个阶段。

---

## 一、基础认知

1. 什么是灰度发布？
答：灰度发布就是不把新版本一次性放给所有用户，而是按比例、按人群、按地区或按环境逐步放量，让团队在可控范围内验证稳定性。它的本质是把一次高风险全量变更拆成可观察、可回退、可止损的多个小步骤。

2. 为什么前端也必须理解灰度发布？
答：因为前端版本直接面向所有用户，一旦静态资源、缓存、接口兼容或关键交互出问题，影响面会非常大。前端如果只会“构建完就发”，而不理解灰度和回滚，线上风险其实是裸奔状态。

3. 灰度发布主要解决什么问题？
答：它主要解决两个问题：第一，降低新版本一次性扩散的风险；第二，让团队能在真实流量下尽早发现只有线上才会暴露的问题。很多性能、兼容性和缓存问题，在测试环境里根本看不出来，只有灰度才能把风险控制在早期小范围内。

4. 灰度发布和正式发布是什么关系？
答：灰度不是正式发布的对立面，而是正式发布过程中的风险控制阶段。理想流程通常是先小流量灰度、再观察指标、再逐步放量、最后全量，而不是把灰度当成可有可无的附加步骤。

5. 前端灰度发布相比后端有什么特殊性？
答：前端最大的特殊性是静态资源会被 CDN、浏览器和中间代理缓存，而且用户拿到什么版本未必完全可控。也就是说，前端发布不是“服务器切一下开关”那么简单，而是要同时考虑 HTML、JS、CSS、接口和缓存的一致性。

6. 面试里如何解释“发布风险”？
答：我会把它拆成代码风险、资源风险、兼容性风险、缓存风险、接口兼容风险和操作风险。真正的线上 owner 视角，不会只盯某一类单点问题，而是会主动识别这些风险如何叠加放大。

7. 什么样的需求更应该做灰度？
答：涉及核心链路、用户面广、依赖复杂、改动范围大、实验不确定性高或引入新基础设施的需求，都更应该做灰度。不是所有需求都必须同样重的发布流程，但高风险改动一定要有更强的保护措施。

8. 灰度放量时一般看什么？
答：我会重点看错误率、白屏率、接口成功率、核心转化、性能指标和用户反馈，同时结合版本维度观察是否相对旧版本显著恶化。灰度不是“放一点看看没炸就继续”，而是一个持续观测和判断的过程。

## 二、灰度、AB 实验与特性开关

9. 灰度发布和 `AB` 实验有什么区别？
答：灰度发布的目标主要是控制风险和验证稳定性，关注的是“新版本能不能安全上线”；`AB` 实验更偏产品策略验证，关注的是“哪个方案效果更好”。两者都可能用到分流能力，但目的完全不同。

10. 为什么不能把灰度和 `AB` 实验混为一谈？
答：因为如果你把实验分桶和版本放量混在一起，出了问题会很难判断到底是代码故障、流量分布问题还是实验策略本身导致。成熟团队通常会把“发布控制”和“业务实验”在概念和平台能力上尽量区分开。

11. 什么是特性开关 `Feature Flag`？
答：它是一种把功能启用状态从代码发布中解耦出来的机制。也就是说，代码可以先上线，但功能是否对某类用户生效由远端配置控制，这样既能支持灰度，也能在出现问题时快速关停功能止血。

12. 特性开关对前端有什么价值？
答：前端很多风险并不是整个版本都必须回滚，只是某个新入口、某个新组件、某个新交互有问题。通过特性开关可以把影响面收缩到功能粒度，不必每次都整包回退，响应速度会快很多。

13. 特性开关应该怎样设计才不会失控？
答：要有生命周期管理、命名规范、归属人和清理机制，避免开关越积越多变成历史债务。真正成熟的开关系统不是“能开能关”就够，而是能知道谁加的、为什么加、什么时候该删。

14. 灰度放量通常按什么维度切流？
答：常见维度有用户 id、设备 id、地区、渠道、白名单、账号等级和租户维度。怎么切取决于业务形态，但原则是分流规则要稳定可复现，否则同一个用户今天一个版本、明天另一个版本，会严重影响体验和问题定位。

15. 前端如何拿到灰度或实验配置？
答：一般通过配置中心、BFF 注入、服务端渲染注水或首屏配置接口下发。关键是保证配置获取足够早、缓存策略可控、失败时有默认行为，不然页面渲染时机和功能开关会互相打架。

16. 为什么说特性开关是一种“线上止血手段”？
答：因为它允许团队在不重新发版的情况下快速关闭高风险功能，尤其适合接口正常但某个新交互引发大量报错的场景。相比重新构建、重新发 CDN，开关止血往往更快、更小范围。

17. 做 `AB` 实验时为什么也要考虑发布风险？
答：因为实验本身可能引入新代码、新埋点和新接口依赖，如果只关注转化提升而忽略稳定性，实验同样可能造成线上故障。真正成熟的实验体系会把效果指标和稳定性指标一起纳入守护条件。

18. 如果面试官问“灰度和开关有什么关系”，怎么回答？
答：我会说灰度是一种发布策略，开关是一种实现手段，两者经常配合使用。灰度决定“放给谁、放多少”，开关决定“哪些功能真的生效”，把这两层分清，发布体系才清晰。

## 三、版本不一致与静态资源缓存

19. 为什么前端比后端更容易遇到版本不一致问题？
答：因为前端代码运行在用户浏览器里，HTML、JS、CSS、接口、配置可能来自不同时间点的缓存和 CDN 节点。一次发布期间，真实线上往往会同时存在多个版本，这和后端统一切流的控制感完全不同。

20. 什么是前端版本不一致？
答：比如用户拿到了新的 HTML，却还缓存着旧的 JS；或者主包更新了，异步 chunk 还是旧版本；又或者页面用的是新接口协议，但用户本地代码还是旧逻辑。前端故障里很典型的一类，就是这些资源组合没有落在同一版本快照上。

21. 静态资源缓存为什么既是性能手段又是发布风险来源？
答：缓存能显著降低加载时间和源站压力，但它会让资源更新传播变成一个渐进过程，而不是瞬时切换。只要缓存策略和资源命名没设计好，发布时就很容易出现“有的人看到新版本，有的人仍在旧版本”的混合状态。

22. 为什么前端静态资源通常要做文件名 hash？
答：因为 hash 能让内容变化直接反映到资源 URL，从而实现长缓存下的安全更新。这样浏览器和 CDN 可以放心缓存老资源，而新版本通过新 URL 获取，避免“文件内容变了但路径没变”导致的脏缓存。

23. 为什么入口 HTML 的缓存策略往往和 JS/CSS 不一样？
答：因为 HTML 通常负责引用最新资源清单，它更适合短缓存或协商缓存；而带 hash 的 JS/CSS 可以设置长缓存。两者配合起来，既能保证资源更新及时生效，也能尽量保留缓存性能收益。

24. 什么是“旧 HTML 引用新资源”和“新 HTML 引用旧资源”问题？
答：本质上都是版本快照不一致，前者可能导致运行时 chunk 丢失，后者可能导致代码逻辑和模板结构不匹配。发布设计里要尽量让一次构建产物作为整体被引用，而不是让不同版本资源在 CDN 边缘随机拼装。

25. 异步 chunk 更新为什么容易出线上故障？
答：因为单页应用运行过程中用户可能停留很久，主包早就加载完了，但某个页面是后来才按需加载对应 chunk。此时如果服务端已发布新版本、旧 chunk 又被清掉，就很容易出现动态导入失败和页面白屏。

26. 如何降低 chunk 加载失败的风险？
答：可以保留旧版本静态资源一段时间、为 chunk 加载失败做刷新或回退兜底、在发布平台控制清理节奏，并尽量避免引用未准备好的新清单。成熟团队会把“资源保留窗口”视作发布系统的一部分，而不是发完就立即清 CDN。

27. 什么是前端发布中的接口兼容风险？
答：就是前端新版本依赖的新字段、新状态或新协议还没完全在后端准备好，或者旧版本前端仍在访问已被后端删除的字段和接口。前端线上同时存在多版本时，接口兼容往往必须按“向后兼容”思路设计，而不是只看当前最新版。

28. 如果线上存在多个前端版本，后端接口应该如何配合？
答：理想上要保证一段时间内兼容多个前端版本，不要在发布窗口里立刻做破坏性变更。真正成熟的前后端协作不是“谁先发谁等谁”，而是一起设计版本过渡期和淘汰期。

## 四、回滚机制与发布风险控制

29. 什么是回滚？前端回滚和后端回滚有什么不同？
答：前端回滚是把用户可见版本尽快切回已知稳定状态，可能涉及 HTML、静态资源、配置和开关，而不只是回退一段代码。相比后端，前端回滚更受缓存和资源传播影响，所以“发回旧包”不一定能瞬间让所有用户恢复。

30. 前端回滚一般有哪些手段？
答：常见手段包括回切 HTML/资源清单、关闭特性开关、回退配置、切回上一个稳定构建版本，必要时还要配合 CDN 刷新和资源保留。不同故障适合的止血手段不同，不能把“回滚”理解成只有一种动作。

31. 为什么说前端回滚不一定比重新发版更快？
答：因为有些问题只影响某个功能，用开关关掉比整站回滚更快；有些问题和缓存有关，即使回滚代码，用户也不一定马上拿到旧资源。真正的目标是最快恢复可用性，而不是形式上执行了某个“回滚”命令。

32. 发布前应该做哪些风险控制？
答：我会做发布清单检查、核心链路回归、接口兼容确认、资源校验、监控告警预设、灰度计划和回滚预案确认。成熟发布不是“测完就上”，而是明确知道出了问题谁看、怎么切、怎么退。

33. 为什么发布预案和回滚预案同样重要？
答：因为真正危险的时候，团队没有时间现场发明流程。提前把故障分级、负责人、开关位置、回滚版本和观察指标都准备好，才能把处置从慌乱操作变成标准响应。

34. 如何判断应该继续放量还是停止灰度？
答：要看新版本相对基线是否出现显著异常，尤其是错误率、白屏率、关键转化和性能指标有没有恶化。如果核心路径指标已经明显下滑，就不该抱着“再观察一下”继续放量，止损永远比侥幸重要。

35. 前端发布平台最值得建设哪些能力？
答：我会优先建设版本管理、灰度分流、静态资源原子切换、发布记录、监控联动、快速回切和资源保留策略。发布平台真正的价值不是把按钮做出来，而是把线上风险控制能力产品化。

36. 什么是“原子发布”思想？
答：就是尽量让一个版本的 HTML、资源清单、静态文件和配置以可识别的完整快照方式协同生效，减少跨版本混搭。前端虽然很难做到绝对原子，但这个思路能显著降低版本不一致导致的事故。

37. 发布过程中为什么要特别关注监控和告警窗口？
答：因为很多前端故障会在发布后的几分钟内快速暴露，如果这个窗口没人看或者告警没接通，问题就会在全量前扩散。发布不是点完按钮就结束，而是要盯住一段关键观察窗口。

38. 如果面试官问“你如何设计一个安全的前端发布流程”，怎么答？
答：我会从发布前校验、按批灰度、指标守护、特性开关、资源版本管理、快速回滚和复盘治理几个阶段讲。这样能体现你关注的不只是“怎么发”，而是“怎么在真实线上把风险压到最低”。

## 五、线上故障止血与应急处理

39. 线上故障出现后，前端第一步应该做什么？
答：第一步不是马上改代码，而是先确认影响范围和故障类型：是白屏、接口失败、资源 404、配置错误、某个功能异常，还是特定地区或浏览器受影响。只有先判断问题属于版本、配置、缓存还是外部依赖，止血动作才不会乱。

40. 什么叫“止血”？
答：止血就是先把故障扩散停下来、把用户可用性尽快恢复，再去做根因修复。它强调的是优先级顺序，线上应急时最忌讳一边没控住影响面，一边直接在生产上试错。

41. 前端线上故障常见的止血手段有哪些？
答：常见包括暂停放量、关闭特性开关、回切稳定版本、恢复旧配置、保留或恢复旧资源、引导刷新以及对高风险入口临时降级。最优手段取决于问题落在哪一层，不能每次都只会“重新发一个版本试试”。

42. 如果是静态资源 404 或 chunk 加载失败，通常怎么止血？
答：我会优先确认是不是资源被提前清理、清单引用错版本或 CDN 缓存异常，然后恢复旧资源可访问性，必要时回切 HTML 到稳定清单，并给前端加刷新兜底。因为这类问题往往不是业务逻辑错，而是资源快照被打散了。

43. 如果是某个新功能导致大量报错，但主站其他功能正常，怎么处理更合适？
答：这种场景通常优先关特性开关或隐藏入口，而不是整站回滚。因为故障范围本来是局部的，最优策略应该也是局部止血，避免为了一个新模块把整站其他稳定改动一起撤回。

44. 故障处理后为什么必须复盘？
答：因为真正的稳定性提升来自复盘后的机制改进，比如补监控、补回滚预案、修缓存策略、补兼容测试和优化发布平台。如果每次都只是“修好了就算”，同类事故大概率还会再来。

45. 如果面试官让你总结“灰度发布与回滚的核心价值”，你会怎么答？
答：我会说核心价值就是把前端线上风险从“全量、不可控、发现晚”变成“分批、可观测、可止损”。这背后体现的不是某个工具，而是团队有没有把发布当成产品稳定性工程去建设。
