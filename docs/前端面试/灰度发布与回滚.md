# 灰度发布与回滚专题

## 使用方式

这个专题真正考察的是，团队有没有把发布当成稳定性工程来建设，而不是只会点发布按钮。成熟回答通常会覆盖发布策略、风险点、技术手段、故障处置和复盘治理这几个阶段。

---

## 一、基础认知

1. 什么是灰度发布？
答：灰度发布就是不把一个新版本一次性推给所有用户，而是按比例、按人群、按地区、按渠道或按环境逐步放量，让团队在可控范围内观察真实表现。它的本质不是“慢一点发”，而是把一次高风险的全量变更拆成多个可观察、可中止、可回退的小步骤，用更小的影响面去验证新版本是否真的稳定。

2. 为什么前端也必须理解灰度发布？
答：因为前端发布直接面向最终用户，而且前端的问题往往会马上变成“页面打不开”“按钮点不了”“资源加载失败”这种高感知故障。很多人误以为灰度主要是后端的事情，但前端同样会遇到缓存不一致、静态资源失效、接口不兼容、浏览器兼容问题，所以如果前端只会构建和上传，而不理解灰度和回滚，线上风险其实非常高。

3. 灰度发布主要解决什么问题？
答：它主要解决的是两个问题，一是降低新版本一次性扩散到全量用户的风险，二是让团队能够在真实流量和真实设备环境中尽早发现测试环境不容易暴露的问题。很多性能波动、浏览器兼容问题、配置异常和缓存问题，在本地与测试环境都可能表现正常，只有放到小规模真实用户上，风险才会被提前看见。

4. 灰度发布和正式发布是什么关系？
答：灰度不是正式发布的对立面，而是正式发布过程里最重要的一段风险控制机制。理想流程通常不是“开发完直接全量”，而是先在小范围流量上验证，再结合监控和反馈逐步放量，最后才到全量，所以正式发布可以理解成一个阶段性过程，而灰度是这个过程中负责探路和止损的关键环节。

5. 前端灰度发布相比后端有什么特殊性？
答：前端最大的特殊性在于代码运行在用户浏览器里，静态资源还会经过 CDN、浏览器缓存和各种中间代理，这意味着用户究竟拿到哪个版本并不总是像后端切流那样可控。也就是说，前端发布不只是“切一个服务版本”，而是要同时考虑 HTML、JS、CSS、资源清单、接口契约和缓存传播是否保持一致。

6. 面试里如何解释“发布风险”？
答：我会把发布风险拆成代码逻辑风险、静态资源风险、缓存风险、浏览器兼容风险、接口兼容风险、配置风险和操作风险几个层面。这样回答的好处是，能体现你理解发布不是“代码写得对不对”这么单一，而是一次上线过程中多个环节可能叠加放大问题，真正成熟的 owner 视角会主动识别这些风险是如何联动的。

7. 什么样的需求更应该做灰度？
答：凡是涉及核心链路、用户覆盖面大、依赖复杂、改动范围广、引入新基础设施、存在较强不确定性或者可能影响关键转化的需求，都更适合做灰度。并不是所有小改动都必须走同样重的流程，但风险越高、回滚越复杂、影响越广的变更，就越应该用更谨慎的发布策略去保护线上。

8. 灰度放量时一般看什么？
答：我通常会重点看错误率、白屏率、资源加载失败率、接口成功率、关键转化、核心路径性能指标和用户反馈，并且一定会结合版本维度和历史基线去对比。灰度不是“先放一点，没炸就继续”，而是一个持续判断的过程，要看新版本相对旧版本是否出现明显恶化，而不是只看绝对值有没有超过某个阈值。

## 二、灰度、AB 实验与特性开关

9. 灰度发布和 `AB` 实验有什么区别？
答：灰度发布的主要目标是控制上线风险、验证稳定性，核心问题是“新版本能不能安全上线”；`AB` 实验更偏产品策略和效果验证，核心问题是“哪个方案的业务表现更好”。两者都可能使用分流能力，但一个强调风险控制，一个强调方案对比，如果概念不分清，发布和实验很容易相互干扰。

10. 为什么不能把灰度和 `AB` 实验混为一谈？
答：因为一旦把版本灰度和实验分桶混在一起，出了问题时你很难判断到底是代码故障、灰度放量策略异常，还是实验本身的设计导致效果变差。成熟团队通常会尽量把“发布控制”和“业务实验”分层处理，哪怕它们都依赖同一套分流平台，也会在概念、配置和观测上区分清楚。

11. 什么是特性开关 `Feature Flag`？
答：特性开关本质上是一种把“代码是否已发布”和“功能是否对用户生效”拆开的机制，也就是代码可以先进入线上，但功能是否真正打开由远端配置控制。这样做的意义非常大，因为它把发布和启用解耦了，团队可以先把代码安全送到生产环境，再按小范围逐步开启，出了问题也能更快关停。

12. 特性开关对前端有什么价值？
答：前端很多风险并不是整个版本都坏了，而是某个新入口、某个新弹窗、某个实验模块或某段新交互逻辑出了问题。这个时候如果只能整包回滚，代价会很大；而有了特性开关，就能把故障收敛到功能粒度，快速关闭有问题的部分，保留其他已经稳定的改动，这对发布效率和止血速度都很有帮助。

13. 特性开关应该怎样设计才不会失控？
答：开关系统最怕的是只加不清，最后变成一堆谁也不敢删的历史债务，所以必须配合命名规范、归属人、创建原因、失效时间和清理机制。成熟的开关体系不只是“支持开和关”，而是能回答这个开关是为什么存在、影响谁、什么时候该删除，以及出了问题谁负责处理。

14. 灰度放量通常按什么维度切流？
答：常见的切流维度包括用户 id、设备 id、地区、渠道、白名单、账号等级和租户维度，选择哪种方式取决于业务形态和风险控制目标。这里一个很重要的原则是规则必须稳定可复现，也就是同一个用户在同一阶段最好落到同一个桶里，否则今天看到新版本、明天又回到旧版本，不仅体验混乱，也不利于问题定位。

15. 前端如何拿到灰度或实验配置？
答：常见做法包括通过配置中心下发、由 BFF 注入首屏配置、服务端渲染时注水，或者在首屏关键接口里返回配置结果。关键点不在“从哪里拿”，而在于拿得够早、失败时有默认值、缓存策略可控，否则页面渲染和开关判断可能相互打架，导致用户先看到一种状态，随后又被切成另一种状态。

16. 为什么说特性开关是一种“线上止血手段”？
答：因为它允许团队在不重新发版的前提下，快速关闭某个高风险功能，把影响面尽快收住。对于前端来说，重新构建、重新发 CDN、等待缓存传播往往没有关闭远端开关那么快，所以当主站整体正常、只是某个新功能导致大量报错时，开关往往就是最有效的第一道止血手段。

17. 做 `AB` 实验时为什么也要考虑发布风险？
答：因为实验不是只改一个文案或颜色，它常常会带来新代码、新资源、新埋点、新接口依赖，如果只盯转化提升而忽略稳定性，实验本身也完全可能变成一次线上事故。真正成熟的实验体系会把效果指标和稳定性指标同时纳入守护条件，也就是说，不仅要看谁转化高，还要看谁更稳。

18. 如果面试官问“灰度和开关有什么关系”，怎么回答？
答：我会说灰度是一种发布策略，解决的是“新版本先给谁、放多少、怎么逐步扩散”；特性开关是一种实现和控制手段，解决的是“某段功能逻辑是否真正生效”。它们经常配合使用，但不完全等价，把这层关系讲清楚，会显得你对发布系统的理解更分层。

## 三、版本不一致与静态资源缓存

19. 为什么前端比后端更容易遇到版本不一致问题？
答：因为后端版本通常运行在服务端，切流路径更集中，而前端代码散落在用户浏览器里，HTML、JS、CSS、接口和配置还可能分别来自不同时间点的缓存和 CDN 节点。结果就是同一次发布期间，线上往往会同时存在多个前端版本，这种“混合态”是前端发布天然要面对的现实。

20. 什么是前端版本不一致？
答：前端版本不一致指的是同一个用户最终拿到的 HTML、主包、异步 chunk、配置或接口语义没有落在同一个版本快照上，比如新 HTML 引用了旧 JS、旧页面逻辑访问了新接口字段、主包更新了但按需加载的 chunk 还是旧的。很多前端线上事故的本质，都是这类看不见的版本组合出了问题。

21. 静态资源缓存为什么既是性能手段又是发布风险来源？
答：缓存可以大幅降低资源加载时间和源站压力，所以它对前端性能非常重要，但缓存也意味着资源更新传播不是瞬时完成的，而是渐进发生的。只要资源命名、缓存策略或清理策略设计得不够稳妥，就会出现一部分用户拿到新资源、一部分用户还停留在旧资源的情况，从而把性能优化手段变成发布风险来源。

22. 为什么前端静态资源通常要做文件名 hash？
答：因为 `hash` 能让“资源内容变化”和“资源 URL 变化”绑定在一起，浏览器和 CDN 看到新 URL 就会把它当成新资源拉取，而不会继续命中旧缓存。它的底层思想其实很简单，就是用内容指纹避免脏缓存，让带 `hash` 的 JS、CSS 可以安全地长缓存，同时保证新版本仍然能够准确生效。

23. 为什么入口 HTML 的缓存策略往往和 JS/CSS 不一样？
答：因为入口 HTML 往往承担“告诉浏览器当前应该加载哪一组资源”的角色，它本身像一份资源索引，所以更适合短缓存或协商缓存，以便尽快指向新的版本清单；而带 `hash` 的 JS/CSS 文件内容稳定、路径唯一，更适合长缓存。两者配合起来，才能同时兼顾资源更新及时生效和缓存性能收益。

24. 什么是“旧 HTML 引用新资源”和“新 HTML 引用旧资源”问题？
答：这两种情况本质上都是版本快照不一致，只是错位方向不同。旧 HTML 引用新资源，可能导致页面逻辑和资源清单不匹配；新 HTML 引用旧资源，则可能出现新模板依赖的逻辑根本不存在，或者加载到错误的 chunk，最终都可能表现为运行时报错、白屏或者某些页面无法进入，所以发布时要尽量让一整套构建产物以完整快照协同生效。

25. 异步 chunk 更新为什么容易出线上故障？
答：因为单页应用里用户可能在页面停留很久，首次进站时加载的是主包，但某个二级页面对应的异步 chunk 可能在十分钟后才请求。此时如果平台已经发布了新版本、旧 chunk 又被清掉或清单已切换，就容易出现动态导入失败，表现成“刚才还好好的，一点某个页面就白屏”，这类问题非常典型。

26. 如何降低 chunk 加载失败的风险？
答：通常可以通过保留旧版本静态资源一段时间、让发布平台支持资源保留窗口、为 chunk 加载失败增加刷新或回退兜底、避免过早清理旧清单来降低风险。这里的核心思想是承认前端线上存在“用户停留很久才去加载后续资源”的现实，而不是假设用户一定会瞬间切到最新快照。

27. 什么是前端发布中的接口兼容风险？
答：接口兼容风险指的是前端新版本依赖的新字段、新状态或新协议还没有在后端完全准备好，或者旧版本前端还在请求的字段和接口已经被后端删掉。因为前端线上天然会同时存在多个版本，所以后端接口往往不能只面向“当前最新版前端”设计，而要考虑一个过渡窗口内的多版本兼容。

28. 如果线上存在多个前端版本，后端接口应该如何配合？
答：更成熟的做法是为前端预留版本过渡期，保证旧版本在一段时间内仍能正常工作，新字段按增量兼容方式引入，破坏性变更则延后处理。前后端协作真正需要的是共同设计“升级窗口”和“淘汰节奏”，而不是简单地互相等版本或赌用户都会立刻刷新到最新页面。

## 四、回滚机制与发布风险控制

29. 什么是回滚？前端回滚和后端回滚有什么不同？
答：前端回滚可以理解成把用户可见版本尽快切回一个已知稳定的状态，它可能涉及 HTML、资源清单、静态资源、远端配置和特性开关，而不只是“把代码回退一个 commit”。和后端相比，前端回滚最大的难点在于缓存传播不可瞬间统一，所以即使平台已经切回旧版本，也不代表所有用户立刻都恢复到了同一状态。

30. 前端回滚一般有哪些手段？
答：常见手段包括回切入口 HTML 或资源清单、关闭特性开关、回退远端配置、切回上一个稳定构建版本，以及在必要时刷新 CDN 缓存或恢复旧资源访问。不同故障适合的手段不同，回滚的核心不是动作形式统一，而是尽快把用户恢复到稳定体验，所以不能把“回滚”理解成只有一种固定操作。

31. 为什么说前端回滚不一定比重新发版更快？
答：因为有些问题只是某个功能开关开错了，直接关开关会比整站回滚快得多；有些问题是缓存导致的，即使你回滚了代码，用户也不一定马上命中旧资源。前端应急真正追求的是“最快恢复可用性”，而不是形式上一定要执行一个名叫回滚的动作，所以有时局部修复、快速补发甚至比传统回滚更有效。

32. 发布前应该做哪些风险控制？
答：我会在发布前做发布清单检查、核心链路回归、接口兼容确认、静态资源和资源清单校验、监控告警预设、灰度计划确认以及回滚预案确认。成熟发布不是“测试通过就上线”，而是上线前就明确如果出问题看哪些指标、由谁负责决策、能用什么方式最快止血。

33. 为什么发布预案和回滚预案同样重要？
答：因为真正线上出问题时，团队最缺的不是技术点子，而是时间和冷静，如果没有预案，大家很容易在高压下临时试错，反而扩大影响。提前准备好故障分级、负责人、开关位置、回滚目标版本、关键看板和观察指标，能把应急从慌乱操作变成标准化响应。

34. 如何判断应该继续放量还是停止灰度？
答：判断标准通常不是“有没有完全挂掉”，而是新版本相对旧版本和历史基线是否出现明显恶化，尤其要看错误率、白屏率、资源失败率、核心转化和关键性能指标是否有显著异常。如果核心路径已经明显变差，就不该抱着侥幸心理继续放量，灰度的意义本来就是尽早止损，而不是硬着头皮往前推。

35. 前端发布平台最值得建设哪些能力？
答：我会优先建设版本管理、灰度分流、发布记录、特性开关联动、资源清单原子切换、监控看板联动、快速回切和旧资源保留策略。发布平台的真正价值不只是“帮大家点按钮更方便”，而是把经验性的发布和回滚能力沉淀成平台化、可复用、可审计的稳定性基础设施。

36. 什么是“原子发布”思想？
答：所谓原子发布，并不是前端一定能做到像数据库事务那样绝对原子，而是尽量让一个版本的 HTML、资源清单、静态文件和配置以完整快照的方式一起生效，减少跨版本混搭。这个思想的核心，是尽量让用户拿到的始终是一组彼此匹配的产物，而不是东一块新版本、西一块旧版本。

37. 发布过程中为什么要特别关注监控和告警窗口？
答：因为很多前端故障会在发版后的几分钟内快速暴露出来，如果这个窗口没人盯、告警没接通、指标看板也没有版本对比，问题就可能在团队还没意识到时继续扩散到更大范围。发布并不是点完按钮就结束，真正危险的阶段往往是发布后那段短但关键的观察窗口。

38. 如果面试官问“你如何设计一个安全的前端发布流程”，怎么答？
答：我会从发布前检查、核心链路回归、按批灰度、指标守护、特性开关配合、资源快照管理、快速回滚和复盘治理几个阶段去讲。这样回答的重点，不是展示自己会不会发包，而是体现你理解前端发布本质上是在做一套线上风险控制流程。

## 五、线上故障止血与应急处理

39. 线上故障出现后，前端第一步应该做什么？
答：第一步通常不是立刻改代码，而是先判断影响范围和故障类型，例如是全站白屏、某个路由异常、静态资源 404、配置错误、接口不兼容，还是只影响某个地区、浏览器或渠道。只有先把问题归到大致正确的层面，后续止血动作才不会乱，否则很容易边查边试，把故障拖得更久。

40. 什么叫“止血”？
答：止血就是先把故障扩散停下来，尽快恢复用户基本可用性，再去做更彻底的根因修复。它强调的是顺序感，线上应急时最忌讳的就是一边让影响继续扩大，一边直接在生产环境里试错，所以止血本质上是一种优先级判断能力。

41. 前端线上故障常见的止血手段有哪些？
答：常见手段包括暂停继续放量、关闭特性开关、回切稳定版本、回退远端配置、恢复旧资源、临时降级高风险入口、增加刷新兜底以及必要时引导用户重新加载。止血手段没有固定模板，关键是先判断问题落在代码、配置、资源、缓存还是外部依赖，再选择影响最小、恢复最快的方案。

42. 如果是静态资源 404 或 chunk 加载失败，通常怎么止血？
答：这类问题一般会优先确认旧资源是否被过早清理、HTML 或清单是否引用错版本、CDN 缓存是否异常，然后尽快恢复旧资源访问、回切到稳定的资源清单，并在必要时给前端增加刷新或回退兜底。因为这类故障很多时候不是业务逻辑有 bug，而是资源快照被打散了，所以处理重点也应放在恢复正确资源组合上。

43. 如果是某个新功能导致大量报错，但主站其他功能正常，怎么处理更合适？
答：这种情况下通常更合适的做法是优先关掉对应特性开关、隐藏入口或局部降级，而不是整站回滚。原因很简单，故障既然是局部的，止血最好也保持局部，这样既能快速恢复大多数用户体验，也能避免把其他已经稳定的新改动一起撤掉。

44. 故障处理后为什么必须复盘？
答：因为真正能提升稳定性的不是“这次修好了”，而是通过复盘把这次事故沉淀成机制改进，比如补监控、补回滚预案、补接口兼容检查、调整缓存策略、完善发布平台和流程。没有复盘，团队下次遇到同类问题时大概率还会用同样混乱的方式再处理一遍。

45. 如果面试官让你总结“灰度发布与回滚的核心价值”，你会怎么答？
答：我会说它们的核心价值，是把前端上线从“全量暴露、难以及时发现、出了问题影响巨大”变成“分批放量、持续观察、能够快速止损和恢复”。这背后体现的不只是某个工具会不会用，而是团队有没有把发布当成稳定性工程，而不是把上线当成一次碰运气的操作。
