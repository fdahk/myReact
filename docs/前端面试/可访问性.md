# 可访问性专题

## 使用方式

这个专题的重点不是记住多少无障碍术语，而是把可访问性拆成真正可落地的实现问题：语义是否正确、键盘能不能完整操作、读屏能不能理解、焦点是否可控、自定义组件有没有把原生能力补回来。

---

## 一、基础认知

1. 什么是 Web 可访问性？
答：Web 可访问性可以理解成让更多用户在更多真实场景下都能顺利理解和操作网页，这里的“更多用户”不仅包括视障、听障、行动受限用户，也包括临时单手操作、只用键盘、网络差、屏幕小、强光环境下使用的人。它本质上不是附加功能，而是产品基本可用性的一部分，也就是页面不能只对“鼠标 + 大屏 + 视力正常 + 网络稳定”的理想用户好用。

2. 为什么前端工程师需要关注可访问性？
答：因为可访问性最终落到 HTML 结构、交互行为、焦点流转、颜色对比、错误提示这些前端直接负责的实现细节上，如果前端不关心，很多问题根本不会在后端环节被发现。除了用户覆盖面和合规要求之外，可访问性做得好还会顺带提升测试稳定性、组件语义质量和普通用户在复杂环境下的使用体验，所以它不是“额外加分项”，而是工程质量的一部分。

3. 可访问性只服务于残障用户吗？
答：不是，可访问性的核心是降低使用门槛，而不是只给某一类人做补丁，比如临时手受伤的人会依赖键盘操作，强光环境下的人更依赖清晰对比度，嘈杂环境里的用户可能看字幕而不是听声音，低性能设备用户也会受益于结构清晰、反馈明确的页面。也就是说，可访问性解决的是“人和环境的限制”，而这种限制远比我们直觉里想的更常见。

4. 什么是语义化 HTML？为什么它是可访问性的基础？
答：语义化 HTML 就是用最符合内容含义和交互角色的标签来表达页面结构，比如标题用标题标签、按钮用 `button`、导航区用 `nav`、表单字段用 `label` 关联输入框。它之所以是基础，是因为浏览器和辅助技术首先读取的不是你写的视觉样式，而是 DOM 的语义结构，语义一开始就错了，后面往往要用很多 ARIA 和脚本去补，而且还不一定补得完整。

5. 为什么说“先用原生语义标签，再考虑 ARIA”？
答：因为原生 HTML 元素本身就带着浏览器长期约定好的语义、键盘行为、焦点规则和辅助技术支持，比如 `button` 默认可聚焦、支持键盘触发、能表达禁用状态，这些都是白送的。ARIA 更像是原生语义不够时的补充说明，如果一开始就用 `div` 造所有东西，再试图靠 ARIA 补回来，既容易遗漏细节，也更容易写出“看起来有语义、实际上行为不完整”的组件。

6. `WCAG` 是什么？前端面试里至少要知道到什么程度？
答：`WCAG` 是 Web Content Accessibility Guidelines，可以理解成 Web 可访问性的主流指导标准，很多行业规范、审查要求和工具检查都和它有关。面试里通常不要求你背条文编号，但至少要知道它强调的是内容要能被感知、功能要能被操作、信息要容易理解、实现要对不同辅助技术足够稳健，也就是最好能把原则翻译成前端实践，而不是只停在标准名词层。

7. 什么是 `POUR` 原则？
答：`POUR` 指的是 Perceivable、Operable、Understandable、Robust，中文常说可感知、可操作、可理解、稳健，它基本概括了可访问性的核心目标。比如有文字替代和足够对比度属于可感知，键盘可操作和焦点明确属于可操作，表单提示清晰属于可理解，正确语义和兼容辅助技术属于稳健，所以它不是抽象口号，而是能直接映射到具体实现检查项的。

## 二、语义化 HTML

8. 为什么按钮应该优先用 `button` 而不是 `div`？
答：因为按钮的本质不是“长得像按钮”，而是“承担触发某个动作的交互语义”，而 `button` 天然就有这个语义，同时自带焦点、键盘触发、禁用态和辅助技术支持；`div` 只是一个普通容器，本来没有这些能力。你当然可以把 `div` 伪装成按钮，但那意味着你要自己补 `role`、`tabIndex`、键盘事件、禁用状态和读屏反馈，稍微漏一点就会让组件在可访问性上变得残缺。

9. 为什么链接应该优先用 `a` 标签？
答：因为链接的语义是“导航到另一个资源或位置”，浏览器、屏幕阅读器、键盘用户都会按这个预期去理解和操作它，而 `a` 标签正好原生承载了这种语义。用 `div` 模拟链接不仅会丢失默认行为，比如新开标签、复制地址、读屏识别，还会让用户和工具都更难判断它到底是跳转还是按钮触发，所以语义上应该尽量让“导航就是导航，动作就是动作”。

10. 标题层级为什么不能乱用？
答：因为标题层级不只是字体大小，而是页面信息结构的骨架，很多屏幕阅读器用户会直接按标题跳转浏览页面，像看目录一样快速理解内容层次。你如果只是为了样式好看就随便跳级，比如页面里突然从 `h1` 跳到 `h4`，或者用一堆标题标签纯做大字，视觉上可能没问题，但结构上会让辅助技术误判内容组织方式。

11. 列表、表格、表单为什么都应该尽量使用语义化标签？
答：因为这些内容本身就带有清晰的结构关系，列表有项与项的关系，表格有行列和表头关系，表单有标签、字段、分组和错误信息关系，而原生标签会把这些关系天然暴露给浏览器和辅助技术。越早用对语义标签，后面越不需要靠大量 ARIA、额外脚本和复杂说明去补救，也更容易让测试工具和自动化检查准确理解页面结构。

12. `label` 和表单控件的关联为什么重要？
答：因为对表单来说，用户最先需要知道的不是“这里有个框”，而是“这个框是干什么的”，而 `label` 正是在给输入控件提供稳定名称。它不仅能帮助屏幕阅读器把字段含义读出来，也让普通用户点击标签文字时直接聚焦输入框，所以 `label` 的关联其实同时提升了语义清晰度和交互便利性。

13. 图片的 `alt` 文本应该怎么写？
答：`alt` 的作用是当用户看不到图片时，用文字替代图片传达的信息，所以写法应该关注“这张图对理解页面有什么意义”，而不是机械描述成“一个图片”或者直接把文件名塞进去。比如商品图的 `alt` 重点可能是商品名称和关键特征，图标型图片如果旁边已有文字，`alt` 甚至可能不需要重复同样内容，核心标准是让非视觉用户获得等价信息，而不是获得冗余噪音。

14. 什么情况下图片可以使用空 `alt`？
答：当图片纯装饰、不承担信息传递任务，或者它表达的含义已经被周围文字完整覆盖时，可以使用空 `alt=""`，这样辅助技术会把它当作可忽略内容跳过。这里的判断标准不是“这张图漂不漂亮”，而是“去掉这张图后，用户会不会失去理解页面所必需的信息”。

## 三、ARIA 基础

15. `ARIA` 是什么？它解决了什么问题？
答：ARIA 可以理解成一套给辅助技术补充“角色、状态、关系”的语义协议，主要用于原生 HTML 表达不够的场景，比如自定义下拉、选项卡、树形控件、弹窗这些复杂组件。它解决的不是视觉问题，而是帮助屏幕阅读器等工具知道“这个元素是什么、现在是什么状态、和谁有关联”，也就是把自定义组件从“能看”进一步提升到“能被理解和操作”。

16. 为什么说“没有 ARIA 比错误的 ARIA 更好”？
答：因为错误的 ARIA 会把错误信息明确告诉辅助技术，比如一个元素明明不是按钮却硬加 `role="button"`，或者状态属性和真实行为不一致，读屏用户就会被系统性误导。换句话说，不写 ARIA 可能只是信息不够，而写错 ARIA 则可能让用户相信了错误的交互模型，所以它绝不是“多加几个属性总没坏处”的东西。

17. `role`、`aria-label`、`aria-labelledby`、`aria-describedby` 分别适合什么场景？
答：`role` 用来告诉辅助技术这个元素扮演什么角色，适合原生语义不足但交互语义明确的场景；`aria-label` 适合元素本身没有可见文本，但仍然需要一个可访问名称，比如只有图标的按钮；`aria-labelledby` 更适合页面上已经有可见标题或标签时，直接复用它作为名称；`aria-describedby` 则是补充说明、帮助文案或错误提示。最关键的是理解“名字”和“描述”不是一回事，前者回答“这是什么”，后者回答“它还有什么补充信息”。

18. `aria-hidden` 的作用是什么？
答：`aria-hidden` 是告诉辅助技术“这部分内容不需要读出来”，常见于纯装饰图标、重复视觉元素或者已经被别的更清晰文本表达过的内容。它的风险也在这里，如果你把仍然需要被理解或操作的内容随手设成 `aria-hidden="true"`，那对读屏用户来说这部分就等于不存在了，所以它应该用来减噪，而不是用来掩盖结构问题。

19. `aria-expanded`、`aria-selected`、`aria-checked`、`aria-disabled` 分别表示什么状态？
答：这几个属性都在做同一件事，就是把视觉上能看到、但辅助技术未必能自动理解的状态显式告诉出去，`aria-expanded` 表示是否展开，常见于折叠面板和菜单，`aria-selected` 表示当前选中项，常见于 Tabs 或列表，`aria-checked` 表示勾选状态，`aria-disabled` 表示不可用状态。对自定义组件来说，这些状态非常关键，因为用户看不到颜色、图标或动画时，只能靠这些语义状态知道当前界面发生了什么。

20. 什么情况下原生 HTML 足够，不需要额外加 `role`？
答：当原生元素已经能准确表达语义时，通常就不需要再额外加重复的 `role`，比如 `button` 本来就是按钮、`nav` 本来就是导航、`ul` 就是列表，再手动重复往往没有收益，甚至可能造成语义冲突。可访问性的重点从来不是属性数量，而是信息是否准确、行为是否一致，所以最理想的状态通常是“先把原生能力用对，再补真正缺的那部分”。

## 四、键盘导航与焦点管理

21. 为什么键盘可操作性是可访问性的核心部分？
答：因为很多用户并不依赖鼠标完成操作，有些人长期使用键盘，有些人是临时无法精细操作指针设备，还有一些辅助技术本身就是基于键盘导航逻辑工作的，所以一个页面如果只能点不能键盘操作，它其实就没有满足最基本的交互可达性。更现实一点说，键盘支持往往也是判断组件是否真正语义正确的一面镜子，因为很多“看起来能点”的假组件一到键盘场景就立刻暴露问题。

22. `Tab` 顺序为什么重要？
答：`Tab` 顺序决定了键盘用户在页面里移动时的路径，它应该尽量和视觉阅读顺序、任务完成顺序一致，否则用户会在页面里突然跳来跳去，不知道自己当前在哪里，也不知道下一步该到哪里。很多页面表面上“支持 Tab”，但如果顺序混乱、隐藏元素还能聚焦、弹窗外内容被穿透聚焦，体验依然会非常糟糕。

23. 什么是焦点管理？
答：焦点管理就是在页面状态变化时，主动确保焦点落在用户最该操作的位置上，让用户始终知道当前交互上下文在哪里。它常出现在弹窗打开关闭、菜单展开收起、表单校验失败、路由跳转、异步内容更新这些场景里，本质上是在维护“用户注意力和操作位置”的连续性，而不只是修一个视觉问题。

24. 模态框打开后为什么要把焦点移入弹框？
答：因为模态框一旦打开，用户当前的任务上下文就已经从背景页面切换到了弹框内容，如果焦点还留在外面，键盘用户继续按 `Tab` 时就可能跑到背景页面元素上，造成“视觉上在弹框里，操作上却在外面”的割裂。把焦点移入弹框，本质上是在明确告诉浏览器和辅助技术：现在这个区域才是当前唯一应该被操作的地方。

25. 模态框关闭后为什么要把焦点还给触发按钮？
答：因为关闭弹框并不代表用户的任务突然结束了，通常只是回到了上一步，所以最自然的做法是把焦点送回打开弹框的那个触发点，让用户能无缝继续原来的流程。否则焦点可能丢到页面顶部、浏览器地址栏或一个不可预测的位置，键盘用户会很容易“失去方向感”。

26. 为什么有些自定义下拉框、菜单、弹层在可访问性上容易出问题？
答：因为这些组件表面上只是样式复杂一点，实际上却把原生元素帮你完成的很多底层行为都接管了，比如角色语义、展开状态、选中状态、方向键导航、焦点移动、关闭逻辑和读屏提示，任何一个环节没补齐都可能出问题。也就是说，它们难的地方不是画出来，而是要像一个真正可交互的小状态机那样完整工作。

27. `Enter`、`Space`、方向键、`Escape` 在交互组件里通常各自承担什么角色？
答：一般来说 `Enter` 和 `Space` 常用于触发或确认动作，方向键常用于在同一组可选项之间移动，`Escape` 常用于关闭当前临时层或退出当前交互模式，但这些行为应尽量遵循用户对原生组件的直觉，而不是随意自定义。比如 Tab 组件、菜单、列表框、弹窗都可能用到这些按键，但实现时更重要的是遵循已有交互规范，让用户不需要重新学习。

28. 什么是焦点陷阱？它适合什么场景？
答：焦点陷阱就是把 `Tab` 导航限制在某个临时独占的交互区域内，防止用户在该区域激活期间意外跳回背景页面，最典型的场景就是 Modal、Drawer、全屏对话框这类“当前必须先处理完再离开”的组件。它的意义不是把用户关起来，而是保持当前上下文闭环，让键盘导航和视觉层级一致。

## 五、常见组件场景

29. 如何实现一个可访问的 Modal？
答：一个可访问的 Modal 至少要同时满足几个条件：语义上能让辅助技术知道它是一个对话框，有清晰标题和必要描述；行为上打开时把焦点移入、关闭时把焦点还回触发点，支持键盘关闭；交互上背景内容不能被误操作，必要时要做焦点陷阱和滚动锁定。重点不是“弹出来了”，而是它是否真的形成了一个临时独占、语义明确、键盘可控的交互上下文，即使使用了组件库也最好逐项验证这些默认行为是否真的成立。

30. 如何实现一个可访问的 Dropdown？
答：下拉组件最重要的是让用户和辅助技术都知道“谁是触发器、现在是展开还是收起、当前焦点在哪个选项上、如何选择、如何退出”，也就是触发按钮要有正确语义，展开状态要可感知，键盘能进入列表并移动选项，关闭逻辑要统一。很多问题表面看是样式或定位问题，本质上其实都是状态和焦点没有被一致管理。

31. 如何实现一个可访问的 Tabs 组件？
答：Tabs 不只是几个切换按钮，它本质上是一组彼此关联的角色和状态：有 `tablist` 作为容器，里面每个 `tab` 都应能表达是否选中，并和对应的 `tabpanel` 建立关系，同时还要支持方向键在同级标签间切换、焦点和选中态同步更新。只有把这些关系和行为一起做完整，Tabs 才不只是视觉上“像标签页”，而是真正可被读屏和键盘正确理解。

32. 如何实现一个可访问的 Accordion？
答：Accordion 的核心是每个折叠标题都应该是可操作的触发器，通常更接近按钮语义，同时要把当前是否展开的状态显式暴露出来，并让标题和内容区域之间存在明确关联。最常见的问题就是只做了视觉上的展开和收起，却没有让辅助技术知道当前面板是开着还是关着，也不知道哪个标题控制哪个内容区域。

33. 自动补全输入框为什么是可访问性的高频考点？
答：因为它把多个复杂点叠在了一起，既有输入框本身的名称和提示，也有异步出现的建议列表、当前高亮项、键盘上下选择、回车确认、读屏播报和关闭时机，所以特别容易暴露“交互、语义、焦点没有统一设计”的问题。很多面试官问这题，其实是在看你是否能意识到复杂组件不是加几个 ARIA 属性就够了，而是要完整设计状态模型和用户路径。

34. 自定义组件如何兼顾样式自由度和语义正确性？
答：我更倾向的做法是“尽量在原生元素之上做封装”，也就是把语义正确的元素当底座，再在外层扩展样式和交互，而不是一上来全部用 `div` 重新发明浏览器已经提供的能力。这样既能保留设计自由度，也能把很多焦点、键盘和语义能力继承下来，减少后面不断打补丁的成本。

35. 表单错误提示如何让屏幕阅读器也能感知？
答：核心是让输入框、错误信息和校验状态之间建立明确关系，比如错误信息要和字段关联，必要时用合适的 ARIA 状态和描述属性提示变化，并在校验失败时把焦点带到合理位置。这样用户获得的就不只是“有一段红字”，而是能明确知道哪个字段错了、错因是什么、接下来应该怎么修正。

## 六、表单可访问性

36. 为什么表单是可访问性的高频场景？
答：因为表单直接决定用户能不能完成注册、登录、提交申请、下单这类关键任务，一旦在字段命名、输入方式、校验反馈、焦点引导这些地方出问题，用户就可能无法完成核心流程。相比很多展示型页面，表单对“理解”和“操作”的要求更高，所以也最容易真实暴露可访问性问题。

37. 输入框为什么要有明确的标签？
答：因为输入框本身只是一种容器，用户真正需要知道的是这个容器收集什么信息，而这个信息最稳定、最可靠的来源就是标签。仅靠位置、占位符或者视觉排版去猜字段含义，对屏幕阅读器用户、认知负担较高的用户和复杂长表单场景都不够友好，所以标签几乎是表单语义的第一层。

38. 占位符为什么不能替代 `label`？
答：因为占位符通常在用户输入后就消失了，而且颜色对比度往往较弱，也不是所有辅助技术都把它当作稳定名称来源，所以它更适合做补充提示，而不适合承担“字段到底是什么”的主标签职责。简单说，`label` 是字段身份，placeholder 更像简短说明，两者解决的问题并不一样。

39. 错误信息、帮助文案、必填信息应该如何关联到输入框？
答：这些信息都不应该只是视觉上摆在旁边，而应该和具体输入框有清晰的程序化关联，这样辅助技术才能把它们作为字段的上下文一起读出来，用户也更容易在填写时理解规则和错误原因。表单可访问性真正关心的不是文案有没有写出来，而是用户在操作当下能不能顺畅地获得这些信息。

40. 表单校验失败时为什么要考虑焦点跳转和错误摘要？
答：因为当用户点击提交却失败时，最需要的是快速定位问题，而不是自己在长表单里重新扫描一遍所有字段，所以合理的焦点跳转和错误摘要能明显降低查错成本。尤其对长表单、分组表单、多步骤流程来说，如果没有清晰的错误定位机制，很多用户虽然“理论上能改”，但实际上会在页面里迷路。

41. 单选框、复选框、开关组件分别要注意哪些可访问性问题？
答：它们虽然外观可能差异很大，但核心检查点很像：是否有明确名称，是否有正确分组关系，当前状态是否能被感知，键盘是否能按预期操作，禁用态或必填态是否表达清楚。很多自定义开关看起来像原生控件，但如果只是画了个动画按钮，却没补齐语义和状态，那对辅助技术来说它可能只是一个普通容器。

## 七、测试与工程实践

42. 可访问性如何体现在测试里？
答：可访问性不该只靠手工体验一次，而应该进入测试体系，比如优先用语义查询验证结构是否合理，用键盘交互测试验证组件是否真正可操作，用焦点流转测试验证弹窗和菜单是否可控，再配合自动化扫描工具抓一些高频规则问题。这样做的价值在于把“偶尔检查”变成“持续回归”，让组件未来重构时也不容易把无障碍能力做丢。

43. 为什么 `getByRole` 既是测试习惯，也是无障碍习惯？
答：因为 `getByRole` 依赖的是元素真正暴露给辅助技术的角色和名称，所以一个组件如果能很自然地被 `getByRole` 找到，通常说明它在语义上至少走在正确方向上。换句话说，测试写得顺很多时候不是测试工具厉害，而是你的组件本身更符合浏览器和用户的理解方式。

44. 前端项目里如何做基础可访问性检查？
答：我通常会先抓收益最高的三类问题：语义标签是否正确、键盘是否能完整操作、焦点是否可控，因为这三类往往覆盖最核心的用户路径；然后再配合自动化工具补充检测颜色对比、无标签表单控件、重复 ARIA 等问题。这样做比一开始就追求“全面覆盖所有规范条款”更容易在真实团队里落地。

45. 你会用哪些方式验证一个组件是否可访问？
答：我会把验证分成几层，先做基础结构检查，看语义和状态暴露是否合理；再做键盘和焦点实测，看不用鼠标能不能完整走通；最后结合读屏体验和自动化工具看辅助技术表现是否正常。原因很简单，可访问性既有规则层问题，也有真实使用体验问题，只靠其中一种方法通常看不全。

46. 设计系统和组件库为什么更应该提前考虑可访问性？
答：因为设计系统和组件库是业务页面的底座，底层组件如果默认就带着正确的语义、状态和键盘行为，上层业务几乎是自动受益；反过来，如果底层控件天生就不可访问，那同样的问题会被复制到整个产品里。对中大型团队来说，把可访问性做进组件库，往往比逐个业务页面补救更省成本、也更稳定。

47. 如何在不明显增加业务开发成本的情况下提升可访问性？
答：最现实的做法不是要求所有业务都从零学习完整规范，而是优先把高频组件、表单模式、弹窗模式、测试习惯和代码模板做成默认正确，比如按钮默认用 `button`、字段默认带 `label`、弹窗默认处理焦点。这样团队日常开发时即使没有时时刻刻想着“无障碍”，也能被基础设施自然托住。

## 八、常见面试追问

48. 如果设计稿和可访问性要求冲突，你会怎么处理？
答：我不会把它理解成简单的二选一，而是先和设计一起确认目标，看看冲突到底来自视觉、布局还是交互，再尽量找到兼顾方案，比如调整对比度、增加可见文案、保留视觉风格的同时补充语义和键盘路径。实在有冲突时，也应该优先守住基本可用性和合规底线，因为“好看但很多人用不了”的方案本质上并不是完整设计。

49. 如果一个项目之前几乎没考虑可访问性，你会从哪里开始补？
答：我会优先从高频组件和关键流程入手，比如表单、弹窗、导航、菜单、下拉，因为这些地方复用高、影响大、问题也最集中，一旦修好收益会很快扩散。直接从最边缘的页面或最复杂的全量规范开始，往往投入很大但团队感知不到价值，不容易形成持续推进的正反馈。

50. 你如何向面试官解释“可访问性不是加几个 aria 属性”？
答：我会强调可访问性本质上是一整套用户体验设计，涉及正确语义、键盘操作、焦点管理、状态反馈、文案表达和组件抽象，而 ARIA 只是其中一层补充语义的工具。一个页面即使堆了很多 `aria-*` 属性，如果按钮还是不能键盘操作、焦点还是乱跳、错误提示还是无法关联，那它依然不是一个真正可访问的页面。

51. 可访问性和性能、测试、组件抽象之间有什么联系？
答：它们在工程上经常是互相促进的，语义越规范，测试越容易写成稳定的用户视角断言；组件抽象越清晰，越容易把焦点和键盘行为沉淀成默认能力；性能和渲染越稳定，辅助技术体验通常也越可靠。成熟项目里，可访问性往往不是一个孤立话题，而是和组件库、测试体系、设计系统一起演进的。

## 九、综合表达

52. 如果面试官问你“你了解可访问性吗”，你会怎么回答？
答：我会先用比较工程化的方式概括：可访问性核心看四件事，语义结构对不对、键盘能不能完整操作、焦点是不是可控、读屏和辅助技术能不能正确理解；然后再结合自己做过的表单、弹窗、下拉或组件库治理举一个具体例子。这样回答的好处是既有框架，也有落地，不会把可访问性说成一串彼此孤立的术语。

53. 如果让你讲一个你做过的可访问性改进案例，你会怎么组织？
答：我会按“问题场景、受影响用户、原来哪里不好用、我改了哪些语义和交互、怎么验证、最后怎么沉淀”这条线来讲，比如一个 Modal 原来打开后焦点还留在背景页，我补了标题关联、焦点移入、关闭回退和键盘关闭，再用键盘实测、测试用例和读屏验证效果。这样不仅能说明你知道怎么做，还能体现你理解为什么先改这个点，以及最后有没有把经验沉淀进组件库或规范里。
