# 可访问性专题

## 使用方式

这个专题的重点不是记住多少无障碍术语，而是把可访问性拆成真正可落地的实现问题：语义是否正确、键盘能不能完整操作、读屏能不能理解、焦点是否可控、自定义组件有没有把原生能力补回来。

---

## 一、基础认知

1. 什么是 Web 可访问性？
答：Web 可访问性就是让不同能力、不同设备、不同场景下的用户都能顺利理解和操作网页。它不是给少数人做特殊适配，而是让产品的基本可用性更完整。

2. 为什么前端工程师需要关注可访问性？
答：因为可访问性直接影响用户覆盖面、合规性、品牌体验和很多基础交互质量。很多所谓“正常用户”在弱网、强光、键盘操作、临时受限等场景下，也会受益于可访问性设计。

3. 可访问性只服务于残障用户吗？
答：不是，它也服务于临时受限用户、低端设备用户、键盘用户和各种复杂使用环境。可访问性的本质是降低使用门槛，而不是只给某一类人做补丁。

4. 什么是语义化 HTML？为什么它是可访问性的基础？
答：语义化 HTML 就是用正确标签表达正确结构和角色，让浏览器、读屏软件和搜索引擎都能正确理解页面。很多可访问性能力并不是后面“补出来”的，而是从一开始就依赖正确语义。

5. 为什么说“先用原生语义标签，再考虑 ARIA”？
答：因为原生标签天然自带角色、状态、键盘行为和辅助技术支持，通常最稳、成本最低；ARIA 更像是原生能力不足时的补充说明。错误的 ARIA 往往比不加更糟。

6. `WCAG` 是什么？前端面试里至少要知道到什么程度？
答：`WCAG` 是 Web 可访问性的核心指导标准，至少要知道它是设计和实现的重要依据，并理解感知、操作、理解、稳健这些原则。通常不需要背条文，但要能说明这些原则如何落到页面实现上。

7. 什么是 `POUR` 原则？
答：`POUR` 指可感知、可操作、可理解、稳健，这四个词基本概括了可访问性的核心目标。面试时如果能把后续实践点和这四个原则对应起来，会显得理解更体系化。

## 二、语义化 HTML

8. 为什么按钮应该优先用 `button` 而不是 `div`？
答：因为 `button` 天然就有按钮语义、键盘行为、禁用态和焦点支持，而 `div` 什么都没有，需要你手工补完整套行为。很多自定义组件问题，本质就是把原生语义丢掉后又没补回来。

9. 为什么链接应该优先用 `a` 标签？
答：因为链接本质是导航语义，用 `a` 才符合浏览器、键盘和辅助技术预期。用 `div` 模拟链接通常会让语义、可访问性和默认行为都变差。

10. 标题层级为什么不能乱用？
答：因为标题层级决定页面信息结构，读屏用户很依赖标题快速理解和导航内容。标题如果只为了样式随便选，会破坏整页结构感知。

11. 列表、表格、表单为什么都应该尽量使用语义化标签？
答：因为这些结构一旦语义正确，很多辅助技术能力就天然具备了，比如行列关系、分组关系、标签关联。越早用对标签，后续需要补的 ARIA 和脚本就越少。

12. `label` 和表单控件的关联为什么重要？
答：因为这能让读屏用户明确知道输入框是干什么的，也能让点击标签聚焦输入框，提升普通用户体验。表单可访问性的很多问题，其实都能先从 label 是否关联开始检查。

13. 图片的 `alt` 文本应该怎么写？
答：要写图片真正传达的信息，而不是机械写“图片”或把文件名直接塞进去。`alt` 的目标是替代视觉信息，不是重复图片存在这件事。

14. 什么情况下图片可以使用空 `alt`？
答：纯装饰、对理解内容没有帮助的图片可以用空 `alt`，让读屏直接跳过。关键标准是这张图去掉后，用户是否会丢失信息。

## 三、ARIA 基础

15. `ARIA` 是什么？它解决了什么问题？
答：ARIA 是一套给辅助技术补充语义、关系和状态的机制，主要用于原生 HTML 表达不够的自定义组件场景。它不是让页面“看起来更高级”，而是让辅助技术真正理解交互。

16. 为什么说“没有 ARIA 比错误的 ARIA 更好”？
答：因为错误的 ARIA 会向读屏软件传递错误信息，导致用户被误导。也就是说，ARIA 不是加上就加分，前提是你真的理解组件语义和状态模型。

17. `role`、`aria-label`、`aria-labelledby`、`aria-describedby` 分别适合什么场景？
答：`role` 用来定义元素角色，`aria-label` 适合没有可见文本时直接给名字，`aria-labelledby` 适合关联页面上已有可见标题，`aria-describedby` 则用来补充说明或帮助信息。这里最关键的是分清名字和描述是不同层次的信息。

18. `aria-hidden` 的作用是什么？
答：它是告诉辅助技术这部分内容不需要读，常见于纯装饰元素或视觉重复信息。要注意不要把仍然需要交互或理解的内容随手隐藏掉。

19. `aria-expanded`、`aria-selected`、`aria-checked`、`aria-disabled` 分别表示什么状态？
答：它们分别表示展开、选中、勾选和不可用状态。对于自定义折叠面板、菜单、选项卡、复选组件来说，这些状态是读屏用户理解当前界面的关键。

20. 什么情况下原生 HTML 足够，不需要额外加 `role`？
答：当原生语义已经正确表达时，就不要再重复加 `role`，否则可能造成冗余甚至冲突。可访问性不是“属性堆得越多越好”，而是信息表达得越准确越好。

## 四、键盘导航与焦点管理

21. 为什么键盘可操作性是可访问性的核心部分？
答：因为很多用户并不依赖鼠标，键盘是否能完整完成流程是页面最低可用门槛之一。只支持鼠标点击的页面，严格来说可访问性通常就是不合格的。

22. `Tab` 顺序为什么重要？
答：因为它决定用户键盘导航时的移动路径，顺序混乱会让用户迷路或跳过关键操作。很多“看起来能 tab”的页面，其实因为顺序不合理仍然很难用。

23. 什么是焦点管理？
答：焦点管理就是在合适时机把焦点放到合适位置，让用户始终知道自己当前操作到哪里。它是弹窗、菜单、下拉、表单校验这类组件的高频考点。

24. 模态框打开后为什么要把焦点移入弹框？
答：因为用户接下来要操作的上下文已经切换到弹框里，焦点也应该同步进入新上下文。否则键盘用户会继续在背景页面上游走，交互会非常混乱。

25. 模态框关闭后为什么要把焦点还给触发按钮？
答：这样用户不会丢失原来的操作位置，可以自然继续原流程。焦点回退本质上是在维护交互连续性，而不是一个“锦上添花”的细节。

26. 为什么有些自定义下拉框、菜单、弹层在可访问性上容易出问题？
答：因为这类组件把原生语义、键盘行为和焦点流转都接管了，少补一环就会出问题。它们通常不是样式复杂，而是交互状态机复杂。

27. `Enter`、`Space`、方向键、`Escape` 在交互组件里通常各自承担什么角色？
答：通常是确认/触发、激活、切换选项、退出关闭，但具体也要结合组件语义来设计，比如菜单、Tab、列表框、弹窗并不完全一样。一个更稳的原则是尽量遵循原生组件的键盘预期，不要发明反直觉交互。

28. 什么是焦点陷阱？它适合什么场景？
答：焦点陷阱就是把 `Tab` 焦点限制在某个临时独占区域内，典型场景是 Modal、抽屉、全屏对话框。它的作用是防止用户在背景页面和当前交互层之间失控游走。

## 五、常见组件场景

29. 如何实现一个可访问的 Modal？
答：关键点是正确语义、焦点移入和回退、键盘关闭、背景不可误操作，以及标题和描述可被辅助技术感知。可访问的 Modal 绝不是“能弹出来”就结束了，它本质上是在管理一个完整的独占交互上下文。
延伸：即使项目里用了组件库，也不能默认它天然可访问，焦点流转、键盘行为和读屏表现仍然要逐项验证。

30. 如何实现一个可访问的 Dropdown？
答：重点是触发按钮语义、展开/收起状态可感知、键盘可导航、选项可聚焦、关闭行为合理。很多下拉可访问性问题，核心都在状态和焦点没有被一致管理。

31. 如何实现一个可访问的 Tabs 组件？
答：要建立 `tablist`、`tab`、`tabpanel` 之间的关系，同时支持方向键切换、选中态同步和焦点管理。它其实是一个小型状态机，而不只是几个样式切换按钮。

32. 如何实现一个可访问的 Accordion？
答：核心是标题按钮可操作、展开状态可被读屏识别、内容区域和触发项关联清楚。最常见问题是只做了视觉折叠，却没把语义状态暴露出去。

33. 自动补全输入框为什么是可访问性的高频考点？
答：因为它同时涉及输入框、建议列表、当前高亮项、键盘选择、读屏提示和异步更新，复杂度很高。很多面试官会借这题看你是否真正理解“交互 + 语义 + 焦点”是一体的。

34. 自定义组件如何兼顾样式自由度和语义正确性？
答：我的原则是优先包一层原生语义元素，再在外面做样式和扩展行为，而不是一上来全用 `div` 造轮子。这样既保留了样式自由，也不容易把底层可访问性全部丢掉。

35. 表单错误提示如何让屏幕阅读器也能感知？
答：要把错误信息和输入框建立明确关联，必要时用合适的 ARIA 属性提示状态变化，并在校验失败时把焦点带到合理位置。这样用户不只是“看见红字”，而是真能知道错在哪、怎么改。

## 六、表单可访问性

36. 为什么表单是可访问性的高频场景？
答：因为表单直接决定用户能不能完成任务，一旦在输入、校验、错误反馈上出障碍，业务损失会很直接。很多最真实的可访问性问题，都是在表单里暴露的。

37. 输入框为什么要有明确的标签？
答：因为用户和辅助技术都需要知道这个输入框收集的是什么信息，仅靠位置、占位符或视觉样式都不够稳定。标签是表单理解成本最低也最关键的语义信息。

38. 占位符为什么不能替代 `label`？
答：因为占位符会消失、对比度通常较弱，而且不是稳定的可访问名称来源。它可以作为辅助提示，但不能承担主标签职责。

39. 错误信息、帮助文案、必填信息应该如何关联到输入框？
答：要让输入框和这些说明建立明确关联，让辅助技术能一起读出来，同时视觉上也要足够清晰。表单可访问性关键不是信息多，而是信息是否被正确关联。

40. 表单校验失败时为什么要考虑焦点跳转和错误摘要？
答：因为用户需要尽快知道先修哪一项、错误出现在哪，尤其长表单里这点非常重要。没有焦点引导和错误摘要，很多用户会在页面里来回找不到问题位置。

41. 单选框、复选框、开关组件分别要注意哪些可访问性问题？
答：重点都是名称是否明确、分组是否清晰、状态是否可感知、键盘是否可操作。视觉上像开关，不代表语义上就自动成立，很多自定义控件都容易在这里出错。

## 七、测试与工程实践

42. 可访问性如何体现在测试里？
答：可以体现在语义查询、键盘交互、焦点流转、自动化规则扫描和关键组件回归用例里。真正有效的可访问性不是靠手工检查一次，而是尽量进入测试和组件规范。

43. 为什么 `getByRole` 既是测试习惯，也是无障碍习惯？
答：因为你能按 `role` 正常找到元素，往往说明组件的语义至少大方向是对的。测试写得顺，很多时候反映的是组件语义设计本身健康。

44. 前端项目里如何做基础可访问性检查？
答：我会先从语义标签、键盘操作、焦点管理这三类高收益问题入手，再结合自动化扫描工具做基础检测。可访问性治理最怕一开始就想全做，结果团队完全落不下去。

45. 你会用哪些方式验证一个组件是否可访问？
答：我会结合键盘实测、读屏体验、自动化工具和测试用例一起看。因为可访问性既有规则层问题，也有真实使用体验问题，单靠一种方式不够。

46. 设计系统和组件库为什么更应该提前考虑可访问性？
答：因为底层组件一旦做对，上层业务会被动受益；底层做错，问题会在整个系统里被重复复制。对大团队来说，可访问性最值钱的地方往往就在组件库层。

47. 如何在不明显增加业务开发成本的情况下提升可访问性？
答：最现实的做法是从原生语义、组件库规范、测试习惯和高频组件治理先入手。先把最常见的问题做成默认正确，比要求每个业务都手动补救更可持续。

## 八、常见面试追问

48. 如果设计稿和可访问性要求冲突，你会怎么处理？
答：我会先和设计对齐目标，尽量寻找兼顾方案，比如样式微调、交互重排、增加辅助文案，而不是简单二选一。实在冲突时，也要优先守住基本可用性和合规底线。

49. 如果一个项目之前几乎没考虑可访问性，你会从哪里开始补？
答：我会先补高频组件和关键流程，比如表单、弹窗、导航、菜单、下拉这些。因为这些地方复用高、问题多、收益最直接，适合先建立正反馈。

50. 你如何向面试官解释“可访问性不是加几个 aria 属性”？
答：我会说它本质上是语义、键盘操作、焦点管理、状态反馈和组件抽象的一整套体验设计。ARIA 只是其中一层，不是全部答案。

51. 可访问性和性能、测试、组件抽象之间有什么联系？
答：它们其实经常互相促进：语义越规范，测试越稳定；组件边界越清晰，可访问性越容易做进默认能力；性能更稳时，辅助技术体验也更可靠。成熟项目里它们通常不是孤立治理的。

## 九、综合表达

52. 如果面试官问你“你了解可访问性吗”，你会怎么回答？
答：我会先讲语义化、键盘操作、焦点管理和读屏理解这几个核心点，再结合项目中做过的组件治理或表单改造来回答。这样能避免把可访问性答成一堆术语。

53. 如果让你讲一个你做过的可访问性改进案例，你会怎么组织？
答：我会按问题场景、影响用户、改造点、验证方式和最终效果这条线来讲。哪怕项目里没有系统做过，也可以讲一次弹窗、表单、下拉、键盘导航的具体优化。
延伸：一个更完整的案例表达，应该补上为什么先改这个组件，以及最后怎么沉淀到组件库或团队规范里。
