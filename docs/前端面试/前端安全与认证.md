# 前端安全与认证专题

## 使用方式

这个专题的重点不是背安全名词，而是说明风险如何形成、信任边界在哪里、前端能做到哪一步、又不能替代哪些后端职责。安全题答得成熟，通常会同时覆盖攻击路径、防御手段、协同边界和应急治理。

---

## 一、基础认知

1. 为什么前端也必须理解安全问题？
答：因为很多攻击入口就在浏览器和页面渲染层，前端如果缺少安全意识，等于把输入、存储、渲染和请求边界主动放开。前端不是安全的最终裁决者，但一定是第一道防线。

2. 前端安全问题的本质通常是什么？
答：本质通常是把不可信输入当成可信内容去执行、渲染或使用，再加上权限边界没有守住。换句话说，很多安全问题最后都能回到“信任边界错了”这件事上。

3. 为什么很多安全问题看似发生在浏览器里，但根因可能横跨前后端？
答：因为输入、存储、鉴权、渲染、接口校验是一整条链路，任何一环放松都可能被串起来利用。安全题答得成熟的关键，是能看到链路而不是只盯单点。

4. 如果面试官问“你理解的前端安全是什么”，你会怎么回答？
答：前端安全是围绕浏览器这一层，控制输入、渲染、存储、请求和权限展示边界，尽量减少可被利用的入口，同时和后端一起把风险控制成闭环。安全理解如果只停留在 XSS、CSRF 名词层面，通常还不够工程化。

## 二、登录态与认证

5. 常见的登录态方案有哪些？
答：常见有 Cookie/Session、Token 方案，以及在此基础上的刷新 Token、SSO、OAuth 等。更完整地看，认证方案通常要从“浏览器怎么带状态”和“服务端是否保存会话”两个维度一起理解。

6. `Cookie`、`Session`、`Token` 的关系是什么？
答：Cookie 是浏览器保存和自动携带状态的一种机制，Session 是服务端保存的会话状态，Token 是服务端签发给客户端携带的凭证。它们不是互斥概念，而是可以组合使用的不同层概念。

7. 基于 `Cookie` 的认证和基于 `Token` 的认证，各自适合什么场景？
答：同站后台系统、BFF 场景下我更容易选择 Cookie/Session，因为浏览器天然支持且服务端更容易统一控会话；前后端分离、多端接入、移动端场景下 Token 更常见。真正的选型关键是部署形态、跨域形态和安全策略。

8. 为什么很多系统会把认证信息放在 `HttpOnly Cookie` 里？
答：因为它不能被前端 JS 直接读取，可以明显降低 XSS 成功后窃取登录态的风险。它不是绝对安全，但能把最常见的一类前端可读风险先压下去。
延伸：如果面试官继续问，我会补 `Secure`、`SameSite` 和 CSRF 防御要一起设计，不能只靠 `HttpOnly`。

9. 为什么不建议把敏感 Token 随便放到 `localStorage`？
答：因为一旦页面有 XSS，`localStorage` 中的内容就能被前端脚本直接读走。它不是“绝对不能放”，而是风险暴露面更大，所以要结合场景权衡。

10. `HttpOnly`、`Secure`、`SameSite` 分别是什么？
答：`HttpOnly` 让 JS 读不到 Cookie，`Secure` 要求只在 HTTPS 下传输，`SameSite` 用来控制跨站请求时 Cookie 是否自动携带。三者分别解决的是可读性、传输安全和跨站请求风险。

11. 刷新 Token、续期、退出登录通常怎么设计？
答：常见做法是短效访问凭证 + 长效刷新凭证，访问凭证快过期时静默续期，退出登录时前后端同时清理会话和相关缓存。成熟设计里还会考虑多端登录、主动失效和异常登出场景。

12. 单点登录和普通站点登录在前端上有什么不同？
答：SSO 前端更关注跨域跳转、票据回跳、登录态同步和重定向安全，而普通站点登录更多只是当前站点内的状态建立。前端在 SSO 里承担的是跳转编排和安全兜底，而不是认证判定本身。

## 三、XSS

13. 什么是 XSS？
答：XSS 是恶意脚本被注入页面并执行，本质是把不可信内容当代码运行。它之所以危险，是因为一旦执行，攻击者就能借当前页面和当前用户上下文做很多事情。

14. 存储型、反射型、DOM 型 XSS 的区别是什么？
答：存储型是恶意内容先落库，再被其他用户页面渲染执行；反射型通常是恶意内容跟着请求被服务端回显；DOM 型则是前端自己在浏览器里把不可信数据拼成了危险 DOM。区别在传播路径，但本质都是不可信内容进入执行上下文。

15. 为什么 XSS 是前端最常见也最危险的问题之一？
答：因为它一旦成功，可以读取页面数据、劫持用户操作、发起请求、窃取可读凭证，影响范围往往不是单个组件。很多安全问题最后还能进一步扩散成认证和数据泄露问题。

16. `innerHTML`、富文本、第三方脚本为什么容易带来 XSS 风险？
答：因为这些场景都在处理动态内容，而且很容易绕过框架默认转义机制，把字符串直接当 HTML 或脚本上下文使用。风险不在 API 名字本身，而在于是否把不可信输入直接送进了可执行上下文。

17. 如何防御 XSS？
答：核心思路是输入不盲信、输出按上下文转义、富文本做白名单清洗，再结合 CSP、依赖治理和危险 API 审查。XSS 防御不是单点策略，而是多层收口。
延伸：如果项目里有 Markdown、富文本、AI 输出，我会主动强调内容清洗和危险链接过滤是高风险点。

18. 为什么 `dangerouslySetInnerHTML` 需要谨慎使用？
答：因为它本质上是绕过框架默认防护，允许你直接把 HTML 注入到页面。只有在内容来源可控、或已经过严格清洗时，它才是可接受的工程手段。

19. 如果业务必须渲染富文本，前端应该怎么做？
答：我会先做内容清洗，只保留白名单标签和属性，禁掉脚本、事件、危险 URL 和高风险样式，再根据业务决定是否做服务端清洗和二次兜底。真正成熟的回答要强调“前后端都要做，不是前端单扛”。

20. XSS 为什么会进一步威胁到登录态和用户数据？
答：因为恶意脚本一旦执行，就能读取当前页面可见信息、冒充用户发请求、窃取可读 token 或引导敏感操作。也就是说，XSS 往往是其他安全问题的放大器。

## 四、CSRF

21. 什么是 CSRF？
答：CSRF 是利用用户已登录状态，诱导浏览器自动带上认证信息去发起用户本不想发的请求。它利用的不是脚本执行能力，而是浏览器的“自动带凭证”行为。

22. CSRF 的攻击前提是什么？
答：前提通常是认证信息会被浏览器自动带上，比如 Cookie，而且服务端没有额外校验请求来源或 CSRF Token。没有自动携带凭证，就很难形成典型 CSRF。

23. 为什么基于 `Cookie` 的认证更容易被问到 CSRF？
答：因为 Cookie 默认会随请求自动发送，这正好满足 CSRF 攻击条件。相比之下，前端手动带的 Token 不会天然解决所有问题，但至少不是浏览器自动带出。

24. 如何防御 CSRF？
答：常见手段是 CSRF Token、`SameSite Cookie`、校验 `Origin/Referer`、关键操作二次确认等。真正有效的防御通常是多手段叠加，而不是押宝单一 header。

25. `SameSite Cookie` 对 CSRF 防御有什么帮助？
答：它可以限制 Cookie 在跨站请求中的自动发送，从浏览器层先挡掉一部分风险。它很有效，但也要看业务是否有跨站跳转、第三方登录等真实需求。

26. 为什么说“把接口都改成 POST”并不能真正解决 CSRF？
答：因为攻击者照样可以构造 POST 请求，攻击的关键不是请求方法，而是认证信息是否被浏览器自动带上、服务端是否做额外校验。只改成 POST 是典型的“改了表象没改根因”。

## 五、点击劫持与其他浏览器风险

27. 什么是点击劫持？
答：就是把你的页面嵌进透明或伪装的 iframe 中，诱导用户点击到真正危险的按钮或区域。它利用的是用户视觉误导，而不是脚本注入。

28. 如何防御点击劫持？
答：最直接的做法是禁止页面被未知站点 iframe 嵌套，常见是配 `X-Frame-Options` 或 CSP 的 `frame-ancestors`。对后台、支付、设置类页面尤其重要。

29. 什么是 `X-Frame-Options`？
答：它是一个响应头，用来控制当前页面是否允许被 iframe 嵌入。虽然现在很多场景会更推荐 CSP，但它依然是很常见的基础防护手段。

30. 什么是 `CSP`？它主要解决什么问题？
答：CSP 是内容安全策略，用来限制页面可以加载哪些来源的脚本、样式、图片以及是否允许内联脚本执行。它对抵御 XSS 和第三方脚本风险非常有帮助，因为它在浏览器侧进一步收紧了可执行面。

31. 为什么 `CSP` 对抵御 XSS 很重要？
答：因为即使页面里混进了恶意内容，CSP 也能进一步限制未知来源脚本、内联脚本或危险资源的执行。它不是替代输入清洗，而是给页面加一层运行时隔离带。

32. 还有哪些常见安全响应头值得知道？
答：像 `Strict-Transport-Security`、`X-Content-Type-Options`、`Referrer-Policy` 这些都值得知道。面试里不需要背所有细节，但最好知道它们分别和 HTTPS、MIME 嗅探、来源泄露控制有关。

## 六、本地存储与数据泄露

33. `localStorage`、`sessionStorage`、`Cookie` 在安全层面的主要区别是什么？
答：前两者都能被 JS 读取，适合保存低敏感、前端确实需要直接访问的数据；Cookie 则可以通过 `HttpOnly` 变成前端不可读，安全边界更强。真正的区别不在能不能存，而在是否暴露给脚本和浏览器如何携带。

34. 为什么浏览器本地存储不适合保存高敏感数据？
答：因为浏览器端天然更容易被脚本接触，高敏感数据一旦放进去，风险面会明显扩大。安全设计的原则不是“客户端保密”，而是“客户端尽量少接触真正敏感信息”。

35. 前端如何避免把敏感数据暴露到日志、URL、埋点里？
答：我会尽量不把敏感字段放在 query 中，日志和埋点做脱敏，错误上报避免原样带用户数据和凭证。很多真实的数据泄露不是黑客直接攻破，而是工程链路自己泄出去的。

36. 如果使用第三方 SDK，为什么也要关注数据泄露风险？
答：因为第三方脚本一旦拿到页面执行权，理论上也能接触到页面里的很多上下文信息。引入第三方 SDK 不只是功能选型，也是安全边界让渡。

37. 前端如何避免把环境变量误打进包里？
答：要区分公开配置和私密配置，凡是前端构建后要运行在浏览器里的东西，本质上都不能算秘密。服务端密钥、私有 token、内部地址这类内容绝不能通过前端环境变量暴露出去。

## 七、接口与业务层安全

38. 前端在接口调用层面能做哪些安全约束？
答：前端能做的包括鉴权头规范、参数基础校验、重复提交拦截、关键操作确认、幂等按钮控制和异常态兜底。它不能替代服务端鉴权，但能显著减少低级风险和误操作。

39. 为什么前端也要关注参数校验和输入边界？
答：虽然最终安全校验一定在后端，但前端先做一层可以降低脏数据、无效请求和误操作成本，也能让用户更早得到反馈。工程上这是体验和安全的共同前置。

40. 文件上传会有哪些安全风险？
答：常见风险有恶意文件、伪装类型、超大文件、可执行内容混入公开目录，以及预览链路导致的内容执行风险。前端至少要做基础类型和大小限制，但最终仍要靠服务端校验和隔离存储。

41. AI 项目里，Prompt、富文本、用户上传内容为什么都可能带来安全问题？
答：因为这些内容都可能被拼接、渲染、回显，甚至进一步喂给其他系统执行，所以天然是注入和泄露高风险点。AI 场景会把传统前端安全问题放大，而不是把它们消灭。

42. 如果一个接口需要鉴权，前端应该承担哪些职责，哪些不该前端决定？
答：前端负责正确携带凭证、控制入口、做无权限体验兜底和错误提示，但“用户有没有权限”最终必须由服务端判定。前端可以优化体验，不能定义安全真相。

## 八、工程实践

43. 如何把安全问题纳入前端工程规范？
答：我会把危险 API 使用、富文本清洗、存储规范、依赖审查、第三方脚本接入规则写进开发规范，并用 ESLint、自定义检查、代码评审卡点去落实。安全只有进入流程，才不容易变成“大家都知道但没人做”。

44. 如何在代码评审中发现潜在安全问题？
答：重点看有没有直接拼 HTML、敏感数据落地、本地存储滥用、第三方脚本越权、权限逻辑只写在前端、危险跳转没校验等。安全 review 的重点不是找语法问题，而是找信任边界有没有被打穿。

45. 如果线上出现 XSS 风险或登录态泄露风险，前端应该怎么应急？
答：先止血，比如下线危险入口、撤掉高风险脚本、清缓存、强制退出登录或切换配置开关，再配合后端排查和修复。安全应急最怕边查边扩散，所以第一步通常不是“优雅修复”，而是快速降风险。

46. 如何向面试官解释“安全不是一次性加固，而是持续治理”？
答：因为业务在变、输入来源在变、第三方依赖在变、攻击方式也在变。真正成熟的安全观不是上几条规则就结束，而是把安全纳入开发、发布、依赖治理和监控应急的持续流程。

## 九、结合真实项目

47. 如果你们项目使用 `Cookie` 维持登录态，你会怎么讲安全设计？
答：我会重点讲 `HttpOnly`、`Secure`、`SameSite`、CSRF 防护、会话续期和退出清理，再补充关键操作是否有二次确认和风控兜底。这样能体现你不是只会说“Cookie 更安全”。

48. 如果你们项目使用 `Token`，你会如何回答“为什么这样存、这样传”？
答：我会先讲业务场景，再讲存储位置、安全权衡、传输方式和续期策略。成熟回答不在于说某种方案绝对正确，而在于把风险和取舍说清楚。

49. 如果项目里接入了 AI 富文本输出、Markdown 渲染、第三方脚本，安全上要注意什么？
答：重点要防 XSS、恶意链接、危险 HTML、第三方脚本越权、敏感数据外流和不可信内容回显。AI 输出不能因为“看起来像文本”就默认安全。

50. 如果你做的是后台系统，前端安全和普通内容站相比有什么不同？
答：后台系统通常更敏感，因为权限操作多、数据更重要、误操作成本更高，所以前端在鉴权展示、敏感操作确认、日志审计、会话安全上要更谨慎。它不一定攻击面更多，但一旦出事损失通常更大。

## 十、综合表达

51. 如果面试官让你讲 XSS 和 CSRF 的区别，你会怎么组织？
答：我会先说 XSS 是脚本注入，本质是执行了不可信代码；CSRF 是利用已登录状态发起伪造请求，本质是滥用了浏览器自动带凭证。一个偏“执行恶意代码”，一个偏“借用户身份做事”。

52. 如果面试官问“为什么不建议把 Token 放 localStorage”，你会怎么回答？
答：核心不是绝对不能放，而是它对 JS 可读，一旦有 XSS 风险暴露面会明显更大，所以要结合项目架构、安全能力和前后端协同一起权衡。比起一句“localStorage 不安全”，这种回答更符合真实工程取舍。

53. 如果让你从前端视角总结一次安全治理思路，你会怎么说？
答：我会按输入治理、渲染治理、存储治理、请求治理和工程流程治理五层来讲，强调前端能做第一道防线，但最终要和后端、网关、监控一起形成闭环。安全题越往高阶答，越要体现“边界”和“协同”。
延伸：把安全原则真正讲透，通常要落到具体场景，比如富文本渲染、后台权限系统或 AI 输出展示，说明这些边界在工程里是怎么治理的。
