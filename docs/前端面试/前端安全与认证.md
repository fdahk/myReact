# 前端安全与认证专题

## 使用方式

这个专题的重点不是背安全名词，而是说明风险如何形成、信任边界在哪里、前端能做到哪一步、又不能替代哪些后端职责。安全题答得成熟，通常会同时覆盖攻击路径、防御手段、协同边界和应急治理。

---

## 一、基础认知

1. 为什么前端也必须理解安全问题？
答：因为浏览器就是用户和系统交互的第一层入口，用户输入、页面渲染、登录态存储、接口调用几乎都先经过前端，如果这层把不可信内容当可信内容处理，风险会被直接带进系统里。前端当然不能替代后端做最终鉴权和数据裁决，但它至少要守住输入、展示、存储、跳转这些最容易被利用的入口，否则很多攻击甚至不需要突破服务端就能先在页面里落地。

2. 前端安全问题的本质通常是什么？
答：本质上通常是信任边界判断错了，也就是把本来不该信任的输入、链接、脚本、配置或身份状态，当成了安全且可直接执行、渲染或使用的内容。再往底层说，安全问题往往不是“某个 API 天生危险”，而是数据在进入浏览器、进入 DOM、进入请求、进入存储时，没有经过正确校验、隔离和约束，最后让攻击者有机会借用户身份或页面执行环境做事。

3. 为什么很多安全问题看似发生在浏览器里，但根因可能横跨前后端？
答：因为真实系统里的安全链路是连续的，输入从页面进入，到接口传输、服务端存储、再回到前端渲染，任何一环放松都可能被串起来利用，比如前端没清洗、后端又回显、浏览器再执行，问题就闭环了。成熟回答不能只盯“页面里出事了”，而要能看到认证、存储、接口校验、重定向、日志和监控这些环节其实共同决定了风险是否能真正被利用。

4. 如果面试官问“你理解的前端安全是什么”，你会怎么回答？
答：我会把前端安全概括成一句话：围绕浏览器这一层，控制输入、渲染、存储、请求和权限展示的边界，尽量减少可被利用的入口，并和后端一起把认证、鉴权、监控、应急做成闭环。也就是说，前端安全不只是记住 XSS、CSRF 这些名词，而是理解哪些数据是不可信的、哪些状态不能只靠前端判断、哪些能力需要前后端协同兜底。

## 二、登录态与认证

5. 常见的登录态方案有哪些？
答：常见方案可以分成两类理解，一类是 Cookie/Session 这种“浏览器帮你带状态，服务端保存会话”的模式，另一类是 Token 这种“服务端签发凭证，客户端后续携带”的模式，实际项目里又会叠加 Refresh Token、SSO、OAuth 2.0、OIDC 之类机制去解决续期、跨系统登录和第三方授权问题。面试里如果只背名词还不够，最好能顺手说明浏览器怎么带状态、服务端是否保会话、是否跨域、多端是否共用，这些才是方案差异真正落地的地方。

6. `Cookie`、`Session`、`Token` 的关系是什么？
答：这三个词经常被混着说，但它们其实处在不同层次上，Cookie 是浏览器保存并随请求自动携带数据的机制，Session 是服务端保存用户会话状态的方式，Token 则是服务端签发给客户端、用来证明身份的一段凭证。它们不是互斥关系，比如 Token 也可以放在 Cookie 里传，Session 的会话标识也常通过 Cookie 保存，所以更准确的理解方式是：一个偏浏览器存储和传输机制，一个偏服务端状态管理，一个偏身份凭证本身。

7. 基于 `Cookie` 的认证和基于 `Token` 的认证，各自适合什么场景？
答：如果是同域或同站点的后台系统、SSR/BFF 架构、企业内网系统，我通常更容易选择 Cookie/Session，因为浏览器天然支持自动携带，服务端统一续期、失效和踢下线也更方便；如果是前后端彻底分离、多端接入、移动端和小程序都要共用认证能力，Token 模式往往更灵活。真正的选型关键不是“哪种更高级”，而是跨域形态、部署结构、登录态续期方式、是否需要服务端集中控会话，以及团队能否把对应的安全细节做完整。

8. 为什么很多系统会把认证信息放在 `HttpOnly Cookie` 里？
答：因为 `HttpOnly` 的核心价值是让前端 JavaScript 读不到这份 Cookie，这样即使页面发生 XSS，攻击者也更难直接把登录凭证偷走，从而显著降低最常见的一类登录态窃取风险。它不是万能防护，因为浏览器仍然可能在请求时自动带上 Cookie，所以通常还要结合 `Secure`、`SameSite`、CSRF 防护和服务端会话策略一起看，不能简单理解成“加了 `HttpOnly` 就绝对安全”。

9. 为什么不建议把敏感 Token 随便放到 `localStorage`？
答：因为 `localStorage` 对当前站点下执行的脚本是可读的，一旦页面出现 XSS，攻击者就可能直接把里面的 Token 读走并带到自己的环境里复用，这和只能“借当前页面发请求”相比，风险面会更大。它并不是绝对不能用，而是意味着你要接受更高的 XSS 代价，所以如果业务允许，通常会优先考虑让浏览器自动带 `HttpOnly Cookie`，或者至少把存储和续期策略设计得更谨慎。

10. `HttpOnly`、`Secure`、`SameSite` 分别是什么？
答：这三个都是 Cookie 属性，但作用完全不同，`HttpOnly` 解决的是“前端脚本能不能读到 Cookie”，也就是防可读性暴露；`Secure` 解决的是“Cookie 是否只能在 HTTPS 下传输”，也就是减少明文传输风险；`SameSite` 解决的是“跨站请求时浏览器要不要自动带上 Cookie”，也就是帮助降低 CSRF 风险。面试里如果能顺手补一句它们分别对应脚本读取、网络传输、跨站携带这三个维度，回答会更清楚。

11. 刷新 Token、续期、退出登录通常怎么设计？
答：比较常见的做法是短效访问凭证配合长效刷新凭证，访问凭证过期时间设得更短以缩小被盗后的风险窗口，接近过期时再通过刷新机制换新，从而在安全和体验之间做平衡。退出登录时不能只删前端状态，还要让服务端会话或刷新凭证失效，并考虑多端登录、被管理员踢下线、密码修改后强制失效这些真实场景，否则用户表面上“退出了”，系统里却可能还有残留会话。

12. 单点登录和普通站点登录在前端上有什么不同？
答：普通站点登录更多是在当前系统内建立登录态，关注的是登录页、表单提交、会话保持和退出清理；SSO 则还要处理跨站跳转、票据回跳、重定向参数安全、登录态同步和多系统之间的用户体验一致性。前端在这里更像认证流程的编排者，要保证跳转链路正确、来源可信、回跳地址不被滥用，而不是自己决定“用户到底算不算登录成功”。

## 三、XSS

13. 什么是 XSS？
答：XSS 的全称是 Cross-Site Scripting，可以理解成恶意脚本被塞进了页面并在用户浏览器里执行，本质上就是把本来应该当普通文本处理的不可信内容，当成了可执行代码。它之所以危险，是因为代码一旦在用户上下文里跑起来，就可能读取页面数据、伪造操作、窃取可读凭证，攻击者相当于借用了当前用户的浏览器身份。

14. 存储型、反射型、DOM 型 XSS 的区别是什么？
答：存储型 XSS 是恶意内容先被提交到数据库或评论区这类持久化位置，之后别的用户访问页面时再被渲染执行；反射型 XSS 是恶意内容跟着一次请求进来，再被服务端直接回显到响应里；DOM 型 XSS 则不一定经过服务端回显，而是前端脚本自己从 URL、`location`、`document.cookie`、接口返回等位置取数据后，拼进了危险的 DOM 或脚本上下文。三者的传播路径不同，但根因都一样，就是不可信数据进入了执行环境。

15. 为什么 XSS 是前端最常见也最危险的问题之一？
答：因为 XSS 直接发生在浏览器执行层，一旦成功，攻击者就不只是“看一眼页面”，而是可能借当前用户身份做很多动作，比如读取页面可见数据、发起带凭证请求、操控界面引导误操作、窃取本地可读信息，危害范围经常会从单点展示问题扩大成账户和数据问题。它常见，是因为前端本来就经常处理文本、富文本、URL 参数、第三方脚本和动态内容，这些全都是典型高风险入口。

16. `innerHTML`、富文本、第三方脚本为什么容易带来 XSS 风险？
答：因为这些场景都有一个共同点，就是“把外部来的字符串交给浏览器按 HTML 或脚本语义解释”，一旦内容来源不可信，浏览器就可能把本来只是字符串的东西当成标签、事件、链接甚至脚本处理。风险的关键不在于 API 名字本身，而在于是否绕过了框架默认转义、是否给了第三方脚本执行权、是否让富文本里的危险标签和属性原样进入 DOM。

17. 如何防御 XSS？
答：真正有效的 XSS 防御一定是多层的，输入侧不能盲信用户和第三方内容，输出侧要按上下文做正确转义，富文本和 Markdown 要做白名单清洗，危险 URL、事件属性和脚本标签要显式过滤，工程上再配合 CSP、依赖治理、危险 API 审查和代码评审一起收口。尤其在 AI 输出、富文本编辑器、公告系统这类场景里，要牢记“看起来像文本”不等于安全，因为它最后仍然可能被浏览器当成可执行内容。

18. 为什么 `dangerouslySetInnerHTML` 需要谨慎使用？
答：因为 React 默认会帮你做文本转义，而 `dangerouslySetInnerHTML` 的意思就是显式绕开这层保护，直接把一段 HTML 注入到 DOM 里，所以它天然就是高风险入口。这个 API 不是不能用，而是只能在内容来源可信、或者已经经过严格清洗和审计时再用，否则你等于在代码里主动打开了一个让不可信字符串变成 DOM 的通道。

19. 如果业务必须渲染富文本，前端应该怎么做？
答：我的做法会是先把“渲染”拆成“清洗再渲染”，也就是只允许业务真正需要的标签、属性和协议通过，明确禁掉脚本、事件、危险 URL、内联执行能力和高风险样式，再决定是否把清洗同时放到服务端做双重兜底。原因是富文本最大的风险不是“有 HTML”，而是 HTML 里的结构和属性可能被浏览器解释成行为，所以成熟方案一定强调白名单、双端清洗和高风险内容隔离。

20. XSS 为什么会进一步威胁到登录态和用户数据？
答：因为 XSS 一旦拿到页面执行权，攻击者就能直接借当前用户环境做事，能读到的就偷，不能直接读到的也可能通过伪造请求、诱导点击、页面劫持等方式间接利用，所以它常常是其他安全问题的放大器。换句话说，XSS 的危险不只是“页面被插了一段脚本”，而是浏览器已经把攻击者当成了当前站点的一部分去执行。

## 四、CSRF

21. 什么是 CSRF？
答：CSRF 可以理解成跨站请求伪造，它利用的是用户已经登录某个站点后，浏览器会自动带上认证信息这一特性，诱导用户从另一个恶意页面向目标站点发出本不想发的请求。它和 XSS 的区别在于，CSRF 不依赖恶意脚本先在目标站点里执行，而是利用浏览器默认的“自动带凭证”行为来冒用用户身份。

22. CSRF 的攻击前提是什么？
答：典型前提有两个，一是浏览器会自动把认证信息带上，比如 Cookie；二是服务端只看“有没有凭证”而不进一步校验请求来源、CSRF Token 或其他上下文信息。只要这两个条件同时成立，攻击者就可能在别的站点里构造一个请求，让用户不知情地替自己完成转账、改密码、发消息这类操作。

23. 为什么基于 `Cookie` 的认证更容易被问到 CSRF？
答：因为 Cookie 的一个重要特性就是浏览器在满足条件时会自动随请求发送，这恰好给了 CSRF 攻击“借用户身份发请求”的基础。相比之下，前端手动放进 `Authorization` 头里的 Token 通常不会被第三方页面天然带出，所以面试里一提到 Cookie 登录态，后面很自然就会追问 CSRF 防护。

24. 如何防御 CSRF？
答：常见方案是多层叠加，比如使用 CSRF Token 让请求里必须带一个攻击者难以伪造的随机值，再结合 `SameSite Cookie`、校验 `Origin/Referer`、关键操作二次确认、验证码或风控校验一起降低风险。这里最重要的不是死背某一个手段，而是理解 CSRF 的根因是“自动带凭证却缺少来源校验”，所以防御也应围绕“确认这次请求真的是从合法页面发出的”来设计。

25. `SameSite Cookie` 对 CSRF 防御有什么帮助？
答：`SameSite` 会限制 Cookie 在跨站上下文里的自动携带，从浏览器层面直接减少很多典型 CSRF 攻击成立的机会，所以它是非常高性价比的一道基础防线。它也不是完全无脑开启就结束了，因为像第三方登录、跨站回跳、支付回调这类场景有时真的需要跨站携带状态，所以工程上要结合业务路径选择合适模式，而不是简单一句“开了就完事”。

26. 为什么说“把接口都改成 POST”并不能真正解决 CSRF？
答：因为 CSRF 的关键从来不是 GET 还是 POST，而是浏览器会不会自动带着用户凭证把请求发出去，只要这个前提还在，攻击者同样可以构造 POST 请求。把方法改成 POST 可能对某些简单 URL 拼接攻击有一点干扰，但并没有触及“请求来源未校验”这个真正根因，所以它本质上只是改了表面，不是解决问题。

## 五、点击劫持与其他浏览器风险

27. 什么是点击劫持？
答：点击劫持可以理解成界面层的欺骗攻击，攻击者把真实页面嵌进透明或伪装的 iframe 里，再在上面覆盖诱导性内容，让用户以为自己点的是一个普通按钮，实际上点到的是被隐藏的敏感操作。它利用的不是脚本执行，而是用户视觉判断和浏览器嵌套页面的能力，所以对设置页、支付页、后台操作页尤其危险。

28. 如何防御点击劫持？
答：最直接的方法就是明确限制页面能不能被别的站点以 iframe 的形式嵌入，常见做法是设置 `X-Frame-Options` 或 CSP 里的 `frame-ancestors`，从源头阻止恶意站点把你的页面当“透明操作面板”来用。对于修改密码、管理后台、支付确认这类页面，这种防护往往属于基础配置，而不是可选项。

29. 什么是 `X-Frame-Options`？
答：`X-Frame-Options` 是一个响应头，用来告诉浏览器当前页面是否允许被 iframe 嵌套，常见值比如 `DENY` 和 `SAMEORIGIN`，前者是完全禁止嵌入，后者只允许同源页面嵌入。现在很多更细粒度的场景会更推荐用 CSP 的 `frame-ancestors`，但理解它依然很重要，因为它代表的是“页面能不能被外层框住”这一类基础防护思路。

30. 什么是 `CSP`？它主要解决什么问题？
答：CSP 全称是 Content Security Policy，也就是内容安全策略，可以理解成浏览器执行资源时的一套白名单规则，它规定页面能从哪些来源加载脚本、样式、图片、字体，是否允许内联脚本，是否允许 `eval` 之类危险能力。它主要解决的是“就算页面里混入了恶意内容，浏览器还能不能随便执行”的问题，所以在对抗 XSS、第三方脚本失控和资源来源污染时非常有价值。

31. 为什么 `CSP` 对抵御 XSS 很重要？
答：因为很多 XSS 最终都要落到“执行脚本”这一步，而 CSP 可以在浏览器侧再加一层约束，比如禁止未知域名脚本、禁止内联脚本、限制动态执行能力，这样即使前面某个环节漏了，攻击代码也不一定能顺利跑起来。它不能替代输入清洗和转义，因为脏内容还是不该进页面，但它能显著提高攻击成功的门槛，属于典型的纵深防御。

32. 还有哪些常见安全响应头值得知道？
答：像 `Strict-Transport-Security` 会要求浏览器优先走 HTTPS，减少降级和中间人风险；`X-Content-Type-Options: nosniff` 可以阻止浏览器做危险的 MIME 嗅探；`Referrer-Policy` 则能控制跳转时带出去多少来源信息，降低敏感路径泄露风险。面试里不一定要一字不差背配置，但最好知道这些响应头分别是在传输、资源解释和信息泄露层面补强浏览器安全边界。

## 六、本地存储与数据泄露

33. `localStorage`、`sessionStorage`、`Cookie` 在安全层面的主要区别是什么？
答：`localStorage` 和 `sessionStorage` 都是浏览器提供给前端脚本直接读写的存储空间，区别主要在生命周期，一个更持久，一个通常随标签页结束而消失，但它们在安全上有个共同点，就是只要脚本能执行，通常就能读到；Cookie 则既可以被前端读，也可以通过 `HttpOnly` 变成前端不可读，同时它还会在满足条件时自动随请求携带。也就是说，真正该比较的不是“谁能存数据”，而是“脚本能不能碰到、浏览器会不会自动带走、风险发生时暴露面多大”。

34. 为什么浏览器本地存储不适合保存高敏感数据？
答：因为浏览器端天然是一个更开放的执行环境，页面脚本、第三方 SDK、XSS、调试工具都可能接触到前端存储，所以高敏感信息一旦落进去，风险面就明显扩大了。安全设计的更好思路不是“客户端也要负责保密”，而是尽量让真正敏感的秘密只留在服务端，让前端只拿到完成业务所必需、而且可失效、可轮换、可限制范围的最小信息。

35. 前端如何避免把敏感数据暴露到日志、URL、埋点里？
答：我会优先避免把用户标识、Token、手机号、身份证号这类敏感信息放进 query 参数，因为 URL 很容易被浏览器历史、代理日志、监控平台和截图带走；日志和埋点则应做脱敏、采样和字段白名单控制，错误上报也不能把接口原始响应和用户输入整包带出去。很多线上泄露并不是被黑客正面攻破，而是研发链路自己把不该公开的数据扩散到了太多系统里。

36. 如果使用第三方 SDK，为什么也要关注数据泄露风险？
答：因为第三方脚本一旦在你的页面上下文里执行，理论上就拥有和你业务脚本接近的接触面，它可能接触 DOM、网络请求、用户输入、埋点数据甚至本地存储，所以引入 SDK 不只是“多了个功能”，也是把一部分执行权和数据可见性让渡给了外部代码。成熟团队通常会关注来源可信度、最小权限、版本治理、CSP 配置和是否真的有必要引入，而不是只看它“能不能用”。

37. 前端如何避免把环境变量误打进包里？
答：关键是要先建立一个正确认知：只要代码最终会发到浏览器里运行，这里面的变量原则上都不应被当成秘密，因为用户完全可以在构建产物和 DevTools 里看到它们。工程上要区分“公开配置”和“服务端秘密”，前端只接收公开配置，比如接口基址、公开埋点 key、功能开关，而真正的私钥、内部 token、数据库地址、第三方密钥必须留在服务端或网关层处理。

## 七、接口与业务层安全

38. 前端在接口调用层面能做哪些安全约束？
答：前端能做的安全约束更多是“降低误用和补齐体验边界”，比如统一鉴权头带法、规范请求封装、参数基础校验、重复提交拦截、关键操作二次确认、按钮幂等、防重放时间窗控制和异常态兜底。它不能替代服务端做真正的权限判断，但它可以减少低级错误、减少误操作，把本不该轻易发生的风险先挡在用户操作层。

39. 为什么前端也要关注参数校验和输入边界？
答：因为参数校验不只是安全问题，也是体验问题和系统成本问题，前端越早拦住明显非法、超长、格式错误、越界或空值输入，后端收到的脏请求就越少，用户也能更早得到明确反馈。这里的关键是理解“前端先校验”和“最终以后端为准”并不冲突，前者是在做第一层过滤和交互优化，后者才是安全边界的最终裁决。

40. 文件上传会有哪些安全风险？
答：文件上传的风险不只是“用户传了个大文件”，还包括伪装文件类型、恶意脚本或可执行内容混入、超大文件拖垮系统、图片或文档预览时触发内容执行、公开访问目录直接暴露危险文件等问题。前端能做的是限制大小、类型、数量和基础交互兜底，但真正决定安全的是服务端的 MIME 校验、内容扫描、隔离存储、重命名、防直链和预览策略。

41. AI 项目里，Prompt、富文本、用户上传内容为什么都可能带来安全问题？
答：因为 AI 项目里很多内容会被“接收、拼接、展示、再发送给别的系统”，这意味着它们既可能成为注入入口，也可能成为数据外流入口，比如模型输出的 Markdown、HTML、链接、脚本片段如果直接渲染，风险和传统富文本没有本质区别。AI 场景不是把前端安全问题变没了，而是让输入来源更多、链路更长、看起来像普通文本的危险内容更多，所以更需要强调清洗、隔离和权限边界。

42. 如果一个接口需要鉴权，前端应该承担哪些职责，哪些不该前端决定？
答：前端应该负责的是正确携带凭证、在 UI 层做入口控制、对无权限和失效态给出明确反馈，并避免把明显不该出现的操作入口暴露给用户；但“某个用户到底有没有权限做这件事”最终必须由服务端判定，因为前端代码、路由和按钮显隐都能被篡改。可以把前端理解成体验边界和误操作防线，而不是安全真相的定义者。

## 八、工程实践

43. 如何把安全问题纳入前端工程规范？
答：我会把安全从“知识点”变成“流程要求”，比如规定危险 API 的使用条件、富文本和 Markdown 渲染必须经过统一清洗、登录态存储有明确约束、第三方脚本引入要走审核、环境变量白名单化，再通过 ESLint、自定义规则、代码评审清单和模板化封装把这些要求落到日常开发里。安全一旦只停留在 README 里，团队很快就会回到“大家都知道，但每次都忘”的状态。

44. 如何在代码评审中发现潜在安全问题？
答：我会重点看几个高风险信号：有没有直接拼 HTML 或使用危险 DOM API，有没有把敏感数据塞进本地存储、URL、日志或埋点，有没有把权限判断只写在前端，有没有引入来源不清的第三方脚本，有没有不校验的外链跳转和富文本回显。安全 review 真正看的不是语法优不优雅，而是信任边界是否被不小心打穿了。

45. 如果线上出现 XSS 风险或登录态泄露风险，前端应该怎么应急？
答：第一目标通常不是立刻把根因讲清楚，而是先止血，比如紧急下线危险入口、关闭高风险渲染能力、撤掉第三方脚本、清理缓存、强制失效登录态、切换安全开关，然后再配合后端和安全团队追查传播范围、用户影响和修复方案。因为安全事件最怕的是边查边扩散，所以成熟应急思路一定是“先降风险，再定位，再修复，再复盘”。

46. 如何向面试官解释“安全不是一次性加固，而是持续治理”？
答：因为系统一直在变，业务会加功能，输入来源会变多，第三方依赖会升级，浏览器能力和攻击方式也会变化，所以安全不可能靠上线前做一次检查就永远解决。真正成熟的做法是把安全纳入需求评审、开发规范、依赖治理、发布检查、线上监控和应急机制里，让它像性能和稳定性一样成为持续运营的能力，而不是一次性项目。

## 九、结合真实项目

47. 如果你们项目使用 `Cookie` 维持登录态，你会怎么讲安全设计？
答：我会先说明为什么选择 Cookie，比如同站部署、BFF 架构或服务端会话管理更方便，然后再把安全细节讲完整：登录态通常放在 `HttpOnly`、`Secure`、合适 `SameSite` 的 Cookie 里，关键操作要有 CSRF 防护，会话要支持续期和服务端失效，退出登录要清理前后端状态，异常登录还要能强制下线。这样回答能体现你不是只会说“Cookie 更安全”，而是真的理解它安全在什么前提下成立。

48. 如果你们项目使用 `Token`，你会如何回答“为什么这样存、这样传”？
答：我会先交代业务背景，比如前后端分离、多端共用、跨域部署，然后说明存储位置是怎么在体验和风险之间平衡的，传输上一般放在 `Authorization` 头里，续期上是否采用短期 Access Token 加 Refresh Token，异常情况下如何失效和重登。成熟回答的重点不在于宣布“某种方案绝对正确”，而是在于把威胁模型、实现约束和工程取舍讲清楚。

49. 如果项目里接入了 AI 富文本输出、Markdown 渲染、第三方脚本，安全上要注意什么？
答：这类场景最大的共同点是“内容来源复杂、看起来像文本、实际上可能被执行”，所以要重点防 XSS、危险链接、外部脚本越权、Prompt 或内容注入、敏感信息被模型或 SDK 带走，以及不可信内容被原样回显到页面。换句话说，AI 输出、Markdown 和第三方脚本都不能因为“业务需要”就默认可信，它们进入 DOM 和请求之前都应该经过清洗、过滤和权限约束。

50. 如果你做的是后台系统，前端安全和普通内容站相比有什么不同？
答：后台系统通常数据更敏感、权限更复杂、操作后果更直接，比如删数据、配权限、发审批、改配置这类动作一旦出问题，损失往往比普通内容站大得多，所以前端在鉴权展示、危险操作确认、日志审计、会话安全、点击劫持防护上都要更谨慎。它不一定攻击面更多，但因为“单次失误的代价更大”，所以工程上通常会对登录态、权限入口和高风险操作做更严格的约束。

## 十、综合表达

51. 如果面试官让你讲 XSS 和 CSRF 的区别，你会怎么组织？
答：我会先从根因讲，XSS 的本质是页面执行了不可信代码，攻击者因此拿到了当前页面的执行上下文；CSRF 的本质是攻击者没有执行你站点里的代码，但利用浏览器自动带凭证，让用户在不知情的情况下发出了本不想发的请求。一个偏“代码注入”，一个偏“身份冒用”，前者重点防不可信内容执行，后者重点防自动带凭证请求被伪造。

52. 如果面试官问“为什么不建议把 Token 放 localStorage”，你会怎么回答？
答：我会回答成风险权衡题，而不是绝对判断题，也就是 `localStorage` 最大的问题是对脚本可读，一旦出现 XSS，Token 很可能被直接窃取并拿到站外复用，所以它让 XSS 的后果变得更严重。比起一句笼统的“localStorage 不安全”，更成熟的说法是结合业务架构说明为什么要选这个存储位置、你接受了什么风险、又用什么手段把风险收住。

53. 如果让你从前端视角总结一次安全治理思路，你会怎么说？
答：我会按输入治理、渲染治理、存储治理、请求治理和工程流程治理五层来讲，先说明哪些数据天然不可信，页面怎么避免把它们直接变成执行内容，再讲登录态和敏感信息怎么存、接口和权限怎么兜、最后再补规范、监控和应急机制如何形成闭环。安全题越往高阶答，越要体现前端知道自己的边界在哪里，既能守住第一道防线，也不会误把体验控制当成真正的安全裁决。
