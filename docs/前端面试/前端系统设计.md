# 前端系统设计专题

## 使用方式

这个专题不是考背概念，而是考你能不能把一个复杂前端系统讲成“边界清楚、模块清楚、取舍清楚”的完整方案。我在回答系统设计题时，通常会先澄清需求，再拆页面和模块，再讲状态与数据流，最后补性能、稳定性、安全和演进方式。

---

## 一、系统设计通用思路

1. 前端系统设计题和普通八股题的区别是什么？
答：八股题更像在考你知不知道某个知识点的定义、特性和常见用法，而系统设计题更像在考你能不能站在 owner 视角，把一个真实业务从需求、边界、方案到风险都讲完整。它不是单独问你会不会 React、缓存、权限或监控，而是看你能不能把这些能力串成一条真正能落地的链路，所以回答时重点不只是“我知道什么”，而是“我为什么这么设计、这样设计解决了什么问题、代价又是什么”。
2. 做系统设计题时，第一步为什么一定是澄清需求？
答：因为系统设计最怕的不是实现难，而是起点就错了，后面越说越偏。比如同样是“做一个聊天系统”，如果目标是企业内部协作工具，和面向用户的 AI 对话产品，实时性、权限、消息类型、合规要求、存储时长都可能完全不同，所以我会先把用户是谁、核心任务是什么、数据规模多大、是否多端、是否强实时这些前提问清楚，这一步本质上是在给后面的技术方案画边界。
3. 功能需求和非功能需求分别要问什么？
答：功能需求主要是问系统“要做什么”，比如有哪些角色、页面、核心流程、关键操作、数据对象和异常分支；非功能需求主要是问系统“要做到什么程度”，比如性能目标、稳定性要求、安全权限、可观测性、可扩展性、终端适配和上线风险控制。很多人只会描述功能，却忽略这些约束，但真实系统之所以难，往往不是因为按钮多，而是因为要在高并发、弱网、灰度发布和安全限制下仍然稳定工作。
4. 为什么需要先确认规模，例如用户量、数据量、实时性、终端类型？
答：因为架构从来不是越重越好，而是要和场景规模匹配。用户量和数据量决定你要不要优先考虑缓存、虚拟列表、分片加载和更强的数据治理；实时性决定你是用轮询、SSE、WebSocket 还是根本不需要实时；终端类型则决定优化重心更偏首屏、弱网、低端机，还是复杂交互和大屏渲染，所以规模其实是在告诉你“这个题应该用什么等级的方案来回答”。
5. 如何在有限时间里快速画出系统边界？
答：我会先把问题收敛到四个最关键的点：用户是谁、入口页面有哪些、核心数据从哪里来、依赖哪些外部系统。然后我会主动说明哪些部分不展开，比如支付、推荐、风控、模型策略或复杂后端编排由服务端负责，前端只讲接入边界、状态管理和交互设计，这样做的意义是避免回答发散成产品全案，也能让面试官知道我有“知道讲什么，也知道不讲什么”的边界意识。
6. 前端系统设计中最常被忽略的非功能需求有哪些？
答：最容易被漏掉的通常不是酷炫能力，而是上线之后最容易出事的那些基础能力，比如权限、安全、灰度、监控、异常兜底、版本回滚、低端机和弱网适配。它们平时在 demo 里不显眼，但生产环境里很多事故都和这些地方有关，所以系统设计如果完全不提这些内容，往往说明方案还停留在“能跑起来”，还没进入“能长期稳定运营”的层次。

## 二、架构与模块划分

7. 一个复杂前端系统通常如何拆模块？
答：我一般会先按页面和业务域拆主结构，再把请求、权限、埋点、组件库、缓存、状态管理和工具函数这类横切能力抽成基础层。这样拆的原因不是为了目录好看，而是为了让不同层级承担不同责任，页面负责装配，业务域负责业务规则，基础层负责复用和治理，否则系统做久了就会出现页面里既写交互又写协议转换、既写权限又写日志的混乱情况。
8. 页面级模块、业务域模块、通用能力模块如何划分？
答：页面级模块更像“组装层”，负责路由入口、页面布局、各子模块如何拼起来以及页面级 loading 和错误边界；业务域模块负责更核心的领域逻辑，比如订单筛选、聊天会话、工作流节点配置这种和业务直接相关的状态与行为；通用能力模块则负责请求、鉴权、缓存、日志、实验开关和基础 UI 组件。这样分层的好处是每层的变化原因更单一，后续改需求时不容易牵一发动全身。
9. 什么时候应该按业务域拆分，而不是按组件类型拆分？
答：当业务复杂、团队分工明确、模块边界相对稳定时，我会优先按业务域拆，因为真实协作往往也是按业务 ownership 分的。按业务域拆可以让“谁负责什么”更清楚，测试、联调和后续维护也更自然；而过早按组件类型拆，常见问题是表面上复用了很多组件，实际上却把上下文强耦合进了公共层，最后形成一个谁都能改、谁也不敢改的大杂烩。
10. 如何避免系统越做越像“大杂烩”？
答：我会尽早定义目录边界、依赖方向和公共层准入规则，比如什么逻辑可以进入 shared，什么逻辑必须留在业务域里，业务模块之间能否互相依赖，页面层能不能直接调底层能力。系统越做越乱，通常不是因为框架选错，而是因为没有治理复杂度，任何需求都临时找最近的地方塞进去，所以架构治理的本质就是持续阻止“短期方便、长期失控”的事情发生。
11. 什么时候需要引入状态管理？
答：当状态开始跨页面共享、多个区域联动明显、异步链路复杂，或者单靠 props 一层层传递已经让代码阅读和维护成本明显变高时，我会引入状态管理。状态管理的存在意义不是显得项目高级，而是让数据归属、更新入口和订阅关系更清楚；如果只是一个局部表单或简单弹窗，其实本地状态就足够，硬上全局状态反而是在给自己加理解负担。
12. 全局状态、局部状态、服务端状态如何划分边界？
答：全局状态适合身份信息、主题、权限、实验开关这类“很多地方都要读，而且变化不频繁”的数据；局部状态适合输入框、弹窗开关、当前 tab、局部筛选条件这种页面私有交互态；服务端状态适合列表、详情、分页结果、查询缓存这类以服务端为权威的数据。核心判断标准其实是三个问题：谁拥有这份数据、谁负责更新、谁对一致性负责，只要这三个问题答清楚，状态边界通常就不会太乱。
13. 如何设计服务层、缓存层、权限层、日志层？
答：我会把它们做成统一入口的横切层，而不是让每个业务模块自己实现一套，因为这几层本质上都是“全局治理能力”。服务层负责请求收口、协议转换和错误映射；缓存层负责数据复用、过期策略和失效时机；权限层负责规则判断和展示控制；日志层负责埋点字段统一、错误上报和排障串联。这样设计的价值在于，很多系统前期看不出差别，但一旦业务变复杂，统一层能显著降低改动成本和线上排障成本。

## 三、接口与数据流

14. 系统设计题里，前端为什么也必须关注 API 契约？
答：因为 API 契约决定的不是一个字段名，而是前端的数据模型、错误处理方式、缓存粒度、兼容成本和后续演进空间。所谓契约，本质上就是前后端对“数据长什么样、什么时候返回什么、失败了怎么表达”的共同约定，如果这个约定混乱，前端页面就会被迫写大量兼容分支和转换逻辑，最后页面层会变成真正的复杂度垃圾桶。
15. REST、GraphQL、WebSocket、SSE 各适合什么场景？
答：REST 更适合资源明确、语义稳定的常规 CRUD 场景，因为接口直观、治理成熟；GraphQL 更适合前端页面字段差异大、需要灵活组合数据的场景，但也会引入 schema 治理和缓存复杂度；WebSocket 适合强双向、高频实时通信，比如协同编辑、实时聊天；SSE 或 HTTP 分块流更适合服务端单向持续推送，尤其是 AI 流式生成。选型时我更关注通信模式是否匹配，而不是哪个名字更流行。
16. 分页、增量加载、无限滚动各适合什么场景？
答：传统分页更适合目标明确、需要定位、回访和跳页的场景，比如后台检索和管理系统，因为用户经常需要知道“我现在在第几页”；增量加载和无限滚动更适合内容消费型场景，用户关注的是连续阅读体验。这里本质上是在平衡三件事：阅读流畅度、定位能力和实现复杂度，所以不能简单把无限滚动理解成更高级的分页。
17. 如何处理实时更新和本地缓存的冲突？
答：我会先明确谁是权威数据源，通常服务端或实时流更接近权威，然后再通过版本号、时间戳、消息序列或业务合并规则决定如何覆盖。冲突处理的本质是在解决一致性问题，也就是同一时刻不同来源的数据谁算真，如果这件事没定义清楚，就很容易出现用户刷新前后看到两个版本、列表和详情不一致，甚至自己刚操作的数据又被旧缓存顶掉。
18. 乐观更新适合什么场景？风险是什么？
答：乐观更新适合成功率高、失败代价低、用户又很需要即时反馈的场景，比如点赞、开关切换、排序拖拽，因为它可以让界面看起来更灵敏。它的风险在于失败回滚、并发冲突和本地状态与服务端状态脱节，所以只有在操作语义清晰、回退逻辑可定义时才值得使用，否则用户看到“先成功后撤回”的反复状态，体验反而会更差。
19. 如何设计前端请求层，避免竞态、重复请求和错误处理散落？
答：我会建立统一请求层，把取消请求、请求去重、超时、鉴权、重试、错误映射和基础埋点统一收口，让业务页面只关心“要拿什么数据”和“失败后业务上怎么表现”。请求层本质上是在做链路治理，如果每个页面都自己写一套 loading、error、cancel 和 retry，短期看很灵活，长期看会造成行为不一致、排障困难以及竞态问题到处重复出现。
20. 如何设计离线缓存或草稿保存？
答：我会先区分哪些数据允许本地保存，哪些必须以服务端为准，然后根据数据体量和结构选择 `localStorage`、`IndexedDB` 或服务端草稿接口。草稿保存真正难的不是“先存下来”，而是恢复时如何知道它对应哪个版本、是否和线上数据冲突、是不是过期草稿，以及用户是否能理解当前恢复的是“最新编辑”还是“历史残留”，所以版本和恢复语义一定要设计清楚。

## 四、性能与体验

21. 系统设计题里前端性能通常要从哪些维度回答？
答：我一般会从加载性能、渲染性能、交互性能和资源策略四个维度回答。加载性能讲的是用户多久能看到并开始操作，渲染性能讲的是列表、表格、图表这类 UI 在更新时会不会卡，交互性能讲的是输入、拖拽、切换时是否及时响应，资源策略则关注图片、脚本、缓存和分包如何配合，所以这样回答比只说“做懒加载”更像一个完整系统的性能设计。
22. 首屏加载慢时，常见的优化策略有哪些？
答：我会先判断瓶颈是在网络、资源、脚本还是接口，然后再选策略，因为不同瓶颈对应的手段差别很大。比如资源重就做拆包、压缩和关键路径收敛，脚本重就做按需加载和减少首屏执行量，接口慢就做并行请求、降级和缓存，必要时再考虑 SSR 或预渲染。性能优化的底层思路不是堆技巧，而是先找到最长木板，再优先处理收益最大的那一段。
23. 长列表、复杂表格、大量图表时，如何设计渲染方案？
答：我会优先从“减少一次渲染多少东西”和“减少每次更新影响多大”两个方向设计，也就是通过虚拟化、按需渲染、局部刷新、计算缓存和组件拆分，把 DOM 数量和重渲染范围控制住。很多人只关注页面能不能显示出来，但真正难的是数据变了以后还能不能流畅更新，所以渲染方案的核心其实是在控制更新成本，而不只是首渲染成本。
24. 虚拟列表的核心思想是什么？适合什么场景？
答：虚拟列表的核心思想是只渲染用户当前看得到、以及马上可能看到的那一小段内容，把真实 DOM 数量控制在一个稳定区间，从而降低内存、布局和绘制压力。它特别适合数据量大、单项结构相对稳定的列表，但如果每一项高度变化非常大、内部又有大量异步内容，那么虽然仍然可以做虚拟化，滚动锚点、动态测量和回流控制的复杂度就会明显上升。
25. 如何处理高频输入、搜索建议、自动保存这类高交互场景？
答：我会把这类问题拆成输入反馈、请求调度和状态一致性三个层面处理。输入反馈要保证字符一敲就有响应，所以局部状态要足够轻；请求调度要用防抖、节流、取消和最新结果优先来避免无效开销；自动保存则要明确“Saving”“Saved”“Save failed”这类状态，让用户知道系统现在处于什么阶段。高交互场景真正可怕的往往不是慢，而是用户不知道刚才那次输入到底有没有被系统正确接住。
26. 什么时候应该用 Worker、懒加载、分片调度？
答：如果瓶颈是 CPU 计算，比如大文件哈希、复杂解析、图片处理，我会优先考虑 Worker，把重计算移出主线程；如果瓶颈是资源体积和首屏路径过长，我会用懒加载；如果任务本身必须留在主线程，但单次执行太长，我会通过分片调度把长任务拆开，减少页面卡死感。它们的底层目标都是改善响应性，但分别解决的是计算阻塞、资源阻塞和长任务阻塞三个不同问题。
27. 系统设计里如何回答“移动端弱网和低端机适配”？
答：我会从资源体积、缓存策略、骨架屏、超时与重试、图片质量分级、降级方案和长任务控制这几个角度回答，因为弱网和低端机的本质问题是“资源来得慢、机器算得慢”。适配不是让页面勉强打开就算完成，而是要保证核心路径在差环境下依然可用、提示清楚、失败可恢复，否则用户虽然 technically 能打开页面，但实际上并不能顺利完成任务。

## 五、安全、权限、稳定性

28. 系统设计题里为什么不能漏掉权限和认证？
答：因为很多系统不是“功能写完就能上线”，而是“权限和认证没做好根本不允许上线”。认证解决的是“你是谁”，权限解决的是“你能做什么”，这两层共同决定了系统的安全边界。如果系统设计完全不提这一层，通常说明方案还停留在 demo 视角，没有真正考虑生产环境里用户隔离、数据保护和操作风险的问题。
29. 页面级权限、路由级权限、按钮级权限如何设计？
答：我一般会让路由级和页面级权限先拦住大的访问边界，比如某个页面能不能进、某个模块能不能看；按钮级权限再负责更细粒度的交互控制，比如能不能编辑、能不能删除、能不能发布。这样分层的意义是既保证整体访问路径清晰，也让局部操作控制更精细，不过我一定会补一句，真正的数据权限和操作权限必须由服务端兜底，前端更多是做体验层控制。
30. 如何设计统一错误处理和异常提示？
答：我会先把错误分层，比如网络错误、业务错误、权限错误和前端运行时错误分别走不同处理路径，再统一文案规范、提示方式、兜底页和日志上报。统一错误处理的意义不只是用户看到的 toast 长得一致，更重要的是系统对错误有明确入口和降级策略，避免每个页面都自己拍脑袋决定是 silent fail、弹窗还是直接崩掉。
31. 前端监控通常要覆盖哪些维度？
答：我会覆盖性能指标、JS 报错、资源加载失败、接口异常、用户行为路径、环境信息、版本信息以及必要的业务关键路径。监控真正的价值不是做几张漂亮图表，而是线上出问题时能快速回答几个核心问题：是哪个版本出了问题，是资源层、接口层还是代码层出了问题，只在某些浏览器发生还是全量用户都受影响。
32. 如何设计日志、埋点和问题定位链路？
答：我会让日志和埋点尽量共享统一字段体系，比如用户标识、页面、会话、requestId、版本号、实验信息和时间点，这样用户反馈、接口日志和前端错误才能串起来。问题定位最怕的是每一层都在记录信息，但字段彼此对不上，结果排障时只能靠猜，所以统一上下文标识其实是在给未来的线上问题提前修路。
33. 如果系统支持灰度发布或特性开关，前端应该怎么接入？
答：前端需要能根据开关决定功能是否展示、是否走新链路、是否打不同埋点，并且最好具备快速关闭和回退能力。灰度的底层思想不是“先给少数用户看新功能”这么简单，而是用流量分层来控制风险扩散半径，所以前端接灰度时不仅要会读开关，还要考虑开关变化后的状态同步、埋点分层和异常回退。

## 六、典型题型：聊天系统

34. 设计一个聊天系统前端，你会先问哪些需求？
答：我会先确认这是单聊、群聊还是 AI 对话，是否要求强实时，消息类型有哪些，历史消息量多大，是否需要已读回执、多端同步、离线消息、草稿和引用回复。聊天系统看起来像一个消息列表加输入框，但实际上对顺序、一致性、实时性和异常恢复都很敏感，所以需求不问清楚，后面很容易一开始就把协议、状态模型和通信方式选错。
35. 聊天列表、消息列表、输入框、会话切换分别有哪些关键状态？
答：聊天列表要关注当前会话、未读数、排序规则和同步状态；消息列表要关注加载态、发送中、发送失败、分页锚点和消息去重；输入框要关注草稿、附件、快捷输入和发送中状态；会话切换则要重点处理旧请求取消、新会话初始化和上下文隔离。把这些状态拆清楚，本质上是在把“一个看似简单页面”还原成多个小状态机，这样系统复杂度才不会全压在一个组件里。
36. 如何处理消息实时更新？
答：我会先根据实时性和基础设施选择 WebSocket、SSE 或轮询，然后在前端设计消息去重、乱序处理、本地发送回填、重连恢复和异常提示。实时更新真正难的不是“收到一条消息”，而是“收到以后顺序正确、状态正确、列表不抖、未读数也正确”，所以这是一道数据一致性题，不只是通信题。
37. 如何处理历史消息分页加载？
答：我通常会设计成向上分页加载，因为聊天阅读习惯更接近“看旧消息时向上翻”，同时我会特别处理滚动锚点，确保加载历史消息后用户当前看的内容不会突然跳走。这里的核心不是分页本身，而是阅读上下文要保持稳定，否则即便接口设计得没问题，用户也会觉得这个聊天体验很“飘”。
38. 如何避免消息重复、乱序和滚动位置跳动？
答：我会要求消息模型有稳定唯一 id，接收时先做去重，再根据时间戳、服务端序列或业务规则插入正确位置，渲染时同时维护滚动锚点。重复、乱序和跳动其实是同一类问题的不同表现，本质上都是数据模型和视图更新策略没有协同好，所以解决它不能只在 UI 层补丁式修修补补。
39. AI 聊天场景下，流式输出如何设计？
答：我会把流式回答设计成明确状态机，至少覆盖请求中、首段到达、持续生成、工具调用中、完成、失败、取消和重试，并让每个 chunk 都带会话上下文和 requestId。AI 聊天前端真正难的地方不在于“文本一段段显示出来”，而在于生成过程中任何一步中断、切会话、重试、工具调用插入后，页面状态仍然要能自洽，这也是为什么状态机在 AI 产品里格外重要。
40. 会话切换时如何避免旧请求污染新会话？
答：我会同时做两层防线：切换时尽量取消旧请求，结果落地前再根据当前会话 id 或 requestId 做一次校验。这样即使网络抖动导致旧请求晚返回，也不会把旧内容错误渲染到新会话里。这个问题的底层原理其实就是异步上下文隔离，也就是让每个请求从发起到落地都知道自己到底属于谁。

## 七、典型题型：Feed / 信息流

41. 设计一个信息流页面时，你会优先考虑哪些问题？
答：我会先考虑内容形态、刷新方式、首屏策略、列表性能、广告或推荐混排、曝光埋点和阅读连续性，因为信息流的难点不只是“数据很多”，而是内容组织、资源加载和滚动体验彼此强相关。比如一条卡片里既可能有图片、视频、广告，又要保证滚动顺滑和曝光准确，所以这类页面更像是“渲染系统设计”，而不只是普通列表。
42. 无限滚动和分页按钮各有什么优缺点？
答：无限滚动的优点是消费体验更连贯，用户不容易被“下一页”打断，缺点是不利于定位、回访和恢复上下文；分页按钮更可控，也更适合检索和后台场景。选择哪种方式，本质上取决于用户是在持续浏览内容，还是在带着目标找内容，所以这不是技术优劣问题，而是交互目标匹配问题。
43. 如何处理图片、视频、广告混排？
答：我会先统一卡片数据模型和渲染协议，再针对不同内容类型分别做懒加载、按需渲染和曝光控制，避免每种卡片都自成体系。混排场景最容易出问题的地方是“不同卡片各写各的逻辑”，时间一长就会出现滚动性能差、埋点口径不一致、广告插入打乱排序等问题，所以先抽象统一协议通常比先写具体卡片更重要。
44. 如何设计懒加载和预加载策略？
答：我的原则是优先保证首屏关键内容尽快可见，对屏外内容做懒加载，对用户高概率下一步会看到的资源做适度预加载。预加载不是越多越好，因为它本质上是在拿当前资源去赌未来命中率，如果赌错了，就会白白消耗带宽、内存和 CPU，所以策略重点在“关键内容优先”和“高命中才预取”。
45. 实时插入新内容时如何避免用户阅读体验被打断？
答：我一般不会直接把新内容硬插进用户当前视口，而是先通过“New updates”提示条、角标或回到顶部后再合并的方式处理。因为信息流场景里，内容新鲜度固然重要，但阅读连续性更重要，用户正在读一半时突然被页面顶动，体感往往会比晚几秒看到新内容更差。

## 八、典型题型：Dashboard / 后台系统

46. 设计一个后台 Dashboard，你会如何拆页面和数据模块？
答：我会从用户的操作视角拆页面，比如总览区负责全局状态感知，筛选区负责输入查询条件，图表区和表格区负责结果展示，详情区负责下钻解释；数据模块则按业务主题和查询维度拆分。这样拆的价值在于页面结构和数据流是对应的，后面联动、缓存和刷新策略也更容易围绕“当前筛选条件 + 当前选中对象”这两个核心上下文展开。
47. 图表、筛选器、表格、详情面板之间如何联动？
答：我会先定义主状态源，通常是筛选条件和当前选中项，然后所有图表、表格和详情面板都围绕这份状态派生刷新，而不是每个区域自己维护一套查询条件。联动系统最怕的就是多份状态互相打架，比如图表显示的是 A 条件，表格却还是 B 条件，所以要先保证状态源唯一，再考虑局部组件怎么高效订阅。
48. 如何设计定时刷新和手动刷新？
答：我会让自动刷新可配置、可暂停，并且和页面可见性、用户是否正在操作联动，比如用户正在编辑筛选器时就不要突然刷新；手动刷新则尽量复用同一套请求和状态逻辑。这样设计的底层原因是自动和手动本质上都只是“重新获取数据”的不同触发方式，如果实现两套链路，很容易让 loading、错误态和缓存行为出现不一致。
49. 多筛选条件、多图表联动时如何管理状态？
答：我会把筛选条件集中管理，再让图表、表格和详情订阅同一份查询状态，同时根据交互频率决定哪些筛选即时生效，哪些采用延迟提交。这样做是为了避免用户每点一下筛选都触发整页多轮刷新，尤其在图表多、接口慢的场景里，若没有请求取消、局部缓存和节流控制，页面很容易出现闪烁和状态错位。
50. 如何控制大表格、大图表的渲染成本？
答：我会从数据裁剪、虚拟化、局部刷新、懒渲染、计算缓存和实例复用几个方向同时控制成本，因为后台系统的卡顿通常不是单点问题，而是数据多、图表多、联动多共同叠加的结果。比如表格可以限制首屏列数和行数，图表可以按区域延后初始化，筛选变化时尽量局部更新而不是整页重建，这样整体体验才会稳。

## 九、典型题型：编辑器 / 可视化平台

51. 设计一个低代码编辑器或工作流编排平台，你会如何划分模块？
答：我会拆成画布区、组件或节点面板、属性配置面板、调试运行区、版本管理和底层协议层，因为这类系统本质上不是单页展示，而是多个区域共同编辑同一份复杂数据。模块划分围绕统一状态源来做，才能保证后面拖拽、选中、配置修改、运行调试这些行为都在同一个数据世界里发生，而不是每个面板各自维护一份近似但不一致的状态。
52. 画布、配置面板、图层树、调试面板之间如何同步状态？
答：我会维护一份统一的编辑态数据结构作为核心状态源，其他面板围绕它读写和派生。比如画布决定当前选中节点，配置面板读写这个节点的属性，图层树反映结构层级，调试面板读取当前版本对应的运行结果。统一状态源的意义，是把“同步”问题变成“订阅同一真相源”问题，这样复杂度会低很多。
53. 为什么这类系统通常需要更强的状态管理和协议设计？
答：因为这类产品往往同时存在拖拽、选中、批量编辑、撤销重做、版本切换、运行态调试和多人协作等复杂交互，如果状态模型不稳，任何一个功能都可能把别的功能带坏。所谓协议设计，讲人话就是先把“节点长什么样、事件怎么表达、每种操作怎么落地”规定清楚，否则项目做到后面很容易进入一种大家都能改、但没人敢重构的脆弱状态。
54. 如何设计撤销、重做、草稿保存？
答：我会把用户操作抽象成可记录的变更单元，维护历史栈和当前指针，再在关键节点或固定周期把草稿持久化。这样做比每次都存整个页面快照更稳，因为它更适合做精细回滚、差异回放和冲突判断。撤销重做本质上是在管理状态变更历史，所以设计重点不是按钮本身，而是变更记录粒度是否合理。
55. 如何处理大量节点或组件时的性能问题？
答：我会考虑局部更新、画布虚拟化、事件节流、计算隔离、缩略视图和重型逻辑 Worker 化，因为编辑器类场景的性能瓶颈往往出现在高频交互，比如拖拽、缩放、框选和连续属性修改。换句话说，编辑器性能优化更像是在保护交互链路，而不是单纯优化首屏加载，所以不能只看 bundle 大小，要看操作时是不是会掉帧。

## 十、典型题型：AI 平台

56. 设计一个 `AI Agent` 平台前端，你会先确认哪些能力？
答：我会先确认会话管理、工具配置、Prompt 配置、执行日志、调试面板、权限控制、版本发布和灰度能力，因为 AI 平台和普通后台的最大区别，就是执行链路更长、结果更不稳定、问题也更难追。很多时候用户不是问“为什么页面没出来”，而是问“为什么这次 Agent 走了这个工具、为什么这里失败了”，所以前端必须把运行过程做得可理解、可回放。
57. 设计一个 `AI Workflow` 编排平台前端，你会如何拆模块？
答：我会拆成节点库、画布编排区、节点配置面板、运行结果面板、日志追踪区和版本发布区，把编辑态、运行态和治理态分开。这样做的意义是不同阶段关注点不同，编辑态关注如何配置，运行态关注怎么执行和哪里失败，治理态关注怎么发布和回滚，如果把这些内容全塞在一个界面层级里，用户和开发者都会很痛苦。
58. AI 平台里为什么前端也要关心模型状态、工具调用、日志和评测？
答：因为这些概念最终都要落成用户可以操作、理解和排查的产品界面。前端如果只把它们当成“后端内部细节”，最后就只能做出一个表面上的配置页，用户看不到执行过程，也不知道失败在哪。真正可用的 AI 平台，前端必须把模型状态、工具调用、日志追踪和评测结果做成可配置、可观察、可解释的工作台。
59. 流式输出、调试面板、版本发布、权限控制如何在同一个系统里共存？
答：我会按运行态、编辑态、发布态和治理能力分层，让它们共享底层协议、日志和权限体系，但在页面结构和状态流上尽量隔离。这样做的原因是这些能力虽然都属于同一个平台，但变化频率、用户角色和交互重点完全不同，只有边界清楚，系统才能长期演进，而不会因为加一个调试字段就影响发布页面一大片逻辑。
60. 这类系统如何做可观测性设计？
答：我会覆盖节点级耗时、执行日志、错误原因、输入输出快照、版本信息、用户操作链路和灰度信息，让系统问题既能从机器视角看，也能从用户视角回放。所谓可观测性，讲人话就是“出了问题后别靠猜”，而 AI 平台因为中间过程多、黑盒感强，所以比普通后台更需要把每一步做成可追踪的证据链。

## 十一、回答框架与表达

61. 系统设计题中，为什么不能一上来就讲技术栈？
答：因为技术栈只是实现手段，不是问题本身。如果一上来就说 React、微前端、GraphQL 或 WebSocket，面试官通常会觉得你在先报工具箱，而不是先理解业务目标。我会先讲清楚系统服务谁、核心矛盾是什么、边界在哪，再解释为什么当前场景需要这些技术，这样技术选型就有了因果关系，而不是像背模板。
62. 如何在 10 分钟内讲清一个前端系统设计方案？
答：我会固定成“需求边界、模块拆分、数据流、关键难点、性能稳定性、方案取舍”这六段，每段只讲最核心的决策和理由。这样做的好处是能先帮面试官建立全局图，再给对方留出深挖空间。系统设计不是追求把所有细节一口气讲完，而是要在有限时间内让对方相信你知道重点在哪里、知道哪些地方值得深挖。
63. 如果面试官追问“为什么不用别的方案”，你应该如何回答？
答：我会先承认替代方案通常也能做，然后再从业务规模、团队现状、开发效率、可维护性、上线风险和后续演进成本解释为什么当前方案更合适。系统设计里很少有绝对正确答案，更重要的是你是否知道每种方案的收益和代价，所以这类追问其实是在看你的取舍能力，而不是在等唯一标准答案。
64. 如果你没有做过完全同类项目，如何基于已有项目经验做迁移回答？
答：我会先抽象共性问题，比如实时更新、复杂状态、多模块联动、权限治理、版本一致性和可观测性，然后把自己做过的项目经验迁移过来。面试官通常并不要求你做过一模一样的系统，更看重你能不能从已有经验中提炼方法论，所以关键不是硬说“我做过”，而是能证明“虽然业务不同，但我理解这类系统的共性难点”。
65. 如何把系统设计回答和你的主项目经历连接起来？
答：我会在讲方案时自然带出自己真实做过的类似链路、踩过的坑和最后沉淀下来的做法，比如在哪个项目里遇到过版本不一致、竞态污染或流式状态错乱，于是这次设计会提前把校验和监控补上。这样回答的好处是方案不再像纸上谈兵，而是有“我为什么会想到这里，因为我真的踩过这个坑”的可信度。
