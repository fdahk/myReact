# 监控埋点与前端可观测性专题

## 使用方式

这个专题真正要讲清的是，可观测性如何帮助团队发现问题、解释问题、定位问题并推动治理，而不只是“接了某个 SDK”。成熟回答通常会自然覆盖采集维度、字段设计、告警策略、排障链路和业务反哺。

---

## 一、基础认知

1. 什么是前端可观测性？
答：前端可观测性不是简单的错误上报，而是让我们能从线上数据还原用户到底经历了什么、页面到底慢在哪里、故障到底卡在哪一层。它既服务于稳定性，也服务于体验优化和业务决策，是前端从“会开发”走向“会运营线上系统”的关键能力。

2. 前端监控和前端可观测性有什么区别？
答：监控更像“收集指标和告警”，关注有没有出问题；可观测性更强调通过日志、指标、事件和链路把问题解释清楚，关注为什么出问题、用户受影响到什么程度。换句话说，监控是入口，可观测性是闭环。

3. 为什么大厂特别看重前端可观测性？
答：因为页面一旦上线，真实环境远比开发环境复杂，设备、网络、浏览器、地区、缓存和用户行为都会放大问题。没有可观测性，团队只能靠用户投诉和人工复现，排障效率和改进效率都会非常低。

4. 前端可观测性通常包含哪些维度？
答：通常至少包括错误监控、性能监控、白屏监控、接口监控、资源加载监控、用户行为埋点、日志与追踪、告警和排障闭环。对于 AI 产品，还会额外关注首字时间、生成成功率、中断率和工具调用阶段指标。

5. 前端为什么不能只看 PV、UV 这类业务指标？
答：因为 PV、UV 只能说明有没有人来，不能说明页面是否稳定可用。真实项目里，经常会出现流量没掉、但用户其实卡在白屏、接口失败或关键按钮不可点击的情况，所以稳定性和体验指标必须并行建设。

6. 面试里如果被问“你做过监控吗”，怎么答更完整？
答：我会先讲监控维度，再讲采集方案和字段设计，然后补告警策略、看板建设和一次真实排障案例。这样才能体现自己不是只接过 SDK，而是参与过从采集到定位再到治理的全过程。

7. 可观测性的最终目标是什么？
答：我认为最终目标不是“采到很多数据”，而是能更快发现问题、更快定位问题、更有依据地优化体验和业务。没有治理动作的数据只是噪音，真正有价值的是能驱动决策和改进的观测体系。

8. 为什么说前端可观测性是“产品系统能力”而不只是前端技术问题？
答：因为它会影响研发、测试、运维、产品和业务分析的协作方式。指标怎么定义、告警谁接、问题如何分级、回归是否通过，最后都会落到团队协作机制上，而不是单个 SDK 本身。

## 二、错误、性能、白屏与接口监控

9. 前端错误监控通常分哪些类型？
答：通常会分成 JS 运行时错误、资源加载错误、Promise 未捕获异常、框架错误边界错误、跨域脚本错误和业务自定义异常。分类清楚的价值在于后续聚类、告警和排障路径不同，不会把所有异常混成一团。

10. 为什么错误上报不能只上传 `message`？
答：因为同一个错误文案可能来自不同环境和不同代码路径，只看 `message` 几乎无法定位。更有价值的是堆栈、源码映射、浏览器信息、页面路由、用户操作上下文、发布版本和 trace id 这些辅助定位字段。

11. 如何理解前端性能监控？
答：性能监控的重点不是“页面打开用了几秒”这么粗，而是拆成导航、资源、渲染、交互和接口几个阶段去看。只有把性能问题拆到可行动的粒度，团队才知道下一步该优化首包、图片、接口还是主线程阻塞。

12. 前端性能指标通常看哪些？
答：常见会看 `FCP`、`LCP`、`CLS`、`INP`、`TTFB`、首屏时间、资源大小、接口耗时和长任务。关键不在于把指标名背全，而在于能说清这些指标分别反映用户看到内容、交互响应、布局稳定和网络后端延迟哪一类问题。

13. 为什么白屏监控不能只靠用户反馈？
答：因为白屏很多时候发生在首屏最早期，用户甚至还没来得及描述问题就关闭页面了。白屏监控的价值在于自动发现“页面没真正渲染成功”这种致命故障，它往往比普通 JS 报错更值得优先处理。

14. 白屏监控通常怎么做？
答：常见思路是结合关键 DOM 检测、首屏元素采样、骨架屏超时、路由加载状态和异常日志综合判断，而不是只看 `document.body` 有没有内容。成熟方案通常还会区分“加载慢”和“真正白屏”，避免误报太多。

15. 为什么接口监控对前端也很重要？
答：因为用户体感上的很多“页面卡、页面空、按钮没反应”，本质上都是接口耗时或失败造成的。前端如果只能看到页面报错，看不到接口分布和错误分类，很多问题会被误判成纯前端问题。

16. 前端接口监控一般关注什么？
答：我会关注接口成功率、耗时分位数、超时率、取消率、状态码分布、业务错误码分布和关键接口对页面链路的影响范围。对于聚合页面，还会把核心接口按首屏阻塞、交互阻塞和后台异步三类分开看。

17. 资源加载监控为什么不能忽略？
答：因为资源 404、CDN 失效、脚本加载慢、样式缺失都可能直接导致页面不可用或表现异常。很多前端故障不是业务代码写错，而是静态资源链路出问题，所以资源层监控是线上稳定性的基本盘。

18. Source Map 在错误监控里为什么重要？
答：因为线上压缩后的堆栈几乎不可读，没有 Source Map 很难把错误还原到真实源码位置。它不是可观测性的全部，但对提升问题定位效率非常关键。

19. 为什么不能对所有错误一视同仁地告警？
答：因为前端线上噪音很多，比如浏览器插件注入、用户脚本、跨域噪声、第三方环境异常，如果全部拉高优先级，会让真正的生产问题被淹没。成熟做法是先做去噪、聚类和影响面评估，再决定告警策略。

20. 如何定义“关键页面”与“关键接口”？
答：我会从业务损失和用户核心路径出发，比如登录、支付、下单、创建、提交、对话发送这些链路上的页面和接口就是关键对象。可观测性建设不是平均发力，而是优先保护高价值路径。

21. 什么是长任务监控？为什么它和卡顿有关？
答：长任务指主线程长时间被 JS 执行占用，导致浏览器无法及时响应输入和渲染。它能帮助我们识别复杂计算、重渲染、同步解析和第三方脚本阻塞，是发现页面“明明没报错但就是卡”的关键线索。

22. 如何监控 SPA 路由切换性能？
答：单页应用里不能只看首次加载，还要把路由切换视作新的体验节点，监控页面加载数据、异步 chunk、首屏组件 ready 和交互可用时间。否则很多“首页没问题，二级页很慢”的问题会被传统导航指标漏掉。

23. 为什么“接口成功但用户失败”这类情况需要单独监控？
答：因为技术成功不代表业务成功，比如接口返回 `200`，但前端渲染条件没满足、按钮态没恢复、表单校验卡住，用户依然完成不了操作。大厂更看重端到端结果，所以要把技术指标和业务结果指标一起看。

24. AI 产品的性能监控相比普通后台有什么不同？
答：它往往更关注首字时间、流中断率、平均生成时长、取消率和最终完成率，而不是只看接口总耗时。因为在 AI 交互里，用户感知最强的是“多久开始返回”和“中途会不会断”，这和普通 CRUD 页面差别很大。

## 三、用户行为埋点与事件设计

25. 什么是埋点？它和监控是什么关系？
答：埋点更偏业务事件和用户行为采集，比如曝光、点击、提交、转化、停留、路径流失；监控更偏稳定性和性能。两者并不割裂，因为真实排障往往要把技术异常和用户行为放在一起看，才能回答“故障影响了谁、影响了哪一步”。

26. 埋点设计最重要的原则是什么？
答：我认为最重要的是事件语义稳定、字段可解释、命名可治理，不要为了一时分析方便把埋点做成随便加字段的日志垃圾桶。埋点一旦规模变大，治理成本会远高于初次接入成本。

27. 曝光埋点和点击埋点分别解决什么问题？
答：曝光解决“用户有没有看到”，点击解决“用户看到了之后有没有行动”。如果只有点击没有曝光，就无法判断入口是吸引力不足还是根本没被用户看到。

28. 埋点为什么要区分页面级、模块级和动作级？
答：因为分析问题时关注粒度不同，页面级看整体漏斗，模块级看局部入口，动作级看关键操作成败。分层设计后，指标体系更清晰，也更容易做权限控制和采样策略。

29. 埋点字段通常需要哪些上下文？
答：至少要有事件名、时间、页面标识、会话标识、用户标识、实验桶、版本号和关键业务属性；如果要和监控串起来，还应尽量带 trace id 或 request id。字段设计越统一，后续联查越顺畅。

30. 为什么前端埋点要重视版本号和环境信息？
答：因为很多问题和版本发布强相关，如果埋点看不到版本维度，就很难判断转化下降是否由某次发布引起。环境信息也能帮助区分测试、灰度和正式流量，避免分析结果被混淆。

31. 自动埋点和手动埋点怎么选？
答：自动埋点适合低成本覆盖基础点击和曝光，能快速铺量；手动埋点适合关键业务事件和复杂上下文，语义更准确。成熟方案通常是两者结合，而不是押宝其中一个。

32. 埋点设计为什么要避免“一个事件包打天下”？
答：如果所有行为都塞进同一个事件名，再靠字段区分，后期分析会非常痛苦，数据口径也很难统一。事件设计应该贴近业务语义，让分析和治理都能围绕稳定概念展开。

33. 如何处理埋点的重复上报、漏报和时序问题？
答：要从组件生命周期、页面缓存、重试机制和去重策略一起考虑，比如曝光只记一次还是每次进入都记、失败重试是否可能重复、离线补报如何保证顺序。埋点问题本质上也是分布式数据一致性问题，只是规模更偏前端侧。

34. 埋点和隐私合规为什么要一起考虑？
答：因为埋点经常触达用户行为和业务数据，如果字段设计过度，很容易采到不必要的敏感信息。成熟团队会在设计阶段就做字段分级、脱敏和最小化采集，而不是等合规检查时再返工。

## 四、日志、告警与排查链路

35. 前端日志应该记录什么，不该记录什么？
答：应该记录有助于还原问题的上下文，比如页面、版本、接口、错误分类、用户动作和 trace 信息；不该直接记录密码、完整 token、身份证号等敏感数据。前端日志的价值是排障，不是把所有运行时状态都无差别上传。

36. 告警策略为什么不能只看阈值？
答：因为前端数据天然波动大，如果只设一个静态阈值，容易出现误报和漏报。更稳妥的方式是结合错误率变化、影响用户数、关键路径权重和版本变更做分级告警，让告警真正服务于应急响应。

37. 前端告警一般怎么分级？
答：我通常会按致命故障、核心路径异常、局部功能异常和观察级事件分层。比如白屏率突增、登录失败率激增属于高优先级；某个低频组件报错增加则可能先挂观察，不要把团队叫醒处理低影响问题。

38. 什么是前端排查链路？
答：就是从“告警触发”到“确认影响范围”再到“定位版本、页面、用户路径、接口和具体代码”的一整条调查路径。可观测性成熟的团队，应该能把错误、埋点、日志、接口和发布记录串起来，而不是每次靠人肉猜。

39. 如何把前端问题和后端链路打通？
答：常见方式是统一 trace id、request id 或 session id，让前端事件、BFF 日志、后端接口和网关日志可以联查。这样一来，就能回答问题到底卡在浏览器渲染、接口超时、BFF 聚合还是下游服务异常。

40. 如果线上出现“用户说点了没反应”，前端怎么排查？
答：我会先看相关操作埋点有没有触发，再看按钮状态和前端错误、接口调用是否发出、接口是否成功、页面是否被长任务阻塞，最后再结合版本和用户环境定位。这个问题最能体现你是否真正有线上排障经验。

41. 为什么发布记录要纳入可观测体系？
答：因为很多故障都和发布时间点强相关，如果看板和告警里看不到版本信息，就很难第一时间判断是不是新版本引入。把发布事件和指标变化叠加，是缩短定位时间非常有效的手段。

42. 可观测性建设如何形成闭环？
答：我理解的闭环是：先定义指标，再采集数据，再建立告警，再通过排障找到根因，最后推动代码、流程或架构优化，并验证指标是否改善。没有最后一步治理回流，监控体系只会越来越吵，团队却越来越麻木。

## 五、AI 产品指标与专项观测

43. AI 产品为什么要有单独的观测指标？
答：因为它的用户体验和传统页面差别很大，很多关键问题并不会体现在普通接口错误率里。比如模型首字慢、流式中断、工具调用卡死、回答质量波动，这些都需要独立指标去表达。

44. AI 产品前端侧通常关注哪些指标？
答：我会重点看发送成功率、首字时间、首屏内容出现时间、流式完成率、用户取消率、重试率、会话中断率和结果采纳率。这样既能覆盖稳定性，也能覆盖真实交互质量。

45. 为什么 AI 产品要同时看技术指标和业务质量指标？
答：因为接口成功并不代表用户满意，回答跑出来也不代表有价值。比如生成完成率很高，但用户连续追问、频繁重试或快速关闭，这说明产品质量或交互设计可能仍有问题。

46. AI 产品里的工具调用链路前端怎么观测？
答：如果产品里有 Agent 或工具执行，我会把“模型规划、工具调用开始、工具返回、结果整合、最终结束”做成阶段事件，必要时在调试面板里可视化。这样一旦用户说“卡住了”，就能很快知道卡在模型生成还是工具执行。

47. 如果面试官问“监控体系最重要的价值是什么”，怎么答更像 owner？
答：我会说最重要的价值不是证明我们接了多少 SDK，而是让线上问题可发现、可解释、可量化、可回归，并最终反哺产品体验和交付质量。能把这条闭环讲清楚，通常比背一堆监控名词更像真正做过大厂项目的人。
