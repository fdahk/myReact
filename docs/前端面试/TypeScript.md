# TypeScript 训练题纲

## 使用方式

这个模块按约束、推导、抽象和工程收益展开，重点是把 TypeScript 讲成真实工程能力，而不是只列语法点。

---

## 一、基础认知

1. 为什么前端项目要使用 `TypeScript`？
答：我会把它定义成“把很多问题前置到开发期”的工具。对中大型项目来说，真正的价值不是少写几个 bug，而是接口边界、组件契约、重构影响面都更清楚，协作时成本明显下降。
延伸：如果面试官继续追问，我会补一句，`TypeScript` 最值钱的阶段通常不是项目初期，而是需求频繁变动、多人并行开发、老代码持续演进的时候。

2. `TypeScript` 相比 `JavaScript` 的核心收益是什么？
答：核心收益是让代码从“能跑”变成“可约束、可推导、可重构”。它提升的不是语法层体验，而是整个工程里的可读性、可维护性和变更信心。

3. `TypeScript` 是如何在不改变运行时语义的前提下增强开发体验的？
答：因为它主要工作在编译期，做的是静态检查、类型推导和编辑器提示，编译后的运行时代码仍然是 JavaScript。也正因为如此，它擅长约束开发过程，但不会自动替你兜住运行时脏数据。

4. `TypeScript` 的类型检查发生在什么时候？
答：主要发生在开发、CI 和构建阶段，本质上是“发布前发现问题”。浏览器执行时已经没有类型信息了，所以不要把它理解成运行时安全机制。

5. 为什么说 `TypeScript` 不能替代运行时校验？
答：因为类型系统默认相信你拿到的数据是符合声明的，但真实项目里接口返回、用户输入、URL 参数、第三方 SDK 数据都可能不可信。我的做法通常是：内部流程靠 `TypeScript` 保持一致性，边界入口再用校验库或业务校验做运行时兜底。
延伸：像表单提交、后端返回、配置中心下发、富文本解析，这些都属于“编译期无法证明正确”的典型边界。

6. 什么情况下你会认为一个项目“不适合强行全量上 `TypeScript`”？
答：如果项目生命周期很短、历史债务很重、交付窗口很紧，或者团队对类型系统还没有形成共识，强推全量迁移反而可能拖垮产能。我的判断标准不是“要不要上”，而是“当前阶段上到什么程度最划算”。

7. 如果项目里仍然使用 `JavaScript`，你会如何向面试官解释这个决策？
答：结论：如果项目里仍然使用 JavaScript，我会从业务阶段、历史包袱、迁移收益和团队成熟度来解释，而不是简单归因成“没时间”。原理：技术决策本质上是收益和成本的平衡，短生命周期项目、历史债务项目和快速交付阶段，不一定适合一上来做全量迁移。场景：更成熟的做法通常是新增模块优先 TypeScript 化，接口层、公共层和高风险边界先收紧，让迁移收益逐步兑现。

## 二、常用类型系统

8. `interface` 和 `type` 的区别是什么？
答：`interface` 更偏对象契约，适合描述可扩展的结构，而且支持声明合并；`type` 更像类型表达式，联合、交叉、映射、条件类型这些复杂组合通常都离不开它。场景：稳定的对象协议、组件 Props 我更偏向 `interface`，复杂组合和类型运算我更偏向 `type`。边界：两者都能描述对象，但关键是团队要形成一致使用意图，而不是混着写到风格割裂。

9. 什么时候更适合用 `interface`，什么时候更适合用 `type`？
答：我一般把稳定的对象协议、组件 Props、服务层 DTO 优先写成 `interface`，因为语义更像“约定”；遇到联合类型、工具类型组合、别名抽象时更偏向 `type`。团队里更重要的是形成一致约定，而不是两者混用到风格割裂。

10. 什么是联合类型？什么是交叉类型？
答：联合类型表示“这个值可能是多种形态之一”，更适合建模分支；交叉类型表示“同时具备多种能力”，更适合做组合。在项目里，状态流转我常用联合，能力拼接和扩展能力我更常用交叉。

11. 字面量类型有什么价值？
答：它能把值域收得非常精确，比如把 `string` 收窄成 `'loading' | 'success' | 'error'`，这样分支和状态会更可靠。很多高级类型设计的起点，其实都是先把“值的范围”描述清楚。

12. 可选属性和只读属性分别适合什么场景？
答：可选属性适合真实业务里“可能缺失”的字段，比如编辑态和创建态共用模型；只读属性适合那些一旦初始化就不希望被随意改写的数据。这里的关键不是语法，而是把“可变性”和“存在性”设计清楚。

13. 什么是索引签名？
答：索引签名用来描述动态 key 的对象，比如字典、配置表、按 id 聚合的数据结构。它的价值是给“动态结构”一个最基本的边界，但如果 key 集合已知，我会更优先用 `Record` 或字面量联合收窄得更精确。

14. `any`、`unknown`、`never` 的区别是什么？
答：结论：`any` 表示基本放弃类型系统，后续几乎没有约束；`unknown` 表示先接住不确定值，但使用前必须判断；`never` 表示理论上不应该出现的值，常用来做穷尽检查。原理：这三者分别对应类型系统里的失控边界、安全边界和不可能分支。场景：接口边界我更愿意先用 `unknown` 接住，再做收窄；做状态穷尽判断时，`never` 会特别有价值。

15. 为什么 `unknown` 比 `any` 更安全？
答：因为 `unknown` 会强迫你先做收窄，先证明这个值是什么，再去访问属性或调用方法。这种约束在接口层、消息总线、第三方输入等边界场景很重要，可以避免脏数据一路污染业务代码。

16. `void` 和 `never` 的区别是什么？
答：`void` 是函数可以正常结束，只是返回值对调用方没有意义；`never` 是函数根本不会正常返回，比如抛异常或无限循环。项目里 `never` 很常见的一个落点是判别联合的穷尽校验。

## 三、泛型

17. 泛型解决了什么问题？
答：泛型解决的是“逻辑可复用，但类型也不能丢”的问题。没有泛型，很多公共函数、请求封装、表格和表单抽象最后都会退化成 `any`，看似复用，实际丢掉了类型收益。
延伸：可以顺带举例说明，比如请求函数的流程统一，但每个接口的 `data`、`params`、错误体结构并不相同。

18. 泛型函数和普通函数的根本差别是什么？
答：普通函数的参数和返回值类型是预先固定的，泛型函数则把“具体类型”交给调用方决定。它的本质可以理解成“值参数之外，再增加一层类型参数”。

19. 泛型接口、泛型类型别名、泛型类分别怎么理解？
答：三者本质上都是参数化类型，只是承载位置不同。接口常用于约束对象或函数协议，类型别名更灵活，类则更适合和实例状态、方法能力一起演化，但前端业务里泛型类的使用频率通常没前两者高。

20. 什么是泛型约束？
答：泛型约束就是告诉类型参数“你不是完全自由的，至少要满足某个边界”。这样既保留了抽象能力，又能安全地访问某些属性或方法。

21. `extends` 在泛型约束里起什么作用？
答：它不是继承实现，而是表达“必须兼容某种结构”。比如要求泛型参数至少有 `id`、`length` 或某些方法，这样公共逻辑才有安全基础。

22. 多个泛型参数在什么场景下有价值？
答：当输入、输出、配置、错误体不是同一类类型时，单个泛型就不够了。比如请求函数里 `Params`、`Data`、`Error` 分开建模，比全塞一个泛型更能体现边界。

23. 默认泛型参数适合什么场景？
答：适合大多数调用有常见默认值，但又希望少量特殊场景可覆盖时使用。它的工程价值是降低使用成本，而不是炫技。

24. 如何向面试官解释“泛型的本质是把类型参数化”？
答：我通常会说，它像函数接收值参数一样接收类型参数，目的是让同一套逻辑适配多种数据形态，同时保持精确约束。这个表述比单纯背定义更容易体现理解深度。

## 四、类型推导与类型收窄

25. `TypeScript` 的类型推导通常发生在哪些场景？
答：常见于变量初始化、函数返回值、上下文类型、泛型实参推导这些场景。好的 `TypeScript` 代码不是所有地方都手写类型，而是知道哪些地方该显式，哪些地方可以让编译器推出来。

26. 什么是类型收窄？
答：类型收窄就是把一个宽类型，在具体分支里缩小成更精确的类型。它是 `TypeScript` 真正“好用”的关键，因为很多业务代码本来就是根据条件走不同分支。

27. `typeof`、`instanceof`、`in` 在类型收窄中分别怎么用？
答：`typeof` 适合基础类型判断，`instanceof` 适合类实例或原型链相关场景，`in` 很适合判定对象上是否存在某个关键字段。项目里如果能结合业务语义写判断，收窄效果会比无意义断言更稳。

28. 什么是用户自定义类型守卫？
答：就是写一个返回 `x is T` 的函数，把运行时判断结果显式告诉类型系统。它很适合把“边界判断逻辑”沉淀下来，比如接口数据校验、富文本节点识别、不同消息类型分流。

29. 什么是判别联合？
答：判别联合是在联合类型中放一个稳定且可区分的字段，比如 `type` 或 `status`。这样分支一旦判断了这个字段，后续可用属性就能自动跟着收窄，非常适合复杂状态建模。

30. 为什么判别联合非常适合状态机或异步请求状态建模？
答：因为它能把“状态”和“状态下可访问的数据”绑在一起，避免 `loading` 时误读 `data`、`error` 时误取成功字段。相比多个零散布尔值，它更不容易出现非法组合状态。
延伸：例如不要同时维护 `isLoading`、`isError`、`data` 三四个松散字段，而是设计成一组受约束的状态联合。

31. `as const` 有什么作用？
答：它会尽量保留字面量信息，并把对象和数组推成只读结构。项目里经常用它来定义枚举映射、常量配置、路由表、权限 key 列表，避免类型被推宽后失去精度。

32. `satisfies` 操作符解决了什么问题？
答：它解决的是“我既想校验结构，又不想丢掉字面量推导”这个矛盾。相比直接类型注解或强转，它更适合配置对象、映射表、路由元数据这类既要校验又要保留细节的场景。

## 五、工具类型

33. `Partial`、`Required`、`Readonly` 分别有什么作用？
答：它们都是在已有类型上做批量变换，分别表示全量可选、全量必填、全量只读。实际项目里最常见的价值不是“会不会写”，而是知道在哪一层做变换最合理。

34. `Pick` 和 `Omit` 的区别是什么？
答：`Pick` 是主动挑出需要的字段，`Omit` 是从原模型里排除不需要的字段。我的习惯是：字段集较小且明确时用 `Pick`，排除少数字段时用 `Omit`，这样可读性更好。

35. `Exclude` 和 `Extract` 的区别是什么？
答：二者都用于联合类型，一个是排除成员，一个是保留交集成员。它们经常和字面量联合、事件类型、权限类型一起用，本质上是在做类型层面的集合运算。

36. `Record` 适合什么场景？
答：当 key 的集合已知、value 结构相对统一时，`Record` 非常适合，比如状态映射表、图标表、权限描述表。相比随手写对象，它能让 key 是否遗漏、value 结构是否一致都更清楚。

37. `ReturnType`、`Parameters`、`ConstructorParameters` 分别如何使用？
答：它们本质上是在“复用已有定义”，避免你手写重复类型。项目里常见用法是从现有函数推导参数和返回值，保证调用层、封装层和测试 Mock 层保持一致。

38. `NonNullable` 解决了什么问题？
答：它能把 `null | undefined` 从类型里剔除，适合在经过显式判断或初始化之后继续收窄使用。它的意义不是回避空值，而是在合适的时机把边界明确下来。

39. 面试里如果只问你 `Pick` 和 `Exclude`，你会如何顺带体现对工具类型体系的理解？
答：结论：我会直接说，`Pick`、`Exclude` 这类工具类型背后，本质上是映射类型、条件类型和 `infer` 等能力的组合。原理：这说明我理解的不只是 API 名字，而是它们如何基于已有类型生成新类型。场景：一旦面试官继续追问自定义工具类型、类型过滤或类型提取，我就可以自然过渡到它们的底层机制。

## 六、条件类型与 infer

40. 什么是条件类型？
答：条件类型可以理解成类型系统里的分支判断，根据类型关系返回不同结果。它的价值在于把“类型规则”也做成可组合的逻辑。

41. 条件类型为什么常被写成 `T extends U ? X : Y`？
答：因为它的核心就是先判断 `T` 是否满足某个条件，再决定走哪条类型分支。这个模式在工具类型、自定义提取器、类型过滤场景里都非常常见。

42. `infer` 的作用是什么？
答：`infer` 可以在条件类型内部把某一部分类型“提取出来”。你可以把它理解成类型层的解构赋值，用来拿函数返回值、参数、数组元素、Promise 内部值都很方便。

43. 如何用 `infer` 提取函数返回值类型？
答：结论：思路是先把目标约束成函数类型，再在返回值位置写 `infer R`，如果匹配成功就把 `R` 提取出来。原理：`infer` 的作用就是在条件类型匹配成功时，把目标结构里某一段类型单独拿出来。场景：这类写法常用于复用已有函数的返回值定义，避免业务层、封装层和测试层各写一套重复类型。

44. 如何用 `infer` 提取数组元素类型？
答：通常写成 `T extends Array<infer U> ? U : never`，或者 `(infer U)[]` 这种形式。它的核心不是数组本身，而是“在容器结构里拿出内部成员类型”。

45. 分布式条件类型是什么？
答：当裸类型参数是联合类型时，条件类型会对联合中的每个成员分别计算，再把结果合并，这就是分布式条件类型。很多过滤联合成员的工具类型，底层就是依赖这个机制。

46. 为什么联合类型进入条件类型时会发生分发？
答：因为 `TypeScript` 默认把裸类型参数视为可对联合成员逐个判断，这样表达能力更强。反过来说，如果你不想分发，就要把类型包起来，比如放进元组。

47. 条件类型和工具类型之间有什么关系？
答：很多内置工具类型本质上都是条件类型、映射类型、索引访问和 `infer` 的组合。理解这一层之后，遇到自定义工具类型、类型过滤或类型提取这类深挖问题也会更从容。

## 七、函数与类

48. 如何给函数声明参数和返回值类型？
答：我的原则是边界函数、公共函数、业务关键函数尽量把参数和返回值写清楚，局部实现细节可以适当依赖推导。这样既能保证可读性，又不会让类型标注变成噪音。

49. 函数重载适合什么场景？
答：当同一个函数因为不同入参形态，真的会产生不同返回值语义时，函数重载才有意义。否则很多场景用联合类型加分支判断会更简单，也更容易维护。

50. 可选参数和默认参数在类型层面有什么差异？
答：二者都允许调用方不传，但默认参数在函数内部通常已经是确定值，而可选参数在函数内部仍然可能是 `undefined`。这会直接影响你是否还需要空值判断。

51. `this` 参数在 `TypeScript` 里怎么声明？
答：可以在参数列表最前面声明一个 `this`，它只参与类型检查，不算真实参数。这个能力在老代码、工具函数、需要显式约束调用上下文的场景里比较有用。

52. 类中的 `public`、`private`、`protected` 有什么区别？
答：`public` 外部可访问，`private` 仅类内部可访问，`protected` 则允许子类使用。边界是访问修饰符更多是封装和约束手段，不是运行时安全机制。

53. 抽象类和接口的区别是什么？
答：接口只描述能力契约，不包含实现；抽象类除了定义规范，还能沉淀默认实现和共享逻辑。前端里接口更常见，抽象类通常只在确实存在“共性实现”时才值得用。

54. `implements` 和 `extends` 的区别是什么？
答：`extends` 更强调继承已有能力，`implements` 更强调承诺满足某个接口规范。前者主要解决能力复用，后者主要解决契约约束，两者关注点并不一样。

55. 为什么很多前端项目很少大量依赖类来组织业务逻辑？
答：因为现代前端更偏组合式、函数式和声明式，很多状态复用、逻辑拆分用函数和 Hook 会更自然。类并不是不能用，而是它未必是最贴合 React 生态和前端协作方式的抽象。

## 八、React 与工程实践

56. 在 `React` 项目里，组件 Props 和 State 类型通常怎么设计？
答：Props 我会尽量贴近业务语义，明确哪些是输入、哪些是回调、哪些是可选扩展；State 只保留最小必要状态，能从 Props 或其他状态推导出来的就不重复存。这样类型会更稳定，组件也更不容易失控。
延伸：复杂组件里我会优先用判别联合描述状态，而不是堆很多布尔值。

57. 自定义 Hook 的返回值如何设计得更清晰？
答：如果返回项比较少且顺序天然稳定，可以用元组；只要成员一多或语义稍复杂，我更倾向返回对象。核心标准是调用方能不能一眼看懂，不要让别人靠位置去猜。

58. 接口返回值类型为什么不能直接随便写成 `any`？
答：因为 `any` 会把最关键的边界直接打穿，错误会从请求层一路传到组件层，最后只能在线上暴露。真实项目里，哪怕暂时拿不到完整类型，我也会先用 `unknown` 或最小可用结构，而不是直接放弃约束。

59. 前端请求层为什么适合配合泛型封装？
答：请求流程本身往往是统一的，比如鉴权、错误处理、响应解包，但不同接口的 `params` 和 `data` 又不一样，所以泛型正好能把“共性流程”和“差异类型”分开。这样既能复用逻辑，又不会丢失调用精度。

60. 如何为复杂表单、表格、配置面板设计可维护的类型？
答：我一般会先抽稳定领域模型，再区分展示态、编辑态、提交态、接口态，不让一个类型承担所有职责。复杂页面真正难的不是字段多，而是同一份数据在不同阶段的语义不同。

61. 类型定义应该更靠近页面、组件还是服务层？
答：我的原则是“先局部、再上提”。只在单个页面里使用的类型就近放置，真正跨模块复用的再沉淀到公共层，这样能避免一开始就把类型中心化成难维护的大仓库。

62. 什么是“类型驱动开发”？它在前端里能做到什么程度？
答：我的理解是先把核心数据结构、状态边界和函数契约定义清楚，再据此反推实现，这样很多设计问题会更早暴露。它能显著改善边界质量，但不等于业务逻辑自动正确，运行时规则和产品判断仍然要靠实现兜住。

## 九、装饰器与元数据

63. 你了解装饰器吗？装饰器解决了什么问题？
答：了解，装饰器主要适合处理注册、注入、元数据标注、方法拦截这类横切逻辑，让声明更集中。原理：它是在不改业务主体代码的前提下，把一部分增强逻辑挂到类、方法或属性上。场景：NestJS 这类依赖元数据和约定式开发的框架用得很多。边界：它解决的是组织方式问题，不是所有前端项目都需要它。

64. 类装饰器、方法装饰器、属性装饰器通常分别做什么？
答：类装饰器更适合做注册和增强，方法装饰器更适合做日志、权限、缓存、埋点这类拦截，属性装饰器常用于标记元数据。真正落地时，要看团队能不能接受额外抽象成本。

65. 装饰器为什么在后端框架中更常见？
答：因为后端框架天然有控制器、路由、依赖注入、验证器这些“声明式元数据”需求，装饰器和这类结构非常契合。相比之下，前端很多问题用组合函数就已经足够清晰。

66. 为什么 `Express` 很少像 `NestJS` 一样大量使用装饰器？
答：`Express` 的哲学更偏轻量和自由，核心不是强约束的声明式框架；而 `NestJS` 从一开始就是面向模块化、DI、约定式开发设计的。这里的差异本质是框架定位，而不是装饰器本身好坏。

67. 装饰器适合什么场景，不适合什么场景？
答：适合规则稳定、横切逻辑多、元数据收益明显的系统；不适合业务变化快、团队不熟悉、调试链路要求很直接的场景。我的判断标准是收益能否覆盖理解和维护成本。

68. 在前端项目里，为什么通常不会把装饰器作为核心设计手段？
答：因为前端更强调组件组合、状态管理、显式数据流，过多装饰器容易把逻辑藏起来，增加调试心智负担。尤其在 React 项目里，函数组合往往更贴合生态。

## 十、项目选择与边界

69. 如果一个项目用了 `JavaScript` 而不是 `TypeScript`，你如何解释业务和工程上的原因？
答：我会从项目阶段、团队结构、历史代码体量和交付节奏去解释这是现实取舍，而不是技术落后。更完整的说法是：虽然主项目可能还是 `JavaScript`，但新增模块、公共库和接口层已经在逐步类型化，这样既控制迁移风险，也能尽早拿到收益。

70. 如果项目后期要从 `JavaScript` 迁移到 `TypeScript`，你会怎么分阶段推进？
答：我一般会先从新功能和公共层开始，优先拿下接口层、工具层、组件库这类复用面广的模块，再逐步向核心业务渗透。迁移时不要追求一次性纯度，而要先把高风险边界补齐，让收益尽早出现。
延伸：落地上通常会配合 `allowJs`、边迁边收紧 `tsconfig`，以及在 CI 里逐步提升检查强度。

71. 类型系统很复杂时，如何避免“类型体操”反而拖慢开发？
答：结论：我处理类型复杂度的原则是优先可读性和收益比。原理：复杂类型只有在确实提升边界安全、复用效率或重构体验时才值得保留。场景：如果某段类型体操只是看起来炫，但团队没人敢改，我会退回更直接的写法。边界：TypeScript 的目标是降低风险，不是把维护成本从运行时转移到类型层。

72. 什么情况下你会优先保证业务交付，而不是继续追求更完美的类型？
答：当需求窗口很紧、风险点已经通过关键边界兜住、继续细化类型的边际收益不高时，我会先交付，再补强。工程上追求的是“足够正确且可持续”，不是每个阶段都要达到最完美状态。

73. 如何向面试官证明你对 `TypeScript` 的理解不是停留在语法层，而是工程层？
答：结论：想证明自己对 TypeScript 的理解不只停留在语法层，重点不是堆 `infer` 和工具类型名词，而是说明你如何用类型约束接口、管理状态、支撑重构、降低协作风险。原理：类型系统真正的价值发生在工程边界上，而不是发生在写出多复杂的类型体操上。场景：如果能结合项目讲清哪些 bug 被类型提前暴露、哪些抽象因为类型设计更稳，再补一个请求层泛型封装或复杂表单状态建模案例，就很像工程层理解。
