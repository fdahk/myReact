# 浏览器训练题纲

## 使用方式

这个文档按浏览器高频链路整理，答案统一尽量覆盖结论、原理、优化方向和边界，方便直接展开。

---

## 一、浏览器基础认知

1. 浏览器的主要进程和线程有哪些？
答：结论：浏览器可以从进程和线程两层来理解。进程层面重点是浏览器主进程、渲染进程、GPU 进程、网络进程；线程层面重点是 GUI 渲染线程、JS 引擎线程、事件触发线程、定时器线程。原理：浏览器用多进程做隔离和安全，用多线程承载渲染、脚本执行和事件调度这些不同职责。场景：按这条结构展开，后面无论继续追问渲染、事件循环还是网络链路，都能自然接上。
2. 为什么现代浏览器要采用多进程架构？
答：结论：核心是隔离和稳定，一个页面崩了别把整个浏览器带死，同时安全性和性能调度也更好做。原理/场景：多进程的核心收益是隔离、稳定性和安全性，代价是进程间通信和资源占用更复杂。
3. 渲染进程主要负责什么？
答：结论：主要负责页面解析、布局、绘制、脚本执行这些，也就是把资源真正变成可交互页面。原理/场景：它本质上负责把资源解析成页面，并承载脚本执行、布局、绘制等前端最关心的工作。
4. GUI 渲染线程和 JavaScript 引擎线程为什么互斥？
答：结论：因为 JS 可能会改 DOM，如果渲染和脚本同时改界面，状态会乱，所以浏览器干脆让它们互斥。原理/场景：如果两者同时改界面状态，会带来一致性问题，所以浏览器通常让它们互斥调度。
5. 浏览器如何实现标签页隔离？
答：结论：一般是一个或一组标签页对应独立渲染进程，这样页面之间崩溃、权限和数据都更容易隔离。原理/场景：现代浏览器往往是“站点隔离 + 进程复用策略”组合，不一定永远一标签页一个进程。
6. 浏览器为什么既是运行环境，又是渲染平台？
答：结论：因为浏览器既提供 JS、定时器、网络这些运行时能力，又负责把 HTML/CSS 真正渲染成页面。原理：它既是 JavaScript 的宿主环境，也是前端页面的渲染平台。场景：这也是为什么前端问题常常同时牵涉脚本执行、网络链路和渲染结果。边界：Node.js 也能跑 JavaScript，但它没有浏览器这套渲染职责。
7. 你如何解释浏览器和 Node.js 在运行 JavaScript 上的差别？
答：结论：语言本身一样，但宿主环境不同。浏览器偏 DOM 和渲染，Node.js 偏文件系统、服务端和进程能力。原理/场景：语言相同，但宿主能力不同，所以浏览器更偏渲染和安全隔离，Node 更偏服务端和系统能力。
8. 前端开发为什么必须了解浏览器底层运行机制？
答：结论：因为性能、白屏、卡顿、内存泄漏、跨域这些问题，本质上都和浏览器的运行机制直接相关。原理：浏览器决定了请求怎么发、资源怎么解析、JS 怎么调度、页面怎么渲染。场景：理解浏览器底层后，排查问题会更像顺着链路拆，而不是只盯业务代码。边界：前端不需要变成浏览器内核工程师，但核心链路必须讲清。

## 二、从输入 URL 到页面展示

9. 从输入 URL 到页面最终显示，中间完整经历了什么？
答：结论：我会按 URL 解析、DNS、TCP/TLS、发请求、服务端返回、浏览器解析 HTML、构建 DOM/CSSOM、布局绘制这条链路顺着讲。原理：这道题本质上是在考你能不能把网络、解析、渲染串成一条完整路径，而不是把知识点拆散背。场景：真实排查白屏、首屏慢、接口慢时，也基本都是按这条链路一段段定位。延伸：这题最容易拉开差距的点，就是你能不能把每一段耗时和优化手段一起讲出来。
10. DNS 查询大致做了什么？
答：结论：DNS 查询本质上就是把域名翻译成 IP。原理：它通常会先查浏览器缓存、系统缓存、本地 hosts，没命中再走递归解析，最后从权威服务器拿到结果。场景：这一步看起来简单，但如果解析慢，首屏和接口首包都会被整体往后拖。
11. TCP 连接建立过程对前端有什么影响？
答：结论：TCP 建连会直接影响首包前的等待时间，连接建得慢，页面和接口的起步就慢。原理：在真正发 HTTP 请求前，客户端和服务端得先把连接状态建立好。场景：弱网、跨地域访问或频繁新建连接时，这段耗时会明显放大首屏等待。边界：HTTP/2、连接复用和 CDN 都是在尽量减少这部分开销。
12. HTTPS 在这个过程中比 HTTP 多了哪些步骤？
答：结论：多了 TLS 握手、证书校验、密钥协商，所以安全更高，但建连成本也更大。原理/场景：HTTPS 的核心增量是 TLS 握手、证书校验和密钥协商，安全更强但建连成本也更高。
13. 浏览器收到 HTML 后会如何处理？
答：结论：先边下载边解析 HTML，构建 DOM，同时遇到外部资源再继续拉取和处理。原理/场景：浏览器通常是边下边解析 HTML，而不是等整个文档下载完才开始渲染。
14. 什么是预解析？
答：结论：可以理解成浏览器在主解析外，会提前扫描文档里的资源引用，尽早去下载脚本、样式、图片。原理/场景：预解析或预加载扫描器的价值是尽早发现资源引用，减少主解析阻塞后的等待时间。
15. 资源加载和页面渲染是如何并行推进的？
答：结论：HTML 解析、资源请求、样式计算、后续渲染并不是完全串行的，而是浏览器会尽量并行调度。原理/场景：真实链路不是绝对串行，而是浏览器在保证正确性的前提下尽量并行下载和调度。
16. 如果页面白屏很久，你会从这个链路的哪些环节排查？
答：结论：我会先看 DNS、建连、接口、静态资源、首屏 JS、CSS 阻塞、渲染异常这几段，按 waterfall 一段段排。原理/场景：排查白屏时最好按 waterfall 一段段拆：DNS、建连、接口、静态资源、执行、渲染，避免只盯前端代码。

## 三、渲染流程

17. 浏览器的渲染流程是什么？
答：结论：通常就是解析 HTML 生成 DOM，解析 CSS 生成 CSSOM，然后合成 Render Tree，接着 Layout、Paint、Composite。原理/场景：高频标准表达是 DOM、CSSOM、Render Tree、Layout、Paint、Composite 这一整条渲染流水线。
延伸：渲染流程高频关键词：DOM、CSSOM、Render Tree、Layout、Paint、Composite。
18. 什么是 DOM 树和 CSSOM 树？
答：结论：DOM 树描述页面结构，CSSOM 树描述样式规则，浏览器后续渲染要把这两棵树结合起来。原理：只有结构和样式都准备好，浏览器才能生成真正参与渲染的 Render Tree。场景：这也是为什么 CSS 会影响首屏渲染，而不仅仅是改个样子。边界：`display: none` 这类节点会在后续渲染树阶段被过滤掉，不会真正上屏。
19. Render Tree 是如何生成的？
答：结论：浏览器会把可见的 DOM 节点和对应样式合并，生成真正参与渲染的树。原理/场景：Render Tree 只会保留真正参与渲染的节点，所以像 `display: none` 的节点不会进入这棵树。
20. Layout 和 Paint 分别在做什么？
答：结论：Layout 是算位置和尺寸，Paint 是把颜色、边框、文字这些内容画出来。原理：Layout 解决几何信息，Paint 解决视觉内容，而且 Layout 往往比单纯 Paint 更重，因为它可能连带影响后续绘制和合成。场景：性能排查里，先分清是布局成本高还是绘制成本高，优化方向才会对。
21. Composite 阶段的作用是什么？
答：结论：就是把不同图层最终合成到屏幕上，这一步对动画流畅度很关键。原理/场景：Composite 更偏图层合成和最终上屏，这也是为什么某些动画能避开重排和重绘。
22. 重排和重绘的区别是什么？
答：结论：重排会重新计算布局，重绘只是外观变了不改几何信息，所以重排一般更贵。原理/场景：重排通常比重绘更贵，因为它会重新计算布局，并可能连带影响后续绘制和合成。
延伸：重排通常比重绘更贵，是性能题常见切入点。
23. 哪些操作会触发重排？
答：结论：改宽高、位置、字体大小，增删节点，或者读一些布局信息像 `offsetHeight`，都可能触发重排。原理/场景：除了改样式，频繁读取 `offsetWidth`、`getBoundingClientRect()` 这类布局信息也可能触发强制同步布局。
24. 哪些操作只会触发重绘？
答：结论：像改颜色、背景色、阴影这类不影响几何信息的样式，通常只会触发重绘。原理：浏览器不用重新计算布局，只需要重新把视觉结果画出来。场景：hover 变色、主题切换、局部高亮很多时候都属于这类更新。边界：只重绘不代表没有成本，如果面积大或频率高，照样会影响流畅度。
25. 为什么 `transform` 和 `opacity` 更适合做动画？
答：结论：因为它们很多时候只走合成阶段，不用频繁重排重绘，性能更稳。原理/场景：`transform` 和 `opacity` 常能直接走合成层，所以更适合高频动画和交互反馈。
延伸：动画优化高频答案通常会落到 `transform` 和 `opacity`。
26. 如何减少重排和重绘？
答：结论：核心就是少频繁改布局、批量改 DOM、避免读写交替、动画尽量用 `transform` 和 `opacity`。原理/场景：核心思路是减少布局变更、避免读写交替、批量更新 DOM，并优先使用合成层动画。

## 四、页面性能与优化

27. 什么是关键渲染路径？
答：结论：就是从拿到 HTML 到首屏可见这段最关键的渲染链路，谁堵在这条路上，首屏就慢。原理/场景：关键渲染路径说白了就是“首屏链路上哪些资源和步骤会卡住可见内容出现”。
28. 首屏性能受哪些因素影响？
答：结论：首屏性能会同时受到建连、HTML 返回、CSS 阻塞、首屏 JS、图片体积、接口时延和渲染复杂度影响。原理：首屏是整条链路叠加出来的结果，不是单个指标决定的。场景：有的页面慢在网络，有的慢在包体和执行，有的慢在布局和接口串行。边界：优化首屏不能只盯一段，要先拆清主瓶颈在哪。
29. 什么是阻塞渲染资源？
答：结论：就是不等它处理完，页面没法正常继续渲染的资源，最典型就是 CSS 和部分脚本。原理/场景：最典型的阻塞资源是 CSS 和部分脚本，所以首屏优化往往先盯这两类。
30. 为什么 CSS 会阻塞渲染？
答：结论：因为浏览器要先拿到样式才能正确算布局，不然会先画错再重来。原理/场景：浏览器要先拿到样式再构建正确渲染结果，否则容易先画错再回流重算。
31. 为什么普通 `script` 会阻塞解析？
答：结论：因为脚本可能改 DOM，浏览器为了保证结果正确，通常会暂停 HTML 解析先执行它。原理/场景：普通脚本会阻塞解析，是因为它可能立即读写 DOM，浏览器必须保证执行顺序和页面结果正确。
32. `defer` 和 `async` 的区别是什么？
答：结论：`async` 下载完就执行，顺序不保证；`defer` 会延迟到 HTML 解析完成后按顺序执行。原理/场景：`async` 更强调尽快执行且不保证顺序，`defer` 更适合依赖 DOM 结构且要保持顺序的脚本。
延伸：`defer` 和 `async` 经常和脚本阻塞解析一起追问。
33. 图片优化有哪些常见手段？
答：结论：压缩、裁剪、懒加载、响应式图片、合理格式选择，比如 WebP 或 AVIF，这些都很常见。原理/场景：实际优化里通常是格式选择、尺寸裁剪、懒加载、响应式图片和缓存策略一起做。
34. 长列表为什么会卡顿？常见解决思路是什么？
答：结论：因为一次性渲染的节点太多，布局和绘制压力大。常见方案就是虚拟列表和分片渲染。原理/场景：长列表卡顿的根因是节点过多导致布局、绘制和事件处理成本同时上升，虚拟列表是高频解法。
35. 懒加载和预加载分别适合什么场景？
答：结论：懒加载适合非首屏资源，预加载适合马上要用但又想提前准备的关键资源。原理/场景：懒加载是延后不急资源，预加载是提前准备马上要用的关键资源，二者目标不同。
36. 资源合并和资源拆分分别适合什么场景？
答：结论：早期请求多时会偏向合并，现在 HTTP/2 下更常按路由和功能拆分，兼顾缓存和按需加载。原理/场景：HTTP/2 之后不再一味追求合并，很多时候更看重按路由拆分和长期缓存命中。
37. 你会如何解释首屏时间、LCP、FCP、CLS、TTI？
答：结论：我会说首屏时间是业务口径，FCP 看首次内容出现，LCP 看最大内容出现，CLS 看布局抖动，TTI 看可交互时间。原理：业务指标更贴近具体产品目标，Web Vitals 则是更通用的用户体验指标，两者不能混成一个概念。场景：真实优化时，通常既要看业务首屏时间，也要看 LCP、CLS 这类通用体验指标。
38. Lighthouse 的核心指标分别意味着什么？
答：结论：Lighthouse 的核心价值就是从加载速度、稳定性、交互响应和最佳实践几个维度帮你量化页面体验。原理：它把一部分常见体验问题标准化成可比较的评分和诊断项。场景：做首屏优化、回归验证、上线前体检时很有参考价值。边界：Lighthouse 是实验室指标，不等于全部真实用户体验，最终还要结合线上监控。

## 五、缓存机制

39. 浏览器缓存有哪些层级？
答：结论：我一般会答内存缓存、磁盘缓存，再往上还有 Service Worker、HTTP 缓存这些层。原理/场景：浏览器缓存不只是一层，更完整的划分是内存缓存、磁盘缓存、HTTP 缓存和 Service Worker。
40. 强缓存和协商缓存的区别是什么？
答：结论：强缓存命中了连请求都不用真发，协商缓存还是会带条件去问服务端能不能继续用旧资源。原理/场景：强缓存解决“能不能直接用”，协商缓存解决“资源可能变了要不要再确认”。
延伸：强缓存和协商缓存几乎一定会和响应头一起考。
41. `Expires` 和 `Cache-Control` 的区别是什么？
答：结论：`Expires` 用的是绝对时间，容易受客户端时间影响；`Cache-Control` 用的是相对时间，表达能力更强，优先级也更高。原理：前者更像早期方案，后者可以进一步表达 `max-age`、`no-cache`、`no-store`、`public`、`private` 等缓存策略。场景：现在工程上一般都以 `Cache-Control` 为主，`Expires` 更多是兼容性补充。边界：如果用户本地时间不准，`Expires` 的判断就可能失真。
42. `Last-Modified` 和 `ETag` 的区别是什么？
答：结论：`Last-Modified` 基于资源最后修改时间判断，`ETag` 基于资源标识判断，后者通常更精确。原理：时间戳方案实现简单、开销低，但精度有限；`ETag` 可以识别更细粒度的变化，不过生成和比较成本通常更高。场景：静态资源或变化不频繁的内容，用时间戳可能已经够用；对精度要求更高时会更偏向 `ETag`。
43. 什么资源适合强缓存？什么资源适合协商缓存？
答：结论：带 hash 的静态资源适合强缓存，HTML 和可能变化但又想复用的接口更适合协商缓存。原理/场景：带 hash 的静态资源适合强缓存，HTML 和变化频率高的资源更适合协商缓存。
44. 前端静态资源为什么常配合哈希命名？
答：结论：因为文件内容一变，哈希名也会跟着变，所以可以放心把缓存时间拉长，而不用担心用户继续命中旧资源。原理：缓存命中本质上依赖 URL，文件名带 hash 后，内容变更就会自然形成新的 URL。场景：这也是前端静态资源常见的“强缓存 + 文件 hash”组合策略。边界：入口 HTML 一般不适合走同样的长强缓存，因为它要负责引用最新资源地址。
45. 浏览器缓存命中失败的常见原因有哪些？
答：结论：缓存命中失败常见原因有响应头没配对、URL 变了、资源被禁缓存、强刷以及代理层改头。原理：浏览器命中缓存要同时满足 URL、缓存头和当前使用条件。场景：发版后不生效、CDN 命中异常、入口 HTML 老是拿旧版本，常常都和这里有关。边界：排查缓存不要只看浏览器，还要同时看 CDN 和网关层。
46. 用户说“我明明发版了但页面没更新”，你怎么排查？
答：结论：我会先看 HTML 有没有更新、静态资源文件名有没有变、缓存头对不对、CDN 有没有刷新到位。原理/场景：这个问题常见根因是 HTML 没更新、静态资源没变更文件名、CDN 没刷新或缓存头配置不合理。

## 六、存储机制

47. `Cookie`、`localStorage`、`sessionStorage` 的区别是什么？
答：结论：`Cookie` 会跟请求走，容量小；`localStorage` 长久存本地；`sessionStorage` 只在当前会话里有效。原理/场景：`Cookie` 会随请求发送，`localStorage`/`sessionStorage` 只在前端可读写，安全边界和使用成本完全不同。
延伸：存储题常会连着问安全问题，尤其是 token 放哪里。
48. `Cookie` 为什么不适合存大量数据？
答：结论：因为 Cookie 容量本来就小，而且会随着同域请求一起发送，数据一多既浪费带宽，也会扩大泄露面。原理：Cookie 的设计目标是维持会话状态，不是本地大容量存储，所以它天然不适合承载大对象或高频变化数据。场景：像登录态标识、会话信息适合放 Cookie，但草稿、大列表缓存、复杂配置就不合适。
49. `localStorage` 的优点和限制是什么？
答：结论：`localStorage` 的优点是简单直接、持久化方便，浏览器关闭后数据通常还在；限制是它是同步 API、容量有限，而且一旦发生 XSS，里面的数据很容易被读走。原理：它更适合轻量级、低频读写、本地可持久化的数据，不适合高频更新或高敏感信息。场景：主题设置、轻量缓存、非敏感配置可以考虑放这里；认证凭证、隐私数据和大数据量缓存就要更谨慎。
50. IndexedDB 适合什么场景？
答：结论：适合离线数据、本地大数据量缓存、复杂结构存储，像文档、草稿、离线列表这种。原理/场景：IndexedDB 更适合大数据量、复杂结构和离线场景，不要拿它和 `localStorage` 做同一量级比较。
51. 前端本地存储有哪些安全风险？
答：结论：前端本地存储最大的安全风险就是一旦发生 XSS，里面的数据基本都可能被读走。原理：`localStorage`、`sessionStorage` 对页面脚本是可读的，恶意脚本一旦执行就能直接访问。场景：token、用户信息、业务配置如果存得过多，泄露面会被放大。边界：本地存储不是不能用，而是不能把高敏感凭证随意暴露给 JS 可读环境。
52. token 为什么不建议随便放在 `localStorage`？
答：结论：因为它对 JS 完全可读，XSS 一旦发生，token 很容易被偷。原理/场景：把 token 放在 `localStorage` 最大风险是 XSS 一旦发生，凭证很容易被直接读取。
53. 浏览器多标签页间如何同步数据？
答：结论：常见做法是监听 `storage` 事件，或者用 `BroadcastChannel`。原理/场景：多标签同步除了 `storage`，新一点的方案还有 `BroadcastChannel`，语义更直接。
54. 如果你要设计前端缓存层，会如何选择不同存储方式？
答：结论：我会按数据敏感度、大小、生命周期来选，敏感认证尽量别放可读存储，大数据用 IndexedDB，轻量配置再考虑 localStorage。原理/场景：选择存储方式时我会同时看数据敏感度、生命周期、容量需求和是否需要跨标签同步。

## 七、事件机制

55. 什么是事件捕获、目标阶段、冒泡阶段？
答：结论：事件从外往内到目标节点的过程是捕获，落到目标节点本身是目标阶段，再从里往外返回就是冒泡。原理：浏览器用这套传播模型让一个事件能被多个层级节点感知和处理。场景：事件委托、全局监听、局部阻止冒泡都建立在这套机制上。边界：并不是所有事件都会冒泡，比如 `mouseenter`、`mouseleave` 就是例外。
56. 事件委托的原理是什么？
答：结论：本质是利用事件冒泡，把子元素的事件交给父元素统一处理。原理/场景：事件委托本质上是利用冒泡把多个子节点监听合并到父节点，常见于列表和动态节点场景。
57. 为什么事件委托适合大量子元素场景？
答：结论：事件委托适合大量子元素场景，因为不用给每个节点都单独绑监听器，内存和维护成本都更低。原理：父节点统一接管子节点事件，本质上是在复用冒泡机制。场景：列表、表格、动态插入节点非常适合委托。边界：如果事件本身不冒泡，或者需要非常细粒度的独立监听，委托就没那么合适。
58. `event.target` 和 `event.currentTarget` 的区别是什么？
答：结论：`target` 是真正触发事件的元素，`currentTarget` 是当前正在处理事件的那个监听节点。原理/场景：`target` 表示真正触发事件的源头，`currentTarget` 表示当前正在执行监听器的节点。
59. 什么情况下你会阻止冒泡？什么情况下不该阻止？
答：结论：只有在明确不希望上层逻辑被触发时我才会阻止冒泡，默认不会乱拦，避免把组件协作链路打乱。原理：事件传播本身是浏览器默认协作机制，乱拦会让父层逻辑失效。场景：弹窗内部点击不想触发遮罩关闭、菜单点击不想触发外层收起时，才值得阻止。边界：阻止冒泡要带着明确意图，不要把它当成万能止血手段。
60. 被动事件监听器有什么作用？
答：结论：它告诉浏览器这个监听器不会调用 `preventDefault`，这样滚动这类场景可以更顺畅。原理/场景：被动监听常用于滚动和触摸场景，告诉浏览器无需等待 `preventDefault`，从而提升流畅度。
61. 为什么滚动监听容易造成性能问题？
答：结论：因为触发频率很高，如果里面还读布局、改样式或做重计算，就很容易卡。原理/场景：滚动监听频率高，如果里面又读布局、改样式或做重计算，很容易形成掉帧和长任务。
62. 前端框架中的合成事件和原生事件有什么差异？
答：结论：合成事件通常是框架做了一层封装，方便兼容和统一管理，但底层还是绕不开原生事件机制。原理/场景：框架合成事件主要解决兼容、统一管理和批量处理，但底层仍然建立在原生事件机制之上。

## 八、浏览器安全

63. 什么是同源策略？
答：结论：就是协议、域名、端口三者都相同才算同源，不同源的很多资源访问会被浏览器限制。原理/场景：同源策略是浏览器安全模型的核心之一，很多跨域、安全和认证问题都要从这里展开。
延伸：跨域、安全、认证很多题都要先从同源策略展开。
64. 跨域为什么会发生？
答：结论：因为协议、域名、端口只要有一个不同，浏览器就会把它判定成不同源，而现代前后端分离、主站与子域、CDN 分发这些场景都很容易出现这种差异。原理：跨域问题本质来自浏览器的同源策略，是浏览器为了隔离站点数据和脚本权限做的安全边界。场景：开发环境里最常见的是前端本地服务请求测试接口，线上则常见于主站、子域和静态资源域名拆分。
65. 常见跨域方案有哪些？各自适合什么场景？
答：结论：最常见是 CORS，开发环境常用代理，老场景才会提 JSONP，核心还是看服务端能不能配合。原理/场景：真实项目里最常见还是 CORS 和开发代理，JSONP 更多是历史知识点。
66. 什么是 CORS？
答：结论：CORS 就是浏览器基于响应头判断这个跨域请求能不能放行给前端脚本读取的一套机制。原理：请求本身常常能发出去，但浏览器会根据服务端返回的 `Access-Control-*` 响应头决定是否放行。场景：前后端分离、主站和子域协作、CDN 资源访问都很常见。边界：CORS 约束的是浏览器端脚本读取，不是服务器之间的访问控制。
67. 预检请求是什么？为什么会出现？
答：结论：非简单请求前，浏览器会先发个 `OPTIONS` 去问服务端允不允许，这个就是预检。原理/场景：预检请求是浏览器在真正发非简单请求前，先帮你问服务端“这类跨域请求能不能放行”。
68. 什么是 JSONP？为什么现在用得少？
答：结论：JSONP 本质上是利用 `script` 标签不受同源限制来跨域拿数据，但它只能发 GET，请求和错误处理能力都很弱，所以现在基本只剩历史知识点价值。原理：它不是在真正放开浏览器的跨域读取限制，而是借助脚本加载机制，让服务端返回一段函数调用代码。场景：早期服务端不能配 CORS 时会用它兜底，但现代项目里更常见的还是 CORS 或反向代理。边界：因为它执行的是脚本，不只是数据，所以安全和可维护性都明显更差。
69. 什么是 XSS？常见防护方式有哪些？
答：结论：就是把恶意脚本注进页面执行。防护上要做输出转义、内容清洗、CSP，别乱拼 `innerHTML`。原理/场景：XSS 防护重点是输入输出都做约束，尤其避免把不可信内容直接拼进 HTML 执行环境。
延伸：XSS 题常和 CSP、转义、`innerHTML` 风险一起追问。
70. 什么是 CSRF？常见防护方式有哪些？
答：结论：就是利用用户已登录状态诱导浏览器发起恶意请求。常见防护是 CSRF Token、`SameSite` Cookie、来源校验。原理/场景：CSRF 的本质是浏览器自动携带登录态，所以防护手段通常围绕“让攻击者无法伪造有效请求”。
71. 什么是点击劫持？
答：结论：点击劫持就是把页面藏在透明 iframe 里，诱导用户点到不该点的按钮或操作。原理：攻击者利用用户看得见的界面覆盖真实页面，让点击目标发生偏移。场景：支付确认、授权按钮、敏感设置页都可能成为目标。边界：常见防护是 `X-Frame-Options` 或 CSP 的 `frame-ancestors`。
72. 为什么前端也必须理解安全问题？
答：结论：前端必须理解安全问题，因为很多攻击入口就在浏览器端，至少要知道风险从哪里进、自己该怎么挡第一层。原理：XSS、CSRF、点击劫持、跨域配置错误都和前端页面直接相关。场景：输入输出转义、凭证存储、跨域配置、埋点 SDK 接入都离不开安全意识。边界：前端不是最后一道防线，但经常是第一道暴露面。

## 九、任务调度与页面流畅度

73. 浏览器一帧大概经历哪些阶段？
答：结论：可以粗略理解成处理输入、执行 JS、样式计算、布局、绘制、合成，最后提交到屏幕。原理/场景：把一帧理解成输入、脚本、样式、布局、绘制、合成，会更容易定位卡顿发生在哪一段。
延伸：一帧超过预算时，用户就会明显感知到卡顿。
74. 为什么 `requestAnimationFrame` 更适合动画？
答：结论：因为它跟浏览器刷新节奏对齐，时机更稳定，也比自己用定时器更不容易掉帧。原理/场景：`requestAnimationFrame` 和浏览器刷新节奏对齐，适合视觉更新；它不是更快，而是时机更合适。
延伸：动画题里 `requestAnimationFrame` 基本是标准答案之一。
75. `requestIdleCallback` 适合什么场景？
答：结论：适合不着急、可以慢慢做的低优先级任务，比如预计算、预加载、埋点整理。原理/场景：`requestIdleCallback` 更适合不着急的低优先级任务，但不能拿来承诺关键逻辑一定及时执行。
76. 页面卡顿时你如何判断是脚本、样式还是渲染问题？
答：结论：我会看 Performance 面板，先看长任务在哪，脚本长就盯 JS，布局绘制高就往样式和渲染排。原理/场景：定位卡顿时最好先看主线程火焰图，再判断是 JS 长任务、布局抖动还是绘制压力。
77. 什么是长任务？
答：结论：长任务就是主线程上执行时间过长、明显堵住交互的一段任务，通常超过 50ms 就值得警惕。原理：主线程被长时间占住后，输入响应、渲染和其他回调都只能排队。场景：大 JSON 解析、复杂 diff、同步循环计算、重布局都可能制造长任务。边界：长任务不只来自 JavaScript，本地布局和绘制压力也可能在时间轴上表现出来。
78. 为什么长任务会影响交互响应？
答：结论：因为浏览器主线程被长任务占住后，点击、输入、渲染和其他回调都只能排队，所以用户就会感觉页面没反应。原理：主线程是脚本执行、样式计算、布局绘制共享的稀缺资源，只要其中一段任务占用太久，整帧节奏就会被拖慢。场景：这也是为什么复杂计算、重布局和大批量 DOM 更新都会直接影响交互响应。
79. 如果输入框卡顿，你会如何分析？
答：结论：如果输入框卡顿，我会先看是不是每次输入都触发大计算、重渲染、接口风暴或者同步存储写入。原理：输入场景最怕把高频交互和重逻辑绑死在同一帧里。场景：搜索联想、复杂表单、富文本编辑器都很容易出现这类问题。边界：排查时要同时看 JS 长任务、布局抖动和请求并发，不要只盯一个维度。
80. 如何用 Performance 面板定位性能瓶颈？
答：结论：先录制复现，再看主线程火焰图、长任务、Layout/Paint 占比、网络 waterfall 和用户操作时间点。原理/场景：Performance 面板常看主线程、长任务、Layout/Paint 占比、用户操作时间点和网络 waterfall。
延伸：如果你能结合一次真实录制结果讲，会更像项目经验。

## 十、工程视角问题

81. 为什么前端优化不能只盯着代码，还要看网络、缓存、渲染？
答：结论：因为用户感知的是整条链路，代码再漂亮，网络慢、缓存错、渲染卡，体验一样差。原理/场景：用户感知的是整条链路，所以性能优化不能只盯业务代码，还要同时看网络、缓存和渲染。
82. 如何为一个复杂页面设计性能监控指标？
答：结论：我会分加载、交互、稳定性三类去定，比如首屏、接口耗时、长任务、错误率、CLS 这些。原理/场景：指标设计通常分加载、交互、稳定性三类，这样更适合监控和归因。
83. 页面报错、白屏、卡顿、加载慢分别该如何归因？
答：结论：报错要看异常链路，白屏要看资源和渲染，卡顿要看主线程，加载慢要看网络和首屏关键资源，四类问题不要混着查。原理：它们落在浏览器链路的阶段不同，排查入口也不同。场景：真正专业的排查方式通常是先分类，再顺着链路收缩范围。边界：如果一上来什么都一起看，往往会把问题查散。
84. 如何向面试官讲清一次完整的前端性能优化案例？
答：结论：我会按问题现象、定位过程、核心优化、结果数据、复盘沉淀这五步讲。原理/场景：高质量案例最好能讲清“现象 - 定位 - 改动 - 数据结果 - 后续治理”这条线。
85. 如果让你从浏览器角度评价一个前端项目是否“专业”，你会看哪些点？
答：结论：如果从浏览器角度评价一个前端项目是否专业，我会看缓存策略、资源加载、异常监控、性能指标、交互流畅度，以及安全和兼容性意识。原理：这些点基本覆盖了浏览器里从请求到渲染再到治理的整条链路。场景：专业项目通常不是某一项特别极致，而是链路上没有明显短板。边界：只会写业务功能还不够，真正成熟的是同时兼顾体验和治理。

## 十一、Web Worker 与多线程

86. 浏览器里的 Web Worker 适合解决什么问题？
答：结论：适合把重计算、可并行的任务挪出主线程，避免页面交互被卡住。原理/场景：Web Worker 的价值是把重 CPU 任务移出主线程，让交互和渲染不被长任务拖死。
延伸：大文件解析、图片压缩、CSV 处理都是典型 Worker 场景。
87. 为什么 CPU 密集型任务放在主线程里容易导致页面卡顿？
答：结论：CPU 密集型任务放在主线程容易卡，是因为主线程既要跑 JS，又要响应交互和渲染，重计算一塞进去，别的事就排不上。原理：浏览器主线程是稀缺资源，任何长时间独占都会影响用户感知。场景：图片压缩、CSV 解析、大 JSON 处理、复杂计算都适合考虑搬去 Worker。边界：如果任务很小，开 Worker 反而可能不划算。
88. 主线程和 Worker 线程是如何通信的？
答：结论：主要就是靠 `postMessage` 发消息，双方监听消息事件来收发数据。原理/场景：主线程和 Worker 常用 `postMessage` 双向通信，复杂场景要设计统一消息协议和错误结构。
89. `postMessage` 的底层传输机制可以如何理解？
答：结论：可以把它理解成基于消息队列的线程间通信，默认走结构化克隆，必要时也能转移所有权。原理/场景：`postMessage` 默认走结构化克隆，大对象场景下还可以通过可转移对象减少复制开销。
延伸：默认是结构化克隆，性能敏感时再考虑可转移对象。
90. 什么是结构化克隆？它和普通浅拷贝、深拷贝有什么区别？
答：结论：它是浏览器提供的一套对象复制规则，比浅拷贝深，但也不是所有类型都能无脑复制。原理/场景：结构化克隆比简单深拷贝规则更系统，但仍不是所有类型都能完整传递。
91. 什么是可转移对象？为什么 `ArrayBuffer` 适合在线程间传递？
答：结论：可转移对象可以直接把所有权交给另一个线程，不用真的复制一份，所以像 `ArrayBuffer` 传大数据更高效。原理/场景：像 `ArrayBuffer` 这类可转移对象适合传大数据，因为可以转移所有权而不是复制内容。
92. 如果主线程连续传两条消息给 Worker，Worker 如何区分和回传？
答：结论：主线程连续给 Worker 发多条消息时，我一般会在消息体里带 `type`、`id`、`payload`，Worker 回传时再按 `id` 对应起来。原理：消息一多，如果没有协议和关联 id，就很难做并发管理和错误归因。场景：批量文件处理、多任务解析、不同类型命令分发都需要这套协议。边界：不要依赖消息返回顺序去做业务判断，显式关联更稳。
93. 主线程和 Worker 之间会有多条消息吗？如何设计消息协议？
答：结论：当然会，所以协议最好固定成统一结构，不然一复杂就很难维护。原理/场景：消息一多就需要统一的 `type`、`id`、`payload` 协议，否则排查和扩展都很痛苦。
94. Worker 返回给主线程的数据结构通常应该如何设计？
答：结论：Worker 返回主线程的数据结构，我会统一成 `id`、`success`、`data`、`error` 这类字段，方便兜底和并发管理。原理：统一协议能让主线程更容易做结果分流、错误提示和超时处理。场景：多任务队列、可重试任务、文件处理进度回传都很适合这种结构。边界：协议要尽量稳定，不要让不同 Worker 各自发明一套返回格式。
95. Web Worker 能直接访问 DOM 吗？为什么不能？
答：结论：不能，因为 DOM 属于主线程渲染上下文，Worker 的设计目标就是别来碰这块共享状态。原理/场景：Worker 不能操作 DOM，本质上是为了避免把 UI 共享状态暴露给后台线程造成复杂同步问题。
延伸：Worker 不碰 DOM，是浏览器线程模型里很重要的边界。
96. Worker 里能使用哪些浏览器能力，不能使用哪些能力？
答：结论：Worker 能用定时器、网络请求、部分二进制处理能力，但不能直接操作 DOM、`window` 和大部分界面相关 API。原理：Worker 的设计目标就是把计算和 UI 线程隔离开。场景：适合做后台计算、文件解析、数据转换，不适合做页面交互控制。边界：如果任务强依赖 DOM 测量或组件状态，Worker 就不是合适选择。
97. 什么情况下你会选择 Worker，而不是分片调度或 `requestIdleCallback`？
答：结论：任务真的重、持续时间长、而且能和 UI 解耦时，我会优先 Worker；只是小任务拆分，就不一定要开线程。原理/场景：如果任务只是零碎小工作，分片调度或 `requestIdleCallback` 可能比开线程更划算。
98. 大文件处理、图片压缩、CSV 解析为什么适合放到 Worker 里？
答：结论：因为大文件处理、图片压缩、CSV 解析本质上都是重 CPU 任务，放在主线程会直接抢占渲染和交互时间，所以页面很容易一边处理一边卡。原理：这类任务和 DOM 没有强耦合，却会长时间占用计算资源，正适合搬到 Worker 里异步处理。场景：这样做的收益是页面依旧能滚动、点击和输入，代价则是要额外处理线程通信、序列化开销和错误回传。
99. 如何避免 Worker 越开越多导致资源浪费？
答：结论：我会控制线程数、能复用就复用，任务结束及时销毁，别一个功能一个 Worker 无脑开。原理/场景：Worker 不是开得越多越好，线程池、复用和及时销毁比“一个功能一个 Worker”更专业。
100. Web Worker、Service Worker、Shared Worker 的区别分别是什么？
答：结论：Web Worker 主要做后台计算，Service Worker 更偏离线缓存和请求代理，Shared Worker 是多个页面共享一个 Worker。原理：三者名字相近，但解决的问题完全不同，一个偏计算，一个偏离线与网络代理，一个偏多页面共享。场景：区分这三者时，我会直接按“后台计算、离线代理、多页面共享”这条线来讲，最不容易混。
延伸：不要把 Web Worker 和 Service Worker 的职责混在一起。
