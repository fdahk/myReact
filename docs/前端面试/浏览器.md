# 浏览器训练题纲

## 使用方式

这个文档按浏览器高频链路整理，答案统一尽量覆盖结论、原理、优化方向和边界，方便直接展开。

---

## 一、浏览器基础认知

1. 浏览器的主要进程和线程有哪些？
答：浏览器通常要从“多进程 + 进程内多线程”两层去理解：进程层面常见有浏览器主进程、渲染进程、网络进程、GPU 进程、插件或扩展相关进程；线程层面则常见有负责页面渲染的线程、JavaScript 引擎线程、事件循环相关线程、定时器与网络回调协作机制等。这样设计的目的，是把页面渲染、网络收发、图形合成和浏览器管理能力拆开，既提高稳定性和安全性，也让一个页面卡住时不至于把整个浏览器一起拖死。
2. 为什么现代浏览器要采用多进程架构？
答：现代浏览器采用多进程架构，核心原因是隔离：隔离崩溃、隔离权限、隔离站点数据、隔离资源消耗。这样一个标签页里的恶意脚本、死循环或渲染异常，不会轻易把整个浏览器拖垮，同时站点之间也更容易做安全边界控制；代价则是进程更多、内存占用更高、进程间通信更复杂，但对现代 Web 的复杂度来说，这个代价通常是值得的。
3. 渲染进程主要负责什么？
答：渲染进程主要负责把网络拿回来的 HTML、CSS、JavaScript 等资源真正变成用户能看到、能交互的页面，里面会涉及文档解析、样式计算、布局、绘制、图层合成、事件处理以及大部分前端最关心的脚本执行。可以把它理解成“页面真正活起来的地方”，因为浏览器是不是白屏、卡顿、布局抖动、脚本阻塞，很多都发生在这里。
4. GUI 渲染线程和 JavaScript 引擎线程为什么互斥？
答：因为 JavaScript 可以随时读写 DOM 和样式，如果脚本执行和界面渲染同时并行修改同一份页面状态，就很容易出现“脚本读到一半旧 DOM、渲染画到一半新样式”的一致性问题。浏览器因此通常让二者在关键阶段互斥调度，也就是脚本忙时渲染要等，渲染关键工作进行时脚本也不能随意插进来，所以前端才会经常说“长时间执行 JS 会卡住页面渲染”。
5. 浏览器如何实现标签页隔离？
答：浏览器通常会给标签页分配独立或相对独立的渲染进程，再结合站点隔离策略，把不同源或高风险站点尽量拆到不同进程里处理。这样页面之间的内存、脚本执行上下文和权限边界更容易隔离，一个页面出问题也更不容易影响别的页面；不过它并不是绝对的一页一进程，浏览器会根据站点关系、资源占用和复用策略做动态平衡。
6. 浏览器为什么既是运行环境，又是渲染平台？
答：因为浏览器一方面是 JavaScript 的宿主环境，提供定时器、网络请求、事件系统、存储等运行时能力；另一方面又负责把 HTML、CSS 和图形结果真正渲染到屏幕上，所以它既像一个应用运行容器，又像一个专门的图形呈现系统。理解这一点很重要，因为前端很多问题都不是单纯“代码逻辑错了”，而是运行环境、网络链路和渲染平台三者共同作用的结果。
7. 你如何解释浏览器和 Node.js 在运行 JavaScript 上的差别？
答：两者运行的都是 JavaScript 语言，但宿主环境完全不同，所以可用能力、事件模型细节和安全边界也不同。浏览器更偏向 DOM、BOM、渲染、安全隔离和用户交互，Node.js 更偏向文件系统、网络服务、进程能力和服务器场景；因此“JS 一样”不代表“运行机制一样”，很多 API 能不能用、为什么这样调度，都要回到宿主环境本身去看。
8. 前端开发为什么必须了解浏览器底层运行机制？
答：因为用户最终接触的不是源码，而是浏览器里的实际运行结果，白屏、卡顿、重绘抖动、跨域失败、缓存不生效、长任务、内存泄漏这些问题，本质上都和浏览器怎么请求资源、怎么执行脚本、怎么排队任务、怎么渲染页面直接相关。前端不需要把自己训练成浏览器内核工程师，但至少要能把“输入 URL 到页面可交互”这条核心链路讲清楚，不然很多问题就只能停留在经验修 bug 的层面。

## 二、从输入 URL 到页面展示

9. 从输入 URL 到页面最终显示，中间完整经历了什么？
答：这条链路可以按顺序理解成：先解析 URL，看协议、域名、端口和路径；然后查询 DNS，把域名解析成 IP；接着建立连接，HTTP 通常要经过 TCP，HTTPS 还要多一层 TLS 握手；连接建立后浏览器发送请求，服务端返回 HTML 或重定向等响应；浏览器开始边下载边解析 HTML，构建 DOM，同时发现外链 CSS、脚本、图片等资源后继续并行请求；CSS 会参与 CSSOM 构建，DOM 和 CSSOM 结合后形成渲染所需的信息，再经历样式计算、布局、绘制、图层合成，最后页面显示出来并逐步变成可交互状态。真正专业的回答，不是只背几个关键词，而是知道每一段都可能成为首屏慢和白屏的来源。
10. DNS 查询大致做了什么？
答：DNS 查询本质上是在把人类可读的域名翻译成机器真正能连接的 IP 地址，浏览器通常会先查自己的缓存、系统缓存和本地 hosts，没命中再把请求交给递归解析器逐级去问，最后从权威 DNS 拿到结果。它虽然只是请求开始前的一步，但会直接影响首个请求什么时候真正发出去，所以 DNS 解析慢，首屏和接口首包时间都会整体被后推。
11. TCP 连接建立过程对前端有什么影响？
答：TCP 建连会占用请求真正发出前的时间预算，所以它直接影响首包之前的等待时长；连接建立得越慢，页面、接口和静态资源真正开始传输的时间就越晚。这个阶段在弱网、跨地域访问、频繁新建连接或连接复用做得不好的场景下尤为明显，所以前端讲性能时不能只盯 bundle 体积，也要看到“建连本身就可能已经很贵”。
12. HTTPS 在这个过程中比 HTTP 多了哪些步骤？
答：HTTPS 比 HTTP 多出来的核心步骤，是在 TCP 之上再进行 TLS 握手，也就是客户端要先验证服务端证书是否可信、确认自己连的是谁，再协商本次会话要使用的加密算法和会话密钥，之后双方才会在这个安全通道里传输真正的 HTTP 数据。它带来的代价是连接建立更复杂、前置耗时更长，但换来的是数据机密性、完整性和身份认证能力，所以不能把 HTTPS 简单理解成“HTTP 套个壳”。
13. 浏览器收到 HTML 后会如何处理？
答：浏览器收到 HTML 后通常不会傻等整个文档下载完，而是边接收边解析，把标记语言逐步转换成 DOM 树；解析过程中如果遇到外链 CSS、脚本、图片等资源引用，还会继续触发相应请求。也就是说，HTML 解析、资源发现和资源下载是交错推进的，这也是为什么首屏性能往往取决于“关键资源多早被发现、多早被加载”，而不只是 HTML 文件本身多快返回。
14. 什么是预解析？
答：预解析可以理解成浏览器在正式 HTML 解析之外，又开了一条更偏“侦察兵”性质的扫描路径，提前发现文档中可能马上就会用到的脚本、样式、图片等资源引用，从而更早发起下载。它的存在意义，是减少“等主解析走到那里才发现资源还没下”的被动等待，所以首屏资源越关键，越依赖浏览器尽早把它们扫描出来。
15. 资源加载和页面渲染是如何并行推进的？
答：真实浏览器链路并不是“先把所有资源下载完，再开始渲染”的绝对串行流程，而是 HTML 解析、资源发现、资源下载、样式准备和后续渲染会在保证正确性的前提下尽量并行推进。比如 HTML 还在解析时，图片和样式可能已经开始请求，部分 DOM 也可能已经生成；但某些资源又会成为关键阻塞点，比如 CSS 会影响渲染结果，普通脚本会打断解析，这就是为什么理解“哪些能并行、哪些会阻塞”比死背流程图更重要。
16. 如果页面白屏很久，你会从这个链路的哪些环节排查？
答：我会沿着整条链路从前往后拆：先看 DNS、TCP、TLS 这些前置耗时是不是异常，再看主文档和关键接口返回是否过慢，然后看首屏 CSS、首屏 JS、静态资源有没有阻塞，接着再看脚本执行有没有报错、有没有长任务把渲染压住，最后确认是不是样式计算、布局或挂载逻辑本身出了问题。白屏排查最怕一上来只盯业务代码，正确做法是按 Network waterfall 和 Performance 时间线一段段缩小范围。

## 三、渲染流程

17. 浏览器的渲染流程是什么？
答：浏览器渲染流程通常可以概括为：解析 HTML 得到 DOM，解析 CSS 得到 CSSOM，再把结构和样式结合成后续渲染所需的数据，然后进行样式计算、布局、绘制，最后做图层合成并把结果提交到屏幕。很多资料会把它简写成 DOM、CSSOM、Render Tree、Layout、Paint、Composite 这条链，但真正理解时要知道它不是一条完全静止的流水线，而是页面每次样式或结构变化后，浏览器都可能重新走其中一部分甚至多部分流程。
18. 什么是 DOM 树和 CSSOM 树？
答：DOM 树描述的是页面结构，也就是有哪些节点、父子关系如何；CSSOM 树描述的是样式规则和最终样式计算需要依赖的信息。浏览器只有同时拿到“页面长什么结构”和“这些结构该长什么样”，才能继续往下决定哪些节点可见、尺寸多大、位置在哪，所以 CSS 不只是给页面“上妆”，它实际上参与了首屏能否正确渲染这件事。
19. Render Tree 是如何生成的？
答：Render Tree 可以理解成浏览器为了后续布局和绘制而整理出来的一棵“可见渲染树”，它会基于 DOM 和样式计算结果，把真正需要参与渲染的节点挑出来，并附上对应样式信息。像 `display: none` 这种不参与显示的节点通常不会进入这棵树，所以它不是简单复制 DOM，而是面向“接下来怎么画到屏幕上”做过筛选和加工的结果。
20. Layout 和 Paint 分别在做什么？
答：Layout 解决的是几何问题，也就是每个可见元素最终多大、放在哪、和谁发生位置关系；Paint 解决的是视觉问题，也就是把文字、颜色、背景、边框、阴影这些真正画出来。前者更偏“算坐标和尺寸”，后者更偏“把外观画出来”，所以性能排查时一定要先分清是布局太重，还是绘制太重，因为它们的优化方向完全不同。
21. Composite 阶段的作用是什么？
答：Composite 阶段可以理解成“把前面已经准备好的各个图层最后拼起来并提交到屏幕”，它更关注图层之间怎么合成，而不是重新计算布局或重新把所有像素都画一遍。很多动画优化之所以强调走合成层，就是因为如果某次变化只需要合成，不需要重新布局和重绘，浏览器的压力会小很多，页面也更容易保持流畅。
22. 重排和重绘的区别是什么？
答：重排也常叫回流，指的是元素的几何信息变了，浏览器必须重新计算位置和尺寸；重绘则是布局不变，只是颜色、背景、边框这类视觉外观需要重新画。因为重排往往会连带触发后续的绘制甚至合成，所以一般比单纯重绘更贵；但也别把它理解成“重绘就没有成本”，如果重绘区域很大、频率很高，照样会拖慢页面。
23. 哪些操作会触发重排？
答：凡是会影响元素几何信息或文档流关系的操作，都可能触发重排，比如修改宽高、边距、定位、字体大小、增删可见节点，或者在前面刚改过样式的情况下又去读取 `offsetWidth`、`offsetHeight`、`getBoundingClientRect()` 这类布局信息。后者尤其值得注意，因为它会逼浏览器立刻把前面还没来得及统一处理的布局先算出来，这就是常说的强制同步布局。
24. 哪些操作只会触发重绘？
答：像修改文字颜色、背景色、边框颜色、阴影等不影响元素大小和位置的样式，通常只会触发重绘，因为浏览器不需要重新算布局，只需要重新把视觉内容画出来。这类更新在 hover 反馈、主题切换、局部高亮里很常见，但要注意“只重绘”不等于“完全免费”，如果改动范围大或者发生得很频繁，同样会影响流畅度。
25. 为什么 `transform` 和 `opacity` 更适合做动画？
答：因为 `transform` 和 `opacity` 在很多情况下可以避开布局计算，甚至避开大面积重绘，直接在合成阶段完成变化，所以浏览器处理这类动画的成本通常更稳定。你可以把它理解成“元素本身不用重新排版，只是换个位置、缩放一下或透明度变了”，因此它们更适合高频动画和交互反馈，而像频繁改 `top`、`left`、`width` 这类属性就更容易把布局链路一起拖进来。
26. 如何减少重排和重绘？
答：核心思路是减少不必要的布局变化、避免频繁读写 DOM 交替、把多次修改合并成批量更新，并尽量让高频动画走 `transform` 和 `opacity` 这类更容易停留在合成阶段的属性。说白了就是两句话：一是别让浏览器反复重新算页面结构，二是让必须变化的内容尽量走更轻的渲染路径。

## 四、页面性能与优化

27. 什么是关键渲染路径？
答：关键渲染路径说的就是从浏览器拿到 HTML，到首屏关键内容真正出现在用户眼前这段最重要的路径，哪些资源和步骤挡在这条路上，首屏就会被谁拖慢。它关注的不是页面里所有东西，而是“哪些东西必须先准备好，用户才能先看到内容”，所以优化首屏本质上是在给这条路径减负。
28. 首屏性能受哪些因素影响？
答：首屏性能会同时受到网络前置耗时、HTML 返回速度、关键 CSS、首屏 JavaScript 体积与执行时长、首屏图片大小、接口是否串行、页面渲染复杂度等多种因素影响，它是整条链路叠加出来的结果，不是某一个指标单独决定的。所以首屏优化最重要的第一步不是马上“上优化手段”，而是先拆清楚到底慢在建连、下载、执行还是渲染。
29. 什么是阻塞渲染资源？
答：阻塞渲染资源指的是那些如果不先处理完，浏览器就没法正确继续首屏渲染的资源，最典型的就是关键 CSS 和某些会打断解析的脚本。它们之所以重要，是因为即使网络还不错，只要关键路径上这些资源回来得慢、执行得晚，用户看到内容的时间依然会被明显后推。
30. 为什么 CSS 会阻塞渲染？
答：因为浏览器在把内容真正画出来之前，需要先知道这些节点最终应该长什么样、尺寸多大、位置在哪，而这些都依赖 CSS 参与样式计算。要是 CSS 还没准备好就先画，后面又要重算和重画，体验和性能都更差，所以浏览器宁愿先等关键样式到位，再开始相对正确地渲染首屏内容。
31. 为什么普通 `script` 会阻塞解析？
答：普通 `script` 会阻塞 HTML 解析，是因为脚本拿到执行权后可能立刻去改 DOM、插入新节点、读取当前文档状态，如果浏览器不暂停解析、继续往后走，脚本看到的页面状态和最终结果就可能不一致。浏览器因此会先停下文档解析，把这个脚本执行完，再继续向后解析，这也是为什么首屏脚本放得不合理会明显拖慢页面。
32. `defer` 和 `async` 的区别是什么？
答：两者都会让脚本下载尽早开始，但执行时机不同：`async` 下载完就尽快执行，所以不保证多个脚本之间的顺序，更适合独立脚本；`defer` 会等 HTML 解析完成后再按文档顺序执行，更适合依赖 DOM 结构、也依赖前后脚本顺序的场景。简单记忆就是，`async` 更像“谁先下完谁先跑”，`defer` 更像“先并行下载，等文档就绪后按顺序上场”。
33. 图片优化有哪些常见手段？
答：图片优化通常不是只做一件事，而是格式选择、尺寸裁剪、压缩质量、懒加载、响应式图片和缓存策略一起配合。比如首屏大图可以考虑更现代的格式，列表图片要避免原图直出，非首屏图片适合懒加载，高清屏和小屏又适合用不同尺寸资源，所以图片优化的核心不是“换成某个格式就结束”，而是让图片在合适的时机、以合适的大小和合适的格式被加载出来。
34. 长列表为什么会卡顿？常见解决思路是什么？
答：长列表卡顿的根因，通常不是“数组遍历慢”这么简单，而是页面里真实 DOM 节点太多，导致布局、绘制、事件处理和内存占用一起升高，用户一滚动或筛选时浏览器压力就会瞬间放大。常见解决思路是虚拟列表、分页、分片渲染、减少单项复杂度和图片懒加载，其中虚拟列表最典型，因为它用“只保留视口附近少量节点”的方式，直接把页面压力从源头降下来。
35. 懒加载和预加载分别适合什么场景？
答：懒加载适合那些现在还用不到、等用户滚到或触发到再加载也来得及的资源，比如首屏外图片、次级模块和非关键脚本；预加载则适合虽然还没真正执行或显示，但马上就会变得关键、提前准备能减少等待的资源。两者看起来都在管“加载时机”，但方向正好相反，一个是延后不急资源，一个是提前准备关键资源。
36. 资源合并和资源拆分分别适合什么场景？
答：资源合并更适合请求成本特别高、文件又高度稳定的小资源场景；资源拆分更适合现代前端按路由、按功能、按访问时机做按需加载，让首次只下载真正要用的代码，并让缓存粒度更细。尤其在 HTTP/2 之后，请求数不再像 HTTP/1.1 时代那样是绝对负担，所以现代工程里往往不再追求一味合并，而是更关注拆分后的按需加载和长期缓存命中率。
37. 你会如何解释首屏时间、LCP、FCP、CLS、TTI？
答：我会先区分业务指标和通用体验指标：首屏时间通常是业务自己定义的“关键内容出现时间”，FCP 表示首次有内容被画出来，LCP 更关注首屏最大关键内容什么时候出现，CLS 反映页面有没有发生让用户烦躁的布局抖动，TTI 则强调页面什么时候真正具备比较稳定的可交互能力。理解这些指标的关键，是知道它们不是同一个维度的不同叫法，而是在分别观察“有没有看到内容”“重要内容何时出现”“页面稳不稳”“交互什么时候顺”。
38. Lighthouse 的核心指标分别意味着什么？
答：Lighthouse 的核心价值，是把页面体验里常见的加载速度、稳定性、交互响应、可访问性和最佳实践问题，转换成一组可量化、可比较的实验室指标和诊断建议。它很适合做本地优化验证、回归对比和上线前体检，但也要知道它测的是受控环境下的实验室数据，不等于所有真实用户在各种网络和设备上的实际体验，所以最终还是要结合线上监控来判断优化有没有真正生效。

## 五、缓存机制

39. 浏览器缓存有哪些层级？
答：结论：我一般会答内存缓存、磁盘缓存，再往上还有 Service Worker、HTTP 缓存这些层。原理/场景：浏览器缓存不只是一层，更完整的划分是内存缓存、磁盘缓存、HTTP 缓存和 Service Worker。
40. 强缓存和协商缓存的区别是什么？
答：结论：强缓存命中了连请求都不用真发，协商缓存还是会带条件去问服务端能不能继续用旧资源。原理/场景：强缓存解决“能不能直接用”，协商缓存解决“资源可能变了要不要再确认”。
延伸：强缓存和协商缓存几乎一定会和响应头一起考。
41. `Expires` 和 `Cache-Control` 的区别是什么？
答：结论：`Expires` 用的是绝对时间，容易受客户端时间影响；`Cache-Control` 用的是相对时间，表达能力更强，优先级也更高。原理：前者更像早期方案，后者可以进一步表达 `max-age`、`no-cache`、`no-store`、`public`、`private` 等缓存策略。场景：现在工程上一般都以 `Cache-Control` 为主，`Expires` 更多是兼容性补充。边界：如果用户本地时间不准，`Expires` 的判断就可能失真。
42. `Last-Modified` 和 `ETag` 的区别是什么？
答：结论：`Last-Modified` 基于资源最后修改时间判断，`ETag` 基于资源标识判断，后者通常更精确。原理：时间戳方案实现简单、开销低，但精度有限；`ETag` 可以识别更细粒度的变化，不过生成和比较成本通常更高。场景：静态资源或变化不频繁的内容，用时间戳可能已经够用；对精度要求更高时会更偏向 `ETag`。
43. 什么资源适合强缓存？什么资源适合协商缓存？
答：结论：带 hash 的静态资源适合强缓存，HTML 和可能变化但又想复用的接口更适合协商缓存。原理/场景：带 hash 的静态资源适合强缓存，HTML 和变化频率高的资源更适合协商缓存。
44. 前端静态资源为什么常配合哈希命名？
答：结论：因为文件内容一变，哈希名也会跟着变，所以可以放心把缓存时间拉长，而不用担心用户继续命中旧资源。原理：缓存命中本质上依赖 URL，文件名带 hash 后，内容变更就会自然形成新的 URL。场景：这也是前端静态资源常见的“强缓存 + 文件 hash”组合策略。边界：入口 HTML 一般不适合走同样的长强缓存，因为它要负责引用最新资源地址。
45. 浏览器缓存命中失败的常见原因有哪些？
答：结论：缓存命中失败常见原因有响应头没配对、URL 变了、资源被禁缓存、强刷以及代理层改头。原理：浏览器命中缓存要同时满足 URL、缓存头和当前使用条件。场景：发版后不生效、CDN 命中异常、入口 HTML 老是拿旧版本，常常都和这里有关。边界：排查缓存不要只看浏览器，还要同时看 CDN 和网关层。
46. 用户说“我明明发版了但页面没更新”，你怎么排查？
答：结论：我会先看 HTML 有没有更新、静态资源文件名有没有变、缓存头对不对、CDN 有没有刷新到位。原理/场景：这个问题常见根因是 HTML 没更新、静态资源没变更文件名、CDN 没刷新或缓存头配置不合理。

## 六、存储机制

47. `Cookie`、`localStorage`、`sessionStorage` 的区别是什么？
答：结论：`Cookie` 会跟请求走，容量小；`localStorage` 长久存本地；`sessionStorage` 只在当前会话里有效。原理/场景：`Cookie` 会随请求发送，`localStorage`/`sessionStorage` 只在前端可读写，安全边界和使用成本完全不同。
延伸：存储题常会连着问安全问题，尤其是 token 放哪里。
48. `Cookie` 为什么不适合存大量数据？
答：结论：因为 Cookie 容量本来就小，而且会随着同域请求一起发送，数据一多既浪费带宽，也会扩大泄露面。原理：Cookie 的设计目标是维持会话状态，不是本地大容量存储，所以它天然不适合承载大对象或高频变化数据。场景：像登录态标识、会话信息适合放 Cookie，但草稿、大列表缓存、复杂配置就不合适。
49. `localStorage` 的优点和限制是什么？
答：结论：`localStorage` 的优点是简单直接、持久化方便，浏览器关闭后数据通常还在；限制是它是同步 API、容量有限，而且一旦发生 XSS，里面的数据很容易被读走。原理：它更适合轻量级、低频读写、本地可持久化的数据，不适合高频更新或高敏感信息。场景：主题设置、轻量缓存、非敏感配置可以考虑放这里；认证凭证、隐私数据和大数据量缓存就要更谨慎。
50. IndexedDB 适合什么场景？
答：结论：适合离线数据、本地大数据量缓存、复杂结构存储，像文档、草稿、离线列表这种。原理/场景：IndexedDB 更适合大数据量、复杂结构和离线场景，不要拿它和 `localStorage` 做同一量级比较。
51. 前端本地存储有哪些安全风险？
答：结论：前端本地存储最大的安全风险就是一旦发生 XSS，里面的数据基本都可能被读走。原理：`localStorage`、`sessionStorage` 对页面脚本是可读的，恶意脚本一旦执行就能直接访问。场景：token、用户信息、业务配置如果存得过多，泄露面会被放大。边界：本地存储不是不能用，而是不能把高敏感凭证随意暴露给 JS 可读环境。
52. token 为什么不建议随便放在 `localStorage`？
答：结论：因为它对 JS 完全可读，XSS 一旦发生，token 很容易被偷。原理/场景：把 token 放在 `localStorage` 最大风险是 XSS 一旦发生，凭证很容易被直接读取。
53. 浏览器多标签页间如何同步数据？
答：结论：常见做法是监听 `storage` 事件，或者用 `BroadcastChannel`。原理/场景：多标签同步除了 `storage`，新一点的方案还有 `BroadcastChannel`，语义更直接。
54. 如果你要设计前端缓存层，会如何选择不同存储方式？
答：结论：我会按数据敏感度、大小、生命周期来选，敏感认证尽量别放可读存储，大数据用 IndexedDB，轻量配置再考虑 localStorage。原理/场景：选择存储方式时我会同时看数据敏感度、生命周期、容量需求和是否需要跨标签同步。

## 七、事件机制

55. 什么是事件捕获、目标阶段、冒泡阶段？
答：结论：事件从外往内到目标节点的过程是捕获，落到目标节点本身是目标阶段，再从里往外返回就是冒泡。原理：浏览器用这套传播模型让一个事件能被多个层级节点感知和处理。场景：事件委托、全局监听、局部阻止冒泡都建立在这套机制上。边界：并不是所有事件都会冒泡，比如 `mouseenter`、`mouseleave` 就是例外。
56. 事件委托的原理是什么？
答：结论：本质是利用事件冒泡，把子元素的事件交给父元素统一处理。原理/场景：事件委托本质上是利用冒泡把多个子节点监听合并到父节点，常见于列表和动态节点场景。
57. 为什么事件委托适合大量子元素场景？
答：结论：事件委托适合大量子元素场景，因为不用给每个节点都单独绑监听器，内存和维护成本都更低。原理：父节点统一接管子节点事件，本质上是在复用冒泡机制。场景：列表、表格、动态插入节点非常适合委托。边界：如果事件本身不冒泡，或者需要非常细粒度的独立监听，委托就没那么合适。
58. `event.target` 和 `event.currentTarget` 的区别是什么？
答：结论：`target` 是真正触发事件的元素，`currentTarget` 是当前正在处理事件的那个监听节点。原理/场景：`target` 表示真正触发事件的源头，`currentTarget` 表示当前正在执行监听器的节点。
59. 什么情况下你会阻止冒泡？什么情况下不该阻止？
答：结论：只有在明确不希望上层逻辑被触发时我才会阻止冒泡，默认不会乱拦，避免把组件协作链路打乱。原理：事件传播本身是浏览器默认协作机制，乱拦会让父层逻辑失效。场景：弹窗内部点击不想触发遮罩关闭、菜单点击不想触发外层收起时，才值得阻止。边界：阻止冒泡要带着明确意图，不要把它当成万能止血手段。
60. 被动事件监听器有什么作用？
答：结论：它告诉浏览器这个监听器不会调用 `preventDefault`，这样滚动这类场景可以更顺畅。原理/场景：被动监听常用于滚动和触摸场景，告诉浏览器无需等待 `preventDefault`，从而提升流畅度。
61. 为什么滚动监听容易造成性能问题？
答：结论：因为触发频率很高，如果里面还读布局、改样式或做重计算，就很容易卡。原理/场景：滚动监听频率高，如果里面又读布局、改样式或做重计算，很容易形成掉帧和长任务。
62. 前端框架中的合成事件和原生事件有什么差异？
答：结论：合成事件通常是框架做了一层封装，方便兼容和统一管理，但底层还是绕不开原生事件机制。原理/场景：框架合成事件主要解决兼容、统一管理和批量处理，但底层仍然建立在原生事件机制之上。

## 八、浏览器安全

63. 什么是同源策略？
答：结论：就是协议、域名、端口三者都相同才算同源，不同源的很多资源访问会被浏览器限制。原理/场景：同源策略是浏览器安全模型的核心之一，很多跨域、安全和认证问题都要从这里展开。
延伸：跨域、安全、认证很多题都要先从同源策略展开。
64. 跨域为什么会发生？
答：结论：因为协议、域名、端口只要有一个不同，浏览器就会把它判定成不同源，而现代前后端分离、主站与子域、CDN 分发这些场景都很容易出现这种差异。原理：跨域问题本质来自浏览器的同源策略，是浏览器为了隔离站点数据和脚本权限做的安全边界。场景：开发环境里最常见的是前端本地服务请求测试接口，线上则常见于主站、子域和静态资源域名拆分。
65. 常见跨域方案有哪些？各自适合什么场景？
答：结论：最常见是 CORS，开发环境常用代理，老场景才会提 JSONP，核心还是看服务端能不能配合。原理/场景：真实项目里最常见还是 CORS 和开发代理，JSONP 更多是历史知识点。
66. 什么是 CORS？
答：结论：CORS 就是浏览器基于响应头判断这个跨域请求能不能放行给前端脚本读取的一套机制。原理：请求本身常常能发出去，但浏览器会根据服务端返回的 `Access-Control-*` 响应头决定是否放行。场景：前后端分离、主站和子域协作、CDN 资源访问都很常见。边界：CORS 约束的是浏览器端脚本读取，不是服务器之间的访问控制。
67. 预检请求是什么？为什么会出现？
答：结论：非简单请求前，浏览器会先发个 `OPTIONS` 去问服务端允不允许，这个就是预检。原理/场景：预检请求是浏览器在真正发非简单请求前，先帮你问服务端“这类跨域请求能不能放行”。
68. 什么是 JSONP？为什么现在用得少？
答：结论：JSONP 本质上是利用 `script` 标签不受同源限制来跨域拿数据，但它只能发 GET，请求和错误处理能力都很弱，所以现在基本只剩历史知识点价值。原理：它不是在真正放开浏览器的跨域读取限制，而是借助脚本加载机制，让服务端返回一段函数调用代码。场景：早期服务端不能配 CORS 时会用它兜底，但现代项目里更常见的还是 CORS 或反向代理。边界：因为它执行的是脚本，不只是数据，所以安全和可维护性都明显更差。
69. 什么是 XSS？常见防护方式有哪些？
答：结论：就是把恶意脚本注进页面执行。防护上要做输出转义、内容清洗、CSP，别乱拼 `innerHTML`。原理/场景：XSS 防护重点是输入输出都做约束，尤其避免把不可信内容直接拼进 HTML 执行环境。
延伸：XSS 题常和 CSP、转义、`innerHTML` 风险一起追问。
70. 什么是 CSRF？常见防护方式有哪些？
答：结论：就是利用用户已登录状态诱导浏览器发起恶意请求。常见防护是 CSRF Token、`SameSite` Cookie、来源校验。原理/场景：CSRF 的本质是浏览器自动携带登录态，所以防护手段通常围绕“让攻击者无法伪造有效请求”。
71. 什么是点击劫持？
答：结论：点击劫持就是把页面藏在透明 iframe 里，诱导用户点到不该点的按钮或操作。原理：攻击者利用用户看得见的界面覆盖真实页面，让点击目标发生偏移。场景：支付确认、授权按钮、敏感设置页都可能成为目标。边界：常见防护是 `X-Frame-Options` 或 CSP 的 `frame-ancestors`。
72. 为什么前端也必须理解安全问题？
答：结论：前端必须理解安全问题，因为很多攻击入口就在浏览器端，至少要知道风险从哪里进、自己该怎么挡第一层。原理：XSS、CSRF、点击劫持、跨域配置错误都和前端页面直接相关。场景：输入输出转义、凭证存储、跨域配置、埋点 SDK 接入都离不开安全意识。边界：前端不是最后一道防线，但经常是第一道暴露面。

## 九、任务调度与页面流畅度

73. 浏览器一帧大概经历哪些阶段？
答：浏览器的一帧可以粗略理解成这样几个阶段：先处理用户输入和队列里的任务，然后执行脚本，必要时做样式计算、布局、绘制和图层合成，最后把结果提交到屏幕。虽然真实实现比这更复杂，但前端排查卡顿时用这条心智已经很够用，因为你只要知道一帧里脚本、布局和绘制都在争主线程时间，就能明白为什么任何一个环节太重，都会让页面掉帧。
74. 为什么 `requestAnimationFrame` 更适合动画？
答：因为 `requestAnimationFrame` 的回调会尽量和浏览器下一次重绘节奏对齐，时机更适合做视觉更新，比自己用 `setTimeout` 或 `setInterval` 更不容易出现“逻辑更新时机和绘制节奏错位”的情况。它不是因为更快，而是因为更懂浏览器什么时候准备画下一帧，所以特别适合动画、拖拽和需要跟屏幕刷新配合的场景。
75. `requestIdleCallback` 适合什么场景？
答：`requestIdleCallback` 更适合那些“不马上做也没关系”的低优先级任务，比如预计算、预加载、埋点整理、缓存清理这类背景工作。它的核心含义不是“空闲时一定立刻执行”，而是“只有浏览器当前比较闲，才尽量给你一点时间”，所以不能拿它来承载关键业务逻辑，更适合做锦上添花的后台任务。
76. 页面卡顿时你如何判断是脚本、样式还是渲染问题？
答：我会先录制 Performance，看主线程火焰图里时间到底花在了哪：如果脚本执行块很长，就先怀疑 JS 计算、同步循环、框架更新或数据处理；如果 Layout、Recalculate Style、Paint 很突出，就往样式和渲染链路排；如果两边都有，就继续看是不是脚本频繁读写布局触发了连锁开销。关键是别凭感觉猜，而是先让时间线告诉你主线程到底忙在什么地方。
77. 什么是长任务？
答：长任务指的是在主线程上持续执行时间过长、足以明显阻塞输入和渲染的一段工作，通常超过 50ms 就会被当成值得警惕的信号。它可能来自大段 JavaScript 计算，也可能来自很重的布局和绘制，本质上都是“主线程太久没空下来”，所以用户会感知成点击不响应、滚动不跟手、输入卡顿。
78. 为什么长任务会影响交互响应？
答：因为浏览器主线程是前端最稀缺的公共资源，点击、输入回调、脚本执行、样式计算、布局和绘制都要抢它，只要某一段任务长时间不结束，其他该处理的工作就只能继续排队。用户看到的结果，就是点了没反应、输入有延迟、滚动掉帧，所以“交互卡顿”本质上往往不是交互逻辑本身的问题，而是主线程没有及时空出来。
79. 如果输入框卡顿，你会如何分析？
答：我会先看每次输入是不是都绑上了重计算、整页重渲染、频繁请求、同步存储写入或布局测量，因为输入本身是高频交互，最怕和重逻辑绑在同一帧里。接着再结合 Performance 和框架 Profiler 看，是 JavaScript 太重、渲染链路太重，还是请求和结果处理把输入节奏拖慢了；像搜索联想、复杂表单、富文本编辑器这类场景，通常都是“输入很轻，但输入后面挂的东西太重”。
80. 如何用 Performance 面板定位性能瓶颈？
答：最实用的方式是先稳定复现问题，再录一次 Performance，接着沿着用户操作时间点去看主线程火焰图，关注有没有长任务、脚本执行块是否过长、Layout/Paint 占比是否异常，以及网络 waterfall 和渲染时机有没有对不上。它的价值不只是“看哪里红了”，而是帮你把卡顿拆成具体阶段：到底是脚本慢、布局慢、绘制慢，还是网络返回太晚导致后续一连串工作都往后推。

## 十、工程视角问题

81. 为什么前端优化不能只盯着代码，还要看网络、缓存、渲染？
答：结论：因为用户感知的是整条链路，代码再漂亮，网络慢、缓存错、渲染卡，体验一样差。原理/场景：用户感知的是整条链路，所以性能优化不能只盯业务代码，还要同时看网络、缓存和渲染。
82. 如何为一个复杂页面设计性能监控指标？
答：结论：我会分加载、交互、稳定性三类去定，比如首屏、接口耗时、长任务、错误率、CLS 这些。原理/场景：指标设计通常分加载、交互、稳定性三类，这样更适合监控和归因。
83. 页面报错、白屏、卡顿、加载慢分别该如何归因？
答：结论：报错要看异常链路，白屏要看资源和渲染，卡顿要看主线程，加载慢要看网络和首屏关键资源，四类问题不要混着查。原理：它们落在浏览器链路的阶段不同，排查入口也不同。场景：真正专业的排查方式通常是先分类，再顺着链路收缩范围。边界：如果一上来什么都一起看，往往会把问题查散。
84. 如何向面试官讲清一次完整的前端性能优化案例？
答：结论：我会按问题现象、定位过程、核心优化、结果数据、复盘沉淀这五步讲。原理/场景：高质量案例最好能讲清“现象 - 定位 - 改动 - 数据结果 - 后续治理”这条线。
85. 如果让你从浏览器角度评价一个前端项目是否“专业”，你会看哪些点？
答：结论：如果从浏览器角度评价一个前端项目是否专业，我会看缓存策略、资源加载、异常监控、性能指标、交互流畅度，以及安全和兼容性意识。原理：这些点基本覆盖了浏览器里从请求到渲染再到治理的整条链路。场景：专业项目通常不是某一项特别极致，而是链路上没有明显短板。边界：只会写业务功能还不够，真正成熟的是同时兼顾体验和治理。

## 十一、Web Worker 与多线程

86. 浏览器里的 Web Worker 适合解决什么问题？
答：Web Worker 最适合解决“任务很重、和 DOM 没强耦合、继续放在主线程会把页面卡住”的问题，比如大文件解析、图片压缩、复杂计算、CSV 处理等。它的核心价值不是让代码看起来更高级，而是把 CPU 密集型工作搬离主线程，让页面还能继续响应输入、滚动和渲染，所以 Worker 本质上是在帮前端争夺主线程时间。
87. 为什么 CPU 密集型任务放在主线程里容易导致页面卡顿？
答：因为浏览器主线程不仅要跑 JavaScript，还要处理用户输入、样式计算、布局和绘制，任何重计算一旦长时间独占它，其他所有工作就都只能排队等待。于是用户表面上看到的是“页面卡了”，本质上其实是主线程被抢光了，所以像图片压缩、大 JSON 处理、复杂数据转换这种和 UI 无直接关系的重任务，天然就更适合考虑 Worker。
88. 主线程和 Worker 线程是如何通信的？
答：主线程和 Worker 通常通过 `postMessage` 发送消息、监听消息事件来双向通信，本质上是一种基于消息传递的线程协作方式。因为它们不能像普通函数调用那样直接共享 DOM 上下文和大量运行时状态，所以一旦业务稍微复杂，就应该把消息设计成清晰的协议，而不是随手丢个对象过去再靠猜。
89. `postMessage` 的底层传输机制可以如何理解？
答：可以把 `postMessage` 理解成浏览器提供的一套线程间消息通信机制，默认会把要传的数据按结构化克隆规则复制一份再送过去，而不是两个线程直接拿着同一块普通对象内存一起改。这样做更安全、更容易隔离上下文，但数据量一大复制成本也会上来，所以性能敏感场景才会进一步考虑可转移对象。
90. 什么是结构化克隆？它和普通浅拷贝、深拷贝有什么区别？
答：结构化克隆是浏览器内置的一套对象复制规则，专门用来在线程、窗口或存储场景里安全传递数据。它不像浅拷贝那样只复制第一层引用，也不是随手写的深拷贝那样只按业务习惯递归，它有自己支持和不支持的数据类型边界，所以理解它时要抓住一点：这不是简单“复制得更深”，而是一套有明确规范的跨上下文数据传递规则。
91. 什么是可转移对象？为什么 `ArrayBuffer` 适合在线程间传递？
答：可转移对象的核心特点是可以把底层资源的所有权直接交给另一个线程，而不是先完整复制一份再发送，所以它特别适合大数据量场景。`ArrayBuffer` 之所以常被拿来举例，就是因为二进制数据块很适合这种“把所有权转过去”的方式，既减少复制开销，也更适合图片处理、文件解析、音视频等对性能比较敏感的任务。
92. 如果主线程连续传两条消息给 Worker，Worker 如何区分和回传？
答：最稳的做法，是给每条消息设计统一协议，比如带上 `type`、`id` 和 `payload`，Worker 处理完成后也按同样结构把 `id` 原样带回。这样主线程就能明确知道这条结果对应的是哪次任务，而不是去依赖返回顺序做猜测；一旦进入并发任务、批量文件处理或多种命令混合分发场景，显式 `id` 几乎就是必需品。
93. 主线程和 Worker 之间会有多条消息吗？如何设计消息协议？
答：真实项目里当然不可能只有一两条消息，所以协议最好从一开始就统一成固定结构，比如请求类型、任务 id、载荷、时间戳、成功失败标记这些字段都规范好。这样后续无论是增加任务类型、做并发控制、排查错误还是加超时重试，都不会因为消息格式混乱而迅速失控。
94. Worker 返回给主线程的数据结构通常应该如何设计？
答：我通常会把 Worker 的返回值统一成比较稳定的协议，比如包含 `id`、`success`、`data`、`error`、必要时再加 `progress` 或 `type`，这样主线程更容易做结果分流、错误展示和并发任务管理。协议统一的意义在于让 Worker 不只是“会算”，还要“好接入、好排错、好扩展”，否则不同 Worker 各发一套格式，后面维护成本会非常高。
95. Web Worker 能直接访问 DOM 吗？为什么不能？
答：不能，因为 DOM 属于主线程负责的渲染上下文，是页面结构和视觉状态的核心共享资源，如果后台线程也能直接改它，就会把同步、一致性和线程安全问题迅速放大。Worker 的设计目标，就是把重计算从 UI 线程里拆出去，而不是让多个线程同时争抢页面结构，所以它天然和 DOM 隔着一层清晰边界。
96. Worker 里能使用哪些浏览器能力，不能使用哪些能力？
答：Worker 里通常可以使用定时器、网络请求、消息通信、二进制数据处理等偏计算和通信类能力，但不能直接操作 DOM、不能像主线程那样使用大部分界面相关 API，也没有完整的 `window` 语义。它更适合做后台计算、数据解析、格式转换，不适合做依赖页面状态和布局测量的交互控制，所以选不选 Worker，关键看任务是否真的能和 UI 解耦。
97. 什么情况下你会选择 Worker，而不是分片调度或 `requestIdleCallback`？
答：如果任务真的很重、持续时间长、而且和 DOM 没强耦合，我会优先考虑 Worker，因为这代表它值得被彻底搬离主线程；如果任务只是一些零碎小工作，或者虽然多但可以拆成很多短片慢慢做，那分片调度和 `requestIdleCallback` 往往更轻量。简单说，Worker 更适合“重任务迁移”，分片更适合“主线程内温和拆解”。
98. 大文件处理、图片压缩、CSV 解析为什么适合放到 Worker 里？
答：因为这些任务本质上都是和 UI 弱耦合、但对 CPU 很不友好的重计算，如果继续放在主线程里，就会直接抢走渲染和交互时间，让页面在处理过程中明显变卡。把它们放进 Worker 后，用户仍然能继续滚动、点击和输入，代价只是你需要额外设计消息通信、错误回传和数据传输策略，但这通常比让整个页面失去响应划算得多。
99. 如何避免 Worker 越开越多导致资源浪费？
答：避免 Worker 失控的关键，是控制数量、能复用就复用、任务结束及时释放，并在有需要时做简单线程池或任务队列，而不是每个功能点都随手新开一个 Worker。因为 Worker 本身也要占内存和调度成本，线程不是越多越快，真正成熟的做法是让 Worker 成为可管理资源，而不是散落在各个模块里的隐形开销。
100. Web Worker、Service Worker、Shared Worker 的区别分别是什么？
答：Web Worker 主要解决后台计算问题，目标是把重 CPU 任务移出主线程；Service Worker 更偏向离线缓存、请求拦截、后台同步这类网络代理和应用外壳能力；Shared Worker 则强调多个页面或标签页共享同一个 Worker 上下文。三者名字很像，但职责完全不同，所以最不容易讲混的方法，就是直接按“计算、网络代理、多页面共享”三条线去区分。
