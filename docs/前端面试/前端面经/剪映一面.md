# 剪映一面参考回答

## 一、核心项目深挖

### （一）AI Agent 系统项目

#### 请详细介绍一下这个系统项目，包括项目背景、你的核心职责、落地成果。

一个面向复杂任务处理的多 Agent 系统，目标是把单轮问答升级成“能拆任务、会调用工具、能沉淀经验”的执行型 AI 系统。项目背景是，单 Agent 在处理复杂任务时容易出现上下文过载、工具选择不稳定、长链路成功率不高的问题，所以我们做了主从协同架构，把规划、执行、结果聚合拆开。

我主要负责三块：

1. 前端侧的任务编排与执行可视化，包括任务状态流转、步骤展示、流式输出和错误回溯。
2. Agent orchestration 相关能力的接入，包括主 Agent 调度、从 Agent 生命周期管理、工具调用结果回传。
3. 效果优化，重点做了 Memory 复用、重复探索控制、上下文裁剪和执行链路可观测性。

落地效果上，这个系统把复杂任务的成功率和稳定性明显拉高了。你在面试里可以这样表达：`复杂任务完成率提升`、`平均执行耗时下降`、`重复调用次数下降`、`token 成本得到控制`。如果面试官追问指标，你再补真实数据。

#### 请解释系统的主从 Agent 架构设计，主 Agent 和从 Agent 分别承担什么职责？

我的理解是，主 Agent 更像 orchestrator，从 Agent 更像 specialist。

主 Agent 主要负责：

1. 理解用户目标，判断任务复杂度。
2. 做任务拆解和执行规划。
3. 选择合适的从 Agent 和工具。
4. 管理全局上下文、预算、重试和终止条件。
5. 聚合各从 Agent 的结果，生成最终答案。

从 Agent 主要负责某个垂直子问题的深度执行，比如：

1. 代码检索 Agent：负责代码定位、依赖分析、实现路径确认。
2. 文档检索 Agent：负责知识库、规范、接口文档检索。
3. Browser Agent：负责页面操作、截图、表单验证。
4. Data Agent：负责结构化数据查询、分析和结果整理。

这样做的好处是把“全局决策”和“局部执行”解耦。主 Agent 不需要把所有细节都装进一个超长上下文里，从 Agent 也能在较小上下文中更专注地完成任务。

#### 从 Agent 的数量是怎么确定的？如何解决多个从 Agent 重复探索的问题？

从 Agent 数量不是越多越好，而是看任务域是否真的存在“能力边界”。我的划分原则有三个：

1. 能力是否异构，比如代码检索、浏览器操作、知识检索本身就是不同能力。
2. 上下文是否应隔离，如果混在一起会互相污染，就值得拆。
3. 是否能带来稳定收益，如果拆出来后成功率没提升，反而增加调度成本，那就不该拆。

解决重复探索，我一般会做四层控制：

1. **任务去重**：主 Agent 在分发前给每个子任务生成结构化 goal，判断语义是否重复。
2. **结果缓存**：同一轮任务里，相同 query 或相近 query 的检索结果直接复用。
3. **工作区隔离**：不同从 Agent 拿到不同的子问题和上下文边界，避免都从头搜一遍。
4. **中间结论回传**：从 Agent 在拿到关键结论后及时上报，主 Agent 根据最新进度动态裁剪后续任务。

所以本质上不是“禁止重叠”，而是“允许少量冗余换稳定性，但不能无控制地重复消耗 token 和时间”。

#### 请解释 MCP 和 Tool Calling 的区别是什么？

一句话讲，**Tool Calling 不是 MCP 的“子集”这种严格从属关系，但在真实系统里，MCP 暴露出来的 tool 往往会被模型通过 Tool Calling 机制使用。**

更准确地说：

1. **Tool Calling** 是大模型推理阶段的一种能力，关注的是“模型能不能在生成过程中决定调用哪个函数、传什么参数、拿回什么结构化结果，再继续推理”。
2. **MCP** 是一个标准化协议层，关注的是“外部能力怎么被统一描述、发现、调用和返回结果”。它不只包含 `tool`，还可以包含 `resource`、`prompt` 等。

所以两者的关系不是：

1. 不是“Tool Calling = MCP 的一部分”的严格定义。
2. 更像是“模型侧的调用机制”与“系统侧的能力接入标准”。

你可以这样理解一条完整链路：

1. 外部系统通过 MCP 把能力暴露出来，比如某个搜索工具、数据库工具、浏览器工具。
2. 客户端或编排层把这些工具描述转换成模型可理解的 schema。
3. 模型在推理时通过 Tool Calling 决定要不要调用这些工具。
4. 工具执行后再把结果回传给模型继续生成。

所以更严谨的表达是：**MCP 负责把能力标准化接进来，Tool Calling 负责让模型在推理时真正把这些能力用起来。没有 MCP 也能做 Tool Calling，比如直接在服务端手写函数 schema；但用了 MCP 后，多工具接入、权限治理、跨平台复用会更好。**

#### Skill 和固定的 Workflow 有什么区别？你们是怎么选择和平衡的？

我会把 Skill 理解成“可复用能力模块”，把 Workflow 理解成“确定性执行流程”。

Skill 适合处理开放性任务，比如“分析一个页面性能问题”或者“阅读代码后给出修改建议”，因为这类任务路径不固定，但会重复用到某些能力。

Workflow 适合处理强约束任务，比如“拉取数据 -> 清洗 -> 生成报告 -> 发通知”这种稳定流程，因为每一步都比较确定。

我们的平衡方式是：

1. **高波动、高探索任务** 优先 Skill 化。
2. **高频、低变化、强 SLA 任务** 优先 Workflow 化。
3. 主流程用 Workflow 保稳定，关键环节插入 Skill 保灵活。

所以不是二选一，而是把 Workflow 作为骨架，把 Skill 作为能力插件。

#### 你们是怎么实现过往 Agent 经验的复用和 Memory 优化的？

我们做的是分层 Memory，而不是把所有历史对话全量塞回上下文。真正落地时，我会把它拆成“存什么、存在哪、怎么写入、怎么检索、怎么淘汰”五件事。

一般分成三层：

1. **短期 Memory**：当前任务内的中间结论、工具结果、失败原因。
2. **情景 Memory**：某一类任务的成功路径，比如“代码定位类任务通常先查路由、再查组件、再查接口”。
3. **长期知识 Memory**：稳定规则、术语解释、领域知识、历史最佳实践。

### 1. 具体怎么存

**短期 Memory**

1. 通常按 `sessionId / taskId` 维度存。
2. 适合放在 `Redis` 或进程内缓存里，因为它读写频繁、时效短、需要低延迟。
3. 数据结构一般是结构化日志或事件流，比如：
   - `taskId`
   - `stepId`
   - `toolName`
   - `input`
   - `outputSummary`
   - `status`
   - `error`
   - `timestamp`

**情景 Memory**

1. 本质上是“某类任务的经验条目”。
2. 通常会拆成两部分存：
   - 文本摘要和 embedding 存进向量库，用于语义召回。
   - 元数据存进 `PostgreSQL` 或文档库，用于过滤和治理。
3. 元数据一般包括：
   - `memoryId`
   - `taskType`
   - `domain`
   - `summary`
   - `embedding`
   - `successScore`
   - `sourceVersion`
   - `createdAt`
   - `expiredAt`

**长期知识 Memory**

1. 更像知识库或规则库。
2. 原始内容一般放对象存储、文档库或关系型数据库。
3. 为了支持语义检索，会额外切 chunk、生成 embedding、建立向量索引。

### 2. 具体怎么管理

我会做四层管理：

1. **写入门槛**
   不是每次执行结果都写进去。只有高置信、可复用、不是一次性偶然路径的信息才入库。

2. **去重与合并**
   对相似经验做归并，避免向量库里出现很多语义差不多但表述不同的脏数据。

3. **版本和时效**
   经验条目要带 `codeVersion / toolVersion / promptVersion`。版本变化大时，旧经验权重自动降低，必要时直接失效。

4. **反馈回写**
   如果某条 Memory 被命中后明显误导任务，要降低它的置信度，严重时直接下线。

### 3. 具体怎么使用

真正执行时不是“先把所有 Memory 全塞进 prompt”，而是一个检索增强流程：

1. 先根据当前任务生成检索 query。
2. 按任务类型、领域、版本先做 metadata 过滤。
3. 再做向量检索召回候选 Memory。
4. 对召回结果做排序，只保留 topN。
5. 把它们压缩成简短可执行提示注入上下文，比如：
   - 过去类似任务优先查哪里
   - 哪种工具组合成功率更高
   - 哪些失败模式要避开

### 4. 底层为什么这样做

因为 Memory 最大的问题不是“没有记忆”，而是“把不该记的东西记住了”。所以它本质上是一个带筛选、检索、衰减机制的经验系统，而不是简单聊天记录备份。

优化点主要有四个：

1. **写入门槛**：不是每次结果都写入，只沉淀高置信、可复用的信息。
2. **检索优先级**：先检索和当前任务最相关的 Memory，再做注入。
3. **摘要压缩**：把长链路历史压缩成可复用结论，避免上下文膨胀。
4. **版本和时效管理**：过期知识要衰减，强依赖外部环境的经验要带版本标签。

这样做的本质是让 Agent“记住有效经验”，而不是“背下所有历史”。

### （二）Next.js 电商项目

#### 请详细介绍一下这个电商项目，包括你的核心工作、遇到的技术难点和解决方案。

这是一个基于 Next.js 的电商项目，核心目标是兼顾 SEO、首屏性能和交易链路体验。业务上覆盖首页、商品列表、商品详情、搜索、购物车、下单等核心流程，同时接入了智能商品查询助手做导购增强。

我主要负责三部分：

1. 基于 Next.js 做 SSR/SSG 混合渲染方案设计，优化首屏和 SEO。
2. 负责商品列表、详情、搜索等核心页面开发，以及状态管理、缓存和接口层封装。
3. 参与智能导购助手接入，包括前端对话 UI、流式响应展示、检索结果卡片渲染。

我认为项目里比较典型的技术难点有两个：

1. **首屏性能和 SEO 的平衡**：商品详情、活动页需要被搜索引擎收录，所以不能纯 CSR；但如果全部 SSR，服务端压力和响应时间都会变高。我的做法是按页面类型拆分：营销页和详情页优先 SSR/SSG，强交互后台页走 CSR。
2. **数据一致性和缓存策略**：电商场景价格、库存、促销经常变化，静态化过强会有脏数据风险。我的做法是把“可缓存内容”和“强实时内容”拆层，像商品基础信息可以缓存，库存和价格走实时接口，并结合 `revalidate`、`CDN` 和接口缓存控制更新频率。

这里可以展开得更技术一点：

1. **`revalidate` 是什么**
   它本质上是“页面或数据缓存多久后需要重新验证/重新生成”的策略。在 `Next.js` 里，常见理解是 ISR（Incremental Static Regeneration）的一部分。页面不是每次请求都重新渲染，而是先把静态结果缓存起来；当缓存超过 `revalidate` 时间后，下一次请求触发后台重新生成，新结果再替换旧缓存。

2. **`CDN` 是什么**
   `CDN` 是内容分发网络，会把静态资源或缓存后的页面分发到离用户更近的边缘节点。用户请求先到边缘节点，如果节点上有缓存，就直接返回，不必每次都回源站，所以能显著降低延迟和源站压力。

3. **接口缓存是什么**
   是对 API 响应做缓存，既可以是浏览器缓存，也可以是 `CDN` 层缓存、网关层缓存、服务端内存缓存、`Redis` 缓存。比如商品详情里的“静态描述”可以缓存 5 分钟，而库存接口可能只缓存几秒甚至不缓存。

4. **这三者怎么配合**
   - 页面层：用 `revalidate` 控制 HTML 何时再生成。
   - 静态资源层：用 `CDN` 缓存 JS、CSS、图片，减少回源。
   - 数据层：用接口缓存控制后端查询压力。

5. **为什么这样能同时保证性能和一致性**
   因为不是所有数据都要求同等实时性。商品标题、主图、详情文案变化频率低，适合缓存；库存、价格、优惠状态变化快，更适合走实时接口。这样把“慢变数据”和“快变数据”拆层，就能在性能和一致性之间取得平衡。

最后带来的收益通常可以从三点讲：搜索收录更稳定、首屏体验更好、业务转化链路更顺畅。

## 二、前端基础能力考察

#### 请解释 CSR、SSR、SSG 分别是什么，核心区别、优缺点和适用场景是什么？

1. **CSR（Client-Side Rendering）**
   页面初始只返回一个壳，主要渲染工作在浏览器完成。
   优点是前后端分离彻底、交互灵活。
   缺点是首屏慢、SEO 不友好。
   适合后台系统、强交互应用。

2. **SSR（Server-Side Rendering）**
   请求到来时由服务端实时生成 HTML 返回给浏览器。
   优点是首屏更快、SEO 更好。
   缺点是服务端压力大、部署和缓存策略更复杂。
   适合商品详情、内容页、营销页。

3. **SSG（Static Site Generation）**
   在构建阶段提前生成静态 HTML。
   优点是性能最好、成本低、天然适合 CDN。
   缺点是数据更新不够实时。
   适合博客、帮助中心、活动专题等低实时页面。

核心区别就是：**渲染发生在什么时候，以及页面内容在返回给浏览器之前是否已经准备好。**

#### SSR 和 CSR 相比，哪个对企业的维护成本更高？为什么？

通常是 **SSR 的维护成本更高**。

原因主要有四点：

1. 服务端渲染会引入 Node 运行环境、服务端部署、监控和扩缩容问题。
2. 要处理浏览器和服务端双端执行差异，比如 `window`、`document` 不可用。
3. 缓存策略更复杂，要同时考虑页面缓存、数据缓存和 CDN 缓存。
4. 排障难度更高，问题可能出在服务端渲染、客户端 hydration 或接口时序上。

但如果业务强依赖 SEO 和首屏体验，SSR 的额外成本通常是值得的。

## 三、计算机网络 & 基础考察

#### 请详细讲一下 HTTPS 的 TLS 握手过程，TLS1.2 和 TLS1.3 的核心区别是什么？

如果要讲得更底层，我会按“为什么要握手 -> 握手阶段发生了什么 -> 最终如何进入加密通信”这个顺序来讲。

### 1. TLS 为什么需要握手

HTTP 本身是明文协议，直接传输会有三个问题：

1. 数据可能被窃听。
2. 数据可能被篡改。
3. 客户端不知道自己连的是不是真服务器。

所以 TLS 握手要解决三件事：

1. 确认双方接下来用什么加密套件。
2. 完成服务端身份认证。
3. 安全地产生一个双方都知道、别人不知道的会话密钥。

### 2. TLS 1.2 握手流程

#### 第一步：`ClientHello`

客户端发 `ClientHello`，里面会带上：

1. 支持的 TLS 版本。
2. 支持的加密套件列表。
3. 一个客户端随机数 `client_random`。
4. 支持的扩展能力，比如 `SNI`、`ALPN`。

这里的本质是：客户端先告诉服务端“我支持哪些协议和算法”。

#### 第二步：`ServerHello`

服务端返回 `ServerHello`，里面会带上：

1. 选中的 TLS 版本。
2. 选中的加密套件。
3. 一个服务端随机数 `server_random`。
4. 服务端证书。
5. 如果走 ECDHE，还会带上临时密钥交换参数。

这里的本质是：服务端从客户端给出的能力集合里挑一个双方都支持的方案。

#### 第三步：客户端验证证书

客户端会验证：

1. 证书域名是否和访问域名匹配。
2. 证书是否过期。
3. 证书链是否能追溯到受信任的根 CA。
4. 证书签名是否正确。

底层逻辑是：CA 用自己的私钥给服务端证书做签名，客户端本地内置了受信任 CA 的公钥，所以可以验证这个签名是否成立。

#### 第四步：密钥交换

这里要分场景：

1. **老式 RSA 密钥交换**
   客户端生成 `pre-master secret`，用证书中的公钥加密发给服务端，服务端用私钥解密。

2. **更主流的 ECDHE**
   双方交换各自的临时公钥参数，分别计算出同一个共享秘密。真正的私钥从头到尾都不会在网络上传输。

现在更推荐讲 ECDHE，因为这才是现代 TLS 的主路径。

### 证书中的公钥是什么？每个证书都自带吗？
**证书里本来就包含“证书持有者的公钥”**，这也是数字证书最核心的信息之一。

一张服务端证书里通常会包含：

1. 证书主体信息，比如域名、组织。
2. 服务端的公钥。
3. 证书有效期。
4. 签发者信息。
5. CA 对这张证书做的数字签名。

所以当我们说“客户端拿到证书中的公钥”时，指的是：服务端在 TLS 握手中把自己的证书发给客户端，客户端从证书内容里读出这个公钥。

这里要区分两把不同用途的公钥：

1. **证书里的服务端公钥**
   用来证明“这张证书绑定的是这个服务端身份”，在老式 RSA 密钥交换里它还会被拿来加密 `pre-master secret`。

2. **CA 的公钥**
   客户端本地信任库里预装，用来验证“这张证书上的签名是不是真的由可信 CA 签出来的”。

### 这里的私钥是哪个私钥？是生成证书时用的私钥吗？

这里说的“服务端用私钥解密”，指的是：**和证书里的服务端公钥配对的那把服务端私钥**。

它通常是服务端自己先生成一对密钥：

1. 服务端生成 `private key` 和 `public key`。
2. 把 `public key` 连同主体信息做成 CSR（证书签名请求）。
3. CA 审核后，把这个 `public key` 和主体信息签成正式证书。

### ECDHE 整个算法过程是什么？

如果要讲底层一点，可以按“椭圆曲线 Diffie-Hellman 密钥交换”来理解。它的目标不是加密一段现成数据，而是让客户端和服务端在不暴露秘密的前提下，分别算出同一个共享秘密。

可以简化成下面这个过程：

1. **双方先约定公共参数**
   比如使用哪条椭圆曲线，这些参数本身不是秘密。

2. **服务端生成临时私钥和临时公钥**
   - 临时私钥：一个随机大数 `b`
   - 临时公钥：`B = bG`
   这里的 `G` 是曲线上的基点，`bG` 表示椭圆曲线上的标量乘法。

3. **客户端也生成临时私钥和临时公钥**
   - 临时私钥：随机大数 `a`
   - 临时公钥：`A = aG`

4. **双方交换临时公钥**
   - 客户端拿到服务端的 `B`
   - 服务端拿到客户端的 `A`

5. **双方各自计算共享秘密**
   - 客户端算：`S = aB = a(bG) = abG`
   - 服务端算：`S = bA = b(aG) = abG`

6. **双方得到同一个共享秘密**
   因为椭圆曲线标量乘法满足这个交换过程，所以两边最后得到的都是同一个 `abG`。

7. **再把这个共享秘密送进密钥派生函数**
   结合 `client_random`、`server_random`，派生出真正用于加密通信的会话密钥。

为什么攻击者拿到 `A` 和 `B` 也算不出 `abG`？

因为攻击者只看到了：

1. 公共参数
2. `A = aG`
3. `B = bG`

但看不到 `a` 和 `b`。想从 `A` 反推出 `a`，本质上要解椭圆曲线离散对数问题，这在计算上非常困难，所以共享秘密能保住。

### 为什么说 ECDHE 具备前向安全？

因为它每次握手都会生成新的临时私钥 `a`、`b`。即使未来某一天服务端长期证书私钥泄漏，攻击者也无法反推出过去每次会话里的临时私钥，因此也无法解密以前抓包得到的历史流量。

#### 第五步：派生会话密钥

双方拿到相同的密钥材料后，会结合：

1. `client_random`
2. `server_random`
3. 共享秘密

通过密钥派生函数生成后续通信真正使用的对称密钥。

为什么最后要用对称加密？

因为非对称加密虽然适合做身份认证和密钥交换，但太慢；真正大量传输业务数据时，必须用更高效的对称加密算法，比如 `AES-GCM`。

### 对称加密是什么？

对称加密的意思是：**加密和解密使用的是同一把密钥，或者可以等价互推的一把密钥。**

例如：

1. 客户端和服务端都拿到同一个会话密钥 `K`。
2. 客户端用 `K` 把明文加密成密文发出去。
3. 服务端收到后也用同一个 `K` 解密还原明文。

它和非对称加密的区别是：

1. **非对称加密**
   有一对密钥，公钥和私钥分开，适合做身份认证、密钥交换。

2. **对称加密**
   双方共享同一个秘密，适合大量、高速传输业务数据。

为什么 TLS 最终都落到对称加密？

1. HTTP 正文、图片、JSON、视频片段这些数据量很大。
2. 如果全部用非对称加密，CPU 开销太高，吞吐太差。
3. 对称加密算法比如 `AES-GCM`、`ChaCha20-Poly1305` 在现代 CPU 上速度快很多，还能同时提供机密性和完整性保护。

所以 TLS 的典型思路是：

1. 前面用非对称体系解决“你是谁”“我们怎么安全拿到同一把密钥”。
2. 后面用对称加密解决“我们怎么高效传大量数据”。

#### 第六步：`Finished`

双方各自发送 `Finished` 消息。这个消息本身已经是用刚协商出的密钥加密的，用来证明：

1. 双方确实拿到了同一套密钥。
2. 握手过程没有被篡改。

到这一步之后，HTTP 数据才会进入 TLS Record 层，被分片、加密、加认证标签后再通过 TCP 发送出去。

### 3. TLS 1.2 和 TLS 1.3 的核心区别

1. **握手轮次更少**
   TLS 1.2 常见是 2-RTT；TLS 1.3 典型可以做到 1-RTT，恢复会话时甚至支持 0-RTT。

2. **去掉了很多老旧算法**
   TLS 1.3 移除了不安全或过时的算法组合，比如静态 RSA 密钥交换，整体安全基线更高。

3. **默认更强调前向安全**
   TLS 1.3 主路径依赖 `(EC)DHE`。即使未来服务端长期私钥泄漏，也不能反推出以前抓到的会话内容，这就是前向安全。

4. **握手消息更简化**
   很多历史兼容逻辑被裁掉，协议更干净，攻击面更小。

#### HTTPS 是怎么防止中间人攻击的？

HTTPS 防中间人攻击的关键不是“只有加密”，而是“可信身份认证 + 安全密钥协商 + 完整性保护”。

如果没有证书体系，会出现什么问题？

1. 攻击者自己和客户端建立一条加密链路。
2. 再自己和真实服务端建立另一条加密链路。
3. 中间人两头转发、两头解密，客户端根本不知道自己连错了对象。

HTTPS 之所以能防住，核心就在于：

1. 服务端必须拿出一张由可信 CA 签发的证书。
2. 客户端会用本地信任根去验证这张证书是不是合法。
3. 攻击者没有目标网站对应域名的合法证书，也没有对应私钥，所以无法在证书校验阶段通过。
4. 后续 TLS 握手里还有握手消息校验和 Record 层完整性校验，防止中途被悄悄改包。

所以真正挡住中间人攻击的，不是“别人看不懂密文”这么简单，而是“别人没法假扮成这个网站完成可信握手”。

#### 什么是超卖问题？根本原因是什么？有哪些解决方案？

超卖就是库存只剩 1 件，但并发请求下卖出了多件。

根本原因是：**库存检查和扣减不是原子操作，多个请求同时读到旧库存。**

常见解决方案：

1. 数据库乐观锁，比如版本号或 `where stock > 0` 条件更新。
2. 悲观锁，先锁库存记录再扣减。
3. Redis 预扣库存，后面异步落库。
4. 消息队列串行化下单或扣库存流程。
5. 前端限流、防重复提交只能缓解，不能从根上解决。

### 1. 数据库乐观锁

常见写法有两种：

1. 版本号方式：
   `update goods set stock = stock - 1, version = version + 1 where id = ? and version = ?`
2. 库存条件方式：
   `update goods set stock = stock - 1 where id = ? and stock > 0`

底层原理是：把“检查库存”和“扣库存”合并成一条原子更新语句，让数据库在执行这一条 SQL 时完成并发竞争。成功更新说明抢到了库存，更新行数为 0 说明没抢到。

### 2. 悲观锁

典型做法是事务里先查：

`select * from goods where id = ? for update`

然后再扣减库存并提交事务。

底层原理是：数据库先把这行数据锁住，其他事务在锁释放前不能同时修改它，因此不会出现多个事务同时读到同一份旧库存。

1. **锁是什么**
   锁本质上是数据库并发控制的一种机制。因为数据库要同时处理很多事务，如果多个事务同时读写同一份数据，就可能出现脏读、不可重复读、丢失更新这类问题，所以数据库内部会维护锁状态，控制哪些事务此刻可以读、可以写、必须等待。

2. **为什么能锁住这一行数据**
   因为像 `MySQL InnoDB`、`PostgreSQL` 这类数据库本身就实现了行级锁机制。执行 `select ... for update` 时，数据库不是简单“查一行数据出来”，而是在事务上下文里给命中的记录加上排它性控制。后续其他事务如果也想更新这行，或者也想对这行加同类排它锁，就会被阻塞，直到当前事务提交或回滚。

3. **`for update` 到底做了什么**
   它的语义可以理解成：“我现在不只是读这行，我接下来准备改它，所以你先帮我把这行保护起来。”  
   于是数据库会在当前事务持有期间，让这行记录处于一种“当前事务可安全修改，其他冲突事务需要等待”的状态。

4. **这是数据库自身的能力机制吗**
   是的，这就是数据库原生提供的事务和锁能力，不是应用层自己手写出来的。应用层只是通过 SQL 语法触发这套机制，比如：
   - `begin`
   - `select ... for update`
   - `update ...`
   - `commit`

5. **为什么这样能解决超卖**
   因为多个事务不会再同时基于同一份旧库存做扣减。第一个事务锁住行后，其他事务必须等它提交；等锁释放后，后面的事务读到的已经是最新库存，而不是旧库存。

6. **补一句更底层的理解**
   锁不是把“Java 里的对象”锁住，也不是把“整张表”一定锁住，而是数据库存储引擎在内部对记录访问权限做控制。至于到底是行锁、页锁还是表锁，要看数据库实现、索引命中情况和具体 SQL 写法。

### 3. Redis 预扣库存

常见实现是：

1. 把可售库存先放进 Redis。
2. 用户下单时先在 Redis 中原子扣减，比如用 `DECR` 或 Lua 脚本。
3. Redis 扣减成功后再异步写数据库。

底层原理是：Redis 单线程执行命令，同一个 key 的扣减操作天然串行，因此高并发下原子性很好，而且比数据库扛并发更轻。

但它会引入新的问题：Redis 成功但数据库失败怎么办，所以通常要结合消息队列、重试和对账机制。

### 4. 消息队列串行化

常见做法是：

1. 下单请求先进队列。
2. 消费者按顺序消费消息，逐个扣库存。

底层原理是：把高并发同时修改共享资源的问题，转成队列里的顺序处理问题，用时间换一致性。

它适合秒杀这类高峰场景，但缺点是链路更长，用户可能要接受“排队中”或“处理中”的状态。

### 5. 前端限流、防重复提交

1. 点击下单按钮后立刻置灰。
2. 一个请求未完成前禁止重复点击。
3. 页面层做节流、防抖或幂等 token。

底层原理只是减少重复请求数量，降低后端压力，但它解决不了多个不同用户同时抢同一件商品的问题，所以不能作为最终一致性方案。

## 四、AI & RAG 技术考察

#### 请介绍一下电商项目里的智能商品查询助手，包括架构、遇到的问题和优化手段。

底层地讲，它是一条“用户自然语言 -> 检索真实商品 -> 排序 -> 交给大模型解释”的链路。

整体实现链路可以这样拆：

1. 用户输入 query，比如“适合出差、轻薄、续航长的笔记本”。
2. 前端把 query 发给服务端。
3. 服务端做 query 预处理，包括分词、同义词归一化、意图识别、属性抽取。
4. 把处理后的 query 转成向量，去向量库里做相似度检索。
5. 同时结合价格区间、品牌、类目、库存状态等条件过滤，缩小候选集合。
6. 对召回出来的候选商品做 rerank，提升前几条结果质量。
7. 把 top 商品的标题、卖点、参数、价格等结构化信息拼成 prompt 交给大模型。
8. 大模型基于这些真实商品信息生成解释型回答。
9. 前端流式展示文本，同时渲染商品卡片。

### 1. 向量检索具体怎么做

向量检索的核心是把“文本相似”问题转成“向量距离”问题。

具体流程：

1. 离线阶段先把商品标题或商品文本喂给 embedding model。
2. 模型把每个商品转成一个高维向量，比如 768 维或 1536 维。
3. 把这些向量连同商品 `id`、类目、品牌等 metadata 一起存进向量库。
4. 查询时把用户 query 也转成向量。
5. 在向量库里按相似度检索 topK 候选，比如余弦相似度或点积。

底层原理是：语义相近的文本，在向量空间里通常距离更近。比如“轻薄本”和“便携笔记本”字面不完全一样，但向量可能比较接近。

#### Embedding Model 到底是什么？它和 Token、LLM 是什么关系？

这个点面试里很容易被混淆。**Embedding model 不是“专门把文本转成 token 的模型”**，它的核心目标是：**把文本映射成一个能够表示语义的稠密向量。**

可以把链路拆开理解：

1. 原始文本先进入 tokenizer。
2. tokenizer 把文本切成 token。
3. token 再进入 embedding model。
4. embedding model 输出的不是下一个 token 概率，而是一整个文本片段的向量表示。

所以：

1. **tokenizer**
   负责把文本切成模型可处理的 token 序列。

2. **embedding model**
   负责把这个 token 序列编码成语义向量，主要用于检索、聚类、相似度计算、召回。

3. **LLM**
   负责基于 token 序列做上下文建模，并预测下一个 token，主要用于生成。

也就是说，`embedding model` 和 `LLM` 都会处理 token，但它们的任务完全不同：

1. embedding model 的目标是“得到一个向量表示”。
2. LLM 的目标是“继续生成后续 token”。

#### 业务处理长文本时，是不是也都要靠 embedding model 把长文本转成 token？

不是。更准确地说，**长文本进入模型前通常都要先经过 tokenizer 切 token，但不一定都要经过 embedding model。**

分两种典型场景：

1. **RAG / 检索场景**
   这里会把文档切 chunk，然后用 embedding model 把每个 chunk 转成向量，存进向量库。查询时再把 query 也转成向量做相似度检索。

2. **LLM 直接生成场景**
   这里文本会先被 tokenizer 切成 token，然后直接进入大模型做推理，不一定会单独经过一个“用于检索的 embedding model”。

1. **切 token** 是模型输入处理的一部分。
2. **生成 embedding 向量** 是为了做语义表示和检索，不是所有文本处理链路都必须有这一步。

### 2. 条件过滤具体怎么做

条件过滤本质上是 metadata filtering。

比如用户 query 里提到：

1. 价格 5000-7000
2. 品类是笔记本
3. 品牌偏向联想
4. 必须有货

这些条件不适合只靠向量判断，因为向量擅长语义相似，不擅长做严格的“必须满足”约束。所以一般做法是：

1. 先通过规则、NER、意图识别把这些结构化条件抽出来。
2. 检索时附带过滤表达式，比如：
   - `category = laptop`
   - `price between 5000 and 7000`
   - `stock > 0`
3. 让向量检索只在满足这些 metadata 的候选里进行，或者召回后再过滤。

底层原理是：把“语义相关”和“业务硬约束”分开处理。语义相关交给 embedding，硬约束交给结构化过滤。

1. **先做规则抽取**
   对一些非常明确的模式，直接用规则或正则就够了，比如：
   - `5000-7000`
   - `五千到七千`
   - `必须有货`
   - `不要苹果`

   这些可以直接映射成结构化字段：
   - `priceMin = 5000`
   - `priceMax = 7000`
   - `stock = true`
   - `excludeBrand = Apple`

2. **再做 NER（命名实体识别）**
   NER 的目标是从自然语言里识别出“这段词属于什么实体”，例如：
   - `联想` -> 品牌
   - `笔记本` -> 类目
   - `轻薄` -> 卖点/标签
   - `出差` -> 使用场景

   它的实现可以是：
   - 小模型做序列标注
   - 规则词典匹配
   - LLM 提取后再做结构化校验

3. **再做意图识别**
   意图识别解决的是“用户到底想干什么”，比如：
   - 是想找商品
   - 还是想比价
   - 还是想问库存
   - 还是想问某个型号参数

   如果识别出是“找商品”，后面就走召回链路；如果识别出是“问库存”，就可以直接转结构化查询，而不是走完整 RAG。

4. **最后统一映射成过滤条件**
   前面抽出来的实体和意图，不会直接拿去做向量检索，而是会整理成统一 filter 对象，比如：

```js
{
  category: "laptop",
  brand: "Lenovo",
  priceMin: 5000,
  priceMax: 7000,
  inStock: true,
  tags: ["lightweight", "long-battery-life"],
  scene: "business-trip"
}
```

5. **检索时如何用**
   后续向量库或检索层拿到这个 filter 后，会做两件事：
   - 要么先按 metadata 过滤，再做向量检索
   - 要么先召回，再用这些条件做过滤和重排

所以这一整步的本质就是：**先把自然语言拆成“语义部分”和“硬条件部分”，再分别交给不同模块处理。**

### 3. Rerank 具体怎么做

Rerank 就是在召回后的候选集合上，再跑一个更重、更精细的相关性判断。

常见流程：

1. 向量检索先召回 top50 或 top100。
2. 用 rerank model 对 `(query, doc)` 成对打分。
3. 按 rerank score 重新排序。
4. 取前 topN 交给 LLM 生成。

底层原理是：

1. 向量召回为了快，通常走 ANN（近似最近邻）索引，重点是“先广泛捞出来”。
2. Rerank 模型更像精排模型，会更细粒度地看 query 和商品文本是否真正匹配当前需求。

所以向量检索更像粗筛，rerank 更像精筛。

“rerank model 又是什么”：

1. **它本质上也是模型**
   但它不是用来生成长文本答案的，而是用来判断“query 和某个候选文档到底有多相关”。

2. **它的输入是什么**
   输入通常是一对文本，也就是：
   - 用户 query
   - 某条候选商品文档 `doc`

   模型会把这两个文本放在一起看，而不是像向量检索那样分别编码后只比距离。

3. **它的输出是什么**
   一般输出一个相关性分数，比如 `0.92`、`0.67` 这种。分数越高，说明这条商品越适合当前 query。

4. **它为什么比向量召回更准**
   因为向量召回通常是“双塔思路”或相似向量匹配，强调快；而 rerank 更像“交叉编码器”思路，会让 query 和 doc 在模型内部做更细粒度的交互，所以更能判断：
   - 这个商品是不是只是词像
   - 还是真的满足用户当前表达的需求

5. **为什么不一开始就对全库做 rerank**
   因为太慢。假设商品库有 100 万条，不可能把 query 和 100 万条 doc 全部两两送进 rerank model 打分。  
   所以工程上通常是：
   - 先用向量检索快速召回 top50 / top100
   - 再用 rerank model 做精排

### 4. 为什么三者要一起用

1. **只用向量检索**
   容易把语义类似但业务不满足条件的商品召回进来。

2. **只用条件过滤**
   很难理解用户的口语化表达和模糊需求。

3. **只用大模型**
   成本高，而且容易幻觉。

所以工程上更稳的方案通常是：**向量检索负责语义召回，条件过滤负责业务约束，rerank 负责最终排序。**

遇到的典型问题有：

1. 召回不准，尤其是口语化表达和商品标题表达不一致时。
2. 大模型容易“说得像懂了”，但推荐结果不稳定。
3. 上下文太长时成本和延迟都会上升。

优化手段主要有：

1. 做 query 改写和标签归一化。
2. 用混合检索而不是只做向量检索。
3. 加 rerank 提升最终候选排序质量。
4. 控制上下文注入长度，只保留最相关商品。
5. 在前端展示“推荐原因 + 商品卡片”，增强可解释性和点击转化。

#### Rerank 的作用是什么？

Rerank 的作用是：**在召回阶段拿到一批候选结果后，再用更强的相关性模型做二次排序，把真正最相关的结果排到前面。**

它的具体机制通常是：

1. 把用户 query 和每个候选商品组成一对输入。
2. 送进 rerank model。
3. 模型输出一个相关性分数。
4. 按分数重新排序。

向量召回更偏“广撒网”，保证别漏；Rerank 更偏“精排序”，保证前几条更准。所以它解决的不是“有没有召回来”，而是“召回来之后排得准不准”。

#### 为什么不用大模型直接做商品推荐，而要选择 RAG 的方案？

因为商品推荐本质上依赖真实商品库，而不是只靠大模型参数记忆。

直接让大模型推荐有三个问题：

1. 它并不知道实时库存、价格、上下架状态。
2. 容易幻觉，可能编造不存在的商品。
3. 成本高，而且可控性差。

RAG 的优势是：

1. 先从真实商品库里检索，保证答案有依据。
2. 能接入实时数据，结果更可靠。
3. 可解释、可调优，工程上更适合业务系统。

## 六、补充高频追问

#### 介绍一下 SSR 和 SSG

SSR 是请求来了以后实时在服务端生成 HTML；SSG 是在构建阶段预先生成 HTML。SSR 更适合需要动态数据的页面，SSG 更适合内容稳定、追求极致性能的页面。

### 1. SSR 和 SSG 的原理是否一致

它们的**渲染原理本质相似**，都是先在服务器侧把组件树执行成 HTML，再把 HTML 发给浏览器；但**触发渲染的时机不同**：

1. **SSR**
   是“每次请求到来时”在服务端执行渲染。

2. **SSG**
   是“构建阶段”提前把页面渲染好，生成静态 HTML 文件。

所以两者不是完全不同的技术本体，而是“同样的服务端预渲染思路，在不同时间点执行”。

### 2. SSR 是不是需要不断在后端渲染并推送给前端展示

通常不是“持续推送”，而是：

1. 浏览器发起一次页面请求。
2. 服务端收到请求后执行 React 组件树。
3. 服务端把组件输出成 HTML。
4. 把 HTML 作为 HTTP 响应返回给浏览器。

这一步完成后，浏览器先看到的是一份已经生成好的 HTML。

然后浏览器还会继续做两件事：

1. 下载客户端 JS。
2. 执行 hydration，把静态 HTML 和 React 运行时绑定起来，让页面变得可交互。

所以 SSR 不是“后端不停渲染并持续推送”，常规页面场景下更准确地说是“每次请求来时服务端渲染一次并返回”。当然如果是 React 18 的流式 SSR，那么服务端可以把 HTML 分块 streaming 给浏览器，但本质仍然是一次请求内的服务端渲染输出。

### 3. SSG 是不是把 React 项目全部编译完给前端，后续计算都在浏览器完成

更准确的过程是：

1. 构建阶段，框架会执行页面代码，把每个静态页面提前生成成 HTML。
2. 同时也会生成前端 JS bundle。
3. 部署后，浏览器访问时直接拿到这份静态 HTML，而不是临时请求服务端渲染。
4. 浏览器拿到 HTML 后，同样还要下载 JS 并做 hydration，之后交互逻辑还是由浏览器运行。

所以 SSG 不是“只把源码编译完扔给前端”，而是“构建时已经把首屏 HTML 也算出来了”，浏览器后续主要负责接管交互。

### 4. 二者的关键差异

1. **SSR**
   - HTML 是请求时动态生成的。
   - 更适合数据实时性要求高的页面。
   - 服务端压力更大。

2. **SSG**
   - HTML 是构建时提前生成的。
   - 更适合内容稳定页面。
   - 性能和缓存友好度通常更高。

### 5. 浏览器后续是不是都自己算

两者都有一个共同点：**首屏 HTML 先由服务端或构建阶段准备好，后续交互阶段仍然需要浏览器运行 React 客户端代码。**

也就是说：

1. 预渲染负责“更快把内容展示出来”。
2. 浏览器 hydration 负责“把页面变成可交互应用”。

#### SSR 为什么能够减少渲染时间

更准确地说，SSR 不是减少“总计算量”，而是让用户更早看到可用内容。因为首屏 HTML 已经在服务端生成好了，浏览器拿到就能先展示，不需要等前端 JS 全部下载、解析、执行完再看到内容。

#### SSR 的缺点

服务端压力更大、缓存和部署更复杂、双端同构调试更困难、hydration 失败更难排查。

#### 介绍一下 Prisma

Prisma 是一个面向 TypeScript/Node.js 生态的 ORM 工具。它的特点是 schema 驱动、类型安全、开发体验好，能把数据库模型映射成类型友好的查询 API，提高开发效率并减少低级错误。

#### Prisma 如何解决 SQL 注入问题

Prisma 默认不是字符串拼 SQL，而是通过参数化查询和 ORM API 生成 SQL，所以天然比手拼 SQL 安全得多。

底层原理是：

1. **SQL 注入为什么会发生**
   因为开发者把用户输入直接拼进 SQL 字符串里，导致输入内容被当成 SQL 语法的一部分执行。比如用户输入里如果带 `' or 1=1 --`，拼接后可能把原本的条件逻辑改掉。

2. **参数化查询为什么能防注入**
   参数化查询会把“SQL 模板”和“参数值”分开传给数据库。
   例如：
   - SQL 模板：`select * from users where name = ?`
   - 参数值：`["alice' or 1=1 --"]`

   数据库在执行时会把参数当作“值”绑定进去，而不是把它重新解析成 SQL 语法，所以用户输入再恶意，也只会被当成普通字符串。

3. **ORM API 为什么通常更安全**
   因为 ORM 不是让你手写完整 SQL 字符串，而是通过结构化 API 先描述查询意图，再由 ORM 在底层生成参数化 SQL。

4. **数据库原生是不是也支持防注入**
   是的，`MySQL`、`PostgreSQL` 本身都支持 prepared statement / parameterized query。也就是说，防注入的底层能力并不是 Prisma 独有，而是数据库和驱动本身就支持；Prisma 的价值是默认把你引导到更安全的使用方式上。

所以更准确的说法是：**Prisma 不是凭空“发明”了防 SQL 注入能力，而是通过 ORM API 和参数绑定，把开发者从最容易出错的字符串拼接方式中解放出来。**

#### 介绍一下浏览器缓存（协商、强制缓存）

强缓存和协商缓存，本质上都是 **HTTP 缓存机制**。大多数情况下它们是浏览器原生自动做的，不需要你自己写 JS 去控制“下一次请求到底发不发”，但前提是服务端要正确返回响应头。

### 1. 强缓存怎么工作

服务端响应资源时返回：

1. `Cache-Control`
2. `Expires`

浏览器收到后会把资源和过期时间一起存起来。下次再请求同一个资源时，浏览器先看本地缓存是否还在有效期内：

1. 如果还没过期，直接用本地缓存，不发网络请求。
2. 如果过期了，再进入协商缓存流程。

这就是为什么说强缓存“命中后浏览器直接返回”，是浏览器自动判断并执行的。

### 2. 协商缓存怎么工作

如果强缓存失效，浏览器会带上条件请求头去问服务端资源有没有变：

1. 基于时间：
   - 服务端上次返回 `Last-Modified`
   - 浏览器下次请求带 `If-Modified-Since`
2. 基于内容标识：
   - 服务端上次返回 `ETag`
   - 浏览器下次请求带 `If-None-Match`

服务端收到后判断：

1. 没变，返回 `304 Not Modified`，不带完整资源内容。
2. 变了，返回新的 `200` 响应和新资源。

### 3. 开发者需要做什么

你通常不用写前端代码手动实现浏览器缓存判断，但你需要：

1. 配置服务端或 `CDN` 返回正确的缓存头。
2. 对带 hash 的静态资源设置长缓存。
3. 对 HTML 或变更频繁接口设置更保守的缓存策略。

### 4. 运行机制本质

可以理解成：

1. 是否直接使用本地副本，由浏览器根据响应头自动判断。
2. 是否需要向服务端确认，由浏览器自动带条件请求头。
3. 开发者主要负责“制定策略”，浏览器负责“执行策略”。

当然，如果你用了 `Service Worker`，那你可以自己接管更复杂的缓存逻辑，但那已经是比 HTTP 缓存更上层的应用缓存能力了。

#### JS - 正则表达式

正则表达式本质上是描述字符串匹配规则的语言。前端常见用途包括表单校验、日志提取、内容替换和路由匹配。面试里不要只背语法，要能讲清贪心/惰性、分组、捕获、前瞻这些概念。

### 1. 贪心和惰性

1. **贪心**
   默认量词如 `*`、`+`、`{m,n}` 都是贪心的，意思是“在能匹配成功的前提下，尽量多吃字符”。

   例如：
   ```js
   "abcabc".match(/a.*c/)
   ```
   `.*` 会尽量多匹配，所以结果会是整个 `"abcabc"`。

2. **惰性**
   在量词后面加 `?`，比如 `*?`、`+?`，表示尽量少匹配。

   例如：
   ```js
   "abcabc".match(/a.*?c/)
   ```
   结果会是第一个 `"abc"`。

### 2. 分组

分组就是用 `()` 把一段模式包起来，作用有两个：

1. 改变运算范围。
2. 让这段匹配结果可以被单独引用。

比如：

```js
/(ab)+/
```

表示整个 `ab` 作为一个整体重复，而不是只有 `b` 重复。

### 3. 捕获

捕获组就是正则匹配时，会把 `()` 里匹配到的内容单独记录下来。

例如：

```js
const match = "2026-04-10".match(/(\d{4})-(\d{2})-(\d{2})/);
```

这里：

1. `match[0]` 是整体匹配结果。
2. `match[1]` 是年份。
3. `match[2]` 是月份。
4. `match[3]` 是日期。

如果只想分组、不想捕获，可以用非捕获分组：

```js
/(?:ab)+/
```

### 4. 前瞻

前瞻是一种“只判断后面是不是某种模式，但不真正消费字符”的机制。

1. **正向前瞻** `(?=...)`
   表示当前位置后面必须满足某个模式。

2. **负向前瞻** `(?!...)`
   表示当前位置后面不能满足某个模式。

例如：

```js
/\d(?=px)/
```

表示匹配一个数字，并且它后面必须跟着 `px`，但匹配结果本身不包含 `px`。

所以前瞻的关键点是：**看后面，但不吃后面的字符。**

#### JS - Event Loop

JavaScript 是单线程执行模型，同步代码先执行，异步任务会进入任务队列。当前调用栈清空后，事件循环会先取微任务，再取宏任务，所以常见顺序是：同步代码 -> 微任务 -> 宏任务。

#### async-await

`async/await` 是基于 Promise 的语法糖。`await` 会暂停当前 `async` 函数后续逻辑，把后面的代码放到 Promise resolve 之后继续执行，本质上还是异步，不是阻塞线程。

#### 微任务还有哪些？

常见微任务包括：

1. `Promise.then/catch/finally`
2. `queueMicrotask`
3. `MutationObserver`

1. **`queueMicrotask`**
   这是浏览器和现代运行时提供的原生 API，用来显式地把一个回调放进微任务队列。它和 `Promise.then` 一样，都会在当前宏任务执行结束、下一个宏任务开始前执行。

2. **`MutationObserver`**
   这是一个监听 DOM 变化的 API，比如节点新增、删除、属性变化。它的回调不是同步立即执行，而是在当前调用栈清空后，以微任务的时机统一触发，这样可以把一批 DOM 变更合并通知，减少频繁同步回调的成本。

如果是 Node 环境，还可以补充 `process.nextTick`，不过它在 Node 里优先级更特殊。

这个题不求标准答案，建议从“产品形态突破”来讲，比如：

#### 用 Cursor 吗？Agent 模式？其他的……

可以答：我会用 `Cursor`，也会根据任务选择 Agent 模式或普通 Copilot 模式。简单补全和局部改代码时，我更偏向轻量辅助；复杂重构、跨文件分析、文档整理时，我会用 Agent 模式，因为它更适合长链路任务，但我不会完全依赖它，关键决策和结果验证还是由我自己把关。

#### Vue 和 React 区别

我会从三个层面讲：

1. **思维模型**：React 更偏函数式 UI 和状态驱动，Vue 更偏模板式和响应式系统。
2. **更新机制**：React 组件函数会重新执行，Vue 更强调依赖追踪和精细更新。
3. **生态风格**：React 更灵活、选择多，Vue 更统一、上手快。

本质上两者都很优秀，只是设计哲学不同。

#### React 17、18、19 之间的区别

1. **React 17**
   它更像过渡版本，最大的意义不是新功能很多，而是为后续升级铺路。最典型的变化是事件委托从 `document` 下沉到 root 容器，这样多个 React 应用共存时更容易隔离，也更方便渐进升级。对业务开发者来说感知没那么强，但对 React 生态从旧版本平滑迁移到 18 很关键。

2. **React 18**
   React 18 的核心是并发渲染能力真正进入正式主线。它引入了：
   - `createRoot`：启用新的 root API。
   - 自动批处理：不再只局限于 React 事件内批量更新，异步回调里也能批处理，减少不必要渲染。
   - `startTransition`：把更新区分成高优先级和低优先级，让输入响应更及时。
   - `Suspense` 能力增强：更适合和异步加载、流式 SSR 配合。

   底层意义是：React 不再要求所有更新都必须同步立即完成，而是可以按优先级调度，把更影响交互体验的更新先做。

3. **React 19**
   React 19 更强调和现代全栈框架、Server Components、表单能力的结合。比较典型的是：
   - `Actions`：更自然地处理表单提交、异步状态和错误。
   - 更完善的 Server Components 生态支持。
   - 对资源加载、文档 metadata 等体验进一步优化。

   它的意义是让 React 不只是“客户端视图库”，而是更自然地参与现代全栈渲染链路。

### 这些变化有什么用

1. React 17 让升级成本更低。
2. React 18 让复杂交互下的响应性更好，尤其适合搜索联想、筛选面板、列表切换这类高频更新场景。
3. React 19 让 React 在表单、服务端渲染、全栈协作上的体验更完整。

### 怎么用

1. React 17 主要是平滑升级，业务层几乎不用特意写新 API。
2. React 18 最常见的是：
   - 用 `createRoot` 替代旧的 `render`
   - 用 `startTransition` 包低优先级更新
   - 配合 `Suspense` 做异步边界
3. React 19 更适合在支持它的现代框架里用，比如和表单 action、Server Components 一起发挥价值。

如果面试官继续问，重点展开 React 18，因为它对实际开发影响最大。

## 七、按字节风格继续追问

#### 你刚才说“复杂任务完成率提升”，这个完成率是怎么定义的？怎么测的？

不会只说一个模糊指标，我会先把“完成”定义清楚。对 Agent 系统来说，完成率通常不是“模型回了内容就算完成”，而是看任务有没有达到预期目标。

我一般会从三个层次定义：

1. **技术完成**：任务链路跑通了，没有中断、报错或超时。
2. **结果完成**：输出结果满足预期，比如找到目标文件、成功调用工具、给出有效结论。
3. **业务完成**：结果对用户是可用的，而不是“看起来说了很多，但实际没解决问题”。

测量上我会用离线评测集加线上真实任务抽样。离线评测保证可比性，线上抽样保证真实场景。这样才能判断提升到底来自能力变强，还是只是提示词更会“说话”。

#### 你怎么证明你们真的需要多 Agent，而不是把单 Agent 的 Prompt 再调一调？

我会说，Prompt 优化当然是第一步，但当问题的根因已经不是“提示不够清楚”，而是“任务复杂度超过单上下文可稳定处理的范围”时，只靠 Prompt 的收益会变得很有限。

我们判断需要多 Agent，主要看三个信号：

1. 单 Agent 在复杂任务下的路径波动非常大。
2. 同一个任务里混合了多种异构能力，比如检索、浏览器操作、结构化查询。
3. 上下文一长，工具选择和结果整合质量明显下降。

所以多 Agent 不是为了替代 Prompt，而是在 Prompt 已经优化过之后，进一步解决结构性问题。

#### 如果主 Agent 任务拆错了，后面不是全都错了吗？怎么兜底？

这是多 Agent 架构里必须正视的问题，所以我们不会把主 Agent 当成永远正确的“中央大脑”，而是会做几层兜底：

1. **阶段校验**：主 Agent 产出计划后，先做结构化校验，看子任务是否缺失关键步骤。
2. **执行中反馈**：从 Agent 如果发现输入目标不清或方向不对，会把异常信号回传给主 Agent。
3. **动态重规划**：主 Agent 不是一次规划完就不管了，而是可以根据中间结果修正后续路径。
4. **回退机制**：高风险任务下，如果拆解不稳定，可以退回更保守的单链路执行。

如果面试官继续追问“结构化校验怎么实现”，可以这样答：

1. 先要求主 Agent 不是输出自由文本，而是输出固定 schema，比如：
   - `goal`
   - `subtasks`
   - `dependencies`
   - `expectedOutput`
   - `risk`

2. 用 `JSON Schema`、`zod` 或后端类型系统先做格式校验，保证字段齐全、类型正确。

3. 再做业务规则校验，比如：
   - 是否存在空子任务
   - 是否多个子任务目标重复
   - 是否缺少必要前置步骤
   - 是否某个子任务引用了不存在的依赖

4. 对高风险任务再加一层 LLM-as-judge 或规则引擎复核，比如“这个计划是否覆盖了用户原始目标”“是否遗漏检索步骤”“是否存在循环依赖”。

5. 如果校验不通过，不直接执行，而是要求主 Agent 重规划。

底层思想其实和后端接口校验很像：**先做结构合法，再做业务合理，最后再进入执行。**

#### Memory 复用很容易把旧经验带进来，如果经验过时了怎么办？

这是 Memory 系统最容易被忽略的问题。我的理解是，Memory 最大的风险不是“记不住”，而是“记错了还在继续用”。

所以我会强调四个机制：

1. **版本标签**：经验要绑定代码版本、工具版本或业务阶段。
2. **时效衰减**：越旧的经验权重越低，不应该和新经验同权。
3. **置信度管理**：不是所有历史结果都能进入长期 Memory。
4. **结果验证**：注入 Memory 后仍然要靠当前任务反馈验证，不是直接当真理。

也就是说，Memory 不是数据库备份，而是带时效和置信度的经验系统。

#### 面试官如果追问“这里哪些是你亲自做的，哪些是团队一起做的”，你怎么答更真实？

这个问题千万不要泛泛地说“我们做了很多优化”。更好的答法是主动拆开：

1. **我直接负责的**：比如前端执行链路可视化、某一段状态编排、某类体验优化、某个评测面板接入。
2. **我参与设计但不是独立负责的**：比如主从 Agent 整体方案讨论、Memory 方案评审。
3. **团队协同完成的**：比如模型效果优化、服务端调度、底层平台能力。

#### 你说 SSR 对首屏更友好，那 Hydration mismatch 你怎么排查？

我一般会从“服务端输出”和“客户端首次渲染”是否一致去查。

常见原因有：

1. 服务端和客户端依赖了不同环境变量或时间、随机数。
2. 渲染过程中直接读取了 `window`、`localStorage` 这类客户端对象。
3. 条件渲染依赖异步数据，但两端数据时序不一致。

排查时我会先定位哪个节点 mismatch，再看这段渲染逻辑有没有非确定性输入。原则上，首屏参与 hydration 的内容必须尽量保证同构一致。

#### RAG 里 topK 是怎么选的？是不是越大越好？

不是越大越好。`topK` 太小容易漏召回，太大又会把噪声带进后续 rerank 和生成阶段，反而拉低效果，还会增加延迟和 token 成本。

我会把它当成一个需要实验的参数：

1. 先看离线评测里 `Recall@K` 的变化趋势。
2. 再看 rerank 后 top 结果质量是否继续提升。
3. 最后结合线上延迟和成本做平衡。

所以 `topK` 的本质不是一个固定数字，而是召回质量、排序质量、时延成本之间的折中点。
