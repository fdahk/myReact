## 知识提问

- MVVM模式 详细讲讲

MVVM 是 `Model - View - ViewModel` 的缩写，本质是把界面展示和业务状态管理做分层。

#### 1. 三层分别做什么

- `Model`：数据层，负责业务数据、接口数据、数据转换。
- `View`：视图层，负责界面展示。
- `ViewModel`：连接层，负责把模型数据转换成视图可直接消费的状态和行为。

#### 2. 它解决什么问题

- 让视图和业务逻辑解耦。
- 让页面状态有统一来源，更容易维护和测试。
- 降低 DOM 手动操作的复杂度。

#### 3. 在前端里的体现

- Vue 的 `data`、`computed`、`watch`、模板绑定，很像典型 MVVM。
- React 并不严格宣称自己是 MVVM，但从“状态驱动视图”的角度看，组件状态 + 事件 + 渲染函数，也在做类似的事。

#### 4. 核心思想

- View 不应该直接改 DOM 来保持状态一致，而应通过修改状态来驱动 UI 更新。
- ViewModel 承担“状态组织、派发更新、派生视图状态”的职责。

#### 5. 优点

- 可维护性强。
- 状态流更清晰。
- 易测试。
- 易于做组件化和复用。

#### 6. 不足

- 对简单页面来说可能显得抽象偏重。
- 状态管理不当时，也可能出现状态分散、更新链路复杂的问题。

- Vue和React 的区别？

可以从以下几个角度回答：

#### 1. 设计理念

- Vue 更偏“模板驱动 + 渐进式框架”。
- React 更偏“UI = state 的函数映射”，强调函数式和组合能力。

#### 2. 模板与渲染

- Vue 常用模板语法，指令丰富，上手直观。
- React 使用 JSX，本质更接近“用 JavaScript 描述 UI”。

#### 3. 响应式机制

- Vue 主要是细粒度响应式追踪，知道哪些依赖变了。
- React 主要通过状态变更触发组件重新执行，再靠 diff 和调度优化。

把这句话展开讲，可以这样理解：

#### Vue：更偏“依赖追踪”

- Vue 会在数据被读取时收集依赖，在数据被修改时触发对应依赖更新。
- 也就是说，系统会记住“哪个视图、哪个计算属性、哪个副作用依赖了这份数据”。
- 当某个响应式字段变化时，只通知真正依赖它的地方更新。

可以把它理解成：

- 先建立一张“数据 -> 依赖者”的关系网。
- 后续数据一变，沿着这张关系网精确通知。

一个直观例子：

```js
const state = reactive({
  count: 0,
  name: "Tom",
});
```

如果模板里只有：

```html
<div>{{ count }}</div>
```

- 那么这个视图主要依赖的是 `count`。
- 当你改 `state.count++` 时，会触发相关更新。
- 当你只改 `state.name = "Jerry"` 时，这块只依赖 `count` 的视图通常不会因为 `name` 的变化而重新做同样的更新工作。

所以 Vue 常说自己是“细粒度响应式”：

- 关注点更偏向“具体哪个字段变了”
- 以及“谁依赖了这个字段”

#### React：更偏“重新执行组件，再比较结果” 

- React 的核心思路不是自动追踪“模板里依赖了哪个字段”。
- 而是当 state 或 props 变化后，让对应组件重新执行一次。
- 组件执行后会得到一棵新的虚拟 DOM，再和旧结果做比较，最后把真正需要变动的部分提交到页面。

可以把它理解成：

- 先重新算一遍这个组件应该长什么样。
- 再通过 diff 找出页面里真正变化的部分。

例如：

```jsx
function UserCard({ user }) {
  return (
    <div>
      <span>{user.name}</span>
      <span>{user.age}</span>
    </div>
  );
}
```

- 当 `user.age` 变化时，`UserCard` 会重新执行。
- React 不会像 Vue 那样天然先记住“这里只读了 `user.age`，那里读了 `user.name`”再按字段精确触发。
- 它更常见的方式是：组件重新跑一遍，然后通过 reconciliation 判断最终哪些节点真的要更新。

#### 所以二者的区别不只是“谁快谁慢”

更本质的区别是更新模型不同：

- Vue 更像“数据改了，精准通知依赖者”
- React 更像“状态改了，重新计算组件输出，再做比较和调度”

#### React 里为什么又提到调度

- React 不只是“重新执行然后 diff”这么简单。
- 在现代 React 里，更新还会进入调度系统。
- React 会根据更新类型决定优先级，比如输入框输入这类交互通常优先级更高，而列表筛选、过渡渲染可以延后。
- 这也是 React 在复杂交互场景里很强的一点：它不只关心“更新谁”，还关心“什么时候更新更合适”。

#### Vue 和 React 可以怎么一句话区分

- Vue 的强项是“依赖追踪粒度细”。
- React 的强项是“组件模型统一，配合 diff 和调度做更新控制”。

#### 面试里更准确的表述

- 不要简单说“Vue 是响应式，React 不是响应式”，这不准确。
- 更准确的说法是：
  - Vue 是基于依赖收集的细粒度响应式更新。
  - React 是基于状态变更触发组件重渲染，再结合虚拟 DOM diff 和调度机制完成更新。

#### 再深入一点

真正的差别不只是“更新方式不同”，而是两套框架在做不同的工程权衡。

#### 1. Vue 这套细粒度依赖追踪，本质上赢在哪

- 它更擅长回答“到底是谁依赖了这个字段”。
- 一旦依赖关系建立起来，更新时就可以更精准地通知。
- 在一些局部状态很多、依赖关系清晰的场景里，这种方式可以减少不必要的重复计算。

它的优势通常体现在：

- 更新粒度更细。
- 某些场景下，不需要让整个组件函数重新跑一遍。
- 对模板作者来说，写法更接近“声明我依赖什么，框架自动追踪”。

但它的代价也很明确：

- 框架内部必须维护依赖收集和触发机制。
- 开发者需要理解响应式边界、依赖收集时机、计算属性缓存、watch 触发时机等概念。
- 一旦遇到响应式丢失、解构丢失响应性、浅层/深层代理差异等问题，排查会偏“机制层”。

#### 2. React 这套“组件重执行 + diff + 调度”，本质上赢在哪

- 它更擅长统一心智模型。
- 对 React 来说，组件就是状态到 UI 的映射函数。
- 只要状态变了，就重新执行组件，重新算出新的 UI 描述。

这里的“状态变了，组件就重新执行”，一定要理解准确：

- 它不是说 React 一看到状态变化，就立刻把整棵 DOM 全部删掉重建。
- 它也不是说组件里每一行逻辑都会无脑重新产生真实副作用。
- 它更准确的意思是：**React 会把这次状态变化当成一次新的渲染任务，让相关组件重新参与计算，重新产出一份新的 UI 描述，然后再决定真实页面到底要不要改、改哪里。**

为什么 React 要这样设计，而不是像 Vue 那样先追踪“模板里具体用了哪个字段”？

- 因为 React 的核心抽象就是：`UI = f(state)`。
- 既然 UI 是状态的函数结果，那么状态变了，最稳妥的办法就是把这个函数再执行一次，重新算最新结果。
- React 不要求你手动告诉它“这一行 JSX 依赖了 count，那一行依赖了 name”。
- React 更愿意统一成一套模型：**输入变了，就重新算输出；至于最终哪里真的变，再由 React 自己去比较和调度。**

如果再往底层看，一次更新大致会经历下面这条链路：

#### 补充：状态变化后，React 内部到底做了什么

#### 1. 先产生更新

- 比如你调用了 `setCount(2)`，或者父组件传了新的 `props`，或者上层 `context` 值变了。
- React 不会立刻同步去改 DOM，而是先创建一条 update 记录。
- 这条 update 会挂到对应组件的 Fiber 节点更新队列上。
- 同时 React 会给这次更新打上优先级标记，告诉调度系统“这件事急不急，要不要马上做”。

#### 2. 调度系统把这次更新安排起来

- Fiber 可以理解成 React 在运行时为每个组件维护的工作节点。
- 每个组件对应一个 Fiber，里面会记住这个组件当前的 `props`、`state`、Hook 链表、更新队列以及它和父子兄弟节点的关系。
- 当某个 Fiber 上出现更新后，React 会沿着 Fiber 树一路把“有更新”这件事标记到根节点。
- 然后调度器决定什么时候开始这次渲染工作。

#### 3. render 阶段重新执行组件

- 真正进入 render 阶段后，React 会从根开始构建一棵新的 `workInProgress Fiber` 树。
- 处理到某个函数组件时，本质上就是再次调用这个组件函数。
- 调用时 React 会按顺序取出旧 Hook 状态、处理更新队列、计算出新的 state。
- 然后根据新的 `state / props / context`，执行函数体，得到新的 JSX，也就是新的 UI 描述。

很多人会被“树”这个词绕晕，所以最好把 React 里常见的几棵树拆开理解：

#### 补充：Fiber 树是什么？React 中常见有哪些树？

#### 1. React Element 树 / JSX 树

- 这是组件执行后产出的那棵“描述树”。
- JSX 编译后，本质会变成 `React.createElement(...)` 或新的 JSX runtime 调用，最终得到一层层 React Element 对象。
- 它主要描述的是：这个组件这次想渲染什么类型的节点、子节点是什么、`props` 是什么。
- 这棵树更像“渲染结果的声明”，而不是 React 内部真正长期维护的工作数据结构。

可以把它理解成：

- **React Element 树 = 这次 render 算出来的 UI 蓝图。**

#### 2. Fiber 树

- Fiber 树是 React 在运行时真正维护的工作树。
- Fiber 不是简单的 JSX 对象，而是一个带很多运行时信息的节点。
- 一个 Fiber 节点通常会记录当前组件的类型、`props`、`state`、更新队列、Hook 链表、父子兄弟关系、对应的真实 DOM 引用、优先级、副作用标记等信息。
- React 调度、可中断渲染、优先级管理、更新队列处理，基本都是围绕 Fiber 树完成的。

所以可以把它理解成：

- **Fiber 树 = React 内部真正拿来干活的运行时工作树。**

#### 3. current Fiber 树

- `current tree` 指的是当前已经 commit 到页面上的那棵 Fiber 树。
- 它代表“用户此刻真正看到的正式版本”。
- 这棵树上的节点，通常会和当前页面上的真实 DOM 保持对应关系。

可以把它理解成：

- **current tree = 当前线上正在展示的版本。**

#### 4. workInProgress Fiber 树

- 当有新的更新到来时，React 不会直接在 `current tree` 上硬改。
- 它会基于旧树，构建一棵新的 `workInProgress tree`。
- 这棵树表示“这次更新正在计算中的下一版草稿”。
- 在 render 阶段，React 会一边遍历旧 Fiber，一边生成或复用新的 workInProgress Fiber 节点。

可以把它理解成：

- **workInProgress tree = 下一版界面的草稿树。**

#### 5. 真实 DOM 树 / 宿主树

- 这棵树不属于 React 自己的数据结构，而是宿主环境里的真实界面树。
- 在 Web 里就是 DOM 树。
- 在 React Native 里则不是 DOM，而是原生视图树。
- React 最终要做的事情，就是在 commit 阶段把计算出的必要变更同步到这棵宿主树上。

可以把它理解成：

- **宿主树 = 用户最终真正看到和交互到的真实界面。**

#### 6. 这些树之间是什么关系

- 组件重新执行时，先产出新的 React Element 树。
- React 再根据新的 Element 树和旧的 `current Fiber tree`，去构建新的 `workInProgress Fiber tree`。
- render 阶段完成后，React 已经知道这次应该更新哪些地方。
- commit 阶段把这些变更同步到真实 DOM 树。
- commit 完成后，原来的 `workInProgress tree` 会切换成新的 `current tree`。

可以粗略记成这条链：

- `state/props/context` 变化
- 组件重新执行，产出新的 React Element 树
- React 基于旧 `current Fiber tree` 构建新的 `workInProgress Fiber tree`
- reconciliation 记录变更
- commit 到真实 DOM 树
- `workInProgress tree` 变成新的 `current tree`

#### 7. 为什么 React 要维护两棵 Fiber 树

- 因为 React 需要把“当前正在显示的界面”和“下一版正在计算的界面”隔离开。
- 这样 render 阶段就可以被打断、重试、延后，而不会把半成品直接显示给用户。
- 如果 render 过程中发现优先级更高的任务，也可以先处理中断再回来继续。
- 只有当整轮计算完成并进入 commit 时，React 才会把草稿版正式切成当前版。

这也是 Fiber 架构很关键的一点：

- **React 不是一边算一边立即改页面，而是先在内部工作树里把下一版算清楚，再一次性提交。**

#### 8. 面试里怎么简洁表达

- React Element 树是组件执行后产出的 UI 描述。
- Fiber 树是 React 运行时维护的工作树，里面挂着调度和更新所需的各种信息。
- `current tree` 是当前已经展示在页面上的 Fiber 树。
- `workInProgress tree` 是这次更新正在构建的下一版 Fiber 树。
- 真实 DOM 树是宿主环境里的实际界面，只有到 commit 阶段才会被真正更新。

#### 补充：Vue 和 React 中有“虚拟 DOM 树”这个概念吗

有，**Vue 和 React 都有虚拟 DOM 树的概念**，只是它们在运行时真正强调的那棵“核心工作树”不完全一样。

#### 1. Vue 里的虚拟 DOM 树是什么

- Vue 在 render 阶段也会产出一棵描述 UI 的树。
- 这棵树里的节点通常叫 `VNode`，也就是 `Virtual Node`。
- 一层层 `VNode` 组织起来，就可以理解成 Vue 的虚拟 DOM 树。
- 后续 Vue 会拿新的 `VNode` 树和旧的 `VNode` 树做比较，再去 patch 真实 DOM。

所以 Vue 里可以简单记成：

- **模板 / render 函数 -> VNode 树 -> diff / patch -> 真实 DOM**

#### 2. React 里的虚拟 DOM 树是什么

- React 里大家常说“虚拟 DOM”，通常指的是组件执行后得到的那棵 React Element 树。
- 这棵树本质上也是一层层 JavaScript 对象组成的 UI 描述树，可以近似理解成 React 的虚拟 DOM 树。
- 但是 React 真正运行时最核心的结构，不只是这棵描述树，而是 Fiber 树。
- 因为 React 的调度、优先级、中断恢复、更新队列、Hook 状态这些运行时能力，都挂在 Fiber 节点上。

所以 React 更准确的理解是：

- **组件执行 -> React Element 树（可近似理解为虚拟 DOM 树）**
- **React 再基于它去构建和更新 Fiber 树**
- **最后在 commit 阶段把必要变更同步到真实 DOM**

#### 3. 为什么很多人会觉得 React 的“虚拟 DOM”不好讲

- 因为在入门层面，说 React 有虚拟 DOM 完全没问题。
- 但在深入层面，如果只说“React 就是一棵虚拟 DOM 树”，就不够准确了。
- React 早就不只是“生成一棵对象树然后 diff”这么简单。
- 它在虚拟 DOM 描述之上，又引入了 Fiber 这层运行时工作结构。

也就是说：

- **React 有虚拟 DOM，但 React 的核心不只是一棵虚拟 DOM 树。**
- **React 更关键的是：用 Fiber 树把渲染、调度、优先级和更新过程组织起来。**

#### 4. Vue 和 React 在这件事上的区别可以怎么理解

- **Vue**：也有虚拟 DOM，通常说的是 `VNode` 树；同时它又结合响应式依赖追踪，让更新触发更细。
- **React**：也有虚拟 DOM，通常可以近似理解为 React Element 树；但运行时核心工作结构是 Fiber 树。
- **两者最终都会把结果落到真实 DOM**，只是中间的组织方式和强调重点不同。

可以粗略记成：

- Vue 更常见表述：`响应式系统 + VNode 树 + patch`
- React 更常见表述：`状态更新 -> React Element 树 -> Fiber 调度 / reconciliation -> commit`

这里的 `patch` 也很容易被问到，最好顺手理解清楚：

#### 补充：patch 是什么

- `patch` 可以直译成“打补丁”。
- 在前端渲染里，它的意思不是“重新生成整页”，而是：**把这次新旧结果之间的差异，按最小必要范围真正应用到真实 DOM 上。**
- 也就是把“计算出的变化”落地成“实际的节点操作”。

你可以把它理解成两步：

- **先比较**：新旧虚拟 DOM / VNode 哪些地方不一样。
- **再 patch**：把这些不一样真正同步到真实 DOM。

所以它和 `diff` 的关系通常可以粗略理解成：

- `diff` 更偏“找差异”。
- `patch` 更偏“应用差异”。

比如这段模板：

```html
<div>{{ count }}</div>
```

当 `count` 从 `1` 变成 `2` 时：

- 新旧 `VNode` 对比后，框架发现节点类型没变，变的只是文本内容。
- 这时就不需要把整个 `div` 删除重建。
- 而是直接把真实 DOM 中对应文本节点从 `1` 改成 `2`。
- 这个“把文本从 `1` 改成 `2` 的真实落地动作”，就可以理解成一次 `patch`。

如果再举几个常见场景：

- 文本变了：patch 文本内容。
- `class` 变了：patch 对应属性。
- 子节点多了：patch 时插入新节点。
- 子节点少了：patch 时删除旧节点。
- 节点类型变了：patch 时卸载旧节点并挂上新节点。

所以 `patch` 的重点不是“比较”，而是：

- **该改文本就改文本**
- **该改属性就改属性**
- **该插入就插入**
- **该删除就删除**

Vue 里为什么常说 `patch`：

- 因为 Vue 渲染更新时，经常会表述成“新旧 `VNode` 比较，然后 patch 到真实 DOM”。
- 也就是：`render -> VNode -> patch`。
- 它很强调把虚拟节点树对应的变化一步步打到真实节点上。

而在 React 语境里，通常更常见的说法是：

- `reconciliation`：更偏“新旧结果对账、决定哪些地方需要变”。
- `commit`：更偏“把最终变更真正提交到宿主环境”。

所以如果把概念大致对齐：

- Vue 里常说的 `patch`，可以近似理解成“把变更真正落到 DOM 上”的那一步。
- React 里更接近这层含义的，通常是 `commit` 阶段里的 DOM 提交动作。

面试里可以直接这样说：

- `patch` 就是把新旧虚拟节点比较后得到的差异，真正同步到真实 DOM 的过程。
- 它强调的是“应用差异”，不是“寻找差异”。
- 所以常可以把它和 `diff` 简单理解成“前者负责改，后者负责找”。

这里要特别注意：

- “组件重新执行”说的是**组件函数重新跑一遍**，不是“真实 DOM 立即重建”。
- 重新执行是为了得到新的 React 元素树，不是为了直接操作浏览器节点。
- 所以 render 阶段更像“重新算方案”，暂时还没有正式改页面。

#### 4. reconciliation 比较新旧结果

- React 拿到这次新算出的子节点后，会和上一次保存的结果做比较。
- 比较时重点看节点类型、`key`、层级位置等信息。
- 如果是同类型同位置节点，通常会复用已有 Fiber 和真实 DOM，只更新需要变的部分。
- 如果类型变了，或者 `key` 对不上，就可能卸载旧节点、创建新节点。
- 这个过程会把“插入、删除、更新”这些变更先记录下来。

#### 5. commit 阶段才真正改宿主环境

- 前面只是完成了“我应该怎么更新”的计算。
- 真正进入 commit 阶段后，React 才会把这些变更同步到宿主环境。
- 在浏览器里，宿主环境通常就是 DOM。
- 也就是说，这一步才会真的发生节点增删、属性修改、文本更新、ref 赋值，以及提交相关副作用。

所以“状态变了，组件重新执行”的完整理解应该是：

- 状态变化 -> React 记录 update -> 调度 render -> 重新执行相关组件 -> 产出新的 UI 描述 -> diff 找出差异 -> commit 把必要变更落到 DOM。
- 其中“重新执行组件”只是更新链路中的计算阶段，不等于页面已经立刻改变。

可以用一个最小例子理解：

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      {count}
    </button>
  );
}
```

当点击按钮后，底层可以粗略理解为：

- `setCount(count + 1)` 不是直接改按钮文字。
- 它先把“`count` 要从旧值变成新值”这件事放进更新队列。
- React 之后重新执行 `Counter`，这次读到的新 `count` 变成了 `1`。
- 新 JSX 从 `<button>0</button>` 变成 `<button>1</button>`。
- diff 后发现其实只需要改一个文本节点。
- 到 commit 阶段，页面上按钮文字才真的从 `0` 变成 `1`。

这也是为什么 React 经常强调两句话：

- **函数组件本质上是一个会被反复调用的渲染函数。**
- **组件函数重新执行，不代表真实 DOM 全量重建，只代表 React 要重新计算这块 UI 的最新结果。**

它的优势通常体现在：

- 模型统一，思维方式直接。
- 很适合函数式组合、Hooks 抽象、跨组件逻辑复用。
- 结合调度系统后，React 在复杂交互和大应用里有很强的可控性。
- 它不只关心“该不该更新”，还关心“现在更应该先更新什么”。

但代价也很明显：

- 组件函数会重新执行，开发者需要更关注渲染成本。
- 需要借助 `memo`、`useMemo`、`useCallback`、列表虚拟化等手段控制无效重渲染。
- 如果状态设计不合理，父组件一更新，子组件可能跟着重新执行。

#### 3. 为什么不能简单说 Vue 一定更快

- 很多人看到“细粒度更新”会直觉认为 Vue 一定更高效，但实际工程里不能这么绝对。
- 因为性能不只取决于更新粒度，还取决于：
  - 组件树结构
  - 状态设计
  - 列表规模
  - 计算逻辑复杂度
  - 调度能力
  - 缓存策略

举个简单例子：

- 如果一个页面的瓶颈在“大量低优先级更新打断用户输入”，那 React 的并发调度能力可能更有优势。
- 如果一个页面的瓶颈在“很多细小字段频繁变化，但每次只影响局部模板”，那 Vue 的细粒度依赖追踪可能会更自然。

#### 4. 为什么也不能简单说 React 一定更先进

- React 的调度模型、并发特性确实很强，尤其适合复杂交互、大型应用、平台化能力建设。
- 但这不等于它在所有业务场景都“全面压过” Vue。
- Vue 的模板系统、响应式系统、单文件组件对很多中后台、内容型应用、常规业务项目都非常高效，开发体验也很好。

#### 5. 二者其实代表两种不同的优化方向

Vue 更偏：

- 尽量让“更新传播”更精准。
- 也就是从数据依赖层面减少不必要更新。

React 更偏：

- 接受组件重新执行这个事实。
- 然后通过 diff、memo、调度、优先级来把整体更新成本控制住。

一句话理解：

- Vue 更像在“减少该通知谁”上做优化。
- React 更像在“即使重新算，也要把重算这件事组织好”上做优化。

#### 6. 那到底谁更优秀

如果从“局部更新精细度”看：

- Vue 的细粒度响应式通常更占优势。

如果从“复杂交互调度能力、抽象能力、生态和大型应用可组合性”看：

- React 往往更强。

如果从“上手成本和默认开发体验”看：

- Vue 对很多团队更友好。

如果从“函数式抽象、跨端生态、并发更新能力”看：

- React 更受大型前端体系和平台型团队青睐。


#### 8. 再补一个更底层的理解

- Vue 的核心问题是：字段变了，哪些依赖它的地方要更新？
- React 的核心问题是：状态变了，组件重新计算后，哪些结果真的需要提交，哪些任务应该先做？

这里的“提交”不是指提交代码，而是指 React 在完成计算后，把最终需要生效的变化真正应用到界面上的过程。

可以把 React 的更新粗略理解成两步：

#### 1. render 阶段

- 先重新执行组件。
- 计算新的 JSX / 虚拟 DOM 长什么样。
- 对比新旧树，找出哪些地方理论上需要变。
- 这个阶段更像“算方案”，还不是真正改页面。

#### 2. commit 阶段

- 把 render 阶段已经确定好的变更真正同步到宿主环境。
- 在 Web 里，宿主环境通常就是浏览器 DOM。
- 也就是说，这一步才会真的去做 DOM 更新、节点插入删除、属性修改、ref 赋值，以及触发生命周期 / effect 的提交相关逻辑。

所以“哪些结果真的需要提交” 的意思是：

- 组件重新执行不等于页面一定大量变化。
- React 会先算出新的结果，再判断哪些变化值得真正落地到 DOM。
- 只有最后进入 commit 的那部分，才是用户最终能看到的界面变化。

为什么 React 要区分 render 和 commit：

- render 阶段可以被打断、重试、延后，方便做调度。
- commit 阶段必须保持一致性，一旦开始就要把这批最终变更稳定地提交出去。
- 这也是 React 并发特性的基础之一：不是所有“重新算出来的东西”都会立刻落地到页面。

面试里可以直接这样总结：

- render 是“重新计算 UI 应该长什么样”。
- commit 是“把计算后的最终差异真正更新到页面上”。
- 所以 React 说的“提交”，本质上就是把可见结果正式落地。

所以二者不是简单的“谁替代谁”，而是两种不同的系统设计哲学。

#### 4. 组件通信

- Vue 有 `props`、`emit`、`provide/inject`、状态库等。
- React 有 `props`、回调、`context`、状态库等。

#### 5. 学习和工程体验

- Vue 对中小团队和传统前端同学更友好，模板、指令、单文件组件都比较完整。
- React 灵活性更高，生态更大，但工程约束更多依赖社区方案选择。

#### 6. 性能理解

- 两者都能做高性能应用。
- Vue 赢在细粒度依赖追踪。
- React 赢在调度模型、生态、并发特性以及函数式组合能力。

#### 7. 一句话总结

- Vue 更像“给你一套相对完整的默认方案”。
- React 更像“给你一个强大的 UI 抽象能力，由你组合出工程体系”。

- 数据驱动视图有什么好处？

核心好处是“让 UI 成为状态的结果”，而不是手动维护大量 DOM 细节。

#### 1. 降低复杂度

- 开发者只需要关心状态，不需要反复手动增删改 DOM。

#### 2. 一致性更强

- 页面更新逻辑集中在状态变更上，排查问题更容易。

#### 4. 更利于复用和测试

- 组件输入相同，输出就更可预测。

#### 5. 适合复杂交互

- 对列表、表单、权限、异步加载、分页、筛选等复杂场景更友好。

- 讲讲如何手写zustand？

#### 1. zustand 的核心思想

- 维护一个全局 `state`。
- 提供 `getState`、`setState`、`subscribe`。
- React 组件通过 Hook 订阅 store，在状态变化时重新渲染。

使用方式：

```js
const useBearStore = createStore((set) => ({
  bear: 0,
  increase: () => set((state) => ({ bear: state.bear + 1 })),
}));
```

#### 3. 两种用法哪种更好

```js
const bear = useBearStore((state) => state.bear);
const { bear } = useBearStore();
```

更推荐第一种：

```js
const bear = useBearStore((state) => state.bear);
```

原因：

- 第一种只订阅 `bear` 这个字段。
- 第二种拿到的是整个 store 对象，只要 store 中任意字段变化，组件就可能重新渲染。
- React 推荐更细粒度订阅，是因为这更符合性能优化原则：只在真正依赖的数据变化时更新组件。

把这两种写法再拆开看，会更清楚：

#### 写法一

```js
const bear = useBearStore((state) => state.bear);
```

- 这里传了一个 selector。
- selector 的意思是：从整个 store 中“选出”当前组件真正关心的那一小部分数据。
- 也就是说，这个组件订阅的不是整个 store，而是 `state.bear` 这个选中结果。

所以它的更新逻辑更像：

- 每次 store 变化时，zustand 会重新执行 selector。
- 比较“上一次的 `state.bear`”和“这一次的 `state.bear`”。
- 只有当这个结果变了，组件才需要重新渲染。

#### 写法二

```js
const { bear } = useBearStore();
```

- 这里没有传 selector。
- 组件拿到的是整个 store state 对象，然后再在组件内部自己解构出 `bear`。
- 也就是说，从订阅视角看，这个组件订阅的是“整个 state 对象”，不是单独的 `bear`。

所以它的更新逻辑更像：

- 只要 store 里任意字段发生变化，整个 state 对象通常就会产生新的引用。
- zustand 会认为订阅结果变了。
- 于是组件重新渲染。
- 虽然你最后只用了 `bear`，但订阅阶段并不知道你只在意它。

#### 为什么“看起来一样”，效果却不同

因为：

- `const { bear } = useBearStore()` 是“先订阅整个对象，再从对象里取 `bear`”。
- `const bear = useBearStore((state) => state.bear)` 是“从一开始就只订阅 `bear`”。

差别不在“最终拿到什么值”，而在“订阅的是谁”。

- 请问索引和外键是什么？

#### 1. 索引是什么

- 索引是数据库为了加快查询速度而建立的数据结构，常见实现类似 B+ 树。
- 作用类似书的目录，可以快速定位数据，而不是全表扫描。

优点：

- 提升查询、排序、分组效率。

代价：

- 占用额外存储。
- 插入、更新、删除时需要维护索引，写入性能会下降。

#### 2. 外键是什么

- 外键是表与表之间建立关联约束的机制。
- 一个表中的字段引用另一个表的主键或唯一键。

例子：

- 订单表中的 `user_id` 指向用户表中的 `id`。

作用：

- 保证引用完整性。
- 防止出现“订单引用了一个不存在的用户”这种脏数据。

#### 3. 区别

- 索引主要解决性能问题。
- 外键主要解决数据一致性问题。
- 外键字段常常会配合索引一起使用，但二者不是一回事。

- 讲讲SSE 和 WebSocket

#### 1. SSE 是什么

- SSE 全称 `Server-Sent Events`。
- 是服务端主动向客户端持续推送消息的一种机制。
- 基于 HTTP，浏览器通过 `EventSource` 接收流式文本。

特点：

- 单向通信，服务端 -> 客户端。
- 自动重连能力较好。
- 更适合消息推送、通知、AI 流式输出。

#### 2. WebSocket 是什么

- WebSocket 是全双工通信协议。
- 建立连接后，客户端和服务端都可以主动发送消息。

特点：

- 双向实时通信。
- 适合聊天室、协同编辑、实时游戏、交易推送等场景。

#### 3. 对比

- 通信方向：
  - SSE 单向。
  - WebSocket 双向。
- 协议基础：
  - SSE 基于 HTTP。
  - WebSocket 通过 HTTP Upgrade 升级协议。
- 数据格式：
  - SSE 主要是文本流。
    - 这里的“文本流”不是特指 JSON，而是指按文本协议持续传输的字符串数据流；JSON 只是其中一种常见文本内容，SSE 的 `data:` 后面也可以是普通文本、日志片段、状态标记或 JSON 字符串。
  data: hello

  data: {"id":1,"name":"Tom"}

  event: update
  data: done

  - WebSocket 文本和二进制都支持。
- 使用复杂度：
  - SSE 更简单。
  - WebSocket 能力更强，但服务端和连接管理更复杂。

#### 4. 实际选型

- 如果只是服务端持续推送，比如 AI 聊天流式输出、通知流、日志流，优先考虑 SSE。
- 如果需要双向实时互动，优先 WebSocket。

如果面试官继续追问“具体有什么用、怎么用”，可以这样展开：

#### 补充：Lighthouse 有什么用

- `Lighthouse` 是 Chrome DevTools 和 PageSpeed Insights 背后常见的一套自动化审计工具。
- 它的核心价值不是“替你直接优化完”，而是**帮你快速发现页面在性能、可访问性、工程规范和 SEO 上的明显问题**。
- 它适合做首轮体检、回归检查、上线前巡检，也适合拿来量化优化前后的变化。

它常见会从这几类维度给分并给建议：

- **Performance**：性能，例如首屏是否慢、资源是否过大、主线程是否阻塞、是否存在渲染链路问题。
- **Accessibility**：可访问性，例如图片是否缺少替代文本、表单控件是否有可识别标签、颜色对比度是否不足、语义标签是否缺失。
- **Best Practices**：最佳实践，例如是否有明显的安全风险、是否用了过时能力、是否存在影响稳定性的问题。
- **SEO**：搜索引擎友好性，例如标题、描述、可抓取性、基础语义结构是否合理。

如果是前端面试语境，最有用的是把它理解成：

- **一个快速发现问题的自动审计器**
- **一个把性能问题从“感觉慢”变成“有指标、有建议”的辅助工具**

#### 补充：Lighthouse 怎么用

最常见有 3 种用法：

#### 1. 在 Chrome DevTools 里直接跑

- 打开页面。
- 按 `F12` 打开 DevTools。
- 切到 `Lighthouse` 面板。
- 选择要审计的维度，比如 `Performance`、`Accessibility`、`SEO`、`Best Practices`。
- 选择设备类型，例如 `Mobile` 或 `Desktop`。
- 点击生成报告。

这适合本地开发、联调页面、快速看某个页面有没有明显问题。

#### 2. 用 PageSpeed Insights 跑线上页面

- 把线上 URL 放到 [PageSpeed Insights](https://pagespeed.web.dev/)。
- 它会结合 Lighthouse 报告和真实用户数据给出结果。

这适合：

- 看线上页面大致表现。
- 看实验室数据和真实用户数据之间的差异。
- 给产品或团队展示更直观的优化结果。

#### 3. 在 CI 或脚本里自动跑

- 团队里也会把 Lighthouse 接进 CI。
- 例如每次发版、合并前自动跑关键页面。
- 如果性能分数或关键指标低于阈值，就报警或阻止发布。

这适合做长期治理，而不是只靠人工偶尔点一次。

#### 补充：Lighthouse 具体能帮你发现什么

以性能为例，它经常能暴露这些问题：

- 首屏图片过大，没有压缩或没有懒加载。
- JavaScript 体积太大，导致解析和执行时间过长。
- 主线程长任务过多，用户输入响应差。
- CSS 或字体阻塞首屏渲染。
- 没有合理利用缓存、压缩、代码拆分。
- 首屏关键内容渲染太晚。

以可访问性为例，它常能发现：

- 按钮或输入框缺少可访问名称。
- 图片没有 `alt`。
- 标题层级混乱。
- 颜色对比度不足。
- 页面结构缺少语义化标签。

所以它很适合做“第一轮排雷”，但不代表：

- Lighthouse 分高，页面就一定真实体验好。
- Lighthouse 分低，就一定是线上用户全都卡。

它更像“实验室检查报告”，不是全部真相。

#### 补充：看 Lighthouse 时重点关注什么

如果是前端性能排查，建议先盯这些核心指标：

- `LCP`（Largest Contentful Paint）：最大内容元素出现的时间，常用来衡量首屏主要内容是否出来得够快。
- `INP`（Interaction to Next Paint）：交互到下一次可见更新的延迟，反映页面交互响应性。
- `CLS`（Cumulative Layout Shift）：累计布局偏移，反映页面是否乱跳。
- `FCP`（First Contentful Paint）：首次内容绘制时间。
- `TBT`（Total Blocking Time）：总阻塞时间，能帮助判断主线程是否被长任务卡住。

#### 补充：Lighthouse 怎么和实际优化结合

一个比较实用的使用方式是：

1. 先跑 Lighthouse，找出分数低和被标红的建议。
2. 再结合 `Network`、`Performance` 面板去验证具体瓶颈。
3. 区分问题属于资源体积、渲染阻塞、主线程长任务、布局抖动还是图片策略。
4. 做针对性优化后再回跑 Lighthouse，对比指标变化。

也就是说：

- `Lighthouse` 适合告诉你“哪里可能有问题”。
- `Performance` 更适合告诉你“问题发生时，主线程和渲染流水线具体卡在哪”。

#### 补充：面试里怎么简洁回答 Lighthouse

- `Lighthouse` 是一个自动化网页审计工具，主要用来检查性能、可访问性、SEO 和最佳实践。
- 我一般会用它先做首轮体检，快速发现图片过大、阻塞资源、长任务、语义化不足、可访问性缺失这类问题。
- 真正定位性能瓶颈时，我会再结合 DevTools 的 `Performance`、`Network` 面板一起看，因为 Lighthouse 更偏“指标和建议”，不是完整的底层时序分析工具。

- 有没有用过performance的录制功能 会显示哪些内容

有，Performance 录制主要是分析“页面卡顿到底卡在哪”。

录制后常见会看到：

- `FPS`：帧率是否稳定，是否掉帧。
- `CPU` 时间分布：JavaScript 执行、样式计算、布局、绘制、合成分别占多少。
- `Main Thread` 时间线：主线程每一段时间到底在忙什么。
- `Network`：录制期间请求行为。
- `Screenshots`：页面在录制期间的视觉变化。
- `Timings`：如 `DOMContentLoaded`、`Load`、自定义埋点。
- `Frames`：每一帧是否超过 16.6ms。
- `Long Tasks`：是否存在长任务阻塞主线程。

排查思路一般是：

- 先看是否掉帧。
- 再看主线程是被脚本卡住、布局抖动卡住，还是绘制过重。
- 如果是脚本问题，再展开调用栈定位具体函数。

- 大模型中的token 是什么 有什么用 

#### 1. token 是什么

- token 不是单纯“一个字”或“一个单词”，而是模型分词后的最小处理单元。
- 一个中文词、一个汉字、一个英文单词、一个标点，甚至一个单词片段，都可能对应一个或多个 token。

#### 2. 有什么用

- 模型读取输入时按 token 处理。
- 模型生成输出时也是按 token 预测下一个 token。
- 计费通常按 token 数量。
- 上下文窗口大小本质也是“最多能处理多少 token”。

- 大模型是会返回一个token还是返回一批token呢

从模型原理上说，是“逐步预测下一个 token”。

但从工程实现上看，前端经常会看到“分批流式返回”：

- 模型内部是一步一步生成 token。
- 服务端为了网络效率，可能把多个 token 合并成一个 chunk 再推给前端。
- 所以用户感知上像是一段一段返回，但底层生成逻辑依然是自回归逐 token 预测。

- 你了解大模型对话中的哪些技术，这些技术是怎么结合token的呢

可以从对话链路来讲：

#### 1. 分词

- 用户输入先被切成 token，模型才能处理。
- 更准确地说，大模型并不是直接理解“词”或“句子”，而是先把文本编码成 token 序列；这个 token 往往不是自然语言里的完整单词，而可能是一个字、半个词、子词片段、标点，甚至某些特殊前缀组合。
- `tokenizer` 就是把原始文本和 token ID 互相转换的编码器/解码器，可以理解成模型的“文本入口和出口适配器”。
- 常见分词方式可以粗略理解成 `BPE`（Byte Pair Encoding，字节对编码）、`SentencePiece`（一种直接从原始文本学习子词切分规则的分词框架）这类子词切分策略，核心目标是在“词表大小”和“编码效率”之间做平衡：词表太大，训练与推理成本高；词表太小，又会把文本切得过碎。
- 同样一句话，在不同模型里的 token 数可能不同，因为它们的 tokenizer 不同；这也是为什么“同样 1000 个汉字，在不同模型里占的上下文空间未必一样”。
- 工程上讲 token 很重要，不只是因为它是模型输入单位，还因为**上下文窗口、计费、延迟、缓存命中、截断策略**很多时候都是按 token 而不是按字符计算。

这里还要补一个很容易被问到的工程边界问题：

#### 补充：把输入文本切成 token，这件事到底是谁做的

- 从职责分层上看，这通常是 **tokenizer / 模型服务层** 在做的事。
- 更底层一点说，它属于“模型输入编码”这层能力，不属于普通业务前端页面自己手写分词规则的范畴。
- 在完整链路里，常见顺序是：原始文本 -> tokenizer 编码成 token ID -> 模型推理 -> 输出 token ID -> tokenizer 解码回文本。

如果再按层级拆一下：

- **模型层**：训练时绑定了自己的词表、分词规则、特殊 token 规则。
- **模型服务层 / SDK 层**：通常负责把你传入的文本按该模型对应的 tokenizer 编码，再喂给模型。
- **应用层 / 前端业务层**：更多关心传什么消息、怎么裁剪上下文、怎么展示结果，通常不亲自做底层 token 切分。

所以对大多数“调用第三方 AI 接口”的前端开发来说：

- **通常不需要你自己手动把文本先切成 token。**
- 你一般传字符串、消息数组、系统提示词、历史对话这些高层结构就够了。
- 真正的 tokenizer 编码，通常由模型提供方的服务端、官方 SDK 或网关层完成。

也就是说，你平时调用第三方 AI 更像是在做：

- 组织输入内容
- 控制上下文长度
- 决定是否摘要、截断、RAG 注入、工具结果拼接
- 处理流式输出和 UI 展示

而不是自己做：

- `BPE`
- `SentencePiece`
- token ID 编码
- 特殊 token 拼接规则

#### 补充：那前端是不是完全不用关心 token

不是“自己编码 token”，但**仍然要关心 token 带来的工程影响**。

你通常不需要自己分词，但经常需要关心：

- 当前 prompt 大概会占多少 token。
- 历史对话要不要裁剪。
- 长文档要不要先切块、检索、摘要。
- 为什么某个模型报价高。
- 为什么同样长度的中文和英文，在不同模型里的 token 消耗不同。
- 为什么回答被截断，或者上下文超限。

所以更准确的说法是：

- **不需要自己实现 tokenizer。**
- **但需要理解 token 机制，因为它直接影响成本、窗口、延迟和产品体验。**

#### 补充：什么时候你才可能需要主动处理 token

只有在更底层或更工程化的场景里，你才可能直接碰 token 级问题，比如：

- 自己做 AI 网关或模型代理层。
- 需要精确预估 token 成本。
- 做长上下文裁剪、记忆压缩、chunk 策略设计。
- 接入需要你自己传 token 计数或上下文预算的 SDK。
- 做多模型路由，比较不同模型 tokenizer 的消耗差异。

所以面试里可以直接这样答：

- 文本切 token 这件事通常属于 tokenizer 或模型服务层，不是普通前端业务代码手写完成的。
- 我调用第三方 AI 时一般不需要自己做分词，但要理解 token 对上下文窗口、成本、截断和延迟的影响。

#### 2. 上下文窗口

- 当前问题、历史对话、系统提示词、工具结果，都会一起转成 token 放进上下文。
- token 超限时，就必须截断、摘要或做记忆压缩。
- 上下文窗口本质上是“本次推理时模型允许同时看到的 token 上限”，不是无限历史记忆。模型并不是天然记住整个聊天记录，而是每次请求时把要给它看的内容重新拼进去。
- 在运行时，一次对话请求常见会按这个顺序组织：系统指令、开发者指令、历史多轮对话、检索材料、工具结果、用户当前问题。最终这些内容会被串成一个长 token 序列送给模型。
- 窗口超限时，常见处理方式包括：截掉最旧对话、保留摘要替代原文、分层记忆、只保留最近几轮、把长文档先检索再局部注入。
- 再深入一层，模型在推理时还会依赖**位置编码**来知道 token 的先后顺序；没有位置编码，模型只能看到“有哪些 token”，却不知道它们在句子里排第几。
- “位置编码”可以理解成给每个 token 额外附带一个“它在序列里排第几”的位置信息，让模型区分“我爱你”和“你爱我”这种 token 相同但顺序不同的输入。
- 工程里还经常提到 `KV Cache`（Key-Value Cache，键值缓存）：可以把它理解成“模型在前面已处理 token 上算出的注意力中间结果缓存下来”，这样流式生成后续 token 时不必每次从头把整段上下文重新全算一遍，这对长上下文和流式输出的性能非常关键。

#### 3. Prompt Engineering

- 提示词设计本质是在组织 token 序列，影响模型后续预测分布。
- `Prompt Engineering` 可以直译为“提示工程”或“提示词工程”，本质是围绕模型输入组织方式做设计，而不是单纯写几句自然语言。
- 更底层地说，提示词不是“给模型下命令的魔法咒语”，而是在**改变模型看到的条件分布**：前面 token 如何排列，会直接影响“下一个 token 最可能是什么”。
- “条件分布”可以粗略理解成：在“已经看到这些上文 token”的前提下，下一个 token 各自有多大概率被选中。
- 所以 prompt engineering 的核心不是背模板，而是控制三件事：**角色边界、任务约束、上下文示例**。例如你给不告诉模型输出格式、是否允许编造、是否必须先思考再回答、是否要调用工具，本质上都是在限制后续 token 的可选空间。
- `few-shot` 可以理解成“少样本示例提示”：不重新训练模型，只是在 prompt 里先给几组输入输出样例，让模型模仿这个模式继续生成。
- few-shot 示例之所以有效，也是因为示例本质上在告诉模型：“前面类似 token 模式出现后，后面常接什么样的 token 模式。”
- 实战里高质量 prompt 往往包含：角色、目标、输入上下文、输出格式、边界条件、失败策略，而不是只写一句“请帮我回答”。

#### 4. RAG

- 先检索外部知识，再把相关内容拼进上下文 token 中，帮助模型回答更准。
- `RAG` 全称是 `Retrieval-Augmented Generation`，中文常译“检索增强生成”；它的本质不是“让模型学会更多知识”，而是**在推理前临时补充当前问题最相关的外部上下文**，减少纯参数记忆带来的过时、幻觉和遗漏。
- 典型链路可以拆成：用户问题 -> query 改写/扩展 -> embedding 向量化 -> 向量检索/关键词检索/混合检索 -> 召回文档块 -> 重排 -> 把高相关片段拼进 prompt -> 模型生成答案。
- 这里有几个关键术语最好一起讲：  
  - `embedding`：嵌入向量，把文本映射成高维向量，方便按语义相似度做检索。  
  - `chunk`：文档切块，长文档不会整篇塞进上下文，通常会先切成若干可检索片段。  
  - `rerank`：重排，召回之后再用更重的模型重新排序，提升前几条结果质量。  
- “召回”可以理解成先尽量多地把可能相关的候选内容找回来；“重排”则是在召回结果里再精细排序，决定谁最值得进入最终上下文。
- `query` 就是当前拿去检索的查询语句；“query 改写/扩展”指先把用户原问题改成更适合搜索的表达，例如补全关键词、拆歧义、增加同义词。
- “混合检索”通常指把关键词检索和向量检索结合起来：前者擅长精确命中术语，后者擅长语义相似。
- “幻觉”在大模型语境里通常指模型生成了看起来像真话、但实际没有依据或与事实不符的内容。
- RAG 的难点不在“能不能查到”，而在“召回是否准、拼进去的片段是否刚好够用、是否污染主问题、是否超过 token 窗口”。所以工程里常见优化点是 chunk 策略、召回策略、重排、引用对齐、来源标注和答案约束。

#### 5. Function Calling / Tool Use

- 模型先输出结构化 token，表达“我要调用哪个工具、传什么参数”。
- 工具结果再被追加为 token，让模型继续生成最终回答。
- `Function Calling` / `Tool Use` 可以理解成“模型具备调用外部能力的协议化接口”，不是说模型内部真有数据库或浏览器。
- 更准确地说，模型并不是真的“自己执行工具”，而是先根据当前上下文生成一段**符合约定 schema 的结构化输出**，例如函数名、参数 JSON；外部 `orchestrator`（编排器/调度器，一般就是应用服务）看到后，才去真正调用数据库、搜索引擎、HTTP API、代码执行器等工具。
- 工具调用完成后，返回结果又会以“工具消息”或“函数结果消息”的形式重新注入上下文，让模型继续下一轮 token 生成。
- “结构化输出”可以理解成输出格式不是自由文本，而是符合某种确定结构，例如 JSON、字段对象、枚举动作；这样外部程序才能稳定解析。
- 所以 tool use 的本质是一种“模型负责决策，外部系统负责执行”的协作闭环。模型像一个会决定“该不该查、查什么”的调度器，但真正的“副作用”仍由外部系统控制。这里的“副作用”可以简单理解成会真正改变外部世界或依赖外部系统状态的动作，比如发请求、写数据库、下单、发消息。
- 这也是为什么函数调用特别适合做：查天气、查数据库、调用搜索、执行代码、访问业务 API。它解决的是“让模型把回答建立在外部真实结果上”，而不是纯靠语言概率瞎猜。
- 工程上要注意 `schema`（数据结构约束，定义参数应该长什么样）、参数校验、权限控制、幂等性、超时、失败重试和多工具编排，否则工具链越复杂，幻觉只是从“编答案”变成“编参数”。

#### 6. 流式输出

- 模型每生成一部分 token，服务端就逐步推给前端，实现打字机效果。
- 模型在推理内部通常仍然是一步一步生成 token，但服务端对外不一定一 token 发一个包，因为那样网络开销太高。实际工程里常见是把若干 token 合并成 chunk，再通过 SSE、WebSocket 或分块 HTTP 响应推给前端。
- 所以前端看到的“一个个字蹦出来”，底层可能不是严格一 token 对一网络包，而是**逐 token 生成 + 按 chunk 传输**。
- 流式输出的价值不只是在视觉体验上更像“实时思考”，还包括：降低首 token 等待焦虑、让长回答更早可见、方便边生成边取消、支持 AI 聊天的增量 UI。
- 工程上还要处理：半个 Markdown 代码块、半个 JSON、断流重试、用户取消、消息拼接、打字速度节流、前端渲染抖动等问题。

#### 7. 采样策略

- 如 `temperature`、`top-p`，本质上都是在“下一个 token 的候选分布”上做采样控制。
- 更细一点讲，模型每一步都会先得到一个对整张词表的概率分布（或 `logits`，也就是还没过 softmax 的原始打分），采样策略就是决定“到底从这个分布里怎么挑下一个 token”。
- `temperature` 可以理解成“把分布拉平或拉尖”：越低越保守，越高越发散。极低时更接近贪心选最大概率 token，极高时随机性更强。
- `top-p` 是 `nucleus sampling`（核采样），可以理解成“只保留累计概率达到某个阈值的一小撮候选，再在里面采样”，避免尾部大量极低概率 token 造成噪声。
- 有时还会看到 `top-k`、frequency penalty、presence penalty 等，本质都是在干预候选 token 的可选集合或相对权重。
- `top-k` 指只保留概率最高的前 K 个候选；`frequency penalty`（频率惩罚）偏向抑制已经重复很多次的 token；`presence penalty`（存在惩罚）则更像是“出现过就降一点权重”，鼓励话题扩展。
- 面试里可以补一句：采样参数不是越高越“聪明”，而是在**稳定性、创造性、格式一致性、幻觉风险**之间做权衡；代码生成、结构化输出往往倾向更低随机性，创意写作则可能接受更高随机性。

#### 8. 从一次真实大模型对话请求看，token 相关技术是怎么串起来的

- 用户输入文本
- tokenizer 把输入切成 token
- 系统提示词、历史消息、RAG 片段、工具结果一起被拼成上下文 token 序列
- 模型基于这些 token 和位置编码做前向推理
- 若需要工具，先生成结构化 token 触发 tool call
- 工具结果回填为新的 token，再继续推理
- 解码阶段按采样策略一步步选出下一个 token
- KV Cache 让后续 token 生成不必重复全量计算前文
- 服务端把生成结果按 chunk 流式推给前端
- 前端再把文本增量渲染成聊天内容

#### 9. 实际工程里最容易被忽略的点

- token 不是字符数，汉字、英文、标点、空格在不同 tokenizer 下成本不同。
- 上下文不是“会自动记住”，而是每次请求重新喂进去。
- RAG 不是知识库本身，而是“检索 + 拼接上下文”的推理增强链路。
- 工具调用不是模型真在执行代码，而是模型输出结构化意图，外部系统替它执行。
- 流式输出不是“模型一次吐一整段”，而是生成和传输都被分段处理。
- 采样不是“随机一点更聪明”，而是对输出稳定性和创造性的调参。

一句话总结：

- 大模型对话的核心，本质就是“围绕 token 的编码、组织、检索、生成和传输”。

- 讲讲`useTransition` 以及高优先级任务和低优先级任务 怎么判定

#### 1. `useTransition` 是什么

- `useTransition` 是 React 并发特性里用于“标记某些更新可以延后”的 Hook。
- 它返回：

```js
const [isPending, startTransition] = useTransition();
```

- 被 `startTransition` 包裹的更新会被视为低优先级更新。

#### 2. 典型使用场景

- 输入框内容要立刻响应。
- 但输入后触发的大列表过滤、复杂渲染、搜索结果刷新可以稍后做。

示例：

```jsx
const [keyword, setKeyword] = useState("");
const [list, setList] = useState([]);
const [isPending, startTransition] = useTransition();

function handleChange(e) {
  const value = e.target.value;
  setKeyword(value);

  startTransition(() => {
    setList(expensiveFilter(value));
  });
}
```

这样用户输入不会因为大列表更新而卡顿。

#### 3. 什么是高优先级任务

- 用户交互直接相关、需要立即反馈的更新。
- 比如输入、点击、聚焦、文本显示变化。

#### 4. 什么是低优先级任务

- 可以稍后完成、不影响用户即时操作反馈的更新。
- 比如筛选结果、大列表渲染、Tab 内容切换后的重内容区域刷新。

#### 5. React 怎么判定优先级

- 不是开发者手动传一个数字优先级。
- React 会根据更新来源和 API 语义分配不同调度优先级。
- 普通用户输入相关更新默认更紧急。
- `startTransition` 明确告诉 React：这批更新可以被打断、可以延后。

#### 6. 核心理解

- 高优先级更新保证交互流畅。
- 低优先级更新保证页面最终一致。
- React 调度器会优先处理更紧急的任务，必要时打断低优先级渲染并重试。
