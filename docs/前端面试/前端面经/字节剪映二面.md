# 字节剪映二面参考回答

## 一、核心项目深挖

#### 请详细介绍一下实习项目，包括项目背景、你的核心职责、落地成果。

我在实习里做的是一个面向复杂任务处理的 AI Agent 系统，目标是让系统不仅能问答，还能做任务拆解、工具调用、结果汇总和经验沉淀。项目背景是，传统单 Agent 在复杂任务下容易出现上下文过长、路径不稳定、工具使用不一致和成功率波动的问题，所以我们把系统升级成多 Agent 协同架构。

我的核心职责主要有三部分：

1. 负责前端侧任务执行链路的产品化，包括任务面板、步骤可视化、流式结果、错误态和可追溯性设计。
2. 参与主从 Agent 的接入与 orchestration 落地，包括任务拆分结果展示、工具调用状态同步、执行阶段管理。
3. 参与系统优化，重点做了 Memory 复用、重复探索控制、上下文压缩和执行可观测性建设。

落地成果方面，我通常会从三个维度表达：

1. **效果指标**：复杂任务完成率提升，回答稳定性更高。
2. **效率指标**：平均执行时长下降，重复调用减少。
3. **成本指标**：上下文更短、token 消耗更可控。

#### 请讲一下系统的完整优化过程，包括根因定位、解决方案和最终效果。

先说根因定位。系统早期虽然能跑通，但复杂任务表现不稳定，具体问题有三个：

1. 主 Agent 一次性背太多上下文，决策质量会下降。
2. 多个从 Agent 会对相似问题做重复探索，浪费时间和 token。
3. 历史经验没法有效复用，导致相似任务每次都要重新探索。

然后是解决方案，我们分三步做：

1. **任务结构化**：把用户请求先转成结构化子目标，再由主 Agent 分发给不同 specialist，减少“一个 Agent 处理所有问题”的混乱。
2. **Memory 体系化**：把历史成功路径、关键结论和稳定知识做分层存储，按相关性检索注入，而不是全量回放历史。
3. **执行链路可观测**：把计划、调用、失败原因、重试、耗时都记录下来，这样优化不再靠感觉，而是能定位具体瓶颈。

最终效果一般可以这样总结：

1. 长任务成功率更稳定。
2. 重复检索显著减少。
3. 平均链路长度和 token 开销下降。
4. 对线上问题的排查速度提升。

面试官更喜欢你讲“为什么会这样优化”，而不是只堆技术名词。

#### 请详细介绍项目里的 Memory 复用机制，包括设计初衷、核心实现、存储与管理方案。

设计初衷很明确：**让系统对相似任务不必每次从零开始。**

如果没有 Memory，Agent 每次都要重新理解问题、重新检索、重新试错。这样不仅慢，而且不稳定。

我们的 Memory 我会分成三层来讲：

1. **Working Memory**
   当前任务内的临时上下文，比如已经找到的文件、已调用的工具结果、失败原因。

2. **Episodic Memory**
   某一类任务的历史执行经验，比如“遇到代码定位问题，先查路由入口，再查页面组件，再查接口层”。

3. **Semantic Memory**
   相对稳定的长期知识，比如术语定义、项目目录规则、接口规范、常见错误处理方式。

如果面试官继续追问“不是概念，具体怎么存、怎么检索、怎么治理”，我会这样展开：

1. **Working Memory 的落地**
   当前任务内的中间状态我通常按 `sessionId / taskId / stepId` 做结构化存储，热数据放 `Redis` 或进程内缓存，内容包括：
   - 当前子任务目标
   - 已调用过的工具
   - 工具输入输出摘要
   - 失败原因
   - 已确认的中间结论

2. **Episodic Memory 的落地**
   这类“历史经验”通常拆成两部分：
   - 文本摘要和 embedding 存向量索引，用于语义召回
   - 元数据存关系型数据库或文档库，用于过滤和治理

   元数据一般会带：
   - `taskType`
   - `domain`
   - `sourceVersion`
   - `successScore`
   - `createdAt`
   - `expiredAt`
   - `owner / source`

3. **Semantic Memory 的落地**
   更接近知识库。通常原始文档先做 chunk 切分，再生成 embedding 建索引；同时保留原始文本和版本信息，避免只剩向量却丢失可解释性。

核心实现上有三个点：

1. **写入策略**
   不是所有内容都写，只有高置信、低时效、可复用的信息才长期沉淀。

2. **检索策略**
   先根据任务类型、领域标签和语义相似度做召回，再做优先级排序，只把最相关的少量 Memory 注入上下文。

3. **压缩策略**
   长执行链路不会原样存，而是抽成“任务 -> 路径 -> 关键决策 -> 结果”的摘要，避免记忆污染和上下文膨胀。

存储方案上，可以这样讲：

1. 结构化信息放在关系型或 KV 存储里，比如任务类型、标签、版本、时效。
2. 语义检索部分放在向量索引里，用于相似经验召回。
3. 热数据放缓存，降低重复读取成本。

管理上最关键的是版本和过期机制。因为工程经验有强时效性，如果代码结构变了、工具变了，旧经验不能无脑复用。

具体使用时也不是“把命中的 Memory 全塞进 prompt”，而是：

1. 先根据当前任务生成检索 query。
2. 先按任务类型、版本、领域标签做 metadata 过滤。
3. 再做向量召回，拿到候选经验。
4. 按相关性和置信度排序，只保留少量 topN。
5. 注入时压缩成可执行提示，比如“优先先查哪里”“避免哪些失败路径”。

所以它本质上不是聊天记录备份，而是“可检索、可衰减、可证伪”的经验系统。

#### 你为什么选择主从多 Agent 架构？对比其他架构做了哪些选型取舍？

我选择主从多 Agent 架构，不是因为它“听起来高级”，而是因为它更适合处理多阶段、跨能力边界的复杂任务。

我主要对比过三类架构：

1. **单 Agent + 多工具**
   优点是简单，延迟低。
   缺点是复杂任务下容易上下文过载，规划和执行混在一起，稳定性一般。

2. **Plan-and-Execute**
   优点是先规划再执行，结构更清晰。
   缺点是计划一旦不准，后续会一路偏，动态调整能力有限。

3. **主从多 Agent**
   优点是可以把全局规划和局部执行解耦，不同 specialist 能并行处理子问题，复杂任务成功率更高。
   缺点是系统复杂度、通信成本和调试成本都更高。

最后我选择主从多 Agent，是因为我们的任务天然存在异构能力，比如代码分析、文档检索、浏览器操作、结构化查询，不拆开很难稳定做好。

但我也不会把所有问题都丢给多 Agent。简单任务还是单链路直出，只有复杂任务才升级到多 Agent，这样能平衡稳定性和成本。

#### 这个项目目前还有哪些技术瓶颈？后续的迭代优化方向是什么？

我觉得目前还存在四类瓶颈：

1. **任务拆解质量仍然依赖模型判断**，有时会拆得过细或过粗。
2. **跨 Agent 协作成本高**，通信和结果对齐本身会带来额外开销。
3. **Memory 复用有误召回风险**，如果旧经验和当前上下文不匹配，会误导后续执行。
4. **评测体系还不够完整**，复杂任务很难像分类任务一样有统一标准答案。

后续我会优先做四个方向：

1. 引入更稳定的任务分类和路由机制。
2. 把工具能力标准化，降低 Agent 间耦合。
3. 做更强的离线评测集和线上回放分析。
4. 强化 human-in-the-loop 机制，在高风险任务上引入可中断和可确认节点。

#### 你了解哪些主流的 Agent 架构？请讲一下 ReAct、Self-Discovery、Plan-and-Execute 等架构的核心原理与适用场景。

我会这样概括：

1. **ReAct**
   核心是 Thought + Action + Observation 的循环，也就是边想边做边根据结果修正。适合需要多轮工具交互、路径不完全确定的任务。

2. **Plan-and-Execute**
   核心是先规划完整步骤，再按计划执行。适合任务结构较稳定、步骤相对明确的场景。

3. **Self-Discovery**
   核心是让模型先发现解决这类问题需要哪些推理结构或能力，再进入正式求解。适合复杂推理类问题，但成本一般更高。

4. **Multi-Agent**
   核心是分工协作，不同 Agent 负责不同子问题。适合跨知识域、跨工具类型的复杂任务。

**开放问题偏 ReAct，稳定流程偏 Plan-and-Execute，超复杂异构任务偏 Multi-Agent。**

## 二、RAG 项目深挖（电商智能导购项目）

#### 请详细介绍一下这个电商 RAG 智能导购项目，包括核心功能、完整实现链路、落地效果。

这个项目的目标是让用户用自然语言表达需求，比如“我想找轻薄、适合出差、续航长的笔记本”，系统能从真实商品库中检索出候选商品，再由大模型生成解释型回答和推荐理由。

完整链路我一般这样讲：

1. 用户输入 query。
2. 服务端做 query 改写、意图识别和实体抽取，比如品牌、价格段、品类、场景词。
3. 检索层先做向量召回，再结合类目、价格、品牌等结构化过滤。
4. 对候选商品做 rerank，把最相关结果排到前面。
5. 把 top 商品的标题、卖点、属性、价格等信息拼成上下文喂给模型。
6. 大模型生成自然语言回答，前端流式展示，同时渲染商品卡片。

落地效果可以从三方面说：

1. 用户找商品的路径更短。
2. 搜索结果解释性更强，不只是“给你一堆商品”，而是“告诉你为什么推荐”。
3. 对长尾口语 query 的支持更好。

#### 请讲一下 RAG 召回准确率的完整优化过程，以及每一步优化的核心思路。

我会把召回优化分成四步讲：

1. **先定义问题**
   早期的问题往往不是模型“不聪明”，而是召回集合里就没有正确商品，后面的生成再强也没用。

2. **优化 query**
   用户输入很口语化，我们先做 query 改写、同义词归一化、类目补全，让检索 query 更贴近商品表达方式。

3. **优化召回**
   只做纯向量检索容易偏语义相近但业务不对，所以我会加结构化过滤，必要时做混合检索，把语义和关键词能力结合起来。

4. **优化排序**
   召回出来后再做 rerank，把真正相关的商品排到前面，尤其优化 top3、top5 的准确性，因为这最影响用户感知。

“每一步到底怎么落地”，我会这样补：

1. **query 改写阶段**
   把用户口语 query 拆成：
   - 核心商品意图
   - 品牌/价格/类目等硬条件
   - 场景词、卖点词这类软条件

   例如“适合出差、轻薄、续航长的电脑”，会被改写成更贴近商品表达的检索语句，并抽出 `scene=business-trip`、`tag=lightweight`、`tag=long-battery-life` 这类条件。

2. **召回阶段**
   我不会只做纯向量召回，而是“语义召回 + 结构化过滤”：
   - 语义召回解决“说法不一致但意思接近”
   - 结构化过滤解决“库存、价格、类目必须满足”

3. **rerank 阶段**
   rerank model 本质上是精排模型。向量检索先广撒网召回 topK，rerank 再对 `(query, doc)` 做细粒度相关性打分，保证前几条结果真正最贴近用户需求。

4. **生成阶段**
   不会把所有召回结果都塞给 LLM，而是只保留前几条高相关商品，降低噪声、成本和幻觉风险。

如果想让表达更完整，可以补一句：**召回阶段关注“不漏”，重排阶段关注“不乱”，生成阶段关注“不编”。**

#### 你如何进行召回率的测量？

我会先构建评测集，再定义指标。

评测集一般包含：

1. 高频 query
2. 长尾 query
3. 口语化 query
4. 容易混淆的 query

更细一点我会这样做：

1. 先从真实搜索日志和人工构造 query 中抽样。
2. 让业务同学或标注同学给每个 query 标注“哪些商品算相关、相关度几档”。
3. 区分训练/验证/评测集，避免一边调一边测同一批样本。

指标上常看：

1. `Recall@K`：正确商品是否出现在前 K 个结果里。
2. `MRR`：正确结果平均排位是否靠前。
3. `NDCG`：如果相关性有分级，可以衡量排序质量。

线上还会结合点击率、加购率、转化率做业务验证，但这些已经不只是检索指标了。

所以一个更完整的回答是：**离线指标回答“检索质量有没有提升”，线上指标回答“业务效果有没有提升”。**

#### 你为什么选择只用商品标题做向量化，而非全字段？背后的底层逻辑是什么？

核心原因是：**不是字段越多，向量效果就越好。**

商品全字段里会混入很多噪声，比如冗长描述、营销文案、参数堆砌，这些内容可能拉低语义表示的纯度。标题通常是高度压缩后的关键信息，往往最能代表商品的核心语义。

底层逻辑有三点：

1. 向量检索依赖语义表示质量，噪声越多，embedding 越容易偏。
2. 标题字段短且集中，更适合做高质量语义匹配。
3. 结构化属性比如品牌、价格、类目，其实更适合做过滤或排序特征，不一定适合直接拼进向量文本。

如果面试官继续追问“底层逻辑再说细一点”，我会补两点：

1. embedding model 最擅长的是把一段相对凝练的语义文本编码成向量。如果把很长的商品描述、营销文案、参数堆砌全混进去，模型会更难抓住“这到底是个什么商品”这个主语义。

2. 向量索引通常要求同一套语义空间尽量稳定。如果字段内容噪声太大，不仅召回质量会波动，后续 rerank 和生成也会接收到更多无关上下文。

所以我的做法不是“只看标题”，而是“标题负责主语义召回，其他字段负责过滤和补充排序”。如果后续要升级，也更可能是“标题 + 核心卖点摘要”，而不是全字段无脑拼接。

#### 这个项目后续还有哪些优化方向？

我会优先考虑五个方向：

1. 做混合检索，把 keyword 检索和向量检索结合起来。
2. 引入用户行为特征，比如点击、收藏、加购信号，做更强排序。
3. 按品类拆 embedding 策略，不同品类的语义重点不一样。
4. 做多轮对话记忆，让导购不只是单轮搜索。
5. 建立更完整的离线评测和线上 A/B 实验体系。

## 三、技术难点与系统设计

#### 你做这个 Agent 项目的过程中，遇到的最大系统痛点/技术难点是什么？你是怎么解决的？

我觉得最大的痛点是：**任务复杂度上来后，系统不是“不会做”，而是“做得不稳定”。**

根因主要有三个：

1. 任务拆解不稳定。
2. 多 Agent 协作容易重复劳动。
3. 缺少统一的执行观测和调试抓手。

我的解决思路是：

1. 把任务执行链路结构化，明确 planning、execution、aggregation 三个阶段。
2. 给每个子任务定义清晰输入输出，减少 Agent 之间的隐式耦合。
3. 把执行日志、工具入参、结果摘要、失败原因标准化记录下来，形成可回放链路。

这样做之后，系统优化从“拍脑袋调 prompt”变成“基于链路证据做优化”。

#### 针对 Agent 系统模块耦合严重、调试迭代成本高的问题，你有什么成熟的解耦设计方案？

我会从四层解耦：

1. **协议层解耦**
   统一 Agent 间消息格式、工具 schema、结果结构，减少隐式依赖。

2. **能力层解耦**
   把检索、浏览器、代码分析、数据查询都抽成标准能力模块，而不是写死在某个 Agent 里。

3. **流程层解耦**
   把 planning、routing、execution、memory、evaluation 拆成独立模块，便于替换和单测。

4. **观测层解耦**
   单独建设 tracing、日志、回放和评测，不把调试逻辑耦合到主执行逻辑里。

如果让我用一句话总结，就是：**让 Agent 负责决策，让平台层负责能力、协议和观测。**

## 四、大模型与微调认知

#### 你有没有大模型微调相关的实践经验？请详细介绍一下。

如果你的实际经历不是线上大规模微调，建议诚实表达，不要硬编。一个比较稳的回答是：

我有过小规模微调实验经验，但还没有独立负责过超大规模生产级微调系统。我的实践更偏向于用 `LoRA/QLoRA` 做特定任务验证，比如让模型在某个固定领域内更稳定地遵循输出格式，或者提升特定风格问答的一致性。

如果面试官继续问“LoRA/QLoRA 到底是什么”，我会这样解释：

1. **LoRA**
   本质上不是全量改模型参数，而是在原模型权重旁边加一组低秩增量矩阵，只训练这部分小参数，降低训练成本。

2. **QLoRA**
   可以理解成在 LoRA 基础上进一步把底座模型做低比特量化，这样显存占用更低，更适合在有限资源下做实验验证。

3. **为什么选它们**
   因为很多业务场景不值得做全量微调。LoRA/QLoRA 更像低成本验证方案，适合快速判断“这个方向值不值得继续投入”。

我对这类实验的理解是：

1. 先明确微调目标，到底是想增强知识、风格、格式遵循，还是工具调用行为。
2. 准备高质量数据，统一输入输出格式，控制噪声。
3. 选择参数高效微调方式，降低成本。
4. 用验证集评估，不只看 loss，还看业务效果。

如果再往工程层讲，我会补一句：微调后的评估不会只看训练 loss，而是看结构化输出准确率、稳定性、拒答/幻觉情况，以及和 prompt-only 基线相比到底有没有真实收益。

如果面试官追问，你可以强调：**我更清楚微调在工程上的适用边界，不会把所有问题都交给微调解决。**

#### 你觉得大模型微调的核心难点是什么？

我觉得最大的难点不是“把模型训起来”，而是“训完之后真的对业务有稳定收益”。

具体来说有四个难点：

1. 数据质量远比数量更关键，脏数据会直接把模型带偏。
2. 很难定义统一评测标准，尤其是开放生成任务。
3. 微调可能带来能力漂移，某项能力提升了，另一项能力可能下降。
4. 成本高，包括训练、验证、部署和回滚成本。

所以微调更像“重武器”，适合在收益足够明确时再用。

#### prompt 工程和模型微调，你怎么做技术权衡？它们分别适用于什么场景？

我的原则是：**先 prompt，后微调；先低成本验证，再重投入固化。**

Prompt 工程适合：

1. 需求变化快
2. 规则还能靠上下文描述清楚
3. 想快速试错和验证方向

模型微调适合：

1. 输出风格和格式要求长期稳定
2. 单靠 prompt 很难稳定达到目标
3. 有较高频、较固定的业务场景，值得为稳定性投入成本

一句话总结：**prompt 更灵活，微调更重但更可能在长期场景下形成壁垒。**

## 五、行业认知

#### 你觉得 AI 的发展对前端工程师有什么影响？

我觉得影响是双重的。

第一，AI 会替代一部分低门槛、重复性的编码工作，所以前端工程师不能只停留在“写页面”层面。

第二，AI 也会放大优秀前端工程师的价值，因为未来更重要的是：

2. 把 AI 能力产品化，包括流式交互、可视化、可解释性和人机协同。
3. 在真实业务里处理性能、稳定性、安全性和体验问题。

所以我不觉得 AI 会让前端变弱，而是会推动前端往更高层的产品工程能力升级。

#### 你对 AI Native 前端工程师这个岗位是怎么理解的？

我理解的 AI Native 前端，不是“会调一个大模型接口的前端”，而是能围绕 AI 产品形态做完整体验设计和工程落地的人。

他既要懂前端工程，也要理解 AI 系统的特点，比如：

1. 流式输出和状态设计
2. 不确定性结果的交互承接
3. RAG、Tool Calling、Agent 的基本链路
4. 评测、监控和安全兜底

也就是说，这个岗位不是传统前端加一个 AI SDK，而是前端能力和 AI 产品理解深度融合。

## 七、二面高压追问模拟

#### 你怎么证明多 Agent 一定比单 Agent 更好？有没有可能只是系统变复杂了？

这是个很关键的问题，我不会直接说“多 Agent 一定更好”，而是会说：**多 Agent 只在复杂任务上更可能更好，在简单任务上反而可能是负收益。**

证明方式不是靠主观感觉，而是做分层对比：

1. 同一批任务集，对比单 Agent 和多 Agent 的完成率。
2. 对比平均耗时、token 成本和失败类型分布。
3. 观察复杂任务下是否真的减少了路径漂移和工具误用。

如果对比结果显示简单任务收益不明显，我会坚持简单任务走单链路，复杂任务再升级多 Agent。这样才是工程化思路，而不是为了架构而架构。

#### 多 Agent 会引入额外调度和通信成本，你们是怎么判断这笔成本值得的？

我会把这笔成本拆成三类：

1. **延迟成本**：多一次拆解、多一次聚合，天然会增加时延。
2. **token 成本**：Agent 间通信和结果摘要都会增加上下文消耗。
3. **系统复杂度成本**：调试、监控、回放、评测都更难。

判断是否值得，核心看它换来的收益是否更大。比如在复杂任务里，如果多 Agent 能明显提高成功率、减少严重错误，那它是值得的；如果只是让答案“看起来更高级”，但收益不明显，那就不值得。

所以我更看重“按任务复杂度分层启用”，而不是全量默认多 Agent。

#### 如果 Memory 误召回了一条错误经验，最坏会发生什么？你怎么防止它污染后续结果？

最坏情况是，系统会在一个错误前提上越走越远，而且因为这条经验来自“历史成功路径”，模型还可能更相信它，这种错误比普通 hallucination 更隐蔽。

我的防护思路有四层：

1. **检索侧降噪**：不是只看语义相似，还要看任务类型、版本、来源可信度。
2. **注入侧限量**：不要一次注入太多历史经验，避免旧经验压制当前事实。
3. **执行侧验证**：Memory 只能作为候选参考，必须允许当前工具结果推翻它。
4. **写回侧审慎**：失败路径不能轻易沉淀成长期 Memory。

所以关键不是“Memory 命中率高”，而是“命中后仍然可证伪”。

#### 既然规划不稳定，为什么你们不直接做微调，而是先做架构和流程优化？

因为我们先判断问题的性质。如果问题主要来自系统结构，比如上下文过长、能力边界混乱、可观测性缺失，那微调并不能从根上解决。

我会把顺序定成：

1. 先解决系统层问题，比如任务拆分、上下文管理、工具协议统一。
2. 再做 prompt 和路由优化。
3. 如果最后仍然存在稳定的模式性缺陷，再评估是否值得微调。

这样做的原因是，微调成本高、回滚重，而且如果系统结构本身有问题，微调只是把问题包起来，不是真的解决了。

#### 你在 RAG 里为什么一定要加 rerank？只靠向量召回不行吗？

只靠向量召回不是不行，但在业务场景里通常不够稳。因为向量召回更像粗筛，它擅长把语义接近的候选捞出来，但不保证前几条就是最符合当前 query 意图的结果。

Rerank 的价值主要在三点：

1. 进一步理解 query 和候选之间的细粒度相关性。
2. 提升 top 结果质量，因为用户感知最强的是前几条。
3. 为生成阶段提供更干净的上下文，减少模型“看了很多但抓错重点”的问题。

所以我的理解是：**召回解决别漏，rerank 解决别乱。**

#### 你为什么不直接把商品全字段喂给模型，让模型自己判断？

因为“信息更多”不等于“结果更好”。全字段直接喂给模型会带来三个问题：

1. 噪声太多，模型可能被无关信息干扰。
2. token 成本和延迟会明显上升。
3. 可控性变差，很难稳定复现为什么这次结果好、下次结果差。

更合理的工程做法是分层使用信息：标题承担主语义召回，结构化属性做过滤和补充排序，最后只把最相关的少量信息交给模型生成。

#### 如果现在只给你一周时间提升 Agent 系统效果，你最优先做哪三件事？

如果只有一周，我不会追求大而全，而会优先做最能闭环的三件事：

1. **补评测集**
   没有评测，优化就没有方向。

2. **补链路可观测**
   至少能看清 planning、tool use、memory 命中、失败原因。

3. **先做高频问题优化**
   优先解决重复探索、上下文膨胀、工具误用这类最常见且可量化的问题。

因为在时间有限时，最重要的是把“问题发现 -> 优化 -> 验证”这个闭环跑起来。

#### 这道 Scheduler 题如果任务执行失败了，会不会把整个队列卡死？怎么改？

是的，如果只考虑成功路径，失败时没有正确释放并发槽位，就可能把后续队列卡住。

更稳的写法是让 `task` 支持 Promise，并在 `finally` 里统一做：

1. `running--`
2. `runNext()`
