## 一、微前端 / 网络 / React / 跨端

### 1. qiankun 这个微前端架构，你对微前端架构有了解吗？

我了解。`qiankun` 本质上是在 `single-spa` 的路由分发能力之上，补了一套更完整的微前端运行时，包括应用注册、生命周期管理、JS 沙箱、样式隔离、资源预加载和通信能力。

一句话概括微前端，它就是把多个相对独立的前端应用，按统一的接入协议组合到一个主应用里，让团队可以独立开发、独立部署，但最终对用户呈现为一个整体系统。

我一般会从 4 个维度来理解它：

1. **应用组织**
   主应用负责路由分发、应用注册、资源加载和挂载卸载；子应用保留自己的技术栈和构建产物。
2. **运行时隔离**
   避免子应用之间互相污染，核心就是 `JS sandbox` 和 `CSS isolation`。
3. **通信机制**
   主子应用之间通常通过 `props`、全局状态、事件总线或 URL 同步进行通信。
4. **工程收益与成本**
   收益是拆团队、拆仓库、拆部署；成本是运行时复杂度更高、调试更复杂、性能和隔离要额外治理。

适用场景通常是中后台、大型平台、多团队并行开发。如果只是单团队中小项目，我一般不会为了“技术酷炫”硬上微前端，因为复杂度会上升。

### 2. 来讲一下微前端它的一些实现方案？

常见方案可以分成 4 类：

1. **路由分发型**
   典型是 `single-spa`、`qiankun`。主应用根据路由决定挂载哪个子应用。优点是成熟、侵入性相对低；缺点是运行时治理复杂。
2. **iframe 型**
   最强隔离，天然有独立的 `window`、`document`、`CSS`、`JS` 环境。缺点是通信麻烦、体验差、SEO 弱、样式和路由整合差。
3. **Web Components 型**
   通过自定义元素和 Shadow DOM 做封装。隔离性不错，但工程生态、框架兼容、状态共享和大型应用治理不一定够顺手。
4. **Module Federation 型**
   更偏构建时 / 运行时模块共享。它适合多个应用共享组件、页面或能力模块，不一定完整等价于“应用级微前端”，但经常和微前端一起使用。

实际落地里最常见的是：

- 业务级整合：`qiankun`
- 强隔离低耦合：`iframe`
- 组件级共享：`Module Federation`

### 3. 主应用和子应用沙箱隔离是怎么实现的？

核心是 **让子应用以为自己操作的是全局对象，实际上操作的是一个“代理出来的假全局对象”**。

以 `qiankun` 为例，常见有两类沙箱：

1. **快照沙箱**
   进入子应用时记录真实 `window` 的快照，退出时把变更恢复。优点是简单；缺点是性能一般，更适合单实例。
2. **代理沙箱**
   用 `Proxy` 包一层 fakeWindow。子应用读取属性时，优先从 fakeWindow 读，读不到再透传到真实 `window`；写属性时写到 fakeWindow 或记录变更集合，不直接污染原生全局对象。

这样做的结果是：

- A 子应用写的全局变量，不会真正污染 B 子应用
- 子应用卸载时，可以按变更记录把副作用清理掉
- 主应用仍然能控制哪些全局能力允许暴露给子应用

不过要注意，沙箱不是绝对安全隔离。像 DOM、原生浏览器 API、副作用定时器、全局事件监听这些都还要额外清理。

### 4. A 应用会影响到 B 子应用，切换时需要把 window 还原，这个怎么理解？

这个问题本质是在问 **全局副作用回滚**。

如果没有沙箱，A 应用执行了这类代码：

- `window.xxx = 123`
- 重写 `window.fetch`
- 注册全局事件
- 注册定时器

那切到 B 应用时，这些副作用还在，就会污染 B。

所以切换时要做两件事：

1. **恢复全局对象状态**
   比如快照沙箱会把进入前的 `window` 状态记下来，退出时恢复；代理沙箱则是销毁该子应用的 `proxy window` 作用域。
2. **清理副作用**
   包括 `setTimeout`、`setInterval`、`addEventListener`、样式节点、动态 script、订阅关系等。

也就是说，真正要“还原”的不只是 `window` 里的字段，而是整套运行时副作用。

### 5. 微前端像这种代理沙箱，它是单实例还是多实例？window 这一块怎么实现？

`qiankun` 的 `ProxySandbox` 可以支持多实例。每个子应用都可以有自己的代理对象，不共享同一个 fakeWindow。

实现思路大致是：

1. 创建一个 `fakeWindow`
2. 用 `Proxy(fakeWindow, handler)` 包起来
3. `get` 时优先取 fakeWindow，没有再去真实 `window`
4. `set` 时只写 fakeWindow 或写入当前应用的变更记录
5. `has`、`ownKeys`、`getOwnPropertyDescriptor` 等也要做代理，保证像 `in`、`for...in`、`Object.keys` 这类行为尽量接近真实 `window`

它不是把真实 `window` 复制一份，而是做 **读透传、写隔离、卸载可回收**。

单实例更常见于早期快照沙箱；多实例主要依赖 `Proxy` 沙箱。

### 6. CSS 怎么隔离？

常见做法有 4 种：

1. **命名空间约定**
   给每个子应用统一加前缀，比如 `.app-a-button`。简单，但靠人工约束。
2. **CSS Modules**
   编译后自动生成局部类名，适合组件级隔离。
3. **Shadow DOM**
   天然样式作用域隔离，最强，但和现有框架、全局主题、弹层挂载可能有兼容成本。

   `Shadow DOM` 是浏览器 Web Components 体系里的能力。它的核心作用是：**给某个宿主节点挂一棵“影子子树”，让这棵子树默认和外部 DOM / CSS 作用域隔离。**

   你可以把它理解成“组件自己的私有 DOM 世界”：

   - 普通 DOM：全页面共享一个大作用域
   - Shadow DOM：某个组件拥有局部作用域

   一个最小例子：

   ```js
   const host = document.querySelector('#host');
   const shadowRoot = host.attachShadow({ mode: 'open' });

   shadowRoot.innerHTML = `
     <style>
       p {
         color: red;
       }
     </style>
     <p>Hello Shadow DOM</p>
   `;
   ```

   这里：

   - `host` 是宿主节点
   - `shadowRoot` 是影子根节点
   - 里面的 `p { color: red }` 默认只作用在这个 shadow tree 内

   它为什么适合做样式隔离？

   因为浏览器做 CSS 选择器匹配时，会把 Shadow Tree 视为一个相对独立的作用域边界。外部样式不会像普通 DOM 那样直接无差别渗透进去，内部样式也不会轻易污染外部。

   1. 样式天然隔离，封装性强
   2. 组件内部结构不容易被外部选择器误伤
   3. 特别适合组件库、微前端、第三方嵌入组件

   缺点：

   1. 全局主题、reset 样式不会自动生效
   2. 弹窗如果挂到 `document.body`，会逃离 Shadow 作用域
   3. 调试、样式覆盖、主题透传会更复杂
   4. 某些第三方库默认按普通 DOM 工作，兼容成本会更高

   `mode: 'open'` 表示外部还能通过 `element.shadowRoot` 拿到影子根；`closed` 表示外部拿不到引用，但浏览器内部仍然存在这棵树。
4. **运行时样式隔离**
   比如 `qiankun` 的 `strictStyleIsolation` 会基于 Shadow DOM；`experimentalStyleIsolation` 会给选择器自动加作用域前缀。

真实项目里还要注意：

- `body`、`html`、`:root` 这种全局样式最容易互相污染
- 弹窗如果挂到 `document.body`，也可能逃出样式作用域
- 设计系统和全局主题变量要提前统一，否则每个子应用都改一套会冲突

### 7. Web Worker 是用来干什么的？

`Web Worker` 用来把耗时计算放到后台线程执行，避免阻塞主线程的渲染和交互。

`Web Worker` 一般这样工作：

1. 主线程创建 worker
2. 主线程通过 `postMessage` 发消息
3. worker 在线程里处理计算
4. worker 再通过 `postMessage` 把结果发回主线程

最小示例：

**main.js**

```js
const worker = new Worker('./worker.js');

worker.postMessage({ type: 'sum', payload: [1, 2, 3, 4, 5] });

worker.onmessage = (event) => {
  console.log('worker result:', event.data);
};
```

**worker.js**

```js
self.onmessage = (event) => {
  const { type, payload } = event.data;

  if (type === 'sum') {
    const result = payload.reduce((a, b) => a + b, 0);
    self.postMessage(result);
  }
};
```

运行机制要点：

1. Worker 线程不能直接操作 DOM
   因为 DOM 仍由主线程管理。
2. 主线程和 Worker 是两个独立执行环境
   不是共享同一个 JS 作用域。
3. 它们默认通过消息通信
   浏览器会做结构化克隆后再传数据。
4. 某些大对象可以用 `Transferable`
   比如 `ArrayBuffer` 可以转移所有权，避免额外拷贝。

适合场景：

- 大量 CPU 计算
- 图片 / 音视频处理
- 大文件分片和哈希
- 富文本解析

不适合场景：

- 很轻的小计算
- 强依赖 DOM 的逻辑
- 需要非常高频双向细粒度通信的逻辑

典型场景：
- 大数据量排序、过滤、搜索
- 图片压缩、音视频处理
- 富文本解析
- 大文件分片、哈希计算
- AI 前端场景中的流数据预处理

浏览器主线程主要负责：

- 执行 JS
- 布局和绘制
- 响应用户输入

如果大量 CPU 计算都堆在主线程，页面就会卡顿。Worker 的价值就是把“重计算”从主线程挪出去。

### 9. 网络请求不做 Web Worker 处理，请求过程中它会放在主线程吗？

不会把“网络传输过程”卡在主线程上。

更准确地说：

1. 你在主线程调用 `fetch` 或 `XMLHttpRequest`
2. 浏览器把网络请求交给底层网络栈去处理
3. 请求等待响应期间，主线程可以继续做别的事
4. 响应回来后，再把回调结果投递回事件循环

- **网络等待本身** 不会持续占用主线程
- **发起请求的那一刻** 和 **响应回调执行的那一刻** 还是在主线程

### 10. 异步回调会影响吗？

会，但影响点在 **回调执行阶段**，不在“等待阶段”。

比如请求完成后，`.then()` 回调里如果做了这些事：

- 大量 JSON 转换
- 大列表 diff
- 复杂同步计算

那主线程一样会卡。

所以异步不等于没有性能问题。异步只是“等待 IO 不阻塞”，但回调中的同步逻辑仍然可能卡主线程。

### 12. Promise.allSet 这一块里面是一些什么样的逻辑呢？

这里通常是在问 `Promise.allSettled`。

它的语义是：**等所有 Promise 都结束，不管成功还是失败，最后统一返回结果数组**。

返回结果类似这样：

```js
[
  { status: 'fulfilled', value: 1 },
  { status: 'rejected', reason: new Error('fail') }
]
```

和 `Promise.all` 的区别是：

- `Promise.all`：有一个失败就整体 reject
- `Promise.allSettled`：不短路，全部执行完再汇总

如果面试官继续追问“并发控制”，要说明：

`allSettled` 本身 **不负责限制并发数**，它只是负责“收集所有结果”。  
“5 个为一批”那是并发池 / 调度器逻辑，不是 `allSettled` 自带的能力。

### 14. 请求池的概念？

请求池就是一个 **有并发上限的任务调度器**。

它通常有 3 个核心状态：

1. `pendingQueue`
   等待执行的请求队列
2. `runningCount`
   当前正在执行的请求数
3. `limit`
   最大并发数

调度规则很简单：

- 只要 `runningCount < limit`，就从等待队列里取任务启动
- 某个任务完成后，`runningCount--`
- 再尝试补下一个任务进去

这样就能既保护浏览器和服务端，又保持较高吞吐。

### 15. 一堆请求到达，是直接同一批次 push 到候选池里面吗？

一般是的。常见做法是：

1. 业务把一组待执行请求都包装成任务函数
2. 一次性 push 到等待队列
3. 调度器按并发上限逐个拉起

重点不是“分几批 push”，而是调度器能否：

- 控制最大并发
- 支持失败重试
- 支持取消
- 支持优先级

### 16. HTTP/1 和 HTTP/2 的区别

最核心的区别是：`HTTP/2` 不再是纯文本协议，而是二进制分帧协议，并且支持在一条连接上并行复用多个请求流。

主要差异：

1. **报文格式**
   `HTTP/1.x` 是文本格式；`HTTP/2` 是二进制帧。
2. **并发能力**
   `HTTP/1.1` 虽然支持长连接，但同一 TCP 连接里仍然容易受到队头阻塞影响；`HTTP/2` 支持多路复用。
3. **头部压缩**
   `HTTP/2` 用 `HPACK` 压缩头部，减少重复 header 开销。
4. **服务器推送**
   `HTTP/2` 支持 `server push`，虽然现在实际使用不算多。
5. **优先级与流控制**
   `HTTP/2` 可以按流设置优先级，并有更细粒度的流量控制。

这里的“二进制分帧”可以详细理解为：

`HTTP/1.1` 更像是在传“整段文本报文”，而 `HTTP/2` 把请求 / 响应拆成很多更小的二进制帧（frame）去传。

几个核心名词：

- `frame`：最小传输单元，帧
- `stream`：一个请求 / 响应对应的一条逻辑流
- `connection`：底层那条 TCP 连接

- 一条 `connection` 上可以并存多个 `stream`
- 每个 `stream` 又由很多 `frame` 组成

比如一个请求可能被拆成：

- `HEADERS frame`：放请求头
- `DATA frame`：放请求体

响应也是一样：

- 先回响应头的 `HEADERS`
- 再分多次回响应体的 `DATA`

为什么要拆成 frame？

因为拆小后，多个请求的数据就能在同一条连接里交错发送，而不是必须“一个完整发完再发下一个”。这就是多路复用的基础。

### 17. 多路复用和长连接是什么？

**长连接** 是指一次 TCP 连接建立后，不是一个请求用完就断，而是多个请求复用同一条连接。

**多路复用** 是指在一条连接上，多个请求 / 响应可以并行交错传输，不需要“前一个完全结束，后一个才能开始”。

“只有一条连接怎么并行处理”，关键就在于：**并行的是逻辑流，不是说再建多条 TCP。**

在 `HTTP/2` 里：

1. 每个请求都会拿到一个 `stream id`
2. 请求和响应会被拆成多个 frame
3. 每个 frame 都带着自己所属的 `stream id`
4. 接收方收到后，再按 `stream id` 把这些 frame 重新组装

所以同一条连接上的传输顺序可能像这样：

```txt
HEADERS(stream 1)
HEADERS(stream 3)
DATA(stream 1)
DATA(stream 3)
DATA(stream 1)
```


### 18. HTTP/2 除了多路复用还有哪些新特性？

除了多路复用，还有这些高频点：

1. **二进制分帧**
   把消息拆成更小的 frame，便于并行和控制。
2. **头部压缩**
   用 `HPACK` 降低重复 header 开销。
3. **流优先级**
   可以告诉服务端哪些资源更重要。
4. **流量控制**
   控制每个流和整条连接的数据发送速度。

如果继续问“具体怎么用、底层原理是什么”，可以这样答：

#### 1. `HPACK` 头部压缩

HTTP 头里经常有大量重复字段，比如：

- `cookie`
- `user-agent`
- `accept`
- `:method`
- `:path`

如果每个请求都完整重复发送，开销很大。  
`HPACK` 的思路是：

1. 通信双方维护一张头字段表
2. 已出现过的头字段，下次可以只传索引
3. 新值则按规则做编码和增量传输

所以它不是普通 gzip，而是 **针对 HTTP 头结构专门设计的压缩机制**。

#### 2. 流优先级

浏览器可以告诉服务端：

- CSS 更重要
- 首屏接口更重要
- 某张大图可以晚一点

也就是给不同 stream 一个相对优先级，帮助服务端更合理安排发送顺序。

#### 3. 流量控制

如果接收方处理不过来，发送方发太快就会把缓冲区压爆。  
所以 HTTP/2 在应用层对：

- 单个 stream
- 整条 connection

都维护接收窗口。接收方会告诉发送方“我还能接收多少字节”，发送方必须在窗口范围内发。
5. **Server Push**
   服务端可以主动推送它判断客户端即将需要的资源。

### 19. HTTP/3 这一块有了解吗？

有。`HTTP/3` 最大的变化不是 HTTP 语义层，而是 **底层传输从 TCP 换成了 QUIC（基于 UDP）**。

它主要解决两个问题：

1. **TCP 层队头阻塞**
   `HTTP/2` 虽然应用层多路复用，但底层还是单条 TCP，一旦丢包，整个连接都可能受影响。
2. **连接建立慢**
   `QUIC` 把传输层和 TLS 握手整合，连接建立更快，支持 `0-RTT` / `1-RTT`。

> HTTP/2 解决了应用层复用问题，但没解决 TCP 级别队头阻塞；HTTP/3 用 QUIC 把这个问题进一步往下层解决了。

这里你问的几个名词，可以拆开理解：

#### 1. `RTT` 是什么？

`RTT = Round Trip Time`，往返时延。  
就是一个数据从客户端发到服务端，再回到客户端，这一来一回所需的时间。

#### 2. `1-RTT` 是什么？

表示至少需要 **1 个往返** 才能把握手和安全上下文建立起来，再开始正式发送应用数据。

#### 3. `0-RTT` 是什么？

不是“完全不花时间”，而是：

> 当客户端之前已经和服务端建立过会话，下次重连时，可以在握手还没完全结束前，先带着早期数据发部分请求。

优点是更快。  
风险是早期数据可能被重放，所以 `0-RTT` 不适合支付、转账这类有副作用请求。

#### 4. TCP 级别队头阻塞是什么？

`HTTP/2` 虽然应用层已经多路复用，但底层还是 TCP。  
而 TCP 是“按顺序交付字节流”的协议。

这意味着如果某个 TCP 包丢了：

- 即使后面的包已经到了
- 接收方也不能把后面的数据先完整交给上层
- 必须等丢的那个包重传补齐

结果就是：

- stream A 丢了一个底层包
- stream B、C 后续数据虽然到了
- 但整条 TCP 字节流都被这个缺口卡住

这就叫 **TCP 级别队头阻塞**。

QUIC 能改善，是因为它不是把所有上层流强行塞进一条必须全局有序的 TCP 字节流里，而是在 UDP 之上自己维护多条逻辑流，某一条流丢包不一定拖死其他流。

### 20. HTTP/2 一次连接可以并行多个请求，建立连接步骤是什么样子？

如果是 `HTTPS + HTTP/2`，典型步骤是：

1. DNS 解析拿到 IP
2. 建立 TCP 三次握手
3. 进行 TLS 握手
4. 通过 `ALPN` 协商应用层协议，确认使用 `h2`
5. 建立好这条连接后，在其上创建多个 `stream`
6. 多个请求分别拆成 frame，在同一连接里交错传输

#### 第一步：先把连接打通

浏览器先做：

1. DNS 解析
2. TCP 三次握手
3. TLS 握手

#### 第二步：协商“这条连接上跑什么应用层协议”

`ALPN = Application-Layer Protocol Negotiation`，应用层协议协商。

客户端会告诉服务端：

- 我支持 `http/1.1`
- 我也支持 `h2`

服务端回复：

- 好，这次我们用 `h2`

这一步完成后，双方才知道后面要按 HTTP/2 的规则来传。

#### 第三步：开多个 stream

连接建好后，不是再给每个请求额外建一条 TCP，而是：

- 请求 A 分一个 `stream id`
- 请求 B 再分一个 `stream id`
- 它们都跑在同一条连接上

#### 第四步：每个请求拆成 frame

比如：

- 请求 A：首页 HTML
- 请求 B：首屏 CSS

它们会被拆成不同 frame，并标记不同的 `stream id`：

```txt
HEADERS(stream 1)
HEADERS(stream 3)
DATA(stream 1)
DATA(stream 3)
DATA(stream 1)
```

#### 第五步：接收方按 `stream id` 重组

收到这些 frame 后，接收方会看它属于哪个 stream，再把同一个 stream 的数据重新拼成完整请求 / 响应。

所以并行请求不是建立多条 TCP，而是在一条已经握好的连接上开多个流。

### 21. 三次握手第二次丢失了会怎么样？

第二次握手是服务端返回的 `SYN + ACK`。

如果这一包丢了：

1. 客户端收不到服务端响应
2. 客户端会认为握手没完成，超时后重传第一次握手的 `SYN`
3. 服务端如果再次收到 `SYN`，会再次回 `SYN + ACK`
4. 直到客户端收到并发出第三次握手 `ACK`

服务端在这段时间一般处于 `SYN_RCVD` 状态，等待客户端最后确认。

这里的名词全称是：

- `SYN = Synchronize`
- `ACK = Acknowledgment`
- `SYN_RCVD = SYN Received`

详细解释：

#### 1. `SYN`

TCP 发起建连时使用的标志位，表示：

> 我想和你建立连接，并同步我的初始序列号。

#### 2. `ACK`

确认标志位，表示：

> 你前面发给我的内容，我已经收到了。

#### 3. `SYN + ACK`

服务端收到客户端的 `SYN` 后，会回一个同时带 `SYN` 和 `ACK` 的包，意思是：

> 你的建连请求我收到了；这是我的初始序列号；同时我确认收到了你的请求。

#### 4. `SYN_RCVD`

这是服务端 TCP 状态机里的一个状态，表示：

> 我已经收到客户端的建连请求，也把 `SYN + ACK` 发出去了，现在正在等客户端最后那个 `ACK`。

所以第二次握手丢了时：

- 客户端：还觉得连接没建好
- 服务端：已经处于半连接状态，在等最终确认

### 22. 哪一边会超时重试？会进行什么处理？

要看哪一步丢。

如果第二次握手丢了：

- **客户端** 因为没收到 `SYN + ACK`，会超时重传 `SYN`
- **服务端** 也可能重传 `SYN + ACK`，因为它已经收到客户端 `SYN`，正在等第三次握手确认

也就是说，两边都可能有重传逻辑，但触发原因不同：

- 客户端重传：认为连接还没建立
- 服务端重传：等最终 `ACK` 等不到

如果重试多次仍失败，最终会放弃并释放半连接状态。

### 23. HTTPS 在 HTTP 上面做了哪些处理？

可以理解为：**HTTP + TLS**。

它在 HTTP 之上新增了安全层，主要提供 4 个能力：

1. **加密**
   防止明文被窃听。
2. **完整性校验**
   防止传输过程中被篡改。
3. **身份认证**
   通过证书证明服务端身份，防止连错人。
4. **密钥协商**
   通过握手安全地协商会话密钥。

典型流程是：

1. 客户端发起 TLS 握手
2. 服务端返回证书
3. 客户端校验证书链、域名、有效期
4. 双方协商加密套件和会话密钥
5. 后续 HTTP 数据都走加密通道

### 27. Astro，简单说下它和我们日常用到的 SSR 的一些区别？

`Astro` 的核心理念不是“所有页面都 SSR”，而是 **默认尽量少发 JS，把更多内容在构建时或服务端直接产出成静态 HTML**，强调“islands architecture”。

和常规 SSR 框架相比，差异主要有：

1. **默认策略不同**
   很多 SSR 框架默认是页面级 hydration；`Astro` 默认是静态优先、局部激活。
2. **客户端 JS 体积更小**
   只有声明为需要交互的岛屿组件才会下发 JS。
3. **跨框架整合能力更强**
   一个页面里可以混用 `React`、`Vue`、`Svelte` 组件。
4. **更适合内容站**
   博客、官网、营销页、文档站是 Astro 的强项。

所以 Astro 更像是在回答“并不是所有页面都值得完整 hydration”。

### 28. Astro 怎么做组件通信呢？每个组件都单独监听吗？

不是每个组件都单独监听全局事件。Astro 里的通信要分两种情况：

1. **纯 Astro 组件 / 服务端渲染组件**
   这类组件本身不在浏览器里常驻运行，更多是通过 `props` 传递数据。
2. **客户端激活组件（island）**
   它们在浏览器端真正运行，可以用各自框架的通信方式，比如 React Context、Vue provide/inject、事件总线、共享 store。

如果多个 island 之间要通信，常见方式是：

- URL / query 参数
- 自定义事件
- 浏览器全局 store
- `localStorage` / `BroadcastChannel`
- 后端持久化状态

### 29. 复杂数据 URL 不方便传，有什么其他方案？

如果数据比较复杂，我一般按“作用域”和“是否需要持久化”来选：

1. **同页面局部状态**
   直接用组件状态或父子 `props`
2. **跨组件共享**
   用全局状态，比如 `Redux`、`Zustand`、`Pinia`
3. **跨路由但不想暴露在 URL**
   可以用路由状态、`sessionStorage`
4. **需要刷新后保留**
   用 `localStorage`、IndexedDB、后端存储

   这里的 `IndexedDB` 可以详细解释一下：

   `IndexedDB` 是浏览器提供的一套 **本地结构化数据库**。  
   它不是简单的字符串存储，而是更像浏览器内置的小型 NoSQL 数据库。

   和 `localStorage` 对比：

   - `localStorage`：同步 API、只能存字符串、容量较小
   - `IndexedDB`：异步 API、可存结构化对象、容量更大、支持索引和事务

   适合场景：

   - 离线缓存
   - 草稿箱
   - 大列表本地缓存
   - 搜索索引
   - 文件 / Blob 缓存

   > `IndexedDB` 是浏览器里适合存“大量结构化数据”的本地数据库，不只是一个更大的 `localStorage`。
5. **更安全或更大数据量**
   直接把关键数据存后端，只在 URL 里传一个 id

### 30. Astro 是跨框架的，React 的状态库 Vue 不好取，有什么其他方案吗？

对，跨框架场景下不要强依赖某个框架私有状态体系。

如果你追问“`Astro` 到底是什么”，可以这样理解：

`Astro` 不是“React 的替代品”，也不是“只和 React 对标的另一个 UI 框架”。  
它更准确的定位是：

> 一个偏内容站和多框架整合的 Web 框架，核心理念是 `islands architecture`。

什么叫 `islands architecture`？

意思是：

- 页面大部分内容尽量静态输出
- 只有真正需要交互的局部区域才下发 JS 并激活

传统 SPA / SSR 框架更常见的是“整页都交给 React 或 Vue 接管”；  
而 Astro 的默认思想是“静态优先，局部交互再激活”。

所以 Astro 可以：

- 用来做完整站点
- 支持 React / Vue / Svelte 等组件混用
- 但自己不是某个单一 UI 库

它更像“页面编排层 + 岛屿激活层”，特别适合：

- 官网
- 博客
- 文档站
- 营销页
- 内容很多、交互点局部存在的页面

更通用的做法有：

1. **框架无关的外部 store**
   比如自己实现一个基于 `EventTarget` 或 `observable` 的 store。
2. **浏览器原生通信**
   比如 `CustomEvent`、`BroadcastChannel`、`localStorage` 事件。

   这些名词可以详细解释一下：

   #### `CustomEvent`

   `CustomEvent` 是浏览器原生提供的“自定义 DOM 事件”能力。  
   你可以像监听 `click` 一样，自己定义一个事件名并携带数据。

   ```js
   window.addEventListener('user-login', (event) => {
     console.log(event.detail);
   });

   window.dispatchEvent(
     new CustomEvent('user-login', {
       detail: { userId: 123 },
     }),
   );
   ```

   适合场景：

   - 同页面
   - 同窗口
   - 不同组件 / 不同框架之间做轻量通信

   #### `BroadcastChannel`

   `BroadcastChannel` 是浏览器提供的 **同源多标签页广播通信机制**。

   ```js
   const channel = new BroadcastChannel('app-channel');

   channel.postMessage({ type: 'logout' });

   channel.onmessage = (event) => {
     console.log(event.data);
   };
   ```

   只要多个同源页面监听同一个频道名，就能互相广播消息。

   适合场景：

   - 多标签页同步登录态
   - 多标签页同步主题
   - 一处退出，全部页面同步退出

   #### `localStorage` 事件

   当一个标签页修改 `localStorage` 时，其他同源标签页会收到 `storage` 事件：

   ```js
   window.addEventListener('storage', (event) => {
     console.log(event.key, event.newValue);
   });
   ```

   注意：

   - 当前写入的页面自己一般收不到
   - 其他同源标签页能收到

   所以它也能做跨标签页同步，但语义和体验不如 `BroadcastChannel` 直接。
3. **服务端状态**
   把共享状态收口到后端，通过请求或 SSE / WebSocket 同步。
4. **DOM 作为桥**
   某些场景可以通过属性、dataset、自定义元素事件通信。

如果是跨框架岛屿通信，我会优先选 **框架无关 store + 事件机制**，而不是硬把 React store 塞给 Vue。

### 31. Astro 它能够共享它父节点一些 context、props 吗？

`props` 可以，因为父级在渲染子组件时本来就可以显式传参。

但 `context` 要分情况：

- **同一框架树内**：可以走该框架自己的 context 机制
- **跨框架**：通常不能直接共享，因为 React Context、Vue provide/inject 不是同一套运行时

所以 Astro 里要区分：

1. 组件组合层面的父子传值：`props`
2. 框架运行时上下文：只在各自框架边界内成立

### 32. React 的 hooks 是怎么实现的？React 底层怎么来识别这些 hook？

React Hooks 的本质是：**React 在函数组件对应的 Fiber 上，按调用顺序维护一组 Hook 状态节点**。

关键点有两个：

1. **按顺序调用**
   React 不是靠名字识别 `useState`、`useEffect` 对应哪个状态，而是靠“第几个 Hook”。
2. **绑定到 Fiber**
   当前组件对应的 Fiber 节点上会挂一条 Hook 链表，更新时按同样顺序依次读取。

这也是为什么 Hooks 不能写在条件判断里。因为一旦顺序变化，React 就会把后面的 Hook 槽位对错位置。

### 33. useEffect 执行机制是什么时候？

`useEffect` 是 **提交 DOM 之后的异步副作用**。

更准确地说，一次更新分两大阶段：

1. `render` 阶段
   计算新的 React Element 树，不能做副作用
2. `commit` 阶段
   把变更提交到真实 DOM

`useEffect` 会在浏览器完成绘制后，再异步执行。所以它不会阻塞首屏绘制。
- `useLayoutEffect`：在 DOM 提交后、浏览器绘制前同步执行
- `useEffect`：在绘制后执行

### 34. React 渲染会先更新一次，有哪些阶段？

React 更新可以概括成 3 段：

1. **调度阶段**
   收集更新，分配优先级，决定何时开始工作。
2. **render 阶段**
   递归执行组件，生成新的 Fiber 树，做 diff，标记副作用。
3. **commit 阶段**
   把变更真正同步到宿主环境，比如 DOM 更新、ref 赋值、生命周期 / Effect 执行。

其中：

- `render` 可中断
- `commit` 不可中断

### 35. React Fiber 的工作原理？为什么会有 Fiber？什么时候可以中断？做了什么改造让它可以随时中断？

`Fiber` 可以理解成 React 自己实现的一套 **可中断的任务节点结构 + 调度系统**。

为什么会有它？因为旧的同步递归栈模型有个问题：

- 一旦开始渲染整棵组件树，中途停不下来
- 大树更新时容易长时间占用主线程，造成卡顿

Fiber 的改造点主要有：

1. **把递归调用改成可恢复的链式节点遍历**
   每个 Fiber 节点保存父子兄弟关系，React 可以做完一个节点就暂停。
2. **拆分 render 工作**
   不再一次性同步跑完整棵树，而是按时间片分段执行。
3. **引入优先级调度**
   高优先级更新可以插队，例如用户输入优先于低优先级列表更新。

什么时候可以中断？

- 在 `render` 阶段可以
- 在时间片耗尽时可以
- 遇到更高优先级任务时可以

为什么能恢复？

因为 Fiber 节点本身保存了继续执行所需的上下文，不依赖原生调用栈。

### 36. Taro 和 React Native 简单讲一下这两个方案如何实现跨端？底层原理了解吗？最后怎么渲染？是 JS 层做还是 Native 层做？

**Taro：编译适配型**

- 开发时写一套类似 React 的代码
- 编译到不同目标平台，比如微信小程序、H5、React Native
- 核心是“语法层统一 + 编译期转换 + 平台 API 适配”

它最终渲染谁来做，要看目标端：

- 编到 H5：浏览器渲染
- 编到小程序：小程序运行时渲染
- 编到 RN：走 RN 渲染链路

**React Native：运行时桥接型**

- JS 代码在 JS 引擎里执行
- UI 最终不是 HTML，而是映射为原生控件
- 布局一般由 `Yoga` 计算，然后交给 Native 渲染层生成真实视图

> Taro 更像“编译到各个平台”；React Native 更像“JS 驱动原生 UI”。

### 38. React Native 和 App 里写个 WebView 有什么区别？为什么要 React Native？Bridge 这种桥接法是怎么连接前端和 Native 的？

`WebView` 本质上还是网页，只是跑在 App 容器里。  
`React Native` 则是 JS 写业务逻辑，但 UI 组件最终落成原生控件。

核心区别：

1. **渲染结果不同**
   WebView 渲染的是 DOM/CSS；RN 渲染的是原生 View。
2. **性能特征不同**
   RN 在交互、手势、复杂动画、系统能力接入上通常比 WebView 更强。
3. **系统能力接入方式不同**
   WebView 常通过 JSBridge 调 Native；RN 本身就有成熟的 Native Module 机制。

为什么要 RN？
- 复用前端开发范式
- 获得接近原生的体验
- 比 WebView 更适合复杂交互 App 页面

Bridge 的核心原理是：
1. JS 层发消息给 Native
2. Native 解析调用目标模块和方法
3. 执行原生逻辑
4. 再把结果异步回传给 JS

早期 RN 依赖经典 Bridge，序列化和跨线程通信成本较高；新架构则通过 `JSI` 减少桥接损耗。

如果你想从“RN 的 JS 代码最后怎么跑到 App”这个角度完整讲，可以按下面这条链路说：

#### 第一步：你写的是 React 组件代码

例如：

```jsx
export default function App() {
  return <Text>Hello RN</Text>;
}
```

这段代码不会变成浏览器 DOM，也不是塞进 WebView。  
它先在 JS 引擎里执行，生成 React Element，再进入 React 的 Fiber / Reconciler 流程。

#### 第二步：React Reconciler 计算“应该渲染成什么样”

这一层和 React DOM 很像，也会做：

- 组件执行
- 状态更新
- diff
- 生成更新任务

区别在于宿主环境不同：

- React DOM 的宿主是浏览器 DOM
- React Native 的宿主是 Native UI

#### 第三步：RN Renderer 把更新意图变成原生操作指令

比如：

- 创建一个 `View`
- 创建一个 `Text`
- 设置颜色、宽高、布局
- 绑定点击事件

这些不会去创建 HTML 标签，而是会转成“要让 Native 去执行的指令”。

#### 第四步：经典 Bridge 架构里，JS 和 Native 通过桥通信

早期 RN 里：

- JS 跑在 JS 线程
- UI 渲染跑在原生主线程 / UI 线程
- 中间通过 Bridge 传消息

Bridge 本质上就是：

> 连接 JS 世界和 Native 世界的一座桥。

它做的事情是：

1. JS 把调用序列化成消息
2. 发给 Native
3. Native 解析模块名、方法名、参数
4. 执行对应原生逻辑
5. 再把结果异步回传给 JS

#### 第五步：Native 真正创建原生控件

在 iOS 上最终可能创建的是：

- `UIView`
- `UILabel`
- `UIButton`

在 Android 上可能是：

- `View`
- `TextView`
- `Button`

所以用户最后看到的是原生控件，而不是网页 DOM。

#### 第六步：用户交互再从 Native 回到 JS

比如用户点击按钮：

1. Native 先收到原生点击事件
2. 再通过 Bridge 或新架构把事件发回 JS
3. JS 执行你的事件处理函数
4. 状态变化后，再触发下一轮更新

#### 为什么经典 Bridge 会有性能瓶颈？

因为它有几个天然成本：

- 跨线程通信成本
- 数据序列化 / 反序列化成本
- 高频小消息来回的调度成本

这也是为什么复杂动画、大列表和高频手势会更容易暴露 Bridge 的瓶颈。

#### `JSI` 是什么？

`JSI = JavaScript Interface`

它的目标是让 JS 和 Native 之间的调用更直接，减少传统 Bridge 那种“统一异步消息总线”的损耗。

所以一句话总结：

> RN 不是把网页装进 App，而是让 React 负责声明式描述 UI，再由 RN 宿主把这套描述映射成原生控件；Bridge / JSI 则负责连接 JS 世界和 Native 世界。


## 二、React / fetch / 安全 / 通用题

### 1. 介绍 keep-alive，React 中如何实现 keepalive 的效果

`keep-alive` 的核心目标不是“少渲染一次”，而是 **组件切走后不销毁实例，回来时还能保留之前的状态、DOM、滚动位置和缓存数据**。

Vue 里有内建的 `KeepAlive`。React 没有官方同名组件，但可以通过几种方式实现类似效果：

1. **组件不卸载，只做显示隐藏**
   最简单，比如 `display: none`。优点是实现成本低；缺点是隐藏组件依然占内存。
2. **缓存组件实例 / 子树**
   使用第三方库，比如 `react-activation`。
3. **路由级缓存**
   在路由容器层缓存页面节点，切换时不直接 unmount。

   如果继续追问“具体怎么实现、原理是什么”，关键要理解：

   所谓路由级缓存，本质不是 React 官方帮你自动缓存了页面，而是你在路由层**人为控制页面不要被卸载**。

   最简单的实现思路有两种：

   #### 方案 1：不卸载，只隐藏

   ```tsx
   <div style={{ display: activeTab === 'list' ? 'block' : 'none' }}>
     <ListPage />
   </div>
   <div style={{ display: activeTab === 'detail' ? 'block' : 'none' }}>
     <DetailPage />
   </div>
   ```

   这里两个页面其实一直都在，只是切换显示隐藏。  
   所以组件 state、DOM、滚动位置都还保留着。

   #### 方案 2：做一个缓存容器

   可以维护一个：

   ```ts
   Map<routeKey, ReactNode>
   ```

   路由第一次进入时把页面节点存进去；再次返回时直接复用，而不是重新 mount。

   它的底层原理可以概括为：

   - 组件一旦 `unmount`，它对应的 Fiber、state、DOM 都会被释放
   - 如果不 `unmount`，这些运行时状态就还在
   - keep-alive 的本质就是“避免卸载”
4. **状态外置**
   如果只是想“回来还能恢复数据”，也可以把状态存到全局 store，而不是强依赖组件实例缓存。

如果面试官追问 React 里为什么没有官方 KeepAlive，可以回答：

> React 更倾向把“UI 是否重建”和“状态是否保留”拆开处理，官方没有直接提供通用页面缓存容器，但通过 Offscreen、Suspense、外部状态和社区方案都可以实现类似能力。

### 2. 介绍 fetch 请求

`fetch` 是浏览器提供的现代网络请求 API，基于 Promise，语义上更贴近“请求 / 响应对象模型”。

它的特点：

1. **返回 Promise**
   更适合和 `async/await` 配合。
2. **请求和响应可对象化**
   有 `Request`、`Response`、`Headers` 等对象。
3. **默认不因为 4xx/5xx reject**
   只有网络错误才 reject，业务状态码要自己判断 `response.ok`。
4. **支持流式读取**
   比如文本流、SSE 类似处理、文件下载等。
5. **支持 AbortController**
   可以取消请求。

### 4. 跨站脚本注入和跨站请求伪造，怎么预防

#### XSS

XSS 是把恶意脚本注入到页面里执行。

预防手段：

1. **输入清洗和输出转义**
   永远不要把不可信内容直接拼成 HTML。
2. **避免滥用 `innerHTML`**
   尽量使用安全的 DOM API。
3. **富文本场景做白名单过滤**
   比如用 `DOMPurify`。
4. **开启 CSP**
   限制脚本来源，降低注入成功后的执行能力。
5. **Cookie 加 `HttpOnly`**
   即使 XSS 成功，也减少前端直接偷 Cookie 的风险。

#### CSRF

CSRF 是诱导已登录用户在不知情的情况下向目标站点发请求。

预防手段：

1. **CSRF Token**
   请求里带服务端签发的随机 token。
2. **SameSite Cookie**
   限制跨站自动带 Cookie。
3. **校验 Origin / Referer**
4. **关键操作二次确认**
   比如短信验证、密码确认。


## 三、工程化 / Node / Vue / 异步

### 1. 线程和进程是什么？JS 为什么是单线程？

**进程** 是资源分配的基本单位。每个进程通常有独立的内存空间。  
**线程** 是 CPU 调度的基本单位，一个进程里可以有多个线程共享该进程资源。

浏览器里通常会有：

- 渲染进程
- 网络进程
- GPU 进程
- 插件进程

而 JS 在浏览器渲染线程对应的执行环境里通常被设计为 **单线程主逻辑**，原因主要是：

1. **避免 UI 状态竞争**
   如果多个线程同时改 DOM，结果会很复杂。
2. **简化编程模型**
   前端开发者不用默认处理大量锁和并发冲突。
3. **历史设计如此**
   早期浏览器脚本能力简单，单线程模型更容易落地。

但“JS 是单线程”并不代表浏览器只有一个线程。  
真正准确的说法是：**JS 主执行栈通常是单线程的，浏览器底层是多线程协同。**

### 5. 垃圾回收机制

JS 主要采用 **可达性分析** 做垃圾回收。

核心思路：

1. 从根对象出发，比如全局对象、当前执行上下文、闭包引用
2. 能继续访问到的对象叫“可达”
3. 不可达对象就可以被回收

常见回收策略：

1. **标记清除**
   先标记可达对象，再清掉未标记对象。
2. **标记整理**
   回收后顺带整理内存，减少碎片。
3. **分代回收**
   新生代对象通常“生得快死得也快”，老生代存活时间更长，所以采用不同策略。

### 6. 介绍 webpack 工程化

`webpack` 本质是一个模块打包器，但工程里它承担的不只是“打包”，而是整个前端构建体系：

1. **模块依赖分析**
   从入口出发递归分析依赖图。
2. **资源转换**
   通过 `loader` 处理 `TS`、`JSX`、`CSS`、图片等。
3. **代码打包与拆包**
   产出 bundle，支持 `code splitting`。
4. **能力扩展**
   通过 `plugin` 注入构建流程能力，比如压缩、注入 HTML、环境变量替换。
5. **开发体验**
   `dev server`、HMR、source map。
> webpack 不只是把文件拼起来，而是把前端源码、依赖图、静态资源、环境配置、产物优化统一纳入一套构建流水线。

### 7. 介绍 monorepo

`monorepo` 就是多个项目 / 包放在同一个仓库里管理。

适合场景：

- 多个前端应用共享组件库、工具库
- 主工程和基础库要统一版本与规范
- 团队希望统一 CI、lint、构建和依赖治理

优势：

1. 共享代码方便
2. 跨包改动原子化
3. 依赖和规范更统一
4. 更适合设计系统、组件库、多应用平台

挑战：

1. 仓库会变大
2. 构建缓存和任务编排要做好
3. 依赖边界和发布流程要规范

常见工具：

- `pnpm workspace`
- `Turborepo`
- `Nx`
- `Lerna`

### 8. 对 Koa 的理解，用过哪些中间件

`Koa` 是一个更轻量、更现代的 Node Web 框架。它的核心特点是：

1. **中间件洋葱模型**
   请求从外到内进入，再从内到外返回。
2. **基于 async/await**
   比早期 callback 风格更清晰。
3. **核心很薄**
   官方只保留最小能力，很多功能靠社区中间件补。

常用中间件：

- `koa-router`：路由
- `koa-bodyparser`：解析请求体
- `koa-static`：静态资源
- `@koa/cors`：跨域
- `koa-compress`：压缩
- `koa-helmet`：安全头
- `koa-logger`：日志

### 1. composition API、Vue3 的响应式原理

`Composition API` 的核心价值是把“按功能组织逻辑”放在比 Options API 更前的位置，更利于逻辑复用和复杂组件拆分。

Vue3 响应式底层核心是：

1. `Proxy`
   代理对象的 `get` / `set`
2. 依赖收集
   读取属性时，把当前副作用函数收集进去
3. 派发更新
   属性变更时，找到依赖并重新执行

最核心链路就是：

```txt
读取数据 -> track 收集依赖
修改数据 -> trigger 触发依赖
```

相比 Vue2：

- Vue2 用 `Object.defineProperty`
- Vue3 用 `Proxy`
- Vue3 对新增属性、删除属性、数组、Map/Set 等支持更完整

### 3. Vue3 组件传参的方式

常见方式有：

1. `props`
   父传子
2. `emit`
   子传父
3. `v-model`
   双向绑定语法糖
4. `provide/inject`
   跨层级传值
5. 全局状态
   `Pinia`、`Vuex`
6. 事件总线
   现在一般只在少量场景使用

### 6. 处理异步的方法

可以按演进路线回答：

1. **回调函数**
   最基础，但容易回调地狱。
2. **Promise**
   让异步操作具备链式能力和状态表达。
3. **async/await**
   语义更接近同步代码，可读性更好。
4. **生成器 + 调度器**
   历史上是 `async/await` 之前的重要思路。
5. **事件 / 发布订阅**
   适合状态变更或消息驱动型异步。
6. **Worker / 多线程**
   适合耗时计算类异步。

### 8. 了解哪些协议、HTTP/1.1 和 HTTP/2 的区别，为什么需要 HTTP/3，以及和 HTTPS 的区别

我一般会按分层回答：

- 应用层：`HTTP`、`HTTPS`、`WebSocket`、`DNS`
- 传输层：`TCP`、`UDP`
- 网络层：`IP`、`ICMP`

为什么需要 `HTTP/3`：

- `HTTP/2` 仍然跑在 TCP 上
- TCP 丢包会影响整条连接
- `HTTP/3` 用 `QUIC` 解决 TCP 层队头阻塞，并优化建连速度

和 `HTTPS` 的区别：

- `HTTP/1.1`、`HTTP/2`、`HTTP/3` 说的是 **HTTP 协议版本**
- `HTTPS` 说的是 **HTTP 是否跑在安全传输层之上**

所以 `HTTP/2` 可以是明文，也可以是 HTTPS；只是现代浏览器里主流使用通常是 `HTTPS + HTTP/2` 或 `HTTPS + HTTP/3`。

## 四、浏览器 / Webpack / Vue / 算法

### 1. webpack 中对 loader 的了解

`loader` 的本质是 **把某种资源转换成 webpack 能继续处理的模块**。

因为 webpack 原生只认识 JS / JSON，所以像下面这些都要靠 loader：

- `ts-loader` / `babel-loader`
- `css-loader`
- `sass-loader`
- `url-loader`、`file-loader`（旧生态）

执行特点：

1. `loader` 本质上是函数
2. 多个 loader 会形成调用链
3. 一般从后往前执行
4. 每个 loader 接收上一个 loader 的输出

一句话：

> plugin 更像“扩展构建流程”，loader 更像“转换模块内容”。

### 1. 浏览器缓存

浏览器缓存一般分两层：

1. **强缓存**
   在缓存有效期内，浏览器直接用本地副本，不发请求。
2. **协商缓存**
   浏览器带上缓存标识去服务端确认资源是否变更，没变就返回 `304`。

常见响应头：

- 强缓存：`Cache-Control`、`Expires`
- 协商缓存：`ETag` / `If-None-Match`、`Last-Modified` / `If-Modified-Since`

工程里通常是：

- HTML 不做长强缓存
- 静态资源走文件名 hash + 长缓存

### 4. 浏览器如何渲染页面，整个过程发生了什么？

简化后的主流程：

1. 解析 HTML，构建 DOM Tree
2. 解析 CSS，构建 CSSOM Tree
3. 合并生成 Render Tree
4. 计算布局，得到每个节点几何信息
5. 分层、绘制、栅格化
6. 交给 GPU 合成显示

要注意：

- CSS 会影响渲染树构建
- JS 可能阻塞 HTML 解析
- 修改几何属性可能引发回流
- 修改颜色等可能只触发重绘

### 5. 输入 URL 后都发生了什么？

1. URL 解析
2. DNS 解析
3. 建立 TCP 连接
4. 如果是 HTTPS，再做 TLS 握手
5. 发送 HTTP 请求
6. 服务端处理并返回响应
7. 浏览器根据响应头决定缓存、重定向、解压、编码处理
8. 解析 HTML / CSS / JS
9. 构建页面并渲染
10. 期间继续加载子资源，页面逐步可交互


### 8. Vue2 和 Vue3 的区别、响应式、diff 算法

核心区别可以答这几个点：

1. **响应式实现不同**
   Vue2 用 `Object.defineProperty`，Vue3 用 `Proxy`
2. **组合式 API**
   Vue3 有 `Composition API`，更适合复杂逻辑复用
3. **性能更好**
   包括编译优化、Patch Flag、静态提升等
4. **TypeScript 支持更友好**
5. **内置能力增强**
   比如 `Fragment`、`Teleport`、`Suspense`
 
`diff` 算法方面，Vue3 在编译阶段会尽量标记动态节点，运行时能减少无意义比较，整体比 Vue2 更精细。

### 9. v-for 循环中 key 的作用

`key` 的本质是 **给虚拟节点一个稳定身份**，让框架在 diff 时知道“谁是谁”。

好处：

1. 更准确地复用节点
2. 减少错误复用导致的状态错乱
3. 提高列表更新的稳定性

尤其是表单、动画、组件实例列表，如果不用稳定 `key`，很容易出现值串位。


## 五、业务场景 / Vue / 浏览器基础

### 1. 场景题：活动页抽奖次数为三次，点击按钮抽奖，从用户进入平台到抽奖的整个过程，考虑极致用户体验，按钮初始状态如何设计？

我会从“快感知、强兜底、防误触”三层设计。

1. 进入页面先并行拉：
   用户登录态、剩余抽奖次数、活动配置、奖品池状态。
2. 按钮初始态：
   数据未返回前显示 skeleton 或 loading，不要直接可点。
3. 数据返回后：
   `count > 0` 按钮可用；`count === 0` 直接禁用并展示 `No Chances Left`。
4. 点击抽奖时：
   立刻进入 loading 态并禁用按钮，防连点；前端本地先扣一次可视次数，失败再回滚。
5. 请求成功后：
   播放中奖动画，再展示结果弹层。
6. 失败兜底：
   恢复按钮、恢复次数、给出明确错误提示。

核心原则是：**先给用户稳定反馈，再做最终一致性校准**。

### 1. Vue 响应式

Vue2 基于 `Object.defineProperty`，Vue3 基于 `Proxy`。

- `track`：读取时收集依赖

### 5. Vuex 和 Pinia 的区别用法

`Vuex` 更偏传统集中式 store，`Pinia` 更轻量、TS 更友好、写法更接近组合式 API。

### 6. Vue3 生命周期分别每个周期的作用

- `setup`：初始化响应式状态

- `onMounted`：DOM 挂载完成，适合请求、订阅、测量
- `onBeforeUpdate`：更新前

- `onUnmounted`：销毁后

### 7. history 和 hash 路由的区别

- `hash`：URL 带 `#`，不会发给服务端，兼容简单
- `history`：URL 更干净，但服务端要做 fallback

### 8. 父子组件挂载顺序

通常是：

- 父 `beforeCreate/created`
- 父 `beforeMount`
- 子 `beforeCreate/created`
- 子 `beforeMount`
- 子 `mounted`
- 父 `mounted`

**创建先父后子，挂载先子后父**。

### 9. webpack 的 loader、plugin 如何实现？

- `loader`：就是一个把源代码字符串转成新代码字符串的函数
- `plugin`：通过 webpack 暴露的 hooks，在构建生命周期插入逻辑

### 3. 浏览器输入 URL 到页面渲染整个过程

前面已答，可浓缩为：

`URL 解析 -> DNS -> TCP/TLS -> HTTP 请求 -> 响应 -> 解析 HTML/CSS/JS -> 构建 DOM/CSSOM -> layout/paint/composite`

### 4. JS 中 defer 和 async 的区别

- `defer`：异步下载，HTML 解析完后按顺序执行
- `async`：异步下载，下载完立即执行，顺序不保证

### 5. 字符串常用方法

`slice`、`substring`、`substr`、`indexOf`、`includes`、`startsWith`、`endsWith`、`split`、`replace`、`trim`、`toUpperCase`、`toLowerCase`。

### 6. 事件代理

把子节点事件统一挂到父节点，通过事件冒泡处理。

### 11. CSS 实现垂直居中有哪些方法

- `flex`: `display:flex; align-items:center; justify-content:center;`
- `grid`: `place-items:center;`

### 1. Vue 和 React 数据流的区别，组件通信的区别，Vuex 的原理

- React：更强调单向数据流、显式状态更新
- Vue：模板更直接，双向绑定语法糖更多，但本质仍建议单向数据流

组件通信：

- React：`props`、回调、Context、store
- Vue：`props`、`emit`、`provide/inject`、store

- 单一 store
- 响应式 state
- 通过 `commit` / `dispatch` 驱动更新
- 组件订阅 state 变化自动更新

## 六、Next.js / SSE / 性能

### 1. Next 水合原理，整个运行流程和机制

服务端先产出 HTML，浏览器先显示静态内容；随后客户端下载 JS，对已有 DOM 绑定事件和状态，让页面“活起来”，这一步就是 hydration。

### 2. App Router 和 Page Router 的区别

- `pages`：传统路由，以页面文件为中心
- `app`：基于布局、Server Components、流式渲染、嵌套路由能力更强

### 3. RSC 和 SSR 的区别，水合时有什么不一样？路由缓存缓存的是什么？

- SSR：服务端生成 HTML，客户端仍要下载组件 JS 再 hydration
- RSC：Server Component 本身不下发客户端运行时代码，只把渲染结果和边界信息传给浏览器

Next 路由缓存通常缓存的是 RSC payload、路由段数据和已获取的资源结果，不是单纯缓存最终 HTML。

### 4. 异步组件加载的时候 Next 是怎么处理的？

通常结合 `dynamic import`、`Suspense`、代码分割和流式渲染。  
服务端可先返回可渲染部分，异步模块到位后再补齐。

### 7. SSE 和 WS 的区别；WS 为什么双向通信；SSE 怎么断线重连？

- SSE：服务端单向推送，基于 HTTP，文本流
- WS：建立升级连接后，客户端和服务端都能主动发消息

WS 双向是因为协议升级后两端都维护长连接帧通道。

SSE 断线重连：

- 浏览器原生 `EventSource` 有自动重连能力
- 服务端可以配 `retry`
- 用 `Last-Event-ID` 或业务 cursor 做续传

如果服务端数据不符合预期：

1. 前端校验 schema
2. 标记断点 cursor
3. 关闭连接并重连
4. 携带上次成功 cursor，从断点继续拉

### 8. 虚拟滚动 + 两种视图切换怎么优化

1. **页面性能**
   数据分页、懒渲染、稳定 key、避免无效重渲染。
2. **滚动性能**
   固定高度优先、减少同步测量、滚动中只做位移不做重排。
3. **虚拟滚动核心**
   只渲染可视区附近节点，其他节点用占位高度撑开滚动条。
4. **视图切换性能**
   缓存列表状态、滚动位置、视图计算结果；必要时用 keep-alive 思路。

### 2. 事件委托的好处，元素不一样怎么区分

好处：

- 少量监听器
- 动态节点也能处理
- 内存占用更低

区分方式：

- `event.target`
- `closest()`
- `dataset`
- `tagName`

这些名词具体是：

#### 1. `event.target`

表示“真正触发事件的那个最内层元素”。

例如：

```html
<li>
  <img />
</li>
```

如果你点的是 `img`，那：

```js
event.target // 就是 img
```

#### 2. `closest()`

`closest(selector)` 会从当前元素开始，沿着父节点一路往上找，直到找到第一个匹配选择器的祖先节点。

```js
const li = event.target.closest('li');
```

这句的意思是：

> 不管你点的是 `li` 里面的 `img`、`span` 还是文本区域，最后都帮你找到它所属的那个 `li`。

这是事件委托里最好用的 API 之一。

#### 3. `dataset`

`dataset` 是读取自定义属性 `data-*` 的快捷方式。

例如：

```html
<li data-index="3"></li>
```

```js
li.dataset.index; // '3'
```

它适合把一些轻量标识挂在 DOM 上，比如：

- 索引
- id
- 类型
- action 名称

#### 4. `tagName`

表示当前元素标签名，比如：

```js
event.target.tagName; // 'IMG' / 'SPAN' / 'LI'
```

如果你想根据点击到的是图片还是文本做不同处理，就可以用它来分支。

### 3. 五个 li 点击弹出索引，基于事件委托怎么做

```js
ul.addEventListener('click', (event) => {
  const li = event.target.closest('li');
  if (!li) return;

});
```

### 4. 为什么 React 看起来不用手动考虑事件委托

因为 React 合成事件系统已经帮你做了统一事件托管。

如果继续问“底层原理”，可以这样答：

#### 1. 什么是合成事件？

React 不直接把浏览器原生事件对象原封不动交给你，而是包装成一个跨浏览器统一的事件对象，这个对象就叫 `SyntheticEvent`。

这样做的目的有两个：

1. 统一不同浏览器事件差异
2. 把事件系统纳入 React 自己的调度体系

#### 2. 为什么看起来像给每个元素都绑了事件？

因为你写的是：

```jsx
<button onClick={handleClick} />
```

但 React 底层通常不会真的在每个按钮上单独挂一套原生监听器，而是会做 **事件委托 / 统一托管**。

传统理解里，React 会把很多事件统一注册到根容器附近，然后：

1. 原生事件冒泡上来
2. React 拿到原生事件
3. 根据当前 Fiber 树找到对应组件的事件处理函数
4. 再按 React 规则模拟捕获和冒泡阶段调用

#### 3. 这样做的好处

- 减少大量原生监听器
- 统一管理事件系统
- 更方便和 React 更新调度配合

#### 4. 为什么说“看起来不用手动考虑事件委托”？

因为 React 已经在框架层帮你把“监听器统一托管、事件分发、冒泡链路”都处理掉了，所以业务开发时只需要按组件写事件回调。

### 4. NAT 怎么把公网报文转发给内网主机

NAT 会维护一张映射表，把“公网 IP:端口”映射到“内网 IP:端口”。  
回包时根据映射表再转回对应内网主机。

如果要把整个运行流程讲清楚，可以这样展开：

#### 1. NAT 是什么？

`NAT = Network Address Translation`，网络地址转换。

它的作用是：

> 让多个内网设备共享一个或少量公网 IP 对外通信。

家庭路由器就是最常见的 NAT 设备。

#### 2. 为什么需要 NAT？

因为公网 IP 很有限，不可能家里每台手机、平板、电脑都分一个独立公网 IP。  
所以常见做法是：

- 内网设备使用私有地址，如 `192.168.x.x`
- 路由器对外只有一个公网 IP

#### 3. 一个内网请求发出去时发生了什么？

假设：

- 你电脑内网地址：`192.168.1.10:50000`
- 路由器公网地址：`8.8.8.8:30001`
- 目标服务器：`1.1.1.1:443`

当你的电脑发请求时：

1. 内网主机先发出源地址为 `192.168.1.10:50000` 的包
2. 包到达 NAT 设备（路由器）
3. 路由器把源地址改写成自己的公网地址和一个可用端口，例如 `8.8.8.8:30001`
4. 同时在 NAT 映射表里记一条记录：

```txt
8.8.8.8:30001 <-> 192.168.1.10:50000
```

5. 然后再把这个改写后的包发到公网服务器

#### 4. 服务器回包时发生了什么？

服务器收到后，回包目标地址是：

```txt
8.8.8.8:30001
```

回包到达路由器后：

1. 路由器查 NAT 映射表
2. 发现这个公网端口对应的是 `192.168.1.10:50000`
3. 把目标地址再改写回内网主机地址
4. 再把包转发给你的电脑

这就是“公网报文如何回到对应内网主机”的关键。

### 4. 如何解决跨域

CORS、反向代理、JSONP、服务端转发都可以，但面试里最好按“浏览器为什么会拦”和“各方案运行机制”来答。

#### 1. 什么叫跨域？

浏览器的同源策略要求：**协议、域名、端口必须都相同**。  
只要有一个不同，就是跨域。

例如：

- `https://a.com` -> `https://b.com`：域名不同
- `http://a.com` -> `https://a.com`：协议不同
- `https://a.com:3000` -> `https://a.com:4000`：端口不同

#### 2. 为什么会被拦？

不是服务器“天然不允许”，而是 **浏览器为了安全，默认不允许一个源随便读取另一个源的响应内容**。

#### 3. 常见解决方案

**CORS**

- 服务端通过响应头明确告诉浏览器：哪些源可以访问我
- 核心响应头是 `Access-Control-Allow-Origin`

**反向代理**

- 开发服务器或网关把前端的 `/api` 代理到真正后端
- 浏览器看起来仍在请求同源地址
- 常见于本地开发、BFF、网关层

**JSONP**

- 利用 `<script>` 标签天然可跨域加载脚本
- 服务端返回一段函数调用
- 只支持 GET
- 现在更多是历史题

**服务端转发**

- 浏览器只请求自己的后端
- 后端再去请求真正目标服务
- 本质是把“跨域读取”从浏览器挪到服务端

### 5. 设计模式有哪些

高频回答可以按“是什么 + 场景”来讲：

- **单例模式**：全局只有一个实例。场景如全局配置、日志器、缓存管理器。
- **工厂模式**：把对象创建逻辑收口。场景如根据类型创建不同实例。
- **观察者模式**：对象状态变化时通知依赖者。场景如响应式系统。
- **发布订阅模式**：通过事件中心解耦发布者和订阅者。场景如 event bus。
- **策略模式**：封装多种可互换算法。场景如支付方式、校验规则、排序策略。
- **装饰器模式**：不改原对象核心逻辑，给它叠加能力。场景如日志、权限、缓存。
- **代理模式**：通过代理对象控制真实对象访问。场景如虚拟代理、缓存代理。
- **适配器模式**：把不兼容接口转换成统一接口。场景如旧接口兼容。
- **责任链模式**：把处理节点串成链。场景如中间件、审批流、拦截器。

### 7. computed 和 watch 的区别

`computed` 和 `watch` 最大区别在于“是不是为了得到一个派生值”。

**computed**

- 用于从已有响应式数据派生出新值
- 有缓存
- 依赖不变就不会重复计算
- 适合模板直接消费

**watch**

- 用于“观察变化并执行副作用”
- 适合发请求、写日志、同步本地存储、手动联动状态
- 关注的是“变化发生后做什么”，不是“最终值是什么”

一句话总结：

> `computed` 是声明式算结果，`watch` 是命令式做事情。

### 1. token 被窃取了怎么办

### 2. IntersectionObserver 怎么实现无限滚动，和 scroll 有什么区别

核心思路是在列表底部放一个“哨兵元素”（sentinel）：

1. 页面初始渲染列表
2. 底部放一个空元素
3. 用 `IntersectionObserver` 观察它是否进入视口
4. 一旦进入，触发下一页加载
5. 新数据插入后，哨兵继续往下移动

示例：

```js
const observer = new IntersectionObserver((entries) => {
  const entry = entries[0];
  if (entry.isIntersecting) {
    loadMore();
  }
});

observer.observe(document.querySelector('#sentinel'));
```

和 `scroll` 方案的区别：

- `scroll`：高频触发，需要自己算 `scrollTop + clientHeight >= scrollHeight`
- `IntersectionObserver`：浏览器帮你做可见性计算，回调更语义化

所以它的优势是：

- 不用自己高频算位置
- 性能更稳
- 更适合懒加载和无限滚动

### 3. 虚拟列表用 IntersectionObserver 怎么实现

严格来说，`IntersectionObserver` 不是虚拟列表的核心，但可以辅助做“可见区段激活 / 回收”。

虚拟列表真正核心是：

1. 只渲染可视区域附近元素
2. 用占位高度撑开滚动条
3. 根据滚动位置不断复用 DOM 节点

`IntersectionObserver` 可以帮助做：

- 预加载临近区域
- 某段进入视口时激活
- 某段远离视口时回收

但如果问“核心原理”，一定要答：

> 核心不是观察器，而是可视区计算、元素复用和占位容器。

### 5. Express 中间件有哪些？运行机制是什么？

中间件本质是：

```js
(req, res, next) => {}
```

它们按注册顺序串成调用链。

常见中间件：

- 日志中间件
- 鉴权中间件
- body 解析
- 静态资源
- 跨域处理
- 错误处理中间件

运行机制：

1. 请求进入 Express
2. 从第一个中间件开始执行
3. 当前中间件处理完调用 `next()`
4. 继续进入下一个中间件
5. 某个中间件如果直接返回响应，链条可提前结束

它的核心价值是把横切逻辑统一抽出来，形成可组合的请求处理流水线。

### 7. 发布订阅是什么？应用场景？

发布订阅模式的核心是：**发布者和订阅者不直接互相引用，中间通过一个消息中心解耦。**

运行机制：

1. 订阅者向事件中心注册某个主题的回调
2. 发布者往这个主题发消息
3. 事件中心找到所有订阅者并依次通知

常见场景：

- 前端事件总线
- WebSocket 消息分发
- Node 事件系统
- 状态变化广播
- 组件解耦通信

## 十、WebSocket / TS / CSS / Git / 算法

### 2. WebSocket 相对 HTTP 的优势

`WebSocket` 相比普通 HTTP 轮询，核心优势是：

- 真正双向通信
- 长连接
- 更低头部开销
- 更适合实时场景

为什么？

因为普通 HTTP 通常是“一问一答”，客户端必须主动发起；  
而 WebSocket 建连成功后，服务端也能主动推消息。

### 3. WebSocket 如何建立连接

完整流程是：

1. 浏览器先发一个 HTTP 请求
2. 请求头里带：
   - `Upgrade: websocket`
   - `Connection: Upgrade`
   - `Sec-WebSocket-Key`
3. 服务端如果支持，会返回 `101 Switching Protocols`
4. 之后这条连接从 HTTP 协议升级为 WebSocket 协议
5. 后续双方通过 WebSocket frame 收发消息

所以 WebSocket 不是“绕开 HTTP”，而是 **先借 HTTP 完成 Upgrade，再切换协议**。

### 4. WebSocket 断线怎么办？怎么兜底？

工程上通常这样做：

1. **心跳检测**
   定期发 ping/pong 或业务心跳，判断连接是否还活着。
2. **自动重连**
   断开后尝试重新建立连接。
3. **指数退避**
   不要疯狂立刻重连，避免把服务端打崩。
4. **消息重发**
   对重要消息做 ack / 重试机制。
5. **降级**
   某些场景可以降级到 SSE 或轮询。

### 5. 非对称加密有哪些算法

`RSA`、`ECC`、`DSA`、`ElGamal`。

### 6. reduce 和 reduceRight 是什么

它们都是“把一组值归并成一个值”的高阶函数。

- `reduce`：从左到右
- `reduceRight`：从右到左

例如：

```js
[1, 2, 3].reduce((acc, cur) => acc - cur); // ((1 - 2) - 3)
[1, 2, 3].reduceRight((acc, cur) => acc - cur); // ((3 - 2) - 1)
```

### 16. 软链接与硬链接区别

这一题如果只背“硬链接指向 inode，软链接指向路径”，很容易自己也听不懂。  
要先把 3 个基础概念搞清楚：

#### 1. 文件名不是文件本体

在类 Unix 文件系统里，一个“文件”通常可以拆成两部分理解：

1. **目录项（directory entry）**
   就是“文件名 -> inode 编号”的映射关系。
2. **inode**
   存文件元信息和数据位置，比如：
   - 文件大小
   - 权限
   - 时间戳
   - 数据块地址

也就是说：

> 你平时看到的文件名，本质上更像“入口”；真正描述文件内容和元信息的是 inode。

#### 2. 什么是硬链接？

**硬链接（hard link）** 的本质是：

> 多个目录项，指向同一个 inode。

也就是说，两个不同文件名，其实对应的是同一份底层文件数据。

例如你可以理解成：

```txt
文件名 A -> inode 100
文件名 B -> inode 100
```

这时：

- 改 A 的内容，B 看到的也会变
- 删掉 A，只要 B 还在，文件数据还在
- 因为 inode 100 仍然被引用

这就是为什么说“硬链接不是快捷方式，它就是同一文件的另一个名字”。

#### 3. 什么是软链接？

**软链接（symbolic link / symlink）** 的本质是：

> 它自己是一个独立文件，里面保存的是“目标路径”。

你可以把它理解成系统级快捷方式。

例如：

```txt
shortcut.txt -> /Users/mac/file.txt
```

它不是和目标文件共用同一个 inode，而是“记录目标在哪”。

所以：

- 打开软链接时，系统会再去找它记录的那个路径
- 如果目标文件被删了，软链接就会失效
- 这叫“悬空链接”

#### 4. 两者最核心区别

**硬链接**

- 指向同一个 inode
- 更像“同一个文件的另一个名字”
- 删除其中一个名字，不影响另一个
- 一般不能跨文件系统
- 通常不能直接给目录创建硬链接

**软链接**

- 指向的是路径
- 更像快捷方式
- 目标没了，软链接就失效
- 可以跨文件系统
- 可以链接目录

#### 5. 它们有什么用？

**硬链接的作用**

- 节省磁盘空间
- 让多个路径共享同一份文件内容
- 包管理器 / 文件系统优化里常见

**软链接的作用**

- 做快捷入口
- 做目录映射
- 组织工程结构
- 包管理器里也很常见，比如 `node_modules` 的链接结构

#### 6. 为什么 `pnpm` 会提到软链接 / 硬链接？

因为 `pnpm` 的高效安装就和链接机制强相关：

- 全局 store 里保存真实包内容
- 再通过硬链接 / 软链接把包组织到项目里

这样就避免每个项目都完整拷贝一份依赖。

### 18. 回放攻击、供应链攻击、中间人攻击、SQL 注入、CSP、XSS

这道题建议按“是什么 -> 怎么发生 -> 怎么防”来答。

**回放攻击**

- 是什么：把一次合法请求录下来后反复重放
- 风险：重复下单、重复扣款

**供应链攻击**

- 是什么：攻击依赖包、构建工具、发布链路
- 风险：你引入的第三方包里被植入恶意代码
- 防御：锁版本、审计、私仓、签名校验、最小依赖面

**中间人攻击**

- 是什么：攻击者夹在通信两端之间窃听或篡改
- 防御：HTTPS/TLS、证书校验、HSTS

**SQL 注入**

- 是什么：把恶意 SQL 片段拼进查询语句
- 防御：参数化查询、ORM、输入校验、最小权限

**CSP**

- 是什么：内容安全策略
- 作用：限制哪些脚本、样式、图片、iframe 来源可被页面加载和执行

**XSS**

- 是什么：把恶意脚本注入页面执行
- 防御：输出转义、富文本白名单过滤、CSP、HttpOnly

4. 驱动视图层重新渲染

两种常见模式：

### 1. Node 的模块加载流程

可以按 CommonJS 主线讲：

1. 先解析 `require()` 传入的路径
2. 判断是不是 Node 内建模块
3. 不是的话，再按文件 / 目录规则去找
4. 找到源码后读入
5. Node 会把源码包进一个函数里执行
6. 执行结果挂到 `module.exports`
7. 放进模块缓存，下次再 require 直接复用

### 2. CommonJS 和 ESM 区别

可以从 4 个维度讲：

**语法**

- CJS：`require` / `module.exports`
- ESM：`import` / `export`

**加载时机**

- CJS：运行时加载
- ESM：静态分析阶段就能确定依赖

**导出语义**

- CJS：更接近值拷贝结果
- ESM：实时绑定

**优化能力**

- ESM 更利于 tree-shaking

### 3. Vue 的 `$nextTick` 作用，底层如何实现

`nextTick` 的作用是：

> 等本轮响应式更新把 DOM 改完之后，再执行你的回调。

为什么需要它？

因为 Vue 更新 DOM 不是“你改一次数据就立刻同步改一次 DOM”，而是会把多个更新合并进一个异步批处理队列里。

运行机制：

1. 响应式数据变化
2. watcher / effect 被放进更新队列
3. Vue 在同一轮事件循环里做去重和批量更新
4. DOM 真正更新完成后，再执行 `nextTick` 回调

底层通常优先使用微任务：

- `Promise.then`
- `MutationObserver`

因为微任务能比宏任务更快地在当前事件循环末尾执行。

### 5. 微信小程序的 rpx 是什么？0.5px 会直接渲染吗？

`rpx` 是响应式单位，以 750 设计稿为基准。  
是否能表现 `0.5px` 要看设备 DPR 和平台渲染实现，不是所有设备都完全精确。

### 6. 如果需要显示 0.5px 怎么实现

常见做法：

- 利用高 DPR
- 伪元素 + `transform: scale(0.5)`

### 7. 跨域方式，如果跨域发生服务器返回什么？

这里要讲清楚一个常见误区：

> “跨域失败”不一定表示服务器完全没返回，而是浏览器不让前端 JS 读取这个响应。

也就是说：

- 网络层可能已经真的收到了服务端响应
- 但如果 CORS 头不对
- 浏览器会把它拦住
- JS 层看起来就像请求失败了

### 8. `display:none` 和 `visibility:hidden` 区别

- `display:none`：不占位，触发回流重绘
- `visibility:hidden`：占位，只隐藏，通常只重绘
- 两者都还在 DOM Tree

## 十二、浏览器 / DNS / TCP / 类型 / AJAX / 工程

### 1. 浏览器根据响应解析生成 HTML 树和 CSSOM 树

浏览器拿到 HTML 后，会启动 HTML 解析器构建 DOM Tree；  
遇到 CSS 资源时，会下载并解析成 CSSOM Tree。  
然后浏览器把 DOM 和 CSSOM 合起来，生成 Render Tree，再进入布局和绘制阶段。

### 2. DNS 查找过程

一般顺序是：

1. 浏览器缓存
2. 系统缓存
3. 本地域名服务器
4. 根服务器
5. 顶级域（TLD）服务器
6. 权威域名服务器

最后拿到目标域名对应的 IP。

### 3. 根服务器是什么

它不负责直接告诉你目标 IP，而是告诉你该去问哪个顶级域服务器。

### 4. TCP 连接流程

完整可以答：

- 三次握手：建立连接，双方同步序列号并确认可收发
- 四次挥手：双方独立关闭发送方向

三次握手的目的不是“凑数字”，而是为了确认：

1. 我能发给你
2. 你能发给我
3. 双方初始序列号都同步完成

### 5. TCP 和 UDP 区别

前面已答：可靠 / 连接 / 顺序 / 重传 / 拥塞控制，是 TCP 的关键词。

### 6. TCP 拥塞控制机制

高频四件套：

- **慢启动**：一开始别发太猛，指数增长试探网络容量
- **拥塞避免**：接近阈值后线性增长
- **快重传**：收到多个重复 ACK 时尽快重传，不等超时
- **快恢复**：发生丢包后不要直接回到最初速率，而是进入更温和恢复

### 7. DOM 树和 CSSOM 树构建过程

DOM 来自 HTML token 解析，CSSOM 来自 CSS 规则解析和级联计算。

### 8. 加载 JS、CSS 对页面有什么影响

**CSS**

- 一般不会阻塞 HTML 继续解析到所有节点完成
- 但会阻塞 Render Tree 构建和页面首次渲染

**同步 JS**

- 会阻塞 HTML 解析
- 因为浏览器担心脚本会修改后续 DOM 结构

优化方式：

- 脚本用 `defer/async`
- CSS 做拆分
- 关键 CSS 内联
- 降低首屏关键资源数量

### 9. CSS 资源什么时候解析？会不会阻塞 DOM 构建？

下载后会尽快解析。  
它一般不阻塞 DOM 节点继续解析，但会阻塞 render tree 构建和后续渲染。

### 10. 跨域解决方法？CORS 是什么？简单请求和非简单请求、预检请求是什么？

CORS 是浏览器基于 HTTP 响应头实现的 **跨域资源共享机制**。

简单请求：

- 方法通常是 `GET`、`POST`、`HEAD`
- 头部和 `Content-Type` 满足浏览器规定
- 浏览器可直接发正式请求

非简单请求：

- 方法、头部或 `Content-Type` 不满足简单请求条件
- 浏览器会先发一个 `OPTIONS` 请求
- 这一步就叫 **预检请求**

预检的作用是：

> 先问服务端“这个跨域正式请求你允不允许我发？”

### 11. JSONP 是什么

利用 `<script>` 标签可跨域加载脚本，通过约定回调函数拿数据，只支持 GET。

### 12. CSS 选择器优先级

- 权重比较是从高位到低位逐级比
- 权重相同时，后写的覆盖先写的

### 25. Map、WeakMap、Set、RegExp、XMLHttpRequest

`Map`

- 键值对集合

- 保留插入顺序

`WeakMap`

- 键必须是对象
- 弱引用
- 不可遍历
- 常用于缓存对象关联数据，避免阻止 GC

`Set`

- 唯一值集合

`RegExp`

- 正则表达式对象

`XMLHttpRequest`

- 传统浏览器请求对象
- 支持状态监听、上传下载进度、取消请求
- 是 `fetch` 之前最常见的 Ajax 基础设施

### 27. class 底层实现 Promise

这题如果现场真要手写，完整 Promise/A+ 很长。  
高分点不在于把所有细节背出来，而在于你先把核心机制说清：

1. **状态机**
   - `pending`
   - `fulfilled`
   - `rejected`
2. **状态不可逆**
   一旦成功或失败就不能再变回去
3. **回调队列**
   `then` 注册的回调要被存起来，等状态落定后异步触发
4. **链式调用**
   `then` 必须返回一个新的 Promise
5. **thenable 处理**
   如果回调返回的还是 Promise / thenable，要继续展开

所以它的本质是：

> 一个带状态、回调队列和链式解析规则的异步状态机类。

### 35. 什么是 WebGL，运行流程，顶点着色器与片元着色器

WebGL 是浏览器里的图形渲染 API，底层接近 OpenGL ES。

运行流程简化版：

1. 准备顶点数据
2. 上传到 GPU buffer
3. 编写并编译 shader
4. 顶点着色器处理顶点位置等信息
5. 图元装配、光栅化
6. 片元着色器计算像素颜色
7. 输出到帧缓冲，再显示到屏幕

两个核心 shader：

- **顶点着色器**
  处理顶点坐标、变换、法线等
- **片元着色器**
  处理最终像素颜色、光照、贴图

### 38. 前向渲染 / 延迟渲染 / 分块渲染

**前向渲染**

- 简单直接
- 每个物体在渲染时直接算光照
- 透明物体处理相对友好

**延迟渲染**

- 先把几何信息写进 G-buffer
- 再统一做光照
- 多光源场景优势明显

**分块渲染**

- 把屏幕或空间分块
- 只处理当前块内相关的光源 / 对象
- 减少无效计算

### 39. 帧缓冲、离屏渲染

**帧缓冲（Framebuffer）**

- 是渲染输出目标
- 里面通常包括颜色缓冲、深度缓冲、模板缓冲

**离屏渲染**

- 先不直接渲到屏幕
- 而是先渲到内存中的 buffer / 纹理
- 之后再做后处理、合成或二次利用

### 40. 光照类型、光照模型

光照类型：

- 环境光
- 平行光
- 点光源
- 聚光灯

光照模型：

- Lambert
- Phong
- Blinn-Phong
- PBR

解释时可以这样说：

- **环境光**：提供基础亮度，没有方向
- **平行光**：像太阳光，方向一致
- **点光源**：从某个点向四周发散
- **聚光灯**：有方向和角度限制

光照模型则是在回答“表面到底怎么根据光线算出颜色”。

### 41. 如何进行大模型的渲染、八叉树优化

如果这里问的是“大型 3D 模型 / 点云 / 场景如何高效渲染”，八叉树是很常见的空间分区结构。

它的思路是：

1. 把三维空间递归切成 8 个子块
2. 每个物体放进对应空间节点
3. 渲染时只检查和处理当前视锥里真正可见的块

这样可以减少：

- 视锥裁剪计算量
- 碰撞检测计算量
- 无意义绘制

### 42. 光线追踪、实时光线追踪如何实现

**光线追踪**

- 从视点出发发射光线
- 计算与场景物体的相交
- 再递归处理反射、折射、阴影等效果

它的画面真实感强，但计算量极大。

**实时光线追踪**

之所以能“实时”，依赖几类关键技术：

- GPU 硬件加速
- BVH 等空间加速结构
- 降噪
- 与传统光栅化混合渲染

所以真实工程里通常不是“全场景纯光追”，而是混合方案。
