# 网络训练题纲

## 使用方式

这个文档按链路、协议、性能和排查四条主线组织，答案尽量做到可直接表达，而不是只背协议名词。

---

## 一、网络基础

1. 什么是 HTTP？它解决了什么问题？
答：结论：HTTP 本质上是客户端和服务端之间传输资源的一套应用层协议，解决的是请求怎么发、响应怎么回、语义怎么约定这件事。原理：它把方法、状态码、头部、消息体这些规则统一起来，让不同系统之间可以按同一套语义通信。场景：前端所有页面加载、接口调用、静态资源获取，本质上都跑在这条协议链路上。
2. HTTP 和 HTTPS 的核心区别是什么？
答：结论：HTTP 是明文传输，HTTPS 是在 HTTP 外面加了一层 TLS，能做加密、校验和身份认证。原理/场景：HTTPS 不是“HTTP 更快版本”，而是把 HTTP 放到 TLS 之上，重点是保密性、完整性和身份认证。
延伸：HTTPS 高概率会被继续追问到 TLS 握手和证书校验。
3. HTTPS 为什么更安全？
答：结论：因为它不只是加密，还会校验证书、保证数据没被篡改，所以中间人更难下手。原理/场景：它不只是加密，还要解决身份验证和内容未被篡改的问题，所以安全性明显高于纯明文 HTTP。
4. TLS 握手大致做了什么？
答：结论：TLS 握手我会概括成四步，先确认服务端身份，再协商加密算法，然后协商会话密钥，最后建立安全传输通道。原理：它解决的不是单纯加密，而是同时解决身份认证、密钥协商和数据完整性问题。场景：这样后续真正的 HTTP 数据传输才有保密性和防篡改能力。延伸：如果时间有限，我就用“身份、算法、密钥、加密通道”这四个关键词快速概括。
5. TCP 和 UDP 的区别是什么？
答：结论：TCP 可靠、有序、面向连接，适合对完整性要求高的场景；UDP 更轻量、延迟更低，但不保证一定送达，也不保证顺序。原理：TCP 通过确认、重传、流量控制这些机制换来了稳定性，UDP 则把这些能力交给上层自己决定。场景：网页、接口、文件传输更偏 TCP，音视频、实时语音和部分低时延场景更容易考虑 UDP。
6. 为什么 HTTP 通常建立在 TCP 之上？
答：结论：因为网页和接口大多数场景更看重可靠传输，不太能接受丢包乱序。原理/场景：Web 页面和接口通常更看重可靠、有序和完整性，所以大多数 HTTP 场景选择 TCP 更合理。
7. 什么是 IP、端口、域名？它们之间是什么关系？
答：结论：IP 像主机地址，端口像具体服务窗口，域名是给人看的名字，最后还是要解析到 IP 再连端口。原理/场景：可以把它类比成“主机地址、服务窗口、人类友好的名字”，再顺着串成解析和建连过程。
8. DNS 的作用是什么？
答：结论：就是把好记的域名解析成机器能连的 IP 地址。原理/场景：DNS 的价值是把人类可读的域名解析成机器可连接的 IP，是整条请求链路的起点之一。
9. DNS 查询一般有哪些步骤？
答：结论：一般先查本地各级缓存，没命中再走递归解析，最后拿到权威结果。原理/场景：完整链路通常会经过浏览器缓存、系统缓存、本地 hosts、递归查询和权威服务器这些环节。
10. CDN 的核心价值是什么？
答：结论：核心就是把资源分发到离用户更近的节点，降低延迟，也减轻源站压力。原理/场景：CDN 不只是加速，还承担就近分发、削峰、容灾和静态资源缓存能力。

## 二、HTTP 基础

11. 常见 HTTP 请求方法有哪些？它们的语义分别是什么？
答：结论：最常见的方法是 `GET` 获取资源，`POST` 提交或创建，`PUT` 整体更新，`PATCH` 局部更新，`DELETE` 删除。原理：HTTP 方法不只是写法差异，更代表服务端应该如何理解这次请求的语义，以及是否适合缓存、重试和幂等处理。场景：真正完整的表达，要把方法语义和幂等性一起讲出来。
12. `GET` 和 `POST` 的核心区别是什么？
答：结论：核心不是能不能带参数，而是语义不同。`GET` 更偏获取，`POST` 更偏提交和创建。原理/场景：`GET` 和 `POST` 真正的差异主要是语义和缓存、幂等、可书签化等工程特性，不是“一个能带参一个不能”。
延伸：`GET`/`POST` 题最容易被追问到缓存、幂等和语义。
13. 幂等是什么意思？哪些方法通常被认为是幂等的？
答：结论：就是同一个请求做多次和做一次结果一致。通常 `GET`、`PUT`、`DELETE` 会被认为是幂等的。原理/场景：幂等性高的接口更适合重试和网络抖动恢复，这也是它在工程上很重要的原因。
14. 常见 HTTP 状态码有哪些？你会如何分类记忆？
答：结论：我会按 2xx 成功、3xx 重定向、4xx 客户端问题、5xx 服务端问题来记。原理/场景：状态码最好按大类记，再补几个高频细分码的真实含义和使用场景。
15. `200`、`201`、`204` 的区别是什么？
答：结论：`200` 表示请求成功，通常会带响应体；`201` 更强调资源创建成功，常见于创建接口；`204` 也是成功，但没有返回内容。原理：它们的区别不只是数字不同，而是服务端在表达“成功”的不同语义。场景：新增资源返回 `201` 会比一律用 `200` 更清楚，删除成功或无须回传内容时用 `204` 也更贴近语义。
16. `301`、`302`、`307`、`308` 的区别是什么？
答：结论：`301` 和 `308` 是永久重定向，`302` 和 `307` 是临时重定向；`307/308` 会更严格保留原请求方法。原理/场景：`307/308` 的关键价值是严格保留原请求方法和请求体，这也是它们和 `301/302` 最容易拉开差异的地方。
17. `400`、`401`、`403`、`404` 的区别是什么？
答：结论：`400` 是请求本身有问题，`401` 是没认证，`403` 是认证了但没权限，`404` 是资源不存在。原理/场景：`401` 和 `403` 是高频混淆点，前者偏“没认证”，后者偏“认证了但没权限”。
18. `500`、`502`、`503`、`504` 的区别是什么？
答：结论：`500` 是服务内部错了，`502` 常见是网关拿到坏响应，`503` 是服务不可用，`504` 是网关等超时了。原理/场景：这些状态码常出现在网关、反向代理和上游服务异常链路里，排查时要结合链路位置去判断到底是服务自身异常还是上游依赖异常。
19. 常见请求头和响应头有哪些？
答：结论：请求头里常见 `Content-Type`、`Authorization`、`Cookie`，响应头常见缓存头、跨域头、内容类型这些。原理/场景：关键不在于只背头名，而是讲清它们分别控制内容协商、缓存、认证还是跨域。
20. `Content-Type`、`Accept`、`Authorization`、`Cookie` 分别起什么作用？
答：结论：`Content-Type` 说明请求体或响应体的数据类型，`Accept` 表示客户端期望接收什么格式，`Authorization` 通常承载认证凭证，`Cookie` 用来携带会话状态。原理：这些头部本质上分别在解决内容协商、身份认证和状态传递问题。场景：比如 JSON 接口会常见 `application/json`，登录态方案会涉及 `Authorization` 或 `Cookie` 的取舍。

## 三、HTTP 缓存

21. 为什么缓存对前端性能很重要？
答：结论：因为它能直接减少请求次数和传输时间，首屏、接口和静态资源体验都会更好。原理/场景：缓存的核心价值是减少 RTT、减轻源站压力、提升首屏和接口体验，是性能题基础能力。
延伸：缓存题最好能顺着讲到强缓存、协商缓存和静态资源策略。
22. 强缓存和协商缓存分别是什么？
答：结论：强缓存是本地直接用，协商缓存是去问服务端“旧资源还能不能继续用”。原理/场景：强缓存解决“直接用不用问”，协商缓存解决“资源可能变了要不要确认”。
23. `Cache-Control` 常见指令有哪些？
答：结论：最常见我会答 `max-age`、`no-cache`、`no-store`、`public`、`private`。原理/场景：`Cache-Control` 是缓存策略里最常用也最重要的响应头，很多缓存题都会落到它身上。
24. `no-cache` 和 `no-store` 的区别是什么？
答：结论：`no-cache` 不是不缓存，而是每次用前都要协商；`no-store` 才是真的别存。原理/场景：`no-cache` 常被误解成“不缓存”，其实它是“缓存可以存，但每次使用前要先协商”。
延伸：`no-cache` 不是“不缓存”，这是非常高频的纠错点。
25. `max-age` 的含义是什么？
答：结论：`max-age` 表示资源在多少秒内可以直接命中强缓存。原理：浏览器在这段时间里不需要再去和服务端确认资源是否变化，所以能直接减少 RTT 和服务端压力。场景：带 hash 的静态资源特别适合把 `max-age` 拉长，从而把缓存收益吃满。
26. `ETag` 和 `Last-Modified` 的区别是什么？
答：结论：`Last-Modified` 看时间，`ETag` 看资源标识，后者精度通常更高。原理/场景：`ETag` 更精确，但也可能带来额外计算成本；`Last-Modified` 简单但粒度没那么细。
27. 为什么协商缓存仍然需要发请求？
答：结论：因为协商缓存要由服务端参与判断资源是否变化，所以浏览器必须带上条件头再确认一次。原理：浏览器自己只能保存旧资源和校验条件，最终能不能继续复用，仍要由服务端基于 `ETag` 或 `Last-Modified` 来决定。场景：它的好处是资源没变时可以继续复用旧内容，代价是比强缓存多一次请求往返。
28. 静态资源缓存策略该怎么设计？
答：结论：我一般会说带 hash 的静态资源长强缓存，入口 HTML 走协商缓存。原理/场景：高频标准策略是“带 hash 的静态资源长强缓存，入口 HTML 走协商缓存”。
延伸：入口 HTML 通常不做长强缓存，这是前端发版链路里的关键点。
29. 接口响应适合缓存吗？应该看什么场景？
答：结论：要看业务一致性要求。读多写少、实时性不高的接口适合缓存，强实时接口就要慎重。原理/场景：接口是否缓存要看一致性要求、实时性和数据更新频率，不能一概而论。
30. 当缓存策略出问题时，你如何排查？
答：结论：先看请求响应头，再看 URL 是否变更、CDN 是否命中、浏览器是不是强刷，按层排。原理/场景：排查缓存问题时最好按浏览器、CDN、代理、源站四层去看，避免只盯前端。

## 四、HTTP 各版本演进

31. HTTP/1.0、HTTP/1.1、HTTP/2 的主要区别是什么？
答：结论：1.1 重点是长连接和更多缓存能力，2 重点是多路复用、头部压缩和二进制分帧。原理/场景：版本演进题最好抓住“长连接、多路复用、头部压缩、二进制分帧”这几条主线。
延伸：HTTP 版本演进题常和队头阻塞、多路复用放在一起问。
32. 什么是长连接？
答：结论：就是一个 TCP 连接不只跑一次请求，而是复用起来，减少反复建连的开销。原理/场景：长连接的价值是减少重复建连成本，但也要配合连接管理和超时策略。
33. 什么是队头阻塞？
答：结论：前面的请求或响应卡住了，后面的也得跟着等，这就是队头阻塞。原理/场景：队头阻塞既可能发生在 HTTP 层，也可能发生在传输层，面试时不要混为一谈。
34. HTTP/2 的多路复用解决了什么问题？
答：结论：解决了同一连接上请求串着排、利用率低的问题，让多个请求能并行跑。原理/场景：HTTP/2 多路复用让多个请求共享一条连接并发传输，是它最核心的性能改进之一。
延伸：多路复用是 HTTP/2 最常见的标准答案之一。
35. HTTP/2 的头部压缩有什么价值？
答：结论：HTTP/2 头部压缩的价值在于，请求多的时候头信息往往高度重复，压缩后能明显减少传输体积。原理：像 Cookie、User-Agent、鉴权头这些内容在多请求场景下反复发送，非常浪费带宽，而头部压缩就是在降低这部分重复成本。场景：资源多、接口密集、同域下反复请求时，这个收益会更明显。
36. HTTP/2 一定比 HTTP/1.1 快吗？为什么？
答：结论：不一定，网络质量、服务端配置、资源拆分方式不对时，收益可能没想象中大。原理/场景：协议升级不是银弹，效果仍然取决于网络环境、服务端配置、资源拆分和缓存策略。
37. HTTP/3 为什么基于 QUIC？
答：结论：因为 QUIC 基于 UDP，自带更灵活的可靠传输和连接迁移能力，能更好缓解传输层队头阻塞。原理/场景：HTTP/3 基于 QUIC，重点价值是缓解传输层队头阻塞、支持更灵活的连接迁移和更快恢复。
延伸：HTTP/3 高概率会被追问到 QUIC 和传输层队头阻塞。
38. 前端项目什么时候能实际感知到 HTTP/2 或 HTTP/3 的收益？
答：结论：当前端项目资源多、小文件多、弱网占比高，或者用户跨地域访问明显时，更容易感知到 HTTP/2 或 HTTP/3 的收益。原理：这类场景更容易受到连接复用、头部压缩、队头阻塞缓解和链路恢复能力的影响。场景：比如大型中后台、模块很多的站点、弱网移动端访问、海外链路访问，这些都是比较容易体现收益的环境。

## 五、请求过程与性能

39. 一个请求的耗时主要由哪些部分构成？
答：结论：一个请求的耗时通常可以拆成 DNS、TCP、TLS、请求发送、TTFB、内容下载这几段。原理：只有把耗时拆到这些阶段，才能判断瓶颈到底在解析、建连、服务端处理还是内容传输。场景：真实排查接口慢时，我也会按这条链路顺着看，而不是笼统地说“网络慢”。
延伸：把耗时拆清楚，比单纯说“接口慢”更像工程表达。
40. 什么是 TTFB？
答：结论：就是从请求发出到收到服务端第一个字节的时间，它很能反映后端和链路前半段效率。原理/场景：TTFB 很能反映服务端处理和链路前半段效率，是判断“慢在哪”时很关键的指标。
41. 前端为什么要关注 DNS、TCP、TLS 建连耗时？
答：结论：因为这些都发生在业务代码前面，建连慢的话，接口逻辑再快也救不回来。原理/场景：这些耗时都发生在业务逻辑之前，所以它们慢时，前端代码再优化也救不了首包体验。
延伸：这些前置耗时常常决定首屏和接口首包体验上限。
42. 如果接口慢，你如何判断是前端问题还是服务端问题？
答：结论：我会先看 Network 里耗时拆解，若等待阶段长更像服务端或网络，若前端处理长再看渲染和逻辑。原理/场景：判断慢点时要看 Network 分段耗时，是等待长、下载长还是前端处理长。
43. 为什么接口并发数过高会影响页面性能？
答：结论：它会占带宽、占连接、抢主线程处理结果的时间，页面渲染和交互也会被拖慢。原理/场景：高并发请求会同时占带宽、占连接、加重主线程解析结果的压力，所以页面也会变卡。
44. 前端如何控制请求风暴？
答：结论：常见做法是防抖节流、请求合并、去重、缓存、并发限制，别一有输入就无脑打接口。原理/场景：防抖、节流、请求去重、缓存、并发限制和合并请求，都是控制请求风暴的常见手段。
延伸：搜索联想、防抖、并发限制都是请求治理高频场景。
45. 请求重试应该什么时候做？什么时候不该做？
答：结论：临时网络抖动、幂等接口可以重试；支付、创建订单这类非幂等操作就不能乱重试。原理/场景：只有临时性、可恢复且接口幂等时才适合重试，支付、下单这类非幂等操作要非常谨慎。
延伸：重试题一定要主动提幂等性，不然很容易被追问。
46. 超时策略如何设计才合理？
答：结论：要按接口类型分层设置，别全站一个超时值，同时要有取消和兜底提示。原理/场景：超时策略最好分层设计，不同接口类型和弱网场景不应共用一个超时值。
47. 如何设计接口取消能力？
答：结论：前端一般用 `AbortController`，页面切换、重复请求或用户主动中断时及时取消。原理/场景：`AbortController` 是现代前端取消请求的高频方案，页面切换和重复请求时尤其有用。
延伸：页面切换取消请求、重复请求取消旧请求，都是常见场景。
48. 什么是降级？前端在降级里能做什么？
答：结论：降级就是在服务不稳或资源不足时保核心功能。前端能做的是弱化非关键模块、用兜底 UI、减少重试和并发。原理/场景：降级的重点是保核心链路、弱化非关键能力，让用户至少能完成最关键操作。

## 六、跨域与安全

49. 什么是同源？
答：结论：协议、域名、端口都相同，才算同源。原理/场景：同源是浏览器安全模型的基本边界，跨域题通常都以它为起点。
50. 跨域问题的本质是什么？
答：结论：本质不是请求发不出去，而是浏览器基于同源策略限制了前端读取响应。原理/场景：跨域问题的本质是“浏览器不让前端脚本读响应”，不是请求根本发不出去。
延伸：跨域本质题最好先讲“请求能发，响应未必能被前端读取”。
51. CORS 的核心机制是什么？
答：结论：CORS 的核心机制就是浏览器看服务端返回的跨域响应头，决定要不要把响应放给前端代码读取。原理：跨域请求经常能发出去，但浏览器会根据 `Access-Control-*` 规则决定是否放行。场景：前后端分离、CDN 静态资源、子域协作都绕不开它。边界：CORS 是浏览器安全机制，不影响服务端之间互调。
52. 简单请求和非简单请求有什么区别？
答：结论：区别主要在方法、头和内容类型。简单请求直接发，非简单请求通常会先预检。原理/场景：简单请求和非简单请求的区别，直接决定是否会先发 `OPTIONS` 预检。
53. 预检请求什么时候会触发？
答：结论：像自定义头、非简单方法、特殊 `Content-Type`，这些都容易触发 `OPTIONS` 预检。原理/场景：自定义头、非简单方法和特殊 `Content-Type` 都是常见的预检触发条件。
54. 服务端如何正确配置跨域响应头？
答：结论：核心是按需返回 `Access-Control-Allow-Origin`、方法、头信息，带凭证时还不能配通配符。原理/场景：带凭证跨域时尤其要注意 `Access-Control-Allow-Origin` 不能写 `*`。
55. JSONP 的原理是什么？
答：结论：就是利用 `script` 标签可跨域加载脚本，再让服务端返回一个函数调用。原理/场景：JSONP 更像历史兼容方案，知道它是利用 `script` 标签跨域即可。
56. 反向代理为什么能解决跨域？
答：结论：因为浏览器看到的是同源代理地址，由代理去转发给真正后端，跨域发生在服务端之间了。原理/场景：代理能解决跨域，是因为浏览器看到的是同源地址，真正跨域发生在服务端之间。
57. token、cookie、session 的关系是什么？
答：结论：Session 是服务端登录态，Cookie 是浏览器带状态的载体，Token 更像服务端签发给客户端自行携带的凭证。原理/场景：这三个概念经常一起问，最好从“凭证放哪、状态存哪、请求怎么带上”三层回答。
延伸：这三个概念很多时候要放到同一条登录态链路里解释。
58. 基于 cookie 的认证和基于 token 的认证，各自适合什么场景？
答：结论：同站后台系统很多会用 Cookie/Session，前后端分离、多端接入更常见 Token。原理/场景：Cookie/Session 更适合同站后台，Token 更适合前后端分离和多端接入，但都要配合安全策略。
59. 为什么跨域认证经常伴随 `withCredentials` 问题？
答：结论：因为跨域默认不带 Cookie，前端和服务端要一起明确允许带凭证才行。原理/场景：跨域默认不带 Cookie，所以前后端必须同时允许带凭证，很多问题都卡在这里。
60. 你如何向面试官解释前端登录态方案？
答：结论：我会先说凭证放哪、怎么传、怎么续期、怎么退出，再补安全边界和跨域处理。原理/场景：高质量回答通常会覆盖存储位置、携带方式、续期、退出和安全边界。

## 七、实时通信

61. 轮询、长轮询、WebSocket 的区别是什么？
答：结论：轮询是定时问，长轮询是服务端先挂住等结果，WebSocket 是建一个长连接持续双向通信。原理/场景：轮询、长轮询、WebSocket 本质是不同实时性与成本之间的取舍，不是单纯“谁更高级”。
延伸：实时通信题里最重要的是按场景做方案取舍。
62. WebSocket 为什么适合实时通信？
答：结论：因为建连后双方都能主动发消息，时延低，也不用反复建 HTTP 请求。原理/场景：WebSocket 适合高频、低时延、双向通信场景，但也要承担连接管理和心跳维护成本。
63. SSE 和 WebSocket 的区别是什么？
答：结论：SSE 更适合服务端单向推送文本流，WebSocket 是双向通信，能力更全但也更重。原理/场景：SSE 更适合服务端单向推送文本流，WebSocket 适合双向实时交互，选择要看业务方向和复杂度。
延伸：AI 流式文本、系统通知、聊天，是 SSE/WS 高频对比场景。
64. 聊天、通知、协同编辑分别更适合哪种方案？
答：结论：聊天和协同编辑通常更偏 WebSocket，纯通知流很多场景用 SSE 也够。原理/场景：聊天、协同编辑通常更偏 WebSocket，单向通知流很多时候 SSE 就够了。
65. WebSocket 断线重连应该怎么设计？
答：结论：要有重连次数、退避策略、心跳检测，还得处理重复订阅和旧消息清理。原理/场景：重连设计要有次数上限、指数退避、心跳检测和重复订阅清理，否则容易雪上加霜。
延伸：断线重连如果没有退避和去重，容易把故障放大。
66. 如果实时连接很多，前端要注意哪些问题？
答：结论：要注意连接生命周期、内存占用、页面切换时的清理，还有消息风暴下的渲染压力。原理/场景：连接多时要特别注意生命周期管理、页面切换清理、消息风暴和内存占用。

## 八、接口设计与协作

67. 一个好的前后端接口应该具备哪些特点？
答：结论：语义清晰、字段稳定、错误可识别、扩展性好，而且不要把前端逼得写一堆兼容逻辑。原理/场景：好接口的标准通常包括语义清晰、字段稳定、错误可识别、扩展性合理。
68. 为什么接口字段设计会直接影响前端复杂度？
答：结论：字段一旦混乱，前端就要做大量判空、转换、兼容和脏数据兜底，复杂度会直线上升。原理/场景：字段设计混乱会把复杂度转嫁给前端，最终表现成大量兼容逻辑和脏数据判断。
69. 分页接口、筛选接口、导出接口分别要关注什么？
答：结论：分页看总数和游标设计，筛选看条件一致性，导出看异步化和大文件体验。原理/场景：分页、筛选、导出接口关注点不同，真正要突出的是体验一致性和后续扩展性。
70. 如何设计统一错误码体系？
答：结论：要让前端能区分业务失败、权限失效、参数错误、系统异常，这样才能做不同兜底。原理/场景：统一错误码的价值是让前端能按错误类型做差异化兜底，而不是一律弹“请求失败”。
71. 为什么前端需要做请求封装？
答：结论：因为鉴权、超时、重试、错误处理、埋点这些逻辑不能散在每个页面里。原理/场景：请求封装本质是把鉴权、超时、取消、重试、日志这些横切能力收拢到基础设施层。
延伸：请求层题最好体现“横切能力沉淀到基础设施层”的意识。
72. 请求拦截器通常处理什么逻辑？
答：结论：常见就是加 token、公共头、请求去重、日志和统一参数处理。原理/场景：请求拦截器更像“发出前统一处理”，适合挂公共头、鉴权和去重信息。
73. 响应拦截器通常处理什么逻辑？
答：结论：一般处理错误码、登录失效跳转、统一提示、数据结构拆包这些。原理/场景：响应拦截器更像“收到后统一归一化”，适合错误分类、登录失效处理和数据拆包。
74. 如何设计统一的异常处理和用户提示？
答：结论：要区分系统异常、业务异常、静默异常，别什么错都只弹一个“请求失败”。原理/场景：统一异常处理要区分业务失败、系统错误、静默错误和用户提示级别，不能一把梭。
75. 如何避免接口重复请求和竞态问题？
答：结论：我会做请求去重、旧请求取消、结果版本比对，避免后返回的旧数据覆盖新状态。原理/场景：避免竞态常见手段是请求去重、旧请求取消、结果版本校验，防止旧结果覆盖新状态。
延伸：竞态题很适合结合搜索框或切换筛选条件举例。

## 九、排查与实战

76. 如果页面加载慢，你如何从网络层排查？
答：结论：先看 waterfall，拆 DNS、建连、TTFB、下载时间，再看是不是缓存没命中或资源体积过大。原理/场景：从网络层排查加载慢时，最好先拆 waterfall，看慢在 DNS、建连、TTFB 还是下载。
延伸：waterfall 是网络排查最常用的切入点。
77. 如果接口偶发失败，你会重点看什么？
答：结论：我会看失败分布、状态码、超时点、重试情况、网络环境差异，还有服务端日志时间点。原理/场景：偶发失败通常要看时间分布、状态码、重试情况和网络环境差异，再结合服务端日志对时。
78. 如果线上出现大量 `499`、`502`、`504`，你会怎么理解？
答：结论：`499` 常是客户端提前断开，`502` 像网关拿到坏响应，`504` 更像上游超时，三者都偏链路或网关层问题。原理/场景：`499`、`502`、`504` 往往不是前端业务代码本身的问题，更像链路、网关和上游服务异常信号。
延伸：这类状态码题如果能带一点网关/代理视角，会明显更像实战。
79. 如果静态资源加载失败，你会从哪些角度排查？
答：结论：我会看资源 URL、缓存、CDN、证书、跨域、部署路径，还有 hash 是否对应当前 HTML。原理/场景：静态资源失败要同时排查 URL、缓存、CDN、部署路径、证书和跨域头。
80. 如果用户反馈“只有公司网络打不开页面”，你会想到哪些原因？
答：结论：我会想到公司代理、防火墙、DNS 污染、证书拦截、某些端口或域名被策略限制。原理/场景：公司网络打不开，常见会想到代理、防火墙、DNS、证书拦截和策略限制。
81. 如何用浏览器开发者工具分析请求问题？
答：结论：重点看 Network 面板的请求头、响应头、waterfall、预览内容和失败时机。原理/场景：Network 面板要重点看请求头、响应头、状态码、waterfall 和失败时机。
82. 如何把一次真实网络排障讲成面试回答？
答：结论：按现象、假设、定位、修复、结果五步讲，别只说“我改了个超时”。原理/场景：排障题最好按“现象、假设、定位、修复、结果”组织，而不是只说改了哪个参数。

## 十、综合表达

83. 如何从网络角度完整讲一次前端性能优化？
答：结论：我会从缓存、CDN、资源加载、接口并发控制、协议升级这些点串成一条链来讲。原理/场景：从网络角度讲优化，最好把 CDN、缓存、协议、资源加载和请求治理串成一条链。
84. 如何从认证角度讲清一个后台系统的登录链路？
答：结论：我会讲登录请求、服务端签发凭证、浏览器存储与携带、鉴权校验、续期和退出。原理/场景：认证链路题最好覆盖登录、签发凭证、浏览器携带、鉴权校验、续期和退出。
85. 如果让你设计一个稳定的请求层，你会包含哪些能力？
答：结论：我会包含鉴权、超时、取消、重试、去重、错误分类、并发控制、监控埋点这些能力。原理/场景：稳定请求层通常要包含鉴权、超时、取消、重试、去重、错误分类、并发控制和监控埋点。
延伸：如果再补监控埋点和日志关联，会更像成熟项目方案。

## 十一、流式传输与 AI 场景

86. 普通 HTTP 响应和流式响应的核心区别是什么？
答：结论：普通响应一般等完整结果再返回，流式响应是服务端边生成边推，前端边收边展示。原理/场景：流式响应的核心价值是“边生成边传、边接收边展示”，这对 AI 场景的首字时延尤其重要。
延伸：AI 场景里经常更关心首字时延，而不是总时长。
87. 什么是 `Transfer-Encoding: chunked`？它在流式输出里起什么作用？
答：结论：它表示响应按块传，不必提前知道总长度，所以很适合流式返回。原理/场景：`Transfer-Encoding: chunked` 让服务端无需预先知道完整长度，就能持续按块推送响应。
88. 前端如何消费一个流式响应？
答：结论：通常是拿到 `ReadableStream` 后持续读 chunk，再做增量解码和 UI 拼接。原理/场景：前端消费流时通常要结合 `ReadableStream`、reader 循环和增量解码一起看。
89. `ReadableStream`、`getReader()`、增量解码分别在做什么？
答：结论：`ReadableStream` 是流本身，`getReader()` 用来一段段读，增量解码是把二进制连续转成不乱码的文本。原理/场景：`ReadableStream` 提供流抽象，`getReader()` 负责一段段读取，增量解码解决中文和多字节字符边界问题。
90. SSE 和 WebSocket 的区别是什么？在 AI 文本流式输出里为什么经常优先考虑 SSE 或普通 HTTP 流？
答：结论：SSE 和普通 HTTP 流实现更轻，天然适合服务端单向吐文本，很多 AI 场景并不需要 WebSocket 那么重的双向能力。原理/场景：AI 文本输出很多时候只需要服务端单向推流，所以 SSE 或普通 HTTP 流往往比 WebSocket 更轻更稳。
延伸：很多 AI 文本输出场景并不需要 WebSocket 的双向能力。
91. 如何设计流式响应的数据格式，才能让前端更容易消费？
答：结论：最好每段都有明确边界和类型，比如 `event`、`id`、`data`，别让前端自己猜。原理/场景：流数据最好有明确边界、类型和序号，否则前端很难增量消费和恢复状态。
92. 流式输出中如果中途报错，前端该如何感知和兜底？
答：结论：要么服务端在流里发错误事件，要么断流时结合状态和上下文判断，并给用户明确提示。原理/场景：错误处理要么在流里发错误事件，要么在断流后结合上下文做兜底，不能只靠 try/catch 想当然。
93. 如何取消一个正在进行中的流式请求？
答：结论：前端直接用 `AbortController`，用户切换会话或主动停止时马上中断。原理/场景：`AbortController` 是取消流式请求的标准手段，用户切会话、停止生成时都要立刻中断。
延伸：流式请求的取消和普通请求取消思路一致，但要更注意 UI 状态回收。
94. 当用户快速切换会话时，如何避免旧流数据污染新界面？
答：结论：要取消旧请求，并给每次会话一个唯一标识，渲染前先校验是不是当前活跃流。原理/场景：避免旧流污染新界面，关键是“取消旧流 + 校验会话 id + 只渲染当前活跃流”。
95. 登录态、鉴权和代理层会对流式请求带来哪些特殊问题？
答：结论：主要是 Cookie/Token 是否能持续带上、代理会不会缓冲流、超时和断流策略会不会把流截断。原理/场景：流式请求很容易被代理层缓冲、超时和鉴权续期机制影响，所以线上排障不能只看前端代码。
延伸：线上流式问题非常常见的一层根因就是代理和网关配置。
96. 如果线上流式接口偶发中断，你会从网络层优先排查哪些问题？
答：结论：我会先看代理层超时、网关缓冲、连接被重置、鉴权失效、弱网抖动这些点。原理/场景：优先排查代理超时、网关缓冲、连接重置、弱网抖动和鉴权失效，是比较常见的线路。
