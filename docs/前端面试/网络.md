# 网络训练题纲

## 使用方式

这个文档按链路、协议、性能和排查四条主线组织，答案尽量做到可直接表达，而不是只背协议名词。

---

## 一、网络基础

1. 什么是 HTTP？它解决了什么问题？
答：HTTP 本质上是一套应用层通信协议，用来约定客户端和服务端之间“请求怎么发、响应怎么回、各字段分别代表什么语义”。它解决的核心不是底层怎么传比特流，而是上层如何把“我要什么资源”“这次请求成功没有”“返回的内容是什么类型”“能不能缓存、能不能重定向”这些问题标准化，所以网页、接口、图片、脚本和大多数 Web 资源访问，本质上都建立在这套语义协议之上。
2. HTTP 和 HTTPS 的核心区别是什么？
答：HTTP 是明文传输，链路上的中间节点只要能截获数据，就有机会看到内容甚至篡改内容；HTTPS 则是在 HTTP 外面套了一层 TLS 安全通道，让数据在传输过程中具备加密、完整性校验和身份认证能力。它不是“HTTP 的小升级版”，而是把原本裸奔的应用层请求放进了一个更安全的运输通道里，所以真正的关键词不是“多了个 S”，而是“保密性、完整性、身份可信”。
3. HTTPS 为什么更安全？
答：因为 HTTPS 解决的不只是“别人能不能看懂内容”，还解决了“你连到的到底是不是目标服务器”“传输过程中内容有没有被偷偷改过”这两个同样关键的问题。TLS 会通过证书体系帮助客户端验证服务端身份，再通过密钥协商建立加密通道，并用校验机制确保数据未被篡改，所以它比单纯明文 HTTP 更能抵抗窃听和中间人攻击。
4. TLS 握手大致做了什么？
答：TLS 握手可以用一个更容易理解的顺序来记：先打招呼，确认双方支持哪些协议版本和加密算法；然后服务端出示证书，客户端验证这个证书是否可信、域名是否匹配、证书链是否有效；接着双方协商出本次会话要使用的密钥材料；最后再基于这些结果建立安全通道，让后续真正的 HTTP 数据在这个通道里传输。也就是说，TLS 握手本质上是在先解决“你是谁、我们怎么安全交流、后面用哪把钥匙”，而不是一上来就直接加密正文。
5. TCP 和 UDP 的区别是什么？
答：TCP 更强调可靠、有序、面向连接，也就是数据丢了会重传、顺序乱了会重排、双方要先建立连接状态再传输；UDP 则更轻量，没有这么多内建保证，所以延迟通常更低，但上层要自己接受丢包、乱序或自己补可靠性机制。网页、接口、文件传输通常更偏向 TCP，因为不能接受内容乱掉；而音视频、实时语音、游戏同步等更在意时延的场景，才更常考虑 UDP 或基于它做的上层协议。
6. 为什么 HTTP 通常建立在 TCP 之上？
答：因为 Web 页面和接口调用大多数都要求内容完整、顺序正确、请求和响应边界明确，如果 HTML 少几段、JSON 字段乱序、下载内容缺块，业务通常是不能接受的，而 TCP 恰好提供了这些可靠传输能力。HTTP 作为应用层协议，更希望把“丢包重传、顺序控制、流量控制”这类复杂问题交给传输层处理，所以长期以来它天然更适合站在 TCP 之上。
7. 什么是 IP、端口、域名？它们之间是什么关系？
答：可以把 IP 理解成网络里的主机地址，端口理解成这台主机上具体某个服务的入口，域名则是给人更容易记忆和使用的名字。用户访问网站时，浏览器先把域名通过 DNS 解析成 IP，再去连接目标 IP 上对应端口的服务，所以这三者不是并列关系，而是“人类友好名字 -> 机器地址 -> 具体服务入口”这样一条连续链路。
8. DNS 的作用是什么？
答：DNS 的作用就是把用户更容易记住的域名，转换成机器真正能拿来建立连接的 IP 地址。它相当于互联网的“电话簿”，没有它，人们就得直接记一串 IP；从请求链路角度看，DNS 又是访问开始前的关键前置步骤，所以解析慢、解析错、缓存异常都会直接影响后面的所有请求。
9. DNS 查询一般有哪些步骤？
答：DNS 查询通常会先查浏览器缓存、系统缓存和本地 hosts，如果这些本地层都没命中，再把解析请求交给递归 DNS 服务器，由它继续向根域、顶级域和权威 DNS 逐级查询，最后拿到目标域名对应的结果。真实业务里我们不一定每次都要把整棵查询树讲完，但至少要知道 DNS 并不是“问一次服务器就完了”，而是一个带缓存、带分层解析的体系。
10. CDN 的核心价值是什么？
答：CDN 的核心价值，是把资源提前分发到离用户更近的边缘节点，让请求尽量少走远路，从而降低访问延迟，同时减少源站的带宽和并发压力。它不只是“让静态资源更快”，还承担缓存、削峰、容灾、跨地域访问优化等职责，所以前端讲首屏性能、静态资源命中率和全球访问体验时，CDN 往往都是绕不开的一环。

## 二、HTTP 基础

11. 常见 HTTP 请求方法有哪些？它们的语义分别是什么？
答：最常见的方法有 `GET`、`POST`、`PUT`、`PATCH`、`DELETE`，它们对应的不是简单“增删改查按钮”，而是服务端应该怎样理解这次请求的语义：`GET` 更偏获取资源，`POST` 更偏提交动作或创建资源，`PUT` 更像整体替换，`PATCH` 更像局部修改，`DELETE` 则表达删除。理解这些方法时，最好同时带上缓存、幂等和重试语义，因为工程里真正重要的是“协议希望你如何使用它”，而不只是记单词。
12. `GET` 和 `POST` 的核心区别是什么？
答：`GET` 和 `POST` 的核心区别不在于“一个能不能带参数”，而在于语义和工程属性不同：`GET` 更强调获取资源，通常更适合缓存、书签化和幂等场景；`POST` 更强调提交数据或触发服务端动作，通常不默认具备幂等性。很多初学者容易把它记成“GET 放 URL、POST 放 body”，但那只是常见用法，不是这两个方法最本质的区别。
13. 幂等是什么意思？哪些方法通常被认为是幂等的？
答：幂等的意思是同一个请求执行一次和执行多次，系统最终结果应该一致，至少不会因为多来几次就不断产生额外副作用。像 `GET`、`PUT`、`DELETE` 通常会被认为具备幂等语义，而 `POST` 不一定具备；这个概念之所以重要，是因为网络抖动、代理重发、客户端重试时，你必须知道多发一次会不会把业务做重。
14. 常见 HTTP 状态码有哪些？你会如何分类记忆？
答：我通常按大类记忆：2xx 表示成功，3xx 表示重定向，4xx 表示请求方有问题，5xx 表示服务端处理失败。这样记的好处是先建立整体认知，再往里面补高频码，比如 `200`、`201`、`204`、`301`、`302`、`304`、`400`、`401`、`403`、`404`、`500`、`502`、`503`、`504`；比起死背数字，更重要的是能说清这些状态码背后到底在表达哪一类语义。
15. `200`、`201`、`204` 的区别是什么？
答：这三个状态码都表示成功，但表达重点不同：`200` 是最常见的普通成功，通常会带响应内容；`201` 更强调“资源已经创建成功”，常用于新增接口；`204` 表示成功但没有响应体，常见于删除成功或不需要额外返回数据的场景。面试里讲这题时，最好突出“状态码不是数字装饰，而是在更精确地表达服务端到底成功了什么”。
16. `301`、`302`、`307`、`308` 的区别是什么？
答：`301` 和 `308` 表示永久重定向，`302` 和 `307` 表示临时重定向；而 `307`、`308` 相比 `301`、`302` 更强调要保留原请求方法和请求体，不要偷偷把 `POST` 之类改成 `GET`。所以这题的关键不是只记“永久还是临时”，还要知道为什么后来又演进出 `307/308`，本质就是为了让协议语义更严格、更少歧义。
17. `400`、`401`、`403`、`404` 的区别是什么？
答：`400` 表示请求本身有问题，比如参数格式不对；`401` 表示还没通过身份认证，或者认证已失效；`403` 表示你是谁服务器知道了，但你没有权限做这件事；`404` 则表示请求的资源不存在。这里最容易混的是 `401` 和 `403`，前者更偏“你还没被确认身份”，后者更偏“身份没问题，但这件事你不能做”。
18. `500`、`502`、`503`、`504` 的区别是什么？
答：`500` 通常表示服务自身内部处理出错，`502` 常见于网关或代理从上游拿到了异常响应，`503` 更强调服务当前不可用，比如过载、维护或熔断，`504` 则表示网关等待上游超时。它们看起来都像“服务挂了”，但含义其实不一样，所以排查时一定要结合链路位置判断，到底是应用本身、网关代理，还是更上游的依赖出了问题。
19. 常见请求头和响应头有哪些？
答：常见请求头里会有 `Content-Type`、`Accept`、`Authorization`、`Cookie`、缓存条件头等，常见响应头里会有 `Content-Type`、`Cache-Control`、`ETag`、`Set-Cookie`、`Access-Control-*` 等。答这题最重要的不是背一串头名，而是知道它们分别在管什么，比如内容协商、认证授权、缓存控制、跨域放行还是会话传递，这样回答才不会像纯记忆题。
20. `Content-Type`、`Accept`、`Authorization`、`Cookie` 分别起什么作用？
答：`Content-Type` 用来说明当前消息体是什么格式，方便接收方按正确方式解析；`Accept` 表示客户端希望收到什么格式的响应；`Authorization` 通常承载认证凭证，比如 Bearer Token；`Cookie` 则更像浏览器自动携带的状态容器，常用于会话维持。它们分别解决的是“这段数据怎么解释”“我希望你回什么”“我是谁”“我现在带着哪些状态”这几类不同问题。

## 三、HTTP 缓存

21. 为什么缓存对前端性能很重要？
答：结论：因为它能直接减少请求次数和传输时间，首屏、接口和静态资源体验都会更好。原理/场景：缓存的核心价值是减少 RTT、减轻源站压力、提升首屏和接口体验，是性能题基础能力。
延伸：缓存题最好能顺着讲到强缓存、协商缓存和静态资源策略。
22. 强缓存和协商缓存分别是什么？
答：结论：强缓存是本地直接用，协商缓存是去问服务端“旧资源还能不能继续用”。原理/场景：强缓存解决“直接用不用问”，协商缓存解决“资源可能变了要不要确认”。
23. `Cache-Control` 常见指令有哪些？
答：结论：最常见我会答 `max-age`、`no-cache`、`no-store`、`public`、`private`。原理/场景：`Cache-Control` 是缓存策略里最常用也最重要的响应头，很多缓存题都会落到它身上。
24. `no-cache` 和 `no-store` 的区别是什么？
答：结论：`no-cache` 不是不缓存，而是每次用前都要协商；`no-store` 才是真的别存。原理/场景：`no-cache` 常被误解成“不缓存”，其实它是“缓存可以存，但每次使用前要先协商”。
延伸：`no-cache` 不是“不缓存”，这是非常高频的纠错点。
25. `max-age` 的含义是什么？
答：结论：`max-age` 表示资源在多少秒内可以直接命中强缓存。原理：浏览器在这段时间里不需要再去和服务端确认资源是否变化，所以能直接减少 RTT 和服务端压力。场景：带 hash 的静态资源特别适合把 `max-age` 拉长，从而把缓存收益吃满。
26. `ETag` 和 `Last-Modified` 的区别是什么？
答：结论：`Last-Modified` 看时间，`ETag` 看资源标识，后者精度通常更高。原理/场景：`ETag` 更精确，但也可能带来额外计算成本；`Last-Modified` 简单但粒度没那么细。
27. 为什么协商缓存仍然需要发请求？
答：结论：因为协商缓存要由服务端参与判断资源是否变化，所以浏览器必须带上条件头再确认一次。原理：浏览器自己只能保存旧资源和校验条件，最终能不能继续复用，仍要由服务端基于 `ETag` 或 `Last-Modified` 来决定。场景：它的好处是资源没变时可以继续复用旧内容，代价是比强缓存多一次请求往返。
28. 静态资源缓存策略该怎么设计？
答：结论：我一般会说带 hash 的静态资源长强缓存，入口 HTML 走协商缓存。原理/场景：高频标准策略是“带 hash 的静态资源长强缓存，入口 HTML 走协商缓存”。
延伸：入口 HTML 通常不做长强缓存，这是前端发版链路里的关键点。
29. 接口响应适合缓存吗？应该看什么场景？
答：结论：要看业务一致性要求。读多写少、实时性不高的接口适合缓存，强实时接口就要慎重。原理/场景：接口是否缓存要看一致性要求、实时性和数据更新频率，不能一概而论。
30. 当缓存策略出问题时，你如何排查？
答：结论：先看请求响应头，再看 URL 是否变更、CDN 是否命中、浏览器是不是强刷，按层排。原理/场景：排查缓存问题时最好按浏览器、CDN、代理、源站四层去看，避免只盯前端。

## 四、HTTP 各版本演进

31. HTTP/1.0、HTTP/1.1、HTTP/2 的主要区别是什么？
答：HTTP/1.0 阶段连接复用能力很弱，请求和响应模型也比较朴素；HTTP/1.1 明显增强了长连接、缓存控制、分块传输等能力，减少了反复建连的成本；HTTP/2 则进一步把传输方式升级成二进制分帧，引入多路复用、头部压缩和更高效的同连接并发传输。理解版本演进时，不要只背新特性名词，而要抓住主线：它们一直在解决“建连贵、并发差、重复头信息浪费、队头阻塞明显”这些实际问题。
32. 什么是长连接？
答：长连接就是一个 TCP 连接建立之后，不是只服务一组请求就立刻断开，而是尽量在后续请求里继续复用这条连接，从而减少反复三次握手、慢启动和 TLS 握手带来的开销。它解决的是“每次都从零开始太贵”的问题，所以在请求较多的页面里，长连接往往能明显减少前置等待时间。
33. 什么是队头阻塞？
答：队头阻塞可以简单理解成“排在前面的那个包或请求卡住了，后面的即使已经准备好了，也只能跟着一起等”。它既可能发生在应用层的请求排队里，也可能发生在传输层的数据交付里，所以回答这题时最好主动说明：HTTP/1.1 和 HTTP/2、HTTP/3 所讨论的队头阻塞并不完全发生在同一层，不能一句话混过去。
34. HTTP/2 的多路复用解决了什么问题？
答：HTTP/2 的多路复用，解决的是过去 HTTP/1.1 在同一连接上请求容易串行排队、连接利用率不高的问题。它把多个请求和响应拆成很多帧，在同一条连接里交错传输，这样浏览器不用再为了并发去开一堆连接，资源也能更高效地共用同一条通道。对前端来说，它降低了“请求一多就必须疯狂合并资源”的压力，但并不代表资源拆分可以完全不讲策略。
35. HTTP/2 的头部压缩有什么价值？
答：HTTP/2 的头部压缩价值在于，请求一多时，很多头信息其实高度重复，比如 Cookie、鉴权头、User-Agent 等，如果每个请求都原样重复发送，会非常浪费带宽。头部压缩就是在减少这部分重复成本，让协议在大量小请求场景下更高效，所以它不像多路复用那样直观，但在接口密集和静态资源较多的页面里同样很有意义。
36. HTTP/2 一定比 HTTP/1.1 快吗？为什么？
答：不一定，因为协议能力更先进，不代表在任何场景里都会自动变快。实际收益还要看网络质量、服务端配置、资源拆分策略、缓存命中率和链路质量，比如资源本来就很少、缓存做得很好，或者服务端 HTTP/2 配置不佳，收益就未必明显。回答这题时最加分的点，是能说明“协议升级是能力上限提高，不是自动满分”。
37. HTTP/3 为什么基于 QUIC？
答：HTTP/3 之所以基于 QUIC，而不是继续绑死在 TCP 上，核心是想从传输层就改善之前的一些先天限制，尤其是 TCP 层面的队头阻塞问题。QUIC 虽然跑在 UDP 之上，但自己实现了可靠传输、流控制、拥塞控制和更灵活的连接管理，同时还天然更适合连接迁移和弱网恢复，所以 HTTP/3 的价值不是“换了个底座”，而是把很多过去 TCP 很难灵活改进的能力搬到了更容易演进的用户态协议里。
38. 前端项目什么时候能实际感知到 HTTP/2 或 HTTP/3 的收益？
答：当前端项目资源很多、小文件很多、请求密集，或者用户处在弱网、移动网络、跨地域访问这类链路不稳定的环境里时，更容易感知到 HTTP/2 和 HTTP/3 的收益。因为这些场景更依赖连接复用、头部压缩、队头阻塞缓解和断线恢复能力；反过来，如果页面本身资源很少、缓存很好、网络环境也稳定，协议升级带来的用户体感提升就未必特别明显。

## 五、请求过程与性能

39. 一个请求的耗时主要由哪些部分构成？
答：一个请求的总耗时通常可以拆成几个阶段：DNS 解析、TCP 建连、TLS 握手、请求发送、等待服务端首字节返回也就是 TTFB，以及后续内容下载。把耗时拆清楚非常重要，因为只有这样你才能判断问题到底慢在域名解析、连接建立、服务端处理，还是响应体太大导致下载慢；否则“这个接口很慢”几乎没有排查价值。
40. 什么是 TTFB？
答：TTFB 指的是从客户端发出请求，到收到服务端返回的第一个字节之间的时间。它能较集中地反映前半段链路效率，包括连接建立、请求到达服务端、服务端处理并开始回包这些阶段，所以当一个接口“迟迟没响应”时，TTFB 往往是判断慢点在哪的很关键指标。
41. 前端为什么要关注 DNS、TCP、TLS 建连耗时？
答：因为这些耗时都发生在真正的业务请求处理之前，它们属于“你代码还没开始发挥，时间就已经花掉了”的前置成本。前端如果只盯接口函数和页面逻辑，却忽略 DNS 慢、TCP 建连慢、TLS 握手慢，就会误以为“后端怎么这么慢”，实际上首包体验的上限可能早就被链路前半段锁死了。
42. 如果接口慢，你如何判断是前端问题还是服务端问题？
答：我会先看浏览器 Network 的耗时拆解，如果 DNS、建连、TTFB 很长，更像网络链路或服务端处理慢；如果请求很快回来，但前端迟迟才把界面更新出来，就要继续看是不是 JSON 解析、状态更新、渲染或布局太重。也就是说，这题不能只凭“感觉像后端慢”来回答，而要用时间分段把“链路慢”和“前端消费慢”明确拆开。
43. 为什么接口并发数过高会影响页面性能？
答：因为接口并发数一高，不只是网络层在忙，浏览器还要同时处理连接、下载、解压、解析响应体、更新状态和触发渲染，这会同时占带宽、占连接配额、占主线程时间。结果就是页面可能既慢又卡，所以请求多不代表就快，很多时候更需要控制并发、合并请求和延后非关键数据。
44. 前端如何控制请求风暴？
答：前端控制请求风暴的常见手段包括防抖、节流、请求去重、结果缓存、并发限制、合并请求和取消旧请求，核心思想是别让每一次微小输入都立刻放大成一次完整网络开销。像搜索联想、筛选条件频繁切换、滚动加载这类场景，如果没有治理，请求不仅会浪费带宽，还会制造竞态和旧数据覆盖新数据的问题。
45. 请求重试应该什么时候做？什么时候不该做？
答：只有在错误很可能是临时性的、可恢复的，并且接口本身具备幂等语义时，请求重试才比较合理，比如网络抖动、瞬时超时、可安全重复获取的读接口。像支付、下单、扣库存、创建订单这类非幂等操作，就不能为了“提高成功率”随便重试，否则一次业务动作可能真的被执行多次，所以重试这题一定要主动把幂等性讲出来。
46. 超时策略如何设计才合理？
答：合理的超时策略应该按接口类型和业务重要性分层设置，而不是整个项目只用一个统一超时值。比如首屏关键接口、后台导出接口、流式接口、弱网移动端接口的容忍时间都可能不同，同时还要配合取消能力、错误提示和兜底 UI，这样超时才不只是“时间到了就报错”，而是真正服务于用户体验和系统稳定性。
47. 如何设计接口取消能力？
答：现代前端通常会用 `AbortController` 来做请求取消，在页面切换、组件卸载、用户主动停止、同一条件下发起新请求需要替换旧请求时及时中断旧链路。取消能力的价值不只是节省一点流量，更重要的是减少无意义工作、避免旧结果回写新状态，以及让用户的当前操作优先于历史遗留请求。
48. 什么是降级？前端在降级里能做什么？
答：降级就是当服务不稳定、依赖异常或资源紧张时，主动放弃部分次要能力，优先保住最关键业务链路。前端可以做的事情包括隐藏或延后非核心模块、减少自动重试和并发、给出兜底 UI、提供缓存数据或只保留核心操作入口，目标不是“页面看起来完整”，而是“系统再差，用户也至少能把最重要的事做完”。

## 六、跨域与安全

49. 什么是同源？
答：同源指的是协议、域名和端口三者都相同，只有这三项都一致，浏览器才会认为它们属于同一个安全上下文。这个概念是前端很多安全问题的起点，因为跨域、本地存储访问、DOM 互操作等限制，很多都建立在同源策略之上。
50. 跨域问题的本质是什么？
答：跨域问题的本质，不是请求完全发不出去，而是浏览器基于同源策略限制了前端脚本对响应结果的读取权限。很多请求其实已经真正到达了服务端，服务端也回了数据，只是浏览器判断这次跨源访问没有获得足够授权，于是把响应拦在了脚本外面。所以最准确的一句话是：跨域本质上是浏览器的读取限制，不是服务器之间通信能力的限制。
51. CORS 的核心机制是什么？
答：CORS 的核心机制，就是浏览器在收到跨域响应后，会去看服务端返回的一组 `Access-Control-*` 头，判断这次跨域访问是否被明确授权，如果授权成立，才把响应放给前端脚本读取。它解决的不是“让网络层 magically 支持跨域”，而是给浏览器一个更细粒度的跨源放行规则，所以前后端分离项目里，CORS 本质上是在协商“哪些跨源访问可以被浏览器接受”。
52. 简单请求和非简单请求有什么区别？
答：两者的区别主要在请求方法、请求头和 `Content-Type` 是否落在浏览器认定的安全简单范围内。简单请求通常可以直接发起，浏览器只在收到响应后检查是否允许读取；非简单请求则通常会在正式请求前先发一次 `OPTIONS` 预检，提前确认服务端是否允许这类跨域访问。所以区分它们的意义，在于你能提前判断“为什么我的接口多了一次预检”。
53. 预检请求什么时候会触发？
答：当请求使用了浏览器认为不够“简单安全”的方式时，就容易触发预检，比如用了自定义请求头、使用了 `PUT`、`DELETE` 等非简单方法，或者 `Content-Type` 不属于简单类型。浏览器这样做的目的，是在真正发业务请求之前先替你问一句服务端：这种跨域访问你到底允不允许。
54. 服务端如何正确配置跨域响应头？
答：服务端配置跨域的关键，是按需返回 `Access-Control-Allow-Origin`、允许的方法、允许的请求头、必要时还要配置凭证相关头，并且在带 Cookie 或其他凭证时，`Access-Control-Allow-Origin` 不能简单写成 `*`。这题真正想考的是你有没有安全意识，也就是跨域放行最好是精确授权，而不是为了省事一把全开。
55. JSONP 的原理是什么？
答：结论：就是利用 `script` 标签可跨域加载脚本，再让服务端返回一个函数调用。原理/场景：JSONP 更像历史兼容方案，知道它是利用 `script` 标签跨域即可。
56. 反向代理为什么能解决跨域？
答：结论：因为浏览器看到的是同源代理地址，由代理去转发给真正后端，跨域发生在服务端之间了。原理/场景：代理能解决跨域，是因为浏览器看到的是同源地址，真正跨域发生在服务端之间。
57. token、cookie、session 的关系是什么？
答：Cookie 更像浏览器自动携带状态的一种容器，Session 更像服务端保存登录态和会话数据的一套机制，而 Token 则更像服务端签发给客户端自行携带、用来证明身份的凭证。它们经常一起出现，但不是同一个层面的东西：Cookie 说的是“怎么带”，Session 说的是“状态存哪”，Token 说的是“凭证长什么样、怎么验证”，所以回答时最好把这三层关系拆开讲。
58. 基于 cookie 的认证和基于 token 的认证，各自适合什么场景？
答：基于 Cookie/Session 的认证更适合同站场景，尤其是传统后台系统，因为浏览器会自动带 Cookie，服务端维护 Session 也比较直观；基于 Token 的认证更适合前后端分离、多端接入和网关统一鉴权场景，因为客户端能更灵活地自行携带和传递凭证。两者都不是绝对更好，而是安全边界、部署方式、跨域需求和多端协作方式不同，导致更适合的方案不同。
59. 为什么跨域认证经常伴随 `withCredentials` 问题？
答：因为浏览器在跨域请求里默认不会自动带上 Cookie 这类凭证，除非前端显式开启 `withCredentials` 或等价配置，同时服务端也明确允许带凭证跨域返回。也就是说，这不是前端单方面开关的问题，而是前后端必须同时同意这次跨域认证访问，很多“明明登录了却还是没权限”的问题，根因就在这里。
60. 你如何向面试官解释前端登录态方案？
答：我会按一条完整链路来讲：用户登录后，服务端如何签发凭证；前端把凭证放在哪里、请求时如何带上；服务端如何校验并返回权限结果；凭证怎么续期、失效后怎么处理、退出时怎么清理；最后再补上安全边界，比如 XSS、CSRF、跨域带凭证和本地存储风险。这样回答的重点不是背某个方案名，而是证明你能把“登录态从创建到失效”说成一整条工程链路。

## 七、实时通信

61. 轮询、长轮询、WebSocket 的区别是什么？
答：结论：轮询是定时问，长轮询是服务端先挂住等结果，WebSocket 是建一个长连接持续双向通信。原理/场景：轮询、长轮询、WebSocket 本质是不同实时性与成本之间的取舍，不是单纯“谁更高级”。
延伸：实时通信题里最重要的是按场景做方案取舍。
62. WebSocket 为什么适合实时通信？
答：结论：因为建连后双方都能主动发消息，时延低，也不用反复建 HTTP 请求。原理/场景：WebSocket 适合高频、低时延、双向通信场景，但也要承担连接管理和心跳维护成本。
63. SSE 和 WebSocket 的区别是什么？
答：结论：SSE 更适合服务端单向推送文本流，WebSocket 是双向通信，能力更全但也更重。原理/场景：SSE 更适合服务端单向推送文本流，WebSocket 适合双向实时交互，选择要看业务方向和复杂度。
延伸：AI 流式文本、系统通知、聊天，是 SSE/WS 高频对比场景。
64. 聊天、通知、协同编辑分别更适合哪种方案？
答：结论：聊天和协同编辑通常更偏 WebSocket，纯通知流很多场景用 SSE 也够。原理/场景：聊天、协同编辑通常更偏 WebSocket，单向通知流很多时候 SSE 就够了。
65. WebSocket 断线重连应该怎么设计？
答：结论：要有重连次数、退避策略、心跳检测，还得处理重复订阅和旧消息清理。原理/场景：重连设计要有次数上限、指数退避、心跳检测和重复订阅清理，否则容易雪上加霜。
延伸：断线重连如果没有退避和去重，容易把故障放大。
66. 如果实时连接很多，前端要注意哪些问题？
答：结论：要注意连接生命周期、内存占用、页面切换时的清理，还有消息风暴下的渲染压力。原理/场景：连接多时要特别注意生命周期管理、页面切换清理、消息风暴和内存占用。

## 八、接口设计与协作

67. 一个好的前后端接口应该具备哪些特点？
答：结论：语义清晰、字段稳定、错误可识别、扩展性好，而且不要把前端逼得写一堆兼容逻辑。原理/场景：好接口的标准通常包括语义清晰、字段稳定、错误可识别、扩展性合理。
68. 为什么接口字段设计会直接影响前端复杂度？
答：结论：字段一旦混乱，前端就要做大量判空、转换、兼容和脏数据兜底，复杂度会直线上升。原理/场景：字段设计混乱会把复杂度转嫁给前端，最终表现成大量兼容逻辑和脏数据判断。
69. 分页接口、筛选接口、导出接口分别要关注什么？
答：结论：分页看总数和游标设计，筛选看条件一致性，导出看异步化和大文件体验。原理/场景：分页、筛选、导出接口关注点不同，真正要突出的是体验一致性和后续扩展性。
70. 如何设计统一错误码体系？
答：结论：要让前端能区分业务失败、权限失效、参数错误、系统异常，这样才能做不同兜底。原理/场景：统一错误码的价值是让前端能按错误类型做差异化兜底，而不是一律弹“请求失败”。
71. 为什么前端需要做请求封装？
答：结论：因为鉴权、超时、重试、错误处理、埋点这些逻辑不能散在每个页面里。原理/场景：请求封装本质是把鉴权、超时、取消、重试、日志这些横切能力收拢到基础设施层。
延伸：请求层题最好体现“横切能力沉淀到基础设施层”的意识。
72. 请求拦截器通常处理什么逻辑？
答：结论：常见就是加 token、公共头、请求去重、日志和统一参数处理。原理/场景：请求拦截器更像“发出前统一处理”，适合挂公共头、鉴权和去重信息。
73. 响应拦截器通常处理什么逻辑？
答：结论：一般处理错误码、登录失效跳转、统一提示、数据结构拆包这些。原理/场景：响应拦截器更像“收到后统一归一化”，适合错误分类、登录失效处理和数据拆包。
74. 如何设计统一的异常处理和用户提示？
答：结论：要区分系统异常、业务异常、静默异常，别什么错都只弹一个“请求失败”。原理/场景：统一异常处理要区分业务失败、系统错误、静默错误和用户提示级别，不能一把梭。
75. 如何避免接口重复请求和竞态问题？
答：结论：我会做请求去重、旧请求取消、结果版本比对，避免后返回的旧数据覆盖新状态。原理/场景：避免竞态常见手段是请求去重、旧请求取消、结果版本校验，防止旧结果覆盖新状态。
延伸：竞态题很适合结合搜索框或切换筛选条件举例。

## 九、排查与实战

76. 如果页面加载慢，你如何从网络层排查？
答：结论：先看 waterfall，拆 DNS、建连、TTFB、下载时间，再看是不是缓存没命中或资源体积过大。原理/场景：从网络层排查加载慢时，最好先拆 waterfall，看慢在 DNS、建连、TTFB 还是下载。
延伸：waterfall 是网络排查最常用的切入点。
77. 如果接口偶发失败，你会重点看什么？
答：结论：我会看失败分布、状态码、超时点、重试情况、网络环境差异，还有服务端日志时间点。原理/场景：偶发失败通常要看时间分布、状态码、重试情况和网络环境差异，再结合服务端日志对时。
78. 如果线上出现大量 `499`、`502`、`504`，你会怎么理解？
答：结论：`499` 常是客户端提前断开，`502` 像网关拿到坏响应，`504` 更像上游超时，三者都偏链路或网关层问题。原理/场景：`499`、`502`、`504` 往往不是前端业务代码本身的问题，更像链路、网关和上游服务异常信号。
延伸：这类状态码题如果能带一点网关/代理视角，会明显更像实战。
79. 如果静态资源加载失败，你会从哪些角度排查？
答：结论：我会看资源 URL、缓存、CDN、证书、跨域、部署路径，还有 hash 是否对应当前 HTML。原理/场景：静态资源失败要同时排查 URL、缓存、CDN、部署路径、证书和跨域头。
80. 如果用户反馈“只有公司网络打不开页面”，你会想到哪些原因？
答：结论：我会想到公司代理、防火墙、DNS 污染、证书拦截、某些端口或域名被策略限制。原理/场景：公司网络打不开，常见会想到代理、防火墙、DNS、证书拦截和策略限制。
81. 如何用浏览器开发者工具分析请求问题？
答：结论：重点看 Network 面板的请求头、响应头、waterfall、预览内容和失败时机。原理/场景：Network 面板要重点看请求头、响应头、状态码、waterfall 和失败时机。
82. 如何把一次真实网络排障讲成面试回答？
答：结论：按现象、假设、定位、修复、结果五步讲，别只说“我改了个超时”。原理/场景：排障题最好按“现象、假设、定位、修复、结果”组织，而不是只说改了哪个参数。

## 十、综合表达

83. 如何从网络角度完整讲一次前端性能优化？
答：结论：我会从缓存、CDN、资源加载、接口并发控制、协议升级这些点串成一条链来讲。原理/场景：从网络角度讲优化，最好把 CDN、缓存、协议、资源加载和请求治理串成一条链。
84. 如何从认证角度讲清一个后台系统的登录链路？
答：结论：我会讲登录请求、服务端签发凭证、浏览器存储与携带、鉴权校验、续期和退出。原理/场景：认证链路题最好覆盖登录、签发凭证、浏览器携带、鉴权校验、续期和退出。
85. 如果让你设计一个稳定的请求层，你会包含哪些能力？
答：结论：我会包含鉴权、超时、取消、重试、去重、错误分类、并发控制、监控埋点这些能力。原理/场景：稳定请求层通常要包含鉴权、超时、取消、重试、去重、错误分类、并发控制和监控埋点。
延伸：如果再补监控埋点和日志关联，会更像成熟项目方案。

## 十一、流式传输与 AI 场景

86. 普通 HTTP 响应和流式响应的核心区别是什么？
答：普通 HTTP 响应通常是服务端把结果准备完整后一次性返回，前端也往往要等完整响应到齐后才能统一处理；流式响应则是服务端一边生成、一边分块往外推，前端也能一边接收、一边解码、一边展示。它真正提升的不是总时长一定更短，而是“首段内容更早可见”，所以在 AI 文本生成、日志流、实时通知这类场景里，用户体感会明显更好。
87. 什么是 `Transfer-Encoding: chunked`？它在流式输出里起什么作用？
答：`Transfer-Encoding: chunked` 表示响应体会按一块一块的方式持续传输，而不是必须先知道完整内容长度再一次性回包。它在流式输出里的作用非常直接，就是让服务端可以在结果还没完全生成完时，就先把已经准备好的部分推给客户端，因此特别适合边生成边返回的场景。
88. 前端如何消费一个流式响应？
答：前端消费流式响应时，通常会从 `fetch` 返回对象里拿到 `ReadableStream`，再通过 reader 持续读取每个 chunk，把二进制数据做增量解码，并把已经拿到的内容逐步拼到界面上。理解这题的关键，是知道流式响应不是“等完整 body 再 `json()`”，而是“边读边处理边展示”，因此 UI 和网络接收节奏会真正联动起来。
89. `ReadableStream`、`getReader()`、增量解码分别在做什么？
答：`ReadableStream` 表示这份响应是一个可持续读取的数据流，`getReader()` 则是从这条流里拿到一个读取器，让你可以一段一段地把数据取出来；增量解码的作用，是把这些连续到来的二进制 chunk 正确拼成文本，尤其避免中文或多字节字符刚好被截断在块边界时出现乱码。所以这三者其实分别对应“流本身、读取动作、文本还原”三个层次。
90. SSE 和 WebSocket 的区别是什么？在 AI 文本流式输出里为什么经常优先考虑 SSE 或普通 HTTP 流？
答：SSE 更适合服务端单向持续推送文本事件，WebSocket 则是更通用的双向长连接通信机制，能力更强但实现和运维心智也更重。很多 AI 文本流式输出场景，本质上只需要服务端把结果持续往客户端吐出来，客户端并不需要高频双向互发消息，所以 SSE 或普通 HTTP 流往往更轻、更符合现有网关体系，也更容易落地。
91. 如何设计流式响应的数据格式，才能让前端更容易消费？
答：流式响应的数据格式最好让每一段都具备明确边界、类型和值，比如带上 `event`、`id`、`data`、必要时再有 `done` 或错误标记，这样前端才能稳定地做增量解析、状态恢复和异常处理。最怕的设计是“服务端持续吐一串不规则文本，让前端自己猜哪里是一条消息”，那样一旦网络分块边界变化，客户端逻辑就会非常脆弱。
92. 流式输出中如果中途报错，前端该如何感知和兜底？
答：前端感知流中错误，最好有两层手段：一层是服务端在流协议里显式发错误事件或结束标记，让客户端知道这次不是正常完成；另一层是客户端在连接异常中断时，结合当前状态、是否已收到部分内容、是否拿到完成标记来做兜底判断，并给用户清晰提示。因为流式场景下的错误，不一定都会体现在一次普通的 `catch` 里，它可能是中途断流、被代理截断或协议层提前结束。
93. 如何取消一个正在进行中的流式请求？
答：取消流式请求最常见的做法，还是用 `AbortController`，在用户点击停止、切换会话、重新发起新请求或页面离开时主动中断当前流。这里不只是网络层断掉就结束了，前端还要同步回收 loading、禁用态和当前活跃会话标识，否则 UI 很容易残留在“还在生成”的假状态里。
94. 当用户快速切换会话时，如何避免旧流数据污染新界面？
答：避免旧流污染新界面的核心是三件事：及时取消旧请求、给每次会话分配唯一标识、在真正写入 UI 前先校验当前 chunk 是否仍然属于活跃会话。因为流式响应不是一次性返回，旧请求就算逻辑上已经“过时”，网络层和读取层也可能还在继续吐数据，所以只有取消和会话校验同时做，才能真正把旧结果挡住。
95. 登录态、鉴权和代理层会对流式请求带来哪些特殊问题？
答：流式请求会把很多平时不显眼的问题放大出来，比如 Cookie 或 Token 是否能稳定带上、长连接期间鉴权是否会过期、代理层会不会把流缓冲后再一次性吐出、网关超时和空闲断开策略会不会把连接提前截断。也就是说，流式问题经常不只是前端代码没写好，而是登录态、网关、代理和超时配置一起作用的结果。
96. 如果线上流式接口偶发中断，你会从网络层优先排查哪些问题？
答：我会优先排查代理层超时、网关缓冲配置、连接被中间层重置、弱网抖动、TLS 或长连接中断、以及鉴权在流过程中失效这些问题，因为它们都是流式接口线上“偶发中断”最常见的根因。排查这类问题时不能只看浏览器控制台，而要把前端时间点、网关日志、上游服务日志和代理配置放到同一条时间线上对齐，才能看清中断到底发生在哪一层。

## 十二、协议体系全景详解

按真实网络分层去系统梳理。：一台机器如何接入网络，如何找到对方，如何建立传输通道，如何保证可靠性和安全性，最后如何把应用数据表达成 HTTP、DNS、WebSocket 这些上层语义。

如果用一条最常见的 Web 访问链路概括，大致是：

1. 设备先通过网卡、以太网或 Wi-Fi 接入局域网
2. 通过 DHCP 拿到 IP、网关、DNS 等网络配置
3. 通过 ARP 在局域网里找到下一跳设备的 MAC 地址
4. 通过 IP 把数据包送到目标主机或中间路由器
5. 通过 TCP 或 UDP 建立传输语义
6. 如果需要加密，再通过 TLS 或 QUIC 建立安全通道
7. 最后在应用层传输 DNS、HTTP、HTTPS、WebSocket、SSE 等具体协议数据

所以协议不是孤立存在的，而是一层套一层协作完成整条链路。

### 1. 以太网协议 Ethernet

#### 它解决什么问题

- 以太网主要解决的是局域网内部“帧该怎么组织、网卡怎么识别是不是发给自己”的问题。
- 它工作在链路层。
- 在现代互联网里，很多 IP 包真正落到网卡传输时，底层承载形式就是以太网帧。

#### 它的关键数据单位

- 以太网传输的基本单位是帧，也就是 Ethernet Frame。
- 一个典型帧里会有：
  - 目标 MAC 地址
  - 源 MAC 地址
  - 类型字段
  - 负载数据
  - FCS 校验

其中最重要的几个概念：

- `MAC` 地址：链路层地址，标识网卡
- `EtherType`：告诉接收方负载里装的是 IPv4、ARP、IPv6 等哪种协议
- `FCS`：Frame Check Sequence，帧校验序列，用来做链路层差错检测；如果校验不过，这一帧通常会被网卡直接丢弃。
- “帧”可以先理解成链路层真正往网卡和交换机之间发送的那一小包数据；上层的 IP 包、TCP 段最终都要先被装进帧里，才能在本地链路上传。

#### 它的开始、运行时、结束

开始阶段：

- 网卡准备发数据时，先把上层交下来的负载封装成帧。
- 帧里会写明“我要发给哪个 MAC 地址”。

运行时：

- 交换机根据目标 MAC 做二层转发。
- 同一局域网中的设备通过二层网络把帧一跳一跳送过去。

结束阶段：

- 目标网卡发现帧头中的目标 MAC 是自己，或者是广播地址，就接收该帧。
- 然后把负载继续上交给对应上层协议，比如 ARP、IPv4、IPv6。

#### 它的边界和局限

- 以太网只适合处理链路层的本地转发问题。
- 它不负责全球路由，不负责跨公网找到远端机器。
- 一旦跨网段，就必须交给 IP 和路由器处理。

#### 面试里的底层理解

- 网页访问不是“浏览器直接把 HTTP 发给服务器”。
- 真正落地到网卡时，HTTP 最终会被一层层封装进 TLS、TCP、IP、以太网帧里，再发出去。

### 2. ARP 协议

#### 它解决什么问题

- ARP 全称是地址解析协议。
- 它解决的是：在局域网里，已知目标 IP，怎么找到对应的 MAC 地址。

#### 为什么必须有 ARP

- IP 负责逻辑寻址。
- 但真正发到局域网里时，网卡和交换机只认 MAC。
- 所以只知道 IP 还不够，还要知道“下一跳的 MAC 是谁”。
- “下一跳”可以理解成当前主机这一次真正要把帧先交给谁：若目标在同网段，下一跳就是目标主机；若目标不在同网段，下一跳通常就是默认网关。

#### 它的开始、运行时、结束

开始阶段：

- 主机要发送一个 IP 包。
- 它先判断目标 IP 是同网段设备，还是需要交给默认网关。
- 一旦确定“下一跳 IP”，就要查 ARP 缓存，看是否已有对应的 MAC。

运行时：

- 如果 ARP 缓存没有命中，主机会发 ARP 广播：
  - “谁是这个 IP，请告诉我你的 MAC”
- 局域网内所有设备都能收到这个广播。
- 真正持有该 IP 的设备会单播回复自己的 MAC 地址。

结束阶段：

- 发送方把 `IP -> MAC` 映射写入 ARP 缓存。
- 后续一段时间内再发给这个目标，就不需要重复广播。
- 这里的 “ARP 缓存” 可以理解成操作系统记住的一张临时映射表，避免每次发包都重新广播问路。
- “广播”指发给当前局域网里所有设备；“单播”指只发给一个确定目标。

#### 运行中的特点

- ARP 是局域网协议，不跨公网。
- 它依赖广播，所以只在本地链路里有意义。
- ARP 缓存过期后，会重新解析。

#### 风险和问题

- ARP 欺骗是局域网常见攻击手段之一。
- 攻击者伪造 ARP 响应，可能把流量骗到自己机器上，造成中间人攻击。

### 3. IP 协议

#### 它解决什么问题

- IP 协议解决的是跨网络、跨网段的逻辑寻址与路由转发问题。
- 它是网络层核心协议。
- 可以把它理解成互联网里的“地址系统 + 投递系统”。

#### IPv4 和 IPv6

- IPv4 地址长度是 32 位。
- IPv6 地址长度是 128 位。
- IPv6 主要是为了解决地址耗尽、路由聚合、自动配置等问题。

前端面试里更常讲的是 IPv4，但知道 IPv6 的演进背景会更完整。

#### IP 包关心什么

- 源 IP
- 目标 IP
- TTL
- 协议号
- 分片相关信息

其中几个高频点：

- `TTL`：防止数据包在网络里无限循环
- `Protocol`：告诉接收方上层是 TCP、UDP、ICMP 等哪种协议
- `MTU`：Maximum Transmission Unit，链路一次能承载的最大负载大小；超过它就可能分片，分片过多会增加丢包和重组成本。
- “分片”可以理解成把一个过大的 IP 包拆成多块分别发送，接收端再尝试重组；工程上通常更希望通过路径 MTU 发现等手段尽量避免它。

#### 它的开始、运行时、结束

开始阶段：

- 上层传输层把 TCP 段或 UDP 数据报交给 IP。
- IP 根据目标地址决定下一跳该往哪里走。

运行时：

- 路由器根据路由表逐跳转发。
- 每经过一跳，TTL 减 1。
- 如果超过 MTU，IPv4 可能发生分片；现代路径上通常尽量避免分片。

结束阶段：

- 包到达目标主机后，IP 层检查目标地址是否是自己。
- 如果是，则根据协议号把负载交给 TCP、UDP、ICMP 等上层。

#### 它不保证什么

- 不保证可靠送达
- 不保证有序
- 不保证不重复
- 不保证低延迟

IP 做的是“尽力而为交付”，可靠性通常交给 TCP 或更上层协议补。

### 4. ICMP 协议

#### 它解决什么问题

- ICMP 主要用来传递网络控制信息和差错信息。
- 它不是用来承载正常业务数据，而是网络层的“诊断和反馈机制”。

#### 常见用途

- `ping` 基于 ICMP Echo Request / Echo Reply
- 路由不可达、目标不可达
- TTL 耗尽
- 某些路径诊断工具如 `traceroute`
- `Echo Request / Echo Reply` 可以简单理解成“你在吗 / 我在”的探测消息。
- `traceroute` 本质上就是故意让包的 TTL 从小到大递增，逼每一跳路由器返回 “TTL 耗尽” 的 ICMP 响应，从而推断路径经过了哪些中间设备。

#### 它的开始、运行时、结束

开始阶段：

- 主机或路由器在发现某些网络异常时，会生成 ICMP 报文。
- 或者主动发诊断请求，比如 ping。

运行时：

- ICMP 报文跟随 IP 进行网络传输。
- 中间设备或目标主机根据 ICMP 类型做响应。

结束阶段：

- 发起方收到 ICMP 响应后，可据此判断网络是否可达、哪一跳超时、是否有错误。

#### 为什么它重要

- 它让“网络出了什么问题”可以被反馈回来。
- 没有 ICMP，很多基础诊断能力会很差。

### 5. DHCP 协议

#### 它解决什么问题

- DHCP 用于动态分配网络配置。
- 它解决的是设备刚接入网络时，“IP、子网掩码、网关、DNS 从哪里来”的问题。

#### 为什么它重要

- 如果没有 DHCP，很多设备都得手动配 IP。
- 在企业网络、家庭网络、公共 Wi-Fi 中，这显然不现实。

#### 它的经典流程

最经典的是 `DORA` 四步：

1. `Discover`
2. `Offer`
3. `Request`
4. `Acknowledge`
- 可以把 `DORA` 记成：先广播找服务器，再收到可用地址，再明确申请其中一个，最后拿到正式确认。
- “租约”指的是 DHCP 分配的地址有时效，不是永久归你占用；到期前需要续租。

#### 它的开始、运行时、结束

开始阶段：

- 主机刚接入网络，没有 IP。
- 因为还不知道 DHCP 服务器是谁，所以通常先广播 `DHCP Discover`。

运行时：

- DHCP 服务器返回 `Offer`，告诉客户端可以用哪个 IP 和配置。
- 客户端选择一个 `Offer`，发送 `Request` 表示我想用这个地址。

结束阶段：

- 服务器返回 `Ack`，正式确认租约。
- 客户端获得 IP、默认网关、DNS 等配置，网络栈初始化完成。

#### 运行中的续租

- DHCP 分配的是租约，不是永久地址。
- 到期前客户端会尝试续租。
- 如果续租失败，地址可能失效或重新分配。

### 6. UDP 协议

#### 它解决什么问题

- UDP 是一种无连接、面向报文的传输层协议。
- 它解决的是“快速、简单地把一份报文从 A 发到 B”。

#### 它的特点

- 无连接
- 不保证可靠
- 不保证顺序
- 不做重传
- 头部开销小
- 时延低
- “面向报文”指应用每次交给 UDP 的是一整份消息，UDP 会尽量按消息边界发出去，而不像 TCP 那样只提供连续字节流语义。
- “无连接”不是说两端之间完全没关系，而是协议层不维护像 TCP 那样的连接状态机，不需要握手建立状态。

#### 它的开始、运行时、结束

开始阶段：

- 应用把一份数据交给 UDP。
- UDP 加上源端口、目标端口、长度、校验和等头部。

运行时：

- UDP 报文通过 IP 发送出去。
- 网络中可能丢包、重复、乱序。

结束阶段：

- 目标主机根据端口号把数据交给对应进程。
- 协议本身没有“连接关闭”概念，因为它从头到尾都没有连接状态。

#### 它适合什么场景

- 实时音视频
- 在线游戏
- DNS 查询
- QUIC 底层承载

#### 为什么它不是“低配版 TCP”

- UDP 不是为了替代 TCP 的所有能力。
- 它的设计目标是把可靠性、拥塞控制等策略留给上层按场景选择是否实现。

### 7. TCP 协议

#### 它解决什么问题

- TCP 提供可靠、有序、面向连接的字节流传输。
- 它解决的是“如何在不可靠的 IP 网络之上，给应用一个相对可靠的数据通道”。

#### 它的关键能力

- 三次握手建立连接
- 序号与确认应答
- 重传机制
- 滑动窗口
- 流量控制
- 拥塞控制
- 四次挥手关闭连接

#### 1. 开始阶段：三次握手

三次握手的目标不是“礼貌地打招呼”，而是确认两端的收发能力和初始序号都正常。

过程：

1. 客户端发 `SYN`
2. 服务端回 `SYN + ACK`
3. 客户端再回 `ACK`

为什么不能只两次：

- 因为服务端需要知道客户端是否真的收到了自己的确认。
- 否则旧连接、重复包、半开连接等问题会更难处理。

#### 2. 运行时：可靠传输怎么实现

##### 序号和确认号

- 每段 TCP 数据都会带序号。
- 接收方通过 ACK 告诉发送方“我已经收到哪里了”。
- `SYN`：同步序号，用于发起连接。
- `ACK`：确认应答，表示“前面的数据/握手我收到了”。
- `FIN`：结束发送，表示“我这个方向没有更多数据要发了”。

##### 重传

- 如果发送方在一定时间内没收到 ACK，会认为数据可能丢失，触发重传。
- 也可能通过快速重传更早地发现丢包。

##### 滑动窗口

- 双方不是一发一收地停等，而是允许一定范围内连续发送。
- 这样能提高链路利用率。
- 这里的“窗口”可以理解成在还没收到最终确认前，发送方最多能有多少数据同时在路上飞。

##### 流量控制

- 接收方通过窗口大小告诉发送方“我还能接多少”。
- 目的是别把接收方缓冲区撑爆。

##### 拥塞控制

- 这是 TCP 非常核心的能力。
- 它关心的不是接收方能不能接住，而是网络本身会不会被打爆。
- 典型过程包括：
  - 慢启动
  - 拥塞避免
  - 快重传
  - 快恢复
- `慢启动`：刚开始时保守试探网络容量，窗口指数级增长。
- `拥塞避免`：达到一定阈值后改成更平滑地增长，避免把网络一下打满。
- `快重传`：通过重复 ACK 提前判断某段数据可能丢了，不必总是傻等超时。
- `快恢复`：丢包后不是完全回到最初状态，而是尽量保留一部分吞吐能力。

#### 3. 结束阶段：四次挥手

关闭连接通常需要四步：

1. 一方发 `FIN`
2. 对方回 `ACK`
3. 对方处理完剩余数据后，再发 `FIN`
4. 最后一方回 `ACK`

为什么通常是四次：

- 因为 TCP 是全双工连接。
- 一边不发了，不代表另一边也立刻没数据了。
- 所以两个方向的关闭通常要分开处理。

#### TIME_WAIT 为什么存在

- 主动关闭方在最后一个 ACK 发完后，通常会进入 `TIME_WAIT`。
- 这样做是为了：
  - 防止最后一个 ACK 丢失
  - 让旧连接中的延迟报文在网络里自然消失

#### TCP 的局限

- 建连和断连有成本
- 在高丢包、弱网下恢复代价可能大
- 队头阻塞问题明显

### 8. TLS 协议

#### 它解决什么问题

- TLS 解决的是在不可信网络上建立一个保密、完整、可认证的安全通道。
- 它位于应用层和传输层之间，经常可以理解成“HTTP 之下、TCP 之上”的安全层。

#### 它提供哪三类能力

- 保密性：别人看不懂内容
- 完整性：别人改了内容会被发现
- 身份认证：客户端能验证服务端是不是目标站点

#### 它的开始阶段：握手

TLS 1.2 和 TLS 1.3 细节不同，但主线可以统一理解为：

1. 协商协议版本和密码套件
2. 服务端出示证书
3. 客户端验证证书链、域名、有效期
4. 双方协商会话密钥
5. 后续数据改用对称加密传输
- “证书链”可以理解成一条信任背书路径：站点证书由中间 CA 签，中间 CA 再由更上级 CA 签，最终能追溯到系统信任的根证书。
- “会话密钥”指这一次连接真正拿来加密业务数据的对称密钥，不会长期固定写死。

为什么握手里会同时用到非对称和对称：

- 非对称更适合做身份认证和密钥协商
- 对称更适合大规模数据加密，因为性能更高

#### 运行时

- 应用数据会被封装成 TLS Record
- 再做加密和完整性保护
- 然后交给 TCP 发送

#### 结束阶段

- 会话关闭后，连接资源释放
- 某些情况下可做会话恢复，减少后续再次握手成本

#### TLS 1.3 为什么更先进

- 握手轮次减少
- 弱算法被清理
- 前向安全成为更核心的默认方向
- 会话恢复和 0-RTT 能力更强
- “前向安全”可以简单理解成：即使未来某个长期私钥泄露，过去抓到的会话流量也不应该被倒推解密。
- `0-RTT` 指某些恢复场景下客户端可以在握手完全结束前就先发一部分应用数据，以减少首包等待，但要额外考虑重放风险。

#### 风险点

- 证书过期
- 域名不匹配
- 中间人伪造证书
- 代理层 TLS 配置错误

### 9. DNS 协议

#### 它解决什么问题

- DNS 把人类可读的域名解析为机器可连接的 IP 地址。
- 它是应用层协议。

#### 为什么 DNS 不只是“查一次表”

- DNS 是分布式、分层、带缓存的命名系统。
- 它包含递归解析、迭代查询、TTL 缓存、权威应答等机制。
- “递归解析”可以理解成“客户端把结果请求完整托管给递归 DNS，让它代自己一路问到底”。
- “迭代查询”是“当前 DNS 服务器不替你问到底，只告诉你下一步该去问谁”。
- `TTL`：Time To Live，记录这条解析结果可以被缓存多久，不是永远有效。
- “权威 DNS”就是最终对某个域名拥有正式解释权的那台/那组 DNS 服务器。

#### 典型解析流程

开始阶段：

1. 浏览器查缓存
2. 操作系统查缓存
3. 本地配置如 hosts
4. 没命中后，请求递归 DNS 服务器

运行时：

1. 递归 DNS 去问根服务器
2. 根服务器告诉它去问哪个顶级域服务器
3. 顶级域服务器告诉它去问哪个权威 DNS
4. 权威 DNS 返回目标记录，如 A、AAAA、CNAME

结束阶段：

- 递归 DNS 把结果返回给客户端
- 客户端缓存结果
- 后续应用用得到的 IP 去发起连接

#### 常见记录类型

- `A`：IPv4 地址
- `AAAA`：IPv6 地址
- `CNAME`：别名
- `MX`：邮件服务器
- `TXT`：文本记录

#### DNS 的运行时问题

- 解析慢
- 缓存污染
- 权威记录配置错误
- CDN 调度偏差
- DNS 劫持

### 10. HTTP 协议

#### 它解决什么问题

- HTTP 解决的是应用层“请求怎么表达、响应怎么表达、状态怎么描述”的问题。
- 它不关心底层怎么比特传输，而关心语义：
  - 我要什么资源
  - 这是成功还是失败
  - 能不能缓存
  - 需不需要跳转

#### HTTP/1.1 的流程

开始阶段：

1. 浏览器拿到 URL
2. 解析域名
3. 建立 TCP 连接
4. 如果是 HTTPS，再做 TLS 握手
5. 然后发送 HTTP 请求行、头、可选 body
- “请求行”通常长这样：`GET /api/user?id=1 HTTP/1.1`，里面包含方法、路径和协议版本。
- “响应状态行”通常长这样：`HTTP/1.1 200 OK`，里面包含协议版本、状态码和状态描述。

运行时：

- 服务端解析请求
- 生成响应状态行、响应头和响应体
- 连接可能被 Keep-Alive 复用
  - 这里的 Keep-Alive 只是指 **TCP/HTTP 连接复用**，不是“把某个字段设为 true 就启动长轮询”。长轮询是应用层请求策略：客户端发起请求后，服务端故意延迟响应，直到有新数据或超时才返回；而 Keep-Alive 只是让一次请求结束后连接先别关，便于后续请求复用同一条连接。

结束阶段：

- 请求结束后，连接可以继续保留
- 也可能根据策略被关闭

#### HTTP/1.1 的特点

- 文本协议
- 请求响应模型清晰
- 支持长连接
- 仍有明显队头阻塞问题

### 11. HTTP/2 协议

#### 它解决什么问题

- HTTP/2 的核心目标是提升同一连接上的并发利用率。
- 它解决的是 HTTP/1.1 在多请求场景下连接浪费、排队明显、头部重复等问题。

#### 它的核心机制

- 二进制分帧
- 多路复用
- 头部压缩
- 流优先级
- 服务端推送
- “二进制分帧”指 HTTP/2 不再像 HTTP/1.1 那样主要靠明文文本报文，而是把消息拆成更底层的二进制 frame 来传输。
- “stream”可以理解成同一条连接中的一条独立请求-响应通道；frame 则是 stream 上实际传输的小片段。
- “头部压缩”主要指 HPACK，目标是减少大量重复请求头带来的带宽浪费。

#### 开始阶段

- 一般建立在 TLS 之上
- 握手时通过 ALPN 协商出 HTTP/2
- `ALPN`：Application-Layer Protocol Negotiation，可以理解成 TLS 握手里顺便协商“这条安全连接上应用层到底跑 HTTP/1.1 还是 HTTP/2”。

#### 运行时

- 一个 TCP 连接里可以有多个 stream
- 每个请求和响应被拆成 frame
- 多个 stream 的 frame 可以交错发送

#### 结束阶段

- stream 可以单独结束
- 整条 TCP 连接也可以复用很久后再关闭

#### 它的局限

- 虽然 HTTP 层多路复用了，但底层还是 TCP
- 如果 TCP 丢包，整个连接中的流都可能受影响
- 所以它解决了应用层部分队头阻塞，没有解决 TCP 层队头阻塞

### 12. QUIC 协议

#### 它解决什么问题

- QUIC 试图在 UDP 之上实现一套现代化的可靠传输和安全机制。
- 它解决的是：传统 TCP + TLS 演进慢、内核实现重、连接迁移差、队头阻塞明显等问题。

#### 它把哪些能力整合到了自己内部

- 可靠传输
- 拥塞控制
- 流控制
- 多路复用
- TLS 1.3 风格安全握手
- 连接迁移
- “用户态实现”是 QUIC 很重要的工程特征：它跑在 UDP 之上，很多协议演进不必像 TCP 那样强依赖操作系统内核升级。
- “连接迁移”指连接的延续不再死绑定 `IP + port` 这组五元组，终端网络切换时更容易不断连。

#### 开始阶段

- 客户端基于 UDP 发起 QUIC 握手
- 握手里就整合了安全协商
- 某些场景下可减少往返次数

#### 运行时

- 单个 QUIC 连接里可以有多个独立流
- 某个流丢包，不一定阻塞其他流
- 连接用 Connection ID 标识，而不只是 `IP + port`
- `Connection ID` 可以理解成 QUIC 在更高抽象层给连接起的稳定名字，这样网络路径变化后也更容易继续识别同一连接。

#### 结束阶段

- 流可独立关闭
- 连接可整体关闭
- 移动网络切换时，连接迁移能力更强

#### 为什么它重要

- QUIC 是 HTTP/3 的基础
- 它把很多过去卡死在 TCP 内核里的演进空间搬到了用户态

### 13. HTTP/3 协议

#### 它解决什么问题

- HTTP/3 是基于 QUIC 的 HTTP。
- 它的核心价值是从底层传输层面继续改善 HTTP/2 在弱网和丢包场景下的体验。

#### 开始阶段

- 客户端与服务端建立 QUIC 连接
- 握手同时完成安全协商和传输层能力准备
- 然后基于 QUIC stream 承载 HTTP 请求响应

#### 运行时

- 请求和响应跑在独立流上
- 某个流出问题，不一定拖死整个连接
- 在连接迁移和弱网恢复上更友好

#### 结束阶段

- 流和连接按 QUIC 机制结束

#### 它相对 HTTP/2 的本质升级

- 不只是“继续多路复用”
- 而是换了传输底座，减少 TCP 级别队头阻塞影响

### 14. WebSocket 协议

#### 它解决什么问题

- WebSocket 解决的是浏览器和服务端之间长时间、低开销、双向实时通信的问题。

#### 为什么不是直接继续用 HTTP 轮询

- 轮询每次都要请求响应
- 长轮询也仍然基于请求-响应思维
- WebSocket 建立后，双方都能主动发消息

#### 开始阶段

- 先通过 HTTP/1.1 发起 Upgrade 请求
- 请求头里会带：
  - `Connection: Upgrade`
  - `Upgrade: websocket`
- 服务端同意后返回 `101 Switching Protocols`

这一步之后：

- 协议从 HTTP 切换为 WebSocket
- 底层 TCP 连接继续沿用

#### 运行时

- 双方通过 frame 进行消息收发
- 消息可以是文本，也可以是二进制
- 需要心跳、保活、重连等配套机制
- “心跳”本质上是周期性发一个轻量消息，证明双方还活着、链路没被中间层回收。
- “保活”关注的是连接不要因为长时间空闲被代理、网关、防火墙提前掐掉。

#### 结束阶段

- 任意一方都可以发 close frame
- 对方响应后，连接进入关闭流程
- 最终底层 TCP 连接断开

#### 实际问题

- 代理和网关是否支持 Upgrade
- 长连接空闲超时
- 心跳保活
- 断线重连和重复订阅

### 15. SSE 协议

#### 它解决什么问题

- SSE 主要解决浏览器接收服务端单向持续推送文本事件的问题。
- 它适合“服务端不断吐消息，客户端只负责收”的场景。

#### 开始阶段

- 浏览器通过普通 HTTP 请求建立连接
- 服务端返回 `Content-Type: text/event-stream`

#### 运行时

- 服务端按事件流格式持续写入数据
- 常见字段有：
  - `event`
  - `data`
  - `id`
  - `retry`
- 浏览器端 `EventSource` 持续读取
- `event`：事件名，客户端可按事件类型监听。
- `data`：真正的消息内容，可以是纯文本，也可以是一段 JSON 字符串。
- `id`：事件 ID，便于断线重连后从某个位置继续。
- `retry`：告诉客户端断线后隔多久再尝试重连。

#### 结束阶段

- 服务端关闭连接
- 或客户端主动断开
- 浏览器还可按规则自动重连

#### 为什么它适合 AI 文本流

- 本质是单向文本流
- 协议简单
- 天然支持持续输出和逐段展示
- 不需要 WebSocket 那种更重的双向通道管理

### 16. Web 访问里一条完整协议链到底长什么样

如果从“用户在浏览器输入一个 HTTPS URL”开始，把协议串起来，完整链路可以这样讲：

#### 第一步：网络接入

- 设备先通过 DHCP 获得 IP、网关、DNS
- 如果要访问外网，还要知道默认网关是谁

#### 第二步：域名解析

- 浏览器、系统、本地配置依次查缓存
- 没命中则走 DNS 递归解析，拿到目标 IP

#### 第三步：局域网寻址

- 主机发现目标不在本地网段，于是确定下一跳是网关
- 通过 ARP 查出网关 MAC

#### 第四步：建立传输连接

- 如果是 HTTP/1.1 或 HTTP/2 over TLS，通常先走 TCP 三次握手
- 然后走 TLS 握手
- 如果是 HTTP/3，则是 QUIC 握手

#### 第五步：应用层请求开始

- 浏览器发送 HTTP 请求
- 服务端返回状态码、头和响应体

#### 第六步：运行时持续传输

- TCP 负责可靠传输、重传、窗口控制
- TLS 负责加密和完整性
- HTTP 负责请求响应语义

#### 第七步：连接结束

- 普通请求可能继续复用长连接
- 长时间不用后，连接被回收
- TCP 通过挥手关闭
- QUIC 或 WebSocket 也有各自的关闭流程

### 17. 前端面试里最容易混淆的点

#### 1. IP 和 MAC 不是一回事

- IP 是网络层逻辑地址
- MAC 是链路层硬件地址

#### 2. TCP 和 HTTP 不是一层

- TCP 负责可靠传输
- HTTP 负责应用语义

#### 3. HTTPS 不是新协议替代 HTTP

- HTTPS 可以理解为 HTTP 跑在 TLS 安全通道上

#### 4. HTTP/2 多路复用不等于彻底没有队头阻塞

- 它解决的是 HTTP 层的一部分问题
- TCP 层丢包时仍会拖累整条连接

#### 5. WebSocket 不是“更高级的 HTTP”

- 它通常借助 HTTP Upgrade 建立
- 但建立成功后就是另一套协议

#### 6. SSE 不是缩小版 WebSocket

- SSE 本质是服务端单向事件流
- WebSocket 是双向消息通道

### 18. 最后给一套面试表达模板

如果面试官问“你把网络协议整体讲一下”，比较稳的回答方式是：

1. 先按分层讲：链路层、网络层、传输层、安全层、应用层
2. 再按真实链路讲：接入网络 -> 拿配置 -> 解析域名 -> 找到下一跳 -> 建连接 -> 加密 -> 传应用数据
3. 最后补每层代表协议：
   - 链路层：Ethernet、ARP
   - 网络层：IP、ICMP
   - 传输层：TCP、UDP
   - 安全层：TLS、QUIC
   - 应用层：DNS、HTTP、WebSocket、SSE

### 19. 如果要把“协议怎么运行”讲得更像工程实践，应该再补哪些底层视角？

这一部分专门补齐很多面试回答里最容易缺的内容：**真实运行时数据长什么样、协议在代码里怎么体现、抓包时能看到什么、各层到底各自负责什么边界。** 前面 1-18 节已经把主线串起来了，这里补的是“把主线落到工程和数据形态上”。

#### 1. 从抓包视角看，一次 HTTPS 请求在链路上大概长什么样

如果从浏览器发起一个最常见的：

```http
GET /api/user?id=1 HTTP/1.1
Host: example.com
Accept: application/json
Cookie: sid=abc
```

在真正发出去之前，它不会裸奔，而会被一层层包起来。工程上可以把它理解成：

```text
HTTP 消息
  被包进 TLS Record
    被包进 TCP Segment
      被包进 IP Packet
        被包进 Ethernet Frame
```

抓包时不同层大致会长这样。

##### 以太网帧的形态

```text
Ethernet II
  Destination MAC: 00:11:22:33:44:55
  Source MAC:      aa:bb:cc:dd:ee:ff
  EtherType:       0x0800 (IPv4)
  Payload:         IP Packet
  FCS:             ...
```

你在代码里几乎不会直接操作这些字段，但抓包时看到它，就知道“这份数据此刻还在链路层承载里”。

##### ARP 报文的形态

```text
ARP Request
  Sender MAC: aa:bb:cc:dd:ee:ff
  Sender IP:  192.168.1.10
  Target MAC: 00:00:00:00:00:00
  Target IP:  192.168.1.1
  Meaning:    谁是 192.168.1.1，请把 MAC 告诉我
```

这类数据你平时不会在前端代码里写，但当开发环境“能上网却请求打不出去”时，ARP 异常、网关异常、虚拟网卡冲突都可能体现在这里。

##### IPv4 包的形态

```text
IPv4
  Source IP:      192.168.1.10
  Destination IP: 93.184.216.34
  TTL:            64
  Protocol:       6 (TCP)
  Payload:        TCP Segment
```

这里最常见的抓包判断点是：

- `Destination IP` 是否正确
- `TTL` 是否不断被消耗
- 是否有分片
- 是否收到 ICMP 不可达

##### TCP 段的形态

```text
TCP
  Source Port:      52341
  Destination Port: 443
  Sequence Number:  ...
  Ack Number:       ...
  Flags:            SYN / ACK / FIN / PSH
  Window Size:      ...
  Payload:          TLS Record
```

前端开发时，你虽然不会直接写这些字段，但“连接慢”“接口时好时坏”“弱网丢包重传”本质上都在这一层发生。

##### TLS 记录的形态

```text
TLS Record
  Content Type: Handshake / Application Data
  Version:      TLS 1.2 / 1.3
  Payload:      加密后的 HTTP 数据
```

TLS 握手时还能看到证书链、密钥协商过程；真正业务数据阶段一般只会看到：

```text
Application Data
```

因为 HTTP 头和体都已经被加密了。

##### HTTP 请求/响应的形态

如果是明文 HTTP，抓包能直接看到：

```http
GET /api/user?id=1 HTTP/1.1
Host: example.com
Accept: application/json
User-Agent: Mozilla/5.0 ...
Cookie: sid=abc
```

响应大概是：

```http
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: no-cache
Content-Length: 27

{"id":1,"name":"Tom"}
```

如果是 HTTPS，抓包一般看不到明文 HTTP 内容，除非你做了代理解密。

#### 2. WebSocket 与 SSE 在运行时的数据长什么样

##### WebSocket 建连前

它先长成一段 HTTP Upgrade 请求：

```http
GET /socket HTTP/1.1
Host: example.com
Connection: Upgrade
Upgrade: websocket
Sec-WebSocket-Key: xxxxx
Sec-WebSocket-Version: 13
```

服务端同意后：

```http
HTTP/1.1 101 Switching Protocols
Connection: Upgrade
Upgrade: websocket
Sec-WebSocket-Accept: yyyyy
```

这之后就不再是 HTTP 请求/响应了，而是 WebSocket frame。运行时消息大概会长成：

```text
WebSocket Frame
  FIN: 1
  Opcode: Text
  Payload: {"type":"chat","msg":"hello"}
```

或二进制帧：

```text
WebSocket Frame
  Opcode: Binary
  Payload: <binary bytes>
```

##### SSE 的运行时数据

SSE 是标准文本事件流，最典型长这样：

```text
event: message
id: 1001
data: hello

event: chunk
data: {"delta":"你"}

event: chunk
data: {"delta":"好"}

```

这里最重要的是：

- 整体 `Content-Type: text/event-stream`
- `data:` 后面是文本
- JSON 只是文本内容的一种常见格式
- 以空行分隔事件

#### 3. 前端开发里，这些协议到底“表现成什么 API”

这是面试最容易加分的一层。不要只说协议名字，要说在工程里你怎么碰到它。

##### DNS / TCP / TLS

前端通常不直接调用这些协议 API，但每次：

```js
fetch('https://api.example.com/user')
```

背后都会隐含：

1. DNS 解析域名
2. TCP 建连
3. TLS 握手
4. HTTP 请求发出

所以你在前端里看到的一个 `fetch`，底层并不是一个简单“函数调用”，而是一整条协议链的入口。

##### HTTP

工程里通常表现为：

- `fetch`
- `XMLHttpRequest`
- `axios`
- 浏览器加载静态资源（HTML/CSS/JS/image/font）

你能直接控制的通常是应用层内容：

- method
- URL
- headers
- body
- cache mode
- credentials
- keepalive（页面卸载时发送小请求）
- abort signal

##### WebSocket

工程里通常表现为：

```js
const ws = new WebSocket('wss://example.com/socket')
ws.onmessage = (e) => {}
ws.send(JSON.stringify({ type: 'ping' }))
```

你直接管理的是：

- 建连时机
- 重连
- 心跳
- 消息协议
- 订阅/退订

##### SSE

工程里通常表现为：

```js
const es = new EventSource('/stream')
es.onmessage = (e) => {
  console.log(e.data)
}
```

或服务端 AI 输出流时，用 `fetch + ReadableStream` 自己读文本流。

##### QUIC / HTTP/3

前端通常不会手动选择“这一请求具体怎么发 QUIC 包”，而是：

- 浏览器与服务端协商
- CDN / 反向代理 / 服务器配置是否支持
- DevTools / 抓包工具里能看到是否用了 h3

也就是说，它对前端开发者更像“运行环境能力”和“性能特征”，不是像 `fetch` 一样直接编码的 API。

#### 4. 实际开发里如何利用这些协议能力

这部分是“知道协议”到“会工程落地”的关键。

##### 用 HTTP 做普通接口

适合：

- 列表查询
- 表单提交
- 页面初始化
- SSR 数据拉取

关注点：

- 幂等性（GET/PUT/DELETE）
- 超时
- 重试
- 缓存策略
- 鉴权头 / Cookie
- CDN / 浏览器缓存

##### 用长轮询

适合：

- 无法上 WebSocket
- 服务端基础设施较老
- 需要兼容代理环境

表现形式通常是：

```js
async function poll() {
  const res = await fetch('/poll')
  const data = await res.json()
  handle(data)
  poll()
}
```

本质是：

- 客户端发一个请求
- 服务端等到有数据或超时再回
- 客户端收到后再立刻发下一个请求

不是把某个 Keep-Alive 字段改成 `true` 就自动有长轮询。

##### 用 SSE

适合：

- AI 文本流
- 服务端单向推送
- 日志流
- 进度通知

优势：

- 协议简单
- 自动重连方便
- 天然文本流

劣势：

- 单向
- 通常不适合二进制

##### 用 WebSocket

适合：

- 即时聊天
- 协同编辑
- 游戏
- 行情 / 看板实时推送
- 双向低延迟交互

你要多考虑：

- 心跳机制
- 连接状态机
- 重连抖动
- 消息顺序
- 幂等处理
- 重复订阅

##### 用 HTTP/2 / HTTP/3

前端工程里不是“手写协议”，而是：

- 确保静态资源部署链路支持
- 关注多资源并发加载表现
- 关注弱网和丢包场景
- 配合 CDN / 反向代理开启支持

#### 5. 使用层的作用过程: 从一行 `fetch` 到真实网络发生了什么

可以把下面这段：

```js
const res = await fetch('https://api.example.com/user?id=1', {
  method: 'GET',
  headers: { Accept: 'application/json' }
})
```

拆成协议层视角的完整运行过程：

1. JS 调用浏览器提供的 `fetch` API
2. 浏览器检查缓存、连接池、同源策略、Cookie、CORS 等浏览器规则
3. 若没有现成连接，触发 DNS 解析
4. 建立 TCP 或 QUIC 连接
5. 若是 HTTPS，再做 TLS 握手
6. 组装 HTTP 请求行、请求头和可选 body
7. 数据向下逐层封装，最终变成网卡可发送的帧
8. 服务端收到后逐层解封装，交给 Web 服务器/应用服务器
9. 应用生成响应
10. 响应再逐层封装回传
11. 浏览器逐层解封装，拿到 HTTP 响应
12. `fetch` Promise resolve，JS 才能继续 `await res.json()`

这段链路的加分点是：**前端代码只写了一行 `fetch`，但浏览器实际上帮你调度了 DNS、连接复用、TLS、HTTP、缓存、Cookie、CORS、流读取等一大堆机制。**

#### 6. 协议运行中的常见工程问题和排查抓手

##### DNS 层

现象：

- 域名偶发打不开
- 某些地域访问特别慢
- CDN 调度异常

抓手：

- `nslookup` / `dig`
- 观察 TTL
- 对比本地、运营商、公共 DNS 结果

##### TCP / TLS 层

现象：

- 首包慢
- 建连慢
- HTTPS 明显比 HTTP 慢很多
- 弱网下大量超时

抓手：

- 看是否复用连接
- 看 TLS 版本 / 证书链
- 看 RTT、重传、丢包
- Chrome DevTools Timing / Wireshark

##### HTTP 层

现象：

- 状态码异常
- 缓存失效
- Cookie 没带上
- 预检请求太多

抓手：

- Request/Response Headers
- Cache-Control / ETag / Vary
- CORS headers
- Set-Cookie / SameSite / Domain / Path

##### WebSocket / SSE 层

现象：

- 连不上
- 断线频繁
- 数据重复
- AI 输出流中断

抓手：

- 是否成功升级到 101
- 代理是否支持 Upgrade
- 心跳/超时策略
- SSE 是否被中间层缓冲
- Nginx/CDN 是否禁掉流式刷新

#### 7. 面试里把“协议原理 + 工程落地”讲完整的高分模板

如果面试官让你把网络协议“讲透”，可以按这套顺序回答：

1. 先讲分层职责：链路层、本地寻址、网络层、传输层、安全层、应用层
2. 再讲一条真实访问链路：接入网络 -> 配置 IP -> ARP 找下一跳 -> DNS 解析 -> TCP/QUIC -> TLS -> HTTP
3. 再补运行时数据长相：
   - Ethernet Frame
   - ARP Request/Reply
   - IP Packet
   - TCP Segment
   - TLS Record
   - HTTP Request/Response
   - WebSocket Frame
   - SSE event-stream
4. 再落到前端工程 API：
   - `fetch/axios`
   - `WebSocket`
   - `EventSource`
   - `ReadableStream`
5. 最后补实战问题：
   - 缓存
   - CORS
   - 长连接
   - 心跳
   - 代理与网关
   - 弱网与重试

如果能这样讲，面试官会感受到你不是“背协议名词”，而是**真的把协议、浏览器、服务端、工程实践和排障视角串成了一张完整运行图**。
