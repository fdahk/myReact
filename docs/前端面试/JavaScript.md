# JavaScript 面试深答手册

## 使用方式

这个文档按高频问题组织，答案统一采用可直接口述的一段式深答写法，不只给结论，还补上必要的概念解释、底层机制、存在意义、典型场景、常见坑点和可自然延伸的追问，目标不是背八股，而是把知识点讲成一条能自洽的逻辑链。文中遇到稍微专业一点、对初学者不够直观的术语，我会尽量在正文里顺手解释；如果某个词读到这里还是觉得抽象，可以再看文末新增的“术语补充题”。

## 一、语言基础

1. `var`、`let`、`const` 的核心区别是什么？

答：`var`、`let`、`const` 都是声明变量的方式，但本质区别在于变量绑定被放进什么作用域、什么时候能访问、能不能重复声明以及后续能否重新赋值。`var` 属于函数作用域，创建执行上下文时会先进入变量环境并初始化为 `undefined`，所以会出现变量提升和重复声明覆盖的问题；`let`、`const` 属于块级作用域，会进入词法环境，在执行到声明语句之前处于暂时性死区，用来强制开发者遵守“先声明再使用”的规则。它们存在的意义，本质上是在解决早期 `var` 带来的作用域污染和维护成本问题，因此现代工程里通常默认用 `const`，需要重新赋值时才用 `let`，尽量避免 `var`。
2. 什么是暂时性死区？
答：暂时性死区就是 `let` 和 `const` 在所属作用域内已经被创建、但在执行到声明语句之前仍然不能访问的那段区域，它的重点不是“没有提升”，而是“提升了但不能被读取”。底层上这和词法环境有关，词法环境你可以先理解成“JavaScript 在当前作用域里用来记录变量的一张表”，标识符也就是变量名、函数名这类名字会先登记进去，但值还没进入可用状态，所以引擎会直接抛出 `ReferenceError`，也就是“引用了当前还不能访问的变量”这类错误。它存在的意义是把很多本来运行后才暴露的变量误用问题，提前变成语言层面的硬错误，从而提升代码的确定性和可维护性。
3. JavaScript 的数据类型有哪些？
答：如果从规范角度回答，JavaScript 的值类型主要包括 `Undefined`、`Null`、`Boolean`、`Number`、`String`、`Symbol`、`BigInt` 和 `Object`，前七类通常被称为基本类型，`Object` 对应对象类型，数组、函数、日期、正则本质上都属于对象。很多人会把“引用类型”当成一类说法，它更偏工程表述，用来强调对象在变量里保存的不是实体本身，而是指向内存里那份对象数据的位置，你也可以先把它简单理解成“变量手里拿着对象的地址”。这个知识点的意义在于，它决定了比较规则、拷贝方式、序列化边界和内存管理方式；尤其函数虽然能被调用，但底层仍然是对象，所以既能执行，也能挂属性、挂原型、参与继承和反射操作。
4. 基本类型和引用类型的根本区别是什么？
答：根本区别不在“一个简单一个复杂”，而在于变量里到底保存的是什么，以及复制和传递时复制的是值本身还是引用关系。基本类型通常直接以值语义参与赋值、比较和传参，新的变量拿到的是一份独立值；对象类型则更像拿到一个地址或引用，多个变量可能同时指向同一块内存，所以看起来在改 A，实际是在改那块共享对象。这个差异是浅拷贝、深拷贝、状态污染、不可变更新这些工程问题的源头，也是为什么前端状态管理特别强调不要直接改原对象的根本原因。
5. `null` 和 `undefined` 的区别是什么？
答：`undefined` 更偏系统层面的“缺省状态”，表示一个值目前没有被赋予明确内容，比如变量声明未赋值、函数没有显式返回值、对象不存在的属性读取结果都可能是它；`null` 更偏业务层面的“主动置空”，表示这里本来应该有值，但我明确告诉你当前为空。换句话说，`undefined` 更像“还没有”，`null` 更像“这里就是空”。它们的语义差异非常重要，因为接口设计、状态流转和空值判断往往需要区分“未初始化”和“刻意置空”。
6. `==` 和 `===` 的区别是什么？为什么实际开发通常更推荐 `===`？
答：`===` 是严格相等，比较时不做隐式类型转换，只有类型相同且值相等才返回 `true`；`==` 是宽松相等，会先按规范做一系列抽象相等比较转换，比如字符串转数字、布尔转数字、对象先走 `ToPrimitive`，这个词可以先理解成“对象在参与比较前，先被想办法转成普通值，比如数字或字符串”，因此结果往往不符合直觉。`===` 更推荐的根本原因，不是因为它“高级”，而是因为它让比较规则更稳定、更可预测，能减少很多由隐式转换引起的维护成本。只有在非常明确地想利用 `null == undefined` 这个特性去同时判断两者时，才可能谨慎使用 `==`。
7. JavaScript 隐式类型转换的常见规则有哪些？
答：隐式类型转换本质上是引擎为了让不同类型的值能够参与统一运算或比较，而自动执行的一套抽象操作，核心可以抓三层：字符串拼接场景倾向转字符串，数学运算和关系比较常倾向转数字，布尔值参与运算时会先变成 `0` 或 `1`。真正复杂的地方在对象参与计算时会先执行 `ToPrimitive`，也就是尝试通过 `Symbol.toPrimitive`、`valueOf`、`toString` 把对象转成原始值后再继续计算。所以很多经典题的本质并不是死记结果，而是看清楚它到底经历了哪几步转换。
8. `NaN` 为什么不等于自己？
答：`NaN` 表示“不是一个有效数值结果”，比如非法数学运算、无法转成正常数值的结果都会落到它身上，它的设计重点是传播错误数值状态，而不是充当普通数值。规范规定 `NaN` 与任何值都不相等，包括它自己，是因为它代表的不是某个确定数字，而是一种数值计算失败的信号。也正因为如此，判断 `NaN` 不能靠 `===`，更稳妥的方式是 `Number.isNaN()`。
9. `typeof` 和 `instanceof` 分别适合解决什么问题？
答：`typeof` 适合判断基本类型和函数，因为它看的是值在语言层面属于什么基础类别；`instanceof` 适合判断一个对象是否出现在某个构造函数的原型链上，它解决的是“这个对象是不是由这条原型体系产生”的问题。两者看似都在判断类型，但本质上一个更偏值分类，一个更偏原型链关系判断，所以 `typeof null` 会得到 `object` 这个历史遗留结果，而跨 iframe 场景下 `instanceof` 也可能失真。实际工程里经常是 `typeof`、`Array.isArray`、`Object.prototype.toString.call()` 组合使用。
10. 如何准确判断一个变量是数组、对象、函数、日期、正则？
答：最稳妥的思路不是迷信某一个 API，而是根据要判断的目标选最合适的工具：数组优先用 `Array.isArray()`，函数常用 `typeof fn === 'function'`，像日期、正则、`Map`、`Set` 这类内置对象，优先用 `Object.prototype.toString.call(value)` 获取内部标签，你可以先把“内部标签”理解成对象内部自带的一种类型标记。它的底层意义在于，JavaScript 的类型判断并不是单点问题，而是值分类、原型链、对象内部标记等多套机制交织在一起，因此不存在一个万能判断函数能覆盖所有场景。

## 二、作用域与闭包

1. 什么是作用域？

答：作用域本质上是名字解析规则，也就是一个标识符在代码的哪一片区域内是可访问的，变量、函数、类声明最终都要受它约束。它存在的意义是做隔离和约束，避免所有名字都堆到全局里互相污染，让不同逻辑片段拥有自己的私有上下文。JavaScript 采用词法作用域，意味着变量能不能访问，主要由它写在什么位置决定，而不是由谁来调用决定；也正因为如此，作用域是理解变量查找、闭包、模块封装、块级变量安全的基础。
12. 什么是词法作用域？
答：词法作用域就是变量的可访问范围在代码写出来时就已经基本确定，核心看函数和块定义在哪里，而不是运行时从哪里被调用。底层上，函数在创建时会保存对其外层词法环境的引用，这里的词法环境你可以理解成“函数出生时周围那层变量上下文”，因此即使它以后跑到别处执行，也仍然沿着定义时那条环境链去找变量。这个机制存在的意义，是让代码的可读性和推理成本大幅下降，因为你看到一段函数代码时，就能顺着源代码结构判断它可能访问哪些外部变量。
13. JavaScript 中有哪些作用域类型？
答：从工程视角看，JavaScript 里常见的作用域可以分为全局作用域、函数作用域、块级作用域和模块作用域。全局作用域生命周期最长，最容易形成污染；函数作用域是 `var` 时代的核心隔离边界；块级作用域是 `let` 和 `const` 带来的更细粒度控制；模块作用域则让每个 ES Module 顶层天然独立，不会像旧时代脚本那样默认把变量暴露到全局。它们存在的意义不是分类好看，而是帮助我们明确变量生命周期、可见范围和系统边界。
14. 什么是作用域链？
答：作用域链就是变量查找时从当前环境开始，沿着外部词法环境一层层向上寻找，直到全局甚至最终找不到为止的那条查找路径。它的本质不是一条真实的数组让你操作，而是执行上下文在运行时根据词法环境形成的一种解析关系。这个机制存在的意义是让内层逻辑能够自然复用外层上下文，而不需要把所有变量都显式传参，因此闭包、嵌套函数、模块封装都能成立。
15. 什么是闭包？
答：闭包不是某种特殊语法，而是函数和它创建时所处词法环境的组合，也就是说，函数即使脱离了原来的执行位置，依然可以访问定义时外层环境中的变量。很多人把闭包理解成函数记住了变量，更准确地说是函数持有了对外层变量记录的引用，你也可以把它先理解成“函数把外层那份变量上下文带在了身上”，所以相关变量不会随着外层函数执行结束就立刻失效。闭包存在的意义非常大，它让 JavaScript 拥有了私有变量、函数工厂、事件回调、异步回调、缓存和模块封装这些能力。
16. 闭包为什么能访问外层变量？
答：因为函数在创建时就已经和外层词法环境建立了关联，这个关联不是把值拷贝进去，而是保存了一条指向外层变量记录的引用，所以函数后续执行时仍然能沿着这条关系拿到变量。只要这个函数还活着并被引用，相关词法环境就仍然处于可达状态，这里的可达你可以理解成“程序还能顺着引用关系找到它”，垃圾回收器就不能随意回收它。这个机制的本质是词法作用域在运行时的延续。
17. 闭包有哪些常见应用场景？
答：闭包最典型的应用场景包括私有变量封装、函数工厂、事件处理器、异步回调、缓存、防抖节流和模块模式，本质上都是让某段逻辑在未来执行时仍然能用到创建它时的上下文。例如计数器函数通过闭包保存内部状态，事件回调通过闭包拿到组件局部变量，防抖节流通过闭包保存定时器句柄。它的价值在于既能保留状态，又不用把状态暴露成全局变量。
18. 闭包可能带来哪些问题？
答：闭包的问题不在于它不好，而在于它会延长变量和对象的生命周期，如果设计不当，就会带来内存占用增加、状态来源不清晰、旧值被意外捕获、调试困难等问题。比如回调函数长期持有大对象引用、定时器和监听器不释放、React 中因为依赖写错而拿到旧闭包，都是典型场景。闭包的本质风险在于引用关系被拉长，所以正确思路不是回避闭包，而是控制它持有的内容、生命周期和释放时机。
19. 循环里绑定事件为什么容易出现闭包问题？怎么解决？
答：这个问题的本质是多个回调共享了同一个外层变量绑定，而不是每个回调拥有自己独立的一份值，典型情况就是 `for` 循环用 `var` 声明索引，最后所有回调访问到的都是循环结束后的同一个结果。因为 `var` 只有函数作用域，不会为每次循环创建新的词法环境，而 `let` 会在每轮迭代中生成新的块级绑定，所以天然能解决这个问题。除此之外，也可以通过 IIFE 或显式传参把当前值固化进去。
20. 你如何向面试官解释“闭包不是特性题，而是日常开发题”？
答：因为只要代码里存在回调、异步、事件、模块封装、函数返回函数、状态缓存，闭包就在工作，区别只在于你有没有意识到它。很多线上问题表面看像事件没解绑、定时器旧值、Hook 依赖错了，其实底层都能追到闭包和词法环境上。换句话说，闭包不是一块孤立知识点，而是 JavaScript 运行模型的一部分；真正理解它之后，你解释 bug、设计 API、控制内存和写抽象代码的能力都会明显更强。

## 三、执行上下文与 this

1. 什么是执行上下文？

答：执行上下文可以理解为代码真正开始运行时所处的那一帧运行环境，你也可以先把它想成“当前这段代码执行时随身带着的一包运行信息”，里面会准备变量绑定、词法环境、外部环境引用以及 `this` 等信息。全局代码会创建全局执行上下文，函数每被调用一次就会创建对应的函数执行上下文，它们由调用栈统一管理，调用栈可以先理解成“函数执行时排队进出的一摞栈结构”，先进后出。这个概念存在的意义，是把代码写出来和代码真正运行起来区分开，因为变量提升、函数调用、闭包捕获、`this` 绑定这些问题，本质上都是执行上下文层面的问题。
22. 执行上下文中包含哪些重要信息？
答：执行上下文里最值得面试时提到的核心信息包括变量环境、词法环境、外部环境引用和当前 `this` 绑定。变量环境和词法环境决定了当前代码有哪些标识符、分别处于什么状态；外部环境引用决定了变量继续往哪一层找；`this` 绑定决定了函数内部通过 `this` 能访问哪个对象。真正掌握这一点后，你会发现变量提升、闭包、`this` 和调用栈并不是分散知识点，而是在同一个执行模型下的不同表现。
23. 函数调用时 `this` 是如何确定的？
答：`this` 的确定核心不看函数定义在哪里，而看函数是怎么被调用的，也就是调用位置和调用形式。普通直接调用走默认绑定，对象点调用走隐式绑定，`call`、`apply`、`bind` 走显式绑定，`new` 调用走构造绑定，箭头函数则根本不参与这套规则，而是捕获定义时外层的 `this`。`this` 这个机制存在的意义，是让同一个函数在不同调用者下能复用逻辑并访问不同上下文，但也正因为依赖调用方式而容易出错。
24. 默认绑定、隐式绑定、显式绑定、`new` 绑定分别是什么？
答：默认绑定是普通函数独立调用时的 `this` 规则，非严格模式下通常指向全局对象，严格模式下是 `undefined`；隐式绑定指对象通过属性访问形式调用函数，此时 `this` 指向调用它的对象；显式绑定是通过 `call`、`apply`、`bind` 主动指定 `this`；`new` 绑定则发生在构造调用中，`this` 会指向新创建的实例对象。它们本质上都是语言为函数复用提供的上下文绑定规则，优先级通常是 `new` 绑定高于显式绑定，显式绑定高于隐式绑定，隐式绑定高于默认绑定。
25. `call`、`apply`、`bind` 的区别是什么？
答：这三个方法的共同点是都能显式指定函数执行时的 `this`，区别主要在于执行时机和参数组织方式。`call` 和 `apply` 都会立刻调用函数，前者按参数列表传入，后者按数组或类数组传入，这里的类数组可以先理解成“长得像数组、有下标和 length，但不是真正数组的对象”；`bind` 不会立刻执行，而是返回一个绑定好 `this` 和部分参数的新函数，适合回调透传、事件处理和函数柯里化场景。它们存在的本质意义，是把函数调用权和上下文绑定权拆开，让函数既可以被复用，也可以在运行时重新指定上下文。
26. 箭头函数为什么没有自己的 `this`？
答：因为箭头函数的设计目标不是模拟普通函数，而是提供一种更轻量的函数表达式，让回调场景下的上下文继承更自然，所以它在语言层面故意不创建自己的 `this`、`arguments`、`super` 和 `new.target`，这里的 `new.target` 你可以先把它理解成“当前函数是不是通过 `new` 调用的一个内部标记”。底层理解上，它不是有一个固定 `this`，而是根本不走普通函数那套动态绑定规则，而是直接从外层最近一层非箭头函数或全局上下文中捕获 `this`。它存在的意义主要是减少回调里 `this` 丢失的问题。
27. 箭头函数的 `this` 到底指向谁？
答：箭头函数的 `this` 不是调用时确定的，而是在定义时从外层词法环境中继承下来的，所以更准确的说法是它使用外层的 `this` 绑定结果。如果外层是普通函数，就继承那个普通函数执行时的 `this`；如果再往外都没有普通函数，就落到当前顶层上下文的 `this`。很多人说箭头函数永远指向定义处对象其实不严谨，因为它继承的是外层 `this`，不是定义位置旁边某个看起来像对象的字面量。
28. 面试里如何解释普通函数和箭头函数的使用边界？
答：判断边界的核心不是哪种写法更新，而是你是否需要函数拥有自己的动态上下文和构造能力。需要自己的 `this`、`arguments`、`prototype`、`new` 调用能力时，应使用普通函数；只是把一段逻辑作为回调，并且明确希望继承外层上下文时，箭头函数通常更合适。换句话说，普通函数更像可独立参与运行时绑定的行为单元，箭头函数更像借用外层语义环境的轻量表达式。
29. 手写 `bind` 需要考虑哪些细节？
答：手写 `bind` 时真正拉开水平差距的，不是能不能把 `this` 改掉，而是能否完整还原语言行为：第一要支持预置参数和后续参数拼接，第二要返回新函数而不是立刻执行，第三要兼容被 `new` 调用时 `this` 优先指向实例而非原绑定对象，第四还要尽量维持原型链关系，否则构造场景会失真。它的本质难点在于 `bind` 同时做了上下文绑定、参数固化和构造兼容三件事，并不是一个简单的包装函数。
30. `new` 一个函数时，底层大致发生了什么？
答：`new` 的执行过程可以概括为四步：先创建一个全新的空对象，再把这个对象的内部原型链接到构造函数的 `prototype` 上，然后以这个新对象作为 `this` 去执行构造函数，最后判断构造函数有没有显式返回对象类型的值，如果有就返回那个对象，否则返回新创建的实例。它存在的意义是让函数具备构造实例和组织原型复用的能力，这道题的底层重点其实是把 `this`、原型链和返回值规则三件事串起来。

## 四、原型与继承

1. 什么是原型？

答：原型可以理解为对象间共享能力的那层公共对象，它不是类的拷贝副本，而是一种属性查找时的委托目标。JavaScript 中很多对象自身并不直接拥有全部方法，比如数组实例能调用 `map`、`filter`，并不是每个数组都拷贝了一份这些函数，而是它们在访问不到自身属性时，会沿着内部原型链接去对应原型对象上查找。原型机制存在的意义，是在不复制方法代码的前提下实现能力复用，从而降低内存占用并支持动态扩展。
32. 什么是原型链？
答：原型链就是对象在查找属性时，从当前对象出发，沿着内部原型引用一层层向上寻找，直到 `Object.prototype`，再到 `null` 为止的这条委托路径。它的本质不是继承复制了一整套父对象内容，而是当自己找不到时，把查找动作交给上层对象继续完成。这个机制存在的意义，是让 JavaScript 用对象之间的委托关系完成方法共享和继承扩展，因此理解原型链，本质上就是理解属性到底是怎么被找到的。
33. `__proto__`、`prototype`、`constructor` 分别是什么？
答：`prototype` 是函数在作为构造函数使用时，用来给实例共享成员的那个原型对象；`__proto__` 是对象内部指向其原型的访问入口，标准层面更推荐用 `Object.getPrototypeOf()` 理解它；`constructor` 则通常是原型对象上的一个属性，默认指回创建该原型体系的构造函数。三者经常被混淆，本质上是因为它们处在不同层次：`prototype` 属于函数，`__proto__` 属于对象实例，`constructor` 属于原型对象上的普通属性。
34. JavaScript 是如何通过原型链实现继承的？
答：JavaScript 的继承本质上不是复制父类内容，而是让子对象或子类型原型对象在查找属性时，能继续沿着父级原型对象向上委托，因此更准确地说它是一种原型委托机制。只要建立好“子实例 -> 子原型 -> 父原型”这条链路，子实例访问不到的成员就会继续去父原型上查找，从而形成方法复用和行为继承。它的意义在于用很轻量的方式实现复用，但代价是链路一深之后，行为来源会变得不够直观。
35. `class` 语法糖的本质是什么？
答：`class` 的本质仍然是构造函数、原型对象和原型链，只是语法上更接近传统面向对象语言，让代码可读性更强、组织结构更清晰。比如类方法本质上还是挂在 `prototype` 上，静态方法挂在构造函数本身上，`extends` 仍然是在建立原型链关系。它存在的意义不是改变底层机制，而是降低使用原型系统的心智负担，所以真正深入理解 `class`，最后还是得回到原型链、`new`、`super` 和属性查找过程上来。
36. ES5 常见继承方式有哪些？各自优缺点是什么？
答：ES5 常见继承方式包括原型链继承、借用构造函数继承、组合继承和寄生组合继承，它们可以理解成一条不断修补问题的演进链。原型链继承能复用父原型方法，但实例属性容易共享；借用构造函数能保证实例属性独立，却拿不到父原型方法；组合继承把两者结合起来，基本可用，但会执行两次父构造函数；寄生组合继承进一步优化了这点，通过中间对象建立原型连接，减少多余构造调用。理解这些方案的意义，不是为了背历史，而是看见 JavaScript 继承模型的本质约束。
37. 组合继承和寄生组合继承的区别是什么？
答：两者都想同时解决实例属性独立和原型方法复用两个问题，区别在于建立子原型链的方式不同。组合继承通常既在子构造函数里调用一次父构造函数给实例拷属性，又通过 `new Parent()` 给子原型连链路，因此父构造函数会被执行两次；寄生组合继承则通过 `Object.create(Parent.prototype)` 之类的方式建立原型关系，只在实例初始化时调用一次父构造函数，所以更高效也更干净。这个区别的本质，是是否把原型连接误写成了再 `new` 一次父类。
38. 为什么说 JavaScript 的继承本质上是对象之间的委托？
答：因为 JavaScript 并不像很多传统语言那样在创建子类时把父类成员真正复制一份给子类，而是在访问属性时，如果当前对象没有，就把查找动作沿原型链交给上层对象继续处理。也就是说，复用的本质是查找路径复用，不是数据复制。这样做的好处是共享方法、节省内存、改动原型后实例可感知，代价是行为来源可能分散在整条原型链上，需要开发者更理解运行时查找过程。
39. 面试官让你解释“继承”和“组合”时，你会怎么回答？
答：继承强调稳定的 is-a 关系，也就是子类型本身就是父类型的一种，适合抽象层次明确、变化频率低的体系；组合强调把多个能力按需拼起来，本质是 has-a 关系，灵活性更高、耦合更低。前端业务里通常更偏向组合，因为需求变化快、状态多、依赖关系复杂，如果用继承层层叠加，后续一改就容易牵一大片。这个问题真正考察的不是概念定义，而是你有没有能力根据业务变化频率和耦合成本去做设计选择。
40. 什么时候不该使用继承？
答：当关系本身不稳定、只是想复用一点局部能力、或者一旦继承下去会出现很深的层级和强耦合时，就不应该优先使用继承。因为继承一旦建立，子类型天然要依赖父类型的结构和演进方向，父类改动会影响整条链，这在变化快的业务场景里代价很高。前端组件、页面逻辑、工具能力更常见的更优方案往往是组合、配置、函数复用或策略模式，而不是强行搭一棵继承树。

## 五、异步与事件循环

1. JavaScript 为什么是单线程？

答：通常说 JavaScript 是单线程，准确表达是同一个主线程上的 JavaScript 执行是串行的，这和它最早在浏览器中的定位有关，因为它需要频繁操作 DOM，如果允许多个脚本线程同时改界面，就会带来复杂的竞争条件和一致性问题。单线程的意义在于让页面状态变更模型更简单，但这并不代表浏览器只有一个线程，网络请求、定时器、渲染、I/O 等可以由宿主环境在别的线程或系统能力中协作完成，这里的宿主环境可以先理解成“让 JavaScript 运行起来的平台，比如浏览器和 Node.js”。JavaScript 的难点因此不在多线程共享内存，而在如何通过事件循环把异步结果安全地送回主线程执行。
42. 什么是同步任务和异步任务？
答：同步任务会直接进入当前调用栈并按顺序立即执行，前一个没结束，后一个就必须等待；异步任务则会先交给宿主环境处理等待阶段，例如计时、网络、I/O、用户交互等，当条件满足后再把对应回调放入任务队列，等待主线程合适的时候执行。它们的本质区别不是一个快一个慢，而是是否会阻塞当前调用栈继续向下执行。异步机制存在的意义，是把等待这件事从主线程中剥离出去，否则一旦请求网络或等用户操作，整个页面都会卡死。
43. 事件循环的完整流程是什么？
答：事件循环可以概括为，先执行当前宏任务里的同步代码，期间如果产生微任务就暂存，等当前宏任务执行完毕后立即清空微任务队列，再进入下一个宏任务，如此往复。底层上，调用栈只负责执行当前任务，事件循环负责协调任务何时进栈；宿主环境则负责在合适时机把定时器回调、I/O 回调、UI 事件等送入对应队列。它存在的意义，是在单线程前提下让同步、异步、界面更新能够有序协作，所以凡是涉及 Promise、`setTimeout`、`await`、DOM 事件执行顺序的问题，最后都要回到事件循环这条主线来解释。
44. 宏任务和微任务分别有哪些？
答：在浏览器里，常见宏任务包括整段脚本执行、`setTimeout`、`setInterval`、部分 I/O 和 UI 事件回调；常见微任务包括 Promise 的 `then/catch/finally` 回调、`queueMicrotask`、`MutationObserver`。两者的关键差异不是名字，而是调度优先级：每完成一个宏任务，事件循环都会先把当前产生的微任务全部清空，然后才会去取下一个宏任务。微任务存在的意义，是让某些需要尽快在当前任务结束后执行的逻辑拥有更高优先级。
45. 为什么 Promise 回调会比 `setTimeout` 更早执行？
答：因为 Promise 回调进入的是微任务队列，而 `setTimeout` 回调进入的是后续某个宏任务队列，事件循环的规则是当前宏任务结束后先清空微任务，再去执行下一个宏任务，所以 Promise 的回调通常会先执行。这里更准确的说法不是 Promise 天生比定时器快，而是同一轮事件循环里微任务优先于后续宏任务。理解这点后，很多执行顺序题就不需要死记答案了。
46. 浏览器环境和 Node.js 环境的事件循环有什么差异？
答：两者的核心思想相同，都是调用栈、任务队列和循环调度，但 Node.js 的实现会更像分阶段调度模型，比如 timers、poll、check 等阶段，而且 `process.nextTick` 的优先级又比常规微任务更特殊。也就是说，事件循环不是一套完全脱离宿主环境的抽象理论，它最终一定要落到具体平台的调度实现上。分析这类问题时一定要先区分宿主环境，否则很容易把浏览器和 Node 的执行顺序混为一谈。
47. `async/await` 本质上是基于什么实现的？
答：`async/await` 的本质是 Promise 语义上的语法糖，它把原本链式的异步流程改写成更接近同步代码的形式，提升的是表达能力和可读性，而不是改变异步底层模型。`await` 后面仍然要等一个 Promise 或类 Promise 结果，异常传播仍然依赖 Promise 的 rejected 通道，所以它没有神奇地消灭异步，只是把异步组合写得更自然。它存在的意义，是降低复杂异步流程的心智负担，让串行逻辑、错误处理和资源收尾更容易读懂。
48. `await` 为什么会让出执行权？
答：因为 `await` 遇到一个 Promise 后，不会阻塞整个线程死等，而是把当前 `async` 函数后续逻辑拆成一段待恢复的 continuation，这个词你可以先理解成“从暂停点往后还没执行的那部分代码”，再挂到 Promise 完成后的微任务中，当前函数先暂停，主线程继续往下执行别的同步任务。它本质上是在当前执行点做了一次阶段性让渡，而不是开启了一个并行线程。这个机制存在的意义，是在保留接近同步写法的同时，仍然遵守 JavaScript 单线程和事件循环的调度规则。
49. 如何处理多个异步请求的串行与并行？
答：如果后一个请求依赖前一个请求的结果，就应该串行执行，用 `await` 或 Promise 链按顺序组织；如果彼此独立，就应该尽量并行发起，常见做法是 `Promise.all`、`Promise.allSettled` 或自定义并发池控制。这个问题的本质不是记住几个 API，而是先判断任务之间有没有数据依赖、失败是否会影响整体、是否需要限流。真正成熟的回答还会补充一点：并行不等于无限并发，请求量大时还要考虑服务端压力、浏览器连接数限制和失败重试策略。
50. 如何设计一个可取消、可重试的异步请求流程？
答：一个成熟的异步请求流程通常至少要考虑取消、超时、重试、幂等性和竞态处理。幂等性你可以先理解成“同一个请求重复执行多次，结果是否仍然安全一致”，竞态可以理解成“多个请求或多个结果谁先回来不确定，可能互相覆盖”，取消可以借助 `AbortController` 之类的机制让失效请求尽早停止，重试要带最大次数和退避策略，避免瞬时故障时直接把服务端打爆，同时要区分哪些错误可重试、哪些不可重试，尤其非幂等操作不能盲目重放。这个设计的本质，是把请求从一次简单调用升级为一个可治理的异步状态机，这里的状态机你可以先理解成“按不同状态和状态切换规则运行的一套流程模型”。

## 六、Promise

1. Promise 的三种状态是什么？

答：Promise 只有三种状态，`pending`、`fulfilled`、`rejected`，也就是进行中、成功和失败。它本质上是对异步结果的一层统一抽象，把原本分散在回调里的未来结果收敛成一个可组合、可链式传递的状态机。Promise 存在的意义，不只是让代码看起来更整齐，而是让成功通道、失败通道和后续处理都拥有统一语义。
52. Promise 状态为什么只能从 `pending` 变成一次最终状态？
答：因为 Promise 被设计成异步结果的不可逆承诺，一旦最终状态还能反复改写，后续链式调用、错误传播和组合语义都会失去确定性。也就是说，状态只能落定一次，不是拍脑袋规定，而是为了让异步流程具备可预测性和可推理性。这个设计和函数返回值只能有一个最终结果是同一个思路，只不过 Promise 把这个结果延迟到了未来。
53. `then`、`catch`、`finally` 的执行规则是什么？
答：`then` 用来处理成功分支，`catch` 用来处理失败分支，`finally` 则不区分成功还是失败都会执行，通常适合资源释放、关闭 loading、收尾日志等逻辑。`finally` 一般不会改写前面链路的结果，除非它自己抛错或返回一个 rejected Promise。这个设计的意义，是把业务分支处理和统一收尾分开，让代码结构更清晰，也更接近同步代码里的 `try/catch/finally` 语义。
54. `then` 为什么可以链式调用？
答：因为每次调用 `then` 都会返回一个新的 Promise，而不是把结果直接塞回原 Promise，所以后面的链路可以继续接着处理上一步的结果或错误。这个设计非常关键，它让 Promise 不只是一个异步结果容器，而是一套异步流程组合协议。也正因为每一层都返回新 Promise，普通值、抛错、返回 Promise、返回 thenable 才能被统一纳入同一条链中继续解析，这里的 thenable 可以先理解成“虽然不是原生 Promise，但只要它有 `then` 方法，Promise 规范就会把它当成类似 Promise 的对象来处理”。
55. Promise 的值穿透是什么？
答：值穿透指的是在 `then` 或 `catch` 中如果没有传入有效的处理函数，当前成功值或失败原因不会被吞掉，而是继续沿着链往后传。这不是一个边角现象，而是 Promise 链能够支持局部处理、统一兜底的基础，否则每一层都必须写完整处理函数，链式模型就失去意义了。很多人只记结论不理解本质，实际上它反映的是 Promise 对结果传递连续性的设计。
56. Promise 错误冒泡是怎么发生的？
答：Promise 的错误冒泡，本质上是失败状态会沿着链自动向后传播，直到被某个 `catch` 或带失败处理器的 `then` 接住。这个机制之所以成立，是因为 Promise 会把同步抛错也统一包装成 rejected 状态，从而把同步异常和异步失败纳入同一条错误通道。它存在的意义是让复杂异步流程可以集中收敛错误，而不是在每一层都手动 try/catch。
57. `Promise.all`、`Promise.allSettled`、`Promise.race`、`Promise.any` 的区别是什么？
答：`Promise.all` 要求全部成功，只要有一个失败就整体失败；`Promise.allSettled` 不要求全成，而是等全部结束后把每一项结果都汇总出来；`Promise.race` 关注谁最先结束，不管成功还是失败；`Promise.any` 则关注谁最先成功，只有全部失败才整体失败。它们的本质区别不是 API 名字，而是业务目标不同：有的是追求全成功，有的是要尽快拿到一个可用结果，有的是要完整收集结果做兜底。
58. 如何手写一个简化版 Promise？
答：手写 Promise 的核心不是把三种状态写出来，而是实现一套规范化的状态流转和回调调度机制，至少要覆盖状态只落定一次、异步收集成功和失败回调、`then` 返回新 Promise、异常自动转 rejected，以及返回 Promise 或 thenable 时的继续展开，这里的 thenable 还是指“带有 `then` 方法、会被当成类似 Promise 处理的对象”。真正难的地方不在表面结构，而在链式解析规则，因为这决定了整个 Promise 模型是否可组合、是否兼容规范。面试里如果能说到 thenable 解析和循环引用保护，深度会明显更好。
59. Promise 和回调函数相比解决了哪些问题？
答：Promise 主要解决了裸回调时代的三类痛点：第一是回调嵌套深、流程横向蔓延导致难读；第二是成功和失败处理分散，错误通道不统一；第三是多个异步任务之间缺少稳定的组合协议。Promise 的价值不是让回调消失，而是把回调包装进一套可链式、可组合、可统一兜底的语义体系中，所以多个请求串行、并行汇总、统一错误收敛才真正变得优雅。
60. Promise 有没有解决回调地狱的全部问题？
答：没有，Promise 解决的是控制流组织和错误通道统一的问题，但取消、超时、并发治理、资源释放、竞态控制这些问题仍然需要额外设计。也就是说，Promise 不是万能解法，它只是把异步流程的表达能力提升了一大步，但不会自动替你完成系统治理。成熟的工程实现里，Promise 往往只是基础设施，上层还要叠加取消协议、重试策略、超时处理和状态机设计。
61. Promise 上常见静态方法除了聚合方法，还有哪些？
答：除了 `all`、`allSettled`、`race`、`any` 这类聚合方法，最常用的静态方法就是 `Promise.resolve` 和 `Promise.reject`。它们的价值不只是快速包一层，而是把同步值、异常值、thenable 都统一拉进 Promise 语义里，方便函数对外维持一致返回类型。一个异步接口如果无论快路径慢路径都返回 Promise，调用方的组合成本就会明显降低，这就是这些静态方法存在的真正意义。
62. `Promise.resolve` 和 `Promise.reject` 的典型使用场景是什么？
答：`Promise.resolve` 常用于把普通值、已有 Promise 或 thenable 统一接入 Promise 链，比如接口封装层想保证无论内部逻辑同步还是异步，对外都返回 Promise；`Promise.reject` 则适合快速生成失败态，用来做前置校验失败、权限拒绝、参数非法短路等场景。它们的本质价值，是把不同来源的结果统一收束到同一套异步协议中，而不是让每个调用方都分别判断同步值和异步值。
63. `Promise.all` 和 `Promise.allSettled` 的返回结果结构有什么差异？
答：`Promise.all` 成功时返回一个按输入顺序排列的纯结果数组，但只要任意一个失败就会整体 rejected，因此拿不到其他任务的最终状态；`Promise.allSettled` 则一定等所有任务结束，返回的每一项都包含 `status`，成功项有 `value`，失败项有 `reason`。两者的根本区别不只是返回结构不同，而是语义目标不同：前者强调全成才有意义，后者强调无论成败都要把结果汇总回来给上层决策。
64. Promise A+ 规范主要约束了哪些行为？
答：Promise A+ 规范真正核心约束的是 `then` 的行为，包括状态落定后回调如何执行、回调返回普通值怎么办、返回 Promise 或 thenable 怎么办、抛异常怎么办，以及如何避免自解析死循环。换句话说，它统一的不是“三种状态”这种表面结构，而是链式调用时结果如何稳定传播。也正因为规范把重点放在 `then` 上，所以很多自己写的 Promise 看起来能跑几个 demo，但一遇到嵌套 thenable 或异常传播就会暴露问题。
65. 为什么说 Promise A+ 的核心在于 `then` 的行为定义？
答：因为 Promise 真正难、也真正有价值的地方，不是有个 `pending/fulfilled/rejected` 状态机，而是每次调用 `then` 后如何生成新的 Promise，并把上一步结果、异常、异步值继续可靠地传下去。只要 `then` 的行为不稳定，链式组合、错误冒泡、值穿透这些 Promise 最核心的能力都会崩掉。所以说 Promise A+ 的核心在于 `then`，本质上是在强调 Promise 的价值来自可组合的协议，而不是来自某个对象长得像 Promise。

## 七、对象、拷贝与不可变

1. 浅拷贝和深拷贝的区别是什么？

答：浅拷贝只会复制第一层结构，内部嵌套对象和数组仍然共享同一份引用；深拷贝则会递归复制嵌套引用，使新旧对象彻底分离。这个问题的本质不是“复制得深不深”这么简单，而是你是否切断了原来那条引用关系。它的工程意义非常大，因为表单编辑、状态管理、缓存复用和撤销重做几乎都会碰到数据共享边界，一旦边界判断错了，就很容易出现状态污染。
67. 常见浅拷贝方式有哪些？
答：对象常用展开运算符和 `Object.assign`，数组常用展开、`slice`、`concat`，它们都只复制外层容器本身，不会继续递归处理内部引用。浅拷贝之所以常见，是因为很多业务更新并不需要把整棵数据树完全复制，只需要沿着变化路径做局部复制即可。也就是说，浅拷贝并不是低级方案，它恰恰是很多高性能不可变更新的基础。
68. 如何实现深拷贝？要考虑哪些边界情况？
答：深拷贝可以自己递归实现，也可以优先使用现代环境提供的 `structuredClone`，你可以先把它理解成“浏览器和运行环境内置的深拷贝能力”，但真正难点不在代码能不能递归，而在于边界是否处理完整，比如循环引用、`Date`、`RegExp`、`Map`、`Set`、`ArrayBuffer`、函数、原型链、属性描述符等，其中 `ArrayBuffer` 可以先理解成“用来存原始二进制数据的一块内存”，属性描述符可以理解成“属性除了值以外的额外规则，比如是否可写、是否可枚举”。深拷贝的本质是重建数据图，而不是简单把对象套娃遍历一遍，所以任何特殊对象和引用关系都会成为复杂点。工程上要先问自己是否真的需要全量深拷贝，因为它往往比想象中更贵。
69. 为什么 `JSON.parse(JSON.stringify(obj))` 不是可靠的深拷贝方案？
答：因为它本质上不是深拷贝工具，而是一种受限的 JSON 序列化再反序列化过程，所以会丢失函数、`undefined`、`symbol`、循环引用、原型链和很多内置对象信息，`Date` 还会被转成字符串。它之所以看起来能用，只是因为对纯 JSON 数据结构凑巧有效。换句话说，这个方案处理的是“可 JSON 化的数据”，不是“任意 JavaScript 对象”。
70. 什么是可变数据，什么是不可变数据？
答：可变数据强调原对象本身可以被直接改写，不可变数据强调每次变化都返回一个新值而保留旧值不动。它们的本质区别不只是写法风格，而是状态变更如何被表达和追踪：可变更新往往更直接，但容易造成共享引用污染；不可变更新则把每次变化都变成一次新的版本，便于比较、回溯和调试。很多前端框架推崇不可变更新，并不是因为它更“函数式”，而是因为它更利于做变更检测和状态管理。
71. 前端项目里为什么要尽量遵守不可变更新？
答：因为前端项目里状态往往会被多个组件、缓存层、渲染逻辑共享，如果直接原地修改，就很难判断到底哪里变了、谁改了、谁应该更新。不可变更新的价值在于把变化显式化，让引用变化本身就成为一种变更信号，这对 React 一类依赖浅比较优化渲染的框架尤其重要。更进一步说，不可变更新不是道德要求，而是一种降低副作用扩散范围的工程策略。
72. `Object.freeze` 有什么作用和局限？
答：`Object.freeze` 可以冻结对象的表层属性，阻止新增、删除和修改现有属性值，从而在一定程度上约束对象不被随意改写。它的意义主要在开发期约束和调试，比如帮助你尽早发现不该发生的直接修改；但它是浅冻结，嵌套对象仍然可能变化，而且它解决不了结构共享、版本管理、变更通知这些不可变体系中的核心问题。所以它更像一种防误操作护栏，而不是完整不可变方案。
73. `Object.assign` 和展开运算符在拷贝对象时有什么区别？
答：两者本质上都属于浅拷贝，都会遍历源对象的可枚举自有属性并把结果复制到目标对象或新对象上。展开运算符语法更简洁，更适合日常对象更新；`Object.assign` 更像显式的合并行为，适合多源对象合并或某些老代码风格。但要注意，两者都会触发 getter，这里的 getter 可以先理解成“读取某个属性时会自动执行的一段取值函数”，也都不复制原型链、都不处理深层嵌套，所以本质差异更多在书写体验而不是能力等级。
74. 数组去重有哪些方式？如何比较它们的复杂度和适用场景？
答：最常见、最推荐的方式是 `Set`，因为语义清晰、实现简单、时间复杂度通常更优；也可以用 `filter + indexOf`、双重循环或哈希表思路，但这些更多是为了展示原理。真正要注意的是，对原始值去重和对对象数组去重是两类问题，对象数组去重的关键不在于用不用 `Set`，而在于按什么 key 或组合 key 判断“相同”。所以面试里高质量回答应该从“判等标准”讲起，而不是只报几个 API 名字。
75. 你如何向面试官说明“引用类型导致的状态污染问题”？
答：我会说，所谓状态污染，本质上不是某段代码突然变脏了，而是两个看起来像独立变量的引用，实际上指向了同一份底层对象，所以你修改其中一处，另一处也被无意间改变。这个问题经常出现在表单草稿、弹窗编辑、缓存复用、props 派生状态、对象默认值复用等场景中。解决思路通常不是一上来就全量深拷贝，而是先明确数据共享边界，再按变化路径做最小必要复制。

## 八、数组与函数式编程

1. `map`、`forEach`、`filter`、`reduce` 的区别是什么？

答：`forEach` 的语义是遍历并执行副作用，不强调返回新结果；`map` 的语义是逐项映射，返回一个等长新数组；`filter` 用于按条件筛选，返回满足条件的新数组；`reduce` 则更通用，它通过累加器把一组输入折叠成任意形态的结果。它们真正的区别不只是返回值，而是表达意图不同。成熟的代码并不是所有遍历都能写，而是让读代码的人一眼看出你是在做遍历、副作用、映射、筛选还是聚合。
77. `reduce` 能解决哪些实际问题？
答：`reduce` 适合做一切“把多个输入收束成一个结果”的场景，比如求和、分组、计数、数组拍平、索引表构建、树转映射、状态机推进等。它的底层思想是维护一个累加器，让每一轮的输出成为下一轮的输入，因此比固定用途的 `map`、`filter` 更通用。也正因为它足够通用，所以既能体现抽象能力，也很容易被滥用，关键在于是否真的让代码更清晰。
78. 为什么说 `reduce` 强大但不应该滥用？
答：因为 `reduce` 的表达能力很强，强到很多不同逻辑都能塞进去，但“一切都能写”不等于“一切都该写”。如果一个场景本来用 `map`、`filter`、`some`、`every` 就能更直接地表达，硬塞到 `reduce` 里通常只会让代码阅读门槛升高。换句话说，`reduce` 的风险不在功能，而在可读性；真正成熟的使用方式，是把它留给那些确实属于聚合、归并和状态累计的问题。
79. `some` 和 `every` 的区别是什么？
答：`some` 表示只要有一个元素满足条件就返回 `true`，`every` 表示必须所有元素都满足条件才返回 `true`，两者都会在结果确定后立即短路停止遍历。它们的本质一个是在做存在性判断，一个是在做全称判断。面试里如果能再补充一句空数组上 `every` 返回 `true`、`some` 返回 `false`，就说明你不仅知道用法，还理解它们背后的逻辑定义。
80. 数组扁平化有哪些实现方式？
答：数组扁平化可以用原生 `flat`，也可以手写递归、迭代加栈、或者基于 `reduce` 递归组合实现。它的本质是把一棵嵌套结构重新线性展开，所以真正需要关注的不只是写法对不对，还包括扁平深度是否可控、递归是否可能栈溢出、大数组是否会产生过多中间对象。很多手写题的价值不在于会不会拍平，而在于你能不能顺手把复杂度、边界和场景一起讲出来。
81. 函数柯里化是什么？
答：柯里化就是把一个原本接收多个参数的函数，拆成一连串每次只接收部分参数的函数，直到参数凑齐后再真正执行。它的本质不是为了把函数写成很多层，而是把“固定配置”和“实时输入”拆开，让部分参数可以被提前固化形成可复用函数。这个思想在前端里很实用，因为很多逻辑其实都包含稳定配置和动态输入两部分。
82. 柯里化在前端开发里有哪些应用？
答：柯里化常见于校验器、日志器、埋点器、权限判断、请求封装和函数工厂场景，比如先固定模块名、埋点类型、校验规则或接口前缀，再返回真正处理运行时数据的函数。它的价值在于把公共配置前置，减少重复参数传递，让函数职责更清楚。需要注意的是，柯里化不是越多层越高级，如果只是一次性调用，直接传完整参数反而更清晰。
83. 函数组合是什么？
答：函数组合就是把多个小函数按顺序串起来，让前一个函数的输出成为后一个函数的输入，最终形成一条数据处理流水线。它的本质是在做职责拆分和流程拼接，让每个小函数专注单一转换，而整体逻辑通过组合形成更强能力。函数组合的意义在于提高复用性、可测试性和表达力，但前提是各函数的输入输出契约足够稳定，否则组合只会制造额外适配成本。
84. 防抖和节流分别是什么？适用场景有什么区别？
答：防抖强调高频触发后只在最后一次稳定下来时执行一次，适合搜索输入、表单校验、窗口 resize 收尾这类“最终结果更重要”的场景；节流强调在连续触发过程中按固定时间间隔稳定执行，适合滚动监听、拖拽、mousemove、曝光上报这类“过程反馈也重要”的场景。它们本质上都是频率治理手段，只是一个更偏减少无效收尾前的重复触发，一个更偏控制持续过程中的执行节奏。
85. 手写防抖和节流时要考虑哪些参数设计？
答：手写这类函数时，真正体现深度的不是能不能写出一个定时器，而是是否考虑了立即执行还是尾执行、定时器保存、取消能力、`this` 和参数透传、返回值处理，以及在异步请求场景下是否需要继续考虑竞态和取消。也就是说，防抖节流的工程实现不是一个十行算法题，而是一个围绕行为语义做设计的问题。

## 九、模块化

1. 为什么需要模块化？

答：模块化的根本目的不是把文件拆碎，而是把系统边界切清楚，让每个模块只暴露明确输入输出，内部实现尽量收敛，从而降低作用域污染、依赖混乱和牵一发而动全身的风险。它的意义在大型项目里尤其明显，因为没有模块边界，团队协作、测试、复用、缓存和替换实现都会越来越难。换句话说，模块化本质上是在做复杂度治理。
87. CommonJS 和 ES Module 的核心区别是什么？
答：CommonJS 更偏运行时加载和导出值快照式使用，典型写法是 `require` 和 `module.exports`；ES Module 则是标准化的静态模块系统，也就是“代码还没真正跑起来，工具就已经能提前看懂模块之间的依赖关系”，而且导出是实时绑定而不是简单值拷贝。这个差异非常重要，因为 Tree Shaking、预加载、循环依赖处理、构建优化都依赖静态结构，这里的 Tree Shaking 你可以先理解成“打包时把没用到的代码摇掉、删掉”。简化理解就是：CommonJS 更像运行时拼装，ESM 更像可分析的模块声明协议。
88. `import` 为什么要静态提升？
答：`import` 必须在编译阶段就能确定依赖关系，才能让引擎或构建工具提前建立模块图，你可以把模块图先理解成“整个项目里谁依赖谁的一张关系网”，并进一步判断导入导出绑定、处理循环依赖和执行优化。它之所以不能像普通函数一样随便写在条件里，本质上不是语法保守，而是因为 ESM 设计的核心价值就是静态可分析。如果导入关系都得等运行时再猜，Tree Shaking、预加载和很多构建能力都会失去基础。
89. ES Module 为什么更利于 Tree Shaking？
答：因为 ESM 的导入导出是静态结构，构建工具能在打包前就分析出哪些导出被真正使用、哪些没有被引用，从而安全地删除未使用代码。Tree Shaking 的本质不是“删没用文件”，而是基于静态依赖图做可证明的死代码消除，你可以把死代码消除先理解成“确认某段代码永远不会被用到后把它删掉”。也要注意，它能否真正生效还依赖副作用标记，这里的副作用可以简单理解成“执行这段代码时，会不会顺手影响外部状态”，以及打包配置和代码写法，并不是只要换成 ESM 就自动零冗余。
90. 循环依赖会带来什么问题？
答：循环依赖最麻烦的地方不是代码风格难看，而是初始化时序会变得复杂，某个模块可能在自身还没完成初始化时就被对方读取到，从而出现 `undefined`、部分初始化对象或行为异常。这个问题的本质是模块图出现了环，导致值的可用时机不再直观。它往往说明公共层抽象出了问题，或者模块职责边界没有切好。
91. 动态导入 `import()` 的使用场景是什么？
答：动态导入最典型的场景是路由懒加载、按需加载大体积依赖、根据用户操作再补加载功能模块，以及根据环境条件决定是否引入某块能力。它的本质是把原本静态阶段就要确定的依赖，延迟到运行时按需触发，因此天然形成代码分割点。它存在的意义是优化首屏和资源利用率，但也要注意拆得过碎会增加请求和调度成本。
92. 浏览器为什么原生支持 ES Module？
答：因为 ESM 已经成为 JavaScript 标准模块方案，浏览器原生支持它后，可以直接在平台层理解模块依赖关系、按模块粒度加载脚本，并为现代开发和调试提供统一基础。它的底层前提是 ESM 具备静态结构，浏览器在解析阶段就能知道接下来还需要哪些模块资源。原生支持的意义不只是省掉打包，而是让模块化从构建工具能力上升为语言和平台共同能力。
93. 模块作用域和全局作用域的关系是什么？
答：ES Module 拥有自己的独立作用域，模块顶层声明的变量默认不会自动挂到全局对象上，因此不同文件之间不会像早期脚本那样轻易互相污染。它的本质是在语言层面给每个文件建立一个私有边界，这对多人协作和大型项目尤为重要。当然，模块里如果你主动往 `window` 或 `globalThis` 上挂东西，依然会重新回到全局污染问题，所以模块作用域提供的是默认隔离，不是绝对屏蔽。
94. 在工程项目里，如何设计模块边界？
答：设计模块边界时，我通常优先看职责是否单一、输入输出是否清晰、变化频率是否一致，以及这个模块对外暴露的是稳定能力还是易变细节。一个好模块不是功能越多越好，而是高内聚、低耦合，让调用方只依赖必要接口，不感知内部实现细节。这个问题本质上考察的是你有没有从系统演化角度看模块，而不是只会按文件夹拆代码。
95. 如何避免公共模块变成“垃圾仓库”？
答：避免公共模块失控的关键，是只沉淀那些真正稳定、跨场景复用、边界清晰的能力，而不是为了少写几行代码就过早抽公共。很多公共模块最后变成垃圾仓库，本质原因是抽象时机太早、职责范围失控、调用方需求互相污染。我的经验是，至少在两个到三个真实场景里验证复用模式成立后，再决定是否上升为公共能力，这样抽象会更稳。

## 十、进阶与手写

1. 什么是垃圾回收？常见策略是什么？

答：垃圾回收就是自动识别哪些对象已经不可达、程序后续也无法再使用，并在合适时机回收其占用内存。这里的不可达和可达性分析你都可以先理解成“程序还能不能顺着引用关系找到这个对象”，JavaScript 引擎主流采用的核心思路是标记清除，也就是从全局对象、当前调用栈、闭包环境等根对象出发做可达性分析，能走到的对象保留，走不到的对象回收；真实实现中还常结合分代回收，也就是“把新创建的对象和活了很久的对象分开回收”，以提升效率。理解它的意义，在于你会知道内存泄漏不是因为没手动 `free`，而是因为某条引用链让对象一直处于可达状态。
97. JavaScript 常见内存泄漏场景有哪些？
答：常见场景包括全局变量滥用、定时器未清理、事件监听未解绑、闭包长期持有大对象、缓存无限增长、已经移除的 DOM 节点仍被引用，以及某些第三方库订阅关系未释放。它们的共性不是“写法不好看”，而是对象本来应该结束生命周期了，但仍然能从根对象沿引用链被找到，所以垃圾回收器不敢回收。排查时真正有用的不是猜，而是看堆快照和保留路径，堆快照你可以理解成“某一时刻内存对象分布的截图”，保留路径可以理解成“这个对象为什么还没被回收、是谁一路引用着它的那条路径”。
98. `WeakMap` 和 `Map` 的区别是什么？
答：`Map` 是强引用键值映射，只要 `Map` 里还持有某个键，对应对象就不会被垃圾回收；`WeakMap` 的键必须是对象，而且是弱引用，你可以先理解成“这种引用不会强行把对象留在内存里”，不会阻止这个对象被回收。它存在的意义是给对象附加元数据而又不人为延长对象生命周期，这在缓存对象私有信息、DOM 节点关联数据、框架内部依赖跟踪中非常有价值。简单说，`Map` 更适合显式数据容器，`WeakMap` 更适合附着型元数据。
99. `Set`、`Map` 在业务开发里分别适合什么场景？
答：`Set` 更适合做去重、成员存在性判断、选中态集合、黑白名单集合这类“值集合”问题；`Map` 更适合做键值映射、对象到元数据的关联、请求缓存、节点索引等“通过键快速取值”问题。它们存在的意义，是用更贴近语义的数据结构替代数组和普通对象的勉强承载。数据结构选得合适，不只是性能更好，更重要的是代码意图会更清晰。
100. Proxy 和 `Object.defineProperty` 的区别是什么？
答：`Object.defineProperty` 更像是给单个属性加拦截点，它能监听读取和设置，但对新增属性、删除属性、数组索引变化等场景支持有限；`Proxy` 则是代理整个对象，可以统一拦截 `get`、`set`、`deleteProperty`、`has`、`ownKeys` 等多种底层操作。两者的本质差异，不只是新旧 API，而是一个在补丁式拦截属性，一个在对象层做完整代理。也正因为如此，`Proxy` 更适合现代响应式系统和元编程场景，这里的元编程你可以先理解成“用代码去控制、拦截或改造代码本身的行为”。
101. Reflect 的作用是什么？
答：Reflect 提供了一套更规范、更函数化的对象操作 API，比如 `Reflect.get`、`Reflect.set`、`Reflect.defineProperty` 等，它的价值不在于发明新能力，而在于把很多原本分散、语义不统一的底层操作收拢成统一接口。尤其在 `Proxy` 里，Reflect 很适合做默认转发，因为它和代理拦截器的语义天然对应。换句话说，Reflect 是让对象底层操作更标准、更可组合的一层工具接口。
102. 为什么 `Proxy` 更适合响应式系统？
答：因为响应式系统的核心需求不是单纯知道“某个已有属性被改了”，而是要尽可能完整地感知对象行为，包括读取、设置、新增、删除、数组索引变化、`in` 判断、`for...in` 枚举等。`Proxy` 能从对象层统一拦截这些操作，因此覆盖面比 `Object.defineProperty` 更完整，尤其适合深层对象和数组。它存在的意义，是让响应式追踪从“拦属性”升级为“拦行为”。
103. 手写 `new` 要点有哪些？
答：手写 `new` 的关键不是把代码凑出来，而是准确复现语言语义：先创建新对象，再把对象原型链接到构造函数的 `prototype`，再以该对象为 `this` 执行构造函数，最后判断构造函数是否显式返回对象类型的值，如果返回了对象就用那个对象，否则返回新实例。这个题真正考的是你是否把构造调用、`this` 绑定、原型链连接和返回值规则完整串起来了。
104. 手写数组扁平化、深拷贝、防抖、节流、Promise、`bind` 时，最常见失误点是什么？
答：这类手写题最常见的问题通常不是主流程不会写，而是边界处理不完整，比如数组扁平化忘了深度控制、深拷贝忘了循环引用和特殊对象、防抖节流忘了 `this` 和参数透传、Promise 忘了异常传播和 thenable 展开、`bind` 忘了 `new` 场景。换句话说，手写题真正拉开差距的地方不是骨架，而是你有没有把语言细节和工程边界一起考虑进去。
105. 如果让你用 5 分钟总结 JavaScript，你会怎么组织表达？
答：我会按一条完整主线来讲：先从值和类型系统切入，因为这决定了比较、拷贝和状态表达；再讲作用域、闭包、执行上下文和 `this`，因为这决定了代码运行时如何找变量和绑定上下文；接着讲原型链和继承，因为这决定了对象能力如何复用；再讲异步、事件循环和 Promise，因为这解释了单线程下异步为什么还能成立；最后再落到模块化、内存管理、响应式和工程实践，因为这些才是 JavaScript 真正进入大型项目后的落点。这样的组织方式好处是每个知识点都能顺着底层原理自然往下接，而不是一堆孤立术语。

## 十一、扩展追问

1. JavaScript 引擎、运行时、宿主环境分别是什么关系？

答：JavaScript 引擎负责解析和执行语言本身，比如词法分析、语法分析、字节码或机器码生成、垃圾回收等，这里的字节码你可以先理解成“引擎为了更高效执行代码而生成的一种中间代码”；运行时是更宽的概念，包含引擎加上一些执行所需的基础设施；宿主环境则是浏览器、Node.js 这类为 JavaScript 提供外部能力的平台，它们会额外提供定时器、网络请求、文件系统、DOM、事件系统等 API。这个区分很重要，因为很多人误以为 `setTimeout`、`fetch`、DOM 都是 JavaScript 语言本身，其实它们来自宿主环境。
107. 浏览器中的 Web API 和事件循环为什么必须一起理解？
答：因为很多所谓的异步能力，比如定时器、网络请求、DOM 事件，本质上都不是 JavaScript 引擎自己完成的，而是 Web API 或浏览器其他模块先代为处理等待阶段，等条件成熟后再把回调交回事件循环调度执行。如果只知道事件循环，不知道异步任务是谁交进队列的，理解会断一截；如果只知道 Web API，不知道回调何时回到主线程执行，也解释不了实际顺序。两者必须连起来理解，异步模型才完整。
108. 为什么说 JavaScript 是动态语言又是多范式语言？
答：说它是动态语言，是因为很多类型、属性、对象结构和绑定关系都可以在运行时决定和变化；说它是多范式语言，是因为它既支持命令式编程，也支持函数式风格，还能通过原型和 `class` 语法去表达面向对象思想。这个特性让 JavaScript 非常灵活，但代价也是系统边界更容易失控，所以工程实践里才会不断引入模块化、类型系统、Lint、测试和规范来给这种灵活性加护栏。
109. 什么是调用栈，栈溢出通常怎么产生？
答：调用栈是用来管理函数执行顺序的一种后进先出结构，每调用一个函数就会压入一帧执行上下文，函数返回后再弹出。所谓栈溢出，通常是递归缺少正确终止条件、递归层级过深，或者某些间接调用形成无限递归，导致栈帧不断压入最终超过引擎允许的最大深度。理解调用栈的意义，在于很多同步阻塞、异常堆栈、递归问题、执行顺序问题都要回到它上面解释。
110. 面试中如何把 JavaScript 的知识体系讲成一条线？
答：最好的讲法不是背散点题，而是按“值如何表示、代码如何运行、对象如何复用、异步如何调度、系统如何工程化”这条线组织。也就是先讲类型和比较，再讲作用域、闭包、执行上下文和 `this`，再讲原型链和继承，再讲事件循环、Promise 和异步控制，最后讲模块化、内存、响应式和性能治理。这样一来，面试官无论从哪个点深挖，你都能把它接回整套运行模型，而不是只会背某一道题的标准答案。

## 十二、术语补充题

1. 什么是词法环境？

答：词法环境可以先理解成“某个作用域里专门用来存变量名字和变量值的一张记录表”。当代码进入一个新的作用域，比如函数、代码块、模块时，JavaScript 都会为它准备对应的词法环境，后续找变量时就会先从这张表开始找，再逐层往外找，所以它本质上是在支撑作用域和变量查找。
112. 什么是环境记录？
答：环境记录可以理解成词法环境内部真正用来存数据的那部分内容，也就是“这个名字对应什么值、现在能不能访问”的那层记录。你可以把词法环境理解成一个盒子，把环境记录理解成盒子里面真正写着变量信息的那张表，这样就比较容易把两个概念区分开。
113. 什么是执行上下文？
答：执行上下文可以简单理解成“代码当前这一轮执行时随身带着的运行信息包”，里面会放变量、外层作用域引用、`this` 指向等信息。每调用一次函数，基本都可以理解成新开了一个执行上下文，所以它是理解变量提升、调用栈和 `this` 的关键基础。
114. 什么是调用栈？
答：调用栈可以理解成“函数执行时排队进出的一摞盒子”，哪个函数先结束，哪个盒子就先弹出去，所以它遵循后进先出的规则。只要你明白函数调用会入栈、函数返回会出栈，很多执行顺序和栈溢出问题就能比较直观地理解。
115. 什么是 `ToPrimitive`？
答：`ToPrimitive` 可以简单理解成“对象在参与比较、运算、字符串拼接之前，先被尝试转换成普通值”的过程。因为很多运算并不能直接拿对象本身去算，所以 JavaScript 会先问这个对象：你能不能先给我一个数字或字符串，我再继续后面的比较和计算。
116. 什么是 thenable？
答：thenable 指的是“虽然它不一定是原生 Promise，但它长得像 Promise，因为它有 `then` 方法”。Promise 规范在处理返回值时，看到这种对象会把它当成类似 Promise 的东西继续展开处理，所以 thenable 本质上是 Promise 体系里的“兼容对象”概念。
117. 什么是宿主环境？
答：宿主环境就是让 JavaScript 真正跑起来的平台，比如浏览器、Node.js、Deno。JavaScript 语言本身没有 `setTimeout`、DOM、文件系统这些能力，这些额外能力都是宿主环境提供的，所以很多 API 其实不是 JavaScript 语言自带的，而是平台额外给它接上的。
118. 什么是幂等性？
答：幂等性可以简单理解成“同一个操作重复执行多次，最终结果是否还是一样且安全”。比如把用户资料更新成某个固定名字，多执行几次结果通常一样；但支付扣款这种操作如果重复执行多次，结果就不一样，所以它不是随便可重试的幂等操作。
119. 什么是竞态？
答：竞态可以理解成“多个异步结果谁先回来不确定，而最终结果又会受先后顺序影响”的问题。比如你连续发两次搜索请求，旧请求后回来却把新请求结果覆盖了，这就是典型竞态，所以它本质上是在讲异步结果之间的时序冲突。
120. 什么是状态机？
答：状态机可以先理解成“一个流程在不同状态之间按规则切换的模型”。例如一个请求可能有空闲、加载中、成功、失败、取消这些状态，而不同操作会让它从一个状态切到另一个状态，所以状态机本质上是在把复杂流程的变化规则画清楚。
121. 什么是 Tree Shaking？
答：Tree Shaking 可以简单理解成“打包时把项目里没有真正用到的代码删掉”。它之所以叫这个名字，是因为模块依赖关系很像一棵树，构建工具会把没有被引用到的枝叶摇掉，但它能否做好，前提是模块依赖关系足够清楚、足够静态。
122. 什么是模块图？
答：模块图可以理解成“整个项目里每个模块之间谁依赖谁的一张关系图”。构建工具只有先看懂这张图，才能知道入口从哪开始、哪些代码应该一起打包、哪些模块有循环依赖、哪些代码根本没被用到。
123. 什么是副作用？
答：副作用可以简单理解成“代码执行后除了返回值之外，还顺手改了外部世界”。比如修改全局变量、发请求、操作 DOM、打印日志、改本地存储，都属于副作用。理解副作用很重要，因为很多优化和推理都依赖“这段代码是不是只根据输入返回输出”。
124. 什么是弱引用？
答：弱引用可以理解成“我引用了这个对象，但不会强行把它留在内存里”。如果除了这条弱引用以外，程序已经没有别的地方再用这个对象了，垃圾回收器就可以把它清掉，所以 `WeakMap` 这类结构很适合存“附加信息”，而不适合做必须长期保留的主数据。
125. 什么是元编程？
答：元编程可以理解成“用代码去控制、观察、拦截、改造代码本身的行为”。像 `Proxy` 拦截对象读写、装饰器给类和方法加额外逻辑、Reflect 统一转发底层操作，都可以看成元编程思路，它不是日常业务必需品，但在框架和底层工具里非常常见。

## 十三、异步与 Promise 易错深挖（补充）

1. 写代码时没有 `isMacroTask()` 这类 API，我该怎么判断一段逻辑是同步、宏任务还是微任务？

答：判断方式不是靠运行时探测，而是靠“这段逻辑是被哪个机制排进事件循环的”。同步代码就是当前调用栈里立刻执行的那部分；`setTimeout`/`setInterval`、多数 DOM 事件、很多 I/O 完成回调等，规范上通常以 **task（宏任务）** 形式入队；`Promise.then/catch/finally`、`queueMicrotask`、`MutationObserver` 等通常走 **microtask（微任务）**。不确定时就去查对应 API 或规范描述里写的是 **queue a task** 还是 **queue a microtask**。另外 Node.js 里 `process.nextTick` 是更特殊的队列，优先级通常比常规微任务更高，需要单独记。

1. `Promise` 相关代码里，executor、`resolve` 和 `then` 回调到底哪些是同步、哪些是微任务？

答：可以把 Promise 拆成三层：`new Promise(executor)` 里的 **executor 会立刻同步执行**；在 executor 里调用 `resolve/reject` 时，**状态更新与入队反应任务**按规范处理，但 `**.then` 注册的回调不会在同一条同步调用栈里立刻执行**，而是进入 **微任务**，等当前同步代码跑完后再跑。已经 `fulfilled/rejected` 之后再调用 `.then`，也不会“错过”，引擎会把回调用 **已落定的结果**在微任务里补执行。很多人误以为“先 resolve 再 then 就不会触发”，本质是把 **注册** 和 **执行**混在一起了。

1. `setTimeout` 这一行是同步的吗？回调什么时候进宏任务队列？说“1 秒后执行”精确吗？

答：**调用 `setTimeout(fn, delay)` 本身是同步的**：它把计时与调度交给宿主，登记完就继续往下执行。计时到期后，引擎把 **回调对应的 task 放进宏任务队列**，真正执行还要等事件循环取到它且主线程空闲，所以不是“到点立刻执行”。**delay 也不是精密时钟**：常见还有最小延迟 clamp、嵌套定时器规则、后台标签节流、主线程长时间阻塞等，因此只能理解为 **至少约 delay 毫秒**，不能当实时计时基准。

1. 定时器回调“入队时刻”和与其他宏任务的先后顺序是不是不可测？

答：**墙钟意义上精确到毫秒的时刻**不要当可测前提，环境差异和节流会让入队/执行抖动。**相对顺序**也不是完全随机：同一段同步代码里按顺序注册、相同 delay 的多个 timer，在常见实现里往往仍能保持 **先注册先排队** 的直觉；但一旦和点击、网络、`postMessage` 等其他 task 源混在一起，就要回到“谁先被调度进队列、队列如何被取”的具体规则上分析，不能一句话概括成“完全不可预测”。

1. Promise 只能 `fulfilled/rejected` 一次吗？一直不结算呢？结算完会自动被垃圾回收吗？

答：**状态迁移只能发生一次**：从 `pending` 到 `fulfilled` 或 `rejected` 落定后，再调用 `resolve/reject` 会被忽略。也可以 **永远 `pending`**，语法上合法，但要注意长期引用可能造成内存长期占用。**没有“结算完 Promise 就被引擎特殊销毁”的规则**：是否回收只看 **可达性**，和普通对象一样；若变量、闭包、链式引用仍指向它，就不会因为已 settled 而消失。

1. 为什么 `promise1.then(handler)` 明明只传了一个函数，却会出现“另一个 Promise”（链上的 `promise2`）？

答：因为 `**then/catch/finally` 的返回值**在规范里就是 一个新的 Promise，用来表示“这一环处理函数跑完之后链路的最终结果”。这个新对象不是你写在回调里的，而是 内置实现创建并返回的：回调里 `return` 普通值、返回 Promise、`throw`、或隐式 `undefined`，都会决定新 Promise 如何落定。所以 `promise1` 与 `promise1.then(...)` 的返回值通常是 两个不同实例；例如 `promise1` 已成功但 `then` 里 `throw`，常见结果是 `**promise1` 仍 fulfilled，返回的新 Promise rejected**。

1. `.then`、`.catch`、`.finally` 必须传函数吗？传原始值会怎样？

答：**不必须**，但 **只有可调用（callable）才会当回调执行**。对 `.then(onFulfilled, onRejected)`：参数不是函数时，相当于这一环 **缺少对应处理器**，成功值或失败原因会 **继续向后传递**（值穿透/拒绝穿透），**一般不会因此抛类型错误**。`.catch(onRejected)` 同理，非函数意味着 **不接拒绝**，错误继续往后传。`.finally(onFinally)` 不同：`**onFinally` 必须是函数**，否则通常在 **调用 `finally` 时直接 `TypeError`**。

1. 经典输出题：`Promise.resolve(1).then(2).then(Promise.resolve(3)).then(console.log)` 输出什么？

答：常见输出是 `**1**`。原因是 `.then` 的第一个参数若不是函数会被忽略并 **透传上一环结果**：`2` 不是函数；`Promise.resolve(3)` 会先同步求值得到一个 Promise 对象，它同样 **不是函数**，也不会把结果改成 `3`。最后 `.then(console.log)` 是函数，微任务阶段打印链上传下来的 `**1`**。

1. 经典输出题：`new Promise` 里 `setTimeout(0)` 再 `resolve`，外面同步 `console.log` 与 `then`，顺序怎么排？

答：典型顺序是：**先同步打印 executor 与外层同步日志**，再执行 **timer 回调（宏任务）**；若在 timer 回调里调用 `resolve`，则 `**.then` 回调作为微任务**，会在 **该宏任务同步部分结束后**才运行，因此常见现象是：timer 回调里 `resolve` 后面的同步 `console.log` 仍会 **先于** `.then`。这类题用来区分：**宏任务回调内的同步代码**与 **Promise 反应微任务**的边界。

1. `promise1` 与 `const promise2 = promise1.then(...)` 是两个 Promise 吗？面试里一句话怎么收尾？

答：**通常是两个对象**：`promise1` 是创建出来的那个 Promise；`promise2` 是 `**then` 返回的新 Promise**。一句话收尾：**链式调用的每一环都会生成新的 Promise，用来承载这一环处理之后的结果；错误是否向上冒泡、值是否穿透，都发生在这条链上。**

1. `Promise.resolve(x)` 是什么意思？`Promise` 不是构造函数吗？是不是等于先 `new Promise` 再立刻挂 `then`？

答：可以把 `Promise` 理解成“既能当构造函数用，也是一个带静态方法的对象”：`new Promise(executor)` 走 **executor 模型**，常见是异步里再 `resolve/reject`；`**Promise.resolve(x)` 则是静态工厂**，用来把 `x` **接进 Promise 协议**：若 `x` 是普通值，通常得到 **已 `fulfilled`、值为 `x`** 的 Promise；若 `x` 本身是 Promise 或 thenable，会按规范做 **采纳/展开**，而不是简单再包一层。它**不等价于语法上的**“先 `new Promise` 再在同一表达式里立刻 `.then`”这种两步写法，但**效果上**很像“我已经有一个可链式调用的成功态 Promise，可以直接 `.then` 继续组合”。面试里若要类比，可以说它与 `new Promise(resolve => resolve(1))` 在“普通值快速变 fulfilled”场景下语义接近，但仍要留一句：**细节以规范为准（thenable、循环解析等边界）**。顺带记一个常见链式陷阱：`Promise.resolve(1).then(...).catch(...).then(...)` 里，只要中间没有 `rejected`，`**catch` 不会执行**，成功值会一路透传到后面的 `then`。

1. `const promise = Promise.resolve().then(() => { return promise; }); promise.catch(...)` 底层发生了什么？为什么和“循环引用”有关？

答：这题要把 **表达式求值顺序**、**闭包捕获**、**Promise 解析算法**三件事叠在一起看。第一，整行赋值语句会先同步执行右侧：`Promise.resolve()` 立刻得到一个已 fulfilled 的 Promise；接着调用 `.then(onFulfilled)` 会 **同步返回一个全新的 Promise**（记作 `P`），并把 `onFulfilled` 排入微任务队列。此时 `onFulfilled` 通过词法作用域 **捕获了名字 `promise`**，但回调还没执行。第二，当右侧调用返回后，`**const promise` 完成绑定**：外层变量 `promise` 指向的正是这个 `**P`**（也就是 `then` 返回的新 Promise）。第三，微任务阶段执行 `onFulfilled`，执行 `return promise` 时读到的 `promise` 已经是 `**P` 本身**，等价于“**用 `then` 正在解析结果的那个 Promise，去作为 `then` 处理函数的返回值再解析自己**”。按 ECMAScript 的 **Promise 解析过程**，当解析目标与当前待解析的 Promise 形成 **自引用/链式循环（chaining cycle）** 时，必须 **拒绝（reject）** 并给出 `**TypeError`**，否则状态机无法得到确定的落定结果。因此 `**P` 通常会以 `TypeError` 失败**，而不是永远 pending。第四，题里若写 `promise.catch(console.err)`，常见笔误是 `**console.err` 并不存在**（应为 `console.error`），于是传给 `catch` 的很可能是 `**undefined`（非函数）**，这会让这一环 接不住拒绝，拒绝会继续向后传递并容易变成 未处理的 rejection；若改成 `console.error` 或 `err => console.error(err)`，才能稳定观察到上述 `**TypeError`**。面试收尾可以一句话概括：`**then` 返回新 Promise；回调若返回“正在等自己的那个 Promise”，规范用 TypeError 打破循环；`catch` 参数必须是可调用函数。**

1. `Promise.resolve(1).then(2).then(Promise.resolve(3)).then(console.log)` 为什么输出 `1`？为什么最后一个 `then` 不是“继续透传”？

答：这题最容易错的地方，是把“**给 `then` 传了什么**”和“**这一环 Promise 最终怎么落定**”混在一起了。要按链一环一环拆开看。先记第一个 Promise：`Promise.resolve(1)` 得到的是一个 **已 fulfilled、值为 `1`，只有resolve会自动包装并return** 的 Promise，记作 `P0`。第一环 `P1 = P0.then(2)` 里，`2` **不是函数**，而 `then` 的规则是：对应处理器如果 **不可调用**，这一环就相当于 **没有成功处理器**，效果近似于把前一环的值 **原样向后传**，也就是工程上常说的“值穿透”；所以 `P1` 最终仍然会以值 `1` fulfilled。第二环 `P2 = P1.then(Promise.resolve(3))` 也一样，虽然 `Promise.resolve(3)` 这个表达式会先同步求值，得到一个 **Promise 对象**，但这个对象 **本身不是函数**，因此它不会被当成 `onFulfilled` 执行，也不会把链路结果改成 `3`；这一环仍然等价于“没有成功处理器”，于是 `P2` 继续以值 `1` fulfilled。第三环 `P3 = P2.then(console.log)` 才是关键：这里传进去的 `**console.log` 是函数**，因此这一环不再是“缺少处理器时的自动透传”，而是**真正执行成功回调**，回调参数拿到的就是前面两次透传下来的 `1`，于是打印 `1`。所以更准确地说，不是“后面的 `then` 不会透传”，而是：**前两环确实在透传，最后一环拿到的正是透传后的 `1`；只是最后一环因为传入了真正的函数，所以它执行的是“消费这个值”，而不是“继续省略处理器式的透传”。** 如果你在 `console.log` 后面再接一环，比如 `.then(x => console.log('next', x))`，那下一环里拿到的往往不是 `1`，而是 `console.log` 的返回值 `undefined`，因为一旦某一环提供了真实函数，链上传递的就不再是“自动透传”，而要看这个函数 `**return` 了什么**；没有显式 `return` 时，下一环接到的就是 `undefined`。面试里一句话总结就是：`**then` 只有在对应参数不是函数时才发生透传；一旦传入函数，这一环就会执行该函数，并以它的返回值决定下一环。**

1. `Promise.resolve('1').then(...).finally(...)` 与 `Promise.resolve('2').finally(...).then(...)` 这类题，为什么 `finally` 的 `return` 不会改掉后面 `then` 拿到的值？

答：这类题的底层关键是分清 `**then` 是“处理并转换链值”**，而 `**finally` 是“收尾但默认不改链值”**。先看第一段：`Promise.resolve('1')` 立刻得到一个已 fulfilled、值为 `'1'` 的 Promise；后面的 `.then(res => { console.log(res) })` 在微任务里执行，先打印 `1`，但因为这个回调 **没有显式 `return`**，所以这一环 `then` 返回的新 Promise 会以 `**undefined**` fulfilled。紧接着挂在它后面的 `.finally(() => { console.log('finally') })`，拿到的是“前一环已经落定”的状态，因此会在后续微任务中打印 `finally`；但它的作用不是把链值改成自己的返回值，而是“执行一下收尾逻辑，再把原来的落定结果继续传下去”。所以如果这一段后面再接一个 `.then(res => console.log(res))`，拿到的会是 `**undefined**`，原因不是 `finally` 改了值，而是 **前一个 `then` 没有 `return`，链值早就变成了 `undefined`**。再看第二段：`Promise.resolve('2').finally(() => { console.log('finally2'); return '我是finally2返回的值' }).then(res => { console.log('finally2后面的then函数', res) })` 中，`finally` 回调同样会作为微任务执行并先打印 `finally2`，但它 **返回的普通值不会替换原链值**，因此后面的 `then` 拿到的仍然是原来的 `'2'`，而不是“我是finally2返回的值”。为什么会这样？因为可以把 `finally(onFinally)` 粗略理解成一种特殊包装：成功时近似于 `promise.then(value => Promise.resolve(onFinally()).then(() => value))`，失败时近似于 `promise.then(undefined, reason => Promise.resolve(onFinally()).then(() => { throw reason }))`。注意这两个近似式里，`onFinally()` 的返回结果只是“等待它做完”，真正继续往后传的仍然是 **原来的 `value` 或 `reason`**，这就是为什么 `finally` 适合写关闭 loading、埋点、释放资源等“收尾逻辑”，却不适合当作“改写链值”的位置。它只有在两种情况下会真正影响后面：第一，`finally` **自己抛错**；第二，`finally` **返回一个 rejected Promise**。这两种情况会把后续链改成 rejected，从而打断原本的成功值或失败原因继续向后传。结合执行顺序看，你给出的那两段代码常见输出是先 `1`，再 `finally2`，然后 `finally`，最后 `finally2后面的then函数 2`；其中 `finally2` 会早于第一个链上的 `finally`，是因为第二段的 `finally` 在初始 fulfilled Promise 上直接登记，而第一段的 `finally` 必须等前面的 `then` 先执行完、对应的新 Promise 先落定后才会进入可执行状态。面试里一句话总结就是：`**then` 的返回值决定下一环，`finally` 的返回值通常被忽略；`finally` 默认只做收尾并把原结果继续传下去，除非它自己抛错或返回 rejected Promise。**
本质规则就三条：
then / catch / finally 每调用一次，都会返回一个新的 Promise。
只有“某个 Promise 自己 settle 了”，挂在它身上的 reaction（对应的 then/catch/finally 回调）才会入微任务队列。
链上后一环不会因为前一环“已经写出来了”就立刻入队，必须等前一环返回的那个新 Promise settle
finally内的return不会执行，只会包装上一层的并自动返回
Promise.resolve(...) 本身是同步执行。
本质规则
Promise.resolve(x) 同步返回一个 Promise
x 是原始值：立刻得到一个 fulfilled Promise
x 是 Promise / thenable：按规范同步创建/采纳外壳，但后续落定可能受被采纳对象影响
同步的是“创建/落定 Promise”
不是同步执行 .then 回调
.then/.catch/.finally 的回调属于 Promise reaction job，进 微任务

1. `Promise.all([runAsync(1), runAsync(2), runAsync(3)]).then(...)` 这类题，完整执行过程是什么？为什么会先打印 `1 2 3`，最后才打印 `[1, 2, 3]`？

答：这题的本质规则只有四条：**第一，`new Promise(executor)` 的 executor 同步执行；第二，`setTimeout` 同步注册、回调作为后续宏任务执行；第三，`resolve` 只会让“挂在当前这个 Promise 身上的 reaction”进入后续微任务调度，不会把整条链一次性跑完；第四，`Promise.all` 自己返回一个新的聚合 Promise，它只有在所有输入 Promise 都 fulfilled 后才 fulfilled。** 代入代码看：`runAsync(1)` 被调用时，先同步执行 `new Promise(r => setTimeout(() => r(x, console.log(x)), 1000))` 的 executor；executor 里没有直接调用 `r`，只是同步注册了一个 1 秒后的 timer，然后立刻返回一个 pending Promise，记作 `P1`。`runAsync(2)`、`runAsync(3)` 同理，分别得到 `P2`、`P3`。所以在整个 `Promise.all([...])` 这一行的**同步阶段**里，实际发生的是：连续创建了三个 pending Promise，并各自注册了一个定时器；随后 `Promise.all([P1, P2, P3])` 也会**同步**创建一个新的聚合 Promise，记作 `PA`，并给 `P1/P2/P3` 分别挂上内部成功处理器，用来记录第几个位置完成、保存结果、递减剩余计数。此时没有任何一个输入 Promise 已经 settle，所以 `PA` 还是 pending，`.then(res => console.log(res))` 只是把回调挂到 `PA` 身上，并不会立刻入微任务。约 1 秒后，三个 timer 回调依次进入宏任务队列；在常见实现里，由于它们是按 1、2、3 的顺序同步注册且 delay 相同，所以通常也按这个顺序执行。关键再看单个 timer 回调内部：`r(x, console.log(x))` 的**参数求值顺序是从左到右**，先求 `x`，再执行 `console.log(x)`，所以会先打印当前的 `x`；`console.log` 的返回值是 `undefined`，但 `resolve` 只接收第一个参数，后面的参数会被忽略，因此真正生效的是 `r(x)`，也就是 `P1/P2/P3` 最终分别 fulfilled 为 `1/2/3`。于是宏任务阶段常见先看到 `1`、再 `2`、再 `3`。每个 timer 回调结束后，对应 Promise 的 reaction 会进入微任务：`Promise.all` 挂在 `P1/P2/P3` 上的内部处理器会在微任务里把结果写入固定下标位置，所以虽然完成时机可能不同，但最终结果数组仍严格保持输入顺序，即 `[1, 2, 3]`，而不是“谁先完成就排前面”。当前两个输入完成时，`PA` 还不会 settle；直到第三个输入也 fulfilled、剩余计数归零，`PA` 才会 fulfilled，值为 `[1, 2, 3]`。这时，之前挂在 `PA` 身上的 `.then(res => console.log(res))` 才有资格进入微任务并执行，因此最后才打印整个数组。常见输出通常是：先 `1`、`2`、`3`，最后 `[1, 2, 3]`。如果要一句话收尾：`**runAsync` 的 Promise 是“同步创建、异步落定”；`Promise.all` 是“同步建壳、异步聚合”；单个任务完成时只会推进聚合状态，只有全部完成后，挂在 `Promise.all` 返回值上的 `then` 才会真正执行。**

1. `obj.fun()` 为什么输出对象上的名字，而 `new obj.fun()` 却输出 `undefined`？

答：这题的本质规则只有两条：**第一，函数一旦和 `new` 连用，`this` 绑定优先走 `new` 绑定，而不是原来的隐式绑定；第二，`new` 调用创建的是一个全新的实例对象，默认并不继承调用者对象自身的数据属性。** 看 `new obj.fun()`：运算符优先级决定它等价于 `**new (obj.fun)()`**，也就是把 `obj.fun` 这个函数当成构造函数来调用。一旦用了 `new`，运行时会先创建一个全新的空对象 `instance`，把这个对象的内部原型连到 `obj.fun.prototype`，然后让函数体里的 `this` 指向这个新对象，而**不再指向 `obj`**。因此 `console.log(this.name)` 实际读取的是 新实例对象上的 `name`。而这个新对象本身并没有 `name` 属性；它的原型链上也没有从 `obj` 那里继承 `name`，因为 `obj` 只是“原调用者对象”，不是这次 `new` 出来实例的原型。所以读取结果是 `undefined`。这道题最容易错的点，是把“函数从 `obj` 身上取出来调用”误以为“`new` 时 `this` 还会指向 `obj`”；实际上 `**new` 绑定优先级高于隐式绑定**，只要进入构造调用语义，`this` 就改指向新实例。若面试官继续追问，可以补一句：这里虽然 `fun` 是对象上的普通方法，但只要写成 `new obj.fun()`，它的身份就从“方法调用”切换成了“构造调用”；除非你在函数内部手动给 `this.name` 赋值，否则打印的就是 `undefined`。
142. 普通函数里定义箭头函数、对象属性本身又是箭头函数时，`this` 为什么会分别指向不同对象？
答：这题只看三条规则：**普通函数的 `this` 由调用方式决定；箭头函数没有自己的 `this`，只捕获定义时外层最近一层非箭头函数的 `this`；若定义时外层没有普通函数包裹，它拿到的就是当前顶层词法环境的 `this`。** 先看 `var o = obj.say; o();`：`say` 本身是普通函数，被赋值给 `o` 后再直接调用，属于默认绑定，因此 `say` 执行时的 `this` 在浏览器普通脚本里通常是 `window`，严格模式下常见是 `undefined`。`say` 里面的 `f1` 是箭头函数，它不会自己重新绑定 `this`，而是直接捕获这次 `say` 执行时的 `this`，所以 `f1()` 打印出来的也是 `window` 或 `undefined`，而不是 `obj`。再看 `obj.say()`：这里是隐式绑定，因此 `say` 执行时 `this === obj`；于是内部箭头函数 `f1` 捕获到的也是 `obj`，所以打印的是 `obj`。因此前两次调用的差异根源不在箭头函数，而在**外层普通函数 `say` 被谁调用**。最后看 `obj.pro.getPro()`：最容易误判就在这里，很多人看到“对象点调用”就条件反射地认为 `this === obj.pro`，但 `getPro` 本身是**对象字面量里直接定义的箭头函数**，它创建时外层并没有一个普通函数执行上下文可供捕获，所以它的 `this` 在定义那一刻就已经固定为当前顶层词法环境的 `this` 了；后面再写成 `obj.pro.getPro()` 也不会改成 `obj.pro`。因此它打印的通常仍是浏览器普通脚本下的 `window`，或模块/严格环境下的 `undefined`。如果面试官要你一句话总结：**普通函数看调用点，箭头函数看定义点；对象不能给箭头函数“绑出一个新的 this”，对象字面量里的箭头方法拿到的通常不是该对象，而是定义处外层环境的 `this`。**

1. `window.number = 2; var obj = { number: 3, db1: (function(){ ... })() }` 这类 IIFE + `this` 输出题，为什么最后 `obj.number === 15`、`window.number === 40`？

答：这题只看四条规则：**第一，对象字面量求值时，属性值表达式会立刻执行；第二，`(function(){ ... })()` 是 IIFE，属于普通函数直接调用，`this` 走默认绑定；第三，IIFE 的返回值会直接成为属性值；第四，后面再怎么调用这个返回的普通函数，它的 `this` 仍然只看调用点。** 这里先把 **IIFE** 说清楚：IIFE 是 **Immediately Invoked Function Expression**，中文通常叫“**立即执行函数表达式**”。它的本质不是一种新函数类型，而是“**先把函数写成表达式，再立刻在后面加 `()` 执行**”。例如 `(function(){ ... })()` 或 `(() => { ... })()` 都是 IIFE。之所以常加外层括号，是因为单独写 `function(){}` 在语法上默认会被当成**函数声明**，而声明后面不能立刻跟调用括号；包上括号后，它就被强制当成**函数表达式**，于是后面的 `()` 才表示“立刻执行这次表达式求值出来的函数”。回到本题：对象字面量在创建 `obj` 时，不是先把 `db1` 记成函数再等你后面调，而是会先把 `db1: (function(){ ... })()` 这整个表达式**当场求值**。因此 IIFE 会在 `obj` 创建阶段立刻执行一次。因为它是普通函数直接调用，不是对象方法调用，所以这次执行里的 `this` 在浏览器普通脚本下通常就是 `window`。因此第一句 `console.log(this)` 打印的是 `window`；接着执行 `this.number *= 4`，改的是 `window.number`，于是原本 `2` 变成 `8`。然后 IIFE 返回一个新的普通函数，这个返回函数才会被放到 `obj.db1` 上，所以此时 `obj.db1` 保存的根本不是 IIFE 本身，而是 **IIFE 返回出来的那个内部函数**。接下来 `var db1 = obj.db1; db1();`：这里把函数取出来单独调用，仍然是普通函数直接调用，因此 `this` 再次走默认绑定，通常还是 `window`。所以先打印 `window`，再执行 `this.number *= 5`，把 `window.number` 从 `8` 改成 `40`。再看 `obj.db1();`：这次调用形式变成了对象点调用，所以这是隐式绑定，函数执行时 `this === obj`。因此它打印的是 `obj`，随后执行 `this.number *= 5`，改的是 `obj.number`，从 `3` 变成 `15`。所以最后 `console.log(obj.number)` 是 `15`，`console.log(window.number)` 是 `40`。这题最容易错的地方有两个：一是误以为 `db1` 属性里写的是函数，所以对象创建时不会执行；其实 IIFE 在对象创建阶段就已经跑了，并且先改了一次 `window.number`。二是误以为 `db1` 来自 `obj`，所以后面单独调用 `db1()` 时 `this` 还会指向 `obj`；实际上函数一旦被取出来直接调用，就丢失了原先的调用者信息，`this` 回到默认绑定。面试时一句话总结：**IIFE 就是“定义完立刻执行”的函数表达式；本题里先执行 IIFE，IIFE 默认绑定改的是 `window.number`；IIFE 返回的普通函数赋给 `obj.db1`，后续 `db1()` 改 `window.number`，`obj.db1()` 改 `obj.number`。**

1. 什么是 IIFE？为什么它看起来像“函数外面套括号再立刻执行”？

答：IIFE 是 **Immediately Invoked Function Expression**，中文通常译作“**立即执行函数表达式**”。它的本质可以压缩成一句话：**把函数写成表达式，然后在定义完成的同一时刻立刻调用它。** 常见写法有 `(function(){ ... })()`、`(function(){ ... }())`、`(() => { ... })()`。它之所以常见，是因为很多场景需要“临时创建一个独立作用域，马上执行一次，顺手返回一个结果”，例如早期模块封装、初始化逻辑、只想运行一次的配置代码、在对象字面量里先算出某个属性值等。为什么一定要加括号？因为单独写 `function foo(){}` 或 `function(){}` 时，解析器倾向于把它当成**函数声明**；而声明语句不是一个可立即参与后续运算的值，后面直接跟 `()` 不符合这里的语法预期。用括号、`!`、`+` 等方式把它包起来，本质都是在告诉解析器：“这里我要的是一个**函数表达式**，表达式会产生一个函数值”，于是后面的 `()` 才能把这个函数值立刻调用掉。面试里一句话总结就是：**IIFE 不是新语法，而是‘函数表达式 + 立即调用’的组合；括号的作用不是装饰，而是把函数从声明语义切到表达式语义。**

1. 所有直接定义的变量、函数都会挂到 `window` 上吗？顶层 `let` 又是“挂”在哪？

答：**不会。** 这题的关键是区分 **“全局绑定”** 和 **“全局对象属性”**，它们不是一回事。很多人把“定义在最外层”直接等同于“变成 `window.xxx`”，这是不严谨的。先看浏览器**普通脚本**环境：顶层 `var` 声明和顶层 `function` 声明，通常既会形成全局绑定，也会变成 `window` 的属性，所以 `var a = 1` 后通常能通过 `window.a` 读到，顶层 `function foo(){}` 之后通常也能通过 `window.foo` 访问。但顶层 `let`、`const`、`class` 不一样，它们虽然同样处在全局层级，却进入的是**全局词法环境（global lexical environment）**，不会自动挂到 `window` 上，所以 `let b = 2` 后通常 `window.b` 是 `undefined`。这里“挂在哪”若要说严谨，不该说“挂到某个对象上”，而应说：**它被记录在全局执行上下文对应的词法环境里，属于全局作用域可访问的绑定，但不是全局对象属性。** 这也是为什么你可以直接写 `b` 访问它，却不能用 `window.b` 访问到它。`const`、`class` 与顶层 `let` 的归属逻辑相同，只是额外受暂时性死区和重复声明规则约束。再进一步，如果代码运行在 **ES Module** 里，那顶层声明默认都有模块作用域，不论是 `var`、`function`、`let`、`const` 还是 `class`，通常都不会自动挂到 `window`。Node 环境里更不能套浏览器这套结论，因为本身就没有 `window` 这个全局对象。面试里最稳的表述是：**浏览器普通脚本中，顶层 `var` 和顶层 `function` 通常会成为 `window` 属性；顶层 `let/const/class` 进入全局词法环境，只是全局绑定，不是 `window` 属性；模块环境里顶层声明默认都不挂到 `window`。**

1. `function a(xx){ this.x = xx; return this } var x = a(5); var y = a(6); console.log(x.x); console.log(y.x);` 为什么在浏览器和 Node 里结果可能不一样？

答：这题的本质是**同一段代码，`this` 绑定规则可能相同，但“顶层 `var` 是否等于全局对象属性”在不同宿主里不同**，于是最终结果分叉。先看共同点：`a(xx)` 是**普通函数直接调用**，不是 `new` 调用，也不是对象方法调用；在非严格模式下，这种调用里的 `this` 会指向宿主的全局对象。于是 `this.x = xx` 的本质，都是在改“全局对象上的 `x`”。关键差异出在 `**var x` 本身是不是这个全局对象的属性**。在浏览器普通脚本里，顶层 `var x` 通常就是 `window.x`。因此第一次执行 `var x = a(5)` 时，先运行 `a(5)`：它把 `window.x` 设为 `5`，并返回 `window`；然后再把这个返回值赋给顶层变量 `x`。但由于顶层 `var x` 与 `window.x` 是同一个槽位，这一步相当于又把 `window.x` 改成了 `window` 本身。第二次 `var y = a(6)` 时，`a(6)` 会再次把 `window.x` 改成 `6`，然后返回 `window`；`y` 得到的是 `window`。因此浏览器里常见状态会变成：变量 `x` 读出来是当前的 `window.x`，也就是 `6`，所以 `x.x` 实际是 `(6).x`，结果是 `undefined`；而 `y` 指向 `window`，所以 `y.x === window.x === 6`。这就是浏览器里常见的 `undefined` 和 `6`。但在 Node 里，同样的直接调用虽然通常也会让 `this` 指向 `global`，可**顶层 `var x` 不会挂到 `global` 上**，它只是当前 CommonJS 模块里的局部变量。于是第一次 `var x = a(5)` 时，`a(5)` 把 `global.x` 改成 `5`，返回 `global`，然后模块变量 `x` 单独接住这个返回值，所以 `x === global`；第二次 `var y = a(6)` 后，`y === global`，同时 `global.x === 6`。因此 Node 里 `x.x === global.x === 6`，`y.x === global.x === 6`，最终输出是 `6 6`。这题真正考的不是“`this` 到底指哪”，而是你能不能再往前追一层，意识到 **浏览器普通脚本的顶层 `var` 与 `window` 属性共用一个绑定槽，而 Node 模块的顶层 `var` 不是全局对象属性**。面试里一句话总结：**两边都可能把 `this.x` 写到全局对象上，但浏览器里顶层 `var x` 还会和 `window.x` 互相覆盖；Node 模块里 `var x` 只是局部变量，所以最终 `x`、`y` 都拿到 `global`，结果就是 `6 6`。**

1. `(function(){ var x = y = 1; })();` 为什么 `y` 存在而 `x` 不存在？

答：这题的本质只有三条：**第一，赋值表达式是从右往左结合；第二，`var` 只修饰紧挨着的那个标识符；第三，非严格模式下，给一个未声明的标识符直接赋值，会在全局对象上隐式创建同名属性。** 所以 `var x = y = 1` 不能理解成“同时声明了 `x` 和 `y`”，它的真实求值顺序更接近：先执行 `y = 1`，再把这个赋值表达式的结果 `1` 赋给 `x`，也就是近似等价于 `y = 1; var x = y;`。这里 `var` 只作用于 `x`，并没有声明 `y`。因此在这个 IIFE 内部，`x` 是一个函数作用域内的局部变量，IIFE 执行完后它就离开了外层可访问范围，所以外面访问 `x` 会得到 `ReferenceError`。而 `y` 不同：因为它在函数内部被直接写成 `y = 1`，前面没有 `var` / `let` / `const` 声明，在**非严格模式**下，这种对未声明标识符的赋值会发生“隐式全局泄漏”，也就是在浏览器普通脚本里近似变成 `window.y = 1`。所以 IIFE 执行结束后，`y` 依然作为全局对象属性存在，外面能读到 `1`。至于 `var z;`，它只是声明了一个变量但没有赋值，因此默认值是 `undefined`，所以打印 `undefined`。这题最容易错的地方是把 `var x = y = 1` 误读成 `var x = 1, y = 1` 或误以为 `var` 会同时把右边出现的变量一起声明；实际上 `var` 没有这种“向右扩散”的能力，它只声明 `x`。还要补一句环境差异：如果在**严格模式**下执行，同样的 `y = 1` 不会创建全局变量，而是直接抛出 `ReferenceError`，因为严格模式禁止这种隐式创建全局属性的行为。面试里一句话总结：`**var x = y = 1` 的本质是“先给未声明的 `y` 赋值，再用这个结果初始化局部变量 `x`”；因此非严格模式下 `y` 泄漏成全局，`x` 只是 IIFE 内部变量。**

1. `var friendName = 'World'; (function(){ if (typeof friendName === 'undefined') { var friendName = 'Jack'; console.log('Goodbye ' + friendName); } else { console.log('Hello ' + friendName); } })();` 为什么输出不是 `Hello World`？顺便说清楚不同声明的提升和作用域规则。

答：这题的本质只有两步：**第一，`var` 没有块级作用域，只有函数作用域；第二，`var` 的“声明”会提升到整个函数顶部，但“赋值”留在原地。** 所以 `if { var friendName = 'Jack' }` 不能理解成“只在 `if` 块里临时声明一个变量”，它在这个 IIFE 里实际更接近下面的重写结果：`(function(){ var friendName; if (typeof friendName === 'undefined') { friendName = 'Jack'; console.log('Goodbye ' + friendName); } else { console.log('Hello ' + friendName); } })();`。关键点就在第一行补出来的 `var friendName;`：进入这个 IIFE 时，函数内部已经存在一个**局部同名变量**，只是值还是 `undefined`。变量查找时总是先找当前作用域，再向外找，因此 `if` 条件里的 `friendName` 读到的不是外层全局的 `'World'`，而是 IIFE 内部这个被提升后的局部变量。因为它**提升到了整个函数顶部并遮蔽了外层同名变量**。顺着这题把其他声明一起理清：`var` 是**函数作用域**，声明提升后初始化为 `undefined`，所以能在声明前访问但值是 `undefined`；顶层 `var` 在浏览器普通脚本里通常还会成为 `window` 属性。`let` 和 `const` 是**块级作用域**，也会在所属块开始时建立绑定，但在执行到声明语句前处于**暂时性死区**，因此声明前访问不是 `undefined`，而是直接 `ReferenceError`；其中 `const` 还要求声明时必须初始化，且绑定不能被重新赋值。`function` 声明在传统场景下面试通常可以记为“**提升优先级高于 `var`，函数体也一并可用**”，因此函数声明前就可以直接调用；但它在块级中的精确行为受规范模式和宿主实现影响，工程上不要依赖“块内函数声明像 `var` 一样泄漏到外层”这种不稳定直觉。`class` 与 `let/const` 更接近：是块级绑定，也受暂时性死区约束，声明前访问会报错，不会像函数声明那样可直接提前使用。把这些规则压成一句话：`**var` 看函数、会提升并初始化为 `undefined`；`let/const/class` 看块、会提升绑定但在声明前不可访问；`function` 声明通常整体可提前使用。**

为什么“在函数内部定义并调用 `b`”能访问 `temp`，而“在外部定义 `b` 再到 `a` 里调用”却访问不到？
答：这题只看一条总规则：**JavaScript 采用词法作用域，变量能不能访问取决于函数“定义在哪里”，不取决于函数“在哪里调用”。** 第一段代码里，`b` 是在 `a` 的内部定义出来的，所以 `b` 创建时就把它外层那层词法环境一起“带上了”；这层环境里正好有 `var temp = 10`，因此 `b` 执行时查找 `temp`，会先看自己的局部作用域，没有；再沿着它定义时保存的外层环境往上找，在 `a` 的作用域里找到 `temp`，所以打印 `10`。这就是闭包最基础的形态：**内部函数引用了定义时外层函数中的变量**。而第二段代码里，虽然 `b()` 的调用语句写在 `a` 里面，但 `b` 的定义位置根本不在 `a` 内部，它是在外层全局作用域定义的。既然 `b` 定义时外层环境不是 `a`，那它的作用域链里自然就没有 `a` 的局部变量 `temp`。因此第二段执行 `b()` 时，查找 `temp` 的过程是：先查 `b` 自己的局部，没有；再查 `b` 定义时所处的外层，也就是全局作用域，仍然没有；于是直接抛出 `ReferenceError: temp is not defined`。这题最容易掉进去的误区，是把“函数在哪里调用”误当成“函数就能访问调用处的变量”；JavaScript 不是动态作用域，不会因为 `b()` 这一行写在 `a` 里面，就临时把 `a` 的局部变量开放给 `b`。换句话说，**调用位置只决定当前什么时候执行，不决定词法查找链；查找链在函数创建那一刻就定好了。** 如果面试官继续追问，可以顺手补一句：第一段能访问 `temp`，并不是因为 `a` 先执行了，而是因为 `b` 在定义时就和 `a` 的词法环境绑定了关系；第二段即便把 `b()` 延迟到以后执行，本质也不会变，因为 `b` 的定义位置始终没变。面试里一句话总结就是：**第一段是闭包，`b` 定义在 `a` 里面，所以能沿词法作用域链拿到 `temp`；第二段 `b` 定义在外面，调用写在 `a` 里面也没用，因为 JavaScript 看定义位置，不看调用位置。**

1. JavaScript 的“变量提升 / 函数提升”到底是怎么发生的？为什么同名函数声明、`var`、赋值混在一起时结果常常和直觉不一样？

答：这题要分成**创建阶段**和**执行阶段**两层看。可以先用工程化说法理解：**JavaScript 不是一行一行边读边临时决定所有绑定，而是在进入一个作用域（全局、函数、模块、块）时，先做一轮“创建执行上下文 / 建立绑定”的工作，再真正按顺序执行代码。** 这不等于把源码物理搬到最上面，而是说：在当前作用域开始跑之前，引擎已经先把这个作用域里该有的名字登记好了。对 `var` 来说，名字会在创建阶段进入变量环境，并初始化为 `undefined`，所以它看起来像“可在声明前访问”。对函数声明 `function a(){}` 来说，创建阶段不只是登记名字，**函数对象本身也会先准备好并绑定到这个名字上**，所以函数声明通常能在源码定义前直接调用。对 `let/const/class` 来说，名字也会在创建阶段建立，但不会像 `var` 那样初始化成可读的 `undefined`，而是进入暂时性死区，执行到声明语句前访问会直接报错。最容易混淆的是同名情况。若同一作用域里既有 `function a(){}` 又有 `var a`，单看“声明”层面，`**var a` 不会把已经建立好的函数绑定重新冲成 `undefined`**；因为 `var` 的声明只是说“这里也有个叫 `a` 的变量名”，而这个名字在创建阶段其实已经被函数声明占住并完成初始化了。所以很多经典题里会出现：源码里明明也写了 `var a;`，但在执行到赋值前，`a` 仍然是那个函数。真正会改结果的往往不是 `var a;` 这条声明本身，而是后面的**赋值语句**，例如 `var a = 1` 到执行阶段时，本质会变成普通赋值 `a = 1`，这一步才会把原来绑定到函数对象的 `a` 改成数字。再看“同时声明两个 `function a(){}` 会不会覆盖”这个问题：**会，但发生在创建阶段的绑定决议里。** 在同一作用域内，后写的同名函数声明通常会覆盖前一个同名函数声明，因此真正进入执行阶段前，这个名字通常已经指向“最后那个函数声明对应的函数对象”了。也正因为如此，很多题里你明明看见前面写了一个函数、后面又写了一个同名函数，结果一调用拿到的是后面那个版本。把这些规则压成一句话：**进入作用域时，JS 会先建立绑定；`function` 声明在创建阶段就拿到完整函数值，`var` 在创建阶段只拿到 `undefined`，`let/const/class` 在创建阶段建立绑定但声明前不可访问；同名时，函数声明通常先占住名字，`var` 声明本身不覆盖它，真正改变值的是后续执行阶段的赋值。** 所以如果被问“是不是先扫描整份代码，把该提升的提升”，最稳的回答是：**可以近似理解为‘进入当前作用域前会先收集声明并建立绑定’，但不是简单文本搬家；更准确地说，是执行上下文的创建阶段先完成了名字与初始值规则的处理。**

1. 一份 JavaScript 代码文件从“被加载”到“真正跑完”会经历哪些阶段？浏览器和 Node 环境分别是什么流程？

答：这题最好的讲法不是死背几个术语，而是按一条统一主线去说：**宿主先接收源码，再交给引擎做解析与编译；进入某个作用域时先做创建阶段建立绑定，再进入执行阶段跑同步代码；执行过程中持续与宿主的任务系统协作；最后由事件循环决定异步回调何时继续执行。** 但浏览器和 Node 在“源码怎么被拿到、外部能力怎么接进来、事件循环怎么分阶段调度”上明显不同，所以要拆开讲。

先说所有环境都共享的内核阶段。第一步是**源码进入宿主**：浏览器通过 `<script>`、模块加载、动态导入等方式拿到 JS 文本；Node 通过 CLI 入口、`require`、`import`、内部模块解析等方式拿到文件内容。第二步是**词法分析和语法分析**，也就是把字符串源码切成 token，再生成 AST；如果源码本身语法不合法，例如括号不匹配、关键字位置非法，通常在这一步就报 `SyntaxError`，后面执行阶段根本不会开始。第三步是**编译准备**，现代引擎一般不会简单地“直接解释整份源码”，而是会把 AST 进一步转换成内部中间表示，例如字节码，并在热点代码上做 JIT 优化；面试里不必把所有引擎细节背成流水账，但要知道“解析 -> AST -> 字节码/机器码”这条链是存在的。第四步是**进入执行上下文的创建阶段**：此时当前作用域里的 `var`、函数声明、`let/const/class`、形参、`this`、外部词法环境引用等绑定关系会先建立好；也就是前面很多题说的“提升”，本质都发生在这里。第五步才是**执行阶段**：按源码顺序真正运行同步语句、求值表达式、调用函数、触发副作用。第六步是**运行期持续交互**：一旦代码里出现定时器、网络、I/O、DOM 事件、Promise reaction、`await` 恢复等，就会把后续工作交给宿主和事件循环继续推进。第七步是**垃圾回收与优化回退/重优化**：这不是源码层面的“一个固定时刻动作”，而是整个运行过程中引擎反复执行的后台机制，用于回收不可达对象、根据热点路径优化或去优化函数。

浏览器环境可以再拆得更细。浏览器拿到 HTML 后，不只是“看到 `<script>` 就立刻把 JS 跑完”这么简单，而是要和 HTML 解析、DOM/CSSOM 构建、渲染流水线互相协调。对传统同步 `<script>` 来说，HTML 解析器遇到脚本标签通常会**暂停文档解析**，先下载（若未缓存）、编译、执行脚本，再继续往下解析 HTML，所以大脚本会阻塞首屏。若用了 `defer`，脚本会并行下载，但等 DOM 解析完成后再按顺序执行；若用了 `async`，则下载完成就尽快执行，不保证和文档解析、其他脚本的相对顺序；若是 `<script type="module">`，则进入模块加载流程：浏览器会先解析 import 图，按模块依赖图取资源、实例化模块记录、建立模块环境，再执行模块顶层代码。代码真正开始执行时，会先创建全局执行上下文或模块执行上下文：普通脚本顶层 `var/function` 可能还会和 `window` 建立属性级联系，而模块顶层有独立模块作用域，默认不挂到 `window`。执行同步代码过程中，如果调用 `setTimeout`、注册 DOM 事件监听、发 `fetch` 请求，真正的等待阶段并不在 JS 主线程里死等，而是交给浏览器对应的 Web API/底层模块。等条件成熟后，浏览器把这些回调作为 task 放回事件循环。浏览器事件循环的一个高频面试模型可以压成：**执行一个宏任务 -> 清空当前微任务队列 -> 浏览器有机会进行渲染 -> 进入下一个宏任务。** Promise 的 `then/catch/finally`、`queueMicrotask`、`MutationObserver` 都会进入微任务；`setTimeout`、用户事件回调、消息事件等通常是宏任务。这里再补一个很多人忽略的浏览器细节：脚本执行并不是和渲染完全无关，长时间同步 JS 会阻塞渲染和用户交互，因为主线程要在“跑 JS”“布局绘制”“处理输入”等任务之间分配时间。

Node 环境则是另一套宿主模型。Node 启动时，先初始化 V8、libuv、内部模块系统和进程对象等运行时设施，然后解析命令行入口，加载主模块。若是 CommonJS，Node 会先读取文件内容，再把源码包进一个类似 `(function (exports, require, module, __filename, __dirname) { ... })` 的模块包装函数中执行，因此 CommonJS 顶层 `var` 并不是 `global` 的属性，而是当前模块作用域内部的局部绑定；这也是很多“浏览器里是 `window.x`，Node 里不是 `global.x`”题目的根源。若是 ES Module，Node 走的更接近标准模块流程：解析依赖图、实例化模块、建立模块环境记录，再执行模块顶层。进入代码执行后，Node 同样会经历创建阶段和执行阶段，但异步能力不是 Web API，而是通过 libuv 等运行时能力接入文件系统、网络、定时器等。Node 事件循环常被说成“分阶段”，典型可以记成 timers、pending callbacks、idle/prepare、poll、check、close callbacks；`setTimeout` / `setInterval` 的回调主要在 timers 阶段推进，`setImmediate` 在 check 阶段，I/O 完成常和 poll 阶段关系密切。还要特别提 `process.nextTick` 和 Promise 微任务：`process.nextTick` 队列优先级通常还高于常规 Promise 微任务，因此很多 Node 执行顺序题如果只套浏览器模型会出错。换句话说，Node 的大框架仍然是“同步代码 + 微任务 + 下一轮任务”，但它的宿主阶段划分比浏览器更细，且没有 DOM/渲染那套约束。

再把“文件会不会整份先扫描完”这个问题说严谨：**会先完成当前要执行单元的解析与创建工作，但不是说整个应用所有未来会加载的文件一次性全扫完。** 浏览器里，当前脚本文件会在执行前先解析；模块环境下，会先根据依赖图拉取并解析相关模块，再按模块规则执行。Node 里，入口模块执行时，也只会在需要 `require/import` 某个模块时再继续加载那个模块，不是天然把项目里所有文件全都预扫描一遍。更准确的说法是：**每个“将被执行的代码单元”在开始执行前都会先完成解析与创建阶段；模块系统再决定接下来还要加载哪些单元。**

最后给面试官一个高密度总结可以这么说：**一份 JS 文件在任何宿主里都大致经历“宿主拿到源码 -> 引擎解析并生成内部表示 -> 进入作用域创建阶段建立绑定 -> 执行同步代码 -> 把异步工作交给宿主 -> 由事件循环继续推进”的流程。浏览器额外要和 HTML 解析、模块加载、渲染流水线、Web API 协同；Node 额外要经过模块包装/模块图解析，并通过 libuv 的事件循环阶段来推进定时器、I/O 和微任务。**

1. `function` 声明算不算“函数作用域”？为什么常说它会提升，但块内函数声明又不能简单当成 `var`？

答：这题最稳的讲法是先把两个维度拆开：**第一，函数声明这个“名字”最终绑定在哪个作用域里；第二，函数执行时又能沿哪条作用域链去查变量。** 在最常见、最稳定的场景里，若 `function inner(){}` 写在某个函数体内部而不是块内，那么它的名字 `inner` 可以理解为绑定在这个外层函数作用域里；外部访问不到它，函数内部和其内部嵌套函数可以访问到它，所以在这种普通面试语境下，说“函数内部的 `function` 声明属于所在函数作用域”是成立的。但这句话不能无限泛化，因为一旦写在 `if`、`for`、裸 `{}` 这种**块内部**，规则就变复杂了：现代语义更接近**块级绑定**，但浏览器历史兼容又让非严格模式下的块内函数声明带有特殊处理，因此它既不能简单当成 `var`，也不建议在工程里依赖“块内函数声明会不会泄漏到外层”这种细节。更安全的面试说法是：**普通函数体内的函数声明，可按函数作用域理解；块内函数声明规则特殊，应把它当成独立考点。**

再把“为什么会提升”说严谨。所谓函数声明提升，不是说源码真的被搬到了顶部，而是说：**当前作用域进入创建阶段时，函数声明对应的名字会先建立绑定，并且函数对象本身通常也已经准备好。** 所以像下面这种代码：

```js
foo();
function foo() {}
```

之所以能执行，不是因为执行到第一行时临时往下偷看了源码，而是因为在进入当前作用域执行前，`foo` 这个名字已经在创建阶段绑定到了对应的函数对象上。也正因为函数对象在创建阶段就就位了，所以它和 `var` 的行为不同：`var foo` 在创建阶段只会得到 `undefined`，而 `function foo(){}` 在创建阶段通常会得到“可直接调用的完整函数值”。这也是为什么同名时，很多经典题里在执行到赋值之前，标识符仍然指向函数，而不是 `undefined`。

接着解释“同名函数声明为什么会覆盖”。若同一作用域内写了多个同名函数声明：

```js
function a() { console.log(1); }
function a() { console.log(2); }
```

进入执行阶段前，当前作用域里最终绑定到 `a` 上的，通常是**后写的那个函数声明对应的函数对象**。也就是说，同名函数声明之间的覆盖，本质发生在**创建阶段的绑定决议**里，而不是到执行阶段才一行一行慢慢覆盖。若再混入 `var a;`，单看声明本身，`var a` 通常不会把已经建立好的函数绑定重新冲成 `undefined`；但如果后面写了 `a = 3` 或 `var a = 3`，那在执行阶段运行到这句赋值时，`a` 才会真正从函数变成数字。顺着这条线，你就能把很多“为什么函数声明和 `var` 混在一起结果不是我想的那样”都解释通。

最后给一个面试级高密度总结：**函数声明在普通函数体内部时，可以按所在函数作用域理解；它之所以“会提升”，本质是创建阶段就先建立了名字到函数对象的绑定。`var` 只是在创建阶段得到 `undefined`，函数声明则在创建阶段就拿到完整函数值。块内函数声明规则特殊，别简单等同于 `var`。**

1. `[] == ![]` 为什么结果是 `true`？底层到底发生了哪些转换？

答：这题不能靠“数组是空的所以等于 false”这种口语化直觉去猜，必须严格按**运算符优先级**和**抽象相等比较（Abstract Equality Comparison）的转换规则来走。第一步先算右边的 `![]`。在 JavaScript 里，所有对象都属于 truthy，而数组本质上也是对象，所以空数组 `[]` 作为布尔判断时并不是 `false`，而是 `true`；因此 `![]` 的结果才会是 `false`。这时原表达式就变成了 `[] == false`。第二步进入 `==` 的抽象相等比较规则：当一边是对象、另一边是布尔值时，并不是直接比较，而是先把布尔值转成数字，所以 `false` 会先变成 `0`，表达式继续变成 `[] == 0`。第三步继续比较“对象”和“数字”时，规范会先把对象做 `ToPrimitive`，也就是尽量转成原始值。数组在没有自定义 `Symbol.toPrimitive` 的情况下，通常会先走 `valueOf()`，但数组的 `valueOf()` 返回的还是数组对象本身，不是原始值，于是继续走 `toString()`；空数组的 `toString()` 结果是空字符串 `''`，所以 `[]` 最终被转成 `''`。此时表达式又变成了 `'' == 0`。第四步比较字符串和数字时，字符串会再转成数字；空字符串转数字的结果是 `0`，于是整个比较最后变成了 `0 == 0`，结果当然是 `true`。把链路压缩一下就是：`[] == ![]` -> `[] == false` -> `[] == 0` -> `'' == 0` -> `0 == 0` -> `true`。这题最容易错的地方有两个：一是误以为“空数组是假值”，其实对象一律 truthy；二是把 `==` 当成“直接比较两边表面长什么样”，但它真正做的是一整套分类型转换。面试里一句话总结就是：`![]` 先把空数组当 truthy 取反成 `false`，然后 `==` 依次把 `false` 转成 `0`、把 `[]` 转成 `''`、再把 `''` 转成 `0`，最终比较的是 `0 == 0`。**

1. 以 `function Person(name){ this.name = name } var p2 = new Person('king')` 为例，`p2`、`Person`、`Object`、`Function` 之间的整张原型关系图到底是什么？

答：这题最容易乱，是因为把 **“实例对象”**、**“构造函数对象”**、**“构造函数的 prototype 属性”**、**“对象内部原型（`[[Prototype]]` / `__proto__`）”** 四层东西混在一起了。先给结论图，再逐条解释。可以把这组关系先压成下面这张图来看：

```text
p2
 ├─ own property: name = 'king'
 ├─ own property: constructor ? 没有，这是沿原型链拿到的
 ├─ own property: prototype ? 没有，实例对象默认没有 prototype 属性
 └─ [[Prototype]] / __proto__ ────────────────▶ Person.prototype
                                              ├─ own property: constructor ─▶ Person
                                              └─ [[Prototype]] / __proto__ ─▶ Object.prototype
                                                                              └─ [[Prototype]] / __proto__ ─▶ null

Person
 ├─ 作为函数对象，本身是 Function 的实例
 ├─ own property: prototype ────────────────▶ Person.prototype
 ├─ own property: constructor ? 没有这个自有属性，`Person.constructor` 通常是沿原型链得到的
 └─ [[Prototype]] / __proto__ ─────────────▶ Function.prototype
                                            └─ [[Prototype]] / __proto__ ─▶ Object.prototype ─▶ null

Function
 ├─ 是内建构造函数，也是函数对象
 ├─ own property: prototype ───────────────▶ Function.prototype
 └─ [[Prototype]] / __proto__ ─────────────▶ Function.prototype

Object
 ├─ own property: prototype ───────────────▶ Object.prototype
 └─ [[Prototype]] / __proto__ ─────────────▶ Function.prototype
```

先解释 `new Person('king')` 到底做了什么。`new` 调用时，会先创建一个全新的空对象 `p2`，再把 `p2` 的内部原型 `[[Prototype]]` 指向 `Person.prototype`，然后以这个新对象作为 `this` 去执行 `Person`，所以 `this.name = name` 最终给 `p2` 自身挂上了 `name: 'king'`。因此 `p2` 是一个**实例对象**，它自己的自有属性只有 `name`，以及之后你手动再加上的别的属性。于是 `console.log(p2.prototype)` 是 `undefined`，因为 `prototype` 是**函数对象**才默认拥有的属性，实例对象默认没有。很多人会把 `p2.__proto__` 和 `p2.prototype` 混淆，前者是实例对象内部原型链的访问入口，后者是构造函数身上的一个普通属性，这两个完全不是一回事。  
**[[Prototype]] 是规范里的内部槽，proto 是部分环境暴露出来的访问入口。**

它们不是两个不同的“原型”，而是：

- **[[Prototype]]**：对象内部真实保存的原型引用，语言规范层面的东西
- **proto**：你在代码里有时能访问到的一个历史兼容属性，用来读/改那个内部原型

所以：obj.**proto**   本质上是在**访问 obj 的 [[Prototype]]**。  
**更准确地说 [[Prototype]]**

- 这是 ECMAScript 规范里的**内部槽**
- 不能直接按这个名字写代码访问
- 每个普通对象都有这个内部链接
- 原型链查找本质就是沿着它往上找

### **proto**

- 不是规范推荐的主接口
- 是浏览器/运行时长期兼容留下来的访问器属性
- 通常定义在 Object.prototype 上
- 用来 get/set 对象的 [[Prototype]]

再看 `p2.__proto`**。因为 `new Person()` 时已经把 `p2` 的内部原型连到了 `Person.prototype`，所以 `p2.__proto__ === Person.prototype`。接着 `p2.__proto__.__proto__ === Object.prototype`，因为绝大多数普通函数的原型对象本身，都是一个普通对象，它再往上就连到 `Object.prototype`。再往上 `Object.prototype.__proto__ === null`，这表示原型链到顶了，`null` 不是对象，没有再往上的原型，所以 `p2.__proto__.__proto__.__proto`** 是 `null`。因此你再继续点：

```js
p2.__proto__.__proto__.__proto__.__proto__
```

本质是在访问：

```js
null.__proto__
```

这时就会报错，因为 `null` 没有属性，也没有原型链可走。这就是为什么“再往后没有了”。

然后看 `constructor`。`console.log(p2.constructor)` 之所以通常打印 `Person`，不是因为 `p2` 自己有一个 `constructor` 属性，而是因为 `p2` 查找不到该属性时，会顺着原型链去 `Person.prototype` 上找；而默认情况下，`Person.prototype` 会有一个 `constructor` 属性指回 `Person`。所以这句拿到的是：

```js
p2.constructor
// 等价于
p2.__proto__.constructor
// 也就是
Person.prototype.constructor
```

它默认指向 `Person`。但要注意，这不是永恒真理；如果你手动重写了整个 `Person.prototype = { ... }` 而没补回 `constructor`，这个关系就会变。

再看 `Person` 自己。`Person` 不是实例对象，而是一个**函数对象**。所有函数对象本质上也都是对象，所以它自己也有内部原型，也会参与另一条“函数对象的原型链”。对 `Person` 来说：

- `Person.prototype`：这是它作为构造函数时给未来实例共享成员用的那块原型对象
- `Person.__proto__`：这是 `Person` 这个函数对象本身的内部原型，通常等于 `Function.prototype`  
`原型对象是用来共享的模板，示例的原型指向这份模板，显然：示例本身是没有原型对象即模板的`

这两个也不是一回事。很多人看见名字都叫 prototype / `__proto__` 就混了：  
`**Person.prototype` 是给实例用的；`Person.__proto__` 是 `Person` 这个函数对象自己作为对象时的原型。**

所以：

- `console.log(Person.prototype)`：打印的是那块“实例共享原型对象”  
Person.prototype 为一个对象 {
constructor: Person,
[[Prototype]]: Object.prototype
}
- `console.log(Person.prototype.constructor)`：默认是 `Person`
- `console.log(Person.prototype.__proto__)`：通常是 `Object.prototype`
- `console.log(Person.__proto__)`：通常是 `Function.prototype`

再往更底层看内建构造器本身：

- `Function.prototype.__proto__ === Object.prototype`
- `Function.__proto__ === Function.prototype`
- `Object.__proto__ === Function.prototype`
- `Object.prototype.__proto__ === null`

这里最绕的两句是：

```js
Function.__proto__ === Function.prototype
Object.__proto__ === Function.prototype
```

原因在于：`Function` 和 `Object` 本身都是**函数对象**，既然是函数对象，它们自己的内部原型当然都应该连到 `Function.prototype`；同时 `Function` 又是“所有函数的构造器”，因此形成了那个著名的“`Function` 自己也是 `Function` 的实例”式自洽结构。这个结构不是循环原型链错误，而是语言内建对象系统有意设计出来的闭环：函数对象这一类，统一沿 `Function.prototype -> Object.prototype -> null` 这条链向上走。

最后把你列出来的每一句快速对上：

- `p2.__proto__` -> `Person.prototype`
- `p2.__proto__.__proto__` -> `Object.prototype`
- `p2.__proto__.__proto__.__proto__` -> `null`
- 再点下去报错 -> 因为在读 `null` 的属性
- `p2.constructor` -> 沿原型链拿到 `Person.prototype.constructor`，结果是 `Person`
- `p2.prototype` -> `undefined`，因为实例没有默认 `prototype`
- `Person.constructor` -> 通常沿 `Person.__proto__` 找到 `Function.prototype.constructor`，结果是 `Function`
- `Person.prototype` -> 那块实例共享原型对象
- `Person.prototype.constructor` -> `Person`
- `Person.prototype.__proto__` -> `Object.prototype`
- `Person.__proto__` -> `Function.prototype`
- `Function.prototype.__proto__` -> `Object.prototype`
- `Function.__proto__` -> `Function.prototype`
- `Object.__proto__` -> `Function.prototype`
- `Object.prototype.__proto__` -> `null`

面试里最稳的一句话总结就是：**实例对象顺着 `__proto__` 找到构造函数的 `prototype`；构造函数本身又是函数对象，所以它自己的 `__proto__` 指向 `Function.prototype`；所有普通对象原型链最终都收敛到 `Object.prototype -> null`。**

1. `function A(){}` 能不能理解成“创建了一个类”？它和真正的类到底是什么关系？

答：这题最稳的答法是：**可以类比成“类”，但本质上它首先创建的是一个函数对象，而不是传统面向对象语言里那种独立的类模板。** 当你写下 `function A(){}` 时，JavaScript 创建的是一个**可调用的函数对象**，这个函数对象默认会自带一个 `prototype` 属性，里面有默认的 `constructor: A`；以后如果你写 `new A()`，引擎会创建实例对象，并把实例的内部原型连到 `A.prototype` 上，然后再以这个实例作为 `this` 去执行 `A`。也就是说，`A` 本身并不是“类的描述对象 + 类的构造函数”分成两层的结构，而是**这个函数自己就直接承担了构造器角色**。因此如果你非要用传统面向对象语义去类比，可以把它近似理解成：“`A` 扮演类构造器，`A.prototype` 扮演实例共享方法区，`new A()` 产生实例。” 但要立刻补一句：这只是类比，不是它最底层的真实模型。JavaScript 在 ES6 之前，本质上一直是 **构造函数 + prototype + 原型链委托**；ES6 的 `class A {}` 也没有发明一套完全不同的底层对象模型，它更像是在语法层把原来的构造函数/原型写法包装得更像传统类而已。所以如果有人问“这个类的构造函数是不是这个函数本身”，在“构造函数模式”这个语境里可以回答**基本对**，但更严谨的说法应该是：**这里并不是先有一个独立类实体，再说它有一个构造函数；而是函数 `A` 自己直接被当成构造函数使用。** 进一步说，之所以很多人容易把 `function A(){}` 误当成“类”，是因为它确实具备两种让人联想到类的能力：一是可以通过 `new` 批量创建形状相近的实例；二是可以通过 `A.prototype` 给实例共享方法。但它和很多传统 class 机制仍有关键差别，比如实例方法并不是从“类模板复制”下去，而是通过原型链委托查找；函数本身既是可调用对象，又是可构造对象；构造函数和普通函数在语法上还是同一个 `function`，只是调用方式不同。面试里一句话总结就是：`**function A(){}` 本质上创建的是函数对象；当它和 `new` 连用时，这个函数就充当构造函数。你可以把它类比成“类的构造器”，但更底层的真实模型仍然是函数对象 + `A.prototype` + 原型链，而不是传统意义上的类模板。**

1. `Foo.getName()` / `getName()` / `Foo().getName()` / `new Foo.getName()` / `new Foo().getName()` / `new new Foo().getName()` 这道题为什么会得到 `2、4、1、1、2、3、3`？

答：四条规则：**第一，创建阶段里函数声明先建好绑定，`var` 只拿到名字；第二，执行阶段的赋值会覆盖前面创建好的绑定；第三，普通调用、对象调用、构造调用的 `this` 不同；第四，成员访问 `.`、`new`、函数调用 `()` 的结合顺序要分清。** 先把定义阶段整理出来。`function Foo(){...}` 创建了构造函数；`Foo.getName = function(){ console.log(2) }` 给 `Foo` 这个函数对象本身加了一个静态方法；`Foo.prototype.getName = function(){ console.log(3) }` 给未来实例共享原型方法；`var getName = function(){ console.log(4) }` 是执行阶段赋值；`function getName(){ console.log(5) }` 是函数声明。这里最关键的是全局同名 `getName`：创建阶段时，函数声明 `function getName(){ console.log(5) }` 会先把 `getName` 绑定成那个函数；但执行阶段运行到 `var getName = function(){ console.log(4) }` 时，又会把这个绑定改写成打印 `4` 的函数

先看 `Foo.getName()`。这是直接访问 `Foo` 这个函数对象自己的静态方法，前面已经明确写了：

```js
Foo.getName = function () { console.log(2) }
```

所以这里直接输出 `2`。

再看 `getName()`。根据上面说的创建阶段/执行阶段规则，虽然函数声明里有一个打印 `5` 的 `getName`，但执行阶段后面的：

```js
var getName = function () { console.log(4) }
```

已经把它覆盖了，所以这里输出 `4`。

第三个最关键：`Foo().getName()`。这里先执行 `Foo()`，注意不是 `new Foo()`，只是普通函数直接调用。非严格模式下，`Foo` 里的 `this` 会指向全局对象；而 `Foo` 函数体里做了这件事：

```js
getName = function () {
  console.log(1)
}
```

因为这里没有 `var/let/const`，它改写的是当前外层已有的那个 `getName` 绑定；在浏览器普通脚本里也可理解成把全局 `getName` 改成了打印 `1` 的函数。随后 `return this` 返回全局对象，因此 `Foo().getName()` 等价于“先把全局 `getName` 改成打印 `1` 的函数，再返回全局对象并调用它上面的 `getName`”。所以这一句输出 `1`。紧接着下一句 `getName()`，由于上一步已经把全局 `getName` 改写成打印 `1` 的版本，所以这里自然也输出 `1`。

接下来要看 `new` 结合顺序。`new Foo.getName()` 的优先级等价于：

```js
new (Foo.getName)()
```

也就是把 `Foo.getName` 这个静态方法当构造函数来 `new`。而 `Foo.getName` 这个函数体本身只是：

```js
function () { console.log(2) }
```

所以构造调用时也只是执行它，输出 `2`。这里不会走 `Foo` 本体，更不会碰 `Foo.prototype.getName`。

再看 `new Foo().getName()`。这句的结合方式是：

```js
(new Foo()).getName()
```

也就是先 `new Foo()` 生成实例。`new Foo()` 时，`this` 指向新实例对象，因此 `Foo` 函数体里的：

```js
getName = function(){ console.log(1) }
```

在非严格模式下由于没有声明符，依旧会去改外层全局 `getName`，但这不影响当前新实例本身；同时 `return this` 在构造调用里返回的是这个新实例。这个实例自身没有 `getName` 自有属性，于是属性查找沿原型链走到 `Foo.prototype.getName`，所以输出 `3`。

最后看 `new new Foo().getName()`。这句最容易懵，其实按结合顺序等价于：

```js
new ((new Foo()).getName)()
```

先执行 `new Foo()` 得到实例；再从实例上取到 `.getName`，由于实例本身没有这个属性，所以拿到的是 `Foo.prototype.getName`，也就是打印 `3` 的那个函数；最后再对这个函数执行一次 `new`。而 `Foo.prototype.getName` 的函数体仍旧只是：

```js
function () { console.log(3) }
```

所以最终输出 `3`。这题里很多人卡住，不是因为不会原型链，而是因为把 `new Foo.getName()` 和 `new Foo().getName()` 误当成一样的分组。其实前者是 **对静态方法 `Foo.getName` 做构造调用**，后者是 **先构造 `Foo` 的实例，再调用实例原型链上的 `getName`**，因此一个输出 `2`，一个输出 `3`。
