# JavaScript 面试深答手册

## 使用方式

这个文档按高频问题组织，答案统一采用可直接口述的一段式深答写法，不只给结论，还补上必要的概念解释、底层机制、存在意义、典型场景、常见坑点和可自然延伸的追问，目标不是背八股，而是把知识点讲成一条能自洽的逻辑链。文中遇到稍微专业一点、对初学者不够直观的术语，我会尽量在正文里顺手解释；如果某个词读到这里还是觉得抽象，可以再看文末新增的“术语补充题”。

## 一、语言基础

1. `var`、`let`、`const` 的核心区别是什么？
答：`var`、`let`、`const` 都是声明变量的方式，但本质区别在于变量绑定被放进什么作用域、什么时候能访问、能不能重复声明以及后续能否重新赋值。`var` 属于函数作用域，创建执行上下文时会先进入变量环境并初始化为 `undefined`，所以会出现变量提升和重复声明覆盖的问题；`let`、`const` 属于块级作用域，会进入词法环境，在执行到声明语句之前处于暂时性死区，用来强制开发者遵守“先声明再使用”的规则。它们存在的意义，本质上是在解决早期 `var` 带来的作用域污染和维护成本问题，因此现代工程里通常默认用 `const`，需要重新赋值时才用 `let`，尽量避免 `var`。
2. 什么是暂时性死区？
答：暂时性死区就是 `let` 和 `const` 在所属作用域内已经被创建、但在执行到声明语句之前仍然不能访问的那段区域，它的重点不是“没有提升”，而是“提升了但不能被读取”。底层上这和词法环境有关，词法环境你可以先理解成“JavaScript 在当前作用域里用来记录变量的一张表”，标识符也就是变量名、函数名这类名字会先登记进去，但值还没进入可用状态，所以引擎会直接抛出 `ReferenceError`，也就是“引用了当前还不能访问的变量”这类错误。它存在的意义是把很多本来运行后才暴露的变量误用问题，提前变成语言层面的硬错误，从而提升代码的确定性和可维护性。
3. JavaScript 的数据类型有哪些？
答：如果从规范角度回答，JavaScript 的值类型主要包括 `Undefined`、`Null`、`Boolean`、`Number`、`String`、`Symbol`、`BigInt` 和 `Object`，前七类通常被称为基本类型，`Object` 对应对象类型，数组、函数、日期、正则本质上都属于对象。很多人会把“引用类型”当成一类说法，它更偏工程表述，用来强调对象在变量里保存的不是实体本身，而是指向内存里那份对象数据的位置，你也可以先把它简单理解成“变量手里拿着对象的地址”。这个知识点的意义在于，它决定了比较规则、拷贝方式、序列化边界和内存管理方式；尤其函数虽然能被调用，但底层仍然是对象，所以既能执行，也能挂属性、挂原型、参与继承和反射操作。
4. 基本类型和引用类型的根本区别是什么？
答：根本区别不在“一个简单一个复杂”，而在于变量里到底保存的是什么，以及复制和传递时复制的是值本身还是引用关系。基本类型通常直接以值语义参与赋值、比较和传参，新的变量拿到的是一份独立值；对象类型则更像拿到一个地址或引用，多个变量可能同时指向同一块内存，所以看起来在改 A，实际是在改那块共享对象。这个差异是浅拷贝、深拷贝、状态污染、不可变更新这些工程问题的源头，也是为什么前端状态管理特别强调不要直接改原对象的根本原因。
5. `null` 和 `undefined` 的区别是什么？
答：`undefined` 更偏系统层面的“缺省状态”，表示一个值目前没有被赋予明确内容，比如变量声明未赋值、函数没有显式返回值、对象不存在的属性读取结果都可能是它；`null` 更偏业务层面的“主动置空”，表示这里本来应该有值，但我明确告诉你当前为空。换句话说，`undefined` 更像“还没有”，`null` 更像“这里就是空”。它们的语义差异非常重要，因为接口设计、状态流转和空值判断往往需要区分“未初始化”和“刻意置空”。
6. `==` 和 `===` 的区别是什么？为什么实际开发通常更推荐 `===`？
答：`===` 是严格相等，比较时不做隐式类型转换，只有类型相同且值相等才返回 `true`；`==` 是宽松相等，会先按规范做一系列抽象相等比较转换，比如字符串转数字、布尔转数字、对象先走 `ToPrimitive`，这个词可以先理解成“对象在参与比较前，先被想办法转成普通值，比如数字或字符串”，因此结果往往不符合直觉。`===` 更推荐的根本原因，不是因为它“高级”，而是因为它让比较规则更稳定、更可预测，能减少很多由隐式转换引起的维护成本。只有在非常明确地想利用 `null == undefined` 这个特性去同时判断两者时，才可能谨慎使用 `==`。
7. JavaScript 隐式类型转换的常见规则有哪些？
答：隐式类型转换本质上是引擎为了让不同类型的值能够参与统一运算或比较，而自动执行的一套抽象操作，核心可以抓三层：字符串拼接场景倾向转字符串，数学运算和关系比较常倾向转数字，布尔值参与运算时会先变成 `0` 或 `1`。真正复杂的地方在对象参与计算时会先执行 `ToPrimitive`，也就是尝试通过 `Symbol.toPrimitive`、`valueOf`、`toString` 把对象转成原始值后再继续计算。所以很多经典题的本质并不是死记结果，而是看清楚它到底经历了哪几步转换。
8. `NaN` 为什么不等于自己？
答：`NaN` 表示“不是一个有效数值结果”，比如非法数学运算、无法转成正常数值的结果都会落到它身上，它的设计重点是传播错误数值状态，而不是充当普通数值。规范规定 `NaN` 与任何值都不相等，包括它自己，是因为它代表的不是某个确定数字，而是一种数值计算失败的信号。也正因为如此，判断 `NaN` 不能靠 `===`，更稳妥的方式是 `Number.isNaN()`。
9. `typeof` 和 `instanceof` 分别适合解决什么问题？
答：`typeof` 适合判断基本类型和函数，因为它看的是值在语言层面属于什么基础类别；`instanceof` 适合判断一个对象是否出现在某个构造函数的原型链上，它解决的是“这个对象是不是由这条原型体系产生”的问题。两者看似都在判断类型，但本质上一个更偏值分类，一个更偏原型链关系判断，所以 `typeof null` 会得到 `object` 这个历史遗留结果，而跨 iframe 场景下 `instanceof` 也可能失真。实际工程里经常是 `typeof`、`Array.isArray`、`Object.prototype.toString.call()` 组合使用。
10. 如何准确判断一个变量是数组、对象、函数、日期、正则？
答：最稳妥的思路不是迷信某一个 API，而是根据要判断的目标选最合适的工具：数组优先用 `Array.isArray()`，函数常用 `typeof fn === 'function'`，像日期、正则、`Map`、`Set` 这类内置对象，优先用 `Object.prototype.toString.call(value)` 获取内部标签，你可以先把“内部标签”理解成对象内部自带的一种类型标记。它的底层意义在于，JavaScript 的类型判断并不是单点问题，而是值分类、原型链、对象内部标记等多套机制交织在一起，因此不存在一个万能判断函数能覆盖所有场景。

## 二、作用域与闭包

11. 什么是作用域？
答：作用域本质上是名字解析规则，也就是一个标识符在代码的哪一片区域内是可访问的，变量、函数、类声明最终都要受它约束。它存在的意义是做隔离和约束，避免所有名字都堆到全局里互相污染，让不同逻辑片段拥有自己的私有上下文。JavaScript 采用词法作用域，意味着变量能不能访问，主要由它写在什么位置决定，而不是由谁来调用决定；也正因为如此，作用域是理解变量查找、闭包、模块封装、块级变量安全的基础。
12. 什么是词法作用域？
答：词法作用域就是变量的可访问范围在代码写出来时就已经基本确定，核心看函数和块定义在哪里，而不是运行时从哪里被调用。底层上，函数在创建时会保存对其外层词法环境的引用，这里的词法环境你可以理解成“函数出生时周围那层变量上下文”，因此即使它以后跑到别处执行，也仍然沿着定义时那条环境链去找变量。这个机制存在的意义，是让代码的可读性和推理成本大幅下降，因为你看到一段函数代码时，就能顺着源代码结构判断它可能访问哪些外部变量。
13. JavaScript 中有哪些作用域类型？
答：从工程视角看，JavaScript 里常见的作用域可以分为全局作用域、函数作用域、块级作用域和模块作用域。全局作用域生命周期最长，最容易形成污染；函数作用域是 `var` 时代的核心隔离边界；块级作用域是 `let` 和 `const` 带来的更细粒度控制；模块作用域则让每个 ES Module 顶层天然独立，不会像旧时代脚本那样默认把变量暴露到全局。它们存在的意义不是分类好看，而是帮助我们明确变量生命周期、可见范围和系统边界。
14. 什么是作用域链？
答：作用域链就是变量查找时从当前环境开始，沿着外部词法环境一层层向上寻找，直到全局甚至最终找不到为止的那条查找路径。它的本质不是一条真实的数组让你操作，而是执行上下文在运行时根据词法环境形成的一种解析关系。这个机制存在的意义是让内层逻辑能够自然复用外层上下文，而不需要把所有变量都显式传参，因此闭包、嵌套函数、模块封装都能成立。
15. 什么是闭包？
答：闭包不是某种特殊语法，而是函数和它创建时所处词法环境的组合，也就是说，函数即使脱离了原来的执行位置，依然可以访问定义时外层环境中的变量。很多人把闭包理解成函数记住了变量，更准确地说是函数持有了对外层变量记录的引用，你也可以把它先理解成“函数把外层那份变量上下文带在了身上”，所以相关变量不会随着外层函数执行结束就立刻失效。闭包存在的意义非常大，它让 JavaScript 拥有了私有变量、函数工厂、事件回调、异步回调、缓存和模块封装这些能力。
16. 闭包为什么能访问外层变量？
答：因为函数在创建时就已经和外层词法环境建立了关联，这个关联不是把值拷贝进去，而是保存了一条指向外层变量记录的引用，所以函数后续执行时仍然能沿着这条关系拿到变量。只要这个函数还活着并被引用，相关词法环境就仍然处于可达状态，这里的可达你可以理解成“程序还能顺着引用关系找到它”，垃圾回收器就不能随意回收它。这个机制的本质是词法作用域在运行时的延续。
17. 闭包有哪些常见应用场景？
答：闭包最典型的应用场景包括私有变量封装、函数工厂、事件处理器、异步回调、缓存、防抖节流和模块模式，本质上都是让某段逻辑在未来执行时仍然能用到创建它时的上下文。例如计数器函数通过闭包保存内部状态，事件回调通过闭包拿到组件局部变量，防抖节流通过闭包保存定时器句柄。它的价值在于既能保留状态，又不用把状态暴露成全局变量。
18. 闭包可能带来哪些问题？
答：闭包的问题不在于它不好，而在于它会延长变量和对象的生命周期，如果设计不当，就会带来内存占用增加、状态来源不清晰、旧值被意外捕获、调试困难等问题。比如回调函数长期持有大对象引用、定时器和监听器不释放、React 中因为依赖写错而拿到旧闭包，都是典型场景。闭包的本质风险在于引用关系被拉长，所以正确思路不是回避闭包，而是控制它持有的内容、生命周期和释放时机。
19. 循环里绑定事件为什么容易出现闭包问题？怎么解决？
答：这个问题的本质是多个回调共享了同一个外层变量绑定，而不是每个回调拥有自己独立的一份值，典型情况就是 `for` 循环用 `var` 声明索引，最后所有回调访问到的都是循环结束后的同一个结果。因为 `var` 只有函数作用域，不会为每次循环创建新的词法环境，而 `let` 会在每轮迭代中生成新的块级绑定，所以天然能解决这个问题。除此之外，也可以通过 IIFE 或显式传参把当前值固化进去。
20. 你如何向面试官解释“闭包不是特性题，而是日常开发题”？
答：因为只要代码里存在回调、异步、事件、模块封装、函数返回函数、状态缓存，闭包就在工作，区别只在于你有没有意识到它。很多线上问题表面看像事件没解绑、定时器旧值、Hook 依赖错了，其实底层都能追到闭包和词法环境上。换句话说，闭包不是一块孤立知识点，而是 JavaScript 运行模型的一部分；真正理解它之后，你解释 bug、设计 API、控制内存和写抽象代码的能力都会明显更强。

## 三、执行上下文与 this

21. 什么是执行上下文？
答：执行上下文可以理解为代码真正开始运行时所处的那一帧运行环境，你也可以先把它想成“当前这段代码执行时随身带着的一包运行信息”，里面会准备变量绑定、词法环境、外部环境引用以及 `this` 等信息。全局代码会创建全局执行上下文，函数每被调用一次就会创建对应的函数执行上下文，它们由调用栈统一管理，调用栈可以先理解成“函数执行时排队进出的一摞栈结构”，先进后出。这个概念存在的意义，是把代码写出来和代码真正运行起来区分开，因为变量提升、函数调用、闭包捕获、`this` 绑定这些问题，本质上都是执行上下文层面的问题。
22. 执行上下文中包含哪些重要信息？
答：执行上下文里最值得面试时提到的核心信息包括变量环境、词法环境、外部环境引用和当前 `this` 绑定。变量环境和词法环境决定了当前代码有哪些标识符、分别处于什么状态；外部环境引用决定了变量继续往哪一层找；`this` 绑定决定了函数内部通过 `this` 能访问哪个对象。真正掌握这一点后，你会发现变量提升、闭包、`this` 和调用栈并不是分散知识点，而是在同一个执行模型下的不同表现。
23. 函数调用时 `this` 是如何确定的？
答：`this` 的确定核心不看函数定义在哪里，而看函数是怎么被调用的，也就是调用位置和调用形式。普通直接调用走默认绑定，对象点调用走隐式绑定，`call`、`apply`、`bind` 走显式绑定，`new` 调用走构造绑定，箭头函数则根本不参与这套规则，而是捕获定义时外层的 `this`。`this` 这个机制存在的意义，是让同一个函数在不同调用者下能复用逻辑并访问不同上下文，但也正因为依赖调用方式而容易出错。
24. 默认绑定、隐式绑定、显式绑定、`new` 绑定分别是什么？
答：默认绑定是普通函数独立调用时的 `this` 规则，非严格模式下通常指向全局对象，严格模式下是 `undefined`；隐式绑定指对象通过属性访问形式调用函数，此时 `this` 指向调用它的对象；显式绑定是通过 `call`、`apply`、`bind` 主动指定 `this`；`new` 绑定则发生在构造调用中，`this` 会指向新创建的实例对象。它们本质上都是语言为函数复用提供的上下文绑定规则，优先级通常是 `new` 绑定高于显式绑定，显式绑定高于隐式绑定，隐式绑定高于默认绑定。
25. `call`、`apply`、`bind` 的区别是什么？
答：这三个方法的共同点是都能显式指定函数执行时的 `this`，区别主要在于执行时机和参数组织方式。`call` 和 `apply` 都会立刻调用函数，前者按参数列表传入，后者按数组或类数组传入，这里的类数组可以先理解成“长得像数组、有下标和 length，但不是真正数组的对象”；`bind` 不会立刻执行，而是返回一个绑定好 `this` 和部分参数的新函数，适合回调透传、事件处理和函数柯里化场景。它们存在的本质意义，是把函数调用权和上下文绑定权拆开，让函数既可以被复用，也可以在运行时重新指定上下文。
26. 箭头函数为什么没有自己的 `this`？
答：因为箭头函数的设计目标不是模拟普通函数，而是提供一种更轻量的函数表达式，让回调场景下的上下文继承更自然，所以它在语言层面故意不创建自己的 `this`、`arguments`、`super` 和 `new.target`，这里的 `new.target` 你可以先把它理解成“当前函数是不是通过 `new` 调用的一个内部标记”。底层理解上，它不是有一个固定 `this`，而是根本不走普通函数那套动态绑定规则，而是直接从外层最近一层非箭头函数或全局上下文中捕获 `this`。它存在的意义主要是减少回调里 `this` 丢失的问题。
27. 箭头函数的 `this` 到底指向谁？
答：箭头函数的 `this` 不是调用时确定的，而是在定义时从外层词法环境中继承下来的，所以更准确的说法是它使用外层的 `this` 绑定结果。如果外层是普通函数，就继承那个普通函数执行时的 `this`；如果再往外都没有普通函数，就落到当前顶层上下文的 `this`。很多人说箭头函数永远指向定义处对象其实不严谨，因为它继承的是外层 `this`，不是定义位置旁边某个看起来像对象的字面量。
28. 面试里如何解释普通函数和箭头函数的使用边界？
答：判断边界的核心不是哪种写法更新，而是你是否需要函数拥有自己的动态上下文和构造能力。需要自己的 `this`、`arguments`、`prototype`、`new` 调用能力时，应使用普通函数；只是把一段逻辑作为回调，并且明确希望继承外层上下文时，箭头函数通常更合适。换句话说，普通函数更像可独立参与运行时绑定的行为单元，箭头函数更像借用外层语义环境的轻量表达式。
29. 手写 `bind` 需要考虑哪些细节？
答：手写 `bind` 时真正拉开水平差距的，不是能不能把 `this` 改掉，而是能否完整还原语言行为：第一要支持预置参数和后续参数拼接，第二要返回新函数而不是立刻执行，第三要兼容被 `new` 调用时 `this` 优先指向实例而非原绑定对象，第四还要尽量维持原型链关系，否则构造场景会失真。它的本质难点在于 `bind` 同时做了上下文绑定、参数固化和构造兼容三件事，并不是一个简单的包装函数。
30. `new` 一个函数时，底层大致发生了什么？
答：`new` 的执行过程可以概括为四步：先创建一个全新的空对象，再把这个对象的内部原型链接到构造函数的 `prototype` 上，然后以这个新对象作为 `this` 去执行构造函数，最后判断构造函数有没有显式返回对象类型的值，如果有就返回那个对象，否则返回新创建的实例。它存在的意义是让函数具备构造实例和组织原型复用的能力，这道题的底层重点其实是把 `this`、原型链和返回值规则三件事串起来。


## 四、原型与继承

31. 什么是原型？
答：原型可以理解为对象间共享能力的那层公共对象，它不是类的拷贝副本，而是一种属性查找时的委托目标。JavaScript 中很多对象自身并不直接拥有全部方法，比如数组实例能调用 `map`、`filter`，并不是每个数组都拷贝了一份这些函数，而是它们在访问不到自身属性时，会沿着内部原型链接去对应原型对象上查找。原型机制存在的意义，是在不复制方法代码的前提下实现能力复用，从而降低内存占用并支持动态扩展。
32. 什么是原型链？
答：原型链就是对象在查找属性时，从当前对象出发，沿着内部原型引用一层层向上寻找，直到 `Object.prototype`，再到 `null` 为止的这条委托路径。它的本质不是继承复制了一整套父对象内容，而是当自己找不到时，把查找动作交给上层对象继续完成。这个机制存在的意义，是让 JavaScript 用对象之间的委托关系完成方法共享和继承扩展，因此理解原型链，本质上就是理解属性到底是怎么被找到的。
33. `__proto__`、`prototype`、`constructor` 分别是什么？
答：`prototype` 是函数在作为构造函数使用时，用来给实例共享成员的那个原型对象；`__proto__` 是对象内部指向其原型的访问入口，标准层面更推荐用 `Object.getPrototypeOf()` 理解它；`constructor` 则通常是原型对象上的一个属性，默认指回创建该原型体系的构造函数。三者经常被混淆，本质上是因为它们处在不同层次：`prototype` 属于函数，`__proto__` 属于对象实例，`constructor` 属于原型对象上的普通属性。
34. JavaScript 是如何通过原型链实现继承的？
答：JavaScript 的继承本质上不是复制父类内容，而是让子对象或子类型原型对象在查找属性时，能继续沿着父级原型对象向上委托，因此更准确地说它是一种原型委托机制。只要建立好“子实例 -> 子原型 -> 父原型”这条链路，子实例访问不到的成员就会继续去父原型上查找，从而形成方法复用和行为继承。它的意义在于用很轻量的方式实现复用，但代价是链路一深之后，行为来源会变得不够直观。
35. `class` 语法糖的本质是什么？
答：`class` 的本质仍然是构造函数、原型对象和原型链，只是语法上更接近传统面向对象语言，让代码可读性更强、组织结构更清晰。比如类方法本质上还是挂在 `prototype` 上，静态方法挂在构造函数本身上，`extends` 仍然是在建立原型链关系。它存在的意义不是改变底层机制，而是降低使用原型系统的心智负担，所以真正深入理解 `class`，最后还是得回到原型链、`new`、`super` 和属性查找过程上来。
36. ES5 常见继承方式有哪些？各自优缺点是什么？
答：ES5 常见继承方式包括原型链继承、借用构造函数继承、组合继承和寄生组合继承，它们可以理解成一条不断修补问题的演进链。原型链继承能复用父原型方法，但实例属性容易共享；借用构造函数能保证实例属性独立，却拿不到父原型方法；组合继承把两者结合起来，基本可用，但会执行两次父构造函数；寄生组合继承进一步优化了这点，通过中间对象建立原型连接，减少多余构造调用。理解这些方案的意义，不是为了背历史，而是看见 JavaScript 继承模型的本质约束。
37. 组合继承和寄生组合继承的区别是什么？
答：两者都想同时解决实例属性独立和原型方法复用两个问题，区别在于建立子原型链的方式不同。组合继承通常既在子构造函数里调用一次父构造函数给实例拷属性，又通过 `new Parent()` 给子原型连链路，因此父构造函数会被执行两次；寄生组合继承则通过 `Object.create(Parent.prototype)` 之类的方式建立原型关系，只在实例初始化时调用一次父构造函数，所以更高效也更干净。这个区别的本质，是是否把原型连接误写成了再 `new` 一次父类。
38. 为什么说 JavaScript 的继承本质上是对象之间的委托？
答：因为 JavaScript 并不像很多传统语言那样在创建子类时把父类成员真正复制一份给子类，而是在访问属性时，如果当前对象没有，就把查找动作沿原型链交给上层对象继续处理。也就是说，复用的本质是查找路径复用，不是数据复制。这样做的好处是共享方法、节省内存、改动原型后实例可感知，代价是行为来源可能分散在整条原型链上，需要开发者更理解运行时查找过程。
39. 面试官让你解释“继承”和“组合”时，你会怎么回答？
答：继承强调稳定的 is-a 关系，也就是子类型本身就是父类型的一种，适合抽象层次明确、变化频率低的体系；组合强调把多个能力按需拼起来，本质是 has-a 关系，灵活性更高、耦合更低。前端业务里通常更偏向组合，因为需求变化快、状态多、依赖关系复杂，如果用继承层层叠加，后续一改就容易牵一大片。这个问题真正考察的不是概念定义，而是你有没有能力根据业务变化频率和耦合成本去做设计选择。
40. 什么时候不该使用继承？
答：当关系本身不稳定、只是想复用一点局部能力、或者一旦继承下去会出现很深的层级和强耦合时，就不应该优先使用继承。因为继承一旦建立，子类型天然要依赖父类型的结构和演进方向，父类改动会影响整条链，这在变化快的业务场景里代价很高。前端组件、页面逻辑、工具能力更常见的更优方案往往是组合、配置、函数复用或策略模式，而不是强行搭一棵继承树。

## 五、异步与事件循环

41. JavaScript 为什么是单线程？
答：通常说 JavaScript 是单线程，准确表达是同一个主线程上的 JavaScript 执行是串行的，这和它最早在浏览器中的定位有关，因为它需要频繁操作 DOM，如果允许多个脚本线程同时改界面，就会带来复杂的竞争条件和一致性问题。单线程的意义在于让页面状态变更模型更简单，但这并不代表浏览器只有一个线程，网络请求、定时器、渲染、I/O 等可以由宿主环境在别的线程或系统能力中协作完成，这里的宿主环境可以先理解成“让 JavaScript 运行起来的平台，比如浏览器和 Node.js”。JavaScript 的难点因此不在多线程共享内存，而在如何通过事件循环把异步结果安全地送回主线程执行。
42. 什么是同步任务和异步任务？
答：同步任务会直接进入当前调用栈并按顺序立即执行，前一个没结束，后一个就必须等待；异步任务则会先交给宿主环境处理等待阶段，例如计时、网络、I/O、用户交互等，当条件满足后再把对应回调放入任务队列，等待主线程合适的时候执行。它们的本质区别不是一个快一个慢，而是是否会阻塞当前调用栈继续向下执行。异步机制存在的意义，是把等待这件事从主线程中剥离出去，否则一旦请求网络或等用户操作，整个页面都会卡死。
43. 事件循环的完整流程是什么？
答：事件循环可以概括为，先执行当前宏任务里的同步代码，期间如果产生微任务就暂存，等当前宏任务执行完毕后立即清空微任务队列，再进入下一个宏任务，如此往复。底层上，调用栈只负责执行当前任务，事件循环负责协调任务何时进栈；宿主环境则负责在合适时机把定时器回调、I/O 回调、UI 事件等送入对应队列。它存在的意义，是在单线程前提下让同步、异步、界面更新能够有序协作，所以凡是涉及 Promise、`setTimeout`、`await`、DOM 事件执行顺序的问题，最后都要回到事件循环这条主线来解释。
44. 宏任务和微任务分别有哪些？
答：在浏览器里，常见宏任务包括整段脚本执行、`setTimeout`、`setInterval`、部分 I/O 和 UI 事件回调；常见微任务包括 Promise 的 `then/catch/finally` 回调、`queueMicrotask`、`MutationObserver`。两者的关键差异不是名字，而是调度优先级：每完成一个宏任务，事件循环都会先把当前产生的微任务全部清空，然后才会去取下一个宏任务。微任务存在的意义，是让某些需要尽快在当前任务结束后执行的逻辑拥有更高优先级。
45. 为什么 Promise 回调会比 `setTimeout` 更早执行？
答：因为 Promise 回调进入的是微任务队列，而 `setTimeout` 回调进入的是后续某个宏任务队列，事件循环的规则是当前宏任务结束后先清空微任务，再去执行下一个宏任务，所以 Promise 的回调通常会先执行。这里更准确的说法不是 Promise 天生比定时器快，而是同一轮事件循环里微任务优先于后续宏任务。理解这点后，很多执行顺序题就不需要死记答案了。
46. 浏览器环境和 Node.js 环境的事件循环有什么差异？
答：两者的核心思想相同，都是调用栈、任务队列和循环调度，但 Node.js 的实现会更像分阶段调度模型，比如 timers、poll、check 等阶段，而且 `process.nextTick` 的优先级又比常规微任务更特殊。也就是说，事件循环不是一套完全脱离宿主环境的抽象理论，它最终一定要落到具体平台的调度实现上。分析这类问题时一定要先区分宿主环境，否则很容易把浏览器和 Node 的执行顺序混为一谈。
47. `async/await` 本质上是基于什么实现的？
答：`async/await` 的本质是 Promise 语义上的语法糖，它把原本链式的异步流程改写成更接近同步代码的形式，提升的是表达能力和可读性，而不是改变异步底层模型。`await` 后面仍然要等一个 Promise 或类 Promise 结果，异常传播仍然依赖 Promise 的 rejected 通道，所以它没有神奇地消灭异步，只是把异步组合写得更自然。它存在的意义，是降低复杂异步流程的心智负担，让串行逻辑、错误处理和资源收尾更容易读懂。
48. `await` 为什么会让出执行权？
答：因为 `await` 遇到一个 Promise 后，不会阻塞整个线程死等，而是把当前 `async` 函数后续逻辑拆成一段待恢复的 continuation，这个词你可以先理解成“从暂停点往后还没执行的那部分代码”，再挂到 Promise 完成后的微任务中，当前函数先暂停，主线程继续往下执行别的同步任务。它本质上是在当前执行点做了一次阶段性让渡，而不是开启了一个并行线程。这个机制存在的意义，是在保留接近同步写法的同时，仍然遵守 JavaScript 单线程和事件循环的调度规则。
49. 如何处理多个异步请求的串行与并行？
答：如果后一个请求依赖前一个请求的结果，就应该串行执行，用 `await` 或 Promise 链按顺序组织；如果彼此独立，就应该尽量并行发起，常见做法是 `Promise.all`、`Promise.allSettled` 或自定义并发池控制。这个问题的本质不是记住几个 API，而是先判断任务之间有没有数据依赖、失败是否会影响整体、是否需要限流。真正成熟的回答还会补充一点：并行不等于无限并发，请求量大时还要考虑服务端压力、浏览器连接数限制和失败重试策略。
50. 如何设计一个可取消、可重试的异步请求流程？
答：一个成熟的异步请求流程通常至少要考虑取消、超时、重试、幂等性和竞态处理。幂等性你可以先理解成“同一个请求重复执行多次，结果是否仍然安全一致”，竞态可以理解成“多个请求或多个结果谁先回来不确定，可能互相覆盖”，取消可以借助 `AbortController` 之类的机制让失效请求尽早停止，重试要带最大次数和退避策略，避免瞬时故障时直接把服务端打爆，同时要区分哪些错误可重试、哪些不可重试，尤其非幂等操作不能盲目重放。这个设计的本质，是把请求从一次简单调用升级为一个可治理的异步状态机，这里的状态机你可以先理解成“按不同状态和状态切换规则运行的一套流程模型”。

## 六、Promise

51. Promise 的三种状态是什么？
答：Promise 只有三种状态，`pending`、`fulfilled`、`rejected`，也就是进行中、成功和失败。它本质上是对异步结果的一层统一抽象，把原本分散在回调里的未来结果收敛成一个可组合、可链式传递的状态机。Promise 存在的意义，不只是让代码看起来更整齐，而是让成功通道、失败通道和后续处理都拥有统一语义。
52. Promise 状态为什么只能从 `pending` 变成一次最终状态？
答：因为 Promise 被设计成异步结果的不可逆承诺，一旦最终状态还能反复改写，后续链式调用、错误传播和组合语义都会失去确定性。也就是说，状态只能落定一次，不是拍脑袋规定，而是为了让异步流程具备可预测性和可推理性。这个设计和函数返回值只能有一个最终结果是同一个思路，只不过 Promise 把这个结果延迟到了未来。
53. `then`、`catch`、`finally` 的执行规则是什么？
答：`then` 用来处理成功分支，`catch` 用来处理失败分支，`finally` 则不区分成功还是失败都会执行，通常适合资源释放、关闭 loading、收尾日志等逻辑。`finally` 一般不会改写前面链路的结果，除非它自己抛错或返回一个 rejected Promise。这个设计的意义，是把业务分支处理和统一收尾分开，让代码结构更清晰，也更接近同步代码里的 `try/catch/finally` 语义。
54. `then` 为什么可以链式调用？
答：因为每次调用 `then` 都会返回一个新的 Promise，而不是把结果直接塞回原 Promise，所以后面的链路可以继续接着处理上一步的结果或错误。这个设计非常关键，它让 Promise 不只是一个异步结果容器，而是一套异步流程组合协议。也正因为每一层都返回新 Promise，普通值、抛错、返回 Promise、返回 thenable 才能被统一纳入同一条链中继续解析，这里的 thenable 可以先理解成“虽然不是原生 Promise，但只要它有 `then` 方法，Promise 规范就会把它当成类似 Promise 的对象来处理”。
55. Promise 的值穿透是什么？
答：值穿透指的是在 `then` 或 `catch` 中如果没有传入有效的处理函数，当前成功值或失败原因不会被吞掉，而是继续沿着链往后传。这不是一个边角现象，而是 Promise 链能够支持局部处理、统一兜底的基础，否则每一层都必须写完整处理函数，链式模型就失去意义了。很多人只记结论不理解本质，实际上它反映的是 Promise 对结果传递连续性的设计。
56. Promise 错误冒泡是怎么发生的？
答：Promise 的错误冒泡，本质上是失败状态会沿着链自动向后传播，直到被某个 `catch` 或带失败处理器的 `then` 接住。这个机制之所以成立，是因为 Promise 会把同步抛错也统一包装成 rejected 状态，从而把同步异常和异步失败纳入同一条错误通道。它存在的意义是让复杂异步流程可以集中收敛错误，而不是在每一层都手动 try/catch。
57. `Promise.all`、`Promise.allSettled`、`Promise.race`、`Promise.any` 的区别是什么？
答：`Promise.all` 要求全部成功，只要有一个失败就整体失败；`Promise.allSettled` 不要求全成，而是等全部结束后把每一项结果都汇总出来；`Promise.race` 关注谁最先结束，不管成功还是失败；`Promise.any` 则关注谁最先成功，只有全部失败才整体失败。它们的本质区别不是 API 名字，而是业务目标不同：有的是追求全成功，有的是要尽快拿到一个可用结果，有的是要完整收集结果做兜底。
58. 如何手写一个简化版 Promise？
答：手写 Promise 的核心不是把三种状态写出来，而是实现一套规范化的状态流转和回调调度机制，至少要覆盖状态只落定一次、异步收集成功和失败回调、`then` 返回新 Promise、异常自动转 rejected，以及返回 Promise 或 thenable 时的继续展开，这里的 thenable 还是指“带有 `then` 方法、会被当成类似 Promise 处理的对象”。真正难的地方不在表面结构，而在链式解析规则，因为这决定了整个 Promise 模型是否可组合、是否兼容规范。面试里如果能说到 thenable 解析和循环引用保护，深度会明显更好。
59. Promise 和回调函数相比解决了哪些问题？
答：Promise 主要解决了裸回调时代的三类痛点：第一是回调嵌套深、流程横向蔓延导致难读；第二是成功和失败处理分散，错误通道不统一；第三是多个异步任务之间缺少稳定的组合协议。Promise 的价值不是让回调消失，而是把回调包装进一套可链式、可组合、可统一兜底的语义体系中，所以多个请求串行、并行汇总、统一错误收敛才真正变得优雅。
60. Promise 有没有解决回调地狱的全部问题？
答：没有，Promise 解决的是控制流组织和错误通道统一的问题，但取消、超时、并发治理、资源释放、竞态控制这些问题仍然需要额外设计。也就是说，Promise 不是万能解法，它只是把异步流程的表达能力提升了一大步，但不会自动替你完成系统治理。成熟的工程实现里，Promise 往往只是基础设施，上层还要叠加取消协议、重试策略、超时处理和状态机设计。
61. Promise 上常见静态方法除了聚合方法，还有哪些？
答：除了 `all`、`allSettled`、`race`、`any` 这类聚合方法，最常用的静态方法就是 `Promise.resolve` 和 `Promise.reject`。它们的价值不只是快速包一层，而是把同步值、异常值、thenable 都统一拉进 Promise 语义里，方便函数对外维持一致返回类型。一个异步接口如果无论快路径慢路径都返回 Promise，调用方的组合成本就会明显降低，这就是这些静态方法存在的真正意义。
62. `Promise.resolve` 和 `Promise.reject` 的典型使用场景是什么？
答：`Promise.resolve` 常用于把普通值、已有 Promise 或 thenable 统一接入 Promise 链，比如接口封装层想保证无论内部逻辑同步还是异步，对外都返回 Promise；`Promise.reject` 则适合快速生成失败态，用来做前置校验失败、权限拒绝、参数非法短路等场景。它们的本质价值，是把不同来源的结果统一收束到同一套异步协议中，而不是让每个调用方都分别判断同步值和异步值。
63. `Promise.all` 和 `Promise.allSettled` 的返回结果结构有什么差异？
答：`Promise.all` 成功时返回一个按输入顺序排列的纯结果数组，但只要任意一个失败就会整体 rejected，因此拿不到其他任务的最终状态；`Promise.allSettled` 则一定等所有任务结束，返回的每一项都包含 `status`，成功项有 `value`，失败项有 `reason`。两者的根本区别不只是返回结构不同，而是语义目标不同：前者强调全成才有意义，后者强调无论成败都要把结果汇总回来给上层决策。
64. Promise A+ 规范主要约束了哪些行为？
答：Promise A+ 规范真正核心约束的是 `then` 的行为，包括状态落定后回调如何执行、回调返回普通值怎么办、返回 Promise 或 thenable 怎么办、抛异常怎么办，以及如何避免自解析死循环。换句话说，它统一的不是“三种状态”这种表面结构，而是链式调用时结果如何稳定传播。也正因为规范把重点放在 `then` 上，所以很多自己写的 Promise 看起来能跑几个 demo，但一遇到嵌套 thenable 或异常传播就会暴露问题。
65. 为什么说 Promise A+ 的核心在于 `then` 的行为定义？
答：因为 Promise 真正难、也真正有价值的地方，不是有个 `pending/fulfilled/rejected` 状态机，而是每次调用 `then` 后如何生成新的 Promise，并把上一步结果、异常、异步值继续可靠地传下去。只要 `then` 的行为不稳定，链式组合、错误冒泡、值穿透这些 Promise 最核心的能力都会崩掉。所以说 Promise A+ 的核心在于 `then`，本质上是在强调 Promise 的价值来自可组合的协议，而不是来自某个对象长得像 Promise。


## 七、对象、拷贝与不可变

66. 浅拷贝和深拷贝的区别是什么？
答：浅拷贝只会复制第一层结构，内部嵌套对象和数组仍然共享同一份引用；深拷贝则会递归复制嵌套引用，使新旧对象彻底分离。这个问题的本质不是“复制得深不深”这么简单，而是你是否切断了原来那条引用关系。它的工程意义非常大，因为表单编辑、状态管理、缓存复用和撤销重做几乎都会碰到数据共享边界，一旦边界判断错了，就很容易出现状态污染。
67. 常见浅拷贝方式有哪些？
答：对象常用展开运算符和 `Object.assign`，数组常用展开、`slice`、`concat`，它们都只复制外层容器本身，不会继续递归处理内部引用。浅拷贝之所以常见，是因为很多业务更新并不需要把整棵数据树完全复制，只需要沿着变化路径做局部复制即可。也就是说，浅拷贝并不是低级方案，它恰恰是很多高性能不可变更新的基础。
68. 如何实现深拷贝？要考虑哪些边界情况？
答：深拷贝可以自己递归实现，也可以优先使用现代环境提供的 `structuredClone`，你可以先把它理解成“浏览器和运行环境内置的深拷贝能力”，但真正难点不在代码能不能递归，而在于边界是否处理完整，比如循环引用、`Date`、`RegExp`、`Map`、`Set`、`ArrayBuffer`、函数、原型链、属性描述符等，其中 `ArrayBuffer` 可以先理解成“用来存原始二进制数据的一块内存”，属性描述符可以理解成“属性除了值以外的额外规则，比如是否可写、是否可枚举”。深拷贝的本质是重建数据图，而不是简单把对象套娃遍历一遍，所以任何特殊对象和引用关系都会成为复杂点。工程上要先问自己是否真的需要全量深拷贝，因为它往往比想象中更贵。
69. 为什么 `JSON.parse(JSON.stringify(obj))` 不是可靠的深拷贝方案？
答：因为它本质上不是深拷贝工具，而是一种受限的 JSON 序列化再反序列化过程，所以会丢失函数、`undefined`、`symbol`、循环引用、原型链和很多内置对象信息，`Date` 还会被转成字符串。它之所以看起来能用，只是因为对纯 JSON 数据结构凑巧有效。换句话说，这个方案处理的是“可 JSON 化的数据”，不是“任意 JavaScript 对象”。
70. 什么是可变数据，什么是不可变数据？
答：可变数据强调原对象本身可以被直接改写，不可变数据强调每次变化都返回一个新值而保留旧值不动。它们的本质区别不只是写法风格，而是状态变更如何被表达和追踪：可变更新往往更直接，但容易造成共享引用污染；不可变更新则把每次变化都变成一次新的版本，便于比较、回溯和调试。很多前端框架推崇不可变更新，并不是因为它更“函数式”，而是因为它更利于做变更检测和状态管理。
71. 前端项目里为什么要尽量遵守不可变更新？
答：因为前端项目里状态往往会被多个组件、缓存层、渲染逻辑共享，如果直接原地修改，就很难判断到底哪里变了、谁改了、谁应该更新。不可变更新的价值在于把变化显式化，让引用变化本身就成为一种变更信号，这对 React 一类依赖浅比较优化渲染的框架尤其重要。更进一步说，不可变更新不是道德要求，而是一种降低副作用扩散范围的工程策略。
72. `Object.freeze` 有什么作用和局限？
答：`Object.freeze` 可以冻结对象的表层属性，阻止新增、删除和修改现有属性值，从而在一定程度上约束对象不被随意改写。它的意义主要在开发期约束和调试，比如帮助你尽早发现不该发生的直接修改；但它是浅冻结，嵌套对象仍然可能变化，而且它解决不了结构共享、版本管理、变更通知这些不可变体系中的核心问题。所以它更像一种防误操作护栏，而不是完整不可变方案。
73. `Object.assign` 和展开运算符在拷贝对象时有什么区别？
答：两者本质上都属于浅拷贝，都会遍历源对象的可枚举自有属性并把结果复制到目标对象或新对象上。展开运算符语法更简洁，更适合日常对象更新；`Object.assign` 更像显式的合并行为，适合多源对象合并或某些老代码风格。但要注意，两者都会触发 getter，这里的 getter 可以先理解成“读取某个属性时会自动执行的一段取值函数”，也都不复制原型链、都不处理深层嵌套，所以本质差异更多在书写体验而不是能力等级。
74. 数组去重有哪些方式？如何比较它们的复杂度和适用场景？
答：最常见、最推荐的方式是 `Set`，因为语义清晰、实现简单、时间复杂度通常更优；也可以用 `filter + indexOf`、双重循环或哈希表思路，但这些更多是为了展示原理。真正要注意的是，对原始值去重和对对象数组去重是两类问题，对象数组去重的关键不在于用不用 `Set`，而在于按什么 key 或组合 key 判断“相同”。所以面试里高质量回答应该从“判等标准”讲起，而不是只报几个 API 名字。
75. 你如何向面试官说明“引用类型导致的状态污染问题”？
答：我会说，所谓状态污染，本质上不是某段代码突然变脏了，而是两个看起来像独立变量的引用，实际上指向了同一份底层对象，所以你修改其中一处，另一处也被无意间改变。这个问题经常出现在表单草稿、弹窗编辑、缓存复用、props 派生状态、对象默认值复用等场景中。解决思路通常不是一上来就全量深拷贝，而是先明确数据共享边界，再按变化路径做最小必要复制。

## 八、数组与函数式编程

76. `map`、`forEach`、`filter`、`reduce` 的区别是什么？
答：`forEach` 的语义是遍历并执行副作用，不强调返回新结果；`map` 的语义是逐项映射，返回一个等长新数组；`filter` 用于按条件筛选，返回满足条件的新数组；`reduce` 则更通用，它通过累加器把一组输入折叠成任意形态的结果。它们真正的区别不只是返回值，而是表达意图不同。成熟的代码并不是所有遍历都能写，而是让读代码的人一眼看出你是在做遍历、副作用、映射、筛选还是聚合。
77. `reduce` 能解决哪些实际问题？
答：`reduce` 适合做一切“把多个输入收束成一个结果”的场景，比如求和、分组、计数、数组拍平、索引表构建、树转映射、状态机推进等。它的底层思想是维护一个累加器，让每一轮的输出成为下一轮的输入，因此比固定用途的 `map`、`filter` 更通用。也正因为它足够通用，所以既能体现抽象能力，也很容易被滥用，关键在于是否真的让代码更清晰。
78. 为什么说 `reduce` 强大但不应该滥用？
答：因为 `reduce` 的表达能力很强，强到很多不同逻辑都能塞进去，但“一切都能写”不等于“一切都该写”。如果一个场景本来用 `map`、`filter`、`some`、`every` 就能更直接地表达，硬塞到 `reduce` 里通常只会让代码阅读门槛升高。换句话说，`reduce` 的风险不在功能，而在可读性；真正成熟的使用方式，是把它留给那些确实属于聚合、归并和状态累计的问题。
79. `some` 和 `every` 的区别是什么？
答：`some` 表示只要有一个元素满足条件就返回 `true`，`every` 表示必须所有元素都满足条件才返回 `true`，两者都会在结果确定后立即短路停止遍历。它们的本质一个是在做存在性判断，一个是在做全称判断。面试里如果能再补充一句空数组上 `every` 返回 `true`、`some` 返回 `false`，就说明你不仅知道用法，还理解它们背后的逻辑定义。
80. 数组扁平化有哪些实现方式？
答：数组扁平化可以用原生 `flat`，也可以手写递归、迭代加栈、或者基于 `reduce` 递归组合实现。它的本质是把一棵嵌套结构重新线性展开，所以真正需要关注的不只是写法对不对，还包括扁平深度是否可控、递归是否可能栈溢出、大数组是否会产生过多中间对象。很多手写题的价值不在于会不会拍平，而在于你能不能顺手把复杂度、边界和场景一起讲出来。
81. 函数柯里化是什么？
答：柯里化就是把一个原本接收多个参数的函数，拆成一连串每次只接收部分参数的函数，直到参数凑齐后再真正执行。它的本质不是为了把函数写成很多层，而是把“固定配置”和“实时输入”拆开，让部分参数可以被提前固化形成可复用函数。这个思想在前端里很实用，因为很多逻辑其实都包含稳定配置和动态输入两部分。
82. 柯里化在前端开发里有哪些应用？
答：柯里化常见于校验器、日志器、埋点器、权限判断、请求封装和函数工厂场景，比如先固定模块名、埋点类型、校验规则或接口前缀，再返回真正处理运行时数据的函数。它的价值在于把公共配置前置，减少重复参数传递，让函数职责更清楚。需要注意的是，柯里化不是越多层越高级，如果只是一次性调用，直接传完整参数反而更清晰。
83. 函数组合是什么？
答：函数组合就是把多个小函数按顺序串起来，让前一个函数的输出成为后一个函数的输入，最终形成一条数据处理流水线。它的本质是在做职责拆分和流程拼接，让每个小函数专注单一转换，而整体逻辑通过组合形成更强能力。函数组合的意义在于提高复用性、可测试性和表达力，但前提是各函数的输入输出契约足够稳定，否则组合只会制造额外适配成本。
84. 防抖和节流分别是什么？适用场景有什么区别？
答：防抖强调高频触发后只在最后一次稳定下来时执行一次，适合搜索输入、表单校验、窗口 resize 收尾这类“最终结果更重要”的场景；节流强调在连续触发过程中按固定时间间隔稳定执行，适合滚动监听、拖拽、mousemove、曝光上报这类“过程反馈也重要”的场景。它们本质上都是频率治理手段，只是一个更偏减少无效收尾前的重复触发，一个更偏控制持续过程中的执行节奏。
85. 手写防抖和节流时要考虑哪些参数设计？
答：手写这类函数时，真正体现深度的不是能不能写出一个定时器，而是是否考虑了立即执行还是尾执行、定时器保存、取消能力、`this` 和参数透传、返回值处理，以及在异步请求场景下是否需要继续考虑竞态和取消。也就是说，防抖节流的工程实现不是一个十行算法题，而是一个围绕行为语义做设计的问题。

## 九、模块化

86. 为什么需要模块化？
答：模块化的根本目的不是把文件拆碎，而是把系统边界切清楚，让每个模块只暴露明确输入输出，内部实现尽量收敛，从而降低作用域污染、依赖混乱和牵一发而动全身的风险。它的意义在大型项目里尤其明显，因为没有模块边界，团队协作、测试、复用、缓存和替换实现都会越来越难。换句话说，模块化本质上是在做复杂度治理。
87. CommonJS 和 ES Module 的核心区别是什么？
答：CommonJS 更偏运行时加载和导出值快照式使用，典型写法是 `require` 和 `module.exports`；ES Module 则是标准化的静态模块系统，也就是“代码还没真正跑起来，工具就已经能提前看懂模块之间的依赖关系”，而且导出是实时绑定而不是简单值拷贝。这个差异非常重要，因为 Tree Shaking、预加载、循环依赖处理、构建优化都依赖静态结构，这里的 Tree Shaking 你可以先理解成“打包时把没用到的代码摇掉、删掉”。简化理解就是：CommonJS 更像运行时拼装，ESM 更像可分析的模块声明协议。
88. `import` 为什么要静态提升？
答：`import` 必须在编译阶段就能确定依赖关系，才能让引擎或构建工具提前建立模块图，你可以把模块图先理解成“整个项目里谁依赖谁的一张关系网”，并进一步判断导入导出绑定、处理循环依赖和执行优化。它之所以不能像普通函数一样随便写在条件里，本质上不是语法保守，而是因为 ESM 设计的核心价值就是静态可分析。如果导入关系都得等运行时再猜，Tree Shaking、预加载和很多构建能力都会失去基础。
89. ES Module 为什么更利于 Tree Shaking？
答：因为 ESM 的导入导出是静态结构，构建工具能在打包前就分析出哪些导出被真正使用、哪些没有被引用，从而安全地删除未使用代码。Tree Shaking 的本质不是“删没用文件”，而是基于静态依赖图做可证明的死代码消除，你可以把死代码消除先理解成“确认某段代码永远不会被用到后把它删掉”。也要注意，它能否真正生效还依赖副作用标记，这里的副作用可以简单理解成“执行这段代码时，会不会顺手影响外部状态”，以及打包配置和代码写法，并不是只要换成 ESM 就自动零冗余。
90. 循环依赖会带来什么问题？
答：循环依赖最麻烦的地方不是代码风格难看，而是初始化时序会变得复杂，某个模块可能在自身还没完成初始化时就被对方读取到，从而出现 `undefined`、部分初始化对象或行为异常。这个问题的本质是模块图出现了环，导致值的可用时机不再直观。它往往说明公共层抽象出了问题，或者模块职责边界没有切好。
91. 动态导入 `import()` 的使用场景是什么？
答：动态导入最典型的场景是路由懒加载、按需加载大体积依赖、根据用户操作再补加载功能模块，以及根据环境条件决定是否引入某块能力。它的本质是把原本静态阶段就要确定的依赖，延迟到运行时按需触发，因此天然形成代码分割点。它存在的意义是优化首屏和资源利用率，但也要注意拆得过碎会增加请求和调度成本。
92. 浏览器为什么原生支持 ES Module？
答：因为 ESM 已经成为 JavaScript 标准模块方案，浏览器原生支持它后，可以直接在平台层理解模块依赖关系、按模块粒度加载脚本，并为现代开发和调试提供统一基础。它的底层前提是 ESM 具备静态结构，浏览器在解析阶段就能知道接下来还需要哪些模块资源。原生支持的意义不只是省掉打包，而是让模块化从构建工具能力上升为语言和平台共同能力。
93. 模块作用域和全局作用域的关系是什么？
答：ES Module 拥有自己的独立作用域，模块顶层声明的变量默认不会自动挂到全局对象上，因此不同文件之间不会像早期脚本那样轻易互相污染。它的本质是在语言层面给每个文件建立一个私有边界，这对多人协作和大型项目尤为重要。当然，模块里如果你主动往 `window` 或 `globalThis` 上挂东西，依然会重新回到全局污染问题，所以模块作用域提供的是默认隔离，不是绝对屏蔽。
94. 在工程项目里，如何设计模块边界？
答：设计模块边界时，我通常优先看职责是否单一、输入输出是否清晰、变化频率是否一致，以及这个模块对外暴露的是稳定能力还是易变细节。一个好模块不是功能越多越好，而是高内聚、低耦合，让调用方只依赖必要接口，不感知内部实现细节。这个问题本质上考察的是你有没有从系统演化角度看模块，而不是只会按文件夹拆代码。
95. 如何避免公共模块变成“垃圾仓库”？
答：避免公共模块失控的关键，是只沉淀那些真正稳定、跨场景复用、边界清晰的能力，而不是为了少写几行代码就过早抽公共。很多公共模块最后变成垃圾仓库，本质原因是抽象时机太早、职责范围失控、调用方需求互相污染。我的经验是，至少在两个到三个真实场景里验证复用模式成立后，再决定是否上升为公共能力，这样抽象会更稳。

## 十、进阶与手写

96. 什么是垃圾回收？常见策略是什么？
答：垃圾回收就是自动识别哪些对象已经不可达、程序后续也无法再使用，并在合适时机回收其占用内存。这里的不可达和可达性分析你都可以先理解成“程序还能不能顺着引用关系找到这个对象”，JavaScript 引擎主流采用的核心思路是标记清除，也就是从全局对象、当前调用栈、闭包环境等根对象出发做可达性分析，能走到的对象保留，走不到的对象回收；真实实现中还常结合分代回收，也就是“把新创建的对象和活了很久的对象分开回收”，以提升效率。理解它的意义，在于你会知道内存泄漏不是因为没手动 `free`，而是因为某条引用链让对象一直处于可达状态。
97. JavaScript 常见内存泄漏场景有哪些？
答：常见场景包括全局变量滥用、定时器未清理、事件监听未解绑、闭包长期持有大对象、缓存无限增长、已经移除的 DOM 节点仍被引用，以及某些第三方库订阅关系未释放。它们的共性不是“写法不好看”，而是对象本来应该结束生命周期了，但仍然能从根对象沿引用链被找到，所以垃圾回收器不敢回收。排查时真正有用的不是猜，而是看堆快照和保留路径，堆快照你可以理解成“某一时刻内存对象分布的截图”，保留路径可以理解成“这个对象为什么还没被回收、是谁一路引用着它的那条路径”。
98. `WeakMap` 和 `Map` 的区别是什么？
答：`Map` 是强引用键值映射，只要 `Map` 里还持有某个键，对应对象就不会被垃圾回收；`WeakMap` 的键必须是对象，而且是弱引用，你可以先理解成“这种引用不会强行把对象留在内存里”，不会阻止这个对象被回收。它存在的意义是给对象附加元数据而又不人为延长对象生命周期，这在缓存对象私有信息、DOM 节点关联数据、框架内部依赖跟踪中非常有价值。简单说，`Map` 更适合显式数据容器，`WeakMap` 更适合附着型元数据。
99. `Set`、`Map` 在业务开发里分别适合什么场景？
答：`Set` 更适合做去重、成员存在性判断、选中态集合、黑白名单集合这类“值集合”问题；`Map` 更适合做键值映射、对象到元数据的关联、请求缓存、节点索引等“通过键快速取值”问题。它们存在的意义，是用更贴近语义的数据结构替代数组和普通对象的勉强承载。数据结构选得合适，不只是性能更好，更重要的是代码意图会更清晰。
100. Proxy 和 `Object.defineProperty` 的区别是什么？
答：`Object.defineProperty` 更像是给单个属性加拦截点，它能监听读取和设置，但对新增属性、删除属性、数组索引变化等场景支持有限；`Proxy` 则是代理整个对象，可以统一拦截 `get`、`set`、`deleteProperty`、`has`、`ownKeys` 等多种底层操作。两者的本质差异，不只是新旧 API，而是一个在补丁式拦截属性，一个在对象层做完整代理。也正因为如此，`Proxy` 更适合现代响应式系统和元编程场景，这里的元编程你可以先理解成“用代码去控制、拦截或改造代码本身的行为”。
101. Reflect 的作用是什么？
答：Reflect 提供了一套更规范、更函数化的对象操作 API，比如 `Reflect.get`、`Reflect.set`、`Reflect.defineProperty` 等，它的价值不在于发明新能力，而在于把很多原本分散、语义不统一的底层操作收拢成统一接口。尤其在 `Proxy` 里，Reflect 很适合做默认转发，因为它和代理拦截器的语义天然对应。换句话说，Reflect 是让对象底层操作更标准、更可组合的一层工具接口。
102. 为什么 `Proxy` 更适合响应式系统？
答：因为响应式系统的核心需求不是单纯知道“某个已有属性被改了”，而是要尽可能完整地感知对象行为，包括读取、设置、新增、删除、数组索引变化、`in` 判断、`for...in` 枚举等。`Proxy` 能从对象层统一拦截这些操作，因此覆盖面比 `Object.defineProperty` 更完整，尤其适合深层对象和数组。它存在的意义，是让响应式追踪从“拦属性”升级为“拦行为”。
103. 手写 `new` 要点有哪些？
答：手写 `new` 的关键不是把代码凑出来，而是准确复现语言语义：先创建新对象，再把对象原型链接到构造函数的 `prototype`，再以该对象为 `this` 执行构造函数，最后判断构造函数是否显式返回对象类型的值，如果返回了对象就用那个对象，否则返回新实例。这个题真正考的是你是否把构造调用、`this` 绑定、原型链连接和返回值规则完整串起来了。
104. 手写数组扁平化、深拷贝、防抖、节流、Promise、`bind` 时，最常见失误点是什么？
答：这类手写题最常见的问题通常不是主流程不会写，而是边界处理不完整，比如数组扁平化忘了深度控制、深拷贝忘了循环引用和特殊对象、防抖节流忘了 `this` 和参数透传、Promise 忘了异常传播和 thenable 展开、`bind` 忘了 `new` 场景。换句话说，手写题真正拉开差距的地方不是骨架，而是你有没有把语言细节和工程边界一起考虑进去。
105. 如果让你用 5 分钟总结 JavaScript，你会怎么组织表达？
答：我会按一条完整主线来讲：先从值和类型系统切入，因为这决定了比较、拷贝和状态表达；再讲作用域、闭包、执行上下文和 `this`，因为这决定了代码运行时如何找变量和绑定上下文；接着讲原型链和继承，因为这决定了对象能力如何复用；再讲异步、事件循环和 Promise，因为这解释了单线程下异步为什么还能成立；最后再落到模块化、内存管理、响应式和工程实践，因为这些才是 JavaScript 真正进入大型项目后的落点。这样的组织方式好处是每个知识点都能顺着底层原理自然往下接，而不是一堆孤立术语。

## 十一、扩展追问

106. JavaScript 引擎、运行时、宿主环境分别是什么关系？
答：JavaScript 引擎负责解析和执行语言本身，比如词法分析、语法分析、字节码或机器码生成、垃圾回收等，这里的字节码你可以先理解成“引擎为了更高效执行代码而生成的一种中间代码”；运行时是更宽的概念，包含引擎加上一些执行所需的基础设施；宿主环境则是浏览器、Node.js 这类为 JavaScript 提供外部能力的平台，它们会额外提供定时器、网络请求、文件系统、DOM、事件系统等 API。这个区分很重要，因为很多人误以为 `setTimeout`、`fetch`、DOM 都是 JavaScript 语言本身，其实它们来自宿主环境。
107. 浏览器中的 Web API 和事件循环为什么必须一起理解？
答：因为很多所谓的异步能力，比如定时器、网络请求、DOM 事件，本质上都不是 JavaScript 引擎自己完成的，而是 Web API 或浏览器其他模块先代为处理等待阶段，等条件成熟后再把回调交回事件循环调度执行。如果只知道事件循环，不知道异步任务是谁交进队列的，理解会断一截；如果只知道 Web API，不知道回调何时回到主线程执行，也解释不了实际顺序。两者必须连起来理解，异步模型才完整。
108. 为什么说 JavaScript 是动态语言又是多范式语言？
答：说它是动态语言，是因为很多类型、属性、对象结构和绑定关系都可以在运行时决定和变化；说它是多范式语言，是因为它既支持命令式编程，也支持函数式风格，还能通过原型和 `class` 语法去表达面向对象思想。这个特性让 JavaScript 非常灵活，但代价也是系统边界更容易失控，所以工程实践里才会不断引入模块化、类型系统、Lint、测试和规范来给这种灵活性加护栏。
109. 什么是调用栈，栈溢出通常怎么产生？
答：调用栈是用来管理函数执行顺序的一种后进先出结构，每调用一个函数就会压入一帧执行上下文，函数返回后再弹出。所谓栈溢出，通常是递归缺少正确终止条件、递归层级过深，或者某些间接调用形成无限递归，导致栈帧不断压入最终超过引擎允许的最大深度。理解调用栈的意义，在于很多同步阻塞、异常堆栈、递归问题、执行顺序问题都要回到它上面解释。
110. 面试中如何把 JavaScript 的知识体系讲成一条线？
答：最好的讲法不是背散点题，而是按“值如何表示、代码如何运行、对象如何复用、异步如何调度、系统如何工程化”这条线组织。也就是先讲类型和比较，再讲作用域、闭包、执行上下文和 `this`，再讲原型链和继承，再讲事件循环、Promise 和异步控制，最后讲模块化、内存、响应式和性能治理。这样一来，面试官无论从哪个点深挖，你都能把它接回整套运行模型，而不是只会背某一道题的标准答案。

## 十二、术语补充题

111. 什么是词法环境？
答：词法环境可以先理解成“某个作用域里专门用来存变量名字和变量值的一张记录表”。当代码进入一个新的作用域，比如函数、代码块、模块时，JavaScript 都会为它准备对应的词法环境，后续找变量时就会先从这张表开始找，再逐层往外找，所以它本质上是在支撑作用域和变量查找。
112. 什么是环境记录？
答：环境记录可以理解成词法环境内部真正用来存数据的那部分内容，也就是“这个名字对应什么值、现在能不能访问”的那层记录。你可以把词法环境理解成一个盒子，把环境记录理解成盒子里面真正写着变量信息的那张表，这样就比较容易把两个概念区分开。
113. 什么是执行上下文？
答：执行上下文可以简单理解成“代码当前这一轮执行时随身带着的运行信息包”，里面会放变量、外层作用域引用、`this` 指向等信息。每调用一次函数，基本都可以理解成新开了一个执行上下文，所以它是理解变量提升、调用栈和 `this` 的关键基础。
114. 什么是调用栈？
答：调用栈可以理解成“函数执行时排队进出的一摞盒子”，哪个函数先结束，哪个盒子就先弹出去，所以它遵循后进先出的规则。只要你明白函数调用会入栈、函数返回会出栈，很多执行顺序和栈溢出问题就能比较直观地理解。
115. 什么是 `ToPrimitive`？
答：`ToPrimitive` 可以简单理解成“对象在参与比较、运算、字符串拼接之前，先被尝试转换成普通值”的过程。因为很多运算并不能直接拿对象本身去算，所以 JavaScript 会先问这个对象：你能不能先给我一个数字或字符串，我再继续后面的比较和计算。
116. 什么是 thenable？
答：thenable 指的是“虽然它不一定是原生 Promise，但它长得像 Promise，因为它有 `then` 方法”。Promise 规范在处理返回值时，看到这种对象会把它当成类似 Promise 的东西继续展开处理，所以 thenable 本质上是 Promise 体系里的“兼容对象”概念。
117. 什么是宿主环境？
答：宿主环境就是让 JavaScript 真正跑起来的平台，比如浏览器、Node.js、Deno。JavaScript 语言本身没有 `setTimeout`、DOM、文件系统这些能力，这些额外能力都是宿主环境提供的，所以很多 API 其实不是 JavaScript 语言自带的，而是平台额外给它接上的。
118. 什么是幂等性？
答：幂等性可以简单理解成“同一个操作重复执行多次，最终结果是否还是一样且安全”。比如把用户资料更新成某个固定名字，多执行几次结果通常一样；但支付扣款这种操作如果重复执行多次，结果就不一样，所以它不是随便可重试的幂等操作。
119. 什么是竞态？
答：竞态可以理解成“多个异步结果谁先回来不确定，而最终结果又会受先后顺序影响”的问题。比如你连续发两次搜索请求，旧请求后回来却把新请求结果覆盖了，这就是典型竞态，所以它本质上是在讲异步结果之间的时序冲突。
120. 什么是状态机？
答：状态机可以先理解成“一个流程在不同状态之间按规则切换的模型”。例如一个请求可能有空闲、加载中、成功、失败、取消这些状态，而不同操作会让它从一个状态切到另一个状态，所以状态机本质上是在把复杂流程的变化规则画清楚。
121. 什么是 Tree Shaking？
答：Tree Shaking 可以简单理解成“打包时把项目里没有真正用到的代码删掉”。它之所以叫这个名字，是因为模块依赖关系很像一棵树，构建工具会把没有被引用到的枝叶摇掉，但它能否做好，前提是模块依赖关系足够清楚、足够静态。
122. 什么是模块图？
答：模块图可以理解成“整个项目里每个模块之间谁依赖谁的一张关系图”。构建工具只有先看懂这张图，才能知道入口从哪开始、哪些代码应该一起打包、哪些模块有循环依赖、哪些代码根本没被用到。
123. 什么是副作用？
答：副作用可以简单理解成“代码执行后除了返回值之外，还顺手改了外部世界”。比如修改全局变量、发请求、操作 DOM、打印日志、改本地存储，都属于副作用。理解副作用很重要，因为很多优化和推理都依赖“这段代码是不是只根据输入返回输出”。
124. 什么是弱引用？
答：弱引用可以理解成“我引用了这个对象，但不会强行把它留在内存里”。如果除了这条弱引用以外，程序已经没有别的地方再用这个对象了，垃圾回收器就可以把它清掉，所以 `WeakMap` 这类结构很适合存“附加信息”，而不适合做必须长期保留的主数据。
125. 什么是元编程？
答：元编程可以理解成“用代码去控制、观察、拦截、改造代码本身的行为”。像 `Proxy` 拦截对象读写、装饰器给类和方法加额外逻辑、Reflect 统一转发底层操作，都可以看成元编程思路，它不是日常业务必需品，但在框架和底层工具里非常常见。
