# JavaScript 训练题纲

## 使用方式

这个文档按高频问题组织，答案统一尽量做到先给结论，再补原理、场景和边界，方便直接表达。

---

## 一、语言基础

1. `var`、`let`、`const` 的核心区别是什么？
答：结论：`var` 是函数作用域，会提升，还能重复声明；`let` 和 `const` 是块级作用域，`const` 还要求声明时初始化，后面不能改绑定。原理/场景：工程上默认用 `const`，需要重新赋值再用 `let`，很少再用 `var`，因为它的提升和可重复声明会放大维护成本。
延伸：可顺带补一句：`const` 只保证绑定不变，不保证对象内部不可变。
2. 什么是暂时性死区？
答：结论：就是变量声明前这段区域不能访问 `let`/`const`，不是没提升，而是提升了但不能用。原理/场景：这能避免声明前误用变量，是块级作用域安全性的体现。
3. JavaScript 的数据类型有哪些？
答：结论：我会分成基本类型和引用类型。基本类型常说 `number`、`string`、`boolean`、`undefined`、`null`、`symbol`、`bigint`，对象、数组、函数这些都算引用类型。原理/场景：函数本质上也是对象，只是它额外具备可调用能力，所以既能当值传递，也能挂属性和方法。
4. 基本类型和引用类型的根本区别是什么？
答：结论：基本类型按值存和按值传，引用类型变量里放的是地址，拷贝和传参时更容易互相影响。原理/场景：这也是拷贝、传参和状态污染问题的根源，前端项目里非常常见。
5. `null` 和 `undefined` 的区别是什么？
答：结论：`undefined` 更像“未定义、未赋值”，`null` 更像“我明确地表示这里是空”。语义上 `null` 更主动一些。原理/场景：`undefined` 更像系统缺省值，`null` 更像业务上主动表达“此处为空”。
6. `==` 和 `===` 的区别是什么？为什么实际开发通常更推荐 `===`？
答：结论：`==` 会做隐式类型转换，`===` 不会。开发里更推荐 `===`，因为规则更直观，不容易出坑。原理/场景：`===` 语义更稳定，除非明确要同时判断 `null` 和 `undefined`，否则我不会依赖 `==` 的隐式转换。
延伸：高频追问是 `[] == ![]`、`null == undefined` 这类转换题。
7. JavaScript 隐式类型转换的常见规则有哪些？
答：结论：我一般抓三类规则就够了，字符串拼接优先转字符串，比较和数学运算常转数字，布尔参与运算时会转成 `0/1`。原理：对象参与运算时还会先走 `ToPrimitive`，也就是先尝试转成原始值，再继续后续比较或计算。场景：高频坑点通常就出在对象转原始值、字符串拼接和布尔转数字上，比如 `'1' + 2`、`'3' - 1`、`[] == 0` 这类题。
8. `NaN` 为什么不等于自己？
答：结论：因为 `NaN` 表示“不是一个有效数值结果”，规范里就规定它跟任何值都不相等，包括它自己。原理/场景：判断 `NaN` 更稳的方式是 `Number.isNaN()`，而不是直接做相等比较。
9. `typeof` 和 `instanceof` 分别适合解决什么问题？
答：结论：`typeof` 适合判断基本类型和函数，`instanceof` 适合看对象在不在某个原型链上。原理/场景：`typeof null` 的历史问题，以及跨 iframe 场景下 `instanceof` 不稳，都是常见追问。
10. 如何准确判断一个变量是数组、对象、函数、日期、正则？
答：结论：我一般优先用 `Array.isArray` 和 `Object.prototype.toString.call()`，函数再配合 `typeof`，这套最稳。原理/场景：工程上通常是 `Array.isArray`、`typeof`、`Object.prototype.toString.call()` 组合使用，而不是只靠一种方式。

## 二、作用域与闭包

11. 什么是作用域？
答：结论：作用域就是变量、函数和块在代码里能被访问的边界，本质上是在规定名字去哪儿找。原理：JavaScript 采用词法作用域，所以查找范围由定义位置决定，不由调用位置决定。场景：它直接影响变量隔离、闭包、模块封装和块级变量安全。边界：作用域管的是变量查找，`this` 管的是调用上下文，两者不要混在一起。
12. 什么是词法作用域？
答：结论：词法作用域就是变量的可访问范围在代码写出来时就确定了，看函数定义在哪里，而不是调用在哪里。原理：函数创建时会记录外层词法环境，所以执行时仍按定义时那条链找变量。场景：这也是闭包、模块私有变量和回调里能访问外层变量的基础。边界：JavaScript 不是动态作用域，不会因为调用位置变化就重算作用域。
13. JavaScript 中有哪些作用域类型？
答：结论：常见可以分成全局作用域、函数作用域、块级作用域和模块作用域。原理：`var` 主要受函数作用域约束，`let`/`const` 受块级作用域约束，ES Module 顶层又天然形成独立作用域。场景：项目里最常见的坑是把 `var`、全局变量和块级变量混用，最后导致污染和变量覆盖。边界：真正高频的是函数作用域和块级作用域，模块作用域更多出现在工程化场景。
14. 什么是作用域链？
答：结论：作用域链就是变量查找时从当前作用域逐层向外查找，直到全局的那条链路。原理：函数创建时会保存对外层词法环境的引用，所以运行时能沿着这条链继续向上找。场景：闭包、模块封装、回调访问外层变量，本质都依赖作用域链。边界：查找到某层为止，不会因为调用方不同而改变链路。
15. 什么是闭包？
答：结论：闭包我会说成“函数和它定义时所在词法环境的组合”，所以函数执行完了，外层变量也还能被它记住。原理/场景：它是事件回调、异步回调、模块封装和 Hooks 闭包问题的基础，也是高频追问点。
延伸：面试里常和事件回调、定时器、Hooks 旧值问题一起问。
16. 闭包为什么能访问外层变量？
答：结论：因为函数创建时就把外层词法环境关联上了，后面执行时还是按这条链去找变量。原理/场景：只要这个函数还在被引用，相关词法环境就不会被随意释放。
17. 闭包有哪些常见应用场景？
答：结论：最常见就是封装私有变量、函数工厂、事件回调、定时器、缓存和防抖节流。原理/场景：真实项目里最常见的是事件处理器、缓存、防抖节流和模块私有变量。
18. 闭包可能带来哪些问题？
答：结论：主要是变量被长期引用，容易造成内存占用和状态难追踪，写复杂了也会让代码可读性变差。原理/场景：重点不是闭包“不能用”，而是引用被长期持有后生命周期变长，更容易造成泄漏和状态难追踪。
19. 循环里绑定事件为什么容易出现闭包问题？怎么解决？
答：结论：本质是多个回调可能拿到同一个外层变量。解决一般用 `let`、IIFE，或者把当前值单独传进去。原理/场景：老项目里经常是 `var` 导致多个回调共享同一个变量，`let` 本质上是每轮循环创建新的词法环境。
20. 你如何向面试官解释“闭包不是特性题，而是日常开发题”？
答：结论：因为事件处理、异步回调、模块封装这些日常代码都在用闭包，不理解它，很多 bug 很难讲清。原理/场景：很多 React、Node 和浏览器回调问题，最后都要落回闭包和词法环境来解释。

## 三、执行上下文与 this

21. 什么是执行上下文？
答：结论：执行上下文可以理解成代码真正开始执行时的运行环境，里面会准备好变量、外层环境引用和 `this` 等关键信息。原理：全局代码、函数调用、`eval` 都会创建对应上下文，并由执行栈统一管理。场景：理解它之后，变量提升、闭包、`this` 指向和调用栈问题都会更好解释。边界：执行上下文不是作用域本身，它更像代码运行时这一帧的上下文对象。
22. 执行上下文中包含哪些重要信息？
答：结论：我一般会答变量环境、词法环境、外部环境引用以及当前的 `this` 绑定。原理：这些信息决定了变量去哪找、当前代码能访问哪些外层标识符、函数里 `this` 指向谁。场景：分析变量提升、闭包旧值、函数调用链时，本质上都是在分析这些上下文信息。边界：现代规范更常讲词法环境和环境记录，不一定再用早期教材里的 VO/AO 术语。
23. 函数调用时 `this` 是如何确定的？
答：结论：核心看调用方式，不是看定义位置。谁调用、有没有显式绑、是不是 `new`，这些都会影响 `this`。原理/场景：我会按默认、隐式、显式、`new` 和箭头函数词法绑定这几类讲，顺便说明优先级。
延伸：一句总结：`this` 主要看调用方式，不看定义位置。
24. 默认绑定、隐式绑定、显式绑定、`new` 绑定分别是什么？
答：结论：默认绑定就是普通函数直接调用，非严格模式下 `this` 通常落到全局对象，严格模式下是 `undefined`；隐式绑定是对象通过点调用，`this` 指向调用它的对象；显式绑定是用 `call`、`apply`、`bind` 人为指定；`new` 绑定是把 `this` 绑定到新创建的实例对象上。原理：它们本质上都是函数调用时确定 `this` 的不同规则，优先级通常是 `new` 绑定高于显式绑定，显式绑定高于隐式绑定，隐式绑定高于默认绑定。场景：最常见误用是把对象方法当普通函数传出去，结果从隐式绑定退化成默认绑定，`this` 就丢了。
25. `call`、`apply`、`bind` 的区别是什么？
答：结论：`call` 和 `apply` 都是立刻执行，只是传参方式不同；`bind` 不会立刻执行，它返回一个绑定好 `this` 的新函数。原理/场景：`bind` 常用于回调透传和预置参数，`call/apply` 更像一次性借用函数执行。
26. 箭头函数为什么没有自己的 `this`？
答：结论：因为它设计上就不走普通函数那套绑定规则，而是直接捕获外层作用域的 `this`。原理/场景：所以它不适合做对象方法或构造函数，但很适合回调里避免 `this` 丢失。
27. 箭头函数的 `this` 到底指向谁？
答：结论：它指向定义它时外层最近一层非箭头函数，或者再往外直到全局上下文的 `this`。原理/场景：更准确地说，是定义时外层最近一层非箭头函数或全局上下文中的 `this`。
28. 面试里如何解释普通函数和箭头函数的使用边界？
答：结论：需要自己的 `this`、`arguments`、构造能力时用普通函数；只是想继承外层上下文做回调时，箭头函数更省心。原理/场景：判断标准不是“写法新不新”，而是是否需要动态 `this`、`arguments` 或构造能力。
29. 手写 `bind` 需要考虑哪些细节？
答：结论：手写 `bind` 至少要考虑四件事，第一是绑定 `this`，第二是支持预置参数，第三是返回函数继续接收新参数，第四是兼容 `new` 调用。原理：一旦返回函数被 `new` 调用，`this` 应该优先指向新实例，而不是继续使用原先绑定对象，同时还要保证原型链关系别断。场景：很多实现只能覆盖普通调用，一到构造调用、参数拼接和原型继承就会出错，这也是手写题最容易拉开差距的地方。延伸：一句话概括就是，`bind` 不只是改 `this`，还要处理参数复用、构造优先级和原型链延续。
30. `new` 一个函数时，底层大致发生了什么？
答：结论：先创建新对象，再把原型连上，然后用这个新对象执行构造函数，最后看构造函数有没有主动返回对象。原理/场景：若构造函数显式返回对象，就优先返回该对象，这是手写题常漏点。

## 四、原型与继承

31. 什么是原型？
答：结论：原型可以理解成对象共享属性和方法的那层公共对象，多个实例会沿着它复用能力。原理：JavaScript 对象查属性时，自己没有就会继续沿内部原型链接向上找。场景：实例方法放在原型上能减少重复创建，像数组方法、日期方法本质都来自各自原型。边界：原型不是类的拷贝副本，而是运行时查找时的一层委托对象。
32. 什么是原型链？
答：结论：原型链就是对象查属性时，从当前对象一路沿原型向上查找的链路。原理：每个对象都可能关联一个上层原型，最终会走到 `Object.prototype`，再往上就是 `null`。场景：实例为什么能访问构造函数原型上的方法、数组为什么能调用 `map`，本质都来自原型链查找。边界：原型链解决的是属性访问，不等于传统 class 语言里的字段复制。
33. `__proto__`、`prototype`、`constructor` 分别是什么？
答：结论：`prototype` 是函数在作为构造函数时给实例共享成员用的原型对象；`__proto__` 是对象内部指向其原型的链接；`constructor` 是原型对象上默认指回构造函数的属性。原理：实例通过内部原型链接连到构造函数的 `prototype`，再通过那层对象访问共享方法。场景：解释 `instanceof`、手写继承、排查原型链问题时，这三个概念必须分清。边界：开发里更推荐用 `Object.getPrototypeOf()`，不要把 `__proto__` 当标准 API 滥用。
34. JavaScript 是如何通过原型链实现继承的？
答：结论：JavaScript 的继承本质上是委托，子对象自己没有的成员会沿原型链去父对象原型上找，而不是把父类内容整份复制下来。原理：只要让子类型原型对象连到父类型原型对象，就建立了继承链。场景：ES5 继承、`class extends`、实例方法复用，本质都落在这套机制上。边界：继承能复用能力，但层级过深会让查找链和耦合关系都变复杂。
35. `class` 语法糖的本质是什么？
答：结论：本质还是原型和构造函数那套，只是写法更像传统面向对象，底层机制没变。原理/场景：`class` 提升了可读性，但理解原型链之后，很多追问都能落回底层机制。
延伸：被追问 `extends`、实例方法和静态方法时，都可以落回原型链解释。
36. ES5 常见继承方式有哪些？各自优缺点是什么？
答：结论：ES5 常见继承我会答原型链继承、借用构造函数、组合继承和寄生组合继承，其中工程上最成熟的一般是寄生组合继承。原理：原型链继承能复用父类原型方法，但实例属性容易共享；借用构造函数能解决实例属性独立问题，但拿不到父类原型方法；组合继承把两者拼起来了，但会执行两次父构造函数；寄生组合继承则在保留方法复用的同时，避免了多余的一次父构造调用。场景：如果是 ES5 手写继承题，我通常会把它们当成一条演进链来讲，重点说明每种写法解决了什么问题，又留下了什么代价。
37. 组合继承和寄生组合继承的区别是什么？
答：结论：组合继承会执行两次父构造函数，一次给子实例拷属性，一次给子原型连链路；寄生组合继承只执行一次父构造函数，所以更省也更干净。原理：寄生组合继承是通过中间对象连原型，而不是直接 `new Parent()` 去做子原型。场景：ES5 手写继承时，寄生组合继承通常被认为是更成熟的写法。边界：现在业务里更常见的是 `class extends`，但底层思想还是一样。
38. 为什么说 JavaScript 的继承本质上是对象之间的委托？
答：结论：因为 JavaScript 访问属性时，不是把父类内容复制给子类，而是当前对象找不到时，沿原型链把查找动作委托给上层对象。原理：这说明继承的核心不是复制，而是查找路径的复用，所以更准确地说它是对象之间的委托关系。场景：这样做的好处是方法可以共享，修改原型上的公共能力后实例也能感知；代价是链路一深，查找关系和行为来源都会变得更绕。
39. 面试官让你解释“继承”和“组合”时，你会怎么回答？
答：结论：继承适合明确的 is-a 关系，组合更灵活，耦合更低。前端业务里我通常更偏向组合。原理/场景：前端更常用组合，因为业务变化快，继承层级太深会放大耦合。
40. 什么时候不该使用继承？
答：结论：当关系不稳定、只是想复用一点能力、或者一旦扩展就会层级很深时，我就不会优先选继承。原理：继承一旦建立，子类型会天然耦合父类型的结构和演进方向。场景：前端组件和业务模块更常见的是组合、配置和函数复用，而不是一层层 class 继承。边界：继承不是不能用，而是更适合稳定的 is-a 关系，不适合快变业务。

## 五、异步与事件循环

41. JavaScript 为什么是单线程？
答：结论：主要是浏览器里的 JS 最早就是操作 DOM 的，单线程能避免复杂的界面状态竞争。原理/场景：这里的重点不是“浏览器里绝对只有一个线程”，而是主线程执行 JS 时是串行的。
42. 什么是同步任务和异步任务？
答：结论：同步任务会直接进入当前调用栈执行，异步任务会先交给宿主环境处理，时机到了再把回调放回任务队列等待主线程执行。原理：JavaScript 主线程一次只执行一件事，所以异步不是并行执行 JS，而是把等待阶段交给浏览器或 Node。场景：定时器、网络请求、事件监听都属于典型异步。边界：异步不等于多线程，真正执行回调时仍然要回到主线程排队。
43. 事件循环的完整流程是什么？
答：结论：事件循环可以概括成一句话，先执行当前调用栈里的同步代码，当前宏任务结束后先清空微任务，再取下一个宏任务继续执行，循环往复。原理：因为微任务的优先级高于下一个宏任务，所以 Promise 回调通常会比 `setTimeout` 更早执行。场景：这套机制决定了异步代码的实际先后顺序，像 Promise、定时器、DOM 事件和 `await` 的执行时机，本质都要回到这条链路上解释。延伸：如果面试官继续追问，我会直接结合一段执行顺序题，把宏任务和微任务的入队、出队过程现场讲清楚。
44. 宏任务和微任务分别有哪些？
答：结论：常见宏任务有整体脚本、`setTimeout`、`setInterval`、I/O 回调；微任务常见有 Promise 回调、`queueMicrotask`、MutationObserver。原理：事件循环每完成一个宏任务，都会先把当前产生的微任务清空，所以两类任务的时序优先级不同。边界：浏览器和 Node.js 都有宏任务、微任务的概念，但具体分类和阶段顺序并不完全一样，分析题目时要先区分宿主环境。
45. 为什么 Promise 回调会比 `setTimeout` 更早执行？
答：结论：因为 Promise 回调进的是微任务队列，而当前宏任务结束后会先清空微任务，再去执行下一个宏任务。原理/场景：准确说法是“当前宏任务结束后先清空微任务”，不是单纯记结论。
46. 浏览器环境和 Node.js 环境的事件循环有什么差异？
答：结论：核心思想一样，但 Node.js 的阶段更多，比如 timers、poll、check，而且像 `process.nextTick` 优先级也比较特殊。原理/场景：分析这类问题时一定要先区分宿主环境，不能把浏览器事件循环和 Node.js 阶段模型完全混在一起。
47. `async/await` 本质上是基于什么实现的？
答：结论：本质上是 Promise 和生成器思想的语法糖，重点还是把异步写得更像同步。原理/场景：`async/await` 解决的是可读性问题，不是性能问题，也没有自动解决取消和并发控制。
延伸：`await` 后面的代码进入微任务，是很常见的延展点。
48. `await` 为什么会让出执行权？
答：结论：因为它会把后面的逻辑挂到 Promise 完成后的微任务里，当前函数先暂停，让调用栈继续往下跑。原理/场景：本质是把后续逻辑挂到 Promise 完成后的微任务里，所以当前函数会先暂停。
49. 如何处理多个异步请求的串行与并行？
答：结论：有依赖关系就串行，用 `await` 顺着写；没依赖就并行发，常配 `Promise.all` 或并发控制。原理/场景：`Promise.all` 很适合无依赖并发，但请求量大时还要补并发池或限流。
50. 如何设计一个可取消、可重试的异步请求流程？
答：结论：我会加 `AbortController` 做取消，重试要带次数、退避和幂等判断，避免把服务端打爆。原理/场景：设计时要区分可重试错误和不可重试错误，尤其注意非幂等请求不能乱重试。
延伸：可顺带讲请求去重、超时和竞态处理。

## 六、Promise

51. Promise 的三种状态是什么？
答：结论：Promise 只有三种状态，`pending`、`fulfilled`、`rejected`，也就是进行中、成功和失败。原理：它把异步结果统一抽象成一个最终只会落定一次的状态机。场景：这样链式调用、错误传递和并发组合才有统一语义。边界：Promise 解决的是异步结果表达，不等于自动解决取消、超时和竞态。
52. Promise 状态为什么只能从 `pending` 变成一次最终状态？
答：结论：这是为了保证异步结果可预测，避免同一个 Promise 一会成功一会失败，链式调用就没法稳定推导。原理/场景：一旦最终态可以反复变化，链式调用和异常传播都会失去确定性。
53. `then`、`catch`、`finally` 的执行规则是什么？
答：结论：`then` 处理成功，`catch` 处理失败，`finally` 不管成功失败都会走，而且一般不改最终结果，除非它自己抛错。原理/场景：`finally` 更适合做收尾，不适合承载真正的业务分支逻辑。
54. `then` 为什么可以链式调用？
答：结论：因为每次 `then` 都会返回一个新的 Promise，后面的链就能继续接着处理。原理/场景：关键点是每次 `then` 都返回新 Promise，这也是链式组合的基础。
55. Promise 的值穿透是什么？
答：结论：值穿透就是 `then` 里如果没传函数，或者传的不是有效处理函数，当前值或错误会继续往后传。原理：Promise 链要求每一环都能把结果继续传递，所以空处理器会被当成透传。场景：这也是为什么很多链路可以只在最后统一写一个 `catch`。边界：值穿透传的是当前结果，不代表错误已经被处理。
56. Promise 错误冒泡是怎么发生的？
答：结论：只要链路里抛错，或者返回一个 rejected Promise，错误就会顺着后面的链一直传，直到被最近的 `catch` 接住。原理：Promise 会把同步异常自动包装成 rejected 状态，所以能统一走错误通道。场景：这让复杂异步流程可以集中收敛错误，而不用每一层都手动套 try/catch。边界：如果中间已经把错误消费并返回正常值，后面的链就会重新回到成功分支。
57. `Promise.all`、`Promise.allSettled`、`Promise.race`、`Promise.any` 的区别是什么？
答：结论：`all` 看全部成功，`allSettled` 看全部结束，`race` 看谁先结束，`any` 看谁先成功。原理/场景：选择时要先想清楚业务目标是“全成”“先返回”“全量兜底”还是“谁先成功”。
延伸：`all` 的短路失败、`any` 的 `AggregateError` 都是常见追问。
58. 如何手写一个简化版 Promise？
答：结论：核心就三块，状态管理、回调收集、`then` 返回新 Promise 并处理返回值和异常。原理/场景：真正难点不在三种状态，而在 thenable 展开和异常传播这些规范细节。
59. Promise 和回调函数相比解决了哪些问题？
答：结论：Promise 主要解决了回调嵌套深、错误分散、异步流程不好串联的问题。原理：它把异步结果抽象成统一对象，让成功、失败和后续链路都能按规则组合。场景：像多个请求串行、并行汇总、统一兜底这些场景，用 Promise 明显比裸回调更清晰。边界：它只是改善控制流，不会自动解决取消、资源释放和并发治理。
60. Promise 有没有解决回调地狱的全部问题？
答：结论：没有，它只是把控制流整理得更清楚，复杂并发、取消、资源释放这些问题还得额外设计。原理/场景：复杂业务里还要继续解决取消、超时、竞态、资源释放这些问题。
61. Promise 上常见静态方法除了聚合方法，还有哪些？
答：结论：除了 `all`、`race` 这类聚合方法，最常用的静态方法就是 `Promise.resolve` 和 `Promise.reject`。原理：它们的作用是把普通值或错误快速包装成统一的 Promise 结果，方便接入链式流程。场景：封装异步接口、统一返回值类型、提前中断流程时很常见。边界：`Promise.resolve` 遇到 thenable 会继续展开，不只是简单包一层。
62. `Promise.resolve` 和 `Promise.reject` 的典型使用场景是什么？
答：结论：`Promise.resolve` 常用来把普通值或 thenable 统一接进 Promise 链，`Promise.reject` 常用来快速返回失败态，中断后续成功分支。原理：它们的价值不只是“包一层”，而是把同步值、异步值和错误都收拢到同一套 Promise 语义里。场景：封装请求层、统一函数返回值、做前置校验失败短路时都很常见，比如参数不合法时直接 `Promise.reject`，而不是继续往下走。
63. `Promise.all` 和 `Promise.allSettled` 的返回结果结构有什么差异？
答：结论：`Promise.all` 成功时返回一个纯结果数组，但只要有一个失败就整体 rejected；`allSettled` 一定会等全部结束，返回每一项的 `status`、`value` 或 `reason`。原理：两者的区别本质上是全成功才返回，还是无论成败都汇总结果。场景：前者适合强依赖全成的请求组，后者适合批量任务统一收集结果再兜底。边界：`allSettled` 不会帮你自动处理失败，只是把失败结果显式交还给你。
64. Promise A+ 规范主要约束了哪些行为？
答：结论：Promise A+ 重点约束的是状态流转、`then` 的返回规则、异常捕获，以及 thenable 的解析行为。原理：它真正统一的是链式调用时的语义，也就是返回普通值怎么办、返回 Promise 怎么办、抛异常怎么办、遇到 thenable 怎么展开。场景：这也是为什么很多 Promise 实现看起来能用，但只要 `then` 解析规则不完整，就经不起规范测试。
65. 为什么说 Promise A+ 的核心在于 `then` 的行为定义？
答：结论：因为 Promise 真正难的不是三种状态，而是链式调用时 `then` 如何返回新 Promise、如何处理异常、如何展开 thenable。原理：只要 `then` 的行为稳定，异步组合、错误传递和结果透传才会稳定。场景：这也是手写 Promise 时最容易写错、最能拉开深度的地方。边界：很多实现看似能跑，但只要 `then` 解析规则不完整，就不算真正兼容规范。

## 七、对象、拷贝与不可变

66. 浅拷贝和深拷贝的区别是什么？
答：结论：浅拷贝只复制第一层结构，嵌套对象和数组仍然共享引用；深拷贝会递归复制嵌套引用，让新旧对象彻底分离。原理：引用类型变量里保存的是地址，所以只复制外层结构并不能切断内部引用关系。场景：状态更新常用浅拷贝配合结构共享，完整数据隔离或序列化场景才更需要深拷贝。边界：深拷贝更重，不要把它当成任何场景的默认方案。
67. 常见浅拷贝方式有哪些？
答：结论：对象常用展开运算符和 `Object.assign`，数组常用展开、`slice`、`concat`。原理：这些方式都只是复制外层容器，内部引用不会继续展开。场景：前端状态更新里最常见的就是按变化路径做一层或几层浅拷贝。边界：一旦对象里有深层嵌套，浅拷贝后直接改内部字段仍然会污染旧数据。
68. 如何实现深拷贝？要考虑哪些边界情况？
答：结论：可以自己递归，也可以用 `structuredClone`。边界上要考虑循环引用、`Date`、`RegExp`、`Map`、`Set`、函数这些特殊值。原理/场景：现代浏览器可优先考虑 `structuredClone`，但状态管理场景更常见的是按变化路径做局部拷贝。
延伸：如果是状态管理场景，很多时候只需要做结构共享，不需要全量深拷贝。
69. 为什么 `JSON.parse(JSON.stringify(obj))` 不是可靠的深拷贝方案？
答：结论：因为它会丢函数、`undefined`、`symbol`，`Date` 会变字符串，循环引用还会直接报错。原理/场景：它更像受限的序列化手段，而不是通用深拷贝方案。
70. 什么是可变数据，什么是不可变数据？
答：结论：可变数据是原对象本身可以被直接改写，不可变数据更强调每次变更都返回一个新值，旧值保持不动。原理：不可变更新让变更前后边界更清晰，也更利于按引用比较判断是否变化。场景：React、Redux 这类场景非常依赖不可变更新来做渲染优化和状态追踪。边界：不可变不代表绝对不能修改，而是要明确在哪一层保证对外呈现为不可变。
71. 前端项目里为什么要尽量遵守不可变更新？
答：结论：因为状态变化更可追踪，也更利于框架做比较优化，减少副作用和状态污染。原理/场景：React 这类框架依赖引用变化做优化，不可变更新能明显降低状态串改问题。
延伸：这里能自然衔接到 React 的渲染优化和状态设计。
72. `Object.freeze` 有什么作用和局限？
答：结论：它能冻结对象表层属性，防止新增、删除、改值，但它是浅冻结，嵌套对象照样能变。原理/场景：它更适合开发期约束和调试，不是完整的不可变方案。
73. `Object.assign` 和展开运算符在拷贝对象时有什么区别？
答：结论：两者本质上都属于浅拷贝，日常体验上展开语法更简洁直观，`Object.assign` 更像显式地把多个对象合并到目标对象上。原理：它们都会遍历可枚举自有属性并复制到新对象或目标对象。场景：简单对象更新我更常用展开，做多源合并或需要兼容老写法时会用 `Object.assign`。边界：两者都不会处理深层引用，也都可能触发 getter 取值。
74. 数组去重有哪些方式？如何比较它们的复杂度和适用场景？
答：结论：最常见是 `Set`，简单高效；也可以用 `filter + indexOf`、双重循环。原理/场景：大多数场景我会优先选 `Set`，因为可读性和时间复杂度都更好；如果是对象数组去重，关键就不是 `Set`，而是先明确按什么 key 去重。
75. 你如何向面试官说明“引用类型导致的状态污染问题”？
答：结论：我会说两个变量看起来是两份数据，实际却指向同一块引用，所以一处修改会把另一处状态一起带脏。原理：对象和数组复制的往往只是引用，不是内容本身。场景：这类问题在表单编辑、缓存复用、状态管理里非常常见，比如弹窗里改了一份草稿，结果把列表原数据也改了。边界：解决思路通常不是全量深拷贝，而是明确共享边界并按变化路径复制。

## 八、数组与函数式编程

76. `map`、`forEach`、`filter`、`reduce` 的区别是什么？
答：结论：`forEach` 主要遍历，`map` 产出新数组，`filter` 做筛选，`reduce` 是把一组值累计成一个结果。原理/场景：选择它们时最重要的是表达意图，而不是把所有遍历都统一成一种写法。
77. `reduce` 能解决哪些实际问题？
答：结论：`reduce` 适合把一组值累计成一个结果，比如求和、分组、计数、拍平、构建映射表。原理：它的本质是维护一个累加器，让每一轮遍历都在前一轮结果上继续计算。场景：对象分组、树结构整理、接口结果归一化这些都很常见。边界：能用 `map`、`filter` 直接表达的场景，不要为了炫技硬塞给 `reduce`。
78. 为什么说 `reduce` 强大但不应该滥用？
答：结论：因为很多简单逻辑硬塞进 `reduce` 可读性会很差，团队维护成本反而更高。原理/场景：如果本来用 `map` 或 `filter` 就能讲清，硬塞进 `reduce` 反而会拉高维护成本。
79. `some` 和 `every` 的区别是什么？
答：结论：`some` 是只要有一个元素满足条件就返回 `true`，`every` 是必须所有元素都满足才返回 `true`。原理：两者都会短路，所以一旦结果确定就不会继续遍历。场景：权限命中、命中一个异常项适合 `some`，表单全量校验、全部状态一致适合 `every`。边界：空数组上 `every` 会返回 `true`，这是经常被忽略的边界。
80. 数组扁平化有哪些实现方式？
答：结论：可以用原生 `flat`，也可以递归、循环加栈，或者用 `reduce` 组合。原理/场景：数据层级很深时要考虑递归栈深度和大数组场景下的性能。
81. 函数柯里化是什么？
答：结论：柯里化就是把一个接收多个参数的函数拆成多层函数，每次先收一部分参数，最后再统一执行。原理：这样做能把一部分参数提前固定下来，形成更细粒度的可复用函数。场景：校验器、日志器、事件处理器、配置化函数工厂都很适合。边界：柯里化不是越多层越高级，过度使用会明显拉高阅读成本。
82. 柯里化在前端开发里有哪些应用？
答：结论：最常见的是参数复用和函数工厂，比如先固定接口地址、埋点类型、校验规则，再生成具体处理函数。原理：它把公共配置和实时输入拆开，能让函数职责更清楚。场景：表单校验、日志封装、权限判断、高阶组件参数注入都很常见。边界：如果只是一次性调用，直接传完整参数往往更清晰。
83. 函数组合是什么？
答：结论：函数组合就是把多个小函数按顺序串起来，前一个输出作为后一个输入，最后形成一条处理流水线。原理：这样每个函数都只关心单一职责，整体逻辑却能层层拼起来。场景：数据清洗、表单处理、请求结果转换、函数式工具库里都很常见。边界：组合要求输入输出契合，否则会引入额外的适配成本。
84. 防抖和节流分别是什么？适用场景有什么区别？
答：结论：防抖是高频触发时只在最后一次停下来后执行，适合搜索输入、表单校验这类“结果比过程重要”的场景；节流是固定时间窗口内最多执行一次，适合滚动、拖拽、resize 这类“过程也要持续反馈”的场景。原理：两者本质都是做频率治理，只是防抖更强调收尾，节流更强调稳定输出。边界：真实业务里还要考虑是否需要立即执行、尾执行和取消能力，否则很容易出现体验不符合预期的问题。
延伸：搜索框通常用防抖，滚动监听通常用节流，这是最常见的业务表达。
85. 手写防抖和节流时要考虑哪些参数设计？
答：结论：我会考虑立即执行还是尾部执行、定时器保存、取消能力，还有 `this` 和参数透传。原理/场景：如果函数里还有异步请求，往往还要继续考虑竞态和取消。

## 九、模块化

86. 为什么需要模块化？
答：结论：因为代码一旦变大，不做模块化就很容易出现作用域污染、依赖混乱和改一处牵一片的问题。原理：模块化本质上是在做边界切分，让每个模块只暴露明确输入输出，内部实现尽量收敛。场景：这样团队协作、复用、测试和按职责拆分都会更稳。边界：模块也不是越碎越好，拆得过细会带来依赖跳转多、理解成本高的问题。
87. CommonJS 和 ES Module 的核心区别是什么？
答：结论：CommonJS 更偏运行时加载和导出值拷贝，ESM 是静态分析、编译时可确定依赖，而且是实时绑定。原理/场景：ESM 的静态结构更适合现代工程化，但 Node 里也要注意和 CommonJS 的互操作。
延伸：如果再补一句“ESM 是实时绑定，CommonJS 更偏值拷贝”，会更完整。
88. `import` 为什么要静态提升？
答：结论：因为编译阶段就要知道依赖关系，打包、Tree Shaking、循环依赖处理都依赖这个前提。原理/场景：只有依赖关系静态可分析，打包、Tree Shaking 和预加载优化才做得稳。
89. ES Module 为什么更利于 Tree Shaking？
答：结论：因为它的导入导出是静态可分析的，构建工具能更准确地删掉没用代码。原理/场景：但 Tree Shaking 也依赖副作用标记和打包配置，不是换成 ESM 就自动生效。
90. 循环依赖会带来什么问题？
答：结论：会导致某个模块拿到未初始化完成的值，轻则 `undefined`，重则逻辑时序混乱。原理/场景：循环依赖真正难的是初始化时序，公共层设计要尽量避免双向引用。
91. 动态导入 `import()` 的使用场景是什么？
答：结论：动态导入最常见的场景就是路由懒加载、按需加载大模块，以及某个条件成立后再补加载能力。原理：它把依赖从静态导入变成运行时触发，所以能天然形成代码分割点。场景：后台页面、图表编辑器、富文本、导出模块这些都很适合延后加载。边界：拆得过碎会增加请求和调度成本，不是模块越多越好。
92. 浏览器为什么原生支持 ES Module？
答：结论：因为 ESM 已经是标准模块方案，浏览器直接支持后，模块依赖关系可以由平台原生识别和加载。原理：ESM 具备静态结构，浏览器在解析阶段就能知道依赖图。场景：现代开发环境、原生模块调试、轻量项目都能直接受益。边界：真实生产环境里仍常配合打包器处理兼容性、压缩、拆包和缓存优化。
93. 模块作用域和全局作用域的关系是什么？
答：结论：模块有自己的独立作用域，模块顶层定义的变量默认不会自动挂到全局对象上。原理：每个模块都会形成独立命名空间，避免不同文件互相污染。场景：这让多人协作和公共模块管理更安全，减少全局冲突。边界：模块里如果主动往 `window` 或 `globalThis` 上挂东西，依然会回到全局污染问题。
94. 在工程项目里，如何设计模块边界？
答：结论：我会按职责拆，保证模块输入输出清晰，别让一个模块同时管数据、UI、请求和状态。原理/场景：高质量模块边界通常同时考虑职责、变化频率和对外暴露的稳定性。
95. 如何避免公共模块变成“垃圾仓库”？
答：结论：关键是只沉淀真正稳定、跨场景复用、边界清楚的能力，不要为了少写几行就过早抽公共。原理：一旦公共层失控，依赖会四处蔓延，后续改动成本会被指数放大。场景：我通常会先在两三个真实场景里验证，再决定是否上升为公共模块。边界：公共层不是越大越好，可维护比看起来复用很多更重要。

## 十、进阶与手写

96. 什么是垃圾回收？常见策略是什么？
答：结论：垃圾回收就是自动释放那些已经不可达、后续程序也拿不到的内存。原理：JavaScript 引擎主流思路是标记清除，也就是从根对象出发做可达性分析，能走到的对象保留，走不到的对象就回收；为了兼顾效率，真实实现里通常还会配合分代回收。场景：理解这一点后，闭包、全局缓存、事件监听不释放为什么会泄漏，就更容易解释清楚。
97. JavaScript 常见内存泄漏场景有哪些？
答：结论：常见场景有全局变量滥用、定时器没清、事件没解绑、闭包长期持有引用、大对象缓存不释放，以及 DOM 节点已经移除但引用还在。原理：它们的共性都是对象本来该结束生命周期了，却仍然能从某条引用链上被找到，所以垃圾回收器不敢回收。场景：真实排查时我会优先看 Chrome Memory 面板、堆快照和 Retainers 路径，确认到底是哪条引用把对象留住了。
98. `WeakMap` 和 `Map` 的区别是什么？
答：结论：`WeakMap` 的键只能是对象，而且是弱引用，不会阻止垃圾回收，适合存对象私有附加信息。原理/场景：这也是为什么很多框架内部会用 `WeakMap` 给对象挂元数据。
99. `Set`、`Map` 在业务开发里分别适合什么场景？
答：结论：`Set` 更适合去重和快速判断某个值是否存在，`Map` 更适合做键值映射，尤其是键不只限于字符串时。原理：两者都比普通对象和数组更贴近特定语义，也通常有更清晰的 API。场景：选中态集合、去重缓存适合 `Set`，对象元数据、节点映射、请求缓存更适合 `Map`。边界：如果只是简单 JSON 结构透传，普通对象反而更直观。
100. Proxy 和 `Object.defineProperty` 的区别是什么？
答：结论：`Proxy` 是代理整个对象，能力更全，新增删除属性也能拦；`defineProperty` 更像给单个属性打补丁。原理/场景：Vue 3 选 Proxy 也是因为它对新增删除属性和数组场景支持更完整。
延伸：高频延展是为什么 Vue 2 用 `defineProperty`、Vue 3 换成 `Proxy`。
101. Reflect 的作用是什么？
答：结论：Reflect 提供了一套更规范、更函数化的对象操作 API，很多能力和对象内部行为是一一对应的。原理：它把诸如 `get`、`set`、`defineProperty` 这些操作收成统一接口，也方便在 `Proxy` 里转发默认行为。场景：写代理、做元编程或需要拿到布尔结果而不是直接抛异常时很有价值。边界：Reflect 不是新能力的来源，更多是把原有底层操作表达得更规范。
102. 为什么 `Proxy` 更适合响应式系统？
答：结论：因为它能直接监听新增、删除、数组变化这些情况，覆盖面比 `defineProperty` 更完整。原理/场景：但能力更强也意味着调试和代理层级管理要更谨慎。
103. 手写 `new` 要点有哪些？
答：结论：核心就四步，先创建新对象，再把它连到构造函数原型上，然后用这个对象执行构造函数，最后判断返回值是否为对象。原理：这四步正对应 `new` 在运行时做的事情。场景：这是原型、`this`、构造调用和返回值规则的综合题。边界：最常漏的是构造函数如果显式返回对象，要优先返回该对象。
104. 手写数组扁平化、深拷贝、防抖、节流、Promise、`bind` 时，最常见失误点是什么？
答：结论：最常见不是主流程不会写，而是边界漏了，比如 `this`、参数透传、异常、循环引用、定时器和构造场景。原理/场景：手写题区分深浅的关键，往往就是这些边界条件。
105. 如果让你用 5 分钟总结 JavaScript，你会怎么组织表达？
答：结论：我会按五块来讲，第一是语言基础和类型系统，第二是作用域、闭包和 `this`，第三是原型链与继承，第四是异步、Promise 和事件循环，第五是工程实践里的模块化、内存和性能。原理：这样的组织方式是先讲日常最常用的能力，再逐步落到底层机制。场景：一旦框架搭起来，后续无论面试官往闭包、异步还是原型链深挖，我都能自然接上去。
