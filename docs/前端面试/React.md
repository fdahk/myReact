# React 训练题纲

## 使用方式

这个模块按高频 React 问题展开，答案统一尽量覆盖结论、机制、业务场景和边界，避免只停留在 API 背诵层。

---

## 一、React 基础认知

1. React 的核心思想是什么？
答：React 的核心思想可以概括成三件事：把界面拆成可组合的组件，用状态而不是手动 DOM 操作去描述界面，再由框架负责在状态变化后把界面更新到正确结果。它真正解决的不是“怎么写几个按钮”，而是复杂页面里数据、交互和视图如何长期保持一致的问题，所以 React 很强调单向数据流、组件边界和可预测更新，因为项目越大，越需要一套能让人推导“数据从哪来、为什么会更新、这次更新会影响谁”的稳定心智。

2. 为什么 React 强调声明式 UI？
答：因为声明式 UI 让开发者更多表达“当前状态下页面应该长什么样”，而不是一步步命令浏览器“先创建节点、再改文本、再改样式、再插入父节点”。这样做的真正价值不是少写几行 DOM API，而是把界面结果和状态直接绑定起来，代码更容易阅读、回放和重构；当需求不断变化时，你只需要关心状态是否正确，至于中间怎么从旧界面变到新界面，由 React 统一处理，这会明显降低复杂交互里的维护成本和出错概率。

3. React 和直接操作 DOM 的主要差异是什么？
答：直接操作 DOM 更像命令式编程，你需要亲自维护“当前页面已经改到哪一步了”；React 则更像声明最终状态，你告诉它现在的数据是什么，它再根据新旧状态差异决定该更新哪些节点。前者在小功能里很直接，但页面一复杂，代码里就会充满查节点、改节点、同步状态的细节；后者虽然引入了组件、状态、渲染这些心智成本，但换来的是更强的一致性和可维护性，所以 React 的重点不是完全替代 DOM，而是把 DOM 操作收敛到一个统一、可调度、可推导的更新体系里。

4. 什么是组件化？组件化真正解决了什么问题？
答：组件化就是把页面按职责拆成一块块有边界的 UI 单元，每个单元都可以拥有自己的输入、内部逻辑和输出结果，然后再像搭积木一样组合成完整页面。它真正解决的是复杂度管理问题，包括代码复用、多人协作、需求迭代时的影响范围控制，以及问题定位时“这一块是谁负责、状态归谁管”的边界清晰度；但组件化不是切得越碎越高级，如果拆分后一个小改动要层层传参、上下文混乱、抽象名字也看不懂，说明边界反而被拆坏了。

5. React 应用的渲染流程大致是什么？
答：React 的渲染流程可以理解成“先算，再提交，再交给浏览器画出来”：当 state、props 或 context 变化后，React 先创建这次更新任务，然后在 render 阶段重新执行相关组件，得到一棵新的 React 元素树，并在协调过程中判断哪些节点可以复用、哪些要新增或删除；等这些计算完成后，才会进入 commit 阶段，把真实 DOM 变更、ref 更新和副作用执行一次性提交，最后浏览器再进行样式计算、布局、绘制和合成。理解这条链路的关键是分清“组件函数执行了”不等于“DOM 已经变了”，更不等于“浏览器已经重绘完成了”。

6. React 为什么要引入 Virtual DOM？
答：Virtual DOM 的核心价值不是神话般地“永远比真实 DOM 快”，而是先用 JavaScript 对象把界面抽象成一棵可计算、可比较、可跨平台描述的树。这样 React 就能先在内存里分析这次更新到底影响了哪些节点，再按自己的策略去提交真实 DOM 变更；这不仅让更新流程更统一，也让 React 能把同一套组件心智扩展到 Web、React Native 等不同平台，所以它更像“UI 抽象层 + 更新调度层”的基础设施，而不只是一个性能技巧。

7. Virtual DOM 一定更快吗？
答：不一定，因为 Virtual DOM 自己也有创建对象、比较新旧树和调度提交的成本，在极简单、目标明确的场景下，手写精确 DOM 操作甚至可能更快。它真正占优的地方，是复杂页面里“如何稳定维护大量状态驱动的更新”，也就是用统一的渲染心智换来更好的工程可维护性、批量更新能力和跨场景一致性，所以回答这题时最好强调：Virtual DOM 的价值首先是开发模型和更新策略，其次才是部分场景下的性能收益。

8. React 的优势和成本分别是什么？
答：React 的优势在于把组件化、状态驱动、工程化生态和可扩展架构结合得很成熟，特别适合复杂交互、多人协作和长期迭代项目；它的成本则是学习曲线更陡，你不仅要理解组件和 Hook，还要理解渲染时机、状态边界、性能优化、请求和副作用管理这些工程问题。换句话说，React 很适合把前端当成一套系统来做，但如果页面很轻、团队经验不足，或者业务只是简单展示，它带来的抽象和配置成本也可能显得偏重，所以技术选型还是要看业务复杂度和团队掌握程度。

## 二、JSX 与渲染机制

9. JSX 的本质是什么？
答：JSX 本质上是一种语法糖，它看起来像把 HTML 写进了 JavaScript，但编译之后其实会变成 `React.createElement` 或新的 JSX runtime 调用，最终产物仍然是普通 JavaScript 对象。它的意义在于把“界面结构”和“状态表达式”放进同一套语言里，让组件更容易写成声明式视图，所以 JSX 不是模板引擎，也不是浏览器直接认识的语法，而是开发时更友好的 UI 描述方式。

10. JSX 为什么不能直接写多个根节点？
答：因为组件返回值最终要形成一份明确的节点结构，React 需要知道这次渲染返回的“根”到底是什么，早期最直观的做法就是要求返回单个根节点。现在虽然可以返回多个并列节点，但通常仍要借助 `Fragment` 或数组把它们包装成一个统一结果，这样既满足 React 对树结构的组织要求，又不会像额外包一层 `div` 那样平白增加无语义 DOM。

11. 条件渲染和列表渲染有哪些常见陷阱？
答：常见陷阱包括把 `0 && <div />` 这种表达式误以为不会渲染、实际却会把 `0` 本身渲染出来，列表里使用不稳定的 `key` 导致节点被错误复用，以及条件切换时以为只是隐藏组件，实际上组件已经被卸载重建，从而让输入框内容或本地状态串位。它们背后的共同点是 React 并不是按“你脑海里的业务语义”处理节点，而是按表达式结果、节点位置和 `key` 身份来决定是否渲染、是否复用，所以这类题最重要的是理解 React 眼里的“同一个组件”到底靠什么判断。

这里很多人最容易困惑的是：为什么 `0 && <div />` 会渲染出 `0`？

先看 JavaScript 本身：

```jsx
0 && <div>hello</div>
```

这不是 React 特有语法，本质上先按 JavaScript 的 `&&` 运算规则求值。

`&&` 的规则是：

- 如果左边是假值，就直接返回左边
- 如果左边是真值，才返回右边

而 `0` 在 JavaScript 里是一个假值，所以：

```js
0 && <div>hello</div> // 结果不是 false，而是 0
```

也就是说，这个表达式最终产出的结果其实是数字 `0`，不是“什么都没有”。

然后再看 React：

- React 会忽略一部分“空节点”，比如 `false`、`null`、`undefined`
- 但像字符串和数字这类普通可显示值，React 会把它们当作文本节点渲染出来

所以：

```jsx
{false} // 不渲染
{null} // 不渲染
{undefined} // 不渲染
{0} // 会渲染成文本 0
{''} // 会渲染为空字符串，视觉上通常像没东西
```

这就是为什么下面这句会出问题：

```jsx
{list.length && <List />}
```

当 `list.length === 0` 时：

- `list.length && <List />` 的结果就是 `0`
- React 收到的是数字 `0`
- 最终页面上就可能直接出现一个 `0`

很多人误以为：

- “左边是假，所以 React 不渲染”

但更准确的说法是：

- JavaScript 先算出结果
- React 再决定如何处理这个结果
- 如果结果是 `0`，React 会把它当普通文本渲染

正确写法通常有几种：

```jsx
{list.length > 0 && <List />}
{!!list.length && <List />}
{Boolean(list.length) && <List />}
```

这些写法的本质是先把左侧结果显式变成布尔值。

例如：

```js
Boolean(0) // false
```

这样表达式最终返回的就是 `false`，而不是 `0`，React 才会把它当作空节点忽略掉。

面试里可以这样总结：

- `0 && <div />` 会渲染出 `0`，不是因为 React 把假值渲染出来了。
- 而是因为 JavaScript 的 `&&` 表达式结果本身就是 `0`。
- React 会忽略 `false/null/undefined`，但不会忽略数字 `0`。

12. `key` 为什么重要？
答：`key` 的本质作用不是“消除警告”，而是告诉 React 同一层级里每个节点的身份是谁，这直接决定了更新时是复用旧节点、移动旧节点，还是卸载旧节点再新建一个。因为 React 会把组件本地状态、DOM 复用关系都挂在这个身份判断上，所以 `key` 一旦不稳定，就很容易出现列表顺序变了但输入框内容没跟着项走、动画错位、局部状态串位等问题，因此它是理解列表渲染和状态保留的关键概念。

13. 为什么不能随便使用数组索引作为 `key`？
答：因为索引只代表“当前排在第几个”，不代表“这个数据项本身是谁”，一旦列表发生插入、删除、排序，原来第 2 项今天可能变成第 3 项，React 却会误以为它们还是同一个节点，于是把旧状态错误地复用到新位置上。只有在列表完全静态、不会增删改也不会重排时，索引 `key` 才勉强可接受；只要列表是动态的，就应该优先使用业务上的稳定唯一标识。

14. React 更新列表时如何利用 `key` 提升复用效率？
答：React 在比较同级列表时，会先拿新旧节点的 `key` 去对齐身份，能对上的就优先复用原来的 Fiber、DOM 和组件状态，只对真正变化的项做新增、删除或移动处理。这样做的意义不是让算法“绝对最优”，而是用一个稳定身份把差异比较控制在工程上足够高效的范围内，所以频繁增删改排序的列表、表格和复杂表单行，尤其依赖稳定 `key`；同时也要记住，`key` 只在同级节点比较里有意义，并不是全局唯一标识系统。

15. 什么是协调过程？
答：协调可以理解成 React 在每次更新时做的一轮“新旧节点对账”，它会比较新的元素树和旧的 Fiber 树，判断哪些节点类型没变可以继续复用，哪些节点需要更新属性，哪些应该删除，哪些又是新增出来的。它的目标不是做理论上最完美的全量树比较，而是在现实业务里用一套足够快、足够稳定的规则，把界面从旧状态推进到新状态。

16. React Diff 的基本策略是什么？
答：React Diff 的核心策略是同层比较而不是跨层全量搜索：如果两个节点类型不同，通常直接视为整棵子树替换；如果类型相同，就继续比较属性和子节点；到了列表场景，再结合 `key` 去判断谁能复用、谁是新增或移动。这样设计不是因为 React 不知道更复杂的算法，而是因为完整树编辑距离成本太高，而前端 UI 更新大多数都发生在同层局部变化里，所以这是一个非常典型的工程取舍。

17. 为什么 React Diff 是同层比较？
答：因为如果允许跨层自由比较，算法复杂度会迅速上升，React 每次更新都可能陷入非常重的全树搜索，而真实页面里最常见的变化其实是同一层级上的增删改。React 选择同层比较，本质上是在“理论最优”与“工程可用”之间取平衡，用更低的计算成本换来足够稳定的更新结果；代价是某些跨层移动的节点不会被聪明地复用，很多时候会被当成删除旧节点、再创建新节点处理。

18. 如果组件频繁重新渲染，你怎么定位原因？
答：我会先用 React DevTools Profiler 找到到底是谁在频繁 render，再顺着看触发源是本组件 state、父组件传参变化、Context value 变化，还是某些对象和函数引用每次都重新创建。真正的排查重点不是看到“渲染很多次”就马上上 `memo`，而是先分清这次渲染是业务必需更新，还是状态边界设计不合理、不稳定引用、Context 粒度过粗等原因造成的无效渲染，这样优化才不会只停留在表面。

## 三、Hooks

19. `useState` 的更新机制是什么？
答：`useState` 的更新不是把某个变量原地改掉，而是生成一条更新记录挂到当前组件对应的 Fiber 上，等 React 下一次调度到这个组件时，再按更新队列顺序计算出新的 state，然后触发新的渲染。也就是说，函数组件里你拿到的 state 更像“这一次渲染的快照”，`setState` 做的是“请求下一次渲染使用新值”，这也是为什么它看起来不像普通变量赋值那样立刻生效。

20. 为什么 state 更新看起来是异步的？
答：因为 React 不会在你调用 `setState` 的那一行就立刻重跑组件并替换当前闭包里的值，而是先把更新收集起来，等合适的时机统一调度、批处理和重新渲染，所以同一段同步代码里你读到的往往还是当前这次渲染对应的旧快照。更准确地说，它不是语言层面的异步，而是 React 的渲染模型决定了 state 更新具有“排队后在下一轮渲染生效”的特征。

21. React 中的批量更新是什么？
答：批量更新就是 React 不急着对每一个 `setState` 都立刻单独渲染一次，而是把同一轮里相关的更新先收集起来，再统一计算最终状态和最终 UI，这样可以减少重复 render 和重复 commit。它的意义不只是性能更好，还能避免一连串状态更新中间暴露出很多过渡态，所以你在一个点击事件里连续调用多次 `setState`，React 往往只会做一次更完整的结果提交；但要注意，批量更新减少的是渲染次数，不代表当前同步代码里能立刻读到更新后的 state。

22. `useEffect` 和 `useLayoutEffect` 的区别是什么？
答：两者都会在 commit 之后执行，但时机不同：`useEffect` 一般在浏览器把页面绘制出来之后再执行，更适合请求、订阅、日志这类不要求阻塞首屏的副作用；`useLayoutEffect` 则是在 DOM 已更新、但浏览器真正绘制前同步执行，适合读布局信息、立刻修正样式或滚动位置这类不能闪一下再改的场景。正因为 `useLayoutEffect` 会卡住绘制，它不是更高级的版本，而是一个更重、更严格的工具，绝大多数副作用优先仍然应该用 `useEffect`。

23. `useEffect` 的清理函数在什么时机执行？
答：清理函数会在两个时机运行：一是组件卸载时，二是下一次同一个 effect 因依赖变化要重新执行之前。这样设计的目的，是先把上一轮副作用留下的订阅、定时器、事件监听、旧请求等资源收掉，再建立新的一轮副作用，避免重复注册、资源泄漏和旧逻辑继续生效。所以理解清理函数时不要只记“卸载时执行”，更要记住它其实也是“更新前的旧副作用退出机制”。

24. 为什么 `useEffect` 依赖数组容易出错？
答：因为依赖数组比的不是“业务上感觉差不多”，而是每一项在本次渲染和上次渲染之间是否还是同一个引用，同时 effect 回调捕获的又是当次渲染的闭包快照，所以一旦漏了依赖，回调里就可能一直读旧值；反过来如果把每次都会新建的对象、函数直接放进去，又会导致 effect 频繁重跑。它难的地方不在语法，而在于你必须同时理解“引用稳定性”和“闭包快照”这两件事。

25. 什么是闭包陷阱？在 Hooks 里如何出现？
答：闭包陷阱本质上是“回调函数记住了旧渲染里的变量值”，所以等定时器、事件监听、异步请求回调真正执行时，它拿到的可能不是最新 state，而是某一次旧 render 形成的快照。Hooks 场景里它特别常见，因为函数组件每次渲染都会重新生成一套新的变量环境，如果你没有正确补依赖、没有用函数式更新，或者没有用 `useRef` 保存最新值，就很容易出现“页面已经变了，但回调里读到的还是老数据”的问题。

26. `useMemo` 和 `useCallback` 分别解决什么问题？
答：`useMemo` 主要缓存“结果”，适合避免昂贵计算在依赖没变时重复执行；`useCallback` 主要缓存“函数引用”，适合在你需要把函数传给子组件、依赖数组或订阅系统时保持引用稳定。它们解决的都不是功能正确性问题，而是“重复计算”和“引用不稳定导致的额外工作”问题，所以常常要和 `memo`、依赖数组、Context value 稳定性一起配合看，单独套上去不一定有收益。

27. 什么情况下不应该滥用 `useMemo`？
答：如果计算本身很轻、依赖经常变化、组件也没有明显性能压力，那就不应该滥用 `useMemo`，因为缓存本身也有比较依赖、占用内存和增加阅读难度的成本。很多项目里真正的问题不是“少了一个 `useMemo`”，而是状态边界太大、无效渲染太多，所以优化前应该先确认瓶颈是不是这段计算，而不是把 `useMemo` 当成机械式优化模板。

28. `useRef` 的核心用途是什么？
答：`useRef` 有两个核心用途：一是拿到某个 DOM 或组件实例引用，便于聚焦、测量、滚动控制等命令式操作；二是保存一个跨渲染周期持续存在、但修改它不会触发重新渲染的可变容器。第二种用途特别重要，因为它能用来存定时器 id、上一次值、最新回调、请求实例等“需要记住但不需要驱动 UI”的数据，所以 `useRef` 不只是“拿 DOM 的工具”，更是函数组件里管理非响应式可变值的基础能力。

29. 如何用 Hooks 封装可复用逻辑？
答：用 Hooks 封装可复用逻辑，本质上是把一段经常重复出现的状态管理、副作用流程和事件处理规则，提炼成一个有明确输入输出的自定义 Hook，让页面只关心“怎么用”，不用每次都重写“怎么实现”。高质量的 Hook 不是单纯把代码挪个文件，而是抽象出稳定的能力边界，比如分页请求、弹窗控制、权限判断、表单联动，并把返回值设计成容易理解和组合的接口。

30. 自定义 Hook 的设计原则是什么？
答：我会强调职责单一、命名贴近业务能力、输入输出稳定、内部副作用可控，不要把某个页面独有的流程硬包装成“通用 Hook”。Hook 的价值在于复用状态逻辑和副作用链路，而不是为了追求文件拆分把复杂度藏起来，所以如果一个 Hook 同时处理请求、路由跳转、埋点、弹窗和表单，通常说明边界已经失控，后面既难测试也难复用。

### Hooks 深入补充

#### Hook 到底是什么？

如果只用一句话给 Hook 下定义，我会这样说：**Hook 是 React 约定的一类函数接口，它让函数组件能够复用状态逻辑，并接入 React 运行时能力。**

这个定义里故意包含了两层意思，因为面试里很多人会把“Hook”这个词只理解成 `useState`、`useEffect` 这些内置 API，但实际上一说到 Hook，通常同时包含两类东西：

1. **React 内置 Hook**
   例如 `useState`、`useEffect`、`useRef`、`useContext`。
   它们的作用是让函数组件接入 React 内部能力，比如状态、副作用、上下文、订阅、ref 等。

2. **自定义 Hook**
   例如 `usePagination`、`useRequest`、`useModal`。
   它们通常不是在创造新的 React 底层能力，而是在**组合和复用已有 Hook 逻辑**，把一段重复出现的状态管理和副作用流程封装成一个更易复用的函数。

所以如果面试官问“Hook 是什么”，一个更贴切、覆盖范围更完整的回答应该是：

**Hook 是 React 体系里以 `use` 开头的一类函数约定。对内置 Hook 来说，它是函数组件接入 React 运行时能力的接口；对自定义 Hook 来说，它是对状态逻辑和副作用逻辑的组合封装方式。**

这也正是你刚才指出的问题：如果直接说“Hook 的本质就是按顺序访问 Fiber 上的状态槽位”，这句话更适合描述 **React 内置 Hook 的底层运行机制**，但并不能完整概括 **自定义 Hook**。

因为自定义 Hook 的本质并不是“自己再去实现一套新的 Hook 运行时”，而是：

- 它首先当然还是一个普通 JavaScript 函数
- 这个函数里可以使用其他 Hook
- 它把一段状态逻辑、闭包逻辑、副作用流程和返回接口组合起来
- 然后把这段能力以一个统一函数形式暴露给外部调用

所以如果你单独问“自定义 Hook 的本质是什么”，更贴切的说法反而是：

**自定义 Hook 本质上就是一个遵守 Hook 调用规则的普通函数，它把一段可复用的状态逻辑和副作用逻辑封装起来，对外暴露统一接口。**

也就是说，要把这两层区分开：

- **Hook 作为 React 概念的总定义**：
  是一类让函数组件复用逻辑并接入 React 能力的函数约定

- **React 内置 Hook 的底层本质**：
  是 React 在函数组件渲染时，按固定调用顺序把内部状态节点和副作用节点暴露出来访问的机制

- **自定义 Hook 的本质**：
  是一个“组合已有 Hook 的普通函数封装”，它本身不创造新的底层机制，而是在复用机制

所以真正更完整的一句话总结应该是：

**Hook 从概念上讲，是 React 中一类可复用逻辑与接入运行时能力的函数约定；从底层实现上讲，内置 Hook 依赖 React 按调用顺序把 Fiber 上的 Hook 状态节点暴露出来，而自定义 Hook 则是在这个机制之上做逻辑封装。**

#### 什么叫“按顺序访问内部状态槽位”？

这是理解 Hook 最关键的一步。初学者经常会以为 React 是靠变量名识别 Hook 的，比如它知道第一个 `useState` 是 `count`，第二个 `useState` 是 `name`。实际上不是。React 根本不认识你写的变量名，它只认识“这次渲染里，第几个 Hook 被调用了”。

例如：

```tsx
function Demo() {
  const [count, setCount] = useState(0);   // 第 1 个 Hook
  const [name, setName] = useState('Tom'); // 第 2 个 Hook
  const ref = useRef(null);                // 第 3 个 Hook
}
```

在教学理解里，你可以先把 React 内部想成一排“格子”：

- 第 1 个 Hook 对应第 1 个槽位
- 第 2 个 Hook 对应第 2 个槽位
- 第 3 个 Hook 对应第 3 个槽位

第一次渲染时，React 遇到第一个 Hook，就去拿第一个槽位；遇到第二个 Hook，就去拿第二个槽位；遇到第三个 Hook，就去拿第三个槽位。下一次重新渲染时，它还必须按照完全相同的顺序再走一遍，这样它才知道：

- 第一个 Hook 的旧状态应该从哪里取
- 第二个 Hook 的旧状态应该从哪里取
- 第三个 Hook 的旧 ref / effect 应该从哪里取

所以“按顺序访问内部状态槽位”真正的意思是：**React 不是靠名字识别 Hook，而是靠 Hook 调用顺序，把每次调用映射到一格固定的内部存储位置上。**

这也是为什么 Hook 不能写在条件判断、循环或嵌套函数里。因为一旦调用顺序变了，原来“第 2 个 Hook”今天可能突然变成“第 1 个 Hook”，那 React 就会把旧状态错配到错误的位置上，整个状态系统都会乱掉。

#### “槽位”到底是什么？它是怎么产生的？

“槽位”不是 React 官方文档里一个特别正式的 API 名字，它更像一个帮助理解底层的教学说法。你可以把它理解成：**React 为每一个 Hook 调用预留的一块内部状态存储位置。**

它不是你手动创建的，而是在组件渲染过程中自然产生的。React 每执行到一个 Hook，就会向前推进一次内部指针，把当前这个 Hook 和当前这一个位置绑定起来。第一次渲染会建立这些位置，后续渲染会再次按顺序访问这些位置。

所以槽位不是某个额外的概念，它本质上就是：**某次 Hook 调用在 React 内部对应到的那一个状态节点位置。**

#### 为什么 Hook 会和 Fiber 绑定？

这是面试里很容易继续追问的一层。只知道“Hook 依赖调用顺序”还不够，还要知道它为什么会落在 Fiber 上。

原因很直接：**Fiber 是 React 当前这一次渲染工作的载体，它上面已经挂着这个组件最核心的运行时信息。** React 在 render 阶段处理某个函数组件时，真正手里拿着的不是“组件源码文本”，而是这个组件对应的 Fiber 节点。这个 Fiber 节点上本来就会存：

- 当前组件类型
- 当前 props
- 当前 state
- 更新队列
- 父子兄弟指针
- 副作用标记

那么 Hook 的内部状态当然也就最自然地挂在这个 Fiber 上。因为 Hook 不是脱离组件实例独立存在的，它一定是“属于某个具体组件这次渲染过程”的。

你可以把 Fiber 理解成“当前组件这次工作的档案袋”。React 一边执行函数组件，一边需要知道：

- 这是哪个组件的 Hook
- 这是这个组件的第几个 Hook
- 上一次渲染这个位置上存的 state / effect / ref 是什么
- 这次更新要不要把新的 Hook 节点挂回去

这些信息都必须附着在这个组件自己的 Fiber 上，React 才能在 render 阶段构建 `workInProgress` 树时，把新的 Hook 链和旧的 Hook 链正确对齐。

所以更本质地说：**Hook 会和 Fiber 绑定，是因为 Hook 的生命周期、状态和更新队列都必须附着在“某个组件当前这次渲染的工作单元”上，而这个工作单元就是 Fiber。**

如果不和 Fiber 绑定，会遇到两个大问题：

1. React 不知道这些 Hook 状态属于哪一个组件
2. React 不知道当前这次 render 应该从哪里拿旧 Hook 链来对齐和复用

所以 Hook 不是“单独飘在 React 全局里的一堆状态”，而是**挂在 Fiber 节点上的一串 Hook 状态节点**。

#### 数组理解和链表理解到底差在哪？

这是另一个特别容易让人混淆的点。很多入门文章会说“你可以把 Hook 想成数组”，而更接近 React 源码的解释又会说“Hook 是链表”。这两种说法并不是互相矛盾，而是站在不同抽象层次上。

##### 1. 数组理解：更适合入门，帮助你理解“顺序”

教学里常会这样讲：

```js
hooks[0]
hooks[1]
hooks[2]
```

这种讲法的优点是非常直观。它能快速帮你建立下面两个认知：

- Hook 是按调用顺序取的
- 每个 Hook 都会对应一个固定位置

所以数组模型最适合解释：

- 为什么 Hook 不能写在条件里
- 为什么调用顺序一变就会错位
- 为什么 React 要按顺序访问内部状态

但它有一个明显局限：**它太像一个静态容器了，不够贴近 Fiber 渲染过程中“边遍历边挂载 Hook 节点”的真实工作方式。**

##### 2. 链表理解：更接近源码，帮助你理解“挂在 Fiber 上的一串 Hook 节点”

在 React 更贴近真实的实现里，Hook 不是存成一个简单数组，而是挂在 Fiber 上的一串节点。每个 Hook 节点大致会包含：

- 当前 Hook 的状态值
- 更新队列
- effect 信息
- 指向下一个 Hook 节点的指针

你可以把它理解成：

```txt
Fiber.memoizedState
  -> Hook1
  -> Hook2
  -> Hook3
  -> null
```

也就是说，Fiber 上会有一个入口指针，指向这串 Hook 链表的第一个节点。React 执行函数组件时，每调用一个 Hook，就沿着这串链往下走一个节点，拿到当前对应的 Hook 状态。

这种链表模型更适合解释：

- 为什么 Hook 状态会和 Fiber 绑定
- React 在 `current tree` 和 `workInProgress tree` 之间如何复用和构建 Hook 节点
- 为什么每个 Hook 不只是“一个值”，而是一整块带状态和更新队列的节点


#### 一句话怎么把这些串起来？

如果面试官一路从 Hook 问到 Fiber，你可以这样收束：

**Hook 的本质不是几个特殊 API，而是 React 在函数组件渲染时，按固定调用顺序把 Fiber 上的一串 Hook 状态节点暴露出来访问的机制。初学时可以先把它理解成“按顺序访问槽位”，更贴近源码时再把这些槽位理解成挂在 Fiber 上的 Hook 链表节点。**

## 四、组件通信与状态管理

31. React 常见组件通信方式有哪些？
答：结论：常见通信方式有 props、回调、Context、状态库、事件总线，但我的选择原则一直是先近后远、先简单后复杂。原理：React 默认就是单向数据流，所以能在父子层解决的问题，没必要过早放大全局。场景：父子通信优先 props 和回调，跨层共享再考虑 Context，真正到跨页面共享、联动复杂、调试要求高时再上状态库。延伸：一句话就是，能局部解决就别全局化。

32. 父子通信、兄弟通信、跨层通信分别怎么做？
答：结论：父子通信靠 props 和回调，兄弟通信通常靠状态提升，跨层共享优先 Context，再复杂点再上状态管理。原理：React 数据流默认是单向向下的，所以谁拥有状态，谁就更适合做通信源头。场景：局部联动优先近距离传递，跨页面或复杂共享再考虑全局方案。边界：不要为了少传几层 props 就一上来把所有状态都塞进全局。

33. `Context` 适合解决什么问题？
答：结论：适合主题、用户信息、权限这类跨层共享但更新频率没那么夸张的数据。原理/场景：它更适合主题、语言、权限、用户信息这类横切数据，不适合高频变化的大状态。

34. `Context` 为什么可能引发性能问题？
答：结论：因为 Provider 的 value 变了，相关消费者都会重新渲染，所以 value 设计要很谨慎。原理/场景：Provider 的 `value` 一变，相关消费者就会重新渲染，所以 value 稳定性和拆分粒度很关键。
延伸：常见优化方式是拆 Provider、稳定 `value`、减少无关消费者。

35. 什么时候只用 `useState` 就够了？
答：结论：状态只在局部页面里流转、依赖关系不复杂时，`useState` 往往就够了，没必要上全局方案。原理：局部状态离使用方越近，心智负担越低，也更容易维护。场景：弹窗开关、表单输入、局部筛选条件、分页状态这些都很适合留在组件内部。边界：一旦状态跨页面共享、需要持久化或联动很多模块，再考虑提升到更高层。

36. 什么时候要引入 Zustand、Redux、MobX 这类状态管理方案？
答：结论：当状态跨页面共享、联动复杂、调试诉求强，或者需要统一数据流时，再引入 Zustand、Redux、MobX 这类方案才更有价值。原理：这类工具解决的不是“state 不够用”，而是共享范围、更新链路、调试追踪和团队协作都开始变复杂。场景：比如全局用户态、权限、跨页面筛选条件、复杂业务流编排，就比单纯 `useState` 更适合交给状态管理层。

### 状态管理补充：为什么 Redux 常配合 Provider，而 Zustand 通常不需要？

这个问题非常适合和上面那题连在一起回答，因为它不是在问“哪个库更高级”，而是在问：**这两个库把 store 暴露给 React 组件树的方式为什么不一样。**

先说一个很容易被忽略的关键点：**Redux 核心本身并不依赖 React，也不强制必须有 Provider。** 纯 Redux 的 `createStore` 只关心几件事：

- 保存 state
- 提供 `getState`
- 提供 `dispatch`
- 提供 `subscribe`

也就是说，Redux store 本质上只是一个独立的外部状态容器，它本身并不知道 React 组件树是什么，更不知道某个组件该怎么拿到 store。真正把 Redux 接到 React 组件树上的，通常是 `react-redux` 这一层。

那为什么 `react-redux` 常常要用 `Provider`？

因为 React 组件是树状结构，组件想拿到外部 store，有两种最常见办法：

1. 一层层 props 往下传
2. 借助 Context 在组件树里向下分发

显然，如果整个应用很多组件都可能要访问 Redux store，一层层 props 传会非常麻烦，所以 `react-redux` 选择了第二种方案：**用 `Provider` 把 store 放进 Context，然后让任意后代组件通过 `useSelector`、`useDispatch` 等方式取出来。**

所以这里要特别注意一句话：

**Redux 需要的不是 Provider，React 组件树想方便拿到 Redux store 时，通常才需要 Provider。**

也就是说：

- Redux 核心：不依赖 Provider
- React 使用 Redux：通常通过 `react-redux` 的 Provider 注入 store

再看 Zustand 为什么常常不需要 Provider。

因为 Zustand 最常见的用法不是“先把 store 放进 Context，再从 Context 里取”，而是直接把 store 定义在模块作用域里，例如：

```ts
const useCounterStore = create(...)
```

然后任何组件只要 import 这个 `useCounterStore`，就能直接订阅这一个外部 store。也就是说，它的 store 通常天然就是一个“模块级单例入口”，组件不需要先从组件树上级拿到 store 引用，而是直接 import 就能访问。

这就是两者最根本的区别：

- **Redux + react-redux**
  常见模式是：store 先创建出来，再通过 Provider 放进组件树上下文中，组件运行时再去读取这个上下文。

- **Zustand**
  常见模式是：store 本身就作为模块导出，组件直接 import 并订阅，不需要额外再经过 Context 分发。

所以从 React 视角看：

- Redux 更像“我先把 store 注入到树里，再让组件去消费”
- Zustand 更像“我把 store 定义在树外，组件直接连接这个外部 store”

那为什么 Zustand 这样就能不需要 Provider？

因为它不靠“组件树里的上下文传递”解决 store 获取问题，而是靠“模块导入 + 外部订阅”解决。只要所有组件 import 到的是同一个 store 实例，它们天然就在共享同一份状态。

当然，这里也不是说 Zustand 永远不需要 Provider。真实项目里如果你想做这些事情：

- 为不同子树注入不同 store 实例
- 做多实例隔离
- 做测试态 / 沙箱态注入
- 做请求级 SSR store 隔离

那 Zustand 也完全可以配合 Context / Provider 模式来用。只是它的“默认主路径”不是这么设计的，所以日常使用时你会感觉它“天然不需要 Provider”。

最后用一句最适合面试收束的话总结：

**Redux 之所以常配合 Provider，不是因为 Redux 核心离不开 Provider，而是因为 React 组件树需要一种统一方式拿到 store，而 `react-redux` 选择了 Context 注入；Zustand 则通常把 store 设计成模块级外部对象，组件直接 import 并订阅，所以默认不需要再通过 Provider 做一层分发。**

37. 全局状态和局部状态如何划分边界？
答：结论：能放局部就别放全局，只有跨页面共享、长期存在、多人依赖的状态我才会提到全局。原理/场景：我通常坚持“局部优先”，只有跨页面共享、生命周期长、多人依赖的状态才会提到全局。
延伸：能放局部就别放全局，是很重要的工程取舍。

38. 服务端状态和客户端状态为什么应该分开考虑？
答：结论：因为服务端状态核心是获取、缓存、失效和同步，和表单输入、弹窗开关这种本地状态不是一类问题。原理/场景：服务端状态核心是获取、缓存、失效、重试和同步，和 UI 开关、表单输入不是一类问题。

39. 如何设计一个复杂页面的状态结构？
答：结论：我会先按领域拆，再区分源数据、派生数据和 UI 状态，避免所有东西都堆在一个组件里。原理/场景：复杂页面最好分成源数据、派生数据和 UI 状态三层，避免所有字段都平铺在一个组件里。

40. 如何避免状态分散和状态冗余？
答：结论：关键是单一数据源，能推导出来的就别重复存，状态归属也要尽量贴近使用方。原理/场景：能推导出来的就不要重复存，单一数据源是避免状态打架的关键。

## 五、生命周期与渲染时机

41. 函数组件时代如何理解生命周期？
答：在函数组件时代，生命周期最好不要只靠类组件那套“挂载、更新、卸载”名词硬背，而要按 React 真实工作阶段理解成 render、commit、effect 建立和 effect 清理。因为函数组件本质上就是一次次重新执行的渲染函数，真正有阶段差异的是“什么时候只是计算 UI”“什么时候真实提交到 DOM”“什么时候启动或销毁副作用”，这样理解之后，`useEffect`、`useLayoutEffect`、Strict Mode 和并发渲染就都能自然串到一条线上。

42. 挂载、更新、卸载分别会触发哪些常见逻辑？
答：挂载时常见逻辑是初始化状态、发首屏请求、建立订阅和事件监听；更新时常见逻辑是根据新依赖重建副作用、同步外部系统或修正布局；卸载时最重要的是清理定时器、监听器、连接和未完成请求。真正需要强调的是，这些逻辑不是靠一个固定生命周期函数自动分类完成的，而是由你的 effect 依赖决定什么时候建立、什么时候清理，所以 effect 粒度设计得清楚，生命周期边界才会清楚。

43. 为什么渲染阶段不能做副作用？
答：因为渲染阶段的职责只是根据输入计算这次 UI 应该长什么样，它可能被重复执行、提前打断，甚至整次结果最后被丢弃，如果你在这个阶段就发请求、改 DOM、写全局变量或注册监听，就会出现重复执行和状态不一致的问题。React 希望渲染尽量保持纯函数心智，也就是“相同输入得到相同输出、没有额外副作用”，真正和外部世界打交道的逻辑应该放到 commit 之后。

44. 什么是 commit 阶段？
答：commit 阶段就是 React 把 render 阶段已经算好的结果真正落到真实世界的阶段，包括更新 DOM、设置 ref、触发生命周期相关逻辑以及安排 effect 执行。它和 render 的关键区别在于，render 只是“算草稿”，commit 才是“正式发布”，所以只有进入 commit 之后，用户看到的页面、DOM 状态和副作用系统才会真正变化；也正因为 commit 会改真实界面，它一旦开始就不能像 render 那样随意中断。

45. Strict Mode 为什么会让某些逻辑执行两次？
答：Strict Mode 在开发环境里会故意让某些渲染和 effect 建立/清理过程多跑一遍，目的不是折磨开发者，而是尽早把不纯渲染、遗漏清理、依赖不稳定和副作用写错时机这类问题放大出来。你可以把它理解成 React 在帮你模拟“如果渲染被重来、如果副作用被重新建立，这段代码还能不能保证安全”，所以它强调的是开发期诊断能力，而不是生产环境真实会重复执行两次。

46. React 这样设计是为了帮助发现什么问题？
答：它主要是在帮你发现几类并发和重渲染时代最危险的问题，比如渲染函数不纯、effect 没有正确清理、依赖数组写错、重复订阅、重复请求、以及依赖“这段逻辑只会执行一次”的脆弱写法。因为 React 越来越强调可中断、可重试、可恢复的渲染模型，开发环境就必须更严格一点，提前把这些上线后会难查的问题暴露出来。

47. 为什么组件渲染不等于 DOM 更新？
答：因为“组件渲染”在 React 里通常指的是函数组件执行、生成新的元素描述和参与协调的过程，这一步只是计算，不是立刻操作真实 DOM。只有等到 React 确认这次更新需要提交，并进入 commit 阶段后，真实 DOM 才会发生变化，所以你看到组件函数跑了一次，并不代表页面一定已经改了，也不代表浏览器一定已经重绘完成。

48. 如何判断一次更新是否真的引发了页面重绘？
答：我会把“React render 次数”“DOM 是否真的变化”“浏览器有没有做样式计算、布局、绘制或合成”分开看，通常结合 React Profiler 和浏览器 Performance 面板一起判断。因为组件函数执行很多次，不一定会产生 DOM 改动；就算 DOM 变了，也不一定一定触发布局和大面积重绘，所以真正的性能定位不能只盯 React，而要看浏览器时间线里有没有 Layout、Paint、Composite 等开销被放大。

## 六、性能优化

49. React 项目常见性能问题有哪些？
答：结论：React 项目常见性能问题通常有无效渲染、长列表、重计算、包体积过大、请求串行和状态设计不合理。原理：这些问题本质上分别落在渲染次数、单次渲染成本、资源加载成本和数据流设计上。场景：排查时我会先区分到底是首屏慢、交互卡，还是某个局部区域重渲染过多，再决定是做拆分、缓存、虚拟列表还是请求治理。延伸：如果能结合一次真实优化案例，回答会更有说服力。

50. 什么是无效渲染？
答：结论：无效渲染就是组件重新执行了，但最终 UI 没有实质变化，这类渲染通常就是优化重点。原理：状态提升过高、父组件引用不稳定、Context 粒度过粗都可能让子树被动重跑。场景：列表页、复杂表单、筛选区联动很容易出现这种问题。边界：不是所有重复渲染都值得优化，要看组件重量和用户感知。

51. `React.memo` 的作用是什么？
答：结论：它会在 props 浅比较没变时跳过函数组件重渲染，适合稳定 props 的纯展示组件。原理/场景：`React.memo` 适合 props 稳定的纯展示组件，不是所有组件都值得包一层。
延伸：`memo` 只有在“跳过渲染”收益大于“比较成本”时才值得用。

52. `memo` 为什么不是万能优化手段？
答：结论：因为它本身也有比较成本，如果 props 总在变或者组件很轻，收益可能很有限。原理/场景：比较本身也有成本，如果 props 总变或组件很轻，`memo` 可能没有收益。

53. `useMemo`、`useCallback`、`memo` 该如何配合使用？
答：结论：一般是父组件用 `useMemo`、`useCallback` 稳定传参，子组件再配合 `memo` 才比较有意义。原理/场景：常见套路是父组件用 `useMemo`/`useCallback` 稳定传参，子组件再配 `memo` 才更有意义。

54. 长列表优化有哪些常见方案？
答：结论：常见是虚拟列表、分页、懒加载、分片渲染，再加上减少单项渲染成本。原理/场景：长列表优化通常不只一招，常见是虚拟列表、分页、分片渲染和减少单项渲染成本一起做。
延伸：大表格、大列表几乎一定会被追问到虚拟列表。

55. 表单很多、交互复杂时如何减少卡顿？
答：结论：我会拆状态、减少全量联动、做局部更新，必要时配合防抖和低优先级更新。原理/场景：复杂表单场景里，拆状态和减少全量联动往往比无脑上缓存 Hook 更有效。

56. 路由级懒加载如何做？
答：结论：路由级懒加载一般配合 `React.lazy`、`Suspense` 或路由层动态 import，把首屏不需要的页面代码后置。原理：它本质上是代码分割，让不同路由按访问时机再加载各自 bundle。场景：后台系统、大型内容页、多模块平台都很适合。边界：懒加载能减小首屏包，但切路由时可能出现额外等待，所以要配合 loading 态和预加载策略。

57. 组件拆分为什么也属于性能优化？
答：结论：组件拆分也是性能优化，因为拆完之后状态影响范围更小，重渲染粒度更细。原理：当一个大组件被拆成多个稳定边界后，局部状态变化不会轻易拖着整棵子树一起重跑。场景：复杂表单、搜索页、表格页特别明显，拆分后既更易维护，也更容易配合 `memo`、局部状态和懒加载做进一步优化。

58. 如何量化一次 React 性能优化是否有效？
答：结论：我会拿优化前后首屏时间、交互耗时、渲染次数和用户感知指标做对比，不靠感觉。原理/场景：性能优化要回到数据，用渲染次数、交互耗时和用户感知指标验证，而不是靠感觉。

## 七、表单、路由与业务场景

59. 受控组件和非受控组件的区别是什么？
答：结论：受控组件值由 React state 驱动，非受控更多依赖 DOM 自身状态，前者更可控，后者更轻。原理/场景：受控更可控、更好做校验和联动；非受控更轻，但复杂表单里通常不如受控稳定。

60. 大型表单最常见的问题是什么？
答：结论：大型表单最常见的问题是状态过多、联动复杂、校验分散，最后同时拖垮性能和可维护性。原理：表单字段一多，任何一次输入都可能牵动全局状态和一串派生逻辑。场景：复杂审批单、配置面板、动态表单最容易出现这种问题。边界：优化重点通常是拆状态、分区域更新和收敛校验，不是无脑加 memo。

61. 表单校验逻辑应该放在哪一层？
答：结论：基础规则我会靠近表单模型，业务校验放服务或提交层，别把所有逻辑都塞组件里。原理/场景：基础规则靠近表单模型，跨字段和业务规则放提交层或服务层，避免组件里堆满判断。

62. React Router 的核心作用是什么？
答：结论：React Router 的核心作用就是把 URL 和页面状态对应起来，让单页应用也能按路由组织页面、导航和刷新恢复。原理：它把浏览器地址变化映射成组件树切换，而不是整页重新刷新。场景：页面导航、参数驱动筛选、刷新后恢复当前页，都依赖路由系统。边界：路由解决的是页面状态组织，不负责替你处理权限和数据缓存策略。

63. 前端路由和后端路由的区别是什么？
答：结论：前端路由主要负责视图切换和页面状态组织，后端路由主要负责资源定位、接口处理和服务端分发，职责根本不在一层。原理：前端路由更多运行在浏览器，后端路由更多运行在服务器。场景：单页应用里切页通常是前端路由，接口 `/api/users` 这种则是后端路由。边界：SSR 或同构场景下两者会协作，但职责仍然要分清。

64. 如何设计后台系统中的权限路由？
答：结论：我会用路由配置做单一入口，把菜单、路由、权限码尽量放同一份元数据里。原理/场景：更稳的做法是把菜单、路由和权限码维护在同一份元数据里，减少配置漂移。

65. 路由守卫在 React 中通常怎么实现？
答：结论：React 里做路由守卫，一般是在布局层、高阶路由组件或 loader 里做权限校验，再配合重定向。原理：核心不是阻止路由本身，而是在进入页面前统一判断登录态和权限。场景：后台系统的登录拦截、菜单权限、角色隔离都很常见。边界：守卫只是一层前端兜底，真正权限校验仍要在服务端完成。

66. 菜单、路由、权限如何保持一致？
答：结论：菜单、路由、权限要保持一致，核心就是配置同源，别三套数据各管各的。原理：只要入口不统一，后续必然出现菜单能见但进不去、路由可进但无按钮权限这类错位。场景：更稳的做法是用一份路由元数据同时驱动菜单、守卫和按钮权限。边界：前端配置统一能降低漂移，但真正权限结果仍要以后端为准。

67. 标签页缓存、多页签管理该怎么设计？
答：结论：我会先明确缓存粒度，再区分路由缓存、页面状态缓存和业务数据缓存，别混成一层。原理/场景：设计时最好区分路由缓存、页面状态缓存和业务数据缓存，别三者混成一层。

68. 复杂搜索页和表格页，组件边界如何划分？
答：结论：复杂搜索页和表格页，我通常按筛选区、表格区、操作区、弹窗区拆，页面层只负责编排和状态协调。原理：这样可以把高频变化区和重渲染区隔离开，降低联动成本。场景：中后台列表页、订单页、审批页都适合这种拆法。边界：页面层仍要保留整体状态编排权，不能把跨区域协作逻辑完全打散。

## 八、React 18 与并发特性

69. React 18 带来了哪些重要变化？
答：React 18 最值得关注的变化，是并发渲染能力真正对业务开放、自动批处理范围变大、`useTransition` 和 `useDeferredValue` 这类优先级工具更成熟，以及新的 root API 让应用能进入新调度模型；如果再往 SSR 方向补充，还可以提到流式服务端渲染和选择性 Hydration 的体验更完整。这里的重点不是背新 API 名字，而是理解 React 18 开始更强调“更新不是只有快和慢两种，而是有优先级、可打断、可恢复”的调度思路。

70. 什么是并发渲染？
答：并发渲染可以理解成 React 不再要求所有更新都必须“一口气同步算到底”，而是能把 render 工作拆成很多小段，在必要时暂停、让出主线程、先处理更紧急的更新，再回来继续或重算之前的低优先级工作。它解决的核心问题是交互响应性，也就是页面很重时，输入、点击这种高优先级操作不该被大列表刷新或复杂过滤计算拖死；但它不是浏览器里突然多开线程并行跑 React，本质上仍然是主线程上的更聪明调度。

71. 并发渲染和并行执行是一回事吗？
答：不是，并发更强调“任务能交替推进、能被打断、能按优先级插队”，并行则通常强调“真的同时在多个执行单元上一起跑”。React 的并发渲染主要发生在浏览器主线程上，所以它并没有凭空变成多线程框架，而是通过 Fiber 和优先级调度，把更新拆成更容易让权和重排顺序的工作单元。它改善的是交互跟手性，不是无中生有减少总计算量。

72. `useTransition` 适合什么场景？
答：`useTransition` 适合“同一次交互里既有必须立刻反馈的更新，也有比较重、可以慢一点跟上的更新”这类场景，比如输入框内容要马上显示，但筛选后的大列表刷新可以稍晚一点；又比如切 tab 时按钮高亮应立即响应，但新面板内容渲染很重。它的本质是把某些更新标记成非紧急，让 React 在高优先级交互和低优先级渲染之间做更合理的安排，所以它解决的是响应性问题，不是直接减少计算量。

73. `useDeferredValue` 适合什么场景？
答：`useDeferredValue` 适合某个值变化特别频繁，但依赖它的下游内容又比较重的场景，比如搜索关键词每敲一个字都变，但结果列表渲染很大，这时可以让输入框继续用实时值，而让列表使用一个“稍微延后一点”的值。你可以把它理解成给重内容一个缓冲，让用户输入先跟手，复杂列表稍后追上，而不是把高频输入和重渲染死绑在同一拍里。

74. 自动批处理带来了什么变化？
答：React 18 之前，很多人只在 React 事件里明显感受到批量更新；到了 React 18，更多异步场景里的多次状态更新也会被自动合并，这意味着你在 Promise、`setTimeout` 或原生事件回调里连续更新状态时，往往也不会再触发多次重复渲染。它带来的核心变化是更新行为更统一、更贴近“以结果为导向”的调度模型，但如果某些场景确实需要立刻同步看到最新 DOM，仍然可以用 `flushSync` 这种更强制的手段。

75. 这些新特性在业务项目里真的常用吗？面试该如何回答？
答：这些能力在真实项目里不一定天天都写到，但它们背后的调度思路非常重要，所以面试时最好的回答不是假装自己到处都用了，而是坦诚说明哪些场景真正能受益，比如高频输入联动重列表、复杂筛选、SSR 流式输出和 Hydration 优化；如果页面本身很轻，就没必要为了“用了新特性”而强行引入复杂度。这样回答既体现理解深度，也更符合真实工程判断。

## 九、工程与架构

76. 一个中大型 React 项目通常如何组织目录？
答：结论：我一般会按页面、组件、hooks、services、stores、utils、types 这些层次拆，但不会机械地按文件类型平铺。原理：目录结构本质上是在表达分层和边界，既要看业务聚合关系，也要看复用边界和依赖方向。场景：页面层负责编排，组件层负责 UI，hooks 复用视图逻辑，services 处理接口和数据转换，stores 管共享状态，这样项目越大越不容易失控。延伸：目录题真正加分点不是报目录名，而是说清为什么这样分层。

77. UI 组件、业务组件、页面组件如何划分？
答：结论：UI 组件强调通用性和样式复用，尽量不掺业务语义；业务组件承载领域逻辑和局部交互；页面组件负责把路由、数据、权限和多个区域编排起来。原理：这样分层后，复用边界、变更范围和测试重点都会更清晰。边界：如果一个 UI 组件里塞满权限判断和接口请求，或者页面组件承担太多细节逻辑，通常就说明分层已经失衡。

78. 自定义 Hooks、工具函数、服务层如何分层？
答：结论：我会把 Hook 定位成视图逻辑复用层，把工具函数定位成纯函数能力层，把服务层定位成接口请求和数据转换层。原理：Hook 可以依赖 React 生命周期和状态，工具函数不应该依赖组件上下文，服务层则尽量隔离请求协议、错误结构和数据整形。场景：这样拆完后，页面层就能更多关注业务编排，而不是同时处理请求细节和视图副作用。

79. 如何避免页面组件越来越臃肿？
答：结论：我会把状态、请求、表单、权限判断逐步下沉到 Hook、服务层和子组件里。原理/场景：页面组件过胖时，常见拆法是把请求、表单、权限和复杂状态下沉到 Hook、服务层和子组件。

80. 如何在 React 项目中引入 TypeScript 并保持可维护性？
答：结论：在 React 项目里引入 TypeScript，关键不是把类型写得多花，而是让边界清楚、接口稳定、公共类型收敛，避免类型体操。原理：类型系统最能发挥价值的地方是组件契约、请求层、状态结构和公共能力边界。场景：中大型团队协作、组件库和长期演进项目收益最明显。边界：复杂类型如果只增加阅读成本而不增加约束收益，就该及时收手。

81. 如何设计请求层、错误处理层、权限层？
答：结论：我会把请求封装、统一错误处理和权限校验做成基础设施，页面层尽量只关心业务结果。原理/场景：请求层、错误层、权限层都更像基础设施，页面层最好只关心业务结果和交互。

82. 为什么“能跑”和“可维护”是两回事？
答：结论：能跑只说明今天这版功能交付了，可维护看的是边界是否清楚、扩展是否可控、团队接手成本高不高。原理：短期把功能堆出来不难，难的是后续需求变化时能否低风险改动、快速定位问题、稳定复用已有能力。场景：这也是为什么中大型 React 项目里，目录分层、状态边界、组件抽象和监控测试都会比“先写出来再说”更重要。

83. 你会如何设计一个能长期迭代的后台前端架构？
答：结论：我会优先保证模块分层、配置同源、状态清晰、公共能力可复用，再加监控和测试兜底。原理/场景：长期可迭代的架构重点是分层稳定、配置同源、状态清晰，并且监控和测试能兜底。

## 十、综合与深挖

84. 如果让你讲一个 React 主项目，你会按什么结构讲？
答：结论：我会按业务背景、技术选型、架构设计、核心难点、解决方案和结果复盘这条线来讲。原理：这样讲的好处是先让面试官知道项目为什么存在，再说明为什么这样选技术，最后把你的价值落到问题解决和结果上。场景：不管是后台系统、内容平台还是 AI 应用，项目表达都应该围绕问题和结果，而不是流水账式地列功能。延伸：一句话就是，项目题先讲背景和矛盾，再讲方案和结果。

85. 如何把 React 的性能优化经历讲得具体可信？
答：结论：一定要讲问题现象、定位方法、改动点和量化结果，不要只说做了 memo 优化。原理/场景：只说“做了 memo 优化”说服力很弱，必须把定位过程和量化结果讲出来。

86. 如何把组件封装经验讲成工程能力？
答：结论：把组件封装经验讲成工程能力，重点不是说封了几个组件，而是说你怎么抽边界、控扩展、降重复和服务团队协作。原理：真正的工程价值在于让组件既能复用，又不会被业务差异轻易打穿。场景：按钮、表单项、表格、弹窗这类组件最容易体现抽象能力。边界：封装不是把所有差异都做成 props，过度兼容反而会让组件失控。

87. 如果面试官追问“为什么不用别的方案”，你该如何回答？
答：结论：如果面试官追问为什么不用别的方案，我会从业务规模、团队成本、复杂度和收益比来答，而不是简单说别的方案不好。原理：技术选型本质上是约束条件下的收益比较。场景：比如 Zustand 和 Redux、React Router 不同模式、表单库取舍，都应该回到项目目标来解释。边界：最怕的回答是绝对化，比如某个方案永远最好。

88. 你如何解释 React 项目里最难的一次问题定位经历？
答：结论：讲最难的一次问题定位时，我会按现象、复现、排查路径、根因、修复和预防这几个步骤讲。原理：这种表达方式能体现你不是凭经验拍脑袋，而是有系统排查方法。场景：渲染抖动、状态错乱、请求竞态、性能掉帧都适合这样组织。边界：只讲最后修了什么不够，真正加分的是中间如何定位。

89. 如何把 Hooks、状态管理、性能优化串成一个完整回答？
答：结论：可以围绕一个复杂页面讲，从状态拆分到 Hook 抽象，再到渲染优化，链路就很顺。原理/场景：用一个复杂页面把状态拆分、Hook 抽象、渲染优化串起来，会比散点作答更像真实经验。

90. 如果只给你 5 分钟总结 React，你会怎么组织答案？
答：结论：我会先讲声明式和组件化，再讲状态与渲染机制，最后补工程实践和性能边界。原理/场景：5 分钟总结时不要陷进 API 细节，先搭框架：声明式、状态驱动、调度更新、工程实践。

## 十一、Fiber 与调度机制

91. React 为什么要从旧的栈式协调转向 Fiber 架构？
答：旧的栈式协调最大问题是更新一旦开始，就会沿着调用栈一路同步执行到底，组件树一大就很容易长时间占住主线程，导致输入、动画和页面响应被明显拖慢。Fiber 的出现，本质上是为了把“更新 UI”从一个不可中断的大任务，拆成很多可暂停、可继续、可插队的小工作单元，让 React 获得真正的调度能力，所以 Fiber 最核心的价值不是数据结构换成链表，而是 React 从此有了“先做谁、后做谁、做一半能不能停”的能力。

92. 如果继续使用调用栈递归更新 UI，会带来哪些问题？
答：如果继续沿用传统调用栈递归同步更新，一次更新开始后就只能从上到下做完，React 几乎没有机会在中途把主线程还给浏览器处理更紧急的输入和绘制任务。这样一来，问题不只是极端情况下可能爆栈，更现实的风险是大列表刷新、复杂页面重算、频繁联动更新时会制造长任务，让用户明显感觉页面卡住，所以旧模式真正欠缺的是调度弹性，而不是单纯的遍历能力。

93. Fiber 节点本质上是什么？它和普通虚拟 DOM 节点有什么不同？
答：Fiber 可以把它理解成“带状态、带指针、带调度信息的工作单元”，它当然也描述这个组件长什么样，但它比普通的 Virtual DOM 节点多承载了一层“React 接下来怎么处理它”的信息。普通 Virtual DOM 更偏静态描述结果，而 Fiber 同时描述结构、更新状态、优先级和副作用标记，所以它不只是界面的快照，更像 React 渲染引擎手里的一张工作卡片，靠它才能把更新拆碎、暂停、继续和重试。

94. Fiber 节点上通常会存哪些关键信息？
答：Fiber 节点通常会存组件类型、待处理和已生效的 props、state、父子兄弟关系指针、对应的真实 DOM 或实例引用、副作用标记以及和优先级、更新队列相关的信息。理解这题的重点不是背源码字段名，而是明白它为什么需要这些内容：React 一边遍历 Fiber 树，一边要知道“这是什么组件、它当前和下一轮数据是什么、它和谁相连、这次有没有副作用、这次更新急不急”，只有这些信息都挂在节点上，调度系统才能真正工作起来。

95. `current tree` 和 `workInProgress tree` 的关系是什么？
答：`current tree` 是当前已经提交到屏幕上的那棵 Fiber 树，可以理解成“线上正式版本”；`workInProgress tree` 则是这次更新过程中正在计算的新版本草稿。React 会在旧树旁边构建这棵新树，只有等 render 全部算完并且进入 commit 后，两者才会切换角色。双树结构的意义在于让“正在显示的界面”和“正在计算的下一版界面”彼此隔离，这样 render 阶段就算被打断、回滚或重算，也不会把半成品直接展示给用户。

96. React 更新时是如何从当前树构建新的 Fiber 树的？
答：当某个组件产生更新后，React 会从当前树里对应的 Fiber 节点出发，创建或复用一套 work-in-progress Fiber 节点作为本轮工作的载体，然后在向下遍历时结合新的 props、state、context 做协调，决定哪些子节点继续复用、哪些需要新建、哪些可以直接 bailout 跳过。你可以把这个过程理解成“以当前树为蓝本，边复制边修订下一版草稿”，这样既保留了当前屏幕上的稳定结果，又能安全地计算下一次提交版本。

97. React 的 render 阶段和 commit 阶段分别在做什么？
答：render 阶段主要做“计算”，也就是重新执行相关组件、构建 work-in-progress 树、比较新旧节点、收集副作用标记；commit 阶段主要做“落地”，也就是把确认好的 DOM 变更一次性提交、更新 ref、执行布局相关逻辑并安排 effect。理解这两个阶段的最大价值，是你能分清哪些代码可能被重复执行或打断，哪些代码一旦开始就代表真实界面和外部系统要发生变化了。

98. 为什么 render 阶段可以被打断，而 commit 阶段不能？
答：因为 render 阶段只是生成候选结果，本质上还是在内存里算草稿，算到一半停下来、稍后继续、甚至整次推翻重来，都不会直接影响用户眼前看到的界面；但 commit 阶段已经开始动真实 DOM、ref 和副作用系统了，如果这时随便中断，界面就可能处在一半新一半旧的不一致状态。于是 React 才把 render 设计成可中断，把 commit 设计成必须一次性完成的原子提交。

99. React 所说的“调度”到底在调度什么？
答：React 调度的不是浏览器网络请求，也不是 DOM API 本身，而是“更新任务的执行顺序、优先级和让权时机”，也就是哪些更新应该立刻算，哪些可以晚一点，算到一半是否应该先暂停，把主线程让给更重要的事情。换句话说，调度解决的是“有限主线程资源该怎么分配给不同更新”，这是 Fiber 和并发能力背后的核心目标。

100. React 为什么需要优先级？高优先级和低优先级任务分别有哪些典型场景？
答：因为并不是所有更新对用户体感都一样重要，输入框回显、点击反馈、光标移动这类交互如果稍微迟一点，用户就会马上觉得卡；但大列表刷新、复杂筛选结果、预渲染内容稍晚一点出现，通常是可以接受的。所以 React 需要优先级系统来区分“必须先做”和“可以后做”的任务，把有限的主线程时间优先让给用户最能感知到的更新。

101. React 18 之后的 Lane 模型是怎么理解的？
答：Lane 可以把它想成“多车道优先级系统”，不同类型的更新不会只被粗暴地分成高、中、低，而是会被放进不同车道里，React 再根据这些车道决定哪些要先处理、哪些可以一起合并、哪些可以被更紧急的更新打断。这样设计的好处是优先级表达更细，多个更新之间既能区分轻重缓急，又能保留合并处理的空间，所以理解 Lane 时不必先陷入位运算细节，先抓住“多车道调度”这个心智就够了。

102. React 是如何实现“可中断渲染”的？
答：React 实现可中断渲染的关键，就是把原本一整棵树同步递归到底的大任务，拆成一个个以 Fiber 为单位的小工作单元，每处理一段就检查当前时间片是否快用完、有没有更高优先级更新插进来，如果有就先暂停当前 render，把控制权还给浏览器或去处理更紧急的任务。也就是说，真正让渲染可中断的不是某一个孤立 API，而是 Fiber 结构、调度器和时间片协作出来的一整套机制。

103. React 如何把控制权还给浏览器？
答：React 会在 render 工作推进过程中不断检查当前时间预算，如果发现这一帧继续算下去可能会压住输入和绘制，就会先暂停自己的工作，把主线程让回给浏览器，让浏览器去处理更紧急的用户输入、样式计算和绘制任务。这个“让权”动作正是时间切片的核心意义，它保证的是页面别被 React 一口气算太久；但要注意，这种让权主要发生在 render 阶段，commit 一旦开始仍然要一次性完成。

104. React 调度为什么经常会和 `MessageChannel`、时间切片一起被提到？
答：因为 React 想把长任务拆散执行，就需要一种比 `setTimeout` 更稳定、延迟更低的方式来安排“下一段工作什么时候继续”，而 `MessageChannel` 这类机制正适合用来做调度器底层的任务衔接。时间切片说的是把长 render 拆成很多小片轮流做，`MessageChannel` 常被拿来承接这些小片之间的调度，所以面试里这几个词经常一起出现；真正要理解的不是背某个底层 API，而是 React 为什么需要一套可靠的“暂停后再继续”机制。

105. React 是怎么判断当前是否还有剩余空闲时间的？
答：React 会记录当前这轮工作开始了多久，再结合一帧可接受的时间预算去判断：如果继续算下去很可能挤压浏览器处理输入和绘制的机会，就先暂停，把剩余工作留到下一轮。你可以把它理解成 React 在不断问自己“这帧我是不是已经做得够多了”，如果答案是快超时了，就主动收手。这种判断并不能保证绝对不掉帧，但能大幅减少一次更新长时间霸占主线程的风险。

106. 面试里如果被问“React 怎么计算一帧里还能不能继续执行”，你会怎么回答？
答：我会回答成一个容易理解的版本：React 会给当前 render 工作设一个时间预算，执行过程中不断比较“已经用了多久”和“还能不能在这一帧里继续干”，如果快到上限了，就暂停当前工作，把主线程留给浏览器去处理输入和绘制，下一轮再继续。这样答的重点不是背出某个具体毫秒值，而是讲清它为什么要这么做，也就是避免把一整帧时间都吃光。

107. React 中的高优先级任务通常会如何打断低优先级任务？
答：当低优先级 render 还在进行时，如果这时来了更高优先级的更新，比如输入或点击，React 会先暂停当前低优先级工作，把新的高优先级任务插到前面去算，等紧急更新处理完，再回来继续原来的低优先级工作，或者基于最新状态重新计算。这里的关键不是“中断 commit”，而是中断尚未提交的 render，所以它才能做到响应更及时，同时不让页面进入不一致状态。

108. 为什么说 Fiber 的核心价值不只是“链表结构”，而是“可调度”？
答：因为链表、父子兄弟指针这些只是实现手段，它们本身并不能自动让页面更流畅，真正产生价值的是 React 借助这套结构获得了把更新拆小、暂停、恢复、重排优先级的能力。换句话说，如果只是把树结构从递归栈换成链表，但更新仍然要一口气同步跑到底，那主线程阻塞问题依然存在；所以 Fiber 真正改变的是 React 的工作方式，而不是单纯换了一种数据组织形式。

109. 组件 setState 或 Hook 更新之后，是如何一路找到需要更新的 Fiber 的？
答：当组件调用 `setState` 或 Hook 更新函数时，这条更新会先被挂到当前组件对应的 Fiber 更新队列上，然后 React 会从这个 Fiber 一路向上冒泡，把“这条路径上有待处理更新”和“这次更新属于什么优先级”这些信息传到根节点。根节点拿到这些标记之后，调度器才知道整棵树里哪里有工作要做、这次该以什么优先级开始 render，所以它不是全树盲找，而是从更新源头沿链路向上登记，再由根统一调度。

110. React 如何决定某个 Fiber 节点是否需要继续向下更新？
答：React 会结合当前 Fiber 的类型、props、state、context 以及本轮更新优先级来判断这个节点和它的子树有没有继续处理的必要，如果发现输入没变、子树里也没有更高优先级待处理工作，就可以直接 bailout，跳过这部分后续遍历。你可以把 bailout 理解成一次“这块不用再算了”的剪枝，它是 React 避免整棵树无意义深挖的重要优化基础。

## 十二、React 19 与新特性

111. React 19 相比 React 18，最值得关注的新特性有哪些？
答：结论：我觉得最值得说的是 Actions、表单能力增强、`useOptimistic`、`useActionState` 和 `use()`。原理/场景：React 19 可重点讲 Actions、表单能力增强、`useOptimistic`、`useActionState` 和 `use()`。
延伸：如果项目还没上 React 19，也可以讲你理解的新特性价值和升级顾虑。

112. React 19 的 Actions 解决了什么问题？
答：结论：它把表单提交、异步状态、错误和 pending 这些常见流程收得更顺，少写很多样板代码。原理/场景：它把表单提交、pending、错误处理和异步结果更新串成更顺的一条链，减少手写样板。

113. `useActionState` 适合什么场景？
答：结论：`useActionState` 适合表单提交这类有返回状态、有成功失败反馈的异步交互场景。原理：它把提交函数、pending 状态和结果状态更紧密地绑定在一起。场景：登录、评论提交、资料修改这类表单流程会更顺手。边界：如果只是普通本地状态切换，没必要为了新 API 强行改写。

114. `useOptimistic` 适合什么场景？它和你自己手写乐观更新有什么区别？
答：结论：适合点赞、评论这类先给反馈再等结果的场景，它比手写更统一，也更贴近 React 状态流。原理/场景：`useOptimistic` 很适合点赞、评论、列表插入这类先反馈后确认的交互，但也要考虑失败回滚。

115. `useFormStatus` 解决了什么问题？
答：结论：`useFormStatus` 主要解决表单内部子组件拿提交状态不方便的问题，比如按钮 loading、禁用态这些。原理：它让表单上下文里的子组件能自然感知当前 action 是否在提交。场景：提交按钮、局部提示、提交中的禁用态都很适合。边界：它更适合表单上下文，不是通用全局 loading 方案。

116. `use()` 的核心作用是什么？它和 Suspense 的关系是什么？

123. 为什么函数组件重新执行了，DOM 却不一定立刻变？到底是组件代码影响了 DOM，还是 React 处理后才影响？
答：这题的关键是把 **render 阶段** 和 **commit 阶段**彻底分开。函数组件重新执行时，首先发生的不是“直接操作 DOM”，而是**重新计算本次 UI 的描述结果**。也就是 React 会再次调用你的函数组件，读取当前的 props、state、context，执行其中的普通 JavaScript 逻辑，然后拿到一棵新的 React Element 树；这棵树本质上只是“UI 应该长什么样”的描述，不是浏览器里的真实 DOM 节点。所以哪怕函数组件执行了很多次，只要还停留在 render 阶段，就只是“重新算方案”，不等于页面已经被改掉了。真正影响 DOM 的，是 React 在后续 **commit 阶段** 里根据新旧结果比较后，决定要不要、以及怎样去修改真实 DOM。换句话说：**组件函数负责产出描述，React 负责把描述变成最小化的 DOM 提交。**

之所以很多人会误以为“组件代码一执行就改了 DOM”，是因为在直觉里 JSX 很像模板，`return <div>{count}</div>` 看起来像“写页面”。但 React 内部真正拿到的是类似 `{ type: 'div', props: { children: count } }` 这样的对象结构，而不是已经插入文档的 DOM 节点。React 会把这次 render 产出的新树和上次 commit 后保留下来的旧树做对比，计算出差异，再在 commit 阶段把这些差异反映到真实 DOM 上。如果比较下来发现没有实际变化，或者这次 render 结果被更高优先级更新打断、丢弃了，那么函数组件即使执行过，DOM 也可能完全不动。这也是为什么 React 强调 render 阶段尽量保持纯：因为它可能被重复执行、被中断、被丢弃，不能把“执行函数组件”本身当成一次可靠的 DOM 提交时机。

再顺着讲清楚“为什么还需要 React 处理才会影响 DOM”。因为从函数组件执行得到的只是**声明式结果**，而不是“命令式修改页面”的步骤。React 的价值就在于替你做三件事：第一，记住上一次界面描述；第二，把这次新描述和上次旧描述做 diff；第三，只把真正变化的部分提交到真实 DOM。若没有 React 这层，你每次状态变化都得自己手动找 DOM、判断哪里变了、处理更新顺序、控制子节点复用和销毁。React 本质上就是把“重新执行组件函数得到的新描述”和“如何最小化更新真实界面”这两件事拆开，由框架接管第二件事。

再把阶段顺序压成面试常用表述：**render 阶段调用函数组件，生成新的虚拟树；commit 阶段把需要提交的变更同步到真实 DOM，并处理 ref、`useLayoutEffect` 等；`useEffect` 则通常在 commit 完成后再异步执行。** 所以如果有人问“commit 阶段会不会重新执行函数组件”，答案是否定的：函数组件执行属于 render，commit 只负责提交 render 的产物。面试里一句话总结就是：**函数组件重新执行，只是在重新计算 UI 描述；真正让 DOM 变化的，不是函数执行本身，而是 React 在 commit 阶段基于新旧结果比较后做出的提交。**
