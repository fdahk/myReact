# React 训练题纲

## 使用方式

这个模块按高频 React 问题展开，答案统一尽量覆盖结论、机制、业务场景和边界，避免只停留在 API 背诵层。

---

## 一、React 基础认知

1. React 的核心思想是什么？
答：结论：我理解 React 的核心就是组件化加声明式，用状态去描述 UI，数据变了视图自动跟着变。原理：它把界面拆成组件树，再通过单向数据流把状态变化稳定地传到视图层。场景：这套心智特别适合复杂交互和多人协作项目，因为组件边界、数据流向和更新来源都更容易追踪。延伸：一句话总结就是，组件化描述 UI，状态驱动视图更新。

2. 为什么 React 强调声明式 UI？
答：结论：因为声明式更关注结果长什么样，不用手动管一堆 DOM 细节，代码更稳也更好维护。原理/场景：声明式的价值不只是少操作 DOM，更重要是让状态和 UI 的关系更可推导、更易维护。

3. React 和直接操作 DOM 的主要差异是什么？
答：结论：直接操作 DOM 是命令式，一步步改节点；React 是基于状态重新描述界面，再统一做更新。原理/场景：React 更像先描述目标状态，再由框架统一决定最合适的更新方式。

4. 什么是组件化？组件化真正解决了什么问题？
答：结论：组件化就是把页面拆成有边界、可复用、可组合的单元，本质上解决的是复用、协作和维护成本问题。原理：React 把 UI 抽象成组件树，让状态和渲染边界都能落到具体组件上。场景：像表单、表格、弹窗、布局区块都适合组件化拆分，团队协作时也更容易并行开发。边界：组件不是拆得越碎越好，过度拆分会放大 props 传递和维护成本。

5. React 应用的渲染流程大致是什么？
答：结论：一般是状态变化触发 render，React 生成新旧树做 diff，最后在 commit 阶段把变更同步到 DOM。原理/场景：标准表达通常是 render 计算结果、reconcile 比较差异、commit 真正落 DOM 和副作用。
延伸：一句简化链路：状态变更 -> render -> reconcile -> commit。

6. React 为什么要引入 Virtual DOM？
答：结论：不是为了神化性能，而是为了用一层抽象统一描述 UI，让跨平台和更新策略都更可控。原理/场景：它的价值更像统一 UI 抽象和调度更新，而不是神化成“任何场景都更快”。
延伸：Virtual DOM 的核心价值更在于统一 UI 抽象和更新策略，而不只是性能优化。

7. Virtual DOM 一定更快吗？
答：结论：不一定，简单场景直接改 DOM 可能更快，Virtual DOM 的价值更多是可维护和批量更新能力。原理/场景：简单场景下直接操作 DOM 可能更快，但复杂 UI 下 React 的可维护性和批量更新能力更有优势。

8. React 的优势和成本分别是什么？
答：结论：优势是组件化、生态成熟、工程化能力强；成本是心智负担更高，状态设计和性能优化更考验经验。原理：React 把 UI 抽象、状态驱动和调度机制做得很完整，所以适合复杂前端工程。场景：中后台、复杂交互、多人协作项目通常很受益；页面很轻、团队经验不足时，React 的工程成本也会更明显。边界：它不是所有项目的默认最优解，要看业务复杂度和团队熟悉度。

## 二、JSX 与渲染机制

9. JSX 的本质是什么？
答：结论：JSX 本质上是语法糖，编译后还是 `React.createElement` 或新的 JSX runtime 调用。原理：它把 UI 结构和 JS 表达式放在一套语法里，方便描述声明式视图。场景：开发时可读性更高，也更利于组件化表达和条件渲染。边界：JSX 不是模板语言本身，最终仍然会回到 JavaScript 调用。

10. JSX 为什么不能直接写多个根节点？
答：结论：因为组件返回值本质上要形成一个统一的虚拟节点结构，不过现在可以用 Fragment 同时包住多个节点。原理：React 早期在协调时更容易围绕单根节点处理树结构，Fragment 则是在不额外产出 DOM 的前提下满足这个要求。场景：页面结构需要并列节点时，优先用 Fragment，而不是为了凑单根去多包一层无语义 `div`。边界：现在不是不能返回多个节点，而是通常仍要通过 Fragment 或数组把返回结构组织起来。

11. 条件渲染和列表渲染有哪些常见陷阱？
答：结论：常见坑就是 `0 && xxx` 会把 `0` 渲染出来、列表 `key` 不稳定导致节点错复用、条件切换后组件状态串位。原理：这些问题本质上都和 React 对表达式结果、节点身份和组件复用方式的处理有关。场景：像表单项切换、动态列表增删改、不同组件条件互切时最容易踩到。

12. `key` 为什么重要？
答：结论：`key` 是 React 识别同级节点身份的依据，直接影响复用、重排和组件状态是否保留。原理/场景：`key` 解决的是同级节点身份识别问题，直接影响复用、重排和本地状态是否保留。
延伸：`key` 直接影响组件状态是否保留，是列表题的核心。

13. 为什么不能随便使用数组索引作为 `key`？
答：结论：因为一旦插入、删除、排序，索引就变了，React 可能错误复用节点，导致状态串位。原理/场景：如果列表会插入、删除、排序，索引 key 很容易导致节点错误复用和状态串位。

14. React 更新列表时如何利用 `key` 提升复用效率？
答：结论：React 更新列表时会先按 `key` 找可复用节点，只更新真正变化的项，减少不必要的卸载和重建。原理：`key` 是同级节点身份标识，稳定的 `key` 能让 diff 更准确地判断谁是新增、谁是移动、谁是复用。场景：增删改排序频繁的列表、表格、表单行最依赖稳定 `key`。边界：`key` 只在同级比较里生效，不是全局唯一 ID。

15. 什么是协调过程？
答：结论：协调就是 React 比较新旧树，决定哪些节点复用、更新、删除、新增的过程。原理/场景：协调本质上是在决定哪些节点复用、哪些更新、哪些删除，不是简单“全量 diff”。

16. React Diff 的基本策略是什么？
答：结论：核心就是同层比较、不同类型直接替换、同类型继续比较属性和子节点，再结合 `key` 复用列表项。原理/场景：React 选择同层比较是工程上的复杂度取舍，结合 `key` 才能在列表场景下提高复用准确性。

17. 为什么 React Diff 是同层比较？
答：结论：因为如果做跨层全量比较，复杂度太高，React 用同层比较是在性能和准确性之间做的工程取舍。原理：UI 更新最常见的是同层节点的增删改，同层比较能把 diff 控制在更可接受的复杂度内，再配合 `key` 提高列表场景的复用准确性。边界：它的代价是跨层移动节点时不会做最优复用，很多时候会直接重建。

18. 如果组件频繁重新渲染，你怎么定位原因？
答：结论：我会先看 React DevTools Profiler，再查状态源头、父组件传参、上下文变更和不稳定引用。原理/场景：排查时我会优先用 React DevTools Profiler，再看状态源头、父组件传参和上下文变更。

## 三、Hooks

19. `useState` 的更新机制是什么？
答：结论：`useState` 不是立刻改变量，而是把更新任务放进队列，下一轮渲染再基于队列算新状态。原理/场景：`useState` 更新更像入队和调度，而不是同步改变量本身，这是很多“为什么拿到旧值”问题的根源。
延伸：`setState/useState` 更像入队和调度，而不是同步改值。

20. 为什么 state 更新看起来是异步的？
答：结论：因为 React 会调度和批量处理更新，所以当前同步代码里通常拿到的还是旧值。原理/场景：更准确地说，是更新被调度和批量处理，所以当前同步代码里经常读到旧快照。

21. React 中的批量更新是什么？
答：结论：React 的批量更新就是把同一轮事件循环里的多个状态更新合并处理，减少重复渲染和重复提交。原理：React 会先把更新收集进队列，再统一计算新状态和新树。场景：一个点击里连续调多次 `setState` 时，批量更新能明显减少额外渲染。边界：批量更新优化的是调度次数，不代表你在当前同步代码里就能立刻拿到新 state。

22. `useEffect` 和 `useLayoutEffect` 的区别是什么？
答：结论：`useEffect` 在浏览器绘制后执行，`useLayoutEffect` 在 DOM 提交后、绘制前执行，适合同步读写布局。原理/场景：`useLayoutEffect` 会阻塞绘制，只应放需要同步读写布局的逻辑，绝大多数副作用仍优先 `useEffect`。
延伸：只要不涉及布局同步读写，副作用优先 `useEffect`。

23. `useEffect` 的清理函数在什么时机执行？
答：结论：`useEffect` 的清理函数会在下一次 effect 执行前先跑一遍，组件卸载时也会再跑一次。原理：React 这样设计是为了让旧副作用先退出，再建立新副作用，避免订阅、定时器和监听器叠加。场景：事件绑定、WebSocket、定时器、请求取消都很依赖清理函数。边界：如果 effect 依赖频繁变化，清理和重建也会频繁发生，所以依赖设计要谨慎。

24. 为什么 `useEffect` 依赖数组容易出错？
答：结论：因为它依赖的是引用和闭包，不是我觉得没变就真没变，漏依赖和错依赖都很常见。原理/场景：依赖数组难点在于引用比较和闭包快照，不是“我主观觉得没变就算没变”。
延伸：依赖数组问题常和闭包旧值一起追问。

25. 什么是闭包陷阱？在 Hooks 里如何出现？
答：结论：就是回调拿到的是旧渲染时的变量，比如定时器、事件监听、异步回调里最容易出现。原理/场景：最常见在定时器、事件监听和异步回调里拿到旧 state，这也是 Hooks 高频追问点。
延伸：解决思路常见有补全依赖、函数式更新、`useRef` 保存最新值。

26. `useMemo` 和 `useCallback` 分别解决什么问题？
答：结论：`useMemo` 缓存计算结果，`useCallback` 缓存函数引用，核心都是减少不必要的重复工作。原理/场景：`useMemo` 缓存结果，`useCallback` 稳定函数引用，通常要和子组件 `memo` 联动才有实际收益。

27. 什么情况下不应该滥用 `useMemo`？
答：结论：计算本身很轻、组件也不敏感时就别上，不然缓存成本和理解成本可能比收益还大。原理/场景：缓存本身也有维护成本，如果计算很轻或依赖总在变，收益可能还不如不用。

28. `useRef` 的核心用途是什么？
答：结论：一个是拿 DOM，另一个是存跨渲染周期但不触发更新的可变值。原理/场景：`useRef` 一个拿 DOM，一个保存跨渲染周期但不触发更新的可变值，这两类场景都很高频。

29. 如何用 Hooks 封装可复用逻辑？
答：结论：把状态、副作用和事件处理按业务能力封成自定义 Hook，对外只暴露清晰输入输出。原理/场景：高质量自定义 Hook 不是把代码搬走，而是抽象出可复用的状态、副作用和输入输出协议。

30. 自定义 Hooks 的设计原则是什么？
答：结论：自定义 Hook 的设计原则我会强调职责单一、命名清楚、输入输出稳定，不把页面耦合细节直接塞进去。原理：Hook 的价值是复用状态和副作用逻辑，而不是简单搬代码。场景：像列表请求、弹窗控制、权限判断、表单联动都适合沉淀成 Hook。边界：如果一个 Hook 同时管请求、路由、弹窗和埋点，通常就说明边界已经失控。

## 四、组件通信与状态管理

31. React 常见组件通信方式有哪些？
答：结论：常见通信方式有 props、回调、Context、状态库、事件总线，但我的选择原则一直是先近后远、先简单后复杂。原理：React 默认就是单向数据流，所以能在父子层解决的问题，没必要过早放大全局。场景：父子通信优先 props 和回调，跨层共享再考虑 Context，真正到跨页面共享、联动复杂、调试要求高时再上状态库。延伸：一句话就是，能局部解决就别全局化。

32. 父子通信、兄弟通信、跨层通信分别怎么做？
答：结论：父子通信靠 props 和回调，兄弟通信通常靠状态提升，跨层共享优先 Context，再复杂点再上状态管理。原理：React 数据流默认是单向向下的，所以谁拥有状态，谁就更适合做通信源头。场景：局部联动优先近距离传递，跨页面或复杂共享再考虑全局方案。边界：不要为了少传几层 props 就一上来把所有状态都塞进全局。

33. `Context` 适合解决什么问题？
答：结论：适合主题、用户信息、权限这类跨层共享但更新频率没那么夸张的数据。原理/场景：它更适合主题、语言、权限、用户信息这类横切数据，不适合高频变化的大状态。

34. `Context` 为什么可能引发性能问题？
答：结论：因为 Provider 的 value 变了，相关消费者都会重新渲染，所以 value 设计要很谨慎。原理/场景：Provider 的 `value` 一变，相关消费者就会重新渲染，所以 value 稳定性和拆分粒度很关键。
延伸：常见优化方式是拆 Provider、稳定 `value`、减少无关消费者。

35. 什么时候只用 `useState` 就够了？
答：结论：状态只在局部页面里流转、依赖关系不复杂时，`useState` 往往就够了，没必要上全局方案。原理：局部状态离使用方越近，心智负担越低，也更容易维护。场景：弹窗开关、表单输入、局部筛选条件、分页状态这些都很适合留在组件内部。边界：一旦状态跨页面共享、需要持久化或联动很多模块，再考虑提升到更高层。

36. 什么时候要引入 Zustand、Redux、MobX 这类状态管理方案？
答：结论：当状态跨页面共享、联动复杂、调试诉求强，或者需要统一数据流时，再引入 Zustand、Redux、MobX 这类方案才更有价值。原理：这类工具解决的不是“state 不够用”，而是共享范围、更新链路、调试追踪和团队协作都开始变复杂。场景：比如全局用户态、权限、跨页面筛选条件、复杂业务流编排，就比单纯 `useState` 更适合交给状态管理层。

37. 全局状态和局部状态如何划分边界？
答：结论：能放局部就别放全局，只有跨页面共享、长期存在、多人依赖的状态我才会提到全局。原理/场景：我通常坚持“局部优先”，只有跨页面共享、生命周期长、多人依赖的状态才会提到全局。
延伸：能放局部就别放全局，是很重要的工程取舍。

38. 服务端状态和客户端状态为什么应该分开考虑？
答：结论：因为服务端状态核心是获取、缓存、失效和同步，和表单输入、弹窗开关这种本地状态不是一类问题。原理/场景：服务端状态核心是获取、缓存、失效、重试和同步，和 UI 开关、表单输入不是一类问题。

39. 如何设计一个复杂页面的状态结构？
答：结论：我会先按领域拆，再区分源数据、派生数据和 UI 状态，避免所有东西都堆在一个组件里。原理/场景：复杂页面最好分成源数据、派生数据和 UI 状态三层，避免所有字段都平铺在一个组件里。

40. 如何避免状态分散和状态冗余？
答：结论：关键是单一数据源，能推导出来的就别重复存，状态归属也要尽量贴近使用方。原理/场景：能推导出来的就不要重复存，单一数据源是避免状态打架的关键。

## 五、生命周期与渲染时机

41. 函数组件时代如何理解生命周期？
答：结论：函数组件时代我更愿意按渲染、提交、副作用建立和清理这几个阶段理解生命周期，而不是硬背类组件名词。原理：函数组件本身就是重复执行的渲染函数，真正有阶段差异的是 render、commit 和 effect。场景：这样更容易把 `useEffect`、`useLayoutEffect`、Strict Mode 和并发渲染串起来解释。边界：如果面试官追问类组件生命周期，再把这套阶段映射回去即可。

42. 挂载、更新、卸载分别会触发哪些常见逻辑？
答：结论：挂载阶段常做初始化和订阅，更新阶段常做同步逻辑和副作用重建，卸载阶段重点做事件、定时器、请求这类清理。原理：不同阶段关注点不同，核心是避免副作用泄漏和旧逻辑残留。场景：页面进入时拉首屏数据，依赖变化时重建订阅，离开页面时取消请求，都是典型例子。边界：不要把所有逻辑都堆到一个 effect 里，不然生命周期边界会很模糊。

43. 为什么渲染阶段不能做副作用？
答：结论：因为渲染阶段可能被重复执行甚至打断，不安全，副作用要放到 commit 后。原理/场景：因为渲染阶段可能被重复执行、打断甚至丢弃，所以副作用必须放到 commit 后。
延伸：渲染阶段要尽量保持纯函数心智。

44. 什么是 commit 阶段？
答：结论：commit 阶段就是 React 把本次 render 计算出的变更真正同步到 DOM 和副作用系统的阶段。原理：render 负责算结果，commit 才负责落地，所以只有到了 commit，页面和副作用才算真正更新。场景：DOM 读写、布局 effect、ref 更新都发生在这一阶段周围。边界：render 可以被打断重来，但 commit 一旦开始就要保证原子性。

45. Strict Mode 为什么会让某些逻辑执行两次？
答：结论：开发环境下它会故意多跑一次，帮你暴露不纯副作用和不安全逻辑。原理/场景：Strict Mode 在开发环境下故意多跑一轮，是为了帮助你暴露不纯逻辑和清理缺失问题。
延伸：Strict Mode 的双调用主要发生在开发环境。

46. React 这样设计是为了帮助发现什么问题？
答：结论：React 这样设计主要是为了提前暴露渲染不纯、effect 没清理、依赖不稳和状态更新时机不安全这些问题。原理：开发环境故意更严格，能在上线前放大潜在副作用问题。场景：很多定时器重复注册、事件重复绑定、依赖漏写的问题，都是在 Strict Mode 下更容易暴露。边界：它增加的是开发期暴露概率，不是生产环境真的会多执行一次。

47. 为什么组件渲染不等于 DOM 更新？
答：结论：渲染只是算结果，真正改 DOM 要等到 commit，所以组件函数执行了不代表页面一定变了。原理/场景：组件函数执行只是算新树，只有 commit 后页面和副作用系统才真正发生变化。

48. 如何判断一次更新是否真的引发了页面重绘？
答：结论：我会结合浏览器 Performance、React Profiler 和具体样式变更一起看，不只盯 React 渲染次数。原理：组件函数执行、DOM 更新、浏览器重绘不是一回事，三者要分开定位。场景：有时 React 渲染很多但浏览器压力不大，有时渲染不多却因为布局抖动导致掉帧。边界：只看 React Profiler 很容易漏掉真实的浏览器瓶颈。

## 六、性能优化

49. React 项目常见性能问题有哪些？
答：结论：React 项目常见性能问题通常有无效渲染、长列表、重计算、包体积过大、请求串行和状态设计不合理。原理：这些问题本质上分别落在渲染次数、单次渲染成本、资源加载成本和数据流设计上。场景：排查时我会先区分到底是首屏慢、交互卡，还是某个局部区域重渲染过多，再决定是做拆分、缓存、虚拟列表还是请求治理。延伸：如果能结合一次真实优化案例，回答会更有说服力。

50. 什么是无效渲染？
答：结论：无效渲染就是组件重新执行了，但最终 UI 没有实质变化，这类渲染通常就是优化重点。原理：状态提升过高、父组件引用不稳定、Context 粒度过粗都可能让子树被动重跑。场景：列表页、复杂表单、筛选区联动很容易出现这种问题。边界：不是所有重复渲染都值得优化，要看组件重量和用户感知。

51. `React.memo` 的作用是什么？
答：结论：它会在 props 浅比较没变时跳过函数组件重渲染，适合稳定 props 的纯展示组件。原理/场景：`React.memo` 适合 props 稳定的纯展示组件，不是所有组件都值得包一层。
延伸：`memo` 只有在“跳过渲染”收益大于“比较成本”时才值得用。

52. `memo` 为什么不是万能优化手段？
答：结论：因为它本身也有比较成本，如果 props 总在变或者组件很轻，收益可能很有限。原理/场景：比较本身也有成本，如果 props 总变或组件很轻，`memo` 可能没有收益。

53. `useMemo`、`useCallback`、`memo` 该如何配合使用？
答：结论：一般是父组件用 `useMemo`、`useCallback` 稳定传参，子组件再配合 `memo` 才比较有意义。原理/场景：常见套路是父组件用 `useMemo`/`useCallback` 稳定传参，子组件再配 `memo` 才更有意义。

54. 长列表优化有哪些常见方案？
答：结论：常见是虚拟列表、分页、懒加载、分片渲染，再加上减少单项渲染成本。原理/场景：长列表优化通常不只一招，常见是虚拟列表、分页、分片渲染和减少单项渲染成本一起做。
延伸：大表格、大列表几乎一定会被追问到虚拟列表。

55. 表单很多、交互复杂时如何减少卡顿？
答：结论：我会拆状态、减少全量联动、做局部更新，必要时配合防抖和低优先级更新。原理/场景：复杂表单场景里，拆状态和减少全量联动往往比无脑上缓存 Hook 更有效。

56. 路由级懒加载如何做？
答：结论：路由级懒加载一般配合 `React.lazy`、`Suspense` 或路由层动态 import，把首屏不需要的页面代码后置。原理：它本质上是代码分割，让不同路由按访问时机再加载各自 bundle。场景：后台系统、大型内容页、多模块平台都很适合。边界：懒加载能减小首屏包，但切路由时可能出现额外等待，所以要配合 loading 态和预加载策略。

57. 组件拆分为什么也属于性能优化？
答：结论：组件拆分也是性能优化，因为拆完之后状态影响范围更小，重渲染粒度更细。原理：当一个大组件被拆成多个稳定边界后，局部状态变化不会轻易拖着整棵子树一起重跑。场景：复杂表单、搜索页、表格页特别明显，拆分后既更易维护，也更容易配合 `memo`、局部状态和懒加载做进一步优化。

58. 如何量化一次 React 性能优化是否有效？
答：结论：我会拿优化前后首屏时间、交互耗时、渲染次数和用户感知指标做对比，不靠感觉。原理/场景：性能优化要回到数据，用渲染次数、交互耗时和用户感知指标验证，而不是靠感觉。

## 七、表单、路由与业务场景

59. 受控组件和非受控组件的区别是什么？
答：结论：受控组件值由 React state 驱动，非受控更多依赖 DOM 自身状态，前者更可控，后者更轻。原理/场景：受控更可控、更好做校验和联动；非受控更轻，但复杂表单里通常不如受控稳定。

60. 大型表单最常见的问题是什么？
答：结论：大型表单最常见的问题是状态过多、联动复杂、校验分散，最后同时拖垮性能和可维护性。原理：表单字段一多，任何一次输入都可能牵动全局状态和一串派生逻辑。场景：复杂审批单、配置面板、动态表单最容易出现这种问题。边界：优化重点通常是拆状态、分区域更新和收敛校验，不是无脑加 memo。

61. 表单校验逻辑应该放在哪一层？
答：结论：基础规则我会靠近表单模型，业务校验放服务或提交层，别把所有逻辑都塞组件里。原理/场景：基础规则靠近表单模型，跨字段和业务规则放提交层或服务层，避免组件里堆满判断。

62. React Router 的核心作用是什么？
答：结论：React Router 的核心作用就是把 URL 和页面状态对应起来，让单页应用也能按路由组织页面、导航和刷新恢复。原理：它把浏览器地址变化映射成组件树切换，而不是整页重新刷新。场景：页面导航、参数驱动筛选、刷新后恢复当前页，都依赖路由系统。边界：路由解决的是页面状态组织，不负责替你处理权限和数据缓存策略。

63. 前端路由和后端路由的区别是什么？
答：结论：前端路由主要负责视图切换和页面状态组织，后端路由主要负责资源定位、接口处理和服务端分发，职责根本不在一层。原理：前端路由更多运行在浏览器，后端路由更多运行在服务器。场景：单页应用里切页通常是前端路由，接口 `/api/users` 这种则是后端路由。边界：SSR 或同构场景下两者会协作，但职责仍然要分清。

64. 如何设计后台系统中的权限路由？
答：结论：我会用路由配置做单一入口，把菜单、路由、权限码尽量放同一份元数据里。原理/场景：更稳的做法是把菜单、路由和权限码维护在同一份元数据里，减少配置漂移。

65. 路由守卫在 React 中通常怎么实现？
答：结论：React 里做路由守卫，一般是在布局层、高阶路由组件或 loader 里做权限校验，再配合重定向。原理：核心不是阻止路由本身，而是在进入页面前统一判断登录态和权限。场景：后台系统的登录拦截、菜单权限、角色隔离都很常见。边界：守卫只是一层前端兜底，真正权限校验仍要在服务端完成。

66. 菜单、路由、权限如何保持一致？
答：结论：菜单、路由、权限要保持一致，核心就是配置同源，别三套数据各管各的。原理：只要入口不统一，后续必然出现菜单能见但进不去、路由可进但无按钮权限这类错位。场景：更稳的做法是用一份路由元数据同时驱动菜单、守卫和按钮权限。边界：前端配置统一能降低漂移，但真正权限结果仍要以后端为准。

67. 标签页缓存、多页签管理该怎么设计？
答：结论：我会先明确缓存粒度，再区分路由缓存、页面状态缓存和业务数据缓存，别混成一层。原理/场景：设计时最好区分路由缓存、页面状态缓存和业务数据缓存，别三者混成一层。

68. 复杂搜索页和表格页，组件边界如何划分？
答：结论：复杂搜索页和表格页，我通常按筛选区、表格区、操作区、弹窗区拆，页面层只负责编排和状态协调。原理：这样可以把高频变化区和重渲染区隔离开，降低联动成本。场景：中后台列表页、订单页、审批页都适合这种拆法。边界：页面层仍要保留整体状态编排权，不能把跨区域协作逻辑完全打散。

## 八、React 18 与并发特性

69. React 18 带来了哪些重要变化？
答：结论：比较关键的是并发能力、自动批处理、`useTransition`、`useDeferredValue` 和新 root API。原理/场景：React 18 最值得讲的是并发渲染能力、自动批处理和新的优先级调度能力。
延伸：React 18 题常和自动批处理、Transition、Fiber 调度放在一起问。

70. 什么是并发渲染？
答：结论：我的理解是 React 可以把渲染工作切开、插队、重试，不再是一口气同步干到底。原理/场景：并发强调的是可中断、可插队、可恢复，不是浏览器里突然多线程并行执行 React。

71. 并发渲染和并行执行是一回事吗？
答：结论：不是，并发强调的是可打断、可插队和优先级调度，不代表 React 真的在浏览器里多线程并行跑。原理：本质上还是主线程执行，只是 React 更会安排谁先算、谁后算。场景：输入框联动大列表时，就能把紧急输入和低优先级列表更新拆开。边界：并发解决的是交互响应问题，不是无中生有减少总计算量。

72. `useTransition` 适合什么场景？
答：结论：适合输入之后还要触发较重 UI 更新的场景，比如筛选、切列表、切 tab。原理/场景：`useTransition` 适合把非紧急更新降成低优先级，避免输入和点击被重渲染拖慢。
延伸：`useTransition` 解决的是“响应别被重更新拖慢”，不是减少总工作量。

73. `useDeferredValue` 适合什么场景？
答：结论：适合把高频变化的值延后给低优先级内容，比如搜索输入和结果列表分开更新。原理/场景：`useDeferredValue` 更像让重内容“跟慢一点”，常见在搜索输入和大结果列表联动里。

74. 自动批处理带来了什么变化？
答：结论：React 18 里更多异步场景也会自动合并更新，减少额外重渲染。原理/场景：React 18 之后更多异步场景也会自动合并更新，但必要时仍可用 `flushSync` 明确同步刷新。

75. 这些新特性在业务项目里真的常用吗？面试该如何回答？
答：结论：这些特性不是每个项目都会重度使用，但我会说清它们解决什么卡顿问题，以及为什么有些场景没必要上。原理：像 `useTransition`、`useDeferredValue` 更偏性能调度，不是基础 CRUD 必备。场景：数据量大、筛选重、输入联动重渲染的页面更容易感知收益。边界：如果页面本身不重，强上并发特性只会增加复杂度。

## 九、工程与架构

76. 一个中大型 React 项目通常如何组织目录？
答：结论：我一般会按页面、组件、hooks、services、stores、utils、types 这些层次拆，但不会机械地按文件类型平铺。原理：目录结构本质上是在表达分层和边界，既要看业务聚合关系，也要看复用边界和依赖方向。场景：页面层负责编排，组件层负责 UI，hooks 复用视图逻辑，services 处理接口和数据转换，stores 管共享状态，这样项目越大越不容易失控。延伸：目录题真正加分点不是报目录名，而是说清为什么这样分层。

77. UI 组件、业务组件、页面组件如何划分？
答：结论：UI 组件强调通用性和样式复用，尽量不掺业务语义；业务组件承载领域逻辑和局部交互；页面组件负责把路由、数据、权限和多个区域编排起来。原理：这样分层后，复用边界、变更范围和测试重点都会更清晰。边界：如果一个 UI 组件里塞满权限判断和接口请求，或者页面组件承担太多细节逻辑，通常就说明分层已经失衡。

78. 自定义 Hooks、工具函数、服务层如何分层？
答：结论：我会把 Hook 定位成视图逻辑复用层，把工具函数定位成纯函数能力层，把服务层定位成接口请求和数据转换层。原理：Hook 可以依赖 React 生命周期和状态，工具函数不应该依赖组件上下文，服务层则尽量隔离请求协议、错误结构和数据整形。场景：这样拆完后，页面层就能更多关注业务编排，而不是同时处理请求细节和视图副作用。

79. 如何避免页面组件越来越臃肿？
答：结论：我会把状态、请求、表单、权限判断逐步下沉到 Hook、服务层和子组件里。原理/场景：页面组件过胖时，常见拆法是把请求、表单、权限和复杂状态下沉到 Hook、服务层和子组件。

80. 如何在 React 项目中引入 TypeScript 并保持可维护性？
答：结论：在 React 项目里引入 TypeScript，关键不是把类型写得多花，而是让边界清楚、接口稳定、公共类型收敛，避免类型体操。原理：类型系统最能发挥价值的地方是组件契约、请求层、状态结构和公共能力边界。场景：中大型团队协作、组件库和长期演进项目收益最明显。边界：复杂类型如果只增加阅读成本而不增加约束收益，就该及时收手。

81. 如何设计请求层、错误处理层、权限层？
答：结论：我会把请求封装、统一错误处理和权限校验做成基础设施，页面层尽量只关心业务结果。原理/场景：请求层、错误层、权限层都更像基础设施，页面层最好只关心业务结果和交互。

82. 为什么“能跑”和“可维护”是两回事？
答：结论：能跑只说明今天这版功能交付了，可维护看的是边界是否清楚、扩展是否可控、团队接手成本高不高。原理：短期把功能堆出来不难，难的是后续需求变化时能否低风险改动、快速定位问题、稳定复用已有能力。场景：这也是为什么中大型 React 项目里，目录分层、状态边界、组件抽象和监控测试都会比“先写出来再说”更重要。

83. 你会如何设计一个能长期迭代的后台前端架构？
答：结论：我会优先保证模块分层、配置同源、状态清晰、公共能力可复用，再加监控和测试兜底。原理/场景：长期可迭代的架构重点是分层稳定、配置同源、状态清晰，并且监控和测试能兜底。

## 十、综合与深挖

84. 如果让你讲一个 React 主项目，你会按什么结构讲？
答：结论：我会按业务背景、技术选型、架构设计、核心难点、解决方案和结果复盘这条线来讲。原理：这样讲的好处是先让面试官知道项目为什么存在，再说明为什么这样选技术，最后把你的价值落到问题解决和结果上。场景：不管是后台系统、内容平台还是 AI 应用，项目表达都应该围绕问题和结果，而不是流水账式地列功能。延伸：一句话就是，项目题先讲背景和矛盾，再讲方案和结果。

85. 如何把 React 的性能优化经历讲得具体可信？
答：结论：一定要讲问题现象、定位方法、改动点和量化结果，不要只说做了 memo 优化。原理/场景：只说“做了 memo 优化”说服力很弱，必须把定位过程和量化结果讲出来。

86. 如何把组件封装经验讲成工程能力？
答：结论：把组件封装经验讲成工程能力，重点不是说封了几个组件，而是说你怎么抽边界、控扩展、降重复和服务团队协作。原理：真正的工程价值在于让组件既能复用，又不会被业务差异轻易打穿。场景：按钮、表单项、表格、弹窗这类组件最容易体现抽象能力。边界：封装不是把所有差异都做成 props，过度兼容反而会让组件失控。

87. 如果面试官追问“为什么不用别的方案”，你该如何回答？
答：结论：如果面试官追问为什么不用别的方案，我会从业务规模、团队成本、复杂度和收益比来答，而不是简单说别的方案不好。原理：技术选型本质上是约束条件下的收益比较。场景：比如 Zustand 和 Redux、React Router 不同模式、表单库取舍，都应该回到项目目标来解释。边界：最怕的回答是绝对化，比如某个方案永远最好。

88. 你如何解释 React 项目里最难的一次问题定位经历？
答：结论：讲最难的一次问题定位时，我会按现象、复现、排查路径、根因、修复和预防这几个步骤讲。原理：这种表达方式能体现你不是凭经验拍脑袋，而是有系统排查方法。场景：渲染抖动、状态错乱、请求竞态、性能掉帧都适合这样组织。边界：只讲最后修了什么不够，真正加分的是中间如何定位。

89. 如何把 Hooks、状态管理、性能优化串成一个完整回答？
答：结论：可以围绕一个复杂页面讲，从状态拆分到 Hook 抽象，再到渲染优化，链路就很顺。原理/场景：用一个复杂页面把状态拆分、Hook 抽象、渲染优化串起来，会比散点作答更像真实经验。

90. 如果只给你 5 分钟总结 React，你会怎么组织答案？
答：结论：我会先讲声明式和组件化，再讲状态与渲染机制，最后补工程实践和性能边界。原理/场景：5 分钟总结时不要陷进 API 细节，先搭框架：声明式、状态驱动、调度更新、工程实践。

## 十一、Fiber 与调度机制

91. React 为什么要从旧的栈式协调转向 Fiber 架构？
答：结论：因为旧架构一更新就容易长时间占住主线程，Fiber 的目标就是把渲染变得可切分、可打断、可调度。原理/场景：旧的栈式递归更新一旦树大就会长时间占住主线程，Fiber 解决的是“怎么把更新变得可调度”。
延伸：Fiber 题的核心不是链表，而是可中断、可调度。

92. 如果继续使用调用栈递归更新 UI，会带来哪些问题？
答：结论：如果继续用调用栈递归同步更新 UI，最大问题就是树一大任务就没法中断，用户输入和动画都会被卡住。原理：旧的栈式协调一旦开始就要一路算到底，主线程长时间拿不回来。场景：大列表刷新、复杂页面重算时，旧架构更容易出现明显掉帧。边界：问题不只是爆栈，更关键是缺少调度能力。

93. Fiber 节点本质上是什么？它和普通虚拟 DOM 节点有什么不同？
答：结论：Fiber 可以理解成带调度信息的工作单元，不只是描述 UI，还描述更新过程本身。原理：它把原来一次性递归执行的大任务拆成可暂停、可继续的小单元。场景：这让 React 能实现优先级调度、时间切片和并发渲染。边界：Fiber 不是另一套 Virtual DOM 替代品，而是让更新过程更可调度。

94. Fiber 节点上通常会存哪些关键信息？
答：结论：Fiber 节点上通常会有类型、props、state、父子兄弟指针、副作用标记和优先级相关信息。原理：它既要描述当前组件是什么，也要支持调度系统知道接下来该怎么处理它。场景：正因为节点上有这些信息，React 才能边遍历边决定继续算、跳过还是打断。边界：这里的重点不是死记字段名，而是理解它同时承载结构和调度信息。

95. `current tree` 和 `workInProgress tree` 的关系是什么？
答：结论：`current tree` 是当前屏幕上已经生效的树，`workInProgress tree` 是本轮正在计算的新树，完成后两者会切换。原理：React 通过双缓存树结构保证正在显示的树和正在计算的树彼此隔离。场景：这样 render 阶段就算被打断，也不会把半成品直接暴露给用户。边界：真正上屏永远是 current 树，work-in-progress 只是候选结果。

96. React 更新时是如何从当前树构建新的 Fiber 树的？
答：结论：React 更新时会基于当前树克隆出可工作的 Fiber 节点，再一路向下做 reconciliation，生成新的 work-in-progress 树。原理：这样既保留了当前可用结果，又能在新树上安全计算下一版结果。场景：状态更新、props 变化、Context 变化都会触发这条链路。边界：不是全树都必然重建，满足 bailout 条件的子树可以直接跳过。

97. React 的 render 阶段和 commit 阶段分别在做什么？
答：结论：render 阶段负责计算变更，commit 阶段负责一次性把变更落到 DOM 和副作用。原理/场景：render 阶段负责计算，commit 阶段负责真正变更 DOM 和副作用，这是很多追问的基础。
延伸：render 可被打断，commit 要保证原子性。

98. 为什么 render 阶段可以被打断，而 commit 阶段不能？
答：结论：因为 render 只是算结果，打断还能重来；commit 已经要改真实界面了，必须保证原子性。原理/场景：commit 一旦开始就要保证界面一致性，所以不能像 render 一样随便中断再重来。

99. React 所说的“调度”到底在调度什么？
答：结论：本质上是在调度更新任务的执行顺序和时机，也就是先算谁、后算谁、能不能暂停。原理/场景：React 调度的其实是更新任务的优先级和执行时机，也就是先算谁、后算谁、能不能暂停。

100. React 为什么需要优先级？高优先级和低优先级任务分别有哪些典型场景？
答：结论：因为用户输入不能被低价值渲染拖慢，高优先级像点击输入，低优先级像大列表刷新和预渲染。原理/场景：高优先级常见是输入、点击，低优先级常见是大列表刷新、筛选结果和预渲染。

101. React 18 之后的 Lane 模型是怎么理解的？
答：结论：可以把 Lane 理解成更细粒度的优先级通道，不同更新会被分到不同车道里统一调度。原理/场景：Lane 可以理解成更细粒度的优先级通道，不同更新会分车道调度。
延伸：Lane 可类比“多车道优先级通道”，方便解释调度。

102. React 是如何实现“可中断渲染”的？
答：结论：核心就是把大任务拆成一个个 Fiber 工作单元，执行一会儿就检查要不要让出主线程。原理/场景：核心是把大任务拆成 Fiber 工作单元，执行一段就检查是否该把主线程还给浏览器。

103. React 如何把控制权还给浏览器？
答：结论：React 会在合适时机暂停自己的 render 工作，把主线程先还给浏览器去处理输入、绘制这些更紧急的事。原理：这正是时间切片和可中断渲染的核心。场景：复杂列表更新时，用户输入还能保持跟手，背后依赖的就是这种让权机制。边界：让权只发生在 render 阶段，commit 阶段仍要一次性完成。

104. React 调度为什么经常会和 `MessageChannel`、时间切片一起被提到？
答：结论：React 调度经常和 `MessageChannel`、时间切片一起被提到，是因为它们是浏览器里安排下一段任务和实现可中断更新的常见基础能力。原理：React 需要一种比 `setTimeout` 更稳定、开销更低的方式安排后续工作。场景：这让重计算可以被拆散执行，而不是一次性卡死主线程。边界：重点不是背某个 API，而是理解 React 为什么需要把长任务切开。

105. React 是怎么判断当前是否还有剩余空闲时间的？
答：结论：React 会根据一帧预算和当前时间判断是否该继续执行，快超时就先让出去。原理：浏览器一帧预算有限，如果继续算下去会挤压输入和绘制时机。场景：数据量大、组件树深时，这种时间判断能显著改善交互流畅度。边界：它判断的是是否继续 render，不是绝对保证每帧都不掉帧。

106. 面试里如果被问“React 怎么计算一帧里还能不能继续执行”，你会怎么回答？
答：结论：如果被问 React 怎么算一帧里还能不能继续执行，我会说它会记录本轮任务开始时间，再和当前时间比较，接近时间片上限就暂停。原理：核心目标是别把一整帧都占满，要给浏览器留出输入和绘制机会。场景：这类回答很适合和 Fiber、Lane、时间切片放在一条链里讲。边界：这里讲清思路就够了，不必死记具体毫秒实现细节。

107. React 中的高优先级任务通常会如何打断低优先级任务？
答：结论：当新的高优先级更新进来时，低优先级 render 可以中断，之后再从合适位置继续或重算。原理/场景：高优先级更新进入后，低优先级 render 可以被打断并在之后继续或重算，这是并发渲染的关键体验。

108. 为什么说 Fiber 的核心价值不只是“链表结构”，而是“可调度”？
答：结论：链表只是 Fiber 的一种实现形式，真正关键的是 React 因此获得了把更新拆小、可打断、可插队的调度能力。原理：如果只是把树改成链表却没有调度，React 依然解决不了大更新阻塞主线程的问题。场景：Fiber 真正带来的价值，是用户输入、动画和低优先级渲染不必再互相硬抢主线程，这才是它和旧栈式协调最大的差异。

109. 组件 setState 或 Hook 更新之后，是如何一路找到需要更新的 Fiber 的？
答：结论：更新会先挂到当前 Fiber，再一路向上标记路径，把优先级冒泡到根节点等待调度。原理/场景：更新会从当前 Fiber 往上冒泡到根节点，把优先级和待处理标记一路带上去。

110. React 如何决定某个 Fiber 节点是否需要继续向下更新？
答：结论：它会看类型、props、state、上下文有没有变化，满足可 bailout 条件就直接跳过子树。原理/场景：如果类型、props、state、上下文都没变，React 就会 bailout，避免无意义地下钻子树。

## 十二、React 19 与新特性

111. React 19 相比 React 18，最值得关注的新特性有哪些？
答：结论：我觉得最值得说的是 Actions、表单能力增强、`useOptimistic`、`useActionState` 和 `use()`。原理/场景：React 19 可重点讲 Actions、表单能力增强、`useOptimistic`、`useActionState` 和 `use()`。
延伸：如果项目还没上 React 19，也可以讲你理解的新特性价值和升级顾虑。

112. React 19 的 Actions 解决了什么问题？
答：结论：它把表单提交、异步状态、错误和 pending 这些常见流程收得更顺，少写很多样板代码。原理/场景：它把表单提交、pending、错误处理和异步结果更新串成更顺的一条链，减少手写样板。

113. `useActionState` 适合什么场景？
答：结论：`useActionState` 适合表单提交这类有返回状态、有成功失败反馈的异步交互场景。原理：它把提交函数、pending 状态和结果状态更紧密地绑定在一起。场景：登录、评论提交、资料修改这类表单流程会更顺手。边界：如果只是普通本地状态切换，没必要为了新 API 强行改写。

114. `useOptimistic` 适合什么场景？它和你自己手写乐观更新有什么区别？
答：结论：适合点赞、评论这类先给反馈再等结果的场景，它比手写更统一，也更贴近 React 状态流。原理/场景：`useOptimistic` 很适合点赞、评论、列表插入这类先反馈后确认的交互，但也要考虑失败回滚。

115. `useFormStatus` 解决了什么问题？
答：结论：`useFormStatus` 主要解决表单内部子组件拿提交状态不方便的问题，比如按钮 loading、禁用态这些。原理：它让表单上下文里的子组件能自然感知当前 action 是否在提交。场景：提交按钮、局部提示、提交中的禁用态都很适合。边界：它更适合表单上下文，不是通用全局 loading 方案。

116. `use()` 的核心作用是什么？它和 Suspense 的关系是什么？
答：结论：`use()` 可以直接读取 Promise 或 context，没准备好就交给 Suspense 挂起处理。原理/场景：`use()` 可以直接读取 Promise 或 context，没准备好就交给 Suspense 挂起处理，这是新心智点。
延伸：`use()` 往往会和 Suspense、Server Components 一起追问。

117. React 19 中 `ref as a prop` 是什么意思？
答：结论：就是 ref 传递这件事更自然了，很多场景下不必再专门绕一层 `forwardRef`。原理/场景：它让很多 ref 透传场景更自然，但升级时要关注第三方库和旧写法兼容。

118. React 19 对表单处理做了哪些增强？
答：结论：React 19 对表单处理的增强，核心就是把 action、pending、错误处理和乐观更新整合得更顺，让表单心智成本更低。原理：以前这些流程往往要自己手写一套状态机，现在框架帮你收拢了很多样板逻辑。场景：登录、评论、资料编辑、管理后台表单都会更容易落地。边界：它减少的是样板代码，不代表表单校验和业务规则可以不设计。

119. React 19 对资源加载和首屏优化提供了哪些能力？
答：结论：它增强了对资源预加载、样式脚本协调这类能力的支持，利于更早把关键资源送到浏览器。原理/场景：这类能力更偏首屏资源协调和关键资源提前送达，本质价值在于缩短关键资源的可见与可用路径。

120. React 19 升级时为什么强调新的 JSX Transform？
答：结论：React 19 升级时强调新的 JSX Transform，是因为一些新能力依赖新的编译产物和运行时约定，旧 transform 兼容不完整。原理：编译阶段决定了 JSX 最终生成什么运行时代码。场景：升级 React 版本时，构建链路和 Babel/TS 配置也必须一起检查。边界：不是只升 React 包版本就够，编译工具链也得跟上。

121. 从工程角度看，项目升级到 React 19 前需要重点检查哪些兼容风险？
答：结论：我会重点看依赖生态、构建链路、测试基线和历史用法，比如 ref、表单、第三方库兼容性。原理/场景：升级前要重点看依赖生态、构建链路、测试基线以及第三方库对新 API 的支持情况。
延伸：回答升级风险时，最好同时提测试和灰度策略。

122. 如果线上项目还没升级到 React 19，面试时该如何回答“你了解 React 19 吗”？
答：结论：我会诚实说项目还没落地，但我知道它重点解决哪些问题，也会结合现有项目讲适用场景和升级顾虑。原理/场景：如果项目还没升级，最好的回答方式是诚实说明现状，再讲你理解它解决了什么问题和为何暂未落地。
