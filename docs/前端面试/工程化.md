# 工程化训练题纲

## 使用方式

这个模块关注的不是会不会用某个工具，而是能不能把前端研发做成一套可持续交付体系。工程化答得成熟，通常要同时覆盖效率、质量、稳定性、治理和长期演进。

---

## 一、工程化基础认知

1. 什么是前端工程化？
答：前端工程化本质上是把“个人把页面写出来”升级成“团队能长期稳定交付产品”的一套体系，里面不仅有构建工具，还包括目录规范、类型系统、测试、发布、监控、权限边界、性能治理和协作流程。它的目标不是堆技术名词，而是让项目在规模变大、人数变多、迭代变快之后，依然能保持可开发、可回归、可发布、可维护。

2. 前端工程化主要解决哪些问题？
答：它主要解决的是多人协作效率、代码一致性、构建和发布稳定性、质量兜底以及长期维护成本，因为这些问题一旦规模变大，靠口头约定和人工流程很快就会失控。换句话说，工程化是在把“人肉经验”逐步变成“系统能力”，让团队不需要每次都从零重复踩坑。

3. 为什么说工程化是大厂前端的核心竞争力之一？
答：因为大厂前端真正面对的难点往往不是单个页面写不写得出来，而是复杂业务怎么稳、多人团队怎么协作、频繁发布怎么控风险、共性能力怎么复用，所以决定上限的通常是体系化交付能力。工程化做得好，最终会体现在更快的构建、更少的线上事故、更高的协作效率和更低的维护成本上，而不是只停留在“工具链看起来很高级”。

4. 工程化和业务开发的关系是什么？
答：工程化不是业务开发的对立面，而是业务能不能持续快跑的基础设施，如果工程化做得对，业务上线会更快、更稳，回归和排查也会更轻松。只有那种脱离实际需求、为了规范而规范的工程化，才会让团队觉得它是在增加负担。

5. 一个中大型前端项目通常需要哪些工程能力？
答：通常至少需要构建体系、代码规范、类型系统、测试体系、CI/CD、性能治理、监控告警、权限和配置治理，以及模块边界控制这些能力，因为它们共同决定了项目的效率、质量和稳定性。不同项目侧重点会不一样，但这几类能力基本构成了一个团队从“能开发”走向“能稳定交付”的基础盘。

## 二、构建工具与打包原理

6. Webpack 和 Vite 的核心区别是什么？
答：Webpack 更偏传统 bundle 思路，也就是开发阶段和生产阶段都围绕“先把模块打成一整套可运行产物”来组织，而 Vite 在开发阶段更多依赖浏览器原生 ESM，按需启动、按需转换，生产环境再交给 Rollup 做正式打包。它们的差异不只是快慢，而是开发链路设计理念不同，一个更偏“开发也以打包为中心”，一个更偏“开发优先追求反馈速度，生产再集中优化产物”。

7. 为什么 Vite 在开发环境通常更快？
答：因为它不需要启动时先把整个项目打成一个大 bundle，而是浏览器访问到哪个模块，就即时处理哪个模块，所以冷启动成本更低；同时第三方依赖可以预构建，HMR 也更接近模块级更新，热更新范围通常更小。简单理解就是，Vite 把很多原本“一上来全做完”的工作，拆成了“真正需要时再做”。

8. Webpack 的构建流程大致是什么？
答：可以概括成从入口文件出发构建依赖图，遇到不同类型资源先交给 `loader` 做模块转换，再在整个生命周期中通过 `plugin` 扩展能力，最后把依赖图输出成若干 chunk、bundle 和其他静态资源。理解这条流程的意义在于，你会知道 Webpack 本质上不是“一个黑盒打包器”，而是“依赖图构建 + 生命周期扩展”的组合。

9. 什么是 Loader？什么是 Plugin？
答：`Loader` 主要处理单个模块的转换问题，比如把 TypeScript、SCSS、图片、MD 文件变成 Webpack 能继续识别的模块；`Plugin` 则更偏整个构建流程的扩展，比如注入 HTML、定义环境变量、做压缩、拆包、分析产物等。简单说，Loader 更像“把资源转成可处理输入”，Plugin 更像“在构建流程里插入额外能力”。

10. Babel 在前端工程里扮演什么角色？
答：Babel 的核心职责是语法转换和兼容性降级，也就是把开发期更现代的 JavaScript 写法转成目标运行环境能理解的代码。它不是打包器，但经常和打包器配合使用，所以它解决的是“代码能否在目标环境执行”的问题，而不是“代码如何被组织成产物”的问题。

11. Tree Shaking 的前提条件是什么？
答：核心前提是模块必须能被静态分析，最典型的就是 ESM，而且代码副作用要足够可判断，否则构建工具不敢随便删。也就是说，Tree Shaking 不是一句“用了 import 就能摇掉无用代码”，它还依赖模块写法、`sideEffects` 标记和代码是否纯净。

12. Code Splitting 是怎么做的？
答：本质上就是把原本会一起加载的代码拆成多个更小的加载单元，常见方式有多入口、动态 `import()`、按路由拆、按功能块拆，再配合缓存和预加载策略让用户只先拿到当前路径需要的那部分资源。它的真正目标不是“把包切碎”，而是把不同访问路径的加载成本拆开，减少首屏压力。

13. 什么是 Chunk、Bundle、Asset？
答：`Chunk` 更偏构建阶段的逻辑代码块，是依赖图在打包过程中的组织单元；`Bundle` 通常指最终输出的某个 JS 打包结果；`Asset` 则更广，表示所有最终产物，包括 JS、CSS、图片、字体等。很多人会混用这些词，但更准确的理解是：Chunk 偏中间结构，Bundle 和其他 Asset 偏最终落地文件。

14. 为什么包体积会越来越大？
答：常见原因包括依赖不断叠加、公共代码膨胀、按需加载没做好、重复依赖没有去重、历史功能删除后产物却没清干净，以及团队缺少长期包体积治理。包越来越大往往不是某一天突然爆炸，而是在没有指标和约束的情况下慢慢堆出来的。

15. 如何分析打包产物？
答：我通常会先看可视化分析报告，定位超大依赖、重复模块、异常公共 chunk 和首屏强依赖资源，再结合路由访问路径和用户实际加载链路判断哪些内容不应该进首包。分析产物的重点不是“把图看懂”，而是找到为什么大、哪里该拆、哪些依赖是否值得保留。

16. 如何做包体积优化？
答：常见手段包括替换重依赖、细粒度引入、做代码拆分、消除重复依赖、删除无用 polyfill、压缩资源以及优化缓存策略，但真正成熟的做法一定会落到数据和机制上，比如优化前后首包大小、`LCP` 或首屏时长变化，以及后续怎么防止体积继续反弹。因为包体积优化本质上不是一次动作，而是持续治理。

17. 如何做按需加载？
答：我会从路由级、组件级和重型第三方能力三级去看，凡是不在首屏、不在高频路径、或者明显属于低频功能的内容，都尽量不要在初始加载里出现。按需加载的关键不是越碎越好，而是平衡首包体积、请求数量、缓存命中和用户真实访问路径。

18. 如何处理第三方库过大的问题？
答：先判断它是不是真的必要，再看是否能被更轻量替代、是否支持按需引入、是否适合拆到异步 chunk、是否能通过服务端或外部化方案减轻前端负担。像编辑器、图表、富文本、AI SDK 这类又大又低频的依赖，通常更适合懒加载隔离，而不是直接塞进首包。

## 三、项目结构与模块设计

19. 中大型 React 项目目录通常怎么设计？
答：我通常会先按页面、通用组件、Hooks、服务层、状态层、工具层、类型层做基础分层，再根据业务复杂度判断是否需要按领域做更清晰的业务切片。目录设计不是为了看起来整齐，而是为了让职责明确、依赖方向稳定，让后来接手的人能快速判断一段代码“应该放哪、应该依赖谁”。

20. 页面、组件、Hooks、服务、工具、常量如何分层？
答：页面层主要负责组装和编排，组件层负责展示和局部交互，Hook 层沉淀可复用的视图逻辑，服务层统一接口访问和数据转换，工具层尽量放纯函数，常量层则保存稳定配置或枚举信息。真正重要的不是名词本身，而是依赖方向要清楚，避免页面逻辑反向渗透到底层工具和基础组件里。

21. 为什么要做模块边界控制？
答：因为项目一大以后，最先失控的往往不是代码量，而是依赖穿透和职责污染，比如页面直接调底层实现、组件偷偷依赖服务层、工具层混进业务状态，一旦边界混乱，改一处就容易牵一片。边界控制的本质是在降低认知负担和修改影响面，让系统能随着规模增长仍然可维护。

22. 你如何避免“utils 大杂烩”？
答：我的做法是先区分纯工具和业务工具，再按领域拆分，只有真正稳定、无业务上下文、复用价值高的内容才放到基础工具层。因为一旦 `utils` 变成“所有不知道放哪里的代码都先塞进去”的垃圾桶，后续几乎一定会出现依赖混乱、命名混乱和难以治理的问题。

23. UI 组件和业务组件如何区分？
答：UI 组件更强调通用性、语义稳定和业务无关，比如 Button、Input、Modal 这类基础能力；业务组件则直接承载领域规则，比如订单状态条、审批节点卡片、Agent 流程节点面板。把这两类分清，才能避免把业务耦合误抽进组件库，也能避免基础组件被业务特例不断污染。

24. 为什么服务层抽象很重要？
答：因为如果接口请求、参数组装、响应解包、错误映射和数据转换都散落在页面里，页面很快就会变成又长又难测的混合体。服务层的价值在于把“数据从哪里来、怎么变成业务可用结构”统一收口，让页面把精力更多放在交互编排和状态呈现上。

25. 公共能力沉淀时要避免哪些问题？
答：最常见的问题是过早抽象、为了复用而复用，以及抽出一个参数爆炸、谁都看不懂的万能组件。更稳妥的做法通常是先让局部方案在真实业务里跑稳，等共性足够稳定再提炼，否则你以为在做基础设施，实际可能只是在把未来的修改成本提前埋好。

26. 单仓和多包仓库分别适合什么场景？
答：单仓更适合应用数量不多、团队规模相对可控、变更大多集中在一个产品里的场景，心智简单、管理成本低；多包或 Monorepo 更适合多个应用共享组件库、设计系统、工程能力和基础包的团队，因为它更方便做联动改动和统一治理。真正的选型关键不是潮流，而是代码共享关系和协作复杂度。

27. Monorepo 的优点和代价是什么？
答：优点是代码复用更自然、规范更统一、跨包联调和联动修改效率更高，特别适合共享基础设施明显的团队；代价是仓库治理、缓存、权限、构建、发布和依赖边界都会更复杂。也就是说，Monorepo 的难点从来不是搭起来，而是长期能不能把依赖规则、缓存策略和发布责任治理住。

## 四、代码规范与质量保障

28. 为什么前端项目必须接入 ESLint？
答：因为很多潜在问题靠 code review 很难稳定拦住，比如危险写法、未处理分支、依赖遗漏、React Hooks 误用、隐式 bug 等，而 ESLint 可以把这类高频风险前置到开发和提交阶段。它真正解决的不是格式漂不漂亮，而是让一些低级问题在代码进主干前就被自动发现。

29. Prettier 和 ESLint 的职责有什么区别？
答：Prettier 主要解决格式一致性问题，比如缩进、引号、换行和逗号风格，目的是减少团队在低价值格式问题上的争论；ESLint 更偏代码质量和规范约束，用规则去发现 bug 风险和不推荐写法。理想状态是格式交给 Prettier 自动处理，质量和规范交给 ESLint 卡住，各司其职。

30. TypeScript 在工程质量上带来了什么价值？
答：它最大的价值是把很多原本只能在运行时暴露的问题，提前到编译阶段，比如接口字段不匹配、组件契约不清、状态类型漂移、重构改漏问题。它不是测试的替代品，但会和类型、测试、Lint 一起形成多层质量兜底，让多人协作和长期演进更可控。

31. 如何设计一套适合团队的代码规范？
答：我会优先保证规范统一、收益清晰、落地成本低，再逐步收紧，而不是一上来堆满一大堆没人愿意遵守的规则。一个实用原则是，能自动修复的尽量自动修复，必须人工判断的才交给 review，这样团队对规范的阻力会小很多。

32. Git 提交规范为什么重要？
答：因为提交信息本质上是变更历史的索引，它影响问题回溯、自动生成 changelog、自动发版和团队沟通效率，而不仅是“commit 好不好看”。当团队规模变大、需求和修复并行推进时，规范的提交信息会显著提升变更可追踪性。

33. Commitlint、Husky、lint-staged 分别是做什么的？
答：`Commitlint` 负责校验提交信息是否符合约定，`Husky` 用来挂 Git Hooks，把检查挂到 commit 或 push 等时机，`lint-staged` 则只处理暂存区文件，避免每次都全量跑检查。它们组合起来的价值，是把规范从“文档建议”变成“流程约束”。

34. 为什么代码审查同样属于工程化能力？
答：因为工程化不只是自动化工具，也包括“团队如何稳定发现风险和统一质量标准”，而代码审查正是这个过程的重要一环。它既是风险控制，也是知识传递和团队认知对齐，所以一个成熟团队的工程能力，通常离不开清晰的 review 机制。

35. 如何降低多人协作中的代码风格冲突？
答：最有效的方法通常不是多开会，而是把格式和基础规范自动化，比如统一 Prettier、ESLint、编辑器配置和提交前检查，让大多数风格问题在提交前就已经自动被修好。只靠口头约定很难持续，因为团队越大，风格分歧放大得越快。

## 五、测试体系

36. 为什么很多前端项目做不好测试？
答：常见根因不是不知道测试重要，而是测试边界不清、业务节奏快、历史代码可测性差、测试收益没有及时显现，最后总被排在后面。很多测试问题本质上都是工程治理问题，也就是没有把测试纳入日常开发和发布流程里。

37. 单元测试、集成测试、端到端测试的区别是什么？
答：单元测试关注局部逻辑正确性，适合守稳定规则；集成测试关注多个模块协作后的局部业务流程；E2E 则站在真实用户视角验证整条链路是否可用。它们不是替代关系，而是分层承担不同成本和不同范围的风险。

38. 哪些前端逻辑适合做单元测试？
答：纯函数、数据转换、权限判断、复杂计算、表单规则、状态机这类边界清晰、输入输出明确的逻辑最适合做单测，因为它们运行快、定位准、维护成本低。单测最怕的反而是和页面结构、异步时序、环境依赖绑得太紧。

39. 哪些场景更适合集成测试？
答：像表单提交流程、组件联动、状态管理和请求配合、路由切换、局部权限判断这类问题，更适合集成测试，因为它们的风险来自模块协作，而不是单个函数本身。它比单测更接近真实使用，又没有 E2E 那么重，是很多业务系统里性价比很高的一层。

40. E2E 测试通常覆盖哪些关键链路？
答：我一般会优先覆盖登录、支付、上传、审批、核心提交、权限流转等“不能坏”的主链路，因为这类流程跨多个模块和环境，一旦出问题业务损失也最大。E2E 的重点不是刷覆盖率，而是守住高损失、高回归风险路径。

41. React 项目里常见测试工具有哪些？
答：比较常见的组合是 `Vitest` 或 `Jest` 作为测试运行器，`React Testing Library` 负责组件行为验证，`Playwright` 或 `Cypress` 负责 E2E。更重要的不是把工具名背全，而是知道它们分别解决执行环境、组件行为和真实链路这三个不同层级的问题。

42. 测试覆盖率高就一定质量高吗？
答：不一定，因为覆盖率只说明代码被跑到过，不说明关键风险真的被守住，比如大量低价值快照也能把数字堆得很好看。比起单独盯覆盖率，更有价值的是看关键路径有没有测试、失败时能不能真正阻断回归。

43. 如何选择最有价值的测试范围？
答：我会优先选择高频、高风险、容易回归、出问题影响大的部分，因为测试资源永远有限，先守住最不能坏的地方才最划算。测试范围选择本质上是在做风险排序，而不是平均撒网。

44. 在后台系统项目里，你会优先测哪些功能？
答：我通常会优先测权限、复杂表单、列表筛选、导入导出、关键配置流程和审批流，因为这些模块规则多、分支多、改动频繁，出了问题业务影响也很直接。后台系统测试的核心常常不是“看起来像不像”，而是流程和边界有没有被守住。

## 六、CI/CD 与发布流程

45. 为什么前端项目需要 CI/CD？
答：因为人工构建、人工测试、人工发布太依赖个体经验，稳定性差、效率也差，尤其团队人数一多、发布频率一高，这种模式很快就会成为瓶颈。CI/CD 的核心价值是让交付流程标准化、可重复、可回溯，而不是单纯为了“自动化看起来先进”。

46. 一条典型的前端 CI 流水线包含哪些步骤？
答：通常会包含依赖安装、缓存恢复、静态检查、类型检查、测试执行、构建产物、制品上传和部署这些步骤，更成熟一些的流程还会加入安全扫描、性能检查、灰度发布和回滚能力。理解这个流程的关键，是把它看成“质量门禁 + 交付链路”，而不是一串机械命令。

47. 自动化构建、测试、部署分别在解决什么问题？
答：自动化构建主要解决环境一致性和产物可重复问题，自动化测试负责把质量检查前置，自动化部署则降低人工失误并提升上线效率。三者串起来，团队才真正拥有一条可持续交付链路，而不是每次上线都靠经验手工拼装。

48. 如何降低发布风险？
答：成熟做法通常不是依赖某一个按钮，而是多层兜底一起上，比如小步快跑、自动检查、灰度放量、监控告警、开关控制和回滚策略。发布风险真正难的地方在于上线前要有门禁，上线中要可观测，上线后要能止损，而不是出问题后大家现场想办法。

49. 灰度发布和全量发布的区别是什么？
答：灰度发布是先让一部分用户或流量接触新版本，用真实环境逐步验证稳定性，然后再扩大范围；全量发布则是一把切过去。灰度的真正价值不是“发布慢一点”，而是给问题发现和快速止损留出观察窗口。

50. 回滚机制为什么重要？
答：因为线上出问题时，第一目标往往不是立刻修好，而是先止损，如果没有可靠回滚，你每次发布都相当于把风险暴露在全量用户前。一个没有回滚手段的发布体系，本质上是不完整的。

51. 多环境配置通常怎么设计？
答：我会把环境变量、构建配置和部署配置分层管理，明确本地、测试、预发、生产的来源和差异，并尽量让配置可追踪、可审计、可复现。好的多环境设计要解决的不是“变量很多怎么记”，而是如何避免串环境、配置漂移和本地线上行为不一致。

52. 环境变量管理有哪些常见风险？
答：常见风险包括串环境、密钥泄露、默认值不一致、无人治理导致配置膨胀，以及本地和线上执行行为不一致。尤其前端项目里一个高频误区是以为写进 `.env` 就安全了，但只要变量最终进了浏览器产物，它本质上就是公开信息。

53. 前端项目如何避免把敏感信息打进包里？
答：核心原则是前端不保管真正的秘密，能放在服务端、网关或 BFF 处理的绝不放到浏览器构建产物里。对于必须暴露给前端的配置，也要明确它只是公开配置，而不是安全边界，比如接口域名可以公开，但私钥、内部 token、数据库地址绝对不该出现在前端包里。

## 七、性能工程化

54. 性能优化为什么也属于工程化？
答：因为真正有价值的性能优化不是偶尔凭经验调一把，而是持续测量、定位、治理、回归和防反弹的过程，只有它进入指标体系、流水线和监控链路，才算工程能力。否则再漂亮的单次优化，也很容易在后续迭代里被一点点吃回去。

55. 如何建立性能基线？
答：我会先选关键页面、关键链路和关键指标，比如 `LCP`、`TTFB`、首包大小、交互时延，再记录当前值作为基线，因为没有对照数据，就很难判断后续优化到底是真提升还是心理安慰。性能基线本质上是在给团队建立一个“优化前后能比较”的公共坐标系。

56. 如何持续监控页面性能？
答：除了实验室数据，我更看重真实用户监控，也就是 Web Vitals、接口耗时、首屏时间、长任务、错误率和关键交互时延这些线上数据，因为它们更接近真实用户体验。持续监控关注的重点是趋势和回归，而不是某一次 Lighthouse 分数好不好看。

57. 为什么要把性能指标接入 CI 或监控体系？
答：因为性能问题非常容易反弹，尤其业务持续迭代时，包体积、关键渲染路径和交互延迟会在不知不觉中变差。只有把关键指标接入发布前门禁或线上告警，团队才会在问题再次出现时第一时间知道，而不是等用户抱怨后才发现。

58. 如何做构建速度优化？
答：常见方向包括缓存、并行、缩小编译范围、减少无效插件、优化依赖、升级工具链和减少重复工作，但真正关键的是先定位瓶颈，确认时间到底耗在依赖安装、代码转换、类型检查还是打包优化上。构建速度优化最怕的就是不做分析、只凭感觉频繁换工具。

59. 如何做运行时性能优化？
答：我会从资源加载、渲染频率、长任务、请求链路、缓存策略和关键交互路径几个方向去看，因为运行时性能很多时候不是包大不大，而是首屏是不是被阻塞、交互是不是卡、状态更新是不是引发大量无效渲染。也就是说，运行时优化不能只盯构建产物，还要回到用户真实使用路径。

60. 前端性能优化有哪些“看起来有效但收益很低”的做法？
答：比如不看瓶颈就到处加 `memo`、只盯包体积不看运行时卡顿、在极小收益的微优化上花大量时间，这些都很常见。成熟的性能优化应该是数据先行、先抓主要矛盾，而不是“我感觉这样应该更快”。

61. 如何向面试官展示一次真正有数据的性能优化？
答：我会按“问题现象、定位方法、根因判断、改动方案、优化前后数据、如何防止回退”来讲，这样既说明你会做分析，也说明你不是只会列一串优化动作。数据最好尽量具体，比如首包下降多少、`LCP` 改善多少、构建时间缩短多少，这样说服力会强很多。

## 八、稳定性与可观测性

62. 什么是前端可观测性？
答：前端可观测性可以理解成让线上问题“看得见、查得到、串得起来”，也就是出了问题后不仅知道发生了什么，还能尽快知道在哪个版本、哪个页面、什么设备、什么操作链路下发生。它不只是报错采集，而是把错误、性能、资源、接口和用户行为放到同一个观察体系里。

63. 前端监控通常包含哪些维度？
答：常见维度包括 JS 报错、资源加载失败、接口异常、性能指标、白屏和卡顿、用户行为埋点以及页面可用性指标。真正有价值的监控不是每个维度各自一堆日志，而是能把它们关联起来，帮助你快速还原用户现场。

64. JS 报错、资源加载失败、接口异常、白屏、卡顿分别怎么监控？
答：它们的采集方式会不同，比如运行时异常可以抓全局错误和 Promise 拒绝，资源失败靠资源加载事件，接口异常看请求层埋点，白屏和卡顿则更依赖性能和页面可用性信号，但核心要求是一致的，就是统一上报、带足上下文、按版本和页面可追踪。真正的难点不在“能不能采到”，而在“采到以后能不能用于定位”。

65. 如何做统一错误处理？
答：我会把接口错误、运行时异常和业务提示分层处理，明确哪些需要上报，哪些需要用户可见提示，哪些需要兜底页面或重试策略，避免每个页面各写各的处理逻辑。统一错误处理的价值，是同时提升用户体验一致性和问题排查效率。

66. 错误上报系统要包含哪些关键信息？
答：至少要包含用户标识、时间、页面路由、应用版本、设备环境、错误堆栈、接口上下文和操作链路，因为只有这些信息串起来，报错才有机会被真正复现和定位。否则你拿到的只是一条孤立错误信息，知道它错了，却不知道为什么错。

67. 用户反馈“页面卡死”时，你如何结合监控排查？
答：我会先看问题是否集中在某个版本、某个页面或某类设备上，再结合长任务、接口耗时、资源失败、错误峰值和用户行为路径逐步缩小范围。卡死类问题通常不是单点信号就能看明白，而是需要从多个监控维度一起交叉判断。

68. 如何设计日志分级和降噪？
答：我会区分 `info`、`warn`、`error` 等不同等级，对重复、低价值、可预期的日志做采样、限流或聚合，避免监控系统被大量无效噪音淹没。因为监控系统最怕的不是没日志，而是有太多没用的日志把真正的问题盖掉了。

69. 为什么没有监控的项目很难称得上成熟？
答：因为没有监控意味着出了问题看不见、量不到、追不回，团队无法建立从发布到发现问题再到止损和复盘的完整闭环。一个不能自我观察和快速定位故障的系统，很难说真正具备工程稳定性。

## 九、权限、配置与业务工程

70. 后台系统的权限控制为什么是工程问题而不只是业务问题？
答：因为权限会同时影响菜单、路由、按钮、接口、配置和页面骨架，如果没有统一设计，就会出现“页面显示一套、路由拦截一套、接口校验又一套”的混乱局面。它之所以是工程问题，是因为要靠统一模型、统一协议和统一流程把这些入口收口，而不是在页面里到处写几个 `if`。

71. 菜单、路由、按钮权限怎么统一设计？
答：更成熟的做法通常是基于同一份权限元数据去映射菜单显示、路由守卫和按钮显隐，让所有前端入口都围绕同一套规则来源展开。这样既能减少重复维护，也更方便做回归测试和权限问题排查。

72. 动态路由和静态路由分别适合什么场景？
答：静态路由更适合结构稳定、页面关系明确的系统，因为它简单直观、可维护性高；动态路由更适合权限差异大、租户差异大、菜单和页面结构受配置驱动明显的后台系统。选型关键不在于谁更炫，而在于你的业务是否真的需要用更高复杂度换取更高灵活性。

73. 为什么配置驱动能提升后台系统可维护性？
答：因为后台很多页面其实共享相似骨架，比如筛选区、表格、分页、表单项、操作按钮和权限显隐，如果这些共性都能通过配置描述，重复开发会明显减少。它的边界也很重要，不是所有复杂业务都应该强行配置化，否则配置会变得比代码还难维护。

74. 如何设计一个通用表格页或表单页框架？
答：我通常会先抽稳定骨架和扩展协议，把筛选区、查询、表格、分页、空态、加载态、权限和操作区这些共性能力固化下来，再把真正强业务的差异留给插槽、配置或领域组件扩展。重点是抽稳定流程，而不是试图把所有页面硬压成一个万能模板。

75. 如何封装请求、鉴权、错误提示、空态、加载态？
答：这些能力更适合沉到基础设施层统一处理，比如统一请求实例、鉴权注入、错误映射、Loading 和 Empty 的默认模式，让页面只关心自己的业务差异。这样不仅能减少重复代码，也能让整个系统的交互体验和异常表现更一致。

76. 为什么后台系统常需要“约定优于配置”？
答：因为后台页面量大、协作人数多，如果每个人都自由发挥，长期维护成本会迅速上升，而适度约定可以显著降低心智负担，让常见页面和常见流程都按统一模式开发。它的目标不是限制灵活性，而是把高频场景做成默认正确。

## 十、综合与实战

77. 如果让你从零搭一个后台前端工程，你会先做哪几件事？
答：我通常会先定技术栈和包管理方案，再明确目录边界、请求层、权限层、路由层、状态层这些基础设施，然后补代码规范、测试和 CI/CD 流程。顺序上更重要的是先让团队能稳定协作和交付，再逐步完善高级治理，而不是一开始就追求最复杂的架构形态。

78. 如果项目越来越大，你如何判断需要做架构治理？
答：当你开始频繁遇到改一处牵很多处、多人协作冲突明显增多、构建越来越慢、发布风险升高、公共能力重复建设、边界越来越模糊时，就说明项目进入了治理窗口。治理不是因为“想重构”才做，而是因为继续放任不管，成本已经开始明显高过治理本身。

79. 如何向面试官说明你具备工程化意识而不只是会用工具？
答：关键不在于列一个工具清单，而在于讲清楚为什么选、解决了什么问题、带来了哪些收益、有什么取舍、结果如何衡量。能从“问题 -> 方案 -> 落地 -> 数据 -> 机制沉淀”完整讲一遍，通常比单纯说自己会 Webpack、Vite、CI 更能体现工程化意识。

80. 如果让你讲一次前端工程化优化经历，你会怎么组织？
答：我会按背景问题、现状痛点、方案设计、推进阻力、落地过程、结果数据和后续机制来组织，因为工程化的重点从来不是工具本身，而是问题如何被闭环解决。特别是最好补上“为什么这么选”和“怎么防止后面再回退”，这样才不像一次性的临时修补。

81. 在字节这类公司面试中，工程化模块最容易被追问什么？
答：最容易被追问的通常是选型依据、架构边界、上线兜底、数据指标和失败经验，因为面试官更想知道你有没有把工程问题真正做成闭环，而不是只会复述术语。也就是说，越到高阶，越看重你对取舍和结果的理解。

## 十一、Vite、发布与项目取舍

82. Vite 的按需加载在开发环境和生产环境分别是怎么工作的？
答：在开发环境里，Vite 主要依赖浏览器原生 ESM，访问到哪个模块再即时转换和返回哪个模块，所以开发期是“按需请求、按需处理”；而到了生产环境，它仍然会把代码交给 Rollup 做正式打包、压缩和优化。两阶段设计的本质是开发优先追求反馈速度，生产优先追求加载效率。

83. 浏览器在 Vite 开发环境里遇到 `import` 时，是否每次都会重新发起网络请求？
答：不一定，因为浏览器缓存、Vite 的模块缓存和 HMR 机制都会参与，所以首次请求模块后，后续是否重新拉取取决于缓存状态和文件是否变化。更准确的理解是，开发期模块请求更细粒度，但并不代表每次 import 都从零开始。

84. 为什么说 Vite 的快不只是“没打包”，还和原生 ES Module、预构建有关？
答：因为它真正快的原因是整条开发链路被缩短了，业务源码按模块即时处理，第三方依赖提前预构建成浏览器更容易消费的形式，热更新也尽量只影响局部边界。只说“因为没打包所以快”太粗糙，少了原生 ESM 和依赖预构建这两个关键环节。

85. 如果依赖很多，Vite 为什么仍然需要依赖预构建？
答：因为第三方依赖经常存在格式不统一、层级深、包数量多的问题，如果开发期原样让浏览器逐个解析，模块请求和解析开销会很高。预构建本质上是在提前整理第三方依赖，让浏览器开发期消费它们时更轻、更快、更稳定，而不是在优化你的业务源码。

86. 除了 Lighthouse 和 Performance 面板，你还会用哪些指标或方式评估性能优化效果？
答：我还会结合 Web Vitals、真实用户监控、错误率、接口耗时、关键交互时延和版本回归趋势来看，因为实验室数据更适合定位问题，真实用户数据才更适合判断优化是否真的改变了用户体验。面试里如果能把“实验室指标”和“线上真实指标”区分开，回答会更完整。

87. 如果一个前端项目需要灰度发布，你会从前端、网关、配置和监控四个层面怎么设计？
答：前端层负责版本识别、能力兼容和降级兜底，网关层负责流量切分和放量策略，配置中心负责开关和策略控制，监控层则负责观测关键指标并在异常时触发告警和回滚。灰度的重点是“逐步放量 + 可观测 + 可止损”，如果系统更复杂，还要补充接口兼容、静态资源缓存失效和后端数据变更是否可回退这些问题。

88. 为什么一些项目会同时使用 `Node.js` 和 `Python`？从工程协作角度如何划分边界？
答：这很常见，因为不同语言适合的能力边界不同，比如前端工程、SSR/BFF、实时网关更常见用 Node，算法、数据处理、训练任务或脚本平台更常见用 Python。工程协作角度更重要的不是“语言尽量统一”，而是接口契约、部署边界、责任归属和调用链路是否清晰。

89. `npm`、`pnpm`、`yarn` 有什么区别？为什么启动命令看起来类似但又不完全一样？
答：它们本质上都是 JavaScript 包管理工具，负责安装依赖、维护锁文件和执行脚本，但在依赖安装机制、磁盘占用、安装速度、Monorepo 支持和团队心智上存在差异。`npm` 是官方默认方案，通用性强、上手成本低；`yarn` 早期因为更快和工作区能力强而很流行，现在更多价值体现在团队历史沉淀；`pnpm` 通过内容寻址和链接复用依赖，更省磁盘、对 Monorepo 也更友好，还能更严格地减少幽灵依赖。命令看起来相似，是因为它们都要做安装和执行脚本这些核心动作，但设计细节不同，比如 `npm` 通常写 `npm run dev`，而 `pnpm`、`yarn` 常可直接写 `pnpm dev`、`yarn dev`。实际工程里该怎么选，往往不是看谁“最先进”，而是看团队现有流程、仓库结构、成员习惯和是否需要更强的 Monorepo 能力，真正决定项目能不能稳定运行的，也始终是锁文件、依赖安装是否一致以及团队是否统一使用同一种包管理工具。
