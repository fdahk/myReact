# 工程化训练题纲

## 使用方式

这个模块关注的不是会不会用某个工具，而是能不能把前端研发做成一套可持续交付体系。工程化答得成熟，通常要同时覆盖效率、质量、稳定性、治理和长期演进。

---

## 一、工程化基础认知

1. 什么是前端工程化？
答：前端工程化本质上是把“个人写页面”升级成“团队持续交付产品”的一整套机制，涵盖开发、构建、测试、发布、监控和治理。它的目标不是堆工具，而是让项目在规模变大后依然可协作、可回归、可发布。

2. 前端工程化主要解决哪些问题？
答：它主要解决协作效率、代码一致性、构建发布稳定性、质量兜底和长期维护成本。项目越大、角色越多、发布越频繁，这些问题越不能靠人肉处理。

3. 为什么说工程化是大厂前端的核心竞争力之一？
答：因为大厂前端比拼的往往不是“会不会写页面”，而是能不能在复杂业务里稳定交付、快速回归、低成本扩张。真正拉开差距的通常是体系化能力，而不是单点编码速度。
延伸：判断工程化是否真正有效，最终要回到结果指标，比如构建时间是否下降、线上故障是否减少、多人协作冲突是否收敛、发布效率是否提升。

4. 工程化和业务开发的关系是什么？
答：工程化不是业务的对立面，而是业务交付效率和稳定性的基础设施。做得好的工程化应该让业务更快上线，而不是把团队拖进“为了规范而规范”。

5. 一个中大型前端项目通常需要哪些工程能力？
答：通常至少要有构建体系、代码规范、类型系统、测试体系、CI/CD、性能治理、监控告警和架构边界治理。不同项目重点不同，但这些能力最终都服务于效率、质量和稳定性。

## 二、构建工具与打包原理

6. Webpack 和 Vite 的核心区别是什么？
答：Webpack 更偏“先打包再开发”的传统思路，开发期也围绕 bundle 工作；Vite 则在开发阶段更多依赖浏览器原生 ESM，按需启动和按需编译，生产环境再交给 Rollup 打包优化。两者差异不只是快慢，而是开发链路设计理念不同。
延伸：Vite 并不是完全没有打包，而是把“开发时整包构建”改成了“生产时再打包”。

7. 为什么 Vite 在开发环境通常更快？
答：因为它不需要每次启动都把整个应用重新打成 bundle，而是按访问路径和模块依赖即时转换。再加上依赖预构建、HMR 边界更细，所以冷启动和热更新通常都更快。

8. Webpack 的构建流程大致是什么？
答：我会概括成四步：从入口出发构建依赖图，`loader` 负责把不同资源转成可处理模块，`plugin` 在生命周期钩子里扩展能力，最后输出 chunk、bundle 和静态资源。更完整地说，它的核心就是围绕依赖图构建和生命周期扩展来组织整个打包过程。

9. 什么是 Loader？什么是 Plugin？
答：`Loader` 面向单个模块转换，比如把 `ts`、`scss`、图片等资源转成 Webpack 能继续处理的模块；`Plugin` 面向整个构建过程扩展，比如注入环境变量、拆包、生成 HTML、分析产物。简单说就是一个偏“转换”，一个偏“扩展流程”。

10. Babel 在前端工程里扮演什么角色？
答：Babel 主要负责语法转换和兼容性降级，让现代语法能运行在目标环境里。它不是打包器，但常和打包器配合，承担“把开发期写法转成运行期可执行代码”的职责。

11. Tree Shaking 的前提条件是什么？
答：核心前提是基于 ESM 做静态分析，并且代码的副作用要可判断。也就是说，不是写了 `import` 就一定能摇掉，模块格式、`sideEffects` 标记、写法是否纯净都会影响结果。

12. Code Splitting 是怎么做的？
答：常见做法是多入口拆分、动态 `import()`、按路由或功能块切 chunk，再结合缓存策略让用户只加载当前需要的资源。它不只是“把包拆小”，而是把不同访问路径的加载成本分摊开。

13. 什么是 Chunk、Bundle、Asset？
答：`Chunk` 更像构建过程里的逻辑代码块，`Bundle` 是最终输出的 JS 打包结果之一，`Asset` 则是所有产物的统称，包括 JS、CSS、图片、字体等。它们的关键区别在于：`Chunk` 偏构建阶段的中间组织单元，`Bundle` 和其他 `Asset` 偏最终落地到线上环境的产物。

14. 为什么包体积会越来越大？
答：常见原因是依赖越堆越多、公共代码不断膨胀、按需加载没做好、重复依赖没收敛，以及历史功能下线后资源没及时清理。包体积变大往往不是一次事故，而是长期缺少治理的结果。

15. 如何分析打包产物？
答：我一般先看可视化分析报告，定位大依赖、重复模块、异常公共 chunk、首屏强依赖资源，再结合路由访问链路看哪些代码其实不该进首包。分析产物的目标不是看图，而是为后续拆包和减依赖提供依据。

16. 如何做包体积优化？
答：常用手段包括替换重依赖、细粒度引入、代码拆分、去重、删除无用 polyfill、资源压缩和缓存优化。真正有效的优化往往不是一次改完，而是建立持续观察和回归机制。
延伸：包体积优化不能只报动作，最好能落到数据上，比如首包从多少降到多少、`LCP` 改善多少、首屏时间缩短多少。

17. 如何做按需加载？
答：常见做法是从路由级、组件级、第三方库级三层做动态加载，核心原则是谁不在首屏、谁不是高频路径，就不要进首包。按需加载不是越碎越好，还要平衡请求数、缓存命中和用户访问路径。

18. 如何处理第三方库过大的问题？
答：我会先判断它是否真的必要，再看能否替换、按需引入、外部化或拆进异步 chunk。对特别重但低频的能力，比如编辑器、图表、富文本、AI SDK，通常更适合懒加载隔离。

## 三、项目结构与模块设计

19. 中大型 React 项目目录通常怎么设计？
答：我会按业务页面、通用组件、Hook、服务层、状态层、工具层、类型层做基础分层，再根据业务复杂度决定是否按领域拆分。目录设计的目标不是“看起来整齐”，而是让代码职责清晰、依赖方向稳定。

20. 页面、组件、Hooks、服务、工具、常量如何分层？
答：页面负责组装和编排，组件负责展示和局部交互，Hook 复用视图逻辑，服务层统一数据获取和数据转换，工具层放纯函数，常量层只放稳定配置。分层的关键是控制依赖方向，避免页面逻辑反向渗透到基础层。

21. 为什么要做模块边界控制？
答：因为项目一大，最先失控的往往不是代码量，而是依赖穿透。没有边界时，页面可以直接调任何东西、组件能偷偷耦合服务层，最后改一处牵一片，重构几乎不敢动。

22. 你如何避免“utils 大杂烩”？
答：我会先区分纯工具和业务工具，再按领域拆分，而不是所有“暂时不知道放哪”的代码都扔进 `utils`。一旦工具文件开始变成垃圾桶，项目后面就很难再收回来。

23. UI 组件和业务组件如何区分？
答：UI 组件强调通用性、语义稳定和业务无关；业务组件则直接承载领域规则，比如审批节点卡片、订单状态条。区分清楚后，抽象才不会过早，也不会把业务耦合误塞进组件库。

24. 为什么服务层抽象很重要？
答：因为接口调用、参数组装、响应解包、错误处理如果都散在页面里，页面会很快失去可维护性。服务层的价值是统一数据访问边界，让页面更多关注交互和状态编排。

25. 公共能力沉淀时要避免哪些问题？
答：最常见的问题是过早抽象、为了复用而复用、抽出一个谁都看不懂的“万能组件”。我的原则是先让局部方案跑稳，再提炼真正稳定的共性，而不是第一版就想抽成平台级能力。

26. 单仓和多包仓库分别适合什么场景？
答：单仓适合团队规模较小、应用数量不多、变更集中统一的场景；多包或 Monorepo 更适合多应用共享基础库、设计系统、工程能力明显的团队。选择关键不是潮流，而是协作关系和发布复杂度。

27. Monorepo 的优点和代价是什么？
答：优点是代码复用、统一规范、联动改动效率高；代价是构建、缓存、权限、版本、发布策略和仓库治理复杂度会上升。Monorepo 的难点从来不在“能不能搭起来”，而在“长期能不能治理住”。
延伸：落地时通常会配合 `pnpm workspace`、`Turborepo`、`Nx` 这类工具，但工具只是手段，真正关键的是依赖边界、缓存策略和发布责任怎么治理。

## 四、代码规范与质量保障

28. 为什么前端项目必须接入 ESLint？
答：因为很多问题靠 code review 很难稳定拦住，比如潜在 bug、危险写法、依赖遗漏、Hooks 使用错误。ESLint 的核心价值不是格式统一，而是把低级错误和高频风险前置到提交前。

29. Prettier 和 ESLint 的职责有什么区别？
答：Prettier 主要解决格式一致性，降低风格争论；ESLint 主要解决代码质量和规范约束。最理想的组合是让格式问题自动化、让质量问题规则化，减少团队把时间耗在低价值争论上。

30. TypeScript 在工程质量上带来了什么价值？
答：它最大的价值是把接口边界、组件契约和数据流约束提前到了编译阶段，让重构和协作更可控。对工程质量来说，它不是替代测试，而是和规范、测试一起形成多层兜底。

31. 如何设计一套适合团队的代码规范？
答：我会先保证统一、易执行、收益清晰，再逐步收紧，而不是一上来就把规则堆满。能自动修复的尽量自动修复，必须人工判断的规则才留给 review，这样团队阻力会小很多。

32. Git 提交规范为什么重要？
答：它影响的是变更可追踪性、自动化流程和团队沟通效率，不只是“commit 漂不漂亮”。当项目接入自动发版、变更日志、问题回溯后，提交规范的收益会非常明显。

33. Commitlint、Husky、lint-staged 分别是做什么的？
答：`Commitlint` 用来校验提交信息，`Husky` 用来挂 Git Hooks，`lint-staged` 用来只处理暂存区文件。三者组合起来的意义是把规范真正卡到流程里，而不是写在 README 里没人执行。

34. 为什么代码审查同样属于工程化能力？
答：因为代码审查本质上是在做风险控制、知识传递和团队质量校准。工程化不只是自动化工具，还包括把“质量如何被发现和被兜住”设计进协作流程。

35. 如何降低多人协作中的代码风格冲突？
答：最有效的是工具自动化加统一规则，比如 Prettier、ESLint、编辑器配置和提交前检查。只靠口头约定几乎不可持续，因为团队规模一大，风格冲突一定会放大。

## 五、测试体系

36. 为什么很多前端项目做不好测试？
答：常见原因不是不知道测试重要，而是边界没想清、业务节奏过快、历史代码可测性差，最后测试变成“想做但总往后排”。所以测试问题很多时候其实是工程治理问题。

37. 单元测试、集成测试、端到端测试的区别是什么？
答：单元测试关注局部逻辑正确性，集成测试关注模块间协作是否正常，E2E 关注真实用户链路是否可用。它们不是谁替代谁，而是站在不同层面控制风险。

38. 哪些前端逻辑适合做单元测试？
答：纯函数、数据转换、权限判断、复杂计算、表单规则、状态机这类稳定且边界清晰的逻辑最适合。单测最怕的是和页面结构、时序、环境耦合得太深。

39. 哪些场景更适合集成测试？
答：像表单提交流程、组件联动、路由切换、状态管理和请求配合这类“局部业务流程”更适合集成测试。它比单测更接近真实使用，但又比 E2E 成本低很多。

40. E2E 测试通常覆盖哪些关键链路？
答：我会优先覆盖登录、支付、上传、关键审批、权限校验、核心提交链路这些“不能坏”的流程。E2E 的重点不是追求覆盖率，而是守住高损失、高回归风险路径。

41. React 项目里常见测试工具有哪些？
答：现在比较常见的组合是 `Vitest` 或 `Jest` 做测试运行器，`React Testing Library` 做组件行为测试，`Playwright` 或 `Cypress` 做 E2E。更重要的是分清它们各自解决哪一层问题：一个负责执行测试，一个负责验证组件行为，一个负责守住真实用户链路。

42. 测试覆盖率高就一定质量高吗？
答：不一定，因为覆盖率只说明“哪些代码被执行过”，不说明“关键风险有没有被覆盖”。如果大量测试都围绕低价值代码，覆盖率再高也不代表回归风险小。

43. 如何选择最有价值的测试范围？
答：我会优先选择高频、高风险、容易回归、出问题影响大的部分。测试资源永远有限，所以先覆盖最不能坏的地方，比平均撒网更有价值。

44. 在后台系统项目里，你会优先测哪些功能？
答：通常是权限、复杂表单、列表筛选、导入导出、关键配置流程和审批流。因为这些模块规则多、分支多，而且一旦出问题对业务影响通常很直接。

## 六、CI/CD 与发布流程

45. 为什么前端项目需要 CI/CD？
答：因为人工构建、人工测试、人工发布太依赖个人经验，稳定性和效率都不够。CI/CD 的核心不是“自动化看起来高级”，而是把交付过程标准化、可重复、可回溯。

46. 一条典型的前端 CI 流水线包含哪些步骤？
答：一般包括安装依赖、缓存恢复、静态检查、类型检查、测试执行、构建产物、制品上传和部署。成熟一点的流程还会加安全扫描、性能检查、灰度发布和回滚策略。

47. 自动化构建、测试、部署分别在解决什么问题？
答：自动化构建解决环境一致性问题，自动化测试解决质量前置问题，自动化部署解决上线效率和人工失误问题。三者连起来，才构成完整交付闭环。

48. 如何降低发布风险？
答：我通常会从“小步变更、自动检查、灰度发布、监控回滚、开关兜底”几个方向一起做。发布风险很少靠单一手段解决，关键是上线前、中、后的风险都要有人兜。
延伸：真正成熟的发布体系一定有明确止损手段，比如配置开关、版本回滚、资源回切或流量切换，而不是出问题后现场想办法。

49. 灰度发布和全量发布的区别是什么？
答：灰度发布是让一部分用户或流量先接触新版本，用真实环境验证风险；全量发布则是一次性全量切换。灰度的核心价值不是“慢一点发布”，而是给问题发现和止损留窗口。

50. 回滚机制为什么重要？
答：因为线上故障的第一目标通常是先止损，而不是现场修复。没有可靠回滚时，每次发布其实都在放大风险敞口。

51. 多环境配置通常怎么设计？
答：我会把环境变量、构建配置、部署配置分层管理，区分本地、测试、预发、生产，并保证配置来源可追踪。好的多环境设计要解决的不是变量多，而是避免串环境和配置漂移。

52. 环境变量管理有哪些常见风险？
答：常见风险包括串环境、密钥泄露、默认值不一致、配置无人治理，以及本地和线上环境行为不一致。尤其前端项目里，很容易误以为“写进 env 文件就安全了”，其实打包后公开变量都会暴露给客户端。

53. 前端项目如何避免把敏感信息打进包里？
答：原则是前端不保存真正的秘密，能只放在服务端的绝不放到前端构建里。对必须暴露给浏览器的配置，也要明确它只是公开配置，不应承担安全边界。

## 七、性能工程化

54. 性能优化为什么也属于工程化？
答：因为性能不是一次性手工调优，而是持续测量、定位、治理和回归控制的过程。只有进入指标体系、流程和监控，它才算工程能力，而不是个人经验。

55. 如何建立性能基线？
答：我会先选关键页面、关键链路和关键指标，比如 `LCP`、`TTFB`、首包大小、交互延迟，再记录当前数据作为对照。没有基线就无法判断后续优化到底有没有真实收益。

56. 如何持续监控页面性能？
答：除了实验室数据，我更看重真实用户监控，比如 Web Vitals、接口耗时、首屏时间、错误率、长任务分布。持续监控的重点是趋势和回归，不是只看一次 Lighthouse 分数。

57. 为什么要把性能指标接入 CI 或监控体系？
答：因为性能问题最大的特点是容易反弹。只有把关键指标纳入发布前检查或线上监控，团队才不会在业务迭代中不知不觉把之前的优化全吃回去。

58. 如何做构建速度优化？
答：方向一般是缓存、并行、缩小编译范围、减少无效插件、升级工具链和优化依赖。构建速度优化的核心是找瓶颈，而不是看到慢就盲目“上新工具”。

59. 如何做运行时性能优化？
答：我会从资源加载、渲染频率、长任务、请求链路、缓存策略和关键交互路径几个方向下手。运行时优化最怕只盯包体，而忽略真正影响用户体验的渲染和交互瓶颈。

60. 前端性能优化有哪些“看起来有效但收益很低”的做法？
答：比如不看瓶颈就到处加 `memo`、只盯打包体积不看运行时、把大量精力花在极小收益的微优化上。真正成熟的性能优化应该数据先行，而不是“感觉这样应该更快”。

61. 如何向面试官展示一次真正有数据的性能优化？
答：我会按“问题现象、定位方法、根因判断、改动方案、优化前后数据、如何防回退”这条线讲。最有说服力的不是动作清单，而是能证明为什么改、改后有什么结果。
延伸：数据最好具体到指标，比如首包下降百分比、`LCP` 改善、构建时间缩短、错误率变化，这样才能体现优化不是凭感觉。

## 八、稳定性与可观测性

62. 什么是前端可观测性？
答：我的理解是让线上问题“看得见、查得到、关联得起来”。它不仅包括报错采集，还包括性能、资源、接口、用户行为和版本信息的关联分析。

63. 前端监控通常包含哪些维度？
答：通常至少包括 JS 报错、资源加载失败、接口异常、性能指标、白屏卡顿、用户行为埋点和页面可用性。能把这些维度串起来，排查效率会比单点日志高很多。

64. JS 报错、资源加载失败、接口异常、白屏、卡顿分别怎么监控？
答：它们采集方式不同，但核心都是统一上报、带足上下文、可按版本和页面追踪。真正难的是把这些信号串成排查链路，而不是单独采到一堆事件。

65. 如何做统一错误处理？
答：我会把接口错误、运行时异常、业务提示分层处理：该上报的上报、该兜底的兜底、该提示用户的提示用户。统一错误处理的价值在于减少各页面各写一套、体验和可观测性都失控。

66. 错误上报系统要包含哪些关键信息？
答：至少要有用户标识、页面路由、时间、版本、设备环境、操作链路、错误堆栈、接口上下文。否则你拿到一条报错，也很难真正复现场景。

67. 用户反馈“页面卡死”时，你如何结合监控排查？
答：我会先看版本分布和是否集中爆发，再结合长任务、接口耗时、资源失败、错误峰值和用户行为链路逐步缩小范围。卡死类问题往往不是单点异常，而是多维信号一起看才能定位。

68. 如何设计日志分级和降噪？
答：我会区分 `info`、`warn`、`error` 的用途，对重复、低价值、可预期日志做采样或限流。监控系统最怕的不是没日志，而是有效信号被无意义噪声淹没。

69. 为什么没有监控的项目很难称得上成熟？
答：因为出了问题看不见、量不到、追不回，工程闭环就不完整。一个不能自我观察和快速止损的项目，很难说具备真正的工程稳定性。

## 九、权限、配置与业务工程

70. 后台系统的权限控制为什么是工程问题而不只是业务问题？
答：因为它会同时影响菜单、路由、按钮、接口、配置和页面骨架，绝不是页面里几个 `if` 能解决的。权限一旦没有统一设计，就会出现“显示一套、路由一套、接口一套”的失控状态。

71. 菜单、路由、按钮权限怎么统一设计？
答：我会尽量基于同一份权限元数据做映射，让菜单显示、路由守卫、按钮显隐共用一套规则来源。这样既减少重复维护，也更容易做权限回归测试。

72. 动态路由和静态路由分别适合什么场景？
答：静态路由适合结构稳定、路径关系明确的系统；动态路由更适合权限差异大、租户差异大、配置驱动明显的后台系统。选型时要看灵活性需求是否真的值得额外复杂度。

73. 为什么配置驱动能提升后台系统可维护性？
答：因为后台很多页面骨架高度相似，如果筛选区、表格列、表单项、按钮权限都能被配置化描述，重复开发会大幅减少。边界是不要把高频变化的复杂业务硬塞成超重配置，最后反而更难维护。

74. 如何设计一个通用表格页或表单页框架？
答：我会先抽稳定骨架和扩展协议，比如筛选区、查询、表格、分页、空态、权限、操作区这些共性，再把强业务逻辑留给插槽或配置扩展。核心是抽“稳定流程”，而不是把所有页面差异抹平。

75. 如何封装请求、鉴权、错误提示、空态、加载态？
答：这些能力我会尽量沉到基础设施层统一处理，让页面只关心业务差异。这样既能减少重复代码，也能保证体验和风格一致。

76. 为什么后台系统常需要“约定优于配置”？
答：因为后台页面量大、协作人数多，如果每个页面都自由发挥，长期成本会非常高。适度约定能换来更快的开发速度、更低的心智负担和更稳定的维护体验。

## 十、综合与实战

77. 如果让你从零搭一个后台前端工程，你会先做哪几件事？
答：我会先定技术栈和目录边界，再搭请求层、权限层、路由层、状态层这些基础设施，然后补规范、测试和发布流程。顺序上我更看重“先让团队能稳定开发”，而不是先追求最复杂的架构形态。

78. 如果项目越来越大，你如何判断需要做架构治理？
答：当你开始出现改一处牵很多处、协作冲突增多、发布风险上升、构建越来越慢、公共能力重复建设时，就说明项目进入治理窗口了。治理不是因为“想重构”，而是因为不治理成本已经开始失控。

79. 如何向面试官说明你具备工程化意识而不只是会用工具？
答：关键不是列工具清单，而是讲清楚为什么这么选、解决了什么问题、有什么取舍、效果如何衡量。能说出一套从问题到落地再到结果的数据闭环，才像真正的工程化意识。

80. 如果让你讲一次前端工程化优化经历，你会怎么组织？
答：我会按背景问题、现状痛点、方案设计、推进阻力、落地结果和后续机制来讲。尤其要强调为什么这么做，以及改完后如何避免问题再次发生。

81. 在字节这类公司面试中，工程化模块最容易被追问什么？
答：最容易被追问的是选型依据、边界控制、上线兜底、指标数据和失败经验。也就是说，面试官更关心你是不是把工程问题真正做成了闭环，而不是只会复述术语。

## 十一、Vite、发布与项目取舍

82. Vite 的按需加载在开发环境和生产环境分别是怎么工作的？
答：开发环境主要依赖浏览器原生 ESM 按需请求模块，访问到哪个模块再即时编译哪个模块；生产环境则仍然会交给 Rollup 做打包、压缩和优化。两阶段策略的核心是开发追求反馈速度，生产追求加载效率。

83. 浏览器在 Vite 开发环境里遇到 `import` 时，是否每次都会重新发起网络请求？
答：不是每次都从零发起，因为浏览器缓存、Vite 的模块缓存和 HMR 机制都会参与。首次访问某模块会发请求，但后续是否重新请求取决于缓存和文件变更情况。

84. 为什么说 Vite 的快不只是“没打包”，还和原生 ES Module、预构建有关？
答：因为真正让它快的是整条开发链路变短了：业务代码按需处理，第三方依赖提前预构建，热更新只影响局部边界。如果只理解成“没打包所以快”，其实少了一半关键原因。

85. 如果依赖很多，Vite 为什么仍然需要依赖预构建？
答：因为很多第三方包格式并不统一，依赖层级也深，如果不先预构建成更适合浏览器处理的形式，开发期模块解析成本仍然会很高。预构建本质上是在优化第三方依赖，而不是优化你的业务源码。

86. 除了 Lighthouse 和 Performance 面板，你还会用哪些指标或方式评估性能优化效果？
答：我还会看 Web Vitals、真实用户监控、错误率、接口耗时、关键交互时延和版本回归趋势。实验室数据适合定位，真实用户数据更适合判断优化是否真的对业务有效。

87. 如果一个前端项目需要灰度发布，你会从前端、网关、配置和监控四个层面怎么设计？
答：前端负责版本识别、能力兜底和兼容处理，网关负责流量切分，配置中心负责开关与策略控制，监控负责观察关键指标并触发回滚信号。灰度设计的重点是“逐步放量 + 可回退”，不是简单让两版代码并存。
延伸：如果系统复杂，还可以补充数据库兼容、接口向后兼容和静态资源缓存失效策略。

88. 为什么一些项目会同时使用 `Node.js` 和 `Python`？从工程协作角度如何划分边界？
答：这很常见，通常按能力边界划分：前端工程、SSR/BFF、实时网关这类更偏 Node，算法、数据处理、训练或脚本任务更偏 Python。工程协作上最重要的是接口契约清楚、部署边界清楚，而不是语言越统一越好。

89. `npm`、`pnpm`、`yarn` 有什么区别？为什么启动命令看起来类似但又不完全一样？
答：三者本质上都是 JavaScript 包管理工具，核心职责都是安装依赖、维护锁文件、执行脚本。区别主要在依赖安装机制、磁盘占用、安装速度、Monorepo 支持和团队使用习惯上。

如果从工程选型角度总结，可以这样说：

- `npm`：Node 官方默认方案，生态最通用，新人上手成本最低。大多数项目都能直接用它，优点是兼容性强、心智负担小，缺点是历史上安装性能和空间复用能力不如 `pnpm`。
- `yarn`：早期主要是为了解决 `npm` 在安装速度、锁文件稳定性和工作区能力上的一些问题，所以曾经在中大型项目里很流行。现在 `npm` 已经补上了不少能力，因此 `yarn` 的优势更多体现在已有团队沉淀和部分工作区场景。
- `pnpm`：通过内容寻址存储和硬链接机制复用依赖，特点是更省磁盘、安装通常更快、依赖管理更严格。它对 Monorepo 特别友好，也能减少“幽灵依赖”问题，但团队成员需要理解它和传统 `node_modules` 结构的差异。

为什么命令不完全一样？因为三者的命令设计不同，但语义很接近：

- 安装依赖：`npm install` / `pnpm install` / `yarn`
- 启动开发环境：`npm run dev` / `pnpm dev` / `yarn dev`

三者在脚本执行方式上也有差异：`npm` 执行 `package.json` 脚本时通常写成 `npm run <script>`；而 `pnpm`、`yarn` 对脚本命令做了更直接的封装，所以可以直接写 `pnpm dev`、`yarn dev`。

实际工程里怎么选，通常不是看“谁最先进”，而是看团队约束和项目背景：

- 如果项目追求通用性、成员背景差异大，`npm` 足够稳妥。
- 如果项目已经长期使用 `yarn`，并围绕它形成了稳定流程，没必要为了跟风轻易迁移。
- 如果是 Monorepo、包很多、依赖体积大，`pnpm` 通常更有优势。

实战上真正决定项目能不能启动，不是你输入的是 `npm`、`pnpm` 还是 `yarn`，而是项目里是否存在 `package.json`、锁文件是否匹配、依赖是否已正确安装，以及团队是否统一使用同一种包管理工具。
