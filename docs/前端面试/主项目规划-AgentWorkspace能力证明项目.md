# 主项目规划：Agent Workspace 双端能力证明项目

## 一、先给结论

目标是用一个项目证明自己有能力进入字节跳动、OpenAI 这类公司，我最推荐你做的不是普通聊天页，也不是纯中后台管理系统，而是：

`Agent Workspace`

你可以把它理解成：

**一个面向 AI Agent 的双端工作台：Web 端负责构建、调试、评测和治理，Flutter 端负责移动查看、告警、审批和运行协作。**

这个项目的重点不是“生成一段回答”，而是：

- 让 AI 执行任务的过程可见
- 让任务链路可调试
- 让模型和工具的组合可配置
- 让失败原因可定位
- 让结果可评测、可回放、可比较

它天然适合证明下面几类能力：

- 复杂前端交互能力
- Flutter 跨端应用能力
- AI 产品理解能力
- 平台型系统设计能力
- 工程化和治理能力
- 面向真实复杂任务的抽象能力

---

## 二、这个项目的核心价值是什么

这个项目的核心价值不是“做一个 AI UI”，而是：

**把 AI 执行复杂任务这件事，从黑盒变成白盒。**

现在很多 AI 产品都有同一个问题：

- 最终只给一个结果
- 用户看不到中间发生了什么
- 出错时不知道错在哪一层
- 不知道是模型不行、Prompt 不行、工具不行、上下文不行，还是流程设计不行
- 团队很难系统比较不同方案的效果

所以这个项目解决的核心问题是：

**AI 任务在真实工作流中不透明、不稳定、不可调试、不可复用、不可治理。**

这个项目的价值就是把这些问题显式化，并做成一个能运行的工作台。

一句话价值主张可以写成：

**Build, run, observe, debug, evaluate, and replay AI agent workflows in one workspace.**

---

## 三、它到底解决什么问题

### 1. 解决 AI 执行过程不可见的问题

普通聊天产品通常只展示最终输出，不展示：

- 任务拆解过程
- 调用了哪些工具
- 每一步拿到了什么上下文
- 哪一步失败了
- 失败后是否重试

而在真实场景里，这些信息非常重要。

### 2. 解决 AI 结果不可控的问题

同一个任务，使用不同模型、不同 Prompt、不同工具组合，结果可能差异很大。

如果没有统一的配置、日志、对比和评测能力，就很难把系统做稳定。

### 3. 解决 AI 系统不可调试的问题

AI 场景最难的地方之一不是“能不能跑”，而是“为什么这次跑歪了”。

你需要知道：

- 输入是什么
- 中间上下文是什么
- 模型收到的提示词是什么
- 工具返回了什么
- 最终输出为何变成现在这样

### 4. 解决多人协作和治理问题

真实团队不是一个人对着 Prompt 不断试。

而是需要：

- 产品看任务配置
- 前端看交互和状态
- 算法或模型同学看效果
- 业务方看结果是否符合预期
- 团队回放失败案例
- 对线上版本做发布和回滚

所以它最终解决的是：

**如何把 AI 系统从个人实验变成团队可协作的产品系统。**

## 四、这个项目本质上是在做什么

本质上，你是在做一个：

**面向 AI 任务执行的“操作系统式前端工作台”。**

这里的“操作系统式”不是指你真的在做浏览器内核，而是指它承担了这几件事：

- 接收用户任务
- 组织任务执行
- 分发模型和工具调用
- 管理运行状态
- 呈现中间过程
- 处理失败和人工接管
- 记录日志和历史
- 对结果做评测和比较

如果把它再抽象一点，它本质上是一层：

`Task -> Planning -> Execution -> Observation -> Evaluation -> Governance`

的统一界面和运行控制面板。

## 六、必要概念解释

下面这些概念一定要理解清楚，否则这个项目会变成“词很多但没讲明白”的状态。

### 1. 什么是 `LLM`

`LLM` 是 `Large Language Model`，大语言模型。

它的本质不是“会思考的人”，而是：

**根据已有上下文，预测下一个最可能出现的 token。**

这意味着：

- 它擅长语言生成
- 它可以在一定程度上做模式推理
- 它的输出受上下文强烈影响
- 它不是天然可靠的程序执行器

你在项目里接入的 `OpenAI`、`Claude`、`Gemini`、`Qwen`、`DeepSeek` 都属于这类模型服务。

### 2. 什么是 `token`

`token` 不是单词，也不是字符，它是模型处理文本时的最小计量单位。

你需要理解它，因为：

- 上下文窗口按 `token` 计算
- 成本按 `token` 计算
- 响应速度和输入长度也会受 `token` 影响

### 3. 什么是 `Prompt`

`Prompt` 就是你给模型的输入指令和上下文。

它不是一段“魔法咒语”，而是：

**控制模型行为的输入协议。**

通常会包含：

- system instruction
- user input
- tool results
- retrieved context
- format constraints

### 4. 什么是 `Tool`

`Tool` 是给模型可调用的外部能力。

例如：

- search
- fetch url
- code execution
- database query
- browser action
- internal API

模型本身只能生成文本，不能直接访问外部世界。  
`Tool` 的作用就是把“语言能力”连接到“真实动作能力”。

### 5. 什么是 `Agent`

`Agent` 不是一个神秘的新物种。

它更准确的理解方式是：

**一个能够基于目标、上下文和工具进行多步决策与执行的运行体。**

工程上可以把它近似理解为：

`LLM + Tools + State + Workflow`

也就是说，`Agent` 的重点不是“有名字”，而是：

- 知道当前目标
- 能根据上下文决定下一步
- 能调用工具
- 能持续维护中间状态
- 能在多步流程中推进任务

### 6. 什么是 `Workflow`

`Workflow` 是一条任务执行流程。

它强调的是：

- 步骤
- 顺序
- 依赖
- 条件分支
- 输入输出传递

如果说 `Agent` 更像“会判断的执行体”，那么 `Workflow` 更像“任务路径结构”。

### 7. 什么是 `Orchestration`

`Orchestration` 指编排。

它本质上是在回答：

**什么时候由哪个模型、哪个 agent、哪个工具，在什么条件下执行哪一步。**

也就是说，编排不是“画条线”而已，而是：

- 决定调用顺序
- 决定状态流转
- 决定失败处理
- 决定分支和回退
- 决定人工何时介入

### 8. 什么是 `Memory`

`Memory` 是系统保留和利用历史上下文的能力。

通常可以分成：

- 短期记忆：单次会话内的上下文
- 长期记忆：跨会话的偏好、历史、摘要、事实

它的作用是减少重复信息，提高连续任务执行的稳定性。

### 9. 什么是 `RAG`

`RAG` 是 `Retrieval-Augmented Generation`。

它本质上是：

**先检索相关资料，再把检索结果拼进 Prompt，让模型基于外部知识生成结果。**

它不是训练模型，而是增强模型在当前任务上的上下文。

### 10. 什么是 `MCP`

`MCP` 是一种让模型或 Agent 以统一协议接入外部工具和资源的方式。

你可以把它理解成：

**给 AI 系统提供标准化工具接口的一层协议。**

它的价值是把不同工具、不同服务、不同资源的接入方式统一起来，避免每个能力都写一套私有协议。

### 11. 什么是 `Skill`

`Skill` 可以理解成可复用的高层能力封装。

它通常不只是一个底层工具调用，而是一组：

- 固定任务意图
- 特定提示模板
- 规则约束
- 工具组合
- 输出格式

的打包能力。

例如：

- summarize document
- review code diff
- generate release note
- analyze support ticket

### 12. 什么是 `Eval`

`Eval` 是评测。

它不是简单的“打个分”，而是系统化判断：

- 结果是否正确
- 格式是否符合约束
- 是否调用了正确工具
- 是否满足业务目标
- 不同模型和配置谁更稳定

### 13. 什么是 `Human-in-the-loop`

`Human-in-the-loop` 指人在流程中介入。

它的价值是：

- 高风险任务不能全自动
- 模型失败时需要人工兜底
- 关键节点需要人工确认
- 调试和标注需要人参与

这个概念很重要，因为它决定你的系统不是“纯自动化幻觉”，而是“可控自动化系统”。

## 七、这些概念之间的关系

你可以用下面这条关系去理解整个系统：

- 用户先提出一个 `Task`
- `Workflow` 决定任务的大结构
- `Agent` 负责在结构内推进执行
- `LLM` 提供语言理解和推理能力
- `Tool` 提供外部动作能力
- `Memory` 提供历史上下文
- `RAG` 提供外部知识增强
- `MCP` 提供标准化接入外部能力的方式
- `Eval` 负责判断结果是否可靠
- `Human-in-the-loop` 负责关键节点的人工兜底

所以这个项目不是在做一个单点能力，而是在做一套把这些能力串起来的前端工作台。

## 八、Agent 从哪来，要不要自己做

这个问题一定要讲清楚。

### 1. Agent 从哪来

`Agent` 一般不是凭空“买来”的，它往往来自三层组合：

- 模型层：`OpenAI`、`Claude`、`Gemini` 等
- 工具层：搜索、执行器、浏览器、数据库、MCP 服务
- 编排层：你定义的状态机、流程、路由、重试和人工介入点

所以严格来说，`Agent` 不是“某个单独的 SDK 名称”，而是一个运行机制。

### 2. 你要不要自己做 Agent

建议是：

**不用自己造大模型，但要自己做轻量的 Agent Runtime。**

不建议自己从零做的东西：

- 大模型训练
- 推理框架
- 通用浏览器自动化底层
- 通用向量数据库底层

建议自己做的东西：

- run state
- step log
- task lifecycle
- tool registry
- model routing
- retry and timeout
- fallback
- replay
- eval pipeline

因为这些才是你项目真正有表达价值的部分。

### 3. 什么是 `Agent Runtime`

`Agent Runtime` 指让 Agent 真正跑起来的那层系统。

它至少要负责：

- 接收任务输入
- 组织执行步骤
- 管理中间状态
- 发起模型调用
- 发起工具调用
- 写入运行日志
- 把结果返回给前端

你不用把它做成一个复杂平台，但应该做成一个足够清晰的最小运行时。

## 九、这个项目发挥作用的原理是什么

### 1. 总体原理

这个系统的总体原理是：

**把原本散落在 Prompt、模型调用、工具调用、日志记录、人工确认中的执行过程，统一成一个结构化运行链路。**

也就是把“AI 调一下接口就返回结果”的模式，升级成：

`任务输入 -> 计划 -> 执行 -> 观察 -> 纠偏 -> 评测 -> 沉淀`

### 2. 为什么它能解决问题

它能解决问题，不是因为 UI 更好看，而是因为它提供了这三层能力：

- 可观测：知道发生了什么
- 可控制：知道哪里能改、哪里能拦
- 可复用：知道成功方案如何保存、失败案例如何重放

### 3. 它的底层运行过程

一次典型运行可以拆成下面几个阶段：

#### 阶段 1：用户提交任务

用户输入：

- task goal
- context
- constraints
- selected workflow
- selected model profile

#### 阶段 2：系统生成执行上下文

系统会整理：

- 系统级 Prompt
- 当前工作流配置
- 节点参数
- 历史上下文
- 工具定义
- 输入变量

#### 阶段 3：运行时推进执行

运行时会：

- 找到起始节点
- 根据图结构或步骤结构推进
- 在某些节点调用模型
- 在某些节点调用工具
- 把结果写入状态
- 决定下一步走向

#### 阶段 4：实时回传给前端

前端会持续收到：

- 当前步骤
- 执行状态
- 流式文本
- 工具调用记录
- 耗时
- 错误信息

#### 阶段 5：失败处理

如果失败，系统可以：

- retry
- fallback to another model
- ask for human review
- stop and expose error

#### 阶段 6：结果沉淀

执行结束后，系统保存：

- input
- step traces
- final output
- metrics
- evaluation result
- replay snapshot

## 十、从前端视角，这个项目真正难在哪里

很多人会误以为这个项目难点在模型调用。

其实对前端来说，真正难的地方主要有下面这些：

### 1. 复杂状态管理

你需要同时维护：

- 当前任务状态
- 当前运行态
- 节点配置态
- 面板 UI 状态
- 日志流状态
- 版本态
- 筛选和查询态

### 2. 实时流式渲染

模型返回常常是流式的。

这意味着你要处理：

- 增量更新
- 中断
- 重连
- 顺序一致性
- 滚动跟随
- 多面板联动

### 3. 可视化编排

如果你要加 `Canvas` 或流程图能力，就要面对：

- 节点拖拽
- 连线
- 缩放和平移
- 选中态
- 配置面板同步
- 图结构序列化

### 4. 日志与调试体验

运行日志不是普通列表。

它会包含：

- 文本流
- 状态流
- 工具调用
- 时间轴
- 错误栈
- 节点级高亮

### 5. 评测和回放

回放不是把旧文本重新展示一遍，而是要恢复当时的：

- 输入
- 配置
- 执行步骤
- 中间状态
- 最终结果

## 十一、推荐你做成什么样的产品形态

建议你把项目定义成：

`Agent Workspace for Build, Run, Debug, Eval and Mobile Ops`

它至少包含下面两个端的核心区域：

### Web 端核心区域

### 1. Task Panel

用来输入任务目标、上下文和约束。

### 2. Workflow Canvas

用来编排步骤、节点和依赖关系。

### 3. Tool / Skill Panel

用来管理可调用能力。

### 4. Run Console

用来展示执行过程、工具调用、状态变化和错误信息。

### 5. Output Panel

用来展示最终结果和结构化产物。

### 6. Eval Panel

用来展示评分、规则校验、对比结果和 benchmark。

### 7. History / Replay Panel

用来回放历史执行并做版本对比。

---

### Flutter 端核心区域

### 1. Run Inbox

展示待处理运行、失败运行、高风险运行和需要审批的任务。

### 2. Run Detail

展示某次运行的概览、步骤时间线、工具调用摘要和最终结果。

### 3. Approval Center

用于人工确认、驳回、补充说明、重新执行或切人工处理。

### 4. Alert Center

用于接收失败告警、超时告警、SLA 告警和高风险操作提醒。

### 5. Mobile Metrics

用于快速查看当天运行成功率、失败率、平均耗时和待处理项。

## 十二、最小可行版本应该做什么

如果你只做一个可用于面试的强版本，我建议先收敛到这个最小闭环：

### MVP 闭环

**Web 端定义任务与执行 -> 流式展示过程 -> 输出结果与评测 -> 历史回放 -> Flutter 端查看结果与审批处理**

### MVP 必须有的功能

- Web 端任务输入页
- Web 端一个最小流程配置能力
- 至少两种模型适配
- 至少两类工具接入
- Web 端实时运行日志
- Web 端失败状态展示
- Web 端结果面板
- Web 端基础评测
- Web 端运行历史
- Flutter 端运行列表
- Flutter 端运行详情
- Flutter 端审批操作
- Flutter 端告警入口

### MVP 明确可以先不做的内容

- 多租户
- 协同编辑
- 真正复杂权限系统
- 完整模板市场
- 自动布局
- 超复杂 memory 系统
- Flutter 端完整画布编辑
- Flutter 端复杂配置中心
- Flutter 端离线全文日志分析

先把闭环做扎实，比功能摊得很大更重要。

---

### MVP 周期建议

如果按单人开发估算，可以按下面三档看：

- 4 周：极简可跑版，只做 Web 主闭环和 Flutter 只读查看
- 6 周：推荐面试版，补齐最小流程配置、回放、评测和 Flutter 审批
- 8 周：强化版，补版本能力、移动告警、更多可观测性面板和性能优化

如果你的目标是尽快拿到一个能写进简历并现场演示的版本，我建议按 `6 周` 方案推进。

双端最好共用同一套领域对象语义，至少在接口层保持一致：

- workflow summary
- run summary
- run detail
- step trace
- approval action
- alert item
- eval result

这样你可以在面试里说明：

- Web 和 Flutter 不是两套割裂系统
- 它们共享同一套业务模型
- 只是承担不同终端职责

## 十四、扩展知识和必须注意的点

### 1. 不要把 Agent 神化

很多人会把 `Agent` 讲得很玄。

你一定要讲得工程化一些：

`Agent` 的本质是基于模型、状态和工具做多步决策，不是一个拥有自主意识的实体。

### 2. 不要把 Prompt 当核心壁垒

Prompt 很重要，但它不是这个项目的核心竞争力。

真正有价值的是：

- runtime
- observability
- workflow design
- evaluation
- governance

### 3. 不是所有任务都适合纯 Agent

有些任务更适合固定流程。

所以你要理解：

- 确定性流程适合规则编排
- 模糊性强的问题适合模型推理
- 高风险节点适合人工确认

### 4. AI 系统的难点往往是失败链路

不要只展示成功 Demo。

真正有说服力的是：

- 失败时怎么提示
- 怎么重试
- 怎么切模型
- 怎么回放问题
- 怎么定位原因

### 5. 可观测性非常重要

你至少要能展示：

- 每一步做了什么
- 哪一步耗时最高
- 哪一步失败
- 调用了什么工具
- 输入输出是什么

还要注意：

- 可观测性不只是“把每一步记日志”
- 更重要的是让系统能查询、分析、对比和回放
- `LLM` 内部计算过程通常仍然是黑盒
- 但围绕 `LLM` 的输入、上下文拼装、工具调用、状态流转和输出结果可以被做成白盒或半白盒

也就是说，这个项目真正要做的是：

**把 AI 应用外围的执行链路做成可理解、可定位、可分析的系统。**

### 6. 评测比“看起来聪明”更重要

一个系统看起来很聪明，不代表它稳定。

你要有意识地区分：

- 演示效果
- 可重复效果
- 可量化效果

### 7. 前端不要只做壳

如果你只是写几个页面包一下后端返回，那这个项目价值会很弱。

你应该主动承担这些前端高价值设计：

- 协议结构
- 面板组织
- 流程可视化
- 状态同步
- 错误体验
- 回放体验

你在简历里不要写成：

“做了一个 AI 对话平台。”

建议写成：

- 独立设计并实现 `Agent Workspace` 双端系统，构建 Web 端 AI 工作台与 Flutter 移动协作端，支撑 AI 任务的定义、执行、调试、评测、回放与审批
- 设计模型适配、工具接入、运行日志、失败恢复和评测对比等核心机制，提升 AI 工作流的可观测性与可控性
- 基于可视化编排与多面板联动实现复杂任务执行工作台，并通过 Flutter 端承接移动告警、审批与运行查看场景
- 针对运行态、配置态和历史态的复杂状态联动进行架构设计与性能优化，提升系统稳定性与可维护性

## 十六、面试时最值得讲的主线

你最好把这个项目统一讲成下面这条主线：

**我做的不是一个 AI 聊天页，而是一个双端 AI Agent 工作台。Web 端负责构建、调试和治理，Flutter 端负责移动协作和审批，它解决的是 AI 执行复杂任务时黑盒、不可控、难调试、难评测、难协作的问题。**

然后顺着这条主线往下讲：

- 为什么单纯聊天不够
- 为什么需要工具和工作流
- 为什么要可视化执行过程
- 为什么要做日志、回放和评测
- 为什么高风险任务需要人工介入
- 为什么移动端更适合承接告警和审批，而不是重型配置工作

只要这条主线讲稳，这个项目就不再像 Demo，而更像一个有系统设计深度的主项目。

## 十八、完整项目规划建议

### 1. 产品定位

`Agent Workspace` 是一个双端 AI Agent 工作台：

- Web 端是构建与治理中台
- Flutter 端是运行与协作移动端

### 2. 用户角色

- Builder：配置工作流、模型、工具、评测规则
- Operator：查看运行、处理失败、审批高风险任务
- Reviewer：查看回放、做人工评分、对比版本

### 3. Web 端页面清单

- Dashboard
- Workspace List
- Workspace Editor
- Workflow Canvas
- Tool Registry
- Model Config
- Run Console
- Run History
- Replay Detail
- Eval Center
- Version Center
- Audit Log

### 4. Flutter 端页面清单

- Login
- Run Inbox
- Run Detail
- Approval Center
- Alert Center
- Feedback Sheet
- Metrics Overview
- Profile / Settings

### 5. Web 端能力边界

Web 端负责：

- 构建
- 调试
- 评测
- 回放
- 版本管理
- 复杂可视化

### 6. Flutter 端能力边界

Flutter 端负责：

- 查看
- 响应
- 审批
- 处理异常
- 接收告警
- 提交反馈

### 7. 需要自己做的核心能力

- agent runtime 的最小状态流转
- run trace 数据协议
- Web 多面板联动
- Flutter 运行态和审批态管理
- 模型适配层
- tool registry
- replay snapshot

## 十九、推荐开发顺序

### 阶段 1：第 1~2 周

- Web 项目骨架
- backend 基础 API
- 跑通一次最小模型调用
- 确定 run / step / tool / eval 的基础协议

### 阶段 2：第 3~4 周

- Web 端完成任务输入、运行控制台、日志流、结果页
- backend 完成最小 runtime、tool call、history record
- 跑通 MVP 主闭环

### 阶段 3：第 5~6 周

- Web 端增加最小流程配置、回放、基础评测
- Flutter 端完成运行列表、详情、审批
- 接入通知或至少本地告警入口

### 阶段 4：第 7~8 周

- Web 端补版本能力、更多可观测面板、性能优化
- Flutter 端补告警中心、metrics、快速操作
- 统一收口文档、演示脚本、简历表达
