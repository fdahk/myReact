# 测试专题

## 使用方式

这个专题真正考察的是测试策略，而不是工具名记得多不多。成熟回答应该先说明测试要守什么风险，再说明不同层级各自覆盖什么边界，最后落到成本、收益和团队执行机制。

---

## 一、测试基础认知

1. 为什么前端项目需要测试？
答：因为前端链路长、交互多、回归范围大，很多问题不是开发时立刻暴露，而是在需求叠加后慢慢出现。测试的核心价值不是替代人工，而是把高风险回归前置到提测和发布前。

2. 测试在前端工程质量里解决了什么问题？
答：它主要解决回归兜底、重构信心和关键流程可验证的问题。类型系统能约束结构，测试则验证行为，两者解决的是不同层面的质量问题。

3. 为什么很多前端项目最后测试都做得不理想？
答：常见原因是业务节奏太快、测试边界不清、历史代码可测性差，以及团队没有形成持续执行机制。很多项目不是不会测，而是没有把测试纳入日常工程流程。

4. 如何回答“测试值不值得做”这种问题？
答：我不会回答成“做或不做”的二选一，而是会讲哪些场景最值得优先做。因为测试资源有限，关键是先覆盖最容易回归、最不能坏、最有业务价值的部分。

5. 测试和代码质量、重构安全性之间是什么关系？
答：好代码不等于不用测试，但有测试的好代码通常更敢重构。测试本质上是在给重构和迭代提供安全网。

## 二、测试分层

6. 单元测试、集成测试、E2E 测试的区别是什么？
答：单元测试关注局部逻辑，集成测试关注模块协作和局部业务流程，E2E 关注真实用户主链路。它们是不同层级的风险控制手段，不是替代关系。
延伸：测试分层最实用的落法是，单测守稳定逻辑，集成守主流程，E2E 守最不能坏的链路。

7. 什么类型的问题更适合单元测试发现？
答：纯函数、数据转换、表单规则、权限判断、状态流转这类输入输出清晰的逻辑最适合单元测试。它的优势是快、稳、定位准。

8. 什么类型的问题更适合集成测试发现？
答：组件联动、状态管理与请求协同、表单提交流程、路由切换这类“一个模块内部的真实流程”更适合集成测试。它比单测更接近使用场景，又没有 E2E 那么重。

9. E2E 测试通常应该覆盖哪些关键链路？
答：我一般优先覆盖登录、支付、上传、核心审批、关键提交和权限流转这些不能坏的路径。E2E 不适合全面铺开，而适合守住业务损失最大的链路。

10. 为什么不是所有功能都值得写 E2E？
答：因为 E2E 成本最高、最依赖环境、维护最重，如果把低价值功能也全部拉进来，团队很快会被执行时间和稳定性反噬。它应该是“少而关键”，而不是“全而脆弱”。

11. 如何平衡测试成本和收益？
答：我的原则是用低成本测试覆盖最多稳定逻辑，把最贵的 E2E 留给关键主链路。也就是说，测试策略本质上也是资源分配问题。

12. 测试金字塔在前端项目里怎么理解？
答：底层单测多一点，中间集成测试兜业务主流程，顶层少量 E2E 守住关键链路。它并不是教条，而是在提醒你不要把所有验证压力都压到最贵的那一层。

## 三、Vitest 与测试执行环境

13. 为什么 `Vitest` 近年越来越常见？
答：因为它和 Vite 生态结合得很自然，启动快、反馈快、配置成本低，特别适合现代前端项目。它的优势不只是速度，还包括和现有工具链复用程度高。

14. `Vitest` 相比 `Jest` 的优势是什么？
答：最直接的优势是更贴近现代 ESM/Vite 项目，配置更轻，冷启动和 watch 体验更好。当然这不代表它绝对替代 `Jest`，而是更适合很多新项目的技术栈。

15. `Vitest` 为什么和 `Vite` 项目配合得更自然？
答：因为它复用了 Vite 的模块解析、转换和部分配置思路，团队不需要为测试环境再维护一套完全平行的构建体系。这样上下文切换成本会低很多。

16. `Vitest` 的 watch 模式和执行速度为什么更有优势？
答：核心还是工具链更轻、增量执行更高效、依赖处理更贴近业务代码实际运行方式。对频繁改代码的前端开发来说，反馈速度本身就是产能。

17. 什么时候你仍然会考虑 `Jest`？
答：如果是历史项目、现有生态深度绑定、团队已经沉淀了大量 `Jest` 工具和约定，我不会为了“追新”强切。工程选型更重要的是迁移成本和团队收益，而不是单看新旧。

18. 在 `Vitest` 里如何组织测试目录和命名？
答：我更倾向测试贴近源码，方便跟着模块演进；对于公共模块或大型业务域，也可以集中到单独测试目录。关键不是目录长什么样，而是团队能快速定位测试和业务代码的对应关系。

## 四、React Testing Library

19. `React Testing Library` 的核心理念是什么？
答：核心理念是从用户可感知行为出发验证组件，而不是盯着内部实现细节。也就是说，重点是“用户看到了什么、能做什么”，而不是组件内部到底用了几个 Hook。

20. 为什么它强调“从用户视角测试”？
答：因为从用户视角写出的测试更稳定，重构时不容易碎，也更接近真实使用方式。实现细节一旦变化就大面积挂掉的测试，通常长期价值不高。

21. `getBy`、`queryBy`、`findBy` 的区别是什么？
答：`getBy` 适合立即存在的元素，不存在时会直接报错；`queryBy` 适合判断元素不存在；`findBy` 适合等待异步出现。真正容易混淆的地方就在于两点：它们会不会等待，以及找不到元素时的行为有什么区别。

22. 为什么 `getByRole` 被认为是高优先级查询方式？
答：因为它最贴近真实的语义和可访问性结构，也能倒逼组件别乱用无语义标签。很多时候，测试写得顺不顺，反映的其实是组件语义写得对不对。

23. 为什么测试不应该过度依赖实现细节或类名？
答：因为类名、DOM 结构、内部状态很容易在重构时变化，但用户感知不一定变。过度依赖实现细节会导致测试高脆弱、低价值。

24. 如何测试异步渲染和加载态？
答：我会先断言加载态是否出现，再用 `findBy` 或等待逻辑断言最终结果。异步测试的重点不是“加个 wait”，而是明确页面在每个阶段应该给用户什么反馈。

25. 如何测试用户点击、输入、键盘交互？
答：优先用 `userEvent` 模拟更接近真实用户的交互，而不是手工触发底层事件。这样测试结果和真实用户操作路径会更一致。

26. 如何测试错误态、空态、权限态？
答：核心是把不同数据输入和权限前提喂进去，看最终用户看到的反馈是否正确。测试这里不要只断言某个变量变化，而要断言用户结果和操作约束。

## 五、组件与业务测试

27. 一个普通业务组件，哪些逻辑适合写测试？
答：我会优先测分支多、易回归、对业务结果直接有影响的逻辑，比如展示条件、交互触发、状态变化和关键文案。不是每个组件都要测满，而是优先测“出错代价大”的部分。

28. 表单组件应该重点测试什么？
答：重点测输入、校验、禁用态、提交流程、错误反馈和提交后的状态变化。表单类测试的本质是在验证“用户能不能顺利完成任务”。

29. 列表、表格组件应该重点测试什么？
答：我会优先测筛选、排序、分页、空态、加载态和行级操作这些高频逻辑。因为这类组件看起来只是展示，实际上业务分支通常非常多。

30. 权限组件应该重点测试什么？
答：重点是不同权限下按钮、菜单、内容、可编辑状态是否符合预期。权限测试很适合用表驱动思路，把不同角色和预期结果枚举清楚。

31. 自定义 Hook 该怎么测？
答：把它当成状态和行为单元来测，关注输入变化后输出、状态流转和副作用触发是否符合预期。Hook 测试要尽量绕开 UI 噪音，抓住可复用逻辑本身。

32. 请求逻辑如何测试？
答：通常会 Mock 网络层，重点验证请求前后的状态切换、成功和失败分支、重试逻辑以及错误提示。真正想验证的是请求行为，不是网络库本身能不能发请求。

33. 如果组件依赖路由、状态管理、请求上下文，测试环境如何搭建？
答：我会封一个统一的 `render` 包装器，把常用 Provider、Router、Store 等上下文补齐。这样测试不会每次重复搭环境，也更接近真实运行条件。

34. 为什么组件测试里通常需要 Provider 包装器？
答：因为真实组件不是在真空里运行的，没有上下文很多逻辑根本不会成立。包装器的意义是把测试环境拉回到真实业务上下文，而不是只让组件“勉强能渲染”。

## 六、Mock 与边界

35. 什么情况下应该 Mock？
答：当外部依赖不稳定、成本高、和当前测试目标无关时，我会 Mock。Mock 的原则不是越多越好，而是只隔离那些不属于当前测试责任范围的边界。

36. 什么情况下不应该过度 Mock？
答：如果 Mock 到连真实行为都看不出来了，测试就会变成“验证自己写的假世界”。一旦测试结果和真实产品脱节，它的价值就会迅速下降。

37. Mock 请求和真实请求相比，各自的优缺点是什么？
答：Mock 请求更快、更稳、可控性更强，适合大多数单测和集成测试；真实请求更接近线上，但也更慢、更脆弱、更依赖环境。二者不是二选一，而是适合不同层级测试。

38. 如何避免测试因为 Mock 太多而失去真实价值？
答：我的原则是只 Mock 边界，不 Mock 自己真正想验证的业务行为。比如可以 Mock 网络返回，但不要把组件内部关键状态流转也一并假掉。

39. 如何为流式请求、时间函数、浏览器 API 做 Mock？
答：核心思路是替换边界能力并控制时序，让测试结果可预测、可重复。像 AI 流式输出、定时器、`localStorage`、`ResizeObserver` 这类能力，都属于适合集中封装后再 Mock 的边界。

40. 你如何理解“测行为，不测实现”？
答：就是看用户能不能完成操作、页面是否给出正确反馈，而不是组件内部是不是用了某个 Hook、某个 state 或某个 class。这样写出来的测试更能跨过重构周期存活下来。

## 七、Playwright 与 E2E

41. `Playwright` 适合解决什么问题？
答：它很适合覆盖真实浏览器环境下的关键业务链路，包括跨页面跳转、权限流转、上传下载、弹窗交互、真实网络时序等。很多只有在浏览器里才暴露的问题，单测很难覆盖到。

42. `Playwright` 和单元测试、集成测试的边界是什么？
答：它不应该替代前两者，而是兜那些只有真实环境和完整链路下才容易暴露的问题。也就是说，Playwright 更像最后一层回归保护网。

43. 登录、上传、表单提交流程为什么适合写 E2E？
答：因为这些流程跨模块、依赖真实交互和浏览器行为，一旦坏了影响也最大。尤其文件上传、登录跳转、复杂提交这类链路，离开真实浏览器环境很多问题测不出来。

44. E2E 测试为什么容易慢、脆弱、维护成本高？
答：因为它依赖环境、网络、页面结构、时序和测试数据，链路越长不确定性越多。正因为成本高，所以更需要精挑覆盖范围和增强排查能力。

45. 如何挑选最值得写 E2E 的关键链路？
答：我会看业务价值、线上损失、回归频率和人工验证成本，优先覆盖最不能坏的路径。E2E 的目标是止大损失，而不是追求“看起来很全面”。

46. 在 CI 里跑 E2E 测试要注意什么？
答：要注意环境稳定、测试数据准备、超时控制、失败重试策略和失败后可排查性。否则 CI 里最先被嫌弃的往往不是测试太少，而是测试太脆弱。

47. 失败的 E2E 用例如何高效排查？
答：我会先看录屏、截图、控制台日志、网络请求和 trace，再判断是代码问题、数据问题还是环境抖动。E2E 的排查效率，很大程度取决于你事先有没有把观测信息准备好。

## 八、测试与可访问性

48. 为什么 `React Testing Library` 的查询方式天然和可访问性有关？
答：因为它优先鼓励你按 `role`、`label`、文本这些用户真正可感知的信息去找元素，而不是按类名和实现细节。测试和可访问性在这里其实是在相互促进。

49. 为什么写测试时用 `getByRole` 往往能顺带提升组件语义化质量？
答：因为当一个组件无法被合理地按角色查询时，通常说明它的语义结构本身就有问题。很多可访问性问题，是在写测试时最先被暴露出来的。

50. 如何测试键盘导航和焦点管理？
答：我会用 `userEvent` 模拟 `Tab`、`Enter`、`Escape`、方向键等操作，观察焦点流转和交互结果是否符合预期。对弹窗、下拉、菜单、表单这类组件来说，这是非常高价值的测试点。

51. 为什么无障碍设计会影响测试稳定性和表达质量？
答：因为语义越规范、交互越标准，测试越容易写成贴近用户的稳定断言。反过来，如果组件没有语义和可访问结构，测试通常也会变得别扭而脆弱。

## 九、工程实践

52. 测试覆盖率高就一定代表质量高吗？
答：不一定，覆盖率只能说明“执行过”，不能说明“关键风险被守住了”。我更看重的是关键路径是否被覆盖、失败时是否能阻断回归。

53. 哪些模块最值得优先补测试？
答：我会优先补核心业务链路、权限逻辑、复杂表单、公共组件和历史高频出错模块。补测试最忌讳平均用力，因为那样通常既慢又没效果。

54. 如何在项目时间紧张时设计“最小但有效”的测试策略？
答：先守关键路径和高风险点，优先选投入小、收益高的层级，比如稳定逻辑的单测和主流程的少量集成测试。不要追求面面俱到，而要追求真正能防回归。

55. 如何把测试接入 CI 流水线？
答：我会把静态检查、单测、关键集成测试和必要的 E2E 放进发布前必经流程，并明确失败阻断策略。测试要真正产生价值，必须和提交流程、合并流程、发布流程绑定起来。

56. 测试失败阻塞发布时，如何权衡效率和风险？
答：我会先判断失败的是不是高价值用例、是真缺陷还是环境波动，再决定是修复、隔离还是临时降级。成熟团队不会因为赶进度就默认忽略测试报警，但也不会一刀切地被低价值噪音绑架。

57. 老项目补测试时，你会从哪里开始？
答：我通常从最核心、最稳定、最容易出回归问题的模块先补，因为这类模块收益最高、最容易形成正反馈。直接从最乱的区域下手，往往会把团队信心先打没。

## 十、项目结合题

58. 在后台管理系统里，你会优先测哪些模块？
答：我会优先测登录权限、复杂表单、列表筛选、审批流和关键配置页，因为这些模块规则多、出错代价高。后台系统测试的重点是守住流程正确性和权限边界。

59. 在 `AI Agent` 或 `AI Workflow` 项目里，你会优先测哪些功能？
答：我会优先测节点配置、执行流程、权限控制、流式结果展示和异常回退。因为这类项目里很多问题不是纯页面渲染，而是状态流、时序和错误处理容易失控。

60. 流式响应、节点配置、权限路由、复杂表单分别如何设计测试策略？
答：我的思路是把稳定规则和数据转换下沉到单测，把组件交互和局部流程放到集成测试，再用少量 E2E 守住跨页面主链路。也就是说，复杂业务更需要分层测试，而不是一层硬扛全部风险。

61. 如果一个问题在线上才暴露，事后你会补哪一层测试？
答：我会先看问题根因落在哪个边界，再补成本最低但能真正防回归的那一层。补测试不是为了“报复性加用例”，而是为了以后同类问题别再漏掉。

## 十一、综合表达

62. 如果面试官问“你们项目测试做得怎么样”，你会怎么回答才不空？
答：我会诚实讲现状、覆盖范围、主要策略、已经做成闭环的部分，以及还没补齐的地方。这样比空喊覆盖率更像真实项目，也更容易引出你的判断力。

63. 如果你没系统做过测试，如何诚实回答并体现认知深度？
答：可以坦诚经验有限，但要把测试分层、价值判断、优先级和落地路径讲清楚。大厂通常能接受经验还在累积，但不会接受只会背工具名、没有测试思维。

64. 如果让你从零搭一套前端测试方案，你会怎么规划？
答：我会先按风险分层，再选工具，先覆盖关键业务和稳定公共能力，然后把测试接入 CI 和团队日常流程。测试体系的目标不是一次到位，而是可持续演进。
延伸：把测试体系讲完整，通常还要补充如何定义阻断策略、如何处理 flaky case，以及如何让团队愿意长期维护测试。
