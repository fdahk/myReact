# 测试专题

## 使用方式

这个专题真正考察的是测试策略，而不是工具名记得多不多。成熟回答应该先说明测试要守什么风险，再说明不同层级各自覆盖什么边界，最后落到成本、收益和团队执行机制。

---

## 一、测试基础认知

1. 为什么前端项目需要测试？
答：因为前端项目的真实风险不是“页面能不能第一次跑起来”，而是需求迭代后旧功能会不会被悄悄改坏，尤其前端同时面对复杂交互、状态联动、浏览器差异、接口时序和权限分支，很多问题只有在回归阶段才暴露。测试的价值就是把这些高频回归风险尽量提前到开发、提测和发布前，让团队不必每次都靠人工点一遍来赌运气。

2. 测试在前端工程质量里解决了什么问题？
答：它解决的核心问题是“行为可验证”，也就是代码改完以后，我们能不能有一套比较稳定的方式确认关键逻辑和关键流程还是对的。类型系统更擅长约束数据结构和接口契约，Lint 更擅长拦危险写法，而测试关注的是页面和业务行为本身有没有按预期工作，所以它们是不同层次的质量兜底。

3. 为什么很多前端项目最后测试都做得不理想？
答：常见原因并不是团队不知道测试重要，而是业务节奏快、测试边界不清、历史代码可测性差、Mock 体系混乱、CI 没接好，最后大家都觉得“这次先不补，下次再说”。所以很多项目测试做不好，本质上不是工具问题，而是工程机制没有把测试变成日常开发的一部分。

4. 如何回答“测试值不值得做”这种问题？
答：更好的回答方式不是简单站队说“必须做”或“太浪费时间”，而是说明测试资源有限，所以应优先投在高风险、高频、容易回归、出问题损失大的部分。也就是说，测试值不值得做，从来不是抽象问题，而是具体看这段代码是否稳定、是否关键、是否经常改、出了问题会不会影响用户或业务。

5. 测试和代码质量、重构安全性之间是什么关系？
答：代码写得整洁、分层合理、可预测，通常会让测试更容易写；而测试写得扎实，又会反过来提升你重构和演进代码的底气，因为你知道改完之后有东西帮你兜行为回归。它们不是谁替代谁，而是互相促进，好代码提高可测性，测试则提高重构安全性和长期维护信心。

## 二、测试分层

6. 单元测试、集成测试、E2E 测试的区别是什么？
答：单元测试主要守局部逻辑，关注一个函数、一个 Hook、一个规则模块在给定输入下输出是否正确；集成测试更关注多个模块组合后的小范围真实流程，比如组件和状态、组件和请求、表单和校验是否协同正常；E2E 则站在真实用户视角，从浏览器入口一路走完整链路。最实用的理解方式是：单测守稳定逻辑，集成守业务流程，E2E 守最不能坏的关键链路。

7. 什么类型的问题更适合单元测试发现？
答：凡是输入输出相对清晰、依赖少、结果容易断言的逻辑都很适合单元测试，比如纯函数、数据转换、表单规则、权限判断、状态机、价格计算、日期处理这类内容。它的优势是执行快、定位准、维护成本相对低，所以特别适合守那些“规则稳定但容易被改坏”的代码。

8. 什么类型的问题更适合集成测试发现？
答：当问题不在单个函数，而在模块之间的协作时，集成测试通常更有价值，比如一个表单里输入、校验、禁用按钮、提交请求和成功提示是不是能连起来工作，一个页面里筛选条件、列表刷新、分页和空态是不是一致。它比单测更接近用户行为，但又比完整 E2E 轻很多，所以很适合承担“局部真实流程”的验证任务。

9. E2E 测试通常应该覆盖哪些关键链路？
答：我一般会把 E2E 留给那些跨页面、跨模块、跨上下文、离开真实浏览器就不容易完全验证的链路，比如登录、支付、上传、核心审批、关键提交流程、复杂权限跳转。它不适合什么都测，而适合守住“最不能坏、坏了代价最大、人工回归又最费劲”的那部分路径。

10. 为什么不是所有功能都值得写 E2E？
答：因为 E2E 是测试里最贵的一层，它运行慢、依赖环境、需要准备测试数据，还更容易受网络、动画、时序、页面结构波动影响，如果低价值场景也大量堆进去，团队很快就会被 flaky case 和执行时间反噬。它更适合少量、关键、代表性强的主路径，而不是追求看起来很全面的覆盖率。

11. 如何平衡测试成本和收益？
答：我的思路是把最稳定、最通用、最容易回归的逻辑尽量往低成本层级下沉，让单测和集成测试承担大部分验证任务，再把最贵的 E2E 只留给关键主链路。换句话说，测试策略本质上也是资源分配问题，不是“每层都铺满”，而是“让每层承担最适合它的那类风险”。

12. 测试金字塔在前端项目里怎么理解？
答：测试金字塔并不是要求你机械地追某个比例，而是在提醒你不要把所有验证压力都压到最贵、最脆弱的 E2E 上，底层应该有足够多的单测守稳定规则，中间用集成测试守真实业务流程，顶层再用少量 E2E 兜住关键链路。它的意义在于帮助团队建立成本和收益更平衡的测试结构，而不是为了背一个概念模型。

## 三、Vitest 与测试执行环境

13. 为什么 `Vitest` 近年越来越常见？
答：因为现代前端项目大量使用 Vite、ESM 和更轻量的开发链路，而 `Vitest` 正好和这套生态天然贴合，启动快、watch 反馈快、配置心智负担也更小。它受欢迎不只是因为“跑得快”，更因为它让测试环境和开发环境尽量共用同一套模块解析和构建思路，减少了两套体系割裂的问题。

14. `Vitest` 相比 `Jest` 的优势是什么？
答：最常见的优势是更适合现代 ESM/Vite 项目，冷启动更轻，watch 模式反馈更快，配置往往也更顺滑，因为很多能力可以直接复用现有 Vite 配置。它并不是在所有场景都绝对优于 `Jest`，而是在新项目、前端现代工具链场景下，通常能用更低的额外成本把测试跑起来。

15. `Vitest` 为什么和 `Vite` 项目配合得更自然？
答：因为它复用了 Vite 的很多底层能力，比如模块解析、代码转换、别名配置和部分插件思路，所以团队不需要为了测试再维护一套和开发环境明显不同的构建规则。这样做的好处是环境更接近真实开发链路，配置漂移更少，大家调试测试时也不容易陷入“为什么开发能跑、测试却不行”的割裂状态。

16. `Vitest` 的 watch 模式和执行速度为什么更有优势？
答：本质上还是因为它围绕现代前端模块体系做了更轻的增量处理，不必像传统方案那样把很多不相关模块都重新走一遍重流程，所以修改一个文件后重新反馈的成本更低。对前端开发来说，测试反馈速度直接影响你是否愿意频繁运行测试，因此快本身就是工程可执行性的一部分。

17. 什么时候你仍然会考虑 `Jest`？
答：如果是一个历史项目已经围绕 `Jest` 沉淀了大量测试工具、快照习惯、Mock 方案、CI 流程和团队经验，我不会为了追新而强行切到 `Vitest`，因为迁移本身也有成本和风险。工程选型看的是整体收益，不是工具谁更新，所以保守选择一个团队已经非常熟悉的方案，有时反而更成熟。

18. 在 `Vitest` 里如何组织测试目录和命名？
答：我通常更倾向测试贴近源码，让测试文件跟着模块一起演进，这样改业务代码时更容易顺手看见对应测试；但对公共模块、跨文件逻辑或大型业务域，也可以有相对集中的测试目录。关键不是目录形式本身，而是团队能不能快速建立“这段代码对应哪些测试、失败时去哪改”的映射关系。

## 四、React Testing Library

19. `React Testing Library` 的核心理念是什么？
答：它的核心理念是尽量从用户视角验证组件，也就是关心页面渲染出了什么、用户能做什么操作、最终看到了什么反馈，而不是优先关心内部 state、Hook 数量或者 DOM 结构长什么样。简单说，它鼓励你测的是“行为和结果”，而不是“实现细节”。

20. 为什么它强调“从用户视角测试”？
答：因为用户从来不关心你内部是怎么实现的，他们只关心按钮能不能点、输入后结果对不对、错误有没有提示，所以测试如果太依赖实现细节，一旦重构但行为没变，测试却会大面积碎掉。用户视角的测试通常更稳定，也更能在长期演进中保留价值。

21. `getBy`、`queryBy`、`findBy` 的区别是什么？
答：可以把它们理解成三种不同的“期待方式”，`getBy` 用于你确信元素此刻应该已经存在，不存在就直接报错；`queryBy` 适合判断元素不存在，因为它找不到时返回空而不是抛错；`findBy` 用于异步等待元素出现，本质上是带等待能力的查询。真正容易混淆的地方就在于它们会不会等、找不到时是报错还是返回空。

22. 为什么 `getByRole` 被认为是高优先级查询方式？
答：因为 `role` 和可访问名称更接近真实用户和辅助技术理解页面的方式，比如一个按钮、输入框、链接、对话框本来就应该有清晰角色和名称，所以按这个层次查询通常更稳、更有语义，也能顺手暴露组件无障碍问题。很多时候，测试能否自然地用 `getByRole` 写出来，本身就是组件语义质量的体检结果。

23. 为什么测试不应该过度依赖实现细节或类名？
答：因为类名、DOM 嵌套结构、内部状态命名这些东西在重构时非常容易变，但用户感知到的行为不一定变，如果测试把自己绑在这些细节上，它就会变成“阻碍重构的绊脚石”而不是“保护行为的安全网”。测试更该关心的是用户能看到什么、能触发什么、最终结果是什么。

24. 如何测试异步渲染和加载态？
答：更好的做法是把异步过程拆成阶段来看，先断言加载态或骨架屏是否出现，再等待异步结果渲染出来，最后验证成功态、空态或错误态是否符合预期。异步测试的重点不是到处塞 `wait`，而是明确页面在请求开始、中间和结束时分别应该向用户表达什么。

25. 如何测试用户点击、输入、键盘交互？
答：我会优先使用 `userEvent` 这类更接近真实用户动作的工具，因为它会更贴近浏览器实际交互时序，比如输入、聚焦、失焦、键盘事件等细节都更自然。这样写出来的测试更能反映真实用户路径，而不是停留在“手工触发一个底层事件，代码好像也动了”的层面。

26. 如何测试错误态、空态、权限态？
答：核心思路是制造不同前提条件，再从用户结果层面断言页面表现，比如接口失败时是否显示明确错误反馈，没有数据时是否出现空态提示，无权限时按钮是否隐藏或禁用且入口是否受控。这里不要只盯内部变量变没变，而要验证用户最终真正能看到、能不能操作、被限制成什么样。

## 五、组件与业务测试

27. 一个普通业务组件，哪些逻辑适合写测试？
答：我会优先测试那些分支多、规则多、经常改、出错后对用户或业务影响直接的内容，比如条件渲染、关键文案、重要交互、状态切换、权限控制和关键回调。不是每个展示型组件都需要同样多的测试，而是优先守住那些“出错代价大、人工又不容易稳定回归”的部分。

28. 表单组件应该重点测试什么？
答：表单的关键不是“能输入字”，而是用户能不能顺利完成任务，所以我通常会重点测字段输入、校验规则、错误提示、禁用态、提交流程、成功后的反馈以及失败时的恢复路径。尤其复杂表单里，很多 bug 不在单个字段，而在多个字段联动、异步校验和按钮状态控制上。

29. 列表、表格组件应该重点测试什么？
答：这类组件表面上像展示组件，实际上往往是业务规则密集区，所以我会优先测筛选、排序、分页、空态、加载态、异常态、行级操作和权限差异。原因是列表页经常承载核心查询和批量操作能力，一旦某个分支出错，对用户效率和业务结果的影响会很明显。

30. 权限组件应该重点测试什么？
答：重点是不同权限前提下，按钮、菜单、入口、内容区、可编辑状态和提交行为是否符合预期，因为权限问题常常不是“显示错了”这么简单，而是会进一步变成误操作或越权风险。权限类测试特别适合用表驱动方式，把角色、场景和预期结果成组列出来，逻辑会更清晰。

31. 自定义 Hook 该怎么测？
答：可以把 Hook 当成一个“状态和副作用组合器”来看，重点验证输入变化后输出是否正确、内部状态如何流转、副作用何时触发和清理。Hook 测试的关键是尽量隔离 UI 噪音，抓住它真正复用的逻辑价值，而不是为了写测试再套一层无意义界面。

32. 请求逻辑如何测试？
答：通常我会把网络层当成边界进行 Mock，然后重点验证请求发起前后的加载状态、成功和失败分支、重试机制、错误提示以及数据转换逻辑。真正想确认的是“这段业务代码面对不同响应时会怎样表现”，而不是去测试底层 HTTP 库本身能不能联网。

33. 如果组件依赖路由、状态管理、请求上下文，测试环境如何搭建？
答：我会封装一个统一的测试渲染入口，把常用的 Router、Store、Query Client、Theme Provider 之类上下文一次性包好，让每个测试只关心自己的业务前提和断言。这样既能减少样板代码，也能让测试环境更接近真实应用运行方式。

34. 为什么组件测试里通常需要 Provider 包装器？
答：因为很多业务组件从来都不是在真空里运行的，它们依赖全局状态、路由信息、国际化、主题、请求缓存等上下文，没有这些前提，组件即使“能渲染”也不代表行为真实。包装器的意义就是把测试还原到一个足够真实的最小运行环境，而不是为了过编译随便补点假数据。

## 六、Mock 与边界

35. 什么情况下应该 Mock？
答：当某个依赖不稳定、执行成本高、和当前测试目标无关或者本来就属于外部边界时，我会考虑 Mock，比如真实网络、时间函数、浏览器 API、第三方 SDK、随机数等。Mock 的核心目的不是省事，而是把测试责任边界收清楚，让当前用例只聚焦自己真正想验证的行为。

36. 什么情况下不应该过度 Mock？
答：如果 Mock 到最后连组件真实的状态流转、交互结果和边界协作都被你一起伪造掉了，那测试其实只是在验证一套你自己搭的假世界，而不是产品行为。过度 Mock 最大的问题不是“写起来麻烦”，而是测试看似很多，实际对真实回归几乎没有保护力。

37. Mock 请求和真实请求相比，各自的优缺点是什么？
答：Mock 请求的优点是快、稳、可控、覆盖边界情况容易，特别适合单测和集成测试；真实请求更接近线上链路，能暴露接口契约、网络时序和环境配置问题，但也更慢、更脆弱、更依赖测试环境。它们不是二选一，而是分别适合不同层级，所以多数情况下是底层多用 Mock，顶层少量关键链路再走更真实的验证。

38. 如何避免测试因为 Mock 太多而失去真实价值？
答：我的原则是只 Mock 边界，不 Mock 自己真正想验证的业务行为，也就是可以把外部网络、时间和浏览器环境稳定下来，但不要把组件内部核心状态和分支直接写死。简单说，Mock 应该帮你去掉噪音，而不是把被测对象本身也一并伪造掉。

39. 如何为流式请求、时间函数、浏览器 API 做 Mock？
答：核心思路是先把这些能力封装成可替换边界，再在测试里控制它们的输入和时序，这样结果才可预测、可重复，比如流式输出可以用可控的分段数据模拟，时间函数可以用 fake timers 驱动，`localStorage`、`ResizeObserver`、`IntersectionObserver` 这类浏览器 API 也适合集中封装后统一替换。重点不是记某个库怎么写，而是让复杂外部时序在测试里变得可控。

40. 你如何理解“测行为，不测实现”？
答：这句话的意思不是完全不看实现，而是优先让断言围绕用户能观察到的结果展开，比如按钮点击后页面是否更新、提示是否出现、回调是否被触发、权限是否生效，而不是先盯着某个内部 state 有没有改。因为实现会随着重构不断变化，但用户行为目标往往更稳定，测试如果绑在行为上，就更能跨过代码演进周期存活下来。

## 七、Playwright 与 E2E

41. `Playwright` 适合解决什么问题？
答：`Playwright` 适合验证真实浏览器里的关键用户链路，比如登录跳转、文件上传、跨页面流转、复杂弹窗交互、下载、真实权限路径等，因为这些场景往往依赖浏览器环境、网络时序和完整页面上下文，单测很难完全覆盖。它更像是在模拟真实用户操作产品，而不是只在代码层确认某个函数返回值。

42. `Playwright` 和单元测试、集成测试的边界是什么？
答：更合理的分工是让单元测试守稳定规则，让集成测试守局部流程，让 `Playwright` 去兜那些只有在真实浏览器和完整链路里才容易出问题的部分。也就是说，它不是替代前两者，而是最后一层高成本但高价值的回归保护网。

43. 登录、上传、表单提交流程为什么适合写 E2E？
答：因为这些流程通常跨越多个模块和上下文，比如输入、校验、请求、跳转、权限、浏览器原生能力、文件系统、重定向等，一旦哪一步错了，对用户和业务影响都非常直接。离开真实浏览器环境，这类问题经常只能测到“局部”，却测不到完整路径是否真的通。

44. E2E 测试为什么容易慢、脆弱、维护成本高？
答：因为它依赖整个应用、测试环境、页面结构、测试数据、网络时序甚至机器性能，链路一长，不确定性就会快速上升，任何小波动都可能导致失败。正因为它昂贵，所以更要克制范围、重视观测和排查能力，否则很快就会演变成“谁都不信测试结果”的负资产。

45. 如何挑选最值得写 E2E 的关键链路？
答：我会优先看几个维度：业务价值高不高、失败损失大不大、人工回归成本高不高、跨模块程度强不强、历史上是否容易出问题，然后把最不能坏的主路径优先纳入 E2E。它的目标不是追求场景数量，而是尽量用少量高价值用例守住大部分高损失风险。

46. 在 CI 里跑 E2E 测试要注意什么？
答：重点通常不在“能不能跑”，而在“跑得稳不稳、失败后能不能查”，所以要关注环境一致性、测试数据准备、超时策略、并发控制、失败重试、截图录屏和 trace 保留等问题。否则 CI 里最容易让团队反感的往往不是测试数量，而是每次失败都分不清到底是产品问题还是环境抖动。

47. 失败的 E2E 用例如何高效排查？
答：我一般会先看录屏、截图、控制台日志、网络请求和 trace，先判断失败发生在页面渲染、用户交互、接口响应还是环境异常，再结合失败前最后一步操作快速缩小范围。E2E 排查效率很大程度取决于你是否提前把可观测信息准备好，否则失败一次就要靠人工重放，成本会非常高。

## 八、测试与可访问性

48. 为什么 `React Testing Library` 的查询方式天然和可访问性有关？
答：因为它鼓励你按角色、名称、标签文本这些用户真实能感知的信息去定位元素，而不是按类名和结构去找，这和无障碍设计强调的“语义清晰、名称明确、状态可理解”本来就是同一件事。很多时候，测试和可访问性是在互相促进，组件越语义化，测试就越自然。

49. 为什么写测试时用 `getByRole` 往往能顺带提升组件语义化质量？
答：因为当一个组件很难被 `getByRole` 自然找到时，通常意味着它的语义结构本身就不够标准，比如按钮其实是 `div`，或者可见文案和可访问名称脱节。测试写得顺不顺，在这里其实是在反映组件本身是否更接近用户和辅助技术的理解方式。

50. 如何测试键盘导航和焦点管理？
答：我会用更接近真实行为的方式模拟 `Tab`、`Enter`、`Escape`、方向键这些操作，然后观察焦点是否按预期移动、组件状态是否同步变化、关闭后是否回到合理位置。对 Modal、Dropdown、Menu、表单错误定位这类组件来说，键盘和焦点测试往往比单纯点鼠标更能发现深层问题。

51. 为什么无障碍设计会影响测试稳定性和表达质量？
答：因为语义越规范、交互越标准，你的测试越容易写成贴近用户的稳定断言，比如按角色找元素、按键盘路径操作、按可见反馈判断结果；反过来，如果组件本身语义混乱、行为反直觉，测试通常也会被迫依赖很多脆弱细节。也就是说，好的可访问性往往会顺手带来更健康的测试写法。

## 九、工程实践

52. 测试覆盖率高就一定代表质量高吗？
答：不一定，覆盖率只能告诉你“这些代码被执行过”，却不能保证关键风险真的被守住了，比如一堆低价值快照把覆盖率堆上去，也可能对主流程回归毫无帮助。相比单看覆盖率，我更关注关键路径有没有被覆盖、失败是否能阻断问题流入生产，以及测试本身是否真的在守业务风险。

53. 哪些模块最值得优先补测试？
答：我通常会优先补核心业务链路、权限逻辑、复杂表单、公共组件、历史高频故障模块和那些经常改但一坏就影响面很大的区域。补测试最忌讳平均用力，因为测试资源永远有限，应该先把风险最高、收益最大的地方做出效果。

54. 如何在项目时间紧张时设计“最小但有效”的测试策略？
答：这时候重点不是追全面，而是追有效，也就是先守关键路径和高风险点，把最稳定、最有复用价值的规则下沉到单测，再补少量关键流程的集成测试，如果还有余力再给最不能坏的链路加 E2E。只要能明显降低回归风险，这套“最小策略”就是有价值的。

55. 如何把测试接入 CI 流水线？
答：我的思路是按成本和阻断级别分层接入，比如提交和合并前先跑静态检查、单测和核心集成测试，发布前再跑必要的 E2E，并且明确哪些失败必须阻断、哪些只是告警。测试真正产生工程价值，靠的不是有人本地偶尔跑一下，而是它被放进团队的提交、合并和发布流程里。

56. 测试失败阻塞发布时，如何权衡效率和风险？
答：我会先判断失败的是不是关键用例、是真缺陷还是环境抖动、是否已有替代兜底，再决定是立即修复、临时隔离、降级发布还是延后上线。成熟团队不会因为赶时间就默认把测试报警当噪音，但也不会完全不区分地让低价值 flaky case 长期绑架交付效率。

57. 老项目补测试时，你会从哪里开始？
答：我一般会从最核心、最稳定、最容易出回归问题的部分入手，因为这类模块既有业务价值，又更容易写出能长期保留的测试，能让团队先看到正反馈。直接从最乱、最耦合、最难测的区域开刀，往往容易把补测试这件事一开始就做成高挫败感工程。

## 十、项目结合题

58. 在后台管理系统里，你会优先测哪些模块？
答：我会优先测登录和权限、复杂表单、列表筛选、审批流、关键配置页和危险操作确认这些模块，因为它们规则密集、角色分支多、业务影响直接，而且一旦出错常常不是“页面不好看”，而是“流程走错了”或“权限出问题了”。后台系统测试的重点通常是流程正确性和权限边界。

59. 在 `AI Agent` 或 `AI Workflow` 项目里，你会优先测哪些功能？
答：这类项目里我会重点盯节点配置、执行流程、权限控制、流式结果展示、异常回退和重试恢复，因为它们的风险往往来自状态流、异步时序和多节点协作，而不仅仅是页面渲染本身。很多问题在线上暴露时，看起来像“结果不对”，根因其实是某个状态转换或边界条件没有被测到。

60. 流式响应、节点配置、权限路由、复杂表单分别如何设计测试策略？
答：我的思路是把稳定规则、数据转换和状态机尽量放到单测，把组件内交互流程和局部业务协作放到集成测试，再用少量 E2E 去守跨页面主链路和真实浏览器能力，比如权限跳转、完整执行流和复杂提交。复杂业务越多，越不能指望某一层测试包打天下，而要把不同风险分散到最合适的层级。

61. 如果一个问题在线上才暴露，事后你会补哪一层测试？
答：我不会机械地说“都补 E2E”，而是会先分析根因落在哪一层边界，再补成本最低但能真正防止同类问题回归的那一层，比如纯规则问题优先补单测，模块协作问题补集成，真实链路问题再考虑 E2E。补测试的目标不是情绪化“补一堆”，而是建立最有效的防回归闭环。

## 十一、综合表达

62. 如果面试官问“你们项目测试做得怎么样”，你会怎么回答才不空？
答：我会先诚实交代测试现状，比如目前单测守了哪些稳定逻辑、集成测试覆盖了哪些主流程、E2E 守了哪些关键链路，再补充测试是怎么接入 CI、哪些地方还有空白、为什么优先级暂时没拉起来。这样回答会比单独报一个覆盖率数字更像真实项目，也更能体现你对风险和投入的判断。

63. 如果你没系统做过测试，如何诚实回答并体现认知深度？
答：可以坦诚自己实际落地经验还在累积，但同时把测试分层、价值判断、优先级和实施路径讲清楚，比如知道什么适合单测、什么适合集成、什么该留给 E2E，也知道如何在项目时间有限时先守关键风险。大厂通常能接受经验还不够完整，但不太接受只会背工具名、没有测试思维的人。

64. 如果让你从零搭一套前端测试方案，你会怎么规划？
答：我会先按风险分层而不是先选工具，也就是先识别哪些链路最不能坏、哪些规则最稳定、哪些模块最常改，再据此选择单测、集成和 E2E 的覆盖重点，然后再确定 `Vitest`、`React Testing Library`、`Playwright` 这类工具组合，并把它们接进 CI、阻断策略、失败排查和团队规范里。测试体系真正重要的不是一次性铺满，而是能随着项目演进持续维护、持续提供回归价值。
