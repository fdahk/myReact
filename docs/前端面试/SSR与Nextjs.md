# SSR 与 Next.js 专题

## 使用方式

这个专题的核心不是背渲染模式名词，而是说明不同渲染策略在首屏、SEO、缓存、服务端成本和交互复杂度上的取舍。真正成熟的回答，通常会先说明业务目标，再说明为什么选某种模式，以及它会带来哪些新的工程边界。

---

## 一、渲染模式基础

1. CSR、SSR、SSG、ISR 的核心区别是什么？
答：核心区别在于 HTML 是什么时候、在哪一侧生成出来的，也就是“页面最开始那份内容是谁先拼好的”。CSR 是浏览器先拿到一个壳和一堆 JS，再由客户端执行脚本把页面拼出来；SSR 是每次请求到来时，由服务端先把当前页面需要的 HTML 生成好再返回；SSG 是在构建阶段提前把页面生成成静态文件，线上请求时直接返回现成结果；ISR 则是在静态页面的基础上允许按策略重新生成，让内容不是永远固定死。进一步看，这几种模式本质上是在首屏速度、SEO 友好度、服务器成本、缓存收益和内容实时性之间做不同权衡。

2. 这四种渲染模式分别适合什么场景？
答：CSR 更适合登录后后台、工作台、富交互应用这类页面，因为这类页面通常 SEO 诉求低，但交互复杂、状态多、页面切换频繁；SSR 更适合首屏和 SEO 都重要、而且内容需要较新数据的页面，比如资讯页、活动页、部分详情页；SSG 适合内容稳定、更新不频繁的页面，比如官网介绍页、帮助文档、部分博客页；ISR 适合内容会更新、但又希望尽量吃到 CDN 缓存收益的页面，比如新闻站、博客站、商品详情的非强实时部分。选型关键不在模式先进不先进，而在页面目标到底是什么。

3. 为什么说渲染模式本质上是在平衡首屏、SEO、服务器压力和交互复杂度？
答：因为这些指标经常互相拉扯，没有一种模式能同时在所有维度都最优。比如 SSR 拿到了首屏直出和更好的 SEO，但也引入了服务端计算、缓存设计、运行环境和监控复杂度；CSR 交互组织相对直接，前后端边界也更清晰，但首屏更依赖 JS 下载和执行，弱网体验与搜索引擎可见性通常更差。所以渲染模式不是“谁先进就用谁”，而是在不同成本之间做取舍，本质上是一道工程权衡题。

4. 纯 CSR 的优点和缺点是什么？
答：纯 CSR 的优点是架构相对简单、前后端边界清晰、开发心智统一，页面切换和复杂交互通常也更自然，尤其适合登录后复杂应用；缺点是浏览器往往要先下载 JS、执行代码、再把首屏内容真正渲染出来，所以首屏白屏时间、弱网体验和 SEO 表现通常不如服务端或静态直出方案。它不是“落后方案”，而是很适合某类场景的取舍，特别是当页面核心价值在交互而不是搜索流量时，CSR 反而可能是最划算的选择。

5. SSR 的优点和缺点是什么？
答：SSR 的优势是首屏能直接返回较完整的 HTML，用户更早看到内容，搜索引擎也更容易抓到页面主体，因此对首屏感知和 SEO 都比较友好；代价是每次请求都可能触发服务端渲染和数据获取，对运行环境、缓存、接口稳定性、依赖兼容性、错误监控的要求都会更高。本质上，它不是凭空让页面变快，而是把一部分原本在客户端完成的工作前移到了服务端，所以如果页面完全不做缓存、服务端接口又慢，SSR 不一定比 CSR 体验更好。

6. SSG 为什么通常拥有更好的性能和缓存收益？
答：因为页面在构建阶段就已经产出了静态 HTML，线上请求时通常可以直接从 CDN 或静态资源服务器返回，不需要每次现算，所以服务端实时计算压力最低，缓存命中率也往往最高。它之所以性能常常很好，不是因为“SSG 这个名词更高级”，而是因为绝大多数请求都在拿现成结果，路径最短、变量最少。它的前提是内容足够稳定，或者业务能接受“更新要等重新构建或再生成”这件事。

7. ISR 解决了什么问题？
答：ISR 解决的是“内容会更新，但又不想每次请求都做 SSR”的问题，也就是在静态化和实时性之间找一个更实际的平衡点。它的思路不是每次都现算，也不是一构建就永远不变，而是先给用户一个可缓存的静态页面，再按规则在后台更新，让大多数请求仍然吃到静态化和 CDN 的收益，同时又不至于让内容长期过旧。理解 ISR 的关键，是把它看成“允许一定延迟的数据新鲜度，换取更高的性能和更低的服务端成本”。

8. 如何向面试官解释“没有最好的渲染模式，只有最适合业务的模式”？
答：我会先说页面目标是什么，是更看重 SEO、首屏、内容实时性，还是更看重交互复杂度、开发效率和服务器成本，然后再推导模式选择。比如面向搜索流量的内容页和登录后工作台，本来就不是同一种问题，硬要统一渲染模式往往反而不合理。这样的回答会比直接站队某个模式更像工程选型，因为它体现的是“按目标反推方案”，而不是“先迷信某个技术名词”。

## 二、Next.js 基础认知

9. 为什么很多团队会选择 `Next.js`？
答：因为它把路由、SSR/SSG、构建、数据获取、部署约定、图片优化、错误边界等一整套 Web 应用基础设施整合好了，团队不用从零拼装一整条 SSR 工程链。它真正的价值是降低 React 做服务端渲染和多种渲染模式混用时的工程门槛，让团队把更多精力放在页面目标、缓存设计和业务实现上，而不是自己反复造路由、构建、首屏直出、部署约束这些轮子。所以它不是“React 的升级版”，更准确地说，它是构建 React Web 应用的一套框架级方案。

10. `Next.js` 相比纯 `React` 项目多解决了哪些问题？
答：纯 `React` 更像 UI 视图层库，重点解决组件开发和声明式渲染；`Next.js` 则进一步把文件路由、服务端渲染、静态生成、数据获取、页面级元信息、图片优化、加载和错误边界、部署运行约束等 Web 应用层面的能力整合起来。换句话说，纯 React 主要回答“组件怎么写”，而 Next.js 会继续帮你回答“页面怎么组织、什么时候生成、数据在哪拿、首屏怎么优化、如何上线部署”这些更靠近完整产品交付的问题。

11. `Next.js` 为什么常被认为更适合内容型站点和 SEO 场景？
答：因为它天然支持 SSR、SSG、ISR 和页面级元信息管理，能更容易把内容在首屏直出给用户和搜索引擎，而不是让页面主体严重依赖客户端 JS 执行后才出现。对内容站、官网、博客、活动页、资讯页这类场景来说，搜索流量、首屏感知和缓存收益通常都很关键，这时 Next.js 提供的多种渲染策略和官方默认工程能力就会特别有价值。它适合这类场景，不是因为“它天生就是 SEO 框架”，而是因为它更方便你做对这些场景真正重要的事。

12. 你如何解释 `Next.js` 的“开箱即用工程能力”？
答：我的理解是，很多原本要团队自己拼起来的基础设施，比如路由规则、渲染策略、构建默认值、资源优化、加载和错误边界、图片处理、部署约束，它都给了相对成熟的官方方案。所谓“开箱即用”，并不是说不用思考，而是你不用先把底层骨架从零搭一遍，就可以直接在统一约束里做业务。这样团队能把更多精力放在业务建模和渲染策略取舍上，而不是花大量时间自己维护一套并不构成核心竞争力的框架层代码。

13. Pages Router 和 App Router 的核心区别是什么？
答：Pages Router 更接近传统页面路由和数据获取模式，心智上更像“每个页面一个入口，再通过 `getServerSideProps`、`getStaticProps` 这类方法决定怎么取数和生成”；App Router 则围绕布局复用、React Server Components、流式渲染和更细粒度边界设计，心智更接近“页面是由布局、服务端组件、客户端交互岛、局部加载和局部错误边界拼起来的”。两者不只是目录结构变化，更重要的是渲染和数据获取模型整体升级了，所以回答这题时最好把“思维模型变化”也讲出来，而不只是说 `pages` 变 `app`。

14. App Router 带来了哪些新能力？
答：比较关键的是嵌套路由布局、Server Components、`loading`/`error` 边界、流式渲染以及更自然的服务端取数方式。这些能力的共同目标，是把页面拆成更细粒度的可组合区域，让稳定骨架、服务端内容、客户端交互、局部加载态和局部错误处理能更自然地协作。尤其是在大型页面里，它的价值不是“目录更新了”，而是让渲染边界和组件职责更清晰，减少整页都必须在客户端完成的负担。

15. Layout、Nested Routes、Loading、Error 边界在 `Next.js` 中各自扮演什么角色？
答：`Layout` 用来复用稳定骨架，比如导航、侧边栏、统一外壳；`Nested Routes` 用来组织分层页面结构，让不同区域能沿着路由层级组合；`Loading` 负责某个局部区域在异步等待时的过渡反馈；`Error` 负责某个局部区域出错时的兜底和隔离。它们一起解决的是“大页面里不同部分能不能独立加载、独立失败、独立复用”的问题，也正因为如此，App Router 更强调边界拆分，而不是把整页当成一个不可分割的大组件。

## 三、数据获取与缓存

16. 在 `Next.js` 中常见的数据获取方式有哪些？
答：常见方式包括服务端取数、构建时取数、客户端补取数，以及 App Router 下在 Server Component 中直接获取数据。不同方式表面上是在“哪里写请求”，本质上是在决定数据是在构建时准备好、请求时现拿、还是首屏后再补拉，这会直接影响首屏内容是否可见、缓存怎么做、数据有多新、服务器压力有多大。所以理解 Next.js 的数据获取，关键不是背 API，而是先想清楚页面需要什么时机的数据。

17. Pages Router 中 `getServerSideProps`、`getStaticProps`、`getStaticPaths` 分别适合什么场景？
答：`getServerSideProps` 适合每次请求都需要较新数据的页面，因为它会在请求时执行；`getStaticProps` 适合可提前生成的静态页面，因为它会在构建阶段准备数据；`getStaticPaths` 则主要服务于动态静态页的路径预生成，让系统知道哪些动态路由要提前产出静态页面。它们最核心的区别，其实就是页面内容在什么时机被生成出来，以及这些页面是更偏“现算”还是更偏“预产出”。

18. 为什么 `getServerSideProps` 用得太多会加重服务器压力？
答：因为每次请求都会执行一次服务端逻辑，通常还会伴随数据获取、模板拼装和序列化处理，请求量一上来，SSR 就不再只是前端页面层的问题，而是变成计算资源、接口延迟和缓存命中率的问题。没有缓存设计时，SSR 成本会被迅速放大，最终可能表现成 TTFB 增长、接口压力升高、服务端实例数上升。所以它不是不能用，而是要知道“每次请求现算”本身就是一种需要被认真计价的成本。

19. ISR 中的 `revalidate` 是怎么理解的？
答：可以把 `revalidate` 理解成这份静态内容的“允许过期窗口”或“再生成间隔”，也就是在这段时间内，系统可以继续返回旧的静态页面；等到窗口到期后，由后续请求触发后台再生成，让新内容逐步替换旧内容。它不是严格意义上的实时刷新，也不是一到时间页面就立刻同步更新，而是一种“允许短暂旧数据，换取更高性能和更稳缓存命中”的策略。理解它时，重点要放在“新鲜度是有上限承诺，但不是毫秒级实时”。

20. 如何为一个新闻站、博客站、电商详情页分别选择数据获取策略？
答：新闻站通常更适合 ISR，因为内容更新频繁、流量大、又需要尽量吃 CDN 缓存收益；博客站如果内容更新没那么高频，往往偏 SSG 或 ISR；电商详情页则要拆信息类型来看，商品介绍、图片、基础文案、评价摘要这类偏稳定内容可以静态化，价格、库存、活动信息这类强实时内容更适合 SSR 或客户端补取。同一页面完全可以混合策略，关键不是整页贴一个标签，而是把高实时和低实时内容拆开设计。

21. 服务端缓存、CDN 缓存和浏览器缓存如何配合？
答：服务端缓存主要是降低渲染和取数计算成本，减少每次请求都去重复做相同工作；CDN 缓存主要是把内容提前分发到更靠近用户的节点，缩短传输距离并抗住流量；浏览器缓存主要提升用户重复访问体验，减少重复下载资源。SSR 项目里这三层通常要配合设计，因为页面即使在服务端渲染出来了，如果每次都重新算、边缘节点也接不住、客户端静态资源还总重复下载，那首屏收益很容易被后端成本和网络损耗吃掉。

22. 为什么 SSR 场景比 CSR 更需要关注缓存设计？
答：因为 CSR 的主要压力在客户端下载和执行，虽然也有接口压力，但页面本身通常不会因为访问量大而额外增加服务端 HTML 渲染成本；SSR 则不同，每个请求都可能触发服务端渲染和数据获取，如果没有缓存，服务端很容易成为瓶颈，尤其在高流量场景下更明显。实际落地时通常会结合 `Cache-Control`、CDN、页面级静态化和数据级缓存一起设计，而不是只押宝单一缓存层，因为 SSR 真正能否跑得稳，很大程度取决于你有没有把“每次请求现算”的范围控制住。

## 四、Server Components 与 Client Components

23. 什么是 React Server Components？
答：React Server Components 可以理解成“运行在服务端、主要负责生成 UI 结果、但自身代码不需要发到浏览器执行”的组件。它和传统 SSR 容易混淆，但重点不完全一样：SSR 更强调“首屏 HTML 由服务端先生成”，而 RSC 更强调“哪些组件逻辑可以彻底留在服务端，不进入客户端 bundle”。它的价值不只是服务端也能渲染，而是把那些不需要交互、不依赖浏览器环境的 UI 逻辑留在服务端，从源头减少客户端需要下载和执行的代码。

24. 为什么 Server Components 能减少客户端包体积？
答：因为这部分组件代码不会进入客户端 bundle，浏览器只拿到渲染结果和必要的交互边界，而不需要把所有展示型组件的实现都下载下来。对于内容展示多、交互局部化的页面，这个收益会非常明显，因为客户端需要承担的 JS 解析和执行压力会下降。换句话说，它不是“少渲染了一次”，而是“有些组件根本不需要出现在浏览器里”。

25. Server Component 和 Client Component 的边界如何理解？
答：我的原则是能放服务端就尽量放服务端，只有依赖本地状态、事件处理、浏览器 API、客户端上下文或必须在浏览器运行的第三方库时，才下沉成 Client Component。这个边界本质上是在回答“这段逻辑到底必须由浏览器执行吗”，而不是在回答“我写习惯哪种组件”。边界划得越合理，客户端包体积、Hydration 压力和首屏交互负担通常就越轻。

26. 什么情况下一个组件必须写 `'use client'`？
答：当它要使用 Hook、事件处理、浏览器对象、动画库、依赖客户端上下文，或者需要在用户交互后即时响应时，就必须标成客户端组件。这个标记的本质是在声明“这部分逻辑需要浏览器执行”，因此它不是一个普通字符串，而是一个明确的运行边界声明。很多人会把它当成“为了解决报错随手加一下”，但从架构上看，每多一个 `'use client'`，就意味着又有一块代码被送回了浏览器。

27. 为什么把过多组件都标成 `'use client'` 会削弱 Server Components 的收益？
答：因为你等于把大量原本可以留在服务端的代码和依赖重新送回浏览器，客户端 bundle、Hydration 工作量和运行时开销都会变大，Server Components 降包体、减轻客户端压力的收益就会被明显削弱。很多项目里最大的误区就是“框架支持服务端组件了，但实际写法上几乎整片区域都加了 `'use client'`”，结果名义上用了新能力，实际运行特征还是老式重客户端方案。

28. Server Component 为什么不能随意使用浏览器 API 和 `useEffect`？
答：因为它本来就不在浏览器里执行，没有 `window`、`document`、`localStorage` 这些环境，也不存在浏览器端副作用触发时机。`useEffect` 的本质是“在组件渲染提交到客户端之后执行副作用”，而 Server Component 并不跑在那个阶段，所以自然不能直接使用。这个限制本质上是在提醒你：服务端组件和客户端组件职责不同，前者更偏“准备内容”，后者更偏“承接交互和副作用”。

29. 如何在 Server Component 和 Client Component 之间传递数据？
答：通常是服务端先获取和整理数据，再通过 props 传给客户端组件，让客户端组件只负责交互和局部状态处理。这样既能保留服务端取数和减包体的优势，也能让浏览器端专注做真正必须在本地完成的事情。理解这一点时，可以把 Server Component 看成“把首屏数据和结构先准备好”，把 Client Component 看成“在这个基础上接管需要互动的那一小块”。

30. 如果一个页面既需要 SEO，又需要复杂交互，该如何划分组件边界？
答：我会让首屏内容、SEO 信息、稳定展示区域和可提前拿到的数据尽量放在服务端，让交互复杂、依赖浏览器状态或用户连续操作的局部区域做成 Client Component。这样拆边界比笼统地说“整页都用 SSR”更符合真实工程实现，因为真实页面往往不是整体要么全服务端、要么全客户端，而是“外层内容要早到、内层交互要灵活”。好的边界设计，本质上是在把页面拆成不同责任区。

## 五、Hydration 与 Hydration Mismatch

31. 什么是 Hydration？
答：Hydration 可以理解成“客户端接管服务端已经画好的页面”，也就是浏览器先拿到 SSR 输出的 HTML，把首屏内容直接显示出来，然后 React 再在客户端把对应组件树接上事件、状态和交互能力，让这份原本只是静态展示的 HTML 变成真正可操作的应用。也就是说，SSR 只是先把页面“画出来”，Hydration 才真正把这张图“接上线”，所以它是从静态内容过渡到可交互应用的关键步骤。

32. 为什么服务端渲染完 HTML 后，客户端还需要 Hydration？
答：因为只有 HTML 还不够，按钮不会点、输入框不会响应、状态不会同步、客户端事件也没有绑定上去。服务端先返回 HTML 的意义是让用户更早看到内容，但页面要真正变成应用，客户端仍然需要基于同一棵组件树接管事件和交互逻辑。可以把它理解成服务端先把“外观”准备好，客户端再把“行为”补上，两者配合才构成完整的 SSR 页面体验。

33. Hydration mismatch 是什么？
答：Hydration mismatch 就是服务端首屏 HTML 和客户端第一次渲染结果不一致，React 在接管时发现“两边对不上”。它本质上是“同一个页面在服务端和客户端跑出了两套不同结果”，比如服务端写的是一个时间，客户端重新算出来又是另一个时间，或者服务端没有 `window` 分支，客户端有 `window` 分支，于是节点结构不一致。这个问题的关键不在于报错文案，而在于它暴露了首屏渲染缺少确定性。

34. 常见的 Hydration mismatch 原因有哪些？
答：最常见的是时间、随机数、时区格式化、浏览器专属 API、依赖环境差异的条件分支，以及不稳定的第三方组件。只要首屏渲染依赖了某种在服务端和客户端可能算出不同结果的值，就有可能出现 mismatch。很多人会把它理解成“React 偶发抽风”，其实大多数情况下根因都很朴素，就是你让首屏内容依赖了不稳定因素，导致两边第一次渲染并不是同一份结果。

35. 为什么 `window`、`document`、`localStorage` 容易导致 Hydration 问题？
答：因为服务端没有这些对象，客户端有，如果首屏逻辑依赖它们，两边渲染分支就可能不同。比如服务端只能走“没有本地存储值”的分支，客户端一运行就读到本地数据，于是页面内容立刻变了，这就很容易让初始 HTML 和客户端第一次渲染不一致。很多 Hydration 问题并不是 React 本身有 bug，而是代码把“只能在浏览器决定的逻辑”提前放到了首屏渲染阶段。

36. 为什么 `Math.random()`、时间戳、时区格式化容易导致服务端和客户端内容不一致？
答：因为这些值天然不稳定，服务端算一次、客户端再算一次，结果很可能不同，而且时区、语言环境、运行节点环境差异还会进一步放大这种不一致。只要这些值影响了首屏输出的文本、节点结构或属性，就可能直接触发 mismatch。理解这类问题时要抓住一个核心：Hydration 需要两边的第一次渲染尽量一致，而随机值和时间值天生就在破坏这种一致性。

37. 如何排查 Hydration mismatch？
答：我会先从首屏里所有非确定性值入手，逐步排查时间、随机数、时区格式化、环境分支、浏览器 API、第三方组件和条件渲染，再看是不是某些数据只在客户端可得。排查的核心不是“到处试”，而是先定位哪些内容在服务端和客户端最可能不一致，然后缩小范围，看究竟是文本不同、结构不同，还是某个组件根本不支持 SSR。只要抓住“同一页面两侧结果为什么会不一样”这条线，排查通常就不会太乱。

38. 什么逻辑适合放到 `useEffect` 里延后到客户端执行？
答：像读取本地存储、获取窗口尺寸、注册事件监听、访问浏览器 API、依赖用户设备环境的逻辑，或者那些不需要影响首屏 HTML 的操作，都适合延后到 `useEffect`。这样可以保证服务端输出和客户端第一次渲染尽量一致，把真正只能在浏览器决定的事情留到接管之后再做。它的意义不是把所有问题都塞进 `useEffect`，而是明确区分“首屏必须稳定输出的内容”和“可以在客户端后置执行的行为”。

39. 如果线上出现 Hydration 报错，你会优先检查哪些地方？
答：我会优先看时间和随机值、`window` 相关逻辑、按环境分支渲染的代码、时区格式化、以及不支持 SSR 的第三方库，因为这些地方最容易制造“服务端和客户端各渲各的”。如果问题只在生产环境出现，我还会进一步看是否和压缩、部署节点环境、Node 版本、默认时区或线上数据差异有关。面试里这题最重要的不是列名词，而是体现你知道要先抓“首屏不确定性来源”。

## 六、SEO 与首屏优化

40. 为什么 SSR 或 SSG 更有利于 SEO？
答：因为搜索引擎和用户一开始就能拿到较完整的页面内容，而不是等前端脚本在浏览器里把主体内容慢慢拼出来。即使现代搜索引擎能执行 JavaScript，SSR/SSG 依然在抓取稳定性、内容可见性和首屏可读性上更有优势，因为页面主体更早、更明确地出现在 HTML 里。对流量型页面来说，这种“内容尽早可见”往往就是 SEO 和用户感知体验的基础。

41. 现代搜索引擎已经能执行 JavaScript，为什么 SSR 仍然重要？
答：能执行不代表执行成本低、时效稳、抓取结果完全一致，也不代表所有场景都能稳定拿到完整页面内容。SSR 的价值除了 SEO 之外，还有首屏直出、低端设备体验更稳、弱网下内容更早可见，以及让用户在 JS 尚未完全执行前就先看到核心信息。所以今天继续谈 SSR，不应该只盯着“搜索引擎会不会执行 JS”，而应该看到它对真实用户首屏感知和内容分发稳定性的价值。

42. 首屏直出 HTML 对用户体验有哪些直接收益？
答：最直接的收益是用户更早看到页面骨架和主要内容，白屏时间更短，页面“有东西可看”的时间更早，弱网和低性能设备下感知尤其明显。它改善的不是所有性能指标，而是“用户多久能看到有意义内容”这个最直观的体验。很多时候用户并不会精确感知脚本解析速度，但会明显感知“页面打开后是不是先白一会儿”，而首屏直出正是在优先解决这个问题。

43. SSR 一定会让性能更好吗？为什么不一定？
答：不一定，因为如果服务端本身慢、缓存命中差、数据源慢、Node 运行环境吃紧、客户端包仍然很大，那么 SSR 只是把等待时间从浏览器挪到了服务端，用户未必真的更快。比如 TTFB 被明显拉长、Hydration 又很重，那用户虽然早看到了一点内容，但可交互时间可能仍然不理想。所以 SSR 的效果必须结合 `TTFB`、`LCP`、包体积、Hydration 开销和缓存一起看，不能只凭“用了 SSR”就默认性能提升。

44. 如何从 `LCP`、`TTFB`、包体积几个角度评价 SSR 效果？
答：我会看首屏关键内容是否更早出现，也就是 `LCP` 有没有改善；再看服务端响应是否被明显拉长，也就是 `TTFB` 有没有恶化；还要看客户端 JS 是否真的减少、Hydration 压力有没有下降，而不是只是把渲染方式换了但包体积几乎没变。只有当用户更早看到关键内容，同时服务端响应没有被拖得太慢，客户端执行成本也没有继续膨胀，SSR 才能说是有效优化，而不是表面上换了一种渲染模式。

45. `next/image` 解决了哪些性能问题？
答：它主要帮助处理图片尺寸控制、响应式加载、懒加载、格式优化和资源分发问题，避免图片以不合理的尺寸、格式和加载时机进入首屏。图片往往是很多页面首屏资源的大头，所以这类优化对真实用户体验非常直接，不只是节省流量，也会影响 `LCP` 和页面稳定性。它本质上是在把“图片应该以什么尺寸、什么格式、什么时机加载”这些容易做错的细节，尽量交给框架做统一约束。

46. `next/script` 和资源预加载策略在首屏优化里起什么作用？
答：核心是控制脚本和关键资源的加载优先级与执行时机，让真正影响首屏的资源尽早到达，而不那么关键的脚本不要阻塞页面主体显示。首屏优化不只是减少资源数量，更重要的是把资源到达顺序设计正确，因为关键资源晚来，用户就算总下载量不大，也还是会觉得页面慢。所以这类能力的本质是在做“资源调度”，而不是单纯做减法。

## 七、工程与部署视角

47. SSR 项目部署时为什么不能只看前端代码，还要看运行环境？
答：因为它不再只是静态资源托管，还依赖 Node 运行时、接口网络、缓存体系、部署节点、日志和监控能力。换句话说，SSR 项目已经跨进了“服务端应用”范畴，页面首屏能不能稳定返回，不只取决于前端组件写得好不好，还取决于服务端实例性能、网络链路、接口健康度和缓存设计。也正因为如此，SSR 项目的问题排查通常天然跨前后端边界。

48. 边缘运行时和传统 Node 运行时分别适合什么场景？
答：边缘运行时更适合低延迟、轻计算、贴近用户分发的响应场景，比如一些轻量请求处理、地理位置更敏感的内容分发；传统 Node 运行时更适合依赖丰富、逻辑更重、第三方库兼容性要求高的场景。选型关键是能力边界，不是边缘一定更高级，因为边缘运行时通常对 Node API、依赖生态和执行模型会有更多限制，所以要先看你的页面和服务到底需要什么能力。

49. 为什么某些依赖在 SSR 场景下会报错？
答：因为它们默认假设自己运行在浏览器里，直接访问 `window`、`document`、DOM、Canvas 或某些只有客户端才有的 API。到了服务端环境，这些前提不成立，自然就会报错。很多库在纯 CSR 项目里没问题，是因为它们从来没有在服务器上执行过；一旦进入 SSR 或 Server Components 场景，隐藏的环境假设就会暴露出来，所以“是否支持 SSR”其实是在问“这个库有没有把浏览器依赖和执行环境边界处理好”。

50. 如何处理只在浏览器环境可用的第三方库？
答：常见做法是动态导入、把它隔离到 Client Component、在 `useEffect` 中延后使用，或者直接替换成支持 SSR 的库。关键是不要让这类依赖参与服务端首屏渲染，也不要让它在服务端执行路径上提前访问浏览器对象。真正的处理思路不是“怎么把报错压下去”，而是先判断这个库到底属于首屏内容必须的一部分，还是其实只应该存在于客户端交互阶段。

51. 如何设计 SSR 项目的监控和错误上报？
答：我会把服务端渲染错误、接口耗时、缓存命中率、页面响应时间、客户端 Hydration 错误、资源加载失败和核心性能指标一起纳入监控。因为 SSR 项目排障通常必须同时看服务端和客户端两侧信号，只看前端报错或者只看后端日志都不够。一个页面慢，可能是服务端取数慢；一个页面报错，可能是首屏 HTML 正常但 Hydration 失败，所以监控设计本身就应该反映这种双端协作的现实。

52. 对于高流量页面，为什么要优先考虑缓存和静态化？
答：因为这类页面真正的瓶颈通常不是组件本身，而是在大量请求下服务端扛不扛得住。如果每个请求都现算、现拉数据、现渲染，再好的组件代码也挡不住流量压力；而静态化和缓存能把大量重复工作提前做掉或直接复用结果，这通常比继续细抠组件实现细节更能解决根本问题。高流量场景里，性能优化很多时候首先是“减少重复计算”，然后才轮到“让单次计算更快”。

## 八、常见面试追问

53. 为什么你们项目要选 `Next.js`？
答：我会从 SEO、首屏、路由约定、数据获取方式、团队落地成本、部署方案这些维度讲，而不是只说“社区火”或“大家都用”。真正好的回答是把项目诉求和框架能力一一对应起来，比如内容页需要搜索流量和首屏直出，工作台部分仍保留客户端交互优势，而 Next.js 正好能在同一个项目里提供多种渲染策略和统一工程方案。这样才能体现你是在做需求驱动的框架选择。

54. 为什么不用纯 `React` + `Vite`？
答：如果项目有明确的 SSR、SEO、首屏直出或多渲染模式混用诉求，纯 CSR 方案需要自己补很多基础设施，比如服务端渲染链路、缓存设计、页面级元信息、路由与部署协同、资源优化和错误边界管理。不是说不能做，而是工程成本和稳定性风险更高，团队需要自己承担更多框架层维护工作。所以这题不是在贬低 `React` + `Vite`，而是在说明当问题从“做一个前端应用”升级成“做一个需要多渲染模式支持的 Web 产品”时，框架层需求已经变了。

55. 你们项目里哪些页面适合 SSR，哪些适合 CSR？
答：我会区分面向外部流量的内容页和登录后工作台：前者更适合 SSR 或 ISR，因为更看重搜索引擎可见性、首屏直出和缓存收益；后者更适合 CSR，因为用户已登录、SEO 不重要、交互和状态复杂度更高。这样的回答能体现你不是把整个站点简单粗暴地统一成一种模式，而是知道不同页面承担的业务目标本来就不同，渲染策略也应该跟着页面目标走。

56. 你们是否遇到过 Hydration 问题？怎么解决的？
答：如果有真实经历，我会按现象、定位、根因、修复和防再犯来讲，比如先出现了哪些报错、具体是哪块首屏内容不一致、最后发现是时间格式化还是浏览器 API 提前参与渲染，再说如何通过后置到 `useEffect`、拆 Client Component、稳定首屏数据来修复；如果没有真实经历，我也会明确说出自己会优先检查哪些高风险点。关键是不要把这题答成纯概念题，而要体现你知道排障路径是什么。

57. `Next.js` 在你们项目里最大的收益是什么？
答：通常我会讲首屏、SEO、路由约定、统一工程方案、图片和资源优化、部署效率这些最直接收益，同时也会补充代价，比如缓存设计更复杂、运行环境和监控要求更高。这样回答才完整，因为框架收益如果不连同代价一起讲，听起来就像宣传稿。真正成熟的结论应该是：它帮我们更快、更稳地满足了某类业务目标，但也引入了某些新的工程边界。

58. 如果线上项目是传统 CSR，你如何回答自己“了解 SSR 吗”而不显得空？
答：我会坦诚说没有大规模落地过，但我能讲清原理、适用场景、性能取舍、Hydration 常见坑、缓存设计思路，以及如果要在真实项目里接入会先从哪些页面开始试点。大厂面试并不要求所有方向都做过，但要求你理解得不能停留在口号层。也就是说，不一定要求你有完整实战，但你至少得能把“为什么需要、怎么落地、会踩哪些坑”讲得像真的做过方案评估。

## 九、和 AI 项目的连接

59. AI 应用为什么也可能需要 SSR？
答：因为 AI 平台不只有聊天工作台，还常常有首页、落地页、案例页、内容页、SEO 营销页，这些场景同样需要首屏和搜索流量。不能因为项目带 AI 就默认所有页面都只适合 CSR，很多 AI 产品真正获取新用户的入口恰恰是内容页和营销页，而这些页面往往更适合服务端直出或静态化。换句话说，产品是否带 AI，不决定渲染模式；页面目标才决定。

60. AI 聊天、AI 内容页、AI 工作台分别适合哪种渲染策略？
答：AI 聊天和工作台通常更偏 CSR，因为交互重、状态复杂、用户上下文强、页面行为高度依赖浏览器端连续操作；AI 内容页和营销页更适合 SSR 或 ISR，因为更看重内容直出、首屏体验和搜索流量。关键还是按页面目标拆，而不是按产品整体拆，同一个 AI 平台完全可能同时拥有服务端优先页面和客户端优先页面，这才更符合真实产品形态。

61. 如果 AI 平台首页需要 SEO，但工作台区域需要复杂交互，你会如何拆分渲染模式？
答：我会让首页、营销内容、公开案例页这类面向外部流量的区域走 SSR 或 ISR，登录后的工作台走 CSR，或者做成外层服务端壳子加内层客户端交互区的组合方案。这样既能保证首页内容尽早可见、利于 SEO，又能让工作台保留更自由的客户端交互和状态管理。这个回答想体现的核心是：同一平台的不同区域完全可以采用不同策略，关键是按页面目标拆，而不是强迫全站统一成一种模式。

62. 流式输出和 SSR 可以如何结合？
答：可以先把页面骨架、导航、关键信息和能快速准备好的首屏内容优先返回，再随着后续数据到位逐步把异步区域流式补上。这样既保住了首屏“先看到东西”的体验，也能兼顾长耗时数据和复杂生成场景，不必等所有内容都准备完才一次性把整页吐给浏览器。理解流式渲染时，关键不是只记“streaming”这个词，而是理解它在解决“慢数据别拖住整页首屏”这个问题。

## 十、综合表达

63. 如果让你用 3 分钟回答 `Next.js` 和纯 `React` 的区别，你会怎么组织？
答：我会先讲定位差异，也就是纯 `React` 更偏 UI 视图层，而 `Next.js` 是更完整的 Web 应用框架；再讲渲染能力差异，比如 `Next.js` 提供了 SSR、SSG、ISR、Server Components 等选择，而纯 React 本身不直接解决这些；接着讲路由、数据获取、资源优化和部署约定这些工程能力；最后补一句业务选型边界和代价，比如它更适合首屏和 SEO 重要的场景，但也会带来缓存和运行环境复杂度。结构清楚会比堆术语更像成熟回答。

64. 如果让你设计一个需要 SEO、内容更新又频繁的站点，你会怎么选 SSR、SSG、ISR？
答：我大概率会以 ISR 为主，把强实时页单独用 SSR，稳定内容页走 SSG，因为这能在性能、缓存和更新效率之间找到比较平衡的方案。原因是这类站点既不能完全静态化到长期不更新，也不适合所有页面都每次请求现算，否则服务端成本会很高。真正合理的设计通常不是三选一，而是根据页面内容新鲜度要求做组合：低实时内容尽量静态化，高实时内容再少量 SSR。

65. 如果让你讲一次 Hydration mismatch 的排查经历，你会怎么讲？
答：我会按“现象是什么、范围怎么缩、根因是什么、怎么修、如何避免再犯”这条线来讲，比如先说明线上报了什么错误、是哪块区域首屏不一致、排查时先锁定了时间值还是浏览器 API，再说最终如何通过稳定首屏输出、后置客户端逻辑、替换不兼容库或拆分组件边界来修复。就算是理论题，也要答出真实排障思路，而不是只背一串原因列表，因为面试官更想听你遇到问题时会怎么定位和收敛。
