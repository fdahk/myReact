# AI 应用前端专题

## 使用方式

这个文档聚焦 AI 应用前端的高频深挖问题。整套内容会尽量按“业务目标、链路拆解、前后端边界、稳定性与取舍”展开，重点体现真实 owner 视角。

---

## 一、AI 项目介绍与实习表达

1. 你在 AI 相关实习里具体做了什么？
答：我会直接讲自己真正 owner 的前端链路，而不是泛泛地说“参与了 AI 项目”。比如我会明确说，我负责过对话页面、会话管理、流式渲染、工具调用结果展示、文件上传解析、调试面板和部分稳定性建设，并补一句这些模块分别解决了什么问题。这样面试官听到的不只是模块名，而是能更快判断我的职责边界、理解我在整条 AI 链路里承担了多深的责任。
2. 你的前端技术栈是什么？为什么这么选？
答：我会先说核心栈，比如 React、TypeScript、状态管理方案、请求层和构建工具，再把这些选择和项目复杂度、协作方式、稳定性要求对应起来。比如 React 适合承接复杂交互和组件复用，TypeScript 能降低多模块协作和重构时的出错率，Node.js 适合作为更贴近前端的中间层。重点不是报一串名词，而是讲清这些技术为什么适合 AI 产品这种异步多、状态多、结果不稳定、需要大量兜底和观测的场景。
3. 请完整介绍一下你的 AI 项目。
答：我会按“业务背景、核心用户场景、整体链路、我的职责、最难的问题、结果和反思”来讲，先说清楚项目解决什么问题、服务谁、为什么值得做，再把从用户输入到模型返回再到结果展示的完整链路串起来。之后我会明确自己真正负责的前端模块，最后挑一个最难点和一个最有结果的点展开。这样的讲法能同时体现业务理解、工程深度和个人贡献，而不会把项目介绍讲成一段模糊流水账。
4. 项目里的核心业务链路是什么？
答：我会从用户输入开始讲，说明前端如何整理当前会话上下文、如何发起请求到 BFF 或服务端、后端又如何编排模型和工具，最后再讲结果怎样流式返回、前端怎样增量渲染、消息和会话怎样持久化，以及异常如何被处理。AI 项目里最怕只讲页面长什么样，不讲链路怎么流，因为这类产品的难点往往不在静态界面，而在状态和数据如何在一条很长的链路里保持一致。
5. 你负责的模块有哪些？最难的点是什么？
答：我会先明确列出自己 owner 的模块，比如会话列表、消息流、工具调用展示、文件上传解析和调试面板，再挑一个最能体现能力的难点展开。这个难点最好不是简单做一个功能，而是能体现链路复杂度的点，比如流式消息状态机、会话切换时的请求隔离、大文件预处理导致的主线程卡顿，或者工具调用中间态和失败态展示。这样说的好处是能避免“什么都做过一点”的空泛感，更能证明我在关键链路上真正承担过责任。
6. 这个项目和普通后台项目最大的区别是什么？
答：我认为最大区别在于 AI 链路天然更长、更不稳定、状态更多，所以前端不再只是把接口数据展示出来，而是要承担流式体验、异常兜底、上下文一致性和可观测性。普通后台很多时候是确定性 CRUD，数据对了就结束了；AI 产品却经常会出现“服务没挂、接口也返回了，但结果并不稳定”的情况，这要求前端把不确定性翻译成用户能理解、能容忍、能恢复的体验。
7. 你在这段实习里最大的成长是什么？
答：我最大的成长是从“做页面”慢慢变成了“理解整条 AI 产品链路”。以前我更容易把注意力放在组件实现和局部交互上，现在我会主动考虑请求编排、状态机、观测指标、异常路径和上线风险。这个变化说明我在做前端时不再只盯着眼前页面，而是会去思考用户从输入到拿到结果的整条链路到底稳不稳、可信不可信。
8. 如果让你用 `3` 分钟讲清这个项目，你会怎么组织结构？
答：我会用“业务目标、用户路径、技术链路、我的职责、难点与取舍、结果”这六段式来组织。业务目标帮助面试官理解项目为什么存在，用户路径能让整条链路更有画面感，职责边界让成果归属清晰，难点与取舍体现工程判断，结果部分则体现我有没有结果意识。三分钟这种题最怕铺得太散，所以我会用固定结构帮助自己讲得更稳。

## 二、AI 应用前后端分工

9. 为什么一个 AI 项目里会同时使用 `Node.js` 和 `Python`？
答：因为两者通常承担的是不同层级的职责。`Node.js` 更适合做 BFF、接口聚合、流式转发、会话层和更贴近前端的协议收口，`Python` 更适合承接模型推理、Prompt 编排、RAG 检索、数据处理和算法能力。这个分层不是为了炫技术栈，而是为了让不同语言去做自己生态和团队协作上更擅长的事，从而让职责边界更清楚、协作成本更低。
10. `Node.js` 更适合承担哪些职责？
答：我一般会把 `Node.js` 放在离前端更近的一层，比如做请求聚合、鉴权、会话管理、流式透传、灰度开关接入和协议转换。因为这一层的工作更偏 I/O、编排和对外协议收口，它更强调“怎么把复杂后端能力整理成前端好消费的形态”，而不是承担复杂算法计算，所以 `Node.js` 在这里通常比较顺手。
11. `Python` 更适合承担哪些职责？
答：`Python` 更适合放在模型调用、RAG 检索处理、数据清洗、任务编排、推理服务和评测分析这类更靠近 AI 能力本身的位置。原因不是它语法更“高级”，而是相关框架、模型生态和数据处理工具更成熟，很多能力能用更低的成本接入、验证和迭代，所以在 AI 核心能力层它通常是更自然的选择。
12. 为什么说模型推理、数据处理、算法侧能力通常更偏向 `Python`？
答：因为主流模型框架、向量库、数据处理工具和实验生态更多集中在 `Python` 这一侧，团队接入这些能力时会更顺、更快，试错成本也更低。工程上选语言，很多时候选的不是语法好不好看，而是现有生态、人才储备和研发效率，所以这个问题的重点通常在“为什么更适合”，而不是“哪门语言绝对更强”。
13. 为什么前端 BFF、流式转发、权限和会话层经常会交给 `Node.js`？
答：因为这一层更强调和前端交互链路的贴合度，要承接协议适配、会话管理、用户态上下文和流式响应收口。`Node.js` 在这类位置开发效率高、和前端团队协作也更顺，还便于把后端原始能力重新整理成更适合 UI 消费的数据结构和错误语义，所以很多团队会把它放在前端和 AI 能力之间做一层缓冲。
14. 如果只用 `Node.js` 做整个 AI 服务，会遇到哪些问题？
答：不是不能做，而是会在模型框架、数据处理和现成算法生态上吃一些亏，很多本来在 `Python` 生态里现成可用的能力，换到 `Node.js` 里可能就要自己补适配、绕路或者接受功能选择更少。也就是说，用一种语言统一全栈并非做不到，但边际成本可能更高，尤其在 AI 能力快速试错阶段，这种差距会非常明显。
15. 面试里如果被问“为什么 Node 有问题”，你应该如何从生态和场景边界回答？
答：我的结论通常不会说 `Node.js`“有问题”，而会说它更适合离前端更近的那一层。它非常适合 BFF、流式转发和会话层，但不一定适合承担整条 AI 算法链路；如果把它放错位置，生态短板和工程成本就会被放大。成熟表达的重点从来不在语言站队，而在于你能不能讲清楚边界和取舍。
16. 什么情况下你会建议做前端 BFF？
答：当前端需要聚合多个后端接口、裁剪协议、统一会话和权限、做流式转发或统一埋点观测时，我会建议做 BFF。它真正的价值不在于“多了一层服务”，而在于把原本会散落在前端页面里的面向 UI 的复杂度集中收口，避免页面层逐渐变成一层难以维护的胶水层。

## 三、流式输出与 AI 聊天链路

17. AI 对话为什么经常使用流式输出？
答：因为模型生成通常天然有耗时，如果等完整结果一次性返回，用户会有很强的“系统是不是卡住了”的不确定感。流式输出能让用户更早看到系统已经开始响应，所以在 AI 产品里，首字时间很多时候比总耗时更影响体感。我会把流式输出理解成一种体验设计，而不只是传输方式，因为它直接影响用户是否愿意继续等下去。
18. 前端如何消费一个流式响应？
答：常见做法是通过 Fetch Stream 或 SSE 持续读取分块数据，按照约定协议解析，再增量更新消息内容和状态。真正重要的不是“会不会读流”这件事本身，而是要同时设计取消请求、异常提示、会话校验、重复 chunk 处理和结束态落地，否则一旦链路复杂起来，就很容易出现串消息、状态错乱或半截内容无法解释的问题。
19. 流式响应的数据格式通常怎么设计？
答：我会要求每段数据至少包含事件类型、内容 `payload`、消息 id、会话或请求标识、是否结束和错误信息；如果链路里还有工具调用、推理阶段或中间状态，我会再补阶段字段。协议设计清楚的意义不是“字段更整齐”，而是前端状态机更容易稳定，排障和观测也更好做，因为每个 chunk 回来时系统都知道自己现在处于哪一步。
20. SSE、WebSocket、普通 HTTP 分块流分别适合什么场景？
答：SSE 适合服务端单向持续推送，浏览器接入简单、语义也比较直观；WebSocket 更适合高频双向实时通信，比如协同编辑或需要前后端持续交换状态的场景；普通 HTTP 分块流则很适合“一次请求、多段返回”的生成式 AI 场景。我的选择标准始终是通信模式是否匹配，而不是哪个技术名词看起来更高级。
21. 为什么很多 AI 聊天接口并不一定非要上 WebSocket？
答：因为很多 AI 聊天场景本质上还是“一次提问，对应一次持续返回”，并不是持续双向高频交互，所以用 HTTP 流或 SSE 就已经够了，而且接入、网关治理和排障成本通常更低。只有当产品真的需要强双向、低延迟、长连接式的互动时，我才会认真考虑 WebSocket，而不是默认一谈实时就上它。
22. 流式输出中如果用户中途切换会话，前端如何防止旧数据串到新会话？
答：我会在切换时主动取消旧请求，同时要求所有 chunk 都带会话 id 或 `requestId`，在真正渲染前再做一次校验。也就是说，我会同时做“源头取消”和“落地校验”两层防线，因为流式场景下旧数据晚到是很常见的，只做其中一层都不够稳。
23. 如果流式请求中断了，前端如何做重试、兜底和提示？
答：我会先判断中断是用户主动取消、网络抖动、服务端异常还是正常结束，再决定是否允许重试以及重试应该怎么带上上下文。UI 上至少要有明确的中断态、重试入口和部分结果说明，避免用户看到半截内容却完全不知道发生了什么；长期上我还会把中断原因和链路日志打通，方便区分问题是在网络层、协议层还是应用层状态机。
24. 一个完整的 AI 聊天链路，从输入到最终渲染大致经历了什么？
答：用户输入后，前端先整理当前会话上下文、用户输入和必要的配置，再向 BFF 或服务端发起请求；服务端完成模型调用、工具编排和结果聚合；返回阶段前端持续解析流式数据、更新消息状态、展示工具调用中间态，并在结束后把消息和会话持久化。这个问题里我最关注的不是链路上的每个名词，而是“数据怎么流、状态怎么变、失败时怎么兜底”，因为这些才是 AI 聊天前端真正难的地方。

## 四、MCP、Function Calling 与 Agent 基础

25. 什么是 `MCP`？
答：我会把 `MCP` 理解成一套让模型或 Agent 通过统一协议接入外部工具和资源的机制。讲人话，它解决的是“模型如果要访问外部世界，到底该怎么用一种标准方式发现工具、知道参数、发起调用、拿回结果”这个问题。前端不一定要实现协议细节，但必须理解它在系统里承担的是工具接入标准层，这样才能把执行态、调试态和错误态设计清楚。
26. `MCP` 和 `Function Calling` 的核心区别是什么？
答：`Function Calling` 更像模型生成一个结构化的“工具调用意图”，由应用侧决定是否执行、怎么执行；`MCP` 更强调外部工具系统如何被标准化描述、发现和管理。前者更像“模型说我想调这个函数”，后者更像“系统先把所有可用能力按统一协议组织好，供模型或 Agent 去接”。一个偏调用意图，一个偏接入标准和协作机制。
27. 为什么说 `Function Calling` 更像“模型返回工具调用指令，由你的应用执行”？
答：因为模型本身通常不会直接去操作你的数据库、文件系统或业务接口，它只是基于当前上下文判断“这一步看起来应该调用哪个工具、传什么参数”。真正的执行、权限控制、结果回填和错误处理仍然由应用侧或运行时系统完成，所以这套机制的本质是“模型给建议，系统去执行”。这点讲清楚以后，前端也更容易说明自己在参数确认、执行态展示和结果回填上的职责。
28. 为什么说 `MCP` 更像“模型通过协议连接外部工具系统”？
答：因为 `MCP` 不只是一次函数调用，而是把工具和资源按统一协议暴露出来，让模型或 Agent 能发现有哪些能力、知道它们怎么用，并在调用后拿回结构化结果。对于前端来说，这种协议化能力最终会体现为调试面板里的工具列表、调用过程展示、参数结构和错误回放，所以理解它时要把它看成系统协作协议，而不是某个单独 API。
29. `MCP` 的完整调用链路是什么？
答：通常是用户先给出任务，模型或 Agent 根据当前目标判断是否需要外部能力，再通过协议发现合适的工具，按 schema 生成参数，发起调用并接收结果，最后再把结果回填给模型继续推理生成最终响应。前端如果做调试平台，就要把这条链路可视化出来，让用户知道模型为什么调用了这个工具、调用时传了什么、什么时候失败、最终结果又是什么。
30. `MCP` 里通常有哪些核心概念？
答：我会重点理解客户端、服务端、工具能力、资源、协议约定、调用结果和错误处理这些概念。关键不在于把每个字段死记硬背，而在于知道这套机制究竟在解决什么协作问题，以及它会怎样影响前端产品形态，比如工具列表怎么展示、执行中间态怎么反馈、失败时用户能看到多少上下文。
31. 用户 Prompt、大模型、工具调用三者是如何协作的？
答：Prompt 提供任务目标和上下文，大模型负责理解意图、规划步骤并决定是否需要调用工具，工具则负责补足模型本身并不具备的真实外部能力，比如查文件、调接口、读数据库。前端真正要做的，是把这三者之间的协作过程尽量可视化，让用户知道系统当前是在理解任务、在调用工具，还是已经拿到结果准备回答。
32. 在 AI Agent 项目里，前端最应该理解哪些非算法概念？
答：我认为最重要的是会话状态、执行阶段、工具调用链路、流式协议、调试能力、权限边界和观测指标。因为这些概念决定前端能不能把系统做成一个真正可用、可排查、可信任的产品，而不是只做出一个“输入框加结果区域”的演示页面。

## 五、大文件上传与分片

33. 为什么服务器经常处理不了超大文件一次性上传？
答：因为一次性上传会让单个请求持续很久，失败成本高，也更容易触发网关超时、连接中断和服务端内存压力。大文件上传本质上是在和长链路、不稳定网络做对抗，所以我通常不会把“一次性传完”当成默认方案，而是会优先考虑怎样把它拆成更容易恢复和追踪的过程。
34. 大文件上传为什么需要切片？
答：切片的核心价值，是把一个高风险的长任务拆成多个可控的小任务，便于并发上传、失败重试、断点续传和进度管理。它解决的不是“技术上能不能把文件传上去”，而是“在真实网络环境里，能不能更稳定、更可恢复地把文件传完”。
35. 前端分片上传的完整流程是什么？
答：通常是先切片并生成文件标识，再向服务端查询哪些分片已经存在，然后按并发策略上传缺失分片，最后通知服务端合并。要把这条链路讲完整，通常还要补上秒传、续传、取消和失败重试，否则你的方案只覆盖了最理想的 happy path，没有覆盖真实用户最容易遇到的问题。
36. 如何决定切片大小和切片数量？
答：我会综合文件大小、用户网络环境、重试成本、服务端限制和浏览器并发能力来决定。切片既不是越小越好，也不是越大越好，本质上是在请求调度开销和失败重传代价之间找平衡，所以回答时最重要的是体现这个 trade-off，而不是硬背一个固定数字。
37. 切片太大和切片太小分别有什么问题？
答：切片太大时，一片失败就意味着要重传很多数据，弱网下代价很高；切片太小时，请求数会暴增，前端调度成本和服务端管理、合并成本都会显著上升。所以我不会死给一个固定值，而是会强调参数要根据业务场景和网络条件动态调整。
38. 如何设计并发上传数量？
答：我会设置可配置的并发上限，并结合用户网络、文件大小和服务端负载做调优。并发控制本质上是在吞吐和稳定性之间找平衡，不会简单追求把浏览器连接数拉满，因为并发过高可能让失败率、重试率和服务端压力一起上升。
39. 如何实现断点续传？
答：关键是让文件拥有稳定指纹，重新上传前先向服务端确认哪些分片已经存在，然后只补传缺失部分。也就是说，断点续传不是靠“前端记住了上次传到 67%”，而是靠前后端共同确认“这是不是同一个文件、哪些片已经是可信状态”，这是它真正可靠的地方。
40. 如何在前端计算文件指纹或哈希？
答：通常会分片读取文件并做增量哈希，避免一次性把整个大文件全部读入内存。这样做既能降低内存压力，也更适合大文件处理；如果文件非常大，我会优先把计算放到 Worker 里做，因为哈希本身就是典型的 CPU 任务，很容易在前处理阶段就把主线程拖卡。
41. 秒传的核心思路是什么？
答：秒传本质上是内容去重。前端先上报文件指纹，服务端确认相同内容文件已经存在后，直接返回成功或复用已有资源。真正起决定作用的是服务端的权威判断，前端只是负责识别和协商，所以回答时要强调“前端不是偷偷跳过上传”，而是在和服务端协作。
42. 上传过程中如何做失败重试和进度展示？
答：我会把每个分片状态维护成待传、上传中、成功、失败，再聚合出整体进度。失败时支持有限次重试，并尽量把失败分片和失败原因暴露出来，让用户知道当前卡在哪，而不是只能盯着一个进度条猜到底发生了什么。大文件上传体验的关键，不只是快，还在于过程透明、失败可恢复。
43. 为什么大文件场景里经常会配合 `Worker`？
答：因为哈希计算、预解析和内容切分都偏 CPU 密集，放在主线程里很容易把输入、滚动和点击一起拖卡。AI 场景里用户通常一边上传一边还要继续配置、提问或查看结果，所以我会优先保护主线程交互体验，把这些前处理任务尽量移出主线程。

## 六、Worker 与前端多线程

44. 什么场景下你会在 AI 项目中使用 Worker？
答：典型场景包括大文件哈希、文档切片、文本解析、OCR 结果预处理、向量前处理和大体量 JSON 转换。我的判断标准很简单，只要任务明显偏 CPU、持续时间不短，而且会影响用户输入、滚动、点击这些核心交互，我就会优先评估是否应该把它下沉到 Worker。
45. 主线程和 Worker 线程是如何通信的？
答：一般通过 `postMessage` 和消息事件通信，双方按约定协议传递结构化数据。工程上还要补上可转移对象、错误事件、超时控制和必要的资源释放，因为这些细节决定这套通信是“demo 能跑”还是“线上能长期稳定跑”的工程级方案。
46. 如果连续给 Worker 发送两条消息，如何在主线程拿到对应结果？
答：我会为每个任务生成唯一 `requestId`，Worker 返回时原样带回，主线程再按 id 做结果路由。这个设计和请求层处理并发异步响应的思路本质上是一样的，核心都是把“发起时的上下文”牢牢绑定到“结果回来时的归属”上，避免多任务并发时串结果。
47. Worker 里会有多条消息并发返回吗？前端该如何设计消息协议？
答：完全可能，所以协议里至少要有 `requestId`、`type`、`status`、`data` 和 `error` 这些字段。没有协议约束时，任务一多结果就会错配，等后面再加取消、重试、进度和阶段状态时会变得非常痛苦，所以协议设计越早统一，后续演进越稳。
48. Worker 返回给主线程的数据一般应该是什么结构？
答：我倾向统一成对象结构，包含请求标识、阶段、成功失败标记、业务数据和错误信息。统一结构的好处是主线程可以复用一套处理逻辑，而不是每个任务都写一堆 `if else` 去单独解析返回，这种统一对于后续加日志、重试和观测也会更友好。
49. 为什么大文件切片、文本解析、向量预处理更适合放到 Worker？
答：因为这些任务大多偏 CPU 密集，而主线程同时还要负责输入响应、布局和绘制。把重计算移出去，本质上是在保护用户操作链路，避免 AI 的前处理阶段先把整个页面交互拖慢，这一点在用户需要边上传边继续操作的场景里尤其重要。
50. Worker 和主线程如何做错误处理与资源释放？
答：我会统一监听错误事件和超时状态，任务结束后根据场景决定是复用实例还是及时 `terminate`，避免悬挂线程、无用监听和内存泄漏。工程上除了“把任务跑起来”，生命周期管理同样重要，因为很多线上问题并不是算法错了，而是线程和资源没有被正确收回。

## 七、线上稳定性与发布

51. 如果 AI 写出的结果有问题，但功能已经在线上，你会怎么处理？
答：我会先做风险控制，比如灰度收紧、能力降级、入口下线、提示兜底或切回稳定链路，然后再定位问题是在 Prompt、模型、工具调用还是前端解析展示上。线上处理的第一原则从来不是一次把原因查透，而是先止损，因为 AI 问题很多时候不是全挂，而是结果开始不可信，继续放大只会让影响范围越来越大。
52. AI 产品上线后常见的线上风险有哪些？
答：常见风险包括超时、中断、内容幻觉、格式不稳定、上下文串线、工具调用失败、成本波动和日志不可追踪。AI 产品很多问题不是“服务直接挂了”，而是“结果还能出来，但已经慢慢不可控了”，所以稳定性治理既要覆盖底层链路，也要覆盖效果层和用户感知层。
53. 内容不稳定、幻觉、格式不一致时，前端和服务端分别能做什么？
答：前端更适合做展示层容错、格式降级、异常提示、样本采集和用户操作兜底，服务端更适合治理 Prompt、模型选择、输出协议和工具编排。我的表达会很明确：前端可以缓冲用户感知、降低直接伤害，但不是根治方，真正的根因治理仍然主要在服务端和模型策略层。
54. 是否做过灰度发布？为什么 AI 相关功能更适合配合灰度机制？
答：即使没有完整实战，我也会按设计思路回答，因为 AI 功能波动大、风险隐蔽，而且效果很依赖真实用户场景，直接全量上线的试错成本通常太高。灰度可以让我们先在小流量验证效果和稳定性，再逐步放量，这是 AI 产品里非常重要的一种风险控制手段。
55. 前端可以怎样接入灰度能力？
答：前端通常会根据服务端下发的开关、白名单、实验分桶或环境配置，决定功能是否展示、走哪条接口链路以及如何打埋点。关键不是“能不能切”这么简单，而是能不能快速切、快速关，并且让观测数据可以按灰度维度清楚分层，否则一旦出问题，很难判断到底是新链路坏了还是其他因素干扰。
56. 如果你没做过灰度，你会如何设计一个简化版灰度方案？
答：我会先做一个最小可用版本，比如白名单、按比例放量和环境开关三种基础策略，再配合监控埋点和一键回退。这个问题我不会空谈复杂平台能力，而是更愿意把最小能落地的方案说清楚，因为真实工程里很多能力都是先从可控的小版本做起，再逐步长出来的。
57. AI 功能上线后你会重点关注哪些监控指标？
答：我会重点看请求成功率、首字时间、总耗时、取消率、重试率、解析失败率、工具调用成功率和用户反馈。因为这些指标既能反映底层链路稳不稳，也能反映用户是否真的愿意继续用这个能力，而 AI 产品的“可用”很多时候就体现在这些细节指标上。

## 八、AI 产品效果与评估

58. 一个 AI 功能“效果不好”时，你如何判断是前端问题、Prompt 问题、模型问题还是后端链路问题？
答：我会按链路来拆，而不是一上来就说“肯定是模型不行”。先看请求有没有发出去、返回是否完整、前端有没有展示和解析错误；如果底层链路没问题，再去看 Prompt、模型输出、工具结果和编排逻辑。复杂 AI 问题如果不先做链路拆分，很容易变成每一方都觉得不是自己的锅，最后谁也定位不出来。
59. 用户反馈“回答慢”“回答不准”“格式错乱”时，你会如何拆解问题？
答：回答慢我会先拆成首字慢还是总耗时长，再判断卡在前端、网络、编排还是模型；回答不准更偏 Prompt、上下文和模型选择；格式错乱则重点查输出协议、解析器和流式拼接逻辑。也就是说，我会先把用户感知到的症状映射到系统阶段，再决定排查顺序，而不是把所有问题都笼统归因给“AI 本来就不稳定”。
60. 你会为 AI 产品设计哪些核心指标？
答：我会设计成功率、首字耗时、总耗时、取消率、重试率、解析失败率、工具调用成功率和用户满意度这几类指标。前几类更偏稳定性和体验，后几类更偏效果和产品价值；对 AI 产品来说，只看请求成功率远远不够，因为“接口 200 但用户依然觉得难用”是非常常见的情况。
61. 为什么 AI 产品前端也要关心成功率、耗时、取消率、重试率这些指标？
答：因为用户最先感知到的就是页面有没有及时响应、结果有没有中断、需不需要反复重试，而这些问题很多都是前端最早暴露出来的。如果前端只关心“接口是不是 200”，就很难把 AI 产品真正做成一个稳定可用的产品，因为用户关心的是体验链路，不是接口状态码。
62. 如何把一次 AI 项目优化讲得有业务价值，而不仅是技术价值？
答：我会把优化结果和用户体验、转化、留存、人工审核成本或操作效率联系起来，而不是只说“我加了缓存”或“我重构了组件”。大厂更在意技术动作最终带来了什么业务收益，所以我会尽量把技术改造翻译成产品侧也能理解的结果，比如等待变短后用户是否更愿意继续提问、失败率下降后人工介入成本是否减少。

## 九、补充高频题

63. 为什么 AI 项目很适合用 `Node.js` 做中间层？
答：因为它很适合承接前端请求、聚合多个后端能力、做流式转发、统一会话和权限控制，也方便接入埋点、灰度和观测能力。它的核心价值不只是“多了一层服务”，而是让所有面向 UI 的复杂度有了一个统一收口层，这样前端页面不用直接面对后端各种不同风格的协议和错误语义。
64. 流式渲染的 UI 状态机应该怎么设计？
答：我通常会拆成待发送、请求中、首段到达、持续生成、完成、失败、取消和中断重试这些状态，并确保每个状态都有明确文案和操作入口。状态机设计得好的意义，不只是让代码分支更清楚，而是让用户始终知道系统当前处于什么阶段，AI 场景下这种可解释性会直接影响用户对系统的信任感。
65. 多轮对话上下文应该由谁管理？前端和服务端如何分工？
答：我认为前端主要负责当前会话展示、交互态和本地临时状态，服务端负责更权威的上下文持久化、裁剪策略和安全控制。前端当然可以做缓存和预展示，但不能把自己当成唯一真相源，否则一旦涉及多端同步、长会话裁剪或权限隔离，就很容易出一致性和安全问题。
66. 如果模型输出不是严格 JSON，前端如何兜底解析？
答：我会先按约定协议做宽松解析，尽量提取出可展示内容；如果解析失败，就回退到纯文本展示，同时把异常样本和上下文一起上报，推动服务端修 Prompt 或协议。成熟回答的重点不在“解析失败就报错”，而在“尽量不让用户看到空白和死链路”，先兜住体验，再推动根因治理。
67. 在 AI 应用里，为什么“工程稳定性”经常比“模型能力”更先暴露问题？
答：因为用户最先感知到的往往不是模型到底聪不聪明，而是它慢不慢、稳不稳、会不会中断、会不会串会话、会不会莫名报错。很多 AI 产品上线初期，决定体验下限的其实不是模型能力上限，而是工程链路有没有把这些基础问题先处理好，所以前端在这类产品里往往承担着比传统页面更重的稳定性责任。

## 十、Skills、MCP、LLM 适配与 Agent 编排深挖

68. 什么是 `Skills`？
答：`Skills` 可以理解成提前为模型或 Agent 封装好的“可复用做事方法包”。它通常不等于某个工具，而更像一份面向特定任务的高质量操作说明书，里面会写清楚任务边界、推荐步骤、注意事项、示例输入输出、该调用哪些工具，以及什么时候不要调用。比如“代码评审 skill”“生成 PR skill”“Figma 转代码 skill”都属于这一类。它解决的问题不是让模型凭空学会新知识，而是把团队已经验证过的做事套路沉淀下来，让模型在面对对应任务时更稳定地按正确方式工作。

69. 理解 `Skills` 需要先知道哪些概念？
答：首先要区分 `Skill`、`Prompt`、`Tool`、`Workflow`。`Prompt` 更像一次性的当前任务说明，`Tool` 是真正能执行外部动作的能力，比如读文件、查数据库、调用接口，`Workflow` 是多步骤执行流程，而 `Skill` 站在它们中间，负责把“某类任务应该怎么做”包装成可复用经验，再在运行时注入给模型。还要理解“上下文窗口”这个背景，因为 skill 本质上也是上下文的一部分，如果写得太散、太长、太泛，模型就可能抓不住重点。

70. `Skills` 的本质是在做什么？为什么它会起作用？
答：本质上它是在做“经验前置”和“行为约束”。普通模型有通用知识，但并不知道你团队到底希望它如何做代码评审、如何写文档、如何处理异常；skill 把这些要求提前结构化，相当于给模型补了一层任务专用操作系统。它之所以会起作用，不是因为模型内部参数被训练改了，而是因为当前推理时多了一份高优先级上下文，模型在生成下一步动作时会参考这些规则、示例和流程，从而更接近你想要的执行风格。

71. `Skills` 的作用过程和底层原理是什么？
答：底层上它仍然是基于大模型的“下一个 token 预测”，只是输入里多了一份更强的任务先验。运行时系统会先判断当前请求适不适合某个 skill，如果适合，就把 skill 内容和当前用户任务一起提供给模型；模型读取后，会更倾向沿着 skill 规定的步骤组织思考、选择工具和生成结果。所以 `Skill` 的本质是运行时提示增强，而不是模型训练；它最像“高质量可复用 Prompt 模块 + 工程触发机制 + 约束模板”。

72. 使用 `Skills` 时有哪些扩展知识和注意点？
答：第一，`Skill` 不是越多越好，太多会导致触发混乱和上下文污染；第二，`Skill` 不能替代权限控制，模型即使被教会某种流程，也不代表它就应该有权限执行危险动作；第三，`Skill` 更适合沉淀稳定方法论，不适合塞大量高频变化的业务数据；第四，`Skill` 最好写成“触发条件明确、步骤清晰、边界清楚”的形式，否则模型学到的只会是模糊口号，而不是可执行流程。再往外扩展，很多团队会把 `Skill` 和检索、工具路由、评测体系结合起来，形成一个可持续迭代的 Agent 能力层。

73. 什么是 `MCP`？如果要更完整地解释，应该怎么讲？
答：`MCP` 可以理解成“模型和外部能力之间的统一接入协议”。它不是某一个具体工具，也不是单纯的函数调用接口，而是一套标准化约定，让不同的工具、资源和上下文可以用一致方式暴露给模型或 Agent。更完整地说，`MCP` 解决的是“模型怎么发现外部能力、怎么知道参数格式、怎么安全调用、怎么拿回结果”这类系统协作问题，它让工具接入从“每个模型平台各写一套私有适配”变成“按统一协议接入一次，多端复用”。

74. 理解 `MCP` 需要哪些核心名词？
答：最关键的是 `client`、`server`、`tools`、`resources`、`prompts`、`schema` 和 `transport`。`client` 是发起调用的一侧，通常是宿主应用、IDE 或 Agent Runtime；`server` 是暴露能力的一侧；`tools` 是可执行动作，比如搜索、读设计稿、查接口；`resources` 是可读取的数据资源；`prompts` 是可复用提示模板；`schema` 描述参数和返回结构；`transport` 则是双方通信方式。理解这些词以后，才会知道 `MCP` 不是在解决“模型会不会思考”，而是在解决“外部世界如何被标准化接入模型工作流”。

75. `MCP` 本质上在做什么？它为什么能统一工具接入？
答：本质上它在做“能力标准化描述 + 统一调用协议 + 运行时解耦”。以前应用接一个工具，往往要针对不同模型平台分别写一套适配逻辑；有了 `MCP` 后，工具提供方只要按协议声明自己有哪些能力、参数怎么传、结果怎么回，宿主系统就能按统一方式发现和调用。它之所以能统一工具接入，是因为大家不再围绕某个厂商私有格式协作，而是围绕公开协议协作。换句话说，`MCP` 统一的不是模型本身，而是模型外部能力的接入面。

76. `MCP` 的完整作用过程和底层原理是什么？
答：完整过程通常是：宿主应用先连接 `MCP server`，读取它暴露的工具、资源和描述信息；当用户发起任务后，模型或 Agent 根据当前目标判断是否需要某个外部能力；一旦决定调用，就按照 schema 生成参数，交给宿主去执行协议调用；工具返回结果后，再把结果回填给模型继续推理，最后生成用户可见答案。底层原理并不神秘，核心就是“结构化能力声明 + 结构化请求响应 + 模型基于上下文做调用决策”，也就是说真正执行工具的还是应用或宿主环境，模型负责理解任务和决定何时使用这些能力。

77. 使用 `MCP` 有哪些扩展知识和注意点？
答：第一，要区分“模型会调用工具”和“工具真的能安全执行”是两回事，权限、鉴权、审计、超时、幂等和隔离仍然需要系统保证；第二，工具 schema 设计得越清楚，模型调用越稳定，尤其是参数约束、错误结构和示例非常重要；第三，`MCP` 不天然解决业务语义对齐问题，它只解决接入标准，不解决工具本身设计是否合理；第四，在前端或 IDE 场景里，`MCP` 往往还意味着要把工具列表、调用中间态、失败原因、回放能力和调试信息可视化，否则用户只会看到“模型突然做了一件事”，却不知道过程发生了什么。

78. 什么是 `LLM 适配`？
答：`LLM` 适配可以理解成“把不同大模型厂商、不同模型能力、不同协议格式，统一收口成应用内部一致接口”的这一层。因为不同模型在消息格式、系统提示词写法、流式协议、工具调用字段、多模态输入、上下文限制、采样参数和返回结构上经常都不一样，如果业务代码直接对接每一个模型，后面会非常难维护。`LLM` 适配层的价值，就是把这些差异尽量屏蔽掉，让上层业务只关心“我要生成什么、要不要流式、要不要工具、结果怎么回”。

79. 理解 `LLM 适配` 需要知道哪些背景概念？
答：至少要知道 `context window`、`token`、消息角色、采样参数、流式事件、工具调用格式、多模态输入和模型能力矩阵这些概念。`context window` 决定一次能塞多少上下文，`token` 影响成本和长度限制，消息角色决定 `system`、`user`、`assistant`、`tool` 等信息如何组织，而不同模型对 JSON 输出、结构化输出、图像输入、推理链暴露方式和工具调用格式的支持也不一样。没有这些背景，就很难理解为什么同样一套业务逻辑，换一个模型后适配层可能就要改很多。

80. `LLM 适配` 的本质是在做什么？底层原理是什么？
答：本质上它是在做“协议翻译、能力抽象、策略收口和兼容性管理”。上层业务发出一个统一请求，比如“流式回答、支持工具、温度为多少、要求结构化输出”，适配层会根据目标模型的能力，把这份统一请求翻译成对应厂商自己的 API 格式；模型返回后，再把不同风格的 chunk、tool call、finish reason、usage 和错误结构归一化成内部标准。底层原理本质上还是输入输出转换，只不过转换的不只是字段名，还包括能力差异、参数容错、降级策略和回退逻辑。

81. 一个典型的 `LLM` 适配层执行过程是什么？
答：通常是这样：先根据任务类型选择模型；再把内部消息结构转换成目标模型要求的请求格式；如果涉及工具调用、结构化输出或多模态输入，就额外做对应字段映射；请求发出去后，适配层持续接收流式返回并转换成内部统一事件；如果模型中途要求调用工具，就把调用意图交给运行时执行，再把结果继续喂回模型；最后输出标准化后的结果给前端或业务层。这个过程中，适配层通常还会顺手做超时控制、重试、日志、成本统计、熔断和 fallback。

82. 什么是 `Agent 编排`？
答：`Agent` 编排可以理解成“让模型、工具、记忆、规则、子任务和执行流程协同工作的调度层”。如果说单次对话只是“问一句，答一句”，那 Agent 编排处理的是更复杂的任务，比如先拆解目标、再选工具、再调用外部系统、再检查结果、必要时重试或改计划，最后才给出答案。它关注的不是单次生成，而是整段任务执行过程如何被规划、驱动、观察和控制。

83. 理解 `Agent 编排` 需要哪些基础概念？
答：至少要理解 `planner`、`executor`、`memory`、`state`、`tool routing`、`reflection`、`retry`、`human in the loop` 和 `workflow engine`。`planner` 负责制定步骤，`executor` 负责真正执行动作，`memory` 负责保存任务上下文和中间结果，`state` 描述当前执行到哪一步，`tool routing` 决定该调用哪个工具，`reflection` 是对结果再检查一遍，`human in the loop` 是遇到高风险动作时要求人工确认。很多所谓“Agent 很聪明”，本质上不是模型单体更强，而是这些运行时机制做得更完整。

84. `Agent 编排` 和普通 `Function Calling`、工作流有什么区别？
答：普通 `Function Calling` 更像“模型在某一轮里决定调用一个工具”；固定工作流更像“流程已经被人提前写死”；而 `Agent` 编排处在两者之间或之上，它允许系统在运行时根据任务状态动态决定下一步做什么，既可能走固定流程，也可能中途改计划、调用多个工具、拆出子任务甚至切换模型。换句话说，`Function Calling` 解决的是一次调用问题，工作流解决的是确定性流程问题，而 `Agent 编排` 解决的是复杂任务如何在不确定环境里持续推进的问题。

85. `Agent 编排` 的本质、发挥作用的原理和底层过程是什么？
答：本质上它是在做“任务状态机 + 决策循环 + 外部执行系统”。常见底层过程是：用户给出目标后，系统先把目标转成当前状态；模型根据状态生成计划或下一步动作；运行时执行工具调用或子任务；执行结果再回写到状态里；然后模型基于新状态继续判断下一步，直到满足终止条件。它之所以能工作，是因为模型擅长在不完整信息下做语义决策，而运行时系统擅长做确定性执行、状态保存和异常控制，两者组合后就能把复杂任务逐步推进。

86. `Skills`、`MCP`、`LLM 适配`、`Agent 编排` 四者之间是什么关系？
答：可以把它们理解成四个不同层次。`Skills` 负责告诉模型“这类事应该怎么做”，`MCP` 负责把外部能力标准化接进来，`LLM` 适配负责把不同模型收口成统一接口，而 `Agent` 编排则负责把模型、skills、tools、memory 和流程真正组织起来。一个比较直观的理解是：用户给出任务后，编排层先选模型和 skill，再通过适配层调用具体模型；模型在推理时发现需要外部能力，于是通过 `MCP` 或其他工具接口去执行；结果回来后，再由编排层继续推进直到任务完成。

87. 如果从面试表达角度总结，这四个概念最容易说错的点是什么？
答：最容易说错的有四个。第一，把 `Skill` 说成“模型训练出来的新能力”，其实它更多是运行时注入的方法包；第二，把 `MCP` 说成“工具本身”，其实它更像工具接入标准；第三，把 `LLM` 适配说成简单的接口转发，实际上它还承担能力差异抹平、错误归一化和降级策略；第四，把 `Agent 编排` 说成“多调几个接口”，其实它真正难的是状态管理、计划调整、权限控制、失败恢复和可观测性。面试里只要把这几个边界讲清楚，整体认知就会显得很扎实，也更像是理解了系统，而不是只记住几个热词。
