# 场景题专题

## 使用方式

场景题真正考的不是你能不能立刻报一个名词，而是你有没有一线处理问题的方法。我在回答时会尽量固定成一个顺序：先复述现象和边界，再按优先级拆原因，接着讲排查动作和判断依据，然后给止血方案，最后补长期治理。

---

## 一、为什么场景题重要

场景题本质上在看四件事：你有没有真实项目经验，你能不能把问题快速收敛，你会不会做取舍，以及你有没有线上稳定性意识。很多题没有唯一答案，但一定有更专业的分析路径。

---

## 二、性能优化场景

1. 页面白屏时间过长，你会如何分析？
答：我会先确认白屏是稳定复现还是偶发，是首开、刷新、切路由还是特定浏览器环境下出现，然后按用户真实访问链路一层层往后拆：先看 DNS、建连、HTML 返回，再看静态资源、首屏 JS、关键接口和首次渲染。所谓“白屏”只是最终症状，根因可能在网络、资源、脚本执行、接口阻塞甚至版本不一致，所以我不会一上来只盯着某个组件代码，而是先判断到底是哪一层先卡住；如果线上影响已经明显，我会先止血，比如降级强依赖、回退高风险版本或给出重试提示，之后再补分阶段监控，把白屏准确定位到哪一段链路。
2. 输入 URL 到页面可交互之前，哪些环节最容易导致白屏？
答：我会重点看 DNS 解析、TCP/TLS 建连、HTML 返回、静态资源下载、首屏脚本执行、关键接口返回和首次渲染这几个阶段，因为用户真正看到的白屏往往是这条链路里某一段没有按时完成。SPA 项目更容易卡在首屏脚本执行、路由初始化和接口依赖，SSR 项目还要额外关注服务端渲染耗时和 hydration 是否阻塞，所以我不会机械背流程，而是会结合项目形态去判断哪一段最值得优先排查。
3. 首屏加载慢时，你会优先排查网络、渲染、脚本还是接口？
答：我不会脱离证据直接拍脑袋判断，因为“感觉是接口慢”这种说法在排查里价值很低。我会先看监控、瀑布图和 Performance，判断时间到底堆在资源下载、主线程长任务还是首屏接口上；资源慢就优先查网络和体积，主线程慢就优先查脚本和渲染，接口慢就看是不是串行依赖、后端慢查询或不必要的首屏强依赖。排查优先级的底层逻辑很简单，就是先抓占比最大、能最快改善体验的那一段。
4. 除了 Lighthouse 和 Performance，你还会怎么评估优化效果？
答：我会同时看真实用户监控、关键埋点、错误率、核心交互耗时和用户反馈，因为实验室数据只能说明“在理想环境下可能变快了”，不代表真实用户真的感知更好了。尤其线上要重点关注 P95、P99 这类长尾指标，因为很多性能问题并不是所有人都慢，而是某些弱网、低端机或特定浏览器特别慢，所以如果只看平均值，很容易把真正影响体验的人群掩盖掉。
5. 聊天组件不断 re-render，应该如何定位和优化？
答：我会先用 React DevTools 的 profiler 看究竟是谁在反复更新，再判断是父组件带着整棵子树一起刷新、Context 扩散过大，还是 props 引用不稳定导致 memo 失效。很多聊天页卡顿不是因为消息多本身，而是输入态、滚动态、会话态这些高频状态和低频展示状态混在一起，所以我会重点看状态边界是否合理；如果线上已经明显卡，我会先把高频状态从大组件里拆出来止血，后面再重构列表结构和渲染粒度。
6. 长列表滚动卡顿时，你会如何拆解原因？
答：我会先把原因拆成几类：DOM 数量过多、单项渲染过重、滚动事件里夹了重逻辑、频繁读写布局导致回流，或者图片和富内容绘制成本太高。判断时我会直接看 Performance 里 JS、Layout、Paint 的时间分布，而不是只凭经验猜；如果问题明显随数据量增长而恶化，我会优先考虑虚拟列表止血，但如果单项本身就很重或者滚动事件里做了太多同步计算，最终还是要回到组件结构和逻辑路径上治理。
7. 图表很多的 Dashboard 页面卡顿，你会如何优化？
答：我会先区分卡顿发生在首次进入还是交互联动阶段，因为这两种问题的治理方向完全不同。首次进入更像加载问题，我会优先做按需加载、首屏裁剪和非关键图表延后渲染；联动阶段更像更新问题，我会重点查重复请求、重复计算、图表实例反复销毁重建和整页级刷新。如果业务已经受影响，短期可以先减少首屏图表数和自动刷新频率，长期再做分区更新、数据缓存和重计算下沉。
8. 某个复杂表单输入明显卡顿，你会如何分析？
答：我会先怀疑每次输入是不是触发了整表单重渲染，或者同步校验、格式化、联动逻辑都挤在输入主链路里，导致用户一敲字页面就要做很多本不该立刻做的事。排查时我会用 profiler 看耗时到底是在渲染、校验还是副作用请求上；短期可以先通过延迟校验、防抖和状态拆分止血，长期则要把字段联动逻辑从输入事件里解耦出来，避免每次输入都牵动整个表单。
9. 用户说“页面偶尔卡死”，你会如何结合监控和本地排查？
答：我会先从监控里收集足够多的上下文，比如设备、浏览器版本、页面路径、长任务分布、错误日志和用户操作轨迹，先判断这是全量问题还是某一类环境问题。因为“偶尔卡死”最难的地方不是不会修，而是复现边界不稳定，所以我不会一上来盲查所有代码，而是先把高频路径和共性环境收窄，再重点排查死循环、异常重试、巨大对象渲染和第三方库异常。
10. 你如何证明一次性能优化是有效的，而不是主观感觉？
答：我会拿优化前后的同口径指标做对比，比如首屏时间、长任务数、交互时延、FPS、卡顿率和关键操作完成率，并尽量保持环境和样本一致。如果是线上问题，我还会看灰度前后的真实用户数据，而不是只拿我自己电脑上的一次录屏当结论。性能优化之所以必须数据化，是因为人的主观体感很容易被页面结构变化或心理预期干扰，只有指标才更接近可复用的工程证据。

## 三、渲染与浏览器场景

11. `transform` 会导致重排吗？为什么很多动画推荐用 `transform`？
答：大多数情况下，`transform` 不会直接触发布局重排，因为它通常不改变元素在文档流里的几何占位，而是更多作用在后面的合成阶段，所以特别适合位移、缩放这类高频动画。大家推荐它，不是因为它神奇，而是因为它通常比修改布局属性更便宜，更不容易拖慢整页；不过我也不会把这条规则说成绝对真理，真掉帧时还是要回到 Performance 看成本到底落在脚本、布局、绘制还是合成上。
12. 如果一个元素位置频繁变化，为什么直接改 `top/left` 可能不如 `transform`？
答：因为 `top/left` 改的是布局结果，浏览器往往需要重新计算这个元素甚至相关节点的位置，而 `transform` 更像是在已有布局结果之上做视觉变换，所以通常代价更低、动画也更平滑。这里真正要理解的是浏览器渲染流水线，不同属性影响的是不同阶段；当然如果元素本来就脱离文档流、影响范围很小，实际成本还要结合具体布局来判断，不能把口诀当铁律。
13. 页面出现明显抖动和重绘，你会从哪些角度排查？
答：我会先排查有没有频繁读写布局信息，比如刚读完 `offsetHeight` 又立刻改样式，导致浏览器被迫提前计算布局；再看动画属性是否选错、样式切换是否过于频繁、是否有大面积重绘，以及阴影、滤镜、图片这类高绘制成本元素。页面“抖动”往往不是一个抽象感受，而是布局、绘制、合成阶段有一段成本异常高，所以我会直接看 Performance 中的 Layout、Paint 和 Composite 数据来判断。
14. 为什么某些布局改动会导致整页重排？
答：因为这些改动动到了布局树里影响范围很大的节点，比如父容器尺寸变化、文档流中的关键块级元素变化，或者某个公共容器的显示方式改变，后续很多节点的位置都要跟着重新计算。所谓“整页重排”，本质上是级联影响范围太大，所以真正重要的不是死记哪个属性会重排，而是理解你改动的那个节点会牵动多大一片布局。
15. 如果动画掉帧，你如何判断瓶颈在脚本、布局还是绘制？
答：我会直接打开 Performance，把掉帧那几帧拆开看主线程时间到底花在哪。JS 占比高说明脚本过重，Layout 高说明布局被频繁触发，Paint 或 Composite 高则说明绘制或图层合成成本大。先做这一步归类非常重要，因为脚本、布局和绘制的优化手段几乎完全不同，如果问题没分清，后面的优化动作很容易南辕北辙。

## 四、请求与网络场景

16. 接口偶发超时，你会怎么判断是前端问题还是服务端问题？
答：我会先确认请求是否真的发出去了，再看 DNS、建连、TTFB、下载耗时和服务端日志，判断超时到底发生在前端发起前、网络传输中还是服务端执行阶段。所谓“接口超时”只是结果，不代表一定是后端慢，也可能是前端重复发请求、超时配置太短、代理层中断或网络抖动；如果问题已影响业务，我会先补超时提示、重试和降级兜底，让用户至少知道发生了什么，再和服务端一起拆完整链路。
17. 页面刷新后登录态丢失，你会先检查哪些地方？
答：我会先检查凭证到底存在哪里、刷新后有没有被正确读取，以及登录态初始化是不是晚于页面鉴权判断，比如 token 是否真的持久化、cookie 的 `domain`、`path`、`sameSite`、`secure` 是否正确，跨域请求是否带上凭证。登录态丢失很多时候不是“没登录”，而是存储、读取、初始化或跨域配置某一段错了；如果线上已经出问题，我会先回退相关改动或加兜底重定向，长期再统一登录态恢复流程。
18. 某接口反复返回 `304`，你会如何解释？
答：这通常说明协商缓存生效了，也就是浏览器先带着上一次资源的缓存标识去问服务端“这个资源有没有变”，服务端判断没变后返回 `304`，告诉浏览器继续使用本地缓存。`304` 本身不是报错，反而常常代表缓存策略正在正常工作；只有当用户因为缓存策略拿到过旧资源，或者它和发版策略冲突时，它才会变成问题。
19. 用户反馈“只有公司网络打不开页面”，你会优先排查什么？
答：我会优先怀疑 DNS、公司代理、防火墙、证书链或某些静态资源域名被内部网络策略拦截，因为这类问题最大的特征就是“外网正常、特定企业网络异常”。第一步我会拿控制台错误、网络面板、失败域名和证书信息判断到底是连不上、握手失败还是资源被拦截；如果要先止血，可以临时切备用域名或给出明确说明，长期则需要和网络侧一起治理域名和证书策略。
20. 页面部署后部分用户一直拿到旧资源，你会怎么处理？
答：我会先确认入口 HTML 有没有被强缓存、静态资源是否带 hash、CDN 是否已刷新完成，以及页面是否出现新旧资源混用。很多用户拿旧资源不是浏览器“抽风”，而是发版策略没有把入口缓存、资源指纹和版本检测一起设计好；如果线上已经报错，我会先用版本提示和强制刷新止血，必要时暂时保留兼容资源，后续再系统性治理资源更新策略。
21. 流式接口偶发中断，你会如何分析网络层和应用层问题？
答：我会先区分是网络层中断还是应用层自己结束得不对，比如连接是否被代理、超时或弱网打断，服务端是否主动 close，返回流格式是否不符合约定，前端 reader 消费是否异常，或者状态机把“中断”误判成了“完成”。流式链路难的地方在于它不是一次返回一个结果，而是持续收数据，所以任意一段断掉都可能表现成半截内容；短期我会先补中断提示和重试入口，长期再把 requestId、链路日志和结束原因打通。
22. 用户快速切换会话时，旧请求结果污染了新页面，怎么解决？
答：我会同时从源头和落地两层处理这个问题。源头上，切换会话时尽量取消旧请求，减少无意义返回；落地上，结果回来时再校验当前会话 id 或 requestId，确保旧结果即使晚到，也不会写进新页面。这个问题本质上是异步竞态，不是简单“切一下页面”的 UI 问题，所以取消请求和结果校验两层防线都要有。

## 五、登录态与权限场景

23. 登录态一般是如何实现的？你会如何解释完整链路？
答：我会按“用户登录获取凭证、前端托管或存储、后续请求自动携带、服务端校验、过期续签、退出清理”这条完整链路来讲，这样面试官能看到我理解的是系统机制，不是某一个 token API。然后我会顺手解释为什么有的系统选 cookie，有的选 token，它们在存储方式、跨域、CSRF 和前端可控性上有什么差异，同时强调前端负责的是体验和状态同步，真正的鉴权裁决权永远在服务端。
24. 如果用户明明已登录，但页面权限按钮突然消失，你怎么排查？
答：我会先确认权限数据是否真的拉到了，再看前端权限判断是否用了旧缓存、初始化顺序有没有竞态、按钮是不是被某个条件误隐藏。这里要特别注意，按钮消失不一定说明用户真的没权限，也可能只是前端展示判断错了；如果问题已经影响业务，我会先从展示层做临时降级或明确提示，但绝不会放松服务端权限控制，长期则要把权限来源和前端判断逻辑统一起来。
25. 如果 token 过期了，前端应该怎么处理？
答：我会在统一请求层做拦截，能静默刷新就先刷新并重放原请求，刷新失败就清理本地状态并引导用户重新登录。这里真正要注意的是并发场景下的“刷新风暴”，也就是多个请求同时发现 token 过期又同时去刷新的问题，所以通常会加单飞控制，让刷新操作在同一时刻只执行一次。
26. 页面级权限、路由级权限、按钮级权限分别应该放在哪里控制？
答：页面级和路由级权限我会放在路由守卫、布局层或页面入口统一拦截，按钮级权限则下沉到组件层做细粒度展示控制。这样做的好处是大边界有统一入口，小动作有就近控制点，规则更容易治理；不过我一定会强调，前端权限更偏体验控制，真正的数据权限和操作权限必须由服务端兜底，否则安全边界是假的。
27. 如果权限逻辑散落在页面里，后续会有什么问题？如何治理？
答：最大的问题是规则不一致、重复判断多、边界容易漏，而且随着时间推移，团队会逐渐失去对权限体系的全局认知，最后谁都不敢动。治理上我会先统一权限数据源和判断方法，再逐步把散落在页面里的逻辑收口到公共方法、配置或权限组件里，让“判断规则”和“页面展示”分离，这样后续权限变更才不会变成全项目地毯式改动。

## 六、工程化与发布场景

28. 线上网站用户正在使用时重新部署前端，出现报错怎么办？
答：我会先止血，优先保证用户还能继续完成核心操作，比如提示刷新、关闭高风险入口、必要时快速回滚，而不是先在群里讨论技术细节。前端重发版后用户报错，常见根因往往是入口缓存、资源 hash、按需 chunk 或接口契约出现了版本不一致，所以第一优先级是控制影响面，第二优先级才是查透具体是哪一层不一致。
29. 为什么前端重新发版后，运行中的旧页面可能报错？
答：因为用户浏览器里当前运行的是旧代码，但发版后新请求拿到的入口、异步 chunk、配置数据甚至接口结构可能已经切到新版本了，一旦两边不兼容，就会出现脚本加载失败、运行时异常或页面逻辑错乱。这个问题的本质不是某一行代码写错，而是前端天然存在“运行中旧页面”和“线上新资源”并存的版本一致性问题。
30. 如何缓解前端资源更新导致的版本不一致问题？
答：我至少会做几件基础治理：静态资源带 hash，入口 HTML 控制缓存策略，客户端做版本检测，发现关键版本变化时主动提示刷新；高风险业务还可以补灰度、双版本资源保留和兼容期。这里要顺手说明 trade-off，也就是治理越完整，工程复杂度越高，所以我会根据业务风险来决定做到什么程度，而不是无脑堆满所有机制。
31. 你做过灰度发布吗？如果没做过，你会怎么设计简化版灰度？
答：如果没完整做过，我不会硬装做过，而是会按最小可用方案回答：先确定灰度维度，比如白名单、按比例分桶或环境开关，再补监控、开关控制和快速回滚能力，最后定义什么情况下继续放量、什么情况下立刻回退。灰度的本质不是平台页面多复杂，而是有没有能力把上线风险控制在一个可接受的半径里。
32. 什么场景适合做灰度发布，什么场景必须做灰度发布？
答：影响面大、依赖链路长、上线后不好快速修复的功能都适合灰度，而像登录、支付、权限、核心交易以及高风险 AI 能力这类，通常更接近必须灰度。原因很简单，这些链路一旦出问题，损失往往不是“某个页面不美观”，而是直接影响用户身份、钱或者核心业务，所以不能把全量用户当测试对象。
33. 如果某次上线后错误率明显上升，你会如何定位和回滚？
答：我会先确认错误率上升和发布时间是否强相关，再按页面、浏览器、用户分层、接口、版本和灰度范围拆开看，判断是全局问题还是局部问题。如果影响面大且短时间收不住，我会优先回滚或关开关止血，再用日志和监控继续追根因。线上场景里，及时止损通常比坚持“先完全定位再处理”更专业，因为业务影响不会等你分析完。

## 七、大文件与上传场景

34. 为什么服务器往往处理不了超大文件一次性上传？
答：因为一次性上传会把一个很长的网络过程压成单个大请求，持续占用带宽、连接和服务端资源，中间只要任意一段失败就得整包重来。弱网环境下这种方式成功率很差，用户体验也非常不稳定，所以大文件方案的本质就是把一个高风险长链路拆成多个更容易恢复、重试和追踪的短链路。
35. 你如何设计一个大文件上传方案？
答：我会从文件稳定标识、切片策略、并发控制、失败重试、断点续传和服务端合并校验这几部分来设计。前端负责切片、计算文件指纹、调度上传、管理进度和失败态，服务端负责记录哪些分片已上传、如何合并以及如何校验完整性；如果文件特别大，我还会把哈希计算放到 Worker，避免上传前的预处理就先把主线程卡住。
36. 切片大小和切片数量如何决定？
答：我会结合文件大小、用户网络情况、服务端限制、浏览器并发能力和失败重传成本来定，因为切片大小本质上是在“单次失败代价”和“请求调度开销”之间做平衡。切片太大时一片失败就要重传很多数据，切片太小时请求数量暴涨、调度和服务端合并成本都会上升，所以这个参数更像是要按场景调优，而不是背一个固定数字。
37. 为什么切片过多也会有问题？
答：因为切片一多，请求数会显著上升，前端要管理更多状态，服务端要记录和合并更多分片，最终整体耗时甚至可能更差。切片策略是一个典型 trade-off，不是越细越稳定，而是在失败成本、调度成本和总体吞吐之间找平衡，所以回答时要体现这种取舍意识。
38. 并发上传数量应该怎么控制？
答：我会设置一个可配置的并发上限，并结合弱网情况、文件大小和服务端承压做动态调整。并发太低，吞吐上不去；并发太高，浏览器、网络和服务端都会更容易抖，所以我更关注的是稳定吞吐，而不是把瞬时速度拉到看起来很漂亮的峰值。
39. 上传中断后如何做断点续传？
答：核心是先给文件一个稳定标识，重新上传前向服务端确认哪些分片已经存在，然后只补传剩下的部分。这里真正要注意的是文件标识是否稳定、本地文件内容有没有变化、服务端记录是否可信，否则表面上看像断点续传，实际上可能把不属于同一个文件的分片拼在一起，变成“假续传”。
40. 如何实现秒传？
答：通常是前端先计算文件指纹，再把这个指纹发给服务端，由服务端判断相同内容是否已经存在；如果服务端确认可以复用，就直接返回成功。秒传的本质其实是服务端内容去重能力，前端只是帮忙做识别和协商，不能自己擅自认为“长得一样就不传了”。
41. 上传进度、失败重试、取消上传如何设计？
答：我会把每个分片都管理成待传、上传中、成功、失败、取消这几种状态，再聚合成整体进度，这样用户和系统都知道当前卡在哪。失败重试最好支持单片级别而不是整轮重来，取消则不仅要停 UI，还要真正终止调度和网络请求。对用户来说，上传过程最重要的不是“快”这一个字，而是过程透明、失败可恢复。

## 八、Worker 与重计算场景

42. 什么场景下你会使用 Web Worker？
答：当任务明显偏 CPU 密集，而且会阻塞主线程影响输入、滚动、点击这些用户核心操作时，我会考虑 Worker，比如大文件哈希、图片处理、CSV 解析和复杂文本处理。Worker 的意义不是“多线程更高级”，而是保护主线程响应能力；但它也有通信、调试和兼容成本，所以我不会把它当默认选项，而是会在问题被验证后再引入。
43. 如果主线程需要和 Worker 通信多次，你会如何设计消息协议？
答：我会把通信协议统一成对象结构，至少包含 `type`、`requestId`、`payload`、`status` 和 `error`，这样任务一多也不会互相串。所谓消息协议，讲人话就是“发过去的每条消息和回来的每条结果都知道自己是谁、在做什么、成功还是失败”，否则后面一旦加取消、进度和重试，通信链路会很快失控。
44. 如果同时发送两条消息给 Worker，如何保证主线程拿到对应结果？
答：我会给每条请求分配唯一 `requestId`，Worker 返回结果时原样带回，主线程再根据这个 id 做结果路由。这个思路和请求层处理异步响应其实是同一个原理，本质上都是把“请求发起时的上下文”绑定到“结果回来时的落点”上；进一步我还会补超时、取消和异常透传，确保并发任务不会串结果。
45. Worker 返回的结果应该如何带上上下文信息？
答：除了业务结果本身，我会一起返回 `requestId`、任务类型、业务标识、阶段状态和错误信息，因为主线程真正需要的不是一段裸数据，而是知道“这是谁的结果、现在走到哪一步、如果失败应该怎么处理”。上下文带得足够全，后面加日志、重试和问题定位都会轻松很多。
46. 为什么大文件哈希计算、图片处理、CSV 解析适合放到 Worker？
答：因为这几类任务通常都偏 CPU 密集型，而且处理时间可能不短，如果放在主线程里，用户输入、点击、滚动这些交互都会被一起拖慢。把它们移到 Worker 的本质，是把重计算和界面响应解耦，优先保护用户手上的操作链路，而不是为了秀一个更“高级”的技术点。
47. 如果浏览器不支持 Worker，是否需要降级？怎么降级？
答：要看这项能力是不是核心链路。核心功能我会做降级，比如改成分片调度、降低并发、减少单次处理量，或者明确提示性能会变差；非核心功能则可以适度弱化体验。我的原则一直是先保可用，再谈体验一致，因为真实项目里不是所有用户环境都能完美支持你的最佳方案。

## 九、AI 前端场景

48. 为什么一个 AI 项目里会同时用 `Node.js` 和 `Python`？
答：我会从分层职责来解释，而不是把它说成“公司喜欢混技术栈”。`Node.js` 更适合做 BFF、流式转发、会话层和更贴近前端交互的接口编排，因为它在 I/O 编排和前端协作上很顺手；`Python` 更适合模型调用、数据处理和算法能力，因为相关生态更成熟。这个组合的意义不是为了炫，而是让不同层做自己更擅长的事，边界更清楚、协作也更顺。
49. 为什么 `Node.js` 通常更适合做 BFF、流式转发和会话层？
答：因为这一层的核心工作是 I/O、协议适配、接口聚合和会话编排，更强调和前端链路贴合、开发效率高、迭代快，而不是复杂计算。`Node.js` 在这种位置优势比较明显，但我也会顺手说明，适合不等于唯一正确，最终仍然取决于团队能力、现有基础设施和整个系统的历史包袱。
50. 为什么 `Python` 更适合模型推理和算法处理？
答：因为 `Python` 在模型框架、数据处理、向量库和算法生态上的成熟度更高，很多能力能以更低成本接入和验证。尤其 AI 项目往往要频繁试错、快速调整，这种生态优势会非常直接地体现在研发效率和实验成本上，所以选语言时很多时候选的是生态和团队速度，而不只是语言语法本身。
51. 如果 AI 输出有问题，但功能已经上线，你会怎么处理？
答：我会先止血，把影响范围控制住，比如收紧入口、降低曝光、补明显提示，必要时直接关闭高风险能力，而不是先争论到底是模型还是前端的锅。然后再拆问题是在 Prompt、模型、工具调用还是前端解析展示上，因为 AI 场景很多时候不是完全不可用，而是“还能出结果，但结果已经不可靠”，这也是为什么灰度、观测和前端兜底在 AI 产品里特别重要。
52. AI 功能上线后，你会关注哪些稳定性指标？
答：我会重点看成功率、超时率、首字耗时、总耗时、取消率、重试率、解析失败率和工具调用成功率，因为 AI 场景的问题很多不是接口直接报错，而是慢、中断、解析失败、结果不稳或状态错乱。也就是说，AI 产品的稳定性指标不能只盯着 HTTP 200，而要能覆盖用户真正感知到的全过程。
53. 为什么 AI 产品更适合配合灰度和观测体系？
答：因为 AI 结果天然存在波动，很多风险不是“系统完全挂了”，而是效果悄悄变差、耗时悄悄变长、长尾异常突然增多。这样的风险如果直接全量上线，很难第一时间察觉和控制；灰度和观测能让我们先在小流量验证，再逐步放量，把试错成本和影响面都压在可控范围内。
54. 流式输出 UI 应该如何设计状态机？
答：我至少会拆成待发送、请求中、首段到达、持续生成、工具调用中、已完成、已中断、失败和重试中这些状态，并定义每个状态下用户能做什么、界面显示什么文案。状态机的价值，不只是让代码更好写，而是让用户知道系统当前到底在思考、在生成、在调用工具还是已经失败，否则 AI 流式链路一复杂，用户很快就会失去信任。
55. 如果模型输出不是稳定 JSON，前端要怎么兜底？
答：我会先做宽容解析，尽量从不完整或不标准的结构里提取出可展示内容；如果结构化解析失败，就回退到纯文本展示，同时把异常样本、原始输出和上下文一起上报。这样做的意义是前端先兜住用户体验，不让页面直接空掉，而后续服务端也能根据这些样本去修 Prompt、协议或模型输出约束。

## 十、项目取舍题

56. 为什么项目用了 `Node.js`，又用了 `Python`？
答：我会强调分层分工而不是简单罗列技术栈。`Node.js` 负责接口编排、流式链路和会话层，更贴近前端交互；`Python` 负责模型和算法能力，更适合承接 AI 处理。只有当这种分工带来的开发效率、边界清晰度和协作收益明显大于维护两套技术栈的成本时，这个组合才是合理的。
57. 为什么项目用的是 `JavaScript`，不是 `TypeScript`？
答：我会先承认现实因素，比如历史包袱、团队现状或迭代节奏导致短期没有完全切到 `TypeScript`，而不是硬说 `JavaScript` 更好。然后再补我的工程判断：如果项目会长期维护，我会优先从公共接口、核心模块和高频变更区域补类型，因为类型系统的价值主要体现在协作、重构和减少隐性错误上，这样回答既真实，也能体现长期视角。
58. 为什么选 `Vite`，而不是 `Webpack`？
答：如果项目更看重本地开发体验、启动速度和配置轻量，我会更倾向 `Vite`，因为它在现代前端场景下开发反馈很快；但如果项目历史体系重、已有很多 `Webpack` 定制能力，那继续沿用 `Webpack` 也可能更稳。技术选型题里我更强调场景匹配和迁移成本，而不是简单站队谁更新谁更好。
59. 为什么选 `Next.js`，而不是纯 `React`？
答：如果项目需要 SSR、文件路由、服务端能力或更好的首屏体验，我会更倾向 `Next.js`，因为它把很多框架能力收在一起，能减少一部分基础设施搭建成本；但如果是内部系统、强客户端交互且 SSR 价值不高，纯 `React` 反而更轻。关键不是谁更高级，而是需求和复杂度是否匹配。
60. 为什么这个功能要做成 Worker，而不是直接在主线程里处理？
答：因为这个任务已经明显影响主线程交互，比如输入延迟、滚动卡顿或页面响应变慢，把它移到 Worker 的核心目标是先保护用户体验。与此同时我也会主动说明 trade-off，也就是通信、调试和兼容处理都会更复杂，所以 Worker 不应该因为“听起来高级”就引入，而应该在问题已经被验证后再使用。
61. 为什么某个功能需要 BFF，而不是前端直接调后端？
答：因为 BFF 可以统一做协议适配、接口聚合、权限会话处理和错误收敛，把原本会散落在页面层的“胶水逻辑”集中收口。尤其当前端要面对多个后端系统、返回格式不统一或需要流式转发时，BFF 的价值会很明显；但如果链路本来就很简单，我也不会为了架构而加层，因为那样只会徒增维护成本。

## 十一、项目表达型场景题

62. 实习中你具体做了什么？
答：我会从业务背景、职责边界和代表性工作来讲，而不是按时间线机械报任务。先说明这个项目解决什么业务问题，再讲我主要 owner 哪几个模块，最后挑两三件最能体现能力的事情展开，比如一次性能优化、一次线上问题排查或一次关键链路重构。这样的讲法更容易让面试官判断我的真实贡献，而不是只听到一串模糊的“我也参与了”。
63. 你在前端上最大的提升是什么？
答：我最大的提升不是多会了几个 API，而是开始用工程视角看问题。以前我更关注功能能不能做出来，现在会主动考虑状态边界、性能瓶颈、异常兜底、监控观测和长期维护成本，这意味着我不再只把自己当页面开发者，而是开始对一条链路最终能不能稳定工作负责。
64. 你做过哪些性能优化？
答：我会挑几个有代表性的真实案例来讲，比如首屏加载优化、减少重复渲染、长列表虚拟化，或者把重计算挪到 Worker，并且每个案例都按“现象、定位、方案、结果”去讲，而不是只报技术名词。因为性能题最能区分一个人是背过技巧，还是做过真实分析，只有把证据链和结果讲出来，优化才有可信度。
65. 这段实习里你遇到的最难问题是什么？
答：我会选一个真正影响过用户、排查链路比较长、而且中间存在判断转折的问题来讲，因为这种问题最能体现我在不确定情况下的处理能力。这个题的重点不是把问题包装得多夸张，而是证明我能建立优先级、收集证据、推进修复，并且在问题解决后还能补上长期治理。
66. 你是怎么排查并解决的？
答：我通常按“现象、假设、验证、定位、修复、复盘”这条线来讲，并特别强调为什么先怀疑某个方向、用了什么证据排除其他可能、短期止血做了什么、长期治理又补了什么。这样回答的价值在于既能体现问题解决过程，也能体现我不是只会修一次，而是会思考如何避免复发。
67. 如果再做一遍这个项目，你会怎么优化？
答：我会把几个当时后期才意识到很重要的动作前置，比如更早补类型约束、监控告警、错误兜底、版本治理和公共能力抽象。这个题我想表达的不是“当时做得很差”，而是“现在我已经知道哪些基础建设越早做，后续迭代越稳”，这样能体现成长和复盘能力。

## 十二、回答框架模板

我回答场景题时，默认会按下面这个框架组织：

1. 先复述问题，确认边界。
2. 先给结论，再拆可能原因。
3. 按优先级讲排查动作和判断依据。
4. 说明短期止血方案。
5. 再说明长期治理和预防复发。

例如“页面白屏怎么分析”，我至少会覆盖资源、接口、脚本、渲染和版本一致性这几层，而不是只给一句“先看控制台”。
