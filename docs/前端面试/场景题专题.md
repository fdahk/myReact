# 场景题专题

## 使用方式

场景题真正考的不是你能不能立刻报一个名词，而是你有没有一线处理问题的方法。我在回答时会尽量固定成一个顺序：先复述现象和边界，再按优先级拆原因，接着讲排查动作和判断依据，然后给止血方案，最后补长期治理。

---

## 一、为什么场景题重要

场景题本质上在看四件事：你有没有真实项目经验，你能不能把问题快速收敛，你会不会做取舍，以及你有没有线上稳定性意识。很多题没有唯一答案，但一定有更专业的分析路径。

---

## 二、性能优化场景

1. 页面白屏时间过长，你会如何分析？
答：我会先确认白屏发生在首开、刷新还是特定环境，再按链路拆层排查：先看 DNS 和建连，再看 HTML、静态资源和首屏 JS 是否正常返回，接着看首屏接口、脚本执行和渲染是否阻塞。如果线上已经影响用户，我会先做止血，比如降级首屏依赖、回退高风险版本或提示重试；长期治理上我会补首屏分阶段监控，把问题定位到网络、资源、脚本还是接口层。
2. 输入 URL 到页面可交互之前，哪些环节最容易导致白屏？
答：我会重点看 DNS、TCP/TLS、HTML 返回、静态资源下载、首屏脚本执行、关键接口返回和首次渲染这几个阶段。SPA 项目通常更容易卡在 JS 执行和接口依赖，SSR 项目还要额外关注服务端渲染耗时，所以我不会背流程，而是会结合项目形态判断最可能的瓶颈点。
3. 首屏加载慢时，你会优先排查网络、渲染、脚本还是接口？
答：我不会脱离证据直接拍脑袋选一个，我会先看监控和瀑布图，判断耗时主要堆在哪一层。资源下载时间长，我先查网络和资源体积；资源很快但主线程长任务多，我优先看脚本和渲染；如果页面强依赖首屏接口，我会查接口串行、慢查询和不必要阻塞。优先级的核心是先抓占比最大、收益最高的瓶颈。
4. 除了 Lighthouse 和 Performance，你还会怎么评估优化效果？
答：我会同时看真实用户监控、关键埋点、错误率、核心交互耗时和用户反馈。实验室数据更适合定位问题，线上数据才更能反映真实体验，所以我会特别关注 P95、P99 这类长尾指标，而不是只看平均值。
5. 聊天组件不断 re-render，应该如何定位和优化？
答：我会先用 React DevTools 看哪些组件在重复更新，判断是父组件带刷、Context 传播、还是 props 引用不稳定导致的；然后再看高频状态是不是和低频状态混在一起、列表项是否缺少 memo 边界。如果线上已经明显卡顿，我会先把高频状态从大组件里拆出去做止血；长期上再把状态边界、渲染粒度和消息列表结构重新梳理，避免后面需求继续放大问题。
6. 长列表滚动卡顿时，你会如何拆解原因？
答：我会先拆成五类可能：DOM 数量过多、单项渲染过重、滚动事件里有重逻辑、频繁触发布局计算、图片或富内容绘制成本高。判断依据主要看 Performance 里的 Layout、Paint 和 JS 时间分布。如果问题先发生在大数据量场景，我会优先上虚拟列表止血；如果瓶颈在单项渲染或事件逻辑，长期还是要回到组件结构和计算路径治理。
7. 图表很多的 Dashboard 页面卡顿，你会如何优化？
答：我会先区分是首次进入卡，还是筛选联动时卡。首次进入卡，我会优先做按需加载、首屏裁剪和非关键图表延后渲染；交互阶段卡，我会重点查重复请求、重复计算和图表实例反复销毁重建。如果业务已经受影响，短期我会先减少首屏图表数和刷新频率，长期再做图表分区更新、缓存和 Worker 化评估。
8. 某个复杂表单输入明显卡顿，你会如何分析？
答：我会优先怀疑每次输入触发了整表单重渲染，或者同步校验、格式化、联动逻辑挤占了输入主链路。排查时我会先用 profiler 看耗时最重的是组件渲染、校验逻辑还是副作用请求。止血上可以先做防抖、延迟校验和状态拆分；长期治理则要重构字段联动方式，避免所有逻辑都耦在输入事件里。
9. 用户说“页面偶尔卡死”，你会如何结合监控和本地排查？
答：我会先看监控里的设备、浏览器版本、用户路径、长任务分布和报错情况，判断是全量问题还是特定环境问题。然后优先复现高频路径，重点看死循环、异常重试、巨大对象渲染和第三方库问题。因为“偶尔卡死”最大的难点是不稳定，所以第一步不是盲查代码，而是先把复现边界收窄。
10. 你如何证明一次性能优化是有效的，而不是主观感觉？
答：我会拿优化前后的同口径指标做对比，比如首屏时间、长任务数、交互时延、FPS、卡顿率或关键操作完成率。如果是线上问题，我会同时看灰度前后真实用户数据，避免只用本地机器结论。对我来说，性能优化必须能被数据验证，否则很容易沦为“看起来做了很多”。

## 三、渲染与浏览器场景

11. `transform` 会导致重排吗？为什么很多动画推荐用 `transform`？
答：大多数情况下，`transform` 不会触发布局重排，它主要影响合成阶段，所以更适合高频动画。我更关注的是它能不能减少布局和绘制压力，而不是死记“某个属性会不会重排”。如果动画已经掉帧，我会用 Performance 直接验证成本到底落在脚本、布局还是绘制。
12. 如果一个元素位置频繁变化，为什么直接改 `top/left` 可能不如 `transform`？
答：因为改 `top/left` 更容易触发布局计算，严重时会影响周围节点；`transform` 通常只影响合成，代价更低，更容易保持动画平滑。当然我不会把这个结论说死，如果元素脱离文档流、影响范围有限，实际成本还要结合具体布局看。
13. 页面出现明显抖动和重绘，你会从哪些角度排查？
答：我会先查有没有频繁读写布局导致强制同步布局，再看样式切换是否过于频繁、动画属性是否选错、是否有大面积重绘以及阴影、滤镜、图片等高成本绘制。判断上我会直接看 Performance 里 Layout、Paint 和 Composite 的占比，而不是只凭经验猜测。
14. 为什么某些布局改动会导致整页重排？
答：因为这些改动影响的是布局树里范围较大的节点，比如父容器尺寸变化、流式布局节点变化或文档流核心节点变化，后续很多节点都要重新计算。真正重要的不是背“会重排”，而是知道影响范围来自哪里，哪些改动会带来级联成本。
15. 如果动画掉帧，你如何判断瓶颈在脚本、布局还是绘制？
答：我会直接看 Performance，把一帧拆开看主线程时间花在哪。JS 很高说明脚本过重，Layout 很高说明布局频繁，Paint 或 Composite 很高说明绘制或图层合成成本大。只有先把瓶颈归类清楚，后面的优化手段才不会南辕北辙。

## 四、请求与网络场景

16. 接口偶发超时，你会怎么判断是前端问题还是服务端问题？
答：我会先确认请求有没有正常发出，再看 DNS、建连、TTFB 和服务端处理耗时，判断超时发生在发起前、传输中还是服务端执行中。如果已经影响业务，我会先加超时提示、重试和降级兜底，避免用户无感知等待；长期再和服务端一起拆链路，确认是网络波动、网关限流还是后端慢查询。
17. 页面刷新后登录态丢失，你会先检查哪些地方？
答：我会先查凭证存储位置和初始化读取时机，比如 token 是否真的持久化、cookie 的 domain、path、sameSite、secure 配置是否正确，以及跨域请求是否带上了凭证。如果问题已经在线上出现，我会先回退最近登录态改动或补兜底重定向；长期治理上要统一登录态初始化流程，避免每个页面各自恢复状态。
18. 某接口反复返回 `304`，你会如何解释？
答：这通常说明协商缓存生效了，浏览器带着缓存标识询问服务端，服务端判断资源没变，于是返回 `304` 让浏览器继续用本地缓存。`304` 本身不一定是问题，只有当用户一直拿到旧资源或缓存策略与发布策略冲突时，它才会成为问题。
19. 用户反馈“只有公司网络打不开页面”，你会优先排查什么？
答：我会优先怀疑 DNS、代理、防火墙、证书链或某些资源域名被公司网络策略拦截，因为这是典型的环境相关问题。第一步我会拿控制台报错、网络面板和失败域名，判断是连不上、证书失败还是资源被拦截；止血上可以先切备用域名或提供离线说明，长期再和网络侧一起治理域名和证书策略。
20. 页面部署后部分用户一直拿到旧资源，你会怎么处理？
答：我会先确认入口 HTML 是否被强缓存、静态资源是否带 hash、CDN 是否完成刷新、页面是否混用了新旧版本引用。如果已经出现线上报错，我会先做版本提示和强制刷新止血，必要时保留兼容资源；长期治理上要把入口缓存、资源指纹、版本检测和刷新策略一起设计好，避免发版靠运气。
21. 流式接口偶发中断，你会如何分析网络层和应用层问题？
答：我会先看连接是否被代理、超时或网络抖动中断，再查服务端是否主动结束、返回格式是否违规、前端 reader 消费是否异常、状态机有没有把中断误判成完成。止血上我会先补中断提示和重试入口，避免用户无感失败；长期治理再把链路日志、requestId 和中断原因打通。
22. 用户快速切换会话时，旧请求结果污染了新页面，怎么解决？
答：我会从两个层面处理。第一，切换会话时尽量取消旧请求；第二，结果返回时再校验当前会话 id 或 requestId，确保旧结果即使晚到也不会落到新页面。取消请求是第一道防线，结果校验是最后一道保险，两个都要有。

## 五、登录态与权限场景

23. 登录态一般是如何实现的？你会如何解释完整链路？
答：我会按“登录获取凭证、前端托管或存储、请求自动携带、服务端校验、续签刷新、退出清理”这条线来讲。然后再补为什么选 cookie 还是 token，安全边界在哪，前端负责体验和状态同步，真正的鉴权还是以服务端为准。
24. 如果用户明明已登录，但页面权限按钮突然消失，你怎么排查？
答：我会先确认权限数据有没有拉到，再看前端权限判断逻辑是否用了旧缓存、初始化时机是否有竞态、按钮是否被错误条件渲染隐藏。如果问题已经影响使用，我会先临时放开前端展示或降级提示，但同时确保服务端权限不放松；长期上要把权限来源和前端判定逻辑统一起来，避免各页面自己写一套。
25. 如果 token 过期了，前端应该怎么处理？
答：我会做统一拦截。能静默刷新就先刷新并重放请求，刷新失败就清理本地状态并引导用户重新登录。这里我会特别注意并发请求下的刷新风暴，所以一般会做单飞控制，避免多个请求同时触发刷新把链路打乱。
26. 页面级权限、路由级权限、按钮级权限分别应该放在哪里控制？
答：页面级和路由级权限我会放在路由或布局层统一拦截，按钮级权限放在组件层做细粒度控制。这样入口清晰、规则集中，也更方便治理。但我一定会补一句：前端权限更多是体验控制，真正的数据权限还是服务端兜底。
27. 如果权限逻辑散落在页面里，后续会有什么问题？如何治理？
答：问题会集中在规则不一致、重复判断、边界遗漏和维护成本高，最后谁也不敢轻易改。治理上我会把权限数据源、权限判断方法和公共组件先统一，再逐步把散落逻辑收口到配置和公共能力里，这样后续变更才可控。

## 六、工程化与发布场景

28. 线上网站用户正在使用时重新部署前端，出现报错怎么办？
答：我会先止血，优先控制影响面，比如提示刷新、关闭高风险入口、必要时快速回滚；然后再确认是不是入口缓存、资源 hash、接口契约或按需 chunk 版本不一致导致的。线上处理里，第一优先级永远是先让用户能继续用，第二才是把根因查透。
29. 为什么前端重新发版后，运行中的旧页面可能报错？
答：因为旧页面内存里跑的是旧代码，但后续加载到的 chunk、配置数据甚至接口契约可能已经切到新版本了，一旦两边不兼容，就会出现脚本加载失败、运行时异常或路由错误。这个问题本质上是版本一致性，不是单个页面代码写错这么简单。
30. 如何缓解前端资源更新导致的版本不一致问题？
答：我会至少做四件事：静态资源带 hash、入口 HTML 合理控缓存、客户端做版本检测、发现大版本变化时主动提示刷新。高风险业务再加灰度、兼容期和双版本资源保留。这里的 trade-off 是工程复杂度会上升，所以我会按业务风险决定治理深度。
31. 你做过灰度发布吗？如果没做过，你会怎么设计简化版灰度？
答：如果没完整做过，我会按最小可用方案回答：先确定灰度维度，比如白名单、分桶比例或环境开关；再补监控、开关和快速回滚；最后定义放量标准。对我来说，灰度的本质不是平台多复杂，而是有没有能力把风险收口在可控范围内。
32. 什么场景适合做灰度发布，什么场景必须做灰度发布？
答：影响面大、依赖链路长、上线后难快速修复的功能都适合灰度；像登录、支付、权限、核心交易和 AI 高风险能力这类，一般更接近必须灰度。因为这些链路一旦出问题，用户和业务损失都比较直接，不能靠全量上线后再观察。
33. 如果某次上线后错误率明显上升，你会如何定位和回滚？
答：我会先确认错误率上升和发布时间是否强相关，再按页面、浏览器、用户分层、接口和版本维度拆开看。如果确认影响面大且短时间内收敛不了，我会优先回滚或关闭开关止血；随后再用日志和监控定位根因。线上问题里，及时止损往往比坚持把问题查清再动手更重要。

## 七、大文件与上传场景

34. 为什么服务器往往处理不了超大文件一次性上传？
答：因为一次性上传会长时间占用带宽、连接和服务端资源，中间任意一步失败都得整包重来，成本很高。弱网场景下，这种方案成功率和体验都会很差，所以大文件方案本质上是在把高风险长链路拆成可恢复的短链路。
35. 你如何设计一个大文件上传方案？
答：我会从稳定标识、切片策略、并发控制、失败重试、断点续传和服务端合并校验这几部分设计。前端负责切片、算文件指纹、调度上传和展示进度，服务端负责记录已传分片、合并文件和做完整性校验。如果文件特别大，我还会把哈希计算放到 Worker，避免卡主线程。
36. 切片大小和切片数量如何决定？
答：我会结合文件大小、用户网络、服务端限制和失败重传成本来定。切片太大，单片失败代价高；切片太小，请求数暴涨、调度和合并成本也高。所以我不会报一个固定值，而是强调这个参数应该根据场景调优。
37. 为什么切片过多也会有问题？
答：因为切片过多会带来更多请求、更多状态管理、更高合并成本，整体甚至可能更慢。切片是一个典型 trade-off，不是越细越好，而是在稳定性、调度成本和吞吐之间找平衡。
38. 并发上传数量应该怎么控制？
答：我会设置一个可配置的并发上限，并根据弱网场景和服务端承压做动态调整。并发太低速度慢，并发太高失败率和服务端压力都会上升，所以我会优先追求稳定吞吐，而不是瞬时跑满。
39. 上传中断后如何做断点续传？
答：核心是先给文件一个稳定标识，重新上传前先问服务端哪些分片已经存在，然后只补剩余部分。真正要注意的是标识是否稳定、本地文件是否变化、服务端状态是否可信，这几件事没处理好，断点续传很容易变成假续传。
40. 如何实现秒传？
答：我会先由前端计算文件指纹，再由服务端判断相同内容是否已存在；如果服务端确认可复用，就直接返回成功。秒传本质上是服务端的内容去重能力，前端只是提供识别和协商，不会自己私自跳过上传。
41. 上传进度、失败重试、取消上传如何设计？
答：我会把每个分片状态独立管理成待传、上传中、成功、失败、取消，再聚合成整体进度。失败重试要支持单片重试，取消要能停止调度和请求。对用户来说，能看到进度、知道哪里失败、还能继续上传，比“黑盒式等待”重要得多。

## 八、Worker 与重计算场景

42. 什么场景下你会使用 Web Worker？
答：当任务明显吃 CPU、会阻塞主线程并影响输入、滚动和点击时，我会考虑 Worker，比如大文件哈希、图片处理、CSV 解析、复杂文本处理。是否引入 Worker，我还会评估通信成本、调试复杂度和兼容性，不会把它当默认方案。
43. 如果主线程需要和 Worker 通信多次，你会如何设计消息协议？
答：我会统一消息结构，至少包含 `type`、`requestId`、`payload`、`status` 和 `error`。这样后续任务多了也不会混乱，排障时也能快速知道是哪一类任务、处于什么阶段。协议化本身就是一种长期治理。
44. 如果同时发送两条消息给 Worker，如何保证主线程拿到对应结果？
答：我会给每条请求分配唯一 `requestId`，Worker 返回时原样带回，主线程按 id 做结果路由。进一步我还会加超时、取消和异常透传，确保并发任务不会串结果。
45. Worker 返回的结果应该如何带上上下文信息？
答：除了结果本身，我会一起返回 `requestId`、任务类型、业务标识、阶段状态和错误信息。主线程真正需要的不是一段裸数据，而是知道这是谁的结果、当前处于什么阶段、失败时怎么处理。
46. 为什么大文件哈希计算、图片处理、CSV 解析适合放到 Worker？
答：因为它们通常都是 CPU 密集型任务，放在主线程会直接拖慢交互。把这些任务挪到 Worker，本质上是在保护主线程响应能力，而不是单纯为了用上一个更“高级”的技术点。
47. 如果浏览器不支持 Worker，是否需要降级？怎么降级？
答：要看这个能力是不是核心链路。核心功能我会做降级，比如改成分片调度、降低并发或明确提示性能会变差；非核心功能可以适度弱化体验。我的原则是先保可用，再谈一致体验。

## 九、AI 前端场景

48. 为什么一个 AI 项目里会同时用 `Node.js` 和 `Python`？
答：我会从分层职责回答。Node.js 更适合做 BFF、流式转发、会话层和前端友好的接口编排，因为它离前端更近；Python 更适合模型调用、数据处理和算法能力，因为生态更成熟。这个组合不是为了炫技术栈，而是为了让边界更清晰、协作更顺。
49. 为什么 `Node.js` 通常更适合做 BFF、流式转发和会话层？
答：因为这层工作本质上偏 I/O、编排和协议适配，强调和前端链路的一致性与开发效率。Node.js 在这里的优势很明显，但我也会补一句，适合不代表唯一选择，最终还要看团队能力和基础设施。
50. 为什么 `Python` 更适合模型推理和算法处理？
答：因为 Python 在模型框架、数据处理和算法库上的生态更成熟，很多能力能以更低成本接入和验证。尤其 AI 项目试错阶段很快，这种生态优势会非常直接地体现在研发效率上。
51. 如果 AI 输出有问题，但功能已经上线，你会怎么处理？
答：我会先止血，控制影响范围，比如收紧入口、降低曝光、加明显提示、必要时关闭高风险能力；然后再拆问题是在 Prompt、模型、工具调用还是前端解析展示。AI 问题很多不是“彻底不可用”，而是“结果不稳定”，所以我会特别强调灰度、观测和前端兜底。
52. AI 功能上线后，你会关注哪些稳定性指标？
答：我会重点看成功率、超时率、首字耗时、总耗时、取消率、重试率、解析失败率和工具调用成功率。因为 AI 场景的问题很多不是报错，而是慢、中断、结果差或状态错乱，所以指标体系要比普通 CRUD 更丰富。
53. 为什么 AI 产品更适合配合灰度和观测体系？
答：因为 AI 结果天然有波动，很多问题不是“有没有响应”，而是效果变差、耗时变长、长尾异常增多。如果直接全量上线，风险很难控制；灰度和观测能让我们先用小流量验证，再逐步放量，把试错成本和影响面都压住。
54. 流式输出 UI 应该如何设计状态机？
答：我至少会拆成待发送、请求中、首段到达、持续生成、工具调用中、已完成、已中断、失败和重试中这些状态，并定义每个状态下用户能做什么、界面怎么提示。AI 流式链路里，状态机设计得不清楚，用户很快就会觉得系统不可信。
55. 如果模型输出不是稳定 JSON，前端要怎么兜底？
答：我会先做宽容解析，尽量从不完整结构里提取可展示内容；如果结构化解析失败，就回退纯文本展示，同时把异常样本、原始输出和上下文上报。这样既不至于让用户看到一片空白，也能为后端修 Prompt 或协议提供数据。

## 十、项目取舍题

56. 为什么项目用了 `Node.js`，又用了 `Python`？
答：我会强调分层分工。Node.js 负责接口编排、流式链路和会话层，更贴近前端交互；Python 负责模型和算法能力，更适合承接 AI 处理。只有当这种分工带来的收益明显大于协作成本时，这个组合才是合理的。
57. 为什么项目用的是 `JavaScript`，不是 `TypeScript`？
答：我会先承认现实，比如历史原因、团队现状或迭代节奏导致短期没有完全切到 TypeScript；然后再补我的工程判断，如果项目会长期维护，我会从公共接口、核心模块和高频变更区域优先补类型。这样的表达既真实，也说明我知道长期更优的方向。
58. 为什么选 `Vite`，而不是 `Webpack`？
答：如果项目更看重本地开发体验、启动速度和配置轻量，我会更倾向 Vite；如果项目历史体系重、已有很多 Webpack 定制能力，那继续用 Webpack 也可能更稳。技术选型题里我更强调场景匹配，而不是站队。
59. 为什么选 `Next.js`，而不是纯 `React`？
答：如果项目需要 SSR、路由约定、服务端能力或更好的首屏体验，我会倾向 Next.js，因为它把很多框架能力收在一起；但如果是内部系统、强客户端交互且 SSR 价值不高，纯 React 反而更轻。关键在于需求和复杂度是否匹配。
60. 为什么这个功能要做成 Worker，而不是直接在主线程里处理？
答：因为这个任务已经明显影响主线程交互，比如输入、滚动或页面响应。把它移到 Worker，是为了先保护用户体验。当然我也会说明 trade-off：通信、调试和兼容处理都更复杂，所以只有在问题被验证后才值得引入。
61. 为什么某个功能需要 BFF，而不是前端直接调后端？
答：因为 BFF 可以统一做协议适配、聚合编排、权限会话和错误收敛，把面向 UI 的复杂度集中收口。尤其当前端要面对多个后端或多种返回格式时，BFF 价值很明显；但如果链路很简单，我也不会强行加一层，因为那样只会增加维护成本。

## 十一、项目表达型场景题

62. 实习中你具体做了什么？
答：我会从业务背景、职责边界和代表性工作来讲，而不是按时间线报任务。比如我会先说明这段实习解决的核心业务问题，再说我主要负责哪几个模块，最后挑两三件最能代表我的工作展开，包括我做过的优化、排查过的问题和带来的结果。这样面试官更容易判断我的真实贡献。
63. 你在前端上最大的提升是什么？
答：我最大的提升不是多会了几个 API，而是开始从工程视角看问题。以前更关注功能实现，现在会主动考虑状态边界、性能瓶颈、异常兜底、监控观测和长期维护成本。这个变化说明我已经不只是做页面，而是开始对链路结果负责。
64. 你做过哪些性能优化？
答：我会挑几个有代表性的真实案例来讲，比如首屏加载优化、减少重复渲染、长列表虚拟化、把重计算挪到 Worker。每个案例我都会按“现象、定位、方案、结果”讲，而不是只报手段。因为性能题最能区分一个人是会背技巧，还是做过真实分析。
65. 这段实习里你遇到的最难问题是什么？
答：我会选一个影响真实用户、排查链路长、而且中间有判断转折的问题来讲。这个题的重点不是把问题讲得多夸张，而是让我证明自己在不确定问题里能建立优先级、收集证据、推进修复并补长期治理。
66. 你是怎么排查并解决的？
答：我通常按“现象、假设、验证、定位、修复、复盘”来讲。过程中我会特别强调为什么优先怀疑这个方向、用什么证据排除了其他可能、止血动作是什么、长期治理又做了什么。这样答案既有过程，也有结果。
67. 如果再做一遍这个项目，你会怎么优化？
答：我会把几个后期才意识到很重要的动作前置，比如更早补类型约束、监控告警、错误兜底、版本治理和公共能力抽象。这个题我想体现的不是“当时做错了”，而是“现在我已经知道哪些事情应该更早做，才能让后续迭代更稳”。

## 十二、回答框架模板

我回答场景题时，默认会按下面这个框架组织：

1. 先复述问题，确认边界。
2. 先给结论，再拆可能原因。
3. 按优先级讲排查动作和判断依据。
4. 说明短期止血方案。
5. 再说明长期治理和预防复发。

例如“页面白屏怎么分析”，我至少会覆盖资源、接口、脚本、渲染和版本一致性这几层，而不是只给一句“先看控制台”。
