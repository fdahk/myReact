网络与缓存（Networking & Caching）
网络：App 如何请求 API、处理错误、重试、超时、鉴权、并发与取消。
缓存：把请求结果/资源存起来，减少重复请求、加速首屏、支持弱网。
关键概念：
缓存策略：Cache-First / Network-First / Stale-While-Revalidate（先用旧数据再后台刷新）
缓存粒度：内存缓存（快但易丢） vs 磁盘缓存（稳定但慢一些）
一致性：数据更新后如何失效缓存/刷新 UI



离线（Offline）
概念：无网或弱网情况下仍能“可用”（至少可浏览/可操作/可排队）。
典型能力：
离线可读：上次内容可打开（本地持久化）
离线可写：操作先记录到本地队列，联网后再同步（需要冲突处理）
网络状态感知：提示、降级、重试策略



原生桥接（Native Bridge）
概念：Flutter 与 iOS/Android 原生能力互通（比如蓝牙、推送、支付、相机底层能力）。
常见方式：
使用现成 plugin
自研 plugin（MethodChannel/EventChannel），处理参数序列化、线程、回调、权限、生命周期




量化结果：首屏、包体（Metrics）
首屏（First Screen / Time to First Render / Time to Interactive）
是什么：从用户点击图标到“第一个有意义的页面内容可用”的耗时。
为什么重要：直接影响用户对“快不快”的第一印象。
怎么量化：定义清晰埋点（启动开始、首帧、首屏关键数据展示完成），统计 P50/P90/P95。
包体（App Size / APK/IPA Size）
是什么：安装包大小、下载体积、增量更新体积。
为什么重要：影响下载转化、更新成本、存储占用。
怎么量化：发布产物的大小、拆分后各模块/资源占比、优化前后对比。





灰度（Gray Release / Canary / Staged Rollout）
是什么：新版本/新功能不一次性给所有人，而是按比例或按人群逐步放量（例如 1%→10%→50%→100%）。
为什么：降低风险；一旦崩溃率、关键指标变差可以快速止损/回滚。
怎么做（概念层面）：
版本灰度：应用商店分阶段发布（常见）
功能灰度：通过 Feature Flag/远程配置对功能开关分人群生效
观察指标：崩溃率、启动成功率、关键链路转化、接口错误率、卡顿等
