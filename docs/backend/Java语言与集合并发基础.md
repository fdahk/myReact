# Java 语言与集合并发基础

## 使用方式

这篇不是语法速查，而是按大厂后端面试和真实工程所需的方式整理 `Java` 基础。你学习时不要只记“这个 API 怎么用”，更要同时回答：

- 这个概念解决什么问题。
- 它和 `Node.js / JavaScript` 的差异在哪里。
- 它在后端工程里最常出现在哪些场景。
- 面试追问时会从哪里继续挖。

---

## 一、为什么你转后端必须先补好 Java 基础

1. 为什么已经有 `Node.js` 经验，还要重新补 `Java` 基础？
答：因为你缺的不是“会不会写接口”，而是国内主流后端岗位对 `Java` 生态的工程可信度。`Node.js` 能让你更快理解 HTTP、鉴权、异步流程和接口编排，但顶级后端岗位会继续追问你 `Java` 语言特性、并发模型、集合、内存、JVM、线程池、锁和工程规范。如果这些地方不扎实，面试官会默认你只是“会做一点后端的前端”，而不是“真正的 Java 后端工程师”。

2. Java 在国内大厂后端里最核心的价值是什么？
答：一是生态成熟，`Spring`、数据库、中间件、监控、运维、规范都很完整；二是工程协作成本低，模块边界、类型系统、接口约束和团队共识都更强；三是长期维护能力强，适合复杂业务系统。很多公司并不是因为 Java 语法最好才用它，而是因为它在中大型团队、多业务线和长期演进场景下更稳。

3. 从 `JavaScript` 切到 `Java`，最重要的认知切换是什么？
答：最重要的是从“动态灵活”切到“静态约束 + 明确边界”。`JavaScript` 更强调运行时灵活性，`Java` 更强调编译期可验证、明确类型和工程结构稳定。对后端来说，这种约束不是负担，而是让复杂系统更可维护的基础，因为服务之间、模块之间、团队之间都需要更高的可推导性。

## 二、面向对象与类型系统

4. Java 是一门什么风格的语言？
答：Java 是强类型、静态类型、面向对象为主、同时逐渐吸收函数式特性的通用语言。它的核心特点不是“类很多”，而是尽量在编译期把接口契约、类型关系和一部分错误暴露出来，这和大型后端服务追求稳定性、可维护性天然契合。

5. 面向对象在后端里真正解决什么问题？
答：它真正解决的是职责划分和复杂度管理问题。后端系统不是只有一堆函数，而是有控制器、服务、仓储、领域模型、DTO、配置对象、异常模型等多个层次。面向对象让你能把“谁负责接请求、谁负责编排、谁负责数据访问、谁负责规则约束”拆得更清楚。它的价值不在于背继承和多态，而在于在复杂业务中保持边界清晰。

6. 封装、继承、多态里，后端最常用的其实是什么？
答：最常用的是封装和接口抽象，多态其次，继承反而要谨慎。因为大厂后端更强调组合优于继承，过度继承会让层级复杂、耦合难拆。真正高频的是：通过接口定义能力边界，通过实现类承载不同策略，通过组合组装业务流程。

7. 为什么接口在 Java 后端里这么重要？
答：因为接口既是抽象，也是协作契约。它让调用方只依赖能力，不依赖具体实现，这对于测试、扩展、切换实现和模块解耦都非常重要。比如支付能力、存储能力、消息发送能力都适合先抽接口，再根据环境和业务用不同实现去承接。

8. 泛型解决了什么问题？
答：泛型解决的是“同一套逻辑在不同类型上复用，同时保持类型安全”的问题。没有泛型时，你只能写成 `Object`，然后到处强转，这会把很多错误拖到运行时。泛型的价值在大厂后端里很明显，因为公共响应体、分页结构、通用查询结果、缓存封装、集合操作都非常依赖它。

## 三、字符串、包装类与常见语言细节

9. 为什么 `String` 被设计成不可变？
答：因为字符串在后端里用得极高频，常用于配置、日志、缓存 key、SQL 片段、HTTP 参数和安全相关逻辑。不可变让它天然线程安全、可缓存 hash、可安全复用，也避免在多处共享引用时被意外篡改。面试时不要只说“安全”，还要说这对线程安全、常量池和性能优化都有帮助。

10. `==` 和 `equals` 的差异是什么？
答：`==` 比较的是基本类型的值，或引用类型的地址；`equals` 默认也是比较地址，但很多类会重写成“值相等”。在后端里这个问题看起来基础，却经常影响缓存 key 判断、对象比较和业务条件分支。面试官问这个题，不是想考小学语法，而是看你有没有对象语义意识。

11. 自动装箱拆箱为什么会成为坑？
答：因为 `Integer`、`Long` 这类包装类在比较、判空、集合操作里很容易引发隐式转换和 `NullPointerException`。例如数据库字段映射成 `Integer` 后，如果你不判空直接参与运算，就可能出问题。真实业务里这些坑比语法题更常见。

## 四、集合体系

12. 为什么集合是 Java 后端面试高频？
答：因为绝大多数业务逻辑最终都在处理数据容器：请求列表、查询结果、缓存内容、去重结果、分组聚合、中间状态、消息批处理。集合不只是“会不会用”，更关键的是你是否知道不同结构的时间复杂度、线程安全特性、扩容行为和适用场景。

13. `List`、`Set`、`Map` 的本质差异是什么？
答：`List` 强调顺序和可重复；`Set` 强调唯一性；`Map` 强调键值映射。业务里经常不是“选一个都会用”，而是要先判断你究竟关心顺序、去重还是索引查找。例如订单明细用 `List`，权限 code 去重适合 `Set`，用户信息缓存适合 `Map`。

14. `ArrayList` 为什么常用？
答：因为它基于动态数组，随机访问快，遍历性能也好，绝大多数读多写少、尾部追加为主的业务场景都适合它。它的 trade-off 是中间插入、删除代价较高，扩容会复制数组，所以大批量写入或频繁中间操作时要特别注意。

15. `LinkedList` 为什么现实业务里不常作为首选？
答：因为它虽然理论上插入删除方便，但节点离散、缓存不友好、随机访问慢，很多真实业务场景下并不比 `ArrayList` 更优。面试里如果你只背“链表插入快”，会显得停留在教材层。更成熟的回答是：具体要看访问模式、内存局部性和真实操作分布。

16. `HashMap` 的核心原理是什么？
答：它通过 hash 把 key 映射到数组桶位，再在桶里处理冲突，JDK 8 以后冲突严重时会从链表转红黑树，以降低极端情况下的查找成本。你要会讲的不只是“查找快”，还要讲 hash 冲突、扩容、负载因子、树化条件以及为什么 key 的 `hashCode/equals` 一致性很重要。

17. 为什么 `HashMap` 线程不安全？
答：因为它的 put、resize、rehash 等操作没有同步控制，多线程并发写时可能发生数据覆盖、可见性问题甚至结构异常。后端里如果多个线程共享可变状态，而你还在用普通 `HashMap`，基本就是隐患。

18. 什么场景下用 `ConcurrentHashMap`？
答：当多个线程需要高并发读写共享映射结构时。它的价值是尽量避免全表大锁，提高并发性能。你在缓存、本地状态表、任务上下文表、幂等记录或统计计数场景里都可能用到它。但也要注意，它不是银弹，如果业务需要复合操作原子性，仍然要额外设计同步策略。

19. 为什么 `Set` 常和权限、去重、标签系统绑定？
答：因为这类场景天然关心唯一性。比如角色 code、已处理消息 id、资源标签、黑名单 key，这些都比顺序更重要。你要有“业务语义驱动数据结构选择”的意识。

## 五、异常处理

20. Java 为什么区分受检异常和非受检异常？
答：它的初衷是把“调用方必须明确处理的错误”与“更偏编程错误或系统异常”的问题区分开。虽然现实里很多团队不会大量自定义受检异常，但这个设计体现了 Java 很重视错误显式表达。后端里更关键的是：不要把异常当流程控制，也不要静默吞异常。

21. 后端里异常处理最重要的原则是什么？
答：边界清晰、语义明确、统一收口。控制层关注把异常映射成合适的 HTTP 响应；服务层关注业务异常和系统异常分层；底层关注保留上下文信息。大厂项目最怕异常四处乱抛、文案不统一、日志打不全，导致排障困难。

22. 为什么不建议 `catch (Exception e) {}` 后什么也不做？
答：因为这会把错误直接吞掉，让问题从“立刻暴露”变成“线上慢性污染”。后端最怕 silent failure，看起来接口返回了，实际上状态已经不一致、数据已经没写进去、消息已经没发出去。

## 六、并发基础

23. 为什么 Java 后端必须系统学习并发？
答：因为服务端天然运行在多线程环境里。一个 Web 服务同时处理很多请求，一个异步任务系统同时消费很多消息，一个线程池同时执行很多任务，如果你不理解并发，就很难解释线程安全、锁竞争、性能瓶颈、任务积压和数据一致性问题。

24. 进程和线程的核心差异是什么？
答：进程是资源分配单位，线程是 CPU 调度单位。对后端开发者来说，更重要的是理解：同一进程内线程共享堆内存，因此共享数据时会有线程安全问题；而多进程隔离更强，但通信成本更高。

25. 什么是线程安全？
答：多线程同时访问同一共享资源时，无论调度顺序如何，程序都仍然能得到正确结果。注意它不只是“不报错”，而是结果符合预期。后端里凡是共享可变状态，都要有线程安全意识。

26. `synchronized` 的本质是什么？
答：它本质是基于对象监视器的互斥同步手段，用来保证同一时刻只有一个线程进入临界区，并建立 happens-before 关系，确保可见性。回答时不要只说“加锁”，还要知道它同时解决互斥和内存可见性。

27. `volatile` 解决什么问题，不能解决什么问题？
答：它解决的是可见性和一定程度的有序性，不能保证复合操作原子性。比如一个布尔开关、配置刷新标记适合用 `volatile`，但 `count++` 这种读改写复合操作只靠它不够。这个题是面试高频，因为很多人会把它当“轻量锁”误用。

28. 什么是原子性、可见性、有序性？
答：原子性是操作不可被中断拆分；可见性是一个线程修改后其他线程能及时看到；有序性是程序执行顺序在并发场景下是否符合预期。Java 并发的大多数问题，本质上都落在这三件事上。

29. `CAS` 是什么？
答：Compare-And-Swap，一种乐观并发控制方式。线程更新值时，先比较内存中的旧值是否仍然是自己预期的值，如果是就更新，否则重试。它避免了很多轻量更新场景下的重量级锁竞争。`AtomicInteger` 等原子类背后就依赖它。

30. 乐观锁和悲观锁怎么理解？
答：悲观锁默认别人会冲突，所以先锁住再做；乐观锁默认冲突不常见，所以先尝试更新，失败再重试。数据库版本号更新、CAS、自旋都更偏乐观思想；`synchronized`、互斥锁更偏悲观思想。没有谁绝对更好，取决于冲突概率和临界区成本。

## 七、线程池

31. 为什么后端不能随便 `new Thread()`？
答：因为线程创建和销毁成本高，数量失控会把 CPU、内存、上下文切换都拖垮。线程池的价值是统一管理线程资源、任务队列、拒绝策略和监控入口，让服务在高并发下更可控。

32. 线程池最重要的参数有哪些？
答：核心线程数、最大线程数、队列容量、线程空闲时间、拒绝策略、线程工厂。面试官问线程池，不是要你背构造函数，而是看你是否知道这些参数如何影响吞吐、延迟、资源占用和故障行为。

33. 为什么线程池参数不能照抄？
答：因为 CPU 密集、IO 密集、队列长度、任务耗时、峰值流量都不同。线程池配置本质上是容量规划问题，而不是模板题。大厂后端更看重你有没有根据业务特征做判断的能力。

34. 拒绝策略为什么重要？
答：因为线程池满了是高压状态的明确信号。拒绝策略决定你是直接失败、抛异常、由调用线程兜底执行，还是丢弃旧任务。这个决策直接关系到系统在过载时的表现，是“优雅降级”还是“悄悄积压到崩”。

## 八、你现阶段最该掌握到什么程度

如果你是从前端和 `Node.js` 转到 Java 后端，这篇至少要达到下面标准：

1. 能系统讲清 Java 为什么适合中大型后端。
2. 能说清集合选择依据，而不是只会 API。
3. 能解释线程安全、锁、`volatile`、`CAS`、线程池的核心区别。
4. 能把这些概念映射到真实业务，比如缓存、幂等、异步任务和并发请求处理。

如果这些做不到，你的后端表达仍然会偏“业务开发者”，还没进入“Java 后端工程师”状态。

## 九、AQS、锁和并发容器深水区

35. 为什么 `ReentrantLock` 和 `synchronized` 经常被放在一起问？
答：因为它们都能解决互斥问题，但能力边界不同。`synchronized` 更偏语言级内置同步，表达简单；`ReentrantLock` 更偏显式锁，能提供更灵活的中断、超时、条件队列等能力。面试里成熟的回答不是说谁“更高级”，而是看你是否知道什么时候需要更多控制力。

36. 什么是 `AQS`，为什么重要？
答：`AQS` 可以理解成 Java 并发包里很多同步器的基础框架，用一个状态值和等待队列去组织线程竞争。你不一定要把它源码背到每一行，但至少要知道：很多锁、信号量、倒计时器并不是各写一套，而是建立在统一同步框架之上。这能明显提升你讲并发题时的层次感。

37. 为什么 `ConcurrentHashMap` 不等于所有并发操作都安全？
答：因为它只能保证单个基础操作在并发下相对安全，不自动保证多个操作组合起来的原子性。比如“先查再改”“先判断再写”这种复合逻辑，仍然可能有竞态问题。这个题很能看出你是否真正理解并发容器的边界。

38. 为什么 `CopyOnWriteArrayList` 适合读多写少场景？
答：因为它在写时复制底层数组，读操作几乎不需要加锁，读性能和并发读取体验很好。但如果写很多，复制成本会很高。它非常适合配置快照、订阅者列表、白名单这类读多写少场景，不适合高频写入主链路。

## 十、线程池与并发问题的真实场景

39. 线程池打满时，为什么不能只说“把线程数调大”？
答：因为线程池打满通常只是结果，不是根因。根因可能是任务过长、下游阻塞、连接池耗尽、锁竞争、队列过大、重试风暴或任务模型错误。盲目加线程数可能只会让上下文切换和资源竞争更严重。

40. 为什么队列大小也是线程池设计的关键？
答：因为队列决定了系统在高压下是更早失败、还是先积压。队列太大容易把问题藏起来，最后形成长尾延迟和内存压力；队列太小又可能过早触发拒绝。线程池设计本质上也是过载治理。

41. 为什么拒绝策略题很有后端味？
答：因为它直接体现系统在过载时的行为选择。是快速失败、调用方回退、丢弃旧任务、还是继续阻塞，背后都是业务优先级和系统保护策略。成熟后端不会把拒绝策略只当成一个枚举值，而会联系业务语义来讲。

42. 多线程并发问题为什么常和缓存、幂等、任务系统绑定？
答：因为这些系统天然存在共享可变状态。缓存要考虑并发更新、任务系统要考虑重复执行和状态推进、幂等要考虑多次请求竞争写入，这些都是 Java 并发基础最容易映射到真实业务的地方。

## 十二、HashMap 底层原理详解

43. HashMap 的底层数据结构到底是什么样的？
答：底层是一个 `Node<K,V>[] table` 数组，每个槽位称为"桶"(bucket)。每个 `Node` 包含 `hash`、`key`、`value`、`next` 四个字段。当多个 key 落到同一个桶时，用链表串联；当同一个桶的链表长度 >= 8 **且** 数组总长度 >= 64 时，链表会转为红黑树（`TreeNode`）。如果数组长度不到 64，即使链表长度达到 8 也只会触发扩容而不会树化。红黑树在节点减少到 6 以下时会退化回链表。这个结构是 JDK 8 引入的，JDK 7 只有数组 + 链表。

44. HashMap 的 hash 计算为什么要做扰动？
答：HashMap 的 hash 计算不是直接用 `key.hashCode()`，而是 `(h = key.hashCode()) ^ (h >>> 16)`，即把高 16 位异或到低 16 位上。原因是定位桶时用的是 `(n - 1) & hash`（n 是数组长度），如果数组长度较小（比如 16，n-1 = 0x0F），只有低 4 位参与索引计算，高位完全被忽略。做一次高低位异或，能让高位的信息也参与桶定位，减少只有高位不同的 key 发生碰撞的概率。这就是所谓的"扰动函数"。

45. 为什么 HashMap 的容量必须是 2 的幂？
答：因为定位桶的公式是 `(n - 1) & hash`，只有当 n 是 2 的幂时，`n - 1` 的二进制全部是 1（比如 16-1 = 0b1111），此时位与运算等价于取模 `hash % n`，但位运算比取模效率高得多。如果 n 不是 2 的幂，`n - 1` 的二进制中会有 0 位，导致某些桶永远不会被映射到，分布不均匀。构造时如果传入非 2 的幂，HashMap 会通过 `tableSizeFor()` 向上取整到最近的 2 的幂。

46. HashMap 默认参数分别是什么含义？
答：
- **初始容量** `DEFAULT_INITIAL_CAPACITY = 16`：table 数组的默认长度。
- **负载因子** `DEFAULT_LOAD_FACTOR = 0.75f`：当元素数量超过 `容量 × 负载因子`（即 threshold）时触发扩容。
- **树化阈值** `TREEIFY_THRESHOLD = 8`：单个桶的链表长度达到 8 时，如果满足最小树化容量条件，转为红黑树。
- **退树化阈值** `UNTREEIFY_THRESHOLD = 6`：扩容或删除后，红黑树节点数降到 6 以下时退化为链表。
- **最小树化容量** `MIN_TREEIFY_CAPACITY = 64`：数组总长度低于 64 时，即使链表到了 8 也不树化，而是先扩容。

47. 为什么负载因子默认是 0.75？
答：这是空间和时间的折中。负载因子越小，桶越稀疏，碰撞越少但浪费空间；越大越省空间但碰撞越多。0.75 的依据是泊松分布推导：在负载因子为 0.75 时，一个桶中出现 8 个节点的概率约为 0.00000006（千万分之六），链表极少会长到需要树化。JDK 源码注释里有这段概率说明。

48. HashMap 的 put 完整流程是什么？
答：
1. 计算 key 的 hash：`(h = key.hashCode()) ^ (h >>> 16)`。
2. 如果 table 为 null 或长度为 0，先调用 `resize()` 初始化。
3. 用 `(n - 1) & hash` 定位桶，取出头节点。
4. **桶为空**：直接 `new Node` 放入该桶。
5. **桶不为空**：
   - 头节点的 key 与待插入 key 相同（hash 相等且 equals 为 true）→ 直接覆盖 value。
   - 头节点是 `TreeNode` → 调用红黑树的 `putTreeVal` 插入。
   - 否则遍历链表，如果找到 key 相同的节点则覆盖；到链尾仍没找到则尾插新节点。尾插后如果链表长度 >= 8，调用 `treeifyBin`（内部再判断数组长度是否 >= 64，不够则扩容，够则树化）。
6. 更新 `modCount`（快速失败机制）。
7. `++size` 后如果超过 threshold，调用 `resize()` 扩容。

49. HashMap 扩容的具体过程是什么？JDK 8 做了什么优化？
答：扩容时创建一个新数组，容量是旧数组的 2 倍，新 threshold 也翻倍。然后遍历旧数组的每个桶，把节点重新分配到新数组。JDK 8 的关键优化是：**不需要对每个节点重新计算桶位**。因为新数组容量是旧的 2 倍，一个节点在新数组中的位置只有两种可能：要么还在原来的位置 `j`，要么在 `j + oldCap`。判断方法是看 `hash & oldCap`（注意不是 `oldCap - 1`，是 `oldCap` 本身），如果结果为 0 则在原位，不为 0 则在原位 + oldCap。这样每个桶里的节点被拆成两条链 `lo` 和 `hi`，分别放到新数组的两个位置，避免了 JDK 7 中对每个节点重新 hash 的开销。

50. HashMap 为什么线程不安全？JDK 7 和 JDK 8 各有什么典型问题？
答：
- **JDK 7 的死循环问题**：JDK 7 扩容时对链表用**头插法**迁移节点。如果两个线程同时扩容，链表的 next 指针可能被交叉修改，形成环形链表。后续 get 如果落到这个桶就会死循环，CPU 飙到 100%。这是经典故障。
- **JDK 8 改为尾插法**，解决了成环问题，但并发 put 仍然不安全：两个线程同时判断桶为空，都尝试写入，其中一个会丢失；并发更新 `size` 字段会导致计数不准；并发 put 和 resize 交叉执行可能导致节点丢失。
- 根本原因是 HashMap 的所有写操作都没有任何同步保护。多线程写场景必须用 `ConcurrentHashMap` 或手动加锁。

## 十三、ConcurrentHashMap 底层原理详解

51. JDK 7 的 ConcurrentHashMap 是怎么实现的？
答：JDK 7 采用**分段锁**（Segment）架构。底层结构是 `Segment[]` 数组，每个 Segment 继承 `ReentrantLock`，内部包含一个 `HashEntry[]` 桶数组。读写操作先定位到对应 Segment，再对该 Segment 加锁。不同 Segment 之间的操作完全互不影响，所以并发度 = Segment 数量，默认是 16，即最多 16 个线程同时写不同的段。缺点是 Segment 数量初始化后不可变，并发度上限固定；get 操作虽然利用 volatile 读大多数场景不需要锁，但 size() 需要多次尝试不加锁统计，统计不一致时最终回退到全部加锁。

52. JDK 8 的 ConcurrentHashMap 为什么要彻底重构？
答：JDK 8 完全抛弃了 Segment 分段锁，改为 `Node<K,V>[] table` + **CAS + synchronized 锁住单个桶头节点**的方案。原因有三：一是锁粒度更细，从锁整个 Segment（包含多个桶）变为锁单个桶（一个链表/树），并发度从固定的 16 提升到等于桶的数量（可以是成千上万）；二是内存更省，去掉了 Segment 的额外对象开销；三是结构与 HashMap 统一，维护成本更低，也能享受红黑树带来的最坏情况下 O(log n) 查找。

53. JDK 8 ConcurrentHashMap 的 put 流程是什么？
答：
1. 计算 hash：`spread(key.hashCode())`，即 `(h ^ (h >>> 16)) & 0x7fffffff`，强制正数。
2. 进入自旋循环（`for(;;)`）：
   - 如果 table 为 null，调用 `initTable()` 初始化（内部用 CAS 竞争 `sizeCtl` 来保证只有一个线程初始化）。
   - 用 `tabAt(tab, i)` 以 volatile 语义读取桶头节点。如果**桶为空**，直接用 **CAS** (`casTabAt`) 写入新 Node。CAS 成功则 break，失败说明有竞争，自旋重试。
   - 如果头节点的 hash 是 `MOVED`（值为 -1），说明该桶正在被迁移（扩容中），当前线程调用 `helpTransfer` 协助扩容，然后重试。
   - **桶不为空且未在迁移**：对头节点加 `synchronized` 锁，然后遍历链表或红黑树完成插入/更新。锁的范围仅限这一个桶，其他桶完全不受影响。
3. 插入后检查链表长度，如果 >= 8 则调用 `treeifyBin` 尝试树化。
4. 调用 `addCount` 更新元素总数，同时检查是否需要扩容。

54. ConcurrentHashMap 的 get 为什么可以无锁？
答：因为 `Node` 的 `val` 和 `next` 字段都用 `volatile` 修饰，`tabAt` 使用 `Unsafe.getObjectVolatile` 读取桶头节点，确保了内存可见性。读线程无需加锁就能看到最新的写入结果。同时 Node 的 key 和 hash 是 final 的，节点一旦创建这两个值不会变。这种设计让读操作完全无锁、无阻塞，是 ConcurrentHashMap 高读性能的关键。

55. ConcurrentHashMap 的 size 是怎么高效统计的？
答：直接用一个全局 `AtomicLong` 来计数在高并发下会成为严重瓶颈（所有线程 CAS 同一个变量）。JDK 8 借鉴了 `LongAdder` 的思路：维护一个 `baseCount` 和一个 `CounterCell[]` 数组。低竞争时直接 CAS 更新 `baseCount`；CAS 失败说明竞争激烈，就把增量分散到 `CounterCell[]` 的某个槽位中。最终 `size()` 返回的是 `baseCount + 所有 CounterCell 的值之和`。这样把热点分散到多个槽位，大幅减少 CAS 竞争。

56. ConcurrentHashMap 扩容（transfer）是怎么支持多线程协助的？
答：扩容时，旧 table 的桶被分成若干段（每段叫一个 stride，最小为 16）。发起扩容的线程从数组末尾开始，CAS 认领一段桶进行迁移。其他线程在 put/remove 时如果发现当前桶的头节点是 `ForwardingNode`（hash = MOVED），就知道正在扩容，会主动调用 `helpTransfer` 加入迁移。每个线程完成自己那段桶的迁移后，会继续 CAS 认领下一段，直到所有桶都迁移完成。已迁移完的桶会被放置 `ForwardingNode`，它的 `find` 方法会将读请求转发到新 table，保证扩容期间读操作仍然正常。这种多线程协助扩容机制让大 table 的扩容不再是单线程瓶颈。

## 十四、synchronized 锁升级机制

57. 为什么 synchronized 有锁升级？
答：因为 `synchronized` 在不同竞争强度下需要不同的代价。如果只有一个线程在用，上重量级锁纯属浪费；如果线程交替执行且竞争不激烈，用自旋比内核态切换划算。JVM 设计了从偏向锁 → 轻量级锁 → 重量级锁的升级路径，让锁在大多数无竞争场景下几乎零开销，只在真正竞争严重时才付出操作系统 mutex 的代价。

58. 对象头 Mark Word 在不同锁状态下长什么样（64 位 JVM）？
答：Mark Word 占 64 位，根据末尾的锁标志位存储不同内容：
- **无锁态**（lock = 01, biased_lock = 0）：`unused(25bit) | hashcode(31bit) | unused(1bit) | age(4bit) | biased_lock(0, 1bit) | lock(01, 2bit)`。调用过 `hashCode()` 后哈希值存在这里。
- **偏向锁**（lock = 01, biased_lock = 1）：`threadID(54bit) | epoch(2bit) | unused(1bit) | age(4bit) | biased_lock(1, 1bit) | lock(01, 2bit)`。存储持有偏向锁的线程 ID。
- **轻量级锁**（lock = 00）：`ptr_to_lock_record(62bit) | lock(00, 2bit)`。指向线程栈帧中 Lock Record 的指针。
- **重量级锁**（lock = 10）：`ptr_to_heavyweight_monitor(62bit) | lock(10, 2bit)`。指向 ObjectMonitor 的指针。
- **GC 标记**（lock = 11）：GC 期间标记使用。

59. 偏向锁是怎么工作的？
答：偏向锁适用于**只有一个线程反复进入同步块**的场景。第一次进入时，线程通过 CAS 将自己的 threadID 写入 Mark Word。后续同一线程再进入时，只需比较 Mark Word 中的 threadID 是否是自己，是的话直接进入，不需要任何 CAS 操作，开销极低。当另一个线程尝试获取同一把锁时，需要在安全点（safepoint）暂停持有偏向锁的线程，检查它是否还在同步块内：如果不在，撤销偏向锁恢复为无锁态，新线程重新竞争；如果还在，升级为轻量级锁。**注意：JDK 15+ 默认禁用偏向锁（JEP 374）**，因为现代应用中真正只有一个线程访问的场景越来越少，偏向锁的撤销开销反而成了负担。

60. 轻量级锁是怎么工作的？
答：轻量级锁适用于**多个线程交替执行、竞争不激烈**的场景。获取过程：
1. JVM 在当前线程的栈帧中创建一个 **Lock Record**，将 Mark Word 拷贝到 Lock Record 的 Displaced Mark Word 区域。
2. 用 CAS 尝试将对象头的 Mark Word 替换为指向这个 Lock Record 的指针（同时末两位变为 00）。
3. CAS 成功 → 获取轻量级锁。
4. CAS 失败 → 说明有其他线程在竞争。先进行**自适应自旋**（自旋次数根据之前该锁的成功/失败历史动态调整），如果自旋期间锁被释放了就 CAS 重试；如果自旋超过阈值仍失败，则膨胀为重量级锁。

61. 重量级锁的 Monitor 结构是什么？
答：重量级锁基于操作系统的 `mutex`（互斥量），涉及用户态到内核态的切换，代价最高。JVM 中对应 `ObjectMonitor` 对象，核心字段有：
- `_owner`：当前持有锁的线程。
- `_EntryList`：等待获取锁的线程队列（阻塞在 `synchronized` 入口处的线程）。
- `_WaitSet`：调用了 `wait()` 方法后进入等待状态的线程队列。
- `_count`、`_recursions`：记录重入次数。

线程获取锁时，如果 `_owner` 为空则 CAS 设为自己；不为空则进入 `_EntryList` 阻塞。锁释放时唤醒 `_EntryList` 中的线程竞争。调用 `wait()` 时线程从 `_owner` 移到 `_WaitSet`，调用 `notify()` 时从 `_WaitSet` 移回 `_EntryList`。

62. 锁能降级吗？
答：锁升级是**单向**的：偏向锁 → 轻量级锁 → 重量级锁，正常运行时不会降级。这是 JVM 的设计简化，因为降级的判定逻辑很复杂且收益不高。唯一的"例外"是 GC 期间：如果对象不再被任何线程持有锁且要被移动（比如 Young GC），GC 可能重置 Mark Word，但这不是通常意义上的"锁降级"，更像是锁状态的重置。

## 十五、AQS (AbstractQueuedSynchronizer) 原理

63. AQS 的核心结构是什么？
答：AQS 的核心就两样东西：一个 `volatile int state` 和一个 **CLH 变体的 FIFO 双向等待队列**。`state` 是同步状态，由子类赋予具体含义：在 `ReentrantLock` 中 state = 0 表示未锁定，state > 0 表示被持有且值等于重入次数；在 `Semaphore` 中 state 表示剩余许可数量；在 `CountDownLatch` 中 state 表示还需要 countDown 的次数。对 state 的操作通过 `getState()`、`setState()`、`compareAndSetState()` 三个方法完成。

64. CLH 队列在 AQS 中是怎么运作的？
答：AQS 的等待队列是一个双向链表（经典 CLH 队列是单向的，AQS 做了变体方便实现取消和条件队列）。每个等待的线程被封装为一个 `Node`，核心字段有：
- `thread`：等待的线程引用。
- `waitStatus`：节点状态，包括 `SIGNAL`(-1, 后继节点需要被唤醒)、`CANCELLED`(1, 线程已取消)、`CONDITION`(-2, 在条件队列中)、`PROPAGATE`(-3, 共享模式下传播唤醒) 和 0（初始状态）。
- `prev`：前驱节点指针。
- `next`：后继节点指针。

队列有一个 dummy `head` 节点（不代表任何线程），`tail` 指向队尾。新线程入队时 CAS 设置 `tail`，出队时修改 `head`。

65. AQS 独占模式的 acquire 流程是什么？
答：以 `ReentrantLock.lock()` 为例，最终调用 `AQS.acquire(1)`：
1. **tryAcquire(1)**：由子类实现。检查 state 是否为 0，是则 CAS 设为 1 并记录当前线程为 owner；如果 state > 0 且当前线程就是 owner，则 state + 1（重入）。成功则直接返回。
2. **addWaiter(Node.EXCLUSIVE)**：tryAcquire 失败，创建一个 EXCLUSIVE 模式的 Node，通过 CAS 加到队列尾部（`enq` 方法会自旋 + CAS 直到成功，第一次还会初始化 dummy head）。
3. **acquireQueued(node, 1)**：自旋循环——如果前驱是 head，再次调用 `tryAcquire`；成功则将自己设为新 head（出队），返回。如果前驱不是 head 或 tryAcquire 仍失败，调用 `shouldParkAfterFailedAcquire` 把前驱的 waitStatus 设为 SIGNAL（告诉前驱释放锁时要唤醒我），然后调用 `LockSupport.park(this)` 挂起线程。
4. 当持锁线程释放锁时（`release` → `tryRelease` → state 减到 0），调用 `unparkSuccessor` 唤醒 head 的后继节点，被唤醒的线程从 park 处继续自旋尝试获取锁。

66. AQS 共享模式和独占模式有什么不同？
答：共享模式用于允许多个线程同时获取的场景（如 Semaphore、CountDownLatch）。核心流程是 `acquireShared`：
1. **tryAcquireShared(arg)**：子类实现，返回负数表示失败，0 表示成功但后续线程不能再获取，正数表示成功且还有剩余许可。
2. 失败则入队（`SHARED` 模式 Node），挂起等待。
3. 获取成功后调用 **setHeadAndPropagate**：将自己设为新 head，并且如果还有剩余许可（返回值 > 0）或后继节点也是 SHARED 模式，会继续唤醒后续节点。这就是"传播"机制：一个线程获取共享资源成功后会级联唤醒后面排队的共享节点，而不是只唤醒一个。

67. ReentrantLock 的公平锁和非公平锁在 AQS 层面有什么区别？
答：区别在 `tryAcquire` 的实现上：
- **非公平锁**（默认）：`lock()` 时先直接 CAS 尝试抢一次 state（不管队列里有没有人排队）；CAS 失败才走 acquire 流程。`tryAcquire` 里如果 state = 0 也是直接 CAS，不检查队列。这意味着新来的线程可能"插队"成功。
- **公平锁**：`tryAcquire` 里即使 state = 0，也会先调用 `hasQueuedPredecessors()` 检查队列中是否有比自己等得更久的线程。如果有，放弃抢锁，乖乖入队排队。

非公平锁吞吐量通常更高（减少了线程切换），但可能导致排队线程饥饿；公平锁保证 FIFO 顺序但开销更大。

68. AQS 的 Condition 是怎么实现等待/通知的？
答：每个 `Condition` 对象内部维护一个独立的**条件队列**（单向链表，节点 waitStatus = CONDITION）。
- **await()**：将当前线程封装为 Node 加入条件队列尾部，然后**完全释放锁**（state 置 0），接着 `LockSupport.park` 挂起。被唤醒后需要重新竞争锁（重新进入 AQS 同步队列的 `acquireQueued` 流程）。
- **signal()**：将条件队列的第一个节点从条件队列中移除，`enq` 到 AQS 的同步队列尾部，并把它的 waitStatus 从 CONDITION 改回 0，然后 unpark 唤醒对应线程（或设置前驱的 SIGNAL 状态让它在释放锁时唤醒）。

条件队列和同步队列是两个独立的链表，await/signal 是节点在两个队列之间的转移。

69. 哪些常用工具类是基于 AQS 实现的？
答：
- **ReentrantLock**：独占模式，state 记录重入次数。
- **ReentrantReadWriteLock**：state 的高 16 位记录读锁持有数，低 16 位记录写锁重入次数。读锁用共享模式，写锁用独占模式。
- **Semaphore**：共享模式，state 表示许可数，acquire 减许可，release 加许可。
- **CountDownLatch**：共享模式，state 表示计数值，`countDown` 减 1，`await` 在 state > 0 时阻塞，state = 0 时唤醒所有等待线程。
- **CyclicBarrier**：不是直接继承 AQS，而是内部用 ReentrantLock + Condition 实现，但底层仍然间接依赖 AQS。

## 十六、ThreadPoolExecutor 执行流程与参数

70. execute(task) 的完整执行流程是什么？
答：这是面试核心中的核心，很多人背错顺序。正确流程是：
1. **workerCount < corePoolSize** → 直接创建新的核心线程来执行这个任务（即使有其他核心线程是空闲的也会创建新的，直到达到 corePoolSize）。
2. **corePoolSize 已满** → 尝试将任务放入 **workQueue**（阻塞队列）。
3. **workQueue 也满了** → 如果 workerCount < maximumPoolSize，创建非核心线程来执行。
4. **maximumPoolSize 也满了** → 执行 **RejectedExecutionHandler** 拒绝策略。

**关键误区**：很多人以为是先创建到 maximumPoolSize 再放队列，这是错的！是先放队列，队列满了才创建非核心线程。这个顺序的设计意图是：队列是缓冲带，尽量用核心线程 + 队列就够了，非核心线程是最后的弹性手段。

71. 四种内置拒绝策略分别是什么？
答：
- **AbortPolicy**（默认）：直接抛出 `RejectedExecutionException`。调用方能感知到任务被拒绝，适合关键任务不能静默丢失的场景。
- **CallerRunsPolicy**：由提交任务的线程自己来执行这个任务。好处是形成天然的反压——提交方在执行任务期间不会再提交新任务，自动减速。适合不允许丢弃且需要降速的场景。
- **DiscardPolicy**：静默丢弃被拒绝的任务，不抛异常不通知。危险：任务无声无息地丢了，排查困难。
- **DiscardOldestPolicy**：丢弃队列中最旧的任务（队头），然后重试提交当前任务。适合只关心最新数据的场景（如最新状态覆盖），但也可能丢失重要任务。

72. 常见的 workQueue 类型和风险是什么？
答：
- **LinkedBlockingQueue（无界队列）**：`Executors.newFixedThreadPool` 使用。因为队列无界，任务永远不会触发创建非核心线程，maximumPoolSize 参数形同虚设。风险是队列无限增长 → OOM。生产环境几乎不应该用无界队列。
- **SynchronousQueue（无缓冲队列）**：`Executors.newCachedThreadPool` 使用。它不存储任务，每个 put 必须等一个 take。效果是来一个任务就创建一个线程（如果没有空闲线程的话）。风险是线程数不受控 → 大量线程创建耗尽资源。
- **ArrayBlockingQueue（有界队列）**：生产推荐。可以明确控制队列大小，配合合理的 maximumPoolSize 和拒绝策略，实现可预测的过载行为。
- **DelayQueue**：用于延迟任务执行（`ScheduledThreadPoolExecutor` 内部使用）。

73. 线程池参数应该怎么配？CPU 密集和 IO 密集的区别是什么？
答：
- **CPU 密集型**（计算为主，少 IO 等待）：核心线程数 ≈ CPU 核数 + 1。+1 是为了当某个线程偶尔因页缺失等短暂阻塞时，仍能充分利用 CPU。线程太多反而增加上下文切换开销。
- **IO 密集型**（大量 IO 等待，如数据库查询、HTTP 调用）：核心线程数 ≈ CPU 核数 × 2 甚至更多，取决于 IO 等待时间占比。公式参考：`线程数 = CPU 核数 × (1 + IO 等待时间 / CPU 计算时间)`。
- **实际应该压测确定**：上面只是起步参考值。真正的线上参数必须结合实际任务的 RT（响应时间）、峰值 QPS、下游依赖情况、队列积压水位来压测调优，而不是套公式。很多大厂已经做了线程池的动态参数调整能力（如美团的动态线程池框架）。

74. keepAliveTime 和 allowCoreThreadTimeOut 分别是什么？
答：
- **keepAliveTime**：当线程数超过 corePoolSize 时，多余的非核心线程如果空闲超过这个时间，就会被销毁回收。它控制的是线程池在低负载时自动收缩的速度。
- **allowCoreThreadTimeOut(true)**：默认情况下核心线程即使空闲也不会被销毁。调用这个方法后，核心线程也会受 keepAliveTime 约束，空闲超时就销毁。适合流量波动大、不想长期持有大量核心线程的场景。

75. shutdown 和 shutdownNow 有什么区别？
答：
- **shutdown()**：平缓关闭。不再接受新任务（再提交会触发拒绝策略），但已提交到队列中的任务和正在执行的任务都会被执行完毕。状态变为 SHUTDOWN。
- **shutdownNow()**：尝试立即停止。对所有正在执行的线程发送中断信号（`interrupt`），并返回队列中尚未执行的任务列表。状态变为 STOP。注意"尝试中断"不等于"一定能停"——如果任务内部没有响应中断（不检查 `Thread.interrupted()`、不处理 `InterruptedException`），线程可能不会停下来。
- 两者最终都会过渡到 TERMINATED 状态（所有线程都结束后）。可以用 `awaitTermination(timeout, unit)` 阻塞等待线程池真正终止。

76. 为什么阿里规范禁止使用 Executors 创建线程池？
答：因为 `Executors` 的工厂方法隐藏了危险的默认参数：
- `newFixedThreadPool` 和 `newSingleThreadExecutor` 使用无界的 `LinkedBlockingQueue`，队列可以无限增长导致 OOM。
- `newCachedThreadPool` 的 maximumPoolSize 是 `Integer.MAX_VALUE`，相当于线程数无上限，可以创建海量线程导致资源耗尽。
- `newScheduledThreadPool` 也是无界队列。

正确做法是手动通过 `new ThreadPoolExecutor(...)` 创建，显式指定核心线程数、最大线程数、队列类型和大小、拒绝策略，让线程池的行为完全可预测、可控。

## 十七、Java 内存模型（JMM）深度

> 说明：本文前部已有「十一、最值得练熟的 10 个追问」，本专题在文末展开 JMM 与虚拟线程等，章节序号顺延为十七。

1. JMM 到底在解决什么问题？
答：Java 内存模型（JMM）规定的是：**多线程之间，一个线程对共享变量的写入，在何种条件下对另一个线程的读可见**，以及**指令是否可以重排序**的边界。物理上，CPU 有多级缓存（L1/L2/L3），JIT 编译器与 CPU 可能为了吞吐做**指令重排**；若没有同步约束，线程 A 写入的值可能长期对线程 B 不可见，或 B 看到「先发布后初始化」的中间态。JMM 通过 **happens-before** 关系把程序员写的 `volatile`、`synchronized`、`final` 等与底层内存屏障、缓存一致性协议对应起来，给出可依赖的可见性与有序性保证。

2. happens-before 的 8 条规则分别是什么？
答：
- **程序顺序规则**：同一线程内，按程序次序在前面的操作 happens-before 后面的操作（仍允许在满足 as-if-serial 的前提下重排，但与监视器/volatile 等规则结合后约束可见性）。
- **监视器锁规则**：对锁的 **unlock** happens-before 随后对**同一把锁**的 **lock**。
- **volatile 规则**：对 volatile 字段的 **写** happens-before 任意线程后续对该字段的 **读**。
- **线程启动规则**：`Thread.start()` happens-before 新线程内执行的任何动作。
- **线程终止规则**：线程中的任意动作 happens-before 其他线程检测到该线程已终止（如 `Thread.join()` 返回、或 `isAlive()` 为 false）。
- **中断规则**：对线程 `interrupt()` 的调用 happens-before 被中断线程检测到中断（如抛出 `InterruptedException` 或 `interrupted` 状态被观测）。
- **终结器规则**：对象构造函数的结束 happens-before `finalize` 方法的开始（实践中应避免依赖 finalize）。
- **传递性**：若 A happens-before B，且 B happens-before C，则 A happens-before C。

3. volatile 底层是怎么实现的？
答：语义上，volatile 读写禁止特定重排序并建立可见性。HotSpot 在 volatile **写**前后、**读**后会插入内存屏障（具体组合与平台相关）：常见模式是在写后加 **StoreLoad** 以保证与后续读的全序关系。在 **x86** 上 TSO 较强，许多场景下读侧屏障较「轻」；volatile 写常被编译为带 **`lock`** 前缀的指令（例如对栈上占位做 `lock addl $0, (%rsp)` 这类形式），`lock` 会触发**缓存行锁定/失效**，把写刷到对其他核心可见，并充当**全屏障**效果。不同 CPU 与 JDK 版本实现细节可能不同，面试答「内存屏障 + 缓存一致性（如 MESI）+ 禁止部分重排」即可落到本质上。

4. 为什么 DCL（双重检查锁）必须用 volatile？
答：单例懒加载的典型写法里，`instance = new Singleton()` 在字节码层面至少包含：分配内存、执行构造器初始化字段、把引用赋给静态变量。在**没有 volatile** 时，编译器/CPU 可能把「引用赋值」重排到「构造完成」之前（即其他线程可能看到一个**非 null 但未完全初始化**的对象）。给 `instance` 加 **volatile** 后，对该字段的写受 JMM 约束，配合屏障禁止这种**发布顺序**上的危险重排，从而保证其他线程在看到非 null 引用时，能观测到构造器已执行产生的 happens-before 关系（常与「对象初始化与引用赋值」的禁止重排一起讨论）。

5. Java 虚拟线程（Virtual Threads）是什么？
答：虚拟线程在 **Java 19** 预览、**Java 21** 正式落地，是 JDK 提供的**极轻量**线程抽象。传统**平台线程**大体 1:1 映射内核线程，默认栈较大（量级可达 MB），海量并发时创建与调度成本高。虚拟线程是 **N:M** 调度：大量虚拟线程运行在少量平台线程（**载体线程 Carrier**）上；当虚拟线程在 **阻塞 IO**、`BlockingQueue.take` 等操作上阻塞时，运行时可将其 **unmount**，释放载体去跑其他虚拟线程，IO 就绪后再 **mount** 回来。优势：**百万级**并发连接、同步风格代码写高并发 IO。注意：**`synchronized` 在载体上 pin** 可能占用载体（JDK 21+ 持续改进中，仍建议热点路径用 `ReentrantLock`）；**CPU 密集**任务不会 magically 变快，反而应与池化平台线程配合。创建方式示例：`Thread.ofVirtual().start(runnable)`、`Executors.newVirtualThreadPerTaskExecutor()`。

6. CompletableFuture 的组合式异步编程？
答：`CompletableFuture`（Java 8）在 `Future` 之上提供**非阻塞组合**：`thenApply` / `thenAccept` / `thenRun` 在当前执行线程或同阶段线程上链接下一步；`thenApplyAsync` 等可显式丢到**线程池**。`thenCombine` / `thenAcceptBoth` 合并两个阶段的输出；`allOf` / `anyOf` 聚合多个 Future；`exceptionally` / `handle` / `whenComplete` 统一处理异常与清理。默认异步阶段常用 **ForkJoinPool.commonPool()**，生产代码建议**传入自定义 Executor** 以便隔离资源与限流。在 AI 网关、编排层常见模式：并行调用多个下游或模型端点，再 `thenCombine` 做融合与后处理，最后用 `exceptionally` 降级或返回部分结果。
