# 操作系统与并发模型

## 使用方式

很多后端候选人会学 `Java` 并发、线程池、锁和 `JVM`，但一被继续追问进程、线程、上下文切换、内存、IO、多路复用，就容易露底。

这篇的目标就是把“操作系统基础”和“后端运行时问题”真正串起来。

你读这篇时要始终带着一个问题：

“为什么一个后端服务会慢、会卡、会抖、会打满线程，却不只是代码逻辑问题？”

---

## 一、为什么后端必须懂操作系统

1. 为什么大厂后端岗位会问操作系统？
答：因为服务最终运行在操作系统之上。线程怎么调度、内存怎么分配、文件和网络怎么读写、上下文切换有什么成本，这些都会直接影响服务的性能、稳定性和排障能力。不会 OS，你对很多线上现象只能停留在表面。

2. 为什么 `JVM` 并发题和操作系统题总是连着问？
答：因为 `Java` 里的线程、锁、阻塞、唤醒、线程池、本地 IO，本质上最后都要映射到操作系统线程、调度和系统调用。你如果只会答语言层抽象，不知道底层代价，面试官很容易继续追问把你问穿。

---

## 二、进程、线程和协程

3. 进程和线程最本质的区别是什么？
答：进程更像资源隔离和运行边界，线程更像进程内部的执行单元。进程之间隔离更强，线程之间共享进程内存和资源。后端工程里，服务通常以进程运行，而请求处理、任务执行和线程池调度主要发生在线程层。

4. 为什么线程不是越多越好？
答：因为线程多并不等于有效吞吐高。线程越多，栈空间、调度成本、上下文切换和竞争都会增加；如果线程大多在等待 IO 或锁，继续加线程只会让系统更乱。成熟后端会根据任务类型和资源模型设计线程数，而不是盲目堆线程。

5. 协程为什么会成为后端高频话题？
答：因为它在很多语言和框架里提供了更轻量的并发抽象，例如 `Go goroutine`、`Kotlin coroutine`、用户态调度模型等。它的价值不是“完全替代线程”，而是降低高并发场景下的调度和资源成本。

---

## 三、调度、上下文切换与性能

6. 为什么上下文切换有成本？
答：因为每次线程切换都需要保存和恢复执行现场，可能还会带来 CPU cache 失效和调度开销。如果线程过多、争抢激烈、频繁阻塞唤醒，上下文切换成本就会非常明显，最终表现成吞吐下降和延迟抖动。

7. 为什么后端排障时经常会提“上下文切换过多”？
答：因为这往往意味着线程模型不合理、锁竞争严重、任务切得太碎、或者系统处于高阻塞高争抢状态。它不是一个抽象术语，而是很多“CPU 不低但吞吐不升”的真实根因之一。

---

## 四、内存、分页与缓存意识

8. 为什么操作系统里的内存概念对后端很重要？
答：因为服务的堆内存、栈内存、直接内存、文件缓存、页缓存都会和系统整体资源竞争。你如果只看语言层对象，而不知道操作系统层的内存压力，就很难解释 OOM、抖动、缺页、文件 IO 异常这些问题。

9. 为什么页缓存会影响后端 IO 表现？
答：因为很多文件读写、日志写入、数据库访问并不是每次都直接落磁盘，操作系统会通过页缓存做缓冲和优化。这也是为什么同样的读写逻辑，在冷热数据、缓存命中不同的时候，延迟会差很多。

---

## 五、IO、多路复用与网络服务

10. 为什么后端必须懂阻塞 IO 和非阻塞 IO？
答：因为服务处理网络请求、数据库连接、文件读写时，线程是否被阻塞，直接影响吞吐和资源占用。你至少要知道：阻塞模型简单但线程成本高；非阻塞和事件驱动更适合高并发网络服务，但复杂度更高。

11. 什么是 IO 多路复用？
答：可以简单理解成用更少线程同时观察多个连接或 IO 事件，等哪个准备好了再处理。它的价值在于避免“一个连接占一个线程”的高成本模型，这也是很多高并发网络框架和网关底层的重要能力。

12. 为什么 `Netty`、网关、Redis、Nginx 这些系统常和多路复用绑定出现？
答：因为它们都需要处理大量网络连接。如果还是简单阻塞线程模型，线程资源和上下文切换很快会成为瓶颈。多路复用正是支撑这类系统高并发能力的底层关键之一。

---

## 六、锁、等待与并发问题

13. 为什么锁竞争会从语言层问题演变成系统层问题？
答：因为锁竞争最终会表现为线程阻塞、CPU 空转、上下文切换增多和吞吐下降。一把锁如果卡在高频路径上，影响往往不是一段代码，而是整个服务的时延和资源利用率。

14. 为什么“很多线程在跑”不代表系统在高效工作？
答：因为这些线程可能都在等锁、等 IO、等连接、等下游返回。真正重要的是有效工作比例，而不是线程总数本身。后端工程里经常会遇到线程很多但吞吐很差的情况，本质上就是资源在等待而不是在产出。

## 十、虚拟内存与页表

17. 为什么需要虚拟内存？
答：三个核心原因。第一，**进程隔离**：每个进程拥有独立的虚拟地址空间（32 位系统 4GB、64 位系统 256TB），进程 A 无法直接访问进程 B 的内存，一个进程崩溃不会破坏其他进程。第二，**按需分配**：物理内存不需要在进程启动时全部分配，操作系统可以在进程真正访问某个页时才分配物理页框，未使用的页可以被交换到磁盘 swap 区。第三，**内存保护与共享**：页表中每个条目包含读/写/执行权限位，操作系统可以精确控制每个内存区域的访问权限；多个进程可以共享同一个物理页框（如共享库 libc.so 只需在物理内存中保留一份）。

18. 页(Page)和页框(Frame)是什么关系？
答：虚拟内存被切分成固定大小的**页(Page)**，物理内存被切分成同样大小的**页框(Frame)**，Linux 默认页大小 4KB。页表(Page Table)记录的就是虚拟页号(VPN) → 物理页框号(PFN)的映射关系。一个虚拟地址被拆分为两部分：高位是虚拟页号(用来查页表)，低位是页内偏移(直接拼接到物理页框号后面得到物理地址)。

19. 为什么需要多级页表？
答：以 64 位系统为例，如果用单级页表，假设虚拟地址空间 48 位有效（256TB），每个页 4KB(12 位偏移)，则需要 2^36 个页表项。每个页表项 8 字节，单级页表需要 512GB，这显然不可能。所以采用多级页表：只有在实际使用到的虚拟地址范围才分配对应级别的页表。Linux x86_64 使用 **4 级页表**：

- **PGD**(Page Global Directory)：第 1 级，占虚拟地址的 [47:39] 共 9 位，512 个条目
- **PUD**(Page Upper Directory)：第 2 级，占 [38:30] 共 9 位
- **PMD**(Page Middle Directory)：第 3 级，占 [29:21] 共 9 位
- **PTE**(Page Table Entry)：第 4 级，占 [20:12] 共 9 位，最终指向 4KB 物理页框

这样一个进程如果只用了很少的虚拟地址空间，大多数中间级页表不需要分配，大幅节省内存。Linux 5.x 起引入了第 5 级页表(PGD→P4D→PUD→PMD→PTE)，支持 57 位虚拟地址(128PB)。

20. TLB 是什么？为什么它对性能至关重要？
答：**TLB(Translation Lookaside Buffer)** 是 CPU 内部的一块高速缓存，专门缓存最近使用过的页表条目(虚拟页号 → 物理页框号)。如果 TLB 命中，虚拟地址翻译只需要 **1 个时钟周期(约 0.5-1ns)**，和访问 L1 缓存的速度相当。如果 TLB 未命中，CPU 需要遍历 4 级页表（称为 page table walk），访问 4 次内存，耗时约 **100-200ns**（如果页表本身不在 cache 中）。

TLB 条目数量有限：L1 dTLB 通常 **64-128 条**（覆盖 256KB-512KB 的 4KB 页），L2 TLB 通常 **512-2048 条**。进程上下文切换时，由于不同进程的页表不同，TLB 中的条目可能需要全部或部分刷新（现代 CPU 通过 PCID/ASID 标记可以避免全量刷新，但切换后 TLB 中可能没有新进程的条目）。

21. 缺页中断(Page Fault)分哪几种？
答：缺页中断发生在 CPU 翻译虚拟地址时发现页表中没有有效映射，此时触发异常，由内核的缺页处理程序介入。分为两种：

- **硬缺页(Major Page Fault)**：页对应的数据不在物理内存中(已被换出到磁盘 swap 或者是 mmap 的文件还未读入)。内核需要发起**磁盘 I/O** 把数据读入一个空闲页框，然后更新页表建立映射。成本：一次磁盘 I/O 约 **4-10ms(HDD)** 或 **50-200μs(SSD)**，代价非常大。
- **软缺页(Minor Page Fault)**：数据已经在物理内存中，但当前进程的页表还没有建立映射。常见场景：①新分配的匿名页(malloc 分配后首次访问)需要映射到零页；②COW(Copy-On-Write)写入时复制页面后更新页表指向新副本；③共享库页已被其他进程加载到物理内存。内核只需更新页表条目，不需要磁盘 I/O，成本约 **1-10μs**。

可以用 `ps -o min_flt,maj_flt -p <pid>` 或 `/proc/<pid>/stat` 查看进程的软/硬缺页统计。

22. 大页(Huge Pages)解决什么问题？
答：默认 4KB 页意味着 1GB 内存需要 262144 个页表条目。TLB 条目有限（假设 1024 条），那么 TLB 最多覆盖 4MB，其余访问全要走页表，导致大量 **TLB miss**。大页将页大小提升到 **2MB** 或 **1GB**：

- 2MB 大页：1GB 内存只需要 512 个页表条目，TLB 1024 条可以覆盖 2GB
- 1GB 大页：1GB 内存只需要 1 个页表条目

Linux 支持两种方式：
- **HugeTLB(静态大页)**：通过 `hugetlbfs` 预先分配，需要手动配置。JVM 可通过 `-XX:+UseLargePages` 使用。
- **THP(Transparent Huge Pages)**：内核自动将连续的 4KB 页合并为 2MB 大页，对应用透明。但 THP 有个问题：`khugepaged` 后台合并线程会占用 CPU，而且在 Redis 等使用 fork() 做 RDB 持久化的场景下，COW 的粒度从 4KB 变成 2MB，一次写入导致 2MB 的页面复制，**大幅增加 fork 后的内存开销和延迟**。所以 Redis 官方建议关闭 THP：`echo never > /sys/kernel/mm/transparent_hugepage/enabled`。

23. 虚拟内存和后端系统有什么直接关联？
答：几乎所有后端核心组件都依赖虚拟内存机制：
- **JVM 堆内存**：JVM 通过 mmap 或 brk 向操作系统申请虚拟内存，堆内存的物理分配是按需的(首次访问触发软缺页)
- **mmap**：数据库(如 MongoDB 早期引擎)和消息队列(如 RocketMQ 的 CommitLog)通过 mmap 将文件映射到虚拟地址空间，实现零拷贝读写
- **Redis fork COW**：Redis 做 RDB/AOF-rewrite 时 fork 子进程，父子共享物理页，写入时触发 COW(软缺页+页面复制)，这就是虚拟内存的页级共享和写时复制
- **数据库 Buffer Pool**：MySQL InnoDB 的 Buffer Pool 本质上是一大块虚拟内存中的页缓存，和操作系统的页缓存协同工作

## 十一、进程调度

24. Linux CFS 调度器的核心原理是什么？
答：**CFS(Completely Fair Scheduler)** 是 Linux 2.6.23 起的默认普通进程调度器，核心思想是**完全公平**——让所有可运行进程获得的 CPU 时间尽量均等。具体机制：

- 每个进程维护一个 **vruntime(虚拟运行时间)**，表示该进程已经消耗的"标准化"CPU 时间
- 调度器总是选择 **vruntime 最小** 的进程投入运行（即获得 CPU 时间最少的那个进程优先运行）
- 使用**红黑树**组织所有可运行进程，树的最左节点就是 vruntime 最小的进程，选择操作 **O(log N)**
- 进程每运行一个时间片，其 vruntime 增加 `实际运行时间 × (NICE_0_LOAD / 该进程权重)`

25. nice 值怎么影响调度？
答：nice 值范围 **-20 到 +19**，默认 0。nice 值越低，权重越大，vruntime 增长越慢，从而获得更多 CPU 时间。具体映射关系定义在内核的 `sched_prio_to_weight[]` 数组中：

- nice 0 → 权重 1024
- nice -20 → 权重 88761（约是 nice 0 的 86.7 倍）
- nice +19 → 权重 15（约是 nice 0 的 1/68）

相邻 nice 值之间的权重比约为 **1.25:1**，即 nice 值每降低 1，获得的 CPU 时间约增加 10%。

CFS 还有一个参数 **sched_min_granularity**（默认约 0.75ms-4ms），防止时间片过短导致频繁上下文切换。即使有优先级更高的进程就绪，当前进程至少运行这么久才会被抢占。

26. 实时调度策略和普通调度有什么区别？
答：Linux 内核支持两类实时调度策略，**优先级永远高于 CFS 普通进程**：

- **SCHED_FIFO(先进先出)**：同优先级内，先获得 CPU 的进程一直运行到它主动让出(yield/sleep/阻塞)，不会被同优先级进程抢占
- **SCHED_RR(轮转)**：同优先级内按时间片轮转，默认时间片 100ms

实时优先级范围 1-99（数值越大优先级越高），普通 CFS 进程优先级为 0。所以任何实时进程都会抢占普通进程。在后端场景中，一般不使用实时调度（可能导致普通进程饿死），但某些低延迟中间件或网络处理线程可能会设置 SCHED_FIFO。

27. 上下文切换的具体成本是什么？
答：上下文切换分为两种：线程切换(同一进程内)和进程切换(不同进程间)。具体成本拆解：

1. **保存/恢复寄存器**：包括通用寄存器、程序计数器、栈指针、浮点寄存器等。这部分成本相对固定，约 **几十到几百纳秒**。
2. **TLB 刷新**：进程切换时页表基地址(CR3 寄存器)改变，如果不支持 PCID(Process Context ID)，TLB 需要全量刷新。即使支持 PCID(Linux 4.14+ 默认开启)，TLB 中新进程的条目可能很少，切换后会经历大量 TLB miss。线程切换(同一进程)不需要切换页表，TLB 不受影响。
3. **CPU 缓存失效**：这是最大的隐性成本。切换后新进程/线程的工作集(working set)不在 L1/L2/L3 缓存中，需要从主存重新加载。L1 cache miss 成本约 3-10ns，L2 miss 约 10-40ns，L3 miss 约 40-100ns+。如果工作集很大，切换后的 cache warm-up 开销可达 **数十微秒**。
4. **内核态操作**：调度器选择下一个进程、更新运行队列等内核操作。

总结：直接成本约 **1-10 微秒**，但间接成本(cache/TLB miss 导致的性能下降)可能持续更久，尤其在缓存敏感的高吞吐场景。

可以通过 `vmstat` 的 `cs` 列或 `pidstat -w` 观测系统级和进程级的上下文切换次数。正常服务器每秒几千到几万次是合理的，如果到了几十万次/秒就需要排查。

28. 进程调度和后端性能有什么直接关联？
答：
- **线程过多导致频繁调度**：如果线程池核心线程数远超 CPU 核数，大量线程竞争 CPU，导致频繁上下文切换和缓存失效，吞吐反而下降。经验法则：CPU 密集型任务线程数 ≈ CPU 核数；IO 密集型任务线程数 ≈ CPU 核数 × (1 + IO 等待时间/CPU 计算时间)。
- **GC 线程的调度优先级**：JVM 的 GC 线程(如 G1 的并发标记线程)和应用线程竞争 CPU，如果 CPU 核数少而 GC 线程多，GC 会抢占应用线程的 CPU 时间，导致延迟毛刺。
- **容器环境的 CFS 带宽限制**：Kubernetes 中通过 cgroup 的 CFS 带宽控制(`cpu.cfs_period_us` 和 `cpu.cfs_quota_us`)限制容器 CPU 使用量。如果 Java 应用线程数超过 CPU limit，线程会被节流(throttled)，表现为延迟突然升高。

## 十二、CPU 缓存与伪共享

29. CPU 缓存的层次结构是怎样的？
答：现代 CPU 有三级缓存，越靠近 CPU 核心越快越小：

| 层级 | 典型容量 | 访问延迟 | 说明 |
|------|---------|---------|------|
| **L1 缓存** | 32-64KB(每核) | ~1ns(约 4 个时钟周期) | 分为 L1i(指令)和 L1d(数据)，每核私有 |
| **L2 缓存** | 256KB-1MB(每核) | ~3-10ns(约 10-30 周期) | 每核私有(部分架构共享) |
| **L3 缓存** | 数 MB-数十 MB | ~10-40ns(约 30-100 周期) | 所有核共享(Last Level Cache) |
| **主存(DRAM)** | 数 GB-数百 GB | ~60-100ns(约 200-300 周期) | 全局共享 |

从 L3 到主存的延迟跳跃约 **3-10 倍**，这就是为什么缓存命中率对性能影响巨大。

30. 什么是缓存行(Cache Line)？为什么它是理解并发性能的关键单位？
答：**缓存行(Cache Line)** 是 CPU 缓存和主存之间数据传输的最小单位，通常 **64 字节**（Intel/AMD 主流架构）。当 CPU 读取一个 4 字节的 int 变量时，会把包含这个变量的整个 64 字节缓存行都加载到缓存中。这意味着：

- 连续访问数组元素(空间局部性)非常高效，因为同一缓存行内的后续元素不需要再次加载
- 但也意味着，如果两个逻辑上无关的变量恰好在同一个 64 字节范围内，它们会共享同一个缓存行，带来潜在的性能问题

31. 什么是伪共享(False Sharing)？怎么解决？
答：**伪共享**是多核 CPU 并发编程中一个经典性能陷阱：

**场景**：线程 A 运行在核 0 上频繁修改变量 `x`，线程 B 运行在核 1 上频繁修改变量 `y`。`x` 和 `y` 是两个不同的变量，逻辑上完全独立，但它们在内存中相邻，**落在同一个 64 字节缓存行**中。

**后果**：
1. 核 0 修改 `x` → 该缓存行在核 0 的缓存中变为 Modified 状态
2. 缓存一致性协议要求核 1 中对应的缓存行失效(Invalid)
3. 核 1 要修改 `y` → 发现缓存行已失效 → 必须从核 0(或 L3)重新加载这个缓存行
4. 核 1 修改 `y` → 又导致核 0 的缓存行失效
5. 如此往复，缓存行在两个核之间**"乒乓"传递**，每次传递成本约 **40-100ns**（走 L3 或核间总线）

性能影响：伪共享可导致性能下降 **2-10 倍** 甚至更多。

**解决方案**：
- **Java 8+ `@sun.misc.Contended` 注解**：在字段前后插入 128 字节的 padding，确保该字段独占缓存行。需要 JVM 参数 `-XX:-RestrictContended` 才能在非 JDK 内部类中生效。
- **手动填充**：在变量前后声明无用的 long 字段，使变量独占 64 字节对齐边界。

**JDK 中的实际案例**：
- `ConcurrentHashMap` 的 `CounterCell[]` 类标注了 `@sun.misc.Contended`，每个 Cell 独占一个缓存行，多线程 addCount 时避免伪共享
- `Thread` 类中的 `threadLocalRandomSeed` 等字段也用 `@Contended` 隔离
- `LongAdder/DoubleAdder` 继承自 `Striped64`，其 `Cell` 类同样使用 `@Contended`

32. MESI 协议是什么？
答：**MESI** 是主流多核 CPU 使用的**缓存一致性协议**，确保同一内存地址在多个核的缓存中保持一致。每个缓存行有四种状态：

- **M(Modified)**：该缓存行只在当前核的缓存中，且已被修改(与主存不一致)。其他核读取这个地址时，当前核必须先写回。
- **E(Exclusive)**：该缓存行只在当前核的缓存中，且与主存一致。可以直接修改(变成 M)，不需要通知其他核。
- **S(Shared)**：该缓存行在多个核的缓存中，且与主存一致。修改前需要先发送 Invalidate 消息让其他核失效。
- **I(Invalid)**：该缓存行无效，访问时需要从其他核或主存加载。

实际 CPU 还使用了 MESIF(Intel)或 MOESI(AMD)等变种来优化核间传输效率。

33. CPU 缓存和后端并发有什么直接关联？
答：
- **volatile 的可见性**：Java 的 volatile 写入会插入 StoreLoad 屏障，底层触发缓存行刷新(从 M 状态写回并通知其他核失效)，保证其他线程能看到最新值。代价：一次 volatile 写约 **20-100ns**（涉及核间通信），远高于普通写入的 ~1ns。
- **CAS 操作**：`compareAndSwap` 在 x86 上编译为 `LOCK CMPXCHG` 指令，LOCK 前缀会锁定缓存行(独占该缓存行)，成本约 **10-50ns**。高竞争下多个核争抢同一缓存行，成本更高。
- **LongAdder vs AtomicLong**：AtomicLong 的所有线程 CAS 同一个 value 字段(同一缓存行)，高并发下缓存行频繁在核间弹跳。LongAdder 把计数分散到 Cell[] 数组的多个 Cell 上(不同缓存行)，每个线程 CAS 自己的 Cell，大幅减少缓存行竞争。sum() 时再汇总所有 Cell。代价是 sum() 不是精确快照。

## 十三、零拷贝(Zero Copy)

34. 传统文件传输到网络要经过几次拷贝？
答：以 `read(file_fd, buf, len)` + `write(socket_fd, buf, len)` 为例，数据经过 **4 次拷贝和 4 次用户态/内核态切换**：

```
磁盘 --[DMA拷贝]--> 内核页缓存(Page Cache)
内核页缓存 --[CPU拷贝]--> 用户态缓冲区(buf)     ← read() 返回，2次上下文切换
用户态缓冲区 --[CPU拷贝]--> Socket发送缓冲区      ← write() 调用，2次上下文切换
Socket缓冲区 --[DMA拷贝]--> 网卡(NIC)
```

其中 2 次 DMA 拷贝由硬件完成(CPU 不参与)，2 次 CPU 拷贝是纯粹的浪费——数据在用户态绕了一圈又回到内核态。4 次用户态/内核态切换(read 进入内核→返回用户态，write 进入内核→返回用户态)各有几百纳秒到微秒的开销。

35. mmap + write 怎么优化？
答：用 `mmap(file_fd)` 将文件映射到用户空间，然后 `write(socket_fd, mmap_addr, len)`：

```
磁盘 --[DMA拷贝]--> 内核页缓存(Page Cache)
(mmap 使用户空间直接映射内核页缓存，不拷贝到用户态)
内核页缓存 --[CPU拷贝]--> Socket发送缓冲区      ← write()
Socket缓冲区 --[DMA拷贝]--> 网卡
```

**3 次拷贝 + 4 次上下文切换**。省掉了一次 CPU 拷贝(内核→用户态)，但 write() 仍然需要用户态/内核态切换。**RocketMQ 的 CommitLog** 就使用 mmap 方式进行消息的读写。

36. sendfile 怎么进一步优化？
答：`sendfile(socket_fd, file_fd, offset, count)` 在 Linux 2.1 引入，数据完全在内核空间传输：

```
磁盘 --[DMA拷贝]--> 内核页缓存(Page Cache)
内核页缓存 --[CPU拷贝]--> Socket发送缓冲区
Socket缓冲区 --[DMA拷贝]--> 网卡
```

**3 次拷贝 + 2 次上下文切换**(只有 sendfile 系统调用的进出)。

如果网卡支持 **SG-DMA(Scatter-Gather DMA)**（Linux 2.4+，现代网卡基本都支持），可以进一步优化：

```
磁盘 --[DMA拷贝]--> 内核页缓存(Page Cache)
(只传输描述符[内存地址+长度]给Socket缓冲区，不拷贝数据)
内核页缓存 --[SG-DMA拷贝]--> 网卡(DMA直接从页缓存读取)
```

**2 次 DMA 拷贝 + 0 次 CPU 拷贝 + 2 次上下文切换**。这才是真正意义上的"零拷贝"——CPU 完全不参与数据拷贝。

37. splice 是什么？
答：`splice(fd_in, off_in, fd_out, off_out, len, flags)` 在 Linux 2.6.17 引入，可以在两个文件描述符之间通过内核管道(pipe)传输数据，不经过用户态。和 sendfile 不同，splice 更通用：sendfile 要求源 fd 必须是文件(支持 mmap)，而 splice 可以在任意两个 fd 之间传输(如 socket→socket、pipe→socket)。Nginx 内部在某些转发场景使用 splice。

38. 零拷贝和后端系统有什么直接关联？
答：
- **Kafka**：Consumer 拉取消息时，Broker 使用 `sendfile` 将消息日志文件直接发送到网络 Socket。这是 Kafka 能达到 **GB/s 级别吞吐**的关键之一(配合顺序写入 + 页缓存 + 分区并行)。Java 代码层面是 `FileChannel.transferTo()` → 底层调用 `sendfile`。
- **Nginx**：发送静态文件时默认使用 `sendfile on;`，直接在内核空间完成文件→网络传输。
- **RocketMQ**：使用 `mmap` 方式(通过 `MappedByteBuffer`)读写 CommitLog 和 ConsumeQueue，避免了读文件时的用户态拷贝。
- **Netty**：`DefaultFileRegion` 封装了 `FileChannel.transferTo()`(底层 sendfile)；`CompositeByteBuf` 通过组合多个 buffer 的引用避免合并拷贝。

## 十四、用户态与内核态

39. 为什么要区分用户态和内核态？
答：现代 CPU 提供多个特权级别（x86 有 Ring 0-3，Linux 只用 Ring 0 和 Ring 3）。**内核态(Ring 0)** 可以执行所有 CPU 指令、访问所有内存和硬件；**用户态(Ring 3)** 只能访问自己的虚拟地址空间，不能直接操作硬件或访问内核内存。

这种隔离的目的：
- **保护内核**：用户程序的 bug 不会直接破坏内核数据结构
- **硬件访问控制**：只有内核能操作磁盘、网卡、内存管理单元(MMU)等硬件
- **进程隔离**：用户态进程无法直接读写其他进程的内存

40. 系统调用(System Call)的过程是怎样的？
答：用户程序需要操作系统服务时(如读文件、发网络包、分配内存)，通过**系统调用**请求内核代为执行。以 Linux x86_64 的 `read()` 为例：

1. 用户态代码调用 glibc 的 `read()` 包装函数
2. 将系统调用号(read = 0)放入 `rax` 寄存器，参数放入 `rdi/rsi/rdx` 等寄存器
3. 执行 `syscall` 指令 → CPU 切换到 Ring 0，跳转到内核的系统调用入口(`entry_SYSCALL_64`)
4. 内核保存用户态寄存器到内核栈，根据 `rax` 中的系统调用号查 `sys_call_table`，调用 `ksys_read()`
5. 内核完成操作(可能涉及页缓存、磁盘 I/O、调度等)
6. 将返回值放入 `rax`，执行 `sysret` 指令 → CPU 切换回 Ring 3，恢复用户态执行

41. 用户态/内核态切换的成本有多大？
答：一次系统调用的切换成本包括：
- `syscall/sysret` 指令本身：约 **50-100ns**
- 保存/恢复寄存器：约 **几十 ns**
- 内核中的实际逻辑执行：视操作复杂度而定
- **Meltdown/Spectre 补丁(KPTI)** 后：内核页表和用户页表分离，每次进入内核需要切换 CR3(页表基地址)，额外增加约 **100-400ns**（TLB 刷新）

保守估计：一次简单的系统调用(如 `getpid()`)约 **200-500ns**；涉及 I/O 的系统调用(如 `read()`)则取决于数据是否在页缓存中。

42. 为什么后端要尽量减少系统调用？
答：因为每次系统调用都有用户态/内核态切换的固定成本。减少系统调用的常见手段：

- **writev()**：将多个不连续的缓冲区一次性写入，避免多次 write() 系统调用
- **mmap**：将文件映射到用户空间后可以像访问内存一样读写，避免反复 read/write
- **epoll**：一次系统调用等待多个 fd 上的事件，而非每个 fd 单独阻塞等待
- **io_uring(Linux 5.1+)**：通过共享内存的提交队列(SQ)和完成队列(CQ)提交/获取 I/O 请求，大幅减少系统调用次数。极端情况下可以在用户态提交 I/O 请求而 **完全不进入内核态**。
- **缓冲区合并**：应用层攒够一批数据再写入(如日志框架的异步批量刷盘)

43. 用户态/内核态和后端框架有什么直接关联？
答：
- **BIO vs NIO**：BIO 模型中每个连接一个线程，线程阻塞在 `read()` 系统调用上(等待内核通知数据就绪+拷贝到用户态)。如果有 10000 个连接，就需要 10000 个线程各自阻塞在 read()——大量线程 + 大量系统调用。NIO 用 **epoll(I/O 多路复用)**，一个线程调用一次 `epoll_wait()` 就能知道哪些连接有数据就绪，然后只对就绪的连接调用 `read()`，大幅减少线程数和系统调用次数。
- **Netty 的 EventLoop**：一个 EventLoop 线程通过 `Selector`(底层 epoll) 管理数千个 Channel，每次循环调用一次 `epoll_wait()` 获取所有就绪事件，然后批量处理。

## 十五、文件描述符与五种 I/O 模型

44. 什么是文件描述符(fd)？
答：在 Linux 中，**一切皆文件**。文件描述符(file descriptor)是进程访问 I/O 资源的句柄，本质是一个**非负整数**，指向进程内核空间中 **文件描述符表** 的索引。每个条目指向一个内核的 **文件表项(struct file)**，其中包含文件偏移量、访问模式、指向 inode 的指针等。

每个进程启动时自动获得三个标准 fd：
- fd 0：标准输入(stdin)
- fd 1：标准输出(stdout)
- fd 2：标准错误(stderr)

之后通过 `open()/socket()/pipe()/epoll_create()` 等系统调用打开新的 fd，编号从 3 开始递增。

45. fd 数量有什么限制？
答：三层限制：

- **进程级软限制(soft limit)**：默认 **1024**，可以通过 `ulimit -n 65535` 调大(当前 shell 及其子进程生效)
- **进程级硬限制(hard limit)**：非 root 用户不能超过硬限制，通过 `ulimit -Hn` 查看，修改需要编辑 `/etc/security/limits.conf`
- **系统级限制**：`/proc/sys/fs/file-max`，表示整个系统所有进程能打开的 fd 总数上限。通常几十万到几百万，和内存大小相关

后端服务(如 Netty 服务器)处理大量连接时，每个 TCP 连接占用一个 fd。如果不调大 fd 限制，连接数超过 1024 时会报 `Too many open files` 错误。生产环境通常设置为 **65535** 或更高。

46. fd 泄漏会导致什么问题？
答：如果代码打开了 fd(文件、Socket、数据库连接等)但未正确关闭(如异常路径中没有 `finally/try-with-resources` 关闭)，fd 会持续累积。当进程的 fd 数量达到限制时：
- 无法建立新的网络连接(accept 失败)
- 无法打开新文件(open 返回 -1)
- 数据库连接池无法创建新连接

排查手段：`ls -l /proc/<pid>/fd | wc -l` 查看进程 fd 数量；`lsof -p <pid>` 查看具体打开了哪些文件和 Socket。

47. Unix 的五种 I/O 模型分别是什么？
答：这五种模型来自 Richard Stevens 的《Unix Network Programming》，以网络 Socket 的 `read` 为例，I/O 操作分两个阶段：①等待数据就绪(数据从网卡到内核缓冲区)；②数据拷贝(从内核缓冲区到用户态缓冲区)。

**① 阻塞 I/O(Blocking I/O)**：
调用 `read()` 后线程阻塞，等待阶段①和阶段②都完成后才返回。一个线程只能等一个 fd，C10K 场景下需要大量线程。

**② 非阻塞 I/O(Non-blocking I/O)**：
将 fd 设为 O_NONBLOCK，调用 `read()` 时如果数据未就绪，立即返回 `EAGAIN` 错误(不阻塞)。应用需要**轮询**(busy polling)反复调用 read 检查是否就绪。数据就绪后 read 仍然阻塞在阶段②(数据拷贝)。轮询浪费 CPU，实际很少单独使用。

**③ I/O 多路复用(select/poll/epoll)**：
一个线程通过 `select/poll/epoll_wait` 同时监控**多个 fd**，阻塞等待任意一个 fd 就绪。有 fd 就绪后，再调用 `read()` 读取数据(阶段②仍是阻塞的)。

三者的区别：
- **select**：fd 集合用 `fd_set` 位图表示，最大 1024 个 fd(FD_SETSIZE)；每次调用需要把 fd_set 从用户态拷贝到内核态；内核遍历所有 fd 检查状态，**O(N)**
- **poll**：用 `pollfd` 数组替代位图，没有 1024 限制；但仍然是线性遍历，**O(N)**
- **epoll**：Linux 2.6+ 引入。`epoll_create()` 创建内核事件表(红黑树)，`epoll_ctl()` 添加/删除/修改感兴趣的 fd，`epoll_wait()` 只返回就绪的 fd 列表。通过回调机制(fd 就绪时内核将其加入就绪链表)，**O(1)** 获取就绪事件(不需要遍历所有 fd)。支持 **ET(边缘触发)** 和 **LT(水平触发)** 两种模式。epoll 是 Linux 高并发服务器(Netty/Nginx/Redis)的基础。

**④ 信号驱动 I/O(SIGIO)**：
注册 SIGIO 信号处理函数，数据就绪时内核发送 SIGIO 信号通知进程。进程在信号处理函数中调用 read。实际很少使用，因为信号处理复杂且不可靠。

**⑤ 异步 I/O(AIO)**：
调用 `aio_read()` 发起读请求后**立即返回**，阶段①和阶段②都由内核完成。数据拷贝到用户态缓冲区后，内核通过信号或回调通知进程。这是唯一真正做到**全程非阻塞**的模型。但 Linux 的原生 POSIX AIO 实现不完善(实际用线程池模拟)。**io_uring(Linux 5.1+)** 是 Linux 异步 I/O 的未来方向：通过用户态和内核态共享的环形缓冲区(提交队列 SQ + 完成队列 CQ)进行 I/O 请求的提交和收割，性能接近或超过 epoll。

48. Java NIO 用的是哪种 I/O 模型？
答：Java NIO 的 **Selector + Channel** 本质是 **I/O 多路复用模型**（在 Linux 上底层是 epoll），不是异步 I/O。`Selector.select()` 对应 `epoll_wait()`，返回就绪的 `SelectionKey` 集合；然后对就绪的 Channel 调用 `read()/write()`，这一步仍然是同步阻塞的(数据在内核和用户态之间拷贝)。

Java 7 引入了 `AsynchronousSocketChannel`(Java NIO.2)，在 Linux 上底层使用 epoll + 线程池模拟异步(不是真正的内核异步 I/O)。

49. Netty 为什么基于 I/O 多路复用而不是异步 I/O？
答：
- Linux 上 **epoll** 成熟稳定，性能优异，是生产级验证过的方案
- Linux 原生 AIO 实现不完善，只对 O_DIRECT 文件 I/O 有用，不支持 Socket
- Netty 的 EventLoop 模型(一个线程 + 一个 Selector 管理数千连接)已经能高效利用 I/O 多路复用
- Netty 的 **EpollEventLoop** 直接使用 JNI 调用 Linux epoll(绕过 JDK 的 Selector 实现)，支持边缘触发(ET)等高级特性
- 实际效果：Netty 服务器用几个 EventLoop 线程就能处理数万甚至数十万并发连接，吞吐和延迟都非常优秀
