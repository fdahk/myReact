# Redis 专题

## 使用方式

不要把 `Redis` 学成“一个更快的 Map”。大厂后端面试和真实业务里更关心：

- 什么时候该用缓存，什么时候不该用。
- 为什么缓存能提升性能，也会带来一致性风险。
- 热点、穿透、击穿、雪崩如何处理。
- 分布式锁、幂等、防重和限流如何落地。

---

## 一、为什么 Redis 是后端高频主干能力

1. 为什么 Redis 在后端岗位里几乎必问？
答：因为它不仅是缓存，还承担分布式锁、计数器、限流、排行榜、会话、幂等、防重、任务协调等作用。它属于“高频基础设施”，很多系统的性能和稳定性都直接依赖它。

2. 为什么说 Redis 的价值不只是“快”？
答：快只是表象。它真正的价值在于提供了低延迟访问能力和多种数据结构，使很多高频读、热点状态和分布式协调问题可以在更低成本下解决。但与此同时，它也引入一致性、容量、持久化和故障切换问题，所以必须成体系理解。

---

## 二、缓存的核心思维

3. 为什么需要缓存？
答：因为数据库、外部服务和复杂计算都可能成为瓶颈。缓存通过把热点结果放到更快的位置来降低访问延迟、减轻下游压力、提升吞吐。后端设计里，缓存本质是“拿一致性的一部分复杂度换性能和可用性”。

4. 什么数据适合放缓存？
答：读多写少、热点明显、容忍短时间不一致、计算或查询成本高的数据适合缓存。比如配置、热点详情、聚合结果、排行榜、用户画像片段等。相反，强一致实时交易数据不一定适合直接缓存。

5. 为什么不是所有数据都上缓存？
答：因为缓存会增加失效策略、更新策略、一致性风险、容量成本和故障复杂度。如果某类数据命中率不高，或者读取本身很便宜，盲目上缓存只会增加系统复杂度。

---

## 三、Redis 常见数据结构

6. `String` 适合什么场景？
答：最通用，适合缓存对象序列化结果、计数器、简单状态位、分布式锁 key、幂等记录等。很多业务问题最终都能抽象成 key-value 状态，所以它在实际项目里出现极高频。

7. `Hash` 适合什么场景？
答：适合一个对象多个字段的场景，比如用户 profile、任务状态字段集合、局部属性更新。它比把整个对象序列化成一个大字符串更灵活。

8. `List`、`Set`、`ZSet` 最常见的业务语义是什么？
答：`List` 常用于消息队列简化场景、时间线、最近列表；`Set` 适合去重、标签、点赞用户集合；`ZSet` 适合排行榜、带分值排序、按时间或权重排序的集合。学习数据结构时一定要和业务场景对应起来。

---

## 四、缓存常见问题

9. 什么是缓存穿透？
答：请求的数据既不在缓存，也不在数据库，导致每次都打到底层。恶意请求或不存在 id 的高频访问很容易造成穿透。常见方案是布隆过滤器、空值缓存、参数校验和访问控制。

10. 什么是缓存击穿？
答：某个热点 key 在失效瞬间，有大量并发请求同时去访问数据库。热点越集中、失效越同步，问题越明显。常见处理是互斥重建、逻辑过期、热点永不过期配合异步刷新。

11. 什么是缓存雪崩？
答：大量 key 在同一时间失效或缓存层整体不可用，导致请求洪峰打到底层。它本质是系统级容量与稳定性问题，不是单个 key 问题。解决方案通常包括过期时间打散、多级缓存、限流降级、熔断保护。

12. 为什么这些问题在大厂面试里必问？
答：因为它们非常能区分“会用缓存”和“懂缓存系统设计”。真正线上缓存问题大多不在 API 使用，而在失效、并发和故障场景。

---

## 五、缓存一致性

13. 为什么缓存最难的是一致性？
答：因为缓存和数据库是两个系统，写入和失效很难天然保持完全同步。你只要引入缓存，就必须接受“数据什么时候可能不一致、不一致能容忍多久、如何修复”的现实。

14. 常见的缓存更新策略有哪些？
答：旁路缓存最常见，即先更新数据库，再删除缓存；也有先删缓存、再写库、延迟双删、消息通知刷新、订阅 binlog 同步等思路。没有绝对银弹，关键看一致性要求和写入模式。

15. 为什么“更新数据库后删除缓存”这么常见？
答：因为它实现简单、读链路清晰、适合多数业务场景。虽然它仍然可能存在极端并发下的短暂不一致，但通常在成本和收益之间比较平衡。

16. 为什么一致性问题不能脱离业务来谈？
答：因为有的场景允许秒级短暂不一致，比如资讯详情；有的场景几乎不能接受，比如库存和支付。后端设计不应该追求抽象上的完美一致，而要和业务风险匹配。

---

## 六、持久化与高可用

17. Redis 为什么既是内存数据库，又会提持久化？
答：因为很多业务不能接受 Redis 一重启所有状态都丢失。持久化是在性能和数据安全之间做平衡，让它在承担缓存之外的职责时更可靠。

18. 为什么高可用问题对 Redis 也很重要？
答：因为一旦 Redis 故障，热点请求可能瞬间打爆数据库，或者分布式锁、限流、会话等能力一起受影响。所以 Redis 在系统里不是“挂了也无所谓的小组件”，而是很多链路的关键依赖。

---

## 七、分布式锁

19. 为什么会用 Redis 做分布式锁？
答：因为在多实例部署下，本地锁只能控制单进程，无法约束多个服务节点。Redis 提供了跨实例共享状态能力，很适合做“同一时刻只允许一个执行者”的协调。

20. 分布式锁适合什么场景？
答：定时任务防重、抢占式任务、热点资源串行更新、幂等保护、避免重复执行某类关键动作等。它不是所有并发问题都该上的手段，而是针对“必须跨节点协调”的场景。

21. 为什么分布式锁不是 setnx 一把梭？
答：因为你还要考虑过期时间、锁误释放、执行超时、重入、续期和异常崩溃。成熟回答一定要体现：分布式锁的问题从来不是“会不会加锁”，而是“锁在异常情况下会怎样”。

---

## 八、限流、计数和幂等

22. Redis 为什么适合做限流？
答：因为它支持高性能计数和过期控制，能快速实现按时间窗口统计访问次数。令牌桶、滑动窗口、漏桶等思路都可以借助 Redis 落地。

23. 幂等为什么常和 Redis 绑定？
答：因为很多接口和消息处理需要短时间内快速判断“这个请求是不是已经处理过”。Redis 非常适合存放幂等 key、请求 token、去重标记等短周期状态。

24. 什么场景特别需要幂等？
答：支付回调、消息重复消费、订单重复提交、重试请求、任务重复执行。后端岗位很重视这个点，因为分布式系统里“重复”是常态，不是例外。

---

## 九、热点 key 与大 key

25. 什么是热点 key？
答：访问量极高的少数 key。它的问题不是业务热点本身，而是会把某个节点、某个分片或某类请求打成局部瓶颈。很多系统在总体 QPS 看着正常时，实际就是被热点 key 拖垮。

26. 什么是大 key？
答：单个 key 对应的数据量特别大，比如超大列表、超大 hash、超大对象。它会拖慢网络传输、序列化和删除操作，也更容易造成阻塞。

27. 为什么热点和大 key 都会成为线上问题？
答：因为它们都会制造“局部极端成本”，让原本均匀的系统突然出现尖刺。顶级后端会非常关注这种长尾风险，而不是只看平均性能。

---

## 十、面试该如何回答 Redis

28. 面试官问 Redis，最少要覆盖哪些点？
答：数据结构、缓存价值、穿透/击穿/雪崩、一致性、分布式锁、限流/幂等、热点 key 和大 key、高可用基本认知。只会说“Redis 单线程所以快”远远不够。

29. 一个更像成熟后端的回答方式是什么？
答：始终把 Redis 放在“系统设计”里讲，而不是“工具题”里讲。也就是说，你要把它和数据库、并发、一致性、限流、故障降级串起来。

## 十一、你现阶段最该掌握到什么程度

你至少要达到下面标准：

1. 能判断什么数据适合缓存，什么不适合。
2. 能说清缓存穿透、击穿、雪崩与对应治理方式。
3. 能解释基本的一致性策略与业务 trade-off。
4. 能把 Redis 用到分布式锁、幂等、限流等高频后端场景。
5. 能把 Redis 设计映射到你的未来 Java 主项目里。

如果你现在还只是“把数据库查出来的数据顺手丢进 Redis”，那还不能算真正掌握缓存设计。

## 十二、持久化、淘汰与高可用细节

30. `Redis` 为什么会有 `RDB` 和 `AOF` 两类持久化？
答：因为不同业务对性能和数据安全的要求不同。`RDB` 更像定期快照，恢复快、文件紧凑，但两次快照之间的数据可能丢；`AOF` 更像记录写操作日志，数据安全性通常更好，但文件更大、恢复和重写成本也更高。成熟回答不是背定义，而是知道什么时候更关注恢复速度，什么时候更关注数据损失窗口。

31. 为什么做缓存时也要关心持久化？
答：因为很多系统里的 Redis 并不只承担“随时可丢的纯缓存”，还可能承载会话、幂等标记、限流状态、分布式锁、排行榜、任务协调等状态。如果这些状态在实例重启后全丢了，业务可能会出现重复执行、限流失效或状态错乱。

32. 内存淘汰策略为什么是 Redis 面试里的高频点？
答：因为 Redis 的内存不是无限的，内存打满后怎么淘汰，会直接影响热点数据命中率和系统稳定性。你至少要有意识：不同业务是否允许淘汰、淘汰的是过期键还是全量键、是否会把真正重要的热点数据淘掉。这不是纯配置题，而是业务优先级问题。

33. 为什么 Redis 高可用不能只理解成“多弄几台”？
答：因为主从、哨兵、集群模式分别解决的是复制、故障发现、自动切换和分片扩展问题，但它们也会带来复制延迟、脑裂、槽位迁移、热点分布不均和运维复杂度。真正成熟的理解是：高可用不是实例数量，而是故障时系统还能否保持可接受行为。

## 十三、主从、集群与热点治理

34. 为什么 Redis 主从也会有一致性问题？
答：因为复制存在延迟。你刚写进主节点的数据，读从节点时未必立刻能看到。如果你的业务把 Redis 用在强约束状态，例如刚写入的幂等标记、任务状态、会话校验，那么复制延迟就可能带来短暂不一致，所以读写路径要结合业务语义设计。

35. Redis 集群里为什么热点 key 更危险？
答：因为热点 key 不会随着节点数量增加自动均匀分散，反而可能把单个分片或单个主节点打成瓶颈。系统总体容量看起来还够，但某个分片已经先顶不住了。所以 Redis 题一定要有“总量正常不代表热点正常”的意识。

36. 热点 key 的成熟治理思路是什么？
答：短期止血可以做本地缓存、多副本读、热点预热、限流和降级；长期则要看能不能拆热点、调整访问模型、引入多级缓存、让热点数据异步刷新而不是同步回源。真正难的不是知道热点会出问题，而是知道要从流量分布和业务模型上治。

37. 大 key 为什么会让 Redis 看起来“偶发卡顿”？
答：因为大 key 带来的问题不只是占内存，还包括网络传输慢、序列化慢、删除阻塞、扫描和迁移成本高。很多系统平均 QPS 正常，但一碰到某个超大 key 的读取或删除，就会突然抖一下，这就是典型的长尾问题。

## 十四、Redis 一致性与事故场景

38. 为什么“先更新数据库再删缓存”也会出问题？
答：因为极端并发下仍可能出现旧值回写缓存、读到旧数据、主从延迟放大不一致等问题。这个策略之所以常用，是因为总体上性价比高，不代表它没有边界。成熟回答一定会强调：要结合业务对不一致窗口的容忍度看，而不是把它说成绝对正确方案。

39. 双删、延迟双删、订阅 binlog 同步缓存分别适合什么场景？
答：延迟双删适合你担心并发回写旧值，但又不想引入太重机制的场景；订阅 binlog 更适合多下游同步、缓存更新链路比较重、需要统一数据变更事件的场景。没有绝对银弹，关键是成本、实时性和一致性要求的平衡。

40. 如果 Redis 故障导致数据库被打穿，你会怎么止血？
答：先保护下游数据库，通常会限流、降级、只保核心接口、临时开启本地缓存或热点兜底；同时快速确认是 Redis 完全不可用、部分分片异常、网络问题还是大面积 key 失效。这个题最看重的是“先保护主存储”，而不是只说“重启 Redis”。

41. 为什么 Redis 相关问题经常和降级一起出现？
答：因为一旦缓存层失效，底层数据库和下游通常承受不了同等流量。后端设计里缓存不仅是性能组件，也是流量缓冲组件。它一挂，系统就需要立刻切换到降级思维。

## 十六、Redis 为什么快：线程模型与 I/O 多路复用

42. Redis 的线程模型到底是怎样的？
答：Redis 的**命令执行**始终是单线程的——一个主线程按顺序从命令队列中取出请求、执行、返回结果。但"单线程"并不意味着整个进程只有一个线程：
- **后台线程**：Redis 在启动时会创建若干 BIO(Background I/O) 线程，专门处理 close fd、AOF fsync、惰性删除(lazyfree) 等不适合阻塞主线程的操作。
- **Redis 6.0+ I/O 多线程**：引入了可配置的 I/O 线程（`io-threads` 默认 1，即关闭），专门负责**读取客户端请求数据（read）和写回响应数据（write）**，但命令的解析与执行仍然在主线程完成。这意味着多线程 I/O 阶段不涉及共享数据结构的并发修改，所以**不需要加锁**。

43. 单线程执行命令为什么能做到高性能？
答：有四个关键原因叠加：
1. **纯内存操作**：数据全在内存中，单次读写耗时通常在亚微秒到几微秒级别，远快于磁盘和网络。
2. **无锁无上下文切换**：单线程不存在线程上下文切换（每次切换约几微秒）和锁竞争开销。多线程模型中锁的 CAS 或 mutex 等操作在高并发下会显著拖慢吞吐。
3. **I/O 多路复用一次处理多个连接**：通过 epoll（Linux）/kqueue（macOS）/evport（Solaris）机制，单线程可以同时监听数万个客户端连接，有数据可读时才处理，不需要为每个连接分配一个线程。
4. **高效的数据结构**：Redis 内部大量使用 ziplist、intset、skiplist 等针对小数据优化的紧凑结构，减少内存分配和 CPU cache miss。

44. epoll 的核心原理是什么？它和 select 有什么区别？
答：
- **select 模型**：每次调用 select() 时，需要把所有被监听的 fd 集合从用户态拷贝到内核态，内核对所有 fd 做 O(n) 线性扫描检查就绪状态，再返回结果。fd 数量上限默认 1024（FD_SETSIZE），且每次都要重新设置 fd 集合。
- **epoll 模型**分三个系统调用：
  1. `epoll_create()`：在内核创建一个 eventpoll 对象，内部维护一棵**红黑树**（存放被监听的 fd）和一个**就绪链表**。
  2. `epoll_ctl(fd, event)`：向红黑树中注册/修改/删除 fd 及其关注的事件。注册时内核为每个 fd 设置一个**回调函数**，当该 fd 就绪时（例如有数据到达触发网卡中断→软中断→协议栈处理），回调函数会把这个 fd 放入就绪链表。
  3. `epoll_wait()`：只需要检查就绪链表是否为空，非空则直接返回就绪的 fd 列表。**不需要遍历所有被监听的 fd**。
- **核心区别**：select 是 O(n) 扫描所有 fd，epoll 是 O(1) 获取就绪 fd（通过回调机制维护就绪链表）。当连接数达到数万时，select 每次调用耗时随 fd 数量线性增长，而 epoll 只与就绪事件数相关。
- epoll 还支持两种触发模式：**LT（Level Triggered，水平触发，默认）**——只要 fd 可读/可写就会持续通知；**ET（Edge Triggered，边缘触发）**——只在状态变化时通知一次，效率更高但编程复杂度也更高。Redis 使用的是 LT 模式。

45. Redis 6.0 的多线程 I/O 具体是怎么工作的？
答：整个流程分为四步：
1. **主线程**通过 epoll_wait 收集所有就绪的客户端连接，将它们的读任务分配到 I/O 线程队列中。
2. **I/O 线程**（包括主线程本身也参与）并行读取各个客户端 socket 的请求数据到 querybuf，读取完成后等待主线程。
3. **主线程**串行地解析命令、执行命令、将结果写入输出缓冲区（这一步是单线程，没有并发问题）。
4. **主线程**再将写任务分配给 I/O 线程，I/O 线程并行将响应数据写回各客户端 socket。
配置方式：`io-threads 4`（开启 4 个 I/O 线程）和 `io-threads-do-reads yes`（允许读也走多线程）。通常在连接数非常多、网络带宽是瓶颈时才需要开启，一般建议设为 CPU 核数的一半左右，不超过 8。

46. 为什么 Redis 不用多线程执行命令？
答：核心原因有三：
1. **加锁成本高**：如果多线程并发执行命令，对共享数据结构（dict、skiplist 等）的每次读写都需要加锁。在 Redis 这种每秒数十万次操作的场景下，锁竞争会严重降低吞吐，甚至可能比单线程还慢。
2. **内存操作本身足够快**：单个命令的执行时间通常在微秒级，CPU 不是瓶颈。实际瓶颈大多在**网络 I/O**（收发数据包、序列化/反序列化），所以 Redis 6.0 选择把多线程用在 I/O 上而不是命令执行上。
3. **实现复杂度**：多线程并发执行会引入死锁、竞态条件、内存可见性等问题，极大增加代码复杂度和 bug 风险。Redis 作者 Antirez 多次强调过，单线程的简单性是 Redis 稳定和高效的重要保障。

## 十七、底层数据结构编码

47. SDS（Simple Dynamic String）的内部结构是什么？比 C 字符串好在哪？
答：SDS 的结构为 `len(已用长度) + alloc(分配总长度) + flags(类型标志，1字节) + buf[](实际数据)`。根据字符串长度不同，Redis 定义了 sdshdr5/8/16/32/64 五种头部类型，尽量压缩头部占用。SDS 相比 C 原生字符串的优势：
1. **O(1) 获取长度**：直接读 len 字段，C 字符串需要 O(n) 遍历到 `\0`。
2. **二进制安全**：不以 `\0` 作为结束标志，可以存储图片、序列化数据等包含 `\0` 的内容。
3. **空间预分配**：扩容时如果修改后 len < 1MB，则分配 2 * len 的空间；如果 len >= 1MB，则多分配 1MB。减少频繁内存重分配。
4. **惰性空间释放**：缩短字符串时不立即释放多余内存，只修改 len，留着备用。需要时可以手动调用 `sdsRemoveFreeSpace`。
5. **防止缓冲区溢出**：修改前自动检查空间是否足够，不够则自动扩容。

48. String 类型有哪几种底层编码？转换条件是什么？
答：String 类型有三种编码：
1. **int 编码**：当值是纯整数且在 long 范围内（64 位系统是 -2^63 到 2^63-1），直接把 `redisObject.ptr` 当作 long 存值，不额外分配内存。对于 0~9999 的整数，Redis 启动时会预创建共享对象，进一步节省内存。
2. **embstr 编码**：当字符串长度 <= 44 字节时使用。`redisObject`（16字节）和 SDS 在一次 `malloc` 调用中连续分配，总共 64 字节正好是 jemalloc 的一个内存分配档位。好处是只需一次内存分配和一次释放，且数据连续、CPU cache 友好。
3. **raw 编码**：当字符串长度 > 44 字节时使用。`redisObject` 和 SDS 分两次 `malloc` 分别分配，`ptr` 指向 SDS 的地址。
- **转换规则**：embstr 是只读的，任何修改操作（如 APPEND）都会先将 embstr 转为 raw 再执行修改。int 编码在 APPEND 等操作后也会转为 raw。

49. List 的底层结构 quicklist 是什么？
答：Redis 3.2 之后 List 的底层统一使用 **quicklist**，它是**双向链表 + ziplist 的混合体**：
- quicklist 本身是一个双向链表，每个节点（quicklistNode）内部存放的不是单个元素，而是一个 **ziplist**（Redis 7.0+ 替换为 listpack）。
- ziplist 是一块连续内存，存储多个小元素时非常紧凑（无指针开销），但插入/删除涉及内存搬移，所以每个 ziplist 不能太长。
- 通过 `list-max-ziplist-size` 控制每个节点内 ziplist 的大小（正数表示元素个数上限，负数 -1~-5 表示字节上限：-2 = 8KB 默认推荐）。
- quicklist 还支持 **LZF 压缩**中间节点（`list-compress-depth`，默认 0 不压缩），头尾各保留若干个不压缩的节点用于高频的 LPUSH/RPUSH/LPOP/RPOP 操作。
- 这样兼顾了双向链表的 O(1) 两端操作和 ziplist 的内存紧凑性。

50. Hash 的底层编码和转换条件是什么？
答：Hash 类型有两种底层编码：
1. **ziplist（Redis 7.0+ 为 listpack）**：当同时满足以下两个条件时使用：
   - 元素数量 <= `hash-max-ziplist-entries`（默认 **128**）
   - 每个 field 和 value 的长度都 <= `hash-max-ziplist-value`（默认 **64** 字节）
   - 存储方式是 field 和 value 紧挨着依次排列在连续内存中，查找时顺序遍历 O(n)，但因为数据量小且内存连续，实际很快。
2. **hashtable（dict）**：当任一条件不满足时，自动从 ziplist 转为 hashtable。dict 使用**链地址法**解决哈希冲突，有两个哈希表（ht[0] 和 ht[1]）用于**渐进式 rehash**——不一次性搬迁所有数据，而是在每次增删改查时顺带迁移一部分，避免大 hash 的 rehash 造成长时间阻塞。

51. Set 的底层编码和转换条件是什么？
答：Set 类型有两种底层编码：
1. **intset**：当同时满足以下两个条件时使用：
   - 所有元素都是整数
   - 元素数量 <= `set-max-intset-entries`（默认 **512**）
   - intset 是一个**有序整数数组**，支持二分查找 O(logN)，内存非常紧凑。根据最大元素的值自动选择 int16/int32/int64 编码，新加入更大范围的整数时会触发**升级**（重新分配内存并转换所有元素），但不支持降级。
2. **hashtable**：当任一条件不满足时（包括加入了非整数元素），转为 hashtable，value 部分为 NULL。

52. ZSet 的底层编码和转换条件是什么？Skiplist 结构是怎样的？
答：ZSet 有两种底层编码：
1. **ziplist（Redis 7.0+ 为 listpack）**：当同时满足以下条件时使用：
   - 元素数量 <= `zset-max-ziplist-entries`（默认 **128**）
   - 每个 member 和 score 的长度都 <= `zset-max-ziplist-value`（默认 **64** 字节）
   - 在 ziplist 中按 score 排序存储，element 和 score 紧挨排列。
2. **skiplist + dict**：当任一条件不满足时转换。这里同时维护两个数据结构：
   - **skiplist（跳跃表）**：用于按 score 范围查询和排序操作（ZRANGEBYSCORE、ZRANK 等）。
   - **dict（哈希表）**：用于按 member 精确查找 score（ZSCORE），O(1) 复杂度。
   - 两者共享 member 和 score 的指针，不会产生双倍内存。

**Skiplist 结构详解**：
- 本质是**多层有序链表**。最底层（level 0）包含所有元素，每往上一层是下层的"快速通道"，只包含部分元素。
- **查找过程**：从最高层开始，沿着当前层向右走，如果下一个节点的 score 大于目标值就下降一层，继续向右。平均时间复杂度 **O(logN)**。
- **新节点层数**：插入时通过**随机函数**决定层数，概率 p = 0.25（即每多一层的概率是 25%），最高 **32 层**（ZSKIPLIST_MAXLEVEL）。这意味着平均每个节点有 1/(1-p) = 1.33 个指针，内存开销很小。
- **为什么用 skiplist 而不是红黑树**：skiplist 实现简单、范围查询天然高效（沿链表遍历即可）、插入/删除只需修改局部指针不需要旋转，且在并发场景下更容易实现无锁版本。Redis 作者 Antirez 在设计时明确选择了 skiplist 的简洁性。

## 十八、持久化机制详解

53. RDB 持久化的完整流程是什么？
答：RDB 通过 `bgsave` 命令（或自动触发）生成某个时间点的内存快照，完整流程如下：
1. **主进程调用 fork()** 创建子进程。fork 的核心开销是**复制页表**（Page Table），而不是复制实际内存数据。页表大小与进程使用的虚拟内存页数成正比——例如一个使用 20GB 内存的 Redis 实例，页表可能占数十 MB，fork 耗时在十几到几十毫秒级别，**期间主进程会阻塞**。
2. **fork 完成后主进程恢复正常处理请求**，子进程和父进程通过 **COW（Copy-On-Write）** 共享物理内存页：
   - fork 后父子进程的虚拟地址指向相同的物理页，这些页被标记为**只读**。
   - 当父进程（主进程）需要**写入某一页**时，CPU 触发缺页中断，操作系统才复制这一页的物理内存，父进程在副本上写入。这就是"写时复制"。
   - 如果 fork 后写入量很小，实际复制的物理页很少，内存开销很小；如果写入量很大（如 bgsave 期间持续大量写入），可能导致内存使用接近 2 倍峰值。
3. **子进程遍历整个内存数据库**，将每个 key-value 序列化为 RDB 二进制格式，写入一个**临时文件**。
4. **子进程写完后**，调用 `rename()` 将临时文件原子替换为正式 RDB 文件（如 dump.rdb），确保 RDB 文件始终完整。
5. **子进程退出**，主进程记录本次 bgsave 完成的时间。

自动触发配置示例（redis.conf）：
```
save 900 1      # 900秒内至少1个key被修改
save 300 10     # 300秒内至少10个key被修改
save 60 10000   # 60秒内至少10000个key被修改
```
满足任一条件就触发 bgsave。生产环境中如果 Redis 仅做缓存，可以关闭 RDB（`save ""`）来避免 fork 开销。

54. AOF 持久化的机制和刷盘策略是什么？
答：AOF（Append Only File）将每个写命令以 RESP 协议格式追加到文件末尾。具体流程：
1. Redis 执行完写命令后，将该命令追加到内存中的 **aof_buf（AOF缓冲区）**。
2. 在每次事件循环结束前，根据 `appendfsync` 策略决定何时将 aof_buf 刷入磁盘：

| 策略 | 行为 | 性能 | 数据安全 |
|------|------|------|----------|
| `always` | 每个写命令都 fsync 到磁盘 | 最慢，每次都有磁盘 I/O | 最高，最多丢失一个命令 |
| `everysec`（默认推荐） | 每秒由后台线程执行一次 fsync | 适中，对主线程几乎无影响 | 故障最多丢失 1 秒数据 |
| `no` | 由操作系统决定何时刷盘（通常 30 秒） | 最快 | 可能丢失较多数据 |

55. AOF 重写（bgrewriteaof）是怎么工作的？
答：AOF 文件会随着写操作不断增长（比如同一个 key 被 SET 了 100 次，AOF 中有 100 条记录），AOF 重写的目的是生成一个最小化的等价 AOF 文件：
1. **主进程 fork 子进程**（与 bgsave 类似，也有 COW 机制）。
2. **子进程读取当前内存数据库的状态**，将每个 key 的当前值直接转化为一条写命令（例如一个有 100 个元素的 List 只生成一条 RPUSH 命令），写入一个**新的临时 AOF 文件**。注意：不是压缩或分析旧的 AOF 文件，而是根据内存中的实际数据状态重新生成。
3. **重写期间主进程继续处理请求**，新产生的写命令会**同时写入两个缓冲区**：
   - **旧的 aof_buf**：保证旧 AOF 文件继续完整追加（万一重写失败，旧文件还能用）。
   - **AOF 重写缓冲区（aof_rewrite_buf）**：记录重写期间的增量命令。
4. 子进程完成重写后**通知主进程**，主进程将 aof_rewrite_buf 中的增量命令追加到新 AOF 文件末尾，然后原子替换旧 AOF 文件。

自动触发条件配置：
```
auto-aof-rewrite-percentage 100   # 当前AOF文件比上次重写后的大小增长了100%
auto-aof-rewrite-min-size 64mb    # 且当前AOF文件至少64MB
```

56. RDB + AOF 混合持久化是什么？
答：Redis 4.0+ 引入了混合持久化（`aof-use-rdb-preamble yes`，Redis 5.0+ 默认开启）。在 AOF 重写时：
- **前半段**使用 RDB 格式写入当前内存快照（体积小、加载快）。
- **后半段**使用 AOF 格式记录重写期间的增量写命令。
- 最终生成的 AOF 文件结构为：`[RDB 二进制数据][AOF 增量命令]`。
好处是兼顾了 RDB 的**快速加载**和 AOF 的**数据安全**：重启时先加载 RDB 部分（速度快），再重放后面的 AOF 增量命令（数据损失小）。这是目前生产环境的推荐配置。

57. fork 在持久化中可能带来哪些问题？
答：
1. **fork 阻塞**：虽然 COW 使得 fork 不需要复制实际数据，但复制页表本身仍然耗时。对于大内存实例（如 20GB+），fork 可能阻塞主线程数十到上百毫秒，导致请求抖动。可以通过 `info stats` 的 `latest_fork_usec` 监控 fork 耗时。
2. **COW 内存膨胀**：如果 bgsave/bgrewriteaof 期间写入量很大，大量页面被复制，内存使用可能接近 2 倍。生产环境建议预留足够内存（一般留 maxmemory 的 1.5~2 倍物理内存）。
3. **关闭 THP（Transparent Huge Pages）**：Linux 默认开启 THP（2MB 大页），COW 时一个字节的修改就会复制整个 2MB 页，极大加剧内存膨胀。Redis 启动时会检查并建议关闭：`echo never > /sys/kernel/mm/transparent_hugepage/enabled`。

## 十九、Redis Cluster 架构

58. Redis Cluster 的数据分片原理是什么？
答：Redis Cluster 将整个数据集划分为 **16384 个 hash slot（哈希槽）**，每个 key 通过 `CRC16(key) % 16384` 计算出所属的 slot 编号，每个集群节点负责一部分 slot。例如三主节点的集群可能分配为：
- 节点 A：slot 0 ~ 5460
- 节点 B：slot 5461 ~ 10922
- 节点 C：slot 10923 ~ 16383

如果 key 中包含 `{...}` 花括号，则只对花括号内的部分做 CRC16（**hash tag** 机制），这样可以确保相关的 key 落在同一个 slot，便于使用 MGET、Pipeline 或 Lua 脚本操作多个 key。

59. 为什么是 16384 个 slot 而不是更多？
答：这是出于**心跳包大小的考虑**。集群节点之间通过 Gossip 协议交换信息，每条消息中需要包含一个 bitmap 标识"该节点负责哪些 slot"。16384 个 slot 的 bitmap 大小是 16384 / 8 = **2KB**，在心跳包中可以接受。如果用 65536 个 slot，bitmap 就是 8KB，每次心跳都要多传 8KB 数据，在大规模集群中会造成不必要的带宽浪费。同时 Redis 官方建议集群节点不超过 1000 个，16384 个 slot 足够分配。

60. 客户端是怎么知道 key 在哪个节点上的？
答：客户端连接任一节点后，通过 `CLUSTER SLOTS` 或 `CLUSTER NODES` 命令获取完整的 slot→node 映射关系并缓存到本地。当客户端请求的 key 恰好在当前连接的节点上时，直接执行。否则节点返回两种重定向：
- **MOVED slot ip:port**：表示该 slot **永久**迁移到了另一个节点，客户端应更新本地缓存并重新发送请求。
- **ASK slot ip:port**：表示该 slot 正在**迁移过程中**，客户端应先向目标节点发送 `ASKING` 命令再发送实际请求，但不更新本地缓存（因为迁移可能尚未完成）。

61. Gossip 协议在集群中起什么作用？
答：集群节点之间通过 **Gossip 协议**以去中心化的方式交换状态信息：
- 每个节点每秒随机选择若干个节点发送 **PING** 消息，包含自己的状态信息和部分已知的其他节点状态。
- 接收方回复 **PONG** 消息，同样携带自己的信息。
- 通过这种"传染"式传播，集群状态最终会收敛到一致（**最终一致性**），不需要一个中心节点统一维护。
- 每条 PING/PONG 消息携带的信息包括：节点 ID、IP/端口、负责的 slot bitmap（2KB）、节点状态标记（PFAIL/FAIL）、集群 epoch 等。
- 代价是收敛有延迟——大规模集群中故障信息可能需要数秒才能传播到所有节点。

62. 故障检测和自动故障转移是怎么工作的？
答：分为三个阶段：
1. **主观下线（PFAIL）**：当节点 A 向节点 B 发送 PING 后，在 `cluster-node-timeout`（默认 15 秒）内没收到 PONG 回复，A 就把 B 标记为 **PFAIL**（Probable Fail，主观下线）。这只是 A 的单方面判断。
2. **客观下线（FAIL）**：A 在后续的 Gossip 消息中把"B 疑似下线"这个信息传播出去。当集群中**超过半数的主节点**都把 B 标记为 PFAIL 时（信息通过 Gossip 汇集），B 被升级为 **FAIL**（客观下线），并向整个集群广播 FAIL 消息。
3. **从节点选举**：如果 FAIL 的是一个主节点，它的从节点会发起选举：
   - 从节点将自己的 `currentEpoch` 加 1，向所有主节点发送 `FAILOVER_AUTH_REQUEST` 投票请求。
   - 每个主节点在一个 epoch 内只能投一票（基于 **Raft** 思想），投给最先请求的从节点。
   - 如果有从节点获得了**超过半数主节点**的投票，它就晋升为新的主节点，接管原主节点的 slot，并广播新的配置。
   - 选举中有**优先级机制**：复制偏移量（replication offset）最大的从节点优先发起投票（数据最新），通过延迟发起投票的时间来实现。

63. 在线扩缩容时数据迁移是怎么做的？
答：通过 `redis-cli --cluster reshard` 命令或手动使用 `CLUSTER SETSLOT` + `MIGRATE` 实现 slot 级别的在线迁移：
1. 目标节点标记 slot 为 **IMPORTING** 状态（`CLUSTER SETSLOT slot IMPORTING source-node-id`）。
2. 源节点标记 slot 为 **MIGRATING** 状态（`CLUSTER SETSLOT slot MIGRATING target-node-id`）。
3. 源节点逐个 key 迁移：`CLUSTER GETKEYSINSLOT slot count` 获取该 slot 中的 key，然后 `MIGRATE` 命令将 key 原子地从源节点移动到目标节点（实际是 DUMP + RESTORE + DEL）。
4. 迁移期间的请求处理：
   - 如果 key 还在源节点上，正常处理。
   - 如果 key 已迁移走，源节点返回 **ASK** 重定向，客户端去目标节点执行。
   - 如果 key 不存在，源节点也返回 ASK（可能 key 已被迁移或本就不存在）。
5. 所有 key 迁移完成后，向集群所有节点发送 `CLUSTER SETSLOT slot NODE target-node-id`，完成 slot 归属变更。

整个过程对客户端透明，只是在迁移期间相关 slot 的请求会多一次 ASK 重定向，延迟略有增加。

## 二十、内存淘汰策略详解

64. Redis 有哪些内存淘汰策略？分别在什么场景下使用？
答：Redis 在 `maxmemory` 用尽时，根据 `maxmemory-policy` 配置决定淘汰行为，共有 8 种策略：

| 策略 | 范围 | 算法 | 适用场景 |
|------|------|------|----------|
| **noeviction** | - | 不淘汰，新写入直接返回错误 | 不允许丢失任何数据，宁可报错 |
| **allkeys-lru** | 所有 key | 近似 LRU | 最通用，适合大多数缓存场景 |
| **volatile-lru** | 设置了过期时间的 key | 近似 LRU | 希望保留永不过期的 key |
| **allkeys-lfu** | 所有 key | 近似 LFU | 访问频率差异大的场景 |
| **volatile-lfu** | 设置了过期时间的 key | 近似 LFU | 同上，但只淘汰有 TTL 的 key |
| **allkeys-random** | 所有 key | 随机 | 访问模式均匀无热点 |
| **volatile-random** | 设置了过期时间的 key | 随机 | 同上，限定范围 |
| **volatile-ttl** | 设置了过期时间的 key | TTL 最短的优先淘汰 | 希望越早过期的越先被清理 |

65. Redis 的近似 LRU 是怎么实现的？
答：Redis 没有用标准的 LRU 实现（维护一个全局双向链表，每次访问都把 key 移到头部），因为对每个 key 都维护链表指针和每次访问都做移动操作成本太高。Redis 的做法是：
1. 每个 `redisObject` 结构体中有一个 **lru 字段（24 bit）**，记录该对象**最后一次被访问时的时钟值**（以秒为精度的 Unix 时间戳，取低 24 位，约 194 天循环一次）。
2. 当需要淘汰时，Redis 随机采样 `maxmemory-samples` 个 key（默认 **5**），比较它们的 lru 字段，淘汰其中**最久没有被访问的**那个。
3. Redis 3.0 引入了**淘汰池（eviction pool）**优化：维护一个大小为 16 的候选池，每次采样后把比池中现有候选更"老"的 key 放入池中，淘汰池中最老的 key。这样多轮采样下来，淘汰精度接近真正的 LRU。
4. `maxmemory-samples` 越大，精度越高但 CPU 开销也越大。Redis 官方测试显示 samples=10 时已经非常接近理想 LRU 曲线。

66. Redis 的 LFU（Least Frequently Used）是怎么实现的？
答：LFU 在 Redis 4.0 引入，复用了 `redisObject.lru` 这 24 bit 字段，但拆分为两部分：
- **高 16 bit：ldt（Last Decrement Time）**——上次计数器衰减的时间戳（以分钟为单位，取低 16 位，约 45 天循环一次）。
- **低 8 bit：logc（Logarithmic Counter）**——对数频率计数器，范围 0~255，新建对象初始值为 `LFU_INIT_VAL`（默认 5）。

**计数器递增逻辑**（不是简单 +1，是概率递增）：
1. 计算 `p = 1.0 / (baseval * lfu_log_factor + 1)`，其中 baseval = 当前 logc 值，`lfu-log-factor` 默认 **10**。
2. 生成一个 0~1 的随机数 r，如果 r < p 则 logc + 1。
3. 这意味着 logc 越大，递增概率越小。当 lfu-log-factor=10 时：logc 从 0 到 10 大约需要 100 次访问，从 10 到 100 大约需要 10000 次，从 100 到 255 需要约 100 万次。这就是"对数"的含义。

**计数器衰减逻辑**：
1. 每次访问 key 时先做衰减：`num_periods = (当前分钟时间戳 - ldt) / lfu_decay_time`。
2. `logc = max(logc - num_periods, 0)`，即每过 `lfu-decay-time` 分钟（默认 **1**），计数器减 1。
3. 这保证了曾经很热但已经冷却的 key 的计数器会逐渐下降，不会永远占据内存。

**淘汰时**同样采用随机采样 + 淘汰池机制，只是比较的是 logc 值（越小越优先被淘汰）。

67. 为什么 Redis 选择近似算法而不是精确 LRU/LFU？
答：精确 LRU 需要维护一个**全局双向链表**包含所有 key，每次 key 被访问时必须将其从链表中删除并移到头部。当 Redis 拥有数千万个 key 时：
1. 每个 key 额外需要两个指针（prev/next），在 64 位系统上是 16 字节/key，百万级 key 就是数十 MB 额外内存。
2. 每次读写操作都需要修改链表指针，高 QPS 下这个操作累积的 CPU 开销不可忽视。
3. 链表操作涉及非连续内存访问，对 CPU cache 不友好。
近似 LRU 只需要每个 redisObject 一个 24 bit 字段（本来就有的空间），淘汰时采样几个 key 就行，时间和空间开销极小。Redis 官方基准测试显示，samples=5 时近似 LRU 的命中率只比理想 LRU 低约 2~3%。

## 二十一、分布式锁的正确实现

68. Redis 分布式锁的基本实现为什么必须用一条命令？
答：正确的加锁方式是：
```
SET lock_key <unique_value> NX PX <milliseconds>
```
- `NX`：只在 key 不存在时才设置（互斥语义）。
- `PX`：设置过期时间（防止持有者崩溃后锁永远不释放）。
- **必须是一条原子命令**。早期有人用 `SETNX lock_key value` + `EXPIRE lock_key seconds` 两步操作，但如果在 SETNX 之后、EXPIRE 之前进程崩溃，锁就变成了永不过期的死锁。SET 命令的 NX PX 组合在 Redis 2.6.12+ 引入，一次原子完成。

69. 为什么锁的 value 必须是唯一的（如 UUID）？释放锁为什么要用 Lua 脚本？
答：假设 A 拿到锁后业务执行超时，锁自动过期，B 此时拿到了锁。如果 A 业务执行完后直接 `DEL lock_key`，就会把 B 的锁删掉。所以释放锁必须**先验证 value 是不是自己设的，再删除**，而且这两步必须是原子的。单独 GET + DEL 不是原子的，两步之间可能有其他客户端修改了状态。正确做法是使用 Lua 脚本：
```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```
Redis 保证 Lua 脚本执行期间不会被其他命令打断（单线程执行），所以 GET + DEL 在脚本中是原子的。

70. 锁过期了但业务还没执行完怎么办？
答：这就是**续期（Watch Dog）问题**。如果过期时间设太短，业务没执行完锁就释放了，其他客户端会拿到锁导致并发冲突；设太长，持有者崩溃后要等很久锁才释放。解决方案是**看门狗（WatchDog）机制**，Redisson 的实现最为经典：
1. 加锁时设置一个默认过期时间（Redisson 默认 **30 秒**）。
2. 加锁成功后启动一个后台定时任务（WatchDog），每隔 **过期时间 / 3**（即 10 秒）检查锁是否还被当前线程持有。
3. 如果仍然持有，就调用 `PEXPIRE` 将锁续期到 30 秒；如果线程已经释放锁或崩溃，定时任务也会停止，锁自然过期。
4. 这样既避免了锁在业务执行中途过期，又保证了持有者崩溃后锁最终会释放。

71. Redlock 算法是什么？它有什么争议？
答：Redlock 是 Redis 作者 Antirez 提出的在**多个独立 Redis 实例**上实现分布式锁的算法，旨在避免单点 Redis 故障导致锁失效。核心步骤：
1. 获取当前时间戳 T1。
2. 依次向 N 个独立的 Redis 实例（通常 N=5）发送 `SET key value NX PX ttl`。
3. 如果在**大多数实例**（>= N/2 + 1，即至少 3 个）上成功加锁，且总耗时 T2-T1 远小于锁的 TTL，则认为加锁成功。
4. 锁的实际有效时间 = TTL - (T2-T1) - 时钟漂移补偿。
5. 如果加锁失败（未拿到多数），立即向所有实例释放锁。

**Martin Kleppmann（DDIA 作者）的批评**：
- **时钟跳跃问题**：如果某个 Redis 实例发生时钟跳跃（NTP 校时、虚拟机暂停等），锁可能提前过期，其他客户端能够同时获得"多数锁"，安全性被破坏。
- **GC 暂停/进程暂停**：客户端拿到锁后如果发生长时间 GC 暂停，锁过期后其他客户端拿到锁，但原客户端 GC 恢复后以为自己还持有锁，导致并发写入。
- **Kleppmann 建议**使用 **fencing token**（单调递增的令牌）：每次获取锁时从一个全局递增计数器获取 token，写入存储层时校验 token 只接受更大的值。但 Redis 本身并不提供这个机制。

**Antirez 的回应**：
- 认为时钟跳跃在合理运维下可控（禁用大幅度 NTP 跳变，使用 adjtime 缓慢校时）。
- 认为 GC 暂停问题在任何分布式锁方案中都存在，不是 Redlock 特有的。

72. 什么时候 Redis 锁就够了，什么时候需要 ZooKeeper/etcd？
答：需要区分两种场景（Martin Kleppmann 的分类）：
- **效率型（Efficiency）锁**：目的是避免重复工作（如重复发短信、重复计算），偶尔锁失效只是多做了一次工作，不会造成数据损坏。这种场景**单节点 Redis 锁就足够**，简单高效，失效了也能容忍。
- **正确性型（Correctness）锁**：目的是保护关键资源不被并发修改（如库存扣减、账户转账），锁失效会导致数据不一致或资金损失。这种场景需要**更强的一致性保证**：
  - **ZooKeeper**：基于 ZAB 协议的共识，创建临时顺序节点实现锁，会话断开自动释放，有严格的顺序保证。
  - **etcd**：基于 Raft 协议的共识，通过 Lease + 事务 CAS 实现锁，提供 revision 作为天然的 fencing token。
  - 这些方案的锁语义是由**共识算法**保证的，不依赖时钟，不怕时钟跳跃。代价是性能更低（需要多数节点确认写入），延迟更高（通常毫秒到十几毫秒级）。

选型建议：
- 大多数互联网业务的分布式锁需求是效率型的，用 Redis（+ Redisson WatchDog）即可。
- 涉及资金、库存等对正确性要求极高的场景，优先考虑 etcd/ZooKeeper，或者在 Redis 锁之外叠加数据库乐观锁/唯一约束等兜底机制。

## 十三、一致性哈希算法详解

1. 为什么简单哈希取模在扩缩容时有问题？
答：hash(key) % N，当N(节点数)变化时，几乎所有key的映射都会改变→大量缓存失效→缓存雪崩。例如从3台扩到4台，约75%的key映射到不同节点。

2. 一致性哈希的核心原理？
答：将整个哈希空间组织成一个0~2^32-1的环。每个节点通过hash(nodeIP)映射到环上一个位置。每个key通过hash(key)映射到环上一个位置，顺时针找到最近的节点就是负责该key的节点。新增/删除节点时，只影响该节点逆时针方向到前一个节点之间的key(约1/N的数据)，其余key映射不变。

3. 虚拟节点解决什么问题？
答：真实节点少时(如3台)，哈希环上分布可能极不均匀→数据倾斜。解决方案：每个真实节点创建多个虚拟节点(如150~200个)，hash(nodeIP#1), hash(nodeIP#2)...均匀散布在环上。key先映射到虚拟节点，再由虚拟节点映射到真实节点。虚拟节点越多，分布越均匀。

4. 一致性哈希在哪些场景使用？
答：分布式缓存(Memcached)、负载均衡(Nginx upstream consistent_hash)、分布式存储(Amazon Dynamo)、Kafka Consumer分区分配。Redis Cluster没用一致性哈希而是用固定16384个hash slot——本质思想类似虚拟节点，但更简单直接。

## 十四、布隆过滤器原理与应用

1. 布隆过滤器的核心原理？
答：一个长度为m的位数组(全0初始化) + k个独立哈希函数。添加元素时：用k个哈希函数计算k个位置，将这k个位置置为1。查询元素时：用同样的k个哈希函数计算k个位置，如果所有位置都是1→可能存在(有误判)；如果有任一位置是0→一定不存在(无误判)。
特性：空间极省(远小于HashMap)、查询O(k)、有误判率但无漏判。

2. 误判率如何计算和控制？
答：误判率 ≈ (1 - e^(-kn/m))^k，其中n=已插入元素数，m=位数组长度，k=哈希函数个数。实践中通常先确定期望误判率(如1%)和预估数据量n，反推m和k。Google Guava的BloomFilter.create(funnel, expectedInsertions, fpp)直接按期望误判率自动计算。

3. 布隆过滤器为什么不支持删除？
答：因为不同元素的哈希位置可能重叠。如果删除元素A时将其k个位置置为0，可能把元素B也「删除」了(误删)。解决方案：Counting Bloom Filter——每个位置不是0/1而是一个计数器，添加时+1，删除时-1。代价是空间×4(每个位置需要4bit计数)。

4. 布隆过滤器的典型应用场景？
答：
- 缓存穿透防护：在Redis前加一层布隆过滤器，不存在的key直接返回，不打数据库
- 爬虫URL去重：已爬过的URL加入布隆过滤器，新URL先查布隆决定是否需要爬
- 垃圾邮件过滤：已知垃圾邮件地址加入布隆过滤器
- HBase/RocksDB/LevelDB：用布隆过滤器判断某个key是否可能在某个SSTable中，减少磁盘IO
- Redis内置：Redis 4.0+通过RedisBloom模块支持`BF.ADD`/`BF.EXISTS`命令
