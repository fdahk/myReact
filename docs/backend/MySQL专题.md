# MySQL 专题

## 使用方式

学习数据库不能只停留在“会写 CRUD”。大厂后端面试里的 `MySQL` 更关注：

- 数据模型是否合理。
- 索引为什么这样建。
- SQL 为什么慢。
- 事务和锁为什么会影响一致性与并发。

你学习这篇时要始终把自己放在“要对线上数据和性能负责”的视角。

---

## 一、为什么 MySQL 是后端主干能力

1. 为什么 MySQL 在后端岗位里地位这么高？
答：因为大多数核心业务最终都要落到结构化数据：用户、订单、资源、权限、任务、日志索引、配置等。`MySQL` 不只是存数据，它还承载查询效率、事务一致性、数据约束和系统演进能力。不会数据库，就谈不上真正的后端。

2. 为什么只会写增删改查远远不够？
答：因为现实问题往往不是“语法会不会”，而是表怎么建、索引怎么选、SQL 为什么慢、并发更新为什么冲突、事务范围应该放多大。后端工程师真正的价值在于这些设计与诊断能力。

---

## 二、表设计与数据建模

3. 设计表结构时最重要的原则是什么？
答：先围绕业务对象和访问模式设计，而不是先按页面字段堆表。你要先明确核心实体是什么、它们之间的关系是什么、最常见的查询路径是什么，再决定字段、索引和拆分方式。

4. 为什么说数据模型是后端设计的地基？
答：因为上层接口、缓存、事务、搜索和报表几乎都依赖底层数据模型。模型一开始设计差了，后面即使业务逻辑写得再漂亮，也会不断被性能、一致性和扩展性问题拖累。

5. 什么情况下需要冗余字段？
答：当读性能、查询便利性或统计需求明显超过范式纯净度时。比如存用户昵称快照、状态冗余、聚合计数等。冗余不是错，但必须明确谁负责更新、何时同步以及不一致时怎么修正。

---

## 三、索引

6. 索引的本质是什么？
答：索引本质上是帮助数据库更快定位数据的额外数据结构，它用空间换时间。回答时不要只说“提高查询速度”，还要知道索引也会影响写入成本、存储空间和维护复杂度。

7. 为什么索引不是越多越好？
答：因为每个索引都要占空间，写入时还要额外维护。索引过多会拖慢插入、更新、删除，也会增加优化器决策成本。成熟的数据库设计看的是“是否命中主要查询路径”，而不是“凡是可能查到的列都建一个索引”。

8. 什么是回表？
答：通过二级索引定位到主键后，再回主键索引或数据页取完整行数据。回表并不一定错，但在高频查询里，如果返回字段很多、数据量大，会明显增加 IO 成本。

9. 什么是覆盖索引？
答：查询所需字段全部都能从索引本身拿到，不需要回表。它的价值在于减少额外数据页访问，是优化高频读场景的经典手段。

10. 为什么联合索引常常优于多个单列索引？
答：因为很多真实查询条件是组合出现的，联合索引能更贴合实际访问路径，也更容易支持排序和范围查询。多个单列索引并不一定能让优化器得到同样好的执行路径。

11. 最左前缀原则为什么重要？
答：因为联合索引的使用依赖前缀匹配顺序。它体现的是索引结构并不是任意列都同等生效，而是有明确组织顺序。这个题高频，是因为它直接关系到你是否真的理解联合索引的行为。

## 四、SQL 与执行计划

12. 为什么后端必须会看执行计划？
答：因为“感觉慢”没有意义，执行计划才告诉你数据库实际是怎么走的。它能暴露是否走索引、是否全表扫描、是否需要排序、是否临时表、是否回表过多。不会看执行计划，就很难做真正可靠的 SQL 优化。

13. SQL 变慢最常见的原因有哪些？
答：索引缺失或失效、返回数据过多、排序和分页成本高、条件写法导致无法走索引、表设计不合理、锁等待、统计信息不准等。面试里如果只答“没建索引”，会显得太浅。

14. 为什么 `select *` 往往不是好习惯？
答：因为你会拿回不需要的列，增加网络传输、内存占用和回表成本，也让覆盖索引更难成立。成熟后端更强调“只查业务真正需要的字段”。

15. 分页为什么在大偏移量时会慢？
答：因为数据库常常需要先扫描并跳过大量记录，尤其 `offset` 很大时成本会迅速上升。更成熟的方案通常是基于游标、主键范围或业务有序字段做分页。

---

## 五、事务

16. 事务解决什么问题？
答：事务解决的是一组操作要么都成功、要么都失败，以及并发访问下的一致性问题。后端业务里很多核心链路都依赖它，比如扣库存、更新状态、写审计日志等。

17. ACID 分别代表什么？
答：原子性、一致性、隔离性、持久性。你不能只背缩写，最好能结合业务理解：原子性保证一组操作不半成品，一致性保证业务约束不被破坏，隔离性保证并发下结果可控，持久性保证提交后不丢。

18. 为什么事务不是越大越好？
答：因为事务范围越大，锁持有时间越长，冲突概率越高，吞吐越差，也更容易把外部调用等不稳定环节带进来。顶级后端更强调“事务边界要小而清晰”。

---

## 六、隔离级别与并发问题

19. 隔离级别在解决什么？
答：它在平衡一致性和性能。并发事务之间既希望互不干扰，又不能把系统变得完全串行，所以数据库提供不同隔离级别来控制脏读、不可重复读、幻读等问题。

20. 脏读、不可重复读、幻读怎么理解？
答：脏读是读到了未提交数据；不可重复读是同一事务内两次读同一行结果不同；幻读是同一事务内按条件查询，两次查出来的记录数量变了。你要把它理解成并发观察到的数据一致性问题，而不是死背名词。

21. 为什么隔离级别越高不一定越好？
答：因为隔离越强，锁和等待通常越多，并发性能会受影响。数据库设计始终是 trade-off，不是“把一致性拉满”就一定是最优。

---

## 七、锁

22. 为什么后端要懂数据库锁？
答：因为很多线上“偶发慢”“更新卡死”“事务超时”本质上不是 SQL 算法问题，而是锁等待和并发冲突。不会锁，就很难解释高并发写场景下的异常表现。

23. 行锁和表锁的差异是什么？
答：行锁粒度小、并发更高，但实现和维护更复杂；表锁粒度大、冲突更明显。大多数业务数据库都希望尽量落到行级控制，而不是整表阻塞。

24. 乐观锁和悲观锁在数据库里怎么体现？
答：乐观锁通常用版本号、时间戳、状态条件更新等方式，先尝试更新，失败再重试；悲观锁则通过 `select ... for update` 等手段直接锁住记录。前者适合冲突不频繁场景，后者适合高冲突关键链路。

25. 什么是死锁？为什么它不是“数据库坏了”？
答：死锁是多个事务互相等待对方持有的资源，形成循环依赖。它本质是并发访问顺序与锁粒度设计问题，不是数据库异常。成熟的后端要知道如何通过统一访问顺序、缩小事务和索引优化来降低死锁概率。

---

## 八、主从、分库分表与演进意识

26. 为什么数据库要做主从？
答：典型目的是读写分离、提升可用性和容灾能力。写入主库，读取可由从库承接，但这也会带来复制延迟和一致性问题。你要有“不是加一台机器就完事”的意识。

27. 为什么不是一开始就分库分表？
答：因为分库分表会带来路由、事务、分页、统计、跨分片查询和运维复杂度。只有在单库单表明显成为瓶颈时才值得引入。成熟设计强调逐步演进，而不是过度设计。

28. 分库分表最难的不是拆，而是什么？
答：是后续的一致性、跨分片查询、全局 ID、迁移过程和复杂运维成本。很多人把它理解成性能技巧，实际上它更像架构级取舍。

## 九、你在面试里至少要答到什么程度

29. 面试官问 MySQL，你至少要能稳定覆盖哪些点？
答：表设计、索引原理与选择、执行计划、事务、隔离级别、锁、分页优化、慢 SQL 排查、主从与演进思路。只会“索引底层是 B+ 树”是不够的。

30. 一个更像大厂后端的回答风格是什么？
答：不是只报名词，而是用“业务场景 -> 数据模型 -> 查询路径 -> 索引设计 -> 并发影响 -> 演进策略”的顺序展开。数据库题真正拉开差距的地方，是你是否能从业务和系统视角做设计，而不只是背原理。

## 十、你现阶段最该掌握到什么程度

你至少要达到下面标准：

1. 能独立设计常规业务表结构。
2. 能解释索引为什么这样建，而不是只会建主键。
3. 能看懂执行计划并定位常见慢 SQL。
4. 能说清事务、锁和一致性问题。
5. 能把数据库设计和未来 Java 主项目绑定起来。

如果你现在还只是“接口拿到参数就查一下表”，那数据库能力还没有进入大厂后端要求的层级。

## 十一、慢 SQL 不是一句“加索引”就结束

31. 如果线上反馈某个接口越来越慢，你如何判断是不是数据库问题？
答：我不会一上来就说“看下 SQL”。更稳的做法是先拆时间消耗：接口总耗时、应用层处理耗时、下游依赖耗时、数据库执行耗时分别是多少。如果数据库耗时明显偏高，再继续看是单条 SQL 本身慢，还是连接池、锁等待、主从延迟、批量查询数量太多。也就是说，数据库问题首先要先被证据确认，而不是靠经验猜。

32. 一条 SQL 慢时，你第一步会看什么？
答：我通常先看三样：SQL 文本、实际参数、执行计划。因为同一条 SQL 在不同参数下执行路径可能完全不同。只有把“具体 SQL + 真实参数 + explain 结果”放在一起，后面的优化才有意义。

33. `EXPLAIN` 最值得先看哪些字段？
答：最常先看 `type`、`key`、`rows`、`Extra`。`type` 大致反映访问方式，能快速判断是不是全表扫描；`key` 看实际用了哪个索引；`rows` 反映大致扫描量；`Extra` 则经常暴露 `Using filesort`、`Using temporary`、回表或额外过滤等信号。真正重要的不是死记字段名，而是能借它们快速回答“数据库到底在怎么执行”。

34. 什么样的 `EXPLAIN` 结果值得警惕？
答：如果你看到 `type=ALL`、扫描行数明显很大、`Using temporary`、`Using filesort`、没有命中预期索引、或者 where 条件过滤率很低，这些都值得重点警惕。因为它们通常意味着数据库在做大量无效工作，而不是精准命中目标数据。

35. 为什么“建了索引还是慢”是高频真实问题？
答：因为索引是否存在和索引是否真正被高效使用，是两回事。很多慢 SQL 不是完全没索引，而是索引列顺序不对、查询条件没贴合联合索引、返回字段导致大量回表、排序和分页没被索引承接、范围条件把后续列利用截断了。所以成熟的优化从来不是“有没有索引”，而是“索引和真实查询路径是否对齐”。

## 十二、三个高频慢 SQL 案例

36. 案例一：为什么“按状态 + 创建时间倒序分页”经常慢？
答：一个典型场景是：

- 表：`task(id, tenant_id, status, created_at, title, payload, ...)`
- SQL：按 `tenant_id`、`status` 查询最近任务，并按 `created_at desc` 分页。

如果只给 `status` 或 `created_at` 建单列索引，数据库往往还要额外排序，或者先扫描大量记录再过滤。更合理的思路通常是围绕真实查询路径建立联合索引，例如 `(tenant_id, status, created_at desc, id)`，这样既能先按租户和状态过滤，又能尽量利用索引顺序支持排序和分页。

37. 案例二：为什么“模糊查询 + 排序”特别容易失控？
答：因为像 `where title like '%agent%' order by created_at desc` 这种写法，前缀不固定时普通 B+ 树索引往往很难高效利用，数据库很可能先扫大量数据，再做过滤和排序。如果业务上这种搜索需求很高频，就不该硬扛在 `MySQL` 普通索引上，而要评估全文索引、前缀搜索改写、搜索引擎或查询约束收紧。

38. 案例三：为什么深分页会越来越慢？
答：例如 `limit 100000, 20`，数据库经常需要先扫描并跳过前面大量记录，再返回后 20 条。数据量一大，代价会非常明显。更成熟的方案通常是基于游标、主键范围或上一次最后一条记录的有序字段继续翻页，而不是一味用大偏移量 `offset`。

## 十三、索引失效和回表要怎么答深

39. 哪些写法经常导致索引失效？
答：高频例子包括：在索引列上做函数运算或表达式变换、隐式类型转换、联合索引没有遵循最左前缀、前导百分号模糊匹配、范围条件后继续指望后续列完全命中、`or` 条件让优化器放弃理想路径等。面试里如果能随口举出两三个真实例子，会明显比只说“可能索引失效”更有说服力。

40. 为什么回表会让查询变慢？
答：因为数据库先通过二级索引定位到主键，再根据主键回主键索引或数据页取整行数据。如果扫描命中的索引记录很多，而你又返回很多列，就会产生大量额外 IO。回表不是绝对不能接受，但在高频读链路里要特别关注它的成本。

41. 什么时候值得为了覆盖索引改查询字段？
答：当某个接口访问频率很高、延迟要求严格、返回字段其实不多，而当前 SQL 又因为 `select *` 或多余字段导致大量回表时，就很值得只取业务真正需要的列，让查询尽量命中覆盖索引。很多时候性能收益不是靠更复杂技术，而是靠少查无用字段。

42. 联合索引列顺序一般怎么定？
答：通常先放过滤能力强、等值匹配稳定、查询里最常出现的列，再考虑排序列和范围列。但这不是死口诀，还要结合真实 where、order by、group by 的组合模式。成熟回答会强调“按实际查询路径定顺序”，而不是“选择性高的列永远放第一”这种简化说法。

## 十四、事务、锁和死锁案例

43. 为什么很多 SQL 慢其实不是执行慢，而是锁等待？
答：因为一个事务即使语句本身很简单，只要它在等别的事务释放锁，接口层看到的总耗时就会很高。尤其在高并发更新场景里，真正拖慢接口的可能不是扫描行数，而是锁等待、死锁回滚或事务范围过大。

44. 一个典型死锁案例怎么讲？
答：例如事务 A 先更新订单表再更新库存表，事务 B 先更新库存表再更新订单表。如果两边并发执行，就可能互相持有一部分锁并等待对方释放，形成循环依赖。这种题的关键不是说“数据库会检测死锁”，而是主动讲如何减少它：统一资源访问顺序、缩小事务范围、让 where 条件尽量命中索引、减少无谓锁范围。

45. 什么是 next-key lock，为什么面试里常问？
答：在 `InnoDB` 里，为了减少幻读等并发问题，它在某些场景下并不只锁住单条记录，还会连记录之间的间隙一起控制，这就是常见的 next-key lock 语义。面试官爱问它，不是因为想考术语，而是想看你是否理解“为什么明明只改一条记录，别的插入也被影响”的锁范围问题。

46. 死锁和锁等待超时在排查上有什么不同？
答：死锁通常是循环依赖，数据库会主动选择回滚一个事务；锁等待超时则更像一个事务长期拿不到需要的资源，超过等待阈值后失败。排查时，死锁更关注资源获取顺序和并发路径；锁超时更关注长事务、热点更新、索引缺失导致锁范围扩大等问题。

## 十五、主从复制与读写分离的真实代价

47. 为什么读写分离不是免费午餐？
答：因为它虽然能把部分读流量分散到从库，但也会带来复制延迟和读一致性问题。比如用户刚提交成功，立刻去从库读结果却还没同步过来，就会出现“写成功但查不到”的体验问题。所以读写分离不仅是容量问题，也是语义问题。

48. 什么是主从延迟？为什么它重要？
答：主库的写入要复制到从库，如果这个同步存在时间差，就会出现主从延迟。它重要在于很多业务默认以为“刚写完就能立刻读到”，但一旦读走从库，这个假设就不一定成立。后端面试里如果你主动提到“读后写一致性”问题，会比只说主从提升性能更成熟。

49. 如何缓解“刚写完立刻查不到”的问题？
答：常见做法包括：关键链路读主库、写后短时间内定向读主、给用户返回本次写入结果避免立刻回查、在业务上接受短暂最终一致、或者通过缓存与会话级路由做读写一致性优化。重点不是背固定方案，而是根据业务场景决定哪里必须强一致、哪里能接受短暂不一致。

50. 主从复制会影响哪些题目表达？
答：它会影响读写分离、缓存一致性、故障切换、报表查询、异步任务这些多个模块。真正成熟的数据库回答，不能把主从只当成“再加一台从库”，而要讲它对一致性、监控、切换和业务语义的连锁影响。

## 十六、分库分表不是“会说就算会”

51. 什么信号出现时，才值得认真评估分库分表？
答：通常是单库容量、单表数据量、写入热点、索引维护成本、备份恢复时间、单机资源上限已经明显成为瓶颈，而且通过索引优化、冷热分层、归档、缓存、读写分离后仍然顶不住。这时候分库分表才更像必要演进，而不是炫架构。

52. 分库分表前为什么要先问业务访问路径？
答：因为分库分表最怕拆完后才发现核心查询都是跨分片。你必须先看核心 where 条件、主查询路径、排序字段、唯一约束和统计需求，决定分片键怎么选。分片键选错，比“不拆”还痛苦，因为后续跨片查询、迁移和回表成本都会很高。

53. 分片键一般怎么选？
答：通常优先考虑高频查询条件、分布相对均匀、能避免热点、且业务上稳定不变的字段，例如用户 id、租户 id、订单号 hash 等。但它一定要结合业务访问模型看，不能为了“均匀”选一个业务根本不常按它查的字段。

54. 分库分表后最麻烦的事是什么？
答：很多人以为最难的是路由，其实更难的往往是分页、排序、全局唯一 ID、跨分片聚合、事务、一致性、扩容迁移和运维复杂度。真正成熟的回答应该体现“它是架构级成本，不是简单性能技巧”。

## 十七、MySQL 题怎么讲得更像大厂后端

55. 如果面试官让你讲一次慢 SQL 排查，你的标准结构是什么？
答：我会按“现象 -> 定位范围 -> SQL 与参数 -> 执行计划 -> 是否锁等待 -> 业务查询路径 -> 优化动作 -> 结果验证”来讲。比如先说明接口 RT 升高，再通过慢日志或监控定位到具体 SQL，接着结合 `EXPLAIN` 看是否走索引、是否 filesort、是否回表过多，最后说明通过改索引、改查询字段或改分页方式后，延迟和扫描行数如何下降。

56. 如果问“索引设计怎么做”，一个成熟回答框架是什么？
答：先从业务查询路径出发，说明核心 where 条件、排序条件和返回字段；再解释为什么选择某个联合索引顺序；接着补写入成本、覆盖索引、回表和后续演进；最后说明如何用执行计划和真实流量验证索引是否真的有效。这样就把“设计 -> 验证 -> 代价”一条链都讲出来了。

57. 如果问“事务和锁怎么讲才不空”，核心要点是什么？
答：不要只背 ACID 和隔离级别，而要结合一个真实更新场景讲：事务边界为什么要小、并发更新时为什么会锁等待、什么情况下用乐观锁、什么情况下用悲观锁、死锁可能怎么形成、如何通过索引和访问顺序降低冲突。这种回答会明显更像线上工程经验。

58. 学完 MySQL 后，主项目里最值得主动补哪几类内容？
答：至少补三类：核心表结构和联合索引设计；慢 SQL 与高偏移分页优化思路；关键状态流转的一致性与幂等控制。因为这三类内容最容易在项目深挖里被追问，也是最能体现你不是只会接口 CRUD 的地方。

## 十八、B+ 树与索引底层原理

59. B+ 树的具体结构是什么样的？
答：B+ 树是一棵多路平衡搜索树。InnoDB 默认一个节点（页）大小为 16KB。非叶子节点只存储索引键（key）和指向子节点的指针，不存储实际行数据，因此一个非叶子节点能装的 key 数量非常多。以 bigint 主键（8 字节）+ 指针（6 字节）为例，一个 16KB 的非叶子节点大约能容纳 16384 / 14 ≈ 1170 个指针。两层非叶子节点就能索引 1170 × 1170 ≈ 137 万个叶子页。叶子节点存放完整的索引键和对应的数据（聚簇索引存整行、二级索引存主键值），并且所有叶子节点按键值顺序通过双向链表串联。这意味着：树高通常只有 3~4 层，定位任意一条记录最多 3~4 次磁盘 IO；范围查询只需找到起点叶子后沿链表顺序扫描，不用回到上层节点。

60. B+ 树为什么比 B 树更适合数据库磁盘存储？
答：核心区别有三点。第一，B 树的非叶子节点既存 key 也存 data，而 B+ 树非叶子节点只存 key。同样 16KB 的节点，B+ 树能放的 key 更多，树更矮，查询需要的磁盘 IO 次数更少。第二，B 树的数据分散在所有层级的节点中，而 B+ 树的数据全部集中在叶子层。这让全表扫描和范围查询可以直接遍历叶子链表，而不需要中序遍历整棵树。第三，B+ 树叶子节点通过双向链表连接，范围查询（如 `where id between 100 and 500`）找到起点后顺序读取即可，对磁盘的顺序读非常友好；B 树做范围查询需要反复在树的不同层级间跳转，产生大量随机 IO。

61. 聚簇索引和非聚簇索引在物理存储上有什么区别？
答：聚簇索引（Clustered Index）的叶子节点直接存放完整的行数据。InnoDB 的主键索引就是聚簇索引，表数据本身就是按主键顺序组织的 B+ 树，所以一张 InnoDB 表的数据文件（.ibd）本质上就是一棵以主键为 key 的 B+ 树。非聚簇索引（Secondary Index，二级索引）的叶子节点存放的是索引列的值加上对应行的主键值，而不是完整行数据。因此通过二级索引查找时，先在二级索引的 B+ 树中找到目标主键值，再拿这个主键值去聚簇索引的 B+ 树里查找完整行数据——这就是回表。物理上，一张表只能有一个聚簇索引（因为数据只能按一种顺序物理存放），但可以有多个二级索引。

62. 什么是页分裂？它对性能有什么影响？
答：InnoDB 以 16KB 的页（page）为基本存储单位。当一条新记录需要插入到某个已经满了的叶子页时，InnoDB 会把这个页拆成两个页：大约一半的记录留在原页，另一半迁移到新分配的页，然后在父节点中增加一个指向新页的指针。如果父节点也满了，分裂会级联向上传播。页分裂的代价包括：分配新页的磁盘空间、移动大量记录、更新父节点指针、页内出现空洞导致空间利用率下降（分裂后两个页通常都只填充约 50%）。频繁的页分裂还会让数据在物理上不再连续，增加后续范围查询的随机 IO。

63. 为什么自增主键能减少页分裂？
答：因为自增主键保证新插入的记录总是追加到当前最后一个叶子页的末尾。只有当最后一页写满时才会分裂出一个新的空页，这是一种顺序写入模式。原有页保持完全填满，新页从空开始填充，不会出现中间页分裂的问题，空间利用率接近 100%。相比之下，如果用 UUID 或随机字符串做主键，新记录的插入位置是随机的，极大概率落到已经满了的中间页，导致大量页分裂和碎片。这也是为什么 InnoDB 官方文档建议使用单调递增的主键。

64. 如果业务上不得不用非自增主键（比如 UUID），有什么缓解方案？
答：几种常见做法：一是用有序 UUID（如 UUID v7 或 Snowflake ID），它们在时间维度上单调递增，保留了顺序写入的优势；二是把 UUID 作为唯一业务键存在二级索引中，主键仍然用自增 bigint，这样聚簇索引的写入是顺序的，代价只在二级索引上产生部分随机写；三是定期执行 `OPTIMIZE TABLE` 重建表以整理碎片，但这在大表上成本很高，通常只作为离线维护手段。

## 十九、InnoDB 存储引擎架构

65. Buffer Pool 的作用和工作机制是什么？
答：Buffer Pool 是 InnoDB 在内存中开辟的一块缓存区域，用来缓存从磁盘读取的数据页和索引页。当执行查询时，InnoDB 先检查目标页是否已经在 Buffer Pool 中，如果命中就直接读内存，避免磁盘 IO；如果没命中，才从磁盘读入 Buffer Pool 并返回。Buffer Pool 使用的是 LRU（Least Recently Used）的变体算法：它把链表分为 young 区（热数据，约占 5/8）和 old 区（冷数据，约占 3/8）。新读入的页先放到 old 区头部，只有在 old 区停留超过 `innodb_old_blocks_time`（默认 1000ms）后再次被访问，才会晋升到 young 区。这种设计的目的是防止全表扫描一次性把大量冷数据页涌入，把真正的热数据挤出去（即 Buffer Pool 污染）。生产环境通常把 Buffer Pool 设置为可用物理内存的 60%~80%，通过 `innodb_buffer_pool_size` 配置。当 Buffer Pool 中的页被修改后变成脏页（dirty page），脏页不会立刻写回磁盘，而是由后台线程在以下时机刷盘：redo log 空间不足、Buffer Pool 空闲页不够、checkpoint 推进、数据库正常关闭。

66. Change Buffer 解决什么问题？
答：对于非唯一的二级索引，当执行 INSERT、UPDATE、DELETE 时，如果目标索引页不在 Buffer Pool 中，InnoDB 不会立即从磁盘读取该页，而是把变更操作先记录到 Change Buffer 中。等到后续有查询需要读取该索引页时，再把 Change Buffer 中累积的变更合并（merge）到真实页上。这样可以把多次随机磁盘 IO 合并为一次，显著减少二级索引维护的 IO 成本。Change Buffer 只适用于非唯一索引，因为唯一索引在插入时必须立即读取页来做唯一性检查，无法延迟。merge 发生在三种时机：该页被读取到 Buffer Pool 时、后台线程定期合并、数据库关闭时。Change Buffer 的大小可以通过 `innodb_change_buffer_max_size` 配置，默认为 Buffer Pool 的 25%。

67. Redo Log 的作用和写入机制是什么？
答：Redo Log 实现的是 WAL（Write-Ahead Logging）策略：事务提交时，不需要等脏数据页刷回磁盘，只要保证 redo log 已经持久化即可。这样即使数据库崩溃，重启后也能根据 redo log 重放未落盘的修改，保证事务的持久性（Durability）。Redo Log 在磁盘上是一组固定大小的文件（默认 `ib_logfile0` 和 `ib_logfile1`），以环形方式写入：有一个 write pos 指针标记当前写到哪里，一个 checkpoint 指针标记已刷盘的脏页对应的 redo log 位置。write pos 追上 checkpoint 时，必须先推进 checkpoint（把脏页刷盘）才能继续写入。关键参数 `innodb_flush_log_at_trx_commit` 控制刷盘策略：值为 1 时每次事务提交都 fsync redo log 到磁盘，最安全但性能最低；值为 0 时每秒由后台线程刷一次，崩溃最多丢 1 秒数据；值为 2 时每次提交写到 OS page cache 但不 fsync，由操作系统决定何时刷盘，崩溃时如果 OS 没挂则不丢数据，OS 也挂了则最多丢 1 秒。生产环境对数据安全要求高的通常设为 1。

68. Undo Log 的作用是什么？
答：Undo Log 有两个核心作用：保证事务的原子性（Atomicity）和支持 MVCC。对于原子性：事务执行过程中每次修改数据，都会先把修改前的旧值写入 undo log。如果事务回滚，InnoDB 就根据 undo log 把数据恢复到修改前的状态。对于 MVCC：其他事务在读取数据时，如果当前版本不满足其可见性要求，就会沿着 undo log 版本链找到历史版本来读。Undo log 存储在 InnoDB 的回滚段（rollback segment）中，事务提交后，undo log 不会立即删除，因为可能还有其他事务的 MVCC 读需要用到这些历史版本。InnoDB 有一个后台 purge 线程，定期清理所有活跃事务都不再需要的旧版本 undo log。如果存在长事务一直不提交，undo log 就无法被 purge，会导致回滚段膨胀、磁盘空间持续增长。

69. Doublewrite Buffer 解决什么问题？
答：InnoDB 的数据页是 16KB，但操作系统和磁盘的原子写入单位通常是 4KB（一个文件系统 block）。如果数据库在把一个 16KB 的页写到磁盘的过程中崩溃，可能只写了前 4KB 或 8KB，产生所谓的 partial write（部分写），这时候这个页既不是旧数据也不是新数据，redo log 也没法修复它（因为 redo log 记录的是对页的增量修改，前提是基础页本身是完整的）。Doublewrite Buffer 的解决方案是：在把脏页写到真正的数据文件之前，先把它们写到一个专门的 doublewrite 区域（在系统表空间中，连续的 2MB 区域）。写入 doublewrite 区域成功后，再把页写到实际的数据文件位置。如果第二步写入时崩溃，恢复时就从 doublewrite 区域找到完整的页副本，先修复数据文件中的损坏页，然后再用 redo log 继续恢复。

70. 一条 UPDATE 语句在 InnoDB 中的完整执行流程是什么？
答：以 `UPDATE user SET name='B' WHERE id=1` 为例，完整流程如下。（1）连接层：客户端发来 SQL，经过连接器鉴权，查询缓存（MySQL 8.0 已移除），交给解析器生成语法树，优化器生成执行计划，交给执行器。（2）读取目标页：执行器调用 InnoDB 引擎接口，InnoDB 先在 Buffer Pool 中查找 id=1 所在的数据页；如果不在，从磁盘读入 Buffer Pool。（3）加锁：对 id=1 这条记录加排他锁（X Lock）。（4）写 Undo Log：把修改前的旧值（name='A'）写入 undo log，记录回滚指针。（5）修改内存页：在 Buffer Pool 中把 name 从 'A' 改为 'B'，此时该页变成脏页。（6）写 Redo Log（prepare）：把本次修改的物理变化写入 redo log buffer，标记状态为 prepare。（7）写 Binlog：把这条 UPDATE 写入 binlog cache。（8）提交——redo log 标记 commit：redo log 从 prepare 状态改为 commit 状态，同时 binlog 落盘（fsync）。至此事务提交完成，但脏页不一定立即刷回磁盘，会由后台线程在合适时机通过 Doublewrite Buffer 刷盘。

## 二十、MVCC 实现原理

71. InnoDB 每行数据的隐藏列有哪些？
答：每行数据除了用户定义的列之外，InnoDB 还会自动维护三个隐藏列。第一个是 `DB_TRX_ID`（6 字节），记录最后一次插入或修改该行的事务 ID。第二个是 `DB_ROLL_PTR`（7 字节），回滚指针，指向 undo log 中该行的上一个版本。第三个是 `DB_ROW_ID`（6 字节），如果表没有定义主键也没有非空唯一索引，InnoDB 会自动生成这个隐藏的自增行 ID 作为聚簇索引的 key。其中 `DB_TRX_ID` 和 `DB_ROLL_PTR` 是 MVCC 机制的核心支撑。

72. Undo Log 版本链是怎么工作的？
答：每次对一行数据执行 UPDATE 或 DELETE，InnoDB 都会把旧版本的数据写入 undo log，并通过 `DB_ROLL_PTR` 将新旧版本串成一条链表。举个具体例子：id=1 这行，初始值 name='A'，由事务 100 插入，此时 `DB_TRX_ID=100`，`DB_ROLL_PTR` 指向一个 insert undo log。然后事务 200 把 name 改为 'B'，InnoDB 先把旧版本（name='A', TRX_ID=100）写入 undo log，然后把当前行改为 name='B'、`DB_TRX_ID=200`、`DB_ROLL_PTR` 指向刚写入的那条 undo log。接着事务 300 再把 name 改为 'C'，同理旧版本（name='B', TRX_ID=200）被写入 undo log，当前行变为 name='C'、TRX_ID=300。最终 undo log 版本链为：当前行(C, trx=300) → undo(B, trx=200) → undo(A, trx=100)。当其他事务做快照读时，就沿着这条链找到自己可见的版本。

73. ReadView 是什么？它包含哪些关键信息？
答：ReadView 是事务在执行快照读（普通 SELECT）时生成的一个"可见性快照"，用于判断版本链上的哪个版本对当前事务可见。一个 ReadView 包含四个关键信息：（1）`m_ids`：生成 ReadView 时所有正在活跃（已开启但未提交）的事务 ID 列表。（2）`min_trx_id`：`m_ids` 中的最小值，即活跃事务中最早开启的那个事务 ID。（3）`max_trx_id`：生成 ReadView 时系统即将分配的下一个事务 ID（即当前最大事务 ID + 1），注意它不是 `m_ids` 的最大值。（4）`creator_trx_id`：创建该 ReadView 的事务自身的 ID。

74. ReadView 的可见性判断规则具体是怎样的？
答：拿到版本链上某个版本的 `DB_TRX_ID` 后，按以下规则判断。规则一：如果 `DB_TRX_ID == creator_trx_id`，说明是自己修改的，可见。规则二：如果 `DB_TRX_ID < min_trx_id`，说明该版本在 ReadView 生成之前就已经提交了，可见。规则三：如果 `DB_TRX_ID >= max_trx_id`，说明该版本是在 ReadView 生成之后才开启的事务产生的，不可见。规则四：如果 `min_trx_id <= DB_TRX_ID < max_trx_id`，则进一步检查 `DB_TRX_ID` 是否在 `m_ids` 列表中——如果在，说明该事务在 ReadView 生成时还活跃未提交，不可见；如果不在，说明该事务在 ReadView 生成前已经提交了，可见。如果当前版本不可见，就沿着 `DB_ROLL_PTR` 找到 undo log 中的上一个版本，重复判断，直到找到可见版本或链尾为止。

75. ReadView 在 RC 和 RR 隔离级别下有什么区别？
答：区别在于 ReadView 的生成时机。在 Read Committed（RC）下，每次执行 SELECT 都会重新生成一个新的 ReadView，所以如果两次 SELECT 之间有其他事务提交了修改，第二次 SELECT 就能看到新值——这就是"不可重复读"的根源。在 Repeatable Read（RR）下，事务只在第一次执行快照读（SELECT）时生成 ReadView，后续所有 SELECT 复用同一个 ReadView。即使其他事务在此期间提交了修改，当前事务也看不到，因为判断可见性的标准始终是第一次 SELECT 时的那个快照。

76. 用一个具体例子说明 RR 下 MVCC 如何避免不可重复读？
答：假设表中有一行 id=1, name='A'，由事务 50 提交。现在有三个事务按以下顺序执行。事务 100（RR 级别）：执行第一次 `SELECT * FROM t WHERE id=1`，此时生成 ReadView，`m_ids=[]`，`min_trx_id=101`，`max_trx_id=101`。当前行 `DB_TRX_ID=50 < 101`，可见，读到 name='A'。事务 200：执行 `UPDATE t SET name='B' WHERE id=1` 并提交。此时当前行变为 name='B'，`DB_TRX_ID=200`，undo log 中保留旧版本 (name='A', TRX_ID=50)。事务 100 再执行第二次 `SELECT * FROM t WHERE id=1`：因为是 RR 级别，复用之前的 ReadView（`max_trx_id=101`）。当前行 `DB_TRX_ID=200 >= 101`，不可见。沿 undo log 版本链找到上一版本 (name='A', TRX_ID=50)，`50 < 101`，可见。所以第二次读到的仍然是 name='A'。两次读取结果一致，不可重复读被避免。如果是 RC 级别，第二次 SELECT 会生成新的 ReadView，此时事务 200 已提交不在 `m_ids` 中，`DB_TRX_ID=200` 会被判断为可见，读到 name='B'，两次结果不同，即产生了不可重复读。

## 二十一、锁的底层实现

77. InnoDB 有哪些行级锁类型？
答：InnoDB 在行级别上有三种核心锁。（1）Record Lock（记录锁）：锁住索引中的一条确切记录。比如 `WHERE id=5`，在唯一索引上精确匹配时，只锁 id=5 这条记录。（2）Gap Lock（间隙锁）：锁住两条记录之间的间隙，但不包括记录本身。比如索引中有 id=3 和 id=7，Gap Lock 可以锁住 (3, 7) 这个区间，阻止其他事务在这个间隙中插入新记录。Gap Lock 之间不互斥（两个事务可以同时持有同一个间隙的 Gap Lock），它只和 INSERT 操作冲突。（3）Next-Key Lock（临键锁）：Record Lock + Gap Lock 的组合，锁住一条记录及其前面的间隙。比如锁住 (3, 7]，即间隙 (3, 7) 加上记录 7 本身。Next-Key Lock 是 InnoDB 在 RR 隔离级别下的默认加锁单位。

78. 为什么 RR 级别下需要 Gap Lock？
答：为了防止幻读。幻读的场景是：事务 A 执行 `SELECT * FROM t WHERE age=25`，读到 3 条记录；事务 B 此时插入一条 age=25 的新记录并提交；事务 A 再执行同样的 SELECT，读到 4 条记录——多出来一条"幻影"行。MVCC 的快照读可以通过 ReadView 避免幻读，但当前读（`SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`）必须读最新数据，无法依赖快照。所以 InnoDB 在当前读时使用 Gap Lock 锁住记录间的间隙：如果 age 索引中现有 age=20 和 age=30 两条记录，查询 age=25 时会对 (20, 30) 这个间隙加锁，其他事务试图往这个间隙插入 age=25 的记录时会被阻塞，从而防止幻读。

79. InnoDB 的加锁规则是什么？
答：在 RR 级别下，加锁遵循以下核心规则。规则一：加锁的基本单位是 Next-Key Lock，即左开右闭区间。规则二：查找过程中访问到的对象才会加锁。规则三：等值查询唯一索引，精确命中时 Next-Key Lock 退化为 Record Lock——因为唯一索引保证不会有相同值的记录插入，不需要锁间隙。规则四：等值查询非唯一索引，命中记录后，向右遍历到第一个不满足条件的记录，该 Next-Key Lock 退化为 Gap Lock（不锁最后那条不等值的记录本身）。规则五：范围查询，对扫描范围内的所有记录加 Next-Key Lock，范围边界处不退化。举个例子：索引中有 id=5, 10, 15, 20。执行 `SELECT * FROM t WHERE id=10 FOR UPDATE`，因为是唯一索引等值查询且命中，只加 id=10 的 Record Lock。如果执行 `SELECT * FROM t WHERE id > 10 AND id < 18 FOR UPDATE`，会加 (10, 15] 和 (15, 20] 的 Next-Key Lock。

80. 意向锁（IX/IS）解决什么问题？
答：意向锁是表级锁，用于快速判断表级锁和行级锁的兼容性。当事务要对某行加行级共享锁（S Lock）时，会先在表级加一个意向共享锁（IS）；要加行级排他锁（X Lock）时，先在表级加一个意向排他锁（IX）。这样当另一个事务想对整张表加表级 S 锁或 X 锁时，只需要检查表上是否已有 IX 或 IS 锁，就能立即判断是否有行级锁冲突，而不需要逐行遍历检查。兼容矩阵：IS 与 IS 兼容、IS 与 IX 兼容、IX 与 IX 兼容；表级 S 锁与 IX 不兼容、表级 X 锁与 IS/IX 都不兼容。意向锁之间永远不冲突，它们只用来和表级锁做快速判断。

81. InnoDB 如何检测和处理死锁？
答：InnoDB 使用 wait-for graph（等待图）进行死锁检测。每当一个事务请求锁而被阻塞时，InnoDB 会在等待图中添加一条边（从等待者指向持有者）。后台会立即检查图中是否存在环——如果存在环，说明发生了死锁。检测到死锁后，InnoDB 会选择回滚"代价最小"的事务（通常选择 undo log 量最少的事务，即修改数据最少的），被回滚的事务会收到错误码 1213（Deadlock found when trying to get lock）。另外可以通过 `innodb_deadlock_detect=OFF` 关闭主动检测（在极高并发下检测本身可能消耗大量 CPU），改为依赖 `innodb_lock_wait_timeout`（默认 50 秒）来处理，但这样死锁的响应会更慢。

82. 能举一个因为间隙锁导致死锁的具体场景吗？
答：假设表 t 有 id=5 和 id=10 两条记录。事务 A 执行 `SELECT * FROM t WHERE id=8 FOR UPDATE`，id=8 不存在，InnoDB 对间隙 (5, 10) 加 Gap Lock。事务 B 执行 `SELECT * FROM t WHERE id=6 FOR UPDATE`，id=6 也不存在，同样对间隙 (5, 10) 加 Gap Lock——Gap Lock 之间不互斥，所以两个事务都成功获得 Gap Lock。然后事务 A 试图 `INSERT INTO t VALUES(7, ...)`，需要在间隙 (5, 10) 中插入，但发现事务 B 持有该间隙的 Gap Lock，于是事务 A 等待。事务 B 也试图 `INSERT INTO t VALUES(9, ...)`，同样要在 (5, 10) 中插入，发现事务 A 持有 Gap Lock，于是事务 B 也等待。两者互相等待对方释放 Gap Lock，形成死锁。InnoDB 检测到后回滚其中一个。这个场景说明 Gap Lock 虽然防止了幻读，但在某些插入模式下会带来额外的死锁风险，需要通过业务层面避免在同一间隙内的并发插入。

## 二十二、Binlog 与两阶段提交

83. Binlog 的三种格式分别是什么？各有什么优缺点？
答：（1）STATEMENT 格式：记录原始 SQL 语句。优点是日志量小；缺点是某些非确定性函数（如 `NOW()`、`UUID()`、`RAND()`）在主从上执行结果可能不同，导致数据不一致。（2）ROW 格式：记录每一行数据的实际变化（修改前后的完整行内容）。优点是精确还原数据变化，不存在非确定性问题，是主从复制最安全的格式；缺点是对于批量操作（如 `UPDATE t SET status=1 WHERE tenant_id=100` 影响 10 万行），日志量会非常大。（3）MIXED 格式：默认使用 STATEMENT，当 MySQL 判断 STATEMENT 可能导致不一致时自动切换为 ROW。看起来两全其美，但实际上判断逻辑不总是可靠，生产环境大多数团队直接使用 ROW 格式以确保一致性。

84. 为什么 Redo Log 和 Binlog 需要两阶段提交？
答：因为 redo log 是 InnoDB 引擎层的日志，binlog 是 MySQL Server 层的日志，它们是两个独立的日志系统。如果不做协调，在崩溃恢复时就会出现不一致。场景一：先写 redo log 再写 binlog——如果 redo log 写成功后崩溃，binlog 还没写，重启后 InnoDB 根据 redo log 恢复了这次修改（数据存在），但 binlog 中没有这条记录，从库通过 binlog 同步后就少了这次修改，主从数据不一致。场景二：先写 binlog 再写 redo log——如果 binlog 写成功后崩溃，redo log 还没写，重启后 InnoDB 没有恢复这次修改（数据不存在），但 binlog 中已经有了，从库会多出这次修改，同样主从不一致。两阶段提交就是为了保证这两个日志的原子性——要么都有，要么都没有。

85. 两阶段提交的具体流程是什么？
答：分为三步。（1）Prepare 阶段：InnoDB 把 redo log 写入磁盘（fsync），并把该事务在 redo log 中的状态标记为 prepare。此时数据修改已经通过 redo log 持久化了，但事务还没有真正提交。（2）写 Binlog：MySQL Server 层把 binlog 写入磁盘（fsync）。（3）Commit 阶段：InnoDB 把 redo log 中该事务的状态从 prepare 改为 commit。至此事务正式提交完成。整个过程中，redo log 的 prepare 标记和 binlog 的存在性共同构成了事务提交的完整证据。

86. 崩溃恢复时如何根据两阶段状态判断事务是否该恢复？
答：MySQL 重启后会扫描 redo log 中处于 prepare 状态的事务，然后逐一去 binlog 中查找是否存在对应的完整记录（通过事务的 XID 关联）。如果 binlog 中有完整记录：说明 binlog 已经落盘，即使 redo log 还停在 prepare，也应该把该事务提交（补一个 commit 标记），因为从库可能已经通过 binlog 同步了这笔数据，不提交会导致主从不一致。如果 binlog 中没有对应记录：说明事务在写 binlog 之前或过程中崩溃了，应该回滚该事务，用 undo log 撤销已经做的修改。这个机制保证了无论在哪个步骤崩溃，恢复后 redo log 和 binlog 的状态始终一致。

87. 两阶段提交有什么性能问题？有哪些优化？
答：两阶段提交要求每次事务提交至少执行两次 fsync（一次 redo log、一次 binlog），磁盘 fsync 是很重的 IO 操作，高并发下会成为瓶颈。MySQL 5.6 之后引入了 Group Commit（组提交）优化：把多个并发事务的 fsync 操作合并成一次，大幅减少 IO 次数。具体实现是把提交过程拆成三个队列阶段（flush、sync、commit），每个阶段由一个 leader 线程收集一批事务后统一执行，后续事务只需等待 leader 完成即可。组提交能让 IOPS（每秒 IO 次数）的利用率大幅提升，是 MySQL 高吞吐写入的关键优化之一。

## 二十三、分库分表工程实践

> 说明：本文前部已有「十七、MySQL 题怎么讲得更像大厂后端」，本专题在文末独立展开分库分表工程细节，章节序号顺延为二十三。

1. 水平分库 vs 水平分表 vs 垂直拆分的区别？
答：**水平分表**是同一个库内把一张大表按某个键拆成多张小表（如 `order_0`～`order_15`），共享同一实例的连接池与存储引擎配置，主要缓解单表过大带来的 B+ 树深度、统计与维护成本。**水平分库**是把不同分片的数据落到**不同数据库实例**上，除了数据分区外还带来连接、容量、故障域的隔离，扩容时往往涉及路由与运维层面的变更。**垂直拆分**是按业务域拆库拆表（用户库、订单库、商品库），把不同访问模式与 SLA 的数据拆开，减少宽表、减少无关列的 IO。工程上常见路径是：**先垂直按业务解耦**，再对**数据量与 QPS 明显集中**的表做水平拆分。

2. 分片键选择的核心原则？
答：核心是三点兼顾——**查询覆盖率高**（线上核心 SQL 的 `WHERE` 条件能稳定命中分片键，避免广播扫全分片）、**分布尽量均匀**（避免热点分片拖垮集群）、**值稳定**（分片键在生命周期内不应频繁变更，否则等于数据搬迁）。常见合理选择：`user_id`（C 端按人查询）、`tenant_id`（B 端多租户隔离）、`order_id` 或与订单强绑定的业务键（交易链路）。典型**错误**选择：纯时间字段（新数据永远挤在「最新」分片，历史分片变冷但不均）、低基数字段如状态（枚举值少导致严重倾斜）。

3. 分片算法有哪些？
答：
- **Hash 取模**：`hash(shardingKey) % N`，数据分布通常较均匀；缺点是**扩容改 N** 时往往要**全量 rehash 迁移**，停机或双写成本高。
- **Range 范围**：按主键区间或时间范围划分分片，**加新分片**即可承接新区间，扩容路径清晰；缺点是**尾部热点**（例如永远往「当前月」分片写）。
- **一致性哈希**：哈希环 + **虚拟节点**，扩容时只迁移环上相邻区间对应的数据，迁移量可降为总量的约 `1/N` 量级（与实现有关）。
- **查表法**：单独维护一张「分片键 → 分片编号/实例」的路由表（或配置中心），变更是改映射而非改算法；灵活但**多一次路由查询**或依赖缓存，需考虑路由表自身的高可用与一致性。

4. ShardingSphere 架构？
答：Apache ShardingSphere 常见两种接入形态：**ShardingSphere-JDBC** 以 JAR 嵌入应用进程，在客户端做 SQL 解析与改写，**无额外网络 hop**，延迟通常最低，适合 Java 单体或 Spring 服务。**ShardingSphere-Proxy** 是独立进程、**兼容 MySQL 协议**的代理，业务侧用普通 JDBC/驱动连代理即可，**多语言**与遗留系统接入成本低。核心流水线可概括为：**SQL 解析** → **路由**（按分片规则绑定到具体数据源/表）→ **SQL 改写**（逻辑表名替换为物理表名，必要时拆成多条语句）→ **执行**（单分片串行、多分片可并行）→ **结果归并**（排序、聚合、分页、去重等）。面试可顺带提：归并阶段往往是跨分片查询的性能与复杂度来源。

5. 分库分表后分页查询怎么做？
答：若 `ORDER BY` 的排序维度与分片键一致且查询能**路由到单分片**，则与普通单库分页无异。**跨分片**时，经典 `ORDER BY x LIMIT offset, limit` 往往需要：各分片各取「足够多」的行（工程上常讨论取 `offset + limit` 或更多），在中间层**内存归并排序**后再截断——`offset` 很大时各分片返回量爆炸，**延迟与内存**都会恶化。可行优化包括：**禁止深分页**（产品层改为「上一页/下一页」的游标，用 `(last_sort_key, last_id)` 条件替代大 offset）、**排序键与分片键对齐**（使热点分页仍落单分片）、**二次查询/全局索引表**（先在各分片取边界再精确拉回，或与 ES 等配合），以及**按用户/租户**强制单分片分页。

6. 分库分表扩容迁移怎么做？（不停服）
答：工程上常用 **「双写 + 校验 + 灰度读切 + 停旧写」** 闭环：（1）新集群与路由规则就绪，对增量**双写**（旧为主、新为从或双主以实际架构为准），保证新集群持续有写入；（2）**历史数据全量迁移**（`mysqldump`、物理备份、或 Canal/DataX 等），期间增量靠双写或 binlog 同步补齐；（3）**对账**（抽样校验、行数校验、checksum、业务抽检）修补差异；（4）**读流量灰度**到新集群（先部分租户/部分读接口），观测延迟与错误率；（5）确认一致后**停止旧库写入**，切换路由，保留旧集群只读窗口以便回滚。**翻倍扩容**（如分片数 2→4→8）配合取模规则设计，可使部分数据「只在本分片与对称分片间」迁移，减少全量搬移比例（本质是重新映射后仅部分 key 跨越边界）。

7. 全局唯一 ID 在分库分表中的方案？
答：单库**自增**在分库后不再全局唯一。常见方案：**Snowflake（雪花）**——64 位：`1` 符号 + `41` 时间戳 + `10` 机器/数据中心 + `12` 序列，**趋势递增**、本地生成、吞吐高；需治理时钟回拨与 workerId 分配。**号段（Leaf 等）**——从 DB/中间件**批量取号段**缓存在应用内存，双 buffer 预取，减少 DB 往返；号段耗尽再申请。**UUID**——唯一但无序，作聚簇主键易导致 B+ 树页分裂与索引膨胀，一般更适合业务侧唯一键而非主键。**Redis `INCR`**——实现简单，强依赖 Redis 高可用与持久化策略，需评估故障与脑裂场景下的重复号风险。
