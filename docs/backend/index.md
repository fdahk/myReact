# Java 后端知识库总索引

## 使用方式

这套文档不是百科，也不是零散笔记，而是按“冲击顶级大厂 Java 后端岗位”来组织的训练型知识库。

你使用它时要一直记住四件事：

- 先搭后端认知地图，再逐个专题深入，不要碎片化乱学。
- 每个知识点都必须同时回答：是什么、为什么、怎么落地、面试里怎么讲。
- 不只背定义，必须把知识点和项目、场景题、系统设计、线上故障排查串起来。
- 当前主线先打透 `Java` 后端主栈，先不把精力发散到 `Go` 或泛化到所有语言。

---

## 一、这套知识库要达到什么标准

你的目标不是“会写几个 `Spring Boot` 接口”，而是逐步补齐下面这五层能力：

1. 业务交付能力：能设计接口、表结构、权限、日志、异常、上线回滚。
2. 工程质量能力：能把代码写到可维护、可测试、可扩展、可排障。
3. 性能与稳定性能力：能分析慢 SQL、缓存问题、线程问题、超时与雪崩。
4. 分布式系统能力：能处理幂等、一致性、消息、服务治理、限流熔断。
5. 架构与表达能力：能做系统设计、容量评估、方案取舍、项目深挖表达。

---

## 二、训练验收标准

如果要把这套知识库做到接近 `docs/前端面试` 的完成度，后端侧至少要满足下面这些量化标准：

### 基础题库标准

- `Java`、`JVM`、`Spring`、`MySQL`、`Redis`、`MQ`、网络、`Linux`、稳定性九大模块都要能单独复习。
- 每个核心专题都要沉淀出 `30~60` 个可口述高频问题。
- 整体后端高频题卡片数，目标做到 `300+`。

### 系统设计标准

- 至少能完整训练 `8~12` 道高频后端系统设计题。
- 每道题都要覆盖：需求澄清、核心模型、数据流、瓶颈、稳定性、演进路线、trade-off。
- 每道题至少准备 `5~10` 个追问点。

### 场景题标准

- 至少形成 `80+` 个高频排障/优化/设计场景。
- 场景题不能只给结论，必须有“现象 -> 假设 -> 验证 -> 止血 -> 治理”链路。

### 项目标准

- 至少有 `1` 个能够支撑 `15~20` 分钟深挖的 `Java` 主项目。
- 主项目必须覆盖：数据库设计、缓存、异步、日志/审计、权限、部署、可观测性中的多数能力。
- 至少有 `2~3` 个量化结果，例如接口耗时下降、慢查询下降、缓存命中率提升、故障恢复时间缩短。

### 面试准备标准

- 自我介绍、项目介绍、系统设计、故障排查都要有结构化模板。
- 至少完成 `8+` 次模拟面试或等价强度复盘。
- 基础题不能只会答“定义”，要能顶住二轮三轮追问。
- 除算法题外，还要补一条“后端手写题训练线”，重点覆盖缓存、并发、限流、线程池、幂等等工程型实现题。

## 六、每周执行模板

- 每天至少 `1.5~2.5` 小时。
- `1` 小时系统学一个核心专题。
- `0.5~1` 小时复述题卡、整理追问、补薄弱点。
- `0.5~1` 小时把当天知识点映射到主项目或系统设计题。

### 周末

- 至少完成 `1` 次系统设计题训练。
- 至少完成 `1` 次场景题/排障题训练。
- 至少输出 `1` 份专题小结，内容覆盖“概念、原理、场景、项目映射、面试表达”。

## 七、审核闭环

为了把这套知识库真正做到“广且深”，后续优化默认按下面的闭环推进：

1. 先补齐知识面缺口，把一级专题和高频专题补全。
2. 再对现有主干文档做深度审查，找出追问链、场景密度、排障深度不足的部分。
3. 按“高频基础 -> 分布式/工程 -> 场景题 -> 系统设计 -> 项目表达”的顺序反复迭代。
4. 每一轮都检查两个问题：
   - 有没有新增知识面盲区。
   - 已有专题能不能支撑更深的追问和真实项目表达。

## 八、判断自己是否学到位

如果一个知识点你只会定义，不算学会。

至少要满足下面三条中的两条，才算真正掌握：

1. 能解释机制、边界条件和关键 trade-off。
2. 能讲出真实业务场景、常见错误做法和更合理方案。
3. 能结合主项目说明为什么这样设计、出了问题怎么排查、未来如何演进。

例如：

- 学 `Redis`，不能只会说它快，还要会讲缓存穿透、击穿、雪崩、热点 key、一致性和集群风险。
- 学 `MySQL`，不能只会说索引重要，还要能讲执行计划、回表、锁、事务、死锁和慢查询治理。
- 学 `Spring`，不能只会写注解，还要理解 `IOC`、`AOP`、事务管理和自动装配解决了什么工程问题。
- 学 `MQ`，不能只会说异步削峰，还要会讲重复消费、积压、顺序、死信和可靠消息。
