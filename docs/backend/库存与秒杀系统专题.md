# 库存与秒杀系统专题

## 使用方式

这篇补的是大厂后端里非常经典的一类题：

- 库存为什么不是一条 `update stock = stock - 1` 就结束
- 秒杀系统为什么天然考热点治理、削峰、限流、幂等和一致性
- 预扣库存、最终扣减、超卖防护、排队和异步化分别解决什么问题

如果你目标是顶级后端岗位，这类题基本绕不过去。

---

## 一、为什么库存和秒杀是高频后端题

1. 为什么库存系统特别能体现后端深度？
答：因为它表面看只是扣减一个数字，实际背后却有并发竞争、热点冲击、超卖风险、支付竞态、订单回滚、缓存一致性和补偿问题。它非常适合检验候选人是否真正理解“高并发下正确性优先”。

2. 为什么秒杀题常被当作系统设计经典题？
答：因为秒杀天然把流量洪峰、热点商品、请求风暴、库存强约束、黄牛刷单、幂等、防重、削峰填谷和异步化全部压在一起。你能把秒杀讲清楚，很多高并发后端能力也就一起展示出来了。

---

## 二、库存链路最核心的 6 个问题

3. 第一个问题是什么？
答：库存语义。到底是现货库存、可售库存、冻结库存、预占库存还是已扣减库存。很多系统一开始只有一个 `stock` 字段，后面就会在支付、取消、超时关单和退款场景里越来越混乱。

4. 第二个问题是什么？
答：超卖防护。多个请求同时抢同一份库存时，如果没有原子扣减、版本控制或串行化处理，就可能出现库存被扣成负数或订单创建成功但实际无货。

5. 第三个问题是什么？
答：预扣和最终扣减。很多交易系统不会在下单瞬间直接“最终消耗库存”，而是先预占，再在支付成功后确认，超时未支付则释放。这样能在正确性和用户体验之间取得平衡。

6. 第四个问题是什么？
答：热点治理。秒杀场景不是所有商品都热，往往是极少数热点 SKU 吞掉大部分流量。真正难的不是整个系统吞吐，而是单热点键、单热点商品、单库存记录如何扛住冲击。

7. 第五个问题是什么？
答：请求削峰。真正的大流量场景通常不会让所有请求直接打到数据库扣库存，而会先在网关、缓存、队列或本地令牌层做限流、排队和预过滤，尽量把无效流量挡在前面。

8. 第六个问题是什么？
答：回补和补偿。超时关单、支付失败、订单取消、退款回滚、消息失败都会影响库存回补。如果没有明确补偿策略，库存就会越来越不准。

---

## 三、秒杀系统主链路怎么讲

9. 一个成熟的秒杀链路应该怎么讲？
答：先在入口层做限流、防刷和活动校验；再在缓存或内存层做快速资格判断；通过队列或异步化把真正的下单请求削峰；在库存层做原子预扣或串行扣减；成功后异步创建订单或支付单；最终由支付结果和超时任务共同收敛库存状态。这样就把热点流量和核心一致性分层处理了。

10. 为什么不能让所有秒杀请求直接打数据库？
答：因为数据库最擅长正确存储，不擅长充当前置流量闸门。秒杀时海量无效流量如果直接打到数据库，不等你解决一致性，系统就先被打崩了。

11. 为什么队列在秒杀里很有价值？
答：因为队列能帮你做削峰、限速和异步化，把瞬时洪峰摊平到系统可承受的速度。但要注意，队列只解决流量形状问题，不自动解决库存正确性问题。

---

## 四、预扣库存和最终一致性怎么讲

12. 为什么很多交易系统喜欢预扣库存？
答：因为它能在用户下单时先占住资源，减少支付成功后无货的情况。同时又给了系统一个“支付失败或超时再释放”的回旋空间。它比单纯最终扣减更稳，比立即硬扣更灵活。

13. 预扣库存最容易出什么问题？
答：最常见是库存长期冻结不释放、支付和关单竞态、重复释放、库存回补失败和补偿任务漏跑。也就是说，预扣只是第一步，真正难的是后续状态治理。

14. 为什么库存题几乎一定会连到消息和补偿？
答：因为库存变化往往会触发订单、支付、通知、搜索索引、活动统计等后续动作，而这些动作大多不能和库存扣减放在一个事务里。于是你就必须思考异步消息、最终一致性和补偿机制。

---

## 五、热点、防刷和黄牛问题怎么讲

15. 为什么秒杀题不能只讲技术，还要讲风控？
答：因为真实秒杀里很多压力根本不是正常用户产生的，而是脚本、黄牛、刷单和重复请求。如果没有验证码、行为风控、设备指纹、接口限流和资格校验，再好的库存设计也会被脏流量淹没。

16. 为什么热点商品要单独治理？
答：因为热点 SKU 往往集中冲击一个库存键、一个缓存键和一条数据库记录。常见做法包括热点缓存、本地令牌、库存分片、活动资格预发、请求预热和热门商品走专门链路。

## 六、和主项目怎么绑定

17. `agent-server-java` 这种项目怎么借库存/秒杀系统的方法论？
答：虽然它不是电商，但很多核心思想一样，例如热点资源配额、稀缺算力额度、高并发任务抢占、并发任务资格控制、预占资源和超时释放。这些都能借库存系统的思路来表达。

18. 如果把项目做成商业化 AI 平台，最自然的映射是什么？
答：高价值模型配额抢占、并发算力槽位、限量资源预约、企业套餐资源池、任务预占与超时释放。这样就能把库存系统的经验自然迁移到平台治理能力。

### 秒杀系统完整架构

1. 秒杀系统分层应该怎么讲清楚？
答：典型分层是「前端静态化 + CDN → 网关限流 → 应用层排队 → 库存扣减 → 下单」。静态页与资源走 CDN，减轻源站读压力；网关上做全局限流、防刷与活动校验，挡住大部分无效或恶意流量；应用层通过队列、本地缓冲或排队模型控制进入核心写路径的速率；库存层在 Redis 预扣或数据库条件更新中完成原子扣减；最后创建订单并进入支付。这样把读流量、非法流量与强一致写路径拆开，系统才扛得住洪峰。

2. Redis 预扣库存的 Lua 脚本长什么样？为什么要用 Lua？
答：下面是一个「有库存则递减并返回成功」的极简示例：

```lua
local stock = tonumber(redis.call('get', KEYS[1]))
if stock and stock > 0 then
    redis.call('decr', KEYS[1])
    return 1
end
return 0
```

使用 Lua 是因为 Redis 对脚本保证执行期间不被其他命令插队，「读库存 → 判断 → 递减」在一个原子单元内完成；若在应用里先 `GET` 再 `DECR`，中间可能被其他请求插入，出现超卖或误判。Lua 把这段逻辑封装成单次原子操作，避免竞态。

3. 预扣库存和「最终扣减」分别指什么？
答：Redis（或缓存）上的预扣主要是快速「占坑」、挡流量，表示「这一瞬间有资格继续往下走」；真正的库存落账往往在数据库用条件更新完成，例如 `UPDATE stock SET quantity = quantity - 1 WHERE id = ? AND quantity > 0`。预扣解决的是洪峰下的吞吐与无效请求过滤，最终扣减解决的是持久化与账务层面的正确性，两者要配套补偿与对账。

4. 乐观锁防超卖怎么讲？还需要 `SELECT FOR UPDATE` 吗？
答：`WHERE quantity > 0` 配合「`quantity = quantity - 1`」的更新，若库存不足则影响行数为 0，等价于用「当前数量」作为乐观条件，避免把库存扣成负数；这是不额外加 version 列时常见的乐观并发写法。秒杀场景往往优先用这种单行条件更新，而不是长时间占用的 `SELECT FOR UPDATE` 悲观锁，以减少锁竞争；若业务需要更强串行语义，再单独评估隔离级别与显式锁策略。

5. 消息队列异步下单怎么配合前端体验？
答：预扣或资格校验成功后，将「创建订单」等重操作封装为 MQ 消息，消费者异步写库、推进状态；同步接口尽快返回（例如返回「处理中」与业务单号）。前端通过轮询订单状态或推送通道拿到「已创建 / 待支付」等终态，避免海量线程阻塞在数据库写入上，实现削峰。注意消费者必须幂等，防止重复消息产生重复订单。

6. 令牌机制解决什么问题？
答：活动开始前或活动期内预生成有限数量的下单令牌（或资格凭证），只有拿到令牌的用户才能进入真正的下单接口。这样把「谁能下单」与「库存扣减」拆开，在源头削减无效请求与脚本冲击，常与用户维度限流、风控组合使用。

7. 库存不足或服务过载时如何降级？
答：在判断活动结束、库存为 0 或系统过载时，可在 CDN、网关或静态资源层直接返回「已售罄」静态页或统一 JSON，不让请求继续打到核心应用与数据库。降级策略应可配置、可观测，并保留监控与埋点便于复盘。

### 库存扣减一致性设计

1. Redis 和数据库双写时一致性怎么保证？
答：常见路径是：先在 Redis 预扣占位，下单链路成功后再在数据库做最终扣减；若数据库更新失败或订单创建失败，必须可靠地回补 Redis 库存（同步回补或发补偿消息），避免 Redis 长期少计可售量。回补与下单逻辑建议幂等，防止重复消息导致多次加回或多次扣减。

2. 超时未支付如何归还库存？
答：订单进入待支付后设置支付超时；可用延迟队列（例如 RocketMQ 延迟消息）、定时扫描任务在超时点检查订单状态，若仍未支付则关单并释放预占：回补 Redis，并在数据库侧回滚或释放冻结库存（视你的库存模型而定）。释放逻辑与「支付成功回调」存在竞态，必须用状态机与条件更新保证只收敛到一个合法终态。

3. 为什么要做对账？怎么做？
答：预扣失败未感知、回补丢失、消息重试、人工运维都可能让 Redis 中的库存与数据库真实销量/可售量不一致。可定时比对 Redis 剩余与 DB 权威数据（或流水汇总），发现偏差则告警并按规则修正 Redis（一般以 DB 或流水为准），必要时结合审计表定位根因。
