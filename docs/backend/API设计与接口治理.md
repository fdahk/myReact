# API 设计与接口治理

## 使用方式

这篇不是教你怎么写 `Controller`，而是补齐一个很容易被低估、但在真实后端系统和面试里都很高频的能力：

- 如何把接口设计得稳定、清晰、可扩展
- 如何处理分页、筛选、排序、幂等、版本兼容、错误码和开放平台问题
- 如何让“接口设计”不只停留在字段命名，而能上升到系统边界和治理能力

如果你想冲顶级后端岗位，这一块必须会，因为很多面试官会借“接口怎么设计”来判断你有没有真实做过生产系统。

---

## 一、为什么 API 设计是后端核心能力

1. 为什么 API 设计不能理解成“把功能暴露出去”？
答：因为接口本质上是系统边界。边界一旦设计得混乱，后面就会连带出现权限难控、幂等不清、版本难演进、前后端反复扯皮、开放平台难治理的问题。很多后端系统后期难维护，本质不是代码写不出来，而是边界一开始就没设计好。

2. 为什么大厂后端面试会追问接口设计？
答：因为它能快速暴露候选人是不是只会写业务代码。真正做过系统的人，会自然思考资源建模、状态语义、错误语义、分页排序、幂等、权限边界和向后兼容，而不是只说“给前端一个接口”。

---

## 二、接口设计最核心的 6 个点

3. 第一个点是什么？
答：资源建模。先想清楚系统里真正的核心对象是什么，是订单、任务、资源、会话、用户还是产物。接口的路径、动作和字段，应该围绕资源来设计，而不是围绕页面按钮来设计。

4. 第二个点是什么？
答：状态语义。很多接口问题都不是参数问题，而是状态问题。例如任务能不能重复启动、订单能不能重复取消、审批能不能越级操作。这些都要求接口设计和状态机设计一起考虑。

5. 第三个点是什么？
答：错误语义。面试里很容易被问到“失败了怎么返回”。成熟回答不是“返回 500”，而是区分参数错误、权限错误、资源不存在、状态冲突、下游异常和系统过载，让调用方知道是改参数、重试、等待还是告警。

6. 第四个点是什么？
答：幂等语义。创建类、支付类、提交类接口特别容易被追问。你要能讲清哪些接口天然幂等，哪些需要业务幂等键，幂等窗口期多长，重复请求返回什么，幂等失败如何恢复。

7. 第五个点是什么？
答：向后兼容。接口一旦被多个调用方依赖，后续加字段、改枚举、调返回结构都可能带来线上事故。成熟接口治理会尽量做“新增而不是破坏式修改”，必要时引入版本管理和灰度。

8. 第六个点是什么？
答：可观测和可治理。接口不仅要“能调通”，还要能看到调用量、耗时、错误率、调用方、热点参数和异常分布。否则接口一多，排障和优化都会很痛苦。

## 三、分页、筛选、排序这些细节为什么值得深挖

9. 为什么分页不是一个小问题？
答：因为分页直接影响数据库压力、用户体验和接口稳定性。只会写 `pageNum + pageSize` 远远不够，你还要知道深分页的代价、游标分页适合什么场景、排序字段为什么不能随便开放、分页条件变多后索引怎么配合。

10. 什么场景更适合游标分页？
答：时间线、消息流、任务列表、日志查询这类数据持续增长、翻页方向明确、深分页成本高的场景，更适合游标分页。它比偏移量分页更稳定，也更容易避免深分页扫描问题。

11. 为什么筛选和排序是高频事故点？
答：因为调用方一多，大家总想“顺便再多加两个筛选条件”。如果后端没有提前想清楚允许哪些字段筛选、哪些字段排序、默认排序是什么、组合条件的索引如何支持，很容易把一个简单列表接口做成慢 SQL 入口。

## 四、幂等、重试和重复提交怎么讲

12. 为什么接口幂等是面试高频题？
答：因为真实系统里重复请求非常常见，可能来自前端连点、用户刷新、网关超时重试、消息重复投递或客户端断线重试。后端如果没有幂等设计，就会出现重复下单、重复扣款、重复建任务、重复发通知。

13. 幂等键应该怎么设计？
答：通常要满足三个要求：能够唯一代表一笔业务动作、能和业务主体关联、能够设置合理过期时间。常见做法是前端或网关生成 requestId，也可以由业务字段组合生成，但前提是你要明确它真正代表的是“哪一次业务提交”。

14. 为什么不能把所有接口都做成无限期幂等？
答：因为幂等本身也有成本和语义边界。有些动作只需要短时间防重复，有些动作则需要按业务主键长期去重。如果不区分场景，可能会导致状态卡死、缓存膨胀，甚至把本该允许的重试也拦掉。

15. 面试里怎么把重试和幂等讲成闭环？
答：可以按“为什么会重试 -> 哪些错误允许重试 -> 幂等如何保证 -> 多次重试如何观测 -> 重试放大故障怎么办”来讲。这样比单独谈幂等或单独谈重试更完整。

---

## 五、版本兼容与接口演进

16. 为什么接口版本兼容是大厂后端必须有的意识？
答：因为大系统里调用方很多，可能有 Web、App、BFF、内部服务、第三方平台。你今天改一个字段，明天就可能有旧客户端、旧服务、脚本任务和外部合作方出问题。接口演进能力，本质上是系统长期维护能力。

17. 什么情况下才应该显式做版本号？
答：当接口存在明显语义变化、返回结构破坏式调整、开放给外部调用方、或者必须长期兼容旧客户端时，才更值得单独做版本管理。很多内部接口其实可以先靠“新增字段 + 保持兼容 + 灰度切换”来演进，不一定一上来就堆很多版本。

18. 为什么“新增字段”也可能有风险？
答：因为有些调用方会做严格反序列化、签名校验、字段白名单或前端强校验。如果不了解调用方约束，哪怕只是多了一个字段，也可能触发兼容问题。所以接口治理不能只看服务端自己。

## 六、开放平台和内部接口有什么不同

19. 为什么开放平台接口比内部接口更难做？
答：因为它不仅要能用，还要考虑租户隔离、调用凭证、签名、防重放、流量控制、错误码规范、文档、沙箱环境、审计和版本生命周期。内部接口出问题可以快速协调，开放平台接口则更像长期对外承诺。

20. 开放平台最值得讲的几个治理点是什么？
答：我会优先讲：应用凭证管理、签名验签、时间戳和 nonce 防重放、配额与限流、回调重试、幂等、日志审计、版本淘汰策略。这些点非常能体现后端治理深度。

---

## 七、和主项目怎么绑定

21. `agent-server-java` 这种项目最值得讲哪些接口设计问题？
答：任务创建、任务查询、任务取消、任务结果获取、流式进度推送、资源上传、产物下载、管理员操作审计。这些接口天然带有状态流转、幂等、防重复、权限边界和分页检索问题，特别适合用来体现后端意识。

22. 如果面试官问“你项目里的接口设计亮点是什么”，怎么答？
答：不要泛泛说“遵循 RESTful”。更成熟的答法是：我重点关注了资源模型、状态约束、幂等、防重复、分页检索、错误码分层和向后兼容，避免接口随着业务迭代越来越难维护。这样回答比背风格规范更像真实 owner。

## 九、这一篇学到位的标准

学完这篇，你至少要做到下面三件事：

1. 能把接口设计讲成“资源、状态、错误、幂等、兼容、治理”的完整体系。
2. 能把分页、排序、版本兼容、开放平台这些容易被忽略的点答深。
3. 能把接口设计自然映射回自己的项目，而不是只会说“遵循 RESTful 规范”。
