# 搜索与 Elasticsearch 专题

## 使用方式

很多后端候选人只会 `MySQL` 和 `Redis`，但一被问到搜索、检索、日志检索、全文索引、复杂筛选，就会明显变薄。

这篇的目标不是把你训练成搜索专家，而是让你具备顶级大厂后端应该有的最小搜索认知：

- 什么场景不该继续硬扛 `MySQL`
- `Elasticsearch` 在系统里通常扮演什么角色
- 搜索和主数据库之间怎样同步
- 搜索链路为什么也会有一致性、延迟和高可用问题

---

## 一、为什么后端要懂搜索系统

1. 为什么搜索是后端高频扩展专题？
答：因为很多业务最终都会遇到“按关键词搜、按多个条件筛、按相关性排序、按时间聚合、按标签检索”的需求。单靠关系型数据库可以顶一段时间，但一旦全文检索、复杂组合查询和高并发检索成为主链路，搜索系统就会变得很重要。

2. 为什么不能把所有查询都继续压在 `MySQL` 上？
答：因为 `MySQL` 更擅长事务、结构化数据和精确查询，不适合长期扛复杂全文检索、相关性排序、大规模多字段搜索和高并发搜索流量。继续硬扛，最终会在索引设计、性能和扩展性上越来越痛苦。

---

## 二、Elasticsearch 在系统里通常做什么

3. `Elasticsearch` 最常见的角色是什么？
答：最常见是做全文检索、日志检索、筛选聚合和搜索型数据访问层。它通常不是主事务数据库，而是从主数据源同步过来的一份“面向搜索和分析优化”的数据副本。

4. 为什么搜索系统通常不是主数据源？
答：因为它更强调检索性能和搜索能力，而不是主事务一致性。很多系统会把真实写入先落在主数据库，再通过消息、binlog 或异步任务把数据同步到搜索引擎。这也是为什么搜索系统经常和最终一致性一起出现。

---

## 三、搜索同步怎么讲

5. 主数据库和搜索引擎最常见的同步方式是什么？
答：常见方式包括：业务写入后同步更新搜索索引、通过消息异步同步、订阅 binlog 做 CDC 同步、离线批量重建索引。成熟回答要说明：不同方式在实时性、一致性、复杂度和失败恢复上的 trade-off 都不同。

6. 为什么搜索同步经常会有延迟？
答：因为它通常不是主事务的一部分，而是异步链路。只要走消息、CDC 或异步任务，就天然会有短暂时间差。所以搜索系统最常见的语义不是“刚写完立刻一定搜得到”，而是“经过短暂延迟后最终可搜索”。

7. 搜索和主库不一致时怎么讲才成熟？
答：我会先区分是短暂最终一致，还是长时间错误不一致。短暂延迟通常可以接受；长时间不一致则需要对账、重放、补偿或全量重建索引。搜索题的关键不是假装零延迟，而是讲清同步链路和恢复手段。

---

## 四、搜索系统的高频问题

8. 为什么搜索系统也会遇到热点问题？
答：因为热点关键词、热点筛选条件、热点聚合查询同样会集中打到某些分片或缓存路径。总量看起来正常，不代表某个热门查询不会打出局部瓶颈。所以搜索系统也需要缓存、预热、限流和热门查询治理。

9. 为什么搜索系统比普通数据库更需要关注索引设计？
答：因为搜索索引结构直接决定查询能力、写入成本、存储成本和聚合性能。搜索题里如果只会说“建索引”，但说不清字段映射、分词、聚合字段和排序字段的差异，就会显得比较浅。

10. 为什么日志检索和业务搜索虽然都能用 `ES`，但关注点不同？
答：因为日志检索更关注写入吞吐、时间范围、结构化字段和排障效率；业务搜索更关注相关性、用户体验、筛选排序和最终一致性。底层引擎可以相似，但业务目标不同，系统设计关注点也不同。

---

## 五、和主项目怎么串起来

11. `agent-server-java` 这种项目为什么也值得引入搜索视角？
答：因为任务、资源、产物、执行日志、错误记录、标签和摘要信息都很适合做检索和筛选。一旦项目往平台型方向发展，搜索能力就不只是“锦上添花”，而会变成非常实际的后台能力。

12. 如果主项目后面要做搜索，最值得先补什么？
答：最值得先补的是：清晰的主数据模型、可同步的事件链路、稳定的文档结构、可重放或重建索引的能力，以及搜索与主库不一致时的恢复策略。这样后面接搜索能力才不会变成“另起一套脆弱系统”。

## 八、倒排索引原理

13. 什么是正排索引(Forward Index)？
答：正排索引是从**文档出发**的索引结构——给定一个文档 ID，直接拿到该文档的全部内容或字段值。MySQL 的主键聚簇索引本质上就是一种正排结构：`docID → {title, body, tags, ...}`。它非常适合"我已经知道要找谁，直接读出来"的场景，但如果想回答"哪些文档包含关键词 X"，就只能全表扫描。

14. 什么是倒排索引(Inverted Index)？
答：倒排索引是从**词项(Term)出发**的索引结构——给定一个词项，直接拿到包含该词的所有文档列表(Posting List)。结构示意：

```
Term        → Posting List
─────────────────────────
"分布式"    → [doc1, doc3, doc7]
"一致性"    → [doc1, doc2, doc5]
"事务"      → [doc2, doc3, doc4, doc7]
```

用户搜索"分布式 一致性"时，只需取两个 Posting List 做交集，就能 O(1) 定位到同时包含这两个词的文档(doc1)，完全不需要遍历所有文档。

15. 倒排索引是怎么构建的？
答：构建过程分四步：
1. **文档输入**：将原始文档(如一篇文章)送入索引管道；
2. **分词(Tokenization)**：将文本切分成独立的词项(Token)。英文按空格和标点切，中文需要专门的分词器(如 IK)；
3. **归一化(Normalization)**：对每个 Token 做小写转换、去停用词(a/the/的/了)、词干提取(running→run)等处理，使搜索时"Running"和"running"能匹配到同一个 Term；
4. **建立映射**：把每个归一化后的 Term 追加到对应的 Posting List 里，记录文档 ID 以及可选的附加信息。

16. Posting List 里只存文档 ID 吗？
答：不是，一个完整的 Posting List 条目可以包含多种信息：

| 字段 | 含义 | 用途 |
|------|------|------|
| DocID | 包含该 Term 的文档 ID | 基本检索 |
| TF(Term Frequency) | 该 Term 在本文档中出现的次数 | 相关性打分(BM25/TF-IDF) |
| Position | 该 Term 在文档中的位置序号(第几个词) | 短语查询(phrase query)、邻近查询(proximity query) |
| Offset | 该 Term 在原文中的起止字符偏移 | 搜索结果高亮(Highlight) |

ES/Lucene 可以通过 Mapping 中的 `index_options` 参数控制 Posting List 存储的粒度：`docs`(仅文档ID) → `freqs`(+词频) → `positions`(+位置，text 字段默认) → `offsets`(+偏移量)。存储粒度越高，功能越强，但索引体积也越大。

17. 为什么倒排索引特别适合全文搜索？
答：因为全文搜索的核心操作是"给定若干关键词，找出包含它们的所有文档并排序"。倒排索引把这个操作变成了哈希查找 + 有序列表合并：
- 查找关键词对应的 Posting List：O(1) 哈希查找或 O(log N) 的 FST(有限状态转换器)查找；
- 多关键词组合：对多个 Posting List 做交集/并集/差集运算；
- 相关性排序：利用 Posting List 中的 TF 和全局 IDF 计算 BM25 分数。

如果用正排索引实现同样的功能，就必须遍历每一个文档、对每个文档做全文扫描，复杂度是 O(文档总数 × 文档平均长度)，在百万级文档上完全不可行。

18. 倒排索引和 B+ 树索引有什么区别？
答：

| 维度 | B+ 树 | 倒排索引 |
|------|-------|----------|
| 核心结构 | 有序平衡树 | Term 字典 + Posting List |
| 擅长场景 | 精确匹配(`=`)、范围查询(`BETWEEN`)、前缀匹配(`LIKE 'abc%'`) | 全文检索、多关键词组合、相关性排序 |
| 对中文全文搜索 | 无能为力(除非手工分词建索引) | 原生支持，配合分词器可以处理任何语言 |
| 排序能力 | 按索引列物理有序 | 按相关性得分排序(BM25) |
| 写入开销 | 维护树平衡(分裂/合并) | 追加新 Segment(近似 append-only) |
| 典型代表 | MySQL InnoDB 索引 | Lucene/Elasticsearch |

两者不是替代关系，而是互补：MySQL 管事务和精确查询，ES 管全文检索和相关性排序。很多系统两者并存。

## 九、Elasticsearch 架构

19. ES 的逻辑层级是怎样的？
答：从大到小：**集群(Cluster) → 节点(Node) → 索引(Index) → 分片(Shard) → Lucene Segment → 文档(Document)**。

- **集群(Cluster)**：由一组共享同一个 `cluster.name` 的节点组成，对外表现为一个整体；
- **节点(Node)**：一个 ES 进程实例。一台物理机可以跑多个节点(生产环境一般一台一个)；
- **索引(Index)**：逻辑上类似一张"表"，是文档的集合。一个索引有自己的 Mapping 和 Settings；
- **分片(Shard)**：索引被水平切分成多个分片，每个分片是一个独立的 Lucene Index，可以分布在不同节点上；
- **文档(Document)**：最小的数据单元，一个 JSON 对象，有唯一的 `_id`。

20. ES 有哪些节点角色？
答：

| 角色 | 配置 | 职责 |
|------|------|------|
| Master Node | `node.roles: [master]` | 管理集群状态(元数据、分片分配、索引创建/删除)。集群需要奇数个 Master-eligible 节点(通常3个)做选举 |
| Data Node | `node.roles: [data]` | 存储分片数据，执行搜索和聚合的实际计算。是集群中资源消耗最大的角色 |
| Coordinating Node | 每个节点默认都是 | 接收客户端请求，将请求路由到相关 Data Node，合并结果后返回客户端。可以设置专用 Coordinating Node(所有 roles 为空) |
| Ingest Node | `node.roles: [ingest]` | 在文档索引前执行预处理管道(Pipeline)，如字段提取、格式转换、富化 |

生产集群一般会分离角色：3个专用 Master + N个 Data Node + 若干 Coordinating Node。

21. 分片(Shard)机制是怎么工作的？
答：
- 每个索引在创建时指定 **主分片数(number_of_shards)**，默认值在 ES 7.x+ 是 1(老版本是 5)。主分片数**创建后不可修改**，要改只能 Reindex；
- 每个主分片可以有 0 个或多个 **副本分片(number_of_replicas)**，默认值是 1。副本数可以**动态调整**；
- 主分片负责写入和读取，副本分片负责读取和容灾。主分片挂掉时，对应的副本会被提升为新的主分片；
- 每个分片底层是一个完整的 **Lucene Index**，拥有自己的倒排索引、正排存储(doc values)和 translog；
- 分片数设置建议：**每个分片 20-50GB** 为宜。分片太小会导致元数据开销大(每个分片需要占用 Master 节点的内存和 CPU)、查询时跨分片扇出多；分片太大会导致恢复慢、单分片查询压力大。

22. Lucene Segment 是什么？为什么说 ES 是"近实时"搜索？
答：每个分片(Lucene Index)内部由多个 **Segment** 组成。Segment 是**不可变(immutable)**的倒排索引片段。写入和搜索的完整过程：

1. **写入**：新文档先写入**内存 Buffer**(In-Memory Buffer)；
2. **Refresh**：每隔 `refresh_interval`(默认 **1秒**)，内存 Buffer 中的数据被刷入一个新的 Segment，此时数据才变得**可搜索**。这就是 ES 被称为 **Near Real-Time(NRT)** 搜索的原因——写入后最快 1 秒才能搜到；
3. **Translog(事务日志)**：写入时除了写内存 Buffer，还会同时写入 **translog**(类似数据库的 WAL)。translog 确保即使进程 crash，已写入但尚未 flush 到磁盘的数据也不会丢失；
4. **Flush**：当 translog 达到阈值(默认 512MB)或经过一定时间，执行 flush 操作——将内存中的数据真正落盘为 Segment 并清空 translog；
5. **Segment Merge**：后台线程不断将多个小 Segment 合并为更大的 Segment。合并过程中才会真正删除标记为"已删除"的文档(因为 Segment 不可变，删除只是打标记)。

23. 文档是怎么路由到特定分片的？
答：路由公式：

```
shard = hash(routing) % number_of_primary_shards
```

- 默认 `routing` 值是文档的 `_id`；
- 可以自定义 routing 值(如用 `user_id` 做路由)，使同一用户的文档落在同一个分片上，减少跨分片查询；
- 这也是为什么**主分片数创建后不能修改**的根本原因——修改后 hash 取模的结果会变，已有文档就找不到了。

24. 文档的写入流程是怎样的？
答：完整流程(以单文档写入为例)：

1. 客户端发送写入请求到任意节点(该节点自动成为本次请求的 **Coordinating Node**)；
2. Coordinating Node 根据路由公式计算目标 **Primary Shard** 所在节点，将请求转发过去；
3. Primary Shard 执行写入：写入内存 Buffer + 追加 translog；
4. Primary Shard 写入成功后，**并行**将请求复制到所有 **Replica Shard**；
5. 所有 Replica 写入成功(或满足 `wait_for_active_shards` 条件)后，Primary 向 Coordinating Node 返回成功；
6. Coordinating Node 向客户端返回写入成功响应。

参数 `wait_for_active_shards` 默认为 1(只要 Primary 写入成功就返回)；设为 `all` 则需要所有副本也写入成功才返回，一致性更强但延迟更高。

## 十、分词与 Mapping

25. Analyzer(分析器)是怎么工作的？
答：Analyzer 由三个阶段组成，按顺序执行：

```
原始文本
  │
  ▼
Character Filter(字符过滤)  ← 如 html_strip 去除HTML标签、mapping 替换特殊字符
  │
  ▼
Tokenizer(分词器)           ← 将文本切分成 Token 序列
  │
  ▼
Token Filter(词项过滤)      ← 如 lowercase(小写)、stop(去停用词)、synonym(同义词)、stemmer(词干提取)
  │
  ▼
最终 Token 序列(写入倒排索引)
```

索引时和搜索时都会经过 Analyzer。如果索引时用的 Analyzer 和搜索时不同(如索引用 `ik_max_word` 搜索用 `ik_smart`)，需要在 Mapping 中分别通过 `analyzer` 和 `search_analyzer` 指定。

26. 常见的分词器有哪些？各自适用什么场景？
答：

| 分词器 | 切分方式 | 适用场景 |
|--------|----------|----------|
| **Standard** | 按 Unicode 文本分割算法切分(英文按空格和标点，中文**每个字一个 Token**) | 英文通用场景，中文效果极差 |
| **IK Analyzer**(插件) | 基于词典的中文分词。`ik_smart` 做粗粒度最少切分，`ik_max_word` 做最大粒度最多切分 | 中文全文搜索的事实标准。索引时用 `ik_max_word`(召回率高)，搜索时用 `ik_smart`(精确度高) |
| **Whitespace** | 只按空格切分，不做任何归一化 | 对大小写和标点敏感的场景(如日志字段) |
| **Keyword** | 不切分，整个字段值作为一个 Token | 精确匹配场景(但一般直接用 keyword 字段类型) |
| **ngram / edge_ngram** | 按字符 n-gram 切分。如 "search" 在 bigram 下变为 `["se","ea","ar","rc","ch"]` | 搜索建议(autocomplete)、模糊匹配、拼写纠错 |
| **Pattern** | 按正则表达式切分 | 自定义分隔符场景 |

27. Mapping 的核心字段类型有哪些？
答：Mapping 定义了索引中每个字段的数据类型和索引方式，直接决定了哪些查询和聚合可以执行：

| 类型 | 是否分词 | 适用查询 | 说明 |
|------|----------|----------|------|
| **text** | 是 | `match`、`match_phrase` 全文搜索 | 会经过 Analyzer 处理后写入倒排索引。**不能直接用于排序和聚合**(需要开启 fielddata 或使用子字段) |
| **keyword** | 否(整个值作为一个 Term) | `term`、`terms` 精确匹配，`range`，排序，聚合 | 适合枚举值、标签、ID、邮箱等不需要分词的字段 |
| **integer/long/float/double** | 否 | `range`、排序、聚合 | 使用 BKD 树存储，优化数值范围查询 |
| **date** | 否 | `range`、`date_histogram` 聚合 | 内部存储为 epoch 毫秒值，支持多种格式(如 `yyyy-MM-dd HH:mm:ss`) |
| **boolean** | 否 | `term` | 值只有 `true`/`false` |
| **geo_point** | 否 | `geo_distance`(范围圈)、`geo_bounding_box`(矩形) | 存储经纬度，支持地理距离排序 |
| **nested** | — | `nested` 查询 | 对象数组中每个对象作为独立文档索引，避免扁平化导致的交叉匹配问题 |

常见最佳实践：给 `text` 字段加一个 `keyword` 子字段(Multi-field)，同时支持全文搜索和精确匹配/排序/聚合：

```json
{
  "title": {
    "type": "text",
    "analyzer": "ik_max_word",
    "search_analyzer": "ik_smart",
    "fields": {
      "keyword": {
        "type": "keyword",
        "ignore_above": 256
      }
    }
  }
}
```

28. Dynamic Mapping 是什么？为什么生产环境建议显式定义 Mapping？
答：Dynamic Mapping 是 ES 在遇到未定义字段时**自动推断类型**的机制。例如写入 `{"age": 25}` 时，ES 会自动把 `age` 推断为 `long` 类型。

问题在于：
- 字符串字段会被同时创建 `text` + `keyword` 子字段，浪费存储；
- 日期格式的字符串可能被误判为 `date` 类型；
- 一旦字段类型确定就**不能修改**(只能 Reindex)；
- 不同批次数据中同名字段类型不一致会导致写入失败。

生产环境建议：`"dynamic": "strict"`(遇到未定义字段直接报错拒绝写入)，所有字段显式定义 Mapping。

## 十一、查询与相关性

29. Query Context 和 Filter Context 有什么区别？
答：

| 维度 | Query Context | Filter Context |
|------|---------------|----------------|
| 核心作用 | 回答"这个文档有多匹配"(计算 `_score`) | 回答"这个文档是否匹配"(yes/no) |
| 是否算分 | 是，会计算相关性得分 | 否，`_score` 固定为 0 或不参与 |
| 是否缓存 | 否 | **是**，ES 会自动缓存 Filter 的结果(Bitset)，后续相同 Filter 直接命中缓存 |
| 性能 | 较慢(需要算分) | **更快**(无需算分 + 可缓存) |
| 典型用法 | `match`、`multi_match`、`match_phrase` | `term`、`range`、`exists`、`bool.filter`、`bool.must_not` |

最佳实践：**能用 Filter 的场景一律用 Filter**，只有需要相关性排序的字段才放在 Query 里。

30. bool 查询是怎么组合 Query 和 Filter 的？
答：`bool` 查询是 ES 中最核心的组合查询，有四个子句：

```json
{
  "query": {
    "bool": {
      "must": [
        { "match": { "title": "分布式事务" } }
      ],
      "filter": [
        { "term": { "status": "published" } },
        { "range": { "publish_date": { "gte": "2024-01-01" } } }
      ],
      "should": [
        { "match": { "tags": "高可用" } }
      ],
      "must_not": [
        { "term": { "is_deleted": true } }
      ]
    }
  }
}
```

| 子句 | 上下文 | 语义 |
|------|--------|------|
| `must` | Query | 必须匹配，参与算分 |
| `filter` | Filter | 必须匹配，不算分，可缓存 |
| `should` | Query | 可选匹配，匹配则加分。在没有 must/filter 时，至少要匹配一个 should(可通过 `minimum_should_match` 控制) |
| `must_not` | Filter | 必须不匹配，不算分 |

31. BM25 算法是怎么计算相关性的？
答：BM25(Best Matching 25)是 ES 5.0+ 的**默认相关性算法**，替代了之前的 TF-IDF。核心公式简化理解：

```
score(q, d) = Σ IDF(t) × [ TF(t,d) × (k1 + 1) ] / [ TF(t,d) + k1 × (1 - b + b × |d| / avgdl) ]
```

三个核心因子：

| 因子 | 含义 | 作用 |
|------|------|------|
| **TF(词频)** | Term 在文档中出现的次数 | 出现越多越相关，但有**饱和上限**(由参数 `k1` 控制，默认 1.2) |
| **IDF(逆文档频率)** | 包含该 Term 的文档占总文档数的比例的倒数 | 越罕见的词权重越高("的"这种高频词 IDF 很低) |
| **文档长度归一化** | 文档长度与平均文档长度的比值 | 短文档中出现某词比长文档中出现更能说明相关性(由参数 `b` 控制，默认 0.75) |

和 TF-IDF 的关键区别：TF-IDF 中词频和得分是线性关系(出现 100 次 = 10 倍于出现 10 次)；BM25 中词频有**对数饱和效应**——出现 10 次和出现 100 次的得分差距很小，避免了词频轰炸(keyword stuffing)。

32. 高亮(Highlight)是怎么实现的？
答：ES 支持在搜索结果中返回匹配片段，通过 HTML 标签(默认 `<em></em>`)标注命中的关键词：

```json
{
  "query": { "match": { "content": "分布式事务" } },
  "highlight": {
    "fields": {
      "content": {
        "pre_tags": ["<em>"],
        "post_tags": ["</em>"],
        "fragment_size": 150,
        "number_of_fragments": 3
      }
    }
  }
}
```

ES 提供三种高亮器：
- **Unified**(默认，ES 6.0+)：基于 BM25 打分选择最佳片段，兼顾性能和准确性；
- **Plain**：基于标准 Lucene 高亮器，逐字段分析，小文档准确但大文档慢；
- **FVH(Fast Vector Highlighter)**：需要在 Mapping 中设置 `term_vector: with_positions_offsets`，大文档性能最好但索引体积增大。

33. 聚合(Aggregation)是怎么工作的？
答：聚合类似 SQL 的 `GROUP BY` + 聚合函数，但更强大，支持多层嵌套。ES 聚合分三大类：

**Bucket 聚合(分桶)**——按条件将文档分组：

| 聚合 | 作用 | 类比 SQL |
|------|------|----------|
| `terms` | 按字段值分桶 | `GROUP BY field` |
| `range` | 按数值范围分桶 | `CASE WHEN price BETWEEN ...` |
| `date_histogram` | 按时间间隔分桶 | `GROUP BY DATE_TRUNC(...)` |
| `histogram` | 按固定数值间隔分桶 | — |
| `filter`/`filters` | 按自定义查询条件分桶 | `WHERE ...` 子查询 |

**Metric 聚合(指标)**——对桶内文档做统计计算：

| 聚合 | 作用 |
|------|------|
| `avg`/`sum`/`min`/`max` | 基本统计 |
| `value_count` | 计数 |
| `cardinality` | 去重计数(基于 HyperLogLog，有约 0.5% 误差) |
| `stats` | 一次性返回 count/min/max/avg/sum |
| `percentiles` | 百分位数(如 P99 延迟) |

**嵌套聚合**——桶内再做指标或再分桶：

```json
{
  "aggs": {
    "by_category": {
      "terms": { "field": "category.keyword", "size": 10 },
      "aggs": {
        "avg_price": { "avg": { "field": "price" } },
        "by_brand": {
          "terms": { "field": "brand.keyword", "size": 5 }
        }
      }
    }
  }
}
```

上例效果：先按 `category` 分桶，每个桶内计算平均价格，同时再按 `brand` 做子分桶。

注意：聚合字段必须是 **keyword**、**数值** 或 **日期** 类型(或开启了 `doc_values` 的字段)。`text` 字段不能直接聚合(需要用 `.keyword` 子字段)。

## 十二、深分页与性能优化

34. from+size 深分页为什么有性能问题？
答：ES 的 `from + size` 翻页机制本质是**全局排序截断**：

假设索引有 5 个分片，查询 `from=10000, size=10`：
1. Coordinating Node 向 5 个分片各发请求："给我你的 top 10010 条结果"；
2. 每个分片返回 10010 条(已排序)；
3. Coordinating Node 收到 5 × 10010 = **50050** 条数据；
4. 在内存中做**全局归并排序**；
5. 取第 10001~10010 条返回客户端，其余全部丢弃。

分片越多、`from` 越大，浪费越严重。所以 ES 默认设置 `index.max_result_window = 10000`，超过这个值直接报错。

35. 有哪些方案解决深分页？
答：

**方案一：Scroll API**
- 原理：首次查询时创建一个**搜索上下文快照**，返回一个 `scroll_id`。后续每次用 `scroll_id` 获取下一批数据；
- 特点：维护快照消耗资源(占用 Segment 不被合并)，适合**全量遍历**(如数据导出、批量处理)；
- 不适合：实时用户搜索翻页(快照是创建时的数据，看不到新写入)；
- 重要参数：`scroll=5m`(快照存活时间)。用完必须调用 `DELETE /_search/scroll` 清理。

**方案二：Search After**
- 原理：每次查询带上**上一页最后一条的排序值**作为起点，继续向后取；
- 特点：无状态，性能稳定，不随页数增大而恶化。适合**实时向后翻页**；
- 限制：只能向后不能跳页(不能直接跳到第 100 页)；
- 使用要求：必须有**唯一性排序字段**(通常在排序条件最后加 `_id` 或 `_shard_doc` 作为 tiebreaker)。

```json
{
  "query": { "match": { "title": "分布式" } },
  "sort": [
    { "publish_date": "desc" },
    { "_id": "asc" }
  ],
  "search_after": ["2024-06-15T10:30:00.000Z", "doc_xyz_123"]
}
```

**方案三：Point in Time(PIT)**
- ES 7.10+ 引入，配合 Search After 使用；
- 原理：创建一个轻量级**一致性快照**(比 Scroll 更轻量)，在快照范围内做 Search After 翻页；
- 优势：既有 Search After 的性能优势，又保证翻页过程中数据视图一致(不会因为写入导致分页错乱)。

| 方案 | 适用场景 | 是否支持跳页 | 是否实时 | 资源开销 |
|------|----------|--------------|----------|----------|
| from+size | 前几页(< 1万) | 是 | 是 | 低(浅页) |
| Scroll | 全量遍历/导出 | 否 | 否(快照) | 高 |
| Search After | 实时向后翻页 | 否 | 是 | 低 |
| PIT + Search After | 实时翻页 + 一致性 | 否 | 快照内一致 | 中 |

36. 写入性能有哪些常见优化手段？
答：

| 优化手段 | 说明 | 参数/做法 |
|----------|------|-----------|
| **Bulk 批量写入** | 合并多条写入为一次请求，减少网络开销和 Segment 生成次数 | 推荐每批 **5-15MB**(不是按条数，按体积)。太大会占用大量内存 |
| **调大 refresh_interval** | 减少 Segment 生成频率，减少 I/O 和合并开销 | 批量导入时可设为 `"refresh_interval": "30s"` 甚至 `"-1"`(关闭自动刷新) |
| **临时关闭副本** | 批量导入时设 `"number_of_replicas": 0`，导入完再改回来 | 避免每条写入都同步到副本，导入速度可提升一倍以上 |
| **调整 translog 策略** | 将 `index.translog.durability` 从 `request`(每次请求 fsync)改为 `async`(异步) | 降低写入延迟，但 crash 时可能丢失最近 `sync_interval`(默认 5秒)的数据 |
| **使用自动生成 _id** | 避免 ES 在写入时检查 `_id` 是否已存在 | 如果业务不需要自定义 `_id`，让 ES 自动生成 |
| **合理设置分片数** | 分片太多会增加写入时的协调开销 | 对于单个索引日写入量不大的场景，1-3 个主分片即可 |

37. 查询性能有哪些常见优化手段？
答：

| 优化手段 | 说明 |
|----------|------|
| **优先用 Filter Context** | Filter 不算分且结果可缓存，对精确筛选条件(状态、日期范围、类目)一律放在 `bool.filter` 里 |
| **避免 wildcard 前缀查询** | `{"wildcard": {"name": "*keyword"}}` 会触发全量扫描。如果需要后缀搜索，考虑 `reverse` Token Filter 或 ngram |
| **避免深度嵌套聚合** | 聚合层级越深，内存消耗指数增长。超过 3 层嵌套需要评估是否改用其他方案 |
| **合理设计 Mapping** | 不需要搜索的字段设 `"index": false`；不需要聚合/排序的字段设 `"doc_values": false`；减少不必要的 `text` 分词 |
| **使用 routing** | 同一查询维度的数据路由到同一分片，减少跨分片扇出(如按 `tenant_id` 路由) |
| **预热缓存** | 使用 `_search` 预热 Filter 缓存和 file system cache(OS Page Cache)，对高频查询效果显著 |
| **控制返回字段** | 使用 `_source` 过滤只返回需要的字段，减少网络传输和序列化开销 |
| **Profile API 诊断** | 在查询中加 `"profile": true`，可以看到每个分片上每个查询子句的耗时，定位慢查询瓶颈 |
