# agent-server Java 重构方案

## 方案定位

这不是把当前 NestJS 代码逐文件翻译成 Java，而是：

**以当前 `agent-server` 的业务题材为蓝本，重构为一个面向顶级后端岗位的生产级 Java 平台型项目。**

目标定位建议为：

**资源整理与 AI 长任务编排平台（Java 后端版）**
标准脚手架工具：
最标准的就是 Spring Initializr + Maven + Spring Boot
---

## 一、重构目标

这次 Java 重构，不是为了“换语言”，而是为了同时完成三件事：

1. 建立你的 `Java` 主栈可信度
2. 把当前项目补到能满足顶级后端面试深挖的复杂度
3. 形成一个可讲系统设计、稳定性、异步任务和平台治理的主项目

所以项目目标不能只是：

- 能跑
- 能调用模型
- 能返回结果

而必须升级成：

- 有任务模型
- 有状态机
- 有数据库
- 有缓存
- 有异步执行
- 有失败重试
- 有权限和治理能力
- 有可观测性

---

## 二、重构后的推荐技术栈

## 后端主栈

原则是：

- 优先选择国内 Java 后端最主流、最稳妥、最适合求职表达的方案
- 优先保证你能把数据库、缓存、异步任务、权限和稳定性真正做扎实
- 不为了“技术更高级”而引入不必要的复杂度

## 后端主栈（第一版定版）

- `Java 17`
- `Spring Boot`
- `Spring Web`
- `Spring Validation`
- `Spring AOP`
- `Spring Security`
- `MyBatis`

这里不再保留 `Spring Data / MyBatis` 的摇摆空间，第一版直接定为 `MyBatis`。

原因是：

- 更贴近国内 Java 后端主流招聘语境
- 更适合你当前阶段把 `SQL`、索引、执行计划和数据访问链路学扎实
- 更利于后续把数据库能力讲深，而不是停留在 ORM 抽象层

## 数据与中间件（第一版定版）

- `MySQL`
- `Flyway`
- `Redis`
- `RabbitMQ`

这里也不再同时保留 `RabbitMQ` 和 `Kafka` 两套方向，第一版直接定为 `RabbitMQ`。

原因是：

- 更适合先把任务队列、重试、死信、异步执行这些核心后端能力做出来
- 学习曲线比 `Kafka` 更平缓
- 对你当前这个长任务平台型项目更直接、更够用
- 不会把你的主要精力过早带偏到消息生态细节本身

`Flyway` 作为数据库迁移工具，第一版就应该接入。

原因是：

- 数据库结构演进要可追踪、可回放、可协作
- 这会显著提升项目的工程规范性
- 也更符合 Java 后端项目的主流治理方式

## 其他基础设施

- `Docker`
- `OpenAPI / Swagger`
- `Micrometer`
- `Prometheus / Grafana` 基础接入
- 结构化日志

## 三、重构后的系统定位

建议把项目升级成下面这个产品级定位：

**一个支持资源采集、任务编排、进度追踪、结果产物管理和任务治理的后端平台**

它服务的不是单次接口调用，而是：

- 长任务执行
- 多阶段状态流转
- 可回放任务过程
- 可查询历史记录
- 可重试失败任务
- 可管理产物输出

这样它就不再是“Agent demo”，而是“平台后端系统”。

## 六、缓存设计建议

建议第一版就接入 `Redis`，但不要乱用。

## 适合缓存的内容

- 任务详情热点查询
- 任务列表筛选结果的短期缓存
- 工作空间权限信息
- 幂等 token
- 流式任务短期运行态

## 不建议直接缓存的内容

- 强一致任务核心状态的唯一真相源
- 需要严格审计的最终结果

核心原则：

- 数据库存真相
- Redis 做加速和协调

## 七、异步执行与队列设计

这是整个 Java 重构最关键的升级点之一。

## 当前问题

现在是：

- 创建任务
- 进程内直接异步执行
- 运行状态在内存中维护

这在生产级项目里不够稳。

## 推荐方案

改为：

1. API 创建任务
2. 任务落库为 `queued`
3. 发送任务消息到队列
4. Worker 消费任务
5. 按步骤更新状态
6. 事件写库并推送 `SSE`
7. 失败任务按策略重试
8. 超过阈值进入死信或人工处理

这样你就可以把项目讲成真正的异步任务平台。

---

## 八、SSE 进度推送如何升级

当前项目的 `SSE` 是可以保留的，而且应该保留。

但要升级：

### 当前亮点

- 能实时推送任务进度
- 能让前端感知长任务过程

### 当前不足

- 事件主要存在内存
- 多实例无法共享
- 服务重启可能丢上下文

### 升级方案

- 任务事件先写库
- `SSE` 连接时先补历史事件
- 再订阅最新事件流
- 支持断线重连后的增量补偿
- 事件带 `sequence` 或时间戳

这样它就不再只是“能看到进度”，而是完整的事件流设计。

## 九、权限与平台化升级

如果你想让这个项目真正符合顶级后端岗位标准，我建议一定补用户和权限。

至少做到：

- 登录鉴权
- 工作空间隔离
- 任务归属
- 任务查询权限
- 产物访问权限

如果没有这部分，项目仍然更像一个技术 demo 服务，而不是平台型系统。

## 十、可观测性设计建议

至少补下面这些：

### 日志

- 每次请求统一 `requestId`
- 每个任务统一 `runId`
- 每次模型调用统一耗时日志

### 指标

- 任务创建量
- 任务成功率
- 任务失败率
- 任务平均耗时
- 模型调用超时率
- 队列积压量
- `SSE` 活跃连接数

### 健康检查

- 数据库
- Redis
- MQ
- 模型服务

这些能力会极大提高项目的后端含金量。

## 十一、面试包装角度

如果按这个方案做完，项目最适合从下面几个角度来讲：

### 1. 异步任务平台

你可以重点讲：

- 为什么同步链路改异步
- 任务状态机怎么设计
- 重试与补偿怎么做

### 2. 长任务进度平台

你可以重点讲：

- 为什么保留 `SSE`
- 如何做事件存储与补偿
- 多实例下如何保持进度一致

### 3. AI 工程平台

你可以重点讲：

- 模型调用隔离
- 外部依赖治理
- 任务编排与结果产物管理

### 4. 平台型后端系统

你可以重点讲：

- 权限
- 工作空间
- 数据隔离
- 任务治理
- 审计和观测

这几个角度叠加起来，项目会比普通管理后台强很多。

## 十三、最终结论

最终建议非常明确：

**不要放弃这个题材。**

你当前这个 `agent-server`，最有价值的不是现有 Nest 代码，而是它已经帮你确定了一个很适合后端成长和面试表达的业务方向：

- 长任务
- 异步编排
- `SSE`
- AI 服务接入
- 资源处理
- 结果产物管理

这些点一旦用 Java 主栈重做，并补上：

- `MySQL`
- `Redis`
- MQ
- 权限
- 状态机
- 稳定性
- 可观测性

它完全有机会成为你冲击高质量 Java 后端岗位的主项目。

换句话说：

**这个项目值得做，但必须按平台级、生产级、Java 主栈的标准重做。**
