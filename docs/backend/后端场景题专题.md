# 后端场景题专题

## 使用方式

后端场景题真正考的不是你能不能立刻报一个名词，而是你有没有生产环境思维。我回答这类题时，默认会按下面顺序组织：

1. 先复述现象和边界。
2. 再按优先级拆可能原因。
3. 说明排查动作和判断依据。
4. 先讲止血方案。
5. 最后补长期治理。

---

## 一、为什么场景题这么重要

1. 后端场景题在看什么？
答：主要看四件事：你是否真的做过系统，你是否能在不确定情况下快速收敛问题，你是否有稳定性和边界意识，以及你是否知道如何做取舍。很多问题没有唯一答案，但一定有更专业的分析路径。

---

## 二、数据库与性能场景

2. 接口突然变慢，你会怎么排查？
答：我会先确认是全量变慢还是部分接口、部分用户变慢，再按链路拆：网关、应用线程池、下游 RPC、数据库、缓存、GC、锁等待。因为“慢”只是结果，不代表一定是 SQL 问题。若影响已经明显，我会先考虑降级、限流或回滚高风险变更，再继续看指标和日志缩小范围。

3. 某条 SQL 很慢，你会怎么做？
答：先看执行计划，确认是否走索引、是否回表严重、是否全表扫描、是否排序或临时表，再结合查询条件、返回字段、数据量和锁等待判断根因。成熟回答不是上来就说“加索引”，而是先看证据。

4. 数据库 CPU 飙高，你会优先怀疑什么？
答：高频慢查询、全表扫描、错误索引、统计信息异常、批量任务、锁竞争和热点写入都可能导致。我会先看慢 SQL 和热点语句，再看应用层是否突然增加重试或流量。

5. 分页越翻越慢，怎么分析？
答：我会先看是不是大 offset 导致数据库不断跳过大量记录，再考虑是否可以改成基于主键、时间戳或游标的分页方式。这个题核心不在分页语法，而在访问模式和数据量的关系。

---

## 三、缓存场景

6. Redis 挂了，系统怎么办？
答：我会先确认 Redis 承担的职责是缓存、锁、会话还是限流，再判断影响面。缓存型 Redis 挂了最怕流量瞬间打爆数据库，所以我会优先限流、降级、保护核心接口；如果 Redis 还承担锁和幂等，则要特别小心重复执行和并发写入风险。

7. 缓存命中率突然下降，你会怎么排查？
答：我会看是否有大面积 key 过期、热点变更、缓存更新逻辑出错、实例切换、容量淘汰或版本发布导致 key 规则变化。命中率下降不是孤立指标，往往会伴随下游 RT 上升。

8. 热点 key 导致单点压力很大，怎么办？
答：短期可以做本地缓存、热点预热、只读副本分担或临时降级；长期要看是否能拆分访问、优化数据结构、增加多级缓存或改业务获取方式。热点问题的关键是局部极端流量，不是整体 QPS。

9. 如何回答缓存一致性？
答：我会先明确业务能接受多大不一致，再讲策略，比如更新数据库后删除缓存、逻辑过期、订阅变更消息或异步刷新。这个题最怕脱离业务空谈“一致性方案大全”。

---

## 四、消息队列与异步任务场景

10. 消息重复消费了怎么办？
答：我会先明确重复消费影响什么业务，再看幂等边界是否存在，比如业务唯一键、幂等表、状态机校验或 Redis 去重。成熟系统不会奢望 MQ 永不重复，而是默认要设计幂等。

11. 消息积压了，你怎么处理？
答：先看是突发流量、消费端故障、下游处理变慢还是错误重试过多，再做扩容、限流、死信隔离或临时旁路。专业顺序一定是先止血，再分析积压根因。

12. 长任务执行到一半失败了怎么办？
答：先看任务是否可重试、是否有幂等保护、是否要回滚部分状态、是否需要人工补偿。异步任务最怕“失败后没人知道如何恢复”，所以状态机和失败记录非常重要。

13. 什么情况下你会把同步接口改成异步任务？
答：当主链路等待时间明显过长、后续动作对用户不是强实时、下游不稳定、流量存在峰谷、执行逻辑复杂时，我会考虑异步化。但我也会同时补任务状态、进度查询、重试和观测能力。

---

## 五、并发与锁场景

14. 同一个用户重复提交订单怎么办？
答：我会从前端防抖、服务端幂等 token、数据库唯一约束、状态机检查这几层一起设计，而不是只靠其中一层。重复提交本质是并发和幂等问题，不是“按钮点太快”的前端小问题。

15. 库存超卖怎么处理？
答：看业务要求，可以用数据库乐观锁、悲观锁、预扣减、队列串行化或缓存预占配合最终一致性。这个题重点在于你是否知道一致性、吞吐和实现复杂度之间的 trade-off。

16. 为什么分布式锁不是银弹？
答：因为它只解决跨节点协调，不自动解决业务幂等、超时、续期、误释放和异常恢复。很多人一说并发就上锁，这是不成熟的表现。

## 六、稳定性场景

17. 服务雪崩时你怎么处理？
答：先限流、熔断、降级、摘除故障实例，保护核心链路，再定位是下游故障、重试风暴、流量突增、缓存失效还是线程池耗尽。雪崩题最看重的是止血意识。

18. 线程池打满了怎么办？
答：先看是什么任务把线程占满，是下游阻塞、任务过长、队列堆积还是配置不合理，再临时扩容、限流或降级。长期则要拆分线程池、优化任务模型和补监控。

19. 如果某个接口偶发超时，你如何判断是应用问题还是网络问题？
答：我会看超时分布、链路追踪、网关耗时、服务内部耗时、下游调用和网络层错误码，再判断卡在连接、应用处理还是下游依赖。偶发问题最怕拍脑袋归因。

---

## 七、SSE 与长链路场景

20. SSE 流式接口偶发断开，你怎么分析？
答：我会同时看代理超时、网络抖动、服务端主动结束、任务状态机异常、客户端重连逻辑和中间层配置。流式问题不能只盯服务代码，它是整条链路问题。

21. 为什么长任务不适合只放在内存里管理？
答：因为服务一重启、扩容或切实例，状态就可能丢失，多实例下也无法共享。内存态适合 demo 或单机试验，不适合作为顶级后端项目的最终形态。

22. 这和当前 `agent-server` 有什么关系？
答：当前 `resource-organizer-agent` 的运行状态主要存在内存 `Map` 中，这非常适合在评估文档里讲“当前题材有价值，但工程深度还不够”。如果重写成 Java 主项目，任务持久化和多实例一致性会是重点补强方向。

---

## 八、回答模板

23. 后端场景题的标准回答骨架是什么？
答：现象、边界、假设、验证、止血、修复、治理。这个骨架最大的价值不是套模板，而是保证你在压力下也不会漏掉最关键的生产思维。

24. 面试官更看重什么？
答：比起最终答案，他更看重你怎么定位和取舍。因为真实工作里，最有价值的人不是“碰巧猜对根因”的人，而是能快速收敛问题、保护业务、推动长期治理的人。

## 十、缓存深水区场景

25. 缓存击穿怎么回答才更像做过？
答：我会先澄清是哪类 key，被多少流量打到，是否是单个热点 key 在同一时间过期。因为击穿的核心不是“缓存没命中”这么简单，而是一个极热点数据突然回源，把数据库瞬间打爆。短期止血可以用互斥更新、逻辑过期、本地缓存和热点预热；长期则要看 key 生命周期设计、热点识别机制和多级缓存架构是否合理。

26. 缓存雪崩和击穿有什么区别？
答：雪崩更像大量 key 在同一时间失效，导致整体回源；击穿更像一个极热点 key 失效后被高并发集中打爆。两者都可能把数据库拖垮，但分析思路不同。雪崩更关注批量失效、实例故障和大面积淘汰；击穿更关注单点热点和互斥保护。

27. 缓存穿透怎么处理？
答：先判断是不是恶意请求、非法参数，还是业务上确实大量查询不存在对象。常见手段包括参数校验、布隆过滤器、空值缓存和限流。这个题最容易失分的地方，是只会报“布隆过滤器”，却不讲它适用场景和误判代价。

28. 双写不一致怎么分析？
答：我会先看写数据库和删缓存的顺序、是否有并发更新、读写分离是否引入主从延迟、是否有重试和失败补偿。很多所谓缓存一致性问题，根因并不在 Redis，而在业务写路径和数据库一致性语义没定义清楚。

## 十一、数据库并发与资源场景

29. 数据库连接池被打满了怎么办？
答：先看连接池耗尽是因为真实流量增加，还是慢 SQL、锁等待、事务过长导致连接长时间不释放；再看是否有应用层重试放大、连接泄漏、批处理任务抢占。止血通常是限流、摘除异常任务、临时扩容或回滚问题版本；长期则要治理 SQL、事务边界和连接池参数。

30. 死锁频繁出现，你怎么排查？
答：我会先收集死锁日志和相关 SQL，确认涉及哪些表、索引和访问顺序，再判断是业务并发模式问题，还是 where 条件没命中索引导致锁范围过大。死锁题关键不是说“数据库会回滚一个事务”，而是要能继续讲怎么统一更新顺序、缩小事务、补索引和降低热点冲突。

31. 读写分离后用户偶发“刚写完查不到”，怎么处理？
答：我会先判断是不是主从延迟导致，再按业务重要性决定策略：关键链路读主库、写后短时间粘主、直接返回刚写入结果、或接受最终一致。这个题真正考的是你有没有把“读写分离提升性能”和“业务读后写一致性”一起考虑。

32. 数据库 CPU 正常，但接口还是很慢，可能是什么问题？
答：可能是锁等待、连接池排队、下游 RPC 卡住、应用线程池满了、GC 抖动、网络重传或慢日志没完全覆盖真实瓶颈。也就是说，数据库资源正常不等于数据库链路没问题，场景题里最忌讳只盯一个指标下结论。

## 十二、MQ 与异步链路场景

33. 消费端不断重试把系统打崩了，怎么止血？
答：先把失败消息隔离出来，避免继续污染主消费链路，再判断失败是脏数据、依赖故障还是幂等失效。止血动作通常包括暂停消费、限流、转死信、关闭高风险重试、手工修正部分异常消息。这个题最体现你是否有“先保护系统，再研究根因”的意识。

34. 为什么消息积压不能只靠扩容消费者解决？
答：因为如果根因是下游数据库慢、外部服务挂、消息格式异常或重试风暴，单纯扩容只会把更多流量压给本来已经出问题的依赖，甚至放大故障。扩容只是手段之一，前提是下游真的还能承压。

35. 顺序消息乱了，怎么分析？
答：我会先看业务到底要求全局顺序还是局部顺序，再看分区策略、并发消费、失败重试和重平衡是否打乱顺序。很多系统真正需要的是“同一订单、同一用户、同一任务局部顺序”，不是全局顺序。如果顺序边界一开始没定义清楚，后面越补越乱。

## 十三、JVM 与运行时场景

36. Full GC 频繁导致接口抖动，你怎么查？
答：先看 GC 日志、堆使用、对象分配速率、老年代增长、是否有大对象和缓存滥用，再看是不是某次发布后引入了内存泄漏或对象暴涨。止血上可以先限流、重启异常实例、摘流量；长期治理则要回到对象生命周期、缓存策略、线程模型和代码热点分析。

37. 内存一直涨但 CPU 不高，这可能意味着什么？
答：常见会想到缓存无上限、集合不断堆积、任务结果没清理、连接或线程对象泄漏、消息堆积在内存队列里。也就是说，资源问题不一定先表现成 CPU 飙升，很多时候是内存和对象生命周期先失控。

38. 线程很多但吞吐不升反降，怎么理解？
答：我会先怀疑线程并不是在有效工作，而是在等待锁、等待 IO、等待下游返回，或者上下文切换过多。线程数多不等于吞吐高，后端题里如果你能主动提到“线程不是越多越好”，会明显更成熟。

## 十四、发布与配置事故场景

39. 发版后错误率升高，你的第一步是什么？
答：先确认是否与发布时间强相关，再按版本、实例、接口、租户、机房维度看异常是否集中。如果影响面明显，就先回滚、关开关或暂停放量，而不是一边让故障扩散一边慢慢分析。发布类场景的第一优先级永远是控影响面。

40. 发布后部分用户出现旧新版本混用问题，怎么处理？
答：我会优先怀疑入口缓存、静态资源版本、配置不一致、接口兼容和数据库兼容问题。对后端来说，也要关注接口字段、新旧服务并存、消息格式兼容和灰度流量路由是否一致。这类问题本质上是版本治理问题，不只是“某个字段返回错了”。

41. 配置改完立刻出故障，你会从哪几层排查？
答：看配置是否生效到预期实例、是否影响连接池/线程池/下游地址/超时阈值、是否存在动态刷新边界问题，以及不同环境配置是否混用。很多事故看起来像代码问题，根因其实是配置变更没有被当成高风险操作治理。

## 十五、回答升级模板

42. 如果想把场景题答得更像大厂后端，最值得主动补什么？
答：主动补四类内容：影响范围、止血动作、判断依据、长期治理。很多人会讲原因，但不会讲为什么先做这个动作、为什么这个指标能支撑结论、后面如何避免复发。真正成熟的场景题回答，必须同时体现工程判断和业务优先级。

43. 面试官为什么喜欢追问“还有别的可能吗”？
答：因为真实故障几乎从来不是单因单果。面试官想看的是你有没有建立假设树和排除路径，而不是靠单点经验硬猜根因。如果你能主动把问题分成应用、数据库、缓存、消息、网络、发布六层来想，答案会稳很多。

44. 如何把场景题和主项目绑定？
答：最有效的方法是挑主项目里真实存在的链路来练。例如 `agent-server-java` 适合重点练：长任务失败恢复、SSE 中断、任务状态不一致、模型调用超时、任务队列积压、任务结果存储和清理。这样场景题就不再是空题，而是项目深挖素材。
