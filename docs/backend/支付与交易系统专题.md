# 支付与交易系统专题

## 使用方式

这篇补的是后端面试里极高频、而且最容易把工程深度拉开的一个方向：

- 订单、支付、退款、回调、对账到底怎么串起来
- 为什么支付系统不只是“调第三方支付 API”
- 幂等、状态机、回调重复、资金安全、风控、补偿为什么都会集中出现在这里

即使你未来不专做支付，这篇也值得学，因为它几乎把后端高频核心问题压缩到了一个场景里。

---

## 一、为什么支付与交易系统是经典后端题

1. 为什么支付系统特别容易成为大厂后端高频题？
答：因为它同时考业务正确性和工程正确性。用户是否扣款成功、订单状态是否正确、回调是否重复、库存是否一致、退款是否安全、对账是否闭环，任何一个点出错都会带来真实业务损失。

2. 为什么支付题比普通 CRUD 更能体现后端水平？
答：因为这里没有“功能跑通就算完”。支付系统必须非常关注状态流转、幂等、防重、延迟回调、补偿、对账、审计和风控。它天然要求你站在“失败也必须正确”的视角设计系统。

---

## 二、交易链路最核心的 6 个问题

3. 第一个问题是什么？
答：交易状态机。至少要把待支付、支付中、支付成功、支付失败、已关闭、退款中、已退款这些状态和合法流转设计清楚。没有状态机，回调重复、超时关单、人工补单时很容易把状态搞乱。

4. 第二个问题是什么？
答：幂等。支付创建、回调处理、退款申请、退款回调、补偿重放，这些动作都可能重复触发。支付系统里最危险的不是失败，而是“重复成功”。

5. 第三个问题是什么？
答：回调可靠性。第三方支付回调可能延迟、重复、乱序，甚至先成功后超时。系统不能只靠同步请求结果判断支付状态，通常要以回调和主动查询结果共同收敛状态。

6. 第四个问题是什么？
答：一致性。订单状态、支付状态、库存状态、优惠券状态、积分状态、账单状态都可能相互影响。成熟系统要明确哪些属于主链路强一致，哪些属于副作用最终一致。

7. 第五个问题是什么？
答：对账。线上再稳定，也不能默认永远正确。支付系统必须有和支付渠道、内部账单、订单系统的对账能力，用来发现漏单、重复单、状态错单和金额不一致。

8. 第六个问题是什么？
答：资金安全和风控。虽然普通业务系统不一定真的自己管资金账户，但支付链路一定要防伪造回调、金额篡改、越权退款、异常频率操作和高风险账户行为。

## 三、主支付链路怎么讲

9. 一个成熟的支付主链路应该怎么描述？
答：先创建交易单或支付单，完成金额、订单归属和幂等校验；再调用支付渠道生成支付凭证；支付完成后以异步回调或主动查询驱动状态流转；后续通知、积分、发货、搜索索引更新等副作用通过异步方式处理。这样就把主链路和副作用链路分开了。

10. 为什么不能只信同步下单接口返回？
答：因为同步返回只代表“请求提交到支付渠道的结果”，不一定代表“最终扣款成功”。真正收敛支付状态通常依赖回调、查询补偿和对账，而不是一次接口响应。

11. 为什么支付成功后的副作用最好异步化？
答：因为支付主链路要尽量短、尽量稳。积分发放、通知、物流推进、风控记录、索引同步这些动作如果全压在主链路里，会显著放大故障面和时延。

---

## 四、支付回调和查询补偿怎么讲

12. 为什么支付回调几乎一定会被追问？
答：因为这是支付系统最容易出事故的地方。回调可能重复、伪造、延迟、顺序错乱、网络失败，所以必须做签名校验、幂等处理、状态校验、日志审计和必要的主动查询补偿。

13. 为什么回调和主动查询通常要一起存在？
答：因为回调不是 100% 可靠，查询也不适合无限制高频轮询。成熟系统通常以回调为主、查询为兜底：回调没来或状态不确定时，再通过定时任务或补偿任务查询支付渠道状态。

14. 为什么支付回调不应该直接更新所有业务状态？
答：因为回调本身可能重复、伪造或时序异常。更安全的做法通常是先经过交易状态机校验，只推进一次核心状态，再异步发出后续业务事件。

## 五、退款、关单和对账为什么很有后端味

15. 退款链路最值得讲什么？
答：退款申请幂等、原支付单关联、部分退款/全额退款语义、退款状态推进、退款回调处理和退款对账。很多人会讲支付成功，却忘了退款同样是高风险交易动作。

16. 为什么超时关单很容易出问题？
答：因为用户可能在超时边界附近刚好完成支付，这时就会出现“系统准备关单，但渠道已经扣款”的竞态。如果没有关单状态校验、支付查询补偿和异常单处理流程，状态就会打架。

17. 为什么对账不是财务的单独事情？
答：因为对账发现的问题最终还是要由后端系统修复。对账是交易系统自我纠错的一部分，它帮你发现漏单、错单、金额不一致和状态异常，然后触发补单、退款、人工介入或告警。

---

## 六、风控和审计在支付里为什么更重要

18. 支付链路最常见的风控点有哪些？
答：高频支付尝试、异常设备/地域、金额异常、短时间多次退款、越权退款、重复支付、异常回调、签名失败暴增。支付不是只关心成功率，也要关心风险事件。

19. 为什么支付系统特别需要审计？
答：因为很多动作都很敏感，比如人工补单、强制退款、状态修正、金额调整、商户配置修改。没有完整审计，后面出了事故很难追责和复盘。

## 七、和主项目怎么绑定

20. `agent-server-java` 这种项目怎么借支付系统思路？
答：即使它不是支付业务，也可以借支付系统的核心方法：严格状态机、回调治理、对账/校验、幂等、防重放、风险事件和补偿任务。这些方法对任务平台、配额平台、开放平台同样适用。

21. 如果真要把主项目做成商业化平台，最自然的支付切入点是什么？
答：工作空间套餐购买、模型调用额度充值、企业版订阅、付费高级能力解锁。这样就能自然引出支付单、回调、配额到账、退款和对账能力。

### 支付回调处理详解

1. 支付回调如何做幂等设计？
答：以支付流水号（如商户侧 `outTradeNo`，或与渠道唯一交易号的稳定映射）作为幂等键。处理回调前先查询本地是否已成功处理过该笔通知：若已成功，直接返回渠道要求的成功响应，不再重复推进订单状态、发货或加积分，避免「重复成功」。

2. 回调为什么要验签？怎么验？
答：防止攻击者伪造 HTTP 回调冒充支付成功。应严格按平台文档使用对应算法与平台公钥验证签名（如支付宝 RSA2），并在验签通过后核对通知中的商户订单号、金额、收款方标识等与本地单据一致，再执行业务更新。

3. 支付平台的回调重试机制对系统有什么要求？
答：渠道往往在较长时间窗口内多次重试（例如 24 小时内多次）。系统必须全程幂等、尽快返回明确结果；处理成功后返回平台约定的成功应答，避免被持续重试；处理失败时要区分可重试与不可重试，并记录日志便于排查。

4. 为什么不能完全依赖回调？
答：回调可能丢失、延迟或网络异常导致未及时到达。需要定时任务或补偿任务主动查询支付平台订单状态，与本地「支付中」等中间态对齐，作为回调的兜底，避免订单长期悬停。

5. 订单状态机可以怎么讲？
答：典型主路径：待支付 → 支付中 → 已支付 → 已发货 → 已完成；另有分支如超时/取消进入已关闭、退款完成后进入已退款等。每一次状态迁移都要定义合法源状态、互斥条件与副作用触发点，避免乱序或重复回调把状态写乱。

6. 什么是掉单？如何处理？
答：用户侧已支付成功，但回调未到达或处理失败，本地订单长时间停留在「支付中」。可通过定时扫描「支付中」超过 N 分钟的订单，主动调用渠道查询接口；若渠道已确认支付成功，则更新本地状态并补发发货、通知等后置流程，形成闭环。

### 退款与对账

1. 为什么说退款不是简单的「反向操作」？
答：需要校验退款金额不超过已支付金额、防止同一支付单重复退款、为退款单维护独立状态与幂等；部分退款、多次退款、渠道拒绝等都要单独建模，不能等价于一次简单的账务回冲。

2. 退款状态机怎么理解？
答：常见路径：退款申请 → 退款中（渠道处理中）→ 退款成功或退款失败。失败后可按业务规则重试或人工介入。与支付回调类似，退款结果通知也可能重复，必须幂等与状态校验。

3. 对账流程应该怎么描述？
答：每日（或按账期）通过支付平台账单下载 API 拉取对账单，与本地支付、退款流水逐笔比对金额、状态、手续费等；标记长款（渠道侧多账）、短款（本地多账）、金额不一致等差异，走人工审核或自动调账流程，并保留审计日志。

4. 资金安全在金额表示上有什么工程约定？
答：金额运算与持久化应使用 `BigDecimal`（或语言中等价的精确十进制类型），避免 `double` / `float` 的二进制浮点误差；存储习惯上以「分」为单位的整数表示，减少小数舍入问题；对外展示再格式化为「元」。
