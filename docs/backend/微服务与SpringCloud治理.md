# 微服务与 Spring Cloud 治理

## 使用方式

这个专题不是为了把所有微服务组件背一遍，而是为了回答下面几个高频问题：

- 什么情况下应该拆微服务，什么情况下不该拆。
- 微服务拆完以后会新增哪些复杂度，如何治理。
- `Spring Cloud` 常见组件分别解决什么问题，代价是什么。
- 面试里如果被追问注册发现、配置中心、网关、限流熔断、链路追踪，如何讲到工程层而不是只讲名词。

你读这篇时，建议始终带着一个问题：

“如果把我的 `Java` 主项目从单体演进到微服务，我到底为什么拆、怎么拆、拆完后如何保证稳定性和研发效率？”

---

## 一、先判断要不要拆

1. 什么是微服务？
答：微服务本质上不是“把一个项目切成很多小项目”，而是把一个复杂系统按业务边界拆成多个可以独立开发、部署、扩缩容和治理的服务单元。它解决的核心问题，不是让架构看起来高级，而是当业务复杂度、团队规模、发布节奏、性能瓶颈已经无法被单体架构很好承接时，给系统提供更强的演进能力。

2. 单体架构和微服务架构的核心区别是什么？
答：单体的优势是开发、部署、调试、联调都更直接，复杂度低，适合业务早期和团队规模较小时快速交付；微服务的优势是模块边界更清晰、可以按服务独立扩容和发布，也更容易让不同团队并行协作。但微服务会引入网络调用、分布式事务、服务治理、链路追踪、配置管理、灰度发布这些新复杂度，所以它不是默认更好，而是规模增长后的权衡结果。

3. 什么情况下适合继续单体？
答：当业务还在快速探索期、核心模块不多、团队规模不大、服务边界尚不稳定、系统吞吐也没到明显瓶颈时，我更倾向先做模块化良好的单体。因为这时候最大的矛盾通常不是“服务拆得不够细”，而是业务本身还在变，如果过早拆服务，很容易把原本内部函数调用的问题，升级成跨网络、跨仓库、跨团队的分布式复杂度。

4. 什么情况下才值得考虑拆微服务？
答：当出现下面这些信号时，微服务才更有讨论价值：第一，不同业务域已经比较稳定，例如订单、库存、支付、用户中心之间边界清楚；第二，单体发布风险越来越高，一个小改动也可能影响全站；第三，不同模块资源模型差异很大，部分链路需要独立扩容；第四，团队协作已经明显受目录、依赖和发布耦合拖累。也就是说，拆微服务通常是为了解决组织和系统同时变复杂的问题。

5. 面试里如果问“为什么不直接微服务”，怎么回答更好？
答：我会先强调架构选择要匹配业务阶段，而不是匹配流行词。早期单体更适合快速验证、降低运维复杂度和缩短反馈链路；只有当业务边界、团队分工、发布风险、扩容诉求都逼近单体上限时，微服务才值得引入。这样回答比直接说“单体不好扩展，所以要上微服务”更成熟，因为它体现的是取舍，而不是盲目崇拜复杂架构。

## 二、微服务拆分原则

6. 微服务一般按什么维度拆？
答：我更倾向先按业务域拆，而不是按技术层拆。比如用户、订单、库存、营销、通知、搜索这些更像天然业务边界；而如果按 controller、service、dao 这种技术层拆，最后往往只是把一个单体硬切成几个高度耦合的远程模块，既没有真正独立性，还多了网络成本和协作摩擦。

7. 服务拆得越细越好吗？
答：不是。服务拆得太细，虽然表面上模块更小，但调用链会变长，事务边界更碎，联调、排障、发布、权限和监控复杂度都会上升。一个成熟的回答方式通常是：服务粒度既不能粗到把多个业务域揉在一起，也不能细到一个完整链路要经过太多服务，最终要看业务边界是否稳定、团队 ownership 是否清楚、调用链是否还能被有效治理。

8. 如何判断一个边界拆得合理？
答：我通常看三件事：第一，服务的变化原因是否相对单一；第二，这个服务能不能对外提供比较稳定的领域能力，而不是暴露大量内部细节；第三，拆分后有没有明显减少团队协作和发布上的耦合。如果一个服务今天改库存字段，明天改用户资料，后天又改营销规则，那通常说明边界还不干净。

9. 数据库应该按服务拆吗？
答：从理想状态看，微服务通常强调每个服务管理自己的数据边界，避免多个服务同时直接写同一张核心表；但真实项目里是否立刻做到库表彻底隔离，要看演进阶段。如果一开始就强行拆到底，成本可能非常高；更稳的做法通常是先明确写入边界和 ownership，再逐步收敛到“谁拥有数据，谁负责写，其他服务通过接口或事件获取”。

10. 微服务拆分后最容易低估的问题是什么？
答：很多人只想到服务数量变多，却低估了调用链、配置、环境、鉴权、流量治理、容灾、日志追踪、接口兼容、数据一致性这些横切复杂度。也就是说，拆服务真正难的从来不是新建几个工程，而是把原来单机内调用时几乎不存在的问题，全部搬到了网络和组织协作层面。

## 三、Spring Cloud 常见组件解决什么问题

11. 注册中心解决什么问题？
答：在微服务环境里，服务实例会频繁扩缩容、重启、迁移，如果每个调用方都写死对方地址，维护成本非常高。注册中心解决的是“服务实例在哪里、当前有哪些可用节点、调用方如何发现它们”这个问题。你可以把它理解成微服务里的地址簿和健康清单。

12. 配置中心为什么重要？
答：微服务一多，环境配置、数据库地址、限流阈值、开关、超时、灰度参数都会迅速膨胀。如果这些配置都散落在各服务本地文件里，不仅难统一管理，也很难安全变更和快速回滚。配置中心解决的是配置集中管理、环境隔离、动态刷新和审计追踪问题，它本质上是服务治理的基础设施。

13. 网关的核心作用是什么？
答：网关是所有外部流量进入后端服务体系的统一入口。它通常承担路由转发、鉴权、限流、灰度、日志、协议适配、统一错误处理等职责。面试里如果只说“网关就是转发请求”，就太浅了；更完整的说法是：网关把原本散落在各服务里的横切流量治理能力集中收口，从而降低重复实现和策略不一致的问题。

14. 为什么微服务里常讲限流、熔断、降级？
答：因为服务之间已经是网络调用关系，上游慢、下游挂、依赖雪崩都可能沿调用链快速放大。限流是控制进入系统的流量上限，避免资源被打爆；熔断是当下游异常率或延迟持续恶化时，主动快速失败，避免调用方被拖死；降级则是在资源紧张或依赖异常时，主动关闭部分非核心能力，优先保核心路径。

15. 链路追踪为什么是微服务的必需品？
答：单体里查问题时，很多调用都发生在同一进程内，而微服务里一次请求可能经过网关、用户服务、订单服务、库存服务、MQ、缓存、数据库多个节点。如果没有统一的 traceId 和链路追踪，你看到的就只是一堆分散日志，很难回答“这次请求到底卡在哪、失败在哪、是哪个依赖先出问题”。

## 四、治理能力怎么讲得更像生产系统

16. 微服务治理具体在治理什么？
答：我会把它拆成四类：第一类是流量治理，例如路由、限流、熔断、重试；第二类是服务治理，例如注册发现、健康检查、配置管理；第三类是可观测性，例如日志、指标、链路追踪、告警；第四类是发布治理，例如灰度、回滚、版本兼容和配置变更管理。这样回答，面试官能感受到你理解的是一整套系统，不是单个组件。

17. 服务间调用为什么不能只靠重试解决问题？
答：因为重试本身也会放大压力。如果下游已经过载，你简单重试只会把瞬时流量进一步放大，甚至造成重试风暴。更成熟的做法是：先区分错误类型，再结合幂等性、超时、退避、限流和熔断决定是否重试。也就是说，重试必须建立在“错误可恢复 + 操作可安全重放 + 系统还有余量”这三个前提之上。

18. 微服务里为什么强调接口兼容性？
答：因为不同服务的发布节奏不一定同步，调用方和被调用方经常处于不同版本。如果接口变更不兼容，很容易导致灰度阶段、一部分节点新一部分节点旧时出现线上故障。所以在微服务体系里，接口设计不能只考虑“现在能不能用”，还要考虑旧版本兼容、字段扩展策略和发布窗口内的共存问题。

19. 微服务和分布式事务为什么经常一起出现？
答：因为一旦数据分散在多个服务里，一个业务动作就可能跨多个写入边界。比如下单要写订单、扣库存、发优惠、记积分，这些已经很难再靠单数据库本地事务一次包住。此时你就必须面对一致性、补偿、幂等和最终一致性的问题，所以微服务和分布式事务天然是绑定出现的。

20. 面试里如何讲“微服务收益和代价”？
答：我会先讲收益：边界更清晰、服务可独立扩缩容、发布影响面更可控、团队能并行协作；再讲代价：调用链更长、数据一致性变难、配置与环境更复杂、排障难度更高、基础设施投入更大。最后再补一句，微服务不是目标，目标始终是用合适复杂度支撑业务增长，这样结构会更完整。

## 五、主项目可以怎么演进

21. 如果把你的主项目从单体演进到微服务，第一步会怎么做？
答：我不会一开始就拆成很多服务，而是先在单体内做清晰的模块边界，把用户、权限、资源、任务、通知、审计这些领域拆清楚，让代码结构先模块化。只有当模块边界在单体内已经稳定、且确实出现独立扩缩容或独立发布诉求时，我才会优先把边界最清楚、收益最明显的服务先拆出去。

22. 拆出去的第一个服务通常怎么选？
答：通常会优先选边界清楚、依赖相对独立、风险可控、收益明确的模块，例如通知服务、文件服务、搜索服务、任务执行服务，而不是一开始就拆订单主链路或用户核心链路。这样做的好处是可以先把服务发现、配置、日志、监控、网关这些治理链路跑通，再逐步把更核心的业务迁移出来。

23. 如果面试官追问“什么时候不该拆”，怎么回答？
答：我会直接说三类不该拆的场景：业务还在快速探索期，边界不稳定；团队规模不大，单体协作成本还可控；系统瓶颈主要在数据库、缓存或代码质量，而不是架构边界。因为很多问题并不是拆微服务就能解决，比如 SQL 设计差、缓存策略差、测试不足、发布流程混乱，拆服务只会让问题扩散到更多节点。

## 八、注册中心、配置中心、网关的故障思维

24. 注册中心最怕什么问题？
答：最怕服务实例实际已经不可用，但注册信息没有及时摘除，导致调用方持续打向坏节点；或者网络分区、心跳异常、实例频繁抖动，造成服务发现结果不稳定。也就是说，注册中心不只是“有个地址簿”这么简单，它本身也是高可用组件，一旦异常会影响整个调用面。

25. 为什么注册中心故障会放大成全局事故？
答：因为它通常是所有服务发现能力的共同依赖。如果新实例注册不上、旧实例下不来、调用方拿不到最新地址或者不断刷新失败，很多服务间调用都会一起异常。所以成熟系统通常会强调本地缓存、健康检查、故障摘除和注册中心自身高可用部署。

26. 配置中心最危险的不是“配置太多”，而是什么？
答：是错误配置被大范围快速传播。比如数据库地址配错、限流阈值调错、超时配置过短、开关误开，全都可能在动态刷新后立刻影响大量实例。配置中心越强大，越需要审计、灰度、回滚和变更边界控制。

27. 为什么配置动态刷新不是越多越好？
答：因为不是所有配置都适合热更新。像线程池、连接池、核心安全参数、路由规则这些配置，如果刷新边界控制不好，可能在运行中制造更复杂的问题。成熟系统会区分“可以动态调”“建议重启生效”“必须分批灰度”三类配置。

28. 网关层最容易被低估的问题是什么？
答：很多人只把网关当成路由层，低估了它在鉴权、限流、灰度、日志、协议转换、跨域、统一错误处理里的系统性作用。网关一旦配置错误、性能不足或熔断策略不当，会同时影响很多服务，所以它往往既是能力中心，也是高风险点。

## 九、微服务治理里的高频事故场景

29. 服务调用 RT 飙升时，你如何判断是单点问题还是全链路问题？
答：我会先按调用链拆：是入口网关慢、某个下游服务慢、数据库慢、缓存失效还是线程池打满；再看异常是否集中在某个版本、某个机房、某类接口。微服务场景里“变慢”经常是级联效应，不先拆调用链，很容易把局部问题误判成全局问题。

30. 为什么微服务里比单体更容易出现重试风暴？
答：因为每一层调用方都可能各自带重试。如果网关重试一次，服务 A 再重试一次，服务 B 还重试一次，最终一个失败请求可能被放大成多次下游打击。微服务治理里，重试策略必须有统一边界，不能每层都“好心”帮忙重试。

31. 接口兼容问题为什么在微服务里特别致命？
答：因为发布节奏是分散的，新旧服务天然会共存。如果提供方直接删字段、改语义、改默认值，调用方即使没改代码也可能出问题。所以微服务里的接口设计必须优先考虑向后兼容、灰度窗口共存和版本过渡，而不是只关注当前功能是否跑通。

32. 为什么微服务里必须有“降级边界”？
答：因为不是所有依赖故障都值得把主链路一起拖死。比如评论服务挂了，不一定要让下单也挂；推荐服务异常，不一定要让搜索不可用。降级边界说到底是在问：哪些能力是系统核心，哪些能力异常时可以临时退化。没有这个意识，系统很容易被非核心依赖拖垮。

## 十、治理能力如何映射到主项目

33. 如果把 `agent-server-java` 从单体往微服务演进，哪几个治理能力最值得先补？
答：我会优先补四层：第一，统一网关和认证入口；第二，任务执行服务的注册发现和健康检查；第三，配置中心和开关治理；第四，任务链路的 traceId 和日志关联。因为这些能力一旦补上，系统才算从“单服务工程”迈向“可治理的服务体系”。

34. 为什么 `agent-server-java` 适合先拆任务执行服务？
答：因为任务执行本身就具有独立扩缩容、长任务隔离、资源消耗模型不同、故障影响面可控这些特点。把它从主 API 服务里拆出来，既能减少主链路压力，也特别适合展示微服务演进、异步系统和可观测性治理能力。

35. 如果面试官追问“微服务演进第一步不是拆服务，而是什么”，怎么答？
答：我会回答：第一步通常不是立刻拆，而是先把单体做成边界清楚的模块化单体，理清领域边界、调用关系、写入 ownership、配置与日志规范。因为如果单体内部还是混乱的，直接拆微服务只会把混乱复制到更多仓库和更多网络调用里。

## 十二、注册中心对比：Nacos vs Eureka vs Consul

1. Eureka（Netflix，已停更）在 CAP 里更偏哪一侧？客户端发现机制是怎样的？
答：Eureka 设计更偏 **AP（可用性 + 分区容忍）**：注册中心集群在网络分区时仍尽量对外提供发现能力，牺牲强一致。客户端会 **缓存本地服务列表**（`DiscoveryClient` 定期从 Server 拉取并更新），因此 **注册中心短时不可用** 时，调用方仍可能用 **上一次拉到的实例列表** 继续调用（直到缓存过期或本地判断不健康），但可能打到已下线的实例，需要配合客户端负载均衡与健康探测兜底。

2. Eureka 的心跳与续约默认参数是什么？
答：实例向 Server **续约（renew）** 的默认周期常配为 **30 秒**：`eureka.instance.lease-renewal-interval-in-seconds`（默认 30）。Server 侧若超过 **`eureka.instance.lease-expiration-duration-in-seconds`**（默认 90 秒，约为 3 个续约周期）未收到续约，会将实例标记为可剔除。客户端拉取注册表的间隔可用 `eureka.client.registry-fetch-interval-seconds` 等配置控制。

3. Eureka 的「自我保护」是什么？相关配置项有哪些？
答：**自我保护（Self-Preservation）**：当 Server 在短时间内收到 **大量续约失败**（例如网络抖动、Client 批量宕机）时，**暂不剔除** 实例，避免把仍存活但暂时失联的节点全部摘掉导致雪崩。核心开关：`eureka.server.enable-self-preservation`（默认 true）。还会涉及续约阈值比例等 Server 侧参数（如 `renewal-percent-threshold` 一类配置，具体以所用 Spring Cloud Netflix 版本文档为准）。代价是可能短时间内容忍 **僵尸实例** 仍在列表中，需要客户端重试、超时与熔断配合。

4. Nacos 与 Eureka 在一致性模型上最大差异是什么？
答：Nacos **同时支持 AP 与 CP 模式**（可切换），而不是像经典 Eureka 那样固定偏 AP。**AP 模式**下更侧重注册发现的高可用与最终一致；**CP 模式**下更侧重强一致（底层采用 **Raft** 等共识），适合对一致性敏感的场景（例如部分配置或元数据场景，具体以官方说明为准）。

5. Nacos 里「临时实例」和「持久实例」分别适合什么？和 AP/CP 怎么对应？
答：**临时实例（ephemeral）**：默认与 **心跳检测** 绑定，类似传统服务发现，实例下线或心跳中断后较快从列表移除，适合 **动态扩缩容的服务发现**，常对应 **AP** 思路。**持久实例（persistent）**：更像人工登记或长生命周期节点，可配合 **CP/Raft** 做更强一致存储，适合需要 **明确存在性** 的注册场景。注册时可用 `ephemeral: true/false`（或 Spring Cloud Alibaba `spring.cloud.nacos.discovery.ephemeral`）区分。

6. Nacos 除了注册发现还提供哪些一体化能力？
答：同一套 Nacos 体系常同时提供：**服务发现**、**配置中心**（DataId、Group、Namespace 隔离）、**DNS 友好名解析**（视部署与集成方式）、以及 **健康检查**（临时实例多用心跳，持久实例可配 TCP/HTTP 等）。面试可概括为「注册 + 配置 + 命名 + 健康」一体化，减少组件数量。

7. Consul 的一致性模型与部署特点是什么？
答：Consul 服务端基于 **Raft** 实现 **CP** 导向的存储与一致性（强调一致性与分区下的行为，与具体部署和只读 Follower 查询有关）。能力上覆盖 **服务发现**、**KV 配置**、**健康检查**（HTTP/TCP/Script 等）、以及 **Connect / Service Mesh**（mTLS、意图策略等）。部署上典型是 **每台机器跑 Consul Agent**（Client 模式）本机注册与健康检查，再与 Server 集群通信。

8. 注册中心选型可以怎么一句话建议？
答：**Java / Spring Cloud 国内项目** 多选 **Nacos**（生态成熟、功能全、与 Spring Cloud Alibaba 集成顺）。**多语言栈 + Mesh 诉求** 可重点看 **Consul**。**老 Spring Cloud Netflix 栈** 可能仍在 **Eureka**（需注意停更与长期维护成本）。最终仍要看团队运维能力、一致性要求、是否已有配置中心与 K8s 服务发现等约束。

## 十三、服务间通信：Feign vs Dubbo vs gRPC

1. OpenFeign 是什么？典型用法与优缺点？
答：OpenFeign 是 **声明式 HTTP 客户端**：用 **接口 + 注解**（如 `@FeignClient`、`@GetMapping` / `@PostMapping`）描述远程 HTTP API，运行时生成代理做 **HTTP 调用**，与 Spring MVC 注解风格接近。优点是 **上手快、与 REST 生态一致、易调试**；缺点是 **基于文本 HTTP/JSON** 时 **序列化与协议开销** 通常大于二进制 RPC，高 QPS 下 CPU 与带宽压力更明显。

2. Feign 底层可调哪些 HTTP 客户端？和超时相关配置一般在哪里？
答：常见底层实现包括 **JDK `HttpURLConnection`**、**Apache HttpClient**、**OkHttp** 等（由依赖与 `spring.cloud.openfeign.httpclient` / `okhttp` 等配置选择）。超时多与 **Ribbon/LoadBalancer 读取超时**、**连接超时** 以及 `feign.client.config.<contextId>.connectTimeout` / `readTimeout`（或全局 `default`）相关，需与 **重试**、**熔断** 策略一起看，避免叠加重试放大流量。

3. Dubbo 的定位与性能特点是什么？
答：Dubbo 是阿里开源的 **高性能 RPC 框架**，默认偏向 **二进制序列化**（如 **Hessian2**、Kryo 等，可按协议与序列化配置），减少报文体积与解析成本。支持多种 **注册中心**（Nacos、Zookeeper、Redis 等）与 **负载均衡、路由、超时、重试、泛化调用** 等治理能力。**Triple 协议** 基于 HTTP/2，**兼容 gRPC**，便于与多语言互通。

4. Dubbo 里哪些配置项最常用来表达「调用契约」与「治理」？
答：消费方/提供方常见的有：**`interface`（接口全限定名）**、**`group` / `version`**（多实现隔离）、**`timeout`**、**`retries`**、**`loadbalance`**（如 random、roundrobin、leastactive）、**`cluster`**（如 failover、failsafe）。注册中心侧常用 **`registry` 地址**、**`namespace`** 等。具体以 Dubbo 3.x 与 Spring Boot Starter 的配置属性为准（如 `dubbo.application`、`dubbo.registry`、`dubbo.consumer`）。

5. gRPC 的协议栈与接口契约长什么样？
答：gRPC 基于 **HTTP/2**（多路复用、头部压缩、二进制帧）+ **Protocol Buffers（protobuf）** 作为 **IDL**，用 `.proto` 文件定义 **service 与 message**，生成各语言 **stub**。强类型、前后向兼容需通过 **字段编号、optional/required 语义演进规则** 等约束。

6. gRPC 支持的四种通信模式分别是什么？
答：**Unary**：一问一答；**Server streaming**：客户端一次请求，服务端流式返回多条；**Client streaming**：客户端流式发送，服务端一次汇总响应；**Bidirectional streaming**：双向流，读写独立。选型时看业务是 **拉取列表/推送** 还是 **长连接双向事件**。

7. Feign、Dubbo、gRPC 怎么按场景选型？
答：**内部以 Java 为主、Spring 生态、REST 已铺开** → **OpenFeign** 或 **Spring Cloud LoadBalancer** 组合足够；**追求性能、内部 RPC、治理丰富** → **Dubbo**（尤其已有 Java 服务矩阵）。**多语言客户端/服务端、强契约、移动端与后端统一接口描述** → **gRPC** 更合适。**极简场景**（少量调用、团队熟悉 HTTP）→ 优先 **Feign**，避免为 RPC 引入额外运维与工具链成本。

## 十四、Sentinel 限流熔断详解

1. Sentinel 里的「资源」是什么？如何定义？
答：**资源**是 Sentinel 保护的目标，可以是 **方法、接口、代码块甚至热点参数**。常见方式：在方法上使用 **`@SentinelResource(value = "resourceName", blockHandler = "...", fallback = "...")`**；或通过 **`SphU.entry("resourceName")` / `Entry` API** 手动定义。`blockHandler` 处理 **被流控/熔断等规则拦截**；`fallback` 处理 **业务异常**（可配置例外类不进入 fallback）。

2. 流控规则里 QPS 限流与线程数限流区别是什么？对应哪些字段？
答：**QPS 限流**：限制 **每秒通过的请求数**，侧重 **到达速率**（`grade` 常取 **FLOW_GRADE_QPS**，`count` 为阈值）。**线程数限流**：限制 **并发执行线程数**，侧重 **占用线程/慢调用挤占**（`grade` 为 **FLOW_GRADE_THREAD**，`count` 为并发线程阈值）。二者解决的是 **“来得太快”** 与 **“同时执行太多”** 两类不同瓶颈。

3. 流控效果「直接拒绝」「Warm Up」「排队等待」分别是什么？Warm Up 的 coldFactor 默认值？
答：**直接拒绝（CONTROL_BEHAVIOR_DEFAULT）**：超过阈值立即 **快速失败**。**Warm Up（预热，CONTROL_BEHAVIOR_WARM_UP）**：从较低阈值 **逐步升到** `count`，避免冷启动突刺；**`coldFactor` 默认 3**（即初始可用阈值约为 `count / coldFactor` 量级，具体计算以实现为准）。**排队等待（CONTROL_BEHAVIOR_RATE_LIMITER，匀速器）**：请求 **按固定间隔排队通过**，类似 **漏桶/匀速排队**，`maxQueueingTimeMs` 控制最长排队时间，超时则拒绝。

4. 熔断规则里三种策略各看什么指标？常用字段名？
答：**慢调用比例（SLOW_REQUEST_RATIO）**：当 **RT 超过 `maxSlowRequestMs` 的请求占比例** 高于 **`minRequestAmount` 窗口内统计的阈值比例 `count`（0~1）** 时打开熔断。**异常比例（ERROR_RATIO）**：异常数占请求数比例超过 **`count`**。**异常数（ERROR_COUNT）**：窗口内异常数超过 **`count`**。共有字段还包括 **`timeWindow`（熔断时长）**、**`minRequestAmount`（最小样本数）**、**`statIntervalMs`（统计窗口）** 等（以 Sentinel 版本文档为准）。

5. 热点参数限流解决什么问题？关键配置是什么？
答：对 **同一接口** 内 **某一参数值**（如 **商品 ID、用户 ID**）单独限流，避免 **个别热点** 拖垮全局。通过 **`ParamFlowRule`** 指定 **`resource`**、**参数索引 `paramIdx`**、**限流模式** 与 **`count`**，并可配 **参数例外项 `paramFlowItemList`**（特定参数值单独阈值）。

6. 系统保护规则（SystemRule）看哪些维度？常见字段？
答：**系统自适应保护**：如 **`highestSystemLoad`（系统 load1 阈值）**、**`avgRt`（全局平均 RT）**、**`maxThread`**、**`qps`**、**`highestCpuUsage`（CPU 使用率）** 等，超过阈值时 **拒绝新请求**。用于 **整实例级别** 兜底，防止单服务被整体打满，与针对单资源的流控互为补充。

7. Sentinel 控制台（Dashboard）能做什么？规则如何下发？
答：Dashboard 提供 **资源实时监控**、**规则配置与推送**（需应用侧引入 **`sentinel-transport-simple-http`** 等模块与客户端连接控制台），支持 **动态修改流控/熔断/系统规则** 并推送。生产环境常将规则 **持久化到 Nacos、Apollo 等** 配置中心，避免仅存在内存中重启丢失。

8. Sentinel 与 Hystrix 在隔离与统计上有什么核心区别？
答：**Hystrix** 常用 **线程池/信号量隔离**，**线程池隔离** 资源开销大（线程上下文切换与池管理），但隔离彻底。**Sentinel** 默认更偏向 **信号量 + 滑动窗口统计**（轻量），配合 **多种流控效果与系统规则**，在 **规则维度、热点、系统保护** 上更细；熔断语义上也基于 **慢调用比例/异常比例/异常数** 等 **更直观的指标模型**。选型上新项目在 Spring Cloud Alibaba 生态多优先 **Sentinel**，老项目可能仍见 **Hystrix/Resilience4j**。
