# 短链接与 ID 生成专题

## 使用方式

这篇解决两个面试里经常被单独拎出来问的问题：

- 短链接系统怎么设计
- 全局唯一 `ID` 怎么生成

它们经常一起出现，因为短链接要解决“长串映射成短码”，很多业务系统也要解决订单号、任务号、消息号、流水号的生成问题。你不能只会背“雪花算法”，还要知道什么时候用数据库自增、号段模式、随机串、哈希映射和业务前缀。

---

## 一、为什么这两个主题值得放一起

1. 为什么短链接和 `ID` 生成经常一起考？
答：因为它们都在处理“唯一标识如何生成、如何路由、如何避免冲突、如何支撑高并发和高可用”这类问题。一个更偏对外可见链接，一个更偏系统内部主键或业务流水。

2. 为什么这类题特别像后端系统设计题？
答：因为它不只是一个算法问题，而是会牵扯数据存储、热点治理、缓存、过期回收、路由规则、碰撞处理、监控和风控。面试官很容易从这里追问到工程细节。

---

## 二、短链接系统最核心的 6 个问题

3. 第一个问题是什么？
答：短码如何生成。常见方式有自增转码、随机串、哈希截断、预生成号段等。不同方案在长度、碰撞概率、可读性和枚举风险上差异很大。

4. 第二个问题是什么？
答：映射关系存在哪里。核心表一般至少要保存短码、原始长链接、创建人、过期时间、状态、访问策略和统计字段。短链不是只存一对映射，后续常常还会加权限、渠道和归因信息。

5. 第三个问题是什么？
答：跳转链路怎么做缓存。热门短链天然会形成热点，如果每次跳转都查数据库，读取成本会很高。一般会采用本地缓存或 `Redis` 做热点保护。

6. 第四个问题是什么？
答：过期和失效怎么处理。很多短链是有生命周期的，比如营销活动、登录邀请、一次性分享链接。如果不设计过期机制，数据会持续膨胀且存在安全风险。

7. 第五个问题是什么？
答：安全与风控。短链接很容易被用来做钓鱼、恶意跳转和流量污染，所以常常要和内容审核、域名风控、访问频控、黑白名单联动。

8. 第六个问题是什么？
答：统计口径。短链系统经常不仅要跳转，还要统计点击、独立访客、渠道来源、设备信息和转化链路。统计链路通常不能阻塞主跳转请求。

## 三、短链接生成方案怎么取舍

9. 自增转码是什么？
答：先拿到一个全局唯一整数，再把它转换成 `62` 进制短码。这种方案实现简单、可控性好、长度可预测，但如果直接暴露连续规律，可能被枚举扫描。

10. 随机串方案有什么特点？
答：生成随机短码并做唯一性校验。优点是不可预测，安全性更好；缺点是会有碰撞概率，需要重试，而且长度和存储成本通常更高。

11. 哈希截断为什么不适合作为唯一主方案？
答：因为长链接很多，不同原链接哈希截断后很容易碰撞。它更适合作为辅助去重或校验工具，而不是绝对唯一键。

12. 什么是号段预生成方案？
答：系统批量拿一段可用编号，本地或业务节点自行编码生成短码。这样能降低中心依赖，但要处理号段发放、回收和浪费问题。

## 四、ID 生成系统最核心的 6 个问题

13. 为什么业务系统不能只用数据库自增主键？
答：因为数据库自增简单，但在分库分表、多机房、多业务线下可扩展性有限，而且可能暴露业务规模和增长规律。很多对外可见单号也不适合直接用裸自增。

14. 数据库自增适合什么场景？
答：单库、单服务、规模不大、只作为内部主键时很合适。它简单、稳定、事务一致性好，但不是所有场景都适合作为全局发号器。

15. 雪花算法解决了什么问题？
答：它在分布式环境下生成趋势递增、全局基本唯一的 `64` 位整数，通常组合时间戳、机器标识和序列号，从而避免每次都去中心服务取号。

16. 雪花算法最大的风险是什么？
答：时钟回拨。只要机器时间回退，就可能导致重复 `ID` 或排序异常。所以生产里必须考虑时钟回拨检测、阻塞、切换备用节点或直接降级。

17. 号段模式适合什么场景？
答：高吞吐发号、业务号生成、多语言调用场景很适合。中心服务一次分配一段号，本地批量消费，吞吐高且实现简单，是很多公司常见的工程方案。

18. 业务单号为什么常常不是“纯数字唯一 ID”？
答：因为业务号还要承载业务前缀、日期、租户、渠道、地区、可读性和人工排查便利性。所以系统里经常同时存在“内部主键 ID”和“外部业务单号”两套标识。

## 五、短链与发号系统的高频工程问题

19. 为什么短链容易出现热点？
答：因为传播爆发时，某个短码可能在极短时间内被大量访问。热点短链通常需要缓存、限流、降级和静态化保护，否则数据库和下游统计系统会被打爆。

20. 为什么发号系统也会成为基础设施？
答：因为订单、支付、任务、消息、审批、审计日志都可能依赖统一发号。如果这个系统抖动，很多业务会一起受影响，所以它常常被当成平台型基础能力建设。

21. 为什么 `ID` 生成也要看排序性？
答：因为有些系统需要按时间大致有序，便于索引写入、日志追踪和问题排查。完全随机的 `UUID` 在数据库索引局部性和排查体验上通常不如趋势递增 `ID`。

22. 为什么不能只回答“用 UUID 就行”？
答：因为 `UUID` 虽然简单，但字符串长、存储大、索引局部性差、可读性弱。它能用，但在高并发订单、消息、日志等系统里不一定是最优解。
