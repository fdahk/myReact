# 开放平台与 Webhook 回调治理专题

## 使用方式

这篇补的是平台型后端里非常高频、但很多候选人答得很浅的一类问题：

- 系统对外开放 API 时，后端要多考虑什么
- `Webhook` 回调为什么不是“发个 HTTP 请求”就结束
- 签名、防重放、幂等、重试、审计和风控为什么会全部串到一起

如果你以后要做开放平台、第三方集成、支付回调、消息推送、异步通知，这篇很关键。

---

## 一、为什么开放平台和回调值得单独讲

1. 为什么开放平台不是普通内部接口的复制版？
答：因为开放平台面对的是外部开发者、第三方系统和不可控调用方。内部接口可以靠团队协作解决很多问题，但开放平台必须从一开始就考虑认证、签名、版本兼容、配额、文档、审计、风控和灰度，否则后面会非常难维护。

2. 为什么 `Webhook` 回调在面试里很高频？
答：因为它几乎天然包含了异步通知、签名校验、重试、幂等、防重放、失败补偿和可观测性这些后端核心问题。一个回调链路答得深不深，很容易看出候选人有没有做过生产系统。

---

## 二、开放平台最核心的 6 个问题

3. 第一个问题是什么？
答：调用方身份识别。常见做法是 `appId + secret`、访问令牌、签名鉴权、IP 白名单或它们的组合。关键不在于用了哪一种，而在于你要能说明为什么这样做，以及密钥轮换和泄露后怎么处理。

4. 第二个问题是什么？
答：请求签名。开放平台请求通常不能只靠 token，还要对请求体、时间戳、随机串、路径和关键参数做签名，避免参数被篡改或被伪造调用。真正高频的追问不是“会不会签名”，而是“签名串怎么拼、时间窗口多长、重放怎么防”。

5. 第三个问题是什么？
答：版本兼容。开放平台一旦被第三方接入，破坏式变更的代价很高。你要优先考虑新增字段、兼容旧版本、提供迁移窗口和淘汰策略，而不是像内部系统那样随手改接口。

6. 第四个问题是什么？
答：配额与限流。开放平台不是“给出去就完了”，还要控制调用频率、每日额度、套餐能力和异常流量。否则很容易被滥用、被刷爆或让成本失控。

7. 第五个问题是什么？
答：回调可靠性。平台给第三方发回调时，要考虑回调失败怎么办、重试策略怎么定、回调顺序是否重要、失败记录如何追踪、人工重放如何做。

8. 第六个问题是什么？
答：审计与风控。开放平台比内部接口更需要知道谁调用了什么、调用频率是否异常、签名失败是否暴增、是否出现重放攻击和恶意刷取。平台型后端很大一部分治理价值就在这里。

## 三、Webhook 回调为什么特别像真实后端题

9. 一个成熟的回调链路至少包括什么？
答：事件生成、投递任务、签名封装、目标地址校验、超时控制、失败重试、幂等约束、回调日志、告警和人工重放。缺任何一个点，线上都容易出问题。

10. 为什么回调不能同步阻塞主业务链路？
答：因为第三方系统不稳定、网络不可控、超时不可预测。如果把回调直接放在主交易或主任务链路里，很容易把核心业务拖慢甚至拖挂。所以成熟系统通常会异步投递回调，并把回调结果独立跟踪。

11. 为什么回调接收方也必须做幂等？
答：因为发送方几乎一定会重试，只要网络超时或接收方没明确返回成功，发送方就不能假设“对方已经处理成功”。所以回调消费方必须基于事件 ID、业务 ID 或幂等键做防重复处理。

12. 为什么发送方也要关注幂等？
答：因为同一事件可能被多次调度、人工重放、故障恢复或消息重复投递。如果发送方自己都没有事件唯一性和投递状态治理，就会让整个回调链路越来越乱。

---

## 四、签名、防重放和时间窗口怎么讲

13. 为什么只做签名校验还不够？
答：因为签名能证明“请求没被篡改”，但不能防止“历史合法请求被重复发送”。所以通常还要引入时间戳、nonce、短时缓存去重或事件唯一 ID，才能真正抵抗重放攻击。

14. 时间窗口为什么不能太大也不能太小？
答：太大会增加重放风险，太小又容易因为时钟漂移和网络延迟误杀合法请求。成熟系统通常会设置一个有限窗口，并配合 `nonce` 或事件 ID 做二次防护，而不是只靠一个时间戳。

15. 回调签名最容易被追问的点是什么？
答：签名原文怎么拼、是否包含请求体、时间戳放哪、重试时签名是否复用、密钥轮换如何兼容旧签名、失败日志是否会泄露敏感信息。这些都是很有生产味的追问。

---

## 五、重试、死信和人工重放怎么讲

16. 为什么回调失败不能无限重试？
答：因为无限重试会放大故障、浪费资源、制造噪音，还可能把对方系统压垮。成熟系统会设重试次数、退避策略、超时预算和最终失败池，而不是一直打。

17. 为什么回调链路要支持人工重放？
答：因为有些失败不是靠自动重试就能恢复，例如第三方地址临时配置错误、对方短时间停机、签名密钥切换错误。此时需要运营或研发在确认风险后手动重放，而不是只能改库或重新造事件。

18. 为什么回调也值得做死信或失败池？
答：因为失败回调本质就是待处理事件，如果没有统一失败池和可观测面板，这些失败只会散落在日志里，最后没人知道哪些通知根本没送到。

## 六、和主项目怎么绑定

19. `agent-server-java` 在这个主题上最自然的切入点是什么？
答：任务完成回调、产物生成通知、外部系统接收任务状态更新、第三方工作流编排联动、开放给外部调用的任务 API。这些都能自然引出开放平台和回调治理问题。

20. 这个主题在项目里最值得强调哪些点？
答：我会优先强调：回调事件表、签名机制、回调投递任务、失败重试、回调日志、人工重放、事件幂等和异常调用风控。这样项目表达会明显更像平台后端而不是普通业务服务。

## 八、这一篇学到位的标准

学完这篇，你至少要做到下面三件事：

1. 能把开放平台和 `Webhook` 回调讲成认证、签名、重试、幂等、风控、审计的一整套治理问题。
2. 能解释防重放、时间窗口、失败池、人工重放这些高频细节。
3. 能把它映射回主项目或系统设计题，讲出平台型后端的味道。
