# 分布式一致性与事务

## 使用方式

这个专题是后端面试里最容易把人问穿的一块，因为它不再是“会不会写接口”，而是“多个服务、多个库、多个消息链路同时参与时，你如何保证业务结果正确”。

读这篇时，你要始终围绕四个关键词：

- 一致性
- 幂等
- 事务
- 补偿

真正重要的不是背出 `CAP`、`TCC`、`Saga` 这些名词，而是当面试官给你一个订单、库存、支付、积分、MQ 的真实链路时，你能说清边界、失败点和解决策略。

---

## 一、先建立一致性认知

1. 什么是一致性？
答：一致性说的不是“所有地方都一模一样”这么简单，而是系统对外是否满足预期的业务语义。比如用户支付成功后，订单状态、库存、积分、消息通知最终是否都落到了正确状态，这就是一致性问题。对后端来说，一致性首先是业务问题，其次才是技术问题。

2. 单体事务和分布式事务的核心区别是什么？
答：单体事务通常发生在一个数据库连接或同一个事务管理器内部，借助数据库本地事务就能做到原子性、一致性、隔离性和持久性；分布式事务则涉及多个服务、多个数据库、多个消息系统，已经没有一个单点可以天然保证所有资源同时成功或失败，所以你必须额外设计协调、补偿和最终一致方案。

3. 为什么微服务里一致性更难？
答：因为原来一个进程内、一个库内的问题，被拆成了跨网络、跨服务、跨存储的问题。你不再只担心 SQL 回滚，还要担心服务超时、消息重复、消费者失败、补偿漏执行、部分成功部分失败这些分布式环境下才会出现的情况。

4. 强一致和最终一致的区别是什么？
答：强一致强调数据一旦写成功，各副本或各参与方立刻看到统一结果；最终一致则允许短时间内不同节点状态暂时不一致，但系统会在后续通过重试、补偿、消息等机制收敛到正确结果。很多业务并不需要绝对实时强一致，比如积分发放、优惠券通知、搜索索引更新，这时最终一致往往更现实。

5. 为什么很多业务更适合最终一致而不是追求绝对强一致？
答：因为强一致往往意味着更高的耦合、更强的同步依赖和更差的可用性，而业务本身未必真的要求每一步都同步完成。比如下单后异步发通知、异步记日志、异步同步搜索索引，用户并不需要在毫秒级看到这些副作用都完成，只要最终结果正确即可。也就是说，一致性设计要和业务容忍度匹配。

## 二、幂等为什么是核心能力

6. 什么是幂等？
答：幂等的核心含义是同一个请求执行一次和执行多次，系统最终结果应该一致，至少不能因为重复执行就产生额外副作用。这个概念在分布式里极其重要，因为超时重试、消息重投、客户端重复提交、网关重放都可能让同一动作被执行多次。

7. 幂等和防重有什么区别？
答：防重更像是从入口上尽量阻止重复请求，例如按钮防抖、请求 token、去重表；幂等则更偏业务结果层，即使重复请求真的进来了，系统也能保证最终结果仍正确。更成熟的系统通常是两者一起做，而不是只靠某一层。

8. 常见幂等方案有哪些？
答：常见方案包括：业务唯一键去重，例如订单号、支付流水号；幂等表或去重记录表；状态机约束，例如只有 `INIT` 才能流转到 `SUCCESS`；数据库唯一索引；分布式锁；消息消费记录表。真正选哪种，要看请求来源、并发量、是否跨服务、幂等窗口时长以及失败重试方式。

9. 分布式锁能直接等于幂等吗？
答：不能。分布式锁更多是在某个时间窗口内减少并发冲突，避免多个线程同时处理同一个资源；但幂等关心的是“重复执行后最终结果是否正确”。如果一次处理已经成功，后续又收到重复请求，即使没有并发，只要业务没做状态判断，结果仍可能被写乱。所以锁只是手段之一，不是幂等本身。

10. 面试里讲幂等时最容易漏掉什么？
答：最容易漏掉的是“幂等键从哪里来、有效期多长、失败后如何恢复、并发下如何保证只有一个成功”。如果只说“我会加一个幂等 token”，面试官很容易继续追问这个 token 由谁生成、丢了怎么办、重复消息怎么识别、结果如何返回，最终把答案问穿。

## 三、分布式事务常见方案

11. 两阶段提交为什么很少直接作为业务系统主方案？
答：两阶段提交理论上能提供更强的一致性，但它需要统一协调器，参与方在 prepare 阶段还要锁定资源，一旦协调器或参与方异常，资源占用和阻塞问题会比较明显。在高并发互联网业务里，这种方式代价较高，所以很多业务系统更常见的是本地事务加消息、补偿和最终一致设计。

12. 什么是 TCC？
答：`TCC` 可以理解成把一个业务动作拆成 `Try`、`Confirm`、`Cancel` 三段：`Try` 先做资源预留，`Confirm` 真正提交，`Cancel` 做回滚或释放资源。它的优点是业务语义清晰、控制力强；缺点是接入成本高，每个参与服务都要实现三套逻辑，幂等、空回滚、悬挂等问题也需要额外处理。

13. 什么是 Saga？
答：`Saga` 更适合长事务场景。它的核心思路是把一个大事务拆成多个本地事务，每一步成功后继续下一步，某一步失败时就按逆序执行补偿操作。它比两阶段提交更轻量、更适合微服务，但代价是补偿逻辑必须设计清楚，而且不是所有业务都能自然补偿。

14. 什么是本地消息表或 Outbox？
答：它的思路是先在本地数据库事务里同时完成业务数据写入和消息记录写入，保证“业务成功”与“待发消息落库”这件事原子发生。之后再由异步任务把消息投递到 MQ，并根据投递结果更新状态。这个方案在互联网业务里非常常见，因为它把“数据库事务”和“消息投递”之间的裂缝尽量缩小了。

15. 为什么“本地事务 + MQ + 幂等消费”这么常见？
答：因为它在复杂度和可用性之间取得了比较现实的平衡。你不需要强依赖全局协调器，而是通过本地事务保证起点正确，再靠可靠消息、重试、消费幂等和补偿让下游最终收敛到正确状态。这套方案比追求全局强一致更适合大多数互联网业务。

## 四、可靠消息到底怎么设计

16. 什么是可靠消息？
答：可靠消息不是“消息一定 100% 不丢”这样绝对，而是你要能清楚地回答：消息从生产到投递到消费的每一个阶段，可能在哪里丢、在哪里重复、在哪里卡住，以及每一层如何通过确认、重试、补偿和监控把风险降到可控范围。

17. 消息为什么会丢？
答：生产者侧可能发送失败却没感知；Broker 侧可能写盘、复制或确认链路出问题；消费者侧可能处理成功但 offset 或 ack 没提交成功，或者处理失败却没有正确重试。很多人只会说“开启持久化和确认机制”，但成熟回答应该按生产、存储、消费三段分别拆。

18. 消息为什么会重复？
答：因为“至少一次”交付模型本身就意味着在确认边界不够牢靠时，系统宁愿多送几次，也不轻易丢消息。比如消费者业务已经执行成功，但在提交 ack 前挂了，下次重启后又会消费到同一条消息。这就是为什么可靠消息几乎总和幂等消费绑定出现。

19. 消息积压属于一致性问题吗？
答：本质上更偏可用性和时效性问题，但它最终也会影响业务一致性体验。比如订单已经成功，但积分和通知因为积压几十分钟后才到，用户看到的系统状态就是“暂时不一致”。所以消息积压既是吞吐问题，也是业务时序问题。

20. 订单场景里常见的一致性设计怎么讲？
答：一个常见讲法是：订单创建作为主事务，先在本地事务里写订单和消息表，再异步投递库存扣减、积分发放、通知发送等事件。消费方必须做幂等，失败进入重试或死信，后台有补偿任务和监控告警。这样你就把“起点正确、异步传播、失败可恢复、最终可收敛”这一整套逻辑讲清楚了。

## 五、CAP 和工程里的真实取舍

21. 如何理解 CAP？
答：`CAP` 不是面试里拿来炫理论的缩写，而是在分布式系统发生网络分区时，你无法同时无限制地满足一致性和可用性。真实工程里更重要的不是死背定义，而是知道不同业务对一致性和可用性的容忍度不同，所以方案也不同。

22. 业务里怎么体现 CAP 的取舍？
答：比如支付扣款、库存扣减、余额变更这类核心状态，更偏向保证一致性；而搜索索引、消息通知、推荐结果、统计报表这类链路，则更常接受最终一致，优先保证系统可用性和吞吐。成熟的回答从来不是“系统选 CP 或 AP”，而是“不同业务链路做不同级别的一致性设计”。

23. 为什么不能把所有链路都做成强一致？
答：因为代价太高。强一致会带来更长链路、更强同步依赖、更高延迟和更低可用性，而且很多链路业务上并不需要这么重。真正成熟的系统会把最核心的状态收在强约束范围里，把可延迟的副作用做成最终一致，从而在正确性和性能之间取得平衡。

## 六、面试里最该怎么答

24. 如果面试官问“分布式事务怎么做”，你应该怎么起手？
答：我会先反问或主动澄清业务场景，因为不同场景选型差别非常大。比如是订单、支付、库存这种核心交易链路，还是积分、通知、搜索索引这种副作用链路；是必须同步完成，还是允许最终一致。只有先确定业务容忍度，后面谈 `TCC`、`Saga`、消息表、补偿任务才有意义。

25. 如果问“如何保证订单、库存、积分一致”，一个完整回答框架是什么？
答：我一般会这样讲：先区分主链路和副作用链路；订单创建与核心扣减使用本地事务或更强的一致性方案保证起点正确；后续通过可靠消息把事件投递出去；消费方做幂等；失败后有重试、死信和人工补偿；最后配合监控和对账任务做兜底。这样既有主线，也有失败恢复机制。

26. 如果问“消息消费重复怎么办”，核心答案是什么？
答：核心不是“我会去重”这一句，而是要明确说明：用什么作为幂等键、幂等结果存在哪里、重复消息来了如何快速判断、并发时如何避免两个消费者同时成功、失败时如何重试。只要这几个点答完整，答案就会更像工程方案。

27. 什么样的回答最像背八股？
答：上来就罗列 `2PC、TCC、Saga、Outbox`，却不讲使用场景、不讲 trade-off、不讲失败模式。因为分布式事务最怕的就是只会报名词。真正好的回答应该始终围绕业务链路、失败点、恢复机制和成本来展开。

## 八、这一篇学到位的标准

学完这篇，你至少要做到下面三件事：

1. 能结合订单类场景讲清分布式事务与最终一致方案，而不是只报方案名词。
2. 能把幂等、可靠消息、补偿任务、死信、对账串成一套完整链路。
3. 能根据业务重要性解释为什么某些链路要更强一致，某些链路接受最终一致。

## 九、订单、库存、支付的完整案例

28. 如果让你设计“下单 + 扣库存 + 支付 + 发积分”，第一步怎么拆？
答：我会先把链路分成两类：核心强约束链路和可延后副作用链路。订单创建、库存扣减、支付结果确认通常更靠近核心链路；积分发放、通知、搜索索引更新、报表统计更像副作用链路。只有先把这两层拆开，后面一致性方案才不会乱。

29. 订单系统里哪些部分更适合强一致？
答：通常订单主状态、库存主状态、支付结果这种直接决定用户资产和履约结果的链路，更倾向强约束设计。因为这些地方一旦错，后果比较严重，比如超卖、重复扣款、订单状态错乱。也就是说，强一致要优先留给业务上最不能出错的地方。

30. 哪些部分更适合最终一致？
答：积分、通知、搜索索引、推荐、统计报表、埋点这类链路更适合最终一致。因为用户通常不要求这些能力和订单主状态在同一毫秒完全同步，只要后续能正确收敛即可。

31. 一个成熟的订单一致性回答框架是什么？
答：我会这样讲：下单请求进入后先做参数校验、幂等和库存预检查；订单主链路使用本地事务或更强约束机制保证订单与库存状态不出现半成功半失败；支付回调通过幂等处理驱动订单状态推进；后续积分、通知等通过可靠消息异步扩散；消费端做幂等，失败后进入重试、死信和补偿流程；最后通过对账和监控兜住长尾异常。

32. 为什么订单系统最后一定会讲到对账？
答：因为任何分布式方案都不可能保证零长尾。再好的事务和消息方案，也可能因为极端故障、人工修复、脏数据或边界条件导致小概率不一致。对账的价值就是把这些长尾漏网之鱼再捞出来，避免问题长期潜伏。

## 十、TCC、Saga、Outbox 到底怎么选

33. `TCC` 最适合什么场景？
答：更适合核心链路、每一步资源语义清晰、参与方都能配合实现 `Try/Confirm/Cancel` 的场景，例如余额预扣、库存预占这类高价值交易型业务。它的优点是可控度高，缺点是侵入性强、实现成本高，而且空回滚、悬挂、幂等这些问题都得认真设计。

34. `Saga` 最适合什么场景？
答：更适合长事务、多步骤业务编排、每一步都能定义补偿动作，但又不值得引入过重全局协调的场景。比如工单流程、审批流、复杂任务编排、部分订单履约链路。它的本质是“允许阶段性成功，但失败时要按业务语义反向补偿”。

35. `Outbox` / 本地消息表最适合什么场景？
答：非常适合“先把核心业务写稳，再把事件可靠扩散出去”的场景，例如订单创建后同步库存、积分、通知。它比 `TCC` 轻，比纯异步裸发消息更稳，所以在互联网业务里非常常见。可以把它理解成最终一致方案里的主力选手之一。

36. 如果面试官逼你做取舍，怎么答？
答：我会先从业务容忍度讲。如果是资金、库存这类极高价值强约束链路，优先考虑更强的一致性设计；如果是多步骤业务编排且每一步都能补偿，`Saga` 更现实；如果是“本地业务成功后通知多个下游收敛”，本地消息表或 `Outbox` 往往性价比最高。也就是说，选型不是看方案名气，而是看业务约束和团队成本。

37. 为什么很多真实系统里并不会只用一种方案？
答：因为不同链路风险不同。一个系统里完全可能同时存在：订单主链路用本地事务 + 唯一约束，库存预占用更强约束，积分和通知走 `Outbox + MQ`，某些长编排流程用 `Saga`。成熟系统不是统一信仰一种方案，而是按业务层级组合使用。

## 十一、三个最容易踩坑的反模式

38. 反模式一：先写业务，再裸发消息，为什么危险？
答：因为如果业务提交成功后进程挂了、网络断了、发送失败了，消息就可能永远没出去。这时你表面上看核心业务已经成功，但下游永远收不到事件，系统状态会长期分裂。这就是为什么本地消息表类方案这么常见。

39. 反模式二：把分布式锁当成一致性总方案，为什么不对？
答：因为锁只是在某个时刻减少并发冲突，不能自动解决业务语义、失败补偿、消息重复和状态机推进问题。你即使拿到了锁，如果后续处理一半失败，系统仍可能是不一致的。所以锁只能解决部分并发问题，不能替代整套一致性设计。

40. 反模式三：无限重试为什么会变成事故放大器？
答：因为很多错误并不是“多试几次就会好”，比如脏数据、逻辑 bug、数据库唯一约束冲突、权限错误。无限重试只会把无效请求重复打向下游，造成重试风暴。成熟系统一定会区分可重试与不可重试错误，并有死信、人工处理和熔断边界。

## 十二、幂等和补偿怎么答得更落地

41. 幂等键一般从哪里来？
答：通常优先选业务天然唯一标识，例如订单号、支付流水号、请求号、任务 id、外部回调流水。如果没有天然业务键，再考虑在入口生成请求幂等键。关键不是“随便搞个 UUID”，而是要保证这个键在业务语义上真的能唯一代表这次动作。

42. 幂等记录表一般至少存什么？
答：通常至少包括幂等键、业务类型、处理状态、结果摘要、创建时间、过期时间、最近更新时间，有时还会记录错误原因和重试次数。这样设计的好处是，不仅能判断是否重复，还能回答“上次做到哪一步、是否成功、失败后能否继续”。

43. 补偿任务和对账任务有什么区别？
答：补偿任务更像针对已知失败链路主动修复，例如重发消息、重新回调、重算积分；对账任务则更像周期性扫描系统间状态差异，找出那些没有被及时发现的不一致。两者都重要，但定位不同：一个偏主动恢复，一个偏兜底发现。

44. 为什么补偿也必须幂等？
答：因为补偿本身也可能失败、重试、重复执行。如果补偿逻辑不幂等，你本来是在修不一致，结果又制造了新的重复副作用。所以成熟系统会把补偿也当成正式业务动作设计，而不是当临时脚本。

## 十四、Raft 共识协议基础

45. 后端为什么要懂 Raft？
答：因为你日常依赖的大量基础设施底层都是 Raft 在做共识：etcd 是 Kubernetes 所有集群元数据的存储，Consul 做服务发现和配置中心，TiKV 是 TiDB 的分布式存储引擎，RocketMQ 的 DLedger 模式用 Raft 保证消息高可用，RabbitMQ 的 Quorum Queue 也基于 Raft。你不需要自己实现 Raft，但你要理解它的选举和日志复制机制，才能在集群脑裂、Leader 切换、写入延迟抖动等线上问题出现时快速定位原因。

46. Raft 有哪三种角色？各自职责是什么？
答：Leader、Follower、Candidate。Leader 负责接收所有客户端写请求、将日志复制给 Follower、定期发送心跳维持权威；Follower 被动接收 Leader 的日志和心跳，不主动发起请求，如果超过选举超时时间(election timeout)没有收到心跳就转为 Candidate；Candidate 是 Follower 触发选举后的中间状态，它向其他节点发起投票请求，获得多数票则晋升为 Leader，否则退回 Follower 或重新选举。

47. Leader 选举的完整流程是什么？
答：当一个 Follower 在 election timeout（通常随机化在 150~300ms 之间）内没有收到 Leader 心跳时，它将自身状态切换为 Candidate，将当前 term 加 1，先给自己投一票，然后并行发送 `RequestVote RPC` 给集群中所有其他节点。每个节点在同一个 term 内只能投一票（先到先得）。如果 Candidate 收到多数节点（N/2+1）的投票，它成为新 Leader，立刻向所有节点发送心跳（空的 `AppendEntries RPC`）宣布权威并重置其他节点的选举计时器。如果在选举超时内没有获得多数票（比如出现了 split vote，两个 Candidate 同时发起且各得一部分票），则本轮选举失败，所有 Candidate 重新随机化 election timeout 后再次尝试。随机化超时是 Raft 解决 split vote 的关键手段——让多个 Candidate 几乎不可能在同一时刻同时发起选举。

48. 日志复制的完整流程是什么？
答：客户端写请求只发给 Leader。Leader 收到请求后先将该操作追加到自己的本地日志（此时状态为 uncommitted），然后并行发送 `AppendEntries RPC` 给所有 Follower，RPC 中携带日志条目以及 Leader 的 `prevLogIndex`、`prevLogTerm` 用于一致性校验。Follower 收到后检查 `prevLogIndex` 和 `prevLogTerm` 是否匹配本地日志，匹配则追加新条目并返回成功，不匹配则返回失败（Leader 会递减 `nextIndex` 重试，直到找到一致的位置后覆盖 Follower 上的冲突日志）。当 Leader 收到多数节点（包含自己）的成功响应后，将该日志标记为 committed，应用到状态机并返回客户端成功。后续心跳会携带 Leader 的 `commitIndex`，Follower 据此推进自己的提交位置。关键点：Follower 每次收到心跳或 `AppendEntries` 都会重置选举计时器，这保证了只要 Leader 正常工作，不会触发不必要的选举。

49. Raft 的安全性是怎么保证的？
答：两个核心保证。第一，Leader 完整性（Leader Completeness）：一个节点要当选 Leader，它的日志必须至少和集群多数节点一样新。`RequestVote` 中会携带 Candidate 的 `lastLogTerm` 和 `lastLogIndex`，接收方会比较——先比 `lastLogTerm`，term 更大的更新；term 相同再比 `lastLogIndex`，index 更大的更新。如果 Candidate 的日志不够新，节点会拒绝投票。这保证了已提交的日志一定存在于新 Leader 上。第二，日志匹配性（Log Matching）：如果两个节点的日志在某个位置有相同的 term 和 index，则从头到该位置的所有日志条目完全一致。这由 `AppendEntries` 的一致性检查递归保证。

50. Raft 和 Paxos 有什么区别？为什么工程上更多选 Raft？
答：Paxos 是分布式共识的理论基石，Lamport 提出的经典算法，理论最优但出了名地难理解和难实现——它的 Multi-Paxos 工程化需要大量未在论文中明确的细节决策。Raft 由 Diego Ongaro 在 2014 年论文中提出，核心目标就是"可理解性"：它把共识问题拆成了 Leader 选举、日志复制、安全性三个相对独立的子问题，每个子问题的规则都很明确。结果是 etcd、Consul、TiKV、CockroachDB 等主流系统几乎都选了 Raft，因为工程师能读懂、能调试、能正确实现。性能上两者在稳态下差异不大（都需要多数确认），但 Raft 的 Leader-based 模型让日志顺序天然确定，实现更简单。

## 十五、TCC 的边界问题与实现细节

51. TCC 三个阶段各自的职责边界是什么？
答：Try 阶段做资源预留，把资源从"可用"变成"冻结中间态"，例如账户余额从 available 中扣出一笔到 frozen，库存从 sellable 中扣出一笔到 withheld。注意 Try 不是最终扣减，而是"占住"。Confirm 阶段做真正提交，把冻结的中间态推进到最终态，例如从 frozen 中扣除（available 已经在 Try 阶段扣了，Confirm 只是清掉 frozen 标记）。Cancel 阶段做释放，把 Try 预留的资源还回去，例如把 frozen 加回 available。Confirm 和 Cancel 都必须是幂等的，因为 TCC 框架在超时或异常时会重试调用它们。

52. 什么是空回滚(Empty Rollback)？怎么解决？
答：空回滚是指 Cancel 被调用时，对应的 Try 其实根本没有执行过。典型场景：参与方 A 的 Try 请求因为网络超时迟迟没到达，TCC 协调器判定事务失败，先给 A 发了 Cancel。此时 A 收到 Cancel 但本地并没有任何 Try 产生的中间态数据。如果 Cancel 不做任何判断直接执行"释放"逻辑，可能操作不存在的冻结记录导致数据错乱，或者返回失败导致协调器反复重试。解决方案：Cancel 执行前先查询 TCC 事务记录表中是否存在对应 `xid + branchId` 且 `status = tried` 的记录。如果没有（说明 Try 没执行过），直接插入一条 `status = cancelled` 的记录并返回成功，这就是空回滚——什么都不做但标记已取消。

53. 什么是悬挂(Suspension)？怎么解决？
答：悬挂是空回滚的延伸问题。场景：Cancel 先到并执行了空回滚，随后因为网络延迟，Try 请求姗姗来迟。如果 Try 不做任何检查直接执行资源预留，这笔冻结的资源就会永远卡在中间态——因为协调器已经认为事务结束了，不会再发 Confirm 或 Cancel。解决方案：Try 执行前先查询 TCC 事务记录表，如果发现已有 `status = cancelled` 的记录（说明 Cancel 已经先执行了空回滚），则直接拒绝 Try 并返回失败。这就是为什么空回滚时必须写入记录的原因——它同时也是悬挂防护的前提。

54. TCC 的幂等怎么保证？
答：Confirm 和 Cancel 都可能被 TCC 框架重复调用（协调器超时重试、网络重传等）。幂等保证的核心是通过 `xid（全局事务 ID）+ branchId（分支事务 ID）+ status` 三元组来判断。每次 Confirm 执行前检查 status 是否已经是 `confirmed`，如果是则直接返回成功；Cancel 执行前检查 status 是否已经是 `cancelled`，如果是则直接返回成功。状态流转必须是单向的：`init → tried → confirmed` 或 `init → tried → cancelled`，不允许回退或跳转。

55. TCC 事务记录表怎么设计？
答：核心字段如下：

```sql
CREATE TABLE tcc_transaction (
    xid         VARCHAR(128) NOT NULL,   -- 全局事务ID，由TCC协调器生成
    branch_id   VARCHAR(128) NOT NULL,   -- 分支事务ID，标识当前参与方
    status      TINYINT NOT NULL DEFAULT 0, -- 0:init 1:tried 2:confirmed 3:cancelled
    retry_count INT NOT NULL DEFAULT 0,  -- 当前阶段已重试次数
    create_time DATETIME NOT NULL,
    update_time DATETIME NOT NULL,
    PRIMARY KEY (xid, branch_id)
);
```

所有 Try/Confirm/Cancel 操作都先查这张表再决定是否执行。空回滚时 Try 没执行过但 Cancel 要插入一条 `status=3(cancelled)` 的记录；悬挂防护时 Try 先查到有 `status=3` 的记录则拒绝；幂等检查时 Confirm/Cancel 发现已经是目标状态则直接返回。这张表和业务表在同一个库中，通过本地事务保证操作的原子性。

56. TCC 的业务改造成本到底有多大？
答：每个参与方原来只需要一个业务接口，引入 TCC 后变成三个（Try + Confirm + Cancel），再加上空回滚、悬挂防护、幂等判断、事务记录表维护，代码量通常是原来的 3 倍以上。而且业务模型要重新设计——必须把"直接扣减"改成"先冻结再确认"的二阶段模型，数据库表也要增加冻结字段。这就是为什么 TCC 通常只用在高价值核心链路（如资金、库存），而不是所有链路都套 TCC。

## 十六、Outbox Pattern 的具体实现

57. Outbox 的核心思路是什么？
答：核心是"同库同事务"。业务操作和消息记录在同一个数据库中、同一个本地事务中完成写入。事务提交后，业务数据落盘的同时消息记录也一定落盘了，不会出现"业务成功但消息丢了"或"消息发了但业务没提交"的裂缝。消息的实际投递由独立的异步组件负责，投递失败可以重试，投递成功后更新消息状态。

58. Outbox 消息表怎么设计？
答：一个经过实践验证的设计如下：

```sql
CREATE TABLE outbox_message (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    aggregate_type  VARCHAR(255) NOT NULL,  -- 聚合根类型，如 "Order"、"Account"
    aggregate_id    VARCHAR(255) NOT NULL,  -- 聚合根ID，如订单ID、账户ID
    event_type      VARCHAR(255) NOT NULL,  -- 事件类型，如 "OrderCreated"、"PaymentCompleted"
    payload         TEXT NOT NULL,           -- JSON格式的事件内容
    status          TINYINT NOT NULL DEFAULT 0, -- 0:待发送 1:已发送 2:发送失败
    retry_count     INT NOT NULL DEFAULT 0,
    create_time     DATETIME NOT NULL,
    update_time     DATETIME NOT NULL,
    INDEX idx_status_create (status, create_time)
);
```

`aggregate_type + aggregate_id` 标识这条消息属于哪个业务实体，方便按实体维度查询和排查问题。`event_type` 让消费者可以按事件类型路由处理。`idx_status_create` 索引用于轮询投递时快速查询待发送的最早一批消息。

59. 两种投递方式各有什么优劣？
答：第一种是轮询投递(Polling Publisher)：一个定时任务（如每秒执行一次）查询 `SELECT * FROM outbox_message WHERE status = 0 ORDER BY create_time LIMIT 100`，逐条或批量发送到 MQ，成功后 `UPDATE status = 1`，失败则 `retry_count + 1`，超过最大重试次数标记 `status = 2` 进入人工处理。优点是实现极简，不依赖额外组件；缺点是延迟取决于轮询间隔（1 秒轮询就有最多 1 秒延迟），高频轮询对数据库有查询压力，低频轮询延迟又太高。第二种是 CDC 投递(Change Data Capture)：通过 Debezium 等工具订阅数据库的 binlog，捕获 outbox 表的 INSERT 事件，实时投递到 Kafka 等 MQ。优点是近实时（毫秒级延迟）、不对数据库做额外查询；缺点是需要部署和维护 Debezium + Kafka Connect 这一套 CDC 管道，架构复杂度上升。中小业务选轮询足够，大规模或对延迟敏感的业务建议上 CDC。

60. 已发送的消息怎么处理？
答：outbox 表如果不清理会无限增长，影响查询性能和磁盘空间。常见做法：对 `status = 1`（已发送）的消息，超过一定天数（如 7 天）后归档到 `outbox_message_archive` 表或直接删除。可以通过定时任务执行 `DELETE FROM outbox_message WHERE status = 1 AND create_time < DATE_SUB(NOW(), INTERVAL 7 DAY) LIMIT 10000`，分批删除避免锁表。对 `status = 2`（发送失败）的消息保留更久或人工介入。

61. 有没有现成的 Outbox 框架？
答：有。Eventuate Tram 是 Java 生态中较成熟的 Transactional Outbox 框架，内置了消息表管理、轮询投递、消费幂等等能力。Debezium Outbox Router 是 Debezium 的一个 SMT（Single Message Transform），专门识别 outbox 表的 INSERT 事件并转换成标准事件格式投递到 Kafka topic。Spring 生态中也可以基于 `@TransactionalEventListener` + 自定义消息表实现类似效果。选型时主要看团队技术栈和是否已经在用 Debezium/Kafka Connect。

## 十七、Exactly-Once 到底能不能做到

62. 消息系统的三种投递语义分别是什么？
答：At-Most-Once：消息最多被处理一次，可能丢失。实现方式是消费者拿到消息后立刻自动提交 offset（或自动 ack），不做重试。如果处理过程中挂了，这条消息就丢了。At-Least-Once：消息至少被处理一次，可能重复。实现方式是消费者处理完业务逻辑后才手动提交 offset（或手动 ack），如果处理成功但提交 offset 前挂了，重启后会再次消费同一条消息。Exactly-Once：消息恰好被处理一次，既不丢也不重复。这是理想状态。

63. Kafka 的 Exactly-Once 是怎么做到的？
答：Kafka 从 0.11 版本开始引入了三层机制。第一层，幂等 Producer：每个 Producer 实例启动时会被 Broker 分配一个 `PID（Producer ID）`，每条消息在每个 Partition 上有单调递增的 `SequenceNumber`。Broker 端对同一个 `<PID, Partition>` 维护已接收的最大 SequenceNumber，重复消息（SequenceNumber 已见过）会被静默丢弃。配置：`enable.idempotence=true`。第二层，事务 Producer：跨多个 Partition 和 Topic 的原子写入。流程是 `producer.initTransactions()` → `producer.beginTransaction()` → 多次 `producer.send()` → `producer.sendOffsetsToTransaction(offsets, consumerGroupId)`（把消费 offset 也纳入事务）→ `producer.commitTransaction()` 或 `producer.abortTransaction()`。Broker 通过 Transaction Coordinator 和 `__transaction_state` 内部 topic 维护事务状态。第三层，Consumer 端 Read Committed：设置 `isolation.level=read_committed`，消费者只能读到已经 commit 的事务消息，abort 的事务消息会被跳过。但要注意：这三层加在一起只是 Kafka 内部的 Exactly-Once——即从 Kafka 读出、处理、再写回 Kafka 这个闭环内不丢不重。

64. 端到端的 Exactly-Once 为什么更难？
答：因为消费者处理完消息后通常要写外部系统（数据库、缓存、搜索引擎等），而 Kafka 的事务机制无法覆盖外部系统的写入。典型问题：消费者从 Kafka 读消息 → 写数据库成功 → 提交 offset 失败 → 重启后重复消费 → 数据库被写了两次。要解决这个问题，必须在消费端自己做幂等：比如用消息的 `<topic, partition, offset>` 或业务唯一键作为幂等标识，写数据库时先查是否已处理过，或通过数据库唯一约束防止重复插入。

65. RabbitMQ 能做 Exactly-Once 吗？
答：RabbitMQ 没有内建的 Exactly-Once 语义。它的 Publisher Confirm 机制保证消息不丢（At-Least-Once 投递），Consumer 的手动 ack 保证消息不丢（At-Least-Once 消费），但没有类似 Kafka 的幂等 Producer 和事务机制。所以 RabbitMQ 场景下要实现等效 Exactly-Once，完全依赖消费端幂等。

66. 工程现实中 Exactly-Once 怎么落地？
答：绝大多数生产系统的做法是 **At-Least-Once + 消费端幂等** 来等效实现端到端 Exactly-Once。具体来说：生产端确保消息不丢（确认机制 + 重试）；Broker 端确保消息持久化和高可用（副本机制）；消费端确保处理幂等（唯一键 + 去重表 / 数据库唯一约束 / 状态机判断）。只要这三层都做到，最终效果就是每条消息的业务副作用恰好生效一次。面试时说"真正的端到端 Exactly-Once 是 At-Least-Once 加消费端幂等"比说"Kafka 支持 Exactly-Once"要成熟得多。

## 十八、分布式事务框架实战

67. Seata 的整体架构是什么？
答：Seata 由三个核心角色组成。TC（Transaction Coordinator，事务协调器）：独立部署的服务端，负责维护全局事务和分支事务的状态，驱动全局提交或回滚。TM（Transaction Manager，事务管理器）：嵌入在应用中的客户端组件，负责定义全局事务的边界——开启全局事务、发起全局提交或回滚请求给 TC。RM（Resource Manager，资源管理器）：也嵌入在应用中，负责管理分支事务的资源，向 TC 注册分支事务、汇报分支状态、执行分支提交或回滚。一个典型流程：TM 向 TC 申请开启全局事务拿到 `xid` → 业务调用链路中各服务的 RM 向 TC 注册分支事务 → 各分支执行本地事务 → TM 根据业务结果通知 TC 全局提交或回滚 → TC 驱动所有 RM 执行分支提交或回滚。

68. Seata 支持的四种模式分别是什么？
答：AT（Auto Transaction）模式：基于 `undo_log` 的自动补偿，对业务代码零侵入，只需要加 `@GlobalTransactional` 注解，但要求参与方都用关系型数据库且支持本地 ACID。TCC 模式：业务侵入式，每个参与方需要实现 Try/Confirm/Cancel 三个接口，灵活性最高，适用于非关系型存储或复杂业务逻辑。Saga 模式：基于状态机引擎编排长事务，每个步骤定义正向操作和补偿操作，适合流程长、参与方多、不适合资源锁定的场景。XA 模式：标准两阶段提交协议的实现，依赖数据库自身的 XA 支持（`XA START`、`XA END`、`XA PREPARE`、`XA COMMIT`），强一致但性能开销大、资源锁定时间长。

69. AT 模式的原理是什么？
答：AT 模式分两个阶段。一阶段：RM 拦截业务 SQL，在执行前查询得到 before image（修改前数据快照），执行 SQL 后查询得到 after image（修改后数据快照），将 before image 和 after image 一起写入本地 `undo_log` 表，然后直接提交本地事务（注意：一阶段就已经提交了，不像 XA 要等二阶段才提交，所以性能好很多）。二阶段提交：如果全局事务成功，TC 通知各 RM 异步删除对应的 `undo_log` 记录即可，因为数据已经是对的。二阶段回滚：如果全局事务失败，TC 通知各 RM 取出 `undo_log` 中的 before image，生成反向 SQL 还原数据，执行后删除 `undo_log`。在还原前还会用 after image 做校验——如果当前数据和 after image 不一致，说明数据被其他事务修改过（脏写），此时需要人工介入。

70. AT 模式的全局锁是怎么回事？
答：AT 模式一阶段直接提交了本地事务，数据库行锁已经释放，但全局事务可能还没有结束。如果此时另一个全局事务尝试修改同一行数据，就会产生脏写风险。Seata 通过 TC 维护的全局锁解决这个问题：RM 在一阶段提交前，要先向 TC 申请这一行数据的全局锁（锁的粒度是 `resourceId + tableName + rowPK`）。如果全局锁已经被另一个 `xid` 持有，当前事务会等待或重试。全局锁在全局事务提交或回滚后释放。这意味着 AT 模式的数据隔离级别介于读未提交和读已提交之间——本地事务已提交但全局事务未提交的数据，其他全局事务会被全局锁阻塞，但本地的普通 SQL（不经过 Seata 代理的）仍能读到这些未全局提交的数据。

71. 什么场景该选哪种 Seata 模式？
答：如果参与方全是关系型数据库、业务改造成本要低、并发不极端高，选 AT 模式——零侵入是最大优势。如果涉及非关系型存储（Redis、ES、外部接口）、或需要精细控制每一步的资源预留和释放，选 TCC 模式——灵活但改造成本高。如果是长链路多步骤编排、每步耗时较长、不适合全程锁资源，选 Saga 模式——适合工作流类场景。如果对一致性要求极高且愿意承受性能代价、数据库支持 XA，选 XA 模式——最接近传统强一致但最慢。大多数互联网业务优先考虑 AT，核心交易链路考虑 TCC，审批流等长事务考虑 Saga，XA 在互联网场景中较少使用。

## 十二、Paxos 算法核心思想

1. Paxos和Raft的关系是什么？
答：Paxos是Lamport提出的分布式共识算法的鼻祖，理论上最优但极难理解和实现。Raft是Diego Ongaro为了降低Paxos的理解难度而设计的等价算法——两者解决同一个问题(在分布式节点间就某个值达成一致)，但Raft通过Leader选举+日志复制的分解方式让理解和实现变得更容易。工程实践中Raft(etcd/Consul/TiKV)远多于Paxos(ZooKeeper的ZAB是Paxos变种)。

2. Basic Paxos的两阶段是什么？
答：
- Prepare阶段：Proposer选择提案编号n，向多数Acceptor发送Prepare(n)。Acceptor如果n大于已见过的最大编号，承诺不再接受编号小于n的提案，并返回已接受的最高编号提案(如果有)
- Accept阶段：Proposer收到多数承诺后，选择值(如果Prepare阶段有返回已接受的值则用那个值，否则用自己的)，发送Accept(n, value)。Acceptor如果n≥已承诺的最大编号，接受该提案
- 关键保证：任何被多数接受的值就是最终共识值，不会被改变

3. 为什么Paxos在工程中难以直接使用？
答：Basic Paxos只能就单个值达成共识。实际系统需要连续决定多个值(日志序列)，需要Multi-Paxos。但Multi-Paxos论文没有给出完整的工程细节(Leader选举、日志压缩、成员变更、快照等)，各实现差异很大。这就是为什么Raft的出现填补了工程空白——它把连续共识问题分解成Leader选举、日志复制、安全性三个子问题，并给出了完整的工程实现细节。
