# AI 应用与大模型后端专题

## 使用方式

这篇不是让你转成算法工程师，而是补齐 2025~2026 年后端岗位越来越高频的一块认知：

- AI 产品和普通后端系统相比，后端难点多了什么
- 为什么流式输出、工具调用、RAG、Agent、评测、成本控制都变成了后端问题
- 如果你的主项目带 AI 能力，怎样讲得既有新意，又不显得“只是在调 API”

这篇对你尤其重要，因为 `agent-server-java` 天然就带有 AI 长任务、流式链路和平台型服务特征。

---

## 一、为什么 AI 后端已经成为高频方向

1. 为什么大厂后端开始越来越常问 AI 系统？
答：因为 AI 功能不再只是独立 demo，而是越来越多地进入搜索、客服、内容处理、办公协同、平台工具和企业后台。面试官现在不仅想看你会不会做传统后端，还会看你能不能把 AI 能力做成稳定、可观测、可治理的服务系统。

2. AI 后端和普通后端最大的差别是什么？
答：普通后端更多是在管理结构化业务规则和确定性链路；AI 后端还要面对结果不稳定、输出不可完全预测、延迟长、成本高、上下文管理复杂、工具调用链长、评测和灰度困难这些问题。也就是说，它不是“再调一个接口”，而是多了一整套新的工程复杂度。

---

## 二、AI 后端的五个核心难点

3. AI 后端第一个核心难点是什么？
答：长链路和高延迟。普通接口几十毫秒到几百毫秒比较常见，但 AI 请求可能要几秒甚至更久，还可能包含检索、工具调用、推理和结果后处理。这会直接影响超时设计、流式返回、任务拆分和用户体验。

4. 第二个核心难点是什么？
答：结果不确定性。传统后端更强调同样输入得到同样输出，而 AI 系统即使输入接近，结果也可能不同。后端必须思考如何做结果校验、异常兜底、重试边界和人工审核。

5. 第三个核心难点是什么？
答：上下文与状态管理。会话历史、工具上下文、用户输入、外部知识、记忆片段和模型配置都会影响最终输出。上下文一旦管理不好，就容易出现成本爆炸、结果漂移、会话串线和调试困难。

6. 第四个核心难点是什么？
答：成本控制。AI 调用不只是 CPU 和内存成本，还直接对应 token、模型调用费用、向量检索成本和延迟成本。大厂 AI 后端很关注“能不能稳定跑”，也非常关注“能不能合理花钱”。

7. 第五个核心难点是什么？
答：可观测性。很多 AI 故障不是 HTTP 500，而是模型变慢、回答质量下降、工具调用异常、解析失败、上下文拼错、RAG 命中变差。这些都要求后端把 AI 链路做成可追踪、可分析、可评估的系统。

## 三、流式输出、RAG、Agent 怎么讲

8. 为什么 AI 场景里经常要做流式输出？
答：因为模型推理耗时长，如果一直等完整结果再返回，用户体验会很差。流式输出可以更早把首字返回给前端，提升交互感知，同时也更适合展示中间过程和工具调用状态。

9. 流式输出为什么不是纯前端问题？
答：因为后端要负责模型流式转发、超时控制、断线恢复、SSE/WebSocket 选型、中间状态语义和多实例下的任务关联。也就是说，流式 UI 只是表象，底层是后端链路治理问题。

10. 什么是 RAG，后端需要管什么？
答：RAG 本质是“检索 + 生成”的组合。后端要管的通常包括：知识入库、切片、索引、召回、重排、上下文拼接、缓存、延迟、评测和结果回放。它不是简单“接个向量库”，而是一条带数据治理和检索质量的后端链路。

11. Agent 后端和普通聊天后端有什么不同？
答：普通聊天更像单轮或多轮会话管理；Agent 后端还会涉及工具调用、任务编排、执行预算、阶段状态、失败恢复和运行日志。它更像一个“可编排、可执行、可追踪”的平台系统，而不是单一聊天接口。

---

## 四、AI 后端的稳定性和治理

12. 为什么 AI 后端更需要可观测性？
答：因为 AI 故障常常不是“服务挂了”，而是“效果悄悄变差了”“延迟越来越长”“工具调用偶发失败”“RAG 召回质量下降”。如果没有结构化运行日志、调用链、评测指标和版本信息，你很难知道问题到底出在模型、Prompt、工具、检索还是业务逻辑。

13. AI 后端最该监控哪些指标？
答：至少包括：首字时间、总响应时长、成功率、取消率、重试率、解析失败率、工具调用成功率、上下文长度、token 消耗、单请求成本、RAG 命中率、输出质量评估分。只有这样你才能同时看到“机器正常”和“效果正常”。

14. 为什么 AI 后端也要灰度发布？
答：因为 AI 变更风险不只是服务是否可用，还包括效果是否漂移、回答是否变差、工具调用是否更容易出错。直接全量切模型、切 Prompt、切检索链路风险很高，所以更需要小流量验证、分桶、回滚和效果对比。

15. 为什么 AI 系统特别适合讲“预算”？
答：因为每次调用都可能直接带来成本，而且成本和上下文长度、模型类型、检索轮次、工具调用次数强相关。成熟 AI 后端会关注预算上限、超额保护、模型降级、上下文裁剪和缓存复用，而不是只追求效果。

## 五、和你的主项目怎么绑定

16. `agent-server-java` 为什么天然适合当 AI 后端主项目？
答：因为它本身就有资源采集、长任务执行、流式进度、模型调用、结果产物、任务治理这些典型 AI 后端特征。只要把任务状态、幂等、缓存、MQ、可观测性、权限和部署补齐，它就不只是“一个 Agent demo”，而是一个很像平台型 AI 后端系统的主项目。

17. 如果把它讲成 AI 后端平台，最值得强调哪些能力？
答：我会优先强调：任务状态机、流式进度推送、模型调用隔离、工具调用链路、失败重试、结果产物管理、调用日志和评测/观测能力。这些都比“我接了某个大模型 API”更像后端深度。

## 八、SSE 流式输出后端实现

1. SSE（Server-Sent Events）协议本质上是什么？
答：本质是在 **HTTP 长连接** 上由 **服务端向客户端单向推送** 文本事件。浏览器端常用 `EventSource` 消费。响应需设置 `Content-Type: text/event-stream`（常配合 `Cache-Control: no-cache`、`Connection: keep-alive` 等）。每条事件通常按行发送，常见约定是 **`data: {json}\n\n`** 表示一帧业务数据；流结束时发送 **`data: [DONE]\n\n`**（或与产品约定一致的结束标记），便于前端关闭读取循环。

2. Spring Boot 里怎么实现 SSE？
答：在 **Servlet 栈** 中 Controller 可返回 **`SseEmitter`**，由 Spring 写出 `text/event-stream` 并支持异步完成/异常结束；在 **WebFlux** 中可返回 **`Flux<ServerSentEvent<T>>`**（或等价流式响应），由响应式栈写出 SSE。实现上要为 **`SseEmitter` 设置合理超时**（如 `new SseEmitter(timeoutMs)`），避免连接无限挂死占满线程或句柄；发生不可恢复错误时应调用 **`emitter.completeWithError(e)`**（或 `complete()` 在可接受场景下正常收尾），并配合日志与监控，避免客户端一直等待无数据。

3. 模型流式输出时，后端整条转发链路是怎样的？
答：**客户端 → 后端业务 API → 上游模型 API（如 `stream=true`）→ 后端逐块读取 → 逐块推送给客户端**。后端职责包括：用 HTTP 客户端或 SDK 读取模型返回的 **`InputStream` / 响应体块**，或 Reactor 的 **`Flux`**；若上游本身也是 SSE，需 **按行解析 `data:` 帧**（及可选 `id:`、`event:`）；将解析出的增量内容 **映射为发给前端的 SSE 帧**（再次 `data: ...\n\n`），必要时做 **鉴权、限流、审计字段脱敏**。核心是：**不要等全量响应再返回**，而是边读边写、背压可控。

4. SSE 断线重连时前后端分别要注意什么？
答：**客户端** 侧 `EventSource` 在连接断开会 **自动重连**；规范支持 **`Last-Event-ID`** 头，把最后收到的事件 ID 带给服务端。**服务端** 若要做到“续传”，需要 **按事件 ID 记录已发送偏移**（内存/Redis/DB），重连后从该点继续推送；若无法续传，可约定 **返回完整最终结果**（或任务 ID + 轮询/另开接口取结果），并明确 **重复内容的幂等与 UI 去重**（避免用户看到两段拼接错乱）。生产上还要区分 **用户主动取消** 与 **网络闪断**，避免无意义重连打满模型 QPS。

## 九、RAG Pipeline 工程架构

1. RAG 从文档到回答的完整工程链路是什么？
答：**文档上传 → 文本提取（PDF/HTML/Office 等）→ 分块（Chunking）→ Embedding → 向量入库 → 查询阶段：用户问题 Embedding → 向量检索 Top-K →（可选）Rerank → 将 Top 片段拼进 Prompt 上下文 → LLM 生成答案**。后端还要管：元数据（租户、权限、版本）、入库任务与失败重试、索引一致性、缓存与成本。

2. 分块（Chunking）有哪些常见策略？过大过小各有什么问题？
答：常见策略包括：**固定 Token/字符窗口**（如 512、1024 tokens，可带 overlap）、**按段落/标题结构**、**按语义边界**（句子/主题漂移处切分）、**递归分割**（如 RecursiveCharacterTextSplitter：按分隔符优先级递归拆，尽量保留可读片段）。**块太大**：单次检索混入噪声多、Embedding 平均化严重、占用上下文窗口。**块太小**：单块信息不完整、跨段关系丢失，召回片段拼起来仍缺关键前提。一般要在 **召回率、噪声、延迟、成本** 之间用评测与 A/B 调参。

3. 向量数据库怎么选型？
答：**Milvus**：云原生、分布式能力强，适合 **大规模、高吞吐** 场景。**Pgvector**：PostgreSQL 扩展，**与业务库同栈、事务与权限模型简单**，适合中小规模或快速落地。**Qdrant**：Rust 实现，**性能与过滤能力**常被认可，可自建或云服务。**Chroma**：**轻量、上手快**，适合原型与小团队。选型还要看：过滤维度（租户/标签）、混合检索需求、运维成本、SLA 与多副本。

4. Rerank（重排）为什么往往必不可少？
答：向量检索多是 **ANN 近似最近邻**，属于 **粗召回**：快但排序不一定最贴合当前问题表述。**Rerank** 常用 **交叉编码器（Cross-Encoder）** 对「查询-文档」对逐条打分，做 **精排**，能 **显著提升相关性** 与最终答案质量。常见方案包括 **Cohere Rerank**、**BGE-Reranker** 等（自建或 API）。工程上要控制 **候选集规模**（只对 Top-K′ 重排）以平衡 **延迟与成本**。

5. RAG 常见评测维度有哪些？
答：**Faithfulness（忠实度）**：答案是否被上下文支持、是否幻觉。**Answer Relevancy（答案相关性）**：答案是否真正回应问题。**Context Precision（上下文精确度）**：检索到的片段里有多少是“有用且少噪声”的。**Context Recall（上下文召回率）**：回答问题所需的关键信息是否被检索覆盖。实际项目还会加 **延迟、成本、拒答率、安全合规** 等指标，与业务阈值一起看。

## 十、Function Calling 与 Tool Use 协议

1. Function Calling（工具调用）在协议层是怎么运转的？
答：在发给模型的请求里声明 **`tools` 数组**：每项含 **name、description、parameters（JSON Schema）** 等，描述可调函数。模型在需要时返回 **`tool_calls`**：**函数名 + 参数 JSON**（可能多个）。**后端执行** 对应函数（查库、HTTP、计算等），把结果以 **`role: tool`**（或厂商约定的 tool 消息）**回填到对话**，再请求模型 **继续生成** 自然语言回答。核心是：**模型只“提议”调用，真正 IO 与副作用在后端**。

2. 多轮、多次工具调用有哪些典型形态？
答：**并行**：一次响应里 **多个 `tool_calls`**，后端可 **并行执行**（注意下游限流与超时），**一次性**把多条 tool 结果填回。**串行多轮**：例如先查天气再查航班再总结，需要 **多轮「模型 → tool_calls → 执行 → 再请求模型」**。实现上要维护 **消息列表顺序**（assistant 带 tool_calls → 多条 tool → 再 assistant），并与厂商 schema 对齐。

3. Agent 循环一般怎么写？要防什么坑？
答：典型结构：**`while` 循环**：发当前消息列表给模型 → 若含 **`tool_calls`** 则执行工具并追加 tool 消息 → 若返回普通 **`content`** 且无待处理 tool_calls 则 **结束循环并返回**。必须设 **`max_iterations` 或最大工具步数**，防止模型反复调用或逻辑错误导致 **无限循环**；同时设 **单工具超时、总耗时预算**，避免拖死线程与账单失控。

4. 工具调用在生产上必须哪些安全约束？
答：**工具白名单**（仅暴露安全能力）、**参数校验**（JSON Schema + 业务规则，防注入与越权）、**执行超时与资源配额**、**权限控制**（按用户/租户隔离数据范围）、**审计日志**（谁、何时、调了什么、入参摘要、结果摘要）。对高风险操作（支付、删数据）应 **二次确认或人工审批**，而不是完全交给模型自动执行。
