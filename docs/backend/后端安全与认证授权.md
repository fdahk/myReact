# 后端安全与认证授权

## 使用方式

这篇不是让你背一堆安全名词，而是帮你建立后端工程师必须具备的安全边界意识。

你至少要能回答下面这些问题：

- 用户是谁，系统如何确认他的身份。
- 用户能做什么，权限边界如何落到接口和数据层。
- 常见攻击面在哪里，后端如何防守。
- 如果系统支持登录、权限、网关、多端接入、微服务，认证和鉴权链路如何设计。

安全题很容易答得很空，所以你读这篇时要一直带着“链路视角”：

登录 -> 签发凭证 -> 请求携带 -> 服务校验 -> 权限判断 -> 审计记录 -> 过期与退出。

---

## 一、认证和授权先分清

1. 认证和授权的区别是什么？
答：认证解决的是“你是谁”，授权解决的是“你能做什么”。比如用户登录时输入账号密码、验证码、短信码，这是认证；登录后是否能看某个页面、调用某个接口、操作某份数据，这是授权。很多人把这两个词混着说，但后端设计里它们分工非常明确。

2. 为什么后端一定要自己兜住权限，而不能只靠前端？
答：因为前端权限更多是体验层控制，防止用户看到不该看的入口，但真正的数据边界和操作边界必须由服务端校验。只要请求能被构造出来，前端隐藏按钮这种保护就会失效，所以后端必须在接口层、服务层甚至数据层再次确认权限。

3. 登录态方案为什么是后端高频题？
答：因为它同时涉及安全性、易用性、扩展性和多端兼容。你不只是要让用户“能登录”，还要回答凭证怎么签发、怎么携带、怎么续期、怎么失效、跨端怎么办、微服务里如何统一校验、出了安全问题如何追踪和回收。

## 二、常见认证方案怎么讲

4. Session/Cookie 方案的核心思路是什么？
答：服务端在用户登录后创建会话，把会话标识返回给浏览器，浏览器后续通过 Cookie 自动携带该标识，服务端再根据它找到对应 Session 数据。这种方案在同站后台系统里很常见，优点是服务端控制力强；缺点是在分布式、多端接入和跨域场景下，状态共享、扩容和跨系统统一鉴权会更复杂。

5. JWT 的核心思路是什么？
答：`JWT` 更像服务端签发给客户端的自描述凭证，里面通常包含用户标识、过期时间、签发方等信息，客户端后续请求时自行携带，服务端通过签名校验其合法性。它的优点是无状态、便于多服务统一验证；缺点是撤销、续期、权限变更后的即时失效处理会更麻烦，所以不能把它理解成“永远比 Session 高级”。

6. Session 和 JWT 怎么选？
答：如果是同域后台系统、浏览器为主、服务端状态管理能力强，`Session` 往往更直观；如果是前后端分离、多端接入、网关统一鉴权、服务较多，`JWT` 或 Access Token 模式会更常见。但真实系统经常是混合方案，比如短期 Access Token + 刷新 Token + 服务端黑名单或 Redis 会话记录，这样既兼顾扩展性，也能增强撤销能力。

7. 什么是刷新 Token，为什么需要它？
答：如果 Access Token 过期太长，泄漏风险高；过期太短，又会频繁要求用户重新登录。刷新 Token 的作用就是把“短有效期、高安全性”的访问凭证和“更长有效期、专门用于换新”的刷新凭证分开。这样用户体验和安全性可以相对平衡，但也要注意刷新风暴、刷新令牌泄漏和退出失效这些问题。

8. OAuth2 和 OIDC 在面试里应该怎么讲？
答：不用把所有授权模式都背成 RFC，而是先说清场景。`OAuth2` 更像授权框架，解决“第三方应用如何在不拿到用户密码的情况下访问受保护资源”；`OIDC` 则是在其基础上补了身份层，让客户端不仅能拿到授权，还能获得用户身份信息。面试里只要你能讲清它们解决的问题和典型使用场景，就比死背协议细节更有价值。

## 三、权限模型怎么落地

9. 常见权限模型有哪些？
答：最常见的是 `RBAC`，也就是基于角色控制权限；更细一点的是 `ABAC`，按属性做判断，比如部门、地区、资源标签、时间窗口；很多企业系统真实落地时是混合模型，例如“角色管粗粒度功能权限，属性管细粒度数据权限”。面试里不要只报模型名，更要说明业务上为什么需要更细粒度控制。

10. 页面级、接口级、数据级权限分别是什么？
答：页面级权限解决“能不能看到这个功能入口”；接口级权限解决“能不能调用这个接口动作”；数据级权限解决“即使调用成功，能看到哪些数据、能改哪些数据”。成熟系统通常三层都要考虑，否则很容易出现按钮隐藏了但接口还能调、接口拦住了但数据边界没拦住的漏洞。

11. 数据权限为什么比按钮权限更难？
答：因为它不是简单的 true/false 判断，而是经常和组织架构、部门树、租户边界、资源 ownership、时间范围等业务语义深度绑定。比如一个运营只能看自己负责的业务线数据，一个主管能看本部门，一个超级管理员能看全量，这已经不是前端按钮能解决的事，而是查询条件和领域模型的问题。

12. 多租户系统的权限边界怎么讲？
答：多租户系统首先要保证租户隔离，也就是 A 租户的数据不能被 B 租户访问。然后再在租户内部做角色和数据权限控制。面试里可以强调三层边界：租户边界、组织边界、角色动作边界。越是 B 端系统，这种分层越重要，因为权限问题一旦出错，往往不是体验 bug，而是严重的数据安全事故。

## 四、常见攻击面和后端防守点

13. SQL 注入的本质是什么？如何防？
答：本质是把用户输入错误地拼进 SQL 语句，让输入突破了原本的数据含义，变成了可执行的查询逻辑。防御的核心不是“记住 SQL 注入很危险”，而是坚持参数化查询、ORM 或预编译语句，不直接拼接动态 SQL，同时对危险输入做校验和审计。

14. XSS 为什么明明像前端问题，后端也要关注？
答：因为很多存储型 XSS 的根源在于后端没有对富文本、评论、消息内容做足够的输入治理和输出策略约束。后端虽然不直接渲染页面，但它在数据入口和内容治理上承担很大责任。如果把所有过滤都推给前端，跨端、多端、管理后台场景下很容易漏。

15. CSRF 的核心成因是什么？
答：本质是浏览器会自动携带某些凭证，例如 Cookie，导致用户在已登录状态下访问恶意页面时，恶意请求也可能被目标站点当成合法请求。防御常见手段包括 CSRF Token、SameSite Cookie、关键操作二次确认和验证码等。

16. 为什么越权是企业系统里特别高频的问题？
答：因为很多系统做了认证，却没有把授权边界真正落到资源层。比如用户能登录，也只能看到自己的订单列表，但接口只按订单 id 查询，没有校验这个订单是否属于当前用户，这就是典型的对象级越权。后端题里如果你能主动提到“对象 ownership 校验”，会比只说 RBAC 更成熟。

17. 密码存储为什么不能明文，也不能简单 `md5`？
答：因为明文一旦泄漏就是灾难，而简单哈希在撞库和彩虹表面前也很脆弱。更合理的做法是使用带盐、专门为密码设计的哈希算法，例如 `bcrypt`、`scrypt` 或 `Argon2`，并配合登录频控、异地登录提醒和审计日志构成完整防线。

18. 文件上传为什么也是安全高发点？
答：因为上传的不只是“一个文件”，还可能带来恶意脚本、危险后缀、超大文件压垮存储、伪造 MIME、路径穿越等问题。后端要做的通常包括大小限制、类型校验、文件名隔离、对象存储托管、病毒扫描或异步审核等，而不是把文件直接写进本地目录就结束。

## 五、网关和微服务里的认证授权

19. 微服务里认证一般放在哪一层做？
答：通常外层网关会先做一轮认证，比如校验 Token、提取用户身份、做基础限流和黑名单判断；进入具体服务后，还要在服务内部继续做授权判断。因为网关更适合做统一入口控制，但真正细粒度的业务权限仍然需要下沉到服务内部。

20. 为什么网关做了鉴权，服务内部还要再校验？
答：因为网关更偏通用身份校验，知道“你是谁”；但具体到某个资源、某条订单、某个部门的数据权限，往往只有业务服务最清楚。只靠网关容易把业务授权逻辑写得过重，也容易漏掉对象级校验，所以成熟系统通常是网关做粗粒度，服务做细粒度。

21. 权限变化后，JWT 为什么会有“旧权限还在生效”的风险？
答：因为 `JWT` 一旦签发，在过期前通常可以被离线校验。如果一个用户刚被降权，但旧 token 还没过期，系统就可能在短时间内继续接受它。解决思路通常包括缩短 token 生命周期、配合刷新机制、维护黑名单、在关键权限变更后主动踢出会话等。

22. 单点登录系统里最重要的点是什么？
答：不是“一个页面能跳来跳去”，而是统一身份源、统一会话生命周期、统一退出、统一审计和统一权限同步。如果单点登录做得不好，用户表面上像是打通了多个系统，实际上身份状态、权限状态和退出状态都可能不一致，这会带来非常大的安全和运维风险。

## 六、审计与风控意识

23. 为什么安全设计里要强调审计日志？
答：因为很多安全问题不是事前完全防住，而是出了问题后要能追。你至少要能回答：是谁、在什么时间、从哪里、对哪个资源、做了什么动作、结果如何。审计日志既服务于安全追踪，也服务于合规、风控和权限争议处理。

24. 高风险操作通常怎么加保护？
答：常见做法包括二次确认、验证码、短信或邮箱二次验证、设备校验、短时 token、权限升级申请、审批流等。比如删库、导出敏感数据、重置权限、批量转账这些操作，都不应该只靠“用户已经登录”这一层保护。

25. 后端安全题里最容易答空的地方是什么？
答：很多人只会说“防 SQL 注入、防 XSS、防 CSRF”，但不会把这些点放到一条真实系统链路里。更成熟的讲法是：登录怎么做、凭证怎么存、接口怎么鉴权、资源 ownership 怎么校验、危险操作怎么审计、出现异常登录和越权时怎么追踪与止损。

## 九、令牌、会话与高频事故场景

26. Token 泄漏后最怕什么？
答：最怕泄漏后攻击者能在较长时间内持续伪装成合法用户，并且系统还缺少吊销、黑名单、设备绑定或异常登录检测能力。很多人以为 token 只是一个字符串，实际上它代表完整的会话权限，一旦泄漏，后果和密码泄漏一样严重，甚至更隐蔽。

27. 为什么 Access Token 不能过期太长？
答：因为一旦泄漏，攻击窗口就会变长。过期太长会让你很难在权限变更、用户退出、账号异常时快速收敛风险。所以成熟系统通常会让 Access Token 生命周期较短，再配合刷新机制平衡体验。

28. 刷新 Token 最容易出什么问题？
答：一是刷新风暴，也就是多个请求同时发现过期，重复刷新；二是刷新 Token 本身泄漏后，攻击者可以不断换新 token；三是退出登录或权限变更后，刷新链路没有被正确收回。也就是说，刷新机制本身也是安全设计的一部分。

29. 为什么权限变更后“旧 token 还有效”是高频风险？
答：因为很多系统为了无状态校验，默认在 token 有效期内一直接受它。但业务上用户可能已经被降权、禁用、踢出组织或角色变更，这时如果旧 token 还在生效窗口里，就会出现“理论上没权限，实际上还能操作”的安全漏洞。

## 十、越权与对象 ownership 场景

30. 什么是水平越权？
答：水平越权通常指同级别用户之间越界访问。比如 A 用户和 B 用户权限等级相同，但 A 通过修改订单 id、资源 id、任务 id，读到了本该只属于 B 的数据。这类问题特别高发，因为很多系统只校验“有没有登录”，却没校验“这个资源是不是你的”。

31. 什么是垂直越权？
答：垂直越权是低权限用户获得了更高权限动作，比如普通运营绕过前端限制直接调用管理员接口、导出全量敏感数据、修改组织配置。它通常暴露的是角色与动作边界没真正落到服务端。

32. 为什么对象 ownership 校验这么关键？
答：因为很多业务资源天生带有“属于谁”的语义，例如订单属于某个用户、任务属于某个租户、审批单属于某个部门。后端如果只按主键查询，不再校验 ownership，就等于默认任何知道 id 的人都能访问。这个问题在多租户和 B 端系统里尤其危险。

33. 面试里怎么把越权题答得更成熟？
答：我会主动分三层讲：第一层是认证，确认请求方身份；第二层是动作权限，确认他能不能执行这个接口；第三层是对象级权限，确认这个资源是否属于他、是否在他的租户/部门/数据范围内。这样就不会把权限题简化成一个 `RBAC` 名词。

## 十一、网关、安全边界与微服务协同

34. 为什么微服务里安全边界不能只放在网关？
答：因为网关更适合做统一入口认证、基础限流和黑名单判断，但对象级权限、数据权限、部门边界、租户边界通常只有业务服务自己最清楚。如果所有安全判断都压在网关，会让网关过重，而且很多业务语义根本放不下去。

35. 网关把用户信息透传给下游时最容易出什么问题？
答：最容易出的问题是下游盲信网关传过来的用户头，而没有验证来源可信性；或者内部服务之间可以伪造这些头，导致身份冒充。成熟系统通常会配合服务间鉴权、可信网络边界、签名或内部令牌，而不是默认 header 来了就可信。

36. 服务间调用为什么也要考虑认证？
答：因为内部网络并不天然安全。服务间如果缺少身份校验和访问边界，一旦某个服务被攻陷，攻击面就可能迅速横向扩散。所以成熟系统不仅考虑“用户怎么进来”，也考虑“服务之间如何互信、如何最小权限访问”。

## 十二、审计、风控与异常登录

37. 审计日志最值得记录哪些字段？
答：至少包括操作者身份、租户/组织信息、目标资源 id、动作类型、时间、来源 IP、设备/客户端信息、请求结果、traceId、变更前后摘要。因为真正出事时，你不只是想知道“谁调了接口”，还要知道“谁改了哪个资源、改前改后是什么、能不能串到完整链路”。

38. 为什么异常登录检测也是后端安全的一部分？
答：因为安全不只是接口能不能拦住，还包括账号被盗、异地登录、设备异常、短时间暴力尝试、敏感操作突增这些行为型风险。成熟系统往往会配合登录频控、设备指纹、异地告警、风险校验等机制，而不是只看密码对不对。

39. 高风险导出、批量删除、权限变更这类动作为什么要二次保护？
答：因为这些动作一旦误操作或被恶意触发，损失会非常大。仅仅“已经登录”这层防护不够，通常还要配合验证码、二次确认、审批、短时签名、操作审计甚至双人复核。越高风险的动作，越不能只靠基础登录态托底。

## 十三、最值得练熟的 10 个追问

1. Token 泄漏后最危险的后果是什么，怎么止血？

## 十四、JWT 内部结构与签名机制

40. JWT 的三段结构是什么？
答：JWT 由三段用 `.` 连接的字符串组成：`Header.Payload.Signature`，每一段都是 Base64Url 编码。拿到一个 JWT 后可以直接用 `base64url_decode` 拆开前两段看到明文 JSON，第三段是签名，用于验证前两段是否被篡改。举例：`eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIn0.dozjgNryP4J3jVmNHl0w5N_XgL0n3I9PlFUP0THsR8U`，三段分别对应 Header、Payload、Signature。

41. Header 里包含什么？
答：Header 是一个 JSON 对象，通常只有两个字段：`alg` 表示签名算法（如 `HS256`、`RS256`），`typ` 固定为 `JWT`。示例：`{"alg": "HS256", "typ": "JWT"}`。服务端验证时会读取 Header 中的 `alg` 来决定用什么算法校验签名，所以这个字段非常关键——如果不在服务端做白名单限制，就会产生安全漏洞。

42. Payload 里常见哪些字段？
答：Payload 也叫 Claims，分为三类。第一类是 **标准注册字段(Registered Claims)**：`iss`（签发者，如 `auth.example.com`）、`sub`（主题，通常放用户唯一标识）、`aud`（受众，指定 token 给谁用）、`exp`（过期时间，Unix 时间戳）、`iat`（签发时间）、`nbf`（生效时间）、`jti`（JWT 唯一 ID，用于防重放）。第二类是 **公共字段(Public Claims)**：在 IANA 注册表登记的，如 `name`、`email`。第三类是 **私有字段(Private Claims)**：业务自定义的，如 `userId`、`role`、`tenantId`、`permissions` 等。

43. 为什么说 Payload 不能放敏感信息？
答：因为 Payload 只是 Base64Url 编码，不是加密。任何拿到 JWT 的人都可以直接 `base64url_decode` 看到 Payload 的全部内容。所以绝对不能在 Payload 里放密码、身份证号、银行卡号等敏感数据。如果确实需要对 Payload 保密，应该使用 JWE（JSON Web Encryption）而非 JWS。

44. Signature 是怎么计算的？
答：以 HS256 为例，签名计算过程是：`HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret)`。也就是把编码后的 Header 和 Payload 用 `.` 拼起来，再用密钥对这个字符串做 HMAC-SHA256 运算。验证时用同样的方式重新计算签名，和 token 中携带的签名对比，如果一致则说明 token 未被篡改。

45. HS256(对称签名)和 RS256(非对称签名)有什么区别？
答：**HS256** 使用同一个 `secret` 进行签名和验证，速度快、实现简单，但所有需要验证 token 的服务都必须知道这个 secret，一旦泄漏影响面很大。**RS256** 使用 RSA 私钥签名、RSA 公钥验证，签名方持有私钥，验证方只需要公钥。优点是验证方不需要也不应该持有私钥，更适合微服务场景——认证服务用私钥签发，各个业务服务只持有公钥做验证，即使公钥泄漏也无法伪造 token。缺点是 RSA 运算比 HMAC 慢，密钥管理更复杂。实际项目中公钥通常通过 JWKS（JSON Web Key Set）端点动态发布。

46. JWT、JWS、JWE 三者是什么关系？
答：**JWT（JSON Web Token）** 是一个规范，定义了 token 的 Claims 结构和紧凑序列化格式。**JWS（JSON Web Signature）** 是 JWT 的签名实现，也是最常用的形式——Payload 明文可见但受签名保护，不可篡改。**JWE（JSON Web Encryption）** 是 JWT 的加密实现——Payload 被加密，只有持有解密密钥的人才能看到内容。日常说的"JWT"几乎都是指 JWS。如果业务需要同时保证完整性和保密性，可以先 JWS 签名再 JWE 加密（嵌套 JWT）。

47. JWT 有哪些常见攻击方式？如何防御？
答：**第一，none 算法攻击**：攻击者把 Header 中的 `alg` 改为 `none`，然后去掉 Signature 段。如果服务端代码没有校验算法就直接放行，token 就被绕过了。防御：服务端必须强制白名单验证 `alg` 字段，拒绝 `none` 和不在预期列表中的算法。**第二，密钥暴力破解**：如果 HS256 使用了弱 secret（如 `secret123`），攻击者可以用工具（如 `jwt_tool`、`hashcat`）快速遍历出密钥。防御：使用至少 256 位的随机密钥，或直接使用 RS256 非对称方案。**第三，算法混淆攻击**：服务端配置为 RS256，攻击者把 `alg` 改为 HS256，并用公钥（公开可获取）作为 HMAC secret 来签名。如果服务端不严格区分对称/非对称算法，就可能被欺骗。防御：签名验证时强制绑定预期算法，不从 token 的 Header 动态决定使用哪种算法。**第四，Token 注入/重放**：攻击者截获合法 token 后重复使用。防御：设置合理的 `exp`、使用 `jti` 配合服务端去重、绑定客户端指纹。

## 十五、OAuth2 授权码流程详解

48. OAuth2 有哪四种授权模式？各自适用什么场景？
答：**第一，Authorization Code（授权码模式）**：最安全、最常用，适用于有后端的 Web 应用，token 交换在后端完成，不暴露给浏览器。**第二，Implicit（隐式模式）**：token 直接通过 URL fragment 返回给前端，已被 OAuth 2.1 废弃，因为 token 暴露在浏览器地址栏，容易泄漏。**第三，Client Credentials（客户端凭证模式）**：没有用户参与，机器对机器场景，如后端服务之间相互调用，用 client_id + client_secret 直接换 token。**第四，Resource Owner Password Credentials（密码模式）**：用户把账号密码直接给客户端，客户端再换 token。已不推荐使用，因为客户端能拿到用户密码，违背了 OAuth2 的核心设计初衷。

49. Authorization Code 完整流程是什么？
答：以"用微信登录"为例，完整链路如下：
**第一步**，用户点击"用微信登录"，前端将浏览器重定向到授权服务器：`GET /authorize?response_type=code&client_id=xxx&redirect_uri=https://myapp.com/callback&scope=read%20profile&state=random_csrf_string`。其中 `state` 是随机字符串，用于防 CSRF。
**第二步**，用户在授权服务器的页面上登录，并同意授权（"允许 myapp 获取你的头像和昵称"）。
**第三步**，授权服务器将浏览器重定向回 `redirect_uri`，URL 中带上授权码和 state：`https://myapp.com/callback?code=AUTH_CODE_abc123&state=random_csrf_string`。
**第四步**，myapp 后端收到 callback 请求后，先验证 `state` 是否和第一步生成的一致（防 CSRF），然后用授权码向授权服务器换 token：`POST /token`，请求体包含 `grant_type=authorization_code`、`code=AUTH_CODE_abc123`、`client_id=xxx`、`client_secret=xxx`、`redirect_uri=https://myapp.com/callback`。注意 `client_secret` 只在后端使用，绝不暴露给前端。
**第五步**，授权服务器验证授权码和 client_secret 后返回：`{"access_token": "eyJ...", "refresh_token": "dGhpcyBp...", "expires_in": 3600, "token_type": "Bearer"}`。
**第六步**，myapp 后端用 `access_token` 调用资源服务器的 API（如 `GET /userinfo`，Header 带 `Authorization: Bearer eyJ...`）获取用户信息。

50. PKCE 是什么？为什么需要它？
答：PKCE（Proof Key for Code Exchange，读作"pixy"）是 OAuth2 授权码模式的增强，主要防止授权码在传输过程中被拦截后被恶意使用。流程是：客户端先生成一个随机的 `code_verifier`（43-128 位的随机字符串），然后计算 `code_challenge = BASE64URL(SHA256(code_verifier))`。在第一步请求授权时，把 `code_challenge` 和 `code_challenge_method=S256` 一起发送给授权服务器。在第四步用授权码换 token 时，把原始的 `code_verifier` 发送给授权服务器。授权服务器对 `code_verifier` 做 SHA256 计算，和之前收到的 `code_challenge` 比对，匹配才发 token。这样即使攻击者拦截到了授权码，因为没有 `code_verifier`，也无法用授权码换到 token。PKCE 最初为移动端和 SPA 设计，OAuth 2.1 已将其作为所有授权码流程的必选项。

51. state 参数为什么重要？
答：`state` 参数用于防止 CSRF 攻击。攻击场景是这样的：攻击者自己走完授权流程拿到授权码，然后构造一个包含这个授权码的 callback URL，诱导受害者点击。如果没有 `state` 校验，受害者的后端就会用攻击者的授权码去换 token，导致受害者的账号绑定了攻击者的第三方账号。有了 `state`，后端在发起授权前生成随机字符串存到 session，callback 回来时验证 `state` 是否一致，就能确认这个 redirect 请求确实是自己发起的。

52. Access Token 和 ID Token 有什么区别？
答：**Access Token** 是 OAuth2 定义的，用于访问资源服务器的 API，资源服务器通过它判断请求方有没有权限访问某个资源。它的内容对客户端来说是不透明的（可以是 JWT 也可以是随机字符串）。**ID Token** 是 OIDC（OpenID Connect）在 OAuth2 之上扩展的，一定是 JWT 格式，包含用户身份信息（如 `sub` 用户唯一标识、`name`、`email`、`picture` 等）。简单说：Access Token 回答"我能访问什么资源"，ID Token 回答"登录的用户是谁"。客户端需要知道用户身份时用 ID Token，调用 API 时用 Access Token，两者不要混用。

## 十六、密码存储与哈希算法

53. 为什么密码不能明文存储？
答：因为数据库泄漏的概率不为零。一旦发生拖库，如果密码是明文，所有用户的密码直接暴露。而且很多用户会在多个平台使用相同密码，一个站点泄漏意味着用户在其他平台的账号也面临风险（撞库攻击）。

54. 为什么不能用 MD5 或 SHA256 来哈希密码？
答：因为 MD5 和 SHA256 是通用哈希算法，设计目标就是速度快。现代 GPU 对 MD5 的计算速度可以达到 **数十亿次/秒**，SHA256 也可达数亿次/秒。攻击者可以轻松暴力枚举常见密码。此外，不加盐的哈希还会被彩虹表（预计算好的 hash → 明文映射表）直接查表破解。

55. 加盐(Salt)是什么？为什么能防彩虹表？
答：Salt 是为每个用户单独生成的随机字符串，存储时和哈希值一起保存。计算过程是 `hash(password + salt)`。加盐的关键在于：即使两个用户使用了完全相同的密码，因为各自的 salt 不同，最终哈希值也不同。这使得彩虹表无法预计算——攻击者需要针对每个 salt 单独构建彩虹表，成本指数级增长，实际上等于不可行。

56. bcrypt 的工作原理和格式是什么？
答：**bcrypt** 是专门为密码哈希设计的算法（基于 Blowfish 加密），核心特点有三个。**第一，内置 salt**：每次哈希自动生成 16 字节随机 salt，不需要手动管理。**第二，工作因子(cost factor)**：参数范围 4-31，默认通常为 10。每增加 1，计算时间翻倍。cost=10 大约需要 100ms，cost=12 大约 400ms，cost=14 大约 1.6s。**第三，固定输出格式**：`$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy`，其中 `$2a$` 是算法版本标识（还有 `$2b$`），`10` 是 cost 值，接下来 22 个字符是 Base64 编码的 salt，最后 31 个字符是 Base64 编码的 hash。bcrypt 之所以比 MD5 好，核心在于它 **故意慢**——100ms 对正常用户登录无感，但对攻击者来说每秒只能尝试约 10 次（而 MD5 每秒可以数十亿次），暴力破解成本高到不可行。

57. Argon2 是什么？为什么说它比 bcrypt 更强？
答：**Argon2** 是 2015 年密码哈希竞赛(Password Hashing Competition)的冠军算法，目前被认为是最先进的密码哈希方案。它有三个变体：**Argon2d** 最大化 GPU 抵抗力（数据依赖型内存访问），**Argon2i** 最大化侧信道攻击抵抗力（数据无关型内存访问），**Argon2id** 是推荐的默认选择，兼具两者优点（先用 Argon2i 填充内存再用 Argon2d 处理）。Argon2 有三个核心参数：**时间成本**(迭代次数，控制 CPU 耗时)、**内存成本**(使用多少 KB 内存，如 65536 即 64MB)、**并行度**(并行计算线程数)。Argon2 比 bcrypt 更强的原因在于：bcrypt 只是 CPU 密集型，而 Argon2 同时是 **CPU 密集 + 内存密集**。GPU 虽然有大量计算核心，但每个核心的内存很有限，当 Argon2 要求大量内存时，GPU 的并行优势就被严重削弱，这使得 GPU/ASIC 暴力破解的成本大幅提升。

58. 实际项目中怎么选择密码哈希方案？
答：**Spring Security 默认使用 `BCryptPasswordEncoder`**，推荐 cost 设为 10-12（在服务器上验证耗时约 100-400ms，用户无感但攻击者痛苦）。如果是新项目且语言/框架支持良好，优先考虑 **Argon2id**（如 Java 的 Bouncy Castle 库、Node.js 的 argon2 包、Go 的 golang.org/x/crypto/argon2）。选择时的核心原则：**让单次哈希耗时在 100ms-1s 之间**——太快则暴力破解容易，太慢则影响正常登录体验。随着硬件算力提升，应定期上调工作因子。

## 十七、CORS 跨域机制

59. 同源策略(Same-Origin Policy)是什么？
答：同源策略是浏览器的安全机制，限制一个源（Origin）的文档或脚本与另一个源的资源进行交互。**源 = 协议 + 域名 + 端口**，三者必须完全一致才算同源。例如 `https://app.example.com:443` 和 `https://api.example.com:443` 不同源（域名不同），`http://example.com` 和 `https://example.com` 不同源（协议不同），`http://example.com:80` 和 `http://example.com:8080` 不同源（端口不同）。注意：同源策略只在浏览器端生效，服务端之间互相调用 API 不受此限制。

60. 什么是简单请求？什么是预检请求(Preflight)？
答：浏览器把跨域请求分为两类。**简单请求**：同时满足以下条件——方法为 `GET`、`HEAD` 或 `POST`；`Content-Type` 仅限 `text/plain`、`application/x-www-form-urlencoded`、`multipart/form-data`；没有自定义请求头（只能使用 `Accept`、`Accept-Language`、`Content-Language` 等安全头）。简单请求浏览器直接发送，不需要预检。**预检请求(Preflight)**：不满足上述条件的请求，比如 `PUT`/`DELETE` 方法、`Content-Type: application/json`、携带自定义 header（如 `Authorization`）。浏览器会先自动发送一个 `OPTIONS` 请求询问服务器是否允许，服务器同意后浏览器才发送真正的请求。

61. OPTIONS 预检请求的具体交互是什么？
答：**预检请求(浏览器自动发送)**：方法为 `OPTIONS`，携带三个关键请求头——`Origin: https://app.example.com`（请求来源）、`Access-Control-Request-Method: PUT`（实际请求要用的方法）、`Access-Control-Request-Headers: Authorization, Content-Type`（实际请求要带的自定义头）。**预检响应(服务端返回)**：`Access-Control-Allow-Origin: https://app.example.com`（允许的源）、`Access-Control-Allow-Methods: GET, POST, PUT, DELETE`（允许的方法）、`Access-Control-Allow-Headers: Authorization, Content-Type`（允许的自定义头）、`Access-Control-Max-Age: 86400`（预检结果缓存时间，单位秒，这里是 24 小时内不再重复预检）。如果预检通过，浏览器才会发真正的 `PUT` 请求。

62. Access-Control-Allow-Credentials 有什么特殊要求？
答：当请求需要携带 Cookie（比如前端设置了 `fetch(url, {credentials: 'include'})`），服务端必须返回 `Access-Control-Allow-Credentials: true`。但此时有一个关键限制：**`Access-Control-Allow-Origin` 不能设为通配符 `*`，必须指定具体域名**（如 `https://app.example.com`）。原因是浏览器为了安全，不允许在携带凭证的情况下对所有源开放。同理，`Access-Control-Allow-Headers` 和 `Access-Control-Allow-Methods` 在带凭证时也不能用 `*`。

63. 后端通常怎么处理 CORS？
答：有三种常见方式。**第一，注解级别**：Spring 的 `@CrossOrigin(origins = "https://app.example.com")` 加在 Controller 或方法上，适合个别接口。**第二，全局 Filter**：注册一个 `CorsFilter` 或实现 `WebMvcConfigurer.addCorsMappings()`，统一配置所有接口的跨域策略，适合前后端分离项目。**第三，网关层统一配置**：在 Nginx、Spring Cloud Gateway、Kong 等网关上统一设置 CORS 响应头，后端服务本身不需要关心跨域，这是微服务架构下最推荐的方式，避免每个服务各自配置导致不一致。注意：不要在网关和服务中同时配置 CORS，否则响应头会重复，浏览器会报错。
