# Linux Docker 与部署

## 使用方式

这篇不是让你转运维，而是让你具备后端工程师最基本的上线、排障和运行环境认知。大厂后端不会因为你写了业务就放过你，服务最终总要部署、观测、回滚和排查。

---

## 一、为什么后端必须懂 Linux 与部署

1. 为什么后端岗位必须会 Linux？
答：因为服务通常运行在 Linux 环境，日志、进程、端口、权限、文件、资源占用、网络连接排查都离不开它。不会 Linux，你只能停留在本地 IDE 开发层，无法对真实运行环境负责。

2. 为什么顶级后端岗位非常看重部署与排障能力？
答：因为生产问题不会只出现在代码层。端口冲突、进程异常退出、配置错误、容器启动失败、磁盘满、连接数异常、CPU 飙高、内存泄漏都需要工程师具备基本环境诊断能力。

---

## 二、Linux 基础认知

3. 后端最常用的 Linux 能力有哪些？
答：查看文件与目录、进程管理、端口排查、权限控制、日志查看、资源监控、网络连接排查、脚本基础。你不需要变成系统专家，但这些必须熟练。

4. 为什么日志查看能力这么重要？
答：因为线上第一现场通常就是日志。不会快速定位日志文件、过滤关键请求、按时间或关键字回溯，很多问题根本无法开始排查。

5. 为什么进程、线程、端口要一起理解？
答：因为服务最终以进程形式运行，线程承载执行，端口对外提供网络能力。服务起不来、端口被占用、线程池打满，这些问题本质都和这三者关系密切。

---

## 三、容器化

6. Docker 解决什么问题？
答：它解决的是环境一致性和交付标准化问题。以前“我本地没问题、线上不行”很多是运行环境差异导致的，容器让应用和依赖更容易以可复制方式交付。

7. 为什么大厂后端会重视容器意识？
答：因为现代服务部署、扩缩容、灰度发布、镜像管理和基础设施治理都和容器强相关。你不一定现在就精通 `Kubernetes`，但至少要知道服务不是简单跑个 jar 就结束。

8. 镜像和容器的差异是什么？
答：镜像更像应用运行环境的静态模板，容器是镜像运行起来后的实例。这个区别非常基础，但关系到你是否真正理解容器不是“压缩包”。

---

## 四、部署链路

9. 一个 Java 服务部署大致会经过哪些环节？
答：构建产物、打包镜像、推送仓库、部署到目标环境、配置注入、启动健康检查、接入监控、灰度或全量放量。成熟后端要能从整体交付链路理解服务上线，而不是把上线当成别人负责的黑盒。

10. 为什么配置管理在部署中很关键？
答：因为不同环境的数据库、缓存、MQ、第三方地址、密钥和开关都可能不同。代码和配置必须解耦，否则环境切换和回滚都会很痛苦。

11. 健康检查为什么重要？
答：因为一个服务进程“活着”不等于真的可用。健康检查帮助负载均衡、容器编排和运维系统判断实例是否能接流量。

---

## 五、日志、进程与故障排查

12. 服务启动失败时，后端应该怎么排查？
答：先看启动日志，再查配置、端口、依赖连接、权限、环境变量、文件路径、镜像内容和资源限制。成熟排查是先收集事实，不是先猜框架问题。

13. CPU 飙高可能意味着什么？
答：死循环、线程忙等、热点请求、GC 压力、日志风暴、序列化成本高、下游阻塞导致重试等都可能引发。不要把 CPU 高只理解成“机器不够”。

14. 内存持续上涨最常见意味着什么？
答：缓存无边界、对象泄漏、消息积压、请求上下文未释放、集合膨胀、线程过多、JVM 参数不合理等。后端要能把资源问题和代码设计联系起来。

---

## 六、Docker 化时要有的工程意识

15. 为什么不是把应用装进容器就结束？
答：因为容器化之后仍然要考虑镜像体积、启动速度、配置注入、日志输出、健康检查、优雅停机和资源限制。很多项目“用了 Docker”，但工程质量并没有真正提升。

16. 优雅停机为什么重要？
答：因为服务在发布、扩缩容和故障转移时会被摘流量和关闭，如果处理不好，可能会中断正在执行的请求或任务，造成数据不一致和用户体验问题。

---

## 七、为什么这对你的 Java 主项目也关键

17. 做后端主项目为什么不能只写业务代码？
答：因为面试官会看你是不是具备完整工程视角。一个只会本地启动的项目，很难证明你理解真实交付链路。至少要做到：环境配置、Docker 化、日志输出、健康检查、基础部署说明这些能力。

18. 当前 `agent-server` 在这方面有什么启发？
答：当前仓库已经有 `Dockerfile` 和 `docker-compose.yml` 的雏形，这说明它具备一定服务化交付意识。但如果要重写成 Java 主项目，需要把部署、健康检查、配置外置、监控接入和优雅停机这些能力做得更完整。

## 九、最常见的 Linux 排障链路

19. 服务进程还在，但接口全超时，你怎么排查？
答：我会按四层看：先看进程是否真的健康，再看线程池、连接池、GC、日志；然后看端口监听和网络连接；接着看下游依赖是否超时；最后看机器资源是否已经接近瓶颈。这个题最怕一上来只说“重启试试”，因为那只是止血，不是排障。

20. 为什么排障时不能只看 CPU 和内存？
答：因为很多问题出在文件描述符耗尽、连接数打满、磁盘写满、日志阻塞、线程池排队、DNS 解析失败、下游连接超时，这些都不一定第一时间反映成 CPU 或内存异常。Linux 排障本质是系统性观察，不是只盯两个指标。

21. 磁盘满了为什么能把服务拖挂？
答：因为日志写不进去、临时文件落不下、数据库或队列本地持久化失败、容器层写入异常都会连锁影响服务。很多后端问题看起来像应用 bug，根因其实是最底层的磁盘资源已经耗尽。

22. 文件描述符耗尽通常意味着什么？
答：常见意味着连接泄漏、短连接风暴、日志文件句柄过多、某些文件流没正确关闭，或者系统限制值太低。它一旦耗尽，新的网络连接、文件读写都可能失败，所以这类问题很容易表现成“服务还活着，但越来越不正常”。

## 十、Docker 不是只会 build 和 run

23. 镜像为什么不能越大越随意？
答：因为镜像越大，构建、拉取、发布、回滚都会更慢，安全面也更大。成熟工程会关注基础镜像选择、分层缓存、无用依赖清理和镜像最小化，这既影响效率，也影响安全和交付质量。

24. 容器里为什么强调日志打到标准输出？
答：因为容器实例通常是短生命周期和可替换的，日志如果只写在容器内文件里，实例销毁后很容易丢失，而且不利于统一采集。把日志打到标准输出再由平台统一收集，是现代容器化部署里的常见最佳实践。

25. 为什么优雅停机会影响一致性？
答：因为一个实例被摘流量和关闭时，可能还有正在执行的请求、MQ 消费、长任务或数据库事务。如果不做优雅停机，任务可能执行一半就被硬切掉，导致重复消费、状态不一致或用户请求莫名失败。也就是说，优雅停机不是体验细节，而是数据正确性问题。

26. 健康检查设计差会有什么后果？
答：如果检查太浅，坏实例可能继续接流量；如果检查太重，又可能把服务自己压垮，或者把短暂抖动误判成不可用。健康检查真正重要的是：能否准确反映实例是否可以安全接流量，而不是简单返回一个 200。

---

## 十一、发布、回滚与基础设施意识

27. 发布时为什么要特别关注配置注入和环境变量？
答：因为很多问题并不是代码错了，而是数据库地址、MQ 地址、限流开关、证书、密钥、线程池参数在不同环境下不一致。后端工程师如果不理解配置注入和环境隔离，很容易把“部署问题”误判成“代码问题”。

28. 容器化后为什么还要理解网络？
答：因为服务之间如何通信、容器如何暴露端口、DNS 如何解析、反向代理如何转发、请求为什么跨容器失败，这些都和容器网络强相关。很多“服务起了但访问不到”的问题，本质上就是容器网络和服务发现层的问题。

29. 为什么基础设施问题经常会伪装成应用故障？
答：因为应用层看到的表象往往只是“请求超时”“连接失败”“日志中断”“实例偶发重启”，但真正根因可能在磁盘、网络、容器 OOM、探针误杀、配置错误、镜像问题。后端岗位要能意识到，代码不是唯一故障源。

---

## 十二、容器与编排的最小认知

30. 为什么即使现在不主攻运维，也建议理解一点编排？
答：因为现代后端服务很少永远只在单机上跑。你至少要理解实例副本、滚动发布、服务发现、探针、资源限制和自动恢复这些概念，这样你在设计服务、写健康检查、做优雅停机和排查实例抖动时才不会脱离运行环境。

31. 资源限制为什么重要？
答：因为容器环境里 CPU、内存、文件句柄、连接数都不是无限的。你如果不了解资源限制，服务在本地能跑，不代表在线上限制条件下也稳定。很多线上 OOM、频繁重启和性能抖动，本质上都和资源设定不合理有关。

32. 为什么后端工程师要关心 OOMKilled 这类现象？
答：因为这说明实例不是“正常退出”，而是被运行环境强制杀掉。它通常意味着内存使用超过限制，后果不只是实例挂掉，还可能带来请求中断、消费中断、任务丢状态和频繁重启。理解这类信号，才能把应用设计和运行环境真正连起来。

## 十三、主项目应该怎样体现部署能力

33. 如果要把 `agent-server-java` 做成能打人的主项目，部署层至少该补什么？
答：至少补六层：环境变量和配置外置、结构化日志、健康检查、优雅停机、Docker 化交付、基础监控接入。如果再进一步，还可以加灰度发布说明、对象存储配置、任务执行器独立部署和资源限制说明。这样面试官才会觉得你不是只写了功能，而是真的考虑了服务如何长期运行。

34. 主项目里最容易被忽视的部署细节是什么？
答：通常是：日志写法不适合容器环境、配置写死、本地路径依赖、优雅停机缺失、任务状态没持久化、健康检查过于简单。这些问题单看都不大，但一到真实部署和扩缩容场景里会集中爆出来。

35. 如果面试官问“你做过部署吗”，怎么答得更成熟？
答：我会按“构建 -> 镜像 -> 配置 -> 启动 -> 健康检查 -> 监控 -> 回滚”这条链讲，而不是只说“我会写 Dockerfile”。因为真正成熟的部署能力是对整条交付链路负责，而不是会一条命令。

## 十五、Linux 排障必备命令

### 进程相关

36. `ps aux` 和 `ps -ef` 分别怎么用？
答：两者都用于查看系统所有进程，区别在于输出格式不同。`ps aux` 是 BSD 风格，输出包含 `%CPU`、`%MEM`、`VSZ`、`RSS` 等字段，适合快速看资源占用；`ps -ef` 是 System V 风格，输出包含 `PPID`（父进程ID），适合看进程父子关系。最常用的组合：

```bash
# 找出所有 Java 进程
ps aux | grep java

# 查看某个进程的父进程关系
ps -ef | grep java
```

典型场景：服务起了但请求不通，先 `ps aux | grep java` 确认进程是否存活、PID 是多少、CPU 和内存占用是否异常，这是排障的第一步。

37. `top` 和 `htop` 怎么用来定位资源问题？
答：`top` 是 Linux 自带的实时资源监控工具，进入后按 `P` 按 CPU 排序、按 `M` 按内存排序。需要重点关注的列：
- `%CPU`：进程 CPU 使用率，超过 100% 说明使用了多个核心
- `%MEM`：物理内存使用率
- `VIRT`：虚拟内存总量（包含未实际分配的部分，不用太紧张）
- `RES`：实际驻留物理内存（这才是真正占了多少内存）

```bash
# 启动 top，默认按 CPU 排序
top

# 只看某个用户的进程
top -u deploy
```

`htop` 是增强版，支持鼠标操作、彩色显示、树形进程视图，但需要额外安装（`yum install htop` 或 `apt install htop`）。日常排障推荐用 `htop`，面试回答用 `top` 即可。

38. `jps` 是什么？什么时候用？
答：`jps` 是 JDK 自带的命令，专门列出当前机器上所有 Java 进程。比 `ps aux | grep java` 更精确，不会匹配到 grep 自身。

```bash
# 列出 Java 进程的 PID 和主类名
jps -l

# 输出示例：
# 12345 com.example.Application
# 12400 org.apache.kafka.Kafka
```

典型场景：机器上跑了多个 Java 服务，用 `jps -l` 可以快速区分哪个 PID 对应哪个服务，后续 `jstack`、`jmap` 等操作都需要先拿到 PID。

### CPU 排查

39. 线上 Java 服务 CPU 飙到 300%，怎么一步步定位到具体代码？
答：这是一个完整的排障链路，分四步走：

**第一步：找出 CPU 最高的线程**
```bash
# 查看进程内各线程的 CPU 占用（-H 表示显示线程，-p 指定进程）
top -Hp <pid>
```
进入后按 `P` 排序，记下 CPU 最高的线程 TID，比如 `12366`。

**第二步：将线程 ID 转为 16 进制**
```bash
printf "%x\n" 12366
# 输出：304e
```
因为 `jstack` 输出的线程 ID 是 16 进制的 `nid`，所以必须先转换。

**第三步：在线程 dump 中找到该线程的调用栈**
```bash
jstack <pid> | grep -A 30 "0x304e"
```
`-A 30` 表示匹配行之后再打印 30 行，足够看到完整的方法调用链。

**第四步：分析调用栈，定位问题代码**
看到的输出类似：
```
"http-nio-8080-exec-5" #45 daemon prio=5 os_prio=0 tid=0x00007f nid=0x304e runnable
   java.lang.Thread.State: RUNNABLE
    at com.example.service.ReportService.calculate(ReportService.java:87)
    at com.example.controller.ReportController.export(ReportController.java:34)
```
这就能看出是 `ReportService.java` 第 87 行在疯狂执行，通常是死循环、正则回溯、无限递归或热点计算代码。

完整流程总结：`top -Hp` 找忙线程 → `printf "%x"` 转 16 进制 → `jstack` 查调用栈 → 定位死循环或热点代码。这套流程是 Java 后端 CPU 排障的标准动作，面试必须能完整讲出来。

### 内存排查

40. 系统内存和 JVM 堆内存分别怎么查？
答：两个层面要分清楚：

**系统层面：`free -h`**
```bash
free -h
# 输出示例：
#               total    used    free   shared  buff/cache   available
# Mem:           16Gi    12Gi    512Mi   256Mi      3.2Gi       3.5Gi
# Swap:         2.0Gi    1.0Gi   1.0Gi
```
重点看 `available`（实际可用），而不是 `free`。因为 Linux 会用空闲内存做 buff/cache，这部分可以被回收。如果 `available` 很低且 `Swap` 使用量大，说明物理内存真的不够了。

**JVM 层面：`jmap -heap <pid>`**
```bash
jmap -heap <pid>
```
输出包含新生代（Eden、Survivor）、老年代（Old Gen）的容量和使用量。如果老年代使用率持续接近 100%，说明可能存在内存泄漏或堆设置过小。

41. 怎么找出内存泄漏的具体对象？
答：分两步走——先看统计，再看 dump。

**快速统计存活对象：**
```bash
# 统计堆中存活对象的类型、数量和大小（注意：会触发一次 Full GC）
jmap -histo:live <pid> | head -30
```
输出按对象大小排序，如果看到某个业务类的实例数异常多（比如几十万个 `UserDTO`），基本可以锁定泄漏方向。

**导出完整堆 dump：**
```bash
# 导出 heap dump 文件（生产环境慎用，会导致 STW 停顿）
jmap -dump:format=b,file=heap.hprof <pid>
```
文件通常很大（几百 MB 到几 GB），需要下载到本地用工具分析：
- **MAT（Eclipse Memory Analyzer）**：找出 Retained Heap 最大的对象，顺着引用链追泄漏源头
- **VisualVM**：轻量级，适合快速查看对象分布

典型场景：服务运行一段时间后 Old Gen 不断增长且 Full GC 后回不来，先 `jmap -histo:live` 看哪类对象最多，再 dump 下来用 MAT 分析引用链，通常是缓存没过期、集合只加不删、监听器没注销等问题。

### 网络排查

42. 怎么查看当前机器上哪些端口在监听？
答：两个等价命令：

```bash
# 传统命令（较老的系统）
netstat -tlnp
# -t: TCP  -l: 监听状态  -n: 数字显示  -p: 显示进程

# 推荐命令（更快，新系统默认有）
ss -tlnp
```

输出能直接看到端口号和对应的进程 PID，比如 `0.0.0.0:8080` 对应 `pid=12345`。

典型场景：服务起了但端口不通，先用 `ss -tlnp | grep 8080` 确认端口是否真的在监听，以及是不是被别的进程占了。

43. 怎么排查 TIME_WAIT 过多的问题？
答：
```bash
# 查看各状态连接的统计汇总
ss -s
```
输出包含 `TCP: X (estab Y, closed Z, timewait W)`。如果 `timewait` 数量上万，说明短连接过多，可能是 HTTP 客户端没用连接池，或者服务间调用频繁创建新连接。

44. 怎么测试 HTTP 连通性和抓包？
答：
```bash
# 测试 HTTP 接口是否可达（-v 显示详细请求/响应头）
curl -v http://localhost:8080/actuator/health

# 抓取某端口的网络包（排查请求是否到达、响应内容等）
tcpdump -i eth0 port 8080 -nn -A
# -i: 网卡  -nn: 不解析主机名和端口名  -A: 以 ASCII 显示包内容
```
`tcpdump` 在排查"请求到底有没有到达服务器"时非常有用，比如客户端说超时了，先抓包看请求是否到了、服务器是否回了响应。

### 磁盘排查

45. 磁盘满了怎么快速定位？
答：两步走——先看哪个分区满了，再找大目录。

```bash
# 第一步：查看各分区使用率
df -h
# 重点看 Use% 列，哪个到了 90%+ 就重点排查

# 第二步：在满的分区下找大目录
du -sh * | sort -rh | head
# 层层进入大目录继续 du，直到找到最大的文件或日志目录
```

46. 为什么磁盘明明删了文件但空间没释放？
答：因为文件被删除后，如果还有进程持有该文件的文件描述符，磁盘空间不会真正释放。用以下命令找出来：
```bash
# 列出已删除但仍被进程占用的文件
lsof +L1
```
输出中会显示 `(deleted)` 标记和持有该文件的进程 PID。解决办法：重启该进程，或者让进程关闭对应的文件句柄。这是一个非常常见的"磁盘已满但 du 算出来的总量对不上"的根因。

### 文件描述符

47. 怎么排查文件描述符（fd）耗尽的问题？
答：文件描述符耗尽时，服务会报 `Too many open files`，所有新建连接和文件操作都会失败。排查方法：

```bash
# 查看某进程当前打开了多少 fd
lsof -p <pid> | wc -l

# 查看该进程的 fd 上限
cat /proc/<pid>/limits
# 找 "Max open files" 那一行，Soft Limit 是实际生效的限制

# 查看/设置当前 shell 的 fd 限制
ulimit -n        # 查看
ulimit -n 65535  # 临时设置（仅当前 shell 生效）
```

典型场景：一个 Java 服务打开了大量数据库连接和 HTTP 连接，`lsof -p <pid> | wc -l` 显示 fd 数接近 `Max open files` 的限制值，说明连接池配置过大或存在连接泄漏。短期止血方案是调大限制，根本修复要查连接池和关闭逻辑。

## 十六、Docker 底层原理

48. Docker 和虚拟机有什么本质区别？
答：Docker 不是虚拟机，它是进程级隔离。虚拟机通过 Hypervisor 模拟完整的硬件和 Guest OS，每个 VM 有自己的内核；Docker 容器直接运行在宿主机内核上，通过内核特性（Namespace + Cgroups）实现隔离和限制。所以 Docker 启动快（毫秒级 vs 分钟级）、开销小（没有 Guest OS）、密度高（同样的机器能跑更多实例）。

49. Docker 依赖的三大核心内核技术是什么？
答：Namespace（隔离视图）、Cgroups（资源限制）、Union FS（分层存储）。这三者组合在一起，让一个普通 Linux 进程"看起来"像拥有独立系统，同时又不会无限制地消耗宿主机资源。

### Namespace（隔离）

50. Namespace 是怎么让容器内进程"以为自己是独立系统"的？
答：Linux Namespace 通过对内核资源进行分区隔离，让每个容器只能看到自己那部分。Docker 使用了以下六种 Namespace：

| Namespace | 隔离内容 | 效果 |
|-----------|---------|------|
| **PID Namespace** | 进程 ID | 容器内有自己的 PID 编号空间，entrypoint 进程在容器内是 PID 1，但在宿主机上是一个普通 PID（如 12345） |
| **Network Namespace** | 网络栈 | 每个容器有独立的网卡、IP 地址、端口空间、路由表。所以多个容器都可以监听 80 端口而不冲突 |
| **Mount Namespace** | 文件系统挂载 | 每个容器有独立的文件系统视图，容器内看到的 `/` 是镜像的根文件系统，不是宿主机的 |
| **UTS Namespace** | 主机名 | 容器可以有自己的 hostname，`hostname` 命令看到的是容器名而非宿主机名 |
| **IPC Namespace** | 进程间通信 | 信号量、消息队列、共享内存等 IPC 资源隔离，不同容器间无法直接 IPC 通信 |
| **User Namespace** | 用户 ID | 容器内的 root (UID 0) 可以映射到宿主机上的非特权用户，增强安全性 |

面试关键点：PID Namespace 是最容易被问到的——容器内 PID 1 如果退出，整个容器就停止了，这就是为什么 entrypoint 不能用后台进程（daemon）启动。

### Cgroups（资源限制）

51. Cgroups 是怎么限制容器资源使用的？
答：Cgroups（Control Groups）是 Linux 内核提供的资源限制机制，Docker 用它来限制容器的 CPU、内存等资源。核心参数：

**CPU 限制：**
- `cpu.cfs_quota_us` / `cpu.cfs_period_us`：CFS 调度器的配额控制
  - `period` 默认 100000（100ms），`quota` 设置为 50000 表示每 100ms 最多使用 50ms CPU，即限制 50% CPU
  - `docker run --cpus=0.5` 等价于 quota=50000, period=100000
  - `docker run --cpus=2` 表示最多使用 2 个核心

**内存限制：**
- `memory.limit_in_bytes`：内存硬上限，容器内存使用超过此值则触发 OOM Kill
- `docker run -m 512m` 表示限制容器最多使用 512MB 内存
- 超限后内核直接杀掉容器内内存占用最大的进程，`docker inspect` 中会看到 `OOMKilled: true`

52. 为什么 Java 进程在容器里特别容易 OOM？
答：这是一个高频面试题。JDK 8u191 之前的版本不识别 Cgroups 内存限制，JVM 会读取宿主机的总内存来计算默认堆大小。比如宿主机 64GB 内存，容器限制 512MB，JVM 默认堆可能算出 16GB，实际只能用 512MB，很快就被 OOM Kill。

解决方案：
- 使用 JDK 8u191+ 或 JDK 11+，这些版本支持 `-XX:+UseContainerSupport`（默认开启），能正确读取容器内存限制
- 显式设置堆大小：`-Xmx384m -Xms384m`（容器 512MB 时，给 JVM 堆留 75% 左右，其余给 Metaspace、线程栈、直接内存等非堆开销）
- 或者使用比例设置：`-XX:MaxRAMPercentage=75.0`

### Union FS（分层存储）

53. Docker 镜像的分层存储是怎么工作的？
答：Docker 镜像由多个只读层（layer）叠加组成，每一条 Dockerfile 指令（如 `RUN`、`COPY`）产生一层。容器运行时在最上面添加一个薄的可写层（thin writable layer），所有写操作都发生在这一层（Copy-on-Write 策略）。

当前主流的存储驱动是 **OverlayFS**，它的工作原理：
- `lowerdir`：镜像的只读层（可以多层叠加）
- `upperdir`：容器的可写层
- `merged`：统一视图，容器内进程看到的文件系统就是 merged 目录
- `workdir`：OverlayFS 内部使用的临时目录

分层的好处：多个容器共享相同的基础镜像层，只有可写层是各自独立的，极大节省磁盘空间。比如 10 个容器用同一个 JRE 基础镜像，JRE 那一层在磁盘上只存一份。

54. 为什么 Docker 比虚拟机启动快得多？
答：因为没有 Hypervisor 和 Guest OS 的开销。虚拟机启动要引导一个完整的操作系统内核（通常需要几十秒到几分钟），Docker 容器本质上只是在宿主机上启动一个受 Namespace/Cgroups 约束的进程，启动时间和直接运行这个进程差不多（毫秒到秒级）。共享宿主机内核是 Docker 轻量和快速的根本原因。

## 十七、Dockerfile 最佳实践

55. 什么是多阶段构建（Multi-stage Build）？为什么推荐用？
答：多阶段构建允许在一个 Dockerfile 中使用多个 `FROM`，每个 `FROM` 开始一个新阶段。最终镜像只包含最后一个阶段的内容，构建工具和中间产物不会进入生产镜像。

```dockerfile
# Build 阶段：包含 Maven、JDK 等构建工具
FROM maven:3.9-eclipse-temurin-17 AS builder
COPY . /app
WORKDIR /app
RUN mvn clean package -DskipTests

# Runtime 阶段：只包含 JRE 和编译产物
FROM eclipse-temurin:17-jre-alpine
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

优势：构建工具（Maven、JDK full）、源代码、中间文件都不会进入最终镜像。一个用 Maven 构建的 Java 项目，构建阶段镜像可能 800MB+，但最终 Runtime 镜像只有 100-150MB。镜像更小意味着拉取更快、攻击面更小、存储成本更低。

56. 怎么利用 Docker 层缓存加速构建？
答：Docker 构建时每一层都会被缓存，只有当某一层的输入发生变化时，该层及其之后的所有层才会重新构建。利用这个特性优化构建速度的关键是：**把变化频率低的操作放前面，变化频率高的放后面**。

Java 项目的典型优化：
```dockerfile
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /app

# 先只 COPY 依赖描述文件并下载依赖（这一层很少变化，会被缓存）
COPY pom.xml .
RUN mvn dependency:go-offline

# 再 COPY 源代码并编译（只有代码变化时才重新执行）
COPY src ./src
RUN mvn clean package -DskipTests
```

未优化前：每次代码改动都要重新下载所有 Maven 依赖（几分钟）。优化后：只有 `pom.xml` 变化才重新下载依赖，日常代码改动只需重新编译（几十秒）。

57. 基础镜像怎么选？
答：按体积从小到大：
- **alpine**（~5MB）：最小，基于 musl libc，适合对体积极度敏感的场景。但 musl libc 和 glibc 有兼容性差异，某些 Java native 库（如 gRPC、Netty 的 native transport）可能出问题
- **slim / debian-slim**（~70MB）：基于 glibc，兼容性好，是大多数 Java 服务的推荐选择
- **完整版**（~300MB+）：包含完整工具链，适合需要在容器内调试的场景，生产环境一般不用

推荐策略：Java 服务优先选 `eclipse-temurin:17-jre-alpine`（如果兼容）或 `eclipse-temurin:17-jre-jammy`（slim Debian），不要用包含完整 JDK 的镜像跑生产。

58. `.dockerignore` 为什么重要？
答：Docker 构建时会把构建上下文（通常是项目根目录）打包发送给 Docker daemon。如果不配置 `.dockerignore`，`.git`（可能几百 MB）、`target`、`node_modules` 等无用目录都会被发送，严重拖慢构建速度。

```
# .dockerignore
.git
target/
node_modules/
*.md
.idea/
.vscode/
```

59. 为什么要用非 root 用户运行容器？
答：容器默认以 root 运行，如果容器被攻破，攻击者就拥有 root 权限。虽然有 Namespace 隔离，但配合内核漏洞仍可能逃逸到宿主机。最佳实践：

```dockerfile
RUN adduser --system --no-create-home appuser
USER appuser
```

这样容器内进程以非特权用户运行，即使被攻破也大幅降低了逃逸风险。

60. 怎么在 Dockerfile 中配置健康检查？
答：
```dockerfile
HEALTHCHECK --interval=30s --timeout=3s --start-period=60s --retries=3 \
  CMD curl -f http://localhost:8080/actuator/health || exit 1
```
- `--interval=30s`：每 30 秒检查一次
- `--timeout=3s`：检查超时时间
- `--start-period=60s`：容器启动后的宽限期（Java 服务启动慢，给足时间）
- `--retries=3`：连续 3 次失败才标记为 unhealthy

健康检查失败后 Docker 会标记容器为 `unhealthy`，编排系统（如 Docker Swarm、Kubernetes）会据此做重启或摘流量处理。

61. 优雅停机在 Docker 中怎么实现？
答：Docker 停止容器时先发 `STOPSIGNAL`（默认 `SIGTERM`），等待 `stop_timeout`（默认 10 秒），超时后发 `SIGKILL` 强杀。

关键点：
- Dockerfile 中确保使用 exec 格式的 ENTRYPOINT（`["java", "-jar", "app.jar"]`），而不是 shell 格式（`java -jar app.jar`）。shell 格式下 PID 1 是 `/bin/sh`，`SIGTERM` 发给 sh 而不是 Java 进程，Java 进程收不到信号就无法优雅停机
- Spring Boot 配置：`server.shutdown=graceful` + `spring.lifecycle.timeout-per-shutdown-phase=30s`
- docker-compose 中可设置 `stop_grace_period: 30s` 给应用足够的停机时间

```dockerfile
STOPSIGNAL SIGTERM
ENTRYPOINT ["java", "-jar", "app.jar"]
```

## 十八、Kubernetes 核心概念速查

62. Pod 是什么？为什么最小调度单位不是 Container？
答：Pod 是 Kubernetes 中最小的调度和部署单位，一个 Pod 包含一个或多个容器，它们共享网络（同一个 IP 和端口空间）和存储卷。

之所以不直接调度 Container，是因为很多场景需要多个紧密耦合的进程协同工作。Pod 支持 **sidecar 模式**：主容器跑业务，sidecar 容器负责日志采集、服务网格代理（如 Envoy）、配置热更新等。这些容器必须在同一台机器上、共享网络和文件，Pod 提供了这个共享环境。

实际使用中大多数 Pod 只包含一个业务容器，但理解 sidecar 模式是面试加分项。

63. Deployment 有什么用？
答：Deployment 管理 Pod 的生命周期，核心能力包括：
- **副本管理**：`replicas: 3` 表示始终保持 3 个 Pod 运行，Pod 挂了会自动重建
- **滚动更新**：更新镜像时逐步替换旧 Pod，不会一次性全部停掉，保证服务不中断
- **回滚**：`kubectl rollout undo deployment/<name>` 一键回到上一个版本

Deployment 是后端工程师最常打交道的 K8s 资源，基本上每个微服务对应一个 Deployment。

64. Service 解决什么问题？
答：Pod 的 IP 是不固定的（重建后会变），Service 为一组 Pod 提供稳定的网络访问入口（固定 DNS 名 + 虚拟 IP）。Service 通过 label selector 匹配后端 Pod，自动做负载均衡。

三种常用类型：
- **ClusterIP**（默认）：只在集群内部可访问，服务间调用用这个
- **NodePort**：在每个节点的固定端口暴露服务（范围 30000-32767），集群外可通过 `节点IP:NodePort` 访问
- **LoadBalancer**：在云环境中自动创建外部负载均衡器（如 AWS ALB、阿里云 SLB），生产环境对外暴露服务的标准方式

65. ConfigMap 和 Secret 分别怎么用？
答：
- **ConfigMap**：存储非敏感配置（数据库地址、功能开关、日志级别等），可以挂载为文件或注入为环境变量
- **Secret**：存储敏感信息（密码、证书、API Key），Base64 编码（不是加密！），可以配合外部密钥管理（如 Vault）使用

两者都实现了配置和代码解耦，不同环境使用不同的 ConfigMap/Secret，镜像不需要改变。

66. K8s 的三种探针分别是什么？
答：

| 探针 | 作用 | 失败后果 | 典型配置 |
|------|------|---------|---------|
| **Liveness Probe** | 检查容器是否还活着 | 重启容器 | HTTP GET /actuator/health/liveness |
| **Readiness Probe** | 检查容器是否准备好接收流量 | 从 Service 摘除（不重启） | HTTP GET /actuator/health/readiness |
| **Startup Probe** | 容器启动阶段的保护 | 启动期间不执行 liveness 检查 | failureThreshold: 30, periodSeconds: 10（给 5 分钟启动时间） |

为什么需要 Startup Probe：Java 服务启动可能需要 30 秒到几分钟（加载 Spring Context、预热缓存等），如果没有 Startup Probe，liveness 检查会在启动期间误判服务不健康并反复重启，形成"启动→被杀→重启→被杀"的死循环。

67. K8s 资源限制怎么配？requests 和 limits 的区别？
答：
- **requests**：调度保证。Pod 被调度到节点时，节点必须有足够的 requests 资源。可以理解为"我至少需要这么多"
- **limits**：硬上限。容器实际使用超过 limits 时，CPU 会被节流（throttle），内存超过则被 OOM Kill

```yaml
resources:
  requests:
    cpu: 500m      # 0.5 核（1000m = 1 core）
    memory: 512Mi  # 512 MiB
  limits:
    cpu: 1000m     # 1 核
    memory: 1Gi    # 1 GiB
```

最佳实践：requests 设为正常负载下的资源用量，limits 设为峰值用量的上限。CPU limits 是否设置存在争议——有些团队只设 requests 不设 limits（避免 throttle 导致延迟抖动），内存 limits 则必须设。

68. HPA（Horizontal Pod Autoscaler）是什么？
答：HPA 根据实时指标自动扩缩 Pod 数量，实现弹性伸缩。

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-service
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```

含义：当所有 Pod 的平均 CPU 利用率超过 70%，HPA 自动增加 Pod 数量（最多到 10 个）；低于 70% 则缩容（最少保持 2 个）。除了 CPU/内存，还支持自定义指标（如 QPS、队列长度），但需要配合 Prometheus Adapter 等组件。

后端工程师需要理解 HPA 的前提是服务必须无状态——状态外置到 Redis/DB，否则扩缩容时会丢状态。
