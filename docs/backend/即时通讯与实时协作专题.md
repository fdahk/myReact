# 即时通讯与实时协作专题

## 使用方式

这篇补的是一类很能体现后端实时系统能力的高频方向：

- 聊天系统、在线状态、已读回执、未读数到底怎么设计
- 实时协作为什么不只是开一个 `WebSocket`
- 顺序、投递、重连、离线消息、协作冲突这些问题该怎么讲

这类题不一定每次都考，但一旦考到，非常能拉开差距。

---

## 一、为什么 IM 和实时协作值得单独讲

1. 为什么即时通讯不是简单的“发消息接口”？
答：因为真正的聊天或实时协作系统要处理连接管理、消息投递、离线补偿、顺序语义、已读回执、未读数、在线状态、消息存储、重连同步和多端一致性。这远比一个 HTTP 接口复杂。

2. 为什么实时协作比普通消息推送更难？
答：因为它不只是把事件发出去，还要维护多人同时在线的共享状态。例如谁在线、谁正在编辑、变更顺序怎么收敛、冲突怎么解决、断线重连后怎么补齐上下文。这些都要求更强的时序和状态管理能力。

---

## 二、IM 系统最核心的 6 个问题

3. 第一个问题是什么？
答：连接管理。连接是长生命周期资源，后端要管理连接建立、鉴权、心跳、断开、重连和节点迁移，不是“收到了就回消息”这么简单。

4. 第二个问题是什么？
答：消息投递。消息是点对点、群聊、广播还是系统通知，不同类型对扇出、存储和顺序要求不同。成熟系统通常会把“消息入库”和“实时分发”拆开设计。

5. 第三个问题是什么？
答：顺序语义。很多聊天场景至少要求单会话内相对有序，但跨会话全局有序通常没必要。实时系统最怕口头说“保证顺序”，但说不清到底保证哪一层顺序。

6. 第四个问题是什么？
答：离线消息和重连补偿。用户掉线、切网、切设备很常见，系统必须考虑离线期间消息怎么补、从哪个游标开始拉、是否支持多端同步、重连后如何补缺口。

7. 第五个问题是什么？
答：已读、未读和回执。用户界面很依赖这些状态，但后端实现并不简单。你要定义已送达、已接收、已读、已撤回各自含义，以及它们对未读数统计的影响。

8. 第六个问题是什么？
答：在线状态。在线不等于“这个用户只有一条连接”，真实系统往往要考虑多端在线、设备维度状态、短时断连和 presence 过期。

---

## 三、聊天系统主链路怎么讲

9. 一个成熟的聊天消息主链路怎么描述？
答：客户端建立实时连接并完成鉴权；发送消息时先做权限和会话校验；消息写入存储并生成会话内序号；再异步或同步地向在线接收端推送；离线用户则记录待同步游标；客户端回执再反向更新送达/已读状态。这样就把“存储正确性”和“实时体验”拆开了。

10. 为什么消息系统通常要先落库再推送？
答：因为实时连接是不可靠的，用户可能断线、实例可能切换、网络可能抖动。先保证消息事实被持久化，再追求在线实时分发，整体会更稳。

11. 为什么聊天系统很适合用 MQ 或事件驱动？
答：因为消息发送之后常常还有很多副作用，例如未读数更新、会话列表更新、Push 通知、风控检查、审计日志、搜索索引更新。把这些副作用异步化，比全部压在发送主链路里更可控。

---

## 四、在线状态和多端同步怎么讲

12. 为什么在线状态很容易答浅？
答：因为很多人只想到“连接在就是在线”。真实系统里，用户可能手机和 Web 同时在线，也可能网络抖动造成短暂断线。成熟系统会区分用户级在线、设备级在线、会话级在线，以及状态保活时间。

13. 多端同步最容易出什么问题？
答：最常见是消息重复展示、已读状态覆盖、会话列表顺序错乱和撤回/编辑状态不同步。多端同步不是“多发几次消息”，而是状态收敛问题。

---

## 五、实时协作和聊天的区别

14. 为什么实时协作比聊天更难？
答：因为聊天通常是离散消息流，而协作往往是共享文档、共享画布、共享状态的连续变更流。它更关注操作顺序、冲突合并、版本演进和局部广播，而不是单条消息送达。

15. 实时协作最值得讲的几个关键点是什么？
答：连接与房间模型、操作日志、版本号、重放、冲突收敛、在线用户列表、存在感状态、断线恢复和最终快照。即使不深入算法，也要知道它比普通通知推送复杂得多。

---

## 六、和通知系统、WebSocket、容灾怎么串起来

16. 为什么 IM 系统和通知系统要分开看？
答：因为通知系统更偏事件触达和多渠道发送，IM 更偏实时会话和状态同步。两者会共享一些基础能力，但目标完全不同，不能简单合并成一个“消息模块”。

17. 为什么实时系统也要考虑容灾和多机房？
答：因为连接型系统对节点切换、路由稳定性和状态恢复很敏感。机房切换后如果连接全部丢失、重连风暴失控、游标恢复不正确，用户体验会非常差。

## 七、和主项目怎么绑定

18. `agent-server-java` 怎么自然映射到实时协作能力？
答：最自然的是任务进度实时同步、多人协同查看任务状态、工作空间在线状态、结果流式输出、协作者 presence、批注/评论流。这些能力都能借鉴 IM 和实时协作系统的设计思路。

19. 这个主题在项目里最值得强调哪些点？
答：我会优先强调：连接管理、任务进度实时推送、会话/房间模型、多端同步、断线重连与历史补齐。这样项目会更像具备实时能力的平台，而不是普通后台服务。

## 九、这一篇学到位的标准

学完这篇，你至少要做到下面三件事：

1. 能把 IM 系统讲成连接、投递、顺序、离线补偿、在线状态的一整套问题。
2. 能区分聊天、通知和实时协作三类系统的边界。
3. 能把实时系统的方法论映射到自己的主项目和系统设计题里。

### WebSocket 连接管理与消息协议

1. WebSocket 握手大致怎么走？客户端和服务端各发什么？
答：握手基于 HTTP 的协议升级。客户端发起带 `Upgrade: websocket`、`Connection: Upgrade` 的请求，并携带 `Sec-WebSocket-Key`（随机 Base64）。服务端校验后返回 `101 Switching Protocols`，响应头里带 `Sec-WebSocket-Accept`，其值为 `Base64(SHA1(Sec-WebSocket-Key + "258EAFA5-E914-47DA-95CA-C5AB0DC85B11"))`（即 Key 与固定 GUID 拼接后做 SHA-1，再 Base64）。此后同一 TCP 连接上按 WebSocket 帧格式收发二进制/文本，不再按普通 HTTP 请求-响应。

2. 长连接管理里，心跳保活一般怎么做？
答：常见做法是客户端按固定间隔（例如每 30 秒）发送 Ping 控制帧，服务端回 Pong；或应用层自定义 ping/pong 消息。若在约定时间内未收到对端 Pong（或未收到任何可读数据），可判定半开或断线，触发关闭、清理路由与重连策略。心跳间隔要平衡省电、带宽与及时发现死连接。

3. 连接路由表要解决什么问题？单机与多实例分别怎么放？
答：需要把「谁在线、连在哪条连接上」记下来，才能推送消息。单机可在进程内用 `ConcurrentHashMap` 等结构维护 `userId → 连接`（或 `channelId`/`sessionId` 与连接的映射）。多实例时，推送目标若不在本机，就要能查到用户落在哪台机器：常把 `userId → serverId`（或连接元数据）存 Redis 等共享存储，本机只持有实际 `Channel`，上线/下线时更新路由。

4. 多端登录时连接模型要注意什么？
答：同一用户可能在 Web、App、桌面等多设备同时在线，不能假设「一个用户一条连接」。路由里通常要按设备维度区分（如 `userId + deviceType/deviceId`），推送时可按策略选择「所有端」「仅当前活跃端」或按业务类型扇出，未读、已读也要考虑多端一致性。

5. 消息投递怎样做「三次握手」式可靠投递？客户端要配合什么？
答：一种常见模式是：服务端推送消息时带上服务端生成的 `server_msg_id`；客户端收到后回 ACK（带上同一 `server_msg_id`）；服务端收到 ACK 后将该消息标记为已投递。若在超时或重试窗口内未收到 ACK，服务端可重推；客户端必须按 `server_msg_id`（或会话内序号）去重，避免重复展示。这与 TCP 三次握手思想类似，但是应用层的「送达确认」语义。

6. 离线消息一般怎么存、怎么拉？
答：用户不在线时，消息写入离线表或收件箱（可按用户、按会话分区），而不是只依赖内存推送。用户上线后按会话维度、带时间戳或消息 ID 游标分页拉取，避免一次拉全量。同时要配置保留时间（过期清理）和容量/条数上限（防止邮箱无限膨胀），必要时与已读游标、同步协议一起设计。

7. 写扩散和读扩散各是什么？实际系统为什么常混合？
答：写扩散是每个接收者存一份消息副本，读简单、推模型直接，但群成员多时写放大严重。读扩散是消息只存一份（或按会话一份），读时按成员列表扩散，适合大群，但读路径和权限校验更复杂。实践中常见混合：私聊或小群偏写扩散，大群或超大群偏读扩散，在中间按成员规模或会话类型切换策略。

8. 消息 ID 为什么要「全局唯一且相对有序」？双 ID 怎么用？
答：唯一用于去重、幂等与跨端对齐；有序（至少会话内单调）用于排序、拉取游标和保证聊天展示顺序。常用 Snowflake、数据库序列、或分段号 + 自增等趋势递增 ID。客户端可先分配临时 ID（client_msg_id）做乐观 UI，服务端落库后返回正式 ID（server_msg_id），客户端再替换、合并重复，兼顾体验与一致性。

9. 多实例下，用户 A 在 Server1、用户 B 在 Server2，A 发给 B 的路径怎么描述？
答：A 的连接到 Server1，消息进入 Server1 后，Server1 查共享路由（如 Redis：`userId B → serverId=Server2`），若目标不在本机，则通过消息队列、RPC 或网关内转发把「投递任务」送到 Server2；Server2 根据本地连接表找到 B 的 WebSocket，再推送给 B。核心是「路由查询 + 跨节点投递」与本地 `Channel` 管理解耦。
