# DDD 领域驱动设计专题

## 使用方式

DDD 是后端架构师面试中越来越高频的一块。它不是学术理论，而是在大型系统中解决「业务复杂度如何不吃掉技术团队」的架构方法论。

很多人会背「聚合根」「限界上下文」，但一追问就暴露没有真正理解。这篇的目标是让你能讲清楚：

- 为什么 DDD 在微服务时代更重要了
- 战略设计和战术设计分别解决什么
- 贫血模型和充血模型的本质区别
- CQRS 和 Event Sourcing 什么时候该用

---

## 一、为什么架构师要懂 DDD

1. DDD 到底在解决什么问题？
答：DDD 解决的核心问题是：当业务足够复杂时，如何让代码结构忠实反映业务逻辑，而不是让技术框架和数据库结构主导系统设计。传统三层架构(Controller-Service-DAO)在业务简单时没问题，但当业务规则越来越多，Service层会变成一个无结构的巨大"事务脚本"，所有业务逻辑堆在一起，改一处动全身。DDD通过领域模型把业务逻辑组织成有边界、有规则、可测试的结构。

2. 为什么微服务时代 DDD 更重要了？
答：因为微服务拆分的最大难题就是「怎么拆」。拆错了比不拆更痛苦(跨服务事务、循环依赖、数据不一致)。DDD的限界上下文(Bounded Context)恰好提供了一个按业务边界拆分服务的方法论——每个限界上下文对应一个微服务，有自己的领域模型和数据库，通过事件或API通信。

## 二、战略设计：限界上下文与上下文映射

3. 什么是限界上下文(Bounded Context)？
答：限界上下文是 DDD 中最重要的概念。它划定了一个模型的适用边界——在这个边界内，每个业务术语有唯一明确的含义。例如「用户」在认证上下文中代表登录凭证，在电商上下文中代表买家，在CRM上下文中代表客户线索。如果把这三个含义混在一个User类里，就会越来越臃肿。限界上下文的价值在于：让不同业务域的模型各自独立演进。

4. 如何划定限界上下文的边界？
答：核心依据是「业务语言的一致性」，而不是技术实现方便。具体方法：
- 列出核心业务概念，看哪些概念在不同场景下含义不同→拆开
- 看团队组织结构(Conway定律)：一个团队负责一个上下文
- 看变更频率：变更节奏差异大的应该拆开
- 看数据一致性边界：强一致的放一个上下文，最终一致的可以拆
- 反模式：不要按技术层拆(一个上下文负责缓存、一个负责数据库)

5. 上下文映射(Context Map)的常见模式有哪些？
答：
- 合作关系(Partnership)：两个上下文团队紧密协作
- 共享内核(Shared Kernel)：共享一小部分模型代码
- 客户-供应商(Customer-Supplier)：上游供应，下游消费，上游优先考虑下游需求
- 遵从者(Conformist)：下游被动接受上游模型，没有话语权
- 防腐层(Anti-Corruption Layer, ACL)：最常用。下游通过适配层隔离上游模型的变化，避免上游模型入侵自己的领域
- 开放主机服务(Open Host Service)：上游提供标准化API，下游按协议接入

## 三、战术设计：实体、值对象、聚合

6. 实体(Entity)和值对象(Value Object)的区别是什么？
答：
- 实体有唯一标识(ID)，身份不随属性变化。例如User(ID=1001)即使改名了还是同一个User
- 值对象没有独立标识，由属性值完全定义。例如Money(100, CNY)——两个金额相同的Money是「相等的」，不需要区分谁是谁
- 值对象应该是不可变的(immutable)。例如Address改了地址就应该创建新对象，而不是修改旧对象
- 设计原则：能用值对象就用值对象，因为不可变带来线程安全和简洁性

7. 什么是聚合(Aggregate)和聚合根(Aggregate Root)？
答：聚合是一组关联对象的集合，有一个聚合根作为唯一入口。外部只能通过聚合根访问聚合内的对象，不能直接引用聚合内部的其他实体。
例如：Order(聚合根) → OrderItem(聚合内实体) → Address(值对象)。外部不能直接修改OrderItem，必须通过Order的方法(order.addItem())。
聚合的边界原则：
- 事务一致性边界：聚合内保证强一致(一个事务只修改一个聚合)
- 尽量小：大聚合会导致并发冲突和性能问题
- 聚合之间通过ID引用，不通过对象引用

8. 为什么聚合边界要尽量小？
答：因为一个事务只能修改一个聚合(这是DDD的关键约束)。如果聚合太大，修改任何一个子对象都要锁定整个聚合，并发冲突会非常严重。例如把所有OrderItem、Payment、Logistics都放进Order聚合→任何操作都要锁整个订单→性能极差。正确做法是Order、Payment、Logistics作为独立聚合，通过领域事件协调。

## 四、领域服务、应用服务与领域事件

9. 领域服务(Domain Service)和应用服务(Application Service)的区别是什么？
答：
- 领域服务：封装不属于任何一个实体但属于领域逻辑的操作。例如「转账」涉及两个Account实体，不应该放在任何一个Account里，而是放在TransferService领域服务中。领域服务包含业务规则
- 应用服务：编排用例流程，不包含业务规则。例如CreateOrderApplicationService负责：接收请求→调用领域对象执行业务逻辑→持久化→发布事件→返回结果。它是薄的一层，只做协调
- 判断标准：如果逻辑抽走后领域模型变得不完整，它属于领域层；如果只是协调多个领域对象的流程，它属于应用层

10. 什么是领域事件(Domain Event)？
答：领域事件表示「领域中发生了某件重要的事」。例如OrderCreated、PaymentCompleted、InventoryDeducted。它有三个核心价值：
- 聚合间通信：聚合A修改后发布事件，聚合B订阅事件异步处理(解耦+最终一致性)
- 限界上下文间通信：订单上下文发布OrderPaid事件→物流上下文订阅创建发货单
- 审计和溯源：事件天然记录了「什么时候发生了什么」

## 五、贫血模型 vs 充血模型

11. 贫血模型的问题是什么？
答：贫血模型指的是实体类只有getter/setter没有业务逻辑，所有逻辑都在Service层。例如Order只有字段，所有下单、取消、支付逻辑全在OrderService里。问题是：
- Service变成了上帝类，所有逻辑堆在一起
- 实体退化为数据容器(DTO)，丧失了面向对象设计的优势
- 业务规则散落在各处，不同开发者可能在不同Service里重复实现同一条规则
- 这实际上是「面向数据库编程」而不是「面向领域编程」

12. 充血模型怎么写？
答：把业务规则放进实体和值对象中。例如：
```java
public class Order {
    private OrderStatus status;
    private Money totalAmount;
    private List<OrderItem> items;
    
    public void cancel() {
        if (this.status != OrderStatus.PENDING) {
            throw new IllegalStateException("只有待支付订单可以取消");
        }
        this.status = OrderStatus.CANCELLED;
        // 发布领域事件
        registerEvent(new OrderCancelled(this.id));
    }
    
    public void addItem(Product product, int quantity) {
        // 业务规则：同一商品不能重复添加
        // 业务规则：总金额不能超过上限
        ...
    }
}
```
cancel()的状态校验规则在Order内部，而不是在Service里用if判断。这样任何人调用order.cancel()都会走同一条规则路径。

## 六、CQRS 与 Event Sourcing

13. CQRS(命令查询职责分离)是什么？
答：CQRS把系统分成Command端(写)和Query端(读)两个独立模型：
- Command端：处理创建、修改、删除操作，使用充血模型，通过聚合根保证业务规则
- Query端：处理查询操作，使用专门的读模型(可以是扁平化的宽表、ES索引、缓存)
- 两端通过事件或消息同步数据
适用场景：读写模型差异大、读比写多得多、需要为不同查询场景构建不同视图。
不适合：简单CRUD系统，增加了复杂度没有收益。

14. Event Sourcing(事件溯源)是什么？
答：不保存实体当前状态，只保存所有领域事件的有序序列。实体状态通过回放事件重建。
例如Account的余额不是直接存100元，而是存一系列事件：Deposited(200) → Withdrawn(50) → Withdrawn(50)，回放后余额=100。
优势：完整审计轨迹、可以回放到任意时间点的状态、天然支持CQRS(事件发布给读端构建查询视图)。
劣势：查询当前状态需要回放(可以用快照优化)、事件Schema演进困难、不适合简单场景。
适用场景：金融交易、审计、协作编辑、游戏状态。

## 七、DDD 分层架构

15. DDD 标准四层架构是什么？
答：
- 用户接口层(Interface/Controller)：接收请求，参数校验，DTO转换
- 应用层(Application)：用例编排，事务控制，调用领域层，发布事件。薄层，不含业务规则
- 领域层(Domain)：核心。实体、值对象、聚合根、领域服务、领域事件、仓储接口(Repository Interface)。不依赖任何外部框架
- 基础设施层(Infrastructure)：实现领域层定义的接口。数据库访问(Repository实现)、消息发送、外部服务调用、缓存

关键依赖原则：领域层不依赖基础设施层(依赖倒置)。领域层定义Repository接口，基础设施层实现它。

16. DDD分层和传统三层架构的关键区别是什么？
答：
- 传统三层：Controller → Service → DAO，业务逻辑在Service，实体是贫血的
- DDD四层：Controller → ApplicationService → Domain(Entity/AggregateRoot/DomainService) → Infrastructure
- 最大区别：DDD中业务规则在Domain层的实体和领域服务中，ApplicationService只做编排；传统三层的Service既做编排又做业务逻辑

## 八、DDD 落地常见问题

17. DDD在团队中最难落地的是什么？
答：
- 统一语言(Ubiquitous Language)难建立：业务和技术用不同术语，代码中的变量名和业务概念对不上
- 聚合边界画不好：要么太大(性能差)要么太小(事务碎片化)
- 团队成员习惯贫血模型，总想把逻辑写在Service里
- 过度设计：简单CRUD硬套DDD，增加了复杂度却没有收益

18. 什么场景适合DDD，什么场景不适合？
答：
- 适合：业务规则复杂、业务概念多、需要长期维护和演进的核心域(订单、支付、风控、审批)
- 不适合：简单CRUD、查询为主的报表、工具类服务、基础设施服务
- DDD的核心判断标准：如果业务逻辑简单到一个Service就能搞定，不需要DDD；如果Service已经膨胀到无法维护，就是DDD发挥价值的时候

## 十、这一篇学到位的标准

学完这篇，你至少要做到下面三件事：

1. 能从「业务复杂度治理」角度解释DDD的价值，而不是只背名词。

3. 能把贫血模型→充血模型的重构讲成具体的代码演进，而不是空谈理论。
