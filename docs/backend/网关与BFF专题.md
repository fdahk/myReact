# 网关与 BFF 专题

## 使用方式

这篇补的是边缘层后端非常高频的一块：

- API Gateway 到底在解决什么
- `BFF` 为什么不是“多写一层转发”
- 鉴权、限流、路由、协议转换、聚合、降级这些能力为什么会集中出现在边缘层

如果你想把系统讲得更有架构层次，这篇很值得补。

---

## 一、为什么网关和 BFF 值得单独讲

1. 为什么网关不是“转发请求的中间人”？
答：因为网关承担的是系统入口治理职责。它往往要处理认证鉴权、路由、限流、灰度、日志、审计、协议适配、跨域和基础安全问题。它不只是转发，而是在定义“请求进入系统之前要经过哪些统一规则”。

2. 为什么 `BFF` 和网关不能混为一谈？
答：网关更偏平台层治理，目标是统一入口；`BFF` 更偏业务和终端适配，目标是给不同客户端提供更合适的数据形状和聚合能力。两者都在边缘层，但关注点不同。

---

## 二、网关最核心的 6 个问题

3. 第一个问题是什么？
答：路由。请求该被转到哪个服务、哪个版本、哪个区域、哪个灰度池，这些都是网关最基础的职责。

4. 第二个问题是什么？
答：认证鉴权。很多系统会把令牌解析、基础权限校验、租户识别、请求签名校验放在网关前置处理，减少下游重复实现。

5. 第三个问题是什么？
答：流量治理。限流、熔断、降级、黑白名单、异常流量拦截、热点接口保护都很适合在入口层统一做第一道防线。

6. 第四个问题是什么？
答：可观测性。网关天然能看到全局请求流量、错误率、耗时分布、调用方维度和版本分布，是观察系统入口健康度的关键位置。

7. 第五个问题是什么？
答：协议与格式适配。例如 HTTP 到 gRPC 的边缘适配、Header 注入、统一错误格式、请求追踪 ID 透传等，都常发生在网关层。

8. 第六个问题是什么？
答：灰度和发布控制。很多团队会把版本路由、流量分桶、实验开关和回滚入口放在网关或边缘控制平面，因为这里最适合统一切流。

---

## 三、BFF 到底在解决什么

9. `BFF` 最本质的价值是什么？
答：让不同终端拿到更适合自己的接口，而不是所有前端都直接打微服务。移动端、Web 端、后台管理端常常需要不同字段组合、不同聚合粒度和不同性能优化，`BFF` 能把这些差异收敛在边缘层。

10. 为什么 `BFF` 不只是“多加一层服务”？
答：因为它如果只是机械转发，就只会增加一跳延迟。真正有价值的 `BFF` 应该承担终端聚合、协议适配、缓存、降级和终端差异化策略，而不是重复下游逻辑。

11. 什么场景特别适合 `BFF`？
答：一个页面要聚合多个服务数据、前端需要大量裁剪和拼装、终端差异明显、需要边缘缓存或弱实时聚合时，`BFF` 往往很有价值。否则直接让前端走网关调用后端服务也未必不行。

---

## 四、网关和 BFF 最容易踩的坑

12. 为什么很多网关最后会变成“超级单点”？
答：因为团队容易不断往网关里塞业务逻辑，最后导致入口层过重。一旦网关既做流量治理又做大量业务编排，故障半径会非常大，发布风险也会提高。

13. 为什么很多 `BFF` 会慢慢变成“小后端”？
答：因为它天然靠近业务聚合，很容易逐步下沉更多逻辑，最终形成一个既不是纯聚合层、也不是领域服务层的混合体。成熟团队会有意识约束 `BFF` 的边界，避免它侵入核心领域规则。

14. 为什么边缘层也必须注意幂等和缓存边界？
答：因为聚合请求、多次重试、弱网络下重复请求都可能在边缘层放大问题；而边缘缓存如果没有清晰失效策略，也容易返回过期或跨租户数据。

---

## 五、网关、BFF 和开放平台怎么串起来

15. 为什么开放平台通常离不开网关？
答：因为开放平台最需要统一接入能力，例如签名校验、配额限流、调用方识别、审计、灰度和流量观测。这些能力放在网关层更容易统一治理。

16. 为什么 `BFF` 和移动端/管理端关系更紧？
答：因为不同终端对字段、时延、聚合结果和交互节奏的需求不同。`BFF` 的价值就在于把“终端差异”收在一层，而不是把这种差异扩散到所有后端服务。

---

## 六、和主项目怎么绑定

17. `agent-server-java` 最自然的边缘层切入点是什么？
答：工作台聚合接口、任务详情聚合、任务进度实时接口、管理端与用户端数据裁剪、对外开放 API 的统一接入。这些都非常适合用网关和 `BFF` 思路来表达。

18. 这个主题在项目里最值得强调哪些点？
答：我会优先强调：统一鉴权入口、任务接口聚合、不同终端数据裁剪、边缘缓存、灰度路由和开放平台统一接入。这样项目会更像有架构分层的平台系统。

## 八、这一篇学到位的标准

学完这篇，你至少要做到下面三件事：

1. 能把网关讲成入口治理平台，而不是简单转发组件。
2. 能区分 `BFF` 的适用边界，知道它为什么有价值也为什么容易失控。
3. 能把边缘层思路映射回主项目、开放平台和终端聚合场景。

### 九、Spring Cloud Gateway 核心机制

1. Spring Cloud Gateway 里「路由」的核心模型是什么？请求大致怎么走？
答：一条 **Route（路由）** 由三部分组成：**Predicate（断言）** 决定这条路由是否匹配当前请求；**Filter（过滤器链）** 在匹配后对请求/响应做加工；**URI** 是转发目标（如 `http://` 或 `lb://service-name`）。请求进入网关后，按配置匹配到某条 Route，依次执行该路由上的 Filter，最后把请求转发到目标 URI。

2. 常用的 Predicate 有哪些典型场景？
答：常见包括：`Path`（如 `/api/v1/**`）、`Header`、`Method`、`Query`、`Host`；以及基于时间的 `After`/`Before`/`Between`（按时间窗口开放路由）；`Weight` 与权重组配合做灰度分流（同一前缀下按权重打到不同 URI）。多条 Predicate 通常是与关系，全部满足才命中该 Route。

3. GatewayFilter 和 GlobalFilter 有什么区别？各举几类用途。
答：**GatewayFilter** 挂在具体 Route 上，只对该路由生效，例如 `AddRequestHeader`、`StripPrefix`、`RewritePath`、Resilience4j **`CircuitBreaker`**、**`Retry`**、基于 Redis 的 **`RequestRateLimiter`** 等。**GlobalFilter** 对所有路由生效，适合做认证、访问日志、全局限流、链路追踪 ID 注入、CORS 等；实现 `GlobalFilter` 时可同时实现 **`Ordered`**，用 `getOrder()` 控制与内置 Filter 的相对顺序。

4. 基于 Redis + Lua 的限流在 Gateway 里怎么理解？要配哪些参数？
答：`RequestRateLimiter` 等实现常用 Redis 执行 Lua 脚本做原子加减，典型是令牌桶：配置 **`replenishRate`**（每秒向桶中补充的令牌数，即可持续 QPS）和 **`burstCapacity`**（桶容量，允许短时间突发）。**`KeyResolver`** 决定限流键，例如按客户端 IP、登录用户 ID、或「用户 + 接口」维度分别计数，实现不同粒度的隔离。

5. 怎么和 Nacos、Eureka 等注册中心集成做动态路由？
答：开启 **`spring.cloud.gateway.discovery.locator.enabled=true`** 后，网关可从注册中心拉取服务列表，并按约定自动生成路由（通常需配合 `lowerCaseServiceId` 等属性规范路径）。转发目标 URI 使用 **`lb://service-name`** 时，由 Spring Cloud LoadBalancer 从注册中心解析实例并做负载均衡，而不是写死 IP。

6. 自定义 GlobalFilter 做 JWT 认证时，典型步骤是什么？
答：实现 `GlobalFilter`（并实现 `Ordered` 以排在 Netty 路由之前或之后合适位置）：从请求 Header（如 `Authorization`）取出 Token；校验 JWT 签名与过期时间；合法则解析出用户标识，通过 `mutate()` 往下游请求头注入用户信息（如 `X-User-Id`），再 **`return chain.filter(exchange)`** 放行；校验失败则设置响应状态码 **401**、写入 JSON 错误体，并 **`return exchange.getResponse().setComplete()`**，不再进入下游。
