# 网络协议与服务通信

## 使用方式

后端面试里的网络题，不是为了考你背协议百科，而是看你能不能把一次请求完整讲清楚：从客户端、网关、服务、数据库到返回结果，中间哪里会慢、会丢、会重复、会超时、会不一致。

---

## 一、为什么后端必须懂网络

1. 为什么网络是后端核心基础？
答：因为后端服务本质上就是在网络中接收请求、调用下游、返回响应。你如果不理解协议、连接、超时、重试、负载均衡和链路时延，就很难真正设计好服务，也很难排查线上问题。

2. 顶级后端岗位看网络到底看什么？
答：不是看你能不能背完 TCP 三次握手，而是看你是否知道一次请求为什么会慢、为什么会重试、为什么会重复提交、为什么会连接打满、为什么流式链路会中断。也就是说，网络知识要能落到系统行为上。

---

## 二、HTTP 与 API 语义

3. HTTP 在后端里最核心的作用是什么？
答：它是应用层通信协议，定义了请求方法、状态码、头部、正文和连接语义。你不能把 HTTP 只理解成“接口怎么调”，它实际上决定了很多 API 契约和缓存、幂等、流式、鉴权设计。

4. 为什么 REST 不是“路径设计规范”这么简单？
答：REST 更本质的是围绕资源建模和语义表达。它要求你思考系统对外暴露的对象、操作边界和状态变化方式。成熟后端在设计 API 时，会先想资源模型，再想 URL 和方法。

5. 为什么幂等和 HTTP 方法常被一起问？
答：因为不同方法通常有不同语义预期，比如 `GET` 应该安全且幂等，`PUT` 通常是幂等更新，`POST` 更常见于创建和非幂等操作。虽然现实业务会比规范复杂，但你必须理解语义设计的重要性。

6. 状态码为什么不能乱用？
答：因为状态码是客户端、网关、监控和调用方理解结果的基础语义。乱用 `200` 包裹所有失败，或者把权限问题、参数问题、系统错误混成一类，会让排障和协作变得非常困难。

---

## 三、TCP、连接与时延

7. 为什么后端虽然写在 HTTP 层，但仍然要懂 TCP？
答：因为很多线上问题最终都落在连接层：建连慢、连接数打满、超时、粘包边界误解、长连接被代理断开。只懂 HTTP 表层，很难解释这些问题。

8. 三次握手和四次挥手为什么对后端仍有意义？
答：因为它们帮助你理解连接建立和释放成本，进一步理解短连接、长连接、连接复用、TIME_WAIT 等现实问题。面试时更成熟的回答不是把握手过程逐字背一遍，而是知道这些机制为什么影响系统性能和资源。

9. 为什么长连接在高频服务里很重要？
答：因为频繁建立和释放连接成本高，会增加时延和系统开销。数据库连接池、HTTP 连接池、RPC 连接复用本质上都在解决这个问题。

---

## 四、DNS、CDN、代理与负载均衡

10. 为什么后端也要理解 DNS？
答：因为服务发现、域名解析、外部依赖调用都和 DNS 有关。很多线上“偶发不可用”其实并不是业务代码错了，而是解析、缓存或网络边界出了问题。

11. 反向代理和负载均衡在后端体系里解决什么问题？
答：统一入口、流量分发、TLS 终止、限流、灰度、故障隔离和多实例扩展。后端不是只写服务本身，还要理解服务是怎样被接入整体流量体系的。

12. 为什么不能把网关理解成“多了一层转发”？
答：因为网关承担的往往是统一鉴权、路由、灰度、协议适配、限流、观测和安全边界。它是系统对外治理的重要节点。

---

## 五、超时、重试与幂等

13. 为什么超时是后端必须显式设计的能力？
答：因为分布式调用里任何下游都可能卡住，如果没有超时，请求会一直占用线程、连接和资源，最终拖垮上游。超时不是“兜底选项”，而是默认必须存在的边界控制。

14. 为什么重试既能救系统，也能打爆系统？
答：因为当失败是偶发网络抖动时，重试可能很有效；但如果下游已经过载，盲目重试只会放大流量，形成雪崩。成熟后端一定会把重试和超时、幂等、退避、熔断一起设计。

15. 为什么一谈重试就要谈幂等？
答：因为重试意味着同一个请求可能被执行多次，如果业务操作不是幂等的，就可能生成重复订单、重复扣款、重复发消息。后端不能把重试当成无副作用动作。

---

## 六、RPC、gRPC 与服务间通信

16. 为什么服务内部常用 RPC，而不是全走 HTTP REST？
答：因为内部服务通信更关注性能、契约、类型约束和统一治理。RPC 让调用更像函数调用，也更容易做服务治理、超时、重试和序列化规范。

17. 为什么后端要知道 gRPC？
答：因为它在内部服务通信、流式调用和强契约接口上很常见。你不一定现在就要把它学得很深，但至少要知道它体现的是“服务间通信模型升级”，不是简单换个协议名。

---

## 七、流式通信

18. 为什么流式接口和普通 HTTP 接口要区别对待？
答：因为普通接口更像一次请求拿完整结果，流式接口更像一段持续过程。它会涉及连接持续时间、代理超时、客户端消费、断线重连、中间状态和结束语义，这些都比普通请求复杂得多。

19. `SSE` 适合什么场景？
答：适合服务端单向持续推送，比如长任务进度、AI 文本流、通知流。它比轮询更实时，比双向 WebSocket 更轻，特别适合“服务端持续发、客户端只接收”的链路。

20. 这和当前 `agent-server` 有什么关系？
答：关系很直接。当前 `resource-organizer-agent` 里就通过 `SSE` 推送运行进度，这本身是个不错的题材亮点。但如果要做成顶级后端项目，还需要把连接治理、任务持久化、重连恢复和多实例一致性进一步补强。

---

## 八、网络题的成熟回答方式

21. 面试问“从浏览器输入 URL 到页面展示”，后端该怎么讲？
答：可以按 DNS、建连、TLS、网关、服务路由、业务处理、缓存/数据库、响应返回、浏览器消费这条链路讲。重点不在于把所有层都展开，而是要体现你知道每一层可能成为瓶颈或故障点。

22. 面试问“接口为什么会慢”，该怎么答？
答：按网络、网关、应用线程池、下游调用、数据库、锁等待、GC、序列化和返回体大小逐层拆。成熟的后端不会一上来就怪 SQL。

## 九、你现阶段最该掌握到什么程度

你至少要达到下面标准：

1. 能讲清 HTTP 语义、状态码和幂等关系。
2. 能理解连接、超时、重试、负载均衡和网关的作用。
3. 能说清普通请求与流式请求的差异。
4. 能把网络问题映射到真实后端链路和主项目设计里。

如果你现在网络知识仍停留在“前端怎么调接口”，那还远远不够支撑大厂后端岗位。

## 十、连接池、超时与重试的真实故障

23. 为什么一个下游慢，最终可能拖垮整个系统？
答：因为后端服务不是孤立的。一个下游调用变慢后，请求会在上游线程里等待，占住线程池、连接池和内存；如果上游没有超时和熔断，等待中的请求越来越多，最终主服务自己也会被拖慢甚至拖死。所以网络问题经常会放大成资源问题。

24. 为什么连接池打满经常和网络问题一起出现？
答：因为连接池本质上是在复用有限连接资源。如果下游超时、响应慢、连接泄漏或短时间请求暴涨，连接会被长时间占住，新请求就只能排队等待。很多“数据库/HTTP 调用超时”的根因，往往是连接池先被拖满了。

25. 超时应该按什么原则设计？
答：不能一个系统只配一个统一超时值。更成熟的做法是按链路和依赖特征分层：核心短链路超时要更小，外部慢依赖可以稍大，但必须有上限；同时还要考虑整体调用链预算，避免上游超时比下游总和还长。超时设计本质上是在管理等待边界。

26. 重试什么时候应该做，什么时候不该做？
答：偶发网络抖动、可恢复的临时错误、而且业务具备幂等语义时，重试比较合理；如果下游已经持续过载、错误是确定性失败、业务不幂等，盲目重试只会放大事故。成熟系统一定会给重试加上限、退避和熔断边界。

27. 为什么“超时 + 重试 + 熔断 + 幂等”总是一起出现？
答：因为它们是同一类分布式调用问题的不同侧面。超时控制等待边界，重试处理瞬时失败，熔断阻止持续打击坏依赖，幂等保证重复执行不把业务做坏。如果只做其中一项，整个链路通常仍然站不住。

## 十一、RPC 和服务间调用的成熟表达

28. 服务间调用为什么不能只理解成“换个协议调用接口”？
答：因为一旦服务之间跨网络，问题就从函数调用变成了序列化、连接复用、超时、重试、兼容性、负载均衡、观测和故障传播问题。RPC 的本质不是“代码更优雅”，而是把服务间调用标准化和可治理化。

29. `gRPC` 为什么经常和流式、内部服务一起出现？
答：因为它更强调强契约、二进制序列化、双向或单向流式能力，以及内部服务统一通信规范。并不是所有系统都必须用它，但一旦业务里出现高频服务通信、流式传输或多语言服务协作，它就很值得被纳入候选方案。

30. 为什么服务间调用也要考虑向后兼容？
答：因为微服务天然是分批发布、新旧版本共存的。提供方如果随意改字段、删字段、变更语义，下游就可能在灰度窗口里出问题。后端网络题真正高阶的地方，不只是“怎么调通”，还包括“怎么长期演进而不互相打挂”。

---

## 十二、流式链路与代理边界

31. 为什么流式链路比普通请求更容易在代理层出问题？
答：因为它持续时间长、连接不断开、中间层可能有读写超时、缓冲策略和连接数限制。很多 `SSE` 或流式响应的问题，不是服务端业务代码本身错了，而是代理、网关、超时配置和连接治理没有配套做好。

32. 流式接口为什么更需要清晰的“结束语义”？
答：因为客户端不能像普通接口那样简单等一个完整响应包。它必须知道什么时候是正常完成、什么时候是中断、什么时候是失败、什么时候可以重试。如果结束语义不清楚，用户看到的就只是“流突然停了”，系统也很难自动恢复。

33. 为什么 `SSE`、WebSocket、轮询不能只按“谁更高级”选？
答：因为它们解决的是不同问题。`SSE` 更适合服务端单向持续推送，WebSocket 更适合双向实时交互，轮询最简单但实时性和资源利用较差。选型本质上是业务通信模式匹配，不是技术名词排名。

## 十四、TCP 三次握手与四次挥手详解

34. 三次握手的每一步到底发生了什么？
答：
- **第一步 Client → Server**：Client 发送 SYN 报文（SYN 标志位=1），携带一个随机初始序列号 `seq=x`，不携带应用数据。发送后 Client 进入 `SYN_SENT` 状态。
- **第二步 Server → Client**：Server 收到 SYN 后，回复 SYN+ACK 报文（SYN=1, ACK=1），携带自己的随机初始序列号 `seq=y`，确认号 `ack=x+1`（表示已收到 Client 的 seq=x，期望下一个字节是 x+1）。Server 进入 `SYN_RCVD` 状态。
- **第三步 Client → Server**：Client 收到 SYN+ACK 后，发送 ACK 报文（ACK=1），`seq=x+1`，`ack=y+1`。这个包可以携带应用数据。发送后双方进入 `ESTABLISHED` 状态，连接建立完成。

35. 为什么是三次握手，两次行不行？
答：两次不行，原因有两个：
- **确认双向通信能力**：两次只能确认 Client→Server 方向和 Server 的接收能力，无法确认 Client 的接收能力。第三次握手就是让 Server 确认"Client 能收到我的响应"。
- **防止历史连接初始化**：如果网络中残留了一个旧的 SYN 包延迟到达 Server，两次握手下 Server 会直接建立连接并分配资源，但 Client 根本没有发起请求，这些资源就浪费了。三次握手允许 Client 在第三步拒绝这个过期连接。
- **SYN Flood 攻击利用的就是这一点**：攻击者大量发送 SYN 包但不完成第三步，Server 在 `SYN_RCVD` 状态为每个半连接分配资源，最终 SYN Queue 被填满，正常请求无法建连。

36. 四次挥手的每一步到底发生了什么？
答：
- **第一步 主动方 → 被动方**：主动关闭方发送 FIN 报文（FIN=1, `seq=u`），表示"我没有数据要发了"。主动方进入 `FIN_WAIT_1` 状态。
- **第二步 被动方 → 主动方**：被动方收到 FIN 后，回复 ACK 报文（ACK=1, `ack=u+1`）。被动方进入 `CLOSE_WAIT` 状态，主动方收到 ACK 后进入 `FIN_WAIT_2` 状态。此时被动方仍然可以继续发送数据（半关闭状态）。
- **第三步 被动方 → 主动方**：被动方数据也发完后，发送 FIN 报文（FIN=1, `seq=w`）。被动方进入 `LAST_ACK` 状态。
- **第四步 主动方 → 被动方**：主动方收到 FIN，回复 ACK 报文（ACK=1, `ack=w+1`）。主动方进入 `TIME_WAIT` 状态，等待 **2MSL**（Maximum Segment Lifetime，Linux 默认 MSL=60 秒，即 TIME_WAIT 持续 120 秒）后进入 `CLOSED` 状态。

37. 为什么挥手需要四次，不能像握手一样三次？
答：因为 TCP 是全双工的，两个方向的关闭是独立的。被动方收到 FIN 时，可能还有数据没发完，所以先回 ACK 确认收到关闭请求，等自己数据发完再发 FIN。这就导致 ACK 和 FIN 分成了两个包，从而需要四步。如果被动方恰好也没有数据要发，某些实现会将第二步的 ACK 和第三步的 FIN 合并，变成三次挥手。

38. TIME_WAIT 为什么必须存在？等 2MSL 的目的是什么？
答：两个原因：
- **确保最后的 ACK 到达被动方**：如果主动方发的最后一个 ACK 丢了，被动方会重发 FIN。主动方如果已经 CLOSED，就无法响应这个重传的 FIN，被动方会一直卡在 `LAST_ACK`。TIME_WAIT 保证主动方还能处理重传。
- **让网络中残留的数据包消亡**：一个 TCP 连接由四元组（源IP、源端口、目的IP、目的端口）标识。如果立刻复用同一个四元组建立新连接，旧连接残留在网络中的数据包可能被新连接错误接收。等待 2MSL 确保旧包全部过期。

39. 高并发场景下 TIME_WAIT 过多怎么办？
答：在高并发短连接场景（如 HTTP/1.0 或频繁创建关闭连接），主动关闭方会积累大量 TIME_WAIT 状态的连接，导致端口耗尽（一台机器最多约 65535 个端口）。解决方案：
- **`SO_REUSEADDR`**：允许绑定处于 TIME_WAIT 的地址，通常用于服务端重启时快速绑定端口。
- **`SO_REUSEPORT`**：允许多个 socket 绑定同一端口，提升并发能力。
- **`tcp_tw_reuse`**（Linux 内核参数）：允许在 TIME_WAIT 状态下复用连接用于新的出站连接（需配合 `tcp_timestamps`）。
- **使用长连接**：HTTP Keep-Alive、连接池，从根本上减少连接创建和关闭的频率。
- **连接池复用**：数据库连接池（HikariCP）、HTTP 连接池（OkHttp/Apache HttpClient），让连接被多个请求复用。

40. SYN Flood 攻击原理和防御措施是什么？
答：
- **攻击原理**：攻击者伪造大量不同源 IP 的 SYN 包发给 Server，Server 对每个 SYN 都会分配资源并放入 SYN Queue（半连接队列），回复 SYN+ACK 后等待第三步 ACK。由于源 IP 是伪造的，第三步永远不会到来，SYN Queue 被填满后，正常用户的 SYN 就会被丢弃。
- **防御措施**：
  - **SYN Cookie**：Server 不在 SYN Queue 中分配资源，而是将连接信息编码到 SYN+ACK 的序列号中。只有当 Client 回复正确的第三步 ACK 时，Server 才真正分配资源建立连接。代价是无法使用 TCP 选项协商（如窗口缩放）。
  - **增大 backlog（SYN Queue 大小）**：`net.ipv4.tcp_max_syn_backlog`，增大半连接队列容量。
  - **缩短 SYN 超时**：`net.ipv4.tcp_synack_retries` 减小重试次数，更快释放半连接资源。
  - **防火墙/流量清洗**：在网络入口层过滤异常 SYN 流量。

## 十五、TCP 可靠传输与拥塞控制

41. TCP 靠什么实现可靠传输？
答：TCP 通过以下机制保证字节流在不可靠的 IP 网络上可靠传输：
- **序列号（Sequence Number）+ 确认号（Acknowledgment Number）**：每个字节都有唯一序列号，接收方通过 ACK 告知发送方已成功接收到哪个字节，发送方据此判断哪些数据需要重传。
- **校验和（Checksum）**：每个 TCP 报文段都携带校验和，接收方校验失败则直接丢弃，等发送方超时重传。
- **超时重传（RTO, Retransmission Timeout）**：发送方发出数据后启动定时器，如果在 RTO 内没收到 ACK，就重传该数据。RTO 根据 RTT（往返时间）动态计算，通常使用 Jacobson 算法：`SRTT = (1-α)*SRTT + α*RTT`，`RTO = SRTT + 4*RTTVAR`。
- **快速重传（Fast Retransmit）**：如果发送方连续收到 **3 个重复 ACK**（即同一个 ack 号出现 4 次），不等超时就立即重传对应的数据段。比超时重传更快速地恢复丢包。
- **滑动窗口（Sliding Window）**：允许发送方在收到 ACK 前连续发送多个数据段，窗口大小决定了"空中飞行"的数据量上限。

42. 流量控制是什么，怎么实现的？
答：流量控制是接收方防止被发送方淹没的机制。接收方在每个 ACK 中携带 **接收窗口（rwnd, Receive Window）** 字段，告诉发送方"我的接收缓冲区还能容纳多少字节"。发送方的发送窗口取 `min(cwnd, rwnd)`，确保不会超过接收方的处理能力。当 rwnd=0 时，发送方停止发送，但会定期发送**窗口探测包（Window Probe）** 来检查接收方是否恢复。

43. 拥塞控制的四个阶段分别是什么？
答：拥塞控制是防止发送方淹没网络的机制，核心变量是**拥塞窗口（cwnd）**和**慢启动阈值（ssthresh）**：
- **慢启动（Slow Start）**：连接刚建立时，cwnd 从 **1 MSS**（Maximum Segment Size，通常 1460 字节）开始，每收到一个 ACK，cwnd 增加 1 MSS，相当于每个 RTT cwnd 翻倍（**指数增长**）。当 cwnd 达到 ssthresh 后，进入拥塞避免阶段。
- **拥塞避免（Congestion Avoidance）**：cwnd 每个 RTT 只增加 **1 MSS**（**线性增长**），谨慎探测网络容量。持续到检测到丢包为止。
- **快速重传（Fast Retransmit）**：收到 3 个重复 ACK 时，判定为丢包，立即重传丢失的数据段，不等超时定时器触发。
- **快速恢复（Fast Recovery）**：触发快速重传后，`ssthresh = cwnd / 2`，`cwnd = ssthresh + 3 MSS`（加 3 是因为已收到 3 个重复 ACK，说明有 3 个包已到达），然后每收到一个重复 ACK，cwnd 增加 1 MSS；收到新数据的 ACK 后，cwnd 恢复为 ssthresh，进入拥塞避免阶段线性增长。
- 如果是超时触发的丢包（更严重），则回到慢启动：`ssthresh = cwnd / 2`，`cwnd = 1 MSS`，重新指数增长。

44. Nagle 算法是什么，什么时候需要关闭？
答：Nagle 算法解决的是小包问题（Silly Window Syndrome 的发送端版本）。规则：如果有已发送但未确认的数据，就把后续小数据先缓冲在发送缓冲区，攒到一个 MSS 或者收到前一个包的 ACK 后再一起发。好处是减少网络中的小包数量，降低开销。但对延迟敏感的场景（如游戏、实时交互、SSH 终端）需要关闭，方法是设置 `TCP_NODELAY` 选项。Nagle 算法和 **Delayed ACK**（接收方延迟发送 ACK，攒一起发）一起使用时可能产生 **40ms~200ms 额外延迟**，这在 RPC 调用中是不能接受的，所以大多数 RPC 框架默认开启 `TCP_NODELAY`。

45. TCP 粘包/拆包是什么，应用层怎么处理？
答：TCP 是**字节流协议**，没有消息边界的概念。发送方调用两次 `write("hello")` 和 `write("world")`，接收方可能一次 `read` 收到 `"helloworld"`（粘包），也可能分多次收到 `"hel"`、`"lowor"`、`"ld"`（拆包）。原因包括 Nagle 合并、MSS 切分、接收缓冲区读取时机等。应用层必须自己定义消息边界，常用方案：
- **固定长度**：每条消息固定 N 字节，不够则填充。简单但浪费带宽。
- **长度字段前缀**：消息头部包含一个长度字段（如 4 字节 int 表示 body 长度）。最常用方案，gRPC、Dubbo、自定义 RPC 协议都用这种。
- **分隔符**：用特殊字符（如 `\r\n`、`\0`）标记消息结束。HTTP/1.1 的 header 就是用 `\r\n\r\n` 分隔的。缺点是消息内容本身不能包含分隔符，或需要转义。

## 十六、HTTP/1.1 vs HTTP/2 vs HTTP/3

46. HTTP/1.1 有哪些关键特性和问题？
答：
- **文本协议**：请求行、头部、正文都是可读文本，方便调试但解析效率低（需要逐字符扫描 `\r\n`）。
- **Keep-Alive 连接复用**：HTTP/1.0 默认每个请求一个连接，HTTP/1.1 默认 `Connection: keep-alive`，一个 TCP 连接可以串行处理多个请求。但**一个连接同一时刻只能处理一个请求-响应对**。
- **队头阻塞（Head-of-Line Blocking）**：一个连接上，前一个请求没有完成响应，后续请求就得等。浏览器通常通过**并行开 6-8 个 TCP 连接**来缓解。
- **管道化（Pipelining）**：理论上允许客户端不等上一个响应就发下一个请求，但**服务端必须按请求顺序返回响应**，实际几乎没有浏览器和服务端正确实现，被 HTTP/2 取代。
- **头部冗余**：每次请求都要发送完整的头部（Cookie、User-Agent 等），大量重复数据浪费带宽。

47. HTTP/2 做了哪些改进？
答：
- **二进制分帧层（Binary Framing Layer）**：HTTP 消息被拆分为二进制帧（Frame），每个帧属于一个**流（Stream）**。帧类型包括 HEADERS 帧、DATA 帧、SETTINGS 帧、PING 帧等。每个 Stream 有唯一 ID（客户端发起的用奇数，服务端发起的用偶数）。
- **多路复用（Multiplexing）**：一个 TCP 连接上可以同时承载多个 Stream，不同 Stream 的帧可以交错发送。请求 A 的 DATA 帧和请求 B 的 HEADERS 帧可以在同一个连接上交替传输，解决了 HTTP 层的队头阻塞。
- **头部压缩（HPACK）**：维护一张**静态表**（61 个常见 header 如 `:method GET`、`:status 200`）和一张**动态表**（连接生命周期内动态添加），加上 **Huffman 编码**压缩 header value。实测可以将头部大小减少 85%-90%。
- **服务端推送（Server Push）**：服务端在客户端请求 HTML 时，可以主动推送关联的 CSS、JS 资源。通过 PUSH_PROMISE 帧实现。但实际效果争议较大，Chrome 已在 2022 年移除支持。
- **流优先级（Stream Priority）**：客户端可以为不同 Stream 分配权重和依赖关系，服务端据此分配带宽。
- **仍存在的问题——TCP 层队头阻塞**：HTTP/2 所有 Stream 共享同一个 TCP 连接。TCP 丢失一个包，整个连接的**所有 Stream** 都要等这个包重传完成。在高丢包率网络下（如移动网络），HTTP/2 性能甚至不如 HTTP/1.1 开多连接。

48. HTTP/3（QUIC）解决了什么问题？
答：
- **基于 UDP + QUIC 协议**：不再使用 TCP，而是在 UDP 之上实现了一个用户态传输协议 QUIC。
- **解决 TCP 队头阻塞**：QUIC 中每个 Stream 独立进行丢包重传，一个 Stream 丢包**不影响其他 Stream**。这是 HTTP/3 相对于 HTTP/2 最核心的改进。
- **更快的连接建立**：
  - TCP + TLS 1.2 需要 3 个 RTT（TCP 握手 1-RTT + TLS 握手 2-RTT）
  - TCP + TLS 1.3 需要 2 个 RTT（TCP 握手 1-RTT + TLS 握手 1-RTT）
  - QUIC 首次连接只需 **1-RTT**（传输握手和加密握手合并），复用连接可以 **0-RTT**（利用之前缓存的密钥直接发送加密数据）
- **内建 TLS 1.3**：加密不是可选的，QUIC 协议本身就包含 TLS 1.3 握手，所有数据默认加密。
- **连接迁移（Connection Migration）**：TCP 连接由四元组（源IP、源端口、目的IP、目的端口）标识，WiFi 切 4G 意味着 IP 变了、连接断了。QUIC 使用 **Connection ID** 标识连接，与 IP/端口无关，网络切换时连接不中断。
- **用户态实现**：QUIC 在用户态实现，不依赖操作系统内核更新，迭代速度更快。但也意味着 CPU 开销比内核态 TCP 稍高。

49. 实际选型时怎么选 HTTP 版本？
答：
- 内部服务通信优先 HTTP/2 或 gRPC（基于 HTTP/2），多路复用减少连接数。
- 面向终端用户的 API，如果客户端和基础设施都支持，优先 HTTP/2；移动端弱网场景考虑 HTTP/3。
- 不要在同一个系统里混用太多协议版本，统一在网关层做协议转换（如 Nginx 对外 HTTP/2，内部转 HTTP/1.1 或 gRPC）。

## 十七、TLS 握手过程

50. TLS 1.2 握手的完整流程是什么？
答：TLS 1.2 需要 **2 个 RTT** 完成握手：
- **第一步 Client Hello（RTT 1 开始）**：Client 发送支持的 TLS 版本（如 TLS 1.2）、支持的**加密套件列表**（如 `TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256`）、**Client Random**（32 字节随机数）、支持的压缩方法。
- **第二步 Server Hello + Certificate + Server Hello Done（RTT 1 结束）**：Server 回复选定的加密套件、**Server Random**（32 字节随机数）、Server 的数字证书（包含公钥）、Server Hello Done。如果选用 ECDHE 密钥交换，还会发送 **Server Key Exchange**（携带 DH 参数和签名）。
- **第三步 Client 验证 + Key Exchange（RTT 2 开始）**：Client 验证证书链（根 CA → 中间 CA → 服务端证书）和有效期。验证通过后，生成 **Pre-Master Secret**（48 字节随机数），用 Server 公钥加密后发送（如果是 RSA 交换）；如果是 ECDHE，则 Client 发送自己的 DH 公钥，双方各自计算出 Pre-Master Secret。
- **第四步 双方计算 Session Key**：双方用 `Client Random + Server Random + Pre-Master Secret` 通过 PRF（伪随机函数）计算出 **Master Secret**，再由 Master Secret 派生出对称加密密钥、MAC 密钥和 IV。
- **第五步 Change Cipher Spec + Finished（RTT 2 结束）**：双方各自发送 Change Cipher Spec（通知后续通信切换为加密模式）和 Finished（用协商好的密钥加密之前所有握手消息的摘要，用于校验握手完整性）。

51. TLS 1.3 握手为什么更快更安全？
答：TLS 1.3 只需 **1 个 RTT** 完成握手：
- **Client Hello**：除了版本、加密套件、Client Random 之外，直接携带 **Key Share**（客户端的 DH 公钥参数，通常是 X25519 或 P-256 曲线）。相当于在第一步就把密钥交换的参数带上了。
- **Server Hello + Key Share + Certificate + Finished**：Server 在一个 RTT 内回复选定的加密套件、Server 的 Key Share、加密后的证书和 Finished。双方在这一步就已经能计算出对称密钥。
- **Client Finished**：Client 发送 Finished 确认。后续开始传输应用数据。
- **安全性改进**：TLS 1.3 移除了不安全的加密套件——去掉了 RSA 密钥交换（不支持前向保密）、RC4、CBC 模式、SHA-1、DES/3DES 等。只保留 AEAD 加密（AES-GCM、ChaCha20-Poly1305）和 ECDHE/DHE 密钥交换（支持**前向保密 PFS**：即使服务端私钥泄露，过去的通信记录也无法被解密）。
- **0-RTT 恢复**：如果客户端之前连接过该服务端，可以使用缓存的 PSK（Pre-Shared Key）在第一个包就携带加密的应用数据。代价是 0-RTT 数据**不具备前向保密性**，且可能被**重放攻击**，所以只适合幂等请求（如 GET）。

52. 后端为什么需要懂 TLS？
答：
- **HTTPS 性能影响**：TLS 握手增加 1-2 个 RTT 延迟，在高并发下 CPU 开销显著（RSA 密钥交换比 ECDHE 更消耗 CPU）。了解握手过程才能做出正确的优化决策。
- **证书管理**：证书申请、续期、链路完整性、中间证书配置、证书过期监控都是后端运维的常见责任。
- **TLS 终止（TLS Termination）**：通常在网关/负载均衡层（如 Nginx、AWS ALB）统一处理 TLS，内部服务走明文或 mTLS，减少每个服务的证书管理负担。
- **mTLS（双向 TLS）**：微服务之间互相验证身份，不仅 Client 验证 Server 证书，Server 也验证 Client 证书。在零信任网络（如 Istio Service Mesh）中广泛使用。

## 十八、select/poll/epoll 对比

53. select 的工作原理和问题是什么？
答：
- **工作原理**：调用 `select(nfds, readfds, writefds, exceptfds, timeout)` 时，用户态将 `fd_set`（一个位图数组，每一位对应一个文件描述符）拷贝到内核态；内核遍历所有被设置的 fd，检查是否就绪，将就绪的 fd 标记在 `fd_set` 中；返回就绪 fd 的数量；用户态再遍历 `fd_set` 找出哪些 fd 就绪了。
- **问题**：
  - **fd 数量限制**：`fd_set` 默认大小 **1024**（由 `FD_SETSIZE` 宏定义），超过需要重新编译内核或应用。
  - **O(n) 遍历**：内核每次都要遍历所有 fd，不管是否就绪。用户态也要遍历整个 `fd_set`。
  - **fd_set 来回拷贝**：每次调用都要把 `fd_set` 从用户态拷贝到内核态，返回时再拷贝回来。
  - **fd_set 被修改**：每次调用后 `fd_set` 会被内核覆写，下次调用前必须重新设置。

54. poll 改进了什么，还有什么问题？
答：
- **改进**：使用 `pollfd` 结构体数组替代位图，每个 `pollfd` 包含 `fd`、`events`（关注的事件）、`revents`（就绪的事件），没有 1024 的硬限制。同时 `events` 和 `revents` 分离，不需要每次重新设置。
- **仍然存在的问题**：每次调用仍然将整个 `pollfd` 数组从用户态拷贝到内核态；内核仍然需要 **O(n) 遍历所有 fd**；返回后用户态也需要遍历找出就绪的 fd。在 fd 数量大但就绪 fd 少的场景下效率很低。

55. epoll 为什么能高效处理大量并发连接？
答：epoll 是 Linux 特有的 I/O 多路复用机制，通过三个系统调用配合工作：
- **`epoll_create`**：创建一个 epoll 实例，内核为其维护一棵**红黑树**（存储所有被监控的 fd）和一个**就绪链表（rdlist）**。
- **`epoll_ctl(epfd, op, fd, event)`**：将 fd 注册到红黑树中（`EPOLL_CTL_ADD`），或修改（`EPOLL_CTL_MOD`）、删除（`EPOLL_CTL_DEL`）。每个 fd 只需注册一次，不需要每次调用都传入。
- **`epoll_wait(epfd, events, maxevents, timeout)`**：阻塞等待就绪事件。内核通过**回调机制**（当 fd 对应的设备驱动就绪时，回调将该 fd 加入就绪链表），`epoll_wait` 只需检查就绪链表是否为空，非空则将就绪的 fd 拷贝到用户态的 `events` 数组中返回。
- **核心优势**：
  - **O(1) 事件通知**：不需要遍历所有 fd，就绪事件通过回调主动加入链表。
  - **只返回就绪的 fd**：用户态不需要遍历全部 fd，只处理返回的就绪列表。
  - **fd 无数量限制**：上限是系统的文件描述符总数（`/proc/sys/fs/file-max`，通常可达百万级）。
  - **避免重复拷贝**：fd 注册到红黑树后，后续 `epoll_wait` 不需要再拷贝 fd 集合。

56. epoll 的 ET 和 LT 模式有什么区别？
答：
- **LT（Level-Triggered，水平触发，默认模式）**：只要 fd 处于就绪状态（如可读缓冲区中有数据），每次 `epoll_wait` 都会返回这个 fd。编程简单，但如果不及时处理，同一个 fd 会反复被通知。
- **ET（Edge-Triggered，边缘触发）**：只在 fd 状态**发生变化时**通知一次（如从不可读变为可读）。即使缓冲区中还有数据没读完，也不会再次通知。因此 ET 模式下必须**一次性将缓冲区数据读完**（循环 read 直到返回 `EAGAIN`），否则数据会一直留在缓冲区却不再被通知。ET 模式减少了事件通知次数，性能更高，但编程更复杂，更容易出 bug。
- **实践选择**：Nginx 使用 ET 模式以获取最高性能；大多数应用框架使用 LT 模式以降低编程复杂度。

57. kqueue 是什么，和 epoll 什么关系？
答：kqueue 是 BSD/macOS 系统上的 I/O 事件通知机制，功能和设计理念与 Linux 的 epoll 类似，都是事件驱动、不需要遍历所有 fd。kqueue 通过 `kqueue()` 创建实例、`kevent()` 注册和等待事件。kqueue 支持的事件类型更丰富（不仅限于 fd，还支持文件系统事件、信号、定时器等）。跨平台框架通常会封装这些差异。

58. 这些 I/O 模型对应哪些上层框架？
答：
| I/O 多路复用 | 对应框架/系统 |
|---|---|
| epoll（Linux） | Netty（Java）、Redis（ae_epoll）、Nginx、Node.js（libuv）、Go runtime |
| kqueue（BSD/macOS） | Netty（kqueue transport）、Node.js（libuv）、Nginx |
| IOCP（Windows） | Node.js（libuv）、.NET |

这些框架在底层使用操作系统提供的高效 I/O 多路复用机制，在上层暴露事件驱动/异步编程模型。Netty 的 `EventLoop` 就是对 epoll/kqueue 的抽象；Redis 单线程能支撑高并发靠的就是 epoll 的事件通知效率；Nginx 的 worker 进程也是通过 epoll 实现高效的连接处理。理解这层底层机制，才能真正理解"为什么单线程也能高并发"以及"连接数暴涨时该调什么参数"。
