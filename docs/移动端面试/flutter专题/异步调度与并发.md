# 异步调度与并发

## 使用方式

Flutter 的异步题很容易被答浅。很多人只会说 Future、async/await、microtask，但高分回答要把 “Dart event loop、Flutter 帧调度、页面生命周期、isolate、compute、Stream” 串成一整条链。

---

## 一、Dart 事件循环怎么讲

建议面试时固定成这个表达：

1. 同步代码先执行。
2. 微任务队列优先于普通事件队列。
3. `scheduleMicrotask()`、`Future.microtask()` 更接近微任务。
4. 普通 `Future`、`Timer` 更接近后续事件队列。

这不是为了背术语，而是为了回答“为什么这段代码是这个输出顺序”“为什么页面状态会在这个时机更新”。

---

## 二、为什么 `await` 不等于“开新线程”

高频误区之一就是把异步等待理解成“丢到后台跑了”。

更准确的说法是：

- `await` 只是让当前 async 函数在结果返回前挂起。
- 它不会自动把任务搬到后台 isolate。
- 如果真正的工作还是 CPU 密集型并且发生在当前 isolate，它仍然会影响响应能力。

---

## 三、`Future`、`microtask`、`Timer` 的真实差异

### 1. `Future.microtask`

更适合非常短、需要在当前同步代码之后尽快执行的逻辑。

### 2. 普通 `Future`

更接近后续事件队列任务，优先级低于微任务。

### 3. `Timer`

也是后续事件队列的一种来源，常用于延迟执行和周期性调度。

高分回答不要只背“谁先谁后”，还要补：

1. 为什么这样设计。
2. 哪种时机适合放 UI 初始化逻辑，哪种不适合。

---

## 四、`addPostFrameCallback` 为什么是 Flutter 高频题

Flutter 官方文档明确说明：

- 它会在一帧结束后执行。
- 按注册顺序执行。
- 只执行一次。
- 不会主动请求新的一帧。

所以它适合：

- 依赖布局完成后的逻辑
- 首帧后导航
- 首帧后弹窗
- 读取尺寸

但不适合：

- 拿来代替普通 Future
- 把一堆业务逻辑无脑塞进去

高分回答关键点：

“它不是简单的异步延后，而是与 Flutter 帧生命周期绑定的时机工具。”

---

## 五、为什么 `FutureBuilder` / `StreamBuilder` 也属于异步调度题

因为它们不只是 UI 组件，更在考你是否理解“异步数据源的生命周期应该放在哪”。

### `FutureBuilder`

官方文档强调：

- future 必须提前拿到
- 不应该在 build 里创建
- 否则父组件每次重建，异步任务都可能重启

### `StreamBuilder`

官方文档强调：

- stream 同样不应该在 build 里临时创建
- builder 收到的是与 pipeline 节奏耦合后的 snapshot 子序列

高分回答：

“这两类题真正考的不是 builder API，而是你有没有把异步数据源和页面生命周期对齐。”

---

## 六、isolate 为什么一定要会讲

### 1. 它在解决什么问题

解决的是 CPU 密集型任务阻塞当前 isolate 的问题。

### 2. 它和 Future 的区别

- Future 只是异步结果抽象
- isolate 才是真正独立的执行上下文

### 3. 它的核心成本

- 不能共享内存
- 需要消息传递
- 可传递对象受限制

---

## 七、`compute()` 怎么讲

Flutter 官方文档明确说明：

- 它适合处理会超过几毫秒、可能导致掉帧的任务。
- 在原生平台上会运行在独立 isolate。
- 在 web 上仍然运行在当前 event loop。
- 传入的 callback、message、result 必须可跨 isolate 传递。

高分回答关键点：

1. `compute()` 很适合大 JSON 解析、文件处理、文本转换。
2. 不是所有耗时任务都适合丢进去，例如需要复杂上下文、对象不可序列化时就要谨慎。

---

## 八、`StreamController.broadcast` 为什么会被追问

官方文档里有几个非常容易考的点：

1. 它允许多个监听者同时订阅。
2. 当没有监听者时，不会缓存事件。
3. 默认 `sync: false` 时，事件会在稍后异步分发。
4. 如果 `sync: true`，事件可能在 `add()` 调用期间直接触发，要非常谨慎。

所以这类题的高分回答一定要补：

- broadcast stream 适合消息广播，不等于自带历史缓存。
- 晚订阅的页面可能直接错过之前的事件。

---

## 九、`Isolate.run()`、长生命周期 isolate、背景 isolate 为什么也值得会讲

Flutter 官方 isolates 文档除了讲 `compute()`，还补了几个更适合中高级面试的点：

### 1. `Isolate.run()`

适合一次性、短生命周期的后台计算。

### 2. 长生命周期 isolate

如果相同类型的后台计算会在应用里反复发生，反复 spawn isolate 也有开销。官方文档明确指出，这时可以考虑更长生命周期的 worker isolate，而不是每次都起一个新的短生命周期 isolate。

### 3. 背景 isolate 的限制

高分回答时非常建议补这些点：

1. isolate 之间不共享内存，只能消息传递。
2. 后台 isolate 不能直接做 UI 工作。
3. 不能直接访问 `rootBundle` 和 `dart:ui` 相关能力。
4. 背景 isolate 虽然可以通过 `BackgroundIsolateBinaryMessenger` 使用部分平台插件，但不能接收宿主主动推送给 Flutter 的 unsolicited messages。

这些内容一说出来，深度会明显比“就是开子线程”高很多。

## 十、真实业务里最常见的异步坑

1. 页面已经销毁，异步回调还在落地状态。
2. build 里创建 Future，导致重复请求。
3. 搜索输入过快，旧请求覆盖新结果。
4. 大计算只用 Future 包了一层，实际还在当前 isolate 里卡 UI。
5. broadcast stream 没有缓存，结果晚订阅页面拿不到关键事件。

---

## 十一、典型追问

1. 为什么 `Future` 完成了，builder 不一定马上执行？
答：因为重建最终还是由 Flutter pipeline 调度，不是 Future 一完成就同步立刻跑 builder。

2. 为什么 `addPostFrameCallback` 比普通 Future 更适合做首帧后逻辑？
答：因为它绑定的是帧完成时机，而不是简单的事件队列延后。

3. 为什么 isolate 适合大 JSON 解析，但不适合滥用？
答：因为它有消息传递和数据可传递性限制，过小任务搬过去反而不划算。

4. 为什么 `compute()` 在 Flutter Web 上不能简单理解成“后台线程”？
答：因为官方文档明确说明，在 Web 上它仍然运行在当前 event loop，只是在代码层面提供统一接口。

5. 为什么有些后台任务适合长生命周期 isolate，而不是反复 `compute()`？
答：因为反复创建 isolate 本身有开销。对于反复执行的同类计算，长期 worker 有时更合适。

---

## 十二、面试回答模板

如果面试官问“请你讲讲 Flutter 的异步模型”，建议按这个顺序说：

1. 先讲 Dart event loop：同步、microtask、event queue。
2. 再讲 Flutter 帧时机：`addPostFrameCallback`。
3. 再讲 `FutureBuilder` / `StreamBuilder` 的数据源生命周期。
4. 最后讲 isolate 和 `compute()`，说明什么时候该真的做后台隔离。
