# 性能优化与渲染性能

## 使用方式

Flutter 性能题最容易被答成“加 const、拆组件、用 ListView.builder”。这些都没错，但只说这些不够。高分回答必须把 build、layout、paint、图片、列表、层、GPU 成本拆开。

---

## 一、Flutter 性能题到底在考什么

通常在考四件事：

1. 你能不能先判断卡顿发生在哪一层。
2. 你会不会用证据而不是口诀做定位。
3. 你是否理解 Flutter 官方的性能建议，而不是只会口耳相传。
4. 你能不能结合真实业务给出取舍。

---

## 二、build 成本怎么讲

Flutter 官方性能最佳实践明确强调：

- 避免单个巨大 widget 拥有超长 `build()`。
- `setState()` 会让所有后代进入 rebuild 路径，所以要尽量局部化。
- `const` 构造器可以帮助框架跳过大量重复工作。
- 可复用 UI 片段优先写成 widget，而不是 helper function。

高分回答不能停在“build 太多”，而要说：

1. 是哪块子树经常重建。
2. 为什么这块重建范围过大。
3. 能不能通过状态下沉、child 预构建、const、局部 builder 来缩小范围。

---

## 三、layout 成本怎么讲

很多人谈 Flutter 性能只谈 build，不谈 layout，这是不够的。

布局成本高的常见来源：

- 树层级很深
- 约束不清导致反复计算
- 使用了会触发 intrinsic pass 的布局
- 大列表里做复杂尺寸计算

Flutter 官方最佳实践特别提醒：对于列表和网格，要警惕 intrinsic passes，因为它们会让布局变成多次遍历。

高分回答建议主动补：

1. “我会先看是不是 build 范围问题，再看是不是 layout pass 太重。”
2. “如果列表 / grid 做了依赖内容大小的统一尺寸计算，就要警惕 intrinsic 成本。”

---

## 四、paint 与合成成本怎么讲

面试里真正拉开差距的地方，是你能否继续往 paint 和合成说下去。

Flutter 官方最佳实践明确指出要谨慎使用：

- `saveLayer()`
- `Opacity`
- clipping

为什么？

- `saveLayer()` 可能触发昂贵的离屏缓冲和 render target 切换。
- `Opacity` 在某些场景下可能产生不必要的额外成本。
- clipping 也不是零成本操作。

高分回答不要机械背“少用 Opacity”，而要说：

1. 为什么这些操作会贵。
2. 是否有更直接的绘制方式。
3. 是否能通过设计调整避免它。

---

## 五、列表性能怎么讲

Flutter 官方明确建议：大列表和大网格一定要“lazy”，优先用 builder 模式。

高分回答建议按下面顺序：

1. 先确认是不是大量不可见项也被构建了。
2. 再看单个 item 是否过重。
3. 再看图片、富文本、动画是否叠加。
4. 最后看 keepAlive、缓存和分页策略。

典型追问：

1. 为什么 `ListView.builder` 比直接 children 更适合长列表？
答：因为它按可见区域懒构建，而不是初始化时一次把全部子项建出来。

2. 为什么长列表卡顿不一定只是 item 太多？
答：因为也可能是 item 本身太重、图片解码过大、状态更新范围太大，或者滚动中夹了昂贵绘制。

---

## 六、图片和资源性能怎么讲

真实项目里，很多 Flutter 卡顿并不是状态管理问题，而是图片问题。

高分回答要覆盖：

- 图片尺寸与展示尺寸匹配
- 是否有缓存策略
- 是否有缩略图和原图分层
- 解码和内存占用是否被低端机放大

---

## 七、`RepaintBoundary` 怎么讲

这是非常高频的性能题。

更准确的回答是：

- 它的价值是帮助把某些局部区域的重绘边界隔离开。
- 适合变化频繁但周围区域相对稳定的场景。
- 不是“包上就一定变快”，边界切太碎反而可能得不偿失。

高分回答要体现 trade-off，而不是把它说成万能解法。

---

## 八、DevTools 与证据链怎么讲

Flutter 性能题高分回答一定要带证据：

1. 在 profile 模式看，不在 debug 模式下拍脑袋。
2. 看 DevTools 的 rebuild、timeline、layout、paint、raster 信息。
3. 结合低端机、真机、弱网、首帧和长列表场景验证。

如果只说“我感觉更顺了”，这类答案通常不够强。

---

## 九、典型追问

1. 为什么 build 快不代表页面一定流畅？
答：因为 layout、paint、图片解码、GPU 合成和动画都可能成为瓶颈。

2. 为什么 `const` 有帮助，但不是万能？
答：因为它只能帮你减少部分 rebuild 工作，真正卡顿还可能来自 layout / paint / 图片 / 动画。

3. 为什么官方建议不用 helper function 代替 widget？
答：因为 widget 更容易让框架在更新时做更细粒度的优化，也更利于复用和局部 rebuild 控制。

---

## 十、面试回答模板

如果面试官问“你做过哪些 Flutter 性能优化”，建议按这个顺序说：

1. 先讲现象和影响范围。
2. 再把问题拆成 build、layout、paint 三层。
3. 再说明你通过什么工具定位到瓶颈。
4. 最后讲优化动作和验证结果。
