# 状态管理与生命周期

## 使用方式

Flutter 状态题最容易出两种问题：一种是只会列状态管理库名字；另一种是只会背生命周期顺序。高分回答要说明“状态为什么这样拆”“某个生命周期为什么适合做这件事”，而不是停在 API 层。

---

## 一、先把状态分层

面试时不要一上来就说 Provider、Riverpod、Bloc，先把状态分层更重要。

常见可以分成四类：

1. 页面瞬时状态：弹窗开关、输入值、选中态、loading。
2. 页面级业务状态：列表数据、详情数据、表单状态。
3. 跨页面共享状态：用户信息、登录态、全局配置、主题、实验分组。
4. 外部事件流状态：WebSocket、下载进度、上传进度、流式输出。

高分回答关键点：

- 不是所有状态都要上全局。
- 也不是所有状态都应该塞在 `setState` 里。
- 先按共享范围、生命周期和更新频率拆状态，再决定用什么工具。

---

## 二、`setState` 真正在干什么

### 1. 它不是“直接刷新 UI”

更准确的说法是：

- 你通过 `setState` 告诉框架“当前 State 依赖的 UI 需要重新评估”。
- Flutter 会把这次变化标记到更新流程里，后续重新走 build。

### 2. 为什么面试官总问 `setState`

因为它最容易暴露你有没有局部更新意识。高分回答通常会主动补：

1. `setState` 本身不是问题，问题在于你把它放在哪个层级调用。
2. 如果高层 State 管得太多，一次小变化也可能拖着大子树一起 rebuild。
3. 真正的优化方向不是“禁用 setState”，而是“控制状态边界和刷新范围”。

---

## 三、`ChangeNotifier`、`ValueNotifier` 怎么选

### 1. `ChangeNotifier`

适合承接一组状态变化，并通过 `notifyListeners()` 主动通知监听者。

适合的场景：

- 页面 ViewModel
- 表单状态
- 列表与详情共用的一组状态

要主动提的点：

- 官方文档指出 add listener 是 `O(1)`，remove 和 notify 是 `O(N)`。
- 所以监听很多、通知很频繁时，要警惕成本。

### 2. `ValueNotifier`

更适合单值状态。

适合的场景：

- 当前 tab
- 搜索关键字
- 计数器
- 某个选中值

高分回答：

“如果状态只是单值变化，用 `ValueNotifier` 往往更轻；如果是一组有关系的状态，更适合抽成 `ChangeNotifier` 或更完整的状态模型。”

---

## 四、`ValueListenableBuilder`、`FutureBuilder`、`StreamBuilder` 怎么区分

### 1. `ValueListenableBuilder`

适合本地、同步、单值可观察状态。

高分点：

- Flutter 官方文档强调可以通过 `child` 传入不依赖值的子树，减少重复 rebuild。

### 2. `FutureBuilder`

适合单次异步结果。

但高频追问是：为什么 Future 不能在 build 里直接创建？

官方文档明确说明：

- Future 应该更早拿到，例如在 `initState`、`didUpdateWidget`、`didChangeDependencies`。
- 如果在 build 里创建，每次父组件 rebuild 都可能导致异步任务重启。

### 3. `StreamBuilder`

适合持续异步事件流。

官方文档同样强调：

- Stream 也不应该在 build 里临时创建。
- builder 收到的是与 Flutter pipeline 节奏耦合后的 snapshot 子序列，不是“事件来一个就 100% 立刻重建一次”的机械模型。

高分回答一定要补：

“`FutureBuilder` 和 `StreamBuilder` 的关键不是会不会写 builder，而是你有没有把数据源生命周期放对位置。”

## 五、生命周期怎么答才不表面

### 1. 高频顺序

常见顺序是：

- `initState`
- `didChangeDependencies`
- `build`
- 若父组件配置变化，可能触发 `didUpdateWidget`
- 销毁时 `dispose`

### 2. 每个阶段更适合做什么

#### `initState`

适合：

- 初始化控制器
- 绑定一次性监听
- 发起基于当前 State 的初始化动作

不适合：

- 依赖 InheritedWidget 的值做完整初始化逻辑

#### `didChangeDependencies`

适合：

- 依赖 `InheritedWidget` / `Provider` / 上下文依赖的初始化

#### `build`

适合：

- 根据当前状态生成 UI

不适合：

- 创建 Future / Stream
- 发请求
- 注册监听

#### `didUpdateWidget`

适合：

- 父组件参数变化后，同步调整本地状态或外部订阅

#### `dispose`

适合：

- 清理 controller
- 取消 stream subscription
- 释放 notifier / timer / animation

## 六、为什么生命周期题经常和 bug 排查绑在一起

因为很多真实 bug 都不是“语法不会”，而是时机放错了：

1. 在 build 里创建 Future，导致反复请求。
2. 没在 dispose 里取消订阅，导致旧事件污染新页面。
3. 父组件参数更新了，但没在 `didUpdateWidget` 里同步处理。
4. 依赖上下文的初始化放在 `initState`，导致时机不对。

---

## 七、`InheritedWidget` 为什么要会讲

即使团队不用它直接写业务，它也很重要，因为很多状态管理工具底层都依赖类似机制。

高分回答要点：

1. 它解决的是“祖先向后代树传递共享依赖”。
2. 比起层层 props 传递，更适合共享上下文级数据。
3. 它的价值不是“全局变量更方便”，而是“依赖关系可以被框架感知并驱动更新”。

---

## 八、`GlobalKey`、`AutomaticKeepAliveClientMixin` 为什么经常和生命周期一起问

### 1. `GlobalKey`

Flutter 官方文档明确说明：

- `GlobalKey` 是全局唯一标识。
- 它可以访问对应的 `BuildContext` 和 `State`。
- 重新创建新的 `GlobalKey` 会让原 subtree 状态被丢弃，并重建 fresh subtree。

所以高分回答要体现：

1. `GlobalKey` 不是普通 Key 的“更高级版本”，而是更重的能力。
2. 它应该由 `State` 这种长生命周期对象持有，而不是 build 里临时创建。

### 2. `AutomaticKeepAliveClientMixin`

Flutter 官方文档明确要求：

- 实现 `wantKeepAlive`
- build 里调用 `super.build(context)`
- `wantKeepAlive` 变化时调用 `updateKeepAlive()`

这类题本质上在考：

1. 你是否理解 lazy list 下的状态销毁机制。
2. 你是否知道“保活”不是加个 mixin 就完事。

## 九、典型追问

1. 为什么 `FutureBuilder` 常常被用错？
答：因为很多人把 Future 写在 build 里，结果父组件一 rebuild，请求就重开，页面看起来像“Flutter 很怪”，实际是异步生命周期放错了。

2. 为什么 `ChangeNotifier` 在大页面里有时会越来越难维护？
答：因为它本身只是通知机制，不会自动帮你拆状态粒度。如果一个 notifier 里既管列表、又管筛选、又管 loading、又管弹窗，很快就会通知范围过大、依赖关系混乱。

3. 为什么 `StreamBuilder` 不等于“来一条消息就一定立刻重建一次”？
答：因为 builder 的执行仍受 Flutter pipeline 调度影响，拿到的是 snapshot 子序列，不是对底层 stream 事件的机械一一映射。

4. 为什么 `GlobalKey` 容易被滥用？
答：因为它确实强大，但代价也大。很多场景其实只需要普通 `Key` 做身份识别，不需要跨树访问状态，一上来就用 `GlobalKey` 往往会把问题复杂化。

5. 为什么 keep alive 题本质上还是状态生命周期题？
答：因为它在问的不是一个 mixin，而是“当一个子项不在视口里时，框架会不会销毁它；如果我要保住它的状态，应该怎么做、代价是什么”。

---

## 十、面试回答模板

如果面试官问“你怎么理解 Flutter 的状态管理”，建议按这个顺序说：

1. 先讲状态要按范围和生命周期分层。
2. 再讲 `setState`、`ChangeNotifier`、`ValueNotifier` 各自适用边界。
3. 再讲 `FutureBuilder` / `StreamBuilder` 的数据源时机。
4. 最后补生命周期里哪些阶段该做什么、哪些阶段不该做什么。
