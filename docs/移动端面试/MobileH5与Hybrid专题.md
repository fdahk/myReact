# MobileH5与Hybrid专题

## 使用方式

这部分最容易被低估，因为很多团队的“移动端前端”真实工作内容就是 `H5 + WebView + 宿主容器`。面试高频点不是标准 Web，而是容器差异、桥接协议、白屏、缓存和登录态一致性。

---

## 一、`Mobile H5` 和普通 Web 有什么不同

1. 运行在移动设备和宿主容器里。
2. 很多能力依赖 `WebView` 和 App 宿主。
3. 登录态、分享、支付、文件、打开页面等常常要通过 JS Bridge 协同。
4. 线上表现会受到容器版本、系统版本和宿主实现影响。

---

## 二、什么是 `Hybrid`

你可以把它理解成“原生容器承载 Web 页面，并通过桥接让 Web 获得更多原生能力”。

它的核心价值通常是：

- 迭代快
- 业务灵活
- 动态化强

但它的核心难点同样明显：

- 容器差异
- JS Bridge 协议稳定性
- 白屏与首屏问题
- 登录态和页面栈一致性

---

## 三、`WebView` 为什么是高频题

因为很多移动端前端问题最终都落在 `WebView`：

- 打开慢
- 白屏
- 资源缓存不稳定
- H5 与宿主状态不同步
- 桥接偶发失败
- iOS 和 Android 表现不一致

面试官问 `WebView`，本质上是在考你是否理解前端页面已经不再处于一个“纯浏览器世界”。

---

## 四、JS Bridge 怎么答

### 1. 它解决什么问题

让 H5 可以调用原生能力，也让原生可以通知 H5。

### 2. 为什么桥接协议很重要

因为一旦宿主版本多、业务页面多，协议不稳定就会导致：

- 线上兼容问题
- 参数错配
- 调用失败难定位
- 老版本容器无法支持新页面

### 3. 高分回答要点

回答桥接时建议主动提：

- 统一协议层
- 版本兼容
- 超时与错误兜底
- 埋点和日志
- H5 与宿主双方约定清晰的成功、失败和取消语义

---

## 五、白屏问题怎么答

`WebView` 白屏通常要按这几层排查：

1. 容器是否成功创建
2. URL 和资源是否成功加载
3. H5 首屏资源是否过重
4. 桥接初始化是否阻塞首屏
5. 是否存在登录态或鉴权前置依赖
6. 是否有版本不匹配或容器异常

高分回答不是一句“看控制台”，而是能把容器、资源、脚本、桥接和登录态拆开。

---

## 六、缓存和离线怎么答

`Hybrid` 场景里常见追问有：

- 静态资源要不要缓存
- 页面资源是否支持离线包
- 容器缓存和浏览器缓存怎么协同
- 资源更新如何做版本控制

高分回答要体现：

1. 资源缓存能提升打开速度，但会放大版本治理难度。
2. 离线包要优先服务高频和稳定业务，不适合所有页面。
3. 缓存策略必须和发布策略一起设计，而不是各做各的。

---

## 七、登录态一致性怎么答

`Hybrid` 登录态问题很高频，因为它经常跨三层：

- H5 页面
- App 宿主
- 服务端

典型风险包括：

- H5 以为已登录，宿主 token 已失效
- 宿主已退出，H5 页面还保留旧状态
- 新开页面和旧页面状态不同步

所以高分回答通常会主动讲：

- 登录态来源统一
- 刷新机制统一
- 宿主通知 H5 的协议
- H5 的兜底校验和失效处理

---

## 八、典型场景题

1. `WebView` 页面首开很慢，如何判断是容器初始化还是 H5 首屏问题。
2. H5 页面在 Android 正常，iOS 偶发白屏，怎么拆解。
3. JS Bridge 调用偶发失败，前端如何设计兜底和日志。
4. H5 与宿主登录态不同步，怎么设计统一机制。
5. 离线包上线后部分用户拿到旧资源，怎么治理。

---

## 九、典型追问

1. 为什么 `Hybrid` 方案仍然很常见？
答：因为它在业务灵活性和迭代速度上很强，尤其适合活动、内容、运营和部分中轻量业务场景。

2. 为什么 `Hybrid` 方案更容易出兼容问题？
答：因为它叠加了 Web、容器、宿主、系统版本和桥接协议多个变量。

3. 为什么 `WebView` 优化不能只按 Web 前端套路做？
答：因为很多瓶颈其实发生在容器初始化、宿主状态或桥接链路，不是纯浏览器渲染问题。

---

## 十、回答模板

如果面试官问“你怎么理解 `Hybrid` 开发”，建议按这个顺序说：

1. 先讲 `WebView` 和宿主容器关系。
2. 再讲 JS Bridge 的价值和风险。
3. 再讲白屏、缓存、登录态一致性。
4. 最后讲为什么 `Hybrid` 仍然是很多团队的重要方案。
