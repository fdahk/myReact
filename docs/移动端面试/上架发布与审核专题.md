# 上架发布与审核专题

## 使用方式

这部分经常被忽视，但它是移动端区别于纯 Web 前端的重要现实约束。高分回答要能体现：你理解提审、发布节奏、审核风险、热修边界和多版本共存对工程决策的影响。

---

## 一、为什么移动端要单独讲上架发布

1. 很多变更不是今天发版、今天全量生效。
2. 审核周期会影响需求上线节奏。
3. 一旦线上出问题，修复节奏远慢于 Web。
4. 发版策略会反过来影响架构、灰度和开关设计。

---

## 二、发布链路通常包含什么

1. 开发和联调
2. 测试验证
3. 灰度和内测
4. 提审
5. 发布
6. 观察指标
7. 止血和回滚

高分回答要体现：发布不是“打个包上传”，而是一整套风险控制流程。

---

## 三、为什么审核会影响前端设计

因为审核慢意味着：

- 高风险功能更需要开关和灰度
- 配置化能力更有价值
- 不兼容变更必须更谨慎
- 上架前就要预埋好止血能力

---

## 四、热更新边界怎么答

这是高频题，因为很多团队都会被问到：

- 哪些内容可以动态化
- 哪些变更不应该依赖热更新
- 热更新为什么不能替代正式发版

高分回答通常会说：

1. 配置、文案、部分低风险资源更适合动态化。
2. 涉及原生能力、协议变化、关键逻辑的高风险改动不能过度依赖热更新。
3. 热更新解决的是灵活性，不是完整的版本治理。

---

## 五、为什么移动端更需要发布检查清单

因为一旦漏了：

- 可能导致老版本直接报错
- 可能导致桥接能力不兼容
- 可能导致资源缓存和新版本错配
- 可能导致审核或权限问题

常见检查项包括：

- 最低版本兼容
- 灰度配置
- 远程开关
- 桥接能力表
- 核心链路回归
- 指标看板是否准备好

---

## 六、典型场景题

1. 为什么移动端发版机制会反过来影响架构设计。
2. 某功能审核前后能力边界变了，前端如何兜底。
3. 上线后发现严重问题但来不及发版，怎么办。
4. 为什么热更新不能成为默认修复手段。
5. 为什么老版本兼容要在提审前就想清楚。

---

## 七、回答模板

如果面试官问“移动端发布和 Web 发布最大的区别是什么”，建议按这个顺序说：

1. 先讲审核、更新节奏和多版本共存。
2. 再讲为什么这会放大灰度、远程开关和兼容的重要性。
3. 再讲发布前后的检查和观测。
4. 最后讲一次真实的止血或回滚经验。
