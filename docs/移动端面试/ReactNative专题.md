# ReactNative专题

## 使用方式

`React Native` 面试通常会同时考三件事：React 前端基础、原生协同认知、性能边界判断。回答时不要只停留在“能写页面”，要能讲清架构、桥接、列表、动画和原生模块。

---

## 一、`React Native` 是什么

可以把它理解成：

- 业务逻辑主要写在 JavaScript / TypeScript
- 宿主环境仍然是 iOS / Android
- 页面最终不是 DOM，而是原生视图体系
- 前端和原生之间需要通信机制

它的核心价值是复用 React 开发范式和业务逻辑，但它不等于“前端代码直接变成原生性能”。

---

## 二、为什么 `React Native` 会有桥接问题

旧架构下，JS 线程和原生线程之间要通过 Bridge 传递序列化消息。如果通信频繁、数据量大、列表重、动画多，Bridge 就可能成为瓶颈。

所以常见性能问题会集中在：

- 高频通信
- 大对象传输
- 长列表频繁更新
- JS 线程阻塞
- 动画和手势处理不够贴近原生

---

## 三、`Bridge`、`JSI`、`TurboModules`、`Fabric` 怎么讲

### 1. `Bridge`

旧架构核心通信机制。问题在于跨线程、跨语言、序列化和异步通信成本较高。

### 2. `JSI`

更接近直接绑定的调用能力，减少传统 Bridge 的消息序列化开销，让 JS 与原生交互更高效。

### 3. `TurboModules`

更现代的原生模块体系，强调按需加载和更高效的模块调用。

### 4. `Fabric`

新的渲染架构，重点是让渲染链路和 UI 更新更现代化，也更利于性能优化。

高分回答不是死背定义，而是说明：新架构的目标，本质上就是减少旧 Bridge 的通信成本和架构限制。

---

## 四、`React Native` 高频性能题

### 1. 列表优化

要点包括：

- 选合适的列表组件
- 控制单项渲染复杂度
- 避免整列表跟随状态刷新
- 做图片懒加载和缓存
- 把高频状态从大列表里拆出去

### 2. 动画优化

高分回答要体现：动画如果强依赖 JS 线程，一旦 JS 线程忙，就容易掉帧，所以能贴近原生驱动或减少 JS 参与的方案通常更稳。

### 3. 启动优化

要关注：

- JS bundle 体积
- 首屏模块加载
- 原生容器初始化
- 首屏接口强依赖

---

## 五、原生模块怎么答

面试官问“为什么还要写原生模块”，你可以这样答：

1. 某些系统能力只能由原生提供，比如相机、蓝牙、通知、Live Activities、桌面小组件。
2. 某些高性能场景更适合原生承接。
3. 团队需要把宿主已有能力复用给 `React Native` 页面。

高分回答通常会再补一句：原生模块设计要注意协议稳定、错误处理和版本兼容，不要让前端直接依赖一堆不稳定的原生细节。

---

## 六、`React Native` 和 Web React 最大差异

1. 渲染目标不同：一个是原生视图，一个是 DOM。
2. 性能瓶颈不同：一个要看 JS 线程、桥接和原生视图层，一个更偏浏览器渲染和资源加载。
3. 发布模式不同：`React Native` 仍然要面对客户端版本和原生宿主问题。
4. 能力边界不同：`React Native` 更依赖原生协同和宿主容器。

---

## 七、典型场景题

1. `React Native` 列表滑动卡顿，你会先查哪里。
2. 某个功能频繁调用原生能力后页面变慢，如何判断是不是桥接开销。
3. 为什么同样是 React 技术栈，`React Native` 不能直接套 Web 的优化方法。
4. 为什么动画在开发机上还行，低端机掉帧明显。
5. 原生模块升级后，旧版本页面偶发异常，怎么做兼容。

---

## 八、典型追问

1. 为什么新架构不是“自动解决所有性能问题”？
答：因为通信机制变好只是降低了架构开销，真正的性能还取决于页面结构、列表、图片、状态设计和原生实现质量。

2. 为什么 `React Native` 项目仍然要有原生同学参与？
答：因为宿主、系统能力、平台差异、打包发布和部分高性能模块最终都离不开原生工程。

3. 为什么 `React Native` 面试里很常问列表和桥接？
答：因为这两个地方最容易同时暴露前端基本功和跨端边界理解。

---

## 九、回答模板

如果面试官问“你怎么理解 `React Native`”，建议按这个顺序说：

1. 先定义它的运行方式和价值。
2. 再讲 Bridge / 新架构为什么重要。
3. 再讲常见性能边界：列表、动画、桥接、启动。
4. 最后讲原生协同和版本治理。
