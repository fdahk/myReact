# 移动端性能优化

## 使用方式

移动端性能题的关键不是背一串优化技巧，而是先判断问题发生在启动、渲染、滚动、网络还是内存，然后给出证据和治理顺序。

---

## 一、移动端性能和 Web 性能有什么不同

1. Web 更常强调资源加载和浏览器渲染，移动端还要同时面对引擎初始化、设备差异、系统调度和宿主容器。
2. Web 的性能问题往往能快速发版修复，移动端很多问题需要通过灰度、远程开关和长期兼容来治理。
3. 移动端更容易受到低端机、系统版本、网络和后台切换影响，所以不能只看实验室环境。

---

## 二、最常见的性能问题分类

### 1. 启动慢

常见原因：

- 首屏初始化做太多事
- SDK 初始化过重
- 引擎初始化成本高
- 首页资源和组件树过重
- 首页强依赖接口太多

### 2. 滑动卡顿

常见原因：

- 列表项渲染过重
- 无回收或回收策略差
- 图片解码和绘制成本高
- 频繁状态更新
- 布局层级过深

### 3. 页面切换慢

常见原因：

- 路由切换时做了大量同步工作
- 页面销毁和重建成本高
- 动画与数据更新重叠
- 页面间共享状态设计不合理

### 4. 内存高

常见原因：

- 图片缓存和大对象未释放
- 页面栈积压
- 定时器、监听器、订阅未清理
- 长列表、富文本、视频等重组件驻留

---

## 三、启动优化怎么答

### 1. 冷启动慢怎么分析

我会先把启动链路拆成：

1. 宿主进程创建
2. 引擎初始化
3. 配置和资源准备
4. 首屏依赖接口
5. 页面树构建
6. 首屏可交互

然后明确每一段的耗时证据来自哪里：启动埋点、性能监控、真机 profiling、分阶段日志。

### 2. 启动优化常见动作

- 延迟非核心 SDK 初始化
- 减少首屏强依赖接口
- 本地缓存首屏必要配置
- 首页分模块懒加载
- 精简首屏组件树
- 提前准备关键资源，但不要过度预加载
- 把“必须启动时做”的事情和“可以进入首页后再做”的事情分开

### 3. 高分回答要点

不要只说“懒加载、异步化、预加载”，而要说明：

- 哪些动作是首屏强依赖
- 哪些动作是体验增强但不影响进入首页
- 哪些动作可以通过灰度和远程配置动态关闭

## 四、列表和滚动优化怎么答

### 1. 为什么列表是移动端高频考点

因为移动端最常见的卡顿就是列表场景：

- 商品流
- 聊天记录
- 评论区
- 订单列表
- 文件或消息列表

### 2. 列表优化常见思路

1. 控制同时渲染的节点数量。
2. 避免单项组件过重。
3. 图片、视频、富文本做分级加载。
4. 拆高频状态，避免整列表跟着刷新。
5. 做缓存和复用，减少频繁创建销毁。

### 3. 典型追问

1. 为什么虚拟列表或列表回收能提升性能？
答：因为它的目标不是减少总数据量，而是稳定屏幕内真实渲染对象的数量，把渲染和布局成本控制在可预测范围内。

2. 为什么列表项里不要随便做复杂计算？
答：因为列表是高频滚动场景，单项看起来不重的计算，一旦乘上几十上百个可见项，主线程就很容易被拖慢。

## 五、渲染性能怎么答

### 1. `React Native`

重点关注：

- JS 线程负载
- Bridge 或 `JSI` 通信成本
- 列表组件选择
- 不必要的重复渲染
- 动画是否走原生驱动

### 2. `Flutter`

重点关注：

- `build` 是否过于频繁
- `setState` 范围是否过大
- `Widget` 树层级是否过深
- 是否正确使用 `const`、局部刷新和 `RepaintBoundary`
- 图片和自定义绘制是否过重

### 3. 小程序

重点关注：

- `setData` 频率和数据量
- 页面通信成本
- 长列表与大对象传输
- 分包和首屏资源体积

### 4. `WebView`

重点关注：

- 首屏资源和脚本体积
- `WebView` 初始化成本
- 图片和字体加载
- JS Bridge 调用频率
- 容器缓存策略

## 六、包体优化怎么答

面试里讲包体优化，建议按下面几个层级回答：

1. 代码层：减少无效依赖、按需加载、删除历史冗余代码。
2. 资源层：压缩图片、合理使用 `webp` 或矢量资源、字体子集化。
3. 模块层：分包、动态加载、插件化、业务拆分。
4. 平台层：区分基础能力和业务能力，避免所有模块都进入首包。

高分回答要体现一个意识：包体不是只影响下载，还会影响安装、启动、解压、内存和首屏准备。

---

## 七、内存优化怎么答

### 1. 常见内存问题

- 页面返回后未释放
- 大图长驻
- 缓存失控
- 监听器和定时器泄漏
- 消息列表或富文本对象堆积

### 2. 典型治理动作

- 页面离开时释放订阅、监听和大对象
- 图片缓存分级和淘汰
- 列表数据分页与裁剪
- 对大对象只保留必要字段
- 对音视频和复杂组件做生命周期回收

### 3. 高分回答要点

不要只说“注意内存泄漏”，要说：

- 泄漏发生在什么对象和生命周期里
- 你如何发现它
- 你如何验证治理是否生效

---

## 八、监控与验证怎么答

性能优化一定要回答“怎么证明有效”，常见验证方式有：

- 启动耗时埋点
- 首屏耗时埋点
- 页面切换耗时
- `FPS`、卡顿率、长任务数
- 崩溃率和内存异常
- 灰度前后对比
- 弱网和低端机专项对比

如果只说“优化后体感更流畅”，这类答案通常不够强。

## 九、最常见的 10 个高频题

1. 冷启动慢怎么分析和优化。
2. 列表滑动掉帧怎么定位。
3. 页面切换明显卡顿怎么拆解。
4. 为什么移动端很看重包体大小。
5. 为什么低端机表现往往决定优化优先级。
6. `React Native` 列表优化有哪些常见动作。
7. `Flutter` 为什么 `build` 过多会出问题。
8. 小程序的 `setData` 为什么要控制频率和体积。
9. `WebView` 首屏白屏和资源缓存怎么做。
10. 如何证明一次性能优化不是主观感觉。

---

## 十、回答模板

如果面试官问“你做过哪些移动端性能优化”，建议按这个顺序说：

1. 先讲问题现象和影响范围。
2. 再讲你如何定位到具体瓶颈。
3. 再讲短期止血和长期治理。
4. 最后讲结果指标和验证方式。

移动端性能题真正拉开差距的点，不是技巧数量，而是你是否有证据、有顺序、有取舍。
