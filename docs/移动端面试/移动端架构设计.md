# 移动端架构设计

## 使用方式

移动端架构题的重点不是画一张漂亮的分层图，而是解释清楚为什么要这样拆，哪些问题应该在页面层解决，哪些问题应该收口到基础设施层。

---

## 一、移动端架构题在考什么

通常在考下面 5 件事：

1. 你有没有全链路视角，而不是只盯页面。
2. 你知不知道状态、网络、桥接、监控、灰度应该分别放在哪一层。
3. 你会不会为了复用和稳定性做适度抽象。
4. 你能不能控制多端协作复杂度。
5. 你有没有版本治理和长期演进意识。

---

## 二、一个常见的移动端前端分层

### 1. 页面层

负责：

- UI 展示
- 用户交互
- 页面内局部状态
- 业务流程编排

不应该承载太多：

- 重复请求逻辑
- 灰度判断细节
- 日志字段拼装
- 原生桥接底层协议

### 2. 业务领域层

负责：

- 业务模型
- 页面共享状态
- 业务规则
- 场景级流程控制

### 3. 基础能力层

负责：

- 请求层
- 缓存层
- 登录态管理
- 灰度与远程开关
- 监控与日志
- 桥接封装

### 4. 宿主协同层

负责：

- 原生能力接入
- 生命周期透传
- 路由跳转
- 登录态同步
- 支付、分享、推送等系统级能力

---

## 三、为什么移动端更需要基础能力层

因为移动端有三个天然复杂点：

1. 页面不能直接假设网络稳定。
2. 页面不能直接假设宿主能力一致。
3. 页面不能直接假设所有用户都在最新版本。

所以如果不把请求、灰度、日志、桥接收口到基础层，业务页面会很快变成一堆重复逻辑，后续既难排查也难升级。

---

## 四、状态应该怎么拆

移动端项目里常见 4 类状态：

1. 页面瞬时状态：弹窗开关、加载态、输入态。
2. 业务领域状态：订单、会话、商品、用户信息。
3. 会话级状态：登录态、全局配置、灰度信息、实验分组。
4. 宿主同步状态：网络状态、前后台状态、权限状态、原生账号态。

高分回答要体现：不是所有状态都应该塞进一个全局 store，也不是所有状态都应该散在页面里。关键是按生命周期、更新频率和共享范围拆。

---

## 五、请求层为什么要统一

一个成熟的移动端请求层通常要统一处理：

- base url 和环境切换
- token 携带
- 刷新与重试
- 超时和取消
- 错误码映射
- 埋点和 trace
- 灰度字段透传

如果这些逻辑散在页面里，后面你几乎不可能稳定治理。

---

## 六、桥接层为什么要统一

高分回答建议强调：

1. 页面不应该直接拼原生协议细节。
2. 统一桥接层可以做能力检测、参数校验、错误转换和埋点。
3. 统一桥接层更容易做版本兼容和宿主差异屏蔽。

---

## 七、监控和灰度应该放在哪

更好的做法通常是：

- 页面层只上报业务语义和页面上下文。
- 基础能力层负责统一字段、trace、错误归一。
- 灰度和远程开关尽量在应用启动和核心链路入口统一注入，而不是分散到每个页面里判断。

---

## 八、技术栈差异怎么体现

### `React Native`

更关注：

- JS 层与原生层边界
- Bridge / `JSI`
- 渲染和状态更新范围

### `Flutter`

更关注：

- `Widget` 树与渲染树
- 状态管理方式
- 平台通道和混编架构

### 小程序

更关注：

- 逻辑层与渲染层通信
- 分包和平台限制

### `Hybrid`

更关注：

- `WebView` 容器
- JS Bridge
- 宿主登录态和页面栈一致性

---

## 九、典型架构题

1. 如果项目越来越大，你会优先沉淀哪些基础设施。
2. 请求、桥接、灰度、日志为什么不能直接写在页面里。
3. 如何设计一个适合移动端的全局状态方案。
4. 页面层和宿主层的边界怎么划分。
5. 多技术栈共存时，哪些能力应该平台化。

---

## 十、回答模板

如果面试官问“你怎么设计一个移动端前端架构”，建议按这个顺序说：

1. 先讲核心问题：多端、弱网、版本、宿主协同。
2. 再讲分层：页面、领域、基础能力、宿主协同。
3. 再讲为什么请求、桥接、灰度、监控要统一。
4. 最后讲这种架构怎样帮助稳定性和后续演进。
