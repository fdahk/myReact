# AI大模型开发分析及规划
## 技术方向分析
- 背景：
1.前后端工程师，当前为大二本科，希望学习ai大模型开发提升核心竞争力。
2.需要最高效、性价比最高的执行方案。

## 目标与约束（先对齐再动手）
- **目标**：用最少成本，在 \(8\sim12\) 周内产出 2~3 个“可上线/可演示/可写进简历”的 LLM 项目，覆盖 **RAG**、**评测**、**部署/工程化** 三个关键词。
- **约束**：
  - 你目前是前后端工程师（优势是工程与交付），不走纯研究路线，优先走 **AI 应用工程 / LLM 工程化**。
  - **性价比优先**：能本地跑就本地跑；需要云 API 的部分只用在“不可替代”的环节（例如效果对齐、评测基准）。

## 可行方案（3 选 1，也可以混合）
> 这里给的是“路径选择”，不是具体实现细节。你可以把它当成学习策略决策表。

| 方案 | 核心策略 | 成本 | 上手速度 | 适合你吗 | 主要风险 |
| --- | --- | --- | --- | --- | --- |
| **A. API 优先（最快）** | 直接用大厂/开源托管的 LLM API 做应用，重点练工程化与评测 | 中（按量计费） | 很快 | 适合“想快速出作品” | 成本不可控、对模型底层理解浅 |
| **B. 本地优先（最省）** | 用本地推理（如 Ollama/LM Studio）完成主流程，API 只做对照评测 | 低（几乎为 0） | 中 | **最推荐**（性价比最高） | 机器性能限制，模型能力上限偏低 |
| **C. 训练/微调优先（最重）** | 走 LoRA/微调/对齐，做数据与训练工程 | 高（显卡/云算力） | 慢 | 不推荐作为第一条路线 | 容易投入大、产出慢、作品不稳定 |

### 推荐选择
- **推荐**：以 **B（本地优先）为主**，配合少量 **A（API 对照评测）**。
- **原因**：你是前后端出身，“能交付的工程能力”更容易在简历与面试中变现；而本地方案能把成本压到最低，允许你做更多次迭代与打磨。

## 基础知识信息（只学“够用的”）
> 目标是 2 周内达到“能写出可用系统”的水平，不追求把所有理论一次学完。

- **LLM 应用工程核心概念**
  - **Token / Context window**：为什么会“截断”、为什么越长越贵
  - **Prompt 结构**：System / Developer / User，约束输出格式（JSON/Markdown）
  - **Embedding**：把文本变向量，用于检索/相似度
  - **RAG**：检索（Recall）→ 过滤/重排（Rerank）→ 生成（Generate）
  - **评测**：离线集（golden set）、自动评测（LLM-as-judge）、线上指标（留存/命中率）
- **后端工程必备**
  - **SSE/Streaming**：流式输出（对话体验与成本控制都更好）
  - **缓存**：embedding 缓存、检索结果缓存、会话状态
  - **可观测性**：日志/trace（至少把 prompt、检索命中文档、耗时打出来）

## 你的“最高效执行版”学习路线（8 周）
> 默认每周投入 10~15 小时；如果更忙，把每周任务拆成 2 周完成即可。

### Week 0（0.5 天）：环境与工具选型（一次性搞定）
- **本地推理**（二选一）
  - **Ollama**：适合本地快速跑模型、做 API 化调用（推荐）
  - **LM Studio**：偏 GUI，适合先体验
- **后端栈**（选你最熟的）
  - Node.js（Express/Fastify）或 Python（FastAPI）
- **基础依赖**
  - 向量库：先用 **SQLite + 向量扩展** / 轻量向量库；后续再迁移到更专业方案
  - 文档解析：PDF/Markdown/HTML 基础解析即可

### Week 1：把“流式对话”跑起来（工程打底）
- **学习重点**
  - Chat completion 的请求/响应结构（含 streaming）
  - Prompt 模板（system + few-shot + 输出约束）
- **交付物**
  - 一个最小 Chat API（支持 streaming）
  - 一个最小 Web UI（展示逐字输出、停止生成、重试）
- **验收标准**
  - 首 token 延迟、完整耗时、错误兜底（超时/模型不可用）都有日志

### Week 2：做一个“可用的 RAG”（先能用，再优化）
- **学习重点**
  - 分块（chunking）策略：按段落/标题/长度
  - 基础检索：top-k、相似度阈值、去重
- **交付物**
  - 支持上传文档 → 建索引 → 提问 → 返回“带引用来源”的答案
- **验收标准**
  - 答案里必须带引用（例如：文件名、段落/页码、命中片段）
  - Prompt 中明确禁止胡编，检索不到就说不知道

### Week 3：RAG 质量提升（重排 + 结构化答案）
- **学习重点**
  - Rerank（可先用轻量方案：规则/关键词/简单模型；后续再升级）
  - 结构化输出（例如 JSON schema 或固定 Markdown 模板）
- **交付物**
  - 增加“问题改写/多查询（multi-query）”或“查询扩展”
  - 增加“答案结构模板”（结论/依据/引用/不确定性）

### Week 4：加上“工具调用”（让它能干活）
- **学习重点**
  - Function calling / Tool use：把 LLM 变成“会调用工具的调度器”
  - 基础安全：白名单、超时、参数校验
- **交付物**
  - 让模型能调用 2~3 个工具（例如：网页抓取、知识库检索、日程/待办的 mock API）
- **验收标准**
  - 每次工具调用都有 trace：输入参数、输出、耗时、失败原因

### Week 5：建立“评测与回归”（拉开与只会写 Demo 的差距）
- **学习重点**
  - Golden set：自己做 30~100 条问题集（覆盖常见/难例/对抗）
  - 自动化评测：准确性、引用命中、格式正确率、延迟
- **交付物**
  - 一个命令即可跑完的评测脚本（输出表格/markdown 报告）
- **验收标准**
  - 每次改 prompt/改 chunk/改检索参数，都能量化对比

### Week 6：部署与成本控制（真正能上线）
- **学习重点**
  - 限流、缓存、队列、降级（本地模型不可用怎么办）
  - 成本点：embedding 次数、上下文长度、重复检索
- **交付物**
  - Docker 化（或最小可部署方式），带基础监控指标（QPS/延迟/错误率）

### Week 7~8：打磨成“作品集级别”
- **打磨方向（任选 2 个）**
  - Prompt 安全：注入攻击、防越权、敏感信息屏蔽
  - 多租户：不同用户/项目隔离索引
  - 多模态：图片 OCR + 文档问答（可选）
  - 业务化：接入你熟悉的前端场景（搜索、客服、知识库、代码助手）
- **最终交付**
  - 录 2~3 分钟演示视频（非常加分）
  - 写一份 README：架构图、API、评测结果、成本说明

## 作品集建议（做“少而精”）
> 与其做 10 个半成品，不如 2 个能对外展示、能跑评测、能部署的项目。

### 项目 1：Knowledge Base RAG（必做）
- **亮点**：引用溯源 + 评测回归 + 可部署
- **面试可讲**：chunk 策略、检索与重排、如何防幻觉、如何做回归评测

### 项目 2：Tool-Using Assistant（必做）
- **亮点**：工具编排、失败重试、trace 可观测
- **面试可讲**：工具白名单、参数校验、超时、幂等与安全

### 项目 3：Evaluation Harness（选做但很加分）
- **亮点**：把“效果”变成“指标”，体现工程思维
- **面试可讲**：golden set 如何构建、指标如何定义、如何避免评测作弊

## 成本控制清单（性价比核心）
- **默认本地模型推理**：开发与联调全部本地跑，降低每次迭代成本
- **API 预算只用于对照**：每周固定少量额度做效果对齐与评测抽查
- **减少上下文**：用检索结果替代长对话历史；对历史做摘要
- **Embedding 缓存**：同一文档不重复算；失败重试要幂等
- **先做“能用的 baseline”**：不要一开始就追最强模型/最复杂框架

## 你可以直接照抄的简历表述（示例）
- **LLM 应用工程**：实现支持 streaming 的对话服务，集成 RAG（检索 + 引用溯源），提供评测脚本进行回归对比与指标化分析。
- **工程化能力**：实现工具调用（tool use）与全链路 trace，支持缓存、限流、超时与降级，完成可部署交付（Docker/最小化部署方案）。
